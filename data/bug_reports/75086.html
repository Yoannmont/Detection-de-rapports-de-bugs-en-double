Description Jeffrey Liu 2004-09-27 10:41:35 EDT 
OS: WinXP

I have an editor that extends the Eclipse editor framework 
(org.eclipse.ui.editors.*). The way the Eclipse editor framework handles/opens 
a document is by loading the entire document into memory. Furthermore, the 
Eclipse editor framework will sometimes create multiple instances of the same 
document (its String representation). When this happens, the peak memory usage 
go up so high that it may exceed the Java heap limit, as a result, 
OutOfMemoryError. For example, check out the implementation of the 
org.eclipse.jface.text.GapTextStore, when you call the get(int, int) method of 
this class, it returns you a new String instance. I have a scenario that 
requires me to open a 13M document, calling the get(int, int) will increase the 
my used Java heap size by 30M:

GapTextStore store = new GapTextStore();
store.set(<string representation of the 13M document>);
String s1 = store.get(0, store.getLength());  // <--- Java heap increases by 30M
String s2 = store.get(0, store.getLength());  // <--- Java heap increases by 
another 30M, a total of 60M just by calling this method twice...

This is not memory efficient. It would be really nice if there's a better way 
of handling documents that are in the mega bytes range.

Comment 3 Christof Marti 2004-09-29 06:08:22 EDT 
A file with an 8bit encoding on disk will take 2 times more memory due to the
in-memory 16bit encoding of Java (i.e., a char is 16bit). A method that returns
the whole document as a String therefore always has a similar memory consumption
as the one observed.

QuickDiff is of interest in the huge file case, because it also holds a
reference document and creates a copy of the actual and the reference document.
A possible approach to avoid OOME would be to use the idea of the
LowMemoryException that guards the diff algorithm and provide a corresponding
guard for the document copying.

A more general approach to huge files could be to use a TextStore that holds
only part of the file in memory. However, this would slowdown and may require
changes to other parts of the Text framework like the partitioning, the syntax
highlighting, etc. The current implementations assume that random access on the
document is fast, this assumption would no longer hold with a TextStore that
loads/discards parts of the underlying file as needed.

Comment 9 Bob Foster 2005-11-24 16:29:59 EST 
There are several problems here and the two commenters aren't talking about the same ones. The original comment is that document get is inefficient because it copies the entire file. The follow-up comment is that GapTextStore stores the entire file in memory as Java chars, which is up to 2x larger than an 8-bit file on disk and observes that fixing this would be non-trivial.

First, get is inefficient but there is no way to give direct access to the buffer without some sort of locking mechanism. If there were, e.g., a document read lock, it would be possible to return a CharSequence for which individual charAt calls could access the buffer directly. Solving this problem, at least, wouldn't require rocket science.

A related problem is that the absence of synchronization primitives and a reliable modification count that can never go backwards means that background threads processing documents must copy the documents on the UI thread before starting. This is not a new issue, it's just one of the reasons why doing a get-that-copies is necessary today. 

The second issue is actually about the design point. The in-memory representation used is perfectly adequate for source files in most languages, which tend to be relatively small. It doesn't work well (or at all) with 100+ MB XML files, but one might argue that these data files are out of scope for this design.

If the large XML file problem were to be addressed, it would seem reasonable to use an entirely different approach for them, and, yes, this approach would probably have to come up with entirely different schemes for partitioning, coloring, etc. All of these things are doable and have been done before but can't be done by slapping a patch on the existing design.

Comment 15 Anton Leherbauer 2005-12-02 08:37:54 EST 
Now some explanation for the above patch:

Trying to find the source of the memory consumption, I found the following:
1.  first the file is loaded into a StringBuffer (first copy)
1a. in case of Java 1.5, another copy is made through StringBuffer.toString().
    Prior to Java 1.5, the resulting String shared the char[] of the
    StringBuffer. 
2.  the String content is then cloned immediately through GapTextStore.set
    (String.toCharArray)
3.  TextViewer.invalidateTextPresentation creates another copy (later)
    by calling document.get() and "leaks" the copy as long as the document
    is unmodified.
    (The only good thing is that the original char[] can be GC'ed now)

Quick diff - if enabled - will create some more instances of the same content.

I think some of the copies can be avoided by applying a simple copy-on-write strategy, such that as long as the document stays unmodified, a readonly text store is used which wraps the String instance from ITextStore.set(). 
As soon as the document is being modified, the readonly text store 
is replaced with the normal GapTextStore (cloning the content).
Taking into account that many files are opened for viewing only,
this might help reduce the number of in-memory copies.
