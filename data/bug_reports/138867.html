Comment 4 Jim Ruehlin 2006-06-01 19:54:32 EDT 
Comments on MDD Plug-in
Jim Ruehlin	060601

Views
•	Integrate most of the method elements (roles, etc) into the existing views defined in OpenUP/Basic. This makes it easier for people to access your information, especially when they have multiple plug-ins installed. For instance, MDD roles should be in the Team view under the Roles node. It’s OK to add an MDD view that summarizes/explains the MDD plug-in.
•	Use the guidelines in the Plug-in Development Guide for EPF Composer sections 5 & 7 to give the MDD plug-in an integrated look/feel. The user should feel like their looking at OpenUP/Basic with extra MDD stuff, rather than just the MDD stuff.
•	I don’t understand the “MDD Tools” view. These are tool mentors but they don’t appear to reference any specific tools. A tool mentor should describe how to use a specific tool (e.g. Rational Rose or Visio) to achieve a specific goal (e.g. Using Rose to Create a UC Model). Perhaps these are really a different kind of guideline or concept that you want to describe?

Content Packages
•	See section 6.1 of the Plug-in Development Guide for EPF Composer. You may want to think about re-organizing the packaging as described in Option 2 of that section. This will make package selection and integration easier for users of the MDD plug-in. 

Work Products
•	Change Request – you may want to revisit this after the Work Item List (or backlogs – we haven’t agreed on a name) is finalized.

Tasks
•	Avoid creating or naming tasks in a way that describes creating a work product. For example, “Define use case scenario model”, “Define a conceptual model”. Tasks should focus on verifiable objectives that demonstrate progress. For example, “Demonstrate the Architecture”, “Plan the Project”. We should do a better job of this in OpenUP overall too. 
•	There appears to be a lot of tasks. Judging from the names of some of the tasks I suspect they might be better as a step or set of steps in another task (see previous bullet). 
•	OpenUP/Basic defines a Tester role that encompasses all testing tasks, and it looks like some of those tasks are repeated in MDD. You may want to see if they’re the same thing, or if you just want to Extend or Contribute to an OpenUP/Basic tasks instead of creating your own. For instance:
o	OpenUP/Basic: Create Test Cases
o	OpenUP/MDD: Define Test Cases

Roles
•	Test Designer may be redundant. See Tester role in OpenUP/Basic. 

Capability Patterns
•	Capability patterns are used when composing a process and generally should’t be shown on the published website. They’re for packaging activities/tasks that may be re-used in multiple iterations or by other delivery processes. Instead of displaying the capability patterns you should create and display a delivery process (see OpenUP/Basic Lifecycle in the Team view).
•	Prepare for Architectural Modelling is misspelled (should be modeling)
•	You should group your task descriptors under activities. Activities only live in CPs or DPs (delivery processes). This is also causing problems with your activity diagrams (see Diagrams section below).
•	CPs should be organized in sub-packages under Capability Patterns folder. See section 6.2 of the Plug-in Development Guide for EPF Composer.

Diagrams (Capability Patterns)
•	Prepare for Architectural Modelling has an incorrect UML activity diagram. The conditional does not create two parallel threads, which is what’s implied when the arrows grow from the decision point to the merge/join bar. The arrows can go directly from the tasks to the end node.
•	Your activity diagrams don’t show activities, they show tasks. Activity diagrams should always show activities.
•	Activity detail diagrams (ADD) normally don’t appear on the same page as activity diagrams (AD).  You should reach the ADD for an activity by selecting that activity in the diagram. This might get cleared up when you create activities in your capability patterns.

OpenUP/MDD plug-in v02

Here you can find an updated version of the OpenUp/MDD plug-in for EPF taking
into account the comments received. (Thank you for your useful comments, Jim)

Modifications of the openUP/MDD plug-in new version (v0.2):

- Re-organization of the content packages aligning it with base plug-ins
content packages.
- Some work products are now referenced to openUP/Basic plug-in work products.
- Modified some task names and re-organize them into set of steps. There are
now less tasks.
- Some tasks refence now to tasks from OpenUP/Basic plug-in.
- Some roles are extensions of roles from base plug-ins
- Capability patterns have been updated solvin the problem with activity
diagrams.

For the time being we have not define any views, so if someone wants to publish
the content, he/she will build its own views.

So, now, the openUP/MDD plug-in is ready to be again reviewed (mainly the
content) by the community and specially by the MDD SIG !!

Best regards,

Aitor Bediaga
European Software Institute (ESI) http://www.esi.es
