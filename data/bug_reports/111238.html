Description Igor Fedorenko 2005-09-30 14:32:30 EDT 
My bundle com.ibm.tivoli.blah depends on thirdparty bundle that
has the following structure

com.ibm.wsn_1.0.0.jar
  META-INF/MANIFEST.MF
  wsnotofication.jar

and the manifest "Bundle-ClassPath: wsnotification.jar" as well as all relevant
Export-Package. This bundle works at runtime as expected. PDE however, does not
make classes from wsnotification.jar available for compilation of my
com.ibm.tivoli.blah bundle. It would be nice if PDE extracted the nested
wsnotofication.jar into a temporary location
(.metadata\.plugins\org.eclipse.pde.core?) and used it to resolve compile
dependency.

I am using 3.2 I20050923-1000 build.

This is an interesting request in that it would require PDE to extract and keep track of nested jars to be able to put these JARs on the classpath.

Comment 9 Peter Nehrer 2008-03-13 21:57:00 EDT 
For what it's worth, we deal with these cases by first unpacking the jarred bundle ourselves. In the IDE, you can import it into your workspace as a binary project (even if it's only a binary jar and no source). In the automated build, we unjar it when populating our base/plugins directory. I know this is not ideal, but it is a feasible workaround.

I think this packaging mode should be fully supported by JDT/PDE as PDE does allow you to create such bundles -- just group your classes into individual jars, place them on the bundle's classpath (rather than just .) and then export the plugin as a jar. When included in features, these plugins can also be marked unpack="false", which will keep them as jars. Lastly, the platform itself comes with a few of them (even though they don't export any public API, so this is not an immediate problem):

cd eclipse/plugins
for foo in *.jar; do jar -tf "$foo" | grep -q \\.jar && echo $foo; done

:-)

You should continue to place the original bundle jar on the classpath container if (and only if) the bundle has nested jars and '.' on the Bundle-ClassPath.

Bundle-ClassPath: ., codeA.jar, codeB.jar

If '.' is missing then you should not place the original bundle jar on the classpath container

Bundle-ClassPath: codeA.jar, codeB.jar

But please don't forget the case where there is no Bundle-ClassPath.  In this case  '.' is assumed and *only* the bundle jar should be placed on the classpath container.(In reply to comment #11)

> (In reply to comment #10)
> By the way, guys, nested jars are evil, please don't use them unless there's no
> way around it. In my opinion, this is an artifact of stupid crap done during
> the JEE era of development.
> 

For cases where there is absolutely no way around it I think you should simply use a non-jar'ed bundle.  This is by far the easiest way around the issues and it avoids the extra extraction that is needed at runtime (and tooling time) to access the nested jar.
Comment 13 Konstantin Komissarchik 2008-03-14 11:43:38 EDT 
Several observations, if I may:

1. Nested jars in jarred bundles are supported by OSGi and work fine with Equinox at runtime. The fact that that PDE UI and Ant Build cannot handle such bundles is a problem since the people who are affected by this limitation are not necessarily the people who produce the bundles in question. 

2. There is a place for every jar is a bundle approach, but saying that nested bundles are always and necessarily evil is going a bit too far. In a lot of situations (especially in commercial software), creating a bundle out of every third-party jar is an overkill when those jars are only used by one plugin. 

3. The known workarounds of using exploded plugins or merging jar contents into the plugin are just workarounds and should not be considered as a solution for this outage in tooling.
