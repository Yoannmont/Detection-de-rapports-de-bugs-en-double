Description Michael Valenta 2005-11-03 19:17:25 EST 
I would like to have the ability for one content extension to override another.
This would allow me to specify a deafult content extension for a superclass and
then override it for subclasses. See bug 114940 for a larger discussion about this.
Comment 1 Nick Edgar 2005-11-04 09:53:18 EST 
In bug 114940 you describe how getAdapter allows one provider (i.e. an
IWorkbenchAdapter) to override another one, simply via inheritance.

I suggest we follow the same approach for matching CN extensions.
That is:
- find the "lowest common denominator" supertype of the selected objects
- find all matching content extensions
- pick the "best" one (i.e. based on priority, or "I'm better than you"
declarations)

Interestingly, the adapter approach wouldn't work either for the library example
when different types of objects are selected, since there's no common supertype
(other than EObject).
Comment 2 Michael Valenta 2005-11-04 10:25:20 EST 
Yes, you are correct about the adapters. I realized that when I looked at my 
plugin.xml and saw a large number of ResourceMapping adapterFactories for my 
model elements as well. 

For the Common Navigator, it would be nice if I could do something like this:

   <extension
         point="org.eclipse.ui.navigator.navigatorContent">
      <navigatorContent
            id="org.eclipse.team.examples.library.navigatorExtension"
 new>>        groupId="org.eclipse.team.ui.modelProviderGroup"
 new>>        overridesIds="org.eclipse.team.ui.defaultModelProviderContent"
 new>>        isPrimary="true"
            contentProvider="....NavigatorExtensionContentProvider"
            labelProvider="....NavigatorExtensionLabelProvider"
            name="Library Example">
         <enablement>
         <or>         
            <objectClass
                  name="....LibraryModelProvider">
            </objectClass> 
            <objectClass
                  name="org.eclipse.emf.ecore.EObject">
            </objectClass> 
            <objectClass
                  name="org.eclipse.emf.ecore.resource.Resource">
            </objectClass>
         </or>
         </enablement>
       </navigatorContent>
   </extension>

This allows me to state the enablement in terms of common superclasses and 
relies on other means to deal with potential conflicts. Some of those means 
may be:

- A group id that can act as a way to ensure that the proper content providers 
are enabled/disabled.

- An overrides mechanism which, in my case, would state that this extension 
overrides the default one used for Model Provider

- An isPrimary designation that means that this content provider should not be 
used with other primary content providers. Thus, if there was another EObject 
or Resource content provider that was primary, only one can be picked.

The last point is a bit weak at the moment since it doesn't really state how 
conflicts would be resolved but I think there is potential there. One possible 
conflict resolution strategy would be to give higher priority to any content 
provider that provided content to a higher level off the tree. That strategy 
would work in my case (but I am sure there are other cases for which this 
wouldn't help).
Comment 3 Michael D. Elder 2005-11-05 16:10:13 EST 

Please take a look at related comments in bug 114940. 

I had a thought which is similar to your notion of a "groupId" when reading bug
114940. Indeed, the "Function Group" mechanism that is breifly touched on in bug
114940 is the idea of a "groupId" on 'roids. 

I would propose that we support this "override" case by something similar to the
solution used for EMF in WTP (described in bug 114940). I stated in bug 114940
that I didn't think it made sense to try to replicate the DynamicAdapterFactory
mechanism for non-EMF objects; but perhaps we could still take something from
this solution. The key point is the separation of concerns between the framework
allowing extensible and a specific extension that is described to handle the
override cases. 

We could create (as part of the default CN distribution) a special extension
that was meant to be overridden. It would read another extension point (say
overridableNavigatorContent), and use some scheme to determine which one it
should delegate too (port down Function Groups to Platform? ;). Then for cases
where there's a Base:ValueAdd relationship that wants tight integration, the
"Base" provider would support the basic case, and then the "ValueAdd" provider
would support the value-add case. The solution might look like:

   <extension
         point="org.eclipse.ui.navigator.navigatorContent">
      <navigatorContent
          enabledByDefault="true"
          name="Library Example"
          id="org.eclipse.team.examples.library.navigatorExtension"
          actionProvider="org.eclipse.ui.navigator.OverrideableActionProvider"
          contentProvider="org.eclipse.ui.navigator.OverrideableContentProvider"
          labelProvider="org.eclipse.ui.navigator.OverrideableLabelProvider"
          >
         <enablement>
         ...
         </enablement>
       </navigatorContent>
   </extension>

   <extension
         point="org.eclipse.ui.navigator.overrideableNavigatorContent">
      <overrideableNavigatorContent
          enabledByDefault="true"
          name="Library Example"
          id="org.eclipse.team.examples.library.navigatorExtension"
          actionProvider="org.eclipse.team...ActionProvider"
          contentProvider="org.eclipse.team...ContentProvider"
          labelProvider="org.eclipse...LabelProvider"
          priority="NORMAL" ?
        
          > 
          <enablement?>
              ... project nature case?
              ... classpath container id case?
              ... state tester? (at this point we might as well bring down
Function Groups from WTP
          </enablement?>
       </overrideableNavigatorContent>
   </extension>

Execution flow: 
(1) The OverrideableContentProvider with id
"org.eclipse.team.examples.library.navigatorExtension" is invoked to provide
content for some object specified in its enablement clause. 
(2) The OverrideableContentProvider seeks the correct override extension. One
override extension must be specified or the OverrideContentProvider would
contribute no children. If it finds one case, it will load it, and delegate to
that content provider (this is your default Model provider case). 
(3) When a higher priority/more specific enablement/(dare I say "Function Group"
determined) extension is found, the
OverrideContentProvider(ActionProvider/LabelProvider) will load that extension
(on a case by case basis -- perhaps only the ContentProvider is overridden to
show a different structure, but the same objects are contributed so there's no
need to override the label provider). Remember that since it will look like the
same Content Extension to the CN framework (since it's really one extension that
varies its behavior dynamically), any enhancements like "using the same label
provider (see bug 115218)" for the contributed content would still work, even if
it was overridden). 

I think this approach provides a solution to your problem without
overcomplicating the Common Navigator framework or basic extension point. It
also requires clients to explicitly designate their content as "unique"; so
clients would make the decision (in cases like yours). I'd like to avoid a
situation where Platform teams (aka JDT) take this approach because I think that
alot of teams are going to want to integrate with the Java content in the tree
(instead of 'override').

Comment 7 Michael D. Elder 2005-11-07 12:19:26 EST 
There are two things to consider here. From my viewpoint, separating the
"override vs. aggregate" from the core framework will actually make
implementation much easier. Adding more attributes and logic to the
navigatorContent extension point means adding more possible variations for
clients. I'd like to keep the basic navigatorContent extension point simple. 

Introducing the two new attributes: 'groupId' and 'overridesIds' means that
there's extra complexity to deciding when an extension is invoked. 

The 'groupId' implies a general "Category" concept. The FunctionGroup framework
in WTP takes this concept and turns it into a generic, re-usable framework, so
that you don't need to implement multiple flavors multiple times. Since there's
been no decision to move this framework down to WTP, the navigator would have to
implement its own version of it. 

The 'overridesIds' attribute means that there's some sense of extension B
overriding extension A, so it would require more coordination when reading
extension points and calculating enablement. Since the extension is still
showing an enablement clause, it means that B may only override A in certain
cases. That means we'd take B, determine if its enabled for some object *before*
we test for A, then if it is enabled, we don't test A and otherwise, we'd test
A. This means more logic for checking enablement for all extensions, even for
the basic case (and I think this is the more widespread case -- at least for all
current clients). Now what happens when there's B and a new C overriding A?
Using isPrimary=true isn't enough information to determine which one is the real
'primary' content provider. I'm also hestitant because I don't really know what
a 'primary' content provider is. Is it the Content Provider that contributed the
object? Or is it a new extension which claims its isPrimary for that enablement?
If it's always the extension that contributed the object, then it it could never
be overridden. 


The 'groupId' also means more logic to determine which extensions are turned on
or off. How would we coordinate an extension or set of extensions for a viewer?
Could I override the content provider, but only in one of the views that uses it
in your scheme? In the scheme I described, this capability is not readily
available, but the OverridableContentProvider could be easily modified to
support it. This was one of the big value adds for Function Groups since they
worked across all types of extensible frameworks, and they could be customized
in specific cases (like I want it to show structure (1) in the navigational
viewer, but structure (2) when shown in the editor).

The key thing to keep in mind is that the proposal that was described has
already proven effective in previous versions. Keeping the the logic simple for
identifying the correct extension(s) to invoke for children is my top priority
so that (1) it's easier to debug and (2) it does as little as possible for
performance concerns. I'd rather introduce a solution where a client must
explicitly decide they have a case where they want "replace" semantics instead
of "aggregate" semantics. This will also prevent extensions from completely
subverting existing extensions that did not want to be overridden (read: JDT/UI).
Comment 8 Michael Valenta 2005-11-07 15:25:19 EST 
The overrides and groupId (or type) cases don't strike me as being overly 
complicated. They're just filters on the set of matching content providers. 
Naively, when displaying the content for an object, the Common Navigator 
Framework could do the following:

1) collect the list of all content providers that match
2) remove any that are overridden by others in the list
3) if group filtering is enabled, remove any that are not in an enabled group
3) get the children for object from the remaining content provider.

I agree that the isPrimary is more complicated but I think is is important, 
especially when considering operations that are being performed on a parent 
node in a tree vsible in the common navigator. Since the operation is 
performed at the model level, some visible children may be excluded from the 
operation (and some invisible children of the model will be included). The 
user needs to understand why some of the children are included in the 
operation and some are not. We should get together to discuss this.
Comment 9 Michael Valenta 2005-11-07 15:46:47 EST 
I think it worthwhile for me to step back a bit here to clearify what is 
driving this. We want to use the common navigator in the context of team 
operations. We have the following requirements for this:

1) Content providers cannot be used in a Team operation unless they know about 
the extra team state we are passing in the IExtensionStateData. For example, 
if there are Java elements being committed, we only want the Java content 
provider to be active if it is somehow flagged as team-aware.

2) In the case where a model's content provider is not team-aware, We need to 
have a default content provider to render that content (basically to show the 
resources instead of the model objects).

3) For models who are team-aware, we don't to force them to have multiple 
content provider definitions.

You can see from this how I arrived at the override and group concepts. I am 
open to alternatives. If you feel that Funtion Groups is the way to go then we 
can meet to discuss that as well.
Comment 10 Michael D. Elder 2005-11-07 17:53:56 EST 
I think this is a good discussion. The reason why I favor splitting the
responsibility from the framework and providing a specialized "overrideable"
provider is demonstrated by your example. If we take a look at the steps:

1) collect the list of all content providers that match
2) remove any that are overridden by others in the list
3) if group filtering is enabled, remove any that are not in an enabled group
4) get the children for object from the remaining content provider

With an overridable content provider, the steps would end with #1. In the case
of groupId and overridesId, we would need to perform an O(N^2) + O(N) operation
for each (compare every overridesId to every extensionId and search all defined
groupIds and check their enablement). These operations are likely to be
comparisons of arbitrarily long strings. 

This scheme also allows clients that didn't want to be overridden (read: JDT/UI)
to be easily wiped out in a way that users cannot change (by specifing the
overridesId and a defunct content provider). You also have to be careful of the
ordering of #2 and #3. The overriding extension could be disabled independent of
the 'logical' group. So then does it still override? It also pops the question
of what to do in the "Types of Content" dialog: show one enablement check for
the logical group or show a tree of enablement checks for 'logical groups'? Can
groups be arbitrarily deep? Can you override an overriding extension? What if
the overriding extension specifies a different groupId? If it specifies a
different group id, which group id determines if its still eligible to override
the existing content provider? 

By dividing the responsibility, I think I can give you want you want; and won't
require other clients to risk (1) being overridden and (2) searching overrideIds
for _every_ call to getChildren(). Instead, the determination for the correct
override will only be in cases where overridition (real word?) will occur. What
I'm suggesting isn't radically different from what you're requesting. The
difference is simply in how the architecture is structured. If you isolate the
override capability to a single specialized, reusable, API content extension,
then you don't need a groupId -- because turning that extension on and off
automatically turns on/off all "overriding" extensions. The "overridesId" is
specified as part of the overrideableNavigatorContent (as just "id"). But all of
the details of how the policy is applied is isolated to just one extension; and
if that extension (OverrideableContentProvider) isn't enough, clients can tap in
their own overrideable schemes for specific technologies (read:
DynamicAdapterFactory for EMF).

If we go with the separation approach, we could use Function Groups to provide
arbitrarily complex override logic that's extensible and not tied to the
Navigator or its extensions directly. The OverrideableContentProvider would use
Function Groups to determine which provider should be used. Clients could decide
how to structure their FunctionGroupInitializer (state tester for the project)
and thus different clients could use different logic to determine how overrides
should be applied. (read: maximum extensibility ;)
