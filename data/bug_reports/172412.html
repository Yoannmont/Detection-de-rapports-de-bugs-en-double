Description Craig Salter 2007-01-31 23:42:03 EST 
Currently within EMF and WTP (and I suspect the whole of Eclipse) we've got a pretty bad story for URI support.  There's several issues with the java.net.URI class that motivates us to create our own class (performance is one issue, but having the flexibility to control the implementation, fix problems and add convenience methods in another).  

Because there's no URI class available in Eclipse platform we're left with various classes within various plugins (since generally plugins don't want to depend on any other plugin that's not the platform).  We've also got little utilities scattered thru our code and it'd be really nice to consolidate these into a single well documented, well tested class that's available once and for all in the platform.

For example, I don't know how many places I've seen snippets of code that convert file locations to URIs (i.e. "C:\foo" to "file:///C:/foo") or fudge the slashes to convert "file://C:/foo" to "file:///C:/foo" (yes the extra slash makes a difference).  

So this bug is intended to track an effort establish such a URI class and promote it to the Eclipes platform. I think a great starting point would be the EMF project's org.eclipse.emf.common.util.URI class.  I'll be happy to work with the EMF, WTP and Eclipse platform to persue this.  I'd appreciate any comments from the Eclipse platform regarding the viability of this request.

Comment 3 John Arthorne 2007-02-01 14:09:03 EST 
I suspect in some cases people wrote their own URI classes before one was available in the JRE.  For example, EMF was once based on JRE 1.3 or earlier. 

For what it's worth, I'll summarize the evaluation of URI that I made while writing the core.filesystem plugin, which makes significant use of URI.

On the "con" side, I generally don't like the idea of writing such a complex piece of code when it's readily available in the JRE,  particularly in the case of a very complex spec like URI.  I flirted with the idea while writing the filesystem plugin, but after a reading of RFC 2396 I realized how abstract URI was and decided I couldn't do much better than the java.net implementation.  Also keep in mind that with java.net.URI you will get some "free" improvements over time as bugs are fixed across releases.

Another downside is that introducing another URI implementation would not actually simplify the overall URI story.  APIs that already use java.net.URI or EMF's URI would not be able to break, so those components would be stuck with what they have.  This inevitably leads to conversion between URI forms at API boundaries and when interacting with JRE classes. I.e., adding a new URI implementation would not allow much consolidation for existing code, but does provide a new option for new components.

On the "pro" side, as Tom mentioned, those with a requirement to support Foundation 1.0 class libraries can't use java.net.URI. I think in many cases this can be addressed by moving bundles up to Foundation 1.1, which is still quite lean but includes java.net.URI.  This is likely not an option within the OSGi framework itself, but certainly viable in the RCP area and above.

Also on the pro side, it's certainly true that java.net.URI lacks convenience methods, making it a pain to work with. Most manipulation of URIs involves manipulation of the path segment of hierarchical, absolute URIs, for which I used IPath in the filesystem plugin. This does mean going from URI->IPath->URI, which is a pain and creates lots of garbage. Other than the path segment, there really isn't much generic manipulation you can do without making assumptions about specific schemes.

Another small "pro" is that java.net.URI is not space-efficient, and a new implementation could perhaps improve on this.  In general this will be a speed/space tradeoff, since java.net.URI stores encoded and raw representations of all URI segments as separate strings. This makes accessor methods fast at the expense of space.  On the other hand you can easily use the external String form for storage, and only create URI objects when further parsing is required (this is my approach in the filesystem plugin, where I hang onto very few URIs in memory and just create them where needed).

In the end, I did decide to go with java.net.URI, and have found it to be acceptable for my particular needs.  Others might conclude differently...

Comment 7 John Arthorne 2010-10-15 09:34:13 EDT 
We now use URI quite extensively in the platform with the following strategy:

 - Use java.net.URI at API boundaries, and as a mechanism to validate inputs since it is very pedantic (and correct) in its adherence to the URI spec. This facilitates API compatibility because everyone has access to java.net.URI
 - Use String if you need to hold onto a large number of URI instances in memory for long periods. This is the most space-efficent representation (I suppose you could go with char[] in extreme cases)
 - Use URIUtil in org.eclipse.equinox.common for URI manipulation.

As such we don`t intend to introduce yet another URI class in the platform.
