Description Walter Harley 2005-06-07 15:06:01 EDT 
It would be nice if a plugin could say what JRE it requires, as well as 
possibly other platform dependencies, and fail gracefully if the requirement is 
not met.

Specifically:  I would like to see the plugin manifest support an optional 
attribute named something like "minimumJreVersion".  At plugin load time, if 
the attribute is present, the platform would check it against the current JRE 
version, and if the version was insufficient an error would be reported and the 
plugin would fail to load.

Background:  Eclipse 3.1 can be compiled and run under JRE 1.4.  However, I am 
working on a plugin that for technical reasons requires JRE 1.5.  If I run 
under 1.4, the plugin fails to load with a cryptic ClassNotFound exception.  
Writing a "shell" that allows my plugin to load and fail gracefully under 1.4 
is not technically practical (in part because my plugin needs to expose Java 5 
dependent APIs to downstream plugins).

More persuasion:  It should be clear that this is not just a Java 1.4 vs. Java 
5 issue; since the OS/language will always be moving targets, and various 
developers and customers adopt new platforms at different rates, there will 
always be the possibility of plugin/platform mismatch.  Anything we can do to 
help users understand why their plugin isn't loading will be a help.
Comment 1 Pascal Rapicault 2005-06-07 15:17:42 EDT 
The closest thing we have for now is the ability for a Manifest to import
packages from the JRE. Then at runtime if the package is not exported (because
you are not running against the proper JRE), the bundle will not get resolved.
Comment 2 John Arthorne 2005-07-20 14:50:04 EDT

Comment 3 John Arthorne 2005-08-22 15:38:34 EDT 
A possible solution to this is to define a well known bundle namespace that
represents the JRE (such as "org.eclipse.jre").  On startup, the Eclipse runtime
would set the version of this fake JRE bundle to match the version of the
currently executing VM.  This would allow plugins to specify JRE dependencies
simply as another bundle pre-requisite. For example, adding this line to the
bundle manifest would specify that the bundle requires a 1.4 level JRE, but
cannot run on a 1.5 JRE:

Require-Bundle: org.eclipse.jre;bundle-version="[1.4,1.5)"

An open question is how various class library profiles would be represented
(foundation, j2me, j2se, etc). Configuring the JRE bundle to automatically have
the correct package exports for a given class library profile is likely more
difficult than just fixing the version number.
Comment 4 Walter Harley 2005-08-22 19:42:40 EDT 
The goal is to come up with a clear error message when a platform requirement 
is not met.  I am concerned that the "org.eclipse.jre" approach might not 
achieve that?  I don't know if "missing bundle org.eclipse.jre" will help a 
user figure out what to do.
Comment 5 Rick Warren 2005-08-22 20:28:56 EDT 
Eclipse already checks for JRE 1.4.1; it would be nice if plug-ins could simply
leverage the same check that already exists. Instead of checking a hard-coded
version, any interested plug-in could register an extension stating which
versions it supported. The runtime would then compare the running JRE version
against the registered extensions instead of against the hard-coded value.

That said, I don't particularly care which implementation technique is chosen.
I'm in the same boat as Mr. Harley and would like to see *some* solution as soon
as possible. It was hard enough as a developer to guess what the cause of the
ClassDefNotFoundError was; I'm not sure how an end user or a customer support
person is supposed to debug the problem.

I am currently working around this missing feature by performing the version
check myself in a small plug-in that extends org.eclipse.ui.startup. I am well
aware that my extension is both somewhat frowned upon and can be switched off by
the user, so I am eager to have a more fully supported solution.
Comment 6 Pascal Rapicault 2005-08-22 21:45:34 EDT 
I think the solution to this problem is two fold.
One part of it sits in the plugin, which needs to indicate that it is compiled for 1.5 and/or uses some 
1.5 APIs and another part sits in the feature.xml where you would like to prevent a feature (or a plugin) 
to be installed if 1.5 is not present.

For the plugin part: you need both a platform-filter (or the corresponding osgi 4.0 mechanism if one 
exits) to capture the fact that it is 1.5 bytecode, and then if you are working with subsets of the JRE, 
you need to import the packages you use (the JRE packages being automatically exported) (this last part 
being optional).

For the feature part: it is unclear if this is a filter like os / ws / arch or something at the top level of the 
feature.
Comment 7 John Arthorne 2005-08-23 09:30:12 EDT 
Re comment #4: I understand the requirement, I was just outlining part of a
possible solution.  In the context of bug 99393 we also need to be improving
error messages when plugin version constraints are not satisfied (at
install/update time and/or at startup).  I would anticipate that even without
special-casing the JRE bundle dependency we could get a message like:

"Unable to [install | start] 'Java development tools' because it requires
version 1.5 of 'Java runtime environment' but only version 1.4 is available.

In any case, as Pascal says there are two parts to the problem.  One, there
needs to be a mechanism for bundles to specify their JRE requirements, and two
this needs to be leveraged at install/startup to improve the user experience.  I
was only digging into the first part of the problem.
Comment 8 Thomas Watson 2005-08-23 09:47:39 EDT 
re comment 6:

There is an OSGi mechanism (introduced in R3).  The bundle header 
Bundle-RequiredExecutionEnvironment can be used to specify an execution 
environment.  But this mechanism is very brittle and does not allow for you to 
specify any types of ranges.  For example you can specify

Bundle-RequiredExecutionEnvironment: J2SE-1.4, J2SE-1.5

This specifies that your plugin will run on J2SE 1.4 and 1.5 but then when 1.6 
comes out it will no longer work.  Also this is an install time check.  If the 
require execution environment is not present then the bundle will fail to 
install.  You may successfully install the bundle when running with one JRE.  
But then you could restart the platform on another JRE.  The framework does 
not check to make sure the existing bundles installed match the current 
execution environment.

The platform-filter does not currently support any java properties outside the 
following:

osgi.nl 
osgi.os 
osgi.arch
osgi.ws

We would have to add some additional support for platform-filters to be able 
to specify JRE versions

Import-Package is good for new packages that appear in newer versions of a 
JRE.  But in many cases new classes or methods are added to existing 
packages.  Simply using Import-Package to make sure a package is present will 
not work in many cases.
Comment 9 Rick Warren 2005-08-23 12:22:41 EDT 
I think there are a couple of different deployment scenarios to keep in mind
here. In the case of a reasonably advanced user who is comfortable installing
and managing a variety of plug-ins that serve a variety of purposes, it may be
acceptable to simply not activate those plug-ins that aren't compatible with the
current runtime environment (after, of course, informing the user of what you're
doing). I think most current users of Eclipse as such probably fall into this
category.

However, the case I am primarily concerned about is somewhat different: my end
users will install a standalone, independently branded Eclipse-based product.
Most of them will likely never have heard of Eclipse and won't know or care that
there is a plug-in mechanism under their application's hood. An error message of
the sort "The feature xxx could not be started...." that then leaves them in an
environment in which 90% of the functionality they care about is inexplicably
(from their POV) missing from the UI is not user friendly. Currently, if you try
to start Eclipse with a pre-1.4.1 JRE, you will get a message that the JRE
version is insufficient and then the application will quit. I would like to be
able to get exactly this current behavior, except that I would like to check
against JRE 1.5.0 instead of 1.4.1.

I understand that solutions such as are proposed in comment #6 and comment #7
may be necessary to solve this bug, and may even be sufficient for some classes
of users. But I just want to make sure that whoever works on this understands
that it may not be sufficient for all classes of users.
Comment 10 Pascal Rapicault 2005-08-23 13:36:37 EDT 
Well, if you think saying my Mom that she should run JRE 1.5 instead of 1.4 is
something helpful to her, then you can specify in the config.ini 
  osgi.requiredJavaVersion = 1.5
and you will get the same error message than you have today when you try to run
eclipse with a 1.3 VM. This function is already in the 3.1 code base.

Note that in the case of very controlled environment like an RCP app, you can
probably get a better error message by doing the check in your workbench advisor
and opening a nice swt dialog.


That said this approach only work in case of very controlled environment and
like you acknowledge we still need to have a mechanism to disable plugins based
on the JRE (and of course provide a helpful message).
Comment 11 Jeff McAffer 2005-08-26 05:23:05 EDT 
I think we should look at using he standard OSGi mechanism and extending it to 
work around the problems Tom outlined in comment 8.  There are two related 
challenges. 
1) no ranges
2) no notion of subsetting.

Tom covered 1 but even if we could spec a range, there needs to be a way of 
enumerating the points in the range such that in [A, B, C], B is a superset of 
A.  This is not obvious from JRE version numbers since we mix in things like 
Foundation 1.0 (soon 1.1), OSGi's minimu EE 1.0, JRE 1.1, personal 
profiles, ...  All of these are called out in the OSGi spec.  

Given an root EE, it is unclear if we would ever need to spec an upper bound.  
For example, if a bundle works on J2se 1.3, would there ever be a case when it 
does not work on say Java 6?  Seems unlikely.  Assuming that, we can work on 
adding a '+' to the end of a EE spec to mean the given EE and anything greater.

Bundle-RequiredExecutionEnvironment: CDC-1.0/Foundation-1.0+

This still needs a total ordering of the EEs.  The ordering scheme has to 
somehow allow for the insertion of new EEs over time (e.g., Foundation 1.1)

Another challenge will be to do this in a compatible way so that it does not 
break other OSGi implementations and such that we can feed it back into the 
spec process for R5.
Comment 12 Thomas Watson 2005-08-26 10:15:55 EDT 
I like the direction of comment 11.  Adding BJ for his comments ...

A couple of points.

- Should we consider checking EE at startup and resolve time instead?  Right 
now it is an install time check which causes a bundle to fail to install.  It 
really seems like the bundle should fail to resolve instead of failing to 
install.  If the EE changed from last startup and a bundle does not support 
the new EE then it should not resolve.

- For J2ME profiles do we need to support a '+' in two places.  One for the 
configuration version (i.e. CDC-1.0+) and one for the profile version (i.e. 
Foundation-1.0+).  I think the syntax could look something like this:

Bundle-RequiredExecutionEnvironment := ee-spec ( ',' ee-spec )*
  ee-spec := ee-name ( '-' version ( '+' ) ) ( '/' ee-spec )*

- Is there really a need to specify a '+' to indicate greater or equal?  If 
the next version of a profile is always backwards compatible maybe we can 
always use greater or equal even without the '+' char.
Comment 13 BJ Hargrave 2005-08-26 10:45:42 EDT 
When using the Bundle-RequiredExecutionEnvironment, a bundle should declare
something like:

Bundle-RequiredExecutionEnvironment: J2SE-1.4

This means the bundle needs (at least) J2SE 1.4. 

At runtime, the org.osgi.framework.executionenvironment property should be set
to ALL of the possible EE names supported by the JRE. So if the JRE is J2SE 1.5,
the value of the org.osgi.framework.executionenvironment property might be:

org.osgi.framework.executionenvironment=OSGi/Minimum-1.0,OSGi/Minimum-1.1,JRE-1.1,J2SE-1.2,J2SE-1.3,J2SE-1.4,J2SE-1.5

which assumes that all of these are subsets of J2SE 1.5.

The point here is that the bundle/plugin only needs to declare his requirements,
i.e. the minimal EE needed, rather than an exhaustive, open ended list. The
framework then has the larger burden of deciding if the current JRE satisfie the
EE requirements of a bundle.
Comment 14 Thomas Watson 2005-08-26 10:59:33 EDT 
hmmm ... ok so this looks like a framework bug.  We currently only set that 
property automatically to actual execution environment of the running JRE.  We 
would have to map that value to all the proceding EE's that it supports.

It can be set in the config.ini to the correct full list of EE's but that is

Comment 15 Jeff McAffer 2005-08-28 21:25:40 EDT 
comment 13 is along the lines of what I was thinking in comment 11 when 
talking about declaring the subsetting relationships.  It seems a bit brutal 
to list them all out like that but it certainly is clear.

One issue is that the framework impl has to know about the JREs.  For example, 
say next week the Foundation 1.1 guys ship.  Existing OSGi implementations do 
not know about Foundation 1.1 so people who want to write bundles that use new 
features of that classlib have no way of specing that dependency.  They have 
to wait for a framework patch.  

A smart impl would externalize this information in a profile file (hint hint :-
) so that users can extend it when new class libs come along.

So we can embed some knowledge of the libs but should allow the profiles to 
override.
