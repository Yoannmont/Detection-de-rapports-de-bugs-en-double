Comment 2 Peter Moogk 2007-08-17 16:51:19 EDT 
Hi Lahiru,
   I reviewed the patch that attached for this defect.  I noticed following code snippet:

public class DefaultCodegenUtil {
	
	private static Model model;
	private static List serviceQNameList = null;
	private static IStatus status;
...

In general having a static field in a class is bad from a memory point of view, since this will lead to memory leakage. (ie. the memory is not recovered when the wizard completes.)

I believe Kathy mentioned that you should use the WSDL parser for this defect.  To get this parser simply call:

WSDLParserFactory.getWSDLParser().

This Factory class uses a static, however, the wizard always frees up this static object when it completes.  This class is in the org.eclipse.jst.ws.internal.consumption plugin.

To used the parser just call:

  public Definition getWSDLDefinition(String wsdlURI)

Note: the WSDL parser caches the WSDL definition for a particular wsdl uri while the user is using the Web services wizard.  This way the WSDL only gets parsed once.

Please, make the changes mentioned above and resubmit the patch.

Thanks.

Comment 7 Peter Moogk 2007-08-21 12:06:17 EDT 
Hi Lahiru,
   Thanks for your comment #6 clarifing things.  I'm still unclear, however, on a few items.  

   You said that you were using the static to enable inter wizard caching of the Definition.  Did you mean caching in the Skeleton Wizard where the user can configure both the skeleton and the client.  Or did you mean caching between one invocation of the skeleton wizard and a separate invocation of the client wizard?

    As a general rule we try to free up most if not all the memory used by the wizard when the wizard is closed.  Data stored in non-static fields will automatically be freed up when the wizard closes.  However, data referenced via a static field will not be automatically freed up when the wizard closes.  This memory will only be freed up when the Eclipse shuts down.  This is why we try to avoid as much as possible the use of static fields.

    In the case of our WSDLParser it caches the WSDL defintion for the time that the wizard is open.  The framework and extensions can then access one parsed version of the cached WSDL.  When the wizard is closed by the user clicking cancel or finish the WSDLParser's cached memory is explicitly freed.  When the Web services wizard is launch again, the WSDL files are parsed and cached again as required.  Even though this has the potential for extra parsing we have found that it is preferable to the memory loss.  Also, the WSDL file may get changed between invocations of the wizard so we are always sure to use the proper version of the WSDL file by parsing it at the beginning of the wizard.

    I hope this answers your questions.  Thanks.

Comment 19 Peter Moogk 2008-06-17 12:04:40 EDT 
Hi Lahiru and Saminda,
   I noticed that Axis2 has it's own version of WSDL4J.  I suspect that the reason you are running into the linkage errors is that sometimes javax.wsdl.Definition is being loaded from your special Ant classloader and sometimes it is being loaded from the Eclipse classload.  Even if though the classes may be identical if they loaded from two different class loaders they will never work together.

   Given that there will probably always be this collision of Eclipse WSDL4J and AXIS2 WSDL4J, I think the solution that I recommended of using our WSDL parser is probably not workable.  Therefore, I think we need to come up with another solution to this problem.  

   In the original defect Kathy indicated that she saw that the input WSDL file was parsed twice.  Perhaps you can just save the WSDL definition for a bit in your model so that it doesn't need to be reparsed.  But make sure that there are not static fields used to save this data.  We don't want this data to be persisted after the Web services wizard is closed.  Thanks.
