Comment 2 Nigel Wade 2004-04-28 12:18:20 EDT 
The problem occurs when using the Find/Replace dialog (i.e. ctrl-f). 

The platform is Redhat FC1. I have gtk+ 1.2.10, gtk2 2.2.4. Eclipse seems to be
linked to /usr/lib/libgtk-x11-2.0.so.0 which is from gtk2.

The problem seems to be related to the Find/Replace dialog box and the X
selection/cut buffer. Opening the dialog box selects whatever text was last
searched for in the Find: field, and doing a paste at this point inserts what is
in that field. Closing the dialog box restores the normal paste text, as does
clicking in the Find: field so that text is no longer selected. If I select text
outside of Eclipse, for example in a terminal window the same happens, that text
is pasted as long as the Find/Replace dialog box is open.

Comment 11 Billy Biggs 2004-10-27 15:48:56 EDT 
This seems to be platform behaviour for GTK+ entry boxes.  The following program
kills the selection buffer just by running it.  If I select some text in an
xterm, go back to this window and hit tab, the automatic select-all also kills
the selection buffer.

This behaves differently from Mozilla Firefox, which only changes the selection
buffer when you explicitly select text, and not when it selects text for you or
by tabbing into an entry.

Bug 63579 is related in the sense that it seems StyledText no longer pastes from
the selection buffer, only from the clipboard (what you paste into by hitting
Ctrl-C and Ctrl-V).  I think this bug should be marked as WONTFIX.


int main( int argc, char **argv )
{
    GtkWidget *window;
    GtkWidget *vbox;
    GtkWidget *entry;
    gtk_init (&argc, &argv);
    window = gtk_window_new (GTK_WINDOW_TOPLEVEL);
    vbox = gtk_vbox_new (FALSE, 0);
    entry = gtk_entry_new ();
    gtk_editable_set_editable (GTK_EDITABLE (entry), TRUE);
    gtk_entry_set_text (GTK_ENTRY (entry), "Some text");
    gtk_editable_select_region (GTK_EDITABLE (entry), 0, -1);
    gtk_container_add (GTK_CONTAINER (vbox), entry);
    entry = gtk_entry_new ();
    gtk_editable_set_editable (GTK_EDITABLE (entry), TRUE);
    gtk_entry_set_text (GTK_ENTRY (entry), "Some text - two");
    gtk_container_add (GTK_CONTAINER (vbox), entry);
    gtk_container_add (GTK_CONTAINER (window), vbox);
    g_signal_connect (G_OBJECT (window), "destroy",
                      G_CALLBACK (gtk_main_quit), NULL);
    gtk_widget_show_all (window);
    gtk_main ();
    return 0;
}
Comment 12 Veronika Irvine 2004-10-27 15:49:39 EDT 
SWT is not setting the data into GtkPrimaryClipboard when you call 
Text.selectAll().  This is being done by the GTK widget when 
gtk_editable_select_region is called.  There was a thread about this on 
mail.gnome.org:

http://mail.gnome.org/archives/gnome-devel-list/2001-May/msg00160.html

Seems like Owen is saying that gtk_editable_select_region is supposed to wipe 
out the primary clipboard and the user is supposed to paste with Ctrl+V into 
the Find dialog.  I have not seen anything that says that programatic 
selection should be different from user selection as far as the Primary 
clipboard is concerned.
Comment 13 Veronika Irvine 2004-10-27 15:53:33 EDT 
Marking this as a duplicate of 63579.  I believe this bug is now fixed in 3.1 
because the Clipboard is referenced rather than the Primary Clipboard when 
pasting with Ctrl+V.
