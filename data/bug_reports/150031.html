Description Gilbert Andrews 2006-07-07 14:50:13 EDT 
I have attached an example of an wsdl with an inline schemas that throw warning messages to the user about not being able to resolve schema locations even though they could be resolved. 

here is a note I sent Ed Merks about the current way we deal with this:

When we go to gather the xsd imports for the wsdl we see the following objects in the xsdchema contents. 

org.eclipse.xsd.impl.XSDImportImpl@62edc6

It turns out the above import is actually a XSDSchemaDirective
here is the snippet of code we are using to resolve the import
 

if (content instanceof XSDSchemaDirective)
        {
          XSDSchemaDirective xsdSchemaDirective = (XSDSchemaDirective)content;
          StringBuffer xsdSchemaDirectiveURL = new StringBuffer();
          String xsdSchemaDirectiveLocation = xsdSchemaDirective.getSchemaLocation();
          if (xsdSchemaDirectiveLocation != null && xsdSchemaDirectiveLocation.indexOf(':') == -1 && xsdSchemaURL != null && xsdSchemaURL.indexOf(':') != -1)
          {
            // relative URL
            int index = xsdSchemaURL.lastIndexOf('/');
            if (index != -1)
              xsdSchemaDirectiveURL.append(xsdSchemaURL.substring(0, index+1));
            else
            {
              xsdSchemaDirectiveURL.append(xsdSchemaURL);
              xsdSchemaDirectiveURL.append('/');
            }
          }
          xsdSchemaDirectiveURL.append(xsdSchemaDirectiveLocation);
          // resolve schema directive
          XSDSchema resolvedSchema = xsdSchemaDirective.getResolvedSchema();
          if (resolvedSchema == null && xsdSchemaDirectiveURL.length() > 0)
            resolvedSchema = getSchema(xsdSchemaDirectiveURL.toString());
          if (resolvedSchema != null)
          {
			if(!checkSchemaURI(xsdSchemaDirectiveURL.toString())){
				schemaList_.addElement(resolvedSchema);
				gatherSchemaDirective(resolvedSchema, xsdSchemaDirectiveURL.toString());
      		}
      	  }


So in the end we are able to resolve the xsdimportimpl objects and so we end up finding the types we need, The problem is that we try and validate the xsdschema objects and present the errors to the users. We do this with all the schemas we resolved as well as the original wsdl xsdschema that held the xsdimportimpl objects. This appears to return errors that it cant locate the xsd files. But we have already resolved them. Im not sure how to handle this, I dont want to show this to the users becasue we were able to resolve them. But I was hoping I wouldnt have to put in code to look for this kind of return. I think the reason we hand in the wsdl xsdschema is because there may be inline schema defs, 

here was his reply:

The schema model generally will only resolve imports on demand, i.e., as a name in the imported namespace is resolved.   So an empty schema with only imports won't resolve any imports and will produce warning messages, not error messages, that the imports have not been unresolved (not cannot be resolved).   In our XSDEcoreBuilder, to force these imports to resolve anyway, we use this approach of trying to resolve a name in that namespace to force demand resolution:

  protected void validate(XSDSchema xsdSchema)
  {
    for (Iterator i = xsdSchema.getContents().iterator(); i.hasNext(); )
    {
      Object content = i.next();
      if (content instanceof XSDImport)
      {
        XSDImport xsdImport = (XSDImport)content;
        xsdImport.resolveTypeDefinition(xsdImport.getNamespace(), "");
      }
    }

Another thing that isn't clear is why you are doing something so fancy as visiting all the imports to resolve types.  I'd expect you to just use xsdSchema.resolveTypeDefinition with a namespace and a name (just like the above) and the rest will just take care of itself, i.e., the imports will be resolved, if possible, and the type will be located, if it exists.

we could easily clean this up and simplify the code.

Comment 10 Markku Saarela 2007-10-09 02:06:46 EDT 
Problem should only occurs when you import to same namespace. In Schema specification XML Schema Part 1: Structures Second Edition (http://www.w3.org/TR/xmlschema-1/#layer2 section 4.2.3) at the end of section there is note.

"Note: The above is carefully worded so that multiple <import>ing of the same schema document will not constitute a violation of clause 2 of Schema Properties Correct (§3.15.6), but applications are allowed, indeed encouraged, to avoid <import>ing the same schema document more than once to forestall the necessity of establishing identity component by component. Given that the schemaLocation [attribute] is only a hint, it is open to applications to ignore all but the first <import> for a given namespace, regardless of the ·actual value· of schemaLocation, but such a strategy risks missing useful information when new schemaLocations are offered."

So it it parser specific if they can import other than first one. Xerces has option to support this.
Comment 11 Gilbert Andrews 2008-01-31 13:41:50 EST 
This change gets rid of all the errors. The change directly affects how xsd import statements are handled. In the EMF model according to Ed Merks xsd import statements are loaded on demand. So In the past our code tried to load the imported xsds and when it was unable to do this it actually took the xsd url and read the xsd and created a new xsdschema object. Then when we validate these schemas because they are not part of the emf model we get errors. 

The correct way to do this according to Ed Merks is to ask the import xsds to resolve themselves. This forces the EMF model to act now and resolve the xsds. Then we use the xsds from the emf model. Because theses xsds are attached to the model all xsds are resolved and no errors are found.

As far as risk we are dealing only with schemas in xsdimports. When I tested the a case there were 6 schemas found in the new codde the same 6 schemas were available. If the url that we use to read in lost schemas is able to find the schemas then the emf model has already found them. This is something the emf model does well and this task should be left to them. 

I actually think this change is less risky  because we are moving from our home made solution to the emf model which has been proven over a long time.

Again this only effects

schema>
	xsdimport>

Comment 17 Gilbert Andrews 2008-05-14 16:03:28 EDT 
We fixed a few problems a while back which cleaned up a lot of the errors. Now using an example I found there were still some warnings being reported. There are a number of warnings stating a reference cannot be resolved as it is unused. These are reasonable errors that are handed back by the XSD model durling validate. The XSD model does not load unused imports. There are several other warnings about unresolved imports. These are not reolved because they are part of the unused imports.

The key to having all warnings removed is to open any wsdl or xsds using the validator and you should be able to see easily where the unused imports are. Remove these from the pertinent files and run on the WSE again there should be no errors remaining.
