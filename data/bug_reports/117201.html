Description Florian Priester 2005-11-19 17:25:03 EST 
SWT-win32, N20051119-0010

From at least v2136 (Eclipse 2.1.3) to ~v3213, the height of a tree item
with the standard 8pt system font and a 16x16 icon was 16 pixels.

In the latest version of SWT the height of such an item is now 17 pixels
which leads to some loss of screen real estate. While not dramatic,
it is still noticeable (one item less per 16 lines). Also note that
having items with an odd height causes ugly irregularities in the
connector lines of the tree.

I've traced this to the addition of the TVS_NONEVENHEIGHT style
in Tree#widgetStyle. Is this a necessary change/an intended effect?

Comment 2 Steve Northover 2005-11-21 10:52:29 EST 
Yes, this was intended.  In previous versions, the height of a tree item was restricted to be an even number rather than the operating system preferred height.

Comment 12 Florian Priester 2006-04-17 16:53:12 EDT 
(In reply to bug 136976 comment #11)
> Florian, you also believe that this is a problem because you changed your app
> to make the icons smaller.  The change makes tree more consistent with table
> (which is good) as well as supporting Custom Draw.  Your thoughts?

I agree that ultimately, the spacing is and should be chosen by the OS
based on aspects such as theme, font height and image size. Moreover,
Windows Explorer indeed has the same look as current SWT trees, although
I remember thinking "this looks bad" when I first saw the irregular tree
lines in Explorer. (I believe this started with XP, earlier versions had
regular lines.)

So, even though personally, I prefer to have less padding and more room
for items (I use 16x15 table icons, too), I can see your reasons for
marking this report WONTFIX.

Still, it might be worth (re)considering whether it didn't/wouldn't make
sense to have even heights for items in a tree that draws its lines using
alternating dots; the visual problems when allowing odd heights have already
been pointed out. Microsoft must have been aware of this too, hence the
defaulting of the tree-view control to even heights. Would dynamically
setting the TVS_NONEVENHEIGHT style based on the presence of MeasureItem
listeners be an option? (Not sure if that's possible.)
