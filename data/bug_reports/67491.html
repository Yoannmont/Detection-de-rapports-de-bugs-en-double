Description John Camelon 2004-06-16 13:11:12 EDT 
This defect serves as a placeholder for ways that we will most likely need to 
expand the IScannerInfo interface, and the clients/builders that populate 
them.  

Using GCC as an example, there are compiler flags beyond -I and -D that our 
parser requires in order to accurately parse a file in the same way that GCC 
does.  -include, -imacros and -idirafter, -iprefix, -isystem, -iwithprefix 
and -iwithprefixbefore are all flags that our builders would need to 
explicitly recognize in the UI and pass into the Scanner through 
IScannerInfo.  

Further investigation is necessary in order to figure out a generalized way of 
presenting this information, particularly as we add support for other 
compilers (VC++,Intel,Visual Age,SunCC, etc.)

Comment 7 Leo Treggiari 2004-10-07 16:00:17 EDT 
We're currently implementing per-file tool options support in the Managed 
Build System (MBS), and I have a question regarding IScannerInfo.  MBS 
currently has code that calls IScannerInfoChangeListener.changeNotification 
when the user changes the value of the Include Paths or Preprocessor 
Definitions for a MBS configuration.  The "granularity" of IScannerInfo seems 
to be per-project.  Is this true?  This doesn't really handle the current MBS 
functionality where the Include Paths and Preprocessor Defintions could be 
different for different configurations in the same project.  And it won't 
handle the new 2.1 functionality where the Include Paths and Preprocessor 
Defintions could be different for different files in the project.  What is the 
current CDT capability in this regard so that I can understand what MBS should 
do?
Comment 8 John Camelon 2004-10-12 19:58:54 EDT 
IScannerInfoChangeListener.changeNotification() is meant to handle changes per 
resource.  From what I remember talking to Sean, this interface is deprecated, 
although not explicitly marked as so in the javadoc.  I am not sure why, as 
the Indexer should be responding to build property change notifications.  

CC-ed Bogdan so perhaps he can explain how this works.
Comment 9 Sean Evoy 2004-10-13 16:54:35 EDT 
Maybe I should clarify what I meant by deprecated. The indexer was the only 
client that was taking advantage of this information in 1.2. The intent was 
that any client of the scanner would locate an appropriate info supplier for a 
given project and subscribe for notifications when the build info changed. Both 
the MBS and standard builder used to be suppliers, but as of 2.0 the indexer 
stopped using this service and switched over to the CPathEntry 
feature/mechanism/framework to get change notifications. That effectively 
renders the whole mechanism deprecated. 

I am not sure if we need a generic way for clients to access this information, 
or just switch the storage of all the includes paths and macros to the 
CPathEntry in the MBS. If we switch, the clients can get the information from 
there, and receive notifications.
Comment 10 John Camelon 2004-10-15 15:36:20 EDT 
After looking at the compiler options for the Intel, IBM XLC, GCC & msvc 
compilers, I have come up with the following plan/design.  

Phase 1: Post 2.1 - Add the following methods to IScannerInfo

  String[]   getMacroFiles(); 
  String[]   getIncludeFiles(); 

This shall allow us to get past the problems regarding -imacros and -include 
w/GCC, /FI w/MSVC.  XLC & Intel do not have seem to support these sorts of 
options as far as I can tell.  

Phase 2: Along with parser extensibility, the GCC & XLC compilers allow for 
different policies for finding inclusions, specified through the use of 
command line options (-idirafter && -I- for gcc, -qidirfirst for XLC).  We can 
allow for IScannerInfo to be subclassed to allow for a more specific mechanism 
for specifying "classes" of include paths (system, user, privileged, etc.), in 
order to guarrantee that we will eventually be able to get as accurate of a 
parse as possible.
Comment 11 John Camelon 2004-10-19 16:29:56 EDT 
Created attachment 15285 [details]
Interface & Scanner2 Support for -imacros & -include

Let's start the ball rolling!

I have attached a zip file that contains the patches that provide the interface
changes and Scanner support for the IScannerInfo enhancements.	I have not done
any work in updating ManagedBuildInfo, MakeScannerInfo or DiscoveredScannerInfo
other than stubbing out the implementations for the new methods.  

I have added an IScannerInfoPluginTest that provides basic sanity of the
support within our IScanner implementation for -imacros/-include inputs.  
I have also extended IASTInclusion and IASTMacro to respond to isImplicit(), it
serves as a short hand to know that these constructs are not actually
offsetable elements in the code and cannot be navigated to in the way that
normal macros and #include directives can be.  I have updated the CModelBuilder
to take advantage of this. 

I have not committed these patches to the HEAD as I do not want to jeopardize
CDT 2.1.  If these patches fall out of date, please let me know and I'll try
and keep them current.
