Description Oyvind Harboe 2004-10-15 06:07:40 EDT 
When I double click on a breakpoint on the left hand side ruler to disable it, 
CDT hogs the Windows input loop, i.e. Eclipse + all other windows applications, 
are unresponsive while the operation takes place.

This is particularly serious when GDB is unresponsive, since it is a pain to 
torture Windows to bring up the task manager to kill GDB.

Hmmm... is this the only place this happens, or is it symptomatic of a broader 
class of problems with too much stuff being executed on the main thread?

Comment 13 Nobody - feel free to take it 2004-11-11 11:39:13 EST 
I saw this problem when I was working on the fix for breakpoints. 
Unfortunately, the "terminate" method of "RuntimeProcess" is called from the UI 
thread by the framework. It issues the "exit" command to gdb and waits unil the 
timeout expires if gdb is unresponsive. During that time the UI hangs. 
I would suggest to close this bug and create a separate bug for "terminate" 
with the target milestone 3.0. Meanwhile I will ask the platform folks if they 
want to make this call asynchronous. In both cases the changes needed are too 
serious to put it in 2.1. Any objections?
Comment 14 Nobody - feel free to take it 2004-11-11 17:13:19 EST 
The Eclipse platform has addressed this issue (async terminate) in 3.1. See bug 
#68808.
