Using 1021, I got this in the .log file:
Error Oct 21, 2003 12:04:11.500 An internal error occurred during: "Status Line

Line 461 of Display is:
	synchronizer.asyncExec (runnable);

The synchronizer must be null!  John, any idea here?

Comment 2 Olivier Thomann 2003-10-22 08:29:20 EDT 
I am using Sun JDK1.4.1_02.
Comment 3 John Arthorne 2003-10-22 16:33:49 EDT 
Synchronizer is set to null when the Display is disposed.  This is the only time
it can possibly be null.  I suspect a UIJob is being scheduled after the display
has been disposed (quite possible, since some are launched by job listeners on
non-UI jobs that may still be running during shutdown). Even if the UIJob
protects itself with if (!display.isDisposed()) ... there will be a race
condition between dispose and asyncExec

Device.dispose does:
  synchronizer = null;
  ... other stuff, including OS calls to destroy widgets ... 
  disposed = true;

asyncExec does:
  if (disposed) throw Error(...);
  synchronizer.asyncExec(runnable)

So there is a non-trivial window where disposed=false and synchronizer=null
where this can happen.  You might be able to convince SWT to change the order in
Device.dispose to set disposed=true before other cleanup work.  This greatly
reduces the possibility of the race condition when other threads try to
asyncExec while it is being disposed.
Comment 4 Nick Edgar 2003-10-22 21:59:42 EDT 
Thanks for the analysis John.  Moving to SWT for consideration.
Comment 5 John Arthorne 2003-10-23 11:32:16 EDT 
This should be moved to back Tod afterwards to add the isDisposed() check before
invoking asyncExec.
Comment 6 Steve Northover 2003-11-12 11:06:41 EST 
The right answer for SWT is probably to add a lock for the shut down process 
so that other threads can't get into asyncExec().  I will talk to Silenio 
about this.

In any case, the job manager needs to be shut down before the display is 
disposed.  This will fix the problem for Eclipse but not for SWT.

Thoughts?
Comment 7 John Arthorne 2003-11-12 11:44:38 EST 
Agreed that a mechanism to explicitly shutdown the job manager will fix the
Eclipse case. I have entered core bug 46492 for this.
Comment 8 Steve Northover 2004-02-25 15:49:19 EST 
Instead of getting an NPE, I think the right thing to do now is get a 
SWT.ERROR_DEVICE_DISPOSED.

Talking to Silenio about it, the syncExec() method is getting called from 
another thread while the Display is being disposed.  If it was called a little 
bit later, syncExec() would throw SWT.ERROR_DEVICE_DISPOSED.  This means that 
application code already needs to be aware that this can happen.

SSQ?
Comment 9 John Arthorne 2004-02-25 16:24:49 EST 
That sounds reasonable.  I suspect that application code of the form:

if (!display.isDisposed()) display.syncExec(...)

will be surprised if syncExec throws a device disposed error, but short of an
atomic syncExecIfNotDisposed method there will always be a window where this can
happen. A disposed display error is certainly more logical than an NPE.
Comment 10 Silenio Quarti 2004-02-25 19:19:15 EST 
I agree. We should protected our code so that we do not cause
a NullPointerException, but application code should get an
exception because it needs to know that the runnable is
not going to be executed, since we are in the process of being
disposed.
