Comment 5 Tod Creasey 2003-08-07 15:48:27 EDT 
After talking to SWT the issue here is that we should not have ever had that 
check in - we should have just applied it without checking first - it was an 
attempt to prevent developers having to set the fonts of widgets that wanted 
to use other fonts. This is also potentially a problem with anyone who sets 
font for particular widgets in thier Xdefaults file on a Linux platform.

We have several choices here
1) Remove this check within the code and update the javadoc. I don't believe 
this effects anyone within Eclipse but it may affect customers.
2) Deprecate this method and write a new one. This will meana change to about 
200 references within Eclipse. Small but still annoying.
3) Make the check platform specific (windows only) as it is not valid on other 
platforms. Not really a good solution as it breaks the Eclipse goal of 
consistency.

I personally like number 1 as it has the least impact on the Eclipse team but 
I am not sure if it constitutes an API change.

Comment 9 Tod Creasey 2003-08-13 15:48:40 EDT 
The problem here turns out to be twofold

1) We need a different default font for each widget type. This is trivial to 
implement
2) We need to skip font propogation when the dialog font is the default. This 
is a simple check that can be added to the applyDialogFont() method. However 
for places they do not use this convenience method we should do this for free.

I propose that JFace call applyDialogFont by default after createContents on 
Dialogs and PreferencePages. The method that does the propogation will be 
overridable by subclasses should they want to do something more fine grained.

This way people will simply remove the dialog font support code from thier 
existing implementations and will just let JFace do the work for them.

We investigated doing this with SWT but SWT has the issue that widgets are 
created in different places in the code and that they would not be able to 
control this for custom controls. Also the font propagation would have to be 
executed in an asyncExec which may cause layout problems etc.

If everyone is happy with this solution I will try to include it for M3.
Comment 10 Tod Creasey 2003-08-19 09:35:37 EDT

If there are any issues with this solution that you can see please let us know 
now as there will be some work required by those who do not surrently use 
applyDialogFont.

As part the changes for M3, initializeDialogUnits(Control) was changed to 
always get the metrics from the dialog font, not the font of the passed 
control.  This is contrary to the method's spec and is a breaking change (if 
the caller does not have the dialog font set in the control).

I guess this was done was to allow applyDialogFont to be called after 
creation, without breaking all the existing code that calls 
initializeDialogUnits at the beginning (without this change, existing code is 
OK if it passes the parent control, but not if it passes a new child).

This could be done in a non-breaking way by adding a new initializeDialogUnits
() method that takes no parameters, and deprecating the old one.  Existing 
code would have to be changed to use this new method, and to call 
applyDialogFont at the end (applying it for free would also be breaking).

However, I agree with Dirk that it would be best if we handled the 
responsibility of choosing the initial font for new controls, and did not 
require clients to assign the dialog font at a later time.  The 
initializeDialogUnits issue illustrates why it's best to set the font as early 
as possible.

The best way to do this would be to track child control creation using a (new) 
child control listener in SWT (as Dirk suggested long ago).  Such a listener 
seems like a reasonable thing to add to SWT, and may have other uses.

Comment #9 above makes it sound like the main issue is that there isn't a 
single place in SWT where it's known when a new child is created (or 
disposed), whether the creation is done in a built-in control or a custom 
control.  Is the issue that sometimes child controls are

	private static Font getDefaultFont(Control control) {
		String fontName = "DEFAULT_FONT_" + control.getClass().getName(); //$NON-NLS-1$
		if (JFaceResources.getFontRegistry().hasValueFor(fontName))

Comment 30 Sebastian Davids 2004-02-11 13:20:15 EST 
Created attachment 7792 [details]
fix to FontRegistry instead of Dialog

My previous patch had the advantage of not creating a font handle for each
Control, but the disadvantage of not doing a fast lookup; instead fonts would
have been compared every time Dialog.applyDialogFont would be called.

This new patch takes a different approach:

The code in Dialog is unchanged, i.e. for each Control an entry in the font
registry is created under DEFAULT_FONT_controlclassname; strategy: lookup in
registry under the key -> if it does not exist create and store in registry.

The behavior of the font registry has changed in the following way:

null (i.e. use the dialog font) is put as the value under key
DEFAULT_FONT_controlclassname into the symbolic name/font data map if the new
font would have the same font data as the system default font.

Thereby no new fonts handles will be created and the fast lookup is preserved.
Comment 31 Sebastian Davids 2004-02-16 09:35:17 EST 
Here's a longer explanation :D

@@@@

The current implementation of createFont(String, FontData[]) in
org.eclipse.jface.resource.FontRegistry will create a new font object (handle)
for each kind of Control it encounters.

Here's a simplified version of the current implementation (using the symbolic
name from Dialog#getDefaultFont(Control)):

Is there a key under DEFAULT_FONT_fully_qualified_control_classname in
stringToFontData?
- Yes -> Is it null?
  + Yes -> return default font
  + No -> return font reference from registry
- No -> determine platform preference for this kind of Control; create a new
font object; put it into the stringToFontData cache under
DEFAULT_FONT_fully_qualified_control_classname

So one ends up with the following (if it has encountered List, Text, Button, and
four Labels all using the system default font) stringToFontData map in pseudo code:

DEFAULT_FONT_org.eclipse.swt.widgets.Button = new Font("MS Sans Serif-regular-8")
DEFAULT_FONT_org.eclipse.swt.widgets.Label = new Font("MS Sans Serif-regular-8")
DEFAULT_FONT_org.eclipse.swt.widgets.List = new Font("MS Sans Serif-regular-8")
DEFAULT_FONT_org.eclipse.swt.widgets.Text = new Font("MS Sans Serif-regular-8")

Now, if it would have encountered 20 Control kinds (using the system default
font) the map would contain 20 font objects (handles).

This is the reason why one cannot display all preference pages on Win98/ME
because the number of Font handles apparently exceeds the platform limit.

@@@@

What does my proposed patch do?

It's algorithm is:

Is there a key under DEFAULT_FONT_fully_qualified_control_classname in
stringToFontData?
- Yes -> Is it null?
  + Yes -> return default font
  + No -> return font reference from registry
- No -> determine platform preference for this kind of Control; Is the Controls
font the same as the system default font?
  + Yes -> store "null" in stringToFontData cache under
DEFAULT_FONT_fully_qualified_control_classname
  + No  -> create a new font object; put it into the stringToFontData cache
under DEFAULT_FONT_fully_qualified_control_classname

One ends up with the following (if it has encountered List, Text, Button, and
four Labela all using the system default font) stringToFontData map in pseudo code:

DEFAULT_FONT_org.eclipse.swt.widgets.Button = null
DEFAULT_FONT_org.eclipse.swt.widgets.Label = null
DEFAULT_FONT_org.eclipse.swt.widgets.List = null
DEFAULT_FONT_org.eclipse.swt.widgets.Text = null

No new font objects (handles) were created.

@@

What if the four Labels and the Button use the system font and List and Text use
a different font? One ends up with:

DEFAULT_FONT_org.eclipse.swt.widgets.Button = null
DEFAULT_FONT_org.eclipse.swt.widgets.Label = null
DEFAULT_FONT_org.eclipse.swt.widgets.List = new Font("Tahoma-regular-8")
DEFAULT_FONT_org.eclipse.swt.widgets.Text = new Font("Lucidia Console-8")
