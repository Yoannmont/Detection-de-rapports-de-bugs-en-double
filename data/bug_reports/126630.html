Comment 1 John Arthorne 2006-02-06 15:01:01 EST 
We debugged this a bit, and found that Prompter#handleStatus is doing something weird.  It's essentially doing a syncExed, but it's doing it by calling asyncExec and then blocking on a monitor that is then signalled from the body of the runnable.  

The deadlock occurred because this happens from the build thread, which owns the workspace root scheduling rule.  The UI thread meanwhile is blocked while waiting for a resource scheduling rule, so it deadlocks.

Interestingly, we have some magic code in the implementation of syncExec that would resolve this kind of deadlock automatically.  Essentially the UI thread will still handle syncExec while it is waiting for a scheduling rule.  Unfortunately, since the Prompter class uses asyncExec, it doesn't benefit from this.
