> input messages or is this a bug?

Show the trace of the MI protocol, it would be of great help in understanding
what is happening.  The way to do this is described in the cdt FAQ
at www.eclipse.org/cdt

The MI protocol is very strict on how a program output should be sent,
gdb does not implement this correctly for local debugging, this
is why we redirect the output to a pseudo pty.
So the program does not interleave with gdb MI output.

For OS that can not create a pseudo pty, we could possibily using
the combination of xterm and shell hackery spawn gdb and redirect the
program output.
By that would require a new launch

Does not HP support pseudo pty ?

I'm not following the first statement.  But look at the GDB/MI documentation
Here I will copy it for you, that will save us some time:

`"~" STRING-OUTPUT'
     The console output stream contains text that should be displayed
     in the CLI console window.  It contains the textual responses to
     CLI commands.
                                                                               
                                             
`"@" STRING-OUTPUT'
     The target output stream contains any textual output from the
     running target.
                                                                               
                                             
`"&" STRING-OUTPUT'
     The log stream contains debugging messages being produced by GDB's
     internals.
                                                                               
                                             

So if you program is doing some printf:
  printf ("Hello world");

The output of the program should appear like this:

@ "Hello world"

And GDB console output from a command should look like this:
  22 info threads
  &"info threads\n"
  22^done

Note the use of "&" and "@"
