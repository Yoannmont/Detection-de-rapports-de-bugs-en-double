Description George Alanshu 2005-08-24 20:39:12 EDT 
XSD editor source tab doesn't work for a wsdl referenced schema outside of the 
workspace.

To reproduce:
1) Create the wsdl file:
<?xml version='1.0' encoding='UTF-8' ?>
<wsdl:definitions name='expenseApprovalService' 
targetNamespace='http://www.example.com' 
    xmlns:tns='http://www.example.com' 
    xmlns:ns0='http://www.foobar.com/expense' 
    xmlns:xsd='http://www.w3c.org/2001/XMLSchema' 
    xmlns:wsdl='http://schemas.xmlsoap.org/wsdl/'>
    <wsdl:types>
<schema xmlns="http://www.w3.org/2001/XMLSchema">
    <import namespace="http://www.foobar.com/expense" 
schemaLocation="expenseApproval.xsd"/>
</schema>
    </wsdl:types>
    <wsdl:message name='Message'>
        <wsdl:part name='approval' element='ns0:approval'/>
    </wsdl:message>
    <wsdl:portType name='Port1'>
        <wsdl:operation name='Task1'>
            <wsdl:input message='tns:Message'/>
        </wsdl:operation>
    </wsdl:portType>
</wsdl:definitions>

2) Create the xsd file:
<?xml version="1.0" encoding="UTF-8"?>
<xsd:schema xmlns:xsd="http://www.w3.org/2001/XMLSchema" 
xmlns:tns="http://www.foobar.com/expense"
    targetNamespace="http://www.foobar.com/expense" 
elementFormDefault="qualified">
    <xsd:element name="approval" type="xsd:string"/>
    <xsd:element name="amount" type="xsd:float"/>
    <xsd:element name="description" type="xsd:string"/>
</xsd:schema>

3) Save both files in the same folder, outside of your Eclipse workspace
4) Within Eclipse choose Open File... and open the wsdl file
5) Through the wsdl editor, open the referenced xsd file
6) When the xsd file is open you'll notice:
   - The graph tab works as expected
   - The source tab doesn't show any content, but a message indicating the file 
does not exist is shown

Thanks,
George.

Comment 4 Valentin Baciu 2006-06-13 22:00:17 EDT 
I've re-tested this functionality with 1.5 RC4.

The schema file opens fine in the XSD editor and both the source and graphical view seem properly populated. I was able to add new content and save it.

The WSDL editor however seems to have regressed a bit. I get a class cast exception related to the type of editor input expected. See https://bugs.eclipse.org/bugs/show_bug.cgi?id=144811 which seems to track this issue. After applying the patch proposed for that bug, the WSDL editor opens fine with both source and graphical views populated. The editor seems to work OK - sort of. I was able to add and save content. It appears though that the IFileEditorInput dependencies are more widespread than just the few references fixed by the patch for 144811 as I see a few exceptions in the console.

When using external files, there seems to be a limitation on navigating from the WSDL file to an imported XSD file using the new hyperlink support: the XSD file is opened in the XML editor instead of the XSD editor. The functionality is handled by the XML hyperlink detector and the ExternalFileHyperlink. This last class uses some platform code to find the proper editor. The mechanism used (from the org.eclipse.ui.ide.IDE class) is based on the file's content type. This mechanism seems to detect the more generic XML content type instead of the more specific XSD content type. I believe this is the same platform behaviour which also causes https://bugs.eclipse.org/bugs/show_bug.cgi?id=139756.

Comment 6 Valentin Baciu 2007-06-27 10:26:06 EDT 
I've given this scenario a try with the latest 2.0 and it seems to me that things have regressed even more. With 2.0, the external files seem to be opened by default with the XML source editor instead of the proper WSDL or XSD editors. It appears that the platform changed the code path used to open editors on external storage and it is now using the same algorithm for figuring out the content type as the one used to display the icons in the navigator. The platform looks at the file extension and matches it with XML source content type (used as a fall back for ill-formed XML-like stuff) because it is higher in the content type hierarchy albeit in a different branch.

Rich, this seems related to the work done by Joshua to allow the WSDL editor to be opened on WSDL files in CVS or outside the workspace. If you concur, you can either duplicate it with the bug he's using or assign to him and cross-reference.
