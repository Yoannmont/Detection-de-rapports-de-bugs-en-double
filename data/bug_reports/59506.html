Comment 36 Andy Carroll 2007-05-16 08:47:21 EDT 
A somewhat complex way to fix this problem is to write a wrapper Launcher program in C/C++ that delegates the job of initializing the JVM to the embedded browser (in this case IE).

Instead of directly creating a JVM in using JNI the Launcher would first initialize its emedded browser (IE) via COM and have the browser intialize its own JVM. it would accomplish this by launching a local HTML splashscreen, that contains an APPLET tag pointing to a local codebase. (ie. launch a splashscreen with a local document rather than requiring a live IP connection to the outside world)

The Launcher program would wait until the splashscreen had been displayed and then connect to in-process JVM that was created by the browser. ie. it would call GetCreateJavaVM to obtain a handle to the JVM created by the browser, then proceed to launch the SWT app within this JVM. 

ie. the launcher would not create a JVM, but rather use the in-process JVM created by the Browser. The Launcher would then create a new ClassLoader, setup the appropriate securtity policy, and launch the SWT app.

Since the Browser created the JVM in the first place, it is assured to support Applets.

By allowing the embedded browser to create the JVM instead of trying to create its own JVM, the issue of mimicing the precise JVM initialization "secret sauce" the browser requires to run applets is side stepped.

NOTE: Thus the actual SWT app would NOT run as an applet. 
It would continue to run as a standalone app in a JVM - however the specific JVM in which it runs would be the JVM that was initialized by the browser.
 
The difference being that IE would have created this JVM and thus the JVM would be correctly configured to run applets.

NOTE: Under this scheme there would still only be a single in-process JVM - however it would be a JVM that was safe to run applets under.

The drawback with this approach include:

a) Startup would be slower. The SWT app would not start until IE and its JVM had started

b) There would be no programmatic control as to the specific version of the JVM that was invoked. The browser would launch the JVM as specified in the Java control panel. 

However the Launcher app could examine the Windows Registry to determine if the JVM configured for running in the browser is appropriate.

c) No programmatic control of JVM args passed to the JVM. The JVM would be initialized by the browser not the launcher, and thus the launcher could not pass JVM arg. The JVM args would be determined by the Java Control Panel, and hence exist in the Windows Registry. Thus the user could configure JVM args, but they would do so in the Java control panel.

Although complex, this approach simply yields to the natural tension that arises due the restriction of a single JVM per process when running an embedded browser, rather than try to fight it.

While an approach like this doesn't make sense as the only way of launching an SWT app, this approach, or something similar may be worth considering for applet friendly deployments. 

I'm not familiar with the existing boot strapcode that is used to invoke the JVM into which a SWT app runs, so I don't know how complex it would be to adapt the existing Launcher code to have an option to invoke IE, and wait for a splashscreen to complete and connect via GetCreateJavaVM.

Perhaps its not as difficult as it sounds :-)

Comment 41 Ron Capelli 2007-12-13 12:35:12 EST 
The observed behavior is exactly what is expected given the root cause of the problem.

The Sun and IBM JVMs on Windows and Linux implement support for applets differently.  On Windows, applets are run in a JVM started in the same process as the browser.  In the case of the SWT Browser, this is already running with a JVM and the attempt to start a new JVM in the same process fails.  On Linux, applets are run in a JVM started in a separate process.  This works.

So, applets work with Mozilla on Linux.  They do not work with Mozilla on Windows.  With IE, Java 1.1 applets work with the MSJVM, but applets that require Java2 will not work.

Comment 63 Grant Gayed 2008-12-19 17:01:45 EST 
To update, I've released changes that expose this functionality now, though this bug will remain open because there are still some issues to be investigated.  It should be in nightly builds >= 20081219.

When the Browser encounters a page with an applet, the applet will now be allowed to execute iff all of the following are true:

1. embedded browser is IE7
  -> I've seen this both work and fail on IE6, so for now applets won't execute when IE6 is embedded

2. IE's Java plug-in is Sun jre >= 1.6u10 with the Next Generation Java Plug-in enabled
  -> this plug-in is enabled by default when the jre is installed

3. eclipse is launched with the same jre as the one from #2 above
  -> there is no explicit check for this, but I've seen the test for #2 fail when these jre's are not the same

4. If the app is a stand-alone swt app (ie.- not an eclipse RCP app) then System.exit() should be called at the end of program execution to ensure that the spawned java process is cleaned up
  -> the clean-up spews a couple of "CAxControl::FinalConstruct / CAxControl::FinalRelease" lines to stdout, presumably coming from the plug-in
  -> these lines are also spewed when exiting eclipse

5. -Xbootclasspath includes the jars listed in comment 52

#1-3 are attributes of the target machine.  A developer wishing to make applets viewable in their app is currently responsible for #4 and #5.  Hopefully a way can be found to get rid of requirements #4 and/or #5.  In the meantime, please feel free to try this out in its current state and report any problems.
