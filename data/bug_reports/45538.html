Comment 7 Chris Recoskie 2005-11-24 10:11:16 EST 
My plan is for this to be a GNU make specific option.  The builder will be invoked with the generated output file as its target.  E.g., if I want to build foo.c, a simple "make foo.o" will be issued.

Similarly, I will be adding clean support.  All this will do is remove the output file.  No external make commands will be issued, the removal will be done with Java code.  This does not follow the dependency tree upward to clean all the files that the output file depends on.

I will also add a "force rebuild of file".  This is just a union of the previous two.  The output file is cleaned, and the build is invoked.

I don't exactly feel these features should go into managedbuilder.gnu.ui because this means that people have to take the GNU build definitions if they want the GNU make features.  I'm not sure I want to create a whole new plugin just for this though either.  Right now my current plan is to put the UI elements into managedbuilder.ui and only show them if the selected project's makefile generator is a GnuMakefileGenerator.

Comment 14 Chris Recoskie 2006-05-01 10:22:48 EDT 
This has been implemented using the new internal builder.

How it works:

The user selects any number of files (instances of ITranslationUnit) and selects Build File(s) from either the context menu or from the Project menu. For each resource, the build description is queried for all dependent steps that depend upon the given resource.  These are only the steps that are direct children of the selected resource, i.e. this will not trigger build steps that are more than one level deep.  E.g., in the conventional compiler/linker paradigm, the .c file will be compiled, but this will not trigger a link.

These steps are then re-built in sequence (i.e. the compile is always forced and is not incremental).  The build is done in the background as a Job with accurate progress reporting and the ability to cancel the job.

I have similarly added the ability to clean selected files.  This queries the build description for the output files for each selected resource, and removes them.

Both of these features are builder agnostic and strictly use information contained in the build model.  At no time is an external builder invoked.
