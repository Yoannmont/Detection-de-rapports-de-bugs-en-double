Comment 30 Kurt Miller 2005-08-26 15:10:03 EDT 
First off thank you Panagiotis for your work on Eclipse on FreeBSD. I have 
largely copied your process for the OpenBSD Eclipse port and the FreeBSD port 
was an excellent guide.

I'm wondering if a generic bsd solution would be an acceptable idea? The 
OpenBSD port is nearly identical to the FreeBSD one, with some minor linking 
differences due to the default location for mozilla installs. I'd assume that 
if NetBSD had a port (haven't checked) that it would also be almost the same. 
Having a bsd solution would reduce the number of directories and files that 
are redundant across Linux/FreeBSD/OpenBSD/NetBSD.

Links for the copy/substitute script and patches follow. Currently OpenBSD 
only support x86/gtk2, but will likely support amd64/gtk2 in the future.

http://www.openbsd.org/cgi-bin/cvsweb/ports/devel/eclipse/sdk/scripts/
http://www.openbsd.org/cgi-bin/cvsweb/ports/devel/eclipse/sdk/patches/
Comment 31 Panagiotis Astithas 2005-08-26 15:51:14 EDT 
Kurt, I am always a fan of minimizing the differences between BSDs, however I
believe there are two issues to consider in this case. The first is that using a
single ECLIPSE_OS="bsd" in the build scripts (as I believe you are essentially
suggesting) would deviate a bit from current practice and I am not sure it would
be welcome from the people here. The second is that the ultimate goal is to
provide binary packages for both these unsupported architectures from
eclipse.org (as per bug 106784). Therefore I don't see the benefit of unifying
the patches for the two systems since we want two (well more actually,
considering gtk/motif and x86/amd64) different binary package releases. In fact
I would expect all of the patches you provided to get integrated along with the
FreeBSD ones into the main tree. 

Either way I'll be happy if I get an official (although unsupported) build for
FreeBSD from eclipse.org.
Comment 32 Kurt Miller 2005-08-26 18:43:40 EDT 
Ahh, I hadn't considered the binary package distribution issues. Using 
ECLIPSE_OS="bsd" was indeed what I was thinking. However, in light of your 
comments that doesn't seem like a good idea anymore.

I don't want to highjack this bugreport (anymore then I already have). I'll be 
very happy if eclipse.org accepts your patches and I hope we will hear from 
the eclipse team what their thoughts are soon.

Comment 34 Billy Biggs 2005-08-29 15:04:39 EDT 
I intend to apply the patches to the launcher and SWT, but I have one question.
 I've been worried about the use of "libgtk-x11-2.0.so.0" vs "libgtk-x11-2.0.so"
on *BSD vs Linux.  For shared libraries, the first number is the major version
number, and an unversioned .so link is supposed to point at the current
development version (it's what -l uses).  We can't dlopen the .so on every
platform because doesn't always exist.  Under many Linux distributions, the .so
symbolic link only exists in the -devel package.

I have heard that the library version weirdness on FreeBSD is due to a libtool
bug, and is fixed by an "ltverhack" script at some point, but I have not been
able to verify this.

Anyway, back to the point, is there a sane #ifdef I can protect this around, or
should I just use the FREEBSD one you define in the makefile?
