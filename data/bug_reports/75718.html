Description Tom Hofmann 2004-10-06 11:47:22 EDT 
3.1 M2

I would like to be able 
a. to open the workbench preference dialog on a specific page
b. to link between preference pages
c. a scheme to address a certain UI element within a prefererenc page

I would want to do this with one single API method:

openPreferencePage(PreferenceLink link, ...)

where PreferenceLink is an address to a certain page or an element within a
page. The method would open the preference dialog on the given page, or open the
page if the dialog is already open.

I will attach a patch that modifies WorkbenchPreferenceDialog and adds a static
method similar to the one above.

Comment 8 Tod Creasey 2004-11-25 14:36:49 EST 
Adding Martin as I have looked at this now and he is involved in the ongoing 
discussion.

We need to decide what to do with these other issues
1) The developer wishes to open a preference page with only certain other 
pages showing/highlighted as if they did a search.

Here is my suggested API 


	/**
	 * Creates a workbench preference dialog to a particular preference 
page. 
	 * Show the other pages as filtered results using whatever filtering 
	 * criteria the search uses. It is the responsibility of the caller to 
then call <code>open()</code>.
	 * The call to <code>open()</code> will not return until the dialog
	 * closes, so this is the last chance to manipulate the dialog.
	 * 
	 * @param preferencePageId
	 *            The identifier of the preference page to open; may be
	 *            <code>null</code>. If it is <code>null</code>, then the
	 *            preference page is not selected or modified in any way.
	 * @param filteredIds
	 * 			The ids of the other pages to be highlighted 
using the same
	 * 			filtering criterea as search.
	 * @return The selected preference page.
	 * @since 3.1
	 * @see #createDialogOn(String)
	 */
	public static final WorkbenchPreferenceDialog createDialogOn(final 
String preferencePageId,
			String[] filteredIds)

2) Do we want to have to concept of a property to send around? Doug does it 
with the key bindings page by getting the current page and using it's API. 
This may be suffecient.
Comment 9 Douglas Pollock 2004-11-25 15:52:46 EST 
I have a preference with regard to (2).  I would prefer programmatic access to 
the page, as this is the most flexible from my standpoint.  Properties would 
add overhead, remove some type safety, and prevent the compiler from the 
catching some types of programmer errors.  In return, the only thing I think 
they offer is an additional level of API control. 
 
However, I'm not sure this is would be helpful.  Imagine a third-party plug-in 
trying to access the keys preference page (currently internal code).  As it 
stands now, the third-party plug-in would get a reference to a preference page.  
Unfortunately, the third-party plug-in shouldn't be down-casting to 
KeysPreferencePage, so they can't control the appearance of the page -- unless 
I make KeysPreferencePage public API. 
 
If we used properties, then I would publish properties as public API.  Either 
way I'm caught with API that I need to maintain.  And, if the class isn't 
public API, then where do I mention the properties it supports?
Comment 10 Michael Van Meekeren 2004-12-02 10:03:38 EST 
Just talked with Tod, I think we should do the following: 

1) API
Add API to set the Array of pages the dialog should filter to or null to show all
This should work on an already open dialog or one that has not been opened yet,
that way we support 
   A) opening from a view (for example) on a single page
   B) opening on multiple related pages
   C) search filtering the dialog down to some set of pages

2) Pass a set of properties to the pages being opened.  Doug, this does not mean
you need to make things public necessarily, it may be ok to say that the
solution is intended to support ones opening of their own page.  Optionally you
could make the page id and properties public if you had that need.
Comment 11 Martin Aeschlimann 2004-12-02 10:27:28 EST 
In general you don't know the set of pages you don't want, as pages are 
contributed by plugins you don't know.
So I think you could only give a set of pages that you want to include.

A problem of naming pages specifically is that this makes it impossible for a 
new plugin to get its page in this list. That's not the general case but will 
happen if you offer preferences for e.g. an extendable view.
For that I think we should allow filters instad of static lists of pages that 
get the extension point and can e.g. test for a certain extra property to be 
set to decide if the page has to be included.

-> IPreferencePageFilter
       select(IConfigurationElement page): boolean

plus a default implementation
   BasicPreferencePageFilter that takes an array of pages to include
