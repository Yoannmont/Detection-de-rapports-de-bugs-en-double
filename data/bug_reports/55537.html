Comment 8 Andrew Eidsness 2004-03-26 14:39:36 EST 
Tod has reproduced this too.  We noticed that the configuration directory and 
the org.eclipse.ui.ide directory were getting marked read-only, which might 
explain why the dialog was not able to write the file.

We tried making the directory writeable, but found it got set back to read-only 
as soon as eclipse ran.  I'm going to investigate this with the runtime guys.
Comment 9 Andrew Eidsness 2004-03-26 15:05:22 EST 
It looks like the value for the checkbox will be properly saved the first time 
the dialog is run, but not on any subsequent attempts.  It always works from the 
preference box inside eclipse.

So, a workaround is to use the Window -> Preferences -> Workbench -> Prompt for 
workspace on startup preference.
Comment 10 Andrew Eidsness 2004-03-26 16:53:38 EST 
Created attachment 8939 [details]
patch

The problem was an early return in the ChooseWorkspaceData.writePersistentData
method.  Before writing the file there is some code that adjusts the MRU list
to put the selected workspace in the right place.  That code contained a return
instead of a break.  The impact was that the following code -- write the file
-- never got executed.

So the problem occurs if the user is selecting a workspace that has been used
within the last 5 launches.  The problem causes both the value of the checkbox
AND the order of the MRU list to be ignored.
