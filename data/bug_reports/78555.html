Comment 7 Amy Wu 2004-11-23 11:28:07 EST 
I am able to recreate this problem easily now.

1. Have at least 2 files that need to be synchronized with CVS
2. Synchronize with CVS
3. In the Synchronize view, double-click one of the files that needs to be 
updated/committed
4. The compare editor should open up with the changes
5. Without closing the compare editor, double-click another file that needs to 
be updated/committed
6. Workbench terminates and you get the first hs error log.

On the surface, the problem appears to have to do with calling setInput a 2nd 
time on the compare editor that is already open, though I guess this is really 
exposing an SWT bug of some sort.  As for seeing the same problem when I 
create or apply a patch, I believe the problem was actually with applying 
patches because that would also be another time where the compare editor is 
used to show what diffs are being applied.

I am currently working around this problem by closing the compare editor 
before comparing each file, but it is tedious if I need to synch up and 
compare every file.
Comment 8 Andre Weinand 2004-11-23 11:45:55 EST

Can you reproduce the problem on a different machine with similar configuration?

(Calling setInput a second time is perfectly OK since the Compare editor is reused in this case.)
