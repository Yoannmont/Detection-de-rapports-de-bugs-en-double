Comment 10 Steve Northover 2004-09-07 18:25:47 EDT 
In bug 4409, it was determined that Windows was sending and extra resize event 
and we were unsure where it was coming from (ie. the size of the top level 
shell has not changed).  This is just a bug/feature in Windows that AWT/Swing 
has probably worked around by saving the previous bounds and not issuing an 
event when is has not changed for the shell.  We could do the same thing but I 
was hoping to determine why Windows was issuing the extra event.

Comment 13 Billy Biggs 2004-09-15 12:59:59 EDT 
Here is another test case for this bug.  This is a regression, the dialog
displays its contents on 2.1.3 but not on 3.0/3.1.  The reason is that SWT/GTK+
is causing a layout when you call setSize():  When the Shell's initial resize
event arrives is being optimized away because the size has not changed since the
last setSize().

import org.eclipse.swt.*;
import org.eclipse.swt.widgets.*;
import org.eclipse.swt.layout.*;

public class ShellOpenLayout {
	public static void main(String[] args) {
		Display d = new Display();
		Shell s = new Shell(d);
		s.setLayout(new GridLayout());
		s.setSize(800,600);
		Button b = new Button(s, SWT.PUSH);
		b.setText("foobar");
		GridData gd = new GridData();
		gd.heightHint = 20;
		gd.widthHint = 20;
		b.setLayoutData(gd);
		s.open();
		while (!s.isDisposed ()) {
            if (!d.readAndDispatch ()) d.sleep ();
		}
		d.dispose ();
	}
}

Comment 15 Steve Northover 2004-09-17 16:19:34 EDT 
Fixed > 20040917

Here is how it works: If you set the size, you will not get a free resize 
later on when the size has not changed (Windows bug fixed).  If you do not set 
the size, you get exactly one resize notification when the shell is made 
visible.  This happens because the size has been allocated in the operating 
system at that time by the window manager.

NOTE:  The example code that "does not draw properly" in this bug report is 
now correctly not drawing properly on all platforms.

Using SWT-win32 v3232, if you create a child control after setting the shell
size but before opening the shell, then you do get a resize event when the
shell is opened. See snippet below. Output:

  Shell resized=Point {200, 200}
  Shell resized=Point {200, 200}

This also means that the shell in the test case from comment #2 comes up
with its controls positioned when comment #15 suggests it should not.

---

import org.eclipse.swt.*;
import org.eclipse.swt.widgets.*;

public class ShellResizeTest {
  public static void main(String[] args) {
    Display display = new Display();
    
    final Shell shell = new Shell(display);
    shell.addListener(SWT.Resize, new Listener() {
      public void handleEvent(Event e) {
        System.out.println("Shell resized=" + shell.getSize());
      }
    });
    
    shell.setSize(200, 200);
    new Label(shell, SWT.NONE);
    shell.open();
    
    while (!shell.isDisposed()) {
      if (!display.readAndDispatch()) display.sleep();
    }
    
    shell.dispose();
    display.dispose();
  }
}
Comment 19 Steve Northover 2006-05-30 14:32:27 EDT 
This is by design and was implemented on all platforms.  If the shell hasn't been opened and a child is added, we give a free resize (and layout).  Too many people has this pattern in their code and were surpised when the shell came up without the children layed out.
