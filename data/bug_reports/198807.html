Description David Carver 2007-08-03 09:12:40 EDT 
Build ID:  I20070625-1500

Steps To Reproduce:
1. Create an XML File that is based on an XML Schema
2. Move the xmlns="" or xmlns:p="" assignment to after the xsi:schemaLocation attribute
3. Turn on Warn if no Schema Available
4. Reload the Grammar dependencies.
5. Switch to design view, there is no content model available for the schema.
6. There is also a warning about a not finding a schema.




More information:
The following order will reproduce the bug:

xmlns:oa="http://www.openapplications.org/oagis/9"
xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
xsi:schemaLocation="http://www.starstandard.org/STAR/5 ProcessSalesLead.xsd"
xmlns="http://www.starstandard.org/STAR/5"

If the order is switched to:

xmlns="http://www.starstandard.org/STAR/5"
xmlns:oa="http://www.openapplications.org/oagis/9"
xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
xsi:schemaLocation="http://www.starstandard.org/STAR/5 ProcessSalesLead.xsd"

The grammar is correctly found, and the content model is loaded correctly.

The order of attributes should not be significant, as XML states that attribute order isn't significant.

Comment 2 David Carver 2007-08-08 08:09:24 EDT 
Nitin this isn't a schema specific issue, as I can reproduce with any schema that isn't in the xmlcatalog (haven't tried it with ones that are in the xmlcatalog).  All you need to do is put the namespaces in the order specified (i.e put the default namespace after the xsi:schemaLocation).

Easiest way to reproduce this is to just use the SpaceWarGame example project.  After creating the new project, edit the ResultSheet.xml and move the xmlns:tns="http://www.example.com/SpaceWarGame" after the xsi:schemaLocation="" section, and reload the grammar.  Then switch to design mode, and you will notice there is no content model.   If you move xmlns:tns="" to before the xsi: attributes, and reload the grammar, the content model shows back up.

The order of attributes and particularly the order in which the various namespaces are defined shouldn't be specific as tools are allowed to put out attributes anyway they want.

Example of no content model created:
<tns:ResultSheet gameId="K77" playerId="Marco" turn="1" xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance" xsi:schemaLocation="http://www.example.com/SpaceWarGame SpaceWarGame.xsd " xmlns:tns="http://www.example.com/SpaceWarGame">

Example where content model created and namespace found:

<tns:ResultSheet gameId="K77" playerId="Marco" turn="1" xmlns:tns="http://www.example.com/SpaceWarGame" xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance" xsi:schemaLocation="http://www.example.com/SpaceWarGame SpaceWarGame.xsd ">


Only the order of the attributes is different.
Comment 3 David Carver 2007-10-18 22:01:14 EDT 
Created attachment 80723 [details]
Fixes the order dependancy on the attributes

The issue is that because attributes could have namespace prefixes, and the content model needs to know all the namespaces for all the prefixes.  The current code reads the attributes in as first come first served, this is an issue when attribute prefixes occur before their namespaces have been resolved.  This patch addresses this issue by reading all the available namespaces first, and then reading the rest of the attributes.   This correctly allows it to resolve the prefixes against their namespace and fixes the issue of attributes being order Dependant.

Comment 7 David Carver 2007-10-19 13:57:39 EDT 
Hot Bug request information:

STAR - Standards for Technology in Automotive Retail
Release to be fixed in: 2.0.2

This is a hot bug because it is affect tool interoperability.  All other XML editors, do not require a specific order for the namespaces and prefixes to appear on an element.  The current situation requires that the attributes appear in a specific order.   This goes against the XML Specification which doesn't require the attributes to appear in a specific order.  The patch that is attached, makes sure that the all the namespaces are resolved on an element before checking the attributes for any prefixes.   This makes sure that the namespaces are resolved and known before the attributes are accessed.  This solves the issue of order dependency between the xmlns and the prefix that resolves to the namespace.

Because of the bug, users are getting false positives that their xml instances are valid due to the xsi:schemaLocation not being resolved to the XSI namespace, and causing the content model to not be shown in the graphical designer.

One of the problems we have here is that the following code is specifically looking for the xsi prefix (this was original code):

      if (prefix != null && unprefixedName != null && !prefix.equals("xmlns"))
      {
        if (prefix.equals(xsiPrefix) && unprefixedName.equals("schemaLocation")) //$NON-NLS-1$
        {
          visitXSISchemaLocationAttribute(attr, attr.getValue());
        }
        else if (prefix.equals(xsiPrefix) && unprefixedName.equals("noNamespaceSchemaLocation")) //$NON-NLS-1$
        {
          visitXSINoNamespaceSchemaLocationAttribute(attr, attr.getValue());
        }
      }

The issue here is that the variable:

xsiPrefix is hard coded to "xsi".   What actually needs to happen is that the value of the namespace needs to be checked to see if it matches the following value:

XML_SCHEMA_INSTANCE_URI

Which is set to the namespace: 

http://www.w3.org/2001/XMLSchema-instance

Programmically, editors should never rely on a specific prefix being set to a particular namespace, but should rely on the namespace that the element or attribute resides in.  There is nothing stopping somebody from using the xsi prefix for another namespace in their XML and it's not an error to do so, but goes against best practices.

I'll rework the code so that it relies on the value of the namespace that the particular attribute resides in to determine if it needs to do anything special.

Sorry for being late to review this, I've been away for a bit and then busy with other things. The new patch seems to address both the original issue and keeps content assist working. Normally I should stop at this point, but here are some further ramblings:

Dave, to address your xsi prefix concern: if I read it correctly, the original code does not seem to really hardcode the xsi prefix, but rather provide a fallback value. If you look in the org.eclipse.wst.xml.core.internal.contentmodel.util.NamespaceAttributeVisitor.visitXMLNamespaceAttribute(Attr, String, String) method you'll see that the code will eventually use the actual value defined in the xml instance for the schema instance namespace. The default value would be used only when the schemaLocation or noNamespaceSchemaLocation attributes occur before the xsi prefix declaration. So yes, there's a small possibility for the wrong schemaLocation attribute being used but one could speculate that the probability for that is fairly small.

If there's one small concern I have with the patch it's that the code now iterates over the attributes list twice, where the previous code was doing it once. It's hard to say for sure without measuring but this could potentially sacrifice a bit of performance for absolute correctness. I'm not sure how many times this code is called during normal use. David, Nitin, do you see any potential fallout from this? If we leave the xsi prefix "hardcoding" aside, the code could be changed a bit to save a reference to the schemaLocation and noNamespaceSchemaLocation attributes at the same time namespace prefixes are retrieved and process them later after the loop ends. This would avoid iterating twice over the attributes and still achieve the end result we're looking for: set the location hints after the namespaces information has been retrieved.

Yes, I saw that.  My only concern is that this particular Class tends to be overridden, and if this happens who knows what is going to happen.  In general from an XML standpoint, the prefix shouldn't be used by an application, it should always be checking the namespace that the element resides.   Again prefixes can be specified to whatever they want, but namespaces are the sole unique identification of a particular element or attribute.   Best practice is not to use the prefix beyond it's use for serlization, you should always compare against the namespace since this should be attached and identified with the element/attribute in the DOM anyway.

With that said, I can see uses for the prefix, especially within an XSLT editor.  I typically steer users of my standard away from relying on a particular prefix and have them look for the namespace to identify where the element/attribute belongs.
