Comment 2 Vlad Klicnik 2002-01-24 14:42:28 EST 
Hmmm .... this information is currently not available at runtime (there is a 
buildmanifest.properties files generated by core in the root but it is really 
not easily usable in the current form).

I had a proposal to JW a while back that would address this, but made little 
progress. It went something like this:

* we support 4-part version identifiers for plugins. This is currently on the 
2.0 list of enhancements (the install/ update layer already supports this). The 
version id would be M.m.s.V, where M=major, m=minor, s=service, V=some version 
string (uninterpreted other than for ability to sort on this). 

* have our build process drop a "magic" properties file into each plugin, 
identifying the build# and CVS version. Downside: plugin is modified as part of 
build (new file), Upside: self-contained (besides, the build is adding .jars to 
the plugins anyways)

* have the plugin.xml parsing support do the following when parsing the plugin 
version# ... parse plugin.xml version="", if 4-part we are done, if 1-3 part 
look for "magic" file, if not found we are done, else use the CVS version stamp 
as part 4 (note: do not use build# because we want to only mark the plugin with 
a new version if there were source changes (ie. new CVS version)).

* if needed have the runtime also associate the build# with the plugin (not as 
part of the version, a separate field)

This way, the precise version identification/ build# would be part of the 
runtime. While self hosting we would NOT have to spin the plugin.xml version="" 
manually to mark differences in the plugin. We would specify 3-part versions in 
our plugin.xml, the build process would drop CVS version and build# into each 
plugin, and runtime would process it. The API to access this would be via the 
plugin registry (4-part version, new method for build id).

Note: one could argue tat the same logic could just look for the 
buildmanifest.properties file and use it if found. The problem is that in the 
install story the code I am running may be a mix of multiple installs, one on 
top of another, so it is not clear where the buildmanifest.properties would go 
and which to use. If we drop the "magic" files into each plugin things are 
unambiguous.

Comment 17 DJ Houghton 2003-03-07 15:53:51 EST 
Created attachment 3918 [details]
method for determining build id

Attached is a method to add to PlatformLogWriter which will determine the build
id based on the "0" key in the about.mappings file in the org.eclipse.platform
feature directory. All this information is hard-coded and should be removed
once bug 32156 is resolved.

Recommend that the new method be referenced in #writeHeader:

// Write out the build id if we can get it
String buildID = getPlatformBuildID();
if (buildID != null) {
	write("org.eclipse.platform build id: "); //$NON-NLS-1$
	writeln(buildID);
}

Comment 21 Jeff McAffer 2004-04-14 17:29:42 EDT 
There does not seem to be a way of getting the build id.  It is currenly in 
some file in the default primary feature's plugin (or something like that). 
Unfortunately, the runtime doesn't know anything about this in general.  In the 
new world we are just told a set of plugins and which application/product to 
run.  Much of this information comes too late for consistent inclusion in the 
log.  Later on in the run we may be able to reverse engineer some thing via the 
product but only if the product were supplied by a extension.  If a product 
provider defined it then we have no way of knowing where the 
product/feature/plugin is that might contain the build information.

Product could be enhanced to know this information but even still having a 
product is not a requirement. 

So, the only other option that comes to mind is to have a system property set 
in the config.ini.  For example, eclipse.build or some such.  The build 
mechanism (releng that is) would have to be changed to update this property 
with the appropriate value.  

cc'ing Sonia to see if she has any thoughts on how this could be accomplished.  
Basically in the place that we fill in the about.ini (I think) file with the 
build id, put it in the config.ini as well.
