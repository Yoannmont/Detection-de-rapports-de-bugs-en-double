Comment 2 Ines Khelifi 2004-04-15 09:49:13 EDT 
I think you have this backwards.
The setContainerMethod is described in the Eclipse API as follows:

http://fullmoon.torolab.ibm.com/downloads/documentation/2.0/html/plugins/org.ec
lipse.platform.doc.isv/reference/api/org/eclipse/ui/dialogs/CheckedTreeSelectio
nDialog.html#setContainerMode(boolean)

"If set, the checked /gray state of containers (inner nodes) is derived from 
the checked state of its leaf nodes."

If I setContainerMode(true), all the nodes will be selected on "Select All", 
since the checked/gray states of containers is derived from the checked state 
of its leaf nodes.

If I set setContainerMode(false), only the root nodes will be 
selected/deselected on "Select All", since the checked/gray states of 
containers is NOT derived from the checked state of its leaf nodes. You could 
argue that the leaf nodes should be selected, and not the root nodes. That is 
just a design issue that the CheckedTreeSelectionDialog decided to handle by 
selecting root nodes only, instead of leaf nodes.

I am attaching pictures of the two different behaviors.

Comment 9 Ines Khelifi 2004-04-15 11:21:04 EDT 
Yeah, it would make sense for the "Select All" button to select all the items 
in the tree. I was under the impression that doing so would break the 
ContainerMethod behavior (as far as what should be checked depending on 
roots/leafs), but it doesn't seem to. I hadn't thought it all out:)

From what I have observed, the CheckedTreeSelectionDialog uses two different 
types of trees:
- ContainerCheckedTreeViewer if ContainerMode is true
- CheckboxTreeViewer if ContainerMode is false

On SelectAll, the CheckedTreeSelectionDialog calls the CheckboxTreeViewer's 
setCheckedElements method on the root items. The ContainerCheckedTreeViewer 
listens for check state changes and automatically updates roots' children if 
it’s checked. The problem is that the CheckboxTreeViewer does not. It only 
checks what it is asked to check.

The obvious and quick fix would be to pass all the elements to the setChecked
() method if "Select All" is pressed and if the ContainerMode flag is set to 
false. Since this was originally your code, do you have any thoughts, or more 
efficient alternatives?
Comment 10 Martin Aeschlimann 2004-04-15 11:47:04 EDT

So I guess I would do it like that.
Comment 11 Ines Khelifi 2004-04-15 14:57:50 EDT 
I am attaching a proposed patch. The widgetSelected method that handles 
the "Select All" button selection listener 
(CheckedTreeSelectionDialog#createSelectionButtons) now looks like:

Object[] viewerElements = fContentProvider.getElements(fInput);
if (fContainerMode)
  fViewer.setCheckedElements(viewerElements);
else {
  for (int i = 0; i < viewerElements.length; i++)
    fViewer.setSubtreeChecked(viewerElements[i], true);
  }
  updateOKStatus();

It seems to work, but the only thing I am confused about, is that for 
the "setSubtreeChecked" documentation, it states that:

"Sets the checked state for the given element and its visible children in this 
viewer. Assumes that the element has been expanded before. To enforce that the 
item is expanded, call <code>expandToLevel</code>for the element."

In my case, the only visible elements on start up are the root elements (the 
children elements are not visible). But when I expand the roots, their 
children are all checked as well (which is what I want). I am just confused 
about the "Sets the checked state for the given element and its visible 
children in this viewer", and trying to understand how it works.

Comment 13 Ines Khelifi 2004-04-16 11:01:43 EDT 
Ok, after some investigation, I now understand what's going on:)

To ensure that the elements have been expanded, the "CheckBoxTreeViewer" 
methods that change the checked state of tree items call an "internalExpand" 
before checking the elements. These methods end up creating the items that 
need to be checked. Which means that in our case, after a "Select All", the 
entire tree will be built.

The patch works for now, but I will log another bug report to maybe re-design 
the code in the "CheckBoxTreeViewer" class, so that the checking methods don’t 
have to create the items when their parents are not expanded. Saving their 
checked state instead for later use would be a more efficient approach.
