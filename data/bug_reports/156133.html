On using prefixes vs. suffixes it should not really matter as long as the tagging is consistent for a particular release and the qualifiers correctly increment for each build of the release.

In this case I'm glad we caught this.  If we had used suffixes then the org.eclipse.core.runtime.compatibility.registry bundle would have been released in 3.2.1 without its version being incremented properly to reflect a bug fix.
Comment 15 David Olsen 2006-09-07 20:05:00 EDT 
> On using prefixes vs. suffixes it should not really matter
> as long as the tagging is consistent for a particular
> release and the qualifiers correctly increment for each
> build of the release.

That is not quite true.  There have been some case where a feature's version number decreased from build to build even when all plugins' version numbers increased.  The hash in qualifier of the feature's version number is taken from the qualifiers of its plugins, not from the full version number.

So if a plugin's version changes from, for example, 3.2.0.v20060801 to 3.2.1.r321_v20060901, and if the feature doesn't change in any way (which would be acceptable if the feature's third digit has already been increased since the last public release), then the feature's version number will decrease because its hash will decrease because the plugin's qualifier has decreased.

It is important that qualifiers always increase, regardless of what happens to the rest of the version number.  After seeing many of the versioning problems in the 3.2.1 maintenance builds, I'm convinced that any release specific strings should be a suffix rather than a prefix.
