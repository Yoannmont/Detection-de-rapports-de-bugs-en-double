Description Stefan Xenos 2005-02-16 16:53:51 EST 
These are similarly-named preference pages with a very simiar function. I am
continually going to one by accident when I'm looking for the other.

These should either be placed directly next to each other in the preference pages.

Alternatively, they could be merged into one page. For example, a single
preference page containing a tabletree should be able to do the trick, where the
current contents of one of the preference pages would be a top-level node in the
tabletree.

These pages should also either permit in-place editing of the templates or make
it clear that the textbox at the bottom is read-only. The flashing cursor makes
the templates appear to be editable.
Comment 1 Stefan Xenos 2005-02-16 16:54:37 EST 
In case this gets renamed, I'm referring to the Code Style -> Code Templates and
Editor -> Templates preference pages
Comment 2 Martin Aeschlimann 2005-02-17 04:22:35 EST 
I don't think we can merge them, they are not the same. But maybe we find better
names. Code Generation snippets instead of Code Templates?
Or naming 'Templates' Editor templates?
Comment 3 Tom Hofmann 2005-02-17 04:38:15 EST 
Other possible terms:
- Patterns (they really are, Erich might not like this one, though)
- Snippets

Editor Templates vs. Code Templates would make sense as well. As it is now, it
looks like Code Templates are a specialized version of Templates.
Comment 4 Martin Aeschlimann 2005-02-17 04:44:30 EST 
The nasty thing is that the 'Editor Templates' would be a child of the 'Editor'
node.

Other suggestion is Code Assist Templates. That one has the problem that we then
have Code Template and Code Assist Templates. So then maybe we rename Code
Templates to New Element Templates, or Generation Templates?
Comment 5 Stefan Xenos 2005-02-17 19:35:02 EST 
I understand that implementation-wise, they are probably quite different.
However, they are the same insofar as they both configure macros that get
expanded in your source code. Merging them would make everything easier to find
and would fit in with the goal to reduce preferences.

One way to merge the pages:

- The current Code Templates could show up in the Editor/Templates page. 
- They would retain their current names and "Edit..." behavior
- They would either gain an additional description or the description column
would remain blank.
- The "Code" subtree would all end up in a new "generated java" context, and the
"Comments" subtree would all end up in a new "generated javadoc" context.
- Allow users to sort by column, so they can quickly find what they're looking for.

Another way to merge the pages:

- Create a page that looks like the current Editor/Templates page, but the
upper-left control would be a tabbed workbook.
- The workbook would have a "code assist" tab that shows the table currently in
Editor/Templates.
- The workbook would have a "generated snippets" tab that contains the tree from
Code Style/Code Templates.
Comment 6 Stefan Xenos 2005-02-17 19:55:37 EST 
If they remain as separate preference pages, putting them right next to each
other would really help.

For example, we could move 
/Java/Code Style/Code Templates/ to /Java/Templates/Generated Code/ 
and move 
/Java/Editor/Templates/ to /Java/Templates/Code Assist/

Also, the differences between the two would be more clear if the text at the top
of the preference page were changed. Rather than "Create, edit, or remove
templates:", it could say "These macros can be expanded by pressing ctrl-space
in the editor:". The new keybinding API should let you query for the correct
binding.
Comment 7 Martin Aeschlimann 2005-02-18 03:40:23 EST 
The differences between Code templates and Editor templates are:
You can rename, add, remove editor templates and modify them as you like. They
are only for code assist. You can define your own variables ans when applied you
can fill these variables using the linked (template) mode.

Code templates are a fixed set of templates the code generation actions require.
They can not be removed nor added. When modified, certain rules are checked,
e.g. you cannot have a new file template without a package_declartion variable.
You can not introduce any new variables, and when applied, the template will not
be in linked mode.

I don't think we help the user by merging the two on one page. I'm also not yet
conviced that bringing them in a own category is that helpful. There are many
aspects a users tries to find certain settings. We are currently trying to group
all editor settings together so you can configure the editor in one pass.
Links and the preference search should help finding settings related.

In fact I think the preference search solves your request.

We should change the description on the editor templates as you stated.

'Macros' defines quite well what the editor tempates are. Tom?
Comment 8 Tom Hofmann 2005-02-18 04:25:36 EST 
(In reply to comment #7)
> We should change the description on the editor templates as you stated.
> 
> 'Macros' defines quite well what the editor tempates are. Tom?

Hm, sounds certainly familiar to emacs veterans and C programmers... For me, a
macro is more like a script that may involve executing commands, while a code
template/pattern is purely textual. But this may just be me...

Not sure whether we should give up the established term 'template' though - I
don't think it is all that evil. I favor qualifying the name by prepending a prefix:

'code template' or 'code generation template' or 'editor template' or 'code
assist template'.

Merging the two pages on a single page does not make sense for me either: the
code templates under 'Code Style' make sense togehter with the formatter and
notation settings. No way we can fit it the code assist templates there or
vice-versa.
Comment 9 Stefan Xenos 2005-02-18 18:26:56 EST 
Okay, I suppose there's a case for leaving them separate. :-)
Comment 10 Martin Aeschlimann 2005-03-07 12:22:17 EST

Comment 11 Tom Hofmann 2006-08-14 10:00:45 EDT
