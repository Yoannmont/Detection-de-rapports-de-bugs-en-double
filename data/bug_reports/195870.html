Description Scott Paxton 2007-07-09 14:15:23 EDT 
While I was investigating the problem in bug 195867 I repeatedly saw a message in the Error Log view: "Error loading model for file "...".  The faces-config.xml file that I used has entity references to other xml files but from within the /META-INF folder in a jar none of these entities can be resolved (the full stack trace below shows the root cause is a FileNotFoundException because the system is looking in completely the wrong place).

I have found from experience with similar tooling that dealing with faces-config files inside jars is very tricky.  To prevent problems like this from occuring way down in the EMF framework I had to parse the file(s) into a Document myself and then turn the existing Document into an EMF model later.  It seems that "jar:" URLs are not handled entirely well, especially for entity resolving cases.  Furthermore, any parsing failures are sometimes apt to cause jar-locking behavior like in bug 195867 if the exceptions are not properly handled.  (I can't tell if this is actually related to 195867 or not - very intermittently the scenario here was enough to cause similar jar-locking).
