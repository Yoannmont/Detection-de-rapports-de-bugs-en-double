Description Bart Kummel 2007-08-13 06:54:52 EDT 
Build ID: I20070621-1340

Steps To Reproduce:
1. Generate an entity from a table in a large database schema in an IBM AS400 DB2 database. (May occur with other databases too, but I did not test that.)
2. Edit the entity (either by typing manualy or using the properties editor)
3. After every couple of keystrokes, the "resyncing JPA" thread runs and blocks the input. This takes several seconds each time.


More information:
As said, we're using an IBM DB2 database running on an AS400 system. We have over 350 schemas in the database. The client machines Eclipse is running on are brand new, very fast Dell desktops with 4 GB RAM, so that cannot be a bottle neck.

Comment 4 Karen Butzke 2007-08-20 10:48:48 EDT 
Max,

We would like to fix this problem for our 1.0.1 maintenance release.  I am not positive that we can fully fix it, but we can at least make it better for this stream and then work to make it even more performant in the 2.0 release.  

The key problem is our access to java and resolving classes in the JDT.  The fix to bug 198934 was to switch to using the ast resolved bindings.  The patch i've included on this bug uses those resolved bindings in more places, thus taking advantage of the performance hit that occurs when you resolve all bindings on a class.  

The problem with searching all the java files for annotations will only occur once when the project is built on startup.  After this one search through all the files, we just listen with the JavaElementChangeListener for more files being added and removed.  The real performance problem is in setting up all the defaults and finding all the cross references involved in a jpa project.  Target entity, inheritance parent classes, testing whether an attribute defaults to basic or embedded.  All of these involve resolving a class name or attribute type to a particular class in the jdt.  This is most of the work happening in the resynching jpa thread. 

I am going to do some more performance testing and see what other ideas I can come up with to improve things.  One possibility that we are not doing yet is to interrupt the resynch thread when a change occurs and restart it instead of waiting for it to finish before running it again.  The resynch thread is already a background thread, so you should be able to continue making changes while it is running.

Comment 8 Karen Butzke 2007-08-21 09:01:12 EDT 
Bart, 

I realized after saying it's a background thread that I needed to catch those exceptions.  If you continue to make changes you can get our model in to a state where it is different from the JDT model because we have not yet received the event notification.  So, at this point I am catching and logging all exceptions that occur in that thread (not checked in yet).  Then they just go to the error log and do not smack you in the face :)  If another change occurs in the JDT model we will update and run the thread again.

Comment 16 Bart Kummel 2007-09-25 15:50:18 EDT 
Hi Neil, Karen and others,

I did had a change to look at the new version. It is better, but still not ideal. The thread runs entirely in the background now, so it never blocks Eclipse entirely. But it still takes up a lot of CPU time and slows down the IDE significantly. When I opened the "progress" window, I sometimes saw more than one "Resyncing JPA" threads running at the same time, I don't think that's efficient. I also notice that the thread is started very often. I still think it may be a good option to add some sort of (configurable) time out. That way I could for example configure the thread to run not more often than once per 2 minutes. The point is: you can spend a lot of effort speeding up the resyncing, but if you're resyncing against a database that is on a busy or slow database server, the resyncing thread itself is probably not the "bottle neck". A more advanced solution would be to cache the DDL information locally to prevent connecting the database every time.

I hope this information is helpful to further improve Dali!

Best regards,
Bart Kummel
