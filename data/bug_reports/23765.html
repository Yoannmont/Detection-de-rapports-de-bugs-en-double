Comment 1 Kevin McGuire 2002-09-18 14:39:54 EDT 
I recall that there was an enhancement report for displaying the list of items 
that are about to be committed to allow the user to both see the set of items 
to be committed, and to allow them to untoggle those they don't mean to.

Presently we show the list of unadded resources that can be checked off to be 
committed.

These two can be combined: by showing the list of resources that are to be 
committed (with checkbox column), with label decorator to indicate those that 
are new.  New ones that are checked will be added and committed.

On a related note, it would be good if the list was a table where first column 
was checkbox, second resource name, and third resource path.  Currently second 
is path but paths are hard to read and put the important info at the end.

Kevin McGuire asks (2003-01-09 18:01):

-comment somehow?  I am interested in what work flow you are envisioning.

It has been ruled that bug 43395 is a duplicate of 23765.

43395 is a request to let me scroll through the "changes" I'm about to check-
in, i.e., the diffs.  I want to peruse these changes so I can document them 
accurately as part of the check-in.  My memory is not perfect, especially if 
I've had the files checked-out for a while or if I've made complex changes to 
many files.  I want to make sure that the person trying to understand my check-
in has a reasonable chance.  Especially if that person is me 3 years from now.
Comment 8 Xiaolong Hao 2004-08-06 18:30:29 EDT 
Two reasons:

1) developers want to put different comments for differnt group of changes they 
made. Using tortoiseCVS, commit on root dir and pick the group of files and put 
the comments and commit them. Repeat that process for other group of changes.
 
2) developers may only want to commit some of their changes to CVS and the 
changes are spreaded over a few places (ejbs, descriptors, servlets etc). In 
stead of going into each place and commit the changes, they simply do a commit 
on project root dir and using tortoiseCVS to pick the files they want to check 
in.

Comment 10 Jean-Michel Lemieux 2004-12-13 22:15:06 EST 
This has been resolved by new support in 3.1 for:
1. change sets in the synchronize view. You can write comments and then group
files into their respective change sets. The sets are persisted at shutdown.
This allows you to write comments and assign files to change sets as you develop.
2. the commit wizard shows the files that will be committed. this includes
outgoing deletions. You can remove files from the commit at this point and
inspect the files to write the commit comment.
