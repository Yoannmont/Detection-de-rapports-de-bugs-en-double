Comment 15 Brad Cupit 2007-10-05 08:40:13 EDT 
(In reply to comment #14)
>The fact that I still have a large number of plugins goes against the
>theory that this is the cause for the PErmGen errors.

The theory is not that a large number of plugins will crash Eclipse, but that a large number of plugins will crash Eclipse 3.3.1 under the default config.

The PermGen garbage collection heap is where the JVM puts all of that data that is permanent and will never get garbage collected, and loaded class files go there. The default setting for Sun JVMs is 64MB and when you run Eclipse (an enormous number of .class files) plus a large plugin (like WTP) the number of loaded class files in memory can exceed 64MB.

As a result, Eclipse started shipping out of the box with a larger-than-default PermGen size. Unfortunately this mechanism appears to have broken with Eclipse 3.3.1, hence the workaround mentioned in comments #2 and #7.

Eclipse details this in their readme file (<your_eclipse_install>/readme/readme_eclipse.html) in the "Allocating enough memory and solving OutOfMemoryErrors" section.

For my organization I'll be taking the default download of "Eclipse IDE for Java EE Developers", unzipping it, tweaking the ini file with the workaround, and zipping it back up. I'll then take that and put it on our network for other devs to use instead of the direct download from the Eclipse site. (Incidentally, this also gave me the idea of pre-installing a few plugins, thus making a custom build just for our organization!)

Next step would be to run JConsole - should be something like C:\Program Files\Java\jdk1.6.0_02\bin\JConsole.exe under Windows XP. 

When you start JConsole you attach to a running Java application, in this case Eclipse itself (which will be the BLANK entry), and you can actually *see* what is happening memory wise. 

For the record there is a really really spectacular crash that occurs AFTER the MaxPerm issue is out of the way and it happens once the Code Cache (the other non-heap memory pool) has been depleted. If you have, like me, a lot of classes and/or linked projects open then this happens within less than a week. The Code Cache never backs off so once I hit 90% now I simply restart Eclipse. 

You *might* be hitting this brick wall then again you might not. To date I have not found the magic incantation that allows a larger Code Cache to be used.

In any case jConsole will likely tell you what you are in fact hitting. I "tried all the fixes" too only to discover, to my amusement, that my MaxPerm size adjustments were not in fact "taking" and that I kept ended up with the defaults.

You can help track it down, since may be specific to your environment, or

In addition to JConcole just mentioned, there's a nifty Memory Monitor tool attached to bug 92250

This tool display perm gen space (and others) so you can watch to see 

  a. That it's really set to what you expect it to be, (just to confirm) and 
  b. if/when perm gen memory grows when you would not expect it to.

The first time you execute some function, you'd expect it to grow, since new classes are presumably loaded. But, if you do the same function over and over again, and the perm gen usage grows each time, then that is a sign of a "leaking classloader" (probably some custom one), and would deserve a bug against what ever component is most related to that function.
