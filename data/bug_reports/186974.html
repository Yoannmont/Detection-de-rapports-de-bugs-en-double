Description Michael Cheng 2007-05-15 06:12:51 EDT 
When the user uses MobileShell in fullscreen mode, some prefere to have the SIP button hidden to achieve a true clean fullscreen.  However, hiding the SIP button will prevent the VKB from coming up and that hinders the usability of application that wishes to input while in the fullscreen mode.  Since both behaviors are desired by programmers, I propose to add new API's to specify whether the SIP button should be hidden.
Comment 1 Mark Rogalski 2007-05-15 12:18:45 EDT 
It does seem like we need to make some improvement in this area> It would be nice if displaying the SIP button would be automatic according to the content of the screen. For instance, if it has text widgets, we know we probably want the SIP button. However, I don't think we can absolutely determine this since the developer could have his own input fields/mechanism that does not use eSWT Text widgets.  Therefore, maybe an API to control the SIP is called for. I imaging it would be an addition to the MobileDevice API.
Comment 2 Gorkem Ercan 2007-05-15 14:16:48 EDT 
What is the actual proposal. Is it a new method? Change to an existing method. I am a bit concerned that not all platforms can support this behavior so the  API change should reflect it.
Comment 3 Wolfgang Röckelein 2007-05-18 12:17:39 EDT 
I suggested to add another constant VK_AND_VKB_ALWAYS_OFF, which an implementation would be free to interpret this as VK_ALWAYS_OFF, where not otherwise applicable.
Comment 4 Mark Rogalski 2007-05-19 09:07:24 EDT 
Does the SIP button ever need to be controlled independently of the VKB itself?  I'm thinking probably not, since without the SIP button, you are also unable to change the type of VKB that is shown. So if you ever show the VKB, then you need to show the SIP as well. 

Therefore, I think Wolfgang's proposal is good, except I think the constant should be SIP_AND_VKB_ALWAYS_OFF.
Comment 5 Michael Cheng 2007-05-21 05:14:35 EDT 
After investigation, the SHFullScreen we use for hiding the SIP button works by reducing the Z order of the SIP button. Which means that we need to have a fullscreen shell covering it to make it hide; Leaving us being able to hide the SIP button only when the shell is fullscreen and without any command.
Comment 6 Mark Rogalski 2007-05-21 08:58:43 EDT 
So the SIP_AND_KBD_ALWAYS_OFF constant needs to be described as a hint with implementation specific behavior.  Gorkem, are you agreeable to this?
Comment 7 Gorkem Ercan 2007-05-21 09:19:12 EDT 
I agree that we should have it as a hint but I think we should reconsider not to have this since it can not be fully implemented on our current platforms (including S60).
Comment 8 Mark Rogalski 2007-05-22 21:34:01 EDT 
I think it is reasonable to add this function since it is very simple and fxies a legitimate usability problem. Let's do it.
Comment 9 Danail Nachev 2007-05-23 02:39:57 EDT 
One comment I want to make is the name of the constant. Is there a better name for it which does include the word "AND". I don't have much knowledge about the Windows Mobile mechanisms, but isn't it better if we name it something like "HIDE_VKB_BUTTON" or something like that. I think this way we more clearly express the intention of the constant.
Another thing which I'm concerned is the name of this button on other platforms. Is it possible that there is a SIP button only on Windows Mobile and on other platforms it has different name?
Comment 10 Mark Rogalski 2007-05-23 13:22:28 EDT 
Yes, the constant name is a sticky point. I don't even know what SIP stands for (secondary input program??). The problem with a separate constant that is unassociated to VKB_ALWAYS_OFF is that it leads to the expectation that the button can be hidden at any time, including when VKB could be on. I don't think that is a good state since the button also allows changing input type and user preferences around using the input type. What is the reason you don't like "AND"?  Maybe we could use "DISABLE_VKB". This implies that VKB is both off and will be kept from being invoked. But I worry a bit about it being confused with the VKB_ALWAYS_OFF constant.
Comment 11 Michael Cheng 2007-05-23 23:13:52 EDT 
I feel that since the hiding of SIP (soft input panel) button requires fullscreen to cover the button.  This functionality should be associated with the setFullscreen() method instead of the setVKStatus() method.  I think maybe changing the signature for the setFullscreen() method to take another argument would do it.  Do we need a name for this?
Comment 12 Gorkem Ercan 2007-05-31 09:49:14 EDT 
I have always assumed that MobileShell fullScreen removes both CBA and SIP areas. Is there a requirement to have CBA area visible bu SIP area visible?

Comment 14 Aleksi Uotila 2007-06-05 09:55:50 EDT 
Cannot we simply agree to do so that full screen is without that SIP? That's how it is going to be in S60. For this we don't need additional method overload nor we would have a method that would be only for Windows mobile. If you insist on having the new method it must state that the showSoftInputSelector is a hint. First, it doesn't mean anything on a device without touch screen support, second it may not be supported even if there's touch screen. Also I would reuse the VKB term as that  is already there and call this vkbSelector.
Comment 15 Mark Rogalski 2007-06-05 22:06:26 EDT 
Aleksi, You have a good point about the "Soft Input Selector" term, since we have not previously used it. I can modify it. However, I think we want to keep the ability to programatically display the SIP since there are several use cases for it in fullscreen mode. One is that developers may want a "kiosk" mode for apps where apps are covered but text widgets and input are still allowed. The proposed javaDoc does say the SIP "will be made visible, if the platform supports it." I think that is enough to inidicate that it is platform dependent.
Comment 16 Aleksi Uotila 2007-06-06 05:32:36 EDT 
I agree, the use cases are valid.
Comment 17 Mark Rogalski 2007-06-06 09:32:04 EDT 
I changed the "Soft Input Selector" terminology to be "Virtual Keyboard Selector":

	 * If the <code>showVKBSelector</code> argument is <code>true</code>,
	 * the Virtual Keyboard Selector used for displaying a virtual keyboard or other input
	 * mechanism will be made visible, if the platform supports it. Otherwise, a
	 * Virtual Keyboard Selector will not be visible.
Comment 18 Michael Cheng 2007-06-28 03:53:57 EDT 
API implemented as discussed.
