Comment 5 Mik Kersten 2007-07-06 16:13:17 EDT 
Jeroen: thanks for the profile info.  Mylyn relies on calls to JavaCore.create(..) to be fast.  In general they should be, however, if Eclipse is running low on memory and there are a lot of elements they might cause slowness.  We will investigate this for 2.1.  In the meantime, could you try to increase your heap size and report if that affects the performance of running Clean.  The size of the Java model cache is proportional to the heap size on Eclipse 3.3.  I do occasional cleans of over a hundred projects and have not seen this performance problem.  

In the meantime, if this is getting in your way the work-around would be to deactivate the task before you do a clean.

Comment 13 Mauro Molinari 2008-07-18 02:50:32 EDT 
I have the exact same problem, as discussed in JDT newsgroup (I didn't know it was a Mylyn fault).

I'm heartily asking to increase the severity (and priority!) of this bug because of two reasons:

1) it is sensitively affecting the user experience: I encountered this problem in every architecture I tried (Windows 32-bit, Windows 64-bit, Linux) and I heard many people talk about the nightmare of rebuilding the whole workspace because of freezing problems of the Eclipse UI (and this is one of them, maybe the biggest I encountered!)

2) the workaround mentioned by Mik in his comment #9 is not a "real" solution: first of all because people often do not know that the source of the problem is Mylyn (I discover myself after long investigation by looking at jConsole thread dumps and with the help of eclipse.tools.jdt newsgroup!); secondly, if Eclipse crashes while you have an active task, when you start it again it begins almost immediately to do a full rebuild of the workspace (if you have "Build automatically" enabled), so you may not be able to deactivate the task before the UI completely freezes (for minutes and minutes, or for hours at worst!)

3) if you have HUGE workspaces (I'm talking about tens of thousands of files), rebuilding the whole workspace becomes an impossible mission... Once, after waiting for almost two hours, I had to kill Eclipse, restart it, quickly disable the "Build automatically" option and MANUALLY build a project at a time in order to continue my work...

Please Mik, consider to investigate this issue as soon as possible! :-(

1) prepare a HUGE workspace and check the "Build automatically" option
2) activate a task
3a) clean the whole workspace
-- or --
3b) kill Eclipse and restart it

Comment 20 Martin Ko?? 2008-09-26 11:08:21 EDT 
Created attachment 113597 [details]
Suggested patch

The problem is in ResourceChangeMonitor - it does not ignore derived files or derived folders (they are modified with build). This patch suggest following:
1) IResourceDeltaVisitor will not visit children of a derived IFolder
2) All derived resources will not be  added for processing with ResourceInterestUpdater

Best performance of build can be achieved with all Output Folders marked as derived (it skips processing of all .class files)

This patch was tested on three different computers - now full build of workspace (with activated task) takes minutes instead of hours.
Comment 21 Shawn Minto 2008-09-26 11:24:45 EDT 
+1 for applying this patch.  Derrived resources are ignored in Mylyn currently, but only when they are added to the context.  By limiting them during the resource visitor, this will stop mylyn from trying to compute too many changes that wont affect the context because of this rule.
