Comment 11 Usha 2003-12-04 11:19:12 EST

Comment 12 Dirk Baeumer 2003-12-04 11:27:24 EST 
The requirement is that the files are writable. We ensure this by calling 
validateEdit. But if your third-party plug-in Team provider doesn't implement 
that hook correctly refactoring will fail.
Comment 13 Usha 2003-12-04 12:34:17 EST 
I would like provide more details on the issue. 
1)For java classes, If the user who is trying to execute the refactor->rename 
process .Then Refactoring rename works OK. 
2)If the user does not have execute access. Then we display an error message. 
It looks like eclipse cannot handle this exception ?

2)For java packages. We dont support rename/move of folders. So we display an 
error message. "Cannot move or rename fodlers". It looks like again that 
Eclipse caanot handle this execption. 

Thanks
Comment 14 Dirk Baeumer 2003-12-04 12:57:16 EST 
Refactoring reacts on the value returned from IWorkspace.validateEdit. Seeing 
a dialog doesn't mean that the repository provider returns the right status. 
The status returned from the repository provider has to be not OK in that case.
Comment 15 Usha 2003-12-04 13:01:18 EST 
Is there any method we could use to state to give an indication to eclipse that 
refractoring should fail and the process should not continue.
Comment 16 Dirk Baeumer 2003-12-04 13:21:20 EST 
Refactoring calls the method IWorkspace.validateEdit to ensure that is can 
modify the file. If the return value of the method is a status != OK then the 
refactoring will not proceed. This covers the rename of a file as well.

I am not aware of any hooks to validate the move or rename of a folder. Moving 
to platform core to comment on this.
Comment 17 DJ Houghton 2003-12-04 14:19:59 EST 
There is currently no pre-validation for the resources APIs. 

This was mentioned in preliminary discussions about the 3.0 plan (in the area of
refactoring) but it did not make it onto the plan.
Comment 18 Usha 2003-12-04 14:23:27 EST

refactoring has failed and hence the reason for the error message we are 
getting. Can you please verify this.
Comment 19 Dirk Baeumer 2003-12-04 17:49:23 EST 
This is correct for rename/moving a package. Pre-Validation existing for 
rename/moving a compilation unit.
