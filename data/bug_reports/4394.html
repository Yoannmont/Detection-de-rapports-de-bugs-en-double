Both Tree and TreeItem act as containers of TreeItems; however, Trees are not 
themselves
TreeItems and cannot be treated as such.  A Java interface can be used to 
capture the
common tree item container API (getItemCount and getItem), and also makes for a
convenient place to put protocol for creating tree items and adding them to 
their container.
Both Tree and TreeItem can implement this interface.

public interface ITreeItemContainer {
	public TreeItem createTreeItemChild(int style);
	public TreeItem createTreeItemChild(int style, int position);
	public int getItemCount ();
	public TreeItem [] getItems ();
}

This allows TreeExample.expandTree to take a ITreeItemContainer as a parameter.
Clients that need to discriminate between Trees and TreeItems must use 
instanceof; 
e.g., see getPath.


I think this nicely fixes problem #1, and leads to a simpler API for the client 
that is no
more difficult to implement that the previous API.

I'll start on #2 once you've agreed that my solution to #1 is a good use of a 
Java interface

Comment 15 Steve Northover 2008-06-18 19:06:44 EDT 
First off, yes there are design flaws in SWT!

The class GC was not intended to be subclassed.  It was not written such that a person could subclass it, reimplement certain methods and get something useful.  Furthermore, it's implementation is platform specific and as such, could fall victim to the dynamic version of the fragile superclass problem.  Instances of GC are created automatically within the toolkit, for example during paint events.  For subclassing to be supported, hooks would be needed to support an alternate GC implementation. But more to the point, it is written to be as fast as possible and not intended to be pluggable.  The idea here was to avoid all possible extra overhead and complexity in design.  Since GC is not intended to be pluggable, then pluggability was not coded into the implementation.  Should this situation change, then refactoring the code will be necessary.

The class Device is not final.  It is subclassed by Printer from outside of the graphics package.  The implementation however, is very platform specific.  The idea here is to allow printing to be removed from SWT by deleting the printing package.

I do agree that the implementation of SWT could be restructured.  In the beginning, there was an effort to avoid layers such as implemenation peers for performance reasons.  Restructuring and retesting everything at this point would be a big effort, something that might make sense for e4.

Comment 17 Viliam Durina 2008-06-19 08:06:47 EDT 
We need common interfaces for another simple thing: We develop a framework that generates UIs and has a control factory to create controls. Control factory can generate either, say, Label or CLabel, Combo or CCombo or even custom implementation etc. Common interfaces would make such thing lot easier: look at the code of org.eclipse.jface.viewers.ComboViewer: it's source would be one half shorter with ICombo, and even able to use other Combo-like implementations.

I workaround it by defining my own interfaces and subclassing like this:

[code]
public interface IButton extends IControl {
	public void setText(String text);
	public String getText();
	public void setImage(Image image);
	public void addSelectionListener (SelectionListener listener);
	//.... other button-like methods
}

public class StandardButton extends Button implements IButton {
	public StandardButton(Composite parent, int style) {
		super(parent, style);
	}
	protected void checkSubclass() {
	}
	// no other code
}
[/code]

It works for me except for CCombo (see bug 235968) and Table/TreeItems of virtual tables (i.e. not created by me).

Comment 19 Steve Northover 2008-06-20 14:30:03 EDT 
L. Mihalkovic, error(int) just calls SWT.error().  

Even if it did more, the intent is for it to clean up any operating system state as necessary before throwing the exception.  For example, a mouse grab might be released on a button.
