Description Danny D'Amours 2005-09-26 16:35:18 EDT 
Unable to add RPM installed Tomcat server.

As an RPM (Fedora Core 4) Tomcat 5 is installed as:

ls -l /usr/share/tomcat5/

total 40
drwxr-xr-x  2 root root 4096 Jul 27 14:46 bin
lrwxrwxrwx  1 root root   23 Jul 27 14:46 common -> /var/lib/tomcat5/common
lrwxrwxrwx  1 root root   12 Jul 27 14:46 conf -> /etc/tomcat5
lrwxrwxrwx  1 root root   16 Jul 27 14:46 logs -> /var/log/tomcat5
lrwxrwxrwx  1 root root   23 Jul 27 14:46 server -> /var/lib/tomcat5/server
lrwxrwxrwx  1 root root   23 Jul 27 14:46 shared -> /var/lib/tomcat5/shared
lrwxrwxrwx  1 root root   23 Jul 27 14:46 temp -> /var/cache/tomcat5/temp
lrwxrwxrwx  1 root root   24 Jul 27 14:46 webapps -> /var/lib/tomcat5/webapps
lrwxrwxrwx  1 root root   23 Jul 27 14:46 work -> /var/cache/tomcat5/work

Trying to add server gives "The Tomcat installation directory is not correct. It
does not point to a valid Tomcat installation."

Workaround appears to be installing Tomcat from a .tar.gz archive where Tomcat
is installed to a single directory.

See: http://dev.eclipse.org/newslists/news.eclipse.webtools/msg05537.html

Comment 2 Henning Heitkötter 2005-10-14 09:03:47 EDT 
I had the same problem on a SuSE 10.0 installation.

Checking the verifyInstall.properties file
(http://dev.eclipse.org/viewcvs/index.cgi/jst/components/server/plugins/org.eclipse.jst.server.tomcat.core/verifyInstall.properties?cvsroot=WebTools_Project#rev1.5)
I discovered, that the file "common/lib/servlet-api.jar" doesn't exist in my
installation (from a rpm shipped with the distribution), neither as a real file
nor as a link (instead a link named "[servletapi5].jar" exists). I created a
symbolic link servlet-api.jar->[servletapi5].jar and it worked.
Comment 3 Larry Isaacs 2006-11-19 00:26:51 EST 
A fix has been released to HEAD which should appear in the next WTP 2.0 integration build.  Currently, the only instance of this issue I'm aware of is the instance cited by Henning with Tomcat 5.0.x.  Now, the installation directory will be checked for both servlet-api.jar and [servletapi5].jar.  Also, the error displayed when the directory isn't considered valid will be more specific about why it isn't valid.

To deal with unanticipated issued of this kind in the future, a system property may be specified in the Eclipse startup consisting of "org.eclipse.jst.server.tomcat.core." followed by the name of the property in the verifyInstall.properties file within the Tomcat plug-in.  The string value it specifies will be used in place of the one in the verifyInstall.properties file.

Since this is a "backdoor" hack that will hopefully not need to be used, the syntax of this string shouldn't be considered public API.  It's syntax may change as other improvements are added.  The current syntax is a comma separated list of file and folder paths.  A file path may have a '|' character appended followed by an alternate name for the file.  If the original file path contains a directory path, the alternate name will be appended this directory path. Additional alternate names may be specified in the same manner.

Comment 5 Larry Isaacs 2007-01-30 14:57:56 EST 
Verified the implemented behavior is working as expected.  Note that this bug specifically references creating a Tomcat runtime.  I'm not seeing a problem any longer creating Tomcat runtimes on packaged Tomcat installs on various Linux systems.  An more specific error message appears if there is a problem.

To be addressed separately is creating a new Server from runtime created from a packaged Tomcat.  If the user running Eclipse doesn't have permissions to read any of catalina.policy, catalina.properties, server.xml, tomcat-users.xml, or web.xml from the installation's conf directory, an error message is displayed claiming the configuration is corrupt or incomplete.  Permission must be granted before the server can be created.  There may be additional problems when the server is run.  The extent of these problems varies between different Linux distributions.
