Description Jeff Dean 2004-09-22 19:10:23 EDT 
Under certain circumstances, when I have aligned columns of code with tabs, the
alignment appears correct but when viewed in an external editor, or when looking
at the column position in the status line, I can see that the columns are not
really aligned.  I have tabs set to 4 and use Tabs, not spaces.

Here is an example; assuming that bugzilla doesn't eat the tabs, there is one
tab before "private" and one tab before the inline comment.  If you display this
in Eclipse with tabs set to four, you'll see that the comments are aligned. 
However, if you place the cursor in front of the comments, you'll see on the
status line that one is in column 37 and the other is in column 33.  If you load
this into an external editor that allows you to control tab width, you'll see
that the comments are not aligned.

public class aaa2
{
	private static Map queueMap;	// list of all NotifyQueue objects
	private static Timer timer;	// timer
}
Comment 1 Dani Megert 2004-09-23 03:23:46 EDT 
The other editors have probably set tab stops to 8. If you change it in Eclipse
to 8 the text will be aligned.

I agree that the naming of the preference "Displayed tab width" is misleading.
We should change it to something like "Tab stops width".

Note: if we change this we also have to adapt the text editor preference page

but the status bar reports different column values.

Comment 8 Felipe Heidrich 2004-10-08 12:30:44 EDT 
Run the snippet:

import org.eclipse.swt.SWT;
import org.eclipse.swt.custom.*;
import org.eclipse.swt.graphics.*;
import org.eclipse.swt.layout.FillLayout;
import org.eclipse.swt.widgets.*;

public class PR74737 {
public static void main (String [] args) {
	final Display display = new Display();
	final Shell shell = new Shell(display);
	shell.setLayout(new FillLayout());
	int style = SWT.BORDER | SWT.V_SCROLL | SWT.H_SCROLL | 
SWT.FULL_SELECTION;
	final StyledText text = new StyledText(shell, style);
	text.setText("private NotifyStr mailFrom; \t// originating eMail 
address\rprivate NotifyStr mailSubject;\t// Subject for eMail");
	Font font = new Font (display, "Lucida Console", 9, SWT.NORMAL);
	text.setFont (font);
	GC gc = new GC (text);
	final int tabWidth = gc.stringExtent("    ").x;
	gc.dispose ();
	text.addListener(SWT.Paint, new Listener() {
		public void handleEvent (Event e) {
			GC gc = e.gc;
			gc.setForeground(display.getSystemColor
(SWT.COLOR_BLUE));
			gc.setLineStyle(SWT.LINE_DOT);
			Rectangle ca = text.getClientArea();
			for (int x = ca.x; x < ca.width; x += tabWidth) {
				gc.drawLine(x, ca.y, x, ca.height);
			}			
		}
	});
	shell.setSize(520, 638);
	shell.open();
	while (!shell.isDisposed ())
		if (!display.readAndDispatch ()) display.sleep ();
	font.dispose ();
	display.dispose();
}
}
Comment 9 Felipe Heidrich 2004-10-08 13:13:07 EDT 
There is no bug in SWT here.
It is totally normal the comment to be aligned for some fonts and not aligned 
with others, (fonts have different metrics, some fonts have fixed pitch others 
are proporcional, etc). No bug here.
About the column number are different and the caret in visually at the y 
position: This is a JavaEditor problem, whenever a tab shows in the text they 
increment the column count by 4 (or 5 depending what you set), run the snippet 
I attached, the blue vertical lines are tab stops, note how the white space 
after "mailFrom; " is just passed a tab stop causing the next chunck of text 
(preceed for tab) to be drawn correctly at the next tab stop.

I'm moving back this pr to the text component.
Comment 10 Jeff Dean 2004-10-11 11:55:53 EDT 
There is definitely a bug here, although I can't say which part of Eclipse is
responsible.  The font is question is a fixed width font, so there is no issue
of differing character widths.  Other products handle this without a problem
(for example, the free PSPad editor, as well as Epsilon, my standard editor).

I really appreciate the effort you folks put into this software, but I can't
fathom how anyone can argue that this is the correct behavior.  If I enter
twenty characters, each of which is x pixels wide, and then I advance 4x pixels
(i.e., enter a tab that is four characters wide), and then enter another
character whose width is x, how can there be any disagreement about the pixel
location of where the characer after the tab starts?
