Description Dirk Grossmann 2004-06-16 03:25:22 EDT 
I run Eclipse with Java 1.4.2, but for my project I have to use 1.3.1.
Particularly immediately after starting Eclipse, the following error occurs:

The project was not built since its build path is incomplete. Cannot find the
class file for java.lang.Object.

Refreshing and cleaning the project sometimes helps when done several times
(project settings are OK and need not be changed). What helps is restarting
Eclipse and cleaning the output folder in the OS.

Comment 5 Derek Adams 2004-06-16 12:21:42 EDT 
I get the same problem in the M9 and later builds. My plugins use projects that 
have a Java nature, but do not use the package explorer. If I open my view and 
make a change that triggers code generation and a resulting Java build, I get 
the "The type java.lang.Object cannot be resolved" message. If I then open 
package explorer and click on "Java Build Path", the JRE is instantly added to 
the classpath and the problem goes away (no need to click on "OK"). My guess is 
that whatever triggers the loading of the JRE into the classpath is somehow 
dependent on package explorer initializing the Java model.

Comment 17 Philipe Mulet 2004-06-17 07:55:55 EDT 
I know what is going on. Actually the successful scenario is the lucky one, 
where the failing one is the normal mode.

Problem lives in the JRE_LIB variable initializer. It get told to initialize, 
but tries to check against previous value to see if it has anything to do.
Actually, we trigger initializations for a good reason from JDTCore 
standpoint, even in multi-threaded mode, but this optimization in JDTLaunching 
is burning us, since we try to be resilient and answer sometimes a value 
inherited from previous sessions for reentering calls (need variable value 
while initializing it). In the past, we were answer null if reentering, and 
overtime we were able to retrieve the persisted value from previous session.
So you could still compare and take any action you want, except forgetting to 
set it in the end. Indeed, we will simply consider it not to be initialized at 
all, and will not perform correctly passed that point; since after first 
attempt for initializing, we flush our cache from previous sessions, since we 
cannot carry these over after initial attempt.

I checked that changing the method below addresses the problem 

on JavaClasspathVariablesInitializer:

private void setJREVariable(IPath newPath, String var) throws CoreException {
  JavaCore.setClasspathVariable(var, newPath, getMonitor());
}
