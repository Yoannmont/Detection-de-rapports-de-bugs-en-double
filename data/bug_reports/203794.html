It appears as though this is an Apache configuration problem. Some of the PDT JAR files have ".php" in their name. This is being intercepted by Apache and forwarded to PHP for processing (when I look in the offending files, they are text files containing PHP errors)

There are four solutions to this problem:

1) Rename the PDT jars to use the form "*.pdt.*" instead of "*.php.*"
2) Convince the operators of the various mirrors to update the matching rule Regexp expression to be a little smarter (which is what we've done on the foundation servers)
3) Disable the offending mirrors.
4) Explicitly select a known-good mirror when updating (the PDT update site works well).

Obviously, #3 is very undesireable, #2 is a PITA for web master, and #4 is a PITA for users.

Any chance you can change the names of the plug-ins?

FWIW, the mirror that's causing me problems is "TDS Internet Services (http)"

Some workarounds:

a) Use a policy file to force updates from a different mirror: http://wiki.eclipse.org/Using_Policy_Files

b) uncheck the "select mirrors automatically" box when searching for updates, and manually pick better mirrors (like the eclipse foundation's main site or an FTP site

----

Of course my +1 vote goes to refactoring the project because this embarrassing bug could go away easily with some project-wide string replacement & file renaming. 

I'd suggest piping all the files through perl...

for f in $(find . -type f); do 
  echo " ### $f ### ";
  cat $f | perl -pe "s/php/pdt/g" > $f.2; # string replacement in files
  mv -f $f.2 $f;
done

for f in $(find . -type f -name "*php*"); do 
  g=${f//php/pdt};
  echo " *** $f -> $g *** "; # file renaming
  mv -f $f $g;
done

[1] http://tldp.org/LDP/abs/html/string-manipulation.html

I realize that it's not QUITE that simple as you'll want to ensure that your plugin.xml file(s) still reference the .php extension, not a .pdt extension. 

You could modify the above script to simply LIST the files which contain "php" in their contents or name, and then audit them before/after for changes. Of course with the Synchronise view in Eclipse you can make all the changes globally (outside Eclipse using the above scriptlets) then audit them one by one before committing them back to CVS.

If the guys in Mylar/Mylyn or Maya/Maynstall can do it, so can you. ;-)
