Description Andrew Cornwall 2002-08-27 14:24:01 EDT 
Right now, I can create file associations that match *.txt or readme.txt. 
However, I can't create an association that allows me to open readme.*.txt with 
one editor and *.txt with another.

If I try to create an association for readme.*.txt, the UI blocks me, saying 
"The file name cannot include the wild card character (*) in the current 
location"

This isn't such a big deal on Windows, but on Unix, many files have no 
extension, or a strange "extension". It would be useful to be able to match 
[Mm]akefile* and use one editor for all my makefiles.

Comment 18 Lorenzo Donati 2005-04-26 07:06:28 EDT 
Since I often use "double extensions" to discriminate among files with different
content, especially XML files (e.g. somefile.ant.xml for ant buildfiles,
somefile.ant.properties or somefile.antdata.xml for properties files to be read
by ant scripts, etc.), I find this a REALLY annoying bug (missing feature?!?).

At least the platform should allow the recognition of files with "multiple"
extensions, if not providing a full-fledged pattern matching engine.

This is not to be confused with the content-based associations (I agree with
Gunnar - comment #16). I think there should always be the option to set these
association manually in the UI, because sometimes the "content type" of a file
with a given extension is peculiar only from a "human perspective", e.g. say you
want to save text files with license information as somefile.license.txt and
"change-logs" as somemodule1.changelog.txt. Both such file types would
presumably contain unstructured human-readable text, with no hope of automatic
recognition.

Comment 20 Johann Petrak 2005-09-21 10:52:09 EDT 
Maybe this should be put in a different bug but I would strongly recommend to
also allow for a mechanism that makes the type decision based on reading the
first few lines of the file. Either by matching against stored patterns or by a
"mode lines" i.e. a (comment) line that contains the type information in a
specific way. (X)Emacs has been doing this for eons, and it would be good to
learn a bit of how file types are determined there.

There are many reasons why file type should be determined by content or mode line:
e.g. many programs that are used like commands do not have an extension and the
file pattern is not intended to say anything about the language used to
implement them either. Such commands could be written in perl, ruby, bash or any
other shell language. There should be a way to automatically determine the type
or at least manually assign a type to such files within a project. 

The type in term should then be used to determine which editor to use, how to
run/syntax check the file etc.

Comment 22 Martin Oberhuber 2005-11-25 09:19:48 EST 
When talking about "reading the first lines of a file", this is exactly what content types (and ContentTypeDescribers) do now. In fact, bug 89859 talks about improving content type matching strategies to allow for more complex file patterns.

I think one problem of "double extensions" is that they produce ambiguous matches - myfile.ant.xml could match *.ant.xml but also *.xml. Using a "longest matching suffix" rule would disambiguate here and also cover cases of file specs like *_pavast.xml (with an underscore).

A really flexible solution should not be limited to extensions but allow for arbitrary glob patterns like read*.txt or [^.]*.sh.

Technically - in order to keep backward compatibility and good performance - this could be solved by a two-stage matching where extensions are tried first, and patterns are used in the second stage to disambiguate.

Finally, it would be good if editor extensions and content types could be unified at some point.

Comment 29 Markus Keller 2013-11-28 07:16:07 EST 
Copying http://dev.eclipse.org/mhonarc/lists/platform-ui-dev/msg05522.html :

> 1) Redefine the "extension" concept to be everything after the FIRST dot

That's a no-go, since that would be a breaking change for existing files 
like "4.4 Plan.txt" and for everybody using e.g. 
IResource#getFileExtension().

> 2) When looking up mappings for some filename, sequentially relax the 
> extension until a mapping is found  (e.g. file.x.y.z will look for 
> mappings for *.x.y.z, *.y.z and *.z -- in that order -- and stop at the 
> first matching mapping as the most specific mapping should "win").

Sounds like the way to go. I.e. never try to extract a "suffix" from a 
given file name, since the correct suffix can be ambiguous. Only use a 
suffix _pattern_ to map from file name to content type.

> This semantic change would in fact mean the API is backwards compatible:
> currently there are only simple rules in the wild (*.somextension), so 
> changing the existing API implementation along these lines would mean 
> things will keep working (but people will be able to now add more 
> specific rules going forward).

It is not completely backwards compatible for clients that use e.g. 
IContentType#getFileSpecs(int) or IFileEditorMapping#getExtension() and 
assume the extension doesn't contain a dot, but I think that will cause 
less problems than introducing a new "suffix" concept to which every 
client has to adapt.

Actually, in my mind IResource#getFileExtension() would remain as-is. I was referring to IFileEditorMapping#getExtension().

The heuristic described below in (2) can be extended to cover cases like "4.4 Plan.txt" as follows:

When calling IEditorRegistry#getDefaultEditor(String fileName) (and friends):

step 1: check if fileName contains a wild char

step 1.1: (if yes) look for exact mapping using filename as key
step 1.2: if exact mapping exists, return it, otherwise fall through to step 2

-->(wild char present/no specific mapping)
step 2: follow approach described below for the suffix pattern
e.g. assuming no specific mapping was defined for "4.4 Plan.txt" we would now check for "*.4 Plan.txt" and "*.txt"...

Since currently nobody is allowed to have mappings where a wild char is present, we can currently only have in the wild a "4.4 Plan.txt" mapping or a "*.txt" mapping, both of which would work.

In newer releases one will be able to define a mapping for "*.4 Plan.txt" which would precede "*.txt" but would still lose to an existing mapping for "4.4 Plan.txt"
