Description Bob Foster 2005-05-31 14:04:21 EDT 
Checking out an existing plug-in project in RC1 yields a number of build errors
of the form: "Project x is missing required project y" where y includes
org.eclipse.jface, etc.

It seems clear that one is now required to explicitly specify dependencies that
were previously satisfied by another project's exports. E.g.,
org.eclipse.jface.text exports org.eclipse.jface, and that worked fine up until RC1.

Aside from the fact that this is a fairly major change to pop up in RC1, it
causes a platform-dependence problem. Instead of just requiring org.eclipse.swt,
one must now also require org.eclipse.swt.win32.win32.x86.

If you put such a dependency in the plugin.xml, you no longer have a
platform-independent plug-in. You would need to have a different plug-in version
for every platform.

Maybe I'm missing something, but if this is the case it is unacceptable. The
ability to build one plug-in that runs on all platforms must be preserved.
Comment 1 Bob Foster 2005-05-31 14:38:38 EDT 
Looking over some other projects, it is apparent that this is not a manifest
problem, but a compiler problem. Where in former releases the compiler was
accepting implicit resolution in cases like:

public class MyPage extends WizardPage {
...
}

now it is requiring that the dependency on org.eclipse.jface be explicitly
specified.

Comment 4 Wassim Melhem 2005-05-31 15:16:06 EDT 
that's the problem with hardcoded dependencies.   You lock yourself into a 
particular setup while the world of plugin dependencies will move around you 
and you have to manually maintain it.

That is why since before 3.0, all plugins now use the PDE classpath container 
to manage their plugin dependencies.

I suggest you do a PDE Tools > Update Classpath on any plug-in project 
whose .classpath does not contain the line:
<classpathentry kind="con" path="org.eclipse.pde.core.requiredPlugins"/>

and then sit back and let PDE take care of your classpath.
Comment 5 Bob Foster 2005-05-31 15:38:40 EDT 
No thanks for the suggestion. The world of plugin dependencies didn't (and
can't) move around without breaking API. What happened here is somebody changed
the rules for the compiler. It's annoying, but at least it's compatible.

As to sitting back and letting PDE do the work.

a) Until you remove a feature from the user interface, it should work when
anyone uses it, not just PDE.

b) I have a number of Java projects that are built
into jar files and incorporated with my plug-in at run time. So they appear in
the plugin.xml as required internal jars. However, it is very inconvenient to
develop if I have to build a jar file every time I make a change in one of these
projects. So I forego PDE's "help" in setting up the build dependencies from the
manifest and instead explicitly specify them in the Build Path.

c) If I turned these "library" projects into plug-in projects then PDE would do
something reasonable, at the cost of a dozen little plug-ins all exporting API I
don't intend to be external. And of course I would be required to use Update to
install, even locally, because no one could keep track of all those plug-ins
manually.

d) Or, Eclipse could add some reasonable support for "library" projects that
aren't distributed as separate plug-ins. Then I'd be happy to let it be managed
for me.
