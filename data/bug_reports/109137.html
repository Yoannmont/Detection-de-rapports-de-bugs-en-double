Description Ed Merks 2005-09-09 08:31:46 EDT 
Consider the long explanation below that I gave on the platform newsgroup for
how we need to use Eclipse for doing our development.  Anybody developing a
runtime against which they must compile generated plugins will have this same
problem, so I'm hoping between the PDE and the JDT that something can be done to
make our development process work more smoothly.  I understand that the PDE
considers this a JDT limitation (can't compile against external folders), but
I've opened the bugzilla against PDE because that's where the function is really
badly needed (and because some might argue: figure out a way to use links to
make the folders not be external; I understand this is difficult because the
workspace is locked while PDE is doing its processing).

When developing EMF, we work around the current limitation of not being able to
compile runtime workspace plugins against the launching workspace's plugins in
the following way.  We check out our CVS code so that the projects are
physically stored in the plugins directory of the Eclipse installation.  We
allow the checkout to complete and the build to complete and then we exit
Eclipse and run a script like this:
for i in  \
plugins/org.eclipse.emf.ant/emf.ant.jar \
plugins/org.eclipse.emf.codegen.ecore.ui/codegen.ecore.ui.jar \
...  lots more plugins ...
plugins/org.eclipse.xsd.example/xsd.example.jar
do
(
cd $(dirname $i)/bin
echo zip -r ../$(basename $i) *
zip -r ../$(basename $i) *
)
done
This builds the jars for all the plugins from the .class files produced by
Eclipse's build.  We can now restart Eclipse, with -clean, and use all our code
in the main Eclipse.  This is very important for EMF because we bootstrap our
own code, e.g.,  to compile our JET templates we need JET installed and to
generate our models, we need EMF itself to be installed.   Because we have
everything set up this way, when we launch a runtime workspace, that workspace
will use the .class files instead of the .jars we produced, so we can test our
latest changes nicely without messing up the EMF in  main Eclipse.  We can also
generate code in the runtime workspace that compiles against our developed
plugins, but note that this compilation is against the .jar files not the .class
files, because of the JDT limitation of not compiling against external folders,
which is typically not such a big issue because, when it actually runs, it will
use the .class files, so it's only an issue if the API is changing...

When we needed/wanted to provide jarred plugins, we thought there'd be an even
better solution because we would no longer need to build the jars, but instead
the problem basically came back to bite us yet again, because, as you can see
from the description above, having the jars is very important for the compiler
to work.  So we've had to modify the manifest to specify a jarred plugin during
our build so that we don't break our only good development-time mechanism.

Comment 13 John R. Fallows 2006-11-29 21:42:47 EST 
We encounter the same issue regularly during our daily development.

When developing plug-ins for Eclipse 3.2, it is common to launch Eclipse in a child workspace, including some of the plug-ins under development from the parent workspace in the launching target platform.

However, if the plug-ins being developed are used to generate code in the child workspace, then that code cannot be compiled unless all dependencies are part of the child workspace target platform definition.

Although the plug-ins from the parent development workspace are in the launching target platform, they are not part of the child workspace target platform used to resolve dependencies for generated code.  See Tools|Preferences... Plug-in Development|Target Platform for the child workspace.

Therefore, the generated code fails to compile unless the plug-in dependencies from the parent workspace have been previously exported.

Each time a change is made to these plug-ins, they must be re-exported, and the child workspace must be restarted to pick up the changes.  This is extremely tedious, especially if you have a grand-child application running that uses the generated code along with the exported dependencies when you decide to make a change.

Would it be possible to include some of the parent workspace plug-ins in the target platform used by the child workspace to resolve compilation dependencies?  This would allow the re-export and restart overhead to be eliminated.

When could we expect to see some progress on this issue?
Comment 14 Wassim Melhem 2006-11-29 22:44:05 EST 
As I explain in comment 4, the problem is not specific to self-hosting and runtime workbenches.  It is a general problem that lies in the inability to reference a filesystem directory in a classpath entry.

We (Platform/Resources, JDT/Core, PDE) held a meeting to discuss this issue.
An ideal solution is for this limitation to be addressed at the platform level.  As a last resort, we could investigate workarounds at higher levels.

Comment 17 Dmitry Stadnik 2006-12-01 08:16:30 EST 
To John R. Fallows:

One possible solution is to import required projects from parent workspace into child workspace (do not copy, just reference). It's also important to have the same compiler options in both workspaces so two instances don't recompile shared projects.

That's how my gmf workspace is organized:
1. Eclipse platform contains platform, gef, emf and emft plugins unpacked from builds.
2. First (development) workspace contains gmf runtime and tooling plugins taken from cvs; there I modify tooling templates.
3. Second (samples) workspace contains gmf sample plugins taken from cvs plus references to gmf runtime and tooling plugins from the first workspace. Thus gmf sample projects generated by tooling plugins are normally compiled against referenced projects.
4. Third (test) workspace contains projects with test data for sample projects.
