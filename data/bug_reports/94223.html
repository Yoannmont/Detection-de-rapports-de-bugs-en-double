Description Josh Micich 2005-05-09 23:04:11 EDT 
When suspended in the debugger at a breakpoint, I want to be able to modify the 
value of fields of any object in the Variables view, even if those fields were 
declared as 'final'.

I know that 'final' values should not be modified.  But I think that when 
debugging the user should be able to use any trick or kludge, without 
restriction.  Constraints of the language should only be a guide while 
debugging.
Even Sun seems to agree with this. Note - according to the "Java Debug 
Interface" Spec the method:
com.sun.jdi.ObjectReference.setValue()
says nothing more than "If static, the field must not be final." Which implies 
that non-static fields can be modified by the debugger, and indeed this is the 
case.

I don't recall hitting this issue in Eclipse 2.1, so I guess this is a recent 
change.  However I could not find any previous bug/enhancement request related 
to this.

-- -- -- --
I had a quick look at the jdt code to see if I could remedy the issue for 
myself.

In the class:
org.eclipse.jdt.internal.debug.core.model.JDIModificationVariable
method:
public boolean supportsValueModification()

there is a check for isFinal():
	if (!isFinal()) { ...

If I remove this line, I get the behaviour desired.  There's probably a more 
correct way to solve this problem.
Comment 1 Darin Wright 2005-05-11 16:25:36 EDT 
Turns out, that although JDI allows us to change finals, it has no effect on 
the value in the target app. It's interesting that you note that you get the 
desired behavior - which VM are you using?

Comment 2 Josh Micich 2005-05-11 18:55:11 EDT 
I am using a Sun 1.4.2_06 VM, but I don't think that is important.  I'm sure 
you'll be able to observe most of these effects on any VM.

The code sample in bug 48935 behaves predictably, but doesn't illustrate my 
point due to compiler optimisations (constant resolution).  The compiler can 
tell that all these final values are constant, and optimises the values 
directly into the bytecode.  The realised data values in the JVM really do 
change.  You can observe this by highlighting the 4 "System.out..." lines, and 
press Ctrl+U (Execute).  The debugger 'Execute' command seems to compile the 
highlighted code independently (and without constant value optimisation).

The following code circumvents the compiler optimisation, and allows more easy 
observation of the successful value change:

public class Test001 {
  final int finalInstanceVariable = three();
  final static int finalStaticVariable = three();
  
  private static int three() {
    // if your compiler is too smart for this (inlines the code),
    // try code like in java.lang.System.nullInputStream()
    return 3;
  }
  public static void main(String[] args) {
    new Test001().foo();
  }
	  
  public void foo() {
    final int finalLocalVariable = three();
    int localVariable= 3;
    System.out.println("test");   // <--- breakpoint here 
    System.out.println(finalInstanceVariable);
    System.out.println(finalStaticVariable);
    System.out.println(finalLocalVariable);
    System.out.println(localVariable);
  }
}

The method "three()" might seem like obfuscation, but there are many other ways 
to achieve the same effect, and some are actually good coding practice.  We use 
a lot of immutable classes, whose fields are set only once (in the 
constructor). 

--  --  --  --
I think the real issue in bug 48935 is the incorrect assumption (on the part of 
a debugger user) that modifying a (compile-time) constant variable at run-time 
will have the same effect as modifying it before compilation.  Some intuitive 
expectations are not realised, but it is incorrect to say that there is no 
effect at all.

Let "modifiable type" to refer to any value type which is modifiable in the 
Variables & Expression views of the debugger.  These are primitives and 
java.lang.String.  By coincidence, "modifiable type" is also the set of types 
which can be optimised as compile-time constants.

Note, that if a value is not a compile-time constant, it will be perfectly 
modifiable in the debugger, regardless of whether it is final or not.
The current resolution (of 48935) is at least incorrect because not all final 
values (of modifiable type)  are compile-time constants.  Indeed, not all 
static final fields (of modifiable type) are compile-time constants. 

I would prefer the debugger to allow me to modify ANY value at any time (and 
accept the consequences of doing so).  If the issues of 48935 are considered 
important and obscure enough that the debugger user needs guidance, perhaps a 
warning message could be included.  Of course, care must be taken to only show 
the warning when modifying a final value that really is compiler optimised.
