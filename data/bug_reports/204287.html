Description Randall Theobald 2007-09-21 12:09:52 EDT 
Created attachment 78990 [details]
Patch

Build ID: M20070212-1330

Steps To Reproduce:
See discussion of problem below.

More information:
This discussion assumes that the patch from bug 202820 has been applied.

I am an adopter whose product may use large workspaces. During the first build of a PI that was originally built in Eclipse 3.0 (hence it needs to have migration builders run on it), I notice that a lot of time is spent in the method:

    J2EEComponentClasspathUpdater.queueUpdateModule

This is mainly because the queueUpdateModule method calls

   J2EEProjectUtilities.getReferencingEARProjects

in order to queue the associated EAR projects. This method needs the DependencyGraph. The problem with doing this in the queueUpdateModule method is that during the build, many changes might result in the DependencyGraph needing to be rebuilt, resulting in the DependencyGraph getting needlessly rebuilt during the build during calls to queueUpdateModule. We can avoid this by delaying the queuing of the associated EAR projects until the ModuleUpdateJob actually runs.

I have created a patch for this which reduces the time spent (during the first build) in DependencyGraphManager.getDependencyGraph by 58%. Time now spent in queueUpdateModule approaches zero. This translates to a savings of about 73 sec for my scenario, or about 10% savings. These results are after applying the patch from bug 202820.
