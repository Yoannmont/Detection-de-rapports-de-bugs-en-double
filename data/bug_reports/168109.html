Description Jon Barrilleaux 2006-12-14 14:41:56 EST 
3.3 M3 testing

- have a project with the following indentation settings:
  - tab policy: mixed
  - tab only for leading indents: yes
  - indent: 2
  - tab: 4

- for the following code:

class Bar {
    void m() {
        if (aaa && bbb && ccc && ddd && eee) {
            return;
        }
    }
}

< expected result (note that line wraps before "eee) {":

class Bar {
--->void m() {
--->--->if (aaa && bbb && ccc && ddd &&
--->--->..eee) {
--->--->--->return;
--->--->}
--->}
}

> actual result:

class Bar {
..void m() {
....if (aaa && bbb && ccc && ddd &&
......eee) {
......return;
....}
..}
}

Comment 4 Jon Barrilleaux 2006-12-15 11:41:47 EST 
Why not?  I did;-)

What is was expecting (hoping for?) is for the initial indent to use only tabs, but the indent after a line wrap to use the initial indent plus a little, in this case something less than a tab, specifically a space or two.

This is exactly the situation I reported in a feature enhancement bug a while back, and I though you guys had addressed it in 3.3 by differentiating initial versus subsequent indent after a line wrap. Perhaps the UI needs a bit of clarification or tweaking, preferably tweaking.

Comment 7 Olivier Thomann 2006-12-15 14:52:22 EST 
What I get is:
class Bar {
..void m() {
--->if (aaa && bbb && ccc && ddd
--->....&& eee) {
--->..return;
--->}
..}
}

And this works as expected.
If you set your identation policy to mixed and tabs used only for leading indentation, then you get what it is above.
If a tab (which is two identations) can be used, it is used only for leading indentation. This is why you get spaces before the "&& eee".
Closing as INVALID.

Add regression test org.eclipse.jdt.core.tests.formatter.FormatterRegressionTests#test641.
Comment 8 Jon Barrilleaux 2006-12-15 15:14:03 EST 
Perhaps I'm being confused by the term "leading".  To me, the leading indentation is the the one initially used to indent a given line, and any subsequent form of indentation is that used for wrapped lines.

It seems that what "most" users would want is for the initial indentation to be so many tabs, and for wrapped lines to be optionally "mixed", with an additional indentation amount that may be less than a tab, such as a space or two, as I showed above.

Might I suggest that in the GUI you provide specification for a "leading" indentation, which could be tabs and/or spaces, and a separate one for "wrapped" indentation, which would be added to the indentation for wrapped lines, and which could be tabs and/or spaces.

(As to why my initial test came out with spaces only, beats me.  I tried to reproduce it and it is as you report.)

Note: I changed this bug to "enhancement", which it seems to have become, and reopened it.
Comment 9 Will Hains 2007-08-08 04:28:18 EDT 
I've noticed that with indentation policy set to "Tabs only", and with tab only for leading indents, tab size set to 4, Eclipse 3.3 now uses spaces for wrapped lines. (Eclipse 3.2.x used tabs only)

This is causing many changes in our source control system that shouldn't be there. Speaking for myself, this is a very high-priority bug.
