Description Samantha Chan 2007-08-02 11:47:03 EDT 
Refer to this bug for background:
https://bugs.eclipse.org/bugs/show_bug.cgi?id=125868

Our model presentation wrappers JDT.  When #getImage is called, we use own own copy of DelegatingModelPresentation to get an image from JDT.

The problem is that, our model presentation could be called on a background thread when #getImage is called.  And if JDT's image registry is not initialized, we create a UI job to get it initialized.  Our model presentation wait for this UI job to finish before returning the getImage call.

In AsynchronousLabelAdapter, we decide if a UI job is needed to compute label depending on if the client's model presentation is initialized.  Before the model presentation is not initialized, we compute labels on the UI thread.  We can have many UI jobs scheduled before client's model presentation is initialized.  

When the deadlock happens, two "Retrieving Labels" jobs are running.  One on the UI thread, and one on the background thread.  When the jobs are running, AsynchronousDebugLabelAdapter needs to acquire a lock on platform's delegating model presentation.

At this time, our model presentation is called on the background, the background thread owns the lock to the delegating model presentation.  The our  model presentation schedules a UI job and waits for it to finish.

On the UI thread, AsynchronousDebugLabelAdapter's computeLabel is being called which also needs the lock to the delegating model presentation.  Because the background thread owns the lock, the main thread waits.

Since the ui job that gets scheduled never gets a chance to be executed. we get into a deadlock.  The background thread holds the lock forever.

Comment 3 Darin Wright 2007-08-08 12:34:29 EDT 
The deadlock problem is reduced in 3.3 (via deprecation). The AsynchronousLabelAdapter and subclasses are only used with the "old" (first version) of the asynchronous viewers. The new "ElementLabelProvdier" and "DebugElementLabelProvider" used in 3.3 tree viewers do not lock the DebugElementHelper's model presentation. The existing classes with the deadlock problem are intended to be deleted.

However, the memory (asynchronous table) viewer uses the old code in 3.3, and could cause a problem.
Comment 4 Darin Wright 2007-08-14 14:40:19 EDT 
Marking as won't fix, as the problem is in internal classes that are intended to be deleted, and the mem view is to migrate off the old code.
