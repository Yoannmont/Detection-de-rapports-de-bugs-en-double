Description Victor Toni 2005-02-28 10:17:16 EST 
I have an additional "Class Folder" to store my project specific
properties/resources which shouldn't be exported.
However this "Class folder" ignores the export settings. It is exported
regardless if checked or unchecked.

Example:

Project1:
- src
- myclasses // export _not_ checked
  + conf.properties 

Now in Project2 which depends on Project1 "conf.properties" is still visible
as long "conf.properties" resides in "myclasses".

Comment 11 Darin Wright 2005-06-08 10:25:27 EDT 
At runtime, we put everything on the classpath (exported or not, since 1.0). 
The reasoning is that at development time, you may/may not need access 
(visibility) to APIs, but at runtime, everything is required. For example, if 
component A requires component B, and B is not exported, it is still required 
by A, and anyone who uses A at runtime.

Do not intend to change.
Comment 12 Victor Toni 2005-06-08 13:52:19 EDT 
I disagree about the need of everything on the classpath.

The user ususally sees that a library is required because the builder enforces
it and makes it hard for the user to bypass it.
To test the behaviour (you mentioned) for jars I had to:
1. Create a dependency in "myProject" on a JAR on the classpath of
"importedProject".
2. Deactivate the builder for "myProject".
3. Uncheck the export of JAR in "importedProject".
4. Start the programm in "myProject" to see that it still works.

This proves you right but here is an example which shows that complete inclusion
is not the desired behaviour:

E.g. I might write a library for some XML use case. Say my library depends in
parts on/may use multiple XML APIs because I would like to support more than
one. So this library includes multiple jars (while working on it).

Using this library inside another application (from within eclipse) may bring up
unexpected results because a library would register itself as the default
provider without being transparent to the user.

Therefore the export functionality should be enforced also at runtime.
It is not clearly stated that this functionality is only to change visibility
during _build_ time but will be ignored at runtime.

There are other scenarios where one certainly knows that a certain entry
shouldn't be exported e.g. the folder containing log4j.xml which should be used
only per project.

Please reconsider, if too late for 3.1 then for the next version.
To have the same behaviour as it is now all entries could be checked and the
user could disable the ones which shouldn't be exported.
