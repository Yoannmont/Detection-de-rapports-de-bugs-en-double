Description Larry Isaacs 2006-01-26 13:15:55 EST 
With the removal of the ".deployables" project directory, the Tomcat support now publishes the web module to Tomcat's "webapps" directory.  At the moment, publish destination is built using the module's name, which is equivalent to the project's name.  Since "autoDeploy" is typically turned on.  A context will be auto-served using the webapp's subdirectory name, i.e. the project's name.  If the context-root is set differently from the project name, then the WTP created context will serve the webapp with the specified context-root and "autoDeploy" will serve another based on the subdirectory/project name.

Unless the webapp has a long running initialization and/or uses lots of memory by default, there is probably little impact from this other than maybe adding something in the release notes.  I'm not sure if anything fix should be attempted for 1.0.1, though I will try to attach a patch within a day to switch to using context-root in the publish destination. I can't be certain yet that such a change won't break something.

We should at least do something for 1.5 and examine the implications of the user modifying the context-path.  For example, if webapps are published to the "webapps" directory, then changing the context-root in a project that has already been published to a Tomcat server should eventually rename or recreate that directory.
Comment 1 Thomas Jachmann 2006-05-10 10:07:13 EDT 
I also ran into this problem with 1.5M6 and debugged my app for over a day until I realized that I was dealing with two instances of the same app. One was working, the other wasn't, since it relies on resources configured via JNDI within the context of the app in the server's server.xml. For such applications, this bug certainly breaks the possibility of specifying a context root, despite the waste of resources.

Comment 3 Larry Isaacs 2006-05-10 14:23:58 EDT 
Sorry for not providing more feedback regarding this problem.  So far I have been unsuccessful at finding a fix simple and robust enough to commit to the 1.5, and thus 1.0.x, version of WTP.  Getting it served using the "context-root" is simple enough, but managing it's deployment and undeployment tends to break when the webapp's deployed directory doesn't match the "deploy-name" in the .component file.

At this point I'm planning to make improvements to the Tomcat configuration for WTP 2.0 such that the deployment directory setting is user configurable.  The default will be something other than the "webapps" directory, so auto-deployment isn't the complicating factor it is now.  In this case, using the context-root won't be the hurdle it is now, and neither will supporting the default webapp (i.e. path="").  If the user specifies the deployment directory to be Tomcat's "webapps" directory, then the ability to deploy a project whose "deployment-name" and "context-root" differ may be lost.

I still intend to try to clean up my best attempt at a fix and provide a patch that works with known breakages.  It may be that for some, the need for this to work is more important than the breakages.

Comment 9 Larry Isaacs 2007-02-10 20:35:50 EST 
This is being address by work for Bug 145834. The location where modules are deployed now defaults to "wtpwebapps" instead of "webapps" for new servers. This setting can be modified in the server editor.  By publishing modules to a directory other than "webapps", autoDeploy being enabled can cause the second copy to be served.

If this is an issue for an existing server, changing the server's deploy path in the server editor will be necessary.  There is a link in the server editor that can be clicked to set the new default deploy path.  Note that the server must be published (i.e. synchronized) with no modules present before this setting can be changed.

Since this is the extent to which this issue will be addressed, I'm marking it FIXED.
Comment 10 Larry Isaacs 2007-03-30 11:09:16 EDT
