Description Brian Bauman 2006-07-11 15:57:27 EDT 
The cursor seems to disappear during tabbing in Linux.  This happens in the PDE Feature editor (Dependencies page > Dependencies Properties section) and the Product Editor on the main page.

Comment 8 Bogdan Gheorghe 2006-12-07 17:25:19 EST 
The difference in the behaviour seen on Windows and Linux is caused by the fact that org.eclipse.pde.internal.ui.parts.ComboPart uses a Combo box on Windows and a CCombo box on Linux. 

If you make use of a CCombo and install a key listener (which ComboPart does indirectly through the FormToolkit), you have to provide your own traversal handler. Combo will handle this for you automatically.

Comment 13 Brian Bauman 2007-05-17 17:36:23 EDT 
Created attachment 67733 [details]
patch

I am now much more familiar with CCombo than I ever hoped to be.

The patch is simple.  We attach a traversal listener to the control if we create a CCombo.  Upon receiving traversal notices, we check to see if it is a tab traversal.  If it is, we then check event.doit.  The first traversal will have event.doit = to false (bug 61733).  If doit == false, we call traverse again.  This time the traversal event's doit value == true and we will do nothing.  

If we take out the event.doit check, we will have a stack overflow since every time we get a traversal event, we call traverse again.  

The resulting behavior is not 100% pretty.  The second tab will actually highlight the Text in the CCombo instead of the next Button.  Therefore, multiple tabs are required to move all the way through a CCombo box.  The bug for this problem is 78885.

Also, bug 61733 is the original bug that talks about CCombo not tabbing correctly with a KeyListener.  I submitted a small patch for that bug to the SWT team.  If the patch is applied, our patch is no longer needed.  I figured we can always remove it when the time comes.

Comment 18 Bogdan Gheorghe 2007-05-18 15:39:12 EDT 
As we are late in the 3.3 release cycle, we do not feel comfortable with making
any changes to CCombo. The patch attached to this bug report seems OK but a
better solution is to not add a the keylistener to the CCombo in the first
place or add a FocusListener to the CCombo that sets focus.

Here is a snippet that illustrates this. If you comment out the KeyListener and
FocusListener - everything will work. If you add back the KeyListener tabbing
will not work. If you add back the focus listener and have it put the focus
back on the CCombo everything works again.

======================

package bog.javasnippets;

import org.eclipse.swt.*;
import org.eclipse.swt.custom.*;
import org.eclipse.swt.events.*;
import org.eclipse.swt.layout.*;
import org.eclipse.swt.widgets.*;

public class Test {

        public static void main(String[] args) {

                Display display = new Display();
                final Shell shell = new Shell(display);
                shell.setLayout(new RowLayout());

                Button b = new Button(shell, SWT.PUSH);
                b.setText("Hello!");

                final CCombo ccombo = new CCombo(shell, SWT.BORDER);
                ccombo.setItems(new String[] { "hello", "goodbye" });
                ccombo.addKeyListener(new KeyAdapter() {
                });

                ccombo.addFocusListener(new FocusAdapter() {

                        public void focusGained(FocusEvent e) {
                                ccombo.setFocus();
                        }
                });

                Text text = new Text(shell, SWT.BORDER);

                shell.pack();
                shell.open();

                while (!shell.isDisposed()) {
                        if (!display.readAndDispatch())
                                display.sleep();
                }
                display.dispose();
        }

}
Comment 19 Brian Bauman 2007-05-18 16:27:45 EDT 
Created attachment 67868 [details]
updated patch

Bogdan, I don't blame you for not wanting to make changes to CCombo (esp. with patches submitted from someone who is not an SWT expert) :)

After reading your response, I was going to reply that the keyListener was necessary.  But to my surprise, the way the Form's listener works, it will actually ignore all keys when it comes from a control which is a Combo.  Therefore, it doesn't do us any good to attach the listener when we use the Combo.  And in order to stay consistent, it makes sense not to attach the listener for CCombo in the same scenario.

Since we don't actually need the key listener, by removing it the tabbing now works fantastic and the focus is set correctly.

Thanks for the suggestion!
