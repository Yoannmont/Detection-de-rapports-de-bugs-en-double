Description Philippe Ombredanne 2005-06-10 11:59:29 EDT 
Ever since the first build on 3.1, the version of all the platform plugins is
3.1.0. And the problem was the same on 3.0.

This is an pretty bad practice! 

Probably one of the worst versioning practice I have ever seen in open source
project, and definitely the worst for a project of this importance and reach: 
- it makes it super-hard for plug-in developers to track milestones and I builds
in an orderly fashion, 
- it guarantees tons of problems and frustration with plug-in incompatibilities
for end-users

Every build IS a new version, and SHOULD have a different version number.

As a result, many plug-in developers will be late to deliver plug-ins on 3.1,
since tracking your builds is just such a pain when nothing is versioned.
It is impossible to track finely the dependency on a certain milestone release
of the platform, unless you build, and re-version all the platform yourself.

Another results is end-user frustration since they cannot adopt a new version of
 the platform and be sure that their existing plugin will either work or not.
For instance there were quite a few changes to internals between 31M6 and 31M7
resulting in dramatic performance increase, but resulting also in many of the
interesting plugin available to stop working, crashing Eclipse and the like.
That they break is not a problem per se, the problem is that there is no way to
express a dependedncy on a certain build, since a build IS NOT identified as a
version. 
Very bad experience for the user, when the plug-in could have stopped working
naturally without a crash if it would have been able to establish a dependency
on a certain build version. 

So please!

After 3.1 is released, could you do someting about it, and implement some of the
good and simple recommendations published by platform folks themselves:

http://dev.eclipse.org/viewcvs/index.cgi/%7Echeckout%7E/platform-core-home/documents/plugin-versioning.html
http://dev.eclipse.org/viewcvs/index.cgi/%7Echeckout%7E/platform-core-home/documents/update.htm

As noted here and other places, there are several problems in this area.  
1. we do not increment our numbers in releases to accurately reflect changes.  
Historically we have just blindly incremented all of them.   That means that 
people cannot express reasonable version number match rules/ranges.  For 
example, the transition from 2.x to 3.x was largely without breaking API yet 
the first digit was incremented.

2. during a development we do not have finer grain differentiation (the 
qualifier is not used).   This limits our ability to do incremental builds and 
incremental updates.

3. we do not spec version dependencies in our own plugins.  This is just goofy 
as it implies that Resources, for example, runs on any version of Runtime.  

4. we appear to have set the precedent for other teams.  Unfortunately people 
have been doing what we do.

So, what is to happen?  Well, I am hopeful on 1 and 3.  4 is an education 
thing that may well take time.  2 is harder to justify in the Eclipse space 
because, believe it or not, we are relatively small by comparison.  Our builds 
are not that long and the update technology would have to get alot better if 
we were to try and us it for delivering the builds that we deliver on a 
regular basis.  None the less, it is worth thinking about.

Note that 2 does have some outstanding issue in the area of qualifier 
progression as we move from release to release and essentially fork a 
maintenance branch at each release.  This really needs to be looked at in many 
different ways and worked through in extreme detail to avoid surprises a year 
from now.

I'm not quite sure what you are proposing should happen in 3.1.

Comment 15 Bob Foster 2005-06-23 22:27:53 EDT 
But it is not a duplicate, except in the sense that if you get a solution that
works for Eclipse it _might_ help other plugins/features.

Eclipse should support alpha/beta/milestone releases - not just for itself, but
for everyone. In fact, if Eclipse chose not to use this feature, there would be
many others who would.

Another way to put this is, update should work for non-final releases.

Currently, a plugin/feature version number must be an integer three-tuple (major
version, minor version, incremental version). This is inadequate for real
software projects, as evidenced by the fact that when Eclipse itself goes into
its milestone cycles, it stops using version number for anything but the target
release and instead identifies versions by build numbers. This is the reason
that update doesn't work for milestones (which is something of a FAQ).

The following sketches the simplest solution that would work. The current
hierarchical scheme needs only a slight modification to handle
beta/milestone releases. For example, suppose you added an optional fourth
integer to the series, indicating a "beta" version. Then:

3.1.0.10 > 3.1.0.9

But:

3.1.0 > 3.1.0.10

This is not rocket science and would add a very useful capability to Eclipse,
one that better models what actually happens in the real world.

Frankly, I can live with the fact that Eclipse is too lazy, um, I mean, too
busy, to version the individual plugins in milestones. But I'm not happy with
the fact that you push Eclipse update at us at every opportunity but we are
forced to use a completely different release mechanism for "beta" releases.

Comment 32 John Arthorne 2005-08-08 11:38:42 EDT 
To further clarify, it's useful to thing of these two problems separately:

1) The major/minor/service number. This is used to express semantic differences
between two plugins.  Plugins express their dependencies in terms of these
numbers.  So a plugin that relies on org.eclipse.core.runtime would state that
they require, for example, "3.0 or greater", or "precisely version 3.1.5". 
Resolution of plugins during install/update/startup would use these numbers to
determine if a plugin's dependencies are satisified.  There is no magic way to
have these numbers incremented automatically - this is a manual step that
committers must perform when changes are made to their plugin. The version would
not need to be changed very often (at most three times during a release cycle:
increment service after the first bug fix, increment minor after the first
significant but compatible change, increment major if breaking or non-compatible
changes are made).

I think that the use of version prereq's depends largely on the current state of
the feature that you develop. There are two different scenarios that I can think of.

1. The development scenario where you often want your dependencies to be "soft"
and not break as soon as you introduce a new release of something. When that's
the case, you would probably like to add a match="<something other than
perfect>" to your import statements.

2. You release something to the public. A release must not have soft matches. A
release is immutable and specification (the top level feature.xml) is a
specification of everything that the release contains.

In my opinion, you should always have a version= in your imports. Possibly with
the exception of the initial stages of development, i.e. before the first release.

A typical development/test/release cycle would look something like this:
During development, when you really want the ground you stand on to move (you
want early access to the latest of much of the stuff that you are dependent on),
you use match="greaterOrEqual" or similar. Once you release, you must pinpoint
the exact version of everything that is included in the release and everything
that the release is dependent on in order go give your QA department a fair
chance to do their job. Also, once the release is out of the door, you want your
bug-reports to reflect the exact configuration that was used.

The release is the start of a new branch (a bug fix branch for that release).
That branch will be very reluctant to use anything but hard version dependencies
since stability is a big issue. A change of dependency must be very deliberate.
Your development branch though, (head perhaps) will continue to use soft
dependencies.

Comment 50 John Arthorne 2005-10-13 12:00:56 EDT 
We realized that encoding the branch information in the qualifer segment does
not work.  Here is the example where it fails:

 1) Plugin ships in Eclipse 3.1 with version "3.1.0"
 2) Plugin ships in maintenance release with version "3.1.1.0301_20050930"
 3) Plugin ships in Eclipse 3.2 release with version "3.1.1.0302_20060630"
 4) Another maintenance release based on Eclipse 3.1 ships, and the plugin has
version "3.1.2.0301_20060930".  

Now this fourth version will appear to be "newer" than the version from 3) (it
will sort higher on a list of versions).  However, upgrading from 3) to 4) is
not possible because 4) is actually based on an older code base.

The solution we found for this problem is to encode the branch information in
the service segment instead, to ensure that newer branches always sort ahead of
branches based on older releases.  When a plugin has bug fixes in the
maintenance release, it increments to "3.1.1" as expected.  When the same plugin
has bug fixes in the HEAD stream, it increments to "3.1.100".  This ensures that
the HEAD stream version always appears ahead of any maintenance stream version.
This is actually the scheme used by several teams such as JDT and SWT for their
CVS tags (increment to the nearest hundred after a release).

Now the qualifier segment can simply be the CVS tag for the plugin, or an
auto-generated build id if no CVS tag is available (such as for nightly builds).
 So, to repeat the above example:

 1) Plugin ships in Eclipse 3.1 with version "3.1.0"
 2) Plugin ships in maintenance release with version "3.1.1.v20050930"
 3) Plugin ships in Eclipse 3.2 release with version "3.1.100.v20060630"
 4) Another maintenance release based on Eclipse 3.1 ships, and the plugin has
version "3.1.2.v20060930".

This solution seems simpler, less confusing, and most importantly we have the
technology to do this today.  The remaining problems that need working out
involve the relationship between plugin version numbers and feature version
numbers, and the relationship of version numbers between dependent features (if
a plugin in the JDT feature requires a major version increment due to a breaking
change, how does this affect the version number of the JDT feature, and the SDK
feature that imports JDT, etc).

There is a valid "optimistic" model where you don't specify version
dependencies, and you hope that it just works.  I believe that approach is valid
for a number of plugin writers in the community, and we certainly don't intend
to impose this proposal on those for whom this approach is working today. If you
are in that group, you can specify your version dependencies as loosely (or not
at all) as you think is appropriate for you and your plugin's users.  However,
there are others (we'll call them "pessimistic") who want to more rigidly lock
down their runtime environment. They don't want their users to be subject to
linkage errors, but instead want to prevent potentially invalid configurations
from happening at install time, even if that means preventing the user from
running a configuration that might turn out to work ok.  The plugins in the
platform must increment their version numbers in a semantically meaningful way
to allow the pessimistic model to work.  The tension of course, is how do we
enable the pessimistic case without undue stress on those happy with the
optimistic model.

> So what happens if your depencies (exported or not) change in a way that is
> more severe than your own component? Do you bump your version number 
> accordingly?

I will side-step the question of whether dependencies should be exported if they
appear in your API, since as Randy points out many platform plugins are guilty
of this.  If you do not export the dependency, then you need to evaluate whether
the breaking change in the required plugin will affect your API. If the breakage
is in a class or method that does not appear in your API, then your API is not
broken and you should not increment your major version number. I.e., there is no
blanket answer because some API changes have a ripple effect, and some do not.

will use it etc.  Consumers (think they) know what they want and how flexible 
they are planning on being about how it is delivered.

Mapping this on to versioning we see a contract between producer and 
consumer.  The producer is saying that they will accurately reflect the degree 
of change in the version number (e.g., as outlined in the proposal).  
Consumers can take that information and spec the base level they need and how 
much risk they are willing to tolerate.  

Note that I have deliberatly used the word "component" here.  It doesn't 
matter if it is a package, plugin or feature -- that dictates granularity.  
Features offer plugin or feature dependency granularity.  If you want to be 
fine-grained, spec plugin dependencies.  Similarly, plugins offer plugin or 
package level dependency granularity.

So to get this right, our producers and consumers have to start acting like 
producers and consumers.  To enable a consistent and coherent model version 
number incrementing has to be done in a consistent and coherent way at all 
levels.  This proposal suggests how we can achieve that goal.

Is that change compatible ?

Comment 117 John Arthorne 2005-12-05 09:51:58 EST 
The proposal certainly doesn't recommend that plugins specify the fourth segment in their prerequisite lists.  This is not generally necessary because development builds are not released "into the wild" for mass consumption.  The relatively small group of developers who consume non-release development builds are by definition using a configuration that has not been extensively tested.  People using plugins from development builds accept the risk of integration failure, so there is no need for you to lock down your prerequisites to a level that does not allow them to run your plugin on these builds.

There are two other reasons for not removing the qualifier segment in the final build.  First, a version with no fourth segment will sort *lower* than an equivalent version with no fourth segment.  If the final development build is "1.0.0.v20050628" and the final build is "1.0.0", then the development build will be chosen over the release by the OSGi resolver mechanism if plugins from both builds are present.

Second, removing the final segment prior to the release adds stress to the release cycle.  A build must be extensively tested before it can be declared a good release.  Performing an extra build after all this testing in order to strip out the version numbers risks introducing other failures and requires further testing.  If a second "final" build is then required, then there is no way to distinguish plugins from these two builds.  For some platform releases, we have had three or four builds that we believed to be final, only to find some late-breaking problem that requires an additional build. Under the current proposal, we simply promote the final development build as the release once we are satisfied with testing, and no further builds or invasive steps are necessary.
