Comment 8 Michael Garvin 2004-03-15 10:03:36 EST 
Our Workaround was to add specific support for it where needed.  For example:

1 - add yourself as a mouse listener to the text widget were you want pasting
to happen:

  entryArea.addMouseListener(this);

2 - now add the appropriate MouseListener methods, and make the "up" event on 
button 2 be the paste action:
  
  public void mouseDoubleClick(MouseEvent e) {}
  
  public void mouseDown(MouseEvent e) {}
  
  public void mouseUp(MouseEvent e) {   
 
    if(e.button!=2) {return ;}
        
    // paste in the system clipboard
       
    Clipboard cb = Toolkit.getDefaultToolkit().getSystemSelection();

    String data;

    try {
      data=(String)cb.getContents(this).getTransferDat(DataFlavor.stringFlavor);
    } catch (Throwable badflav) {

      // report error
                    
    }

This is obviously not ideal, but it does work.  We use it on HP-UX and on Linux.


    // insert the "pasted" text
                                       
    entryArea.insert(data);
  }

Comment 18 Florian Loitsch 2004-09-02 19:53:26 EDT 
you are right. The problem is, that Eclipse doesn't make the difference 
between PRIMARY and CLIPBOARD: when copy/cutting it puts the content into 
CLIPBOARD (and hence automatically into PRIMARY). When pasting it uses the 
PRIMARY (which is always the most up to date). 
The correct (although incomplete) implementation would paste CLIPBOARD on 
CTRL-V. 
The complete solution would furthermore set the PRIMARY on selection, and 
paste the PRIMARY on middle-click. 
Not ranting, but this is really important (eventhough it can be easily 
worked around): It's like having a keyboard with different keys than expected 
(or different shortcuts. just try to switch your copy/paste from your current 
configuration to something different).

Proposed approach for Linux-GTK+

Here's my suggestion for an API and implementation on Linux-GTK+.

- Clipboard includes hasPrimary() to indicate the existance of a
  primary clipboard, get/set implementations for Primary.
- StyledText copies to primary whenever the selection changes.
- if hasPrimary(), StyledText pastes from PRIMARY on middle mouse button up.
Comment 23 Veronika Irvine 2004-11-16 16:22:19 EST 
Support has been added to Clipboard for ICCCM Primary Clipboard.  See 
Clipboard.setContents(..., int clipboards), Clipbaord.getContent( ..., int 
clipboards) Clipboard.clearContents(..., int clipboards) and 
Clipboard.getAvailableTypes(..., int clipboards).  Here "clipboards" is any 
bitwise OR combination of supported clipboards such DND.CLIPBOARD or 
DND.SELECTION_CLIPBOARD or DND.CLIPBOARD | DND.SELECTION_CLIPBOARD.

The API is available on all platforms.  On platforms that do not support the 
selection clipboard, nothing happens if DND.SELECTION_CLIPBOARD is specified.  
The API is fully implemented on GTK.  I still have to implement it on Motif - 
the delay is because on Motif, I need to completely rewrite the Clipboard 
class to support this.

Also, StyledText has been modified to use this new API to support placing data 
on the selection clipboard when user selects text (with mouse or with 
keyboard) as well as pasting from the selection clipboard when user clicks 
with the middle mouse button.

Note that the native Text widget already supports the selection clipboard.

Comment 31 Veronika Irvine 2005-01-11 14:52:57 EST 
Sergey,

I am closing this bug report because it is fixed.  Please enter a new bug 
report with a specfic example showing your problem.

Andrew,

There are two separate clipboards.  1) Things you copy with the menu or with 
Ctrl+C are placed on one clipboard (XA_CLIPBOARD).  These things can be pasted 
with Ctrl+V or with the menu. 2) Text you select is placed in a different 
clipboard (XA_PRIMARY).  This text can be pasted by clicking the middle mouse 
button.

If you have selected some text and then hit Ctrl+C, the text will exist in 
both clipboards.  Selecting a different set of text will modify the XA_PRIMARY 
clipboard and will not affect the XA_CLIPBOARD clipboard.

I do not use xclip so I can not comment on it but other clipboard applications 
like klipper can be configured to specify whether or not they will detect 
the "selection" clipboard - also called the primary clipboard.  Having a quick 
look on google at xclip, I see that there is "xclip -sel" (to specify the 
primary or selection clipboard).

Regarding rxvt, I do not have rxvt.  Can you copy to and from xterm to rxvt?
