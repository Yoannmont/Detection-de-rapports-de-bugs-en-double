Description Marcio 2005-11-01 09:45:18 EST 
We have a language that had a Win32 IDE and now Eclipse, since 2.x. Depending 
on the VM state, the color to be used to highlight the executing line changes. 
It could be green or blue or red or brown (actually, the 4 different modes can 
be customized via a pref page).

This requirement of ours is an old one, see bug#33598 and bug#36068 for 
example.

Now the 3.x stream has support to customize teh marker annotation used for teh 
highlighting. However, doing so is a lot more work than it should. We did it 
the hard way, and here's what was needed:

1) Copy/Paste of these dbg internal classes into my plugin:
InstructionPointerAnnotation
InstructionPointerContext
InstructionPointerManager

2) Change InstructionPointerAnnotation's constructor to take teh marker type 
as parameter

3) Change InstructionPointerManager.addAnnotation so that first it asks the 
IStackFrame what the markerType should be (a String), and pass that to the 
constructor in step 2 above. (the more correct solution is to ask the debug 
model presentation)

4) In plugin.xml I define my 4 markers much like the debugger does for its 
default one.

5) In my FooBarDebugModelPresentation.addAnnotations I have to do this:

com.foo.bar.debug.ui.workaround.InstructionPointerManager.getDefault
().addAnnotation(myEditor, myStackFrame);
return true;

6) In FooBarDebugModelPresentation.removeAnnotations I have to do this:

7) com.foo.bar.debug.ui.workaround.InstructionPointerManager.getDefault
().removeAnnotations(myDebugTarget);
   


Whil ethis works, it is too much work. I should not have needed to copy/paste 
existing internal method.

My suggestion: the DBG platform should ask my debug model presentation for the 
marker type (string) to use for the Instruction Pointer. This way people only 
need to implement 1 method to override teh marker. In this method they can 
return any of the N markers they have for the different colors they may want 
to use depending on the VM state. The DBG core 
InstructionPointerManager.addAnnotation would ask me for this value first, and 
then call its InstructionPointerAnnotation's constructor passing this 
parameter, If I do not implement this method (do not implement the IWhatever 
interface) then you can use your current, existing default.

Comment 7 Darin Wright 2006-01-03 17:16:25 EST 
Not sure... we no longer use markers - we use annotations. Annotation provide the image and color for a text range. An annotation's presentation is defined in plug-in XML (i.e. an extension of type markerAnnotationSpecification), so it feels like the annotation is the smallest divisible unit. We could use an annotation image provider to get the image for an annotation dynamically, but this would not help control highlight colors.

I think that bug 113679 is now a dup of this one since the code for adding to the ruler is now controlled by the IInstructionPointerPresentation. You can't call the code yourself, but I think you just want to control what is displayed (rather than when).
