Description Dejan Glozic 2005-06-10 12:58:20 EDT 
Here is how I envision an ideal support for mirrors:

1) Our preference page has a group with two combo boxes like this:

--- Update Site Mirrors --------
  Mirrors for automatic updates             [ combo ]
  Mirrors for manual updates                [ combo ]
------------------------------------------------------

Where each combo has three choices: 

  - Don't use mirrors
  - Prompt the user with a list of mirrors
  - Test each mirror and pick the best return time

2) The third choice would go over the list of mirrors, ping them somehow and 
pick the one that has the best return time OR the first one whose roundtrip 
time is below some set value that is considered 'acceptable'. The former 
strategy will probably be better because it would not require us to open 
connection to each and every mirror - just pick the first that is fast enough.

I am going to copy this into an enhancement request.
Comment 1 Dorian Birsan 2005-06-10 13:18:23 EDT 
I slightly disagree with making the user worry about mirrors. If you think 
about it, all the world's larger web sites use their own load balancing 
without the clients (i.e. us browsing to a site) being aware of that. The 
simpler the user needs to do, the better. Let the servers worry about load 
balancing and handling mirrors under the coveer.

In fact, the eclipse.org site has a download.php script that does a round-
robin through mirrors for the sdk downloads, that can be used by the update 
manager as well. The algorithm assign some larger weight to the eclipse main 
download site.
Note that this is all done at the serve, so clients are not even being 
prompted to pick a mirror. In essence, this relies on a more intelligent 
update server. 
I have experimented a bit with this support, and it almost worked, but the 
update code does not correctly handle inaccessible mirror sites.
Comment 2 Dejan Glozic 2005-06-10 16:43:28 EDT 
Agreed - I was not thinking too far ahead. Choosing the right mirror 
transparently is the preferred solution.
Comment 3 Eclipse Webmaster 2005-08-20 10:52:18 EDT 
More and more projects are setting up their own Update Manager sites that link
to download.eclipse.org. To name only two:

EMF: http://download.eclipse.org/tools/emf/updates/index.html
ECF: http://www.eclipse.org/ecf/downloads.html#updates

This doesn't scale, considering everything on download.eclipse.org is mirrored.
And the list of Update sites on download.eclipse.org is only going to grow.

If the UM could support HTTP redirects, the eclipse.org servers could be used
for *all* Eclipse updates without ever prompting the user for a mirror.
eclipse.org servers would simply send redirects to other mirrors.

For a sample redirect, visit the following link:

http://www.eclipse.org/downloads/download.php?file=/arch/id.php&r=1

The &r=1 indicates to redirect without prompting; Remove it to see the familiar
mirror list. Note that the above script is configured with a variable to
throttle the amount of blind redirects sent to mirrors. By default, it is set at
35%, meaning eclipse.org will handle 35% of the UM downloads, and the other 65%
would be redirected to mirror sites.

The advantage here is that, down the road, I could use GeoIP to determine the
client's IP address and intelligently redirect to a mirror geographically
closer, or if eclipse.org gets ISPs in different countries, we could redirect to
a download.country.eclipse.org, such as download.uk.eclipse.org or
download.jp.eclipse.org.

D.

Comment 12 Alex Blewitt 2005-10-10 04:23:17 EDT 
Another possibility is that DNS could be used to provide a way of grouping servers from different wide-
area regions instead of treating the world as a single unit. SourceForge does this, as does Debian, by 
providing eu.dl.sf.net and us.dl.sf.net (and uk.debian.org ...) That way, you put your two letter country 
code in to the update manager's location, and it resolves to one of the 'local' mirror sites. This would 
certainly work in place of GeoIP in the short term.

If the DNS is too much of a problem to sort out, it could easily have a country/area code appended to the 
update request (e.g. http://download.eclipse.org/....?r=1&c=uk)

The goal would be for the Eclipse installation, once installed, to ask the user which area/country they live 
in, and then make all further choices automatically without user intervention.

Good work.  I love it.

+1 for server side. This would ensure we can still provide statistics for
download counts and modify the algorithm as needed.


+1 for the client passing the country code to the server. My experience dealing
with our mirrors is that if a download comes from the same *continent*, you're
already ahead of the game. If a download comes from the same country, you can't
go wrong.

little benefit, but you propose it be optional for servers to use.

supplied country code. I could imagine a server returning mirrors in this order:
1. same as supplied country code, 2. in the same continent as supplied country
code, 3. the rest; allowing us to place the Main Download Site in there, at our
discretion, depending on how busy we are.

+1 on Ed's proposal of Update Manager being proactive in looking at #bytes
transferred/seconds elapsed. UM could potentially try the top-3 mirrors from the
list for the first three files and then stick to the fastest one as the user's

I believe Platform is working on plugin signing and verification so this could

I am adding following as http get parameters to each mirror list request:
