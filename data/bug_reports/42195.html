Description Jared Burns 2003-08-27 19:56:19 EDT 
Build 20030827 (M3)

In recent builds, the install/update configuration data has been moved to a .config folder inside the 
Eclipse install directory. This means that upgrading Eclipse manually (delete the old install and 
unzip fresh) loses your installed features info.

I understand the reason for moving this data out of the workspace, but I don't think the install 
directory is the right place for it either.

Instead, I recommend creating a $HOME/.eclipse directory and placing the config folder inside 
that. This has the following happy characteristics:
1. It allows a user to maintain their set of installed plugins independent of their workspaces.
2. It keeps the install directory clean so that reinstalling Eclipse or doesn't lose the installed plugins.
3. Creating a ".eclipse" directory and putting the config folder inside it is a scalable solution that 
avoids cluttering the user's home dir with many more folders. We only add one top-level folder 
which other plug-ins can use in the future.

One problem I can imagine is that you might have Eclipse 3.0 and Eclipse 3.1 installed at the same 
time and the config formats might not be compatible. To address this, we could either append the 
Eclipse version number to the top-level folder (".eclipse-3.0" - this seems to be the common 
solution on Linux) or append it to the nested folder (".eclipse/config-3.0").

Comment 8 Jared Burns 2003-09-05 19:04:37 EDT 
To answer comment #2, yes. I'm installing a 3rd party plug-in to a separate location. The purpose 
of installing it to a separate location is so that it won't be lost when I upgrade my host. However, by 
moving the pointer to the plugin into the host install directly, that purpose has been defeated. :(

Changing the permissions of my $HOST/eclipse directory causes grief for me when I go to update 
my host, so I don't want to do that. Besides, I don't think users shouldn't have to do it to "trick" 
Eclipse into storing the data in the right place. Since $HOME/.eclipse works for both my case and 
your "usual" case, I suggest using it all the time.

This also has the side effect of making the config location consistently locatable, which is easier to 
support.

Comment 10 Jared Burns 2003-09-23 18:36:02 EDT

Comment 13 Dorian Birsan 2004-02-13 11:38:56 EST 
The issue here is that moving from build to build or using PDE (which can 
remove the .config on each invocation) destroys the .config, unless -
configuration is used.  

The .config directory location is no longer controlled by the update manager, 
we use the IPlatform.getConfigurationLocation(). This method has similar code 
to what we did in update.

I will move this to the Platform Core component for thoughts on this.
