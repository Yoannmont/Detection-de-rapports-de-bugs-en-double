Description Stefan Xenos 2007-09-11 11:38:20 EDT 
I work on an RCP app that does a lot of text file merging. We've been getting feedback that users don't know what to do when they see the compare editor.

Our observations:

- The "copy nonconflicting changes from right to left" button is the most useful one, but users have trouble discovering it since it is hidden in with a number of other similar buttons, none of which contain text.

- After users run the "copy nonconflicting... to left" button, they need to save the file before the compare markers get updated.

- After resolving the conflict in the editor and saving the editor, the user needs to run an additional action to tell the revision control system that the file has been merged.

- If the user used the preview pane to focus on a particular method, there is no longer a way to automerge the whole file.

- When merging Java files, most conflicts occur in the imports.


This enhancement requests:

1. The "copy nonconflicting changes from right to left" button should be easier to discover. We'd suggest making it a full-size pushbutton containing the text "auto-merge" rather than an icon.

2. After clicking "auto-merge", the conflict annotations should update themselves automatically without requiring a save.

3. There should be API such that clients can tell if a compare editor currently contains unresolved conflicts... such that if the user saves an editor that has been successfully auto-merged, we can offer to tag the file as merged.

4. The auto-merge button should always act on the entire file, regardless of what the lower-left viewer pane is currently showing.

5. The auto-merge button should have the capability of being extended for particular content types. In much the same way that we can supply an IStreamMerger that merges the entire file, it should be possible to supply a partial merger for the auto-merge action that applies to certain file types. In particular, Eclipse should include a merger that is aware of Java files and can auto-resolve all conflicts on import statements.



We've tried to implement an "auto-merge" button of our own on top of the Eclipse compare support, but since we do not have access to the working copy buffer from outside the compare editor, doing so appears impossible. It appears that this support needs to come from within Eclipse.

Comment 3 Michael Valenta 2007-09-11 16:11:36 EDT 
Here are my response to each of the items you proposed.

1) Where would you suggest we put the push button? I don't think it would be reasonable putting it in the toolbar and I hesitate to create a new area that was part of the compare editor input since this would take up considerable real estate for a single button. Perhaps a better approach would be to provide programmatic access to the action which would allow you to place the button in the top area. Thoughts?

2) Agreed. This shouldn't be difficult.

3) API to indicate that there are only outgoing changes left should be achievable. API indicating that there are no more unresolved conflicts is a bit more complicated since resolving a conflict will often result in what appears to be a conflict. API that indicated that all conflicts have been visited in some form may be possible.

4) Because of the architecture of the Compare framework, this is probably the most complicated part of your request. The copy method that is called from the action is on TextMergeViewer and, in the case you mention, the input is a document range. Having an operation that is associated with the view operate on the entire document when the input is a range could cause grief for some users. If we take the programatic approach to step 1 then perhaps we could provide a flag on that API that indicates that the operation should target the complete document.

5) Technically, this is possible given the current architecture. The copy(boolean) method is a protected method on TextMergeViewer that can be overridden by subclasses (e.g. JavaMergeViewer). This should be addressed in conjunction with bug 202948.

> 4. The auto-merge button should always act on the entire file, regardless of

There is an "Update" action available in Sync view (in popup menu for resource). It copies non-conflicting changes and updates content of the Sync view.

> 5. The auto-merge button should have the capability of being extended for
