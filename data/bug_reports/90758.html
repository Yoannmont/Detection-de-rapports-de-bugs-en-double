Comment 4 Morten Moeller 2005-04-25 18:36:15 EDT 
I'm not sure who owns this, but a little investigation shows this:

The JFacePreferences seems to not be used much as far as I can see, and worse is
that the only place the setPreferenceStore() in JFacePreferences is called from
is within a nodeChangeListener in WorkbenchPreferenceInitializer. I'm not sure
when this nodeChangeListener fires, but it doesn't seem to do so on startup..

---------------------------------
		rootNode
				.addNodeChangeListener(new IEclipsePreferences.INodeChangeListener() {
					/*
					 * (non-Javadoc)
					 * 
					 * @see
org.eclipse.core.runtime.preferences.IEclipsePreferences.INodeChangeListener#added(org.eclipse.core.runtime.preferences.IEclipsePreferences.NodeChangeEvent)
					 */
					public void added(NodeChangeEvent event) {
						if (!event.getChild().name().equals(workbenchName))
							return;
						JFacePreferences.setPreferenceStore(WorkbenchPlugin
								.getDefault().getPreferenceStore());
						((IEclipsePreferences) event.getChild())
								.addPreferenceChangeListener(PlatformUIPreferenceListener.getSingleton());

					}
---------------------------------------------

So the question is whether this is a bug that WelcomeEditor shouldn't be using
JFacePreferences (and rather use
WorkbenchPlugin.getDefault().getPreferenceStore()) or should some code have been
calling JFacePreferences.setPreferenceStore() during startup...
Comment 5 Tod Creasey 2005-04-26 07:44:13 EDT 
Adding DJ as he is the one who owns the Core preferences. Thanks for the analysis.
Comment 6 DJ Houghton 2005-04-26 14:58:57 EDT 
Tod and I have talked about this. The preference initializer should be used to
initialize the default preferences for your plug-in only. In this case it is
initializing the prefs for the workbench plug-in as well as setting up the JFace
default preference store.

The preference initializer is only called the first time the default preferences
are accessed for a plug-in so that is why this isn't being called.

We are considering other places to move this code.
Comment 7 Tod Creasey 2005-04-26 15:46:35 EDT 
Created attachment 20381 [details]
Patch that moves this to JFace initialization
Comment 8 Tod Creasey 2005-04-26 15:53:27 EDT 
This patch does the initialization in the following order

1) start the workbench bundle
2) Creates the workbench preference store and assigns it to the Jface
preferences (due to a Workspace#init)
3) Initialize default preferences (also in Workspace init)

Nick and I were concerned with adding this on startup as we wanted to avoid
initialization prematurely but I think this is OK as it does it in a node change
listener rather than right away so the workbench bundle gets started first.

Nick let me know if you have any concerns - if not I will submit this patch.
Comment 9 Nick Edgar 2005-04-26 16:30:44 EDT 
With this patch, JFacePreferences.setPreferenceStore is never called.

ActivityPersistenceHelper.hookListeners() during Workbench.init, which happens
after JFaceUtil init.

In any case, it seems wrong to trigger the initialization of the JFace pref

Comment 10 Tod Creasey 2005-04-27 08:32:42 EDT

Comment 11 Tod Creasey 2005-04-27 09:44:02 EDT 
I have followed Nicks suggestion and made the call to setPreferenceStore happen

Comment 12 Tod Creasey 2005-05-02 16:51:16 EDT 
In the case where the workspace chooser is used this solution initialized the
preference store too early (in the WorkbenchPlugin()). As a result in this case
the preferences were initalized from the wrong location.

The better solution was to initialize it in Workbench.init() so that the
preferences are pciked up everytime the workbench location changes.

I have verified that this fixes both the report this was logged for and the
potential loss of preferences from the workbench chooser.

To test:

Enable Run In Background
Restart Eclipse with no default
Select the workbench and check

forgotten that JFaceUtil.initializeJFace was called from WorkbenchPlugin.start
instead of Workbench.init.
