Description Javier Kohen 2005-04-29 10:58:13 EDT 
GTK+ has a feature where you can enter a Unicode character by typing it's
hexadecimal ordinal number. This works ala DOS' Alt+numpad, but GTK+ uses
Alt+Shift and the hexadecimal digits ([0-9A-F]{1,4}).

Eclipse doesn't completely disable the feature, as it's still possible to enter
the character, but the character is not then inserted in the text.

I can reproduce this with the Java and text editors at least. I think this bug
is also present in Eclipse 3.0.x and perhaps older versions as well.

As a side note, there's a mapping for key Alt+Shift+B in the default key map
that conflicts with this feature.
Comment 1 Billy Biggs 2005-04-29 11:12:25 EDT 
See bug 46580 and this GNOME bug for some more of our thoughts on this issue :)

  http://bugzilla.gnome.org/show_bug.cgi?id=82011

Doug can comment on default key bindings in Eclipse, many are changed from the
key bindings used on Windows to avoid conflicts with the GTK+ unicode character
entry feature.

However, I was under the impression that this feature works for entering unicode
characters in the Eclipse editor (it of course works in any Text widgets). 
However, it is also failing for me.  Felipe?
Comment 2 Billy Biggs 2005-04-29 11:13:29 EDT 
Also, the key binding (at least on my system) is Ctrl+Shift+[0-9,A-F] not Alt. 
Is this different for you?
Comment 3 Javier Kohen 2005-04-29 11:31:33 EDT 
No, no, no, my wrong... it is Ctrl-Shift here as well. I have Ctrl mapped to the
caps-lock key and that confused me when entering the report :)
Comment 4 Douglas Pollock 2005-04-29 11:49:06 EDT 
Yes, the binding should be "Ctrl+Shift+" unless the GTK+ team has changed it.

There are a few "Ctrl+Shift+[A-F]" bindings in the default set on Linux GTK+.  I
have tried to advise teams to avoid using these bindings, but ultimately it's
their choice.  I could try to remind them a bit more.  If you would like me to
do so, I would recommend filing a separate bug.

As an additional note, more to SWT.  I have platform-specific code in
ActionContributionItem.  This allows "Ctrl+Shift+[A-F]" accelerators to be set
as native menu accelerators -- avoiding the normal Eclipse key binding
architecture.
Comment 5 Felipe Heidrich 2005-04-29 12:02:29 EDT 
The bug is in StyledText#handleKey(), key events with mask equals to ALT, 
CTRL, ALT|SHIFT, CTRL|SHIFT are ignored (it's probably an accelerator anyway). 
It is also important to avoid the StyledText to insert(for example) ^A when 
the user types Ctrl+A.
I not sure how to fix this problem without reopening a old (more serious) bug.
Comment 6 Billy Biggs 2005-04-30 22:53:33 EDT 
Well, it's easy enough to just not ignore Ctrl+Shift in handleKey on GTK+ only,
since they do act on these keys for input.  (However, if they ever do change the
binding, it just gets more complicated).  If I disable handleKey's blocking of
CTRL+SHIFT, the GTK+ input method works, and I can't get any weird characters to
appear.
