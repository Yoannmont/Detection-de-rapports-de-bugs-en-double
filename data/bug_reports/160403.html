Description Michael Scharf 2006-10-10 18:56:20 EDT 
If you have multiple connection of the same type to different computers, it is rarely the case that the computers has the same directory structure. By making the filter shared by default, you get a lot of invalid filters when you create a new connection. 

This could be a preference setting.

See also bug 160402
Comment 1 Martin Oberhuber 2006-10-10 19:03:09 EDT 
Shared filters might be the default by design - RSE addresses this by a concept called "filter pools", the idea is that administrators might pre-create filters for their users to be re-used on different machines.

Also note that the files subsystem is not the only subsystem we deal with. On a processes subsystem, it may be a lot more likely to define filters which are valid on any Linux target, regardless of the host name. Same might be true for filename-filters that do not take a path into account or that are relative to the home directory.

A Preference setting might be a good idea, though it might need to be specifiable per subsystem type. On the other hand, different preferences for different subsystem types might be confusing for the user.

I have personally not had problems with the default shared filters so far. But I do agree that it makes sense to be able and modify a shared filter into a connection-private one later, once it has been created - see bug 160402.

I'm assigning this to DaveD and putting DaveM on CC, can we have more comments from you folks?
Comment 2 Michael Scharf 2006-10-10 19:20:32 EDT 
Makes sense. I have to look into filter pools (I have somehow ignored them up to now)....

Comment 7 Martin Oberhuber 2007-11-19 10:57:48 EST 
It should perhaps depend on the Subsystem whether a filter is connection-private by default or not.

In general, I find it confusing if the settings for a particular UI element come from multiple "shared" or "layered" locations. It's typically easier to understand if all settings are in one location only. That's why I find shared filters confusing -- and problematic when thinking about importing/exporting connection data.

Especially for the "Files" subsystem I cannot see why anybody would want to create shared filters. A "Files" filter will typically be useful on a single host only.

But for a "Processes" subsystem, for instance, the filters are less connection-dependent and more OS-dependent. Here, sharing the filters can make a lot more sense. But even here it's hard to understand why creating a new filter on connection X would immediately appear as a new filter on connections Y and Z of same type as well.

Especially in the context of our recent thoughts around import/export of profiles rather than direct team-sharing it seems more reasonable to import/export filters as well rather than doing direct filterpool-sharing. I think that by default, any kind of filter should be connection-private, and
we should improve the way how filters can be copied from one connection to
another (through drag and drop) or connections can be cloned.

I'd like to discuss this again in for the 3.0 time frame due to our general thoughts about persistence. I think that the whole setup, team-sharing, and import/export of connections would be a lot simpler (and user-friendly) if filters were stored connection-private with the connection by default.
Comment 8 David Dykstal 2008-02-08 12:44:54 EST 
It sounds like there are two separate items here:
1) change the new filter wizard to have the "create in connection" box checked by default
2) We'll need some improvements to filter copy and move. I've created bug 218349 for this.

There are related bugs that don't quite address this problem but should also be looked at:  bug 142452, bug 142447.
Comment 9 David Dykstal 2008-02-08 17:35:42 EST 
After some thought and talking to current products, there seems to be a consensus that all new filters should be connection private. Therefore I'll only be changing the default setting for the new filter wizard. If this needs to be tailored by subsystem or made a product specific preference we can open another bug report.
