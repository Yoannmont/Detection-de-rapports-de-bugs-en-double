Comment 5 Nick Edgar 2004-03-31 16:25:41 EST 
It does not look like SWT.Close is getting sent at all anymore on Windows.

SWT must not do a System.exit().  The Workbench does all the work to close

won't have a chance to shut down cleanly.

Comment 7 Steve Northover 2004-03-31 18:55:37 EST 
Is the EndProgram dialog appearing on top of your dialog, obscuring it?  Here 
is some code that shows SWT.Close working on Windows.  Can Eclipse be changed 
to follow this patern (is the pattern flawed)?

public static void main(String[] args) {
    Display display = new Display();
    final Shell shell = new Shell(display);
    Listener listener = new Listener() {
        public void handleEvent(Event event) {
            int style =
               SWT.OK | SWT.CANCEL | SWT.APPLICATION_MODAL;
            MessageBox box = new MessageBox(shell, style);
            box.setMessage("Exit the application?");
            event.doit = box.open() == SWT.OK;
        }
    };
    display.addListener(SWT.Close, listener);
    display.addListener(SWT.Dispose, new Listener() {
        public void handleEvent(Event event) {
            System.out.println("Saving global state ... ");
        }
    });
    shell.addListener(SWT.Close, listener);
    shell.addListener(SWT.Dispose, new Listener() {
        public void handleEvent(Event event) {
            System.out.println("Saving window state ... ");
        }
    });
    shell.setSize(200, 200);
    shell.open();
    while (!shell.isDisposed()) {
        if (!display.readAndDispatch())
            display.sleep();
    }
    display.dispose();
}

Comment 16 Nick Edgar 2004-04-02 15:58:32 EST 
With Steve's help, we figured out why I was not getting the SWT.Close event.
It turns out that it's not sent if you're running the java.exe VM rather than
javaw.exe.  Windows apparently doesn't recognize java.exe as a regular windowed
app and just blows it away.  I tend to run with java instead of javaw in order
to get the console, which is handy to have around when deadlocks happen.

Very strange that it doesn't even bother to send the WM_QUERYENDSESSION or
WM_ENDSESSION messages in this case, since there are still windows up and
processing the event loop.

I was also launching from a batch file, but that doesn't seem to make any
difference.
Comment 17 Nick Edgar 2004-04-02 16:17:49 EST 
I still think the System.exit needs to be removed from the WM_ENDSESSION
handling though.  But our reliance on being able to do further processing after
the SWT.Close event does seem dicey.

I just reviewed the doc for WM_QUERYENDSESSION and WM_ENDSESSION on MSDN:
http://msdn.microsoft.com/library/default.asp?url=/library/en-us/sysinfo/base/wm_queryendsession.asp
http://msdn.microsoft.com/library/default.asp?url=/library/en-us/sysinfo/base/wm_endsession.asp

From my understanding of it, WM_QUERYENDSESSION is used just to pop up a
confirmation dialog, if any, giving the app the ability to veto the shutdown.
It's not necessary for the app to actually shut down at this point.

If not vetoed, then you get a WM_ENDSESSION telling you to quit.  The doc for
WM_ENDSESSION states:
"the session can end any time after all applications have returned from
processing this message. Therefore, an application should perform all tasks
required for termination before returning from this message."

SWT currently sends SWT.Close in its processing of WM_QUERYENDSESSION, and
SWT.Dispose in its processing of WM_ENDSESSION.

Currently the Workbench shuts itself down in response to SWT.Close, but I guess
that doesn't hurt.  We could change this to have just the prompt and possible
veto in response to SWT.Close, and the actual shutdown in response to SWT.Dispose.

Either way, it seems that rather than just falling out of the Workbench and
letting the Runtime shut down, the Workbench needs some way of telling the
Runtime to force shutdown, which we would call after shutting down the Workbench.

Jeff, is this feasible?

Steve, please advise if I'm completely off base here.

Aside: I also noticed that there have been some changes to the order of
processing of these two messages.  The doc for WM_QUERYENDSESSION states:
"When an application returns TRUE for this message, it receives the
WM_ENDSESSION message and it is terminated, regardless of how the other
applications respond to the WM_QUERYENDSESSION message.
Windows Me/98/95: After all applications return TRUE for this message, they
receive the WM_ENDSESSION and they are terminated."
Comment 18 Steve Northover 2004-04-02 17:02:22 EST 
I think you need to be able to shut down in either SWT.Close (the SWT cake for 
WM_QUERYENDSESSION) or SWT.Dispose (the SWT cake for WM_ENDSESSION).  If you 
query in SWT.Close and shut down in SWT.Dispose, then you will be doing 
exactly what Windows wants you to do in the right WM_ message.  Try both 
ways.  It may be that if you do the shut down work in SWT.Dispose (the cake 
for WM_ENDSESSION), Windows won't taunt you with the "kill" dialog.
