Description iyerv 2003-01-20 15:53:23 EST 
In several of our views, we have to either get the zoomed state of a pane or 
toggle the zoom state of a pane. To do this we need access to the WorkbenchPage 
for ex: 
   ((WorkbenchPage) getSite().getPage()).isZoomed();
   ((WorkbenchPage) getSite().getWorkbenchWindow().getActivePage()).toggleZoom
(MultiPageViewPart.this);
WorkBenchPage is an internal class and a public API to determine and toggle the 
zoom state is needed.
Comment 1 Uwe Stieber 2005-06-29 03:56:47 EDT 
We do have a strong interest to get access to these two methods as well. We do 
require to maximize views while our automated tests are running. Prior to 
Eclipse 3.1, this may could have been gently ignored, but with Eclipse 3.1 it's 
getting very annoying because now Eclipse warn discouraged access to internal 
classes. It seems only a matter of time, this will become an error and the 
necessary classes becomes really internal. Before this may happen, access to 
these API's or any comparable method to set the zoomed state of a view is an 
requironment.
Comment 2 Paul Webster 2005-07-14 10:16:34 EDT 
Created attachment 24755 [details]
Add a public zoom API v01

This exposes the zoom API from the external workbench page class.  This is just
the first draft, we can change some of the names.

IWorkbenchPage now returns an IZoomManagement interface.

IWorkbenchPart part = ...;
IZoomManagement pageZoom = part.getSite().getPage().getZoomManagement();

pageZoom.toggleZoom(part.getSite().getPartReference());

if (pageZoom.isPageZoomed()) { ... }

Comment 9 Uwe Stieber 2005-08-01 05:01:07 EDT 
The use case is quite simple. We require to maximize views during automatic 
running testcases (junit) to asure that during the test all nodes of a tree 
within the view are always visible (the tree is programmatically expanded). The 
underlaying content and label provider needs to be stress-tested in order to 
collect trend data on how our views scale in case of very fast appearing and 
disappearing of tree-nodes. The pre-requirement for these test is that only this 
particular view is visible in maximized state. Public API does not allow control 
the zoom state of a view yet, but we strongly need this API for our product 
quality assurance and testing before Eclipse begins to turn discouraged access 
warnings into errors.
Comment 10 Paul Webster 2005-08-29 09:51:17 EDT 
(In reply to comment #8)
> I meant scenarios from end users or plug-in developers for why they need to
> programmatically drive maximization or minimization, or query the state.

I was thinking about this last week.  The first approach I would take is just to
take our zoom API with a small tweak or 2 and make it publicly accessible
through IWorkbenchPage somehow.

Another approach would be to consider it more from a window management
perspective, even though it's not quite that.  i.e. present the API in terms of
minimized/maximized/restored, etc.  We would make sure that the queries
currently provided by the zoom API were still derivable.

Would we offer per view manipulation or per stack manipulation?  Another request
we get all the time is "how do we programmatically move a view from one stack to
another?"  maybe this could also be answered by the window management approach.

Comments?

PW
