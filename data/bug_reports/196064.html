Comment 23 Nobody - feel free to take it 2007-09-14 05:02:13 EDT 
(In reply to comment #22)
> I am currently working on a port of the WR disassembly view to work with the
> CDI debugger. It won't be fully functional, just something to play with.

It shouldn't be a problem from the CDI side. It is designed to work with gdb/mi and your backend is very similar as far as I know.
The CDT disassembly view is built on the top of the IDisassembly interface, it's an extension of the standard debug model and is a higher level than CDI. Maybe it makes sense to consider using it instead of CDI. CDT will translate high level calls into CDI calls.
And of course it would be very helpful to hear your comments on these interfaces.
Comment 24 Anton Leherbauer 2007-09-14 08:55:00 EDT 
(In reply to comment #23)
> The CDT disassembly view is built on the top of the IDisassembly interface,
> it's an extension of the standard debug model and is a higher level than CDI.

The IDisassembly interface lacks API to request disassembly of an arbitrary address range, therefore I have to fall back to CDI directly.

Another point is that our backend provides a special implementation of the disassembly retrieval command (-wrs-data-disassemble) which is quite different from GDB's -data-disassemble. E.g. the response can contain the memory bytes (opcodes) of the instructions which the disassembly view can then display. In the case of GDB this would need an extra query. 

Those differences need to be factored out in the API, but at the same time it should be generic enough to allow the support of features specific to the backend.

Comment 30 James Blackburn 2007-09-25 05:35:15 EDT 
Hi All,

Having played with this, I discovered that GDBs -data-disassemble command doesn't do what you ask it to.  I've hacked on GDB a bit and now have it outputting the full address range you ask for, even if it spans multiple symtabs/symbols.

Now found that this plugin is quite fussy about what you give it back :), for example, it says:
3580-data-disassemble -s 0xaa8 -e 0xac0 -- 1
GDB replies:
3580^done,asm_insns=[
src_and_asm_line={line="97",file="<memcpy.c>",line_asm_insn=[
{address="0x00000ab0",func-name="memcpy",offset="192",inst="<instr>"}]},
src_and_asm_line={line="98",file="<memcpy.c>",line_asm_insn=[
{address="0x00000aa8",func-name="memcpy",offset="184",inst="<instr>"},
{address="0x00000ab8",func-name="memcpy",offset="200",inst="<instr>"}]}]

Now clearly I'm giving back the full range asked for, just in source line order, so I think what I'm doing is correct from GDBs POV.

Given that I now guarantee to disassemble the requested range -- irrespective of debug info being present (so there may be some bare {address="",func-name="",offset="",inst=""} packets in there) it would be good to just have the assembly added to the model.  Would also be good if additional disassembly outside the requested range was squirreled away (rather than causing an error/infinite loop of -data-disassemble commands :) ).

Going to have a go at making it work with our setup -- and contributing back the GDB changes if those guys are interested.  If you have any pointers Anton, I'm all ears!

Cheers,

James
