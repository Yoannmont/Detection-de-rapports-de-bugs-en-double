I tried all the usual tricks - close&reopen, clean&rebuild, delete & checkout from CVS, and it still has these errors.

Comment 4 Thomas Watson 2006-11-22 10:48:06 EST 
John, does this fragment actually resolve at runtime?  It seems to require org.eclipse.core.runtime with a version range [3.0.0,4.0.0).  But the org.eclipse.core.resources bundle requires org.eclipse.core.runtime with no version range.

These two contrants confict with each other and should cause the fragment not to resolve at runtime.  I was able to get the fragment to resolve if I removed its Require-Bundle org.eclipse.core.runtime constraint.  It is not needed because the host already requires that bundle.  There is a problem with the resolve in develoment mode that does not allow fragments to dynamically attach if they add new constraints to the host.  In order to get the fragment to resolve in the workspace I had to remove its requirement on org.eclipse.core.runtime and force the host to reresolve (I touched the resources manifest file to do that).

I will look at a fix to dynamically attach fragment constraints when in development mode when I return.
Comment 5 John Arthorne 2006-11-22 12:05:22 EST 
The fact that org.eclipse.core.resources specified no version range for its dependency on org.eclipse.core.runtime was a bug, and I have fixed this. Now org.eclipse.core.resources.restorer compiles as it did last week.

However, I don't see why this would cause a conflict for the resolver.  Since org.eclipse.core.resources specified no range, then it should happily resolve for any version of org.eclipse.core.runtime.  Since org.eclipse.core.resources.restorer has a range of [3.0.0,4.0.0), then any version  of runtime in that range should satisfy both the resources plugin and the restorer fragment.  I assume a lack of version range is interpreted as [0,infinity) by the resolver.
Comment 6 Thomas Watson 2006-11-30 16:56:00 EST 
The resolver could be improved in the scenario.  Currently the resolver flags an inconsistentcy if a fragment adds a constraint with a version range that is a subset of an existing constraint version range on the host.  In other words the host's constraint version range must be included in the fragment's constraint version range.

This simplifies the resolver because in this case we can simply drop the fragments constraint because we can guarantee if the hosts constraint is resolved then it will satisfy the fragments constraint.  We could enhance the resolver to be more lenient in the case and detach fragments if their constraints are not resolved to the final constraint picked by the host bundle when it was resolved.
Comment 7 Thomas Watson 2006-12-01 10:57:15 EST 
Released fix to allow dynamic resolution of fragments that have constraints while in development mode.  I opened bug 166499 to address the other issue from comment 6
