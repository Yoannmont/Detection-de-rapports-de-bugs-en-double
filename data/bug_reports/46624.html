Comment 16 Nick Edgar 2003-11-19 12:56:05 EST 
Figured it out (concurrently with Andre <g>).  The flow of control is:

In the UI thread:
- while processing the event loop in Workbench.runEventLoop,
- SWT gets the quit command from the OS and calls Display.close().
  - Display.close() fires a Close event
    - The Workbench has a Close handler which calls Workbench.close().
    - Workbench.close() saves the workbench state, closes all windows, and sets 
runEventLoop to false.
    - It returns event.doit = true.
  - Display.close() disposes the display.
    - Various disposeExecs release resources, including JFace's image registry.
  - At this point, Window.getDefaultImage() is disposed, and so is the display.
- It falls back to Workbench.runEventLoop, which realizes that runEventLoop is 
false and returns to Workbench.runUI().
- Workbench.runUI() calls Workbench.shutDown()
  - Workbench.shutDown() calls advisor.postShutdown()
    - IDEWorkbenchAdvisor.postShutdown() opens a ProgressMonitorDialog to save 
the workspace
      - this is created using a null parent shell, so SWT recreates the Display 
(!!)
      - but when it comes to set the image, Window.defaultImage has been 
disposed

The main problem here is that we are continuing to do UI work after the display 
has been disposed.  Due to Display.close(), it's possible to fall out of the 
event loop with the display having been disposed.  
Note that this can happen on Windows as well: Display.close() is called if you 
shut down Windows while Eclipse is still running.

We need some improved lifecycle to notify the advisor to do any interesting 
shutdown work while still in Workbench.busyClose().
One option would be to move the call to shutdown() to the end of busyClose().

Will discuss this with Jeem.

Comment 20 Steve Northover 2003-11-19 14:19:31 EST 
Vetoing doesn't seem like a good idea.  First of all, it doesn't work on the 
Mac (not sure it's possible, Andre?).

The intent is that you should never fall back to your event loop.  The Window 
System is exiting so their is nothing to fall back to.  The Display has been 
disposed and all your windows are gone.

The time out behavior on Windows depends on the Windows version.  If you don't 
respond quickly enough, XP offers the user a chance to kill your application.  
Older versions of Windows used to wait forever, causing the user to hit 
Ctrl+Alt+Del and kill your application anyways.

Please move this PR to SWT or open a new one about "inconsistent shut down" 
behavior.  Perhaps we need a judicious System.exit() here? (almost always the 
wrong choice but seems right in this case)

I'm leaving open for M9 because I still think that Workbench.shutdown() should

getting passed to the SWT.Close event's doit flag.  So we need to ensure that

save changes dialog, but cancel if you choose Cancel.
