Comment 8 Karice McIntyre 2006-06-29 11:39:50 EDT 
It is not possible to fix this for 3.2 (Callisto) since tomorrow is the ship date.  We cannnot have the path looking mangled, which is why the directional characters were introduced in the first place.  Also, are you using a JDK1.5 VM on Hebrew OS?  The only way the directional characters should be inserted in a string is if the locale is set to a bidi locale (check osgi.nl value in Configuration Details of the About dialog to verify the locale under which Eclipse is running).  You should be able to work around this problem by running Eclipse with command line argument -nl en (unless you prefer the workbench to be rendered RTL-oriented).

Comment 15 Karice McIntyre 2006-09-05 13:19:25 EDT 
Lina contributed the original patch to TextProcessor and I made some modifications to the comments to reflect the change in the algorithm used and the code to short-circuit the processing in cases where the string should not be processed.  Lina, if you have any objections to the revised patch please let me know in this bug.  Otherwise, I will proceed to get the patch released into the 3.3 code stream (I don't have commit rights to the osgi plugin).

Note that the patch also includes one API addition: a #deprocess method that removes all directional characters inserted by the TextProcessor from a string.  This was added as a convenience to clients who want to use the #process methods to display text properly to the user, but who also want to allow user input into that same string (for example, a text field) to save or process (in some other manner) the resulting user-inputted string in subsequent code.
