Comment 2 Johan Walles 2004-03-19 05:14:08 EST 
Regarding GTK, here's what Owen Taylor had to say about it at
"http://bugs.gnome.org/show_bug.cgi?id=129473":

- Applications that want to look reasonable usually draw
  to a backing pixmap, this prevents the user seeing
  clear/redraw flashing. I assume that SWT does this
  in some fashion, but then again, maybe not...

  GDK provides a nice convenient interface to this
  with gdk_window_begin_paint_rect/region(), though
  apps can also do it manually if they want.

Those calls are documented at
"http://developer.gnome.org/doc/API/2.0/gdk/gdk-Windows.html#gdk-window-begin-paint-rect",
and IMO they look like just the thing to use for implementing SWT's setRedraw().
Comment 3 Billy Biggs 2004-03-30 11:41:47 EST 
Johan: The reason those calls have no effect is that they only apply within a
GdkWindow.  Eclipse uses a new GdkWindow for every widget (motif/gtk+-1.2
style).  To move these GdkWindows around, create new ones, or otherwise change
their layout, Gdk needs to ask X to do this using XMoveWindow/XMapWindow.  These
X calls cannot be batched, so whenever they occur they can cause visible repaints.

As an example, you will never see a half-drawn button or other tearing effects
in SWT widgets.  Widgets themselves are correctly double buffering.  But if you
move 20 buttons around and create a listview, there's (AFAICT) no easy way in X
to avoid seeing intermediate states if each widget is a different Window and
you're not quick enough or doing something more advanced.

Comment 9 Billy Biggs 2004-05-07 09:49:02 EDT 
Johan, SWT does use gdk_window_begin_paint_rect() and friends, but they don't
apply to child windows.  It's a flaw in X and not a flaw in SWT.  If SWT wanted
to work around this by avoiding creating GdkWindows, I believe they would end up
having to duplicate a lot of X server code, and it still wouldn't work for Gtk+
widgets that create their own windows (for example GtkTreeview, GtkNotebook, I
think GtkButton).  I do not think these calls were intended to give the behavior
SWT needs for setRedraw().

Comment 14 Silenio Quarti 2004-05-17 17:50:39 EDT 
Just this time Billy :-).

We are creating a X window that is a child of the handle window (paint window on
GTK). This window is raised to be the top most window on the stack, so that it
is in front of all children windows. We also set the the background pixmap of
the  window to None (NULL). The fix also make sure that the window stays on the
top ot the stack and resizes when necessary.

So, this window blocks Expose events to the handle itself and all children. It
also clips out any drawing done with GC. And since the window has no background,
the X server will not draw the background.

Comment 18 Billy Biggs 2004-05-22 18:56:20 EDT 
Even if 'everything' is slow, fixing it is often dependent on the operation 
you are doing.  Some of the visible issues are programmatic: for example bug 
59528 solved some annoying flashing on editor tab switching simply by changing 
the order in which we hide the old editor and display the new one.  As a 
result, switching editor tabs is visually much improved in M9, but the order 
is still not correct for tab close (bug 63137).  Please file bugs for each 
case you notice to help make sure they all get fixed.

The whole 'paint the background colour first' thing is partly a design 
decision by Gtk+.  As an example, try dragging a window in front of Mozilla 
Firefox: you will see a trail until Mozilla redraws.  Firefox uses some 
drawing from Gtk+ themes, but is not really a Gtk+ application.  On #gtk+ it 
was argued that it is better to show the background colour than to show a 
trail, but maybe unset the background when layout changes occur to avoid the 
flash to the background colour.  This is not being done in many places though, 
see these GNOME bugs:

http://bugzilla.gnome.org/show_bug.cgi?id=113310
http://bugzilla.gnome.org/show_bug.cgi?id=113040

Still, we can do a lot ourselves.  A working setRedraw is a big step, 
continuing to identify problem cases and improving them will also help.
