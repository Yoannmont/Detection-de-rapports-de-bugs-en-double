Comment 6 Min Idzelis 2007-08-27 17:13:23 EDT 
I found this very annoying so I took the liberty to fix it. 

I made a few random improvements as well. 


1) Don't schedule the task at all if the preferences say that it's not enabled. (This is the root problem of this bug report.)

2) Use the new SubMonitor API so that status can actually be reported accurately.

3) Don't use the preferences to store session-based information. (Before, the list of "already analyzed" projects was stored/saved in preferences event though it was not a preference at all, and was really just used for the active workbench session to keep track of which projects have/have not been scanned)

4) Better code read-ability/efficiency. Use visitors to get a list of changes files from IProject and ResourceDeltas. Process the IFiles from the concurrent queue homogeneously. 

5) Removed the joins() for the various build and refresh jobs. There is no reason to prevent this task from running concurrently with the builds. Note... If this is really desired, the build scheduling rule (which really is WorkspaceRoot) should be set onto this job. But that I think is overkill for this "read-only" job. 

Note: This uses new 1.5 concurrent libraries for better concurrency. This means execution environments have to be updated, along with classpath. This has the consequence that a few non-related files needed to be updated to prevent the "Unnecessary cast" error from popping up.

Comment 11 Min Idzelis 2007-08-28 08:40:05 EDT 
I can answer why this task was "rogue."

It installs a resource change listener that runs all the time (regardless if the preference is enabled or disabled.)

The listener sends the changes to the background job's queue. Each time an item is added to the queue, the job is scheduled. This is the typical pattern for having a "background queue processing job." When the job runs, it processes and empties the queue and terminates.

The problem is a combination of 2 things.

1) the changes are sent to the job's queue regardless if the changes actually *need* to be processed (per preference setting.) 

2) The job joins() autobuilds, builds, and refreshes. 

This means, that even if the job will have "nothing to do"... it is active and "waiting" for the joins() to finish before it will terminate. 


My patch fixes both these issues, and several other ones. (See my previous comment)

Comment 19 Nitin Dahyabhai 2007-08-30 06:41:30 EDT 
(In reply to comment #16)
> BTW - doesn't WTP 2.0 *require* Java 5 anyways? Why not just move up to it?

*Parts* of WTP may require Java5, but not all of it does, and not all of it must.  That decision is handled by each component/subproject, and as I said before I'm not going to drag anyone upstream into Java 5 just for SSE Core.

(In reply to comment #17)
> Job priority has nothing to do with Thread priority. The task scanning job is
> now set to the minimum priority (maybe too extreme?) This means that if
> *anything* else is going on, the task will run slowly or not at all until it
> can get some timeslices from the threading system. 

The goal of the joins was fairly unrelated to running priorities, it was to ensure that scanning wouldn't begin while a build was still running.  Both can be I/O intensive and regardless of how many CPU cores you throw at the problem, that's going to have an effect on the entire system.  For something that produces no direct output, it must be of a lesser priority (in layman's terms) than building.

(In reply to comment #18)
> Also, got rid of that silly list tracking which projects have/have not been
> scanned. Upon closer examination, it really doesn't do anything and adds
> overhead. 

The "silly list tracking" is intended to prevent rescanning projects that hadn't been modified between workbench sessions.  There's no reason to rescan an entire project when it hasn't been modified at all, and once we know that a project has been scanned completely, it's smarter to simply process resource deltas as they come along.  Without it, you've just increased the workload being done exponentially, in addition to adding a lot of extraneous debugging output to the console.

It is entirely about attitude: users don't like their files, even .project, modified without their consent, and adding a builder to a project modifies the corresponding .project file.  It shows up in team synchronization, can cause problems with pessimistic team providers, and causes annoying log output for users who don't have that builder. Given the unpleasant choices of surprisingly modifying users' files without their permission or introducing the vocabulary of Builders to authors of file types that aren't "built" (plus having to explain why automatic building has to be enabled for changes to be picked up), this was the solution that was chosen.
