Description Daniel Holmen 2007-01-08 10:07:54 EST 
When entering long Strings into a Text control in a Table, the Text control always just shows the first characters instead of scrolling like the Text control does in other contexts (and in the TableViewer on other platforms).
Comment 1 Daniel Holmen 2007-01-10 01:02:00 EST 
I'm not sure if I managed to explain this bug properly, so I'll give it another go.

When editing a cell where the model's text is wider than the column, you will only  see the first characters of the text in the CellEditor text control.

The normal behavior on other platforms is for the cursor to scroll the text in the control so that you won't have to edit blindly.

Comment 4 Eclipse Webmaster 2007-07-29 09:21:27 EDT

Comment 5 Daniel Holmen 2007-08-09 15:08:01 EDT 
Illustrating this bug is quite simple. Just run Snippet019TableViewerAddRemoveColumnsWithEditingNewAPI, or any of the other inline editing snippets and try to enter a string that is wider than the text box. You will then see the cursor disappear "under" the next column.

On windows and linux, the contents of the text box scroll with the cursor - but not on OS X.
Comment 6 Daniel Holmen 2007-09-07 18:37:25 EDT 
Do you need any more information from me before you have a look at this bug?

It makes inline editing pretty much useless on OSX, so if there's anything you need from me just ask.
Comment 7 Silenio Quarti 2007-09-11 12:05:24 EDT 
Hi Daniel, I don't need more info. I can reproduce the problem. The bad news is that there is no easy fix for this. The native widget (HITextViewCreate()) does not scroll the text horizontally. That is the widget we use when the SWT.BORDER style is not specified for a single line text editor. When the SWT.BORDER is specified, we use a different widget (CreateEditUnicodeTextControl()) which has built in horizontal scroll.

Scott, do you know if there is any plans to fix this in Leopard? I believe I have seem radar bugs for this problem in the apple mailing list.
Comment 8 Scott Kovatch 2007-09-11 17:27:55 EDT 
I will ask on the internal carbon mailing list to comment here.
Comment 9 Curt Rothert 2007-09-11 18:19:07 EDT 
As a workaround, if the HITextView contains a single line TXNObject, you can get the desired behavior by embedding the HITextView in an HIScrollView that is created without the kHIScrollViewOptionsHorizScroll bit set. This will correctly scroll horizontally while typing (with no scrollbars!). If you need this behavior for  a multiline control, a Radar will need to be filed against HITextView/X.

Is there a reason that the EditUnicodeTextControl isn't being used in this case anyway? Is there some behavior that the HITextView provides when the SWT.BORDER style is not specified?
Comment 10 Silenio Quarti 2007-09-12 10:27:39 EDT 
EditUnicodeTextControl does not seem to have any way to remove the frame (border) around the widget. Something similar to the tag "kControlDataBrowserIncludesFrameAndFocusTag" for the DataBrowser control. That is the only reason we don't use EditUnicodeTextControl when the SWT.BORDER is not specified.

I will try embedding the HITextView in a HIScrollView.
