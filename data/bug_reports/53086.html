Description Andrew Clement 2004-02-25 12:40:35 EST 
This bug is to track the essential features that are required for 1.2 and it 
partially represents their priority.  Under each feature is a list of 
questions that need answering as part of the implementation of that feature.

Firstly, looking at the basic AJDT features.

1) Creating an AspectJ project
  Is it the same as defining a new Java project?

2) The AspectJ perspective and AspectJ browsing perspective
  How do these differ from their Java counterparts?

3) Creating an Aspect
  Uses the new Aspect wizard that Matt created? (which exposed aspect only 
modifiers like 'privileged')
  Related to this, we have the 'New AspectJ Class' and 'New AspectJ Interface' 
wizards - how do we avoid confusion with the existing 'New Java Class' 
and 'New Java Interface' wizards?
  How is the 'choice' between .java and .aj managed across these wizards - 
Aspects are always going to be created with .aj suffix - what about new 
classes and interfaces - is there a global preference on the workbench and 
then per project overrides?
  What happens if the user starts creating .java files for their AspectJ 
classes and interfaces - we can't open the AspectJ editor for these can we and 
they might contain pointcut declarations which would give us the dreaded red 
squigglies ?

4) AspectJ Editor and Outline
  Do we have eager weaving that will show annotations in the gutter for woven 
advice even before a full 'proper' build?
  What are all the icons for the new gutter annotations - how is 'dynamic 
residue' identified?
  How do we support navigation back from annotations to source aspects?
  Does the outline view contain the same cross reference information as the 
cross reference view? is the inclusion of this information in the outline view 
optional?

5) Code assist and editor templates
  Does code assist 'just work' inside Aspects? (i.e. advice) - are 
thisJoinPoint/tJPSP/tEJPSP always suggested within advice?
  Does code assist work when defining pointcuts? (e.g.  "pointcut p(): call(* 
ma<CtrlSpace>")
  Does code assist work for intertype declarations? (e.g. when introducing an 
int x field on a Hello type -> "int Hel<CtrlSpace>") 

6) Cross reference view 
  How does the cross reference view move with workspace selections?
  Is CtrlX its shortcut?
  Does it only show reference info on java elements or does it show reference 
info for that java element and any below it?

7) Building
  Is this any different to building of a Java project - do we have proper 
incremental compilation?
  Are their any incremental subtleties?
  If the compiler is 'extended' with declare error/warning statements, do they 
appear differently in the task view to show they are user extensions?

8) Executing AspectJ programs
  Is this identical to executing Java programs? (with the minor extension 
that 'search' on MainClass understands Aspects can have entry points)

9) Build lists
  See bug #50454 - do they look like that?
  Do we go for something simple in first release - context menus on AspectJ 
elements to add/remove them to/from the eclipse exclude list?  Do we have a 
build config editor?  Is there a concept of multiple build configurations with 
this scheme? 
Do we support importing from and exporting to .lst files?
  
10) Examples plugin
  Does it work as per the older AJDT?
  Does it just contain the AspectJ examples?

11) Debugging
  Does this just work like debugging Java only inside an AspectJ debugging 
perspective?
  What happens in a large multi project setup where we step from debugging a 
Java based project resource into an aspect that exists in an AspectJ project - 
do we flip to the AspectJ debugging perspective?

12) AspectJ/JDT Interactions
  Are there any interactions between JDT and AJDT that aren't captured above?
====

Beyond that we have the following advanced features:

13) Preferences and Properties
  What do we have beyond the possibilities for JDT?  Do all the JDT 
preferences apply to AJDT?
  How do options like 'aspectpath' manifest themselves?  is there a .ajpath in 
the project directory that contains this info?
  Do we externalize other compiler options?  What about lint?

14) Binary weaving
  How do we set it up?  Do we use binary projects?

15) Aspect Libraries
  How do we create an aspect library?  Do we use a standard jardesc mechanism?
  How are they specified as input?  Are they a special kind of library on the 
project properties path?
  Are there gutter annotations when binary aspects apply to source ?

16) Working with multiple projects
  What are the options for working with multiple projects?  where the projects 
might be Java or AspectJ.

17) visualisation
  Is visualisation entirely as per the old AJDT but using the new visualiser 
codebase? Are we going to visualiser anything new?  
  Are we having multiple visualiser modes which the new visualiser now 
supports?

18) ant integration
  Can we generate simple build.xmls for AspectJ projects to stop any silly 
user errors?  Are there any subtleties in configuring Eclipse ANT to recognize 
AspectJ correctly?

19) Extra pages on wizards (new aj project, etc)
  Which standard wizards will have extra pages in an AJDT world?  
  Are there extra tabs on the AspectJ project wizard for setting aspectpath?

20) Working with large projects (Xmx)
  How will we cope with large projects, setting Xmx to a higher value is a 
crap solution.  We should be able to better manage space consumption by the 
compiler and weaver.

21) Load time weaving launcher
  Are instances of this defined just like Application/JUnit launchers?

22) type hierarchy
  Beyond basically showing inheritance relationships amongst aspects and 
classes - do we reflect any declare parents introduced inheritance?

23) Searching
  Is this just searching supporting Aspect types (so I can searched for 
privileged things or 'before advice')?

24) JUnit
  How do we operate with JUnit - is there scope for streamlining how users 
might involve AJDT in testing (construction of mock objects, etc)?

25) PDE
  How do plugin developers exploit AJDT in their development - from code 
enforcement to full AO designs?  Are there any problems when plugins contain 
woven code?  Where does aspectjrt come from?
====
Comment 1 Andrew Clement 2004-02-29 11:27:59 EST 
Observations of Lancaster codebase - 29th Feb 2004
===
Trying to put a Java class into an AspectJ project gives the error (in the New 
Java Class wizard) "Folder is not a Java project".  Same for 
package/interface/scrapbookpage/sourcefolder

Same error behavior if you work the other way round too - cant create AspectJ 
elements in a Java project.

Java projects do appear in the AspectJ package explorer and vice versa.  but 
if you view a compiled java project in the AspectJ perspective, it doesnt 
collapse Java packages into that "a.b.c.d" type package representation, it 
shows them as nested folders "a", "b", "c", "d".  Same problem with the 
binaries - it doesnt know to hide the .class files in java projects.

Poor error reporting if you put advice in a class.

If you try and set up dependent projects in your AspectJ project properties - 
you can only make it depend on other AJ projects. (This is serious!)

With the duplicate set of preferences we have - we maintain separate lists of 
installed JREs  (Installing a new JRE for AJ doesnt make it appear for Java).  
Same goes for things like classpath variables, they are separate lists...

Doesnt work very well in M7 !
shadows project wont compile (guess underlying JDT changes have caused this).  
But even just running our mirror AJDT seems to misbehave - try creating a new 
project.
====
