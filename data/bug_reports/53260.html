Comment 5 Alex Blewitt 2004-03-29 07:40:26 EST 
The eclipse launcher is to blame. On other systems, the 'eclipse' is a pseudo-script that launches the 
Eclipse window, and then if it terminates with an error code, relaunches it. (e.g. for when the plugins 
are updated).

This is no longer necessary since M8 has the ability to determine plugin updates on the fly without 
needing a restart.

The Eclipse launcher should be reverted to the 2.1 implementation that did not support the auto-
restarting, but merely launches the application as one app. That way, it would be both dockable and 
have the right start/stop features that Mac apps have.

If a Mac app needed to be restarted, it could easily shut itself down, and terminate with a dialog box 
saying 'please re-start the application manually'. Given that this almost never needs to happen in the 
future, loosing that functionality whilst re-gaining the correct docking behaviour would be a very 
worthwhile tradeoff.

Comment 7 Alex Blewitt 2004-03-29 09:22:33 EST 
Comment on bug 53260 says
... within the constraints of a platform independent launcher

This is the problem. The platform independent launcher needs to be able to restart itself, and Mac apps 
are not able to restart themselves. They can launch second applications (which is what the current fix 
is) but this is not a desired feature of a Mac application.

So you can't fix this and still have the 'Eclipse' command line argument being the same as the Mac 
application.

What you can do is to have an 'eclipse' command line program, and Eclipse.app/Contents/MacOS/
eclipse being something different.

Trying to solve the problem of having a ubiquitous launcher isn't really going to solve the problem on 
Mac, because there's a difference between a Mac App and a command line launcher anyway. In any 
case, in the Eclipse.app, there's no way of the user (easily) being able to change the arguments, and 
whilst it's nice that some args can be passed in under the Info.plist, they aren't anything that an end 
user will want to have.

The solution in 2.1 previews worked fine; just don't try to get the Mac version being able to restart 
itself. It doesn't need to do that anyway now, because plug-ins are automatically updated rather than 
requiring a restart (which on all architectures was ugly enough).
