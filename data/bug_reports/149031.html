Description Stephen Pride 2006-06-28 12:08:51 EDT 
Platform: Linux
Other: Tomcat 5.0 w/IBM Java 1.4.2

After creating an application and running it, Eclipse intermittently crashes with a libxpcom SIGSEGV error.  Usually, killing any Tomcat process and restarting with "eclipse -clean" will resolve the issue.

Steps to reproduce:

1. Create application. For example, a Dojo app with a simple Tree.
2. Select the application, and select "Run as -> Run in Mozilla".
3. After selecting the Tomcat server, run the application.
4. Eclipse may crash with the following errors:

(in popup)
  "JVM Terminated. Exit code 1"

(on screen)
  SIGSEGV
  Problematic frame:
    C [libxpcom.so+nnnn] _ZN13nsCOMPtr_base16begin_assignmentEv+0x1A

Comment 2 Adam Peller 2006-06-29 11:45:20 EDT 
What version of xulrunner are you using?  I found that I had to move up to 1.8a3 to get things to run well on Linux, but then the debugger doesn't work: bug #149171
Comment 3 Javier Pedemonte (inactive) 2006-07-12 13:08:57 EDT 
I can confirm these crashes.  Happens with the XULRunner 1.8.0.x and 1.8.1 branch builds.  The crashes only happen the first time;  when restarting, Eclipse doesn't crash in the same spot (this was confirmed by Stephen in a private email).

Debugged this on a 1.8.1 branch build, and it is crashing on GTK calls inside Mozilla.  Not sure why.  Could this be caused by Eclipse 3.2 itself?  I don't remember seeing these issues in the 3.2 betas and release candidates.  Unfortunately, I don't have any of those old Eclipse builds any more, and I can't figure out how to download them from eclipse.org (they've been removed from the main download page).
Comment 4 Stephen Pride 2006-07-12 13:18:25 EDT 
I haven't done existensive testing, but Adam's suggestion of using Xulrunner 1.8.1a3 seemed to cause more stability.  Beforehand, it was almost impossible to run an ATF application in Debug-mode without crashing, but seems a lot better now.  I'll try to test more to see if there is a common theme.
Comment 5 Javier Pedemonte 2006-07-12 14:56:51 EDT 
(In reply to comment #4)
> I haven't done existensive testing, but Adam's suggestion of using Xulrunner
> 1.8.1a3 seemed to cause more stability.

This is to be expected, since I have added several crash fixes to 1.8.1 branch.  However, I see the same crashes (on startup and debugging) on 1.8.0.x and 1.8.1.

What's wierd is that the first crash I get (in AbstractMozillaBrowser.<init>), seems to be crashing inside GTK.  And I can't figure out why, and why subsequent runs don't crash in the same location.

Adam, Stephen, do you see the same thing?  If I run with a clean runtime workspace, and try to bring up "google.com" in the browser, I get a crash in "moz_drawingarea_new+0x2C".
Comment 6 Javier Pedemonte 2006-07-14 13:14:27 EDT 
Some info on the first crash I get (in AbstractMozillaBrowser.<init>):

In mozilla/widget/src/gtk2/mozdrawingarea.c:
  moz_drawingarea_get_type():
    Call to g_type_register_static() fails (returns 0).

This causes g_object_new() (in the function moz_drawingarea_new()), to fail (returns null), which leads to a crash for dereferencing a null pointer.

If I then restart Eclipse after the crash (without cleaning the workspace), then g_type_register_static() works as expected, and there is no crash.
Comment 7 Javier Pedemonte 2006-07-14 14:17:20 EDT 
Tracing this up the stack, it seems to me that XULRunner is getting a bad GTK window handle from SWT (at baseWindow.initWindow(getHandle(), 0, 0, 0, rect.width, rect.height)).  I think that then throws off all the other GTK function calls.  But I don't know much about GTK, so debugging this might take a while.
Comment 8 Javier Pedemonte 2007-08-16 18:52:35 EDT 
Since upgrading to a newer Linux distribution, I have not seen this crash.  This leads me to believe that it is a bug in GTK, which has now been fixed.  Closing this for now, but if anyone still sees this error with a recent distro, please reopen.
