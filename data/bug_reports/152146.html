Description Robert Taniwa 2006-07-28 10:50:46 EDT 
If I import an EAR with an EJB project and publish it to the server, the publish occurs and the state of the server switches to synchronized, but then immediately back to Republish if autobuild is on.

I believe the behaviour comes from the ServerBehaviourDelegate.publish and how tasks are executed. The delta list is built first, and then the tasks are retrieved and executed. However in the above scenario the delta is a list of modules to add to the server, then the EJB deploy occurs after which will generate new artifacts that will not get included in the current publish. As a result, as soon as the app is installed it will then need a delta publish for the new artifacts.

If you manually issue a deploy prior to adding the application to the server it avoids this scenario. But it would be better if there could be some way to allow additional flexibilty as to when a task gets executed during the publish so that we could deploy before adding the project to the server.

Comment 3 Angel Vera 2006-08-10 09:42:29 EDT 
after changing the order of how task are executed, the problem does go away. I noticed that the code does a query for all tasks for this type of server, and it the performs the task during the publishing. Meaning:

The code today on ServerBehaviourDelagete.publish()

1. Start publish       -------------------------------         , by calling the (ServerBehaviourDelagate).publishStart
2. Execute extension points of PublishTask , call the pluginRegistry and retrieve and execute all tasks one by one
3. Publish the server ----------------------------       , by calling the  (ServerBehaviourDelagate).publishServer
4. Publish the modules  ------------------------       , by calling the  (ServerBehaviourDelagate).publishModules
5. End publish           -----------------------------        , by calling the (ServerBehaviourDelagate).publishFinish


The fix will alter the behaviour to:

1. Execute extension points of PublishTask , call the pluginRegistry and retrieve and execute all tasks one by one
2. Start publish       -------------------------------         , by calling the (ServerBehaviourDelagate).publishStart
3. Publish the server ----------------------------       , by calling the  (ServerBehaviourDelagate).publishServer
4. Publish the modules  ------------------------       , by calling the  (ServerBehaviourDelagate).publishModules
5. End publish           -----------------------------        , by calling the (ServerBehaviourDelagate).publishFinish


The fix is simple but it can possible affects other deployers. I am discusing with QA contact how to approach this defect, wether we can fix it this release 1.5.1 or delay it to 2.0

Comment 6 Angel Vera 2006-09-06 14:01:21 EDT 
The problem is related that during the publish of the server we need to calculate the delta chances for the projects, during this process we block the workspace for changes. 

When the task being executed adds code to the project, and the workspace is locked the new code will not get compiled and thus it will not be picked up by the deltachanges calculations. 

The task being excuted needs invoke a project build and rebuild the cache after it is executed.

Adding 3 lines of code to the task seems to be the easiest way to resolve the problem as WTP 1.5.1 is closing down. In a later release we will have to re-think how to deal with this situation.

The workaround code looks like:
----------------
project.build(IncrementalProjectBuilder.INCREMENTAL_BUILD,null); 
server.getServerPublishInfo().clearCache();
server.getServerPublishInfo().startCaching();
----------------
