Comment 16 Philippe Ombredanne 2005-07-26 21:01:18 EDT 
I am changing the title of this request to project refactoring.
I think it better conveys what this is about, and becomes more self-explanantory
than project transforms.
Users understand what refactoring is about!
I am also bringing that discussion to the UI-dev mailing list with a summary of
this comment:

Projects state changes are akin to a pipeline of transforms applied to a project.
State changes are done commonly with wizards, and often sequences of wizards:
- adding some project nature & builder
- configuring some stuffs (i.e. classpath)
- adding some files (from templates for samples or external imports)
- performing some other actions.

The current state forces some fire and forget approach: there is only one entry
point, and each wizard can pipeline on its own to other wizards using it's own
mechanism.

Having a way to pipeline wizards that can be composed (and re-composed) based on
an extension should support most of the use cases, I think. 

Pipelining needs the ability to :
-- invoke and finish a wizard, 
-- pass data between loosely coupled wizards in and out. 
The ability to declare pipelines of wizards as an extension should help compose
wizards together is a loose way.  

For instance, in the case of PDE new plug-in project:
- create a plug-in project
- create a Java project
- apply templates for samples and create artifacts as required
For instance, in the case of CVS import:
- create a new repo location
- create a project 
- apply a new project wizard


* Triggers
Refactorings can be triggered by the user in the UI (New, Import, etc menus,
buttons, ...) or based on some change in context for a project. In the case of
project refactoring, context changes should be a natural trigger point. The most
similar available mechanism is activities/capabilities.

Comment 32 Konstantin Komissarchik 2007-03-19 20:40:31 EDT 
WTP has a solution for this in the form of "project facets" (http://www.eclipse.org/articles/Article-BuildingProjectFacets/tutorial.html). We are actively trying to find a way to make this framework available outside of WTP as there is nothing WTP-specific about it (simply where it incubated). We've approached the platform team about moving the faceted project framework to the platform in the past, but there wasn't the interest, so we are trying a different approach (see https://bugs.eclipse.org/bugs/show_bug.cgi?id=167144). 

Note that retrofitting natures to support user-initiated add/remove as is requested would be a very expensive undertaking. Solving more or less the same use case via project facets took something on the order of 1 man year to develop. This was spread out over roughly 2 calendar years to allow for the feedback in order to refine the api.
Comment 33 Konstantin Komissarchik 2007-03-19 20:54:41 EDT 
Should have answered the obvious question that's bound to come up: 

Q: How are project facets and natures related? Can they co-exist?

A: Facets and natures co-exist quite nicely. The approach we took in WTP is to wrap a facet around a nature where a nature already exists. For instance, we've done that with the java nature. This means that there is no effect on any existing code that might be checking for the nature, etc. Where a nature did not exist prior (such as the case with functionality added by wtp), we have not been creating a nature to go along with a facet as facets are a functional superset to natures.

Comment 36 Konstantin Komissarchik 2007-03-20 11:36:30 EDT 
One of the problems you will find with natures if you really tried to allow users to add and remove them for earnest is that what's there is a very small percentage of what's necessary. Natures were never intended to be controlled by the user and it shows in the api. Retrofitting the existing api to support this would not only be a lot of work as I pointed out, but would almost certainly require api breakage. 

I will use one example of this to demonstrate:

IProjectNature.configure() and IProjectNature.deconfigure() methods do not take any parameters. This makes it impossible to pass configuration information gathered from the user in the wizard. This means that most of existing natures don't actually make much uses of these methods. For instance, you cannot create a fully-configued java project by adding the java nature. Half of the necessary code is embeded in the java wizard.

Can all of these problems be fixed? Sure, given a large time investment, an api-breaking 4.0 release, and a lot of churn for everyone. Adopting project facets means existing natures and existing nature api is unaffected. No need to wait for the 4.0 release and no churn for everyone. Facets also bring a lot of cool features beyond the scope of natures, such as version support and ability to represent runtime support relationships (for instance a project with a java facet v6.0 should not be able to target a 1.4 jre). 

Regarding the builder-nature integration that you mentioned, I presume you are referring to the exstension point support or declaring that a builder is associated with a nature. We didn't add that to facets, because we didn't see advantage of doing this via an extension point. A facet install delegate can easily add a builder along with anything else it needs to do. If you know of other cases where there was something that you could do with a nature that you couldn't do with a facet, I'd appreciate a note about that.

Note that it's possible and often desired to have both a facet and a nature. This most typically happens when a nature already exists. For instance, the java facet that we implemented in wtp, as part of it's install adds the java nature. Basically you can think of the facet as the wrapper that enables user-interraction in that case since natures by themselves cannot.

The IProjectNature interface has a #configure() and a #deconfigure() methods. The contract of IProjectNature implies that thanks to these methods adding/removing a nature is an acceptable operation from API perspective, so it should be from UX as well.
Of course, some more complex migration can still introduce additional workflows (such as wizards in the Configure content-menu), but IMHO the existence of this menus is more a workaround from the lack of covering of this user-story in Platform.
