Description quartz quartz 2003-03-03 11:00:49 EST 
now that the bug of saving the window location is fixed,
could you fix the bug about its dimension? ;-)

On a 1280x1024, I size the window taller than 1024. It doesn't get saved, it
gets back at 1024.

All that because or the waste space of the stupid left toolbar, the window that
at some point becomes stay on top, over my taskbar "already on top" and the
borders everywhere. I don't maximize, slide the window so that I can click on
the taskbar, but since it's not maximized, I want to avoid loosing space by
frame borders.
Comment 1 Nick Edgar 2003-03-16 14:06:44 EST 
We trim the dimensions to fit the current display's client area.
This is important if you switch to a lower resolution.
However, we could remember the old resolution and only trim on restore if 
changed.
Comment 2 Billy Biggs 2004-10-22 15:23:55 EDT 
I'm not sure that it's a good idea to not clip to the size of the client area.  Saving
the resolution seems like a tricky hack, and I think it would cause confusion, especially
for users that often switch between dual and single head mode (laptops).  As well,
if you were just doing this to get back some extra space on the left hand size from
the old perspective bar, it's now more flexible in Eclipse 3.0.

I am marking this as WONTFIX.  Please re-open if you disagree.
Comment 3 quartz quartz 2004-10-22 22:10:39 EDT 
single vs dual screen are easily detectable, just (3*width)/(4*height): if you
get  higher then 2 (side-by-side) or lower than 0.5 (stacked), or other
multiple, you know it is not just a screen resolution change.

as for the comment 1: got my vote.

as for my original complaint: I don't maximize the window, yet I drag it
partially out of desktop for some debugging. So it keeps coming back on the edge...
Annoying.
Comment 4 Billy Biggs 2004-12-03 23:35:00 EST 
Not really as simple as you describe, since not all monitors are 4:3.  I have
used 16:9 monitors, and many people use 5:4 resolution, nor do I really feel
compelled by your use case.  If it's just for debugging, why is it bad that
every time the app restarts, it's moved to be visible?

I'm going to mark this WONTFIX again because I'm not convinced it would be right
to assume that if we're off-screen, the user actually wanted it there and did
not do it in error.

Comment 7 quartz quartz 2004-12-04 18:47:09 EST 
1-set your task bar to appear on top, auto hide.
2-start eclipse, maximize it (or not, but occupy some taskbar space)
3-use it for a for a while, run, debug, external tools, step through (frankly I
don't know what causes that stay on top... seams random)

Eventually, you move your mouse to the left edge (bottom edge if you still use
the task bar at the bottom... ;) and instead of the taskbar sliding/appearing on
top of the IDE, it appears underneath)

You can witness that even non-maximized, as long as the sliding taskbar would
intersect the IDE window.

The only workaround is to press the 'windows' key to pop the start menu by force.

I say it's the SWT [native] code, because no other java app does that.
