As of version 3.2, Eclipse allows the creation of nested projects. However, the CVS plugin does not handle this correctly. It reports updates for the parent project and the projects in the subfolders.

Comment 2 Axel Mueller 2006-06-08 08:03:54 EDT 
Sorry for being unclear.
My source structure is e.g. as follows and lays outside of my eclipse
workspace:
TOP/
  prj1
  src1/
    prj2
  src2/
    prj3
The source is under CVS control.
I create new Eclipse projects by checking out the module from CVS (here:
modules prj1, prj2, prj3). In the import wizard I do NOT choose the default
workspace folder that is suggested (uncheck box). For prj1 I choose for e.g.
/home/user/sources/TOP as destination folder and for prj2 and prj2 the folder
TOP/src1 and TOP/src2, respectively (Eclipse versions <3.2 did not allow this
and complained about a conflict). In the properties dialog of project prj1 I have excluded the source folders src1 and src2.

Now, if I synchronize all the projects in my workspace and there is e.g. a changed file in src1 then I will see an update remark for prj2 (that is OK) and and update remark for prj1 (that is NOT OK). If I update then the file in prj2, prj2 will still say that there is a file to be updated and will give me an error if I try to update the file (in prj1/src1).

In the project properties for prj1 I can select not to update subfolders but this will help only a little bit. Because I still will see local changes of the excluded subfolders in the CVS view.

In my opinion the CVS plugin should respect the projects settings i.e. excluded source folders should also be excluded from CVS.

Comment 5 Axel Mueller 2006-06-16 09:30:17 EDT 
Sorry for the delay. 
The source structure in CVS is as follows:
prog1/
  file1       # source file of prog1
  lib1/       # sources for lib that will be linked to prog1
    testfile  # source file of lib1
  lib2/       # sources for lib that will be linked to prog1

Creating Eclipse project files:
Of course, I need a project file for prog1. 
Import->Projects from CVS (or select New Project)
Browse the CVS tree and select prog1 (Use an existing module)
Check Out as a Project configured using the New Project Wizard.
Click Finish.
The New Project Wizard pops up. 
Select Standard Make C++ Project (I use the CDT plugin, but I guess you can select a Java project as well.).
Project Name will be "Prog1" and then deselect "Use Default Location" and browse your local file system. I choose /home/sources/prog1.

Repeat the steps above for lib1 and lib2. The local file system should reflect the same structure as the cvs tree.

We will have now 3 Eclipse projects called Prog1, Lib1 and Lib2. The source (and project) files are lying outside of your workspace.

Now open the Project Properties of Prog1. Go to Project Paths, click on Exclusion filter and then Edit ...
Click Add Multiple and select the folders Lib1 and Lib2.
Now Eclipse will ignore all source files in the subfolders Lib1 and Lib2 (for the Prog1 project).

Modifying the source files:
Select project Lib1 in the Navigator View and double click the source file testfile to open it. 
Modify and save the file.
If you have CVS label decorations activated you will see in the Navigator view that both projects Lib1 and Prog1 contain modified files!
Go to the Team Synchronizing Perspective and synchronize your workspace.
Select Outgoing Mode in the Synchronize View.
Both projects Prog1 and Lib1 will report that testfile contains local changes (Lib1/testfile and Prog1/Lib1/testfile)! 
I would expect to see only changes in Lib1 because the source files in /Prog1/Lib1 are excluded from the project Prog1!

Now you can select Lib1/testfile and commit it.
However, the Synchronize View will still show an remaining update in Prog1/Lib1!
If you try to commit this file (Prog1/Lib1/testfile) you will get an error dialog (file system out of sync).
Comment 6 Bogdan Gheorghe 2006-06-16 14:13:39 EDT 
Axel, I tried out your scenario but wasn't able to reproduce what you're seeing. Here's what I tried, step by step (please correct me if I got of the steps wrong):

1. Created a project outside of Eclipse (to avoid the .project file) with the following structure:

project1
    file1.txt
    folderA
        test1.txt
    folderB
        test2.txt

2. I check out project1 by using the import wizard and then the new Project wizard. I now have a project called project1 that resides at E:\project1  (outside of the workspace)

3. I now go back and check out folderA by using the import wizard. When it comes time to specify the location of this folder I say: E:\project1\folderA (ie. overwriting folderA from the project) - is this correct? I now have another project called folderA that resides at E:\project1\folderA.


3. I now go back and check out folderB by using the import wizard and follow the exact same steps as in step 2. I now have another project called folderB that resides at E:\project1\folderB.

The first thing to note is that the exclusion mechanism that you are using is not a core feature - it is provided by the CDT (if I remember rightly it is used to determine which folders are considered src folders). The CVS plugin does not consult this exclusion mechanism at all. To get your expected behaviour we would need a similar type of mechanism at the platform level.

I made a change in to test1.txt in my folderA project and saved it. The corresponding test1.txt in project1/folderA also gets updated and show up as dirty. I commit test1.txt from my folderA project; the corresponding file in project1 has its revision number incremented. So it seems to work as intended, did I misunderstand any of your setup instructions? BTW, I'm running 3.2 RC7.
