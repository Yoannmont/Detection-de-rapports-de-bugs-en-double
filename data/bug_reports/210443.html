Description Mike Morearty 2007-11-20 14:58:09 EST 
Build ID: I20071101-2000

Steps To Reproduce:
1. Mac only.  I have not seen this happen on Windows; I have not tried Linux.
2. This is a timing-sensitive bug, so you need to set your keyboard repeat rate to the fastest it will go, under Mac's System Preferences > Keyboard & Mouse.
3. In Eclipse, make a new Java project, and add this sample code:

public class J {
    public static void main(String[] args) {
        nest(0);
    }

    private static void nest(int i) {
        if (i<200)
            nest(i+1);
        else {
            Integer[] m = new Integer[100];
            for (int j=0; j<100; ++j) {
                m[j] = new Integer(j);
            }

            System.out.println("Breakpoint here");
        }
    }
}

4. Set a breakpoint on the "Breakpoint here" line, and debug.
5. In the Variables view, click on the disclosure triangle to expand array "m"
6. Click on the first array element.
7. Press and hold the keyboard's down-arrow to move down the array at full speed.

Actual result:
It will scroll correctly for a while -- perhaps 20 or 40 elements -- and then suddenly the selection will jump all the way back to the top of the Variables view.  It will then continue down, getting farther than it did the first time, but then eventually will jump to the top again.

Once you have made it to the bottom of the list, and all the elements have been read in, all navigation within the list will function correctly.

Expected results:
Selection continues to move down until it reaches the end of the array.

8. Try the same thing in the "Debug" view.  I have had less luck reproducing the problem there, but I have seen it on occasion.  I don't think this bug has anything to do with the debugger; I suspect it is a JFace or SWT issue.

More information:
This does not reproduce every time, but I can get it to happen fairly easily.  It is similar to bug 185673, but is not the same bug.  That one has been fixed; this one still reproduces.  (This one also reproduced in 3.3, but I don't think it reproduced in 3.2; I may be wrong about that though.)

Bug 185673 had to do with preservingSelection() in TreeViewer.replace(); it seems like this one might involve that code as well, because the behavior -- jumping to the top of the list -- is similar.

I find that the problem reproduces most easily after restarting Eclipse.  So if it reproduces for you but then stops reproducing, try restarting.

Comment 2 Mike Morearty 2008-10-13 17:25:59 EDT 
I debugged this a little, and I made some progress.  I tried setting a conditional breakpoint in StructuredViewer.preservingSelection(Runnable, boolean), just after this line:

    oldSelection = getSelection();

On the next line, I set a conditional breakpoint, with the condition being:

    oldSelection.isEmpty()

If I then open the Variables view and hold down the "down" arrow as described in the repro steps, I do eventually hit that breakpoint.

Digging further, I was able to see that the call to getSelection() (which is actually AbstractTreeViewer.getSelection() in this case) has a loop to take the actual selection of the Tree control, and then keep only those elements for which getData() returns non-null.  In the case of the bug, the Tree's selection did contain one TreeItem, but that item's getData() was null.  In the Variables view, if you select that TreeItem, the detail pane says "TreeItem {*virtual*}".

Given all that, here is my guess as to what is happening: Because the debug views handle SWT.SetData in a separate job, sometimes getData() is null when preservingSelection() was called, even though there is a job that has already been scheduled, which will (very soon) call setData() to set the TreeItem's data to some non-null value.

Here is the whole sequence that I think may be happening:

- The user presses (and holds) down-arrow.
- That causes a previously hidden item to come into view.
- That causes Tree.drawItemProc to get called.
- Since the tree is virtual, it dispatches a SWT.SetData event for the tree node that just came into view.
- The debugger's variables view is, by design, as lazy as possible (does everything as deferred tasks).  So when it gets the SWT.SetData, it does not immediately handle it; instead, it schedules a separate task to handle it.  You can see this: TreeViewer.hookControl() has an inline Listener.handleEvent() to handle the SWT.SetData, and that calls TreeModelViewer.virtualLazyUpdateWidget(), which eventually ends up calling TreeModelContentProvider.doUpdateElement(), which creates and then schedules a ChildrenUpdate.
- The point of all that is, the SWT.SetData is not immediately handled, so the node's getData() is still null, for a short time.
- Now it is time for the race condition to come into play.  We start processing another down-arrow.
- We eventually end up in preservingSelection().
- That code gets the current selection, but (for some reason that I don't know) deliberately excludes any elements of the current selection that have getData() == null.

So if that is in fact what is going on, then what is the fix?  I'm not sure.  I can think of a few possibilities:

- Perhaps code like AbstractTreeViewer.getSelection() should not be excluding nodes whose getData() is null (although I'm sure there is a good reason it does that, but I don't know what the reason is).
- Perhaps the handler for SWT.SetData should do the work immediately, without scheduling a background job.  This would be undesirable because it would take what is currently a background task and make it a foreground task, hurting performance of the variables view.
- Perhaps the handler for SWT.SetData should immediately call setData() passing in a dummy non-null object.  This might be the best solution.

By the way, this bug is currently marked as belonging to the SWT component, but now that I have looked at the code, I think it may be a debugger bug.  It depends, I think, on whether JFace has a rule that code must immediately respond to SWT.SetData.  If JFace has no such rule, then I think it's a JFace bug; if JFace does have such a rule, then I think it's a debugger bug.

Comment 5 Pawel Piech 2008-10-14 16:31:59 EDT 
(In reply to comment #2)

Thank you for the detailed analysis.  Yes, I do think it's a debug problem, though I don't think there's any simple fix.

> - Perhaps code like AbstractTreeViewer.getSelection() should not be excluding
> nodes whose getData() is null (although I'm sure there is a good reason it does
> that, but I don't know what the reason is).
getSelection() cannot return paths with null values, it would break too many assumptions resulting in numerous NPEs.

> - Perhaps the handler for SWT.SetData should do the work immediately, without
> scheduling a background job.  This would be undesirable because it would take
> what is currently a background task and make it a foreground task, hurting
> performance of the variables view.
Agreed, this would defeat the purpose of the lazy loading view.

> - Perhaps the handler for SWT.SetData should immediately call setData() 
> passing in a dummy non-null object.  This might be the best solution.
The trouble with this solution is that if the viewer had a selection on an item with a dummy value.  When that dummy value would get replaced by the real value, the selection would still be lost.  The dummy value would have contain identifiable information, such as an index or something, which could then be used to match it to the real value using a custom IElementComparer.  But the IElementComparer only takes the element, instead of a full path, so implementing it could be error prone as well.

Comment 8 Mike Morearty 2008-10-17 16:56:48 EDT 
Sorry, I hadn't yet gotten around to posting the comment that went with the sample program.  To reproduce the problem with the sample:

- run it (it compiles like the SWT snippets from http://www.eclipse.org/swt/snippets)
- select any node(s)
- press 'x' to delete the selected nodes, or 's' to deselect all.  After pressing either one of those, if running on Windows, you will see a dotted rectangle around one item near the previous selection, the "caret" position; but on Mac, you will see no selection.
- Then, press up arrow or down arrow.  On Windows, you will see that the selection moves from the caret.  On Mac, if you press down arrow, the very first item of the tree is selected, and if you press up arrow, the very last item of the tree is selected.

It appears that Windows supports the idea of a caret position within a tree control, and Mac doesn't.

I have found at least three places in JFace where this causes a problem:

- Two places where the debugger's desire to lazy-load everything collides with JFace's rule that getSelection() will never return items whose getData() == null.  StructuredViewer.getSelection() and TreeViewer.replace() both get the selection, do some work, and then restore the previous selection.  In both cases, this can have the unintended consequence of clearing the selection, if getSelection() encountered nodes whose getData() was null (which happens quite often with the debugger views).  A workaround that works for me is, in both of those functions, if getSelection() returned an empty selection, then do NOT attempt to restore the selection.

- One place where deleting the currently selected node caused the Mac to lose its place, in TreeViewer.remove().  In this case, a workaround that is working for me is to select the next item (or the previous item if there is no next item).  My code to do this was pretty verbose; perhaps someone with more intimate familiarity with SWT and JFace could do this in a more elegant way than I did.

Several more places look like potential trouble spots to me:

- Near the end of TreeViewer.remove() is some code that calls setSelection(), which I believe could have the problem mentioned above of accidentally clearing the entire selection.  (This is slightly different from the above-mentioned case where TreeViewer.remove() deleted the currently selected nodes.)

- AbstractTreeViewer.setSelectionToWidget(List, boolean) is given a list of nodes to select, but it first filters down the list to exclude those whose getData() == null.  I am pretty sure that at least once during my testing, I ran into this as a problem, but I can't remember how that happened, and I can't reproduce it now.  A workaround would be, once again, if the passed-in list had >0 items but the filtered list has 0 items, then do nothing (but this is weird, because someone has asked you to change the selection, and yet you are refusing to do it).

Another avenue I looked at was modifying the Mac implementation of Tree, so that it attempted to internally keep track of a caret position.  This would be a better fix if it is decided that part of SWT's contract should be that Tree widgets have a caret; that's a tough call.  In any case, I made fairly good progress with getting this to work, but my code was pretty hacky, and even when I was done, it wasn't working in all cases, and I never did figure out why.

I will post my JFace code soon...

Comment 10 Mike Morearty 2008-10-17 18:48:33 EDT 
Created attachment 115474 [details]
Proposed patch

Okay, attached is a proposed patch.  As I described previously, this is tricky to fix -- it requires a number of changes in different places, due to numerous pieces of code that fall into a treacherous gap between JFace's requirement that the selection that it exposes consist only of objects whose getData() is non-null, the debugger's lazy trees that frequently result in nodes whose getData() is null for a little while, and the Mac's tree widget which loses its place, so to speak, if you clear the selcetion.

The proposed patch fixes the three places mentioned in my previous comment, in order to attempt to get JFace to never accidentally clear the selection.

As described previously, in the two places that were saving the selection and then restoring it, my change is that if the selection being restored is the empty selection, don't bother restoring it.  And in TreeViewer.remove(), if deleting the selected item, find another item to select in its place.

Not sure if this is the right fix, but it works for me.  I have not run any automated tests (not sure how).

I did not add "@since" tags on the handful of functions I added to TreeViewer, because I am not sure whether there should be "@since" tags, or if so, what version they should specify.  (The functions are protected.)

One more thing: The remove() change is to deal with the case where variables are being filtered out by a variables-view filter, e.g. statics or constants being filtered out.  You will never run into that with the original repro case that I posted.  I'll see if I can post another repro case that includes that case.

Comment 15 Mike Morearty 2008-10-20 18:45:59 EDT 
I agree that this is operating system behavior.  For that reason, I don't necessarily think this should be worked around in SWT, although I would be fine with that resolution.

> From a conceptual high level point of view: "Why should TreeViewer need to
> delete the selected item or set the selection when the user is holding down the
> arrow key?"

Here is one simple scenario:

1. user presses down-arrow, which causes another line of the Variables view to scroll into view
2. SWT sends SetData message
3. debugger decides "oops, that variable is static, so it should be filtered out, not displayed"
4. so, the debugger deletes that item (which is the current selection) from the variables view

There are other scenarios as well.  Most of them have to do with the fact that the debugger views (variables etc.) are highly asynchronous -- they do tons kinds of stuff lazily.

Comment 18 Pawel Piech 2008-10-21 14:32:03 EDT 
(In reply to comment #17)
> I'm going to pursue the fix in SWT a while.
> 
> In the meantime, the debugger needed to tell the list how many items it had and
> you are telling me that it changes its mind based on scrolling, is that right?
> 

Correct.  Debugger views request elements for items lazily.  So as items are made visible in the view, the elements are requested from the content provider.  As the elements are received, view filters are applied on them, and if a given item is filtered out it is removed.  This also has a visible side effect animating the scroll bar in rather strange ways.  We have discussed changing this behavior so that if filtering is used elements are not retrieved lazily, so I think it's fair to say that that part of the bug belongs to Debug (and already covered by bug 189033).
