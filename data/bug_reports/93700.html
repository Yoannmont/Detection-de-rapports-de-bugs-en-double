Comment 1 Stefan Xenos 2005-05-05 19:10:13 EDT 
I was able to reproduce this stack trace as follows:

1. Open two editor stacks.
2. Give the rightmost editor focus.
3. Edit the file on the left outside Eclipse (use Notepad or something)
4. Click the close button for the folder on the left (without first giving it focus)

There seems to be three problems here:
1. In general, the workbench page cannot handle being modified while it is in
the middle of notifying its listeners of a previous change.
2. CTabFolder is still firing a close event even though it is still in the
middle of notifying its listeners of a mousedown AND there is a modal child
shell open. Presumably either scenario should prevent the widget from firing
user events.
3. AbstractTextEditor should not be opening a modal dialog inside an activation
listener. (It should open a nonmodal shell or - preferably - a control that
covers the editor).

I've added an explicit check that prevents a part from being closed while it is
still in the process of being activated (it also logs a more informative error
message). This prevents the workbench from entering an invalid state when this
bug occurs.

Moving to SWT for investigation of issue 2. cc'ing Daniel for issue 3.
Comment 2 Veronika Irvine 2005-05-06 09:19:37 EDT 
CTabFolder fires the Close event on MouseUp not MouseDown so I do not believe 
it is possible for the Close to come while MouseDown events are being sent.  
It is entirely possible for Close to come while MouseUp events are being 
processed.  CTabFolder detects the MouseUp in a MouseUp Listener and fires off 
the Close.  Since CTabFolder adds its listener for MouseUp in the constructor, 
it is the first MouseUp listener to ever get called.  All application MouseUp 
listeners will be called after the Close event is sent.  There is currently no 
mechanism for a custom widget to do anything else (except asyncExec which 
would not be desireable because of the uncertainty in timing).

I will take a look at this scenario and see if I can understand what is going 
on.
Comment 3 Veronika Irvine 2005-05-06 12:17:39 EDT 
The following simple example demonstrates the problem.  It fails for 
CTabFolder but not for Composite.  Click on the main shell that opens.  You 
should get a MouseDown and not a MouseUp printed to the console.

public static void main (String [] args) {
	final Display display = new Display ();
	final Shell shell = new Shell (display);
	shell.setText("main");
	shell.setLayout(new FillLayout());
	//Composite c = new Composite(shell, SWT.BORDER);
	CTabFolder c = new CTabFolder(shell, SWT.BORDER);
	Listener listener = new Listener() {
		public void handleEvent(Event e) {
			switch (e.type) {
			case SWT.MouseUp:
				System.out.println("Mouse Up");
				break;
			case SWT.MouseDown:
				System.out.println("Mouse Down");
				Shell shell2 = new Shell(shell, 
SWT.APPLICATION_MODAL | SWT.DIALOG_TRIM);
				shell2.setText("modal dialog");
				shell2.pack();
				shell2.open();
				while (shell2 != null && !shell2.isDisposed()) 
{
					try {
						if (!display.readAndDispatch())
							display.sleep();
					} catch (Throwable th) {
						System.out.println(th);
					}
				}
				
				break;
			}
		}
	};
	c.addListener(SWT.MouseUp, listener);
	c.addListener(SWT.MouseDown, listener);
	shell.setSize(150, 150);
	shell.open();
	while (!shell.isDisposed ()) {
		if (!display.readAndDispatch ()) display.sleep ();
	}
	display.dispose ();
}

Comment 5 Veronika Irvine 2005-05-09 11:21:08 EDT 
OK, here is the simple example with Composite that shows the problem - you 
need a DragDetect listener.

public static void main (String [] args) {
	final Display display = new Display ();
	final Shell shell = new Shell (display);
	shell.setText("main");
	shell.setLayout(new FillLayout());
	Composite c = new Composite(shell, SWT.BORDER);
	Listener listener = new Listener() {
		public void handleEvent(Event e) {
			switch (e.type) {
			case SWT.MouseUp:
				System.out.println("Mouse Up");
				break;
			case SWT.MouseDown:
				System.out.println("Mouse Down");
				Shell shell2 = new Shell(shell, 
SWT.APPLICATION_MODAL | SWT.DIALOG_TRIM);
				shell2.setText("modal dialog");
				shell2.pack();
				shell2.open();
				while (shell2 != null && !shell2.isDisposed()) 
{
					try {
						if (!display.readAndDispatch())
							display.sleep();
					} catch (Throwable th) {
						System.out.println(th);
					}
				}
				
				break;
			}
		}
	};
	c.addListener(SWT.MouseUp, listener);
	c.addListener(SWT.MouseDown, listener);
	c.addListener(SWT.DragDetect, listener);
	shell.setSize(150, 150);
	shell.open();
	while (!shell.isDisposed ()) {
		if (!display.readAndDispatch ()) display.sleep ();
	}
	display.dispose ();
}

Comment 8 Steve Northover 2007-06-20 18:56:05 EDT 
The problem is that OS.DragDetect() eats the mouse up under certain conditions.  We use SendMessage() to send a fake mouse up.  When a grab or the active window changes, the WM_LBUTTONUP goes to the wrong window.

I believe the right thing to do is post a fake mouse event to the queue and let Windows decide who gets it.  I have a hack/fix of this somewhere but I remember that I caused problems with WM_COMMAND and tool bar (of all things).
