Comment 11 Darin Wright 2005-04-20 23:15:14 EDT 
Fixed in RemoteTreeViewer. After children (variables) are fetched in the 
backgound, we replace the previous children with the newly fetched children in 
the tree. In the case that the tree item's data is the same as the newly 
fetched variable, we did not update the tree item. However, we should still 
update the item's label and plus sign, as the value of the variable may have 
changed (even though the variable is the same). Since labels are computed in 
the background as well, we usually computed the new label, but a race 
condition means the old label may remain. The fix is to update an items label 
and plus sign when the old and new child are equal.

Comment 38 Mike Morearty 2005-09-29 16:31:15 EDT 
Okay, I think I've got this one by the tail.  I have almost-consistently 
reproducible case of the problem, and I have a pretty good understanding of 
where the bug is.

The fundamental problem is that AbstractTreeViewer and StructuredViewer are 
full of places where they assume that a given user 'element' only appears in a 
single place in the viewer.  This is not true if, for example, you have a 
variable which has a member that points to itself -- or if there are two 
variables that both point to the same object.

This repro case will exhibit numerous problems with the variables view:

* expanding a variable doesn't show all its members;
* sometimes variable values don't update (bug 82850);
* sometimes a modified variable doesn't show in red;
* sometimes the CPU goes to 100% (bug 106150).

Here are the repro steps:

1. Make sure the Windows Task Manager, or any other CPU monitor, is running.
2. Make a project with the sample file X.java (see attachment 4 [details]).
3. Set a breakpoint on the line with the println() statement.
4. Launch for debugging.
5. After you hit the breakpoint, open the Variables view.
6. Expand variable "this".
7. Expand variable "this.x".

At this point, you will see the first problem -- under this.x, the only member 
you will see is 'i' -- you will not see 'j', 'k', and 'x'.  Also, you may see 
the CPU pegged at 100%.

8. Step over a few times.

Sometimes either 'this.i' or 'this.x.i' will not have its value updated 
properly.  Sometimes either 'this.i' or 'this.x.i' will have its value 
updated, but will still be shown in black (indicating that its value has not 
changed), instead of red.


I think these are SOME of the changes that need to be made:

* Currently, the VariablesViewer ends up calling down to 
AbstractTreeViewer.internalAdd().  However, that function has a comment 
indicating that it is "EXPERIMENTAL," and should only be used by the JDT.  So 
either internalAdd() needs to be fixed to be no longer experimental, or 
VariablesViewer needs to be changed to not call this function.  At the end of 
this comment I have pasted the full callstack that got me into this function.

* As mentioned above, there are numerous places where it is assumed that a 
given 'element' (user data) has only one corresponding 'Item' (list/tree 
node), when in fact this is not a valid assumption.  One place where this 
assumption is made is in AbstractTreeViewer.itemExists(), which attempts to do 
a constant-time lookup if usingElementMap() is true:

	private boolean itemExists(Item[] items, Object element) {
		if(usingElementMap())//if we can do a constant time lookup 
find it
			return findItem(element) != null;
		for (int i = 0; i < items.length; i++) {
			if(items[i].getData().equals(element))
				return true;
		}
		return false;
	}

The problem with the first two lines is that they don't look at "Item[] items" 
at all.  They will find 'element' *anywhere* in the view, not just within the 
subset of items specified by "Item[] items".

This function is called from createAddedElements(), which can then incorrectly 
conclude that the element already exists within the specified array of items 
(when in fact that element already exists somewhere *else* in the view).

* AbstractTreeViewer.createAddedElements() has this code fragment:

	if(itemExists(items,element)){
		refresh(element);
		newItem = false;
	}

As already discussed above, itemExists() is sometimes returning true when it 
should return false.  But in addition, this call to StructuredViewer.refresh() 
is problematic.  It is only passing down 'element', and 
StructuredViewer.refresh() is assuming that there is only a single matching 
Item anywhere in the tree.

I am not sure how StructuredViewer.refresh() should be rewritten.  Either it 
needs to be changed so that you pass down the actual Item that needs to be 
refreshed, *or* it needs to walk through *all* matching Items and refresh all 
of them.  Not sure what the proper fix is.

* StructuredViewer.findItem(Object element) has the same problem: It returns a 
single Widget, when in fact an 'element' may have more than one corresponding 
Widget (if the element is in more than one place in the list).  This function 
looks up the element in the field "elementMap", which is a mapping 
from 'Object element' to 'Widget item'.  However, I think elementMap probably 
needs to map to a *List* of Widget items.


Perhaps I am missing something basic here.  As far as I can tell, the 
assumption of a one-to-one mapping between user 'elements' and 
list/tree 'items' seems to be deeply ingrained in the source, and fixing this 
will probably have a fairly big ripple effect.
