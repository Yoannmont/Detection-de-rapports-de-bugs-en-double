Steps To Reproduce:
org.eclipse.ui.internal.forms.widgets.FormUtil is currently made hidden from extending clients which significantly restricts possiblities of reuse of WidgetFactory class (that is heavily using FormUtil). 

Due to certain behavior (various color changes: foreground and background) of widgets created via WidgetFactory, our current only choice is to reimplement WidgetFactory. Because WidgetFactory is heavily relying on FOrmUtil class -- we get a lot of "discouraged access" warnings in our code.

Comment 3 Barys Dubauski 2007-03-16 19:36:51 EDT 
Sorry for confusion. When I said WidgetFactory in the initial description I did mean FormToolkit. Here are limitations on FormToolkit that restrict its direct use in our code and forces us to override default FormToolkit behavior:

1) FormToolkit doesn't handle widget background colors well and also by default changes the Border style on the widget. 

E.g. SWT.CHECK style button will have inconsistent border style in our UI compared to other widgets. The widget will unconditionally get .FLAT style from FormToolkit.

E.g. No matter what Display object is passed to FormToolkit,  new SWT.CHECK buttones will get white background color. This will be incorrect if the widget is used in a JFace dialog where dialog's background is grey. In addition, the disabled state of such button will also have white background -- which again is incorrect for grey colored dialogs.

2) not all widgets are covered by the FormToolkit. 

E.g. Text and Combo widgets aren't available.


Due to the fact that background color is being set in 'public void adapt(...)' methods, the only way to fix problem #1 for us was to override FormToolkit. However, one of the adapt() methods is using 

	/* default */
	VisibilityHandler visibilityHandler;

	/* default */
	KeyboardHandler keyboardHandler;

which both have package scope -- thus makes them unavailable in an overriding class. This forced us to reimplement this handlers in our code. As you are aware , both VisibilityHandler ad KeyboardHandler classes use FormUtil for their underlying implementation:

    private static class VisibilityHandler extends FocusAdapter {
        public void focusGained(FocusEvent e) {
            Widget w = e.widget;
            if (w instanceof Control) {
                FormUtil.ensureVisible((Control) w);
            }
        }
    }

    private static class KeyboardHandler extends KeyAdapter {
        public void keyPressed(KeyEvent e) {
            Widget w = e.widget;
            if (w instanceof Control) {
                if (e.doit)
                    FormUtil.processKey(e.keyCode, (Control) w);
            }
        }
    }



This is how we arrived at FormUtil and its closed state for reuse. Hope this clarifies the context.
