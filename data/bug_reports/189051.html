Comment 11 Pascal Rapicault 2007-05-31 16:13:27 EDT 
Kevin, Michael, could you please answer to Wassim's question? Also note that there is currently no other way than setting the EE in your manifest.

Wassim, I have been able to track down how one could end up in the situation.
Here are the details:
 - PDE build sets the compiler setting in a generated build.xml using the manifest EE settings or the javacSource and javacTarget properties from the build.properties. When none of these are available, the generated build.xml contains two ant properties javacSource and javacTarget respectively set to 1.3 and 1.1.

 - In the case of interest here, the user did not set the EE, nor did he set properties in the build.properties. However project specific compiler settings (1.4, 1.4) have been defined and differ from the workspace compiler settings (5.0, 5.0).
At export time, when PDE UI invokes the build.xml it sets the two ant properties javacSource and javacTarget to match the workspace compiler settings, which causes the code to be compiled using the wrong settings (5.0 instead of 1.4 in this particular example).

Andrew and I have discussed the issue together and believe that the setting of javacSource and javacTarget to the workspace compiler setting is wrong, and will always cause problems to users. In fact the only good answer, as we have recommended in the past, is to either force people to have an EE and / or inject into the build.properties project specific compiler settings.

In the absence of that we think that the logic used to set javacSource and javacTarget could be improved to accommodate most of the use cases. Here is a little logic that could be used to determine which values to set:

 if (ee is specified on all projects being exported)
   do nothing special and return
 if (javacSource and javacTarget specified in build.properties on all projects being exported)
   do nothing special and return

 for all projects
   get the compiler settings  (from the most relevant place)
      if they are all the same, return these
      else
         return only the workspace level setting

Hi Chris, apologies on my part. I received an issue from a customer who is using one of our products based on Eclipse and is developing Eclipse plug-ins. Turns out Brian already gave them a response similar to yours.

The following reply was given by Brian Bauman:
>>>>>>>>>>>>>>>>>>>>>>>>>>
To fix

bunch of code change which would be very risky in a point release.  

Just so you are aware, there is a relative simple work around for this problem.
 The user can set the Bundle-RequireExecutionEnvironment (which can be set
using the Execution Environment section on the overview page of the Manifest
editor) to the java version they wish to compile against.  When they export,
PDE Build compile the source code the project based on the value of that
header.  One note, to make sure the change is in the workspace, the user must
click "update the classpath settings" after adding the BREE.  This will make
sure the code in the workspace is compile against the proper java version.

For example, if I have a project I want to compile against 1.4 but have a 1.5
JRE, I can set the BREE to "J2SE-1.4".  When I export, the source code will
automatically be compiled for 1.4.
