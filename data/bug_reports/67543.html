Description Jim des Rivieres 2004-06-16 17:19:51 EDT 
Build 3.0RC2
IE 6.0.2800.1106.xpsp2.030422-1633

If the Help browser gets stuck with an error dialog, some parts of the 
workbench window stop responding to events.

1. In the Platform Plug-in Developer Guide, navigate to Reference / API 
Reference.
2. Select link to package org.eclipse.core.runtime
3. Select link to class Platform
4. Select link to Object
This tickles bug 67539 which brings up an IE error dialog:
"An error has occurred in the script on this page."
5. Without dismissing the error dialog, switch to the workbench window.
6. Help > About Eclipse Platform

Observe: The about dialog does not appear. There's no feedback as to what is 
going on.

7. Dismiss the IE error dialog.

Observe: The about dialog now appears.

Expectation: Workbench windows should remain fully operational even when there 
is a modal error dialog showing in the Help Browser.

Added to Help section of 3.0.1 release notes:

"Modal interference between Help browser and workbench

If the Help browser gets stuck with an error dialog, some parts of the 
workbench window stop responding to events. Similiarly, Help topics that teach 
and describe usage of complex wizards and dialogs are unusable in the default 
(built-in) browser due to modal wizards and dialogs. To work around this 
problem, use the Help > Always use external browser preference setting to 
configure Help to open in a standalone browser. (bug 67543)"

Comment 16 Konrad Kolosowski 2004-08-30 12:33:13 EDT 
Using internal browser has mostly advantages, except that one (serious) 
modality issue.

External help browser, on the other hand:
is visually less attractive than embedded,
looks less integrated with the rest of Eclipse, (with embedded some users do 
not realize it is an HTML browser),
displays other toolbars, has extra text on the title bar,
does not allow control over modality of search scope pop up windows,
we do not have control over the trim of search scope window (after XP SP2 
installed),
our implementation of external browser adapters are not capable of closing the 
browser when workbench is closed,
external browser is either opened new for every help is launched (think 
clicking hep hits in search view), or is reused every time help is launched, 
but there is no browser instance dedicated to Eclipse help (means help can be 
opened in some browser that user used to view other sites),
is another running process on user's machine,
we have system browser adapter and custom browser adapter on Windows, and do 
not have implementation that will ensure that external browser is Internet 
Explorer (IE offers the best experience viewing help),
different default than in earlier releases - embedded browser (but run as 
external process) was the default on Windows in 2.1.,
external browser are more difficult to control, in terms of window position, 
size or focus - with our system browser adapter help might even open in very 
small pop-up window existing on user desktop,
generally controlling external browsers is difficult and time consuming,
using SWT browser exclusively for workbench help would simplify a life a lot.
