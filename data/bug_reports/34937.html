Description Kent Johnson 2003-03-13 13:23:29 EST 
RC2 build.

I started my test workspace from inside my normal dev workspace, which includes 
jdt.core as a source plugin. The rest of my dev workspace are imported binary 
plugins.

In my test workspace, I chose not to add binary plugins but instead use the 
Target Platform page to 'import' plugins.

After I 'added' jdt.core from the list, I could add it to the Java Search, but 
it didn't work since there was no jar file to add to the 'External Plug-in 
Libraries' project.

Are source plugins supposed to be available in the Target Platform plug-in list?

If so, how can you make them visible to the Java search?
Comment 1 Dejan Glozic 2003-03-13 15:19:56 EST 
When we launch the second instance and try to use PDE features there, we switch 
to 'Other Application' and reload to avoid picking up source plug-ins in the 
dev workspace.
Comment 2 Kent Johnson 2003-03-13 15:51:04 EST 
Sorry do you mean 'we' as in the PDE implementation? Or 'we' as in every user 
has to know to switch?

Cause I'm just a typical user who doesn't know how to use anything! ;)

If I'm not 'allowed' to see/use source plugins then they shouldn't be available 
in the list OR the switch to use my install should be done for me.
Comment 3 Dejan Glozic 2003-03-13 16:08:09 EST 
When I said 'we' I meant 'PDE team'. Since PDE is used for self-hosting, we 
normally want to test these features in the second instance. Sometimes, we 
launch another (third) instance from the second one.

The main problem we have is that some of the plug-ins in the second instance 
are really source plug-ins from the first (dev) that we load from the registry 
to improve performance. To avoid this problem, you need to switch to 'Other 
application' in the target platform and press 'Reload'. This will make PDE load 
external plug-ins instead of those in source from the first instance.

I know it sounds complex, and it is, but we don't see many people using PDE 
functions in the second 'test' instance.

Comment 6 Dejan Glozic 2003-03-14 16:12:31 EST 
All the problems in this one and the related defects are caused by PDE 
optimization.

In the past, we used to always load external plug-ins from a target platform 
specified in the Target Platform page. In cases where the instance to start was 
the same as the design instance (i.e. the default case), it was considered 
unnecessary to reload all the plug-ins when they are already loaded by the 
design-time instance (in the registry). So we decided to make a special case 
and initialize our model from the 'live' registry. It was faster than loading 
and resolving plug-ins on the file system.

The only issue with this is that some of the plug-ins in the second (test) 
instance may be in source (since they are coming from the workspace of the 
first - design instance). We are still treating them as external and expect to 
see JARs but we cannot because they are in the source form. 

The solution is to not allow the optimization when in the 'dev' mode (i.e. when 
BootLoader.inDevelopmentMode() returns true). We will force usage of 'other 
application' in this case and disabled 'this application' radio button).

Again, this does not affect most normal Eclipse users because they rarely use 
PDE in the launched instance. It mostly affects Eclipse developers while 
testing Eclipse function.
