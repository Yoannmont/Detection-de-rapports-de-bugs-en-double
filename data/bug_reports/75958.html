Description Jennifer Calder 2004-10-08 15:55:44 EDT 
Some users choose to customize help that is shipped to them. If they do this, 
their options for regaining a valid search index are:
1. Deleting the index that they currently have and restarting Eclipse; or
2. Creating their own index (a pre-built index) and using it; or
3. Incrementing the version of the plug-in that contains the updated help (ie. 
fool the help system into kicking off a build of a new index).

Customizing shipped help should be considered a perfect reasonable thing to 
want to do. And, if users are doing this, we should give them a simple way to 
regenerate search indices. Quite frankly, some customers are leery about 
deleting things in their product install - they really don't want to delete 
and search index and build another to replace the deleted one. We should be 
providing an easy mechanism for doing this.

Comment 6 Adam Archer 2007-06-12 11:47:05 EDT 
The only use case for this is if users manually edit help documents in existing plug-ins. This is too specific a use case to provide explicit UI controls which would probably just confuse less advanced users.

What if we made the "eclipse -clean" argument clear the existing index? Would that be a suitable solution?
Comment 7 Lee Anne Kowalski 2007-06-12 12:09:30 EDT 
Yes--from my experience, having the -clean argument clear the search indexes would help on this. We currently use -clean to update the ToC (for example, when a new doc plug-in is added), but have to manually clear out the existing search index to get the system to recognize the new doc plug-in also had a new pre-index with it.

For -clean to be the suitable solution for this particular situation about search, we'd also want it to load any of the pre-indexes, so that it avoids doing the "building the search index" again.

Jennifer, what do you think?

--Lee Anne
Comment 8 Chris Goldthorpe 2007-06-12 12:45:06 EDT 
The search index is stored in the configuration area and cleaning the configuration will cause the index to be rebuilt. I notice that eclipse has a command line argument -initialize which looks like it would work. When launching from Eclipse there is a checkbox in the configuration tab to clear the configuration area.
Comment 9 Lee Anne Kowalski 2007-09-11 11:52:43 EDT 
Hi Chris,

I am having trouble constructing a way that the -initialize argument would work to do what we think is needed. Whenever I add -initialize to a command from the command line to start the help system in standalone/infocenter, I cannot get anything to start up at all. Doing an -initialize does not seem to actually clear  the org.eclipse.help.base directory that might be already written in configuration from a previous iteration. (I admit that my construct of the order of command might be in error--I'm still testing that).

The 3.3 Help Doc here describes how -initialize is used wrt configuration area, but it seems to indicate that its effect on Help indexes might not do what we want:
http://help.eclipse.org/stable/index.jsp?topic=/org.eclipse.platform.doc.isv/reference/misc/multi_user_installs.html

The scenario is that we are in iterative development of the doc plug-ins, and iteratively starting and stopping the infocenter. When it's started once, and either a search is done first time, or has pre-built search indexes, the indexes get written to org.eclipse.help.base in configuration directory. Then, when we stop the system, and over-write a doc plug-in with a new one, and restart, Eclipse is still reading the search indexes in org.eclipse.help.base from the previous iteration.

Would it be do-able to implement for 3.4 what Adam proposed in comment #6? That the -clean will also clear out the org.eclipse.help.base directory that's written to the configuration area? 
I think that might be the most elegant solution to addressing the usage scenario.

Thanks!
--Lee Anne
Comment 10 Chris Goldthorpe 2007-09-11 12:18:04 EDT 
In response to Comment #9 - if you increase the plugin version (you can use the minor part of the version for this) the documents in that plugin will get reindexed. 

Making eclipse -clean clear the search index is not so straightforward to implement but we could add an argument specifically for cleaning the search index.

Thanks!
