Description Christof Marti 2004-07-05 10:46:50 EDT 
3.0

The AnnotationModel manipulates the document's positions, its annotation map and
its annotation model event unsynchronized. For example in
AnnotationModel#addAnnotation(..), the following calls are not synchronized:

if (!fAnnotations.containsKey(annotation)) {
	addPosition(fDocument, position);
	fAnnotations.put(annotation, position);
	synchronized (getLockObject()) {
		getAnnotationModelEvent().annotationAdded(annotation);
	}
...
Comment 1 Dani Megert 2004-07-06 06:08:13 EDT 
Each call itself is using a lock object but some of the calls should be locked
all together to guarantee atomic behavior.

Note: AnnotationModel.cleanup(boolean, boolean) has been changed in one of the
3.0 RCs: it can fire the model changed in a new thread.

Comment 13 Markus Keller 2010-07-08 14:43:24 EDT 
The problem is that
    ProjectionSummary.internalUpdateSummaries(IProgressMonitor) line: 164	
can change the lock object of the AnnotationMap while the lock is already being taken. If another thread then synchronizes on the new lock, it gets free access to the same HashMap.


Steps to reproduce in HEAD:

- Set a breakpoint in ProjectionSummary#internalUpdateSummaries(..) at first call to "sync.setLockObject(..)" (line 164)
- Set a breakpoint in BracketInserterTest#testMultiplePeerEntry() at
'type("((((");' (line 255)

- Debug BracketInserterTest#testMultiplePeerEntry() (only this single test method)

=> One of the suspended threads is in
    ProjectionSummary.internalUpdateSummaries(IProgressMonitor) line: 164	
- Open the local variable 'sync' and write down the id of the field
    fAnnotations= AnnotationMap

- Set breakpoint in AnnotationMap#keySetIterator() inside the synchronized block (line 82)
- Resume the main thread a few times until the 'this' variable has the id you've written down before. WARNING: The Debug view tries sometimes steals your selection and selects the ProjectionSummary thread (bug 229536). Be careful and resume only when you're sure the main thread has focus.

- In the main thread, step INTO the new ArrayList(..) constructor and into "c.toArray(elementData)", and then step over "Iterator it=iterator();" (so that the debug pointer ends up on the next line with the for statement).
- Disable all breakpoints
- Resume the ProjectionSummary thread

- In the main thread, you can now see that the modCount in the HashMap has changed. When you resume, you get this exception:
