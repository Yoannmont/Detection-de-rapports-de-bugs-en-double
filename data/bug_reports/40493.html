Comment 6 DJ Houghton 2004-11-02 08:56:56 EST 
This is what is happening:

- user chooses File -> Import -> Existing Projects operation
- user selects file location (e.g. c:\myProject)
- we create a project whose content is located at c:\myProject
- we read the .project and find the name of the project

When you do a CVS -> Check out as... it:
- creates a project with the new name
- sets the contents to be the contents from the server

This actually exposes a secret...although the project name is stored in the
.project file, we don't actually use it to determine the name of the project.
(whereas the File -> Import -> Existing.. action does)

Come to think of it, here are some steps which might actually cause problems?
- check out MyProject from CVS into workspace (default)
- check out MyProject from CVS into workspace as MyProject2
- change something in the project description of MyProject2 (for instance,
project references)
- now the .project file for MyProject2 is marked as dirty and the <name> element
in the project description says "MyProject2"

Comment 19 Gabriele Garuglieri 2004-11-03 03:12:07 EST 
Oh boy... i'll try to summarize this mess.
1- we need a consistent and predictable way to handle multiple versions/branches
of the SAME project in the SAME workspace.

2- right now the only way to handle this, is to checkout multiple times the same
project with different names.

3- but if you have multiple checked out directories, that contain different
versions of the same project and you need to import them all in a different
workspace, you cannot use "Import from existing project" to import them, because
the project name in .project file is the same for all and you are not allowed to
change the project name during import.
This is inconsistent with point 2 and the only way out of this is to manually
change the name in .project, in each of those imported after the first, to what
you want, import it so that eclipse picks that name and then restore the
.project from CVS.

4- but, again, handling the same project with different names, the way it is
done now, is dangerous, as reported by DJ in comment #6 and as i have verified
in the 3.0.1 version.
Suppose you have a project, lets call it pja, checked out twice, from the main
trunk and from a branch, as pja and pja-v1. If you change some project
properties of pja-v1, that causes the .project to be modified, it has the
unexpected side effect of changing in it the project name to the one with which
you checked it out, ie pja-v1, which is not what you want or expect.

So now the conclusions:
a- the above points uncover an architectural flaw of eclipse, that is not able
to deal in a consistent way with multiple version of the same project. May be a
marker, a flag, a tag or whatever you like could be added in the .project so
that when you import or checkout it multiple times, eclipse knows they are just
different incarnations of the very SAME project.

b- checkout as and Import from existing should behave the same regarding to
project name handling.

May be that point a should be addressed in a different bug, and if so i'll be
glad to open one.

Thanks,  Gabriele.
Comment 20 John Arthorne 2004-11-03 10:16:27 EST 
This bug has indeed become confused.  As Gabriele points out, there are two issues:

1) Import existing project doesn't allow you to pick a different name

2) If the project name does not match the name in .project, the .project file
will eventually change to the new name. We should instead tolerate a project
having a different name from the one in .project to better support this case.
(The name in .project is merely a hint that can be used when the project name is
not known.)

I will move this bug back to UI to handle 1). I will leave the helpwanted
keyword here because this is a very simple and localized fix that anyone in the
community is welcome to take a stab at. The import code is found in the class
WizardExternalProjectImportPage in the org.eclipse.ui.ide plugin. The fix would
mainly involve making the project name field writable and respect that name when
creating the subsequent project (the createExistingProject() method).

I have entered a second bug report for platform core to better handle the
situation where you create a project whose name is different from the one in the
.project file. This issue is independent of whether the project is created via
import or CVS checkout. The bug number for that issue is bug 77637.

Conclusion: This bug is ONLY about allowing the project name to be changed in
the "Import Existing Project" wizard. If anyone has related problems, please
enter a different bug report to reduce confusion.

(I am restoring the priority, owner, and title that the bug had before it was
moved to the resources component.)
