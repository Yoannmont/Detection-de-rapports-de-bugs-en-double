Comment 6 Jeffrey Liu 2005-06-22 04:21:22 EDT 
Hi David,

Thanks for investigating this. Yes, we agreed not to run perf tests as part of 
the build. And to the best of my knowledge, they are not. As far as I can tell, 
<jst-jsp-tests> is a functional test, not a perf test, so it is normal to have 
this test show up in the build log. As you and Keith have suggested, this test 
ran to completion on a display-less machine, and hangs the build when a display 
is available. So I think this is a bug in the testcase itself.

Re #1 - <antcall target="runtests-runAsPerfApp"/> and all other targets with 
the word "perf" in them are not invoked in the normal build process. These 
targets gets run only if you explicitly kick off a "performance build". This is 
done by running the org.eclipse.wtp.releng/performance/build.xml script. If you 
take a look at the declaration of the "runtest-runAsPerfApp" target:

<target name="runtests-runAsPerfApp" if="runAsPerfApp">

This target will run if the "runAsPerfApp" property is set. The normal build 
process should never set this property unless for some reason the build machine 
has this property to being with. I highly doubt this is the case because I just 
checked the test log for 0622 on the build machine, there's no indication that 
the perf tests are being run. Using your attachment as an example, it shows:

runtests-runAsPerfApp:

genHtml:
    [style] Transforming into /home/build-home/test-current-I/results/html

This means the "runtests-runAsPerfApp" target has been evaluated to see if it 
needs to be run or not. Since the condition failed (the "runAsPerfApp" property 
is not set), the content of the "runtests-runAsPerfApp" target is not being 
invoked. If the condition passed, then the log should show something like:

runtests-runAsPerfApp:

init:

suite:

core-test:

eclipse-test:

<messages from running the test>

genHtml:
    [style] Transforming into /home/build-home/test-current-I/results/html

Re #2 - The "runtests-runAsPerfApp" target does not check for locked workspaces 
(same goes for other test/perf targets). These targets basically delegate to 
the individual test plugins and the Eclipse test framework test script to run 
the actual JUnit tests. And it is those individual test plugins and Eclipse 
test script that're responsible for creating the workspace. So the targets in 
test.xml has no way to tell whether a workspace is being locked or not.

Since this appears to be a bug in the JSP test and you're the owner of the JSP 
component, I'll reassign it back to you. Keith, looks like you can reproduced 
the problem. My guess is that the JSP test is waiting for a user click, 
correct? What kind of click is it waiting for? Confirmation to close an editor? 
Confirmation to save?
