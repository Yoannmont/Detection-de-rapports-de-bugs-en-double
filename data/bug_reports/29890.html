Description Erin Harris 2003-01-21 10:24:59 EST 
The following are suggestions for source location improvements that would 
alleviate some of the difficulties we have run into with source location for 
super adapters (debug adapters that wrapper one or more sub debug adapters).  
The main idea behind these changes is to keep source location in the sub 
adapters rather than having the super adapter try to find source for all the 
different sub adapters.  The main problems we have run into are sharing source 
locations, handling the different dialogs that the source locators use and 
having only one source locator associated with a launch (even though you can 
have more than one debug target associated with a launch).


Suggested changes:

1.  To allow more than one source locator for a debug session we had two 
suggestions:
1A.  Associate a source locator with a debug target and add a getSourceLocator() 
method to IDebugElement. 
1B.  Have the launch associate a source locator with a debug target as in #1 but 
also have objects for which source can be displayed implement ISourceLocator.  
So a stack frame would implement ISourceLocator as well as any other object for 
which source can be displayed.  This would make it clear which objects have 
associated source. In this case getSourceElement would not need a stack frame 
parameter.

2.  Make the source location types common instead of specific to JDT (e.g. 
JavaProjectSourceLocation, ArchiveSourceLocation, DirectorySourceLocation).  
This way the source locations can be shared.  If for a particular source locator 
one of the source location types is not meaningful, it can ignore them.  This 
would allow super adapters to collect the source locations the user wants to 
search in the launch configuration and pass them to each of the source locators 
as they are created.  There should also be a common source location interface so 
that debug adapters can create their own types of source locations.  Debug 
adapters that have their own source location types would also have to provide a 
tab for the super adapter launch configuration so that users could specify these 
types of source locations at launch time.  This could be done through an 
extension point.

3. A common way to add source locations to a source locator.  This could be 
addSourceLocations(ISourceLocation[]) and setSourceLocations(ISourceLocation[]) 
methods on ISourceLocator.

4.  When debug adapters cannot find source, they should display a page with a 
button on it, much like when a class file is displayed with a button for 
attaching source.  When the button is pressed a source search dialog will be 
displayed which is specific to the debug adapter.  This means that the user will 
only see the source search dialog if they ask for it.  Current source search 
dialogs have a checkbox for never showing the dialog again but this has some 
problems:
- What if the user wants the dialog in some cases but not in others.
- For super adapters there may be several different dialogs (for each of the 
sub adapters), if the user checks the "don't show dialog again" button on one 
dialog does this mean they want to stop seeing all of the dialogs?

Displaying a file with a button in it would alleviate these problems.  This file 
could be a predefined type in Eclipse that the source locator could instantiate 
and return after initializing it with their dialog (to link the button to the 
correct dialog).  If the source locator wanted a more elaborate file then they 
could create their own.  Also a predefined type for a file that would just show 
"Source not found" would be useful so that debug adapters which cannot search 
for source could return this file.

5.  This relates to #4.  When a source search dialog is presented and this 
causes new source search locations to be created, a mechanism is needed to 
inform the other source locators in the super adapter that there are new source 
locations.  One solution would be for the debug adapter to change the launch 
configuration attributes, all sub debug adapters would be listeners on the 
launch configuration and would be notified of the change.  A common attribute on 
launch configurations for source locations would be needed for this.

6.  A mechanism to allow debug adapters to add menu items dynamically.  This 
would allow the super adpater to display the correct menu items for the current 
stack frame type.  Although the super adapter has created a wrapper stack frame 
for a particular sub-adapter stack frame, it can ask the sub adapter to populate 
the menu with its specific menu items (this goes back to inheriting features 
from sub-adapters).  Note that this applies to more than just source location 
but was brought up here because we were discussing a "Change Source File" menu 
item for stack frames.  One way to do this would be to have stack frames, 
threads, etc. implement an interface such as IDebugMenuContributor.
Comment 1 Darin Wright 2003-01-21 21:11:36 EST 
Initial comments:

Re: (1) This could be solved similar to the suggestion in 1B, but with the use 
of adapters. The debugger could ask the stack frame for its ISourceLocator 
adapter. If it has one, it is used - otherwise, the launch's source locator is 
used.

Re: (2) A JavaProjectSourceLocation cannot be made generic, as it has knowledge 
of org.eclipse.jdt.*. The debug core cannot pre-req JDT. Other locations could 
be made generic.

Re: (3) If a new source locator interface were defined, it could have such an 
API. However, 2.1 has no breaking API changes allowed - thus the current 
interface cannot be changed. We would have to define a new interface that could 
be used as an alternative to an ISourceLocator. This could also be retrieved 
via the adapter mechanism.

Re: (4,5,6) The debug model has no knowledge of the UI classes, and cannot pre-
req the UI plugin. Care must be taken to design an architecture such that the 
UI and model can be separated.
Comment 2 Erin Harris 2003-01-22 10:29:59 EST 
Re: (2) Could there be a generic ProjectSourceLocation?
Comment 3 Darin Wright 2003-01-22 12:22:28 EST 
There could be a generic project location, but the API as currently spec'd 
would have to change. The API for source locations is based on fully qualfied 
type names of java elements. We would need to change this to IPaths, and spec 
if it is relative, abosulte, or unknown, etc. The java source locations, as 
they stand today, are not generic enough for the platform.

Comment 14 Darin Wright 2003-12-05 15:03:30 EST 
I have migrated changes into debug.core:

* created a new package org.eclipse.debug.internal.core.sourcelookup where all 
the code resides, so we can keep it in one place
* renamed ISourceLocation* -> ISourceContainer*. After considering the terms 
location, archive and repository, Jim DesRivieres and I agreed that "container" 
is the most appropriate term
* changed extension points to "plural" form (i.e. if there can be more than one 
extension of a type, it should be plural - sourceContainerTypes and 
sourcePathComputers)
* re-wrote most of the javadoc
* removed #getId() from source containers (source container types have Ids, but 
source containers do not)
* changed source path computer to return a collection of source containers 
rather than a single container. I envision the abstract director will place the 
result in a default (composite container), rather than having each impl having 
to return a default composiste container
* changed the return type of #findSourceElements(...) from IStatus to Object[]. 
If there are dups, the abstract director will prompt for which source element 
to use via a status handler - but this should not be part of the source lookup 
API


Issue: how is a source path computer associated with a launch config? Should it 
be contributed for a launch config type? or should there be an ID on a launch 
config? of both. Thus, we can get the default computer for a config, or a 
specific one (in case it is overridden).
Comment 15 Darin Wright 2003-12-05 15:13:32 EST 
We should add an optional attribute to launch config types to specify a source 
path computer. This way, launch config types can share source path computers. 
As well, I will create an attribute that can be set on a launch config for 
specifying a specific source path computer.

Comment 24 Darin Wright 2004-01-21 15:23:14 EST 
I have integrated the changes into core. Although the design is the same, most 
of the code has been altered/refactored/java-doc'd.

* ISourceContainerType has been split into ISourceContainerType and 
ISourceContainerTypeDelegate. Extensions contribute a delegate. Similarly, 
ISourcePathComputer is split into a computer and delegate.
* IPersistableSourceLookupDirector is renamed to IPersistableSourceLocator2
* The lifecycle of the source lookup director is more explicit (although I 
still need to work on disposing of the source containers & possible 
caching/updating of source containers)
* Memento creation code is now in the source container types (instead of the 
containers)
* Added tests to verify memento creation/restoration
* Need more tests to verify source lookup
* Factory methods for creating containers are now Constructors on the 
associated container - as the containers will likely be public, but the types 
will not (they are internal implementations, instantiated by the debug plug-in)
* Removed #performApply code from director. This work should be in the 
tab/dialog.
* A director only listens for changes in its associated config (working copy 
or not). If connected to a tab, changes in the working copy will cause an 
update. If connected to a persisted config, changes in the persisted config 
will cause an update.
* Did not yet add implementation for archives
