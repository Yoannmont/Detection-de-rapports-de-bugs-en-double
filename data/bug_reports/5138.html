Comment 15 Dani Megert 2006-02-09 09:31:27 EST 
Editor implementors which want to provide the native word double-click drag can do the following:
- in your editor call getSourceViewer.getTextWidget().setDoubleClickEnabled(true)
- make sure not to call TextViewer.setTextDoubleClickStrategy(...)

TextViewer extenders can do it by extending createControl() with getTextWidget().setDoubleClickEnabled(true).
Comment 16 Marc Guenther 2006-02-13 14:47:16 EST 
Created attachment 34605 [details]
patch for triple/double click and drag

This is basically my old 3.0 patch from issue 14654,comment 12 with some more fixes and upgraded to work with Eclipse 3.1.1 and 3.1.2. Tripleclick still only works on MacOSX (carbon). I would have liked to include a patch for 3.2M4, but I couldn't figure out what CVS tag was used to build that version (R3_2_M4 doesn't exist, any hint?).

Here are a couple of thoughts on the various behaviours and bugs of triple/double click'n drag in Eclipse:

a) triple click behaviour: This is more of a feature request than a bug. Triple click isn't supported by the SWT. To work around this, you can either patch the SWT (like in my patch) or put some timing code into a higher level api (eg. jface.text, like in Chris Wiebe's patch bug 14654,comment 18, but then triple click doesn't work in normal text views). I don't like either way especially much. (bug 14654, fixed)

b) double click behaviour in Java Editor: Works quite well for words, but gets a bit strange when trying to double click on punctuation characters (see g). Double clicks on whitespace don't work at all (see f). (both fixed)

c) double click behaviour in normal text views (eg. Console): This is broken for text like "com.webobjects.foundation" or "/Applications/Eclipse.app". Double clicking anywhere in these strings selects only the part up to the first punctuation character. Curiously enough, the selection works if dragging a little bit. (unreported bug, not fixed)

d) double click and drag behaviour in Java Editor: This is completely broken. As already described here and in several other places, the selection jumps back to the position of the first click and it doesn't show word-wise selections during the drag. (bug 5138, fixed with this patch)

e) double click and drag behaviour in normal text views (eg. Console): works as it should. Shows selected words already during dragging.

f) double clicking on white space: Either doesn't do anything or selects the word next to it. (fixed for Java but not for normal text, not yet reported as seperate bug)

g) calculating the cursor position from mouse coordinates:
  The SWT method StyledText.getOffsetAtX() which converts mouse x/y coordinates into character position behaves different if you click on the first half of a character or the second half. This is not what you want for double clicks. For a double click, you don't care in which part of the character you clicked, cause you are selecting complete characters, and not positioning the cursor between characters.
  You can see this when trying to select either the variable or the braces in this line:
      set(xy);
  There are two places where nothing is selected at all, after the t and after the y.
  Even more annoying was clicking on a Tab character to select it (which wasn't possible before my patch). It would either select the tab or the word right next to it.
  (not yet reported as a seperate bug, but fixed with this patch)
 
h) shift click and drag after a double/triple click: One would expect the previous selection mode (word- or line-wise) to continue, but shift click and drag always expands the existing selection character-wise (not reported as seperate bug, not fixed)
--------------------------

About the patch: This now implements all the comments from Peter Eastman (bug 14654,comment 15) and Markus Keller (bug 14654,comment 20) except for the dragging, which is always shown character-wise instead of word-wise. Line selections also happen on >3 clicks as suggested by Chris.

The triple click stuff works only on MacOSX (carbon). The double click might
work on other OS' too, but I don't have access to other boxes.

The patch is against version R3_1_1 or R3_1_2. There is a INSTALL.txt inside the patch on how to install it.

Basically I made these changes:

1. for triple click:

org.eclipse.swt: 
  I introduced an 'int click_count' member to Event.java and MouseEvent.java.
  This is set from inside of sendMouseEvent() in Control.java and works
  with any number of mouse clicks

org.eclipse.jface.text:
  Added a new class ITextTripleClickStrategy, which adds one method
  tripleClicked()
  In TextViewer$TextDoubleClickStrategyConnector I added
  fClickCount and mouseDown(). fClickCount remembers the click count
  of the last mouseDown
  In mouseUp() if the fClickCount is 3, I send a tripleClicked() if the
  Strategy is a ITextTripleClickStrategy
  DefaultTextDoubleClickStrategy implements ITextTripleClickStrategy
  and adds tripleClicked() method

org.eclipse.jdt.ui:
  JavaDoubleClickSelector also implements ITextTripleClickStrategy
  and adds tripleClicked() method

2. for double click n drag support (this should work on all platforms)

org.eclipse.jface.text.DefaultTextDoubleClickStrategy
  and
org.eclipse.jdt.ui.JavaDoubleClickSelector
  
  Changed the doubleClicked() method to use the start AND end point
  of the mouse selection to calculate the resulting selection.

  JavaStringDoubleClickSelector calls super instead of selectWord()
  directly

3. for the conversion from xy coordinates to character position:

org.eclipse.swt.custom.StyledText:
  introduced an additional parameter to getOffsetAtX() which states if a click in the 
  right half should be considered as a click on this or the next character.
  Also had to add this parameter to a few additional methods to get it to the right place.
  
4. for clicking on whitespace in Java

org.eclipse.jdt.internal.ui.text.JavaPairMatcher.matchPairsAt()
org.eclipse.jdt.internal.ui.text.java.JavaDoubleClickSelector.AtJavaIdentifierDetector
  Can't remember exactly what I did here. Played around with the state machine till it did select
  whitespace the way I wanted it to. Still not very satisfactory.


Comments/Bugs:
- double clicking after the last character in a line should select the single
  newline character, but behaves like the last character was clicked.
- The double click in normal Views is still wrong
- Double clicking on braces now also selects the braces and not only the content (for me thats a feature, but for other people it might be a bug)
- Double clicking on the first character after a brace also selects the braces (a bug)
- To make this work on Win* and other OS', basically what I do here in
  SWT/carbon: org.eclipse.swt.widgets.Control.sendMouseEvent()
  (getting the click count and store it into the Event) has to be duplicated.
  Everything else should work from there.
Comment 17 Marc Guenther 2006-02-22 06:22:09 EST 
Created attachment 35138 [details]
patch for just the double click'n drag for 3.2M5

From my previous patch I extracted the part which handles double click and drag problem and updated it for 3.2M5. It doesn't require any api changes, especially no SWT changes. There are no new methods (except a private one in DefaultTextDoubleClickStrategy which could even be inlined)

It patches the following classes:
jdt.ui: JavaDoubleClickSelector, JavaStringDoubleClickSelector, JavadocDoubleClickStrategy
jface.text: DefaultTextDoubleClickStrategy, TextViewer

What it does is:
- change the behaviour of the doubleClicked methods to also consider the endpoint of the mouse selection for its calculation of the text selection.
- change the JavaDoubleClickSelector.AtJavaIdentifierDetector to enable selection of whitespace
- change TextViewer to call the doubleClicked() on mouseUp instead of mouseDown

This does NOT fix the problem about the click position being in the right half of a character
see comment 16: 
  g) calculating the cursor position from mouse coordinates
