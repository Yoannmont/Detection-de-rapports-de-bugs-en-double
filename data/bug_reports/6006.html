Description Rodrigo Peretti 2001-11-16 09:33:43 EST 
Original message on the newsgroup:

I'm using an Ant script to try to run JavaCC (sort of a lex/yacc
for Java).  JavaCC is itself written in Java, so you run it by
doing something like:

   java -classpath "blahblahblah" COM.sun.labs.javacc.Main Parser.jj

My Ant script runs fine with hard-coded paths for the path to the
JavaCC.zip file and the JVM, but that's not good, because different
machines have these things installed in different places.  After a
bit of poking around into the Eclipse's classpath mechanism, I
thought that it might serve to define the locations of global (not
project-specific) resources.

So, I created a "JAVACC" classpath variable to point at the folder
where JavaCC is installed.  I noticed while doing this that there
was classpath variable "ECLIPSE_HOME", which I thought could give
me access to java.exe.

I then wrote an Ant script like this:

<?xml version="1.0"?>
<project name="ParserProject" default="build.parser">
  <property name="JCC_BASECLASS" value="${JRE_LIB}\classes.zip"/>
  <property name="JAVA"          value="${ECLIPSE_HOME}\jre\bin\java.exe"/>

  <target name="build.parser">
    <echo message="Starting JavaCC..."/>
    <exec dir="." executable="${JAVA}">
      <arg line="-classpath ${JAVACC}\bin\lib\JavaCC.zip;${JCC_BASECLASS}"/>
      <arg value="COM.sun.labs.javacc.Main"/>
      <arg value="Parser.jj"/>
    </exec>
    <echo message="JavaCC complete."/>
  </target>
</project>

This doesn't work, on two counts:

 - ${ECLIPSE_HOME} doesn't seem to get expanded as I would think, so
   Ant can't find java.exe.
 - ${JAVACC} doesn't get expanded the way I thought it would, so that
   even if I use a hard-coded path rather ${ECLIPSE_HOME}, I get a
   "class not found" error for COM.sun.labs.javacc.Main.

Am I mistaken in thinking that Eclipse classpath variables can be used
for this purpose?  If so, what can I use to get what I'm after?

[If this were a makefile, I'd use environment variables, but I've gotten
the impression that there's no way to get at environment variables from
Java (and presumably, from within Ant).]

In case it matters, I'm running Eclipse 1.0 R0.137 on Windows NT, but
of course I'd like a portable solution (e.g. that works under Linux).

Comment 10 Dan Rubel 2002-07-17 16:07:34 EDT 
I suggest a new Ant task:

eclipse.getClasspathVariable
Places the current content for the specified classpath variable into a property

        <eclipse.getClasspathVariable
                classpathVariable="ECLIPSE_HOME" 
                property="aPropertyName"/>

        Places the current content
        of the classpath variable named "ECLIPSE_HOME"
        into the property named "aPropertyName"

Comment 14 Olivier Thomann 2003-02-04 12:32:34 EST 
I defined an ant task that set a property with the resolved contents of a
classpath variable.
Its specification could be:
<eclipse.classpathvariable property="propertyName" variable="variableName"/>

After this task is called (both attributes have to be set, otherwise a
BuildException is thrown), the property named "propertyName" is set with the
resolved value of the classpath variable named "variableName". This is a string
which represents the OS value of the corresponding variable.

I'd like to define in the same way a eclipse.resolvedClasspath task.
This task would take a project name as an attribute instead of a variable and
return the OS value of the resolved classpath of the corresponding project
inside a property like the first task.
So it would look like this:
<eclipse.classpathvariable property="propertyName" project="projectName"/>

Is this all right for everybody?
Comment 15 Olivier Thomann 2003-02-04 15:29:09 EST 
The second task would be:

<eclipse.resolveclasspath property="propertyName" projectName="projectName"/>

project is already used.
