Description Gunnar Wagenknecht 2004-02-11 09:27:43 EST 
There is a planned item to evolve the Eclipse user experience (bug 37997). 
Discussions inside this bug associate the topic with the request to gain an 
updated native look and feel. 

Eclipse should honour modern native environments and update it's look and feel 
to recent native looks. This includes but is not limited to Windows XP and GTK 
styles activated on the current platform.

Eclipse/SWT should detect the current setting and adopt to it automatically 
without any user interaction (eg. remove the requirement to add a 
java.exe.manifest manually for Windows XP style widgets .. it could be done 
through the launcher if necessary and no integrated solution is possible).

Also the UI should be reworked and updated to a more recent native look. 
Custom widgets (eg. view toolbars, title bars) should be avoided where 
possible.

This is essential for creating RCP applications that need to have a real and 
most recent native look.

Comment 4 Veronika Irvine 2004-02-11 11:02:02 EST 
One of the main reasons for not using the native TabFolder for stacked Views 
and the Editor workbook is that the native widget does not support closing 
tabs.  In addition, we are adding the ability to minimize, maximize and restore 
the folder and to provide a more advanced list for accessing a folder that is 
not visible.

As far as removing the requirement for a manifets on Windows XP, we have two 
choices.  We can provided a different swt dll for windows XP (versus other 
Windows platforms) (this is required because the dll needs to be compiled 
against a newer version of teh common controls dll to pick up the theme) or we 
can copy the manifest file into the users Java VM directory.  Neither one of 
these solutions is very appealling.  However, this issue is already being 
tracked under bug 45918 and this discussion should move there.
Comment 5 Chris McLaren 2004-02-11 12:11:10 EST 
If this bug is moving to SWT because of the manifest issue on XP, then we still need another PR in the UI 
domain. 

There is a larger philosophical issue here. We can choose as a product to either:
a) minimize the use of custom controls (or at least make them look close to their less-functional native 
equivalents), or
b) have license to go hog-wild on colors, curves, etc. everything and forget about looking native ever 
again.

The 'new look' that is floating around now (which will be in HEAD by the first I-build after M7) takes the 
latter stance: go hog-wild.

But there is not a specific PR which tracks (and allows voting for) the former stance, taken by those of 
us that think it might be a mistake to force a custom look on RCP products and walk all over SWT's 
value proposition.

Comment 8 Gunnar Wagenknecht 2004-02-11 14:38:40 EST 
It is not easy to mention specific details that don't fit into a native look. 
Lots of them are good improvements in reducing UI clutter. But the new look 
just doesn't feel "native".

Having said that, I'd like to recall discussions and decisions already made in 
the past. One of the major benefits of Eclipse was and still is its native 
look. IMHO this was one of the critical factors for the success of Eclipse. If 
we are now going to drop this principle we have to ensure that this change on 
the one hand will be accepted by the majority of users and on the other hand 
will not prevent other RCP applications from beeing successful.

Comment 17 Gunnar Wagenknecht 2004-02-18 01:38:38 EST 
I'd like to point to to an attachment of bug 52175 comment 14.
(https://bugs.eclipse.org/bugs/attachment.cgi?id=7979&action=view)

I never saw or worked with Visual Studio .NET before but this look is somewhat 
similar to the one I imagine for a true native Eclipse look. I know we can't 
just copy it and I never want Eclipse to do that. But we should investigate 
what makes this look native and what makes the new Eclipse look less native. 
Remember, a less native look ist IMHO a show stopper for RCP clients.

Views
-----
The view's title bar is rendered like a real window title bar with minimize, 
maximize and close button. A clean toolbar is below that title bar and a tab 
folder with small tabs. IMHO that is absolutly ok and what we already had with 
the "old" look. (Anybody knows what an "Auto Hide" view is?)
Rendering a view this way is more consistent when the view becomes an undocked 
view. When a view is undocked it should look like a real new window.

Menus
-----
They use the new Windows menus which where first introduced in Office XP I 
think. Should SWT Win32 switch to use this menu style on Windows platforms that 
support this style?

Editors
-------
Multiple, clean and small tabs by default on VS. Eclipse already has the drop-
down list which IMHO is better than the left/right scroll buttons.
