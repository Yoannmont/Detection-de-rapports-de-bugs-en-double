Comment 4 Todd Merriweather 2008-04-03 10:20:54 EDT 
I'm not positive that the findings below are THE cause, but at least they are A cause:

Steps to repro

1. download nmap from http://nmap.org/download.html and 
2. run it against both a machine running the AC with the following command nmap -A -T4 -v -p 10002,10005,10006 {hostname/IP address}.  
3. Shortly afterward the agent controller on the machines jumps to 100% CPU utilization.  

On the Linux agent controller, I saw the following stack trace:

java.lang.OutOfMemoryError
        at java.nio.HeapByteBuffer.<init>(HeapByteBuffer.java:54)
        at java.nio.ByteBuffer.allocate(ByteBuffer.java:327)
        at org.eclipse.hyades.internal.collection.framework.FileClientHandlerExtendedImpl.run(FileClientHandlerExtendedImpl.java:258)

Loosely connecting this data together leads me to believe that the issue seen on the customer's z/OS machine is not platform specific.

Another note, the problem doesn't occur if for whatever reason, the AC isn't listening on port 10005, or port 10005 isn't scanned.
Comment 5 Jonathan West 2008-04-03 10:51:36 EDT 
Hi Todd, port 10005 is the Hyades file transfer port; it uses a separate message protocol from the other ports, the implementation of which is Java-based through JNI. I suspect this message protocol (and therefore, port) is particularly vulnerable to data flooding, as evidenced by the OOM error from the JVM. I've confirmed this is a problem on 4.4.0.3 (and newer ACs).

Comment 7 Jonathan West 2008-04-03 12:40:54 EDT 
Hi Harm, thanks for clarifying. My note should have read that I've confirmed the problem on the new AC, by testing 4.4.0.3, and this indicts _at least_ all newer versions of the AC including 4.4.*. I've not confirmed this behaviour on the older AC code, nor have I confirmed that the Hyades fileserver is the cause of this z/OS specific behaviour. 

For both the old and new agent controllers, the file server code is Java code, which is contained in the org.eclipse.hyades.execution package. The code in this package is independent of either the new or old implementation of the agent controller. Both ACs connect to the Java Hyades package (specifically the classes org.eclipse.hyades.internal.execution.file.FileServer or org.eclipse.hyades.execution.security.SecureServer) through JNI. The new AC connects through the backwards compatibility layer, and the old AC connects as parts of its normal course of execution. As of 4.5, this layer is ALSO implemented natively with new code, but the old code is still used when security is enabled. With the possible exception of the non-secure 4.5 native implementation, I'd expect the same behaviour in both old and new ACs (unless someone broke something in the newer versions of the codebase ;) ).
