Investigate replacing the current custom asynchronous viewers with an asynchronous model that uses a standard JFace viewer. The model and viewer will use existing asynchronous content/label adapters.

Comment 2 Darin Wright 2006-09-07 14:51:13 EDT 
Created attachment 49644 [details]
patch

This work in progress use a new variant of the content adapter interfaces based on the JFace lazy tree content provider. The content adapter provides the number of children for each element, and then each child as required. It avoids the requirement for a model in the "middle".

I added IModelDelta.INSTALL and UNINSTALL to control when a model proxy is installed/removed from a viewer. We used to base this on visibility, but that was a problematic solution, as a non-visible item would not get its model proxy installed. Now the model can control this.

Comment 5 Darin Wright 2006-09-08 15:04:24 EDT 
Notes:

* The latest patch no longer maintains a cache of the underlying model: the content provider accesses the model as required via content adapters
* The model delta has been enhanced to contain INSTALL/UNINSTALL flags that control when model proxies get installed. This used to be based on visibility (mapped/unmapped from model/view). I think this is a better/more robust approach (allowing the model to tell us when to install/remove a model proxy)
* In order for selection/expansion to work, the model deltas have to contain index information (because a virtual viewer is based on indicies).
* Virtual viewers do not support viewer filters/sorters. It means the model has to deal with these issues

We still need support for asynch labels from JFace (see bug 156750). As well, they are experimenting with better support for columns in trees and a cell editor than can be dynamic based on the cell.

Comment 9 Pawel Piech 2006-09-13 19:19:09 EDT 
Supporting traditional viewer filters can certainly be helpful, I don't believe it would address most use cases for filters in debugger views.  In our debugger we currently have two different "filters", which we would like to port to a standard filtering framework.  
 1 - to show only the threads which are suspended 
 2 - to retrieve a limited number of stack frames
For 1, it may be practical to implement the filter at the viewer level, but for 2 the problem is a little more complicated.

Consider the situation where you want to debug a StackOverflowError, you create the exception breakpoint, but when it is hit, the stack trace is so long, that retrieving the full stack trace from the back and, and then creating the IStackFrame objects, pretty much hangs the machine.  The only way that a filter would be effective here, is if it would limit the query to the back end to retrieve a fixed number of frames (I actually don't know if java debugger supports this, but gdb definately does).  And the only way to implement such a filter would be to have the model implementation be aware of it and use it when retrieving children.

So I wonder what would you think of adding another object type: filter.  And similarly to source containers, adding the factories and UI editors for them.  Then simply passing the active list of filters to the IAsynchronousContentAdapter as one of the arguments, and letting the model deal with it.  I know that it's not nearly as convenient as clean and convenient as jface viewer filters, but it's probably a more scalable solution.  Also, platform could define a set of standard filters, and they could still be easily shared between various debuggers.
Comment 10 Pawel Piech 2006-09-13 19:50:41 EDT 
With respect to bug# 143711, the performance issues in debug views.  It is my guess that the biggest performance bottleneck is the creation and managing the many separate jobs that are used to retrieve labels and "isContainer" status of each node.  Even when the threads already exist and are in a thread pool, the fact that all these threads try to access the same set of objects, creates a lot of contention for a few synchronization objects, which is known to be expensive.

So perhaps the quickest and the easiest way to address this issue would be to introduce coalescing in the AsynchronousContentProvider and AsynchronousLabelProvider base classes.  These two could share a global (or per-debug target) coalescing queue which could then be serviced by a single job, or at least by a much smaller number of jobs.  

This is just an idea and something that could simplify the porting of the asynchronous viewers on top of the jface viewers, because the coalescing problem would be out of the picture.  Also, from my point of view, DSF doesn't really benefit much from coalescing at the view level, because the requests are already serialized into a single thread, and because there also needs to be coalescing performed at the back end communication level, which takes into account the length of buffer of commands to the back end and the types of commands being executed.
Comment 11 Darin Wright 2006-09-13 21:26:29 EDT 
Thanks for the insights Pawel.

I think one of the nice properties of the viewer filters (JFace), is that they can work on all debuggers without the model having to know about them. It avoids adding infrastructure for sharing filters. For example, imagine a thread filter based on thread names - it could be used by all debuggers and be provided by the platform.

Currently, when using the JFace viewer, the model deltas need to specify indicies for expand/select to work. This is really problematic when filters are on, because the model has to know about the filters and what elements are filtered etc. By having the viewer deal with filters, the model only needs to deal with raw model coordinates.

As well, to maintain backwards compatibility, viewer filters need to work.

(However, the JFace filters do potentially access the model in the UI thread).

For the case of the stack overflow we'll ask a thread how many frames it has, but only retrieve the visible ones. So, if a model can answer how many frames it has without having to retrieve them, it should perform OK.

We'd really like to use JFace viewers if possible, to avoid the duplication of a generic viewer framework in the debug platform. So, we are working with the UI team to see if we can make that happen.
Comment 12 Darin Wright 2006-09-13 21:28:09 EDT 
I agree that the existing (old?) asynch viewer implemenation suffers with too many jobs. The implementation provided here reduces the number of jobs significantly. I don't have the exact numbers, but I agree that reducing the number of jobs will improve performance.

Comment 20 Darin Wright 2006-09-25 15:48:52 EDT 
Created attachment 50861 [details]
patch

Updated patch with support added for Java thread groups and locks/monitors.

I'm still in the "investigation" phase, however the benefits to using JFace are:

* leveraging a shared code base in the platform, rather than rolling our own "virtual viewer support"
* The JFace virtual support is based on a content provider acting as a proxy to the model, where as the 3.2 debug implementation used a "model cache". A content provider avoids an intermediate cache of model elements, reducing footprint.
* The current patch performs coallescing of content requests in the viewer/UI code rather than delegating the responsibility to the model (although, that could be done in any implementation)
* The imeplementation is *more* virtual as it only retrieves visible children (instead of all children for an expanded node - the 3.2 viewers retrieved all children, but only rendered visible ones).

Issues:
* The current patch requires that model deltas provide index/child counts such that expansion & selection can be performed properly for non-visible elements (since the 3.2 model had a cache of all expanded children, it could determine indexes, but the content provider does not have a cache, so it requires indexes). I've experimented with this by re-implementing the java debugger's content adapters and model proxy to provide indexes. Since the content provider manages the filters, models only need to provide non-filtered child counts and indexes. It seems manageable, but is more work.

Comment 36 Pawel Piech 2006-11-21 11:59:06 EST 
Hi Darin,
I've been porting DSF to the new APIs for about a week now, and I have some questions and comments. 

First of all I really like the API changes and especially the IViewUpdate interfaces.  The addition of the child count and the indexes definately complicates the adapter implementation, but it's the tradeoff of course for the scalability.  As for the questions: 

1) It seems to me that the idea of coalescing could be extended to all the IViewUpdate APIs.  Currently, the IHasChildrenUpdate and IChildrenCountUpdate both operate on a set of elements, while ILabelUpdate and others operate on a single element.  Even if the implementation of the viewer does not implement coalescing of the other updates yet, I would argue that adding that capability to the API makes a lot of sense.  Further, if all the updates operate on a set of elements, the getParents()/getElements(), method could all be moved into the IViewUpdate interface itself, which may make it easier for adapter implementations to handle all the coalesced requests.

2) I'm not really sure if this is a bug or intended behavior, but I think there is some inconsistency as to when a TreePath is used to identify an element, vs. when the element alone is given.  The IElementContentProvider updates all supply the TreePath, while the label IElementLabelProvider, IElementEditor and others supply only the element (at the end of the path).  There also seems to be a bug at TreeModelContentProvider.java:228, where the element alone is used to identify an element to set the child count:
    protected void expand(IModelDelta delta) {
        ... 
        treeViewer.setChildCount(delta.getElement(), viewCount);
        ...
    }
But again, I'm not sure if this is intentional, and if it's not then I believe there may be other bugs like it.  
So the underlying question is: should the viewer allow for different data to be displayed for the same element in different tree-paths?  My assumption would be yes, but I'm not sure if that's clearly stated anywhere.

3) ModelDelta: ModelDelta object has a few methods to add children named addNode(), which internally instanciate other instances ModelDelta.  This pattern makes is pretty much impossible to sub-class ModelDelta, which is what I'm doing in DSF.  BTW, the fact that IModelDelta references ModelDelta (bug 159054) also makes it necessary that ModelDelta be sub-classed for additional functionality.
Comment 37 Darin Wright 2006-11-21 14:01:38 EST 
Thanks for the feedback.

(1) Re: coallescing requests:

Yes we should account for coallescing in all our APIs to avoid having to add this in the future. I'm thinking that we may want to change the API style to be of the form "update(IViewerUpdate[])" where each IViewerUpdate is an update for a single element. This may make the APIs simpler to use. As well, the update method would be specicialized in each type of "request processor" so the updates are type specific. For example, the IElementLabelProvider would have "update(ILabelUpdate[])".

(2) Yes, the intent is that content can be different for different instances of the same element in a tree. The left over bugs are due to the fact that the lazy content provdier APIs in JFace did not exist until recently (we had to use objects instead of tree paths until recently, for virtual content). So yes,  the bug you point out should be fixed.

(3) For model deltas - I'm was almost thinking that we could get rid of the interface and just have the class... I was assuming that everyone would just use the implementation that we provide, making the interface extra baggage. Why do you subclass the model deltas? I'm curious what the use case is.
Comment 38 Pawel Piech 2006-11-21 14:51:16 EST 
(1) +1 from me on the "update(IViewerUpdate[])" format

(2) Thanks for the clarification.  In this case I will file any bugs if I find them, which I guess would include the fact the ILabelUpdate and IElementMementoRequest do not return TreePath.

(3) I have two use cases:
 
One is that I need to be able to change the deltas after creating them, so I need setChildCount() and addFlags().  The latter was actually added with the new API.  

Second use case, which is actually no longer valid after refactoring for the new APIs, was that I was associating an additional object besides the element, which each delta.  I was later reading this additional object for further processing of the delta.

So actually, I could do away with my own IModelDelta implementation if the methods to modify the delta properties were there.  But another thing to keep in mind is that there could be an adapter implementation which might require a thread-safe version of the delta, which ModelDelta isn't. 

(4) One additional comment about ModelDelta: 
IModelDelta.CONTENT, and IModelDelta.STATE flags do not require that a full delta be present, but just the element.  I think this is a very important optimization, and I think it would be good to spell it out in the IModelDelta interface comments.
