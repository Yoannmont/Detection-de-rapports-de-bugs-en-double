Comment 14 Michael Valenta 2005-06-27 09:27:17 EDT

Not sure yet, but it *may* require that there be other modifications in the

Comment 16 David Corbin 2005-07-07 07:30:51 EDT 
At this point in the stack
CVSChangeSetCollector.addSingleRevision(SyncInfo,
RemoteLogOperation$LogEntryCache, ICVSRemoteResource) line: 294
CVSChangeSetCollector.addSyncInfoToCommentNode(SyncInfo,
RemoteLogOperation$LogEntryCache) line: 270
CVSChangeSetCollector.addLogEntries(SyncInfo[],
RemoteLogOperation$LogEntryCache, IProgressMonitor) line: 248
CVSChangeSetCollector.access$0(CVSChangeSetCollector, SyncInfo[],
RemoteLogOperation$LogEntryCache, IProgressMonitor) line: 243
CVSChangeSetCollector$2.run(IProgressMonitor) line: 234
SynchronizeModelUpdateHandler$8.run(IProgressMonitor) line: 674

it appears to loop over the log entries for the specific resource, looking for
one that "isDeleted".  It shows four logentries in that list, when in fact there
are 6, Missing from it are one change and the last one, the delete.  

Further tracing shows that the LogEntryCache does not contain the last (delete)
log entry for another deleted remote resource.

I've backtraced this cache to LogEntryCacheUpdateHandler, which itself contains
a SoftReference cache.  What's not clear to me, is when this cache is
"discarded" as out-of-date (other than basic SoftReference causes), nor why the
log entries are missing from it.  More debugging tips appreciated.
Comment 17 Michael Valenta 2005-07-07 10:58:56 EDT 
It is difficult to debug because of the concurrency. The request to fetch log 
entries comes through the CVSChangeSetCollector#add(SyncInfo[]) method. If you 
debug there, you can check to make sure the comment is at least being 
requested. From there, you can trace it into the fetch job.
Comment 18 David Corbin 2005-07-11 09:08:50 EDT 
Well, I just spent an hour going in circles, and I'm quite dizzy.  As near as I
can tell, the code is trying to update the cache, and it's looking at the data
in the cache to update itself.

In LogCacheEntryUpdateHandler#getRemoteResource(SyncInfo), there is a call to
getRemoteResource(info.getLocal()).  Following that call chain, I cannot find
anywhere it's actuallying going back to the CVS repository.
Comment 19 Michael Valenta 2005-07-11 09:50:49 EDT 
The update handler will only go to the server if it doesn't have the entry 
cahced already. If you look in the getRemotesToFetch method, you will see that 
if removes any entries for which the comment was already fetched.

From comment 16, it looks like the SyncInfo got dropped somewhere along the 
line. When LogEntryCacheUpdateHandler#fetch is called, are all the expected 
SyncInfo there? If you trace through the 
LogEntryCacheUpdateHandler#processQueuedFetches method, are all the SyncInfo 
included in the set trhat is used to fetchLogEntries and 
fireFetchedNotification? It is really the later call that is the most 
important since it is what calls back to the UI to display the SyncInfo. Even 
if a log entry was not fetched, the SyncInfo should appear in an unassigned 
group.
Comment 20 David Corbin 2005-07-14 07:04:57 EDT 
If the SyncInfos passed to LogEntryCacheUpdateHandler#fetch() are supposed to
have the latest remote revision in them, they do not.  They have the "local"
revision.

Then, when the fetch gets into RemoteLogOperation#execute(ICVSRepositoryLocation
location, ICVSRemoteResource[] remoteResources, IProgressMonitor monitor), it
decides it has that revision in the cache and does not go out and try to get the
more recent information.
Comment 21 Michael Valenta 2005-07-14 09:01:22 EDT 
Ahhhhh, that sounds promising. Deletions are stored differently that newer 
revisions. Perhaps this leads to a situation where, if the cache holds the 
entry for the local revision, it will not fetch the entry for the deletion. If 
this is the case, I should be able to reproduce this. I give it a try and let 
you know what the results are.
