Description Phillip Avery 2005-02-23 17:07:30 EST 
Validation framework should utilize content type instead of the object filter
class for IFile.

eg.
	  <filter
		objectClass="org.eclipse.core.resources.IFile"
		nameFilter="*.jspf">
	  </filter>

Since content type is more robust, it handles (multiple) file extensions among
other things.

Also on a side note, it seems that the "caseSensitive" attribute default value
should be "false", since it doesn't seem likely that someone will NOT want to
validate a file if it has an uppcase extension.

Comment 11 Richard Cardone 2006-12-14 11:57:48 EST 
Since the request to allow the XML validator to operate on files with arbitrary extensions is nearly two years old, maybe an interim solution that quickly delivers the needed function should be considered.  The content type approach might be the most comprehensive solution, but a single public method that dynamically adds new file filters to the list of those that the validator services would provided the needed function without too much work.  The method signature could look at follows:

 public void addFilenameFilter(boolean caseSensitive, String nameFilter)

Here's an example use of the method that would direct the validator to process files that have the ".esn" extension:  

 addFilenameFilter(false, "*.esn");

Most of the above capability already exists in the org.eclipse.wst.validation.validator extension point that is defined in the plugin.xml file of the org.eclipse.wst.xml.ui plugin.  

The above method could be implemented as a static method that simply maintains a static list of user-specified name filters.  Another approach is to integrate the user-specified filter names with those loaded from the plugin.xml file.  In either case, it should not take more than a couple of hours of programming to implement something that we really need.
Comment 12 Lawrence Mandel 2006-12-14 15:19:41 EST 
Richard, until something is done on this bug there is already a way (albeit a clumsy one) that you can specify another file extension for the XML validator to validate. You can declare a new validator in your own plug-in that uses the XML validator class. (You may need to specify a dependency on org.eclipse.wst.xml.core.) For example,

<extension
 id="esnValidator"
 name="%ESN_Validator.name"
 point="org.eclipse.wst.validation.validator">
  <validator>
    <filter
      objectClass="org.eclipse.core.resources.IFile"
      caseSensitive="false"
      nameFilter="*.esn">
    </filter>		
    <helper class="org.eclipse.wst.xml.core.internal.validation.core.Helper"/>	
    <run
      async="true"
      class="org.eclipse.wst.xml.core.internal.validation.eclipse.Validator"
      enabled="true"
      fullBuild="true"
      incremental="true"/>
  </validator>
</extension>
