Description David Pickens 2003-02-27 11:58:41 EST 
While using conditional breakpoints, I frequently get the
following popup message when the breakpoint is hit:

	Conditional breakpoint runtime exception
	
	Reason: com.sun.jdi.ObjectCollectedException occurred retrieving reference type

I have confirmed, and reconfirmed, that the conditional expression is valid.
I have received the error in several contexts and in all cases the expression is
valid. And I have confirmed that the expression does not throw a runtime
exception.

I cannot reproduce this problem in a small testcase, but hopefully there's
enough information for you to track down the cause.

The problem started appearing when I updated to 2.1 RC1.
Comment 1 Jared Burns 2003-03-03 18:28:15 EST 
Are you sure that your expression isn't referencing an object that's been
garbage collected?
Comment 2 David Pickens 2003-03-03 19:12:13 EST 
The problem occurs in multiple cases, and the breakpoint expressions
are referencing local varables or arguments.

One breakpoint expression in particular looked like this:

	"string".equals(ref.getName())
	
where "ref" is an argument to the method in which the breakpoint is set. 
So, it could not be garbage collected.

Comment 11 Darin Wright 2003-12-09 09:18:01 EST 
I have seen similar problems when doing evaluations. I'm not sure what we can 
do about it. The problem is that the VM does a GC and collects objects that are 
being used by the debugger for evaluation purposes (but otherwise have no 
references).
Comment 12 Darin Wright 2004-03-26 11:53:04 EST 
I have a proposed implementation, but we cannot currently use it, as it blows 
some VMs out of the water.

The proposal is to disable garbage collection of temporary objects that are 
created during an evaluation. As an object is pushed onto our evaluation 
stack, we disable the garbage collection of that object. When an evaluation 
completes, all "disabled" objects are re-enabled. I experimented with this, 
but I found VMs that crash when the JDI "disableCollection" feature is used. 
For example:

JVMCI161: FATAL ERROR in native method: Illegal ref passed to 
DeleteWeakGlobalRef
JVMDG217: Dump Handler is Processing a Signal - Please Wait.
JVMDG303: JVM Requesting Java core file
JVMDG304: Java core file written to c:\eclipse3.0-darin\runtime-
workspace\Junk\javacore.20040326.104616.2128.txt
JVMDG215: Dump Handler has Processed Error Signal 22.

I will attach my suggested implementation for future reference, created 
against the 3.0 M8 candidate.

Note, I found the Sun VM handles the GC requests, but that IBM VMs do not. 
Marking as later.
