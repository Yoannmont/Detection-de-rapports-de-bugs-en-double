Description Mark Melvin 2006-07-21 10:42:12 EDT 
We are shipping a product (on Windows only right now) that installs the Eclipse platform, branded as our product and augmented with our plugins.  I have tweaked our config.ini so that after our installer installs Eclipse (with our plugins) it then configures the eclipse/configuration folder as read-only so that the configuration area will be cascaded, and essentially per-user on the machine.  This avoids writing cruft into the eclipse/plugins and eclipse/configuration folders of our installed product, provides per-user configuration areas on the machine, and is generally better.  My eclipse/configuration folder only contains config.ini, which contains the following (amongst the other, required stuff):

.
.
# The default workspace location
osgi.instance.area.default=@user.home/workspace

# The user-writable configuration location
osgi.configuration.area=@user.home/.eclipse/configuration

# Ensure the shared, platform configuration location is read only
osgi.sharedConfiguration.area.readOnly=true

# Make the configuration area cascaded
osgi.configuration.area.cascaded=true
.
.

This has been working well for us for the last couple releases, however I just noticed something that is rather annoying.  As part of our build process we pre-index our documentation plugins (and we have quite a few of them) to avoid the big huge index operation that occurs when you perform an initial search.  These indexes are stored with our installed plugins in an "index" subfolder (we ship our docs in single-jar plugins as well).  Well, it appears now that we have moved to this per-user configuration area, these indexes are completely ignored.  The first search that is performed takes forever, and it is because it is indexing the documentation.  I can see that a new documentation index is created in <userhome>\.eclipse\configuration\org.eclipse.help.base\index, and from the length of time it takes to create it, it is obvious that it is re-indexing all of our documentation plugins from scratch.

Can the Help indexer not re-use the prebuilt indexes somehow?  This initial wait (it can be many minutes) is unacceptable, and the whole reason pre-built indexes was provided, correct?
Comment 1 Mark Melvin 2006-10-20 12:59:42 EDT 
Is there any workaround for this?  It is killing our out-of-the-box experience - especially for dynamic help.  Pressing F1 for "quick help" the user has to wait for 5 minutes while the Help system is indexed - even though it was already pre-indexed as part of our build process.

Can I do some sort of trick with our pre-built indexes to avoid this?
Comment 2 Curtis d'Entremont 2006-10-23 13:25:43 EDT 
I've tried reproducing this with the eclipse SDK 3.2.1, but without success. It seems to grab the plugin indexes properly for me.

One thing I should ask - is your documentation html or xhtml? Do you have a lot of content that is searched via lucene search participants?

Currently there's a limitation with per-plugin indexes that it doesn't handle lucene search participants, so even in eclipse SDK it doesn't preindex the few xhtml files as well as welcome content or cheat sheets. These are small enough for our case and do not cause a major performance problem.

If this is your problem, the workaround is to use the per-product index instead. This one indexes the same way as when doing a search for real - by launching it headless, indexing, then shutting down. Preindexing per plug-in currently works by indexing from outside eclipse, similar to the way the PDE operates on a workspace that is not running. This is difficult because we would need to actually load bundles and classes from the workspace in order to index. We may need to change the way this is done in the future to be more like per-product indexes.

If this is not the problem, I am at a loss to explain what is going on, and any additional info you can provide would help, like anything you are doing out of the ordinary, etc.

I can open these JAR files and see that inside them I have a subfolder called "index" containing the following files (at least the one plugin I looked at):

_aa.cfs
deletable
indexed_contributions
indexed_dependencies
indexed_docs
segments

All of our help content is in subfolders of a folder called "html" inside the single-JAR'd plugin.

Now, if I fire up my installed product, after running a help search and waiting for the index to complete, I can see in my user configuration area the big help index under:

There is a group of the same files there, and the main index (called _4lt.cfs) is 5.2MB.  Opening it in a hex editor I can see it is full of stuff from all of my individual plugin indexes.  So it looks like Eclipse has gone and re-indexed all of my plugins into one huge index.

I thought it would use the pre-built indexes in my plugins.  Or if it was going to go ahead and build a new index, it could at least glomm them all together quickly - but from the length of time it takes, it is obvious to me that it is doing a full index of all of the plugin again.

The only other bit of useful information may be that my documentation plugins are also defined as full Java projects and have a plugin activator.  Thus, I can see all of them in:

However, the inflated files in these directories only contain a single jar file containing the .class file for the plugin activator (and a manifest file).  I assume this is normal...

Is there any other info I can provide?
Comment 4 Curtis d'Entremont 2006-10-23 14:20:46 EDT 
Your setup seems fine to me.. it should not be reindexing everything.

About the search participants, what I meant was that it just skips content that would be searched via participants in the case of preindexing, but will still index all the html content.

The master index gets created by, as you suggested, glomming all the plug-in indexes together into one big one. This should not take very long. For eclipse SDK it takes, I would guess less than one second. This happens before the progress monitor reaches 1%, so it sticks at 0% for a sec then as it finds docs to index it starts climbing up toward 100%. So if your progress monitor is climbing very slowly this suggests it is indeed reindexing every document.

One thing I would have you try is running without read-only or cascaded configuration areas to confirm that this is indeed what is surfacing the problem. Try running eclipse SDK - eclipse docs + your doc plugins. If it picks up your indexes and searches quickly, this will confirm that your indexes are properly contributed. If not, we can change the focus and find out why it's not picking up your indexes.
