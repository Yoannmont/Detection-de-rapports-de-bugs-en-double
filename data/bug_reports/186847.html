Description Tim Dossett 2007-05-14 13:50:19 EDT 
Build ID: 200705021527

While IDE-based builds are great for development, eventually all larger projects need to perform automated, command-line builds. Since the new CDT 4.0 internal builder creates no makefile artifacts, automated, command line builds cannot be performed. 

Is there a headless way to perform CDT builds, as there is for ant via org.eclipse.ant.core.antRunner?
Comment 1 Mikhail Sennikovsky 2007-07-05 06:12:00 EDT 
I can see several ways of enabling the command line builds for the Internal Builder-based projects:

1. run/start eclipse from command line and tell it to build the specified project and then shut down. A support for such command line options should be implemented in eclipse/CDT. This approach is the most generic one since it would support building any kind of projects, not CDT only.

2. even though the Internal Builder is used for the project, it could be possible to export the buildfile to be used for building the project from the command line. This could be a makefile of any format, e.g. a gnu make, ant, etc. For the gnu makefile generation the current GnuMakefileGenerator could be used. As for the ant cripts, the ant script generator could be implemented using the Build Model framework available in the build system.

Mikhail
Comment 2 Doug Schaefer 2007-07-05 10:30:47 EDT 
I would have thought that option 1 would already be there. If not, it should be an easy task to create an application that invokes the build on the workspace.
Comment 3 Josu Onandia 2007-09-11 09:48:55 EDT 
I wonder how do the Eclipse+CDT users manage to have automated builds, without this feature.

Ideally, I'd like a command like this:

     $ eclipse build myproject mytarget

(could have another name, cdtbuild, or whatever).

Other IDEs open the GUI, execute the build, and close it automatically, but I find it preferable if the command executes without GUI, only with the console output. 

The command should exit with a value of 0, if the build finish with success.

With such command it would be easy to integrate Eclipse in our centraliced build and packaging system.
Comment 4 Doug Schaefer 2007-09-11 11:18:31 EDT 
This should be easy to implement and I'm disappointed no one has given this a try.

Eclipse allows you to specify the id of an org.eclipse.core.runtime.application extension on the commandline using -application. The application is simply an IApplication class that has a start method that will let you do pretty much anything. One of the things it could do is automate everything you need to do a build.

The big question is whether the APIs necessary to do things like check out source area available to you. Running the build, though, should be as easy as invoking the build method on the IWorkspace.

Comment 10 Chris Recoskie 2007-11-02 10:36:11 EDT 
(In reply to comment #9)
> Would the build not ask for these variables?
> 

The answer is "it depends."  Unless something has changed recently, the builder by default doesn't try to use any of these variables.  However, the ISV can use variables in their toolchain definition, and the user can use variables in the values of their build options etc.

So, I think you can perhaps get it to work, but you'd have to get around the PDOM issue first, which you probably want to do for performance reasons anyway.  If you're just doing an automated build, you don't necessarily need to build the index.

However, this brings up an interesting problem.  I had been thinking in terms of the generated makefile builder, but the reporter is using the internal builder.  The internal builder uses the index for dependency calculation... so if you prevent the PDOM manager from starting up, you won't be able to respond to dependency changes and do incremental builds based on them.  Maybe this doesn't matter... automated builds are usually of the "clean all" variety.

Comment 12 Keith W. Campbell 2008-02-04 11:27:37 EST 
My investigation turned up some useful ant tasks defined in
org.eclipse.core.resources. For example, if you create a
'hello world' project called 'hello', the build script below
can be executed with this command:

eclipse \
  -data ${workspace} \
  -application org.eclipse.ant.core.antRunner \
  -buildfile build.xml

------> build.xml <------
<project name="sample" default="build" basedir=".">
  <target name="build">
    <cdt.build project="hello"/>
  </target>
  <macrodef name="cdt.build">
    <attribute name="project"/>
    <sequential>
      <echo>Building @{project} . . .</echo>
      <eclipse.incrementalBuild
          kind    = "incremental"
          project = "@{project}"
          builder = "org.eclipse.cdt.managedbuilder.core.genmakebuilder"/>
      <echo>Refreshing @{project} . . .</echo>
      <eclipse.refreshLocal resource="@{project}" depth="infinite"/>
    </sequential>
  </macrodef>
</project>
------> build.xml <------

This almost works. When I tried it, the project builds,
but there are errors in the log about not being able to
start org.eclipse.debug.ui because there's no workbench.
That plugin is consulted to resolve references like
  ${workspace_loc:/hello/Debug}

The feedback in bug 208484 suggests that the dynamic
variables contributed by org.eclipse.debug.ui are
going to continue to require the workbench. Perhaps
our best recourse is for CDT to introduce its own dynamic
variables that should be used instead of any of the following.
  container_loc
  container_name
  container_path
  file_prompt
  folder_prompt
  project_loc
  project_name
  project_path
  resource_loc
  resource_name
  resource_path
  selected_text
  string_prompt
  workspace_loc
I'll test this idea to see if it allows the command-line
build described above to work cleanly.

Comment 24 James Blackburn 2009-05-28 07:36:37 EDT 
Created attachment 137461 [details]
Patch 1 - Headless builder for managedbuilder.core

This patch adds a headless builder target to CDT's managedbuilder.core plugin.

The application target should be run with:

eclipse -nosplash -application org.eclipse.cdt.managedbuilder.core.headlessbuild
 {program arguments}

It takes the following arguments:
   -import     {[uri:/]/path/to/project}
   -build      {project_name | all}
   -cleanBuild {projec_name | all}

This allows you to import and (clean)Build a project in a temporary workspace.

It does a few things by default:
 - It sends build stdout to the console
 - It disables auto-build
 - It builds all CDT configurations

Do let me know of any issues.

NB users will still need to ensure they don't use UI variables -- in 3.4.2 / 3.5 the workspace_loc variable was moved from the UI, the other variables don't make sense in a non-UI context.

Yes you can:

  - import an existing project into an existing workspace
  - create a fresh workspace;  import projects; & build.

You can use all options on the command line at the same time if you want so you can do:
eclipse -application ... -data /tmp/workspacedirectory -import /path/to/some/project -cleanBuild project -build all

If you try to import a project and the workspace already contains a project with the same name, the headless builder exits with an error.
