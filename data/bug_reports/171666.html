Comment 4 Michael Valenta 2007-02-01 13:53:03 EST 
Yhe patch is a good start. Here are some comments:

The CVSStatus already has a payload of the root of the command. The root is a CVS resource handle that provides access to the repository location and the underlying IResource if the command was run on the workspace (i.e. CVS commands can be run on a in-memory sandbox as well. We use this to populate the Repositories view, for instance). Whenever possible, the commandRoot should be included in the status (although I suspect there are many places where we have access to it but didn't include it). I see you did that in a few places. However, we need to add a getCommandRoot() to CVSStatus to give access to the field. The command root is a field on the session object (session.getLocalRoot()).

In the cases where the commandRoot is not available, then we could revert to including the location.

Now to answer your questions:

1) Yes, you can access the location from a resource. The code would looks something like this (with null checks omitted):

  ICVSFolder folder = CVSWorkspaceRoot.getCVSFolderFor(resource.getProject());
  FolderSyncInfo info = folder.getFolderSyncInfo();
  String repoString = info.getRoot();
  ICVSRepositoryLocation location = KnownRepositories.getInstance().getRepository(repoString);

I think the best approach would be to include the resource as payload of the status and have a single smart method that can extract the location from eitehr the commandRoot or the resource (if the location field of the status has not been set).

2) This is not API so if there is a method that is no longer used, just remove it.

3) The location is an instance variable of PServerSSH2ServerConnection so you do have access. I wouldn't worry about Client. It exist for backwards compatibility but is never used.

4) Extracting the location from the resource should work. Wouldn't we have similar problems when considering the Synchronize view.

One thing we do have is a class called TeamStatus which is API and can be subclasses. We should probably make CVSStatus subclass that so we can at least get the affected resource via API. What are you using the location for in the status handling? One thing we could consider is adding API to TeamStatus that could be overridden by CVSStatus in some way. Perferably, it would return a String or some other generic description.

Comment 8 Michael Valenta 2007-02-05 09:38:55 EST 
I think it is a good idea if you know for sure that you can make use of the information in the status handler. However, I feel I must warn you that error handling in CVS is one of the trickiest things because errors returned from the server are just simple text messages that we interpret. Any time you do this, you run the risk of misinterpreting the error, which, in the end turns out to be worse than no interpretation at all. A better approach may be to identify the common error cases and improving those specific cases instead of trying to do a blanket improvement.

Having said that, increasing the information that is included in exceptions is fairly safe from a code contribution standpoint, so we are happy to accept any improvements in this area that you feel you need to prove that the status handler work in Eclipse is adequate.
