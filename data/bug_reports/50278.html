Description Christophe Cornu 2004-01-20 11:49:36 EST 
From newsgroup:

Hi,

SWT seems much slower than native Windows apps when drawing its windows, and
I'd like to know why this is and if it can be fixed somehow.

I'm on Windows XP.

I think an example best illustrates this, and I've been trying to come up with
one. The best I can do so far is this:

If you take snippet 123 (embed a web browser in a window) and point it to a
huge web page:

http://dev.eclipse.org/viewcvs/index.cgi/~checkout~/platform-swt-home/snippits
/snippet123.html

I don't know of many huge web pages. It is better with graphics and text, but
here is one with just text:

http://onlinebooks.library.upenn.edu/webbin/book/makeauthorpage

Now, minimize/restore/min/res the window (once the page is loaded).

You will notice that it is very slow.  I am trying to make the speed
difference as noticable as possible by using huge web pages, but it seems true
on a lesser scale when just using normal widgets or normal web pages.

If you quickly minimize/restore/min/res (click the taskbar entry) this web
page in IE, it is instantaneous.  But the SWT hosted one is gruesomely slow.
This happens when using a StackLayout too (for a tabbed interface), and
rotating between topControls.  SWT just seems slower at redrawing.

Furthermore, if you leave the SWT hosted webbrowser open and maximize it, and
place some other native windows app on top of it, and minimize/restore the
windows app, it is slow to redraw because the SWT app underneath it is slow at
redrawing itself (you can visually see the difference, the SWT app slowly
repaints the area).

Is this some innate problem with SWT, or is it a bug, or why isn't it acting
like all the other windows apps at the same speed?
Comment 1 Christophe Cornu 2004-01-20 12:03:05 EST 
- I've tried to reproduce with the snippet 123. Minimized to maxmized 
transition was feeling as fast as the real IE. For some reason, the transition 
maxmizied to minimized was taking about 1 full second, with CPU at 100% - 
unlike IE. 
- I also tried the BrowserExample (org.eclipse.swt.examples project) that is 
also based on IE. Minimized to maximized transition was fast for the native 
widgets part, the IE portion seemed to take about half a second to redraw 
(where it was instantaneous on the snippet 123). The transition maximized to 
minimized was instantatenous.

One of the key differences between the 2 examples is the layout and the way 
the widgets are created/resized/ordered. Just a guess at this point.

- I did not notice any slowness with the ControlExample 
(org.eclipse.swt.examples project). I did not notice problems in redraw when a 
native app on top of an SWT app is minimized.

The behaviour reported in this PR could be due to a low memory situation - if 
the SWT app consumes a lot of memory, the redraw events will be slow as the 
system swaps more and more inside virtual memory. From memory, I've seen this 
in the past when CPU gets max'ed out, RAM is limited and the app does too much 
work inside paint events.

We want to keep an eye on this. We need to find more test cases 
(preferably 'simple' SWT tests).
