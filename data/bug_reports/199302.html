Comment 28 John-John Tedro 2010-10-01 13:31:22 EDT 
Hey, I've put up artefacts for this on my own personal maven repo, so i though I should share my experience with swt+maven.
This is basically my suggestion for how I feel that you should do it:

http://toolchain.eu/maven2/

Each library is divided into a set of artifacts which reflects platform/api that it is based on (groupId is org.eclipse.swt).

swt-carbon-macosx/
swt-cocoa-macosx/
swt-gtk-linux/
swt-windows/

Each of these are also divided into architectures by their "classifier" property, as seen by an example of the swt-cocoa-macosx setup:

swt-cocoa-macosx-3.6-x86.jar
swt-cocoa-macosx-3.6-x86.jar.md5
swt-cocoa-macosx-3.6-x86.jar.sha1
swt-cocoa-macosx-3.6-x86_64.jar
swt-cocoa-macosx-3.6-x86_64.jar.md5
swt-cocoa-macosx-3.6-x86_64.jar.sha1
swt-cocoa-macosx-3.6.pom
swt-cocoa-macosx-3.6.pom.md5
swt-cocoa-macosx-3.6.pom.sha1

These are just straight up deployments of the jars provided in the packages found at (swt.jar):
http://download.eclipse.org/eclipse/downloads/drops/R-3.6-201006080911/

The jars were deployed using the following maven command (showcasing the swt-carbon-macosx artifact):

mvn install:install-file \
  -Dfile=swt.jar \
  -DgroupId=org.eclipse.swt \
  -DartifactId=swt-carbon-macosx \
  -Dversion=3.6 \
  -Dclassifier=x86 \
  -Dpackaging=jar \
  -DlocalRepositoryPath=/var/www/maven2 \
  -DcreateChecksum=true

Example pom for this setup can be found at:
http://github.com/udoprog/c10t-swt/blob/master/pom.xml

I found this to be one of the more flexible ways of developing swt-based applications - and after a quick google, i could not find anything nearly as straight forward.

For now I'm using the toolchain.eu repo, but looking forward to when you guys get this sorted out!
Comment 29 Olivier Cailloux 2011-05-30 06:26:38 EDT 
John-John, you have a nice setup proposal, but I'd suggest differentiating between swt builds with only one configuration entry instead of having different artifact AND classifier for different platforms. E.g. use different "classifier" properties and use constant "artifactId"="swt" (the name of the jar, as suggested here https://docs.sonatype.org/display/Repository/Choosing+your+Coordinates).

I think there is no need to have two different configuration entries to set up when specifying the platform. E.g., linux-32 and linux-64 could be considered as two different platforms, IMHO, so the classifier should be linux-32 and linux-64 instead of artifact specifying "linux" then classifier specifying if it is 32 or 64 bits. I don't think there is any need to group all the "linux" versions together under the same artifact name, you need anyway to completely specify the platform. This is illustrated in the proposed POM https://github.com/udoprog/c10t-swt/blob/master/pom.xml where the profiles are "linux-32", "linux-64", etc.

I'd suggest having a constant artifactId=swt to specify that the dependency is really on SWT. Then the classifier tells the inner details of which specific jar is needed depending on the platform you run on (by platform here I mean whether you run linux, windows, and also whether you run on 32 bits, etc.). So when my project states that it has a dependency on group org.eclipse.swt, artifact swt, it really says that it depends on the SWT functional block being there and implemented "somehow adequately depending on the platform". The classifier tells what "adequate for the platform" means, this can be achieved as usual with the mechanism proposed in the referenced POM.

An other possibility is to have different artifactIds depending on the target platform (thus swt-linux-32, swt-linux-64, etc), and no classifier. I like it less because it means the artifact changes whereas the dependency (see from a functional POV) does not change, thus it does not really abstract the platform which is ran on. But I don't care much though. The main point is that I don't think artifact AND classifier should both change depending on the target platform.

Probably when what's needed is the source for SWT, the classifier should be free to be set to "sources". AFAIU this is what's classically implemented by tools which automatically download sources on demand, and that's what's recommended in the manual (http://maven.apache.org/pom.html). Using my proposal, the ref for the swt sources would be artifactId=swt, classifier=sources. This assumes that the sources to be downloaded do not depend on the architechture, which seems reasonable to me as I guess the typical developer only wants the sources for the common, java part of it, not the sources which depend on the architecture. But I may be wrong.

-----
For the reference, the manual says:
# classifier:
The classifier allows to distinguish artifacts that were built from the same POM but differ in their content. It is some optional and arbitrary string that - if present - is appended to the artifact name just after the version number.

As a motivation for this element, consider for example a project that offers an artifact targeting JRE 1.5 but at the same time also an artifact that still supports JRE 1.4. The first artifact could be equipped with the classifier jdk15 and the second one with jdk14 such that clients can choose which one to use.

Another common use case for classifiers is the need to attach secondary artifacts to the project's main artifact. If you browse the Maven central repository, you will notice that the classifiers sources and javadoc are used to deploy the project source code and API docs along with the packaged class files.

As I understand it tycho works with p2 repos and OSGi bundles, but it doesn't convert p2 metadata to maven metadata to allow cross-dependency resolution (i.e. you are either using maven dep resolution or you are using tycho dep resolution and p2 repos).  There are some tools out there that try and convert the data for you, with varying degrees of success.

I think that's why the Eclipse Foundation is still working out how to publish bundles into maven, so that they can be consumed by maven dep resolution for normal projects (as opposed to tycho/maven + p2).  But there's a lot there to wrap one's head around :-)

Why?  Because it's not working yet :-)  There's lots of missing information, especially in the project POM (which level is that, each artifact?  SWT?  EclipsePlatform?). Amongst the requirements that we've seen:

1) If the project packaging is jar, and the jar file contains java classes, there must be a -javadoc.jar for main artifact.
2) If the project packaging is jar, and the jar file contains java classes, there must be a -sources.jar for main artifact.

We can do #2, but not #1.

Generating the jar is easy, we already have that ... but you're not allowed to push just a jar to maven central, you need a lot more stuff, and PDE build doesn't generate it.

I'm hoping that by converting our build to maven/tycho we will be much closer to producing the artifacts that we need to push to maven central, although we still don't produce the javadoc jar.  If anyone wants to get the javadoc generation added to our maven/tycho builds, have a look at http://wiki.eclipse.org/Platform-releng/Platform_Build to be able to run the maven build locally.

As for past releases, https://code.google.com/p/swt-repo/ is a good example of getting started, but it doesn't look like it meets the requirements of what can be pushed up to maven.  I'm also curious does it correctly convert the OSGi dependencies (like JFace depends on org.eclipse.equinox.common) into maven dependencies?

Once we have what's to be published [1] (because as I understand once it's published it's forever) then we would get someone to publish stuff.
