Comment 4 Neale Upstone 2005-09-09 11:51:21 EDT 
Think I've nailed it:

the server file.

correct order once)
Comment 5 Larry Isaacs 2005-09-09 12:33:41 EDT 
Its a problem that spawns from the handling which attempts to preserve user 
modifications to the run configuration.  If the server has been started 
without the debug option, then the just "start" is already present as the 
startup argument.  When you add the debug option, the handling is smart enough 
to add the "-debug", but not smart enough to put it before the "start".

Also, it appears that turning off the debug option won't remove the "-debug" 
because the handling treats it as if the user added it.  For now, the work 
around it to manually add and remove the "-debug" option in the correct 
location.

Tim,

Feel free to assign this one to me.  I'll also look to see if there is any 
console output being lost that would provide a clue to the problem.  The user 
could make this mistake as well.

Comment 10 Larry Isaacs 2005-12-14 10:54:19 EST 
Hi Tim,

I asked about this one in a e-mail sent 11/29.  Since I knew you were swamped, I didn't pursue.  The "-debug" still appears for me in the program arguments of the launch configuration, after I've started the server.  The problem senario is still the same, if you have previously started a new server and then add the debug option, Tomcat 5.0.x and Tomcat 5.5.x won't start because they aren't tolerant of the "-debug" coming after the "start" argument.  The "Failed to start" error provides no clues to the problem.  The workaround to manually remove the "-debug" or move it to preceed "start" is simple enough, so I didn't think this bug was too critical.

The complexity of the fix will depend on how compatibly we want to work with user modifications to the program arguments.  If the Tomcat plug-in can "own" the "-debug" argument, then I think a simple fix could be included in WTP 1.0.1.

Larry
