Description Chris Recoskie 2005-09-26 10:43:15 EDT 
Currently it is difficult to subclass GnuMakefileGenerator  because all of the 
data structures and several methods which directly operate on them are 
declared private.

Previously I have copied the entire generator wholesale into my own class, but 
this is a real pain to maintain because I have to migrate any changes 
manually.  I would prefer to subclass GnuMakefileGenerator and just override 
the bits that I need.

The part that's currently giving me trouble is that I'm trying to override 
generateDependencies(), which needs access to the private list of 
subdirectories that need to be built.

I would like to go through and change some or all of these data structures and 
methods to be protected rather than private.  I don't think it's worth 
creating new APIs because this is not an interface class - really what I'm 
trying to do is reuse the implementation, and if I wanted to I could certainly 
create my own implementation based on the interfaces (whether I do a wholesale 
copy or not).  I'm open to discussion on that point though.  If we did revamp 
the APIs then changes to IManagedBuilderMakefileGenerator would be in order.

Leo and Mikhail, please provide some feedback as to what you think of the 
idea.  If we go ahead with the idea I would also like to collaborate on 
deciding just what exactly should be protected and what should stay private.

I would rather address the issue as a whole at once rather than revisiting it 
piecemeal every time someone needs access to some additional fucntionality, so 
I am willing to work on a solution that is beyond the scope of just the 
functionality I personally require currently.
Comment 1 Leo Treggiari 2005-09-28 14:00:45 EDT 
I thought I responded this morning, but something didn't work.  Maybe I 
responded to the wrong bug.

I'm not too keen on supporting class derivation in this case.  
GnuMakefileGenerator is a fairly big, complex class and not necessarily ideally 
architected.  Sean did the initial implementation and I have "bolted on" the 
new object model in CDT 2.1 and much new functionality in 2.1 & 3.0.

The public classes that I can remember seeing in Eclipse are the default 
implementation of some interface that a client may want to implement.  This 
makes sense if much of the implementation is fairly standard and can be 
inherited while overriding a small number of methods.

I think class derivation in this case would be very "fragile".  I suspect there 
will be lots of changes in the class as we implement new features.  Maybe the 
class could be broken up in such a way that the places where a client would 
need to override would be available?

Anyway, if you'd like to make a proposal, I'll be happy to review it.
Comment 2 Chris Recoskie 2005-09-28 16:24:26 EDT 
I agree that what we really need is an overhaul.  It is not as well designed 
as a "public" class should be.

The class is not marked as internal in any way, so in theory clients should be 
able to extend it freely.  This also makes sense in the context of this 
specific class.  E.g. I am using GNU make, but because my actual compiler and 
linker are not GNU tools, I can't use the generator as-is because it has a few 
GCC-isms in it.  But, since I'm using GNU make, it doesn't make sense for me 
to make my own generator because what I need to do is the same except for a 
few lines of code here and there.  Hence, it makes sense to extend the 
existing generator.

I can live with the situation for now, but I would like to perhaps revist the 
architecture of the class for say CDT 3.1  In general, given the information 
that came out of the CDT 3.0 project review (i.e. that we have a lot of 
classes which are technically API but are not properly supported/documented), 
we're probably going to have to start giving more thought to issues like these.

If there are no objections, I'll move this to 3.1 and we can give it a bit of 
thought at that time, even if we decide to put it off further.
Comment 3 Mikhail Sennikovsky 2005-10-03 07:35:42 EDT 
Hi Chris, Leo,

I suppose it makes sense to look at the problem more widely.
The managed build system is intended to support any builder a tool-integrator
wants to use/integrate, but not just make-like builders. Currently it is really
difficult to implement the generator for some custom builder, because the
algorithm for calculation the buildfile information form the managed build info
is really complex, and the managed build model changes from version to version
becoming more and more complicated.

I remember some discussions regarding implementing some CDT/MBS internal build
system that users could use instead of make. Also there were some discussion
regarding making the MBS support ant buildfile generation (see e.g bugzilla#
71386, comment#2)
The Gnu makefile generator contains a lot of complex functionality that could be
generalized for the most builders, but I don’t think that subclassing one
abstract builder class that contains some general functionality would work in
all cases.

I wonder if we could split the buildfile generation functionality in two parts:

1.	Calculating the sequence of tool invocations and input/output resources used
in each invocation needed to build the final build artifact based on the given
set of sources and the managed build settings.
2.	Storing data calculated in the first step in the builder-specific buildfile
format.

The part#1 is the most complicated but at the same time it could be made common
for all buildfile generators.
I wonder if we could implement the first part separately in the MBS and provide
the public API for obtaining that information. The data calculated in the first
step could be presented e.g. as some simple object model that would present tool
invocations to be launched, input/output resources used for each invocation and
their dependencies/interrelations.
In this case it would be much easier for the tool-integrators to implement the
custom generator, because the build generator will have only to store the build
data in its buildfile format thus focusing on the custom buildfile syntax rather
than on the build data calculation.
This approach would also allow us to create the MBS internal builder or ant
buildfile generator (if needed in the future) with much less pain.

What do you guys think about that?

Thanks,
Mikhail
Comment 4 Leo Treggiari 2005-10-03 13:38:23 EDT 
Hi Mikhail,

I think that is very interesting!  Two issues come to my mind immediately:

1.  Command line patterns vs. individual invocations:  The GNU Make generator 
currently attempts to use GNU pattern syntax for tool invocations when 
possible, but different builders might support different pattern semantics or 
require separate tool invocations for each source file.  How would your "part 
1" model represent the build steps such that patterns could be used or not used 
depending upon the builder?

2.  Which code determines what needs to be rebuilt in this build:  The GNU Make 
generator generates a complete project makefile and then leaves it up to make 
to determine what need to be rebuilt.  An MBS "internal" builder really only 
needs to know what command need to be executed to get the project up-to-date.  
How do we determine how much of the work "part 1" does?

I think about your proposal some more.

Leo
Comment 5 Mikhail Sennikovsky 2005-10-04 06:37:50 EDT 
(In reply to comment #4)
Hi Leo,

> 1.  Command line patterns vs. individual invocations:  The GNU Make generator 
> currently attempts to use GNU pattern syntax for tool invocations when 
> possible, but different builders might support different pattern semantics or 
> require separate tool invocations for each source file.  How would your "part 
> 1" model represent the build steps such that patterns could be used or not used 
> depending upon the builder?
I assume there would be the possibility to specify some kind of “pattern command
invocation” that could be specified input and output patterns and how those
patterns are to be used in the command line. The pattern command would be able
to expand to the set of real commands to be invoked for the project. So the
pattern command would be either converted to the makefile format syntax (e.g.
for the make builder), or expanded and presented as a set of commands (e.g. for
the MBS internal builder).
Note: the “pattern command” and the “input/output patterns” are assumed to be
presented as some interfaces rather than strings of some specific format that
are to be parsed, so a makegen writer would be able to easily operate with
pattern and determine files to which the pattern applies.

E.g. the pattern could be represented by the interface having the following methods:
 
String getNamePrefix();

String getBaseName();

String getNameSuffix();

If the method, e.g getBaseName() returns null, this would mean “for any”. E.g.
{getNamePrefix() == “”, getBaseName() == null, getNameSuffix() == “.c”} would
mean that the pattern applies for all files with the “.c” extension.

I think that the above pattern definition would be enough to suit MBS needs.

> 
> 2.  Which code determines what needs to be rebuilt in this build:  The GNU Make 
> generator generates a complete project makefile and then leaves it up to make 
> to determine what need to be rebuilt.  An MBS "internal" builder really only 
> needs to know what command need to be executed to get the project up-to-date.  
> How do we determine how much of the work "part 1" does?
I assumed initially that the “part 1” calculates the complete set of
commands/invocations/resources used for the full build, and this is up to the
builder (e.g. make, as you mentioned) to determine what is really needed to be
rebuilt. The object model that would be built in “step 1” would represent tool
invocations/inputs/outputs correlations and it should be designed in the way
that it would be easy to calculate what is needed to rebuild based on those
correlations, e.g. for the case some resource or build settings are changed.
I agree that this might not be the best approach. I see three ways of handling a
“what needs rebuild” problem:
1. Leave this up to the builder to decide what needs to be rebuilt (as described
above)
2. Move the “what needs rebuild” calculation to step 1. In this case, MBS will
be told either to generate the complete sequence of tool
invocations/inputs/outputs (e.g. for the makefile generator), or only the
sequence of invocations needed for the incremental build (e.g. for the MBS
internal builder). 
3. Build the complete sequence of tool invocations/inputs/outputs and provide a
mechanism for calculating whether the specified tool invocation is needed,
whether specified resource is up-to-date, etc. E.g. an interface representing a
tool invocation in the model could have some method that would specify whether
the tool invocation is needed or not.

Looks like #3 is the most flexible and probably the best way to go.

What do you think?

Thanks,
Mikhail
