Description Anthony Hunter 2005-09-19 10:41:54 EDT 
There exists a tabbed properties view implementation in the Web Tools common 
component. The plug-in is org.eclipse.wst.common.ui.properties.

This plug-in should be moved to the core platform so we do not need to depend 
on WTP.

Comment 2 Randy Hudson 2005-09-27 16:33:12 EDT 
I think that the look and feel of the tabs needs to be addressed before it can
be proposed for adoption by the platform. It does not embrace/use the platform
look and feel, and at the same time I don't see how it improves greatly over the
existing SWT widgets.

I have the following constructive criticism (is there any other type;-) ) on
usability of the view:
1) The tabs are being used for different purposes. Some clients are using tabs
for grouping of properties. Others are using it to represent the object
hierarchy of what is currently selected. It should be more consistent.

2) The pages in the tabs are often designed differently/inconsistently (use of
controls, padding, mnemonics, columns, etc). There is no enforcement of
consistency. The pages are also very sensitive to the aspect/size of the view.
Some pages require more width/height than others.
In constrast, look at PDE's handling of extensions in the plugin.xml editor.
They dynamically generate a forms-based "property sheet" based on the extension
point's DTD. It's a single list of 2 columns that uses a predictable amount of
screen space and generally only requires vertical scrolling.

Perhaps this should be split into 2 proposals. One for the widget to be adopted
in SWT (since JFace doesn't provide widgets), with the rest going into "UI"
somewhere.

Comment 11 Eric Moffatt 2005-10-05 11:03:22 EDT 
Time for my two cents...

The current PropertySheetPage (PSP) is what I call "industrial ugly"...it works 
functionally and it relatively inexpensive to adopt but certainly 
isn't 'glitzy'. For example it currently doesn't even allow showing the 
actual 'color box' for color properties until the CellEditor becomes active 
(inactive 'cells' are always displayed as text).

I've attached an image showing how the same info would appear in the current 
PSP and in a 'form'-based mapping of the same PropertySource for comparison 
reasons.

Look and Feel: This is (and always has been) a religious issue (which is why 
the workbench supports "Presentation") but if -we- don't provide a path of 
least resistance for folks that don't want to use the current default then 
they're going to 'roll their own'; leading to a much higher risk that two 
different contributors will choose radically different looks (i.e. UIForms 
vs 'Dialog'-like vs Swing).

Note that this is already happening...enough folks have bought into 
the 'UIForm' style that this defect exists (i.e. multiple clients are already 
using the WTP version) and we have absolutely no control over what individual 
contributors will choose left to their own devices.
