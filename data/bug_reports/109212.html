Description Susan McCourt 2005-09-09 17:30:10 EDT 
While investigating priorities and implementation ideas for adding additional 
undo support in Eclipse, I'm starting to think that implementing a generic 
resource-based undo will solve a big chunk of the undo feature requests with 
the lowest impact on clients.  I'm opening this bug to discuss the idea and 
obtain suggestions/pointers from those who have a lot of experience in this 
area.  I'm not sure at this point if more API is needed from resources or not.

Most of the open bugs in undo (besides bugs or enhancements to text editing 
undo) involve underlying changes to workspace resources, including markers.  
This includes:
- task list undo (bug #7691)
- navigator view undo  (bug #23172, bug #65421)
- undo import(bug #28995)
- undo apply patch 

Further, our priorities are to implement undo where data is lost or else where 
the corresponding counter-action is not obvious (such as retrieve from local 
history, etc.)

Changes to the content of resources can also affect the undo history for any 
editor that is open on an affected resource, so coordination of the workspace 
history with the text undo history must occur.  We handled these complications 
for refactoring undo, but we were starting with an existing refactoring undo 
infrastructure that was already recording the changes and building the 
description of the undo.  So our integration focused on mapping the framework 
to the new API and coordinating undo inside the text editor with 
the "overlying" refactoring changes.  

As part of that work, we introduced the idea of an "Open Operation" that 
batches all undoable changes added to the history.  When a client opens an 
operation, they are saying that any subsequent undos generated should be 
batched together and considered part of the open operation.

I think we could generalize this approach and provide undo for all workspace-
affecting operations by doing the following:
- a "WorkspaceUndoBuilder" listens for workspace changes and traverses 
resource deltas, building an undoable operation for the change described by 
the delta.  The local file history can be used to point to corresponding file 
content changes.  The resource modification stamps can be used to determine at 
any point whether an undoable operation is still valid.  
- Unless explicitly instructed otherwise, the workspace undo builder opens an 
operation to represent the change described in the resource delta as an atomic 
change.  This open operation by default is named generically, such as "Changes 
to Xxx Project" or "Changes to file.txt" or "Move Folder Xxx" so that undo 
would appear as "Undo Changes to Xxx Project," etc.  This means that even if 
no one else did any work, there would be generic, batched resource-based-undo 
available.
- Explicit protocol on WorkspaceUndoBuilder coudl allow clients to open and 
name a semantic operation, such as "Apply Patch..." or "Import Project" 
or "Replace with latest from HEAD."  Clients must also close this operation.  
In this case the WorkspaceUndoBuilder will consider all deltas received to 
belong to this operation until it is explicitly closed.  
- Clients that run workspace runnables would generally open the operation 
before running the runnable, closing it afterward, and assigning it its name 
(or open and close it as the first and last part of the runnable itself).  If 
they didn't do this, there is still generic resource-based undo available for 
that operation. 
- Likewise, a workspace job could open and close the operation as the first 
and last part of running.

The operation history model of an open operation would have to evolve to 
support multiple open operations, since workspace jobs with non-conflicting 
rules can run concurrently.  In effect a scheduling rule (or something mapping 
to it if we don't want operations to reference ISchedulingRule) could be used 
to open an operation on a specific rule.  We'd also need to use job state 
listeners to keep track of which open operation is active.  The idea is that 
when operation history protocol is used to add an undoable operations to the 
history, we need to know which open operation they are assigned to, if any.  
Evolving the existing concept of one open operation is basically the same 
problem as evolving the global lock of the old IWorkspace.run to handling more 
granular scheduling rules.

Does anyone see any problems with this approach or have further suggestions?
Comment 1 John Arthorne 2005-09-14 16:53:21 EDT 
Sorry for the slow response, I've been tied up in other things.  There are some
general challenges with the approach of a delta-based undo builder as far as I
can see:

 1) Deltas don't currently contain enough information to support general undo. 
Deltas do not specify the local history revision corresponding to the change in
the delta (a single CHANGED delta might be the result of several batched changes
to the same resource).  I'm also not sure if marker deltas have enough detail
for doing a completele undo.  Other pieces of resource metadata such as session
properties, persistent properties, and sync info certainly don't give enough
delta information to revert a given change.

 2) It is difficult to automatically distinguish workspace changes made by the
user, versus changes made by tools behind the scenes.  As a very simple example,
undo for builds doesn't make much sense to me.  Search generates markers under
the covers that it wouldn't make sense to undo, and CVS generates hidden files
for storing sync information.  The net effect is that the user could select
"Undo Changes..." and it would undo something they didn't even know had been
done.  Further, the tooling that made those workspace changes could be broken by
undo mucking with their "private" metadata and resources.

 3) It's not clear to me how undo works in the face of concurrent workspace
changes. From the point of view of the workspace implementation, complex
operations are broken into discrete atomic units that happen in linear order.
However, the work units for a given user-level operation might be interwoven
with an unrelated operation.  At the moment of a resource change event, you
might have finished operation A but be half way through operation B.  Some of
the changes in that delta can be attributed to A and some to B.  I guess what
you'd need to do is: at the moment the resource change event occurs, find out
what threads currently own resource scheduling rules.  Split the resource delta
according to those scheduling rules and attribute the various pieces to each
running thread.  It might be possible, but it sounds scary.  

4) If you impose an undo stack structure on concurrent operations, you may force
the user to undo a completely unrelated operation in order to get back to the
one they really wanted to undo.  I guess this problem currently exists even with
linear operations. If I delete file 1, then create file 2, and later realize
that deletion of file 1 was a mistake, I don't want to be forced to undo my
creation of file 2. I don't know enough about the undo API to know if this
problem has been addressed.
Comment 2 Susan McCourt 2005-09-21 13:11:59 EDT 
Thanks for reviewing this, John.  Some thoughts on each of your points.

1.  Deltas don't contain enough information - I thought this might be the 
case, and if we pursue the idea further then we could investigate specific 
enhancement requests.

2.  In the original suggestion, I suggested an "automatic/generic" undo 
generated for all deltas, with the explicitly opened ones having the semantic 
name.  We could drop the automatic one and only generate undos when a client 
explicitly opens an operation, signifying that subsequent workspace changes 
are associated with their operation.  This solves the "surprising/unknown 
changes" problem but not the next problem...

3.  Is it really true that a resource delta might contain changes from two 
separate concurrently running workspace jobs?  I assumed that a workspace job 
would batch a delta describing only that job, and if it was interrupted to do 
another job or foreground work, then those would be in their own deltas.  At 
least that seems how it should work.  ;-)  Have there been any other use cases 
where clients expect it to work this way?  Without this working, then there's 
no way for us to rely on a generic solution and I'll go back to looking at 
specific solutions for different actions.  (Too bad...)  I agree that it's not 
the business of the undo code to sort through the changes in a delta and 
assign to the correct thread.  

4.  The linear undo model is what is presented by the workbench today, but the 
API allows for more generality.  So I see this problem as a separate one that 
would be addressed once linear undo was widely supported.
Comment 3 John Arthorne 2005-09-22 17:00:22 EDT 
I'll jump straight to 3, since that seems to be the main sticking point.  Yes,
it is really true that resource changes may contain changes made by several
different jobs or threads.  Essentially, resource change deltas are assembled in
one central place.  At the moment a resource change event occurs, we calculate
all the tree changes that have occurred since the last resource change event
occurred, and present that delta to users.

Here is an example of why it is important for it to work this way.  Say we start
a very long operation in thread A (such as a CVS checkout).  As part of that
long operation, thread A creates a file.  While that operation is running,
another thread B adds a marker to that same file (let's say it adds a
breakpoint). The operation in thread B is very short, so it ends while the
operation in thread A is still going.  At this point, we need to broadcast a
resource change event so the breakpoint appears in the UI.  If we didn't, the
breakpoint wouldn't be visible until the end of the CVS checkout. We can't just
broadcast the delta for the operation in thread B, since that would cause the
marker creation to be announced before the creation of the file. Resource change
listeners rely on the events arriving in the same order they occurred. What we
actually do here is broadcast a single delta with all changes from the start of
operation A to the end of operation B.  This includes creation of both the file
and the breakpoint.

So... I don't see how a general delta-based undo can work, at least in the case
where multiple operations happen concurrently.  It is possible we could at least
detect the case where a delta is due to changes in multiple threads, and then
provide undo for those cases only.  However, this would likely undermine user
confidence in undo, if an operation that was previously undoable did not work at
a critical moment due to a concurrent change.  It feels like an "explicit" undo
approach for a small set of key operations is more tractable than a generalized
system.

Comment 7 Susan McCourt 2006-02-12 20:29:39 EST 
moving to M6.  I've done some prototyping, enough to feel that API additions are not needed in the undo framework for this to work.

Undo of marker update/create/delete is implemented in the prototype.  Rename/move is working.  I've investigated delete.  The challenge seems mostly in me learning the API enough to ensure I've restored all the right bits (besides the content itself).

There is an open area of risk involving dependent changes.  If a plug-in responds to an operation such as a delete by doing something else and does not participate in undo, then undoing the operation may not completely revert to the original state.  I think the best we can say here is that if you use simple atomic navigator-level operations, that is the state of the world for now, and encourage use of refactoring for more complex scenarios.

It  seems like the undo API's are holding up and I don't foresee any changes to API.  There are some nice abstractions in the prototype that may be generally useful - undoable workspace operations that handle infrastructure for workspace runnables, error reporting, etc., and abstract operations that can handle create/delete/update of certain resources.  We can decide in M6 whether we want to keep these internal, or make at least part of the abstraction public so that other plug-ins can get on board quicker with undo. 

Another challenge will be the amount of duplicated code in the workbench today.  For example, the tasks view uses a different dialog to create/update markers than the text editor and JDT do.  So the changes to the triggering actions and dialogs have to be made in multiple places with slightly different factoring.  This is unfortunate, but doable.
