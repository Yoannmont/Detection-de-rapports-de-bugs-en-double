Comment 7 Michael Golubev 2007-04-23 09:57:04 EDT 
The described problem looks like general GMF issue. 
Diagram initialization code traverses semantic model starting from its root, collecting the elements for which nodes should be created. 

In the attached sample, traverse finds the model, package and finally 2 classes -- Class and Class2. The generalization's destination (MyClass) is not traversed because it belongs to different root in the different resource. 

Finally, initializer attempts to create all links, that are satisfied by mappings. Generalization is OK from semantic point of view, so it creates notation Edge's for it. However, the link's target can not be found, forcing error described by log. 

The proposed workaround filters links that does not have source/target on diagram. As a result, the diagram for model2 can be opened, and shows package and 2 classes without any links. 

It is correct behavior, but the problem should be resolved at the templates level for all diagrams, not just as a single line hack for ClassD initializer.
