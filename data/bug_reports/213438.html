Description Harald Krapfenbauer 2007-12-19 06:24:42 EST 
I want to use an "SSH only" connection to connect to an embedded target. Obviously, TM requires the SFTP service on the target to be able to list and transfer files.

The issue is that on some embedded targets that run Linux, smaller SSH servers like "dropbear" are installed which offer no SFTP service.

To use target management with my target, I have to use a full-blown OpenSSH installation, but this is not desired due to its higher memory footprint.

Hence, I would appreciate if Target Management could provide file transfers also if only the SCP protocol is supported by the target. I am aware of the fact that since SCP is not as standardized as SFTP, it is a lot more complicated to make it work.
Comment 1 Martin Oberhuber 2008-01-22 18:05:19 EST 
We currently don't have committers who could spend time to implement SCP, but we would consider patches from the community.

It looks like the biggest problem will be directory browsing, because in SCP that needs to be implemented through shell commands like "ls" or "dir". Given that the kind of remote system and remote shell is initially unknown, it might be hard to find the proper commands and parse the output appropriately.

Note that for Dropbear, Sftp can be added by installing the sftp-server from openSsh for instance:
http://wiki.openwrt.org/SFTPWithDropbearHowTo
Comment 2 Martin Oberhuber 2009-03-19 08:58:03 EDT 
For SCP, the JSch library actually has some sample code for doing the actual SCP operation, so that might help:

http://www.jcraft.com/jsch/examples/
http://www.jcraft.com/jsch/examples/ScpFrom.java
http://www.jcraft.com/jsch/examples/ScpTo.java

The problem when integrating this with RSE is really in getting and parsing a directory listing (requires "ls" parsing at least which requires a shell channel).

As the "helpwanted" keyword indicates, community contributions for this are highly welcome.
Comment 3 Lothar Werzinger 2009-03-24 11:09:26 EDT 
It may be worth looking into the FISH protocol.
It works by running a Perl script on the server that implements the FISH protocol by running the FISH server over SSH.

http://en.wikipedia.org/wiki/FISH_protocol
Comment 4 Martin Oberhuber 2009-03-24 11:13:50 EDT 
... and the FISH perl script is dynamically deployed by uploading to the remote when the connection starts. This initial deploy step ("bootstrapping") uses SSH only, and therefore works on any server even if sftp is not available.
Comment 5 Daniel Jacobowitz 2010-06-28 14:09:07 EDT 
Note that FISH is not helpful in Harald's original scenario.  A target running dropbear is pretty likely to not have Perl, either.

Another option is to send a binary agent to the target somehow; we can use scp to do that.  dstore has the same problem as fish; it requires Perl (and a JVM?).

The TCF project has native binary agents that run on the target.  Maybe the TCF agent could be used as a file service instead?  And then we could bootstrap that over SCP/SSH.  This does mean providing and selecting a compatible agent binary on the host.  It's a richer solution but more complicated.

ISTM there is a good motivation for /bin/ls parsing.
Comment 6 Martin Oberhuber 2010-07-05 09:55:29 EDT 
When you want to use a TCF agent, why can't you have the TCF agent boot up with the target (like dropbear boots up today)?

Implementing a whole scp service just for bootstrapping seems a lot of effort, and I'm not sure how universally applicable such a thing would be.

If you can't compile TCF into your target fs at boot time, what about having a boot script on the target "pull" whatever you need?(In reply to comment #5)

> dstore has the same problem as fish; it requires Perl (and a JVM?).

dstore requires an 1.4 level JRE on the target, but not necessarily perl.

Comment 9 Martin Oberhuber 2010-10-20 09:50:24 EDT 
I have had a first look at the contribution. My main concern is that it is quite obviously a clone of the SftpFileService, but doesn't always include proper Copyright attribution where code has been copied. 

ScpFileAdapter looks OK; but, for instance, ScpFileUtils#concat() is quite obviously a clone of SftpFileService#concat() without attribution. Copy & paste should indicate where the code came from. At the very least, the copyright header must state "Mentor Graphics and others" when code came from somewhere else (in this case, Wind River).

The ScpFileService currently doesn't implement a number of features such as move(), copy(), delete(), rename(), createFile(), createFolder(), setReadOnly(), setLastModified(), setFilePermissions(). This is particularly bad since getCapabilities() returns FS_CAN_SET_PERMISSIONS so it claims to be able and set permissions, yet it can't.

I'm OK with accepting a contribution that only provides limited features, since apparently the contribution is valuable to you and the community even if it can only browse the remote file system, upload and download. But I do think that Copyrights deserve proper attribution, and API contracts must be honored (in this case, getCapabilities()). 

Also, I'd suggest a 0.x version number and an "(Incubation)" note in "about" since it doesn't look ready for general consumption while so many API's are not implemented. Have you actually tested what happens when a user tries creating/deleting/moving/copying/renaming a file or folder? Limitations should be documented such that Community knows what to expect.

Comment 16 Anna Dushistova 2010-11-24 15:03:31 EST 
I found one problem/limitation during the testing. The time is shown different on different machines.
And if with standalone ls we can standardize it with using --time-style=long-iso option, the busybox ls command doesn't support it. I am perfectly fine with checking in initial version that would work with either ls or busybox, just don't know which one is more needed. Any opinions on that?
Comment 17 Martin Oberhuber 2010-11-25 00:51:00 EST 
I assume that non-working ls on some machines would lead to a total failure, whereas incorrect date just leads to degradation. Few users will care about times; In general, I think scp was written for busybox-like systems so these should take priority.

Probing the connection and falling back to standard ls if the long format might be an option, but it makes behavior of the subsystem nondeterministic. Other parameters like the time zone where the remote system is located might be impossible to find out. Perhaps we'd want the possibility of entering a "correction value" for times in the subsystem configuration? Or document that a user must use sftp (which is well-defined) if determinism is needed.

I think it's more interesting to start documenting such limitations (we'll likely find more). We can surely use bugzilla for documenting the developer side of things. We might need to come up with a good solution for documenting limitations in a form that the user can see.

The "description" field of the subsystem would be one option for doing so. Perhaps introduce a URL pointing to an online resource (eg on Wiki) that has the latest information on limitations? Other thoughts?

For now, I suggest filing a bug with an [scp] tag for the ls limitations, and another bug with an [scp] tag for documenting limitations for the user.
