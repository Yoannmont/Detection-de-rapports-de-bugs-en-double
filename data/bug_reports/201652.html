Comment 3 Mark Melvin 2007-08-31 12:03:36 EDT 
Thanks for the reply.  I haven't tried a Java launch yet - but I assume it is fine.  I didn't write our launch but I see some weird behaviour...

The launch in question is launched from the delegate as follows:

if (ILaunchManager.RUN_MODE.equals(mode)) {
    IProcess p = new MyLoaderProcess(...);
    launch.addProcess(p);
}

However, it does all of its work in the constructor - it connects, downloads, runs, disconnects, and terminates itself.  It also fires a terminate debug event.  This all happens before "launch.addProcess(p)" is executed.  So by the time the process is added to the launch it is already terminated.  I'm assuming this is the problem.

Do you have any recommendations on how I can fix this?  I guess maybe I could do something like this (if it will help)?

Comment 11 Mark Melvin 2007-10-03 17:43:03 EDT 
OK, just to wrap this up.  *Actually* changing my code from:

if (ILaunchManager.RUN_MODE.equals(mode)) {
    IProcess p = new MyLoaderProcess(...);
    launch.addProcess(p);
}

(where all the work, in addition to terminating, is done in the constructor)

to:

if (ILaunchManager.RUN_MODE.equals(mode)) {
    IProcess p = new MyLoaderProcess(...);
    launch.addProcess(p);
    p.doWorkAndTerminate();
}

...does solve the problem.  ;)

I had originally tried to do this (admittedly very quickly at the time...) by wrapping the work+terminate in an immediately-scheduled Job in the constructor - which did not have the same effect.  There were still brief occasions where the scheduled job seemed to execute before the process was added to the launch.  Scheduling the job with a delay confirmed this.

So I think the root cause was that since the process was not added to the launch, the infrastructure was missing most of the debug events, in particular the terminate event - which normally cancels the timer and disposes the model proxy.
