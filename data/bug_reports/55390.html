I did the standard PDE export of a set of plug-ins, shutdown eclipse, and
restarted. I got many exceptions in the log about classes not being found. I

Comment 4 Thomas Watson 2004-03-22 12:05:39 EST 
When should a plugin be reinstalled?  From comment 1 it looks like we are 
suggesting that a plugin must be reinstalled one of the three things happen:

1.  The plugin manifest (plugin.xml, fragment.xml) is modified.
2.  The bundle manifest (/META-INF/MANIFEST.MF) is modified.
3.  Any code is modified.  Does this include a resource is modified (icons etc)?

The timestamp is based off of the base file of the plugin.  In the standard 
case this is the directory the plugin is located.  This does not work because 
if the plugin.xml is modified in the plugin's directory the timestamp of the 
directory is not changed.  The only time a directory timestamp seems to change 
is if a file is deleted or added.

Comment 10 Jeff McAffer 2004-04-06 08:40:19 EDT 
Just to clarify, when you redeployed the plugin you did so into a configuration 
that had already been run correct?  So the issue here is how we check for 
changes in the configuration.  By default I believe we no longer check at all 
as this is the 100% case for products and the 70% case for developers.  The 
osgi.checkConfiguration flag enables the checking.

However, the checking only looks at the plugin.xml etc files, not the code.  
Looking at the code is possible but costly both to compute and to store.

I'm leaning towards a -clean style action which is basically equivalent to 
deleting the configuration area.  We definitely do NOT want people to get into 
the habit of deleting the configuration area and -clean can do a surgical 
removal of info.  Further, using a property like osgi.clean other plugins can 
check to see if they should be purging their caches etc.

Havig PDE modify the plugin.xml puts that file "out of sync" with its 
original.  That is, someone looking at the timestamps will see a change but in 
fact the file did not change.  This can be disconcerting.  PDE cannot touch the 
plugin.xml in the workspace because then it looks like an outgoing change to 
the server.

Comment 17 Jeff McAffer 2004-04-26 23:22:42 EDT 
Ok, lets move on the -clean approach.  Heres what we do
- -clean command line option
- osgi.clean System property
- consume the arg in EclipseStarter
- instrument OSGi level (state, bundeldata, manifest caching) to check and 
ignore/regen caches if property spec'd = true (absence or any other value => DO 
NOT ignore)
- instrument runtime to flush registry cache.  This may come for free depending 
on how the consistency check is done.  For example, if they check the state 
timestamp then it will change so perhaps nothing is needed.

Tom, do you want to go at this or should I take it?
