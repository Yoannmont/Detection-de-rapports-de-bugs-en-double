Comment 1 Stefan Matthias Aust 2004-09-11 08:10:47 EDT 
Actually, it seems to be simple to replace IE with Mozilla's rendering engine.

All you have to do is to register "mozctlx.dll" from the Mozilla distribution as
an IE-compatible ActiveX control using "regsvr32.exe" (or use
http://www.iol.ie/~locka/mozilla/control.htm#download) and then replace a single
line in Browser.java.  Use

 site = new WebSite(frame, SWT.NONE, "Mozilla.Browser");

instead of

 site = new WebSite(frame, SWT.NONE, "Shell.Explorer");

It would be a really nice extension to SWT, if this name would be a System
property, for example "org.eclipse.swt.browser.name".  Then anybody prefering
Mozilla over IE could replace it without modifying SWT.  IMHO, this would
improve cross-plattform portability because then Windows and Linux would use the
same identical rendering engine.

That doesn't give you popup blocking and other features though, because they are
not related to the rendering engine.  You'd have to implement them yourself
using an addOpenWindowListener.

I gathered 2 requests here:
1. embed Firefox (not only Windows, it would be a good candidate for Linux)
2. support popup blocking

Regarding 1.
We are open to help from people who successfully embedded Firefox - a standard 
Firefox build, not one built manually with non standard options. 

Opened a feature request to Mozilla/Firefox to 
https://bugzilla.mozilla.org/show_bug.cgi?id=262453

for HTML rendering. We'll need to see if popup blocking is exposed to 
embedders by the native engine or if we can provide a way to support it 
through the WindowEvent and LocationEvent API.
Comment 5 Benjamin Pasero 2004-10-01 11:32:45 EDT 
Hi Christophe,

one big improvement in the existing Browser widget would be to provide more
information about a WindowEvent in the OpenWindowListener. Currently it does
not make a difference wether a popup or a target=_blank link requested a
new browser window. I would then simply ignore any non-blank calls. This would
block popups by 100%. It's a very strict way, since sometimes new windows
created by Javascript are wanted. But that's something the programmer could
decide then. Therefor I opened a bug report some time ago:
https://bugs.eclipse.org/bugs/show_bug.cgi?id=63233

On the other side, if there is a native way to have popups blocked, I would of
course prefer that way. But popup blocking in the IE came with Servicepack 2 and
is not available on Servicepack 1 and below afaik.

A different approach would be to have a LinkListener. It listens for any Link
the user is clicking. We could then ignore any WindowEvent's, only listen for a
click on a Link and create a new browser if necessary. Of course the LinkEvent
would then have to provide the information where to open it (blank or self).

Ben

Comment 22 Daniel Spiewak 2005-07-27 16:09:30 EDT 
I don't think this is exactly what Ben had in mind.  He seems to be asking for a
substitute Browser backend.  Instead of using IE/Mozilla/Safari as the native
peer, perhaps it would be possible to use Firefox instead.  JRex is cool, but it
looks like it's more of a XUL-For-Java solution and not really an embeding for
Firefox.  Also, JRex is Swing based (at least as far as I could tell).  SWT
*has* to remain independant of Swing and AWT in its core controls otherwise it
can't be used with native cross-compilers.  Also, if we introduce Swing into
core SWT, we add a whole host of problems with event loops, thread deadlock,
inconsistancies in the LAF, etc.  Seeing as Firefox is staticly compiled, I
don't think it's possible to embed it in a separate app without recompilation. 
Unless Mozilla starts offering a dynamically linked version of Firefox in the
future, the only solution I see as viable for this bug is to have a completely
separate control with the GRE included in a separate native library with the
control.

Comment 26 Grant Gayed 2005-09-07 10:21:57 EDT 
The issue with comment 25 is that the Browser needs some way of locating the 
mozilla or firefox that it's going to make use of.  If you don't specify this 
explicitly by setting MOZILLA_FIVE_HOME then the eclipse launcher tries to 
determine this on its own by looking at /etc/gre.conf or /etc/gre.d/gre.conf, 
which points to the distro's mozilla install, if any.  However, the linux 
distros do not provide a similar way of locating firefox installations yet.

So setting MOZILLA_FIVE_HOME is the prescribed way of overriding gre.conf, or 
of handling the case of having no available gre.conf to point to an available 
browser.

Comment 39 Billy Biggs 2005-12-12 13:20:37 EST 
There is a lot of confusion on this bug, and I think the actual problems here are either best described by other bugs. I want to close this bug as INVALID,  but there are an incredible number of votes on this bug!


  ** PEOPLE WHO HAVE VOTED ON THIS BUG, PLEASE READ THIS CARFEFULLY AND ADD
  ** YOUR VOTE TO ONE OF THE FOLLOWING BUGS INSTEAD


  1. IF YOU WANT FIREFOX EMEBDDING UNDER WIN32

  Firefox from mozilla.com does not support embedding, and never will! It is
  built statically to keep the download small and improve performance. Any
  Windows Mozilla solution will require users to either install the full
  Mozilla suite, or the not-yet-released "XULRunner" runtime environment.
  It is not enough to just install Firefox!

  If you still want support for Mozilla embedding under Windows, please vote
  on bug 79213 which is tracking this issue specifically.


  2. IF YOU WANT THE BROWSER TO BLOCK POP-UPS

  Pop-up blocking is an application-level feature and I do not think it is
  built-in to the Mozilla embedding support. Implementing this requires new
  API in SWT to control pop-ups.  Please add your vote to bug 63233.


  3. IF YOU WANT TO USE MOZILLA-FIREFOX ON LINUX INSTEAD OF FULL MOZILLA

  As on Windows, the Linux Firefox download from mozilla.com is built
  statically and does not support embedding.  However, the Firefox packages
  from Red Hat, Debian and Ubuntu are built dynamically and can be used with
  SWT. However, Red Hat has decided not to register their Firefox as an
  embedding location, and Debian/Ubuntu do not register any of their browsers.
  SuSE ships a XULRunner package for embedding, as their Firefox is built
  statically like the one from mozilla.com.

  If you want this to "just work", please file a bug against your distribution.
  See:

    http://bugzilla.ubuntu.com/show_bug.cgi?id=13750
    http://bugs.debian.org/cgi-bin/bugreport.cgi?bug=328471
    http://developer.mozilla.org/en/docs/GRE_Registration


Given the above, and given some of the confusion in this bug, I am unfortunately going to close this as INVALID and ask people to please vote on the other bugs.
Please re-open if you disagree.
