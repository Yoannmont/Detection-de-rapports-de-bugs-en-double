Description Kim Moir 2005-09-13 16:57:48 EDT 
Today, many eclipse.org projects have their own update sites distinct from the
platform update site. In order for updates from non-platform teams to appear as
additional features to the platform there are two options

1) Feature team update sites can be added to the feature.xmls of the platform as
disovery sites. This does not allow updates sites to be added after a product
has shipped.  
OR
2) Feature team jars must be copied to the platform update site and reflected in
the site.xml.  For instance, today  we copy the jars for EMF, GEF, UML2, VE and
WTP to the platform update site.  This results in redundant jars on the server.
 Also, it forces the feature teams to coordinate with the platform releng team
each time that a change is required to their portion the update site. Not very
flexible. 

This is a request for discovery site or similar functionality in the update
manager to reference the urls for multiple update sites within the site.xml.

See discussion in https://bugs.eclipse.org/bugs/show_bug.cgi?id=107514 for the
background on this issue.

This is would be especially useful in the 3.2 timeframe given the "release
train" approach.

Comment 9 Eclipse Webmaster 2006-01-11 16:30:14 EST 
(In reply to comment #8)
> But then there is the problem of mirrors. Currently the mirrors are on a per
> site basis. The VE update site actually has a different set of mirrors than the
> platform site. Don't know why this is so but I do get a different list. So my
> features could possibly require a different mirror.

Not all mirrors have the 115 GB disk space to mirror everything; not all mirrors *want* all the projects; not all mirrors have the bandwidth to support mirroring all projects; as the projects grow in size, some mirrors drop some projects because of space. Mirrors can mirror each top-level project independently, except for platform, which is further broken down to Everything, Updates, Release and Stable.

In this perspective, mixing files from different top-level projects does not mesh well with requesting a list of mirrors for Platform Updates.

download.php supports transparent redirects to a random mirror, so each file could point to download.php?file=somefile.jar&r=1. However, this is not a smart use of mirrors.  If UM could send us the client's country code (in every request?), then we could tweak download.php to redirect to a mirror in the same [country|continent] as the client. See bug 117494 and 99412 comment 15.

D.

Comment 12 Eclipse Webmaster 2006-01-19 21:48:13 EST 
> To absolutely guarantee that a mirror has all the update jars referenced in a
> site.xml should they be stored locally within that update site?

Yes. I am willing to tradeoff disk space to save bandwidth.

* HOWEVER *

If I look at your updates/3.1/site.xml, I see this:

<feature url="features/org.eclipse.team.extras_3.1.0.jar" ... />

You can safely replace all the relative urls for absolute urls like this:
<feature url="http://www.eclipse.org/downloads/download.php?file=/eclipse/updates/3.1/features/org.eclipse.team.extras_3.1.0.jar&r=1" ... />

And the mirrorsURL attribute in site.xml becomes obsolete - each JAR would be fetched from a random mirror (or download.eclipse.org) without any user intervention whatsoever.

As per bug 123962, download.php will soon be using the http client's IP address to detect the client's country code automatically (assuming countryCode is not defined), which will make the mirror selection process obsolete across www.eclipse.org. This will be implemented and tested shortly.

D.
