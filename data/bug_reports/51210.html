Comment 3 Enrique 2005-09-16 05:14:13 EDT 
I`m using Eclipse 3.1 with EMF 2.1, 
and RSA (I think based on Eclipse 3.0) with EMF 2.0

The xsd:choice element isn`t even contained in the ecore model.
The following example ->

<?xml version="1.0" encoding="UTF-8"?>
<xsd:schema xmlns:this="http://XMLValidatingTest/XML-Schema" 
			xmlns:xsd="http://www.w3.org/2001/XMLSchema"
			xmlns:ecore="http://www.eclipse.org/emf/2002/Ecore" 
			ecore:documentRoot="VehicleExperiment">	
	<xsd:complexType name="VehicleType">	
		<xsd:choice>
			<xsd:element name="car" type="xsd:string"/>
			<xsd:element name="motorcycle" type="xsd:string"/>
		</xsd:choice>
	</xsd:complexType>
	<xsd:element name="vehicle" type="VehicleType"/>
</xsd:schema>

produces ecore eAttributes like:

<eStructuralFeatures xsi:type="ecore:EAttribute" name="car" unique="false"
       eType="ecore:EDataType http://www.eclipse.org/emf/2003/XMLType#//String">
      <eAnnotations    
              source="http:///org/eclipse/emf/ecore/util/ExtendedMetaData">
        <details key="kind" value="element"/>
        <details key="name" value="car"/>
        <details key="namespace" value="##targetNamespace"/>
      </eAnnotations>
    </eStructuralFeatures>
    
    <eStructuralFeatures xsi:type="ecore:EAttribute" name="motorcycle" 
       unique="false" eType="ecore:EDataType 
       http://www.eclipse.org/emf/2003/XMLType#//String">
      <eAnnotations 
              source="http:///org/eclipse/emf/ecore/util/ExtendedMetaData">
        <details key="kind" value="element"/>
        <details key="name" value="motorcycle"/>
        <details key="namespace" value="##targetNamespace"/>
      </eAnnotations>
</eStructuralFeatures>

But there are no entries for xsd:choice. If I understood the document "XML
Schema to Ecore Mapping" well, then something like this should be there:
        <details key="exclusion" value="motorcycle">  <!-- for car and    -->
        <details key="exclusion" value="car">         <!-- for motorcycle -->

Comment 11 Jeff Norris 2008-06-03 16:50:38 EDT 
We're fairly new users of EMF and we'd love to see this feature added.  It seems reasonable for a class containing an xs:choice-like structure to either:

1.  Infer a common interface or superclass for the choice value and provide a single setter/getter with perhaps a separate method to ask for the type of the value.  (perhaps the most complicated but best approach)

2.  Stick with the multiple set/get methods but set the other values to null when a setter is called with a new value.

-or-

3.  Throw a runtime exception if you attempt to set two different kinds of values on a xs:choice element.
Comment 12 Ed Merks 2008-06-03 18:09:04 EDT 
The problem is that a choice as simply one of a mutually exclusive set of elements is just one special case use of choice so special case solutions don't have a lot of appeal.  Inducing a common base class isn't so nice if you have a choice of float, int, and string say, because there'd useful type safety in the API.

It also seems a bit too tricky to try to enforce correctness.  For example, if you are only allowed two elements, we don't prevent you from adding a third, because your next operation might be to remove one.  I.e., intermediate invalid states are common.

And to top off all the complexity, XML Schema 1.1 is making some changes to make all this even more complex, e.g., <xsd:any> at any level of nested.  :-(

Comment 15 Philipp W. Kutter 2009-02-11 15:50:13 EST 
In my understanding there are three initial steps:

1. If generating ECore from XSD, keep information about choices constraints in  EAnnotations
   -> This allows to use this information for any software based on ECore.
 
Based on 1, there are two possible next steps:

2) Improve the ECore to XSD exporter, to reestablish the choices, that where in the original Schema

3) Improve the generated Validation such, that it shows if the choice constraints are not followed

Then, based on 3) one can improve the editor:

4) Improve Editor such, that it proactively supports the creation of contained elements in a way that follows the choice constraints.


Even if 4) is complex, starting with 1) and 2) would be great, would extend the usability of EMF for XML-Schema development, and then possibly create more momentum for 3 and 4

What do you think?

Regards, Philipp

Comment 18 Ed Merks 2009-02-12 05:47:49 EST 
The editor currently does nothing to enable to disable commands based on constraints, so even if a multi-valued feature has an upper bound of 4 values, the user can create more than that.  All the constraints are checked only when the user invokes Validate...  The transaction framework supports checking if constraints are violated and it supports rollback for that case.  GMF makes use of this...

One significant concern I have is that XML Schema 1.1 changes some of the behavior of particles so I can expect clients will eventually want to see corresponding changes in XSD and then in EMF.
