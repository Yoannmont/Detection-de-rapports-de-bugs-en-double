Comment 4 Darin Wright 2007-09-11 10:29:59 EDT 
The PDE tab group performs a backwards compatibility check each time a configuration is displayed in the tabs. The config is updated to the latest format and saved even if there are no changes to save. When I checked out the config Dani mentioned, I saw an outgoing change in the XML header of the config, even though no config attributes changed. I'm not sure if this is due to a different XML writer?

However, this can be avoided by testing if the config is dirty (has any changes) before saving it.

Comment 8 Brian Bauman 2007-09-11 15:58:21 EDT 
I think Darin found a niche in the launcher area :)

Dani, this should solve the original issue.  In order to make the launch configuration more complete, I also suggest you use "add required plug-ins" button to add the additional plug-ins required to make the org.eclipse.jdt.ui.tests plug-in run successfully.

As for the change in the workspace tag when running a config with a "deselected" project listed which is not currently in the workspace, this is a current limitation.  A shared launch configuration will require some level (as minimal as possible) of workspace configuration.  If the target platforms are not the same, you may get different results.  Similarly, if the projects in your workspace are not the same, you may get different results.  To enhance portability as much as we can, we do have the "Add new workspace plug-ins to this launch configuration automatically".
