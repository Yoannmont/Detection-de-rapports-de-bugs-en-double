Description Thomas Watson 2007-10-12 11:23:44 EDT 
I am concerned about the amount of public API in the org.eclipse.equinox.util bundle.  It looks like a large portion of the packages exported by this bundle are only imported by the DS implementation.  Can we consider moving these packages into the DS implementation as internal packages?  This would reduce the number of real API packages down to 5 packages from the org.eclipse.equinox.util bundle.  From an API point of view this would help to reduce the amount of scrutiny needed to review a large amount of public API.

The following packages look like they are used by more than one bundle:

 org.eclipse.equinox.util.hash;version="1.0",
 org.eclipse.equinox.util.pool;version="1.0",
 org.eclipse.equinox.util.ref;version="1.0",
 org.eclipse.equinox.util.time;version="1.0",
 org.eclipse.equinox.util.timer;version="1.0"

Can the rest be moved to DS as an implementation detail?
Comment 1 John Arthorne 2007-10-12 11:31:01 EDT 
Dare I ask, what would be the distinction between equinox.util and equinox.common?
Comment 2 John Arthorne 2007-10-12 11:33:08 EDT 
I should add: +1 for anything that reduces public API. Another option is to leave things in util, but mark the util packages with x-friends. That would allow sharing and reduced duplication between equinox bundles, but reduce the amount of public API we must support indefinitely.
Comment 3 Chris Aniszczyk 2007-10-12 11:53:24 EDT 
or move things to common that make sense?

I don't like "util" bundles or x-friending things unless necessary.
Comment 4 Thomas Watson 2007-10-12 12:23:05 EDT 
(In reply to comment #1)
> Dare I ask, what would be the distinction between equinox.util and
> equinox.common?
> 

Yes, this is a concern.  If we can determine that the packages left in equinox.util are generally useful then we should consider moving them into equinox.common.  But I'm not a fan of this unless we think other bundles outside the prosyst contributed bundles will use the package.  (In reply to comment #2)

> I should add: +1 for anything that reduces public API. Another option is to
> leave things in util, but mark the util packages with x-friends. That would
> allow sharing and reduced duplication between equinox bundles, but reduce the
> amount of public API we must support indefinitely.
> 

This is another option.  But the name org.eclipse.equinox.util is still a source of confusion with org.eclipse.equinox.common.  Maybe we need a directive on Bundle-SymbolicName to declare x-internal/x-friends?  For example, specifying Bundle-SymbolicName: org.eclipse.equinox.util; x-internal:=true would indicate that the bundle should not be used for Require-Bundle, only Import-Package should be used to access the publicly exported packages.  This way the BSN org.eclipse.equinox.util has less meaning from an API point of view.
Comment 5 Jeff McAffer 2007-10-14 12:04:52 EDT 
How big is the code you are talking about being left in util and how much reuse is there (i.e., how many other people use it)?  What would happen if we just copied the code to all the places that it is used?  This is not particularly desirable but may be the right direction.  For example, we have in the past resisted making common/util bundles by copying the various classes (sometimes quite a few times).
Comment 6 Pavlin Dobrev 2007-10-14 15:01:54 EDT 
The code that is donated comes from the embedded world and when we make it our idea was not to have any repletion in our bundles in order to kept complete OSGi implementation small. Because only few services were donated to equinox some packages now may be used only in one additional bundle.
I removed any util code that is not used by the other bundles prior donating to equinox and currently in util remains only used packages by other bundles.

Restructuring the code also depends of the moving the code from incubator to the Equinox release. If not all of the bundles/services came to Equinox we can put util packages required only for these bundles. At this time I see that only the DS implementation is used and no have any comments for other bundles. 

A possible option is to deside to make DS implemetation completle independent from util bundle (e.g including all required classes inside the DS).

I agree we should do something with the public API in util bundle. 
I reviewed the packages it exports and I think that some of them can be removed or at least not exported.

It is an option to move to DS bundle some of the packages, but still we have some packages that will stay in the util bundle. 

I would prefer the exported packages by util bundle to be marked as "x-friends". 

Let me summarize what could be done:
1. Completely remove or just remove from the exported packages some of the util's bundle packages 
2. Move some of the packages to DS bundle.
3. Make some code changes in the bundles using util bundle to avoid usage of some of its packages (e.g. org.eclipse.equinox.util.time, org.eclipse.equinox.util.pool)
4. The end result would be that util bundle will have to export only these packages:
 org.eclipse.equinox.util.hash;version="1.0",
 org.eclipse.equinox.util.ref;version="1.0",
 org.eclipse.equinox.util.timer;version="1.0"

They will have x-friends attributes as well.

Comment 11 Thomas Watson 2007-11-30 17:31:07 EST 
(In reply to comment #10)
> Personally I do not like "2. Move some of the packages to DS bundle." mainly
> because the purpose of the code in the util bundles is to provide common
> utilities and these utilities can be used by other bundles than the DS.
> 

It is my understanding that the only classes that would be moved into DS would be the ones which are *only* used by DS.  When you say "other bundles than the DS" do you mean other bundles in the Equinox project, or other bundles provided by another vendor?

I agree that it would be good to have widely used classes in a central spot where all can benefit.  But the level of commitment to support the API is quite large and API is generally hard to evolve over time.  For now we should put x-internal or x-friends on all the util packages because the risk of supporting the packages as API in the long term is probably too high.  In the future we can consider moving selective packages to the common bundle as we see the need.

If a package has a x-friends directive which only lists one friend and that package is not used internally by the exporting bundle, then it makes sense to move the package to the friend bundle.

org.eclipse.equinox.internal.util.security - used internally from the Util bundle (org.eclipse.equinox.util.ref)


org.eclipse.equinox.util.hash - used by DS, IO, Wireadmin
org.eclipse.equinox.util.event - used by DS and internally from the Util bundle (org.eclipse.equinox.util.ref)

org.eclipse.equinox.util.threadpool - used by DS
