Comment 1 Oleg Besedin 2007-09-12 14:13:18 EDT 
Created attachment 78215 [details]
Initial patch - manifests and settings

The only somewhat substantial code change in the patch is the change AuthApplication to use "modern" IApplication. There are few other code changes in ui and sample bundles but those are very trivial.

- I added Equinox formatting and build settings to all bundles (http://www.eclipse.org/equinox/documents/coding.php). 
- BUT: To avoid problems in code merges I turned off autoformatting on save for all bundles except .sample and .ui.default (code in those two bundles are reformatted in this patch.)
(Setting is in the Project properties -> Java Editor -> Save actions)
- formatted code in the org.eclipse.equinox.security.sample and org.eclipse.equinox.security.ui.default bundles; cleaned up a few trivial things while at it
- all manifests: added .qualifier to the bundle version; exported packages; added 1.4 as the min EE
- manifests and JUnits: removed x-frinend:o.e.e.s.junit from the internal packaged on the .auth bundle; replaced with x-internals; modified options on the test bundle: Java compiler -> Errors/Warnings -> Deprecated and restricted API -> Discourage access -> Ignore

TBD: Renames
- add ".internal." to package name for internal classes
- would it be worth to rename ".junit" bundle into ".tests" bundle like others in SDK?

TBD: Code formatting 
- all bundles (aside from ui and sample) need to have code formatted

Comment 5 Oleg Besedin 2007-09-18 15:14:18 EDT 
Created attachment 78683 [details]
Patch

This new patch does code formatting and cleanup for thw .boot, .junit, and provider bundle.

Note that this patch reverses two changes recently added to CVS:

- removed is the addition of ".jre15x" to the bundle symbolic name of .boot fragment and its project file
Reason: while such structure shoudl work fine in principle, it does not work well with PDE - too many edge conditions I guess.

- jdt core preferences I restored this one as a reminder:
org.eclipse.jdt.core.compiler.problem.unsafeTypeOperation=warning
Once we clear on how 1.4 vs 1.5 question works out the code either will be udpated for 1.5 (having warning is a plus there) or compiler setting will be set to 1.4 (warning would disappear).
