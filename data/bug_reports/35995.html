Comment 1 Deven T. Corzine 2003-04-02 16:47:26 EST 
Nevermind, I found the problem.  I thought I was using Sun's 1.4.1 JRE, but
after filing this bug, I noticed "libgcj.so.3" sprinkled throughout the stack trace.

I forgot to make the "jre" symlink in the eclipse directory to point to the JRE
to use, and by default it found the GCJ one which obviously isn't functional
enough to run Eclipse.  The old M5 release, on the other hand, had the link, so
it worked fine.

Two thoughts:

(1) Is there a better way to specify the JRE that would survive upgrades?  (Set
the JAVA_HOME environment variable?)

(2) Could GCJ be supported just enough to warn the user that they may need to
use Sun's JRE instead, if such a crash occurs?
Comment 2 Rafael Chaves 2003-04-02 17:09:54 EST 
Answering 1):

the best way to run Eclipse is specifying explicitly which VM to use. For example:

eclipse -vm /opt/j2sdk1.4/bin/java

Please check the platform user guide, tasks, Running Eclipse for more
information about command line options. Unfortunately, you will only be able to
see it after Eclipse is running. There is a feature request regarding a command
line -help option (bug 32110) for the Eclipse launcher.

Regarding 2), I would say that it is not practical, because there are other VMs
that are not capable of running Eclipse.
Comment 3 Deven T. Corzine 2003-04-02 17:25:01 EST 
(1) Thanks for the tip.  I'll use that from now on.  (Does the README mention it?)

(2) Would it be possible to simply mention the JVM in use, when a fatal error
occurs?  If it's the right one, it's extra information that can be ignored; if
it's the wrong one, it might grab one's attention...
