Here is a snippet that I would have expected to work.  However, I get a

public static void main(String[] args) {
	Display display = new Display();
	Shell shell = new Shell(display);

	createControl(true, true, shell, 10, 10);
	createControl(true, false, shell, 10, 200);
	createControl(false, true, shell, 200, 10);
	createControl(false, false, shell, 200, 200);
	
	shell.open();
	while (!shell.isDisposed()) {
		if (!display.readAndDispatch())
			display.sleep();
	}
	display.dispose();

static void createControl(boolean header, boolean lines, Composite parent, int 
x, int y) {
	Table table = new Table(parent, SWT.BORDER);
	table.setHeaderVisible(header);
	table.setLinesVisible(lines);
	TableColumn column1 = new TableColumn(table, SWT.NONE);
	TableColumn column2 = new TableColumn(table, SWT.NONE);
	for (int i = 0; i < 20; i++) {
		TableItem item = new TableItem(table, SWT.NONE);
		item.setText(new String[] {"Item " + i, "jjjjjadadadads"});
	}
	column1.pack();
	column2.pack();
	
	// show four rows
	int height = table.getItemHeight(); // should not include gridlines
	int grid = table.getGridLineWidth(); // should return space between 
rows even if no line drawn there
	
         // use 60 for width to get a vertical scrollbar
	Point p = table.computeSize(60, 4 * height + 3 * grid); // should 
account for headers and any trim

	table.setSize(p); 	table.setLocation(x, y);
}


The results I get are:

Windows:
Size returned is the same for all cases.  With headers, can only see three 
rows and in the last row the "j" is cut off.  Without headers, you can see 
four rows plus a few pixels of the fifth row.  If I remove the need for a 
horizontal scrollbar, with headers, you can see four rows exactly and no 
more.  Without headers you can see five and a half rows.

WIn32 Conclusion - computeSize is always returning the size with headers and 
without scrollbars.

GTK:
Size returned is the same for all cases.  With headers, you can only see one 
and a half rows.  With no headers, you can see two and three quarter rows.  If 
I remove the need for a horizontal scrollbar, size is the same as reported 
before.  With headers, you get two and a half lines visible, without headers 
you get four rows but the "j" is cut off.

GTK Conclusion: computeSize is always returning the size without headers and 
without scrollbars but is short a few pixels.


Motif:
Size varies depending on the presence of headers.  In all cases, get four rows 
plus a few pixels of the fifth row.
If I remove the need for a horizontal scrollbar, get five and a half rows.

Motif Conclusion: computeSize is taking into account the presence/absence of 
headers and is always including the size of a scrollbar but is adding a few 
extra pixels.

We need to decide what the correct answer is and make them all consistent.

Comment 10 Felipe Heidrich 2004-01-08 18:11:44 EST 
Martin, the right code is:
public static int getTableHeightHint(Table table, int rows) {
	return table.getItemHeight() * rows + table.getHeaderHeight();
}

Notes: table.getHeaderHeight() returns zero if the header is not visible. 
ComputeSize always adds space for borders and of the scrollbar around the hint 
you passed in (even when scrollbars are not necessary).

windows was not working correctly... it is working fine for me).
