Comment 1 Martin Oberhuber 2006-12-12 09:21:26 EST 
Dave - this is high priority for us. If you're not too swamped with 1.0.1 today it would be cool if you can have a look.

Comment 2 David McKnight 2006-12-12 12:54:06 EST 
I've committed a change that makes this faster although for large selections (i.e. 1000+) there still needs to be improvements.
Comment 3 David McKnight 2006-12-13 08:38:35 EST 
It looks like a very expensive part of this operation is in processing that occurs as a result of the property sheet getting notified of the selection change.     The RSE view adapter for each selection is called to determine the number of children underneith each - but to do that, we end up having to search the tree to find each corresponding tree item.  I've optimized this a bit generically and specifically for IRemoteFile.  Rather than relying on the tree to determine the children count, we determine the children cound from the cache.  This is much faster.  This still doesn't solve the slow part of bringing up the popup but it's another step in the right direction.
Comment 4 Martin Oberhuber 2006-12-13 08:49:49 EST 
When doing (select first) -> (shift-select last), shouldn't there only be a single SelectionChanged notification? And, as a result, the Property Sheet be computed exactly once only, for the last item only?

When I do the operation as a user, I only see the properties of the last item so I'd wonder why RSE would want to compute properties of the nodes "in between".
Comment 5 David McKnight 2006-12-13 09:34:43 EST 
When there is a multiple selection, property sheet entry does a refreshValues on the entire selection, not just the last one:

private void refreshValues() {
		// get our parent's value objects
		Object[] currentSources = parent.getValues();

		// loop through the objects getting our property value from each
		Object[] newValues = new Object[currentSources.length];
		for (int i = 0; i < currentSources.length; i++) {
			IPropertySource source = parent
					.getPropertySource(currentSources[i]);
			newValues[i] = source.getPropertyValue(descriptor.getId());
		}

		// set our new values
		setValues(newValues);
	}
Comment 6 David McKnight 2007-01-26 09:30:19 EST

Comment 7 Michael Valenta 2007-05-01 15:51:27 EDT 
I just tried a delete of a large number of files and it appears to have blocked the UI. A progress dialog is up and has a Cancel button but it does not respond to mouse clicks. I will attach a stack trace that indicates that the server is being contacted in the UI thread and the thread appears to be blocked on a remote call. In the very least, you should be using the IProgressService to fork the work to a background thread so cancellation is responsive. Ideally, you could use the Job framework so that the user is not blocked by the operation.
