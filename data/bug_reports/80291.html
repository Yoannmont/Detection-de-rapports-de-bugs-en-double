Comment 9 John Arthorne 2004-12-08 15:42:02 EST 
Users of editors care about creating an IEditorInput that they can pass to
IWorkbenchPage.openEditor(...).  The text editor framework cares mainly about
documents.  The org.eclipse.ui.editors.documentProviders extension point
decouples this relationship between inputs and documents. Anyone can plug in a
document provider that knows how to map a given input to a corresponding document. 

The org.eclipse.ui.editors plugin supplies document providers for
IStorageEditorInput and JavaFileEditorInput (which is not API).  It feels like
the editors plugin just needs to add a document provider for the existing
IPathEditorInput.  It would also help for someone (perhaps workbench) to provide
a concrete implementation of IPathEditorInput as API (which is essentially what
JavaFileEditorInput is). This is a common enough case that it doesn't make much
sense for everyone to roll this themselves (or copy the existing implementation
from the ui.editors plugin).
Comment 10 Nick Edgar 2004-12-08 16:00:34 EST 
I may not understand the problem, but fortunately it sounds like John does.

> It is specifically the "getPath()" call in FileEditorInput that is returning
null.  

As Darin said, this is because they're using FileEditorInput incorrectly.  

> Either FileEditorInput cannot be an IPathEditorInput (since it relies on
IFile, which must exists within the workspace) or IPathEditorInput is specified 
incorrectly. 

FileEditorInput can be an IPathEditorInput.  Any properly-specified
FileEditorInput should return the file-system path for the file via its
IPathEditorInput.getPath() method.  I believe IPathEditorInput is specified 
correctly.  But we probably could benefit from a concrete implementation, as
John suggests.
