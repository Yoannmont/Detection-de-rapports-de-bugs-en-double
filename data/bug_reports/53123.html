Description Jim Wright 2004-02-25 16:20:11 EST 
While porting merge unit test #3 from the CLI test suite to the t-client JUnit 
suite, I found a bug which appears to be an undetected flaw in the core 
MergeManager code.  (I first assumed this was a bug in my just-written code, 
since the MergeManager code has been around for quite a while; investigation 
proved otherwise).

** Test sequence

(The original test is in the Ant script ".../stellation.repos/test/03.xml".)

The problem concerns an artifact 'file13' which is moved in one branch,
deleted in a second branch, and which then reappears as a "broken CA link"
when the two branches are merged.  By 'broken CA link", I mean that
the CA contains <filename,ID> entry for the artifact in question, but the
actual artifact is missing from the BranchImage artifactMap.

Test sequence (excerpted from complete unit test)

1. Create main branch and populate with artifacts.
Artifact 'file13' in ./dir1 with ID=16 is created here.
Artifact 16 'file13' is listed as a member of CA #21 'dir1'.
The local image is checked in to become main@@1.

2. main@@1 is checked out into a separate folder, and forked
as branch 'new'.  Artifact 16 'file13' is moved from folder ./dir1 to 
folder ./dir2.  Artifact 16 is now a member of CA #9 'dir2'.
Artifact 16 is still listed in the BranchImage artifact map.
The local image is checked in to become new@@1.

3. main@@1 is now checked out into a 3rd folder.
Artifact 16 is deleted in this workspace (removed from CA #21 'dir1',
and from the local Branch Imageartifact map.
The local image is checked in as main@@2

3. main@@2 is now checked out into a 4th folder.
Artifact 16 is not present (in either the file system or the
local BI).

4. Branch new@@1 is merged into the local image (based on main@@2).
The "merge results" image is flawed: 
* The 'dir2' CA (#9) lists artifact 16 as a member.
* The BI artifactMap has no entry for artifact 16.
Result: an NPE while attempting to update the local filesystem
and workspace from the merge results BI.

** A closer look.

// TA16 denotes the Text Artifact with ID 16 and name 'file13'.
// CA9 denotes the Compound Artifact 'dir2'.
// CA21 denotes the Compound Artifact 'dir1'.

I spent some time tracing through MergeManager.mergeChanges 
(in ...stellation.repos).
On entry to mergeChanges, the following was observed:
- the target BI is based on main@@2.  It does not contain TA16 or any 
references to TA16.
- the 'changed' BI is based on new@@1. It contains TA16 (in artifact map);  CA9 
includes TA16 as a member.
- The target BI has a later timestamp than the 'changed' BI.

The set of 'common' artifacts is correctly determined around line 172 of 
MergeManager.mergeChanges.  This set includes CA9 and excludes TA16 (both are 
correct).

Next, differing versions of the 'common' artifacts are merged to form a single 
set, drawing on artifact versions contained within the target image, changed 
image and ancestor image (based on main@@1).

The resulting set is contained by the 'resultArtifacts' ArtifactMap, which is 
fully populated by line 192.
NOTE:
* resultArtifacts contains a version of CA9 which includes TA16 as a member.
* resultArtifacts does NOT contain TA16 itself.

The version of CA9 contained by resultArtifacts is taken from the 'changed' 
(new@@1) image.  Selection of this version of CA9 is performed by the 
CompoundArtifactAgent merge method.   

From this point on, the broken link inside CA9 remains; the actual artifact 
TA16 is never added to the artifactMap for the branch image.

----------

Subsequently, an NPE occurs while attempting to update the local filesystem 
from the merge-results BI; this occurs because when adding an <id,artifact> 
pair from the merge-results image to the internal 'tcore' ScmResourceMap, which 
checks that the artifact reference is not null.  Apparently, the CLI code did 
not perform a similar null-value check when using the merge-results BI.  I 
assume this is why the MergeManager flaw did not manifest in the CLI version of 
this unit test (03.xml).

Suggested resolution:
B. This situation (deletion in one, relocation in another) should be treated as 
an ambiguous conflict and punted to the user for manual resolution.  Fixups 
should be performed to restore any artifacts referenced by a CA, but missing 
from the artifact map bound to the merge-results BranchImage.

Comment 13 Jim Wright 2004-03-22 17:53:09 EST 
There is still at least one flaw in the CompoundArtifact merge.

Test case (test 3 of original Ant/CLI test suite):
* A TextArtifact is originally named 'file3' and located in the root project 
dir.
* In the leaf branch, this artifact is both moved and renamed.
* In the main branch, this artifact is renamed, but not moved.
When merge is performed, a conflict should be reported.  However, only a change 
is reported (message: "Resource renamed in target....").

This conflict should be detected CompoundArtifactAgent.merge.
In the existing code, there is a section which begins:

if( inTarget && ! inLeaf )
   ...
   ...
   if(! leafBranchContainsArt) {
      ... logic to handle 3 cases (target modified, target renamed, no target 
change)
   }

However, there is no ELSE for if(! leafBranchContainsArt).
Thus, no confict is raised if the artifact exists elsewhere in the leaf image 
(and there is no way to determine if the artifact was moved AND renamed in the 
leafImage.

Similarly, CompoundArtifactAgent.merge contains code for:
   if( inLeaf && ! inTarget )
but has no ELSE clause for if( ! targetBranchContainsArt ).
Thus, the conflict is not detected.

One possible fix is to raise a conflict when the artifact a) exists in leaf but 
not target CA AND b) exists elsewhere in the targetImage AND c) has been 
renamed in the leafCA.  Conditions a and b imply a move, while c indicates a 
rename.  Thus, I'm suggesting that a combined move-and-rename should always 
raise a conflict requiring manual resolution.

It would probably be better to only raise the conflict if the corresponding 
target artifact has also been renamed.  However, there's no way to determine 
that here (within CompoundArtifactAgent.merge), because the complete set of 
name bindings for the leaf and target images is not available.

Note also that no ChangeNotice is generated for any case within if( inLeaf && ! 
inTarget).  There should probably be a ChangeNotice for the case:
   inLeaf && (! inTarget) && targetContainsArt && (! leafName.equals(baseName)).

This compound condition is true when the artifact has been moved (! inTarget) 
and renamed (! leafName.equals(baseName)) and still exists in the target (with 
the original or yet a 3rd name, and possibly moved to a different location as 
well).

ChangeNotices may be appropriate for other cases as well.

Ideally, this type of conflict (rename in target, move AND rename in leaf) 
could be detected and labeled accurately in mergeBaselines.  This might be 
feasible by using the mergeConflicts list as one of the inputs to 
mergeBaselines, and doing some extra checks for any artifact contained on that 
list (and possibly any artifact contained on the mergeChanges list as well).

Comments ?
Comment 14 Mark C. Chu-Carroll 2004-03-23 10:28:30 EST 
I'm confused by this latest problem. "Moved but not renamed" is a meaningless phrase in terms of the 
semantics. In terms of semantics, we've defined "renamed" as "the binding to the resource in a folder 
was preserved, but the name associated with that binding was modified", and "moved" as "the binding 
to the resource in its original compound was removed, and a new binding to the resource was created 
in a different compound". 

In the "moved" case, the name of the binding is irrelevant. Whether it's the same name that it had in 
the original compound, or an entirely different name, it doesn't matter: it's a binding delete from one
compound, and a binding add to another.
Comment 15 Jim Wright 2004-03-23 11:32:23 EST 
OK, I understand now that "Moved but not renamed" is meaningless in terms of 
the intended semantics.

However, we still have the following potential conflict:
* Renamed in target AND moved in leaf
(or, if you prefer, a binding modified (rename) in one branch and a binding 
removed/created (move) in another branch).

which is not currently detected or reported.

I'm assuming we want to change CompoundArtifact merge to issue appropriate 
conflict notices, so that a name change is not simply lost when the associated 
artifact has a different parent in the other different branch.

The current version of CompoundArtifactAgent.merge identifies four possible 
cases:
resource (artifact) is bound in both target and leaf, bound just in target, 
just in leaf, or neither). A number of subcases are considered for each of the 
four major cases.  However, several subcases are not considered:
* inTarget, ! inLeaf, leafBranchContainsArt, targetName != ancestorName
* inLeaf, ! inTarget, targetBranchContainsArt, leafName != ancestorName.

I suggest conflicts and change messages should be issued as follows (in 
addition to the actions already taken by CompoundArtifactAgent.merge.)  Note: 
Cases 1-4 assume the bound artifact is also in the ancestor CA; case 5 assumes 
the bound artifact is only in the leafCA.

Case 1: artifact is inTarget only, targetName == ancestorName, 
leafBranchContainsArt
Action: None (Leaf binding is used in resulting image)  
[already handled]

Case 2: artifact is inTarget only, targetName != ancestorName, 
leafBranchContainsArt
Action: Issue conflict: "Artifact renamed in target, moved in leaf" 
[new code needed]

Case 3: artifact is inLeaf only, leafName == ancestorName, 
targetBranchContainsArt
Action: Assume artifact moved in target.  Discard leaf member binding from 
resulting CA.
[already handled]

Case 4: artifact is inLeaf only, leafName != ancestorName, 
targetBranchContainsArt
Action: Issue conflict: "Artifact moved in target, renamed in leaf"
[new code needed]

Case 5: artifact is inLeaf but not target or ancestor, targetBranchContainsArt
Action: Assume artifact moved in leaf.  Accept change (and issue ChangeNotice).
If this is part of a case-2 conflict, the conflict will be raised when the 
corresponding target CA is processed.
[already handled]

If you agree, I'll go ahead and add code cases 2 and 4.
Comment 16 Mark C. Chu-Carroll 2004-03-23 12:21:02 EST 
OK. I think I see the problem now...

The name of the artifact is not the issue. But there are some missing cases... And I think those are 
coming from the fact that the fix that I sent is really just part1; I wanted to debug this much before 
adding one more messy bit. The missing bit is the code to ensure that there is no more than one 
binding to any artifact.

I think that my idea was to handle this kind of conflict by having the compound merges keep *both* 
bindings. So if compound merge sees a rename in one of target and leaf for a compound, it doesn't 
really care about what bindings there are in the other, so long as it wasn't deleted. It should just put the 
updated binding into the result compound. 

Then, after all the artifact merges are done, there should be another pass, that ensures that there is no 
more than one binding per artifact. If there is, then it should signal a conflict, and drop one of the 
bindings. (In theory, it's non-deterministic choice which one gets dropped; in practice, I think that the 
binding from the target should take precedence.

I'm not going to go check my code to see if this whole theory is right, because you've mentioned that 
you've made a lot of changes to it, and I don't want to cause any more merge headaches! But if you 
could post the latest code as an attachment here, I'll take a look, check that my theory is correct, and 
add in the code to do the single-binding check.
