 public static void main(String[] args) {

    final Display display = new Display();
    final Shell shell = new Shell(display);
    shell.setLayout(new FillLayout());

    final Tree tree = new Tree(shell, SWT.H_SCROLL | SWT.V_SCROLL | SWT.VIRTUAL | SWT.BORDER | SWT.MULTI
        | SWT.FULL_SELECTION);

    tree.setHeaderVisible(true);
    tree.setLinesVisible(true);

    final int columns = 20;

    for (int i = 0; i < columns; i++)
      new TreeColumn(tree, SWT.LEFT).setWidth(150);

    tree.addListener(SWT.SetData, new Listener() {
      public void handleEvent(Event event) {
        TreeItem item = (TreeItem) event.item;
        int index = event.index;
        if (index == 15) { // <--- change to 0 to see AIOOE
          item.dispose();

        if (parentItem == null) {

          item.setText(i, text);
        }
        item.setItemCount(55);
      }
    });
    
    
    int itemCount = 15;
    tree.setItemCount(itemCount);
    for (int i = 0; i < itemCount; i++)
      tree.getItem(i).setItemCount(20);
    
    tree.getItem(0).setExpanded(true);
    
    
    

    final int timerDelay = 2000;

    final Runnable r = new Runnable() {
      public void run() {

Comment 2 Steve Northover 2006-09-21 12:51:39 EDT 
The problem is that the HITEM has been disposed and the handle is no longer valid but Windows is in the middle of calling in for custom drawing.  The HITEM field in the Windows custom draw struct is not updated by Windows.  We use it, it addresses memory and we get back garbage to use as an index.

NOTE:  Make sure to change the index in the sample code to see the AIOOE.

Comment 5 Steffen Klössel 2006-09-29 02:46:38 EDT 
Hi Steve,

it seems so that "my" testcase does'nt match to that bug. My testcase throws a java.lang.ArrayIndexOutOfBoundsException and this bug (131209) crashes my VM because of problems in COMCTL32.dll (under XP).

Furthermore I solved "my" problem (159136)! Yesterday I descriped the solution within the bugreport but I did a mistake. Maybe I don't understand enough about SWT internals to understand why you think thats the same problem !? Hope you can explain it to me.

My problem is based on two unchecked accesses to the this.items field by _getItem(..). When a custom draw signal overlaps the SetData event that reduces the length of this.items field, the event-data of the custom draw signal contains an item index thats higher than the array length of this.items.

I found two locations in Table.java where methods used by the custom draw system accessing this field by _getItem(...). I think we have to avoid to check the access with in _getData(..) -> performance. 

Here my solution (SWT 3.2)
First location:
Table.java, line 447:
		if (hooks (SWT.PaintItem)) {
			TableItem item = _getItem (nmcd.dwItemSpec);
			sendPaintItemEvent (item, nmcd);
				//widget could be disposed at this point
		}

replaced with :
		if (hooks (SWT.PaintItem)) {
			if (nmcd.dwItemSpec < items.length) {
				TableItem item = _getItem (nmcd.dwItemSpec);
				sendPaintItemEvent (item, nmcd);
				//widget could be disposed at this point
			}
		}


Second location:
Table.java, line 467:
	TableItem item = _getItem (nmcd.dwItemSpec);
	if (item == null) return null;

replaced with :
	TableItem item = nmcd.dwItemSpec < items.length ? _getItem (nmcd.dwItemSpec) : null;
	if (item == null) return null;


Tank you

Comment 7 Viliam Durina 2008-09-10 02:06:20 EDT 
Here is a short snippet to reproduce the problem with Table. Run it and scroll down quickly.

	public static void main (String [] args) {
		Display display = new Display ();
		Shell shell = new Shell (display);
		shell.setLayout(new FillLayout());
		
		final Table table = new Table(shell, SWT.BORDER | SWT.VIRTUAL);
		table.setItemCount(100);
		table.addListener(SWT.SetData, new Listener() {
			public void handleEvent(Event event) {
				try {
					Thread.sleep(50);
				} catch (InterruptedException e) {
				}
				if (event.index < 50)
					((TableItem)event.item).setText("Item " + (event.index + 1));
				else
					// Decrease the item count, as we find out the real
					// number of rows only after we are after the end
					table.setItemCount(50);
			}
		});
		table.addListener(SWT.PaintItem, new Listener() {
			public void handleEvent(Event event) {
			}
		});
		
		shell.setSize (200, 200);
		shell.open ();
		while (!shell.isDisposed()) {
			if (!display.readAndDispatch ()) display.sleep ();
		}
		display.dispose ();
	}


Note the Thread.sleep(50) inside of SetData event: if it were not there, it would not reproduce (or you would have to scroll very quickly).

To reproduce with Tree, just replace Table for Tree and TableItem for TreeItem.
