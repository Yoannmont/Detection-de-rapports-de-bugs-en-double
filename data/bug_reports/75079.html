Description Jeff McAffer 2004-09-27 10:00:27 EDT 
I am developing an RCP app that can be deployed many different ways and with 
many different features which may appear on update sites.  Each deployment 
structure is captured as a feature.  I have set up my workspace for feature 
based launching -- all plugins are in a "plugins" dir which is my workspace 
area and the features are in a "features" dir which is a sibling of "plugins".

When I launch I am not given a choice as to which features I want to include.  
It seems that update just discovers all possible features and makes them 
available.  This is unfortunate since I am actually testing the updating and 
installing of the features I am developing (i..e, some of the ones in my 
workspace).  I tried closing/deleting the projects (but not removing the 
content) but Update still finds them.

How do you recommend someone develop features and test their deployment using 
PDE?
Comment 1 Jeff McAffer 2004-09-27 10:29:31 EDT 
I have worked around the problem by moving most of the feature projects out of 
the "features" dir and back into "plugins".  Unfortunately, this means that 
whenver I want to change the list of features against which I run, I have to 
move the projects around.

Related to this, it seems that the launched configuration also includes all 
plugins in the "plugins" directory.  This is unfortunate because of course, I 
am developing various plugins that I want to have installed while running my 
target.  Since these are already installed, nothing happens (and the included 
function is present too early).  

It seems like I must be doing something different from what was intended by the 
feature based development mechanism.
Comment 2 Wassim Melhem 2004-11-02 23:32:59 EST 
This is not a bug nor a limitation of self-hosting.
In feature-based self-hosting , Update Manager, not PDE, is in charge.  This 
is how you can test feature lifecycles.

Therefore, Update goes through the filesystem and discovers features and 
plugins, and will discover all of them unless you manually move them.
Of course, closing feature projects does not work since the features are still 
in the filesystem.

If PDE does anything to intefere with Update, then it would not be a true 
feature-based selfhosting.  It would become more like regular self-hosting.

Nothing PDE can do.
Comment 3 Jeff McAffer 2004-11-03 09:19:27 EST 
You are looking at it from a technology/implementation point of view.  The 
original bug is basically noting that doing feature based self hosting and 
actually getting it right is hard.  I agree that having high fidelity is 
important.  That is sort of the point.  The current approach requires the user 
to go through considerable pain to configure my *workspace* to contain only 
those things which are actually expected to be present in the real world.

A similar problem arise with the target.  Say your target has 50 features but 
you are actually developing something which really only has 5.  How would you 
do feature based self hosting?  In the current setup you would have to go an 
remove the extra features from your target.

The sort of thing envisioned here is PDE produces/passes as set list of 
features to the target. Update uses the list either verbatim or as a filter on 
what it otherwise discovers.

In the end, the user is not trying to test the Update code, they are trying to 
test their features and related function.  To do that they need update to 
identify the right set of features but are not really concerned with how that 
happens.

I am happy with this as a lower priority but think that it is still an issue.
Comment 4 Dorian Birsan 2004-11-03 11:09:47 EST 
Perhaps feature based self hosting can be branched into
1. run with what can be discovered from the directory structure
2. run with a PDE made platform.xml

Scenario 1 is what is currently there for feature based self hosting.
Scenario 2 would be a combination of feature based self hosting (i.e. proper 
directory structure for features and plugins, much like in current feature 
based self hosting) and current self hosting (i.e. PDE prepares the list of 
features, and can use the newly added update policy MANAGED_ONLY to only pick 
up plugins from features listed in platform.xml)
Comment 5 Wassim Melhem 2004-11-03 11:41:26 EST 
MANAGED_POLICY, is that a 3.1 concept or a 3.0.x concept?  ie. can we only 
used if the target platform is 3.0.x-based?

In any event, Dorian, if plan 2 is a feasible plan, then let's do it.

We could use bug 52201 or the more generic bug 70641 to address it.
Comment 6 Dorian Birsan 2004-11-03 13:08:00 EST 
MANAGED_ONLY is a 3.1 concept. In essence, only the plugins contributed 
(managed) by the features listed in platform.xml will be run.

PDE uses USER_INCLUDE for regular self hosting, so only the plugins listed in 
platform.xml are included at runtime.

Feature based self hosting, like any regular eclipse launch, uses USER_EXCLUDE 
policy, which means that all the plugins found in the installation location 
are used (minus those listed as excluded).

I have not investigated the feasibility, but we can talk more about it.
