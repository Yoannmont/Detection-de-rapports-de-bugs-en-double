Comment 17 Jeff McAffer 2004-10-28 09:56:25 EDT 
I am more trying to clarify rather than refute.  In most cases we have found 
that people actually do not need the classloader itself.  They need to load 
classes from a particular bundle (possible now) or be able to identify the 
bundle which loaded a particular class (not possible now).  The latter is very 
rare.

In comment 15, Scott, it seems you are making some sort of leap here.  A and B 
are plugins.  Some class in B calls some method, say foo(), in one of A's 
classes.  foo() wants to load a class programmatically (e.g., Class.forName
()).  The challenge here is not getting B's classloader but rather identifying 
B (i.e., as you ask, how do you know who B is).  

If you are in A then there are only two ways to load classes, using A's 
classloader or using some other classloader.

1) If you want to use A's classloader then you simply do Class.forName() and 
ensure that A's classpath (including plugin prereqs) includes the classes you 
are looking for

2) If you need to use a different classloader you identify the classloader to 
use and use it.  There are several ways of doing this.

2a) extensions.  have A expose and extension point that people extend with 
classes or class factories.  A's code then uses the extensions to try and load 
the desired classes via createExecutableExtension or by querying the extension 
to discover its bundle and then using Bundle.loadClass()

2b) implicit.  You "figure out" which bundle to use to load classes.  There may 
be any number of ways to derive this.  Perhaps it is in the context, or perhaps 
you use Class.getClassLoader() on some object you are given?

2c) explicit.  Someone tells you explicitly which bundle to use.

Exposing the actual classloader for a bundle helps ONLY in the case that you 
need to somehow do a reverse lookup and find the bundle which owns a particular 
class.  It has the unfortunate downside of exposing more internals and thus 
constraining future implementations (e.g., what if we want to have multiple 
classloaders per bundle?).

Bundle looks more and more like classloaders in that they propose methods to
load classes and resources, however there is no ways to go from classes to
bundle, mimicing the class <--> classloader relation.

Aside from the completeness of the relation, such a mechanism is necessary for a
complete handling of dynamicity. 
For example, the eclipse extension mechanism allows plug-ins to give away some
objects to some other plug-ins. For example, a plugin contributing a View will
give an instance to the UI plugin. 
In most of the case, the view instance' class will be part of the plugin
contributing the extension, however given that classes can be loaded from any
plugin in the import / required (optional or not, dynamic or not), if there is
no way to identify which bundle owns the instance, then leakage can happen on
bundle uninstallation.

The implementation of such a service is pretty trivial and straightforward, and
I think for completeness it should be proposed to the spec. 
Note that I recall that ModJava offered a capability to go from a class to a 
module.

Comment 26 Jeff McAffer 2004-10-29 16:36:26 EDT 
Observations:

- This usecase does not require the ability to go from object/class -> 
classloader (or even bundle).  You note that it may be handy in some situations 
but we have not highlighted those with real usecases.

- classloaders and bundles both allow the loading of classes

The thing at stake here is one's ability to talk in terms of classloaders 
rather than bundles.  For example, you want to call 
    Class.forName("Foo",true,classloader)
rather than 
    bundle.loadClass("Foo")
so that you can use any technique to find the classes.

This is a reasonable desire/requirement.

Where we are differing then is in how to get there.  Two ways:
- give access to the bundle's classloader

- implement a simple wrapper classloader which backs the implementation of 
findClass() with Bundle.loadClass().  Then pass around that classloader.  See 
PluginClassLoader for an example.

The first is direct and simple. It however opens up the implementation in a way 
that we may not want (differing numbers of classloaders per bundle, ...).  The 
second accomplishes the goal but requires an additional wrapper class/object.  
Fortunately this class is totally trivial.

Side note:  In comment 24 I stated that the only usecase for getBundleFor
(Class) was logging.  Actually, a much more interesting usecase is 
serialization.  When serializing an object from a plugin you really need to 
know which plugin so that the guy deserializing can load the correct class.   
We are actively seeking to have this added in the OSGi spec.

Comment 28 Jeff McAffer 2004-10-29 22:23:57 EDT 
Nothing too grand is planned.  Basically something like the proposal in comment 
19.
    public Bundle getBundleFor(Object o)
Actually, comment 20 proposed a better
    public Bundle getBundleFor(Class c)

The usage model would be to annotate serialized objects with the bundle that 
owns their class as they are written out.  The reader can then get that 
information and use it to get the bundle and do Bundle.loadClass() or use the 
fabled wrapper classloader trick.

Of course there will be variations on this theme depending on the serialization 
mechanism.

For completeness I will note that the ultimate and somewhat less than optimal 
approach in cases where you really cannot do this kind of thing is to use 
Dynamic-ImportPackage.

This is a mechanism whereby Bundle B (from comment 25) can say 
    Dynamic-ImportPackage: <some package specifications>
this will allow B to discover packages from pretty much anywhere as long as 
they are exported.  Caveats:
- there are several downsides to this approach and you should read and 
understand the relevant sections in the OSGi specification before using
- some of the objects you may want to handle are not API and so will not be 
exported

Comment 30 Jeff McAffer 2004-10-30 21:10:00 EDT 
Answers:

1) No.  Dynamic-ImportPackage does not have a corresponding element in 
plugin.xml.  In general we have no intention of extending or changing 
plugin.xml to include/expose additional execution concepts/mechanism.

2) Note:  PluginClassloader is very much a legacy class.  EclpiseClassloader 
(and its friends in the hierarchy) are internal and do not expose any API.  So, 
there is no support for changing the set of dynamic imports in code.  you have 
to change the manifest and update the bundle.  Note that you can do Dynamic-
ImportPackage: * but this has potentially serious performance implications.  

Sidenote:  Of course, you could write the equivalent of this mechanism using 
Bundle.loadClass() and BundleContext.getBundles()...

The following API has been added to the PackageAdmin service.
/**
 * Returns the bundle for which the specified class is loaded from. The
 * classloader of the bundle returned must have been used to load the
 * specified class. If the class was not loaded by a bundle classloader then
 * <code>null</code> is returned.
 * 
 * @param clazz the class object to get a bundle for
 * @return the bundle from which the specified class is loaded or
 *         <code>null</code> if the class was not loaded by a bundle
 *         classloader
 * @since 1.2
 */
public Bundle getBundle(Class clazz);

Comment 42 Thomas Watson 2008-02-28 08:50:58 EST 
The Bundle object in OSGi abstracts out the class loader with the methods loadClass, getResource and getResources.  In OSGi an attempt has been made to not expose the class loader in API because the class loader is viewed as an implementation detail.  Unfortunately some Java APIs require a class loader (like ResourceBundle).  In the solution you implemented you are basically reexposing the underlying class path from the bundle in OSGi so that you can access the properties files using a class loader object that simply uses the Bundle object to load the classes/resources.

HTH.
