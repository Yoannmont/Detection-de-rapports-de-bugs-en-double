Description Lars Millberg 2007-03-26 10:03:44 EDT 
The bundle org.eclipse.core.databinding has a "Required-Bundle" dependency to "org.eclipse.core.runtime", which really isn't needed. Remove it and everything still compiles. Don't confuse the org.eclipse.core.runtime <i>bundle</i> with the <i>package</i> with the same name, living in the org.eclipse.equinox.common bundle (which is already referenced as a required bundle)!
If you don't think it will be needed in 3.3 I suggest it be removed.
Comment 1 Boris Bokowski 2007-03-26 10:54:27 EDT 
Both org.eclipse.core.runtime and org.eclipse.equinox.common are optional dependencies ("resolution:=optional").  This enables running data binding against a 3.2 Eclipse.  What we wanted to express (but couldn't) is that data binding requires *either* org.eclipse.core.runtime *or* org.eclipse.equinox.common.

Thanks anyways for the bug report, it's always good to have people who are sensitive to these kinds of things.

Resolving as later (lacking a 3.4 target milestone) because post 3.3 I think we should remove this and just require org.eclipse.equinox.common 3.3.

Note that the Activator of the databinding bundle extends org.eclipse.core.runtime.Plugin which is only in the org.eclipse.core.runtime bundle.  Right now databinding will not work without org.eclipse.core.runtime.

There likely is no need to extend Plugin in this case.  If that can be avoided then the org.eclipse.equinox.common should become a non-optional requirement.  The optional requirements can mess things up when the runtime bundles are updated.  In some cases the databinding bundle can become resolved but not wired to the optional require bundles.  I have seen this happen when self-hosting and one of the core bundles changes within the workspace.  The platform in this case flushes the core bundles at launch and forces the platform to reresolve without the core bundles.  In this case the databinding bundle becomes resolved.  Later the core bundles are reinstalled but the databinding bundle is already resolved so it does not dynamically get wired to the core bundles.

To reproduce use the following steps

1) In a clean workspace checkout org.eclipse.equinox.registry and org.eclipse.core.databinding.
2) Self-host eclipse and allow the workbench to come up, then exit.
3) Modify the bundle manifest in org.eclipse.equinox.registry.  Simply add and remove a space and save the file to increment its file last modified value.
4) relaunch the self-host eclipse instance

You will get this error which is very confusing and always takes me a few minutes to figure out what happened, if others are seeing this they likely just give up and clear the configuration and blame it on some Framework corruption :(

Comment 14 Thomas Watson 2008-04-11 17:05:09 EDT 
Currently you have an Activator class that extends Plugin.  You will have to implement org.osgi.framework.BundleActivator instead because Plugin is in org.eclipse.core.runtime bundle.  You already have a dependency on org.eclipse.osgi in order to have a bundle activator.  I suggest you use Import-Package to get access to the packages you want from org.eclipse.osgi:

Import-Package: com.ibm.icu.text,
 org.osgi.framework;version="1.4.0",
 org.osgi.util.tracker;version="1.3.3"
