Comment 7 Nick Edgar 2004-06-23 10:43:15 EDT 
I don't see how you could be getting a null element passed to getText.
The viewers check that elements coming from the content provider are not null.
You could try setting a breakpoint in BaseWorkbenchContentProvider.getChildren
and see what elements the content provider is returning.

For getAdapter returning null, this is probably due to the IWorkbenchAdapter and
other adapters not being registered with the platform.  This is done in
WorkbenchAdapterBuilder.registerAdapters(), which is called from
IDEWorkbenchAdvisor.initialize.  Since your RPC app is using a different
WorkbenchAdvisor, this is not getting called.  I suggest calling
WorkbenchAdapterBuilder.registerAdapters() from your own advisor's initialize
method.  Notice, though, that this is an internal class, so your code may be
broken in the future if we make changes here.

Back in WorkbenchContentProvider.getChildren, I would expect it to return an
empty array if it can't find the IWorkbenchAdapter, so I still don't see how
you're getting a null element passed to getText.
Comment 8 Jose M Beleta 2004-06-23 14:22:30 EDT

WorkbenchAdapterBuilder.registerAdapters() ALL problems disappear.

What I cannot understand is why this class is internal and not documented.

Comment 9 Nick Edgar 2004-06-23 14:44:34 EDT 
When we split up the workbench into the generic RCP workbench and the
IDE-specific workbench, we did not envision RCP apps building on the IDE
functionality.  We considered the Navigator to be IDE-specific, which is why
it's in org.eclipse.ui.ide.  Likewise, the resource-specific adapters are also
defined here.

org.eclipse.core.resources can optionally be used by RCP apps, but the Resource
Navigator is not currently available as an optional RCP component.
You may be able to get it to work by including org.eclipse.ui.ide and doing
various hacks, but it's not recommended.
Comment 10 Nick Edgar 2004-06-23 14:45:52 EDT 
For more background on why we split things up the way we did, please see the
executive summary and other design docs on the RCP proposal page:
http://dev.eclipse.org/viewcvs/index.cgi/%7Echeckout%7E/platform-ui-home/rcp-proposal/rich_client_platform_facilities.html
Comment 11 Jose M Beleta 2004-06-23 15:04:43 EDT 
A lot of non IDE Windows applications, use window explorer like navigators to 
deal with resources, most of them are files. It is very difficult for me to 
understand why this capability as been left out from RCP.

Any plan to reconsider this in the future?
Comment 12 Nick Edgar 2004-06-23 15:22:46 EDT 
While many apps have a navigator-like view, it's unclear that most would want
the Navigator as it sits now.  It is currently fairly tied to the rest of the
IDE.  That is, it has actions for import, export, build, etc, which may not make
sense for an RCP app.  Plus, many RCP apps have their own underlying datamodel,
not necessarily using the workspace (org.eclipse.core.resources).

We had originally wanted to provide a general purpose navigator as an optional
RCP component/framework, but that unfortunately got deferred due to lack of
resources.  The original plan item for it was bug 36961.  The code for it is
still around, but needs fixing due to some later changes in 3.0.
