Comment 8 Jared Burns 2004-11-19 15:08:04 EST 
I've explored the idea of using working sets to store custom breakpoint groups and the results I found 
were all negative:
1. Because we're not just storing resources, we don't save ourselves any work by using the working 
set framework. We still have to do all the work of creating, persisting, and restoring breakpoint 
working sets ourselves.
2. It doesn't make any sense to register our working set type with the working set manager. If we use 
the working set framework, breakpoint working sets will appear in the dialog along with other working 
set types. This doesn't make sense anywhere. Resource working sets make no sense in the 
Breakpoints view and breakpoint working sets make no sense in the Navigator.
3. If we allowed working sets to contain arbitrary breakpoints, a breakpoint could appear in multiple 
working sets. This creates two problems for the Breakpoints view:
  a. TreeViewers currently can't handle having the same element shown in the tree multiple times. 
Because it populates lazily, the viewer needs to be able to ask for an element's parent in order to 
make it visible (which we ask it to do everytime the view refreshes). Since you can only return one 
parent for an element, it's impossible for us to correctly expand the tree to keep breakpoints visible.
  b. This creates a major booby-trap for the user, whereby they can disable a breakpoint without 
realizing it. Disabling one working set could disable a breakpoint in another working set that the user 
had just enabled. The request here will be for breakpoints to have multiple enabled states, which is 
just a mess.
  c. Breakpoints appearing multiple places in the tree + nested breakpoint groups = total disaster. 
Imagine the the user sets the view to group by project, then by working set. You could have 
something like so:
- ProjectA
  - WorkingSetA
    - BreakpointA
  - WorkingSetB
    - BreakpointA
- ProjectB
  - WorkingSetA
   - BreakpointB
  - WorkingSetB
   - BreakpointB

What does it mean when you uncheck the checkbox next to ProjectA/WorkingSetB? Do you just 
mean that BreakpointA should be disabled? Or did you *really* mean that you wanted to disable the 
working set WorkingSetB, in which case we should also disable BreakpointB. If it means the latter 
(and if people want to be able to disable working sets, it does), then selecting any one group in the 
tree automatically toggles *all* groups in the tree. It's a total disaster.

In my investigation, I saw nothing positive.

Move to close.
Comment 9 Darin Wright 2004-11-23 12:16:51 EST 
I think there are still gains to be had here...

* looking at the package explorer's support for working sets, we could provide 
a very similar UI and support
* the package explorer allows to "Show > Projects | Working Sets". The 
breakpoints view similarly could have "Group By > Projects | File | Breakpoint 
Type | Working Sets".
* When grouping by working sets, we can use existing working sets that are 
resource based (i.e. the breakpoints contained in those resources), and/or use 
a new "breakpoint working set" for "custom groups". 
* The package explorer allows you to choose which working sets to display, 
which we could also do. This groups the elements in the view by working set, 
and places other elements into an "other" group. Thus, it does not filter 
anything from the view.

The benefits are: common UI/interface and use of working sets for logical 
grouping of breakpoints and resource based grouping of breakpoints. The 
package explorer allows packages to be in more than one working set, so we 
should be able to leverage the same support to allow a breakpoint to be in 
more than one working set.

To keep things simple at first, a breakpoint should still has one enablement 
setting. It's enablement is the same in all groups.
Comment 10 Jared Burns 2004-11-23 12:21:43 EST 
How do you propose addressing the problems I outlined in comment #8?
Comment 11 Darin Wright 2004-11-23 12:31:28 EST 
Problem (1): we also support resource based working sets, and thus benefit 
from other types of resource based working sets.

Problem (2): this is a scability problem in the workbench - not something the 
debugger can solve. We should file a bug against the workbench.

Problem (3): allow breakpoint to appear in mutliple working sets/groups. The 
tree viewer can be enhanced to support it (the package explorer does it 
somehow). The enablement is breakpoint based - *not* group based. Toggling a 
group simply toggles the breakpoints in the group. If it disables a breakpoint 
in another group, it is also disabled there, and the group rendering will 
display that with a "grey" check box. If we want to be more complex in the 
breakpoints view, we could introduce a voting shceme, where the enablement of 
a breakpoint is group dependent. I.e. A breakpoint still has one state - 
enabled, but can be rendered as disabled in a disabled group. When an editor 
is opened, it is still "enabled" if it is enabled in at least one group.

Comment 17 Darin Wright 2005-01-14 14:41:43 EST 
Fixed.

Breakpoint groups have been completely re-worked. I have replaced the previous 
exentions point (breakpointContainerFactories) with a new extension point - 
breakpointOrganizers.

The previous imlpementation did not handle the dynamic nature of groups (i.e. 
groups that changed for some reason behind the seen, did not get changes 
reflected in the breakpoints view). For example, as working sets are 
created/changed, this was not reflected in the UI.

Rather than having clients implement container factories to statically group 
breakpoints in containers, the new extension point provides an "organizer" 
that allows a client to categorize breakpoints. The implementation takes care 
of creating/maintaining the containers, and the client only tells us what 
category the breakpoint is in.

This also allows us to remove the notion of groups (group attributes on a 
breakpoint) from the core as well, as the new implementation is purely a 
presentation issue. Breakpoints are allowed to be in more than one group.

Instead of "custom groups", there are only working sets.

I will file individual bugs for remaining issues and polish items.
