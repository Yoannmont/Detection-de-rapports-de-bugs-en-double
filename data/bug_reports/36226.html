Comment 4 Ed Burnette 2003-04-08 12:44:04 EDT 
Created attachment 4499 [details]
Some ideas for reducing the add complexity from O(n^2) to O(n)

[Originally posted on the swt-dev mailing list --Ed]
I was having trouble putting more than a couple thousand items in a table on
Windows so I took a look at the code and found a couple areas that could be
sped up. Could someone take a look at these changes to see if they look
reasonable? If so I'd like to contribute them to the code base. With the
changes I can add about 400,000 table items a minute to a table.

The first part of the patch is in Table.createItem(TableItem, int). I noticed
that it was reallocating and copying the items array every fourth time an item
was added, leading to an O(n^2) time for creating an item. I fixed this by
increasing the items array by 50% each time. You could try 75% or 100% but this
is what I typically use in my own code.

The second 2 parts are in the Table.indexOf(TableColumn) and
Table.indexOf(TableItem) methods. I noticed that typically one will add items
to the bottom of the table and then call TableItem.setText() to set their text.
Unfortunately this turns around and calls Table.indexOf(this), which was
searching all the way from the beginning of the list for the item that was just
added to the end. Another O(n^2) algorithm. I fixed this by searching backwards
through the table items instead, and did the same for columns for consistency.
If any code is relying on the indexOf searching forwards it would break that
code, but I couldn't think of a reason why you'd have duplicate items or
columns anyway, can you? If there's a reason then perhaps another way could be
devised to prevent the extra search such as keeping track of the last item
added or using a hash instead of a linear search (though a hash wouldn't
guarantee the order either).

Comment 8 Ed Burnette 2003-04-08 16:19:54 EDT 
Created attachment 4504 [details]
A more aggressive patch (see comments)

Three more changes in this patch. I haven't done any extensive testing so it
might have bugs (gasp). And it's not especially pretty, it's just for
experimentation. The changes are:

1. I noticed the code kept going to Windows to ask for the number of items in
the table, so I cached that instead. This resulted in a small improvement (see
below).
2. I noticed that in the main2 test, it kept figuring out the code page over
and over, so I cached that too. Moderate improvement.
3. Finally, I noticed that in the main2 test it spent an ungodly amount of time
disposing, and there were comments in the code that said disposing each
individual item was faster than disposing everything, so I decided to test that
hypothesis and try the dispose-all call and see what happened. Big improvement
in main2.

Comment 20 Ed Burnette 2003-04-16 16:44:36 EDT 
I took a look at the latest HEAD version of the Table class for Windows to see 
what "Fixed > 20030410" means.

- It looks like the indexOf() optimization was applied, sort of (it keeps a 
record of the last indexOf result and does a backwards search only if the last 
indexOf was near the end). This seems ok to me.

- Also the table growing optimization was applied, sort of (it only does the 
optimization if redraw is off, so it will only help on cases where you turn off 
redraw and dump a bunch of stuff into the table like these benchmarks). However 
it turns around and resizes and copies the whole array into a smaller array 
when you finally do the redraw. I think the typical case is that redraw is on 
and you're adding a line at a time but you just have a lot of them. So I don't 
think this is a good solution. If you're worried about increasing by 50% each 
time, just increase by 25% or use a different data structure than an array. 
Anything would be better than just adding 4 and copying.

- The dispose-all optimization was not done yet - is this something that is a 
win on some versions of Windows and not on others perhaps? It's a big win on XP 
at least.

Comment 22 Steve Northover 2003-04-16 17:05:37 EDT 
Ed, we are concentrating on the time it takes to create a table on every 
platform, not just Windows.  The code that was release in HEAD satisfies this.

The indexOf() is optimized to search in the order of the last search, if it 
gets it wrong, no harm is done.

SWT is optimized for space as well as speed.  The growing and shrinking code 
in setRedraw() allows us to grow agressively when items are added and shrink 
back down to use less memory the rest of the time.  Using a "don't draw" flag 
to optimize drawing and other calculations is common in many Window systems 
(MS Windows).  We also use this in SWT.

The dispose case on Windows requires that I go and find an old Windows 
machines and run the benchmarks again.

Sorry but there are way more problems in SWTVirtual table than flickering.  
For example, none of the arrow keys work properly, the selection is wrong and 
it seem to have a bug when you scroll to the end really fast.  Home and end 
don't work.

Quite a while ago I looked at doing the exact same thing that you did and came 
to the conclusion that it wouldn't ever really work well.  Basically, you need 
to implement a complete table control, including all of the keyboard 
interactions etc. and use the real Table for display purposes only.  To 
summarize, you will "die the death of 1000 cuts".

I'm attaching my old hacks (including bugs) for you to look at.  A better idea 
is for us to just keep fixing performance problems in the native table and 
look at supporting virtual tables using the operating system (both of which we 
are doing right now when not preparing for EclipseCon...)

I was playing a bit with the virtual table using the code snippet 144 
http://dev.eclipse.org/viewcvs/index.cgi/~checkout~/platform-swt-

There is a problem when disposing a big tabe. That actually happens each time
the size of the table changes. The bottleneck is the removeAll() method that is
called when setItemCount is called. The problem is 

  	ignoreSelect = true;
	int code = OS.SendMessage (handle, OS.LVM_DELETEALLITEMS, 0, 0);

the usefulness of virtual tables (for me)...

(see also http://msdn.microsoft.com/library/default.asp?url=/library/en-

and 
http://msdn.microsoft.com/library/default.asp?url=/library/en-
