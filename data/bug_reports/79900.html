Comment 8 Wassim Melhem 2005-01-13 14:40:08 EST 
Hi Sherry,
Enhancements for bundle tooling in the PDE editor were discussed today and 
here is how we envision it should look like:

1. The extra bundle manifest information (eg. copyright, bundle-contact 
address etc.) that you have on the overview page is not needed as it does not 
contain any vital information.  So we don't need to show it in the editor - 
unless you know of a good reason to do so.  If people want to add them, they 
could do it on the source page.  So the overview page stays the same - for the 
most part.  details below.

2. We don't want a separate page for Import/Export packages/services.
Import service/package would go on the dependencies page, while Export 
packages/services would go on the runtime page.

3. The Dependencies page will contain four sections.
    i. top left: the dependencies list as it is today.  Will add an Edit... 
button to it, which when pressed will pop up a dialog to edit properties such 
as 'exported', 'optional' etc.
    ii.  top right:  dependency analysis section with all the links to invoke 
search functionalities such as find unused dependencies, which would also 
encompass unused package-imports.
    iii.  bottom left: the Import-Package section which you implemented.
    iv.  bottom right: the Import-Service section which you implemented.

4. The Runtime Page contains four sections:
    i. top left: the list of libraries (bundle-classpth)
    ii. top right: the list of Export-Package
    iii. bottom left: Bundle Activation (i.e. Eclipse-Autostart) header.
    iv. bottom right: Export-Services section which you implemented.

5. The link to create a manifest.mf on the fly will move from the plugin 
activation section to the overview page to make it more prominent.

6.  The editor will have two incarnations of the dependencies/runtime pages: 
one that will be as described above, and another copy that will contain no 
bundle manifest info (i.e the latter is intended for traditional plugin.xml 
developers).  We show the right pages depending on whether or not a 
manifest.mf is present.

7.  The 'Secondary References' that Brian is implementing will be defined in 
the classpath container page itself, not in the editor.  This page is invoked 
from the Java build path properties page of the plugin project, go to the 
libraries tab, click on the pde container and click Edit...  The page will 
change dramatically as a result.  Right now, it just allows you to attach 
source, since the value of the resolved container is up to PDE.  In that 
viewer, people should be able to define new bundles that should be added to 
the classpath.  Where would those bundle names be stored?  a suggestion would 
be as a project preference so that they would be sharable.

Comment 20 Wassim Melhem 2005-03-28 17:04:50 EST 
The revamped plugin editor will be released tomorrow.

Here are some highlights/observations/comments:

1. The first three pages of the editor (Overview, Dependencies, Runtime) vary 
significantly whether or not the plugin has a manifest.mf or not.  The plugin 
editor looks exactly the same for users who don't have a manifest.mf.  No need 
to change their workflow.

2. The editor makes use of the new API (courtesy of Nick and Dejan) that 
allows dynamic addition/removal of pages at any index. 
So if you click on the link to create a manifest.mf, we create the file, 
remove the first three pages and create new ones.  This feature is looking 
pretty slick.

3. The link to create a manifest.mf file is now in the content section of the 
Overview page.  I plan to make it even more prominent for the 3.1 release by 
having a separate section ("New in 3.1") on the Overview page.  This section 
would contain links to Platform/Runtime docs telling people about the new 
runtime features in 3.1 and the 3.0 -> 3.1 migration guide for manifest.mf 
files.  Jeff, let me know when such documentation becomes available.

4. The General Info section for manifest.mf will have a Platform-Filter field.

5. The Dependencies page will have a section for the Require-Bundle and 
another for the Import-Package headers, both occupying all the vertical space 
on the page.  The Dependency Analysis section is a collapsible section at the 
bottom.

6. Properties exposed for a required bundle are: resolution, version, 
visibility

7. Properties exposed for Import-Package are: version and resolution
When people are adding a package, we display a list of all the exported 
packages in the state.  This is a pretty long list, even though we don't 
analyze the JARs for plugins without a manifest.mf when we construct the 
state.  So I trimmed it as follows:
   If a package is exported by a plugin that is already on the Require-Bundle 
list (or by any of the plugins it reexported), exclude this package from the 
selection list.

This list will become shorter as people adopt manifest.mf and become more 
restrictive as to what they want to expose to clients.


8. Properties are now done via a popup dialog.  This also makes the validation 
of a version possible.  We validate both versions and version ranges.

9. To get Jeff off my back ;-), the labels for objects in the table viewer are 
more descriptive so that you could get more information at a glance.  For 
example, if you set a version on a package, etc., the package version appears 
in the label along with the package name.

10.  The Runtime page contains Export-Package and Bundle-Classpath support.  
The Eclipse-AutoStart support has been removed due to lack of real estate and 
its limited advanced audience.

11.  A table viewer for the Export-Package header occupies the left.
Top right, you can set the x-internal, x-friends directives.
Bottom right, the Libraries section.
