- in default Java perspective,
- activate Package Explorer

The perspective should remember a placeholder for the detached view, so that it
gets docked back where it was before.

Comment 2 Nick Edgar 2005-10-25 13:45:25 EDT 
Note that you need a separate placeholder of the form "primaryId:*" if you're
specifying a secondary id for a multi-instance view.
See http://eclipse.org/rcp/faq.html#multiViews

Comment 4 Nick Edgar 2005-10-25 14:24:11 EDT 
So does it work as expected, or not?  If not, can you clarify with a concrete

Comment 5 Joshua Salit 2005-10-25 14:34:41 EDT 
Sorry I wasn't clear - Nope, it does not work as expected.  

If I open up an instance of MyView, and say secondary ID is "0", it opens in the
placeholder as expected.  If I then detach that view, and open a second and
third (secondary IDs "1", "2", etc), they are no longer put into the
placeholder.  Instead, each new MyView shown gets placed in a fresh stack,
directly below the last one opened.

Also, I think it only happens if the -first- MyView opened is detached before
opening any other MyViews.

I hope that makes more sense..!
Comment 6 Joshua Salit 2005-10-25 14:41:39 EDT 
Ok, this is interesting - I just tried to reproduce with the Browser example,
and I could not.  Unexpectedly (to me), the Browser example does not use the
secondary ID wildcard to create a placeholder for new browser tabs.

So I removed the secondary ID from my the call in my application, so it now
looks like this:

...and NOW it works as I had expected.  Multiple instances of the same view,
each with unique secondary IDs, open up in the expected placeholder.  From my
understanding, however, the syntax is the opposite of what it's documented to be?
Comment 7 Nick Edgar 2005-10-25 14:52:00 EDT 
Note that if a matching placeholder is not found, the view opens in the folder
at bottom right by default.  Maybe that's what you're seeing?
Comment 8 Nick Edgar 2005-10-25 14:53:07 EDT 
But you are correct with the expected behaviour.  If a view has a secondary id,
it looks for a match against the compound id: "primaryId:secondaryId", so a
placeholder of just "primaryId" will not match, but "primaryId:*" will.
Comment 9 Joshua Salit 2005-10-25 14:57:12 EDT 
I'm pretty sure the matching placeholder is being found - I also have a MyView2,
which I set up a placeholder for below the placeholder for MyView, and if I have
not detached the first view opened, the views always open in the expected
placeholder.

Strange, I can't explain that.  We'll have to investigate.

Comment 19 Eric Moffatt 2011-10-07 14:14:04 EDT 
We have a far better chance in 4.x because the UI Model manages all the storage issues...

It's interesting to note that we already have a flavor of this in order to handle views that get dragged into the shared (editor) area. In 4.x a 'reset perspective' will create a non-rendered placeholder in the view stack where it would have been.

The trick is that when we come to 'open' a view we always search in the shared area first, then the perspective.

Perhaps there's a way to generalize this to include DWs...

As an example of the complexity...what should we do when the user *drags* the view to make it detached ? One might say it should leave a placeholder behind but that can only happen if it was dragged to a DW.
