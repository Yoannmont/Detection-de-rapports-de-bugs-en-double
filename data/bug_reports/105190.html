Comment 6 Philip Barr 2005-08-05 12:30:21 EDT 
I and my colleague both have this problem. Saving after adding and deleting a
space in a document takes at least six seconds. I've noticed that this seems to
be related to publishing to the server.
Steps to recreate:
1. Start / restart Eclipse
2. Edit and save a document (eg. a JSP page but it can be anything) to
demonstrate normal save time.
3. Publish to server.
4. Edit and save a document (eg. a JSP page but it can be anything) to
demonstrate lengthy delay in save time.

Workaround:
1. Go to Windows - Preferences - Server and uncheck all of the top 3 boxes
(anything to do with automatically publishing)
2. Add projects and set up server preferences.
3. Publish the Server configuration.
4. Restart Eclipse.
5. Don't publish anything even when it says "republish". If you need to add
projects or make changes to the server settings then do so, then publish, then
restart.

I'm using:
Win XP Pro
Eclipse 3.1 and WTP 0.7 (wtp-all-in-one-0.7-win32.zip)
VSS Plugin (vssplugin.sourceforge.net)
Fat Jar (fjep.sourceforge.net)

Comment 17 Tim deBoer 2005-10-05 11:14:37 EDT 
Fixed and tagged for the weekly I-build. The problems I found are as follows:

1) There were two separate server resource change listeners, which should be
merged to reduce workload on the Eclipse runtime. Not a major issue, but fixed.

2) The resource change listeners did not filter changes that they are never
interested in, e.g. marker updates, duplicate change events, etc. Fixed.

3) The J2EE APIs for getting lists of modules has greatly degraded performance -
this is the bit that caused the actual performance issue to show. Not fixed.

4) The optimizations for project structure change (i.e. new module added, module
removed from EAR, etc.) had been removed when the J2EE team moved to the
flexible project structure, thus causing server tools to request updates when it
wasn't necessary. New optimizations have been added in. Fixed.

Although the underlying performance problem is still there, the results of these
fixes and optimizations is that we call this code very rarely - once the first
time it is used, and then only when .wtpmodules files change. There is still a
delay in these cases, but it is much less that the previous delay that was on
just about every action in the workspace. As a result, I don't have any
performance problems when testing with a large workspace that was previously
unusable.

FYI, I've already followed up with the J2EE team and we'll be monitoring the
underlying module operations as well.
