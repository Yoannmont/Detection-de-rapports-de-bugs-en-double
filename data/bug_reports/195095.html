Description Richard Birenheide 2007-07-02 05:13:13 EDT 
Build ID: n/a

Steps To Reproduce:
Currently it is not possible to assign a database schema as default schema when selecting a database connection for a JPA project/facet. Special action is required by a database provider (schemaname=username) to obtain correct validation of entities against a specific schema.



More information:
It would be desirable to have the possibility to select a default schema when assigning a connection to a JPA project.
Comment 1 Neil Hauge 2007-07-02 10:48:42 EDT 
This is a bit tricky, as we are trying to replicate what the user will experience at runtime, where the schema (by default) will be determined by the database, based on the login username.  Unfortunately right now we aren't doing this quite right, as described in bug 193879.  We do make the faulty assumption that schemaName=userName, which is not always the case.

JPA is very default driven, and handling these defaults correctly is important to making sure the user understands what is going to happen at runtime.  In the case of schema default, the appropriate place to define this default for a persistence unit as defined by the spec is in the ORM XML file.  This is generally what I have recommended since this is required at runtime for a specific schema to be used that is not the default schema.

All of that said, there could be an argument for a design-time schema setting that will override only the default default schema name (where no schema is set in a JPA artifact).  This could potentially be helpful in situations where your deployment login was going to be different than your design-time login.  The question is whether this is more or less confusing to a user who is trying to make things work at runtime.

Comment 5 Neil Hauge 2008-01-28 18:56:42 EST 
There doesn't appear to be any helpful API for obtaining the default schema name from DTP.  I have entered bug 216839 to request this functionality for the future.

For the short term, we will need to determine if there are any DB's we currently don't support with our getDefaultSchema() method.

Currently we support the userName default and "public" for PostgreSQL.  We have successfully used default schema with MySQL and MSSQL, so I think those are covered.

Let me know if there are DB's that don't currently work with the schema defaults.  For 2.0, if we don't have new API from DTP, we will probably create a setting for schema in the Generic Dali platform.

Comment 7 Neil Hauge 2008-04-21 18:43:29 EDT 
Proposed solution for Schema setting:

In the JPA Project Properties, in the Connection group, we could have the following:

----Connection-------------------------------

 Name:            [                  V]

 Default Schema:  [                  V]

                      Add Connection...
                           Reconnect...
---------------------------------------------


The default Default Schema selection would initially be set to our best guess based on a variety of factors.  If the default is changed, we would then persist this name and use it as the default schema for the connection.

We need to decide if this config should only exist on the JPA project properties page, or also exist on the New JPA Project config page.  I think it probably makes sense to be on both pages.  In the case of a new project, the user would then be able to see what the default schema was going to be up front, and if necessary, change the default.

I left catalog out because we don't really have good (specified) catalog
support in Dali at the moment, and adding it for this would make it less likely
to be completed in time (and would require more API changes).  When it is
necessary to specify a catalog, setting the Database field in the connection
will usually get the job done.


(In reply to comment #9)
> I like the proposal, of having it both in the New JPA project wizard and JPA
> Property page.  This default schema would drive JPA validation, the initial
> schema selected when running Generate Entities, etc?  Would changing this also
> set the default schema in the orm.xml (possibly optionally)?  Or is this purely
> intended as a development-time concept?
> 

This default would only replace the calculated connection default.  That means
that other schema settings specified at a lower level in the JPA artifacts
could override this default.  It would affect validation, UI, and the defaults
in the model.  It would also change the default for Entity Generation. 
Changing this would not set the default schema in the orm.xml.  It is indeed
just a development-time concept.
Comment 12 Neil Hauge 2008-04-23 15:52:30 EDT 
This is a multiple major adopter request for 2.0, and the final Dali UI change to be considered for WTP 3.0.  It is a small UI change to allow a user to choose the schema that Dali will use for validation and defaults processing.  Given that DTP enhancement request bug 216839 has not been addressed, we need a way to let the user pick a default schema for their project, instead of using the calculated default, which is not always correct.

Some API changes (additions only) are a part of the change.  They are as follows:

JpaProject
  getUserOverrideDefaultSchema()
  getUserOverrideDefaultSchemaName()
  setUserOverrideDefaultSchemaName(String defaultSchemaName)

JpaProject.Config
  getUserOverrideDefaultSchemaName()

JptCorePlugin
  getUserOverrideDefaultSchemaName(IProject)
  setUserOverrideDefaultSchemaName(IProject, String defaultSchemaName)


API Changes will be communicated to mailing list and newsgroup.  Requesting PMC approval for this final change.

Just a comment on this.  It seems like this setting being strictly dev-time could lead to a somewhat confusing story for users:

If I create a JPA project with a new connection and set my default schema at that time.  I generate my entities and that schema is used.  I then run my application and it fails. At minimum, it might be a good idea to explicitly state in the Default Schema setting text that this is purely dev-time and that runtime schema will be used from the orm.xml or Entity specifications.
