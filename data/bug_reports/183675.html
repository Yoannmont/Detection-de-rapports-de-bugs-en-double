Comment 3 Ralf Petter 2017-01-01 14:32:39 EST 
The problem is that in the internalSetExpanded(expanded) method only layout() on the ExpandableComposite is called. So the parent Widget will not be relayouted. In Section this method call a protected method reflow() to correct this problem. The simpliest fix ist to change layout() to the new requestLayout() method. After this change ExpandableComposite behaves correctly.

I will provide a Gerrit Code Review for this change.

I will investigate further if reflow can be removed from Section, because it is no longer needed when ExpandableComposite works correctly.

In master, the Javadoc of Section#reflow() is now starting with: "Reflows this expandable composite ...".

It's not completely wrong, but it's very surprising that an API is not documented on the actual method that constitutes the API, but on an overridden method that has only package visibility. I'd even say it's a bug in the Javadoc tool that the description gets copied from an internal method for which no Javadoc is generated.

Please restore the Javadoc on org.eclipse.ui.forms.widgets.Section#reflow() or make ExpandableComposite#reflow() API if that makes sense. In the latter case, you would have to add @since 3.8 and change the bundle version to 3.8.0 (in MANIFEST.MF and pom.xml).
