No. That file was used before M5 (or M4?). We then migrated to a different
implementation that uses a completely different layout in the file system. When
migration occurs, the .index file is renamed to something else called

caused the small (and old, and corrupted) .index file to be created again. You
definitely don't need it.

Comment 11 Rafael Chaves 2005-04-21 11:15:50 EDT 
Clarifying...

Both old and new history implementations keep the *contents* of local history
under .history. The index location/format has changed. 

Your .history directory contains local history contents from both before and
after you migrated to the new format. Since the index for the old data is
corrupted, the new index does not allow you to retrieve the old history
contents, but they are still there along with the new history states.
Unfortunately there is no (reasonable) way to tell them apart, so you will have
to live with those unaccessible files in your file system.

So, Ron, you have seen these log errors during *every* startup on that workspace
since you migrated to the new format? This was not clear from your original
comments.

Comment 15 Rafael Chaves 2005-04-21 13:42:37 EDT 
Old summary: '"Could not open indexed store" error'

Fixed. Released to HEAD.

Problem was in IndexedStore#open. It uses the fact that the name is null to
short circuit the #close operation as a no-op (normal behavior closes the file
and then sets the file name to null). But the name is only set after the failing
operation here, #acquireContext, is called. This way, the name was not being set
due to the exception, and the call to #close was innefective. 

Fix was to set the name immediately after the underlying file has been opened.

(never thought I would have to fix a bug in the legacy code)
