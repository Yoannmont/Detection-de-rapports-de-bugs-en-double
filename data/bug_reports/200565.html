Description Chris Aniszczyk 2007-08-20 11:20:24 EDT 
I'm going to leave this open to discussion, but I would like to put this on the table. Mike and Noam have done an excellent job on moving the PDE text stuff along. I'd like to hear Mike on the possibility of this. The challenge is... we don't necessarily want to make the Text stuff API. Can we get away with x-friends or is this going against "the Eclipse way"

This would enable us to separate out the update/feature stuff into a separate plug-in and make it easy to drop once provisioning stabilizes. Furthermore, we can move the cheatsheet stuff into a separate plug-in also.

Thoughts?
Comment 1 Dejan Glozic 2007-08-20 11:49:12 EDT 
Chris, this is an interesting idea, but I don't see how feature/update stuff is related to it. I would expect to see o.e.pde.ui.feature or o.e.pde.ui.update prefix and feature/site code moved there.

Note also that there are related feature and site models in o.e.pde.core (I think but I may be mistaken).
Comment 2 Mike Pawlowski 2007-08-20 14:48:37 EDT 
The 'org.eclipse.pde.core' plug-in currently contains the following components:

* Interfaces (API)
  -> Used by text-based and non-text-based model frameworks

* Interfaces (Internal)
  -> Used by text-based and non-text-based model frameworks

* Model Utilities
  -> Used by text-based and non-text-based model frameworks

* Text-based Model Framework

* Non-text-based Model Framework

* Text-based Models
  -> Plug-in
  -> Simple Cheat Sheet
  -> TOC

* Non-text-based models
  -> Schema
  -> Target
  -> Feature
  -> Site
  -> Product
  -> Composite Cheat Sheet
  -> Build

* Builders


I do not think separating the text-based components out of org.eclipse.pde.core will help modularize the provisioning effort.  Ideally, all non-text-based models should be converted into text-based models. 

I concur with Dejan.  If we want to separate the cheat sheet related stuff and site / feature related stuff into separate plug-ins we will have to extract their model parts from org.eclipse.pde.core and their wizard and editor parts from org.eclipse.pde.ui
Comment 3 Chris Aniszczyk 2007-08-20 14:56:39 EDT 
"I concur with Dejan.  If we want to separate the cheat sheet related stuff and

This idea may not be containable but 3.4, but this is something we should definitely consider I think.

Thanks for your indepth comments Mike.

Comment 6 Dejan Glozic 2007-08-20 18:00:16 EDT 
After the wars, I totally lost the appetite for breaking things into smaller things :-).

Chris, you have started three topics here. All of them have merit, we just need to look which one is something we should try to attack in 3.4:

1) Factoring out the PDE XML text support
2) Partitioning the PDE along the plug-in/features divide
3) Partitioning PDE along the plug-in/UA tooling divide

1) makes sense if we want to expose reusable multi-page editor with a single incremental model available as an API. Otherwise it does not help us much.
Plus it has input contexts which I want to be buried with so that's all goodness :-).

Activity 2) makes sense but again we would need to offer the chopped-off feature tooling as an add-on feature in 3.4+ if needed.

Activity 3) makes limited sense if we want to build your own mashup (start with the platform, add JDT, add BDE (no features, no value-add stuff).

In general, these are all ideas worth looking at, but I would either stay with one big thing or break it in one big effort and be done with that (i.e. we don't want to spawn a new bundle every milestone or so :-).

Comment 8 Jeff McAffer 2007-08-20 22:48:17 EDT 
my thoughts FWIW...

- breaking out the various functional pieces is a good thing.  

- long term the model management parts of PDE will become more and more interesting in and of itself

- creating XML text support API is an interesting thing but is beyond the scope of PDE

- the tooling for the new provisioning work will likely have to be somewhat separate from standard PDE-like things so we will be directed down this path (of separate functional pieces) anyway.

- need to be careful about biting off more than can be chewed.  These refactoring efforts have a tendency to take on a life of their own.  While I fully support the ideas in the abstract, its not clear how all the work will get done.
