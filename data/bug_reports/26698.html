deep tree.  Since you're running JVM 1.4.1, a small stack is probably not the 
issue.

Comment 7 Nick Edgar 2002-11-26 12:32:52 EST 
The first stack overflow is due to internalSetExpanded recursing too deeply.
This is called from setExpanded.  setExpanded works in two passes:
1. for each element in the list of elements to expand, ensure that the 
corresponding TreeItem is created (and all its parent items).
2. call internalSetExpanded to recurse over all TreeItems in the tree, setting 
the expanded state of all TreeItems appropriately, and populating child items 
for expanded parent items.

Looking at the code, the second pass could possibly recurse infinitely if a 
TreeItem and one of its child TreeItems have the same element, and that element 
is in the expanded set.  This can occur if there is a cycle in the variables 
(not an unusual case).

It looks like the only caller of setExpanded in Debug land is in 
ViewerState.restoreState.  So it should be possible to construct a case where 
the viewer state for an object graph with a self-referencing element is 
restored, where that element was previously expanded.

Reproduced the problem by debugging the following program.

public class DebugVariableCycleTest {
	DebugVariableCycleTest ref = this;

	public static void main(String[] args) {
		DebugVariableCycleTest t = new DebugVariableCycleTest();
		m1();
	}
	
	static void m1() {
		m2();
	}
	
	static void m2() {
	}
	
}

Set a breakpoint and step into m1. 
In the stack frame for main, expand t until it looks like:

- t= DebugVariableCycleTest (id=13)
  - ref= DebugVariableCycleTest (id=13)
    + ref= DebugVariableCycleTest (id=13)

Select the stack from for m1, then back to main.
It crashes.

Should be able to fix this in the viewers by removing elements from the 
expanded set as they are encountered in internalSetExpanded.
