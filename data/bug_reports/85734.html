Comment 12 Darin Wright 2005-02-17 15:04:07 EST 
Fixed thread flicker in debug view. We insert/remove threads, and refresh 
other nodes on create/termiante (which means, processes and targets). This is 
OK since target/process creation/termination is rare compared to threads. 
However, it only works if the threads are returned from the target via 
#getThraeds() in the same order the create events are fired. This is the same 
behavior as 3.0 and earlier, so I will release the fix. However, I'd also like 
to look for a fix for the flickering Linux tree.

Comment 16 Jared Burns 2005-02-17 15:49:09 EST 
I'm looking across all the changes in this area in the past couple days and
trying to understand what it's all about. It looks like we went from always
inserting to always refreshing. But that caused problems, so now we're back to
inserting threads but refreshing otherwise. There are a couple things I don't

1. Why do we ever want to refresh instead of inserting? (I've seen the original

2. Why are we still calling refresh for other objects? If it's okay to insert
threads, which come and go more than anything else, shouldn't it be okay to
insert everything else (targets/processes?)?
Comment 17 Darin Wright 2005-02-17 15:54:54 EST 
When you insert(parent, child), the child will be inserted as the last child 
of the parent, unless their is a sorter in the view (we don't have a sorter in 
the view, so the child will be inserted last). 

The problem:
* a launch is started
* a process fires a create event and is inserted (1st)
* a debug target fires a creat event and is inserted (2nd)
> but, if you call #getChildren() on ILaunch, the debug target comes back 
first. So the event firing order does not match the real order of the children 
(and there is no sorting).

> for threads, the order seems consistent, so it does not cause trouble.
