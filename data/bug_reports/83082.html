Comment 4 Helen Beeken 2005-01-20 08:57:50 EST 
When we create our own version of the structure model (in AJProjectModel) and 
process the relationships, we have a workaround which says "if this already 
exists, then don't add it to the list". This was a workaround for AJ bug 
80916. This bug is now fixed so we should no longer require this workaround.

Removing this, now means that the problem of only seeing the one entry (since 
our code was seeing the relationships as the same thing, and therefore only 
adding it once) is no longer there. However, now we get duplicate entries (two 
for the before advice and two for the after advice) for each of the "advised 
by" entries when you click on the main method. I've also seen the crash occur 
under these circumstances. 

The duplicate entries may be a problem with the underlying structure model 
which we get from AspectJ. I still need to investigate the crash.

Thanks, Helen
Comment 5 Helen Beeken 2005-01-20 11:35:13 EST 
The relationship map is correct - it's just that since we're unable to tell 
the difference between the two entries (lines 17 and 18 in the Main class - 
the two calls to sysout) (we currently don't record any difference between 
these) then we're returning all the entries for each one. Hence each one is 
said to be advised by 4 places. Obviously, the fix requires recording more 
information in our AJDT version of the map so we can tell the difference.

Thanks, Helen
Comment 6 Helen Beeken 2005-01-26 08:42:49 EST 
The problem was that AJCodeElement extends LocalVariable and LocalVariable
overwrites the hashCode() and equals() methods. The bug was that we had, what we
considered to be two different AJCodeElements (same parent, same name but
different line numbers) were actually being considered equal and having equal
hashCodes(). The fix was to overwrite the hashCode() and equals() methods in
AJCodeElement to include the line number. This fix is in and we're just waiting
for a build. I believe that the crash possibly has to do with this (I've done
some testing since this fix and am unable to reproduce the crash).

The final thing which I'm attaching to this bug is that the ordering of what is
displayed is slightly unintuitive. In the case described in this bug, the two
calls to sysout in the main method of Main are on say, line 18 and 19. However,
the one corresponding to line 19 appears in the Cross Reference view above the
one corresponding to line 18. I've therefore written an AJComparator to use to
give a more logical order to the view. The tests for this are being finalized
and will be checked in shortly.

Thanks, Helen
Comment 7 Helen Beeken 2005-01-31 09:26:47 EST 
The final fix is in, just waiting for the build machine to be back on line and

The fix was to order the children in the getExtraChildren(..) method in
AJXReferenceProvider.
