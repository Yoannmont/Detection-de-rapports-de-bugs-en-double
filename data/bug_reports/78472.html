Description Sergey Prigogin 2004-11-11 22:53:37 EST 
When ProgressManager.busyCursorWhile method is called with a modal dialog open,
it shows a hourglass cursor, but does not disable the modal dialog. A user can 
still click on the the dialog controls causing them to produce their normal 
actions.
Comment 1 Tod Creasey 2004-11-18 08:36:40 EST 
We only disable the active window by design. Why do you open this from a 
dialog if you are going to get 2 dialogs potentially? I would have thought 
showing this behavior in your own dialog would be a better idea.
Comment 2 Sergey Prigogin 2004-11-18 13:22:27 EST 
Javadoc for IProgressService.busyCursorWhile says:

  Set the cursor to busy and run the runnable in a non-UI Thread. The calling
  thread will be blocked for the duration of the execution of the supplied
  runnable. After the cursor has been running for getLongOperationTime()
  replace it with a ProgressMonitorDialog so that the user may cancel. Do not
  open the ProgressMonitorDialog if there is already a modal dialog open. 

I'm calling this method from a modal dialog so it doesn't open the 
ProgressMonitorDialog. I expect it to show the hourglass cursor and DISABLE ALL 
SHELLS for the duration of run. Unfortunately, I cannot work around the current 
wrong behavior because even if I disable all shells, they will be reenabled by 
IProgressService.busyCursorWhile BEFORE my run method returns.

If you look at the source code for ProgressManager.busyCursorWhile you can see 
that it initially disables all shells by calling setUserInterfaceActive(false) 
but then reenables them. The problem can be solved, at least partially, by 
calling setUserInterfaceActive(false) again from busyCursorWhile(Runnable 
dialogWaitRunnable, ProgressMonitorJobsDialog dialog) just before 
BusyIndicator.showWhile(display, dialogWaitRunnable);
Comment 3 Tod Creasey 2004-11-18 13:29:39 EST 
Segey we adjust the priorities for our scheduling - if you want to point out 
something is more important to you please adjust severity (this is why we 
don't show proirities when you log the bug).

The re-enablement is by design - we only want to lock out the user for the 
initial time (800 ms) because we do not want the UI to become unresponsive. If 
the user is still locked out after 800ms they will perceive a deadlock in the 
UI.

Keeping the UI disabled after this point will defeat the purpose of the API. 
The user may want to cancel if something is taking too long.
Comment 4 Sergey Prigogin 2004-11-18 13:54:51 EST 
The current behavior simply doesn't make sense. You reenable the UI but the 
cursor remains to be hourglass. Do you expect a user to use an hourglass cursor 
to click on something?
Comment 5 Tod Creasey 2004-11-18 14:21:22 EST 
I think I don't understand what your issue is then. Is it that the hourglass 
cursor is the issue for you?

The enable/disable behaviour is correct and is working as designed. If your 
operation is still running there is no other way for me to indicate that we 
are not done. If I open the dialog I run the risk of blocking out something 
the user needs to respond to so I do not. 

If I clear the cursor the user thinks that the operation is complete but it is 
not. I really think you want to consider using a ProgressMonitorPart in your 
dialog to show progress and not this API.
Comment 6 Sergey Prigogin 2004-11-18 14:47:53 EST 
Would you consider adding a variation of ProgressManager.busyCursorWhile method 
with an extra boolean parameter, so that I don't have to reimplement 
ProgressManager.busyCursorWhile method from scratch?
Comment 7 Sergey Prigogin 2004-11-18 14:54:28 EST 
Let me clarify the meaning of the proposed boolean parameter. It should control 
the state of the UI after 800ms interval expires and the ProgressMonitorDialog 
does not open due to an already active modal dialog.
Comment 8 Tod Creasey 2004-11-18 15:48:08 EST 
So where would the user get feedback that something modal is still going on?

The purpose of this API is to prevent the user from doing anything except 
cancelling or moving the job to the background and continuing work.

If I enable the UI then this is the same as saying Run in Background. The 
problem is that you are not asking the user first. I really don't think this 
is a good idea.
Comment 9 Sergey Prigogin 2004-11-18 16:35:50 EST 
In my situation an ability to move the task to background would not gain 
anything since the modal dialog cannot be closed before the task completes. The 
dialog waits for the results of the task to display them. The task cannot be 
canceled since it spends most of its time inside third party code, which I 
don't have control over. I could move the task to a separate thread and pretend 
that it was canceled when, in fact, it would continue to run. This approach is 
problemetic though due to reasons I don't want to explain here. Even if I 
choose this approach, I would need ProgressManager.busyCursorWhile to change 
the cursor back to normal after 800ms. The simplest although possibly inferior 
from user's perspective approach is to keep the UI disabled for the duration of 
the task and use the hourglass cursor as an indication for the user that it is 
still running. The task itself is deterministic and cannot possibly cause a 
deadlock.

>If I enable the UI then this is the same as saying Run in Background.
>The problem is that you are not asking the user first.

I don't get it. I'm asking for IProgressService.busyCursorWhile to keep the UI 
disabled as long as there is no ProgressMonitorDialog that would allow user to 
cancel or to move the task to the background. This would affect only the case 
when there is already a modal dialog open.
Comment 10 Tod Creasey 2004-11-19 07:31:51 EST 
Shouldn't have mentioned the idea of the background work it clouded the 
discussion -sorry.

So what I don't get is the difference between what you want and 
BusyIndicator.showWhile(). After 800 ms the operation is still not done so 
stopping the busy cursor is not really appropriate as the operation is still 
running. 

If you want the user to be able to click on your dialog when whether they wait 
for 800ms or not doesn't matter if the operation is still running.

The decision about not blocking the UI if there is a dialog open is not as 
simple as seeing if there is a progress monitor dialog somewhere - we need to 
re-enable if there any modal shells as we can never be sure why it is there - 
we can't guess what is going on in 3rd party code. 

What we are trying to avoid is the case where the runnable opens a dialog and 
we prevent the user from responding. If we ever had a condition where we did 
not re-enable the UI it would be possible for 3rd party code to lock the UI 
indefinately.
Comment 11 Sergey Prigogin 2004-11-19 13:09:51 EST 
The difference between the current behavior of BusyIndicator.showWhile() called 
from ProgressManager.busyCursorWhile and the behavior I need is the state of 
UI. I need UI disabled for the duration of BusiIndicator.showWhile() while 
currently it is enabled. Notice that if you put setUserInterfaceActive(false) / 
setUserInterfaceActive(true) around the call to BusyIndicator.showWhile(), it 
will produce the behavior I need but wont affect the ProgressMonitorDialog.
Comment 12 Tod Creasey 2004-11-19 13:51:38 EST 
Right - now I see. All showWhile gives you is a cursor change and you want to 
disable the UI for the user.

Is your dialog modal? If it is then all you need to do is disable your dialog.
Comment 13 Sergey Prigogin 2004-11-19 14:03:30 EST 
I'm not calling BusyIndicator.showWhile(), I call 
ProgressManager.busyCursorWhile since I need my code to work with and without 
an active modal dialog. Even if I disable my dialog, it will be reenabled after 
800ms by ProgressManager.busyCursorWhile.

Please refer to the comment #2 and comment #11 for the change I want you to do.
Comment 14 Tod Creasey 2004-11-19 14:24:41 EST 
The behaviour you want would disable the UI for longer than 800ms with no 
ability for the user to cancel. This is not something we would ever encourage.
Comment 15 Sergey Prigogin 2004-11-19 14:27:11 EST 
This discussion in going in circles. Would you mind assigning this bug to 
somebody else.
Comment 16 Tod Creasey 2004-11-19 14:41:51 EST 
Sergei I am the component owner for this component so assigning it to someone 
else won't resolve anything.

I still don't see a compelling case to add your API request so I am going to 
mark it won't fix.
Comment 17 Sergey Prigogin 2004-11-19 14:44:14 EST 
I guess I'll have to implement my own ProgressManager.
Comment 18 Tod Creasey 2004-11-19 14:53:20 EST 
Sergey

I would really like us to both be clear on the design goals we each have - you 
are right that we talked in circles I think and I want to be sure you 
understand why we have done things the way we have and the concerns I have. If 
I understand what you are asking for I think you are going down a potentially 
problematic road in your implementation.

One of the primary goals of the progress support is to prevent the user from 
ever getting locked out. We have settled on an 800ms limit to locking out the 
user as this is what most current research believes is the longest time a user 
will tolerate non responsive UIs.

If any of our code locks out the UI for more than that time than we have 
failed in that goal. 

Doing something like

disableUI
Run for 5 minutes
enableUI

Would appear to the user like a deadlock. How would your suggestion not fall 
into this situation?
Comment 19 Sergey Prigogin 2004-11-19 15:56:07 EST 
I agree that
 
disableUI
Run for 5 minutes
enableUI

is not aceptable, but I think that

Show a message explaining what's going on
Show an hourglass cursor
disableUI
Run for 5 seconds
enableUI
Switch to normal cursor

is acceptable when a developer doesn't have another alternative. I would be 
glad if I could move the lengthy activity to background, but I can't. I've 
displayed it at length in my previous comments.

I don't think the purpose of the platform is to dictate third party developers 
how to implement their plugins. The purpose of the platform, as I understand 
it, is to empower developers and encourage them to follow the best UI pactices 
without closing doors for reasonable things developers may need.
Comment 20 Tod Creasey 2004-11-19 16:21:56 EST 
I agree completely - we are doing our best to not force you into any decision.

As a framework we also need to provide what we think are generally good 
mechanisms for people to use. We need to make sure that people who use our 
convenience methods get a similar experience with thier code as the get with 
ours. As a result we have to decide on some principles and draw the line 
there. We also need to make sure that poorly designed plug-ins (i.e. one with 
a 5 minute operation) do not freeze out users.

Some people would disagree with 5 seconds being acceptable - we chose 800 ms 
as that number has some good research to back it up. You of course are free to 
do what you like but we don't have that flexibility as any change we make is 
potentially used by everyone.

Sorry we can't be of more help to you Sergei but I think this feature is 
something that might make sense for you but is open to locking all of Eclipse 
if it is used by someone who uses a long Runnable.
Comment 21 Sergey Prigogin 2004-11-19 17:22:06 EST 
OK, I accept your position on babysitting developers.

Still ProgressManager.busyCursorWhile violates a fundamental principle of one-
to-one relationship between the cursor shape and the state of UI:

1. Hourglass cursor means disabled UI
2. Any other cursor means enabled UI

If you don't agree, point me to a Windows application that does not follow 
these rules.
Comment 22 Tod Creasey 2004-11-22 08:03:07 EST 
Ok then - I should change the busy cursor back when there is a blocking 
dialog. See Bug 79176 - I logged a new one as this one is pretty long.
