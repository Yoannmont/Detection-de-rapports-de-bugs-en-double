Description Markus Karg 2003-10-20 07:58:19 EDT 
Typing in all that include lines in large projects is boring and sometimes 
failure prone: Often developers type in all possible includes into one large 
"project.h" header that in turn is includes by any other .h or .cpp file. This 
can lead to problems if one needs a same-named class from another library 
different to one included in the "project.h". This behaves a bit like doing 
imports in Java. So it would be a really great feature to have a similar 
functionality in CDT. It could be "source -- organize includes" and could ask 
for the include that is needed if a class in defined in two different .h files.
Comment 1 John Camelon 2004-04-02 15:23:10 EST 
The order of inclusions matter in C++, but they don't (for imports) in Java. 
While this feature would be nice to have, it requires some significant 
maturity in our parser/indexer infrastructure to allow for us to make some of 
these decisions for the user.

Comment 13 Mathias Kunter 2012-03-04 07:52:26 EST 
I just wanted to let you know that I've been playing around with this a little because I wanted to have this feature for a current project I'm working on. I've patched my local CDT, and the fix is working surprisingly well IMHO. Here is what I did:

-> Reused the existing code from the "Add include" feature for determining how to include a header, inserting the include directives into the editor, etc.

-> Reused the existing code from the "Open declaration" feature to resolve a symbol (i.e. an IASTName) to a header file.

-> Looped through all IASTNames of the current file and resolved their declarations in order to obtain the list of headers which should be included.


As this solution was already working nicely, I also thought about using forward declarations instead of full includes where possible. I added some code which decides whether an include directive is required depending on the parent / ancester / sibling nodes of an given IASTName. While this code also works quite well, it's not yet completely accurate. I simply didn't want to take the time and look into all the details of the AST nodes hierarchy to decide when exactly a forward declaration can be used in C++. I instead just covered the more basic things like e.g. using a forward declaration when declaring a function or defining a pointer type etc. Btw, http://stackoverflow.com/a/553869 has a nice (but unfortunately incomplete) summary about using forward declarations.


If it should be of any interest, I can of course gladly share my local patch. This code however certainly won't meet any Eclipse coding standards, it's more just a quick copy and paste solution which I came up with to see whether this thing generally works. So you wouldn't want to commit this code anyway, but maybe it can still be useful for anybody :-)


What I personally would really like to see when seriously implementing this feature is a separate preferences page where I can fine-tune the behavior of this feature. I think this is important to meet the requirements of different users. I was thinking about the following preference options:

-> Should forward declarations be used when possible? If yes, what should be forward-declared? Classes, structs, unions, enums (only possible since C++11), etc.?

-> Should unused headers be removed or simply left alone?

-> How should a project header be included? Relative to the current file? Relative to a specific include path? A mixture?

-> Should the includes be ordered in some specific way? For example, should library headers be placed before project headers? Should the includes be alphabetically ordered? Should the forward declarations be placed after the includes?

-> etc.
Comment 14 Mathias Kunter 2012-03-14 18:48:21 EDT 
Created attachment 212691 [details]
provisional implementation of the organize includes feature

I've kept working on my provisional implementation of this feature for my local CDT. I've now came up with a solution which worked really well for me to organize the includes and forward declarations of a ~ 100 K lines of code project. I thought I'd share my implementation by uploading the patch here. This is provided with the hope that it can be useful somehow.


To quickly summarize this patch:

-> Adds an "Organize includes" entry within the "Source" submenu of the context menu
-> Adds all missing header files and forward declarations to the current file
-> Considers qualified names, templates, declarations, functions, constructor chain initializers, new keyword, catch / throw, field references, expressions, and some more C/C++ code constructs in order to decide whether a forward declaration is sufficient or not (see the getASTNames() function for details)
-> Supports several different parameters which control the mode of operation (currently hard-coded, but can of course be easily turned into user preference options).


Things which are yet missing:

-> Remove existing unused includes and forward declarations in order to have a true "organize includes" feature - this current implementation is actually an "add missing includes" feature
-> Handle namespaces
-> Improve the handling of templates (very difficult regarding forward declarations)
-> A few other things like e.g. forward declarations of enums which became possible with C++11 etc.

Things which can / should be improved are marked with TODO annotations within the code. If there are any questions about this patch, please just ask.

Comment 47 Sergey Prigogin 2013-03-14 21:24:51 EDT 
The bug_45202 branch has been merged to master. Organize Includes command is added to the appropriate menus and sometimes even produces reasonable results. The major remaining areas of work are:

1. Forward declarations have to be formatted according to user preferences.
2. Unused forward declarations are not removed.
3. New forward declarations are not properly ordered with respect to the existing ones.
4. Header substitution is currently working only for the standard gcc headers. User-defined substitition rules and pragma-driven header substitution haven't been implemented yet.
5. Add Include command doesn't yet obey new include style preferences.

Please try Organize Includes on your code and start filing bugs. Any help with the remaining work will be highly appreciated.
