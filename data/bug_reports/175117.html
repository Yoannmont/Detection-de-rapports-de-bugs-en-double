Description Tod Creasey 2007-02-22 09:02:23 EST 
M5

Currently we have several ways to get ErrorDialogs that do not go through the status handling - we can either call the constructor or use our own class (there are 10 in the SDK alone).

Our strategy of overriding the static method hits many cases but there are many quite common ones we will miss (like CVS errors for instance).

I think we want to look at a way to extend the current error dialog rather than override the static method.

Comment 3 Tod Creasey 2007-02-28 09:58:15 EST 
Created attachment 59983 [details]
Suggested implementation

This patch adds a new error support provider to JFace. When this value is set in the Policy error dialogs will have a third column that can be filled in by the provider. I want to pass this by Christophe before I make the API request.

We likely won't ship one with Eclipse but this will allow other products to plug in to the error dialog in a non breaking way - I even endeavour to keep the order of layout creation the same for backwards compatibility.

The number of columns was hard coded before - I now have a package visible method to provide this so that we do not need to add API.

ErrorSupportProvider has a single method

/**
 * Create an area for adding support components as a child of parent.
 * @param parent The parent {@link Composite}
 * @param status The {@link IStatus} that is being displayed.
 * @return Control
 */
public abstract Control createSupportArea(Composite parent, IStatus status);
Comment 4 Tod Creasey 2007-02-28 10:24:28 EST 
API Request

Add the following to JFace

New class: org.eclipse.jface.dialogs.ErrorSupportProvider. A singleton used to provide the contents of a Composite on the left hand side of ErrorDialogs to allow applications to plug-in thier own support information.

New methods: 

int org.eclipse.jface.dialog.IconAndMessageDialog.getColumnCount()
   the number of columns in GridLayout of the top level Shell

org.eclipse.jface.util.Policy 
   void setErrorProvider. Set the ErrorSupportProvider singleton
   ErrorSupportProvider getErrorProvider. Get the ErrorProvider singleton

RISKS
   Existing error dialogs that re-implement createDialogArea without calling super() will not get this support.
   Existing error dialogs that hard code the number of columns in thier button bar will no longer be right justified when this support is installed.

RISKS OF NOT SUPPLYING THIS SUPPORT

ErrorDialogs will not be part of the 3.3 Servicability support and all explicit calls to ErrorDialog and it's subclasses will not provide this service.

Comment 9 Tod Creasey 2007-03-02 09:09:24 EST 
createDropDownList (the method we call to create the details area) is API. it would mean that in the case where the status handling is implemented we will not call this method so implementors of it will not be referred to.

createDropDownList returns an SWT List so it gives no flexibility for us to change what it sends back. Quite a bit of our code assumes that the list is there.

This is a less subtle breakage than adding a third column - if we were to replace the details area we will break API by not referring to an API method. Adding a third column will break people who hack the dialog by reading the code. 

We could document this in the porting guide however - the API makes no promises about when if createDropDownList is called but in think we would just be banking on the fact that our API was vague rather than being good compatibility citizens.

Comment 12 Tod Creasey 2007-03-02 10:34:02 EST

Patch with getCoumnCount removed from new API

After discussion with McQ we decided that the button bar will keep it's column span of 2 and that the support area will be the full height of the dialog.

This will allow the following improvements
   1) Consistency with the Tray Dialog
   2) No breakages of button bars
   3) No exposing our layout to API (getColumnCount is now package private)

This patch now only adds the changes required in Policy and ErrorSupportProvider. There are no changes required in our other dialogs to support this.

I have also tidied up some layout issues in this patch.

Comment 19 Tod Creasey 2007-03-08 10:04:02 EST 
Correct. We need to handle the case where you want to block the Ui and the cases where you don't otherwise everything has to be modal to be safe.

Szymon and I talked about this and I think we can simply the flags even further

LOG for logging
SHOW for non blocking presentation
BLOCK for a blocking presentation (such as a modal dialog)

Then we should punt NONE and NON_BLOCK as both are meaningless. This approack means that we have one set of state determined by our style flag so the strange cases like LOG | BLOCK don't need to be dealt with.
