Comment 1 Boris Bokowski 2007-05-02 08:21:13 EDT 
Michael has nothing to do with this :-)

Why does StandardJavaElementContentProvider.getChildren() cause a Workspace operation (which is potentially long-running)?  As can be seen in the stack trace, this leads to a reentrant call into the same TreeViewer instance that called StandardJavaElementContentProvider.getChildren(), except this time, refresh() is called. This can alter the internal data structures, which in the worst case leads to NPEs later on (when the control returns to AbstractTreeViewer.isExpandable()) because the internal data structure of the TreeViewer may be corrupt now.

The JFace viewers are not designed to handle reentrant/recursive calls like this, but in the past never detected this situation, they just threw NPEs as they happened. ;-)

If JDT cannot fix their content provider for M7, I can remove the check for being busy in the viewers.  However, the problematic reentrant calls will still happen.
Comment 2 Boris Bokowski 2007-05-02 08:59:46 EDT 
I looked at the stack trace again, and here is why it is correct that the viewer ignores the call to refresh() and writes to the log:

CommonNavigator.createPartControl() creates the tree viewer and initializes it. As part of that (probably the last step) it sets the input, upon which the viewer initializes the tree. It does that by getting the root elements and iterating over them, asking whether any of these elements have children so that it can show a plus sign for the expandable items. It is unsafe to let a refresh operation happen while the viewer is still iterating over the root elements, because the refresh could change the structure of the tree in unexpected ways.

The exceptions are due to the fix for bug 154571, in ColumnViewer. ColumnViewer is new for 3.3, and the isBusy() method is new as of >20070426-0010. Clients really shouldn't be causing re-entrant calls through the viewer, via async() or not, but many clients currently do. For 3.4, we will have to identify these, using the log message. 

Assigning back to Boris to carry out this fix. 

My concern with just identifying this specific issue in the JDT extension, while I think it should be corrected, I'm sure there are going to be downstream clients that will not be able to react in time for 3.3 (in CDT/WTP/adopters). Once the Platform moves into RC mode, it will be hard for these consumers to push for a fix in the platform layers. Particularly, I know for a fact that any downstream contributors of Java classpath containers will trigger workspace operations that can result in notifications back through to the viewer. WTP has this issue for J2EE Classpath Containers.

Comment 28 Martin Aeschlimann 2007-11-05 07:36:26 EST 
Yes, in the package explorer we could always delay the UI refreshes while the viewer is busy.

That's what is happening in the stacktrace of comment 12:
a.) content provider is asked to get the children of a project
b.) the Java model is populating on request. Getting the children of a project the first time will initialize class path containers, what can result in a Java element deltas. If I understand right, this happens when the resulting roots are different to the roots seen previously (last workbench start). JDT core knows more.
c.) the delta is processed in the package explorer and all changes are kept in a change queue. An 'async-exec' runnable is posted to process the queue at a later moment.
d.) The EventLoopProgressMonitor serves the UI queue on calls like isCancelled and processes the async exec. Because of this, the change runnables are already processed, while getChildren has not yet finished.

With API access like 'isBusy', we can delay the processing of the update runnable when we detect that the async exec is called in busy time: we would just repost the async exec.

Some notes:
- I can rebuild the 'busy' flag in my tree viewer by overriding getRawChildren. But it would be easier to access the existing flag.
- Changing the Java model to not send out deltas while children are accessed would be, if possible at all, a complicated change that JDT core isn't planing to do.
- The situation shows a problem of using 'async-exec' to post something after a given action. The display loop is served at several unexpected places like the EventLoopProgressMonitor.
Implementing a 'post event after' is not really possible with async-exec, but I'm not aware of a better replacement. We have similar problems when posting UI events out of a build notification (resource or Java delta). 
- Other viewers don't post UI changes but refresh in sync. This would lead to the same problem here. I don't know what the solution would be with such viewers.
Comment 29 Boris Bokowski 2007-11-06 12:53:25 EST
