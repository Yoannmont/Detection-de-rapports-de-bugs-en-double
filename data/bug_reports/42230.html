Description Timour Mouratov 2003-08-28 13:43:32 EDT 
Build: I20030826 
When I try to edit read-only files in java editor, content of the file is 
replaced by the window with one line inside:  "File ........... is read-only." I 
have to close and open that file again in order to see the source.

Regards.
Comment 1 Dani Megert 2003-08-29 08:44:43 EDT 
This is a bug which also happens in the standard Text editor
==> changing Product to "Platform".

The new CompilationUnitDocumentProvider2 calls validateState on the FileBuffer
which sets the status to the return value of validateEdit which is an error status.

During sanitCheckState the status of the file buffer is requested which is an
error status and therefore the editor's input is discarded and the error status
is shown as content.

Comment 15 John Arthorne 2003-10-16 11:53:57 EDT 
Core change has been released.

Michael is currently applying the change for the team hook. It should be in
tonight's nightly build.  Moving to Platform text for handling of this new
status on their side.

Note:

Since validateEdit can take several files as a parameter, there are some subtle
cases, but this shouldn't affect the text editor usage.  For the record, this is
what you'll see:

-> If one read only file is provided, and there is no team provider, you will
get back a status or MultiStatus whose code is IResourceStatus.READ_ONLY_LOCAL
(severity ERROR).

-> If one read only file is provided, and there is a team provider, they will
return OK if the file has been made modifiable, and ERROR if it could not be
made writable (checkout veto).

-> If multiple files are provided, a MultiStatus will be returned whose severity
will be OK if all children are OK, and ERROR otherwise.  Child statuses may or
may not have READ_ONLY_LOCAL code, depending on whether there is a team provider.
