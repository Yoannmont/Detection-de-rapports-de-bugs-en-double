Description Ed Burnette 2004-05-26 17:11:00 EDT 
[Post 3.0]
I've read bug 33429 and bug 33428 and bug 13524 but still need the 
functionality of sending a signal to a running process launched by Eclipse, 
especially an external program.

There are many use cases, but one is to send a ctrl-break to a Java program to 
make it do a stack dump (I know about the java.exe workaround but it is 
insufficient because I want the output spooled in the console so I can cut-n-
paste and search etc.).

Another use case is to send a ctrl-c interrupt to any external program and 
capture the output as it traps the interrupt does whatever it does (which may 
or may not involve terminating). For example I have a user request to catch 
ctrl-C in a utility I wrote and produce a summary report to stdout before 
exiting and I want to capture that.

Another use case is launching a shell; to really make that work we'd need to 
be able to make key bindings that would take ctrl-C and ctrl-D and ctrl-\ and 
send the right signal to the shell. I could sell Eclipse to a lot more command-
line centric developers internally if we could make a descent shell inside it 
like Emacs has (see also bug 36669).

If there are no portable ways to do this then non-portable ways are 
acceptable. If you can only make it work with external programs and not with 
java launches that would also be ok.

Comment 28 Paolo Denti 2018-01-24 13:37:06 EST 
Instead of trying to solve the problem trying to get the PID in java, falling into native implementation issues, a different approach could be taken.

The "stop" button could be made user configurable with a custom behavior, in the "run configuration".

As a basic example, a Linux/OS X user could customize the kill behavior in such a way executing the following command:

PID=`ps ax | grep <main class> | grep -v grep | awk '{print $1}'` ; [[ "" != "${PID}" ]] && kill -SIGTERM "${PID}"

In such a case, it would be fully portable avoiding any native issue.
Comment 29 Alexander Wessel 2018-08-03 22:43:31 EDT 
(In reply to Paolo Denti from comment #28)
> The "stop" button could be made user configurable with a custom behavior, in
> the "run configuration".

Hi Paolo,
I think this is a very good idea. A launch configuration should provide the option to get the default behaviour (whatever the stop button does now), or to perform another action. The most flexibility and configurability would actually be provided by simply allowing to delegate the stop action to *another* launch configuration! For someone that knows where to make the change, that should be fairly simple to implement, actually.

Using nginx I have the problem that it should not be terminated, but gracefully closed. Thus, I have a "start nginx" and a "stop nginx" launch configuration, both favorited, so they are readily available and don't switch positions in my launch history. But I do hit the terminate button all the time, requiring me to use the console to kill the leftover process (nginx starts two)... Especially in the console view, I'd love to have the "stop" button perform whatever is configured in the launch configuration, and a separate terminate option in the menu, just in case.

Also, I second the opinion that for the default behaviour, Java 9's new Process methods (which are also platform independent), _should_ be used when available and where appropriate.
