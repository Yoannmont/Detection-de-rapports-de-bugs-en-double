Comment 1 Ed Merks 2007-08-22 13:16:05 EDT 
Martin,

The Xerces validator doesn't validate the unique particle attribution constraint by default, and that constraint is very expensive to validate because it involves building a state machine for the content model. I'll try to found out the option to turn this on for Xerces so you have a more apples-to-apples point of comparison.  In any case, I classify making things faster as an enhancement request and given my resource constraints (i.e., I have to do all the work personally) I doubt I'll have time any time soon to make this faster.
Comment 2 Elena Litani 2007-08-22 13:51:27 EDT 
To turn on full schema checking you need to set the following Xerces feature http://apache.org/xml/features/validation/schema-full-checking to true.
See http://xerces.apache.org/xerces-j/features.html#schema-full-checking for details.

Comment 4 Mark Goodhand 2008-01-03 13:35:13 EST 
Our product uses XSD to validate and manipulate schemas that sometimes contain extremely large content models.

One public schema currently takes 1.5 hours to open, which may as well be forever, as users will terminate the program before it completes.

To work around this we are considering the rather drastic step of (selectively) disabling XSD validation, but we would welcome a fix for this bug, or suggestions for a better work-around.

We will conduct further profiling to see if anything pops out, but the XSD code in this area is really quite complicated, and we don't feel well placed to produce a patch.

Comment 6 Ed Merks 2008-01-04 15:42:20 EST 
Created attachment 86226 [details]
Improvements for state machine building...

Guys,

Would you be able to test the improvements from this patch?  I did quite a few things to make the basic processing faster before I noticed a bottle neck in the algorithm itself.  In particular, I was avoiding adding null transitions during the epsilon closure computation when it struck me that adding the null transitions for the states reachable via a null transition and then removing the null transition itself to avoid revisiting states whose transitions can only change by virtue of the closure of their null transitions (which I don't need to know about if I've added them to the originating state) would be a better approach.  This resulted in a massive improvements by avoiding a lot of duplicated processing.  

It would be nice if you guys could confirm that.  Did you know that Eclipse accepts donations? :-P

http://www.eclipse.org/donate/donorlist.php

Comment 9 Stephen White 2008-01-07 10:14:26 EST 
Thank you very much for your work on this.  I've applied your patch to the head of the 2.3 maintenance branch of XSD and the results are very encouraging when compared to the 2.3.1 XSD release, that we've been using to date, or to an unpatched copy of the 2.3 maintenance branch.

One real-world taxonomy with a reasonably large tuple content model (the uk-ir-ct-v1.6 XBRL taxonomy) previously took about 150 seconds to load and validate on my workstation is down to less than 45 seconds.

The (nz-gp-2004-05-12) taxonomy that I'm told took around 1.5 hours previously (I haven't had sufficient time to leave this running on my workstation yet) opened in 7 minutes when I tested it earlier this morning (with your patch from comment 7).  With your latest patch (comment 8) this now opens in under 5 minutes.

The times I've got include work done by our XBRL processor and validation engine.  I'm now going to do some proper profiling to see how much of the remaining time spent opening these taxonomies is down to XSD and how much is inefficiences in our own code.

Thanks again for your work on this, it has clearly made a huge difference.  I will pester management to see if we can arrange a donation in appreciation of your work.
