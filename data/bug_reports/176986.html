Description Tod Creasey 2007-03-12 09:52:42 EDT 
We should consider moving the tasks into the problem view as currently no one is using them in the tasks view.

The main issue this could create is performance - problems change rapidly but tasks are generally pretty static and it may be a fair amount of extra time spent sorting them.
Comment 1 Mik Kersten 2007-03-19 00:30:32 EDT 
++1  This should make the Task Tags mechanism considerably more usable.  I also believe that the performance profile could be a non-issue if the result of the move is configurable enough and if the defaults are set right.  

A typical manifestation of the current problem with the Tasks view is that users have a kazillion of TODO tags all over their code because Eclipse generates these by default.  Most of the time they don't want to see all of these tags.  This could be addressed ensuring that only tasks of a certain priority are shown in the Problems list (i.e. merging the current Tasks view filter mechanism with the Problems filter mechanism).  I think that this is doable simply by adding a "Where priority is: ..." right below "Where severity is: ...".  This will enabletags such as "FIXME" or "XXX" showing up in the Problems list without having that list be overloaded by tons of low-priority tags that are only relevant when editing the corresponding code, or when doing code reviews.  If the filter is set on priority="High" by default then the default settings in Preferences -> Java -> Compiler -> Task Tags should provide the right default behavior.

One effect of this merge is that all Task Tags will now show in the Problems view, and that the tasks that can currently be added via "Add Task" may no longer have a natural place.  In my opinion this would be a big improvement, because mixing resource marker kind tasks with manually-created tasks does not work well.  When doing the initial usage study for Mylar we noticed that this feature of Eclipse was hardly used at all, and I think that Mylar has demonstrated that to be useful it needs a considerably more thorough UI.  So this merge would work beautifully for Mylar users since it would make the separation clear and get rid of one of our oldest FAQ entries: http://wiki.eclipse.org/index.php/Mylar_FAQ#Does_the_Task_List_replace_the_Eclipse_Tasks_view.3F  Do you see this as getting rid of the Tasks view entirely?
Comment 2 Mik Kersten 2007-06-19 18:09:11 EDT 
Mylyn has a major incentive to do this since it will reduce confusion between the Task List and Tasks views.  

Tod: please feel free to assign to me and set for 3.4.  We could commit to doing it within the first 6 months of the cycle and could consider doing it sooner.  Before starting on it we should iterate on a design to make sure that the change is suitable for Platform UI when Mylyn is not installed.
Comment 3 Tod Creasey 2007-06-20 10:19:01 EDT 
I should hold onto it as the performance implications can be pretty significant. I'll be honest with you Mik that this is a hard enough update/performance problem that I am not convinced you will be able to do it without a lot of help from us.

I am of course happy to see what you come up with but I expect it to be hard.

Remember that Tasks and Problems have different columns - I am not sure that you will be able to come up with a decent UI for both.
Comment 4 Mik Kersten 2007-06-20 18:48:52 EDT 
Thanks for being clear on that Tod, it definitely makes sense that this would require help to design and do properly.  I have scheduled it for September, if you think it makes sense to start looking at it before then please post.

Regarding design, my inclination would be to make priority be severity and lose completion or make that an overlay over the severity icon and contribute an action for toggling it for markers that it is relevant for.  Mylyn's Task List does not have a column for completion and nobody complains.
Comment 5 Tod Creasey 2007-06-21 08:30:32 EDT 
A loss of function always gets us into trouble so losing the completion would mean the tasks view would have to stay (although it would not need to be part of the resource perspective then as we could replace it with the problems view).

A bigger issue is the priority of tasks and the sorting mechanism in the problems view right now. Currently people adding markerGroups are only expecting to have to handle problems. How do we handle this when tasks are thrown in too?

I appreciate the effort on your part - I just want to make sure I am not bringing this all up too late in the game. We are going to do some problems view work in 3.4 so having an idea of what everyone wants to do will be quite helpful.
