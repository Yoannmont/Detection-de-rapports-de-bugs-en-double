Description Lina Kemmel 2006-03-14 10:16:39 EST 
Suppose that some structural feature element has BiDi name 'ABC' (upper case stands for BiDi)
and BiDi value for property 'EFG'.

Then in Model Editor presentation view the element's name and property value change their place
in following way:  'DEF' 'ABC'.
while the corrrect display should be 'ABC' 'DEF'.

Steps to recreate:

0. On Preferences page corresponding to Workplace settings specify 'UTF-8' for Text file encodeing.
1. Create a new EMF project in the workspace.
2. Import the Model from BiDi XML Schema (фешен.xsd attached)
3. Generate the EMF Model Code and Editor for it.
4. Run the second instance of eclipse as "Eclipse Application" including workplace plugins.
5. On Preferences page corresponding to Workplace settings specify 'UTF-8' for Text file encodeing.
6. In the new eclipse instance create containing project.
7. In new project, create a new instance of EMF model that has been created in 
steps 2-3 (select it from "Example EMF Model Creation Wizards").
8. Choose the root model object (фешен structural feature), fill in the value for object's
property 'Titleабв' (use BiDi characters to fill property values).

Verify that in Model Editor presentation the object's name appears on the right while property name is 

shown on the left( the correct result should be - object name on the left, property value on the right).

he/she find label and vice versa.
This problem appears to be very serious one since it compromise the usebility
drasticaly.
The problem stems from the difference in how BiDi text is shown versus how it 
is stored in buffer (logical display).
The two adjasent BiDi strings are combined in one buffer and shown as one entity.

Suppose that the first string is stored in buffer as "ABC" and shown on display as "CBA"
The second tring is stored in buffer as "DEF" and shown on display as "FED"
When concatenating two strings in buffer as "ABC DEF" one gets dispaly "FED CBA"
while in our case the correct display should be "CBA FED".

To resolve this problem the one have to use some special possibilities provided by ICU engine.

The answer to your question *why* is rather simple.
You rely on the fact that BiDi user reads text from right to left.

1. The BiDi user is supposed to follow the text flow from right to left

unequivocaly deside in which direction test have to be read.

Comment 23 Matitiahu Allouche 2006-04-10 04:55:26 EDT 
After discussing the content of comment #21 with Alexander Shensis, it appears
that I misunderstood the requirement and provided a correct answer to the wrong
question.
Here is how I now perceive the requirement. The GUI displays pairs of
name/value elements, each of which can be either English text or Arabic/Hebrew
text. If the GUI is mirrored, the SWT component used for display will have RTL
orientation, otherwise it will have LTR orientation, and that orientation is
not known to the software function which assembles the elements for display.
If the orientation is LTR, the display must be "name value" from left to right;
if it is RTL, the display must be "value name". The reason is to keep this
display in sync with a header line whose order depends on the GUI being
mirrored or not (or some similar reason).
The challenge is thus to insure proper presentation order when we don't know
the orientation of the target SWT component and each of the name and the value
can be either English or Arabic/Hebrew, which will only be known at run time.
Proposed solution: 
a) If the name is English and the value is Arabic/Hebrew or vice-versa, a
simple concatenation of name + value will give a correct presentation.
b) If both name and value are English, add RLM (Right-to-Left Mark, U+200F)
between name and value.
c) If both name and value are Arabic/Hebrew, add LRM (Left-to-Right Mark,
U+200E) between name and value.
This solution assumes that each of name and value contains only letters of the
same type (either Latin or Arabic/Hebrew) with possible space and punctuation
(neutrals) or digits between such letters (but not at the ends). If this is not
true, we have to add the following.
d1) For each of name and value, if it contains only letters of the same type
but starts with digits and/or neutrals, prepend LRM for an English element and
RLM for an Arabic/Hebrew element.
d2) For each of name and value, if it contains only letters of the same type
but ends with neutrals, append LRM for an English element and RLM for an
Arabic/Hebrew element.
Note: d1 and d2 can be both true.
e) For each of name and value, if it contains letters from both types, some
algorithm is needed to determine if it is primarily an LTR phrase or an RTL
phrase (e.g. according to the first "strong" letter). For an LTR phrase,
prepend LRE and append PDF; for an RTL phrase, prepend RLE and append PDF.
This assumes that the charset is Unicode (because other charsets have no
support for LRE/RLE/PDF) and that the presentation subsystem supports LRE, RLE
and PDF.
