Comment 2 DJ Houghton 2003-10-08 15:26:19 EDT 
Recently we changed the lock file strategy to use the new 1.4 java.nio.FileLock
class. 

I created a stand-alone application on HP using these classes and there were
still problems. Code looked something like this and the message from the
IOException was "Invalid Argument":

// Calculate the location of the lock file
File lockFile = new File("/tmp/foo");
FileOutputStream stream = null;
try {
  stream = new FileOutputStream(lockFile,true);
  FileLock lock = stream.getChannel().tryLock();
} catch (IOException e) {
  e.printStacktrace();
} finally {
  if (stream != null)
    try {
      stream.close();
    } catch (IOException e) {
      e.printStacktrace();
    }
}

Will add a check for a system property. If the property is set, then the lock
file will NOT be checked. Value for the property is not checked. Key is
"org.eclipse.core.runtime.ignoreLockFile".

This can be passed into Eclipse as a VM arg:
  eclipse -vm \myvm\java -vmargs -Dorg.eclipse.core.runtime.ignoreLockFile

Simple program to test file locking. Run with:

java TestLock <file-path>

import java.io.*;
import java.nio.channels.*;
class TestLock {
	static void acquireLock(String path)  throws Exception {
		File lockFile = new File(path);
		FileOutputStream stream = null;
		stream = new FileOutputStream(lockFile,true);
		FileLock lock = stream.getChannel().tryLock();
		stream.close();
	}
	public static void main(String[] args) throws Exception {
		System.out.println("trying to acquire lock once");	       

		acquireLock(args[0]);
		System.out.println("trying to acquire lock twice");	       

		acquireLock(args[0]);
	}	
}
