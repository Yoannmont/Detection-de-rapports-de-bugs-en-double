Description Justin Smith 2004-03-26 02:13:57 EST 
There doesn't seem to be a way to set the Tab title of a view. If you use
WorkbenchPart.setTitle("...."); it doesn't change the title. This used to work
in V2.x.

I need this feature for a RCP application that uses the new view(id,
secondaryid, mode) functionality that has been introduced in V3.0 and need to
set the view tabs to read different titles.
Comment 1 Nick Edgar 2004-03-26 13:39:12 EST 
2.1 and 3.0-M7 and earlier did not let you change the tab text, just the view
titlebar text.  In 3.0-M8, changing the title does not change the tab, but does
show an extra status field if needed. 

However, this is not adequate for most multi-instance view cases where you want
the second view to have a different name, not just extra status.

We will be revisiting this for multi-instance views in M9.

Can you provide more info on exactly what you would show?
Comment 2 Justin Smith 2004-03-28 22:28:08 EST 
I am using a view to house a Browser style user interface (however, not for web
pages but actual Eclipse Forms and UIs). I have implemented a Control-Click
paradigm like is evident in Mozilla/FireFox. Using the new View creation method
in M8 and the visibility statuses I can get a Mozilla browsing effect. But I
want to be able to set the new View name to whatever I would like. I image it
would be something like "Explorer-[name]", for example: "Explorer-Inbox" and if
you control -clicked on a email it would be "Explorer-Email". This is a simple
example I know.
 
I know this is against the general UI style constraints you are trying to
enforce with views in Eclipse/JDT, however it will give heaps of flexibility for
some RCP applications.

Hope this explains more adequately what I want to do.
Comment 3 Justin Smith 2004-03-28 23:00:50 EST 
Also, before it is suggested having this type of control in a Editor part
wouldn't work in my RCP app because I use the Views action control and menu
functionality and think it is idealy suited for this dynamic "browser" like
control. For example if you are looking at a photo it will give photo functions
in the most logical position, and if you change another control type then you
will get its corresponding actions. In a editor part it wouldn't be able to work
that way (the actions would get added to a different part of the window and the
user experience wouldn't be as intuative).

I have had a look at the extra status information that is included with the
title change but the user will need to see it actually on the tab for that to be
useful to them.
Comment 4 Nick Edgar 2004-04-01 17:24:03 EST 
Would you ever need to change the view name after creation?
Comment 5 Justin Smith 2004-04-01 22:00:44 EST 
Yeah, would need to change after being created. Maybe a extra method to maybe
"force" a title change might be appropriate since most would not want to change
the view name after creation. Suggest overload: setTitle(String, boolean force)
rather than using the setTitle(String) method if you are worried about how
people will use it?
See FireFox and it tabbed-browsing support to see how I am trying to use it.
Comment 6 Nick Edgar 2004-04-02 09:38:38 EST 
Yeah, I like Firefox too.  It's nicely designed.

While I believe this bug is worth fixing, note that you could still use editors
-- there's nothing preventing an editor from having its own local toolbar.
Editors have other advantages, e.g. they can take advantage of the editor
navigation history (see IWorkbenchPage.getNavigationHistory()).
Comment 7 Justin Smith 2004-04-02 18:46:14 EST 
Yeah, agreed. But the comments I made about how the usability difference between
a local toolbar compared with the View tool-items is the real difference here.
The views tool items and menu are in the most intuative place rather than
another coolbar. If we could make the local toolbar for a editor display like
the views tool items then I would use Editors. What do you think? It is a shame
we cannot have some way to show the local toolbar like the views toolbar.
Comment 8 Nick Edgar 2004-05-04 22:26:49 EDT 
Please see the proposed API in bug 55656.  If this doesn't work for you, 
please let us know ASAP.  Our API freeze is this Friday.

*** This bug has been marked as a duplicate of 55656 ***
Comment 9 Justin Smith 2004-05-05 10:16:22 EDT 
Just downloaded build 05052004:
Tried:
		setPartName(newTitle);
		setTitle(newTitle);
		setStatusText(newTitle);

None of these changed the Tab's text. How do I do it??
Comment 10 Nick Edgar 2004-05-05 10:19:54 EDT 
Sorry, I wasn't clear.  The implementation is still TBD.  Could you comment on
whether the API looks like it would work for you?
Comment 11 Justin Smith 2004-05-05 11:43:25 EDT 
So which call should I use to change the Tab title (setPartName or setTitle)? If
one of these achieve the tabs title to be changed then it will satisfy the
requirements.
Comment 12 Nick Edgar 2004-05-05 11:53:20 EDT 
setPartName will change the tab text.
Comment 13 Justin Smith 2004-05-05 12:12:49 EDT 
Therefore, if I have three tabs open - all the same original part id but all
with different secondary id's, can I change each tab name independantly or do
they all change when I call setPartName on one of them?
Comment 14 Nick Edgar 2004-05-05 12:20:58 EDT 
It's a property on the part instance.  It does not change the name in the
registry, so only the tab for the specific view would get updated.
