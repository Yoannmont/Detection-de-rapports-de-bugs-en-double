Description Dani Megert 2007-05-23 04:38:23 EDT 
3.3 RC1.

Search view stepping in fast view mode is broken: it activates the editor and minimizes the view.

This is broken since > 3.0.2.

There are two cases:
1. if reuse is off this is completely broken
2. if reuse is on opening the first editor minimizes the view but further
   next/prev reuse the editor (as long as it is the same type) and hence the
   view stays open

I suspect a bug in Platform UI in the interaction of fast views with opening an editor without activation.

Marking as 'major' as with the new MVBs fast views became more prominent.
Comment 1 Martin Aeschlimann 2007-05-30 12:42:10 EDT 
We call

with activate = false.

In there,
  WorkbenchPage.busyOpenEditorBatched(IEditorInput, String, boolean, int, IMemento)
  line: 2688	
calls

and ignores the activate flag.

Moving to platform/ui.
Comment 2 Eric Moffatt 2007-05-30 15:37:18 EDT 
Dani, thanks for the head up...

A little investigation turns up the following stack as the culprit (I think):

	WorkbenchPage.updateActivePart() line: 1159	
	WorkbenchPage.partAdded(WorkbenchPartReference) line: 1569	
	EditorAreaHelper.addEditor(EditorReference, String) line: 210	
	EditorManager.createEditorTab(EditorReference, String) line: 774	
	EditorManager.openEditorFromDescriptor(EditorDescriptor, IEditorInput, IMemento) line: 673	
	EditorManager.openEditor(String, IEditorInput, boolean, IMemento) line: 634	
	WorkbenchPage.busyOpenEditorBatched(IEditorInput, String, boolean, int, IMemento) line: 2704	
	WorkbenchPage.busyOpenEditor(IEditorInput, String, boolean, int, IMemento) line: 2618	
...

Essentially, whenever we create a new Tab (which is why 'reuse' helps) we end up calling 'partAdded' which updates the active part (causing our pain).

Now, on to the fix..I'm not sure that I can come up with a fix that's safe enough for RCx...

Having 'addEditor' call 'partAdded' seems to make sense but why adding a part should -always- update the active part is more questionable. 

While commenting out the 'updateActivePart()' call in 'addPart' does seem to fix -this- problem I don't consider removing the 'update' call to be safe at this time simply because it may affect -any- code path that results in opening an editor or a view, causing possible regressions (most likely would be a new view/editor not becoming active when it should be). Not calling 'partAdded' when adding a part is certain to cause problems.

I'll gladly take suggestions if anyone can point me to an alternative solution...

Comment 12 Eric Moffatt 2007-08-15 13:53:00 EDT 
Got it...the 'Fast Views' are -explicitly- filtered out of the view history (WorkbenchPage#getActiveReference Line: 4165). Removing this filtering code gives us what we expect (opening a new editor still calls 'updateActivePart' but, since the editor wasn't activated, the currently active part -is- the Search view and it remains up).

With the new story I think it's incorrect to treat FV's in this way. If ot was the last active view in reality then it should activate in its due turn. Note that there's an interesting side-effect here...if you open a 'fast' view and then select another view and close it then the fast view becomes active (since it was the previously active view...).

I'll have to kick this around to make sure that there aren'y really wierd side-effects but I'm on the right track...

Comment 22 Eric Moffatt 2007-08-30 14:11:11 EDT 
Daniel, I've taken a look as to why the Search View disappears when the file type changes and 'reuse' is set. Unfortunately I don't think that I can make a specific enough fix without help from you. Here's the scoop:

The basic problem is that 'updateActivePart' has been specifically coded to ignore fast views as candidates to be the active part. This means that anytime 'updateActivePart' is called in the code flow it'll tear down any active fast view. My 'fix' was to wrap the updateActivePart call in 'partAdded' with a magic boolean that allows fast views to be candidates for active parts whenever an editor gets added. Since this is 3.3.x I've made it as surgical as I can so as not to introduce side-effects.

In this case the call to 'updateActivePart' is being called in 'handleDeferredEvents' (the deferring is done in 'closeEditors' which is called from EditorOpener#showInEditor via 'showWithReuse'). The problem I'm having is that I can't add the magic code in 'handleDeferredUpdates' because it's used from all over the place so I'd risk the side-effects I'm trying to avoid.

The question is whether or not you'd be willing to add code to 'openWithReuse' to set the magic variable (through an internal method in WorkbenchPage that I'd add). 

Note that this would likely be -only- for the 3.3.x stream; in 3.4 I'll be re-examining the whole idea that fast views should be removed from the activation list (but that's too big a change for 3.3.1.
