Description Michael Valenta 2003-10-31 10:30:54 EST 
I have changed CVS replace to be performed in the background. During testing, 
I got a deadlock. Auto-build was off at the time (I don't know if this is 
significant). The stacktrace is below but here is a summary of what is 
happening (as far as I can see). I'm CCing Philippe since one of the threads 
involves JDT and he may have insight into what's happening.

- Worker-6 is performing the CVS replace
   - holds a scheduling rule on the project involved
   - is attempting to create a file
   - is blocked on the workspace lock
- Worker-4 is firing POST_CHANGE events
   - holds the workspace lock
   - is notifying JDT of changes
   - is attempting to do a project#touch

The problem is that Worker-4 is doing an invalid operation and should be 
prevented from doing so (especially since the thread holds the workspace 
lock). I was not able to find where the workspace lock was obtained in Worker-
4 from the stacktrace but I used the debugger to look at the lock and this 
indicated that the lock was held by Worker-4.

Comment 1 Philipe Mulet 2003-10-31 12:20:17 EST 
CVS is causing a .classpath change which gets reconciled as a consequence of 
the notification we receive from platform. I don't think we can avoid it from 
the JDT standpoint. CVS would have to avoid listeners to react in any way 
(batch operation ?)
Comment 2 Michael Valenta 2003-10-31 13:58:43 EST 
JDT may be reacting to a .project change but it is my understanding that you 
cannot do a touch (or any other workspace modifying operation) in a 
POST_CHANGE event because the tree is locked for modification. In this case, 
we could probably fix the problem by having CVS use the new core API which 
prevents deltas from being fired until the end of the operation. However, 
there is always the chance that operations from other sources could cause this 
(although I admit it's unlikely). Would it be possible to restict you handling 
of .classpath changes to the PRE_BUILD delta phase (when the tree is open for 
modification).
Comment 3 Philipe Mulet 2003-10-31 15:24:18 EST 
No, we cannot do so, since we are likely in a different thread than the 
POST_CHANGE (this was actually the reason why we had troubles when switching to 
background autobuild).

However, you're right, we shouldn't attempt a touch if the tree is locked.
Adopting.
Comment 4 Philipe Mulet 2003-10-31 15:26:16 EST 
Just to confirm, Michael, are you using the nightly build ? The #touch() 
addition wasn't released for integration yet...
Comment 5 Philipe Mulet 2003-10-31 15:30:31 EST 
Actually, this isn't us. This is the PDE workaround for bug 45707.
We added a #touch in our latest, but you presumably aren't using it yet, and 
our code is protected so that it only touches if the resource tree isn't locked.

Moving to PDE UI.
Comment 6 Wassim Melhem 2003-10-31 15:48:54 EST 
Just removed the PDE workaround.
