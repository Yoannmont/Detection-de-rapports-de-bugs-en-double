OK, I can reproduce this now.  It also happens for me under xfwm4 on Debian
unstable.  I am not sure yet if this the same problem as bug 100328, but I do
think that the xfwm4 error is the same as what Jim saw under kwin 3.4.

The symptom here is that the Shell is being resized to a size of 200x200.  GTK+
uses a size of 200x200 as a default size for windows that have not requested any
other size.  Unfortunately, Eclipse _does_ resize the window, what seems to be
happening is that another resize event comes in while GTK+ is in an intermediate
state.

Comment 17 Billy Biggs 2005-11-12 16:04:12 EST 
Fixed > 20051112

It's pretty obscure but I don't think it's a WM bug, although it is clearly WM
specific.  Some X calls causes the WM to issue ConfigureNotify requests, which
translate into resize events.  Before the window has been mapped, the default
size of a window in GTK+ is 200x200.  If you call gdk_window_lower() before the
window is mapped, and if the timing is right, it seems that sometimes you can
get a 200x200 resize event even though you may have already given a valid size
to GTK+.

The fix is to avoid calling gdk_window_lower() since it's not useful in this
case anyway, and also to avoid gtk_window_resize() if the shell isn't resizable
(it isn't necessary and simply confuses things more).  I think this fix is
solid, but please reopen this bug if you see more cases in newer builds.
