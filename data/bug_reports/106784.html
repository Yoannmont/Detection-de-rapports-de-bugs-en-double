Description Kim Moir 2005-08-11 15:30:59 EDT 
We have multiple bug requests from people to build the Eclipse SDK on os A ws B
and arch C.  We don't have the resources in terms of people, time, hardware or
software to build all these ports.  In many cases, these ports would appeal to a
smaller community than the more popular platform downloads.  Since we can't test
them,  it doesn't make sense for us to make them available as part of the build
since we can't fix any platform specific bugs.  However, it would be nice to
have a page where people could put their custom builds to share with the
community with the caveat that they are not officially supported and provide the
contact information for the authors.

I think mozilla.org does something similar with firefox.

See
http://www.mozilla.org/products/firefox/releases/1.0.6.html#othersystems

Could there be a contributed ports page for platform builds that is not
mirrored? The releng team would take care of updating this page for the
platform. We could just create a page off the our platform-releng page but I
thought I would open a bug.  This page could be be useful across projects.

Comment 7 Eclipse Webmaster 2005-09-06 14:13:34 EDT 
Here is the proposed propcess/solution for this:

2. User uses an anonymous FTP to upload his file.  This file goes to the
"staging area".  As suggested by Gunnar, once a file is uploaded, it is no
longer visible by the FTP user to avoid eclipse.org becoming a free download
site for anything.

3. User opens a bug in a component called "contributions" or something - this
would have to be consistent from one project to the next.  Platform/Contributions

4. In the bug report, user states the filename and any other information
requested from your instructions webpage

5. Upon receiving the bug, you test/move/delete the file out of the staging area
and put it where it belongs (downloads area, trash, etc.)

6. You provide links to the contributed files in your download scripts.

7. Contributed files older than 14 days are deleted automatically.

1. Your project webpages contains instructions on how to submit a community
contribution using this process.


How does this sound?

D.

Comment 12 Kim Moir 2005-09-12 10:54:56 EDT 
That is certainly an option. I just initially suggested storing them on
eclipse.org because this is how other open source projects have approached
providing contributed builds.  The external contributors that I have worked with
in the past (HP, Intel) have asked that their builds be hosted on eclipse.org

barrier for some contributors, I don't know.
Comment 13 Kim Moir 2005-09-12 22:01:18 EDT 
Talked to Janet today.  We decided that there will be two groups that will be
contributing ports.

1. Committers that craft builds on a platform that has been approved by the PMC
to supplement the builds provided by the project's build team.  These builds
will be hosted on eclipse.org.  Since they have agreed to committer documents
they will be adhering to the due diligence guidelines. An example today would be
HP and Intel who contribute platform builds.

2. Non-committers who have crafted a build on platform that is not approved by
the  PMC.  These builds will be listed on "contributed ports" web page on
eclipse.org.  On this page, we will indicate clearly that these builds were
built by contributors and no warranty is made for their content, for platform
specific bug support, you should contact the author etc.  The actual content
will reside on the contributors web server.  

Comments?
