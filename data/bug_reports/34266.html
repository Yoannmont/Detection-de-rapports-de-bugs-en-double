Description Jin-serk Baik 2003-03-10 05:41:28 EST 
Eclipse version-up is so frequently that we should reinstall the eclipse and 
all of the pre-installed plug-ins to get more stable functionality. The 
eclipse itself has very easy installation process -- just extracting it to a 
specific directory. If we made a our own workbench dir, we just point the 
directory on the eclipse execution command with -date option. However, if we 
have many external plug-ins as each favorites, then we should reinstall all of 
them one by one to the eclipse/plugins directory.

If Eclipse has any external plug-in dir option in it's preferences menu (but I 
can't find it), as I add my ext_plugin dir to the option so that I can install 
any plug-in to that dir except the default plugins included in the eclipse 
distribution.

I think that there is no option about the user defined plug-in dir in
eclipse, and the 'plugins' dir included in the eclipse package is the only
plug-in dir which eclipse can understand. But if we have any option to add
plug-in dir, or command line-option like -data cmd option, we can divide the
workbench and ext_plugin dirs from the eclipse distribution file so that the
only thing to update the eclipse is just extracting the distribution file, I
think.

I wonder whether the dev team can add this option to preferences or cmd option 
or not. I suggest this for more comfortable eclipse upgrade.

Thank you!

Leo
Comment 1 John Arthorne 2003-03-10 10:19:30 EST 
There is already a feature for installing a set of plugins separate from the
base eclipse install.  The facility is called "Product extensions".  Look in the
Help system under:

Platform Plugin Developer's Guide 
  Programmer's Guide
    Packaging and delivering eclipse-based products
      Product extensions
Comment 2 Jin-serk Baik 2003-03-11 00:53:42 EST 
But, I hope that we can use the function of extension with user-friendly 
option. The description of the help system says that we should 
make .eclipseextension file instead of .eclipseproduct. 

BTW, I have a question. If I make a dir to use my own plugin installation dir, 
and make a copy of .eclipseproduct file to that dir renamed 
as .eclipseextension, then how I can make eclipse understand that dir is the 
extension? I read the relative help in eclipse, but I didn't catch how I can.
Comment 3 Jin-serk Baik 2003-03-11 01:57:54 EST 
Ah, I can make the extension dir and eclipse understand it. However, there are 
two manual process I think unconfortable.

1. make .eclipseextension file as a copy of .eclipseproduct
2. make link dir to the eclipse product dir and make each of plug-in link 
within it

The first one is no more problem because we make it only once when we make 
extension dir. However, the second one is the problem because it requires 
additional process when we install the external plug-ins.

I want make that eclipse has an option to point the extension dir in its 
preferences so that this option setup is saved in workbench dir. Then we can 
call the workbench dir with -data option when we execute eclipse.exe. eclipse 
can understand the extension dir as a external plugin dir so that there is no 
need to make a link dir in the product dir path. However, the present 
structure make it unconfortable to use of general purpose - when we install an 
external plugin to the extension dir, we should also add the link file to the 
product_path/link dir. To make more comfortable, we should use an external 
script which oleg shteynbuk serves in his homepage:
http://home.nyc.rr.com/olegs/#Eclipse

That's why I suggest this feature.

Comment 6 Sean McRae 2003-11-12 11:57:18 EST 
I too would very much like to see an additional preference for adding 
additional plugin directories. While not to trivialize the change it seems 
rather straightforward to add a field for "additional plugin path" that would 
be searched for plugin files after the .plugin directory. This way users could 
have directories outside of the main Eclipse install that don't get manhandled 
during a dot release upgrade and prevents the need for having to copy plugins 
between installations.

Comment 9 Sean McRae 2003-11-13 11:44:08 EST 
My understanding from the documentation is that the steps you outline below are 
for product entensions where the writer has included a .eclipseextension marker 
file. These extensions typically include a list of features AND plugins and are 
inherently different from the third-party plugins we all use today.

That said, I think what we are requesting is a way to install plugins in an 
additional location to the default - not a way to handle product extensions.
Comment 10 Dorian Birsan 2003-11-13 14:21:12 EST 
When a feature is installed via the Update Manager, on the last page of the 
install wizard you can pick a new location, and everything is done 
automatically.
The information about sites, features, etc. is kept in the .config directory. 
By default, it is located inside the product directory (<eclipse_install>), 
but if this is not writeable, it is somewhere in the user's home directory.
When the work on user setting is finished, we will use some of those ideas to 
also locate .config according to what the recommendation is for application 
settings.

In the meanwhile, you can start eclipse with -configuration 
file:d:/path/to/config/ so that .config is saved outside the eclipse directory.
This way, you now have both the workspace, and the config data outside 
eclipse, and can reuse the stuff with multiple installations.
Comment 11 John Arthorne 2003-11-13 14:42:53 EST 
Dorian, here's the problem we're trying to solve:

- Simple Joe user downloads Eclipse from eclipse.org
- Joe user then grabs some random plugin from sourceforge or whatever
- They want to use that plugin, but they don't want to put it in the "plugins"
directory because it will be lost next time they upgrade
- They don't know what a feature is, or an update site, or a configuration, or
any of that stuff. They just want some simple steps to follow to get that plugin
working.

Product extensions "almost" work perfectly, but they require the user to add the
.eclipseextension file to their external plugins directory, which can't be done
from within the UI.
Comment 12 Dorian Birsan 2003-11-13 15:06:19 EST 
John, thanks for clarifying the requirement.

Then, the behavior of "unmanaged plugins" (i.e. not under update manager's 
control) must be specified by the Platform. One solution is to allow a 
-plugins <plugins_dir> to support extra plugins.

To me, the current product extension mechanism is for something else, and if 
anybody goes to the trouble of setting it up (ie. creating the features and 
plugins subfolders), then it is no big deal to also put the marker 
file .eclispseextension. 

Ideally, plugin providers should publish their plugins using features, so one 
can get updates and validate the installs via the update manager, with little 
manual intervention.
And, for simple plugins, I'd advocate the -plugins flag at startup.
