Description Dave Orme 2007-04-03 17:26:55 EDT 
It would be nice if Mylar could infer interest in Java structures based upon semantic information that the JDT maintains about those Java structures.

For example, if I open class A which extends class B and implements interfaces i1, i2, and i3, I may be interested in class B and in interfaces i1, i2, and i3.

Similarly, if I am working on method M, I am probably also interested in methods that method M calls and also in methods that call method M.

I would imagine that Mylar would initially add a small amount of interest to objects that have a semantic connection to whatever I open, at the time I explicitly open that thing.  After that initial interest is added, it would decay normally with usage, so only the things that I actually use will remain in my task context.

This would make it easier to build up new task context much more rapidly, especially in an unfamiliar large code base.

The ideal would be for the interest to be tuned so that my interest in the things I never touch decays to 0 at about the time that my task context is getting well populated, so that the garbage disappears automatically at about the right time.
Comment 1 Mik Kersten 2007-04-04 13:12:44 EDT 
This is exactly what the Active Search facility does via its "predicted interest propagation".  The problem is that it made too many elements show in the Package Explorer, so we created a view called Active Search.  That also ended up showing too much stuff so just before 1.0 we moved it into the Mylar Sandbox.  Still, I couldn't agree with your use case more.  The question is how we best present this information.  My current take on it is that we try to use the Quick Context View to show elements structurally related to the currently-selected element, and only show those with a sufficiently high degree of interest.  Your idea of pre-populating very small contexts is also a good one, but I'm not sure if it's worth the special case if the relevant structural information is just one click away?

We're on the same page in terms of mechanisms, but discussing different UIs.  The mechanism exists already (Mylar's Active Search) so it's good for us to keep discussing the UI.  One of the main problems with showing structurally related elements predicted to be interesting in the Package Explorer is that the user has no idea why they showed up there, and that way too many can show up on a large system.  That's why the Active Search view added nodes to show relationships, and also ranked predicted elements by interest: http://www.eclipse.org/mylar/doc/images/0.3/active-search.gif  

So my question is, would your use case be satisfied if instead of seeing the elements go into the Package Explorer, you could pop up a few that would show you structurally related elements, ranked by interest.  It would be similar to the screenshot linked there in that it would show nodes for relationships.  The idea with this view is that it would show you next places that you might go from the current place (especially important with a fresh context), whereas the Package Explorer is only good at showing you the containment hierarchy.
Comment 4 Dave Orme 2007-04-05 23:22:43 EDT 
UI thoughts:

The argument against putting semantic hits in the package explorer is that there is no explicit user interface gesture that put them there, so it can be puzzling and probably disorienting to see them show up.  This makes sense to me.

The ideas you have proposed all relate to providing a separate view (quick or docked) that basically provides a "semantic interest search", kind of like integrating Quick Type Hierarchy and the Call Hierarchy views into a single thing.

There's probably a good idea there worth exploring.

But I'm not convinced that the package explorer idea is not worth continuing to explore too.

I'm wondering if we could address the usability issue by making an explicit user interface gesture for adding elements that are semantically related to the current selection.  Here are some thoughts along those lines:

- What about a pushbutton on the view toolbar.  If you want to see stuff semantically related to the current thing, you bonk the button.  By default, it would traverse one level into the hierarchy and/or call graph.  If you want yet more information, you bonk the button again.  Lather, rinse, repeat.  Stuff that is added this way is given a very low interest level, so that it decays and goes away quickly.  If the user opens something that was added to the package explorer this way, its interest level becomes a bit higher than if she opened it directly, because of the semantic link.  Or something like that.

- Or what about having a pop-up slider like the Windows volume control activated by the local tool bar button on Package Explorer.  Turning the volume up adds semantically-related stuff to package explorer.  Turning the volume down removes semantically-related stuff.  A preference could make the volume gradually decay after turning it up so that eventually only the stuff you actually are using remains.  The speed of the decay could be inversely proportional to the size of the current task context.  The disadvantage of this approach is that I think a volume control connotes a "global" setting, so you might wind up getting a lot of stuff you don't care much about.  The first approach has the advantage of being local to whatever you're focused on at this given moment, so it's much more likely to add stuff you actually care about.

- Maybe putting either of these options on the view toolbar is giving it too much visibility in the interface?  Another option would be to use the view menu bar.  This has the advantage of scaling much more nicely.  We could add multiple menu choices but it probably isn't as good idea to add multiple buttons.  I would imagine multiple menu choices allowing you to explicitly pop up a volume control for either the current selection or for all elements that are explicitly in the current context.  Or maybe a third option: pop up a volume control just for adding semantic search information based upon landmarks only.
Comment 5 Dave Orme 2007-04-05 23:28:16 EDT 
One last thought:  Another way to make semantic hits explicit in Package Explorer would be to decorate the icons of things that were added that way in some manner.

I don't know how decoration works, but if you can alpha-blend your decoration on top of the icon, you could actually dim the icons of semantic hits by putting a black square with 50% transparency on top of the icon.
