Comment 1 Darin Fisher (IBM) 2004-06-25 14:21:12 EDT 
I took a look at the libswt-mozilla-gtk-3061.so that is included in
swt-3.0RC3-linux-gtk.zip, and i found that the library is depending on a bunch
of non-frozen mozilla symbols, and some of these symbols no longer exist in
Mozilla 1.7 (e.g., nsCString::nsCString()).

It appears to me that libswt-mozilla-gtk-3061.so was not linked against the
Mozilla static xpcomglue library (-lxpcomglue) and is not using nsEmbedCString
(and linking against libembedstring.a).  That is a requirement since this
library appears to be using functions like do_QueryInterface and
do_CreateInstance, which are defined in the glue library.  Moreover, this
library appears to be using Mozilla's private nsACString implementation (namely,
nsCString).  It should be using nsEmbedCString instead.  Note, the library only
appears to work in Mozilla 1.4 because libxpcom.so statically links to
xpcomglue_s (and includes the nsCString implementation), which provides these
same symbols.

Two solutions: link libswt-mozilla-gtk-3061.so to the libxpcomglue.a and
libembedstring.a that are provided in the Mozilla 1.4 Gecko SDK, or do not use
any of the code provided by these libraries.

Or, maybe provide a separate library for Mozilla 1.7 and later releases.

More information is available here:
http://www.mozilla.org/projects/xpcom/book/cxc/

I admit the Mozilla documentation is fairly bad.  We are working on improving
the situation ;-)

Please let me know if you have questions about how to properly embed Gecko.
Comment 2 Darin Fisher (IBM) 2004-06-25 14:29:05 EDT 
If you are interested in providing a version of this library that _only_ works
in Mozilla 1.7 and later (not with older versions of Mozilla), then you can use
the new C-style mozilla string API.  It is designed to eliminate many of the
headaches associated with working with APIs that use nsAString and nsACString.

See http://lxr.mozilla.org/mozilla/source/xpcom/string/public/nsStringAPI.h
for information on the API.  The advantage of this API is that it can be used
without requiring that applications or components statically link with an
additional library (such as libembedstring.a).  While you can still use
nsEmbedCString and call methods directly on nsACString, the result is that your
application or library will only have simple extern "C" link dependencies on
libxpcom.so.  And, these APIs are frozen, so they will always be available in
Mozilla builds starting from Mozilla 1.7.

Even if you use the APIs provided by nsStringAPI.h you will of course still need
to statically link against libxpcomglue.a, but that can only be avoided if you
do not use nsCOMPtr and the nsCOMPtr helpers like do_QueryInterface,
do_CreateInstance, etc.

Comment 7 Darin Fisher (IBM) 2004-06-25 17:12:41 EDT 
Build changes include:

1) Add "-I$(MOZILLA_HOME)/include/embedstring" to MOZILLACFLAGS

2) Change MOZILLALIBS to:
   "-L$(MOZILLA_HOME)/lib -lembed_base_s -lxpcomglue_s -lembedstring"

This change makes libswt-mozilla depend only on frozen mozilla symbols.

Note: however, that Browser.java appears to make use of several non-frozen
interfaces, and some of those may have changed since Mozilla 1.4/1.6.

My advice when using non-frozen symbols is to use QueryInterface to protect your
code against interface changes.  Mozilla policy states that any change to a
non-frozen (private) interface should be accompanied by a change to the
interface's UUID.  So, if you hard code UUID values into your code and call
QueryInterface you should be able to at least protect your code against crashes
resulting from interface changes.

I plan to make freezing the interfaces that SWT depends on a high priority bug,
but I cannot promise that there will be no changes to those interfaces before
they are accepted as frozen.

Comment 9 Darin Fisher (IBM) 2004-06-28 19:33:22 EDT 
One correction to comment #7:

> 2) Change MOZILLALIBS to:
>   "-L$(MOZILLA_HOME)/lib -lembed_base_s -lxpcomglue_s -lembedstring"

The link line should really be:

    "-L$(MOZILLA_HOME)/lib -lembed_base_s -lxpcomglue_s -lembedstring -lxpcom"

Otherwise, the Browser component would not load inside Eclipse.  Listing xpcom
last on the link line ensures that the resulting library only links to
libxpcom.so for the required frozen symbols.

Comment 12 Darin Fisher (IBM) 2004-07-20 17:06:45 EDT 
The problem with the 1.5 build was
http://bugzilla.mozilla.org/show_bug.cgi?id=228210 (I think).  The patch was
never applied to the Mozilla 1.5 branch (since there was no 1.5.1 release).

However, that bug can be avoided by not linking to the standalone xpcomglue.  If
you link to the version of the xpcomglue that I have recommended, then you will
be immune to the problems related to NS_GetFrozenFunctions.

Notice that in comment #9, I corrected myself and showed that you still need to
link to libxpcom.so.  That is because I was linking to xpcomglue_s instead of
the full xpcomglue library, and as a result the frozen functions used in your
code are linked directly to libxpcom.so (without going through the
NS_GetFrozenFunctions function table indirection).

I suspect that we will be encouraging other embedders to use xpcomglue_s with
xpcom instead of using xpcomglue all by itself, but that remains to be sorted
out.  There may still be cases where linking to the full xpcomglue makes sense.

The string changes in Mozilla 1.7 represent a turning point for Mozilla.  With
Mozilla 1.7 it is now possible to use Mozilla strings in external code without
linking to an implementation of the string classes.  However, Mozilla 1.7 still
supports the old string binary interface.  So, building a version of this
library against Mozilla 1.4 will continue to be supported for a long time to
come (likely forever).

However, it seems that Browser.java uses a good number of non-frozen interfaces,
and that may cause you to have no choice but to provide different versions of
your code as Mozilla evolves.

Firefox is most likely never going to be embeddable by Eclipse.  The problem is
that Firefox is a static executable.  Nearly all of Gecko is embedded inside the
static Firefox executable.  This is why Firefox is so much faster than Mozilla
Seamonkey.

Long term we want to build Firefox such that it includes Gecko as a library that
can be embedded by Eclipse and other projects, but it is not clear how that will
be made possible given the significant performance overhead of PIC on Linux ELF/x86.

Short term, Mozilla Seamonkey will continue to be the release vehicle for Gecko.

I've sent mail to drivers@mozilla.org asking for information about when a GTK2
version of Seamonkey might happen.  Don't hold your breath... I've repeatedly
asked about it since the beginning of the year! :-/
