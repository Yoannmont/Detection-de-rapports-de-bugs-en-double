Description Warren Paul 2007-08-29 16:31:46 EDT 
Build ID: I20070625-1500

Steps To Reproduce:
1. Turn off build automatically
2. Build a project with errors or warnings
3. Double click on a problem marker
4. Notice that Build Project is disabled.
5. You can click back and forth between the problems view and the editor and it's still disabled.




More information:
You have to click on the navigator view or similar to get the action enabled again.

The problem is that the selection is calculated incorrectly.  When you click on the editor after being on the problems view, the selection is a marker.  When you click back on a marker in the problems view, the selection is a text selection.  Set a breakpoint in org.eclipse.ui.internal.ide.actions.BuildUtilities.findSelectedProjects(IWorkbenchWindow) to see what I mean.  It would appear that the wrong window is being found - see org.eclipse.ui.actions.BuildAction.isEnabled() for how the active window is determined.

It certainly doesn't seem right.  You could fix this particular issue without changing that logic by having Marker adapter to IResource, but that doesn't seem like the correct approach.

I set the severity as major since this is such a common development task - going between the problems view and the editor containing the errors, and building.

Comment 7 Heiko Böttger 2008-11-21 05:15:25 EST 
This bug still exists in 3.4.1 M20080911-1700, I debugged a little bit and found out that the Window Selection Service is registered in the same list where all PartListener2 listeners are registered. There are two issues with this.

1. The Window Selection Service is a PartListener2 and is informed after all PartListeners
2. The Window Selection Service should be one of the first listeners informed on my eclipse is see that is is the second one after the navigation history. 

I propose to extend the constructor to support internal partlistener2 that are informed first or better to put both listener types into one list and ensure that the navigation history and selection service added first. The order of these special listeners should be documented.
Comment 8 Heiko Böttger 2008-11-25 04:26:46 EST 
This is a reflection hack for Eclipse 3.4.1 M20080911-1700. It adds a part listener to the first list ("partListeners") and then make it to the first listener using reflection.

The runnable could be started via the startUp-ExtensionPoint and proves that this the problem can be fixed with notification order. 

Note that if the selection of a part is changed in an IPartListener or IPartListener2, the change have to be reported to all actions otherwise this wouldn't work. 

May be it's a bug to query the selection in the activation-Event. If this is the case the api should throw an assertion.

---

package fix;

import java.lang.reflect.Field;

import org.eclipse.core.commands.common.EventManager;
import org.eclipse.core.runtime.ListenerList;
import org.eclipse.ui.*;
import org.eclipse.ui.internal.*;

import ch.actifsource.core.util.collection.*;

public class ApplyFix implements Runnable {
  private IMap<IWorkbenchWindow, IPartListener> fListeners = new HashMap<IWorkbenchWindow, IPartListener>();

  public void run() {
    PlatformUI.getWorkbench().addWindowListener(new IWindowListener() {
      public void windowActivated(IWorkbenchWindow window) {
      }

      public void windowClosed(IWorkbenchWindow window) {
        window.getPartService().removePartListener(fListeners.get(window));
      }

      public void windowDeactivated(IWorkbenchWindow window) {

      }

      public void windowOpened(IWorkbenchWindow window) {
        addAndRememberPartListener(window);
      }

     
    });
    for (IWorkbenchWindow window : PlatformUI.getWorkbench().getWorkbenchWindows()) {
      addAndRememberPartListener(window);
    }
  }

  private void addAndRememberPartListener(IWorkbenchWindow window) {
    try {
      fListeners.put(window, addPartListener(window));
    } catch (SecurityException e) {
      // TODO Auto-generated catch block
      e.printStackTrace();
    } catch (IllegalArgumentException e) {
      // TODO Auto-generated catch block
      e.printStackTrace();
    } catch (NoSuchFieldException e) {
      // TODO Auto-generated catch block
      e.printStackTrace();
    } catch (IllegalAccessException e) {
      // TODO Auto-generated catch block
      e.printStackTrace();
    }
  }
  
  private IPartListener addPartListener(IWorkbenchWindow window) throws SecurityException, NoSuchFieldException, IllegalArgumentException,
      IllegalAccessException {
    IPartService partService = window.getPartService();

    IPartListener listener = new IPartListener() {
      public void partActivated(IWorkbenchPart part) {
        ((AbstractSelectionService) part.getSite().getWorkbenchWindow().getSelectionService()).setActivePart(part);
      }

      public void partBroughtToTop(IWorkbenchPart part) {

      }

      public void partClosed(IWorkbenchPart part) {

      }

      public void partDeactivated(IWorkbenchPart part) {

      }

      public void partOpened(IWorkbenchPart part) {

      }
    };
    partService.addPartListener(listener);
    WWinPartService wwinPartService = (WWinPartService) partService;
    Field partServiceField = org.eclipse.ui.internal.WWinPartService.class.getDeclaredField("partService");
    partServiceField.setAccessible(true);
    PartService internalPartService = (PartService) partServiceField.get(wwinPartService);
    Field partListenerListField = PartService.class.getDeclaredField("listeners");
    partListenerListField.setAccessible(true);
    EventManager partListenerList = (EventManager) partListenerListField.get(internalPartService);
    Field listenerListField = EventManager.class.getDeclaredField("listenerList");
    listenerListField.setAccessible(true);
    ListenerList listenerList = (ListenerList) listenerListField.get(partListenerList);
    Field listenersField = ListenerList.class.getDeclaredField("listeners");
    listenersField.setAccessible(true);
    Object[] events = (Object[]) listenersField.get(listenerList);
    if (events[events.length - 1] != listener) throw new RuntimeException();
    System.arraycopy(events, 0, events, 1, events.length - 1);
    events[0] = listener;
    return listener;
  }
}
Comment 9 Dani Megert 2009-09-30 08:42:37 EDT 
This is a chicken and egg problem: in WWinPartService.updateActivePart() we first update the partService and then the selectionService. Both fire events to their listeners hence a part listener that accesses the selection service will get wrong results. Same is of course true if we invert the two calls, however since the selection changed notification already provides the new part it is very unlikely that someone asks for the active part via part service. I think in general we would get better results if we invert those two calls. This would not only fix this bug here but might also have positive effect on other code.

Paul and Boris, what do you think?
