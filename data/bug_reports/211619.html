Comment 1 Daniel Ford 2007-12-03 11:23:04 EST 
There is the potential that the BIRT reports views are still holding references to adapters that then reference a Scenario instance even after the Scenario has been "stopped".  This would hold the memory for the canonical graph in the heap and prevent garbage collection.

Comment 3 Daniel Ford 2007-12-06 10:41:37 EST 
The eclipse debugger "Variables" view now shows references to class instances, maybe we can use that to find the link.  The main reference to a canonical graph should be a Sceanario instance.  Other things like the MapView also get a reference to the graph, but they should release it when the the Simulation "stops".
Comment 4 John Thomas 2007-12-09 13:14:33 EST 
Current Status of this problem.

Running jmap ( a java memory tool in Java 6) also showed that the heap has quite a bit of the graph instances still referenced even when there are no simulations running.  

But strangely,  I ran the same test that I ran earlier that showed memory leaks as each simulation ended and a new simulation started.  But now it does not show any overall memory leak.  making 20 runs of USA scenario using the Batch Simulation tool showed approximately the same memory utilization for the 20th as the 1st.
I tried this both with Java 6 and Java 5 and the results were the same.

If I run the Map view,  it does show a severe memory leak but it appears to be restricted to the Map View.  Stopping the Map view stops the memory leak but does not recover the previously leaked memory.  

The above status is inconsistent.   Obviously Map View should be looked at to see what is causing it to hang onto memory but this is not a show stopper for the batch Simulation support.   But why the more severe memory leak without Map View active just went away concerns me. 

John
Comment 5 John Thomas 2007-12-10 13:27:30 EST 
I have resolved the inconsistent results between my first test/report of a memory leak and the 2nd report that the memory leak does not occur.
The first time, to ensure that the Batch simulation code was not at fault, I ran the multiple simulations by manually starting and stopping them.   
In the 2nd case, I used the batch simulation code.  In this case it did not have a memory leak because the batch simulation is reusing the scenario object which allows it to reuse the graph instances and therefore no memory leak occurs.

This is good news for the batch simulation support since it means that it can run simulations over and over again without running out of memory.   

Someone should however look at the MapView app to see why it is not freeing the memory that it uses.   

I will leave this bugzilla open for the mapview case since it contains some potentially useful info.
Comment 6 Daniel Ford 2007-12-11 12:52:49 EST 
Looking through the code, there is the possibility that a reference (or more than one) to a stopped Simulation is kept by various views.  The MapView seemed to be a place to look, but I wasn't able to verify this definitively.  I looked at how the SimulationManager notified it's listeners that the Simulation had stopped and tried to make sure that they were properly removing their references to the stopped Simulation.  Things mostly looked to be in order, but probably a better tool would help.  

Dropping it's priority as John indicated that he had a workaround.
