Description Christal Pomeroy 2002-02-21 21:21:53 EST 
Eclipse does not support building of an alphabetical on-line help index. This 
will be very limiting to the end-user trying to locate relevant help topics.

Requirements of an index:
- offers a simple mechanism by which to access topics based on important 
keywords in the topic, where the keywords are listed as alphabetical entries in 
the index. User picks the index item (a keyword) to open one topic, or access a 
list of relevant topics.
- multiple keywords can map to one topic
- multiple topics can use the same keyword
- index entries can be nested to at least 3 levels
- index entries from each contributing 'book' can be merged into one 
consolidated index for the whole system
- [optional] multiple indexes can be defined and accessed for a system
- [optional] ideally, the user would be able to start typing in field at the 
topic of the index, and have the index scroll automatically (after each letter 
entered or after a short pause) to the location in the index corresponding to 
the letters typed so far.

A short list of arguments for having this mechanism:
- Without an index, the user must find topics by knowing the exact spelling or 
syntax of a word or phrase to seach on, or must know which book and where in 
the table of contents hierarchy to begin looking. With an index, the user can 
begin typing or scroll through a predefined, alphabetical list of terms to 
locate a term just by knowing the first few characters.
- Without an index, the user is left only with a string search mechanism, which 
could potentially return a huge list of topics that reflects every occurrance 
of the string in the help system. With an index, the help system authors can 
intelligently choose the topics that provide the best information for a 
particular term, thus providing the most efficient end-user experience.
- Without an index, the user is unable to see related or extension topics to 
the core term. With an index, related or extension topics can be listed through 
nested entries.

I was responding to this note and accidentally "submitted" the changes, but 
here it is again:

This feature cannot be contained in 2.0 and will be deferred to a further 
release.

A possible workaround is the following:

- define a TOC that contains <topic> nodes for all your index keywords
- the structure could be a simple list of <topic>'s or you could have 
  nested <topic>'s when index keywords can nest (you mentioned that up to
  3 levels may be supported).
- the href attribute of a <topic> will be the topic that the keyword matches,
  but when a keyword matches more than a topic you have two options:
      - either create another html document that lists the matched topics and
        have the href point to it, or
      - create nested <topic> nodes for each matching topic

- if you need to support merging of indexes, then create anchor nodes for each 
  letter, or combinations of letter (like "ab", "di", etc.), and other indexes 
  can be linked to those anchors.
- for the above to work nicely, you may need to create separate, non-primary 
  TOC's for each letter and have them linked to the master TOC.
  The master TOC could be just the list of anchors, or could be anchors plus
  some main topics

The files would look something along these lines:

<toc label="Index">
    <anchor id="a"/>
    <anchor id="b"/>
    <topic label="build" href="../com.company.plugin/buildProject.html"/>
    <anchor id="c"/>
    .......
</toc>

Comment 24 Robert Lauriston 2005-04-12 11:59:09 EDT 
Indexes and TOCs are radically different as regards merging.

Since each TOC is a tree, you can easily merge multiple TOCs as branches of a
larger tree. The way Eclipse handles this is already problematic, since when you
invoke help the TOC focus is often not on the current plug-in's branch.

Since indexes are alpha-sorted lists, it's not so simple. A 300-entry index for
a small application would likely become useless if it were merged with a
5000-entry index for a larger application. Different index authors may use the
same word to mean different things, so merging will sometimes group unrelated
items together. "See" and "See also" entries would be particularly problematic
since they could end up pointing to unrelated items.

From a docs point of view, it would be useful to be able to merge a defined set
of indexes written with that in mind. In other situations, merging would be
highly problematic.

Comment 27 Lee Anne Kowalski 2005-04-12 13:26:43 EDT 
No, I think it would be conceptually limiting to think of the master index as 
very different from ToC or search, as a document in itself.

From an information architecture point of view, the overall reader (user) task 
is "locating information". A master keyword index then is another mechanism for 
locating a piece of content in a whole body of informational topics--just like 
the ToC is a hierarchical method for locating a piece of content and the search 
mechanism is a "term in topics" method for locating a piece of content. A 
keyword index then is parallel with both ToC and search in terms of the reader 
tasks. A keyword index then is a "keywords in topics" method for locating a 
piece of content, but where you can see the hierarchical relationship between 
different keywords; e.g.:
databases,
   creating
   dropping
     locally
     networked
tables
   creating
   dropping
...

A keyword index is a tool that marries the ToC hierarchical method and the 
search term method of locating a piece of content.

It is true that given those facets of a good master keyword index (hierachy and 
keywords), the content developer needs to set the relationships between the 
keywords (index level 1, index level 2, etc) and needs to be able to include 
keywords that do not themselves exist in the content (like synonyms). However, 
from a componentized, doc plug-in conception of information delivery, those 
relationships and synonyms would best be deployed along with the topics they 
are connected to--as metadata in the topics or in some type of map in the doc 
plug-in (similar to toc.xml 'map' in the doc plug-in). And therefore, to 
support that pluggability behavior, merging is important, even if it would be 
challenging to achieve.

Comment 65 Lee Anne Kowalski 2005-11-16 10:35:43 EST 
Hello,

Our organization has reviewed the design of the contribution and has prototyped 
some of our content. Overall, the contribution does the "heavy lifting". :-)

Out of our prototypes, we have identified some enhancements that we would like 
to recommend for consideration. Here are the items in words, and then I will 
attach some graphic files to illustrate these points.

All recommendations requested to apply to both the separate Help System window 
and the dynamic help view, as appropriate.

1. Move Display button from bottom to the position right of the entry field 
(similar in look to the standard Search field & Go button). The rationale is 
the same concept as the Search field/Go button user experience: the user's 
focus is the entry field at the top of the index, and moving the Display button 
there puts the 'submit' action closer to the point where the user is entering 
his terms.

2. Grey-out the Display button when the user selects terms that do not have 
content (a topic) associated with them.

3. As the user types in terms into the entry field, as matches are made with 
the entries in the list box, the list box scrolls so that the matches are 
displayed at the top of the list box.

4. Font: Use black for entries in the index that are "containers" that do not 
have a topic associated with them. Use blue font (like HTML hypertext) for 
entries in the index that have topics associated with them. Rationale here is 
that the entries that have topics associated with them (like back-of-the-book 
index entries with page numbers) are the ones that the user can click to 
actually *get* a topic he wants to read. The blue distinction will convey that 
these are the "active" ones. (Using underlining to indicate the "clickable 
link" nature might also be useful, but perhaps the underline would reduce the 
white space and make it harder for the user to scan.)

5. As the user types in the entry field, index scrolls so that the matches go 
to the top. Instead of the current dark highlighting and the word font turning 
to white, keep the index entry color the same (black or blue as described in 
#4) and use a light gray for the background band on the matched word.

6. Remove the word "Please" from the UI. It's polite :-), but not normally used 
in UI text.

7. Spacing in the type-ahead entry field: when the user types a word that is 
for a container entry, then types a comma and a space, the type-ahead doesn't 
work. Many people who type are used to automatically adding a space after a 
comma. The processing needs to handle this so that the behavior is the same 
whether the user types spaces after commas or not. Also, currently when a user 
selects an index item in the list box, the entry field is populated with the 
nested entries, separated by commas. We recommend to have a space added after 
each comma for readability. 

8. Spacing in the index list box: Increase the indentation for sub-entries by 
one more space. Currently, it looks like the nested entry is 2 spaces in from 
its parent; for easier scanning, we are suggesting 3 spaces be used.

9. When an index entry points to multiple topics (like a back-of-the-book index 
entry has two or more pages associated with it): we strongly recommend an 
alternative to the current implementation using the secondary window. Our 
rationale is that we want the user to be able to focus their attention in a 
specific area, and minimize context shifts. Having a secondary window makes a 
context shift for the user. To meet this goal of minimizing context shifts, for 
these situations where one index entry has 2 or more topics associated with it 
(through installation of multiple components, for example), the Index function 
should automatically make that index entry a container and create the two or 
more next-level entries, so that they are displayed in the index list box 
itself. In this step, we recommend using the topic title or topic search title 
as the index entry for the generated next-level entries.

10. Index tab: We recommend that the Index tab be located between the Contents 
and Search tab, and be present wherever the Contents and Search tab are 
currently displayed in the Eclipse UI (for example, the separate Help System 
window and the dynamic help view). The rationale is that this placement between 
the Contents and Search tab more aligns with industry-standard (MS HTML Help, 
etc). Also, we recommend an updated graphic for the Index tab for NLS 
considerations. The current one is specific to the Western alphabet.

Thank you in advance for your consideration! As always, Eclipse Community 
discussion welcome.

Comment 97 Kiryl Kazakevich 2006-02-01 09:26:17 EST 
I would like to come to an agreement about public API of the index feature.

1. "org.eclpse.help.index" extension point.

This extension point is eliminated. Instead index is contributed through a new attribute of the "toc" element in the "org.eclipse.help.toc" extension point. The new attribute is "keywords" (other variants: "keywordIndex" "keyword_index"). For example:

<extension point="org.eclipse.help.toc">
    <toc
            file="toc.xml"
            primary="true"
            keywords="index.xml"/>
</extension>

This allows us to associate tocs with indexes.

2. Format of index.xml file.

Looking to discussions about format of this file, we can highlight the following requirements:

1) do not change this format because there were significant efforts on creation of tools based on this format (comment #77)
2) have structural compatibility with JavaHelp index format (significant part of this requirement is to replace the "topic" element with corresponding attribute of the "entry" element, comment #74)
3) support "see also" index entries (bug 120878)

I would like to propose the following solution. Do not change what is already presented in the format, but add a new attribute ("href") to the "entry" element which plays the same role as the "href" attribute of "topic" subelements. This allow us to have two possibilities to designate content linked with keyword:

* through the "topic" element, as it was before 
* through the "href" attribute of the "entry" element; this makes the "entry" element structurally compatible with the "indexitem" element in JavaHelp index.

A contributor of the index is free to choose a suitable possibility.

For example, the following fragments are identical:

<entry keyword="level 1">
    <entry keyword="level 2 item a">
        <topic href="..."/>
    </entry>
    <entry keyword="level 2 item b">
        <topic href="..."/>
    </entry>
</entry>

<entry keyword="level 1">
    <entry keyword="level 2 item a" href="..."/>
    <entry keyword="level 2 item b" href="..."/>
</entry>

The format of index files is as following:

<!ELEMENT index (entry)* >

<!ELEMENT entry (entry | topic)* >
<!ATTLIST entry keyword CDATA #REQUIRED >

<!ATTLIST entry href CDATA #IMPLIED >

<!ELEMENT topic EMPTY >
<!ATTLIST topic href CDATA #REQUIRED >
<!ATTLIST topic title CDATA #IMPLIED >
<!ATTLIST topic location CDATA #IMPLIED >

3. Public Java API.

At this moment I do not see what is really needed to be added to the public API of the org.eclipse.help plug-in in relation to the index feature. I suppose that index feature may be enhanced in the next versions of Eclipse after 3.2, and therefore I don't want to hurry to public some API.

I do not want to insist on association index with TOC.

Here my thoughts on the association.

As I understand ideology of help system, there are two conceptual things: content and way to navigate through content. Content is presented by documents, and way to navigate was presented before by TOCs and search. Index is a new way to navigate. From this point of view index is not subordinated to TOC, they are siblings. And we do not need to associate them with each other.

But TOC in the Eclipse help system is not just table of content. It provides mechanism to allow/disallow availability of content to consumer. Here I mean "ignoredTOCS" preference, anchors. That allows products to have all content included but not all visible depending on configuration, target platforms, etc. This feature of TOCs influences on search, and as I understand original discussion should influence on index.

For me the right way to associate index and TOC is to introduce a new conception "book", which tights them and corresponding content together. And separate mechanism similar to what we have in TOCs now to fill help system by suitable books with accompanied TOCs, index, etc.

So I agree to do not provide association of index with TOC. There is "ingnoredIndexes" preference which allows to control availability indexes in products. May be we will have ability to rearrange help system contribution mechanisms more rationally in next versions of Eclipse.

There is a mechanism available for products to explicitly hide index contributions. In pre-3.3 it's done the same way as table of contents entries are hidden, by using the ignoreIndexes preference in plugin_customization.ini (equivalent to the ignoreTOCS for table of contents), like this:

org.eclipse.help/ignoreIndexes=/my.doc.plugin/path/myindex.xml,/my.other.doc.plugin/path/myindex.xml

In 3.3 I recently added a new mechanism that is less error prone, easier to read and write tooling for, and more consistent with welcome. With the latest builds if you do a search for "HELP_DATA" you should find some info about it. This was added in bug 165168.
