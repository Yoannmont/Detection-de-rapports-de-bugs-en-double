Description Eugene Kuleshov 2003-08-14 12:26:01 EDT 
When working with multiple projects in the same worbench it is very inconvenient
to have global breakpoint list for the whole workbench. Just imagine debugging
of the common component project shared between several others.

It will be very helpful to have a filtering option similar to threads to enable
and disable breakpoints per-project.

Opposite to thread filtering this option should be enabled in offlene (when
debugger hasn't been started yet).
Comment 1 Darin Wright 2003-09-02 14:11:02 EDT 
Clarification - your request looks like it is asking for a filter in the

runtime filter. Do you want a runtime project filter, or a development time 
filter in the breakpoints view?
Comment 2 Eugene Kuleshov 2003-09-03 07:13:59 EDT 
I'm looking for the functionality to specify one or several projects in the 
breakpoint properties, so breakpoint will only activated/suspended for those 
projects. In runtime breakpoint should be activated if project from breakpoint 
properties match to project used to start application.

Probably it is not a bad idea to set such property to current project for 
selected source file (if any) when breakpoint is created.
Comment 3 Darin Wright 2003-09-03 11:42:57 EDT 
Currently, breakpoints are created on one resource (file), and this live in one 
project. At run (debug) time, the breakpoint is installed in all types that 
match the fully qualified name associated with the breakpoint. Thus, if you 
have > 1 type with the same fully qualified name in your workspace, you can get 
breakpoints from one project causing suspends in another project. Since the 
debugger cannot tell where the classfiles in a runtime originated from, it 
cannot determine what project a class file came from and if it should be 
installed (i.e. based on a project attribute). Classes are loaded at runtime, 
based on class loaders (which can use classpaths, or their own rules such as 
plug-in class loaders in Eclipse).

Thus, this feature is difficult. Even if the project -> classfile relationship 
were straighforward, we would not be able to (efficiently) determine if there 
are other classes in the workspace with the same name. For example, setting 
breakpoint in type a.b.c.X, in project A, requires that we search the workspace 
for all types with the name a.b.c.X, in order to populate the "project filter" 
list. This is possible (but slow).

It would be possible to group breakpoints in the breakpoints view by project 
(based on the file they are associated with). We would also need "workspace" 
and "external" groups for breakpoints not associated with a specific project 
(for example, breakpoints in a shared class library, or in an external source 
file).
Comment 4 Darin Wright 2003-09-03 11:44:15 EDT 
NOTE: the same problem remains: at runtime, we cannot tell which project a 
classfile originated from, and thus, we could not filter based on project.

Comment 12 Darin Wright 2003-09-04 10:20:53 EDT 
I believe the same result can be acheived in a simpler more intuitive manner -
by grouping and enabling/disabling breakpoints by project. For example, a 
breakpoints view that grouped breakpoints by project, with a check box on the 
project allows you to quickly disable all breakpoints in a project. The user 
can understand this easier, rather than the concept of a runtime project 
filter. (It means that you would need to disable the projects that you are not 
interested in). We could offer a convenience action to enable a project (and 
disable all others).

Note that a project attribute is optional on a launch config - and does not 
even exist on some launch configs (such as a Runtime workbench config). In 
these cases, the feature could not be supported as a runtime filter - but would 
work as described above.

Comment 15 Eugene Kuleshov 2003-09-04 13:07:03 EDT

Comment 16 Darin Wright 2003-09-04 14:18:15 EDT 
(Answer to questionss)
* Breakpoints in dependent projects come from the project they are defined in. 
A "dependent project" does not know about upstream projects that require it.
* Breakpoints in jars come from the project the jar is defined in. If the Jar 
originates from a classpath variable or container (such as JRE_LIB or 
JRE_CONTAINER, used for system libraries), the jar is considered "global", and 
does not have a project association.

Your last comment confirms a previous inclination I had - that is, you need to 
filter based on the config, not the project.

Comment 18 Darin Wright 2003-10-29 11:38:58 EST 
I'm marking this as a duplicate. It has the same underlying problem/issue as 
bug 22536. The problem is a breakpoint appears on a class with a name that is 
not unique in the workspace. At runtime, the debugger does not know which 
project the class came from (it could guess, but it is not definitive - class 
loaders do not need to obey "Eclipse project" rules). For example, in the case 
of plug-ins, two classes could have the same name. Both classes may be 
loaded/running, but have different classloaders (plug-in classloaders), and we 
cannot tell which class to install the breakpoint in (so we put it in both).
