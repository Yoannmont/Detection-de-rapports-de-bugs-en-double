Description Jared Burns 2002-07-24 10:34:30 EDT 
Build 2.0
Linux-GTK

When I launch a selfhosting Eclipse with the standard SDK plugins, I see 6 
"HttpProcessor" processing running in my target for no apparent reason. I 
also see two "StandardManager" processes. All told, it looks like Help is 
running 8 processes and I haven't even touched help. I'm just doing Java 
development. The rest of Eclipse has a combined 3 processes (one of which 
is the UI thread). Is there a reason for this?

I see:
Thread [StandardManager[]] (Running)
Thread [HttpConnector[0]] (Running)
Thread [HttpProcessor[0][0]] (Running)
Thread [HttpProcessor[0][1]] (Running)
Thread [HttpProcessor[0][2]] (Running)
Thread [HttpProcessor[0][3]] (Running)
Thread [HttpProcessor[0][4]] (Running)
Thread [StandardManager[/help]] (Running)
Comment 1 Dorian Birsan 2002-07-24 12:05:03 EDT 
Help uses the Tomcat plugin and those HttpProcessor threads are likely listener 
threads in Tomcat. I think this is configurable, so we can look at limiting the 
number of threads. If Tomcat is designed correctly, those threads should be 
blocked on I/O or some other events, or should be mostly sleeping and only 
wakeup at specified intervals.

The other think I don't understand, is why is the help plugin running 
(the /help context seems to be registered, and is done by the help plugin) when 
you haven't touched help? Are there other plugins (like eclipse.ui) triggering 
the help plugin ?
Comment 2 Dorian Birsan 2002-07-25 13:19:30 EDT 
I checked the Tomcat dev mailing list, and it looks like the mult-threaded 
design is harmless. Here is the answer by Craig, one of main folks behind 
Tomcat:

----- begin quote ---------

List:     tomcat-user
Subject:  Re: Limiting the number of live processes
From:     "Craig R. McClanahan" <Craig.McClanahan@eng.sun.com>
Date:     2000-09-22 17:52:37
[Download message RAW]

Gary Lawrence Murphy wrote:

> >>>>> "C" == Craig R McClanahan <Craig.McClanahan@eng.sun.com> writes:
>
>     C> Gary Lawrence Murphy wrote:
>     >> My idle Tomcat server is generating 35 processes (no clients);
>     >> this is on a test machine where I really only need about a
>     >> tenth that many.  Can I limit the number of live processes?
>     >>
>
>     C> Is this on a Linux box?  Native threads JDK?  If so, those are
>     C> really threads, not processes -- there is only one "real"
>     C> process, and "ps" output is *really* misleading.
>
> As soon as I posted, I realized that I should have said so, and the
> answers are yes, probably and probably yes.  Linux & IBM JDK 1.3.
>
> So now I have just a trivia question: Why so many threads on an idle
> system?
>

Threads are generally created by the following components:

* The connector pool precreates threads so that, as simultaneous
  requests come in, they will already exist.  Then, these threads are
  recycled rather than being created and destroyed all the time.  The
  pool can be configured through property settings for how many it
  creates.

* Background threads for things like checking classfile changes for
  autoreload, session invalidation, etc.

* Depending on its implementation, the JVM might itself have background
  threads running for garbage collection or other reasons.

Most of the time, the threads that Tomcat creates are blocked waiting for
input (or waiting for a timer to expire), so they don't occupy any CPU time
(or any memory, other than the runtime stack that is allocated for each
thread).  So, it's no big deal to have the threads active but idle.  In
fact, it improves performance to have them this way, rather than created on
the fly for every request.
