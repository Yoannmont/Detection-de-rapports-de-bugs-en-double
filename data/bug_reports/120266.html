Description Naci Dai 2005-12-11 08:49:28 EST 
We were able to replicate the problem with a public project.  This test uses the betterpetshop example application from https://betterpetshop.dev.java.net.

1) We created a dynamic web project in a clean intsall and created the petshop project in WTP. During this process we have created a project by modifying default setting in the wizard. i.e:
- we have used a different context name for the war (defaultt was the project name)
- Defined multiple source folders (changed defaults) (src/main/java for java sources and src/main/webapp for webcontent)
- Copied the java sources, libraries and webcontent from the petshop)

2) Defined Tomcat 5.5 as the server.

Project was created successfully and run correctly.

IN THE SECOND PART OF THE TEST.
--
We created a brand new workspace.  And use "Import Existing Project into Workspace" to import the petshop project.  Project imports, WTP validator starts, but hangs at 50%.  Never completes.  

If we attempt to create a new project while the workspace is in this states eclipse locks.  The only fix is to kill the workspace.

-Workaround is to import with automatic builds or validation turned off.

Attached the project for testing

Comment 4 Vijay Bhadriraju 2005-12-11 19:35:27 EST 
After a thorough investigation into this lock, the reason for the lock is due to the Validation Framework doing a lot of processing when it does the filtration to calculate the list of files for a validator to validate the files it is interested in. We have already added quite a bit of optimization in this area in M9 to reduce this processing to the extent needed but in this case it is the jar files that are going through the extra processing when they don't have to.

The validation framework does not do anything specific for jar files like cracking them open and looking for files inside the jar so we need to filer out jar files from this filter processing. I put in a simple fix to ignore jar files in the filters code of the framework and saw like 75% improvement. I sent the fix to Lawrence too and he saw the same. At this point i am not sure if this defect is a stop shipper but if it is i have a simple and safe fix for it and more elegant fix can be worked on for 1.0.1. Here are the details of the fix.

Class changed ValidatorMetaData -> private boolean checkIfValidSourceFile(IResource file) method

private boolean checkIfValidSourceFile(IResource file) {
  if (file.getType() == IResource.FILE) {
   //Follwing 2 lines avoids the lock during validation
   //**********************************************
   if(file.getFileExtension().equals("jar"))
    return false;
   //************************************************
   IProjectValidationHelper helper = ValidationHelperRegistryReader.getInstance().getValidationHelper();
   IProject project = file.getProject();
   if (helper == null || project == null)
    return true;
   IContainer[] outputContainers = helper.getOutputContainers(project);
   IContainer[] sourceContainers = helper.getSourceContainers(project);
   for (int i=0; i<outputContainers.length; i++) {
    String outputPath = outputContainers[i].getProjectRelativePath().makeAbsolute().toString();
    for (int j=0; j<sourceContainers.length; j++) {
     String sourceContainerPath = sourceContainers[j].getProjectRelativePath().makeAbsolute().toString();
     if (outputPath.equals(sourceContainerPath))
      continue;
    }
    String filePath = file.getProjectRelativePath().makeAbsolute().toString();
    if (filePath.startsWith(outputPath))
     return false;
   }
  }
  return true;
 }

Comment 19 Chuck Bridgham 2005-12-15 12:40:05 EST 
The first patch is the J2EE implementation of ProjectValidationHelper that now can cache its information during the scope of a validation run.

The second patch is to add a method in the interface to help flush the cache at the end of a validation run.

The existing interface is used as an extension mechanism, for calling java specific api, that can't be invoked from the wst layer.

Essentially we changed the implemetation to allow caching of the java source/output roots, which was an expensive call, and it can't change during the lifetime of a validation run.
