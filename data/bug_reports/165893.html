Comment 19 Michael Scharf 2007-04-24 23:39:49 EDT 
If I understand the request correctly, both streams of data (remote-to-terminal and terminal-to remote) have to be decorated. 

Currently there is an ugly asymmetry in the way those streams of data are handled (bug 183901).

To solve this request there has to be a way modify both OutputStreams. The simplest way would be to use decorators on both streams (e.g. java.io.FilterOutputStream). The real question is to to manage a potential set/stack of decorators. 

Here's one idea: both directions of the communication could be handled by one Interface:

/**
 * Manages both ways of the communication between the terminal and
 * the remote site.
 */
public interface ITerminalIO {
	/**
	 * Direction terminal -> ITerminalConnector
	 * 
	 * @return the stream that comes from the terminal (e.g. the user input)
	 */
	OutputStream getTerminalToRemoteStream();

	/**
	 * Direction ITerminalConnector -> terminal. Data written to this stream
	 * appears in the terminal.
	 * 
	 * @return the stream that writes to the terminal (e.g. text to be displayed
	 *         in the terminal).
	 */
	OutputStream getRemoteToTerminalStream();
}

And TerminalIODecorators could be provided by an extension point:

public abstract class TerminalIODecorator implements ITerminalIO {
	private ITerminalIO fTerminalIO;

	/**
	 * @param terminalIO
	 *            the terminalIO that will be decorated
	 */
	public void setTerminalIO(ITerminalIO terminalIO) {
		fTerminalIO = terminalIO;
	}

	/**
	 * @return the decorated terminal IO
	 */
	public ITerminalIO getTerminalIO() {
		return fTerminalIO;
	}

	/**
	 * called when the decorator is removed
	 */
	public void dispose() {
	}

	public OutputStream getRemoteToTerminalStream() {
		return fTerminalIO.getRemoteToTerminalStream();
	}

	public OutputStream getTerminalToRemoteStream() {
		return fTerminalIO.getTerminalToRemoteStream();
	}
}



The extension point would provide the name and description of the decorator and a subclass of TerminalIODecorator. 

The user interface would have to be extended to allow managing TerminalIODecorator fora single connections or for all connection (in the preferences).

If no extension is available, the UI might not be shown.

The only problem I can see is that decorators might be order dependent. But in many cases there's at most one decorator anyway....

I'm not sure I understand your proposal entirely, but here are a few comments:

- a "decorator" is a misleading term and probably makes you think that the impact of selecting one of those is slight. In fact it could completely change the handling of input & output never sending anything or receiving anything from the terminal window or opening a terminal window of it's own or no terminal window at all.
- I don't think the pipe concept works. It very tightly confines a plugin to humbly modifying the input/output stream. The plugin should be able to take over the show, using bits from the existing tm-terminal as it sees fit. If I wanted to add a plugin that implemented some fancy terminal(e.g. with flashing text, progress messages, etc.), then I would need to create my own terminal SWT GUI implementation.
- only one plugin should be able to be selected at the time. If that plugin needs to use other plugins, it will do so and define the "ordering" of which it uses those plugins. This leaves the decision making/design firmly in the plugin.
- the current terminal window should be a plugin itself. It could expose parts that a plugin could use. The terminal GUI window could be something that exists while the default plugin is now selected so other plugins could communicate with it.

  org.eclipse.ui.commands
  org.eclipse.ui.bindings
  org.eclipse.ui.handlers
  org.eclipse.ui.popupMenus
  org.eclipse.ui.viewActions
in order to register some commands in the Terminal View. By registering these through plugin.xml, the corresponding extension would not be loaded / activated unless selected by the user. Once selected, the extension would use API in order to get to the TerminalControl and its streams.

So, we're talking about adding API but not about adding extension points.
