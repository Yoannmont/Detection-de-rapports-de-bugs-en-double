Comment 10 Silenio Quarti 2004-04-14 16:01:29 EDT 
Here is what is happening:

Eclipse is disabling the focus control.  When this happens, another control 
within the same Shell is assigned focus.  If this new control is a radio 
button and the platform is Windows, it becomes selected.  This is Windows 
platform behavior.  So, as Eclipse disables the controls one by one, it pushes 
the focus along in the Shell, selecting radio buttons.

We can't (and shouldn't) work around this Windows plaform behavior in SWT.
Comment 11 Sebastian Davids 2004-04-14 16:39:25 EDT 
>>We can't (and shouldn't) work around this Windows plaform behavior in SWT.

But it should be handled somewhere _above_ SWT then.

Here's an analogy to the observed behavior:

I'm in room A and place a basket in the middle of the room. I leave the room and
enter room B. After a short while I go back to room A. Instead of finding the
basket in the middle of the room -- as I would expect -- it is next to an edge
of the room.

Reason: while you're in the room it's perfectly still but once you leave the
room the floor tilts and the basket will move to an edge of the room.

@@@

What astounds me is the following: why hasn't this odd behavior surfaced
earlier? Could it be that there's something "special" about this wizard ...
considering there are so many wizard pages/dialogs not exhibiting this weird
behavior?

Comment 17 Ines Khelifi 2004-04-20 10:10:48 EDT 
We have traced the issue to this line in JavaProjectWizardFirstage

 
	public void setVisible(boolean visible) {
		super.setVisible(visible);
		if (visible) fNameGroup.setFocus();
	
	}

So this wizard is setting the focus to the text which is generating a sequence 
of events that are changing the value of these widgets. Removing this line 
removes this problem.

ControlEnableState currently remembers these values ad restores them - they 
appear to be set afterwards.
Comment 18 Ines Khelifi 2004-04-20 11:44:51 EDT 
I dug a little deeper, and Steve was completely right about the involvement of 
the ControlEnableState class. I do not understand the entire events sequencing 
for the widgets in the JavaProjectWizardFirstPage, but from what I observed, I 
do not think this bug is directly related to the Wizard implementation. It 
seems that the many events thrown in the JavaProjectWizardFirstPage conflict 
with the ControlEnableState enablements.

A quick fix is to move the super.setVisible() call at the end of 
JavaProjectWizardFirstPage#setVisible() to avoid conflicting changes from both 
parts. 
The method would look like this:

/*
 *  (non-Javadoc)
 * @see org.eclipse.jface.dialogs.IDialogPage#setVisible(boolean)
*/
public void setVisible(boolean visible) {
  if (visible) fNameGroup.setFocus();
  super.setVisible(visible);
}

I tested it out, and it seems to work. But as I said, I do not fully 
understand the sequence of events. Other suggestions are welcome.
