Description Michael Burkhart 2007-03-21 11:58:43 EDT 
Our product uses many preference pages and dialogs that use a subclass of org.eclipse.ui.forms.widgets.SharedScrolledComposite to create the scroll area. The scrollbars only show up if you resize the window, or if you run in high contrast mode. The problem is that the thing doesn't automatically scroll when controls are tabbed to inside the composite. Also, arrow and page down/up keys don't work to scroll the page. I found a way to make this work for simple layouts (un-nested Composites) but it seems like something that should be handled by the widget.

Here is an example to try out. The lower checkboxes in the scrolled composite do not come into focus when you tab to them, but when you uncomment the crazy code in there that puts activation listeners on them, then the scrolled composite scrolls.

Comment 7 Carolyn MacLeod 2007-03-27 14:21:14 EDT 
If you want slightly less crazy code, you could try hooking focus listener instead of activate.

import org.eclipse.swt.*;
import org.eclipse.swt.widgets.*;
import org.eclipse.swt.layout.*;
import org.eclipse.swt.events.FocusAdapter;
import org.eclipse.swt.events.FocusEvent;
import org.eclipse.swt.graphics.*;
import org.eclipse.ui.forms.widgets.SharedScrolledComposite;

public class ScrolledPageTest extends SharedScrolledComposite {

        public ScrolledPageTest(Composite parent, int style) {
                super(parent, style);
        }

        public static void main(String[] args) {
                final Display d = new Display();
                Shell shell = new Shell(d, SWT.SHELL_TRIM);
                shell.setText("Scrolled page test");
                shell.setLayout(new FillLayout());
                shell.setSize(400, 200);
                final ScrolledPageTest scrolled = new ScrolledPageTest(shell, SWT.V_SCROLL | SWT.H_SCROLL);
                Composite comp = new Composite(scrolled, SWT.NONE);
                comp.setLayout(new FillLayout(SWT.VERTICAL));
                scrolled.setContent(comp);

                for (int i=0; i<20; i++) {
                        Button b = new Button(comp, SWT.CHECK);
                        b.setText("This is button number " + i);
                        b.addFocusListener(new FocusAdapter() {
                        	public void focusGained(FocusEvent e) {
                                Control child = (Control)e.widget;
                                Rectangle bounds = child.getBounds();
                                Rectangle area = scrolled.getClientArea();
                                Point origin = scrolled.getOrigin();
                                if (origin.x > bounds.x) origin.x = Math.max(0, bounds.x);
                                if (origin.y > bounds.y) origin.y = Math.max(0, bounds.y);
                                if (origin.x + area.width < bounds.x + bounds.width) origin.x = Math.max(0, bounds.x + bounds.width - area.width);
                                if (origin.y + area.height < bounds.y + bounds.height) origin.y = Math.max(0, bounds.y + bounds.height - area.height);
                                scrolled.setOrigin(origin);
                        	}
                        });
                }

                shell.open();
                while (!shell.isDisposed()) {
                        if (!d.readAndDispatch()) {
                                d.sleep();
                        }
                }
                d.dispose();
        }
}

Comment 9 Carolyn MacLeod 2007-03-28 15:29:45 EDT 
We do think we should fix this, but after 3.3, because it's too late to make this kind of change before 3.3. ships. Our current thinking is that we have two reasonable choices for the fix:

1) make ScrolledComposite scroll automatically on keyboard traversal
   - this would cause problems because others have already written their code to scroll automatically using setOrigin, and if we also scroll, the user would see 2 scrolls. Worst case is if both scrolls end up in a slightly different place. Not to mention that it would slow things down to scroll twice.
   - we wouldn't make this kind of breaking change this close to the 3.3 release date.

or 2) provide ScrolledComposite.show(Control) API, similar to showItem in Table and Tree or showSelection in List and Text, that would give you a prettier way to do the scrolling. For example, application code would look something like this:

   b.addFocusListener(new FocusAdapter() {
       public void focusGained(FocusEvent e) {
          scrolled.show((Control)e.widget);
       }
   });

   - this would not have the double-scroll problem listed above, however M6 was the API freeze for 3.3, and we can't add API now.
Comment 10 Steve Northover 2007-03-29 11:14:01 EDT

The default would be off to preserve the existing behavior.  Clients who wanted the behavior would need to explicitly turn it on using the new API.
