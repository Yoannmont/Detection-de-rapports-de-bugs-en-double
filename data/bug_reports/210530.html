Description Cong Chen 2007-11-21 08:55:32 EST 
In ServerExplorerContentProviderNav.getChildren(Object parentElement), class Load is created for loading children contents. But this loading is executed by the way of Job, so that it is in another thread. 
So if a content provider A was overrided by B, as the logic of current implementation of CNF, A's children is still loaded but will be replaced by Bs'. So here is the problem, in our implementation, A returns immediately with a meaningless Holder object, then B uses it's own children to replace whatever A returned, and returns to parent. It's correct so far. But then the 'loading' thread is executed, which appends A's real children to parent. So at the end, you will see children of both in DataSourceExplorer.

Patch will be provided soon.

Comment 9 Loic JULIEN 2008-01-25 17:17:54 EST 
Hi Cong,

My true apologies for the very late answer.

This is a very interesting problem. Let's make sure we fully understand the design of the Loading service and see how it affects this use-case.

1- The loading service allows to release the UI Thread until the objects are ready to be populated in the tree. A little non-blocking animation is being displayed while loading occurs. (Just for the record, I delivered this through https://bugs.eclipse.org/bugs/show_bug.cgi?id=208786 and sent an email back then on our DTP mailing list on Nov. 6)

Clients who wants to leverage this functionality need to follow this pattern: 

    * Implement ILoadingService in their content provider
    * Implement the method public String getLoadingDescription()
      -> The description will be inserted in the following message: Loading {0} provider
    * rename the method getChildren with load
    * Make hasChildren non expandable for LoadingNode -> element instanceof LoadingNode ? false : ...
    * Copy the following new getChildren()

public Object[] getChildren (Object parent)
{
    return new Loading ().getChildren(this.viewer, parent, this);
}

2- Working with the overriding feature of the CNF

Here what I would suggest to better work with the override. I have to admit, we do not have any override so those are just ideas and I will be more than willing to write a little override would you not be able to validate this.

A- We could prevent any children (A in your case) to be contributed if the placeholder is not found (conveniently removed). In your pipeline, you would then have to remove the placeholder when you intercept it.

B- Since the design does a modification on the viewer directly, you only react to interceptAdd in your pipeline and modify the Shape accordingly.

My prefeference would be to go with -B- however, I am open to suggestion. Please, let me know so that we can bring this to closure.

Thanks,
~Loic
