Comment 12 Jed Anderson 2002-06-14 16:17:51 EDT 
I have reproduced this bug with JRun 4.

To reproduce:

1. Install JRun4 into default location (don't change the settings in the 
installer), but point the JRE/JDK setting at a JDK 1.3.0 VM (exact version 
below)
2. Change <JRun4 Install>\bin\jvm.config by changing the java.args line to read
java.args=-classic -Xnoagent -Xmx128M -Xdebug -
Xrunjdwp:transport=dt_socket,address=53006,server=y,suspend=n
3. Copy the following files from your jdk1.3\bin directory into your jdk1.3
\jre\bin directory:
dt_shmem.dll
dt_shmem_g.dll
dt_socket.dll
dt_socket_g.dll
jdwp.dll
jdwp_g.dll
3. Open two consoles (one to start jrun, and one to stop jrun).  In each set 
the path variable to include the jdk1.3\jre\bin directory.  e.g. set path=%
path%;c:\jdk1.3\jre\bin
4. Cd to <JRun4 install>\bin in both the consoles.
5. In one console enter "jrun -start", jrun will start in debug mode.
6. Open a web browser on http://localhost:8200/
7. Click _Web services_.
8. Under the Class-based web service, click _Test_
9. Change a field, and click calculate
10. Open eclipse.
11. Connect with the Remote Java Application launcher, specify port 53006.
12. When you connect, find the web-0 thread.  Pause it, hit F6 (step over), 
pause again, hit resume.
13. Hit the web page again.
14. Disconnect.
15. Hit the web page again.
16. Connect again, Eclipse will lock up because the VM is in a funky state.
17. In the second console type "jrun -stop", this will unlock Eclipse as the VM 
terminates.

This test does not reproduce all the time.  Whenever the threads appear in the 
the debug view with names like Thread-1, Thread-2, etc the next time you 
connect it will fail.

java version "1.3.0"
Java(TM) 2 Runtime Environment, Standard Edition (build 1.3.0-C)
Java HotSpot(TM) Client VM (build 1.3.0-C, mixed mode)
