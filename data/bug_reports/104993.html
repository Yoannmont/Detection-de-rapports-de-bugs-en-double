Comment 8 Kathy Chan 2005-09-09 14:34:44 EDT 
I've tried putting mail.jar and activation.jar in org.apache.axis_1.2.1/lib,
updated MANIFEST.MF and restart with "-clean".  The problem went away for
Geronimo server which has a similar problem of NoClassDefFound with
javax.activiation.DataSource (see bugzilla 108156 for more detail).  

The detail steps are:

1. Shutdown Eclipse.
2. Add activation.jar and mail.jar to org.apache.axis_1.2.1/lib/.
3. Add an entry for it (ie. "lib/activation.jar,", "lib/mail.jar") to
org.apache.axis_1.2.1/META-INF/MANIFEST.MF.
4. Restart with -clean.

I am suspecting there's javax.activation.DataHandler in one of the JARs in
the Java build path of the project containing the Java bean you want to
create Web service for.  See defect 105335 and Axis defect 2146 
(http://issues.apache.org/jira/browse/AXIS-2146).

You could find out what JARs are included in the
Java build path by right-clicking on the project, select properties, then
Java Build path and go to the library tab.  We would appreciate if you could
find the name of the JAR(s) containing javax.activation.DataHandler and
append the info in this defect.

Comment 18 Kathy Chan 2006-01-13 14:19:29 EST 
The user would run into this problem if targetting Geronimo server since the following Geronimo JARs contains javax.activation.DataSource and are put on the Web project's classpath if targetting Geronimo server (if using 01/11 or later Geronimo plugins):

- geronimo-activation_1.0.2_spec-1.0.jar
- geronimo-j2ee_1.4_spec-1.0.jar

The workaround is to exclude the above JARs in ClasspatnUtils when figuring out the classpath to pass to the Axis emitter.

The attached patch (source and JAR) can be used as a workaround when generating Web service using Geronimo until a more elegant solution can be found for the more general problem.

I tried switching the thread's context class loader to a custom classloader(EclipseClassloader + AntClassloader based on the classpath of the Web project) before calling the Java2WSDLAntTask in the Java2WSDLCommand.  

However, JAFSerializerFactory and org.apache.axis.wsdl.fromJava.Emitter are still using the Eclipse classloader (not the custom classloader).   

I then tried calling Java2WSDLAntTask in secondary thread.  However, the Emitter did not solve the problem.

Even after changing the code in JAFSerializerFactory to use ClassUtils.forName() (which uses the classpath from the Web project), the problem is still there.

The failure occurs when the org.apache.axis.wsdl.fromJava.Emitter is being initialized.

When DefaultTypeMappingImpl.initMappings() is called, JavaUtils.isAttachmentSupported returns true because ClassUtils.forName() (the one that has the Web project's classpath containing activation.jar) is used to load javax.activation.DataHandler.

if (JavaUtils.isAttachmentSupported()) {
            myRegister(Constants.MIME_PLAINTEXT, java.lang.String.class,
                    new JAFDataHandlerSerializerFactory(
                            java.lang.String.class,
                            Constants.MIME_PLAINTEXT),
                    new JAFDataHandlerDeserializerFactory(
                            java.lang.String.class,
                            Constants.MIME_PLAINTEXT));
        }

In JAFDataHandlerDeserializerFactory:

private static Class getSerializerClass(Class javaType, QName xmlType) {
        Class ser;
        if (Image.class.isAssignableFrom(javaType)) {
            ser = ImageDataHandlerSerializer.class;
        }
        else if (String.class.isAssignableFrom(javaType)) {
            ser = PlainTextDataHandlerSerializer.class;
        }
//...
        return ser;
}

While loading the class PlainTextDataHandler, the classloadder is trying to load org.xml.sax.Attributes and javax.activation.DataHandler which are in the serialize method.  Commenting the method out proves that the classloader tries to load those classes during initialization even though the method is not called.  This is a problem because activation.jar and mail.jar are not shipped in the org.apache.axis plugin in WTP because of license agreement issues.  That's why we are getting a NoClassDefFound on javax.activation.DataSource.

So, to workaround the problem, we put together a few empty interfaces and classes so that the org.apache.wsdl.fromJava.Emitter can initialize properly.  The dummy javaxzombie.jar contains:

javax/activation/DataSource.class
javax/activation/DataHandler.class
javax/mail/Multipart.class
javax/mail/internet/MimeMultipart.class

After putting the javaxzombie.jar in the lib directory and updating the manifest file for org.apache.axis to include the JAR, the NoClassDefFound for javax.activation.DataSource is solved.
