Description Wendell Beckwith 2006-08-12 22:07:06 EDT 
We are working with the platform mbean server which is required of all Java 5+ jvms and we have encountered a usability problem with the WTP server connectors.  Mainly when the jvm is started we pass the following 3 options as vm args

-Dcom.sun.management.jmxremote.port=<port #>
-Dcom.sun.management.jmxremote.authenticate=false
-Dcom.sun.management.jmxremote.ssl=false

which is correct when starting the server.  However when shutting down the server you can't pass these options because the jvm used to shutdown the server will again try and bind to the port used to startup the server which will fail with an address already in use exception.  The end result is that we can't shutdown nor restart the server from inside eclipse.  We either have to use external scripts to perform a shutdown/restart or we just have to terminate the server instance.
Comment 1 Tim deBoer 2006-08-13 23:38:06 EDT 
What exactly are you asking here? Components have no control over the VM arguments that you use to launch the JVM with. Shouldn't the server (or at least the start/stop commands) be run in a different JVM?
Comment 2 Wendell Beckwith 2006-08-14 12:06:59 EDT 
The problem is easy to reproduce and from that it should be simple to understand the usability problem.  Start an Apache Tomcat 5.5.x server instance with the following additional vm args "-Dcom.sun.management.jmxremote.port=1099
-Dcom.sun.management.jmxremote.authenticate=false -Dcom.sun.management.jmxremote.ssl=false".  You do not need to actually deploy anything to it.  Once the server has started up, now _try_ and shut it down or restart it via the server connector.  You can't.  The reason is that the tomcat server config has on the Arguments tab a text field for "Program arguments", which contains "start" and a text field for "VM arguments", which contains "-Dcatalina.base="E:\dev\workspace-eclipse\.metadata\.plugins\org.eclipse.wst.server.core\tmp0" -Dcatalina.home="E:\dev\apache-tomcat-5.5.17" -Djava.endorsed.dirs="E:\dev\apache-tomcat-5.5.17\common\endorsed" -Dcom.sun.management.jmxremote.port=1099
-Dcom.sun.management.jmxremote.authenticate=false -Dcom.sun.management.jmxremote.ssl=false".

Starting and stopping tomcat requires 2 jvms, 1 and 2 respectively.  JVM 1 uses the last 3 options to bind to port 1099 and start the tomcat server.  All is fine at this point.  However, when you try and shutdown/restart tomcat another jvm is used, JVM 2.  JVM 2 is also given the same vm args which will lead jvm 2 to try and bind to port 1099 again.  This will fail because jvm 1 (the tomcat server) is still bound to it, thus JVM 2 will exit before it ever got the chance to run the Apache code to shutdown/restart JVM 1.

So all in all the launch configs need separate startup and shutdown program arguments and vm arguments.

I hope this is clearer.
Comment 3 Tim deBoer 2006-08-15 10:01:52 EDT 
This in an implementation detail of the Tomcat adapter, and we could use a different adapter or just strip out the properties... is that what you're looking for?
Comment 4 Wendell Beckwith 2006-08-15 14:24:57 EDT 
(In reply to comment #3)
> This in an implementation detail of the Tomcat adapter, and we could use a
> different adapter or just strip out the properties... is that what you're
> looking for?
> 

I haven't looked at the code but I figure this is a general problem that just manifests itsself easily with the server connectors that execute java processes to start, stop, restart, publish, etc to the server instance.  I don't think we can just strip them out though.  In this case that would work but what if there are properties defined for a security manager, endorsed directories, etc.  We would simply be rearranging deck chairs with out addressing the real issue.  The real issue is that for server connectors that execute java process, the user needs to be able to add/remove vm args for each process action (Start, Stop, Restart, Publish).  As I said this only affects connectors that execute java processes to do the actual work.  Connectors that work across HTTP, RMI or custom protocols would not exhibit this problem since a new jvm with the same vm args would not be created.

Comment 8 Larry Isaacs 2006-09-13 19:28:41 EDT 
Tim,

It should be no problem to incorporate something for WTP 2.0.  I'm curious whether you think we need a limited fix for WTP 1.5.2.

At least "com.sun.management.jmxremote*" properties could be removed, since these properties aren't totally unexpected, which would get us past this specific issue.  I would need to verify, but it may be that what is returned by ITomcatVersionHandler.getRuntimeVMArguments() is sufficient for "stop" and we could just skip the "merge" of the other existing arguments.
Comment 9 Wendell Beckwith 2006-09-13 23:37:35 EDT 
Isn't the more general solution to simply add 2 textfields to the tomcat server adapter?  One field contains startup vm args and the other contains shutdown vm args.  The reason this seems like a WTP issue is because in the normal case no thought to test it running a platform mbean server.  However, trying to create a one off would put the code in the same boat because java 6 could introduce new vm args that no one thought to test the combinations out with the tomcat server adapter.  However, were startup and shutdown vm arg fields available, I would have just added the correct args to startup and made sure they weren't in shutdown.
Comment 10 Larry Isaacs 2006-09-14 06:13:40 EDT 
I agree with your approach, but this level of change would need to go in WTP 2.0.  If this is to be addressed in WTP 1.5.x, then something less ambitious will be needed, which is the context for my specific suggestions.  In your case, how important is it to do something about this in WTP 1.5.x?
Comment 11 Wendell Beckwith 2006-09-14 06:45:13 EDT 
(In reply to comment #10)
> I agree with your approach, but this level of change would need to go in WTP
> 2.0.  If this is to be addressed in WTP 1.5.x, then something less ambitious
> will be needed, which is the context for my specific suggestions.  In your
> case, how important is it to do something about this in WTP 1.5.x?

It's very important as we can't use WTP for stopping/restart the connector.  We either kill the process or have to keep the shutdown.[sh|bat] script on the desktop to properly stop the tomcat instance.  Can you comment on how this is a 2.0 "level of change"?  The way I see it, the 2 textfields can be added with 100% backward compatibility.  The first text field, "Startup VM arguments" is already there labeled "VM arguments". That textfield needs to be renamed.  In keeping with backward compatibility, any previously defined 
VM arguments" will get auto-magically moved to the new "Startup VM arguments" textfield.  Another textfield needs to be slid in below "Startup VM arguments" labeled "Shutdown VM arguments".  This field will be empty since it didn't exist  before.  Finally the code would have been updated to call getStartupVMArguments() and getShutdownVMArguments() at the appropriate times.  Were I not patching maven and pde build to work together nowadays, I would have already done this and submitted a patch.  I agree I have,t looked at the code but the mental thought process works out to be a very small addition, not a deletion or big modification.
Comment 12 Larry Isaacs 2006-09-14 08:31:35 EDT 
What gets implemented for WTP 2.0 should be the done the "right" way.  To me, this means potentially making these fields available for all servers, not just Tomcat.  Also there are details to be worked out. For example, if the "Enable security" is checked in the Tomcat configuration and a user has added their own separate "java.security.policy" system property to the startup arguments, what happens?  Does one silently win, a warning dialog pops up, or something else.  Is there validation that displays an error before such a conflict is created?  Also, should there be separate arguments for debug, run, and profile.  If there is a need for this, it would be nice if you didn't lose the ability to switch directly from run to debug because you have to manually modify the arguments.  The scale of all this I think is too big for the 1.5.x stream.

Doing something simpler but similar in WTP 1.5.1, but just for Tomcat, will still be a fair amount of work and complicate doing things "right" in WTP 2.0.  Unfortunately, though my WTP work is focused strictly on Tomcat support, I'm a part time WTP committer and my day job in the past hasn't left many cycles to devote to WTP.  This is the main reason not a lot changed in the Tomcat support between 1.0 and 1.5.  I'm finding more cycles these days, but it's not unlimited.  I have to decide what is the best use of those cycles.  Tim, as a component lead, has a much wider focus in WTP and I believe would face the same kind of decission for that reason.  Thus, if this issue is to be addressed in WTP 1.5.x, I think it will need to be a limited, low risk, minimum cycle, type fix, such as what was suggested earlier.
