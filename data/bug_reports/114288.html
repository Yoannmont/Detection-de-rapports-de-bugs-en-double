Since Internet Explorer is the only one having problem with scaling text which size is specified with an absolute length unit, I suggest you set the main text size as follows:

/* For IE: 90% or whatever size you want to achieve assuming
 * 12pt default: solve 12pt * x = desired_size (x is the percentage/ratio) */
body { font-size: 90% }

/* For other browsers set just your desired size. This rule
 * won't be recognized by IE. */
html > body { font-size: 11pt }

You won't fall into trap of reducing the default size too much assuming certain default size where it is not like that. The 'pt' unit (as opposed to 'px' unit) will relatively guarantee the actual size will be consistent on screens with different DPI, as in the case of Jeff McAffer.

Comment 28 Stanimir Stamenkov 2005-12-17 08:13:55 EST 
O.k. Seems people "want small fonts". Here's my suggestion, set the main text font using a system font keyword:

body {
  font: 9pt sans-seif;
  font: message-box;
}
/* Reset styling the system font may have set.
 * That's in a separate rule because of IE bug. */
body {
  font-weight: normal;
  font-style: normal;
}

IE has no problem scaling the font size when set that way and one will get font/font size as compact as the native UI font (whatever its actual size is), if the keyword is recognized, otherwise one will get your other common preference of '9pt sans-serif'. The system font keyword also ensures the text is minimally readable.

Other things to take into account are the font size of other elements like the PRE, CODE, etc. which usually use other font with different default size, which size is scaled relatively to the more general font size, e.g. a common case is '16pt serif' default font and '10pt monospace' default fixed-pitch font (these look ~ of equal size because the different 'font-size-adjust' factor). So when one sets a relative font size (the same hold true for the case with system fonts) like:

body { font-size: 75% }

the default font size will be 9pt, but "monospace" elements will get 7.5pt font size which is too small as indicated by Bug 118842. So one have to adjust the corresponding elements' font size:

pre, code, tt { font-size: 100% }

One thing which improves the readability is setting a sufficient line height (leading):

body { line-height: 1.3 }   /* 1.4 is also nice */

And the last thing, given the intention is to make the main text as smallest as possible - don't use any smaller font sizes, e.g. for side navigation/panels.

Comment 45 Stanimir Stamenkov 2006-05-12 13:46:14 EDT 
The other alternative I've previously proposed in the e.t.phoenix newsgroup gives "better consistency", i.e. overrides the user preferences unconditionally to achieve the "desired" look. The main problem with this is with IE on Windows, because it disables text scaling when the font size is given with an absolute unit.

Since IE doesn't really make easier for the user to adjust the default font we could assume a default font size of 12pt for it and specify the "desired" font size using percentage relative to that, keeping the text scaling funtionality, and specify the font size with an absolute unit for every other browser (where most modern browsers I know don't have such text scaling limitation). This dual behavior is achieved by employing "CSS hacks" employing known bugs in browsers' CSS parsing:

body { font-size: 83.333%/1.4 "Arial", "Helvetica", sans-serif }
html > body { font-size: 10pt }  /* this will be skipped by IE */

This guarantees the text is the least readable. I'll note again (for other parts of the site) - there shouldn't be used any smaller font sizes. My initial suggestion was not to bother the default font at all but I've given two other suggestions as compromise, to meet some requirements stated in the course. Past that I don't have other ideas to completely satisfy any preferences.

Again, either:

1. Use a system font as base because it is user set and is at least readable;
2. Use absolute font size to set exact targeted font size using 'pt' units which are likely to scale best with different DPI settings, but use the workaround I've described for Internet Explorer to keep the text scalable with it.

The last change I've made which affects both suggestions is the use of '%' units (instead of 'em') for the font size of other elements relative to the base font size, so Internet Explorer scales the text in smaller (more acceptable) steps.
