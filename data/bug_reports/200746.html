Comment 5 Wei Yan 2007-08-22 02:45:16 EDT 
The report execution includes two steps:

1. generate the report, this include execute the query, execute the generation time script, create the report content.

2. rendering. This include execution the rendering time script, localization, convert the report content to user defined format, such as HTML or PDF.

the visiblity is associated with output format, the user may run the report to generate the report document, then render the report document to multiple formats. If an report item is in-visible in HTML but visible in PDF, if BIRT doesn't generate the item, it will fail the following PDF rendering.

Comment 6 Krishna Venkatraman 2007-08-24 14:21:19 EDT 
You are not understanding what I am saying. I would prefer it if you did not close out a bug as invalid. Let us communicate first before you decide unilaterally to close out a bug as invalid. It is rude.

I am not saying that setting visibility to hidden should not do the generation.

The situation is as follows:

Assume that there are 2 report items in a design. Item A is visible for all outputs and item B is hidden for all types of outputs. There are 2 use cases for this. U1) Design time activity - when the report developer is inside the BIRT perspective and is designing the report. U2) Runtime - when the report has been deployed and is run. Further, there are 2 kinds of outputs from a report item - N1) normal rendering of the report item and N2) abnormal error output from the report item due to some error

N1 - Normal rendering of report item:
I agree that both the items A and B should be evaluated during generation time for both U1 and U2. But during rendering the visibility property should determine that item B is hidden. Therefore no output should be rendered for item B.

N2 - Abnormal rendering of report item:
When in U1 i.e. during design time, if there is an error during generation of hidden item B let the report developer know about this error in the Problem view of the BIRT perspective. Don't throw the error up inside the regular flow of the report output. It makes the user experience unpleasant especially since the stack trace is useless to debug what is wrong. Use the Problem view. In U2, i.e. during runtime, if there is an error during generation of hidden item B, you have the choice of showing the report consumer i.e the end user the error so that they can contact the administrator to fix the error. Or, you can throw the error in a log file somewhere.

Bottom line, please talk to me first before just closing out a bug because you think it is invalid. It prevents an exchange of good ideas and it is rude.

Comment 8 Krishna Venkatraman 2007-08-28 13:52:39 EDT 
so we need do the following enhancement for error handling:

I am not suggesting that we write every error into a log file. I am suggesting that we use the problem view in BIRT to report more errors. So, if there are preview errors when inside the BIRT perspective, then why not use the problem view to report the error. The description should identify the error with a message that is easy to understand to the report developer instead of a stack trace. The location for the error could identify the name of the report item.

The above covers the error output related to a report item inside the designer. The error output should always be directed to the Problem view even if the report item is hidden. When the report is deployed i.e. errors from the report engine when running inside Tomcat for example, should be displayed to the report consumer because, they have no way of knowing that there was an error. In contrast the report designer can look at the problem view to determine if there was an error.

If the report item is hidden then it should not be rendered. Report generation should continue as usual.

Hope this helps.
