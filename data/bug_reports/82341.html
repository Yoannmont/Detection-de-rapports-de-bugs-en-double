Comment 7 Michal Stochmialek 2005-05-02 08:03:39 EDT 
I have similar problem on Linux with eclipse 3.1M6 and ajdt 1.2M3.

The cause of the problem was linux filesystem links (created using ln -s).
When workspace directory was specified using path with link,
then resource of problems wasn't shown. And following message
was printed in AJDT Event Trace view.

Processing progress message: Can't find eclipse resource for file with path
compiled: /tmp/bla/foo/FooBar.java

But when workspace directory was specified without using links,
then problem disappears.

I'm attaching simple workspace for testing. It contains
'bla' directory with workspace and 'foo' link to 'bla' dir.
When you specify bla as a workspace dir, then everything
is alright, but when foo link is specified, then problem
accours.

best regards, Michal

The fix was in ProjectProperties.findResource(String, IProject). There was a
check to see if the given String started with the path of the resource. In the
situation where there were case inconsistencies this returned that it didn't
match. The fix was to ignore case when on windows. The corresponding tests are
in ProjectPropertiesTest.java.

The remaining issue is the linux filesystem links - see comment #7.
Comment 13 Helen Beeken 2005-08-18 05:16:41 EDT 
The problem is happening on linux when there are file system links because when
we try to find the resource in ProjectProperties.findResource(String path,
IProject p), we try to find the resource by asking the given project for it's
member who has the given path. In the case where there are file system links,
the given path is the physical location, so in the example of comment #7, this
is ...../workspace/bla/foo/FooBar.java, whereas the project's location (coming
from Eclipse) is ...../workspace/foo/foo - consequently there is no match. This
is because when we call this method we use
ISourceLocation.getSourceFile().getPath() which we get from aspectj to calculate
the path. Therefore, this is an aspectj bug, that the ISourceLocations don't
understand file system links.

Comment 15 Helen Beeken 2005-09-02 08:01:23 EDT 
Bug 107313 has been closed since investigation into fixing this in the compiler
showed that the existing mechanism is probably correct for the command-line.
However, following the suggested algorithm:

* try to find the resource as normal today
* if that fails, compute (if you haven't already) a map from project directory
to canonical file name (just 
get a File for the project dir, and call getCanonicalFile ). Now see if the
source location in the message 
starts with any of these canonical prefixes, and if it does, you have your match
* if that fails, it could be sym-links within the project structure itself.
Drive "find resource/type" using 
just the unqualified file name - that will give zero or more matches (hopefully
1 or more). Get the file 
for each and canonicalize it, the one that matches is your resource...

we should be able to fix this in AJDT.
