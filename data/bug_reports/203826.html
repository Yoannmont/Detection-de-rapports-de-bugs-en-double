Description Kathy Chan 2007-09-18 12:23:28 EDT 
In the current WTP Web service wizard framework, an extender defines what are the required facet when defining a Web service runtime.  The framework tries to calculate whether the project/EAR is compatibale with the server/runtime solely based on facet.  However, some adopters found that different server requires different facets even for the same Web service runtime.  They have no way to define that with the current Web service runtime extension point.  

To generalize the problem, we are proposing an extension point in the WTP Web service creation framework to provide the extender a way to determine whether a project/EAR, Web service runtime, Server combination is supported or not.  They could provide a class in the Web service runtime extension that could be called with project, EAR, server type information and return an IStatus.  This runtime compatibility checking class would be called by the framework if the current facet checking (based on required-facet)found that the runtime is compatible with the project/EAR.  This class is then given a final vetoing option to see if it is compatible or not.  If the IStatus returns OK, then it's OK.  If IStatus returns an error, then the error string is displayed.

Comment 3 Kathy Chan 2007-11-07 16:41:25 EST 
The above patch has been committed and released to HEAD as v200711072031.

The change involved adding a new "checkerClass" attribute to the "org.eclipse.jst.ws.consumption.ui.clientRuntimes" and
"org.eclipse.jst.ws.consumption.ui.serviceRuntimes" extension points.  It takes a class that implements the org.eclipse.wst.ws.internal.wsrt.IWebServiceRuntimeChecker interface.  This interface has a method,

public IStatus checkRuntimeCompatibility(String serverTypeId,
   String projectName, String projectTypeId, String earProjectName);

Extenders to serviceRuntimes and clientRuntimes can optionally add the "checkerClass" attribute to check for the Web service runtime's compatibility with server type, project, project type and EAR.  If a IWebServiceRuntimeChecker is provided, then in ValidationUtils.checkErrorStatus() after all the existing checks have passed, IWebServiceRuntimeChecker.checkRuntimeCompatibility() is called and the status returned will be displayed to the user.  If a checker class is not provided, then the runtime compatibility check is bypassed.

An abstract class AbstractWebServiceRuntimeChecker that implements the IWebServiceRuntimeChecker interface is also provided.  It always returns Status.OK_STATUS as a default implementation.

Comment 9 Kathy Chan 2007-11-30 15:14:21 EST 
The above 2 additional patches contains the following changes (in respond to Kelvin's request in comment #5):

- Adding server instance Id to checkRuntimeCompatibility().
- Changing the attribute name in service runtime and client runtime from "checkerClass" to "runtimeChecker".
- Adding getServerRuntimeId() and setServerRuntimeId() to WebServiceInfo and WebServiceClientInfo.  If the extender wants the framework to use a particular server runtime ID when creating a server, they should set the server runtime ID in the WebServiceInfo or WebServiceClientInfo before the Install phase (i.e. in develop, assemble or deploy phase).  If the server runtime ID is not null, it will be used when creating a server.  If it is null, then the first runtime that satisfy the server type ID would be used.

Kelvin, the patch has been committed and will be released once this week's integration build is declared.  Please test with it and see if this satisfy the extender's requirement.  Thanks!
