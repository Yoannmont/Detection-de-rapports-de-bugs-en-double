Comment 10 Darin Wright 2003-10-08 15:11:23 EDT 
There is an invalid assumption in our code - we do not build if "autobuild" is 
on (it used to be that if autobuild were on, we were sure to be done building 
already, before a launch reqest came in). Thus, the (long-running) background 
autobuild is in progress whil the launch occurred in this case. We do handle 
the case of building before launching properly when background building is off -
 we submit a job that builds, and then launches.

As well, our code that builds should be smarter. Currently, it agressively 
builds the workspace, rather than joining on an existing background build (if 
autobuild is on).

Deferred to M5, as this is not critical.

Comment 14 Jared Burns 2003-10-23 12:10:18 EDT 
I think there are a few common situations for launching while a build is 
happening:
1. You're launching something unrelated to the build.
2. User error. If your attention is distracted, you might easily forget that 
you've got things going on in the background (importing, building, etc.) that 
will affect your launch.
3. (Less common) You know that the classes being built won't be loaded 
immediately by the app being launched (say, building a preference page in 
Eclipse). Or, you know that your VM supports hot code replace. So if the class 
finishes being compiled after it's been loaded, it will just be reloaded. 

Anyway, it seems friendlier to me to ask the person if they want to proceed than 
to tell them they can't.

Comment 21 John Arthorne 2003-11-03 19:07:39 EST 
An auto-build should not be directly or indirectly triggering another
auto-build.  If this is happening it would be a bug.  However, it is always
possible that some other (non auto-build) job is running that is modifying the
workspace and triggering auto-builds after you have performed your check.  For
example, if you started a CVS update, and then tried to launch, there is an
operation running that will eventually trigger an auto-build, but there is not
currently an auto-build running.  Thus your while loop would exit and then
eventually a build would run. It feels like we'd be getting too fancy trying to
solve this general case (joining any job that may modify the workspace before
launching).

Comment 40 John Arthorne 2003-11-10 18:58:19 EST 
It only seems to catch this case:

1) Some background job "foo" starts running
2) Autobuild starts running
3) User tries to launch
4) Your code detects the running autobuild and joins it
5) Before the join completes, background job "foo" changes the workspace again,
causing the current autobuild to be canceled and a new one to be started
6) Your loop detects the restarted autobuild and joins it

Comment 42 John Arthorne 2003-11-10 19:16:14 EST 
You'd have to ask Wassim.  The last I heard they were doing this:

1) Create projects in one operation
2) Allow build to run so that JDT has a chance to create .classpath files
3) Join the build from step 2.
4) Run another operation to mangle the classpaths to add appropriate PDE project
references. (Causing a second build)

So, there really should only be one autobuild still running after the import
completes.
