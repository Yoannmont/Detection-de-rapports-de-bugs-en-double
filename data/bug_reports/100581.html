Comment 3 Chris Recoskie 2005-06-17 22:30:28 EDT 
The way the GNU tool definitions are specified is with inputTypes that are not 
marked as primary.  This allows the contentTypes behaviour to take over. 
That's good.

However, the C++ content type does not have .c files mapped to it.
This, in theory, makes sense.  However, this prevents the .c files from being 
included, because the Tool reports that this extension should be filtered for 
the C++ project nature.  That's bad.

I took a look at the extension point schema and I see the following:

*******************
sourceContentType - The id of an Eclipse content type that describes this type 
of input. Either this attribute or the sources attribute must be specified by 
this element or a superclass. If both are specified, and the content type 
specified here is defined by Eclipse, the sources attribute is not used. 

sources - A comma-separated list of file extensions that identify files of 
this input type. Note that the user will not be able to modify the set of file 
extensions as they can when sourceContentType is specified. 
*******************

If we stick with this model, then we either a) can't support building .c files 
in C++ projects, or b) have to use the "sources" element (i.e. a hardcoded 
extension list) and can't use content types.

There is some opposition to giving .c files an additional C++ content type so 
that option is off the table.

I am looking for feedback on what the right approach is.

My personal opinion so far is to change the model so that the implementation 
is allowed to use the union of the set of extensions from the hardcoded list 
and the content type.  It might be a bit late in the game to be making major 
changes in the model like that though, given that API freeze was M6, and the 
extension points are a form of API.
Comment 4 Mikhail Sennikovsky 2005-06-20 09:43:01 EDT 
I suppose that the easiest way to make the gnu tool-chain be able to
build the .c files for the managed C++ projects without changing the
model and API is to define one more input type for the gnu cpp compiler
tool definition that will specify that the c++ compiler is capable of
building the .c sources, e.g.

<inputType
	sourceContentType="org.eclipse.cdt.core.cSource"
	sources="c"
	dependencyContentType="org.eclipse.cdt.core.cHeader"
	dependencyExtensions="h"
	dependencyCalculator="org.eclipse.cdt.managedbuilder.makegen.gnu.Default
GCCDependencyCalculator"
	id="cdt.managedbuild.tool.gnu.cpp.compiler.c.input">
</inputType>

But although this might be correct from the MBS point of view, this
might be incorrect for CDT/Eclipse in general, because by default the
g++ will treat the .c sources as c++, but not as c. Anyway, changing the
model in the way Cris proposed, that both contentType and hard-coded
extensions are taken into account is OK for me. I suppose we might
currently wait for Leo to come from vacations (later this week) because
he developed that code and he might know some fine points and aspects
that I'm not aware of.
Comment 5 Leo Treggiari 2005-06-22 15:32:58 EDT 
FYI: I switched the GNU tool-chain integration to specifying the 
sourceContentType at the last moment since the content type support in Eclipse 
was not complete until M7.  Unfortunately, I didn't anticipate this problem.

I don't see any obviously "best" solution to this.

We could:

1.  Remove the sourceContentType attribute from the Gnu C++ compiler 
definitions, but that would mean that a user could not modify the set of file 
extensions using the content type UI.

2.  As Chris suggests, we could use the union of the sourceContentType and 
sources file extensions, but this would mean that a user could not REMOVE an 
extension that was specified in the sources attribute through the content type 
UI.

3.  As Mikhail suggests, we could define a second InputType that specified the 
C source file content type.  I'm not sure that this will work with the curent 
implementation.  I think only input files of the "primary" input type will 
cause the generation of a rule.  A variant of this suggestion that might work 
better is to change sourceContentType to sourceContentTypes and allow a list.  
Even so, there are the other issues in the rest of CDT regarding allowing C 
source files in a C++ project.

I'll think about this some more and welcome any further comments.
Comment 6 Leo Treggiari 2005-06-22 15:41:44 EDT 
Here's another idea, but it would certainly require a non-trivial amount of 
work.  We could define "Mixed C++/C" project types.  The tool chain would 
contain both compilers.  This could solve the problem for the MBS, but doesn't 
help the problems with the rest of CDT.
Comment 7 Leo Treggiari 2005-06-23 14:22:27 EDT 
I have a different proposal for solving this problem.  

The proposal is to not change the Gnu tool chain definition, but instead to 
have the project user add *.c to the project specific C++ file types.  This is 
a new feature that was added right before M7.  The user would need to select 
the project -> Properties -> C/C++ File Types, and add *.c as a C++ file type.  
Then the C++ project will compile .c sources.  I think this is better than 
perpetuating the prior semantics where *.c files were automatically compiled in 
C++ projects.

I can investigate whether the MBS project converter can detect when an old C++ 
project contains *.c file(s) and display a message box explaining what needs to 
be done to make the project build in CDT 3.0.

I have to make some changes to get this to work properly.  The MBS code was not 
picking up project-specific content-type settings.  In order to make this work, 
I will need to add an ITool argument to some IInputType & IOutputType methods.  
These are new in 3.0, so I doubt that anyone would be using them at this time.

If you have an opinion, please reply today, so that I can make the changes, if 
everyone agrees, for RC1.

Comment 9 Mikhail Sennikovsky 2005-06-23 15:34:26 EDT 
I suppose that the old style (.c are building by default) is not intuitive 
either, because users might expect that .c files are to be compiled as c, not 
as c++ as done in the managed C++ Gnu projects, so I suppose that forcing user 
to explicitly specify that .c files are to be treated as c++ is a good 
approach. 
I also suppose that we should have a separate mixed project type since this is 
really a common practice to have mixed c/c++ projects, but I'm not sure whether 
we could implement this for 3.0. 
Also note that mixed project types will not have the same behavior as the old-
style C++ gnu managed project-types, because I expect that .c files will be 
compiled as c for the mixed gnu project type, while they were actually compiled 
as c++ for the old-style C++ gnu projects (g++ was used for compiling .c 
files). So if we decide to switch from the C++ project to the mixed project, 
the user will still need to make almost the same adjustments in order to 
make .c files compiled as c++ (as was for the CDT 2.1 C++ managed gnu projects)

Regards,
Mikhail
Comment 10 Doug Schaefer 2005-06-23 16:31:11 EDT 
Actually, we'd love to remove the whole aspect of language orientation for
projects. We generally use the content type in the core to figure out what kind
of file we are working with. The only problem we have there is with .h files
which could be C or C++ depending on who includes those files. Theoretically, we
could ask the index for this information.

The build model was originally designed to allow this. Unfortunately, Sean
didn't have enough time to fully implement it. But you should be able to have
mixed languages in a project where each language invokes it's own compiler
(tool). A common example of this is with lex/yacc files which are intermixed
with C and/or C++ files. Understanding the relationships between the different
types of files, you should be able to generate a Makefile to handle it.
Comment 11 Leo Treggiari 2005-06-23 16:49:35 EDT

> The build model was originally designed to allow this. Unfortunately, Sean
> didn't have enough time to fully implement it. But you should be able to have
> mixed languages in a project where each language invokes it's own compiler
> (tool). A common example of this is with lex/yacc files which are intermixed
> with C and/or C++ files. Understanding the relationships between the different
> types of files, you should be able to generate a Makefile to handle it.

I think make file generation can handle this sort of thing now.  What we don't 
have is a UI for using a tool-chain other than one that is defined by a project 
creation wizard.  Currently, we have project creation wizards for a "C" variant 
of the tool-chain, and one for a "C++" variant.  We don't have a wizard that 
puts both the C and C++ compilers into the tool-chain, and we have no way yet 
of adding additional tools to a tool-chain (except that Custom Build Steps do 
this today in a very limited fashion).  

So, I definitely agree with the need to support mixed languages in the future.  
Right now, I just want to find the best solution for the immediate problem of 
handling CDT 2.x C++ projects that contain C source files.
