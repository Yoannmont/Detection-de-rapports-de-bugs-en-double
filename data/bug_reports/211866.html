Description syllant 2007-12-04 06:38:16 EST 
Build ID: M20071023-1652

Steps To Reproduce:
1. Create library "mainlib.rptlibrary" with a datasource "DS"
2. Create libraries "alib.rptlibrary" and "blib.rptlibrary", include datasource "DS" into them, and create a dataset and a chart in both librairies
3. Creates a report which include both charts from "alib.rptlibrary" and "blib.rptlibrary"

Report will throw an error : org.eclipse.birt.report.model.api.command.LibraryException (code = Error.LibraryException.DUPLICATE_LIBRARY_NAMESPACE, message : The library namespace "mainlib" exists already.)
 
You can rename "mainlib" in both librairies but this tip will not work if "mainlib.rptlibrary" includes another library !

More information:
Comment 1 Rick Lu 2007-12-07 03:04:43 EST 
It is corrected that BIRT doesn't allow to include libraries with the same name space. In the above case, report indirectly uses "mainlib" twice by including "alib" and "blib". This is not allowed. So, the exception is the correct behavior.

What's the meaning of "You can rename "mainlib" in both librairies but this tip will not work if "mainlib.rptlibrary" includes another library !" ???

Em, I guess in your case, it is better to merge "alib" and "blib" into one "clib" and let report uses "clib".

Java packages are different from BIRT library hierarchies. First, java uses sub-package idea, which could quite complicated for the designer users since we don't want to introduce a library like "a.b.c.lib1". Second, we want to make sure only one reference to one specified library for one report because of "reloading library" features. If the same libraries can be included twice, it will be disaster when the user reloads the library. 

Meanwhile, to keep the library hierarchy simple can improve BIRT efficiency. It is not good to let one design include too many libraries with too many levels.

I've used comparison with packages only for their resolution strategy, not for their organization :-)

I understand there may be technical contraints, but as a end-user, I think libraraies are not as powerful as they could be.

We use Birt to manage reports composed of charts. We have defined all of our charts inside unit libraries. Eeach charts library includes a library with its datasets. Each datasets library includes a library with the datasource. This design allow us to define only once each component and freely reuse them. Birt is really great for that. The only problem is that datasource library causes conflicts as we can't alter its namespace beacause of multiple levels of inclusion, so we have to duplicate datasource inside dataset libraries.

The enh request is NOT about loading the same lib twice or sharing libs.  It is about allowing a common lib to be included mulitple times in libs as long as BIRT engnine does not load them twice.

For example, lib1 include commlib, and lib2 also include commlib, and reportA includes lib1 and lib2, the enhancement request is to allow reportA to be loaded by only loading commlib once.   If commlib is loaded when lib1 is loaded, when lib2 is loaded, BIRT engine should not try to load commlib again, instead it should use the commlib loaded when lib1 was loaded.

May have some confusion here. The library has the name space and library file. 

cases 1: the name space should be global unique. That is, the name space should be unique in a design scope, for both directly/indirectly included libraries. This may not the discussion in the comment#7.

case 2: two name spaces refer to the same library file. Example: lib1 include commlib.rptlibrary, and lib2 also include commlib.rptlibrary. This is the case comment #7 indicated, right? In the current design, one specified design file can be only included once in a design scope. This change disagrees current including library rules. To do such change, need to modify loading library mechanism, including library. etc. The new loading rule affects both Engine and UI parts.
Comment 9 Rick Lu 2008-07-01 05:19:42 EDT 
In BIRT 2.3.0 and before, for including library feature, the name space/the library file must be unique globally.  This check-in is an enhancement: Allowing a common lib ( the same file with the same name space ) to be included mulitple times and the lib is loaded only once.
