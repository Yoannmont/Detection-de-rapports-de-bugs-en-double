Description Bryan Jacobs 2007-07-24 13:36:25 EDT 
Build ID: I20070625-1500

Steps To Reproduce:
1. Create a project of a type OTHER THAN PHP.
2. Create a PHP file within that project
3. Attempt to use code assist in that PHP file


More information:
I imported a large PHP project from an existing Mercurial repository using VecTrace.  When I opened up a PHP file (with a .php5 extension) from this project, syntax highlighting functioned correctly but code assist was very strange: no documentation showed up, and it ignored what had been typed before the cursor.  As in, typing set_<CTRL-SPACE> would still have completion at the 'a's with no filtering.  Local variable names weren't included.  In other words, code assist was set to the PHP type but was broken.

This problem did not exist in PHP projects.  So, I edited .project and added the PHPNature.  The little 'P' annotation appeared on the project and IMMEDIATELY code assist started working properly.

In other words, it appears that code assist depends on the project being PHP, and not the file.  I think the correct behavior is to have PHP code completion in any file opened with the PHP editor - PHP source code is part of heterogeneous projects.  And I almost ditched PDT completely because the behavior it produced made it appear to be functioning properly but badly designed, rather than buggy.
