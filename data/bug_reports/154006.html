Description Lawrence Mandel 2006-08-15 23:38:30 EDT 
The validation framework was updated in WTP 1.5 to run validators in jobs. These jobs were displayed in the progress view allowing users to see their status and kill validation jobs. Recently (confirmed on wtp-sdk-M-1.5.1-200608151939) the validation framework stopped displaying the validator jobs in the progress view. I believe the offending change was the addition of

validatorjob.setSystem(true);

in ValidationOperation, which looks to set the root job to be a system job. As system jobs are not displayed in the progress view this prevents all validators from displaying status in the progress view.
Comment 1 Lawrence Mandel 2006-08-15 23:44:50 EDT 
Setting target to 1.5.1 as this is a regression.
Comment 2 Lawrence Mandel 2006-08-16 00:17:21 EDT 
For some further information, the change to run as a system job was set in revision 1.42 for the fix for bug 151934.
Comment 3 Chuck Bridgham 2006-08-16 09:31:16 EDT 
After receiving much feedback that the progress view is cluttered with many validation jobs, most of them short lived...  we decided to move these to a system job, and if the user needs to view them, they can change their preferences.

I don't agree with moving these back to public jobs.
Comment 4 Valentin Baciu 2006-08-16 09:45:09 EDT 
Chuck, can you please clarify on "if the user needs to view them, they can change their preferences". How does one do that?

Just my $0.02, on some very large projects some validators tend to take a long time to complete and use quite lot of CPU cycles. Because the user does not get any feedback that the validation is in progress they might launch more manual validation operations, effectively overloading the workbench.
Comment 5 Chuck Bridgham 2006-08-16 10:01:08 EDT 
Turning on the system jobs is very easy....

Progress view  -> Preferences...  -> Select the checkbox "Show sleeping and system operations"

We can doc this if needed, but for most users, these jobs should be a background event, that shows results only when problems occur....
Comment 6 Lawrence Mandel 2006-08-16 10:19:44 EDT 
Ccing Craig for comments.
Comment 7 Chuck Bridgham 2006-08-16 10:32:59 EDT 
Ok after chatting with Lawrence...   I can see a need for a "Parent" job that
shows progress (much like we used to have via the "build" job).

I can take a look how easy this is for this release... if its simple - we can do it
Comment 8 Lawrence Mandel 2006-08-16 10:34:32 EDT 
Chuck, I'm -1 for the change to system jobs. This further removes feedback users receive when manually invoking validation. At this point it appears as though nothing happens when they click "Validate". (Remember that they used to see a dialog with progress.) Also, this appears to run validation slower then before I believe (although I may be mistaken) because system jobs have a lower priority than decorated jobs.

If the many validation jobs pose a problem for the progress view I suggest creating one parent Validation job that all the others can report progress to. This validation job can stick around until all of its child validation jobs complete in order to provide feedback to the user. The child jobs can be hidden if needed as system jobs. It seems the framework is heading in a direction to remove the manual option altogether. (What good is it without any feedback?) This is a mistake. Users have requested the manual option as a way to validate resources in projects where the validation builder is disabled or when a specific validator is disabled for build.

In general I think the valdiation framework has significant visibility and we need to start exposing these changes in behaviour on the mailing list before they are made. (We made the change to show all the jobs in 1.5. We may have been able to avoid this extra change in behaviour with some requirements gathering and review before the change was made.) In this case I think we should revert the behaviour until a better solution can be found. Running all validation jobs as system jobs seems like the wrong approach to me.
Comment 9 Lawrence Mandel 2006-08-16 10:35:18 EDT 
Chuck has volunteered to investigate a better fix.
Comment 10 Chuck Bridgham 2006-08-22 09:12:25 EDT 
I'm deferring any more work here - after trying A few options - I couldn't easily setup a parent-child monitor.
Comment 11 Gary Karasiuk 2008-04-29 16:40:47 EDT 
The behavior of this has changed in 30. If you still feel that this is a problem please reopen this defect.
Comment 12 Gary Karasiuk 2008-05-07 10:07:33 EDT 
closing
