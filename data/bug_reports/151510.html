Description Missing name Mising name 2006-07-22 18:22:49 EDT 
Currently the only way to outsource tabs which are synchronized with another view in the workbench is to use detached views which are not as organizable as the main workbench window.

My request is to add a subwindow which can be moved to another monitor but still keep it synchronized with the main one like a detached window is atm.

Maybe an example:

I would like to have my java editor view on monitor #1 but the outline, package explorer and hierarchy (which all are linked to the editor) on the second monitor in a single, organizeable window instead of a bunch of detached popup windows.
Of course the layout of that sub-window should be stored too, this way developing with multiple monitors might be more efficient.

Comment 9 Eric Moffatt 2007-07-10 16:16:11 EDT 
Ben, you can't get away with -not- doing it I think. Once we introduce multiple 'panes' into the DW folks will intuitively expect it to work the same...

That being said, clients have been asking for this capability for some time (so most might be willing to live without min/max for a release...or two...;-).

One thing that I haven't had time to scope out is whether the new DW could be based on a cut-down WorkbenchWindow rather than an augmented DetachedWindow (i.e. no Editor Area, CoolBar, PerspectiveSwitcher or basic trim like the status line). It would have the advantage of handling zoom automatically and it -might- be possible to do it with a 'custom' perspective type. I don't know how invested you are in the current code but it might be worth a look...
Comment 10 Missing name Mising name 2007-07-10 20:32:56 EDT 
From a user perspective it's pretty simple: The window should basically add screen real estate to the main workbench window and provide all functionality that can be moved within the main window: this is view-tabs, toolbars and editor windows.
The perspective switcher, the menus and the status bar can't be moved and thus don't have to be moved outside the main window either.
It should be maximizable of course (fitting/moving windows that should gobble up as much space as possible is a PITA), maybe even fullscreen-able since all necessary controls are on the main window itself.

Yes, i know i'm asking for much here, but i'm just describing the ideal situation ;)

Comment 12 Eric Moffatt 2007-07-16 11:33:35 EDT 
Ben, watch out...you may be over-comitting here. Allowing Toolbars (and, presumably a 'main' menu) is gonna get you a true can of worms. Our command management is one of the most complex areas of the workbench and extending it our to 'satellite' windows will -not- be trivial...some questions:

Which main ToolBars / Menus should be shown?

How do we manage the general command context when it's shared across multiple shells? (see Bug 195672; we can't even manage it now)?

If you add Editors into the picture then I think we've come full circle...this is a new workbench window; not a 'satellite'. What wouldn't be allowed in the satellite windows?
