Description Jerome Louvel 2006-11-02 10:55:26 EST 
As the WST Server API doesn't imply that a Module is necessarily a Project, it would be better to adjust the messages like the "Add and remove projects..." menu entry to say "Add and remove modules..." instead.

In our case, Module have a much finer grain than projects and our users will be confused. The strings to modify are in:

/org.eclipse.wst.server.ui/serverui/org/eclipse/wst/server/ui/internal/Messages.properties

Comment 9 Rob Stryker 2008-10-15 16:04:30 EDT 
JBoss can accept single files, such as XML files, as deployments, and in our tool set we have the ability to publish a single file. This single file is turned into a "module" using the module adapter and ModuleFactoryDelegate APIs.

So when a user wants to re-publish "mydatasource.xml" he has to go to "add / remove projects" and then select "ProjName/src/folder/mydatasource.xml" as a "project" name.

I suppose none of the terms (module, application, project) work for all cases and the term should be something more like Deployment?
Comment 10 Chris Brealey 2008-10-16 08:57:45 EDT 
I agree the term "projects" needs to give way to something a bit more open ended. Granted users have projects in their workspace, but these are organizational entities in the workspace, and - depending on the progamming model, deployment mechanism and server in question - may or may not be deployable. A couple other options to throw on the table are (1) "resources" (or "deployable resources" to build on Rob's last comment) simply for the reason that projects, folders and files all boil down to being plain ol' resources in the Eclipse workspace, or something short and sweet like (2) "assets". Just me two cents' worth.
Comment 11 Angel Vera 2008-10-16 09:12:39 EDT 
This morning I was thinking of 'Add and Remove Resources' or 'Add and Remove Objects'  or even 'Add and Remove Components' would apply.

The best term I think is 'deployable resources' (as per Chris suggestions), but it is too long to use in the context menu (ie; 'Add and Remove deployable resources', so my suggestions would be to use 'resource'(s) as a short form and 'deployable resources' in the long form, if there is no objection I can suggest this changes for 3.0.3.
Comment 12 Elson Yuen 2008-10-16 10:18:19 EDT 
A single word "Resource" in Eclipse references to all the resources that you can see within the workspace. If we simply use "Add and remove resources" to add any resource within their workspace to the server.  Most of the server have their own list of resource types that can be added to the server instead of an arbitrary file/project within the workspace.  Therefore, it may confuse the user if we simply use resources.

I like "deployable resources" more to clarify that those are the blessed resources that can be added to the server. One the other hand, Add and remove deployable resources may be tool long as you have suggested. In the existing GUI, the menu item is probably the only place that may have that concern.  One possible approach is to use "Add and remove resources..." on the menu item.  Once the wizard is opened, we will use "deployable resources" within the wizard to clarify what that dialog is supposed to be doing.

Comment 14 Rob Stryker 2008-10-16 17:23:44 EDT 
Most of the APIs use the term "module"... module artifact adapter, ModuleFactoryDelegate,  IModule...

Comment 15 Angel Vera 2008-10-17 10:26:37 EDT 
Max suggestions splits the current single action in two, which we want to keep as one action. 

The deploy/undeploy may work but is not as clear as add/remove, it may imply that deployable actions (ie: PublishTasks & Publishers) will run, which it doesn't occur until the actual publish action is called.

Rob, the concern with modules is that WTP users may not understand what a module is, where does it come from. By using 'resources' it generalizes and allows space to include projects.
Comment 16 Tim deBoer 2008-10-17 10:38:35 EDT 
'module' is used internally since we had to pick a term for the API, but it is currently (and carefully) not exposed to users.

FWIW, one of the problems (with just about any term) is that most terms are overloaded. For instance 'module' is used for Web & EJB modules in Java EE, but since EARs are not modules it is potentially confusing.

Comment 19 Angel Vera 2008-10-20 16:34:32 EDT 
    * Explain why you believe this is a stop-ship defect. Or, if it is a "hotbug" (requested by an adopter) please document it as such. 

There has been growing interest from several adopters to get the UI fixed in the 3.0.3 stream and 3.1, thus I am submitting for PMC approval for the 3.0.3 stream.

This is a long standing UI bug. The term "Project" in the UI (ie: "Add and Remove Projects...") limits the interpretation of what the API allows. Comment #0, Comment #6, Comment #9, Comment #10 are all examples of adopters who have created modules that do not necessarily match to a project.

The API doesn't limit (and shouldn't) a 1-1 mapping between Modules and Projects. Thus, the proposal is to fix several messages (ie: wizard titles, wizard descriptions, context menu items and error messages) to remove the implied 1-1 mapping between Modules(Server Installable Units) and Projects.

Comment 25 Tim deBoer 2008-10-22 10:47:02 EDT 
Hi guys,
I'll stick my neck out and admit it was me as project lead that decided this change cannot be done in the 3.0.x maintenance stream.

As Angel pointed out, making a nomenclature change like this affects *everything*: documentation, translations, screen caps, tutorials, every adopter, etc. While this change can and should be made (and is long overdue), there is absolutely no way that we could verify that every stakeholder agrees with this change and would be willing to react in a maintenance release. It's not stop-ship (we've gone through 5 releases this way), and we can't risk breaking other adopters and having an even bigger backlash when they disagree with the change or would have preferred another term.

My request to Angel has been to send a note to wtp-dev to ensure that all stakeholders see this and have a chance to voice their opinion on the new term. Provided there are no major objections to the current patch (or an even better term is proposed), we should release as soon as possible to 3.1 to give ample time for everyone to react.
