Comment 1 Michael Valenta 2003-05-06 09:09:41 EDT 
The problem is that an external editor (such as a web browser) needs a local 
file to open. Either we try to fake this by saving the remote file to a 
temporary local file or we require the user to check out the resources. The 
problem with faking it is that we have no way of knowing what other files may 
be referenced from the original file (this is especially a problem for html 
files).

Comment 15 Markus Keller 2006-05-30 04:44:00 EDT 
Now in 3.2, an alternative could be to implement a CVS file system provider and plug it into the Eclipse. This way, references could be fetched transparently from the repository.
Comment 16 Michael Valenta 2006-05-30 08:07:55 EDT 
Markus, we do have a CVS EFS implementation. However, it doesn't help in this case since external tools know nothing about EFS but instead are reliant on the local file system.
Comment 17 Markus Keller 2006-05-30 08:37:35 EDT 
I see. So, this seems to be a general problem of EFS implementations then, and Core/Resources should solve it for you and other EFS implementors (either by allowing to plug an EFS into the host OS's file system, or by saving files to open to a temporary location in the local file system).

I'm hesitating to file the bug for them by myself, since I don't know much about EFS mechanics.
Comment 18 Michael Valenta 2006-05-30 09:21:43 EDT 
I've copied John in case he has any thoughts on the last 3 comments. Personnaly, I don't think the two things are related. That is, I don't think it should be the reponsibility of the EFS to have a local on-disk cache that can be read by external tools. However, I can easily see that a higher level API (or an EFS wrapper) could be designed to provide such an on-disk cache.
