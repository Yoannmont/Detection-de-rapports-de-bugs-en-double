Description Mark Proctor 2005-11-16 21:46:51 EST 
I was looking to see if SWT Table was suitable for basic Worksheet
functionality, just displaying a stream of data. I found the rendering when
scrolling on win32 to be unnacceptable compared to a similar application with
JTable. I'm currently investigating KTable.

import org.eclipse.swt.*;
import org.eclipse.swt.widgets.*;
import org.eclipse.swt.layout.*;

public class TestWorksheet
{
    public static void main (String [] args) {
        Display display = new Display ();
        Shell shell = new Shell (display);
        shell.setLayout (new FillLayout ());
        Table table = new Table (shell, SWT.NONE);
        table.setLinesVisible (true);
        table.setHeaderVisible (true);
        for (int i = 0; i < 50; i++) {
            TableColumn col = new TableColumn (table, SWT.NONE);
            col.setWidth (80);
        }
        for (int i = 0; i < 1000; i++) {
            TableItem item = new TableItem (table, SWT.NONE);
            for (int j = 0; j < 50; j++) {
                item.setText (j, "Cell " + i + " " + j);
            }
        }
        shell.open ();
        while (!shell.isDisposed ()) {
            if (!display.readAndDispatch ()) display.sleep ();
        }
    }
}

On WindowsXP when I scroll horizontally sections of the screen visibly flash
white as they redraw - far too slowly. Vertically scrolling is better but if I
scroll fast enough it suffers from the same problem. KTable, in the provided
demos, seemed to fair much better.

Comment 11 Ed Burnette 2005-11-21 13:20:36 EST 
The 50 column one is slow for me too. With 5 columns I couldn't tell.

I turned on the profiler (YourKit) and looked at it very quickly. There are two phases, creating the table, and everything else.

In the creating the table phase, the hotspot is Table.getColumnCount() (57% of cpu), and TableItem.<init>(Table, int) (35% of cpu).

After the table was created I noticed it was very sluggish to scroll, refresh, or even to just switch to it. So I turned on a little more detail in the profiler, switched to the app, waited for it to redraw itself, and switched back to the profiler, and captured the results. It took 1.5 seconds to do that. The traceback was pretty deep, showing that this simple operation turned into 10,831 calls through Display.windowProc() with several trips back and forth across the JNI boundary. I'll attach the results, maybe it will make sense to you.
