Description Naomi Miyamoto 2004-03-29 02:19:21 EST 
Driver : M8 GM

When I select a folder witch includes over 700 plug-ins, only 92 plug-ins are 
listed in Target Platform Preferences page.

Same symptom can be seen in Import External Plug-ins and Fragments Wizard.

[Operation to Reproduce]
1. Invoke Import External Plug-ins and Fragments Wizard
2. Unselect The target platform check box
3. Press Browse and select a folder
4. Select Select from all plug-ins ... radio button
5. Press Next
   ==> Over 700 plug-ins are listed
6. Press Cancel to close Wizard
7. Invoke Preferences Dialog and show Target Platform page
8. Press Browse and select same folder as step 3
   ==> Only 92 folders are listed
9. Press OK to close Dialog
10. Invoke Import External Plug-ins and Fragments Wizard again
11. Select The target platform check box
12. Press Next
   ==> Only 92 folders are listed
Comment 1 Wassim Melhem 2004-03-29 08:42:45 EST 
Correct.  When we list the plug-ins on the target platform page, we pass the 
list of plug-ins (over 700) through a resolution filter supplied by Core, 
which throws away all the plug-ins that have unsatisfied constraints.  In this 
case, only 92 were deemed as good plug-ins.

When importing from an arbitrary location, we just show all plug-ins (both 
resolved and unresolved). 

What is your target platform?  Either something is wrong with it, or the Core 
resolver is too strict.  Moving to Equinox to continue the discussion...
Comment 2 Rafael Chaves 2004-03-29 11:05:24 EST 
One problem here is that we have no feedback on what failed during resolution.

Wassim, one thing that the old resolver used to do and the state/resolver does
not is logging unresolved bundles. That is now left to clients. We provide API
that may be used by clients for that purpose:

BundleDescription#isResolved
StateHelper#getUnsatisfiedConstraints

or you can query the state delta returned by State#resolve.

Comment 13 Rafael Chaves 2004-03-30 13:35:21 EST 
That is it, Wassim, I talked to Jason and he said that there is no MANIFEST.MF
for org.eclipse.osgi. That is true for builds before around March 10th. 

This explains why we were failing: since we don't find a manifest for that
plugin, we are converting the plugin manifest for that plugin to a bundle
manifest (but the plugin converter does not generate exported packages, which
are required by org.eclipse.osgi.util and org.eclipse.osgi.services).

This reliance on bundle manifests/plugin converter is in place since 3.0 M8, and
this is why this was not an issue before. Unfortunately, the outcome is that you
cannot use 3.0 M8 to target M6/M7. I am looking for an workaround right now...

Comment 19 Rafael Chaves 2004-03-31 12:02:31 EST 
It seems that doing the following changes to the bundle manifests in Eclipse
allows one to target a pre-M8 build (tested with M7):
- for each plugin containing a bundle manifest (a META-INF/MANIFEST.MF file),
add a "Bundle-SymbolicName" entry having the same value as the existing
"Bundle-GlobalName" entry;
- for org.eclipse.osgi, create a META-INF/MANIFEST.MF file containing the
contents of core.jar/META-INF/SYSTEMBUNDLE.MF plus the values defined in
eclipseAdaptor.jar/META-INF/ADAPTOR.MF - the redundant entries must be merged.

If you want to try it, tell us what exact build you are targetting and we may
provide the new manifests for you. Or you can apply the changes yourselves. But
beware this has not been thoroughly tested and is not a supported configuration.
Comment 20 Rafael Chaves 2004-03-31 12:10:16 EST 
BTW: the special case for org.eclipse.osgi only holds if a bundle manifest
META-INF/MANIFEST.MF is not found. Otherwise, the only necessary change is the
addition of a Bundle-SymbolicName entry.
