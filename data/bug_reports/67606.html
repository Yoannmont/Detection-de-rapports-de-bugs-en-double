Description Frederic Fusier 2004-06-17 05:59:25 EDT 
Using I200406162000.

This is a behavior I noticed while fixing bug 66898. For this bug, problem was
located in method processCompilationUnitResource(ICompilationUnit,
IPackageFragment) of org.eclipse.jdt.internal.core.CopyResourceElementsOperation.
In this method we copy (or move) a resource using copy (or move) API method of
IResource and rewrite the content of the target resource with modified string
got from source.
Of course we need to know the encoding of the file to be able to write
destination file contents accurately...
Initially, we got encoding from destination file (destFile), but it appears that
 after the IResourec copy (or move) call, the resource encoding was not
preserved (we got the container encoding instead)...

I made some tests in Resources perspective, while copying (ot moving) a file and
I saw that when an encoding was specified on a file, it was well preserved
during copy and move actions...
I debugged to understand what happens and I saw that encoding refresh was done
by CharsetManager$Listener during endOperation of Workspace runnable...
Unfortunately, our copy (or move) is done during executeOperation (ie. before)
and so explain while the encoding was not valid for destination file


So, currently (see patch or fix or CopyResourceElementsOperation), we have put a
workaround which consists to store the encoding of the source before the copy
(or move) and use this stored encoding to write contents of destination.

However, this solution is not really satisfactory as:
1) the destination file encoding might be modified concurrently and then we
finally write contents with wrong encoding,
2) the source file encoding might be not really specific but inherited from its
hierarchy and then we'll force a specific encoding although user wanted that
destination file inherits its encoding from destination container...

If you think that current platform behavior is really a problem and can be fixed
for 3.0, then we could remove our fix for bug 66898 and problem 1) above will no
longer happen :)
Otherwise, I think we need to open a bug against JDT/Core to fix problem 1) by
setting *explicitely* destination file encoding with stored one to avoid
possible inconsistencies...

Hope, this not too long and enough clear... ;)

Comment 12 Markus Keller 2010-08-13 14:05:42 EDT 
I20100810-0800, and has probably always been like this:
Encodings are actually never set on the target of an IResource#copy(..) operation. CharsetManager.Listener#processEntryChanges(..) only copies properties in move operations. See also bug 54048.

JDT/Core tried to work around this (bug 67823), but the workaround only works for some cases (only when JDT/Core actually changes the contents of a compilation unit).
Comment 13 Markus Keller 2010-08-13 14:09:24 EDT 
Also note that the encoding is also only correctly moved if it is specified on the moved file directly. The encoding is still wrong after a move if the encoding is inherited from a container.
Comment 14 Szymon Brandys 2010-08-31 10:57:21 EDT 
For files that inherit their encoding from a container, we could set the encoding directly just before they are copied/moved. It would work only for the move operation though.
