Description Robert Elliot 2007-04-24 06:54:39 EDT 
If a Dynamic Web Project has a child Java project module, and that Java project module represents a tag library with a valid tld in its META-INF directory, WTP fails to resolve the tablib declaration in any JSP, resulting in a validation error "Cannot find the tag library descriptor for http://foo.com/testtags".

I've attached an example workspace (Eclipse 3.3 M6, WTP2.0M6).  It contains a dynamic web module test-war with a jsp test.jsp that uses a tag library with uri http://foo.com/testtags.  It has a child module test-jar which has a taglib. ltd in its META-INF path.  Publishing via WTP results in a fully functioning app (test.jsp compiles and displays the tag output).  However, validation of the JSP in the workspace results in the error described above.

I would expect WTP to be able to find taglibs in the META-INF folders of jars on its classpath by URI.

Comment 2 Robert Elliot 2007-04-25 03:25:08 EDT 
Further testing shows that WTP only finds a tld if it is in the META-INF folder of a jar that is actually in the WEB-INF/lib folder of the project.  If you add the same jar to the J2EE Module Dependencies  but do not actually place it in the WEB-INF/lib folder of the project then you get the same "Cannot find the tag library descriptor" error, though if you then publish using WTP the jar is correctly copied into the WEB-INF/lib folder on the server and the JSP compiles and runs correctly.

Looks like this is a duplicate of Bug 164580.
Comment 3 Robert Elliot 2007-04-25 03:45:21 EDT 
On a suggestion in bug 183756, tried closing and re-opening the project.

With the jar packaged up and included as a J2EE Module Dependency, but not in the WEB-INF/lib folder of the project, this now works fine; validation passes and content assist works.  However, closing and opening projects seems a bit of a brute force solution; is WTP failing to send a classpath change notification under some circumstances when it should?

However, the initial problem as described in the description and in comment #1 remains - a client java project containing a tld is not picked up in the web application, even after closing and reopening all projects, resulting in the validation error in the JSP.

Comment 10 Nitin Dahyabhai 2007-08-08 21:01:43 EDT 
This hotbug must be fixed to correct a regression from previous IBM products.  The only workaround is to manually insert utility projects as jar files within the dynamic web project, a step that's redundant with and could interfere with the deployment support for utility projects.  

The JSP ProjectDescription class holds the information about tag libraries for a single project.  One of its jobs is keeping track of loose .tld files so they can be loaded by the JSP models according to the URI values defined within them.  The fix simply adds those records from projects on the build path to those searched by the web project.  The looping through those projects was already in place for other reasons, so this is a pretty safe change.
