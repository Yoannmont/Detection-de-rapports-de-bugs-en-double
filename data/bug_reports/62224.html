Description Mike Wilson 2004-05-14 04:50:16 EDT 
Note: This bug report outlines problems similar to, but not described in bug 
report 39459, although this report also concerns x-mouse.

Summary
-------

There are several unwanted situations where Eclipse 3.0MS8 in combination with 
the tweakui X-Mouse seeting will raise its main window to the top of the z-
order, effectively making all other windows disappear behind it.

Detailed description
--------------------

How to activate "x-mouse":
- download and install Tweakui from Microsoft's Windows XP PowerToys homepage
- start Programs->PowerToys for Windows XP->Tweak UI
- in the new window open section Mouse->X-Mouse
- make selections:
  . [X] Activation follows mouse
  . [ ] Autoraising when activating
  (autoraising not selected!)
Now you have activated the activation style often used on X-Windows (Unix) 
systems. It will give the input focus to the window containing the mouse 
pointer without having to click the window first. One of the main advantages 
is also that you can have the input focus on a window that is not on the top 
of the z-order (which is difficult to achieve in standard Windows focus style 
as clicking a window will both raise it to top of z-order AND assign input 
focus).

Problem 1: Dialog box will raise main window
Reproducing the bug:
- activate X-Mouse as described above
- start Eclipse 3.0MS8 and make it full screen
- open the dialog box Window->Preferences in Eclipse
- start another application, f ex Windows Explorer, and make its window 
stay "on top" of the Eclipse window by using Alt-Tab to activate this other 
application (Windows will still do a combined activation and raise when using 
Alt-Tab).
- now when you move the mouse into Eclipse, its main window will be raised to 
top-level (this also happened without open dialogs up to 3.0MS7)
Analysis:
This is probably an old-style way of coding the dialogs that was very common 
some years ago on Windows, where you when receiving a SetFocus message always 
did a RaiseToTopLevel. For some time a also think this was the standard 
behaviour in Microsoft's MFC library but is not anymore.
Anyway, this behaviour is undesired with X-Mouse and today I think this 
behaviour also violates Windows design guidelines, in that applications should 
not steal focus or popup on top of other applications, but I'm not up-to-date 
with the exact phrasing...

Problem 2: Moving mouse into tooltips will raise main window
- activate X-Mouse as described above
- start Eclipse 3.0MS8 and make it full screen
- start another application, f ex Windows Explorer, and make its window 
stay "on top" of the Eclipse window by using Alt-Tab to activate this other 
application
- move the mouse over the Eclipse toolbar so a tooltip is displayed (Eclipse 
main window is still below the other application)
- now quickly move the mouse over the tooltip so you "touch" the tooltip 
before it disappears, and the Eclipse main window will be raised to top-level
Analysis:
Probably a similar thing here; when the tooltip window gains focus (caused by 
the mouse inside it) it will raise its parent window to top-level.

Problem 3: Mouse inside content assist "documentation" will raise main window 
and close content assist
- activate X-Mouse as described above
- start Eclipse 3.0MS8
- find a piece of source code where you can popup content assist with 
documentation, for example
    String str;
    str.
        ^ do content assist here
- when the content assist displays methods of class String it will also after 
a short while display the JavaDoc of the selected method
- start another application, f ex Windows Explorer, and make its window 
stay "on top" of the Eclipse window by using Alt-Tab to activate this other 
application
- make sure the other application is positioned so it doesn't cover the place 
where you want to work with content assist
- position the text cursor back on the end of "str." as before
- move the mouse into the area where you know the JavaDoc will be displayed
- open content assist again, it will open up but will close down as soon it 
tries to display the JavaDoc documentation and at the same time raise the main 
window
Analysis:
Probably similar coding here; the content assist JavaDoc window will gain 
focus when popped up under the mouse, and will then activate its parent 
window. It will thereby give away focus from itself and will then decide to 
close as this is often the behaviour for these kinds of hover windows.

Comment 5 Mike Wilson 2004-06-21 09:58:19 EDT 
I did a little bit of searching in the code myself, and I think I found the 
piece of code responsible of raising the main window when a dialog box is 
open. It is located in Display.java (see excerpt below).
From the explanation in the code I understand the raising of the main window 
is a solution to another bug (possibly inherent in Windows). One should be 
very restrictive with raising and popping up top level windows as, as we can 
see here, it is very easy to create other bugs... ;-)

Cheers Mike

case OS.WM_ACTIVATEAPP:
	/*
	* Feature in Windows.  When multiple shells are
	* disabled and one of the shells has an enabled
	* dialog child and the user selects a disabled
	* shell that does not have the enabled dialog
	* child using the Task bar, Windows brings the
	* disabled shell to the front.  As soon as the
	* user clicks on the disabled shell, the enabled
	* dialog child comes to the front.  This behavior
	* is unspecified and seems strange.  Normally, a
	* disabled shell is frozen on the screen and the
	* user cannot change the z-order by clicking with
	* the mouse.  The fix is to look for WM_ACTIVATEAPP
	* and force the enabled dialog child to the front.
	* This is typically what the user is expecting.
	* 
	* NOTE: If the modal shell is disabled for any reason,
	* it should not be brought to the front.
	*/
	if (wParam != 0) {
		if (modalDialogShell != null && modalDialogShell.isDisposed 
()) modalDialogShell = null;
		Shell modal = modalDialogShell != null ? modalDialogShell : 
getModalShell ();
		if (modal != null) {
			int hwndModal = modal.handle;
			if (OS.IsWindowEnabled (hwndModal)) {
				modal.bringToTop ();
			}
			int hwndPopup = OS.GetLastActivePopup (hwndModal);
			if (hwndPopup != 0 && hwndPopup != modal.handle) {
				if (getControl (hwndPopup) == null) {
					if (OS.IsWindowEnabled (hwndPopup)) {
						OS.SetActiveWindow (hwndPopup);
					}
				}
			}
		}
	}

The three lines marked with "//!" should be removed.


What did they want to achieve?
------------------------------

The story of this code is described in the long comment. As you can see it is 
a fix for unwanted behaviour on the way MS Windows handles disabled top-level 
windows.

You can set up the described scenario this way:
- Open Eclipse.
- Open an additional main window through the menu; Window->New Window.
- Open the about box (Help->About) from one of the main windows.
You now have one application with two disabled top-level shells, both being 
disabled by the same modal dialog box. The dialog box is only a child of one 
of the windows (the one we issued the Help->About command from).

You can now see the following behaviour:
- Click in main window owning the popup will raise popup and 
  its parent main window.
- Click in other main window will raise popup and its parent 
  main window.
- Click on task bar icon for main window owning the popup will 
  raise popup and its parent main window.
- Click on task bar icon for other main window will raise 
  popup and its parent main window.

It is the last case that the marked code handles. MS Windows default behaviour 
is to raise the selected disabled top-level window, although this means going 
on top over all other windows including the modal dialog causing its disabled 
state (probably because the modal dialog is owned by another top-level window 
so the code can't enforce the child-in-front-of-parent rule). 
As the author mentions, clicking in the just raised top-level window will 
automatically raise the dialog box and its parent top-level window, so it 
seems MS Windows used a different algorithm for that part of the code.


Problem analysis and consequences of suggested change
-----------------------------------------------------

The three marked code lines are responsible for raising a modal dialog box to 
the top of the z-order (in front of all other non-topmost windows) when its 
application gets activated, without regard to how the application gets 
activated. Raising the dialog box also raises the owning parent window.

The author has probably made the assumption that activation of an application 
always leads to its main window(s) being raised anyway, so making an explicit 
raise of a modal dialog box belonging to the application will not harm 
anything. Unfortunately, this is not true when X-Mouse is activated.

The application can get activated in (at least) the following ways:
1. Clicking in one of our windows.
2. Clicking on one of our top-level window task bar icons.
3. Switching focus with Alt-Tab.
4. Closing other applications until our application is the 
   most recently used.
5. Moving mouse into one of our windows with X-Mouse active.

For the first four cases MS Windows default behaviour is to raise the 
application's main window at the same time as activating it. (Sometimes there 
are exceptions, f ex Windows Explorer doesn't raise when clicking an item in 
its right pane, this is to make drag-and-drop easier.)

For the last case the default behaviour is to let every window stay in its 
current Z-order and just switch activation state to the application window 
containing the mouse (assuming X-Mouse Auto-Raise isn't activated).

As the marked code doesn't distinguish the way the application got activated, 
it will raise the dialog (and main window) for all types (1-5) of activations, 
including mouse-entries with X-Mouse. But it is actually only case (2) that 
doesn't already behave in the desired way by default.

So, having the current fix in the SWT toolkit is probably nice but far from 
essential. It's a nice bonus of using the toolkit and I would have recommended 
it myself if it hadn't messed things up in other places. And the fix just 
corrects one of many ways to activate an application.
My suggestion is to remove the "raise" parts of the fix (corresponding to the 
marked lines).

When doing this, it is important to note that we will not risk users 
accidentally believing their application being hung when seeing the disabled 
main window and no modal dialog, as the modal dialog pops up in front as soon 
as the user clicks in the disabled window.


Possible improvements
---------------------

It may be possible to keep this "click task bar icon fix" if it's improved to 
be better compatible with X-Mouse. I can think of two paths to elaborate on:

1. Detect the different activation methods
   ---------------------------------------
   As it is only activation through the task bar that needs 
   fixing, it would be appropriate to only trigger the fix 
   for this case. Unfortunately the WM_ACTIVATEAPP, 
   WM_ACTIVATE and WM_MOUSEACTIVATE messages don't seem to 
   give us enough information on how the activation was done, 
   so I guess hooking into the Explorer task bar would be 
   needed. The question is if there is a solution for that 
   which doesn't become a Windows version / Explorer Shell 
   version nightmare.

2. Only raise to enough Z-order to be top of our own windows
   ---------------------------------------------------------
   Using Win32 function SetWindowPos() a window can be raised 
   to an arbitrary Z-order which doesn't have to be top of 
   all non-topmost windows. This way the fix code could ensure 
   that the modal dialog is always displayed above the top-level 
   windows it is disabling, but not raising onto top of all 
   windows. As most of the activation methods raise the top-
   level windows to the top of Z-order, then raising the dialog 
   just above them would put it on top of all windows just like 
   today. But the X-Mouse activation would work much better as
   the main windows and dialog would stay on the same level and
   just potentially switch places with each other.
   The challenge here is to find an application's top-level 
   windows in an efficient way. Maybe Win32 EnumWindows() can be 
   used, if SWT doesn't already have the information internally?

Comment 25 Mike Wilson 2005-05-11 11:05:38 EDT 
Ok, first a reply to comment #23:

Well, I made the patch as non-intrusive as possible and only made the bringToTop
() call conditional as this is what messes things up for X-Mouse.

My thoughts, after looking at the remainder of the code (GetLastActivePopup() 
etc) over and over again, are that it seems like this part of the code has not 
much to do with what is described in the preceeding comment. Without an 
explanation it is not so easy to know what problem the author wanted to solve, 
but my guess from looking at the code is that it tries to restore focus to non-
modal popup windows that were previously focused.

Consider the following theoretical case (which I haven't tried):
- we pop up a modal dialog from our main window
- from the modal dialog we pop up some non-SWT non-modal dialog box 
  (with non-SWT I mean we have no corresponding SWT object)
- this non-SWT dialog has focus when we switch to another app
- clicking our main window triggers the code in  WM_ACTIVATEAPP 
  and brings our own modal dialog to the top and also focuses it 
  through our call to bringToTop()
- we would like to restore the exact focus setup from before 
  but the non-SWT dialog will not be activated as it is not 
  known by SWT, and will not automatically override focus from 
  its parent as it is not modal, therefore we need to programatically 
  find and activate it
- this is done by asking for the last popup that was active from 
  our own modal window
- the check hwndPopup != modal.handle stops us from activating our 
  already activated modal dialog
- the check getControl (hwndPopup) == null stops us from activating 
  any control registered in SWT

So, it is the last check that makes me believe this is a fix for common dialogs 
and similars. But as I said, without a better explanation of what the purpose 
has been it is hard to know exactly what behaviour was being corrected here.

Another thing is, I think the code very seldom reaches the call to 
SetActiveWindow(), as I think the author wasn't aware of the behaviour of 
GetLastActivePopup(). In short, this code will only trigger when hwndModal is a 
top-level window. And modal top-level windows are not so common, right?

When, like in my example, the modal window being processed is a dialog being a 
child of another window, the call to GetLastActivePopup() will always return 
the same handle as has been passed in. So 
  int hwndPopup = OS.GetLastActivePopup (hwndModal);
will result in hwndPopup == hwndModal, and the test on the next line
  hwndPopup != modal.handle
will always return false.

So, in short, as I couldn't really make any sense of this I tried not to touch 
this part of the code. It is  quite possible that I have been missing something 
and that the mentioned code is really doing something useful.

Regarding comment #24:

Hm, can you tell me what your environment is? I'm getting the "dialog raise" 
problem all the time with my 3.1M6 installation.

This is what you should do and expect:
- Install Tweakui (for XP if on XP).
- In Tweakui switch on Activation follows mouse (X-Mouse).
--> If you have a couple of non-maximized application windows
    on your desktop you should now see that they are activated/
    focused merely by moving the mouse pointer into the
    respective window.
- Now start Eclipse in a non-maximized window and open the
  Help->About dialog box.
- Move one of the other application windows so it partially
  obscures Eclipse.
