Description Patrice Kerremans 2003-01-02 11:12:15 EST 
often it happens that you want to change a word that appears throughout the 
code (or documentation) in different caps, an example:

MyWord word = null;
MyWord currentWord = null;

when performing a replace now, say, "word" by "phrase", you get:

Myphrase phrase = null;
Myphrase currentphrase = null;

This is definitely not what you'd like it to become...
When using case sensitive replacement, the second 'word' does not get replaced, 
resulting in the following piece of code:

MyPhrase word = null;
MyPhrase currentPhrase = null;

I'd like to see a 'retain-caps-while-replacing option' that cause that replaces 
the first character's caps of the replacing word if required; if the first 
letter of the substring to be replaced is a capital, then make the first 
character of the replacing string a capital, if it is a smallcap, make it a 
smallcap.  This results in the desired piece of code:

MyPhrase phrase = null;
MyPhrase currentPhrase = null;

remark: the second 'word' has been replaced by 'phrase' as we hoped it to be.

thanks!

Hi!

I would like to state this requirement more precisely (as I already did in the bug 52612, that is marked as duplicate of this):

Most tools are able to SEARCH case sensitive,
but nearly everybody needs to REPLACE case sensitive, too!

This is not as trivial as it sounds, as Martin Aeschlimann correctly notes for my example: "That looks a bit like magic."
But in opposite to the example above, my following example shows the difficulty, that has to be clear and managed!

E.g. i want to replace PDF by Document:

interface WorkflowCase {
  String pdfName;
  byte[] pdf;

  String getPDFName();
  void setPDFName(String pdfName);

  byte[] getPDF();
  void setPDF(byte[] pdf);  
}

I want to search for PDF case INsensitive, so I find: 
pdfName, pdf, getPDFName, setPDFName, ...
and I want to REPLACE it case sensitive with document:
documentName, document, getDocumentName, setDocumentName, ...

I believe, all others and me meant:

******************************************************
*** Replace (only first character) case sensitive! ***
******************************************************

example: pdf -> document, PDF -> Document

This seems more precisly, what is meant, as you can see in all other bugs, that are marked as duplicates of this. Perhaps, it helps, looking at the algorithm of other tools, that are mentioned in the other bugs, e.g. emacs and Visual SlickEdit.


Extension (2nd feature):

Sometimes I find "constants" like PDF_EXTENSION and
I want to replace it with document but all upper case,
so I need a button replace all uppercase or all lowercase (vice versa).

E.g. add new button in find dialog that does not replace as usual,
but replaces as constant/all capital letters.

So it is not magic, I only want to control the process manually myself 
with different buttons on the find dialog.

best regards

Thorsten van Ellen

P.S. want to search only with F3 or similar (without dialog), too. 
(3rd Feature request!)

Comment 12 Sam Mesh 2007-01-28 03:21:04 EST 
I think that it could be sufficient to get quite trivial Emacs definition from "Replace Commands and Case" (link and quote are below) and do quite trivial generalization for CamelCase.
***
http://www.gnu.org/software/emacs/manual/html_node/Replacement-and-Case.html

11.7.3 Replace Commands and Case

If the first argument of a replace command is all lower case, the command ignores case while searching for occurrences to replace用rovided case-fold-search is non-nil. If case-fold-search is set to nil, case is always significant in all searches.

In addition, when the newstring argument is all or partly lower case, replacement commands try to preserve the case pattern of each occurrence. Thus, the command

     M-x replace-string <RET> foo <RET> bar <RET>

replaces a lower case `foo' with a lower case `bar', an all-caps `FOO' with `BAR', and a capitalized `Foo' with `Bar'. (These three alternatives様ower case, all caps, and capitalized, are the only ones that replace-string can distinguish.)

If upper-case letters are used in the replacement string, they remain upper case every time that text is inserted. If upper-case letters are used in the first argument, the second argument is always substituted exactly as given, with no case conversion. Likewise, if either case-replace or case-fold-search is set to nil, replacement is done without case conversion.
***

Comment 19 Dani Megert 2007-09-24 08:47:08 EDT 
While thinking back and forth how make this feature understandable and while looking at the new layout of the Find/Replace dialog I came to a different approach to solve this: we introduce a new regular replace expression (e.g. \P or \C). This has several advantages:
1. UI stays as is and to which users got used
2. in the replace regex content assist we can use a longer name and we can
   provide more details in the additional info popup
3. the code gets much simpler as we don't need all the additional APIs

So, instead of having the boolean flag you interpret the regex string, add a content assist proposal to the replace regex proposals and add a corresponding entry to the help doc. To see which files to touch take a look at the following patch which also introduced a new replace regex:
https://bugs.eclipse.org/bugs/attachment.cgi?id=72953
