Description Dani Megert 2005-09-21 03:33:55 EDT 
I20050921-0010

If the root is pointed to a directory that contains several workspaces then the
wizard has some limitations:
- there's no indication of duplicates
- the list is not sorted which makes it hard to detect duplicates (especially if
  the list is long)
- the list does not show from which workspace duplicate project come
- the wizard does not allow me to import the projects with same name - it should 
  allow to assign other names like the CVS Checkout As... operation does

Comment 6 Mike Cepek 2008-01-23 11:30:17 EST 
I'd like to see this item updated to Priority P3 or better, and have Severity be Major (or Minor if a workaround can be documented).  The problem I outline below is a blocking issue for some people upgrading Eclipse.

I had been using Eclipse 3.3.1.1, but need to workaround #189025, so I'm trying to install Eclipse 3.4M4.  I want to create a NEW workspace for this new version, because I'm not sure that 3.4M4 supports all the plug-ins I need to totally convert yet.  And I want to check myself that #189025 is fixed in 3.4M4.

The 3.4M4 "Import Projects" wizard fails to import my old projects.  It seems to think four of my five projects are "the same" when they are not (see attached "import.png").  Attempting the import causes the error shown in "importError.png", and my new workspace ends up with just one project in it.  (Yes, I know I could get one other - but I would still be missing three, including the one I'm actively working on).

Four of my projects happen to be based on the same CVS module (cvsnew), as shown by the 3.3.1.1 "origProjects.png" screen shot.  But I gave them different project names in 3.3.1.1 - why does the 3.4M4 import wizard ignore those?

From an end-user point of view, this is a bug that prevents clean upgrading.  Eclipse needs fewer of these roadblocks to adoption, and I strongly request that at least this portion of this bug (item #4 from the original 2005-09-21 description) be fixed for Eclipse 3.4.

Comment 13 Mike Cepek 2008-01-24 09:03:14 EST 
Perhaps I misunderstood Tod's comment.  I don't have a code patch to offer, but I do know what different behavior I want...

When creating a new project in Eclipse, each project is given a unique a name.  In my case those names are shown in my "Original list of projects..." attachment.  My request is for the "Import Projects" wizard to base it's list of importable projects on those project names.

I don't understand why the Import Projects wizard uses different names for the projects.  That's where I think the problem is.  I don't know where the import wizard is getting those shortened names (in my case naming four of my projects as "sesnew" instead of "sesnew_xxx").

To reproduce this yourself, simply create 2 different projects based on the same CVS trunk or branch (I haven't tested with other VCSs).  Then use "switch workspace" (UNCHECK the "Copy Settings" boxes when creating the new workspace).  Then use "File > Import > General > Existing Projects into Workspace" and BE SURE to check the "Copy projects into workspace" checkbox.

When the wizard names the imported projects correctly and succeeds, this problem will have been corrected.

Comment 16 Dani Megert 2008-01-30 10:42:24 EST 
Created attachment 88289 [details]
Patch that makes the wizard use same structure as import source

When trying to come up with the UI I came to the conclusion that we might not even need this. In my opinion if a user imports this:
[project container foo]
  - project.description says JUnit
[project container foo2]
  - project.description says JUnit

I think he will expect to get the same i.e.
[project container foo]
  - project.description says JUnit
[project container foo2]
  - project.description says JUnit

And not:
[project container JUnit]
  - project.description says JUnit
EXCEPTION due to duplicate project


This patch changes to wizard to show and produce:
[project container foo]
  - project.description says JUnit
[project container foo2]
  - project.description says JUnit

Let me know what you think. If you feel that an additional check box is needed I can do this but I fear that most people wouldn't understand the meaning of it.

Comment 28 Paul Webster 2013-09-16 09:26:09 EDT 
(In reply to rupert THURNER from comment #26)
> is there any trick you use to find in which git repository this hides, given
> the 14 page list at http://git.eclipse.org ?

To find the plugin that contains a class, see http://wiki.eclipse.org/FAQ_How_do_I_find_a_particular_class_from_an_Eclipse_plug-in%3F

Once you've found the plugin, the MANIFEST.MF will contain a statement like:
clipse-SourceReferences: scm:git:git://git.eclipse.org/gitroot/equino
 x/rt.equinox.bundles.git;path="bundles/org.eclipse.equinox.common";ta
 g="I20130409-0800";commitId=c36ff638c1be8c4e54e60786a058469aff8cd957

That spells out the git repo it came from.
