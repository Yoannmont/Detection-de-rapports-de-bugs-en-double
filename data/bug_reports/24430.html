Description Bob Foster 2002-10-05 04:22:01 EDT 
If a SAXParser has set a user entity resolver (setEntityResolver),
and the resolver is given an absolute system id with an http: URL,
and the resolver resolves the id and returns a non-null InputSource,
and the input source contains an external entity with a relative system id,
then the entity resolver is next called with an absolute system id computed from
the relative id and the URL of the source XML document, instead of the URL of
the input source.

For example, when parsing a document at
file:///G:/eclipse/runtime-workspace/Test//foo.xml which contains <!DOCTYPE html
PUBLIC "-//W3C//DTD XHTML 1.0 Transitional//EN"
"http://www.w3.org/TR/xhtml1/DTD/xhtml1-transitional.dtd">, the entity resolver
is first called with the public and system id from the DOCTYPE. If it returns
non-null, it is next called with a public id "-//W3C//ENTITIES Latin 1 for
XHTML//EN" and system id "file:///G:/eclipse/runtime-workspace/Test/xhtml-lat1.ent"!

On the other hand, if the entity resolver is changed to return null and with no
other changes, when the above is repeated, the resolver is called the second
time with the above public id and the correct system id
"http://www.w3.org/TR/xhtml1/DTD/xhtml-lat1.ent".

The bug seems likely to be in the code in EntityResolverWrapper, but it isn't
obvious and I can't see values of variables in the debugger.
Comment 1 Bob Foster 2003-03-15 17:50:49 EST 
Do bugs filed against xerces just go in the bit bucket? It's been five months 
on this one with no response.

The reason I ask is, I have found a new case where entity resolvers are not 
used correctly: in the reading of xml schemas. I tracked this one down. The 
problem is in org.apache.xerces.impl.xs.traversers.XSDHandler getSchema(). The 
errant code is:

                String schemaId = null;
                XSDKey key = null;
                if (schemaSource.getByteStream() == null &&
                    schemaSource.getCharacterStream() == null) {
                    schemaId = XMLEntityManager.expandSystemId
(schemaSource.getSystemId(), schemaSource.getBaseSystemId());
                    key = new XSDKey(schemaId, referType, schemaNamespace);
                    if (fTraversed.get(key) != null) {
                        fLastSchemaWasDuplicate = true;
                        return(Document)(fTraversed.get(key));
                    }
                }
                fSchemaParser.reset();
                fSchemaParser.parse(schemaSource);
                schemaDoc = fSchemaParser.getDocument();
                // now we need to store the mapping information from system id
                // to the document. also from the document to the system id.
                if (schemaId != null) {
                    fTraversed.put(key, schemaDoc );
                    fDoc2SystemId.put(schemaDoc, schemaId );
                }

This code essentially says that if an entityresolver supplied the contents of 
an entity, don't remember its location in fDoc2SystemId and don't remember 
that you've read and parsed it before. Then, when the entity attempts to load 
a new entity with a relative URL, the code has no idea what it is relative to 
and falls back to the install location (of Eclipse, in this case). This is 
completely bogus. The code should be:

                String schemaId = XMLEntityManager.expandSystemId
(schemaSource.getSystemId(), schemaSource.getBaseSystemId());
                XSDKey key = new XSDKey(schemaId, referType, schemaNamespace);
                if (fTraversed.get(key) != null) {
                    fLastSchemaWasDuplicate = true;
                    return(Document)(fTraversed.get(key));
                }
                fSchemaParser.reset();
                fSchemaParser.parse(schemaSource);
                schemaDoc = fSchemaParser.getDocument();
                // now we need to store the mapping information from system id
                // to the document. also from the document to the system id.
                if (schemaId != null) {
                    fTraversed.put(key, schemaDoc );
                    fDoc2SystemId.put(schemaDoc, schemaId );
                }

With this change, entityresolvers work correctly.

Without putting too fine a point on it, is there any chance of getting this 
bug fixed in 2.1? The alternative for me is to ship with my own copy of xerces 
which I would really rather not do.
Comment 2 DJ Houghton 2003-03-17 10:02:53 EST
