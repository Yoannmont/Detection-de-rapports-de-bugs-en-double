Description Rebecca Weinhold 2006-02-22 17:00:59 EST 
This pertains to build M200602141429.

Repro:
1. Create a Dynamic Web Project.
2. Create a default JSP in the WebContent directory of the web project.
3. Add the following line (or similar) to the body of the JSP:
<% out.println("Hello."); %>
4. Set a breakpoint on the line you just added.
5. Create a Tomcat server (I used Tomcat 5.5) and start it in debug mode.
6. Right-click the JSP and choose Run As > Run on Server.

Result: A switch-to-debug-perspective dialog comes up and the debugger hits the breakpoint.
Expected: The JSP runs as though there were no breakpoints set/enabled.

Note: There is a workaround for the user (switching to run mode or manually disabling any breakpoints before running), but this ought to work without much user intervention.  This also applies to the generic WebLogic Server adapter.
Comment 1 Tim deBoer 2006-02-22 17:41:34 EST 
This is "working as designed". In past usability studies, users have said that they would rather leave the server in debug mode and continue instead of restarting the server. Note that if you do the opposite (are in run mode and choose debug on server) it prompts the user to see if they want to restart in debug mode.

I do not think we should change modes without user interaction, since this will lose all server state (e.g. session data) and make take several seconds or minutes to restart on some servers. Do you want this bug to be used for prompting the user  in this case and restarting the server in run mode, or do you have any other suggestions?
Comment 2 Rebecca Weinhold 2006-02-22 18:14:52 EST 
I agree that it is best to leave the server in debug mode -- I didn't mean to suggest that the server be restarted without user interaction, but merely to point it out as one of the two workarounds available to the user.  

The second workaround, manually disabling all of the breakpoints, is the model I used for the expected behavior.  In a perfect world, the breakpoints would be disabled temporarily when the user chooses Run on Server instead of Debug on Server, keeping track of which breakpoints were enabled/disabled at the start in order to restore the state of each breakpoint after the Run on Server is completed. 

I'll leave it up to you, but I wouldn't agree that this is an enhancement request -- if it isn't possible to Run on Server from within debug mode, that option shouldn't be available in the context menu.
Comment 3 Tim deBoer 2006-02-23 08:58:58 EST 
Although disabling the breakpoints is a valid option, I don't want to direct the user only down this path since it disables all breakpoints on the process (i.e. not just in the resource you selected to run) and many users will not know how to turn it back on.

Provided you agree, I'll use this defect to add a prompt with three choices:
 * Restart in Run mode
 * Disable breakpoints and continue
 * Continue as is

This also indicates that we should probably enhance the existing dialog Debug on Server to check whether breakpoints are disabled and re-enable them.
Comment 4 Ted Bashor 2006-02-23 12:34:59 EST 
Tim, I take it you don't think Run on Server should be disabled when the server is in debug mode?
Comment 5 Rebecca Weinhold 2006-02-23 12:40:55 EST 
Is it possible to disable the breakpoints in a temporary manner (i.e. for one
run only), so that the user wouldn't have to know how to re-enable them?  If it
can't be done behind the scenes, then a prompt seems necessary to alert the
user to the side effects (a wait for server restart, disabled breakpoints, or
actually hitting the breakpoints, respectively).  It also might be useful to
have a "Remember my decision" type of checkbox for the prompt.
Comment 6 Tim deBoer 2006-02-23 13:41:51 EST 
> you don't think Run on Server should be disabled when the server is in debug mode?

Eclipse Run> and Debug> support only allows for visibility changes (i.e. you can't disable it, only make it disappear from the menu). I don't think removing it from the menu is consistent with the other Run > and Debug > options, which only change visibility based on the resource selected, not on any other state.

> Is it possible to disable the breakpoints in a temporary manner (i.e. for one
run only), so that the user wouldn't have to know how to re-enable them?

No, Eclipse debug doesn't support this.

> If it can't be done behind the scenes, then a prompt seems necessary to alert the user to the side effects (a wait for server restart, disabled breakpoints, or actually hitting the breakpoints, respectively).

Yes, will prompt as in comment #3.

> It also might be useful to have a "Remember my decision" type of checkbox for the prompt.

Good idea. Will investigate how the platform does this since we'll also need a way to force a reprompt.
Comment 7 Ted Bashor 2006-02-28 14:46:24 EST 
We'd request that for 1.0.2, we at least add a minimal warning dialog that alerts the user that the server is in debug mode and breakpoints will be hit if there are any enabled.

The multiple option dialog would be great, but not required for 1.0.2.

Does this sound appropriate/feasible?
Comment 8 Rebecca Weinhold 2006-04-21 18:18:23 EDT
