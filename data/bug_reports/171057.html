Description Hristo Sabev 2007-01-19 10:15:08 EST 
Hi Colleagues,

We (SAP) need an extension to allow vendor specific naming or ear projects.
Whenever a new web service is created(bottom-up and top-down scenarios) a default name for the EAR project is automatically proposed by WTP. For us this has several severe consequences:
1. This breaks the SAP naming convention and we are forced to check the name proposed by WTP and report an error. 
2. In some SAP specific scenarios the name of the ear project should be configured calculated by a remote back-end system and is thus not trivial for the user to guess.
3. Our tools are targeted mainly to java 5/jaxws web service development. In order to get this currently working we have introduced some new facets (ejb 3.0, web 2.5). These are currently ignored by wtp. Thus any existing ear 5 project currently referencing the service project gets ignored and a new ear 1.4 project gets created. We guard agains this condition by checking whether the ear project specified in the wizard actually exist and if not we report an error asking the user to enter an existing ear project

Regards,
Hristo
Comment 1 Chris Brealey 2007-01-24 17:21:00 EST 
Hristo,
are you looking for this in WTP 2.0?

The algorithm [1] where we compute the name of the EAR first gives preference to existing module project / EAR project associations then, in the absence of any, tacks "EAR" on to the project name. What you're asking for sounds quite reasonable and may not be that hard to manage, but historically speaking the code in question has always been a bit sensitive to change. As an alternative to introducing a new extension point, would adding an optional method to override our algorithm [1] to the IWebService and IWebServiceClient interfaces suffice?

Are you or anyone in your midst interested and available in working up a proposed patch?

[1] See "DefaultingUtils.java"

Comment 4 Chris Brealey 2007-03-15 09:43:16 EDT 
Hristo,
in theory we could add methods to IWebService and IWebServiceClient without breaking adopters since all adopters should be extending the abstract derived classes AbstractWebService and AbstractWebServiceClient rather than implementing the interfaces directly.

Unfortunately, this may be moot. The concrete derivations of IWebService and IWebServiceClient don't get constructed via the extension registry until after the user clicks "Next" on the first page, however, we compute a default EAR project name earlier than that.

Could you elaborate a bit on what visual effect you would like to see? For example, when a user chooses your Web service runtine via clicking on the appropriate hyperlink next to the sliders, would you expect the value of the EAR project a couple lines below to change? When the user changes their Web project name, would the value of the EAR project field change in response to individual keystrokes?

I'm beginning to suspect that implementing this RFE will require plugin activation to occur in response to the initialization of page 1 or to user interactions on page 1, which we've avoided so far, and which requires a bit of design care to be sure we don't regress the performance characteristics of the first page of the wizard.

Peter,
I'm bouncing this over to you in part because Kathy has a few other RFEs on her plate and in part because the UI may be getting more relevant here, and the UI is one of your knacks / focus areas.

Thanks - CB.
