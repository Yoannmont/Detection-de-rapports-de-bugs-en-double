Description Pawel Piech 2007-11-29 19:55:41 EST 
Created attachment 84118 [details]
Patch containing the proposed extension point, and its use to implement CDT breakpoint filter.

_Problem:_

CDT breakpoints are generic breakpoints which hold standard attributes such as file, line number, function name, condition, ignore count, etc.   With these basic features CDT breakpoints (which implement the interface org.eclipse.cdt.debug.core.model.ICBreakpoint), could be re-used by any debugger using any debug model, not just the standard debug model and its extensions.  

So DSF-based debugger, or a debugger implementing flexible hierarchy adapters directly to populate the debug views, could read the CDT breakpoints from the breakpoint database and install corresponding breakpoints on its targets.

However there is one aspect of CDT breakpoints which is model specific, that is filtering.  The ICBreakpoint interface contains the following methods used for defining a breakpoint filter:
    public void setTargetFilter( ICDebugTarget target )
    public void removeTargetFilter( ICDebugTarget target )
    public void setThreadFilters( ICThread[] threads )
    public ICDebugTarget[] getTargetFilters()
    public void removeThreadFilters( ICThread[] threads )
    public ICThread[] getThreadFilters( ICDebugTarget target )


_Proposal:_

To make the CDT breakpoints more model-agnostic I propose to create a new extension mechanism in CDT to allow the CDT breakpoints to have model-specific extensions.  

A new 'CBreakpointExtension' extension point would be created in org.eclipse.cdt.debug.core.  An example of use of this extension point would look like this:

 <extension point="org.eclipse.cdt.debug.BreakpointExtension">
  <breakpointExtension
      id="com.example.ExampleBreakpointExtension"
      markerType="com.example.ExampleBreakpointMarker"
      debugModeId="com.example.debug"
      class="com.example.BreakpointExtensionImpl">
  </breakpointExtension>
 </extension>

This extension would then allow debuggers based on CDT, CDI, or DSF to add model-specific extensions to the breakpoint.  The extensions would be retrieved from the ICBreakpoint object using a new method:
    public ICBreakpointExtension getExtension(String debugModelId, Class extensionType)
Which would instantiate the extensions for a given debug model as needed in order to avoid unnecessary loading of plugins.

With a few changes to the breakpoint property page actions, the breakpoint property dialog would allow property pages to be contributed only for a specific debug model.  These property pages would only be activated when a debug context with the given debug model is active.


The attached patch includes an ICBreakpointFilterExtension which contains the CDT standard debug model-specific methods listed above and which used to be on the ICBreakpoint interface.  It modifies the CBreakpointFilteringPage property page so that it uses this extension to configure the breakpoint filter.  The only differnce in functionality is that after this patch, the filter page is shown in the property dialog for a CDT breakpoint when a CDT debug session is active.  I don't believe this would be a problem for the user since by its nature, the breakpoint filter can only be configured when there is an active CDT debug session.  BTW, the declaration of the property page for a CDT breakpoint filtering page would have an additional parameter '<filter name="debugModelId" value="org.eclipse.cdt.debug.core"> and would otherwise look the same.


_Summary_

In general I think this extension point would be very useful not , because it would allow vendors extending the CDT debugger to add features to CDT breakpoints, such as target-specific hardware breakpoint options, without the need to patch any of the CDT plugins.

Comment 4 Pawel Piech 2007-12-03 13:01:44 EST 
(In reply to comment #3)

> 1. User sets a breakpoint in a source file when no session is active.

The extensions are created on demand.  Which means they are created either when a property page is opened which uses a given extension, or when a debugger session tries to access the extension data in order to process the breakpoint correctly.  
So if the breakpoint is created when there is no active session, then the breakpoint is simply created without any extensions.  It's important to note that the extensions can store properties with the marker just like the breakpoint itself.  So even though the extension is created on demand, its serialized data is not lost when the breakpoint is created.  

The main difference is that if the breakpoint properties dialog was opened and there was no active CDI session, the "Filter" page would be missing.


> 2. Two types of debug models are used to debug the same application. Both are
> running at the same time. A breakpoint is set for the first model type. How it
> would be reflected in the second session? In general, can a breakpoint be
> shared by two different models?

Yes, a breakpoint would be shared by the two (or more) different models.  The extensions are keyed with the debug model id, so the two debug models' extensions would not interfere with each other.


I should point out that this extension mechanism is not a cure-all.  The biggest problem with it is that there is no way for the breakpoint icon or label to be affected by the extensions, because we cannot pass active debug context information to the CDebugModelPresentation.  I still plan to propose a second extension mechanism, which will allow debuggers to create completely custom breakpoints from within the CDT editor.
Comment 5 Nobody - feel free to take it 2007-12-03 13:27:41 EST

Comment 6 Pawel Piech 2007-12-03 15:09:05 EST 
Using the adapter mechanism only would probably be sufficient to implement the UI (property page enablement) part of this functionality.  This is because the CBreakpointContext.getAdapter() could examine the currently selected debug context, and determine whether an extension should be active.  

However, it would not be sufficient to implement the mechanism where the specific debugger implementation needs to access its model-specific extension.  In this case it is non-UI code accessing its extension information and it cannot get to the active debug context, which may be wrong anyway.

In short, no I don't believe this functionality can be implemented just by leveraging IAdaptable and without adding a new extension point.

Comment 15 John Cortell 2008-01-11 07:48:36 EST 
I consider "ignore count" and "condition" standard. At a very minimum, they can be supported by putting an unconditional breakpoint on the target and performing the evaluation in the debugger engine (DE). If the criteria isn't met, the DE resumes the target, all along keeping the user in the dark about the temporary suspension. Obviously, such a breakpoint will significantly impact the target application's performance if the breakpoint is set in high traffic code. A smart DE will opt to use the target's capabilities for the criteria matching if it supports it, but I imagine all DEs provide the mechanism I described.

Comment 19 John Cortell 2008-01-11 10:08:58 EST 
(In reply to comment #18)
> It depends on debug engine. For example, with gdb we have had problems I
> mentioned but in different situations. But of course, gdb supports ignore count
> and conditions.
> The situation is similar to one when the condition is not a valid expression.
> What do you do in this case - set unconditional breakpoint or report an error? 

You cannot feasibly determine the validity of an expression when setting the breakpoint. At one extreme, the user may set the bkpt before launching the session. So, conditions have to be treated as "will try" criteria. When the breakpoint is hit, if the condition cannot be evaluated, then the criteria is effectively not met, and the target should be resumed (i.e., to the user, the breakpoint will never get hit).
