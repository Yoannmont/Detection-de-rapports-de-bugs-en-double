normal for a little while.

previously

Comment 2 John Arthorne 2004-04-06 18:20:47 EDT 
As you're the first to report this, and many are using M8, I suspect a localized
problem.  First, can you give more details, including:

 - What plugins do you have installed outside the ones included in M8?
 - Are these plugin projects, or just normal Java projects?
 - Are the projects shared in a repository (CVS or otherwise)
 - Anything special about your file system (network drive, samba, etc)

If you can reproduce with any frequency, you can instrument the build mechanism
to find out if builds are running:

1) Copy the file plugins/org.eclipse.core.resources_3.0.0/.options to the
Eclipse install directory (eclipse/.options).

2) Turn on builder instrumentation flags:

org.eclipse.core.resources/debug=true
org.eclipse.core.resources/build/failure=true
org.eclipse.core.resources/build/invoking=true
org.eclipse.core.resources/build/delta=true
org.eclipse.core.resources/build/needbuild=true

3) Start eclipse with the "-debug" command line argument, and use java.exe as
the VM instead of javaw.exe:

 eclipse -vm d:\java142\jre\bin\java -debug

-> You should now get diagnostics printing in the Java console.  When it gets
into a state where builds no longer happen, see if any diagnostics are generated.

Comment 59 John Arthorne 2004-05-28 12:32:24 EDT 
With great drama, I believe we have identified and fixed this bug, within
minutes of the first Eclipse 3.0 candidate build! 

Sleeping jobs are inserted into a priority queue, and worker threads "peek" at
the queue to see if the first job should start running. There was a condition in
which a job could have its wake priority changed without the queue being
reshuffled. This could result in a job with a distant future wake time to be at
the head of the sleep queue. Worker threads would peek at this job, and then
give up after observing that the wake time had not yet been reached. Meanwhile,
other jobs further back in the sleep queue could be well past their wake time.

The fix was to ensure the sleep queue is always resorted when a job changes it
desired wake time (this was happening in most cases, but in one case it was missed).

Ian's recent debug trace showing a load of SLEEPING jobs tipped us off that some
jobs were sleeping that should have already run. Thanks to Ian for giving us
