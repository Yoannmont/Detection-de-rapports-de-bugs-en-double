Description Darin Swanson 2002-12-06 11:00:24 EST 
From the newsgroup:
eclipse-SDK-I20021204-linux-motif.zip on my
Athlon 1400 MHz - 768 MB RAM with SuSE Linux 7.2 (kernel 2.4.19) and
(SUNS SDK 1.4.101)/(IBM's SDK 1.4.0), and what recognized is:

      Eclipse v2.0.2 (I200210251102) has a better overall performance,
      and its debugger is much faster: launch time is 50%-100% faster;
      and instruction steps are 5 - 10 times faster.

More details:
[1] After my experiments with eclipse-SDK-I20021204-linux-motif.zip
     I switched back to v2.0.2 and everything has it's perfromance as
     expected, there was no reboot or X-server restart between my
     experiments with eclipse-SDK-I20021204-linux-motif.zip and my
     switch back to v2.0.2. I think that implies, that there is no 
     problem with my local maschine (for example a daemons consuming
     CPU power, ...).
  [2] A test case on my system is to compare the performance of v2.0.2
     and v2.1, I have a not too big  workspace (5MB source code), I use 
     handmade classloaders and securitymanager (I think nothing else
     should slow down the debugger). And my observation happen under
     all conditions.
  [3] I tried for the SDKs (SUN/IBM see below) different JVM options as
     -client/-server -Xms320MB or other thing note somewhere in this 
     newsgroup there was no differents for 10 combinations or so.
  [4] If I say "50% increase in launch time" I don't talk about plugin
     launch time, i.e.: first start off debug session is a little bit
     slower then the following ones. My programm measures it's startup
     time from the very beginning of the main()-method until all services
     are started, that needs approcimately 4 seconds, without debug (run
     mode), on v2.0.2 it needs with debugger 5 seconds (or so), but with
     v2.1 it needs 10 seconds with debugger and 5 seconds without it.
     That is what I mean with 50% slow down.
  [5] I have on my linux box a windows 98 living in a vmware-pc, I use it
     to test my programm under windows. If I remote-debug the JVM with
     eclipse I have approximately the same bad performance with 2.0.2 as
     I have it in a locale-debug session with v2.1. The reason I tell
     this here is: 
     I think my remote-performace slow down is much bigger then I 
     expected, so eclipse v2.0.2 has a perhaps a performance problem 
     with remote debugging, and again perhaps this problem "moved" in 
     the locale-debug mode of v2.1. That means that my first guess is 
     not nessesary true, my first guess was that the noticable overall 
     performance decrease is not the reason for the big performance 
     decrease in the debugger.

Comment 2 Bob Foster 2002-12-06 13:04:53 EST 
Stepping does seem slower in M3 (Win2K, JDK1.4.1) compared to M2. It doesn't

Comment 3 Jared Burns 2002-12-06 14:54:23 EST 
Luc has done some profiling here and it looks like updating the marker that
is used for the instruction pointer is very expensive. If we can't make it
faster (do we *really* need to use a marker?), I would suggest removing the
feature. I don't think it's an acceptable tradeoff to make stepping prettier
if it also makes it noticably slower.
Comment 4 Darin Wright 2002-12-06 15:12:54 EST

instruction pointer marker (used to select text in the editor). I was under the 
impression that we are not using markers for the "arrows" - just annotations.
Comment 5 Jared Burns 2002-12-06 15:21:36 EST 
It's definitely a marker. Take a look at the code in LaunchView.
Comment 6 Luc Bourlier 2002-12-06 16:28:30 EST 
I did some profiling for the bug 25516. Beside the GTK problem, it appears that
the instruction pointer arrow takes up to 500 ms to be updated (see the
screenshot attached below).

My test case is :
- launch a runtime workbench in debug mode.
- launch a simple application in the runtime workbench.
- set a breakpoint line 150 in ASTEvaluationEngine.getCompiledExpression(String,
IJavaStackFrame) ('EvaluationSourceGenerator mapper = null;').
- evaluate an expression in the runtime workbench.
when the breakpoint is hit :
- launch the profiler, hit [F6] (step over)
when the step is performed :
- stop the compiler

We should do the same profiling on win32 (and motif) to confirm the problem.
