Comment 6 Stefan Daume 2009-09-29 09:10:57 EDT 
Created attachment 148320 [details]
Buckminster EMF-based common & rmap models and editors

Thomas, as requested I attach the EMF-based Rmap (and Common) models and editors for review. The generated models are based on the Buckminster EMF schemas. I will also add the adapted common-1.0.xsd and rmap-1.0.xsd. They contain the necessary EMF references and are required to generate the models, and do so with useful default names. I have not checked them in either.

There are a number of caveats with the current version of those models and editors:

1. The "Common" model has a documentation element which extends the xh:Flow type, which essentially means that the documentation element ought to hold valid XHTML. For the generation of the models I changed this type to xsd:anyType. I am fully aware that this is not the final solution, but it allowed to generated a complete and consistent set of models without having to include a full EMF model for XHTML into the Buckminster model set. A final decision on the best possible approach is required.

2. This is a plain RMap model and editor. That is, it does not contain extension models (PDEMap, MavenProvider) that may be required to edit advanced samples of an Rmap (i.e. teh dogfood rmap). I do have models for those as well, but have not included them for the initial review.

If you want to test the model editor and you have Buckminster in the platform you are using then make sure you select the "RMap Model Editor" and not the "Buckminster RMap Editor".

A final comment: I do have models for the other Buckminster model elements. I suggest that after reviewing the RMap-based stuff new Bugzillas are created cover those other models in support of a Buckminster EMF-ization.

   Stefan

Comment 9 Thomas Hallgren 2009-10-02 12:15:20 EDT 
Looking at the current state of this I don't think it's very useful. The model is 100% true to the xsd's which means that all deprecated declarations are included, that the top of the RMap editor is a DocumentRoot where very strange things can be placed, and that elements are represented with their XML element name in the editor. All in all, fairly hard to make sense of.

I think this model needs a lot of work to become functional. So even for the RMAP, where the XSD's is more suited as the base, I'm far from convinced that this is the right path. Instead, we should create a model that makes sense and can assist us in both generating the RMAP Java code and the user editor. I'm sure we can apply the right transformations to adopt such a model to our current way of persisting the RMAP.
Comment 10 Thomas Hallgren 2009-10-05 07:37:12 EDT 
I rewrote the model from scratch using an ecore centric approach and then added the extended meta-data needed to make the XML compliant with the current XSD.

We now have the following new plug-ins.

For the common model shared by all Buckminster models:
org.eclipse.buckminster.model.common
org.eclipse.buckminster.model.common.edit

For the RMAP:
org.eclipse.buckminster.rmap
org.eclipse.buckminster.rmap.edit
org.eclipse.buckminster.rmap.editor

The MavenProvider extension to the RMAP:
org.eclipse.buckminster.rmap.maven
org.eclipse.buckminster.rmap.edit.maven

The PDEMapProvider extension to the RMAP:
org.eclipse.buckminster.rmap.pde
org.eclipse.buckminster.rmap.edit.pde

The editor and the extensions are automatically included in the following features:
org.eclipse.buckminster.core.feature
org.eclipse.buckminster.maven.feature
org.eclipse.buckminster.pde.feature

All affected bundles and features were branched off to Platform-3.5.x before this was introduced. All features and bundles that make use of the EMF models have an increased minor version number, i.e. version 1.2.360.

Work that remains to be done:
1. Replace rmap classes in core with the ones generated from our RMAP ecore model (bug 291329).

2. Create a pure headless maven feature (bug 291330)

3. New icons are needed for the nodes in the ecore generated RMAP editor (bug 291332)

The new rmap editor (all projects mentioned above) is released to trunk, revision 10738
