Description Tom Maier 2006-11-10 08:24:59 EST 
Unsatisfying consideration of baseLocation vs. buildDirectory

Dependencies of a plugin are directed to other plugins
inside the buildDirectory instead of directing it to
the target platform.


Example:
You want to create build scripts for a feature. The feature.xml
tells, that it consists of only one plugin. However, it depends
on further plugins. These further plugins are available at 2 places:
They are available compiled in the target platform ("baseLocation"), 
and they are available as sources (and maybe, but maybe not, also compiled)
under "buildDirectory".

Now the problem is, that the build script automation generally assumes,
that these needed plugins shall be used from the buildDirectory. So it
creates a build.xml for the feature, that only calls a build.xml for
the one named plugin (what is correct), but it expects to have the other 
plugins already compiled in buildDirectory (what is not correct), so in 
the build.xml for the named plugin it creates lines in the element
 <path id="named_plugin.jar.classpath"> for the other plugins, that refer 
to the buildDirectory. But I think, these lines should refer to the target 
platform.

As a test, I deleted the other plugins (completely!) from the buildDirectory. 
Then it worked fine, meaning, that the other plugins were refered to in 
the target platform, and so it compiled.

Proposal:
All plugins mentioned under <plugins> should be taken from the buildDirectory,
and all others should be taken from the target platform. 

(Or alternatively the behaviour should be switchable. Maybe that would be the best solution.)


Background info:
I want to headless compile single plugins in a nightly compile, to verify
the compile cleanness against the other _released_ plugins. They shall _not_
be compiled against the other _working_stage_ plugins.


It seams to have relations to bug #139923, but I don't think, that
the bugfix proposed in there will solve this bug too.

Also bug #108625 may be explained by the behaviour analysed in here.

Further Info:
With "Save As Ant Script" in the GUI, it depends on if the other plugins
are "opened" or "closed" in the workspace. But as the workspace is a
gui thing, the working of <eclipse.buildScript ...> should not depend on
this information. (See also #112767, the last comment in 2005)

At last, be aware, that in a _launch_ configuration (from gui) you can very
well define, which plugins shall be taken from the workspace, and which ones
from the target platform. Something similar is needed for the compiling step.
Comment 1 Andrew Niefer 2006-11-10 11:00:12 EST 
baseLocation and buildDirectory are at a certain level interchangeable.  You can consider them as simply specifying a list of places to look for plug-ins and features.

The problem comes when you have 2 versions of the same plug-in (ie a source and a binary, the difference is distinguished by the presence of a build.properties file).  PDE.Build can only consider one of them.  Off the top of my head I'm not sure if you can predict which one will get resolved in the state (though a higher version number could be the deciding factor).

We should do something about this entire multiple versions of a plug-in problem.

Comment 3 Andrew Niefer 2006-11-10 11:53:17 EST 
> Thanks for agreeing. So what may all be relevant for the outcome, as it is by
> now?
In a headless build, we create an OSGi state that contains all the plugins from the following locations: pluginPath, buildDirectory, baseLocation.  Plugins are added to the state in the order that they are found, that is each of those directories in order and the order of File.listFiles.

The state is then resolved and we simply use whatever bundle was resolved in the state.  All the resolved bundles are sorted in pre-requisite order (this is based on the manifests), and the script is generated to build them in that order.

If more than one plug-in exists with the same symbolic name, then the one we use depends on the which got resolved.   Assuming both satisfy all the dependecy requirements, then if they are singletons, I expect the one with the higher version number is used.  Otherwise I'm guessing it is just the one that was added to the state first (though this might not be guaranteed).

In a headless build nothing in the workspace affects this.  The workspace only matters during an export since then the PDE/UI is giving us a state instead of us building our own.
Comment 4 Tom Maier 2006-11-20 11:56:28 EST 
Hello,

I want to make a proposal for the problem. As you may have realized,
it is actually a configuration management topic, and this is, where
the solution should go to. 

My proposal:
In plugin.xml have this optional section:

   <preferedLocations warnNotExistant="false">
      <plugin name="mystuf.some.*"   location="baseLocation" />
      <plugin name="mystuf.*"        location="buildDirectory" warnNotExistant="true" />
      <plugin name="eclipse.some.xy" location="baseLocation" version="10.2.8" warnNotExistant="true" />
      <plugin name="*"               location="baseLocation" version="LATEST"/>
   </preferedLocations>

The entries in this section are evaluated top-down, so for
each plugin the first entry with matching name pattern is 
selected. If this plaugin is not available at this location,
then the next entry is tried, and so on.
(This is actually the mechanism of clearcase config specs.)
    
So, you can choose, if the plugin shall be taken from the
extra pluginPath, the buildDirectory (workspace), or the 
baseLocation (target platform).
If from the target platform, you can additionally give a 
version you want to use. (Imho this piece of information is 
not covered in the Manifest.mf, as there is only a "same or
newer" information. If not, just delete this extra from my
proposal.)

Also, you can define a default and per-entry, if you want 
to be informed, if a plugin is taken from a different location.

What do you think about this?

Tom
