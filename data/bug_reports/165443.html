Description Georg Lenz 2006-11-22 08:51:12 EST 
Hello

the feature org.eclipse.platform.source that comes with the eclipse SDK download has no dependencies.
That is wrong. 
It contains the plugin org.eclipse.platform.doc.isv that has the following dependencies:
org.eclipse.help;bundle-version="[3.2.0,4.0.0)",
org.eclipse.ui;bundle-version="[3.2.0,4.0.0)"
Therefore the feature must also have these dependencies.

A simple compute dependencies would do the job.

That teases us in our build infrastructure where we have to correct features
that come from org.eclipse and that should therefore play along the rules of 
org.eclipse.

Regards
Georg

Comment 2 Jeff McAffer 2006-11-22 20:32:19 EST 
The "source" features are really the "sdk" features.  They often include source, isv doc and may even include examples.  The way to think of the is "everything you need if you are programming using the corresponding binary feature".

As for the original comment, yes, this does appear to be a problem.  There should be something that checks that features have a consistent dependency set.

Comment 7 Andrew Niefer 2007-04-05 16:07:20 EDT 
There is nothing for pde.build to do here.
1) pde.build is currently not capable of figuring out which features would be required in order to add them to the generated source features.
2) If including org.eclipse.platform.doc.isv adds dependencies to the feature, then org.eclipse.sdk (and org.eclipse.platform.sdk as appropriate) needs to provide a sourceTemplateFeature/feature.xml with the dependencies in it.

However, comment #4 and comment #5 suggest that these dependencies are not real.  In that case, the fix is just to remove the dependencies from doc.isv's (and any other doc plugins) manifest.

Comment 17 Thomas Watson 2013-08-15 17:19:00 EDT 
(In reply to comment #16)
> Adding Tom and John as the current "OSGi guardians" :) 

hmmm this is really reaching back in time.  I remember Jeff arguing at one point that Require-Bundle was needed for also non-code dependencies.  The help bundle provides the help system for the docs so it made some sense that the help docs would require the help system.

But with that said, it is no longer a best practice for osgi bundles to use require-bundle for non-code capabilities (and to be honest osgi best practices almost always recommend against using require-bundle at all, but that is another debate).

Instead bundles should use generic capabilities and requirements for non-code dependencies.  But such generic dependencies is not currently recognized by p2.

With that in mind I don't really see the require-bundle here as completely incorrect.  I certainly don't understand why it would cause build issues.
