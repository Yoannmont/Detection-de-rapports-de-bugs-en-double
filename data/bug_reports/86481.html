Description Bryan Hunt 2005-02-24 10:39:53 EST 
I have an emf model that extends another emf model, and the generated editor for the extending 
model does not behave as I would like.

My base emf model contains two classes:  Parent and Child.  The Parent class has an EList of Child 
classes (children).  My second emf model contains a single class: NeedyChild that extends (is a 
specialization of) Child.  Given a Parent, i should be able to add instances of Child or NeedyChild to it's 
list of children.

As expected the editor for the base project allows me to create a Parent and instances of Child, but not 
instances of NeedyChild.

The generated editor for the project that contains the model with NeedyChild does not allow me to 
create a Parent, only NeedyChild.

My workaround was to create a class in the same model as NeedyChild called PatientParent that extends 
Parent.  When I use the generated editor, I can create an instance of PatientParent and add instances of 
Child and NeedyChild.  From a modeling point of view, PatientParent is not needed as it provides no 
additional information.

Is there something simple I'm missing with the generated editor that would allow me to create elements 
in a base model without having to extend the top level element as I have done with PatientParent?  I'd 
really like to simply create an instance of Parent and be able to add instances of Child or NeedyChild.


A comment from Ed Merks:

I'm not sure how we will fix this.  It's like you would need to generated an extended item provider 
adapter for Parent that would know about NeedyChild.  Your approach of defining a PatientParent 
effectively forced this to happen, with the very unfortunate side effect of introducing a new model 
object.

Comment 3 Dave Steinberg 2005-03-23 09:27:02 EST 
Hi Eike,

Thanks for the contribution.

I'm not sure if you were offering up this template for Bryan's use, or if you
wanted it to be included in EMF.  I'm not inclined to do the latter, as I
believe it would complicate the general case for this fairly unusual one.

I believe that the generated wizard need not be perfect or complete.  What's
more important is that it's flexible and easy to modify to suit the needs of the
particular model.  Including every class from every package referenced may not
be ideal in all cases.  So, what I'm proposing instead is a slight refactoring,
where the string values for the combo box are collected in one place, near where
the inital model is created.

Armed with the knowledge of what's applicable for the model, the user is then
able to relate these as is appropriate.  For example, your extended library
example could be handled with this, pretty trivial change:

  protected Map initialObjectTypes;

  /**
   * Returns the names of the types that can be created as the root object.
   * <!-- begin-user-doc -->
   * <!-- end-user-doc -->
   * @generated NOT
   */
  protected Collection getInitialObjectNames()
  {
    if (initialObjectNames == null)
    {
      initialObjectTypes = new HashMap();

      initialObjectTypes.put("Library", LibraryPackage.eINSTANCE.getLibrary());
      initialObjectTypes.put("Book", LibraryPackage.eINSTANCE.getBook());
      initialObjectTypes.put("Author", LibraryPackage.eINSTANCE.getAuthor());
      initialObjectTypes.put("Topic (Library)",
LibraryPackage.eINSTANCE.getTopic());
      initialObjectTypes.put("EBook", LibraryPackage.eINSTANCE.getEBook());

      initialObjectTypes.put("Author2", Library2Package.eINSTANCE.getAuthor2());
      initialObjectTypes.put("Other", Library2Package.eINSTANCE.getOther());
      initialObjectTypes.put("Topic (Library2)",
Library2Package.eINSTANCE.getTopic());

      initialObjectNames = new ArrayList(initialObjectTypes.keySet());
      Collections.sort(initialObjectNames, java.text.Collator.getInstance());
    }
    return initialObjectNames;
  }

  /**
   * Create a new model.
   * <!-- begin-user-doc -->
   * <!-- end-user-doc -->
   * @generated NOT
   */
  protected EObject createInitialModel()
  {
    return
EcoreUtil.create((EClass)initialObjectTypes.get(initialObjectCreationPage.getInitialObjectName()));
  }

Life's so much easier when you don't have to handle names generally.  ;)

I'm attaching my new version of the template, against which this change would
work.  As a free bonus, I've fixed up the validation and focus issues that
existed in it.  I'm inclined to commit it and close this Bugzilla already.

Comment 5 Dave Steinberg 2005-04-01 14:51:56 EST 
Having heard no complaints, I've comitted the new wizard template that I proposed.

This allows you to very easily modify the generated wizard, as I described in
comment 3, so that you'll be able to create an instance of the base class from
the extended wizard.

The rest of the problem is that, once the editor comes up, you can't create an
instance of the extended child class.  As far as I'm concerned, that's not
something that can be fixed, out of the box.  The problem is that, if you're
generating the base and extended packages in separate projects, the base package
can have no dependency on the extended package.  So, the item providers for the
base package cannot know how to create an object from the extended package.

If you're willing to add the dependency to your base edit project, then you can
easily hand-modify your item providers as follows.

In ParentItemProvider:

  /**
   * @generated NOT
   */
  protected void collectNewChildDescriptors(
    Collection newChildDescriptors, Object object)
  {
    super.collectNewChildDescriptors(newChildDescriptors, object);

    newChildDescriptors.add
      (createChildParameter
        (BasePackage.eINSTANCE.getParent_Children(),
         BaseFactory.eINSTANCE.createChild()));

    // Add this...
    newChildDescriptors.add
    (createChildParameter
      (BasePackage.eINSTANCE.getParent_Children(),
       ExtendedFactory.eINSTANCE.createNeedyChild()));
  }

Then, everything should work.  One thing you might notice is that there was no
icon generated for this particular parent/child combination.  So, you'll see the
generic, non-colourized one, and you'll get a line like this on standard out:

full/ctool16/CreateParent_children_NeedyChild

This will go away if you create the missing icon.

The upshot is, with the new wizard pattern, it's very easy to do what you want
with hand-modifications to the generated edit code.  So, I'd like to close this bug.

Comment 8 Dave Steinberg 2005-09-30 14:07:06 EDT 
I believe you don't understand the fix correctly.  You should not see code
similar to comment #3 in the wizard until you write it (as indicated by
@generated NOT).  The fix is a refactoring that allows you to customize the
wizard's behaviour with this pair of trivial method overrides.

The reasons for not generating that code automatically are that:
1. it's not a given that you would want the wizard to be able to instantiate
classes from other packages
2. it's not necessarily clear how you would want to resolve any naming conflicts
between classes from different packages
3. it's quite likely that you might want to modify getInitialObjectNames()
anyway, in order to restrict which classes can be instantiated to a subset of
those in the package.

Basically, the idea is that you know your model better than we do, so you're in
the best position to decide how to present it.
