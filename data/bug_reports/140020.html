Description Min Idzelis 2006-05-03 12:43:13 EDT 
In PopupBarLabelHandle#paintFigure() the method isToolEnabled() is called which eventually tries to execute a command - which means it tries to acquire a transaction lock. (And its holding the UI lock) This is fine in the typical scenerio. However, I have a background Job in the middle of executing. It also is telling the an edit part to refresh, which eventually tells a Figure to do something which requires the UI lock. 

The fix would be to not execute a command in the paint() method. (Have it check a field that is updated via listeners instead)

Comment 2 Cherie Revells 2006-05-05 12:16:45 EDT 
In order to override paintFigure() the client would need access to private classes.

Min:  Painting should not occur when an operation is being executed.  There is an operation history listener in the DiagramEditor that disables updating (repainting) in the DiagramGraphicalViewer when an operation is being executed.  See DiagramEditor.createDisableUpdateHistoryListener().  

Looking at your stack traces, I would have expected that when the operation is executed in your background job, this history listener would have disabled updating in the DiagramGraphicalViewer so that when your main thread called:
org.eclipse.gmf.runtime.diagram.ui.parts.DiagramGraphicalViewer$ToggleUpdateManager(org.eclipse.draw2d.DeferredUpdateManager).performUpdate()
line: 167 
it would not have done the paint.

Could you try putting a breakpoint in the code in DiagramEditor.createDisableUpdateHistoryListener() where the DiagramGraphicalViewer updates should have been disabled?

Comment 4 Min Idzelis 2006-05-11 09:28:35 EDT 
I set a breakpoint in the listener in:
DiagramEditor.createDisableUpdateHistoryListener()
but it's never reached in my case and I know why. 

When I execute my operations that update the model on a background thread, I know that I'll be only changing transient (and mutable) semantic and view objects. I do not want my "updates" to the model to be able to be undone. Therefore, I new up a new OperationHistory when performing my background update. This new OperationHistory doesnt have the listener added to it.
