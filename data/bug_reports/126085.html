Comment 7 Ed Burnette 2006-02-10 09:41:30 EST 
I looked at the html source. It uses urls like this, which are invalid:

file:/c:/Program Files/eclipse/...

I don't know why this worked in the other browsers. According to the RFC1738 spec, the syntax is

file://<hostname>/<pathname>

where hostname can be omitted for local files. So the correct url would be:

file:///c:/Program Files/eclipse/...

This works on both IE7 and Firefox 1.5 on Windows.

BTW I noticed many links in the HTML that could have been relative instead of absolute. It probably doesn't matter but I thought I'd point it out.

References:
ftp://ftp.funet.fi/pub/doc/rfc/rfc1738.txt
http://www.cs.tut.fi/~jkorpela/fileurl.html
Comment 8 Dejan Glozic 2006-02-10 09:52:58 EST 
Note that the file you are looking at (intro) is dynamically created as a text stream that is directly fed to the browser widget. As such, there is no HTML file to talk about, so the concept of 'relativeness' is moot. We do set baseURL for the file so some URLs would theoretically work, but most of the time images come from various plug-ins (some are even from JARed plug-ins and need to be exploded first before referencing).

I will take a look at the URL issue after M5 - it could be simply an issue of saving an absolute (file) way in a different format.
