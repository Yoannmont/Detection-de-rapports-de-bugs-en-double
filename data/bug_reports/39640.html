Insight killer version 1.0 :-)

The attached patch will stop the GUI from:

- making assumptions about what constitutes a valid executable. This is the job
of the debugger.
- making assumptions about where the executable can be located. It can be
outside the CDT project.

So the procedure to debug an executable that was compiled outside CDT is:

1. Create a dummy empty CDT project
2. Create debug session for this CDT Project
3. Browse and select executable


Voila!

Comment 14 Ken Ryall 2006-02-07 17:28:21 EST 
Created attachment 34305 [details]
Implements previously attached Debugging when no project

Notes:

This patch contains changes in two main areas: a new wizard to import executables and additions to let you browse the source files within them.

IBinaryParser has an additional method to get a list of source files. Implementation of this ripples through several existing files and includes two additional ones. The additional files parse the symbols in files with debug information to produce this source file list. There are new classes that know how to parse CodeView and Stabs formatted debug info. Support for other formats is not yet implemented. 

These new classes implement a new interface, ISymbolReader. This is somewhat grandly named as it currently only parses enough debug information to produce a list of source files. We originally called it ISymbolReaderLite in order to dampen expectations that this would provide a full interface to all symbol data. However it may evolve to do more useful things so now it’s ISymbolReader.

Changes to Binary.computeChildren trigger the call into the new code to get the list of source files.

The user experience part of this patch comes from a new wizard in org.eclipse.cdt.ui, the ImportExecutableWizard. There are four new classes: an abstract version of the core wizard, the two wizard pages, and an reference implementation of the wizard. The wizard lets you select one or more executables, import them into the workspace, and sets up a launch configuration that’s preconfigured to debug them. CDT based products can use the abstract base class to create their own wizards to import their special executable types.

It occurred to us that these additions may push the organization of their host plug-ins a bit but we’ve placed them where they seem to make the most sense for now.

-	Ken Ryall

Nokia Developer Platform Tools

Comment 21 Ken Ryall 2006-03-09 12:30:29 EST 
Created attachment 35996 [details]
Revised patch

This patch contains changes that address Dave's feedback and cleans up a few other things:

In CModelManager.create there is a change that attempts to handle the TODO for external resources. This is required if the source file opened from the binary list is also in the project. Otherwise they will appear as two separate files.

My previous comment about the make stuff being necessary for the binaries to show up was incorrect. That code and the make dependency have been removed. Dave's suggestion about having a nature for these projects sounds like a good idea, but we haven't done anything with it yet.

Currently only Windows PE binaries are supported. We are working on ELF-Stabs and ELF-Dwarf2 next and will contribute those when ready.

The code that was waiting for building and parsing jobs to complete was unnecessary and has been removed.

There is a problem where the call to getBinaries will sometimes return an empty list. I'll file a new bug for this and provide more detail there.
