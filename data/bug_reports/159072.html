Comment 6 Brian Bauman 2006-10-02 16:34:21 EDT 
The performance of trying to keep track of multiple target is not practical.  I have been called crazy many times by Wassim for implying such an idea.  Resolver states are far from light weight and do not scale well.  

As a work around for such a problem (though it does not seem like it will work for this one) would be to create one big state that is a super set of all the states.  Then some how define a subset of which you wish to develop against.  If you then reference any plugins outside of your subset, you would receive a warning/error of some kind.  This would represent mini target runtimes, allowing the user to develop against more than one mini target at a time.

Since Wassim should be coming back soon, I will let him do what he sees fit for this bug.  I know if I agreed to this bug, he would come down here and kill me :)
Comment 7 Wassim Melhem 2006-10-27 20:46:06 EDT 
as Brian points out, having a target platform per plugin project is not as simple as having a separate JRE per project.  It is both impractical and confusing.

While a project's relationship with a JRE is local (politics), a target platform is an envrionment and a state (ie. federal politics).

Target platforms can be thousands of plug-ins large.  Each target platform comes with a huge memory footprint: a state, models, extension registries, etc.

Also, workspaces may contain thousands of projects.  Each with its own distinct target?  the costs become astronomical.

Performance-aside:
I have a plug-in project in my workspace that contributes a view.  Sometimes i want to run it against the entire SDK.  Other times, I want to run it as a standalone RCP application.  Why would I want to tie this plug-in to a particular target?

Intra-workspace relations:
Also, how would the workspace plug-ins relate to each other?  Do they belong to the same state and reference each other?  

i. If yes, then effectively all separate target platforms/states will now effectively be merged and we are back where we were.

ii. if no, then the workspace has become a confusing place.

I would like to see a proposal from people who need this function that attempt to address these issues.

Comment 19 Danail Nachev 2010-01-12 11:22:50 EST 
Would some like this work:

By default all plug-in projects use the current default target platform. If there is a need, a plug-in project can be changed to use a different target platform. All plug-ins, which reference the same target platform, will be visible to each other. Plug-ins which reference different target platforms won't see each other.

This limits the state, which needs to be kept to only the number of target platforms, which the bundles reference. This doesn't save from exploding in memory usage if you reference ten target platforms, but if you reference ten target platforms, it is expected that you have around 100+ projects, which will introduce other problems before target platform state become a problem. Do we have specific measurements on this?

You can add the reference to the target platform similar to how JRE System Library is shown: by showing the name of the target platform next to the Plug-in Dependencies node name. This would enable the user to quickly check whether the plugin reference the correct target platform.

For me, the main problem with current state is that you are not permitted to develop application, which spans multiple environments. For example, how can you easily manage application frontend using RCP and backend based on Equinox server-side stuff?
