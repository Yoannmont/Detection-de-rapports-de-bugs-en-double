Description John Arthorne 2007-09-27 11:15:35 EDT 
I'm self-hosting this week using an SDK provisioned by p2. 

 - I wanted to install the releng tools, so I generated metadata/artifacts, and added the site from the end user UI.  
 - In the "Available Features" tab, I see the feature I want. It's name is "org.eclipse.releng.tools.featureIU", version "3.2.100.v20070122....".  
 - I select this feature, and install it. 
 - When prompted, I give it the name "Releng Tools".  
 - When install is completed, I now have an entry called "Releng Tools" in the "Installed Features" tab, with version number "0.0.0.1190902249937".

It's confusing to me that the name and version change when I move something from the "available features" to the "installed features" by performing an install. Why not just use the human-readable feature name in both "installed" and "available" features tabs (and use the feature version number)? That would avoid having to ask the user to give a name to the software they are installing.
Comment 1 Susan McCourt 2007-09-27 12:25:45 EDT 
There are still some pretty meaty issues to be worked through.

The idea is that the groups are a producer concept and help filter from the vast choices of things in a repo.  The entry point records what a user installed.  They don't have to rename it, though the current UI implies they do.  Btw, the driving requirement for all of this is "name what is installed."  

The fact that the entry point version # deviates from what you chose is confusing and is covered in bug #203821.  The current UI (a text box for entering a name) is temporary and future direction is discussed in bug #204321 comment #1.  Once we have more info to present about an install/update, this prompt will become a more helpful wizard and the naming part will take a back seat to the more important info.  (I wouldn't expect renaming an entry point to appear on the first page, for example, if at all).

But there are some greater issues with entry points as soon as you allow (which we currently do) multiple groups to be coalesced into one entry point.  Then it gets harder to deal with versions and names.  Pascal and I discussed this a bit, and some of the points were:

- we need to make sure that grouping groups into a single entry point is actually helpful to a user.  It sounds good on the surface (name a big wad of stuff you installed so you can remember what it was, specifically if you don't care about the details.)
- if we allow it, there should be a way to go back and see which groups you picked when you installed an entry point
- as soon as you allow a user to drill down into an entry point, it begs the question:  can I go back and install something and add it to this entry point?  In other words, are entry points becoming something more dynamic like install categories?  I don't like this direction, but you can see how one might get there with requirements so we need to be careful about how all this is presented.
- I think of a named entry point as being a helpful hint as to what drove something to be installed.  "Oh yeah, I had to install those C tools."  The fact that they included 3 or 4 groups may be irrelevant.  On the other hand, you could argue that if the user can't deal with the detail of the 3 or 4 groups, then an administrator should build a group called "C tools" and make it available in the repo.  
- in some instances renaming wouldn't be permitted (to avoid support confusion, etc. etc.)  In my mind this is the biggest danger about entry points, they allow the identity from the producer's point of view to change to a consumer view.  Nice for certain users, but could be a support nightmare if not done well.

I'm marking this M3, not because I think we'll solve all the issues but because I think the install flow will begin to develop into something more real and we need to keep these issues in mind.
Comment 2 Susan McCourt 2007-09-27 12:37:39 EDT 
One more point.  The entry point API/concept does not necessarily have to be visible at all to a user.  In fact, I'm trying not to make it very visible in the code either (ie, clients can filter the installed content on anything they like, we happen to use entry points).

We could simply decide that when a user chooses multiple groups to install, we create an entry point for each one, name it the same name as the group, and use the same version (bug #203821).  The only reason we didn't start out implementing it that way is that we wanted the optional name to test entry point support in the admin UI, and I never built a different flow for the end user UI.  This is probably the best way to go for M3 and then we can expand on the grouping function if it makes sense.

Comment 8 Susan McCourt 2007-10-04 12:28:53 EDT 
Pascal and I talked about all this again, and here's what we came up with.  Entry points as implemented today are serving two different purposes:

1) they allow me to designate which IU's a user actually chose to install.  Another way to say this is that in the absence of a notion of profile roots, they are a way to let me create something I recognize as a root.  If I could instead tag an IU as a root, I could achieve the same thing as creating an entry point IU.
2) they allow more forgiving version ranges to be expressed for what I've chosen to install.  Rather than looking for updates to "SDK 3.3.1", the entry point can say that 3.3.x is OK.

We think that purpose #2 is really inappropriate.  Deciding how strict to be on updating or downgrading of something a user installed is really a policy decision made elsewhere.  

If the remaining goal is simply to remember the install roots for a profile, then what we need is a mechanism to "tag" an IU as a profile root.  Rather than introduce Profile.getRoots(), we think there could be a general mechanism for associating profile-specific information with an IU. 

This means that the notion of entry points, supplying an entry point name, etc. goes away.  But neither it is a UI-only concept, because web page installs, disk discovery installs, etc. also need to tag the roots.

So...if the user calls
myDirector.install( new IInstallableUnit [] {A,B,C,D}, myProfile, monitor)

then part of the process would be the tagging of A,B,C,D as roots.

We're going to prototype something in this regard.
Comment 9 Susan McCourt 2007-10-09 15:55:13 EDT 
Created attachment 79991 [details]
Patch with new director API, cleanup, and UI to match

The attached patch attempts to address the issues discussed in this bug, and also the issue of how a client can inquire about what will happen in an install/update/uninstall without actually doing it.  These issues touch the same API such that it was easier to address them both.  

NONCONTROVERSIAL CHANGES
- IDirector keeps the simple API for install, update, replace, uninstall.  This API is modified to remove references to an entry point name.
- IDirector2/DirectorResult/Oracle are placed by IPlanner/ProvisioningPlan.  The idea is that you always talk to a planner if you want to know what can be done or what will happen.  You can run a sizing phase, or examine the operands, or whatever you need to do.  If you like the plan, then you can have the engine run its operations for you.
- NewSimpleDirector and SimpleDirector2 are refactored into SimplePlanner and SimpleDirector.  SimpleDirector is implemented using SimplePlanner to reduce code duplication.  SimpleDirector's API merely calls the planner and runs the result.
- When the director install API is called, the director marks the IU's being installed as roots after it has the engine run the plan
- If an install root is replaced (via director API or directly via the engine), the profile registry listeners handle the transferring of the install root property to the updated IU (more on this later...)
- If you are doing an install by getting a plan first and then running it, you must mark the install roots yourself, since the engine knows nothing about install roots.
- The UI always obtains a plan in advance.  Right now it is still the same UI, but the whole idea is that now I can start providing more info since I have a plan in hand.  

CONTROVERSIAL CHANGES??
Marking an IU as a profile root seems necessary, but the implementation is a bit general and "in your face" as far as the API goes:

- the tagging is done by introducing "Installable Unit Profile Properties."  This API surfaces in IProfileInstallRegistry and Profile.  When we first discussed it, Pascal and I like the generality of introducing profile-dependent properties on an IU rather than introducing the notion of an install root as a first class concept. 
- the implementation of these properties is done with a new map in the profile registry that maps IUs to an OrderedProperties.  If an IU is removed from the profile, its properties are removed.  The properties are created lazily
- when an IU is uninstalled, the profile registry event listener looks for the install root property and if it is present, transfers it to the IU that is replacing the root (second operand).  This is a little weird because at the time it happens, the second operand has not yet been installed in the profile, yet the profile is now carrying properties for it.  You can't wait until the second operand gets installed because the properties from the first are removed when the first operand is removed.  Note that the registry will not be persisted in this state because it only persists on the CommitEvent, and by then the second IU is in profile.  The code is commented to point out the oddity.  This is weird, but the workarounds (temp storage of the property during transition) were just as weird and more complex.
- Because the tag is stored in the profile registry, it is not persisted in the rollback profile.  See bug #197700.  

ALTERNATIVES
- I considered having the moving of the root marker occur in its own phase after an IU is updated, but ran into the same timing problem.  By the time an IU is updated, its profile-related properties are gone.  We'd have to have an action that ran in the install and in the uninstall phases that snapshotted and restored the root property, and this seemed heavyweight.
- I considered having the planner keep track of root IU's as part of a provisioning plan.  However we discussed the fact that there shouldn't be additional IDirectorAPI to run a plan, so a caller of the plan API would have to mark the roots.  Since they know the roots anyway, there's no value in adding that knowledge to the plan.
- We could force callers to set the root property on IU's after an update rather than have the profile registry do it, but it seemed nice to do this automatically.
