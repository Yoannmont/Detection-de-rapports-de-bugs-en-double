Description Lynne Kues 2003-12-03 18:28:58 EST 
Version: 3.0.0
Build id: 200311211210

I have a project that has an Ant script.  The Ant script uses a temporary 
folder that is defined underneath the project.  When the Ant script starts, if 
the temporary folder exists, it will be deleted and then recreated.  At the end 
of the Ant script, a call to the eclipse.refreshLocal task is used so that you 
see the temporary folder and its contents in your project after the ant script 
runs.

.cvsignore includes the name of this temporary folder.

If I run my ant script TWICE, the second time CVS will pick up the temporary 
folder and all of its contents as resources that have changed and need to be 
committed.
Comment 1 Lynne Kues 2003-12-04 11:02:35 EST 
More information.  The script does a command line checkout of a project and 
puts that in the temporary folder.  So the temporary folder contains CVS 
directories.
Comment 2 Lynne Kues 2003-12-04 11:11:48 EST 
Changing my script to delete the CVS directories solves my problem (i.e., the 
second time I run the script the temporary folder is not picked up as needing 
to be committed).

Comment 14 Michael Valenta 2004-12-02 12:31:46 EST 
The main cause of this issue is the inability of CVS to efficiently 
communicate new subdirectory trees. We've had to keep track of all pruned 
directories in order to make the synchronize operation responsive. Not adding 
folders to the CVS/Entries file would hav a potentially severe impact on 
sychronizing. I think the only way to get around this is to give the user to 
explicitly ignore directories that are shared and these will be excluded from 
all CVS operations including synchronize.
