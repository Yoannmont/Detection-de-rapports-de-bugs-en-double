Comment 2 Semion Chichelnitsky 2003-01-21 05:37:06 EST 
If alignment depends on mirroring, it is a problem, that should be solved 
separately (see bug #29782). For example, if we create rtl-oriented Label with 
LEFT alignment, currently we should define SWT.RIGHT style for it. Say "right" 
in order to receive "left" - it is not very friendly, is it?
Now let us assume, that this problem is solved and we always receive the same 
alignment, that was defined. Also assume, that default alignment(it means, if 
it was not defined explicitly) already depends on currect orientation of Label 
and is determined as LEFT for LTR and RIGHT for RTL orientation of it. All 
seams OK in simple case, when Label is created with it's own orientation and 
alignment's styles, but what we can do, when orientation is inherited from 
parent? For example, our rtl-oriented Label has LEFT alignment, it is a child 
of Shell, and orientation of this Shell is determined (not changed!) in the 
runtime. Inside of Label it is not understood, if developer really wanted to 
create widget with left alignment or SWT.LEFT style was defined, because it is 
the only way to provide alignment of the Label's contents to trailing edge of 
the control. In first case alignment should not depend on orientation of 
widget, but in second case, to provide correct mirroring, alignment of native 
control should be changed to opposite.
What we can do here, is to determine alignment of Shell's children directly in 
the external application, but it is not seems very friendly too, is it?
On the other hand, adding new styles (with SWT.LEADING as default, 
independently of current orientation if control) give to developer possibility 
to receive such alignment, as he want:
----------------------------------------
Style                     Real alignment 
----------------------------------------
LEFT_TO_RIGHT | LEFT           left
RIGHT_TO_LEFT | LEFT           left
LEFT_TO_RIGHT | RIGHT          right
RIGHT_TO_LEFT | RIGHT          right
LEFT_TO_RIGHT | LEADING        left
RIGHT_TO_LEFT | LEADING        right
LEFT_TO_RIGHT | TRAILING       right
RIGHT_TO_LEFT | TRAILING       left   
LEFT_TO_RIGHT                  left
RIGHT_TO_LEFT                  right
                               left
Comment 3 Steve Northover 2003-01-27 12:00:03 EST 
I propose that we do what Microsoft suggests.  From the MSDN entry for 
SetDefaultProcessLayout() they suggest aliases rather than adding new bit 
values.

This is from the MSDN:

Once the LAYOUT_RTL flag is selected, flags normally specifying right or left 
are reversed. To avoid confusion, consider defining alternate words for 
standard flags, such as those in the following table.

Standard flag Suggested alternate name 
WS_EX_RIGHT WS_EX_TRAILING 
WS_EX_RTLREADING WS_EX_REVERSEREADING 
WS_EX_LEFTSCROLLBAR WS_EX_LEADSCROLLBAR 
ES_LEFT ES_LEAD 
ES_RIGHT ES_TRAIL 
EC_LEFTMARGIN EC_LEADMARGIN 
EC_RIGHTMARGIN EC_TRAILMARGIN 

Following this approach, we would define SWT.LEAD to be the same value as 
SWT.LEFT and SWT.TRAIL to be the same valu as SWT.RIGHT.  Application code 
could use these new constants to avoid code that looks strange in a mirrored 
environment but they would be aliases.

Your thoughts?  This approach would be more consistent with the operating 
system mechanism, in this case MS Windows.

Comment 6 Steve Northover 2003-01-29 12:23:23 EST 
I still believe that we should add only aliases.  This is exactly what the 
platform is doing.  There is code already written that is using SWT.LEFT.  If 
we force SWT.LEFT to always be on the left, regardless of orientation, this 
code will no work with mirroring.

The main idea behind mirroring is that applications written for LTR can be 
mirrored and work 99% RTL.  Forcing SWT.LEFT to always be on the left would 
break this.  Defining SWT.LEAD == SWT.LEFT would allow new applications to be 
written so that the code didn't look strange in a BIDI/MIRRORED world however, 
code that is dealing with coordinates will already look wrong in a MIRRORED 
world because it will have references to points such as (0,0), be written as 
though this is the top left, yet at run time, it will draw top right.

I understand that you are interested in introducing a mechanism such 
that "left" will always be "left" and "right" will always be "right" but such 
a mechanism cannot be done using the existing constants because this breaks 
mirroring.

Given that we cannot change orientation on the fly right now and that we are 
committing to mirroring as the API for SWT and that the number of applications 
that need "left" to always be "left" in a BIDI world are small, I believe we 
should either not add the constants at all or add aliases.

Your thoughts?  Perhaps discuss this with Mati or others from IBM Middle 
East?  It's critical to note here that we would be providing capability over 
and above what the operating system does without a clear requirement other 
than it seems sensible that "left" should be always "left" or that we need a 
mechanism to force this.  MS does not provide one.  Thanks.

Comment 9 Matitiahu Allouche 2003-01-31 07:05:21 EST 
This is an interesting discussion, to which I would like to add my 2 cents.
Steve writes: <quote>It's critical to note here that we would be providing 
capability over and above what the operating system (Windows) does without a 
clear requirement other than it seems sensible that "left" should be 
always "left" or that we need a mechanism to force this.  MS does not provide 
one.</quote>
While what the platform provides is an important reference, I don't think that 
it should be the determining factor for SWT, which after all is supposed to be 
platform-independent.  Also, there is a good chance that a developer using 
Eclipse has no less a Java/Web background and expectations than Windows-based 
ones.  In Swing, "Left" and "Right" alignment are absolute and are not changed 
by the value of ComponentOrientation.  "Leading" and "Trailing" exist for 
location relative to the orientation. In HTML, "Left" and "Right" also have an 
absolute meaning and are not affected by the value of the "dir" attribute.  
There is no equivalent to "Leading" and "Trailing" for alignment in HTML, 
which is too bad.  In any case, I think it is established that the 
requirements in this bug report are not a whim stemming out of nowhere, but 
reflect concepts anchored in standards and embodied in real-life 
implementations.

Here is what I suggest:
a) SWT.LEAD and SWT.TRAIL will be defined with the same values as current 
SWT.LEFT and SWT.RIGHT (this is Steve's proposal).  SWT.LEAD should be the 
default when no alignment is explicitly specified.
b) New values SWT.ABSLEFT and SWT.ABSRIGHT will be defined and will specify 
absolute Left and Right.
c) Documentation will be updated to recommend use of LEAD and TRAIL when 
relative behavior is wanted and ABSLEFT and ABSRIGHT when absolute behavior is 
wanted.  The use of LEFT and RIGHT should be deprecated.

It seems to me that this proposal does not break any existing application, 
does not cause any surprise to anyone, and provides the necessary tools to 
Bidi-aware developers.

Comment 11 Matitiahu Allouche 2003-02-03 05:58:18 EST 
a) I will not be stubborn about the names I proposed for SWT.ABSLEFT and 
SWT.ABSRIGHT.  Out of Steve's proposals, my choice goes to SWT.ABS_LEFT and 
SWT.ABS_RIGHT ("SWT.ABSOLUTE_xxxx" is lengthy to my taste).

b) We agree about the meaning and the implementation of SWT.LEAD and SWT.TRAIL 
(aliases for the current SWT.LEFT and SWT.RIGHT).

c) Here is an example of use for absolute alignment.  Imagine that I have a 
group of items that I want displayed in successive lines, e.g. items in a 
catalog.  Some items have English names, others have Hebrew names.  The 
English ones must have SWT.LEFT_TO_RIGHT while the Hebrew ones must have 
SWT.RIGHT_TO_LEFT.  However, to avoid a zigzag effect, I want all items 
aligned on the same side, which can be achieved only with absolute alignment, 
or by fiddling with the alignment in the application code for each item.  I 
prefer that work to be done by the widget, because widgets are created to make 
developers' life easier, and because that will make the orientation more 
transparent to application code.

d) I agree that the absolute alignment capability can be added at a later 
date, which is not a reason to delay it unduly, especially if it is not a 
heavy effort to implement.

Comment 15 Felipe Heidrich 2003-04-29 16:13:39 EDT 
Here is code that does the aligment.  The application must set the right-to-
left flags in order to get a right-to-left widget.  At that time, the 
application can also set the desired alignment.

public static void main (String[] args) {
	Display display = new Display ();
	int shellOrientation = SWT.LEFT_TO_RIGHT;
	Shell shell = new Shell (display);
	shell.setLayout(new GridLayout (1, false));
	
	GridData data = new GridData ();
	data.widthHint = 90;
	Button b1 = new Button (shell, SWT.LEFT_TO_RIGHT | SWT.PUSH | SWT.LEAD);
	b1.setText ("English1");
	b1.setLayoutData (data);
	
	data = new GridData ();
	data.widthHint = 90;
	Button b2 = new Button (shell, SWT.LEFT_TO_RIGHT | SWT.PUSH | SWT.LEAD);
	b2.setText ("English2");
	b2.setLayoutData (data);
	
	data = new GridData ();
	data.widthHint = 90;
	Button b3 = new Button (shell, SWT.RIGHT_TO_LEFT | SWT.PUSH | 
SWT.TRAIL);
	b3.setText ("додсЧхФ");
	b3.setLayoutData (data);
	
	data = new GridData ();
	data.widthHint = 90;
	Button b4 = new Button (shell, SWT.LEFT_TO_RIGHT | SWT.PUSH | SWT.LEAD);
	b4.setText ("додсЧхФ");
	b4.setLayoutData (data);
			
	shell.open ();
	while (!shell.isDisposed ()) {
		if(!display.readAndDispatch ()) {
			display.sleep ();
		}
	}
	display.dispose ();
}
