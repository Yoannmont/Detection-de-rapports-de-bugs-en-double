Description Mark Drew 2004-05-11 16:00:29 EDT 
I have observed three different cases in which auto refresh causes file
deletions to fail.  I'm using a polling delay of 30000 (the default, I believe).

First case: In a shell (outside of eclipse), when I try to do a *clean* Ant
build, the build frequently (almost always) fails because it cannot delete the
build directory.  However, I can usually work around this by simply attempting
the clean build again--the second attempt always works.

Second case: I just recently noticed that I sometimes have to delete a project
twice in order to successfully delete the project (along with its resources). 
The first time the deletion fails, having deleted only some of the projects
resources.  The second time, the deletion completes successfully.

Third case: (similar to case 2) When I try to rename a project, the new (copied)
project is created fine, but the deletion of the old project fails with only
some of its resources actually deleted.  The old project no longer shows up in
Eclipse, but its directory remains on the file system.

In all three of these cases, the symptoms disappear when I turn auto refresh off.

Comment 20 John Arthorne 2004-06-17 11:40:42 EDT 
To be clear, this is only reproducible when changes have recently been made
external to the workbench? If this is the case, I suspect we will not be able to
solve it. Here is what probably happens: Autorefresh discovers the external
changes, and refreshes those resources. The refresh results in a resource change
event. Various plugins react to resource change events and do processing on the
changed resources. Some examples:

1) CVS plugins update CVS metadata information in CVS folders
2) Java plugin search index re-indexes changed files
3) Java type cache parses files to build supertype information
... probably other examples ...

If any of these plugins are reading the file when the deletion is attempted,
then  the deletion will fail. I suspect this would be most likely with Ant
scripts that create or modify files, and then delete those same files within the
script. If autorefresh detects the changes within that window, it would not
surprise me if the deletion fails.

Comment 26 John Arthorne 2004-12-14 12:28:58 EST 
Autobuild will run when *any* files in the workspace change.  However, the Java
builder does not respond to deletion of class files (i.e., this won't trigger a
compile from a Java perspective). The Java builder considers its class files
(and the output folder in general) as private property - it reserves the right
to delete anything in there, but is not expecting anyone else to be mucking with
the files in there.

Comment 28 John Arthorne 2004-12-14 17:27:38 EST 
I don't think the stack trace in comment #24 is related to this particular
issue.  Those exceptions are being captured while an editor is being closed and
involve failure to delete markers.  This is not related to any failure to delete
resources which is what this bug is about.

Re "resource manager can't cope with this": as specified on
IResource.deleteMarkers, a CoreException is thrown if you try to delete markers
on a non-existent resource.  You can choose to capture this exception and do
what you like with it (ignore it, log it, etc).

If the bug only appears with the XMLBuddy plug-in installed, it might be the
case that XMLBuddy is listening to the resource change event caused by the
auto-refresh, and is then trying to read the file contents. If delete happens
while another thread has the file open, deletion fails.  This is a Windows
limitation.  The editor "quick diff" feature had the same problem, and they
worked around it by reading the file in a background job, and using the IFile as
the job's scheduling rule.  This prevents the job from running concurrently with
a deletion in another thread.

Please reopen if the problem is reproduced with a plain Eclipse install with no
extra plug-ins.

Comment 31 John Arthorne 2004-12-16 15:04:11 EST 
> It sounds like you nailed the problem. Now, how to fix it?

As I said in comment #28, the only solution that I know is to disallow file
reads concurrent with file writes and deletes.  We can't enforce this inside the
implementation of getContents(), since doing locking where we didn't previously
do locking would break existing plugins. However, a plugin doing reading of a
file can use a scheduling rule on the file to prevent concurrent read and
delete. This can't be done from within a resource change listener, but can be
forked into a separate job (this is what quick diff does).

class ReadJob extends Job {
  public ReadJob(IFile file) {
    super(" ... some name ...");
    setRule(file);
  }
  public IStatus run(IProgressMonitor pm) {
    ... read the file and do whatever processing is required
  }
}

Then in the resource change listener:

new ReadJob(file).schedule();

It's not pretty, but it would work.  Of course, the standard Windows solution is
to just let the deletion fail in this case.  The user can just try deleting
again later.
