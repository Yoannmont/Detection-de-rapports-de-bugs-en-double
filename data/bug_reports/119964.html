Description Kathy Chan 2005-12-08 14:57:39 EST 
This is related to the problem reported in bug 118673.  The problem is when a Web project containing deployed Axis Web service is added to a new server, the Axis servlet does not know about the deployed Axis Web service unless server-config.wsdd is there.  We provided a fix in bug 118673 that copies server-config.wsdd from Tomcat's temporary publish directory to the workspace.  However, we will need a more elegant mechanism to feed the wsdds in a Web project to the Axis Admin Client.  This could be done automatically when a Web project is added to a server or via a Web services pop-up.

Comment 2 Brice Ruth 2006-04-24 17:34:51 EDT 
Since the Axis AdminService is available to any project using the AxisServlet, it seems like the elegant thing to do would be to call org.apache.axis.client.AdminClient, passing the location of the particular project's AdminService (e.g. -lhttp://localhost:8080/${project_name}/services/AdminService) and providing the deploy.wsdd that already exists in WEB-INF/*Service/ws/deploy.wsdd. The code would simply need to use the context root, instead of project_name. Other than that, just call this for each deploy.wsdd that's found in the project. That should be simple enough. Since the deploy.wsdd files are already being created and Axis already provides an easy way of registering the services, this seems like a no-brainer. Since the registration w/ Axis is quick, I would recommend doing this on each re-publish, if a web-service (deploy.wsdd) is present.
Comment 3 Kathy Chan 2006-05-31 16:31:32 EDT 
We are currently calling Axis AdminClient with the wsdd file to generate server-config.wsdd.  The problem is this file is generated in the server's publish directory, which in Eclipse, is not in the workspace.  I've been using a server tooling API IModulePublishHelper.getPublishDirectory() to get the publish directory given an EAR and a module.  This has been implemented for Tomcat but other servers need to support it too so that the server-config.wsdd can be copied into the workspace and can be "persisted" when the server do the next publish.

However, as Tim Deboer pointed out:

This support is internal and will not work in several cases, e.g. remote servers or any case where the tools do not directly publish the files to the server.

We cannot just rely on a pop-up on a project or a WSDD file to call the Axis AdminClient so that a server-config.wsdd is generated because it would disappear with each server publish. The proper solution will be to generate server-config.wsdd and put it in the workspace ourselves.  The server-config.wsdd file contains information that's in the wsdd file plus some entries specific to the Axis servlet.

Comment 8 Kathy Chan 2007-03-28 14:01:51 EDT 
Created attachment 62260 [details]
Patch

Here's a patch that uses the Apache Axis Admin class rather than the Apache AdminClient.  Here's what it does:

1. Construct an AxisServer engine and populates it with the existing server-config.wsdd file from the targetting project if it already exists.

2. Process the deploy.wsdd generated in the service scenario (bottom-up and top-down).

3. Get the resulting server-config.wsdd and write it back to the WebContent/WEB-INF folder of the targetting project respecting file resource overwrite preference.

Since AxisDeployCommand now writes the resulting server-config.wsdd back to the workspace, there's no need to call GeronimoAxisDeployCommand and CopyDeploymentFileCommand anymore.

Comment 10 Kathy Chan 2007-05-17 10:11:09 EDT 
The patch works for bottom-up scenario.  For top-down scenarios, it works when the WSDL file does not refer to complex types.  However, when there's complex types, there's an error with unable to load the type mapping class.  For example, with AreaService.wsdl, there's an exception while trying to load org.tempuri.AreaService.Dimensions.  This is because the type mapping classes are not in the classpath of the AxisEngine.

I further found out that if I run a bottom-up scenario with the same Web project first before running a top-down scenario with complex types, then the top-down will work.  So it looks like the bottom-up scenario changes the classpath of the AxisEngine to include the classpath of the Web project.  I need to look into this in more detail to see how the classpath for the AxisEngine can be changed programmatically.

Since this is getting late for fixing tis in WTP 2.0, defering to WTP 3.0.
