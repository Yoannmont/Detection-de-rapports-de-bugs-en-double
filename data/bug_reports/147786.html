Comment 19 Jacob L E Blain Christen 2006-10-30 21:04:27 EST 
I experience this behavior with Eclipse 3.2.1 and WTP 1.5.x on an updated ubuntu 6.06 amd64 system.

First off, however, I had to go to the trouble of downloading the libswt-*.so dso's from the head (3313 and 3314a both work for me) and putting them in the library path somewhere that java will pick them up (symlinked *-3314.so to *-3235.so).  I had to do this because there is no libswt-mozilla-gtk-3235.so with the amd64 eclipse distribution for a gcc4 system (there is one for gcc3).  With mozilla 1.7 installed this works magically and even the license agreement dialog works as expected.  If I set MOZILLA_FIVE_HOME=/usr/lib64/firefox, however, all of the webish features (welcome page, internal browser) in eclipse work as before *except* the license agreement: eclipse saturates the cpu until shutdown and the lack of an agreement button as described in the original bug report ensues.

Incidentally, the lack of an agreement button happens when eclipse is configured to use the default system browser (which by default on my system was how things were until I figured out the .so trick) BUT it does load the agreement in the external browser!  I consider this behavior an artifact of the overarching bug reported firstly here because the "I Agree" button does not work and there is apparently no sane "Plan B" for Eclipse to walk me through.

As an aside, is it necessary for this dialog to be webified?  I don't use many non-eclipse.org plugins so maybe I am naive on this point.

And lastly, if you would like to see a more detailed description of the steps I went through to get the firefox and/or mozilla control to work in eclipse you can visit this link: http://www.urcode.com/confluence/x/9iQ (I hope my 144kbps can handle it! hah)

Comment 25 Lawrence Mandel 2006-11-06 11:33:04 EST

Patch to fix the license dialog disabled accept button problem

OK. I've reproduced the problem on SLED 10. Seems that there are linux/browser combinations for which the browser widget will not update the progress monitor unless a URL is set (even if the contents of the browser are updated via method calls). I've created a one line fix for this problem (amazing that it was one line after all this time).

I've attached a patch with the fix. I'd appreciate testing to confirm the fix from any of you that have experienced this problem.

Comment 27 Lawrence Mandel 2006-11-10 00:58:25 EST 
Created attachment 53604 [details]
Updated cache plug-in jar

Hi Oliver. Not a silly question at all. If anything I shouldn't assume everyone knows how to test patches. :)

I've attached a jar that contains the fix. To test the fix, download this jar and place it in your eclipse/plugins directory. (You may need to remove the existing cache plug-in. The easiest way to do this is to simply add a file extension such as .bak to the end of the filename.) Launch Eclipse again and try creating a new Dynamic Web project to see if the cache dialog works as you'd expect. (If this doesn't work the first time try launching Eclipse with the -clean flag when launching from the command line.)
