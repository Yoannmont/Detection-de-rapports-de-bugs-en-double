Comment 4 Markus Keller 2006-03-16 14:07:33 EST 
I found some problems with the OpenAndExpand Action:

(1) CommonNavigatorManager#init() registers an OpenListener that executes the retargetable open action even if the action is not enabled. It should only call run() if the action is enabled.
    commonNavigator.getCommonViewer().addOpenListener(new IOpenListener() {
        public void open(OpenEvent event) {
//Missing:if (openAction.isEnabled())
            openAction.run();
        }
    });

(2) Our OpenAction is enabled for IFile and IStorage as well. These cases should probably be disabled if the action is used in the Project Explorer.

(3) With the fix from (1), doubleclicking non-java nodes does not expand/collapse items with children. This only worked because the new JDT action OpenAndExpand expands/collapses any selected item. There needs to be some other mechanism that also works without the jdt contribution.

(4) PackageExplorerActionProvider#fillContextMenu(..) has
    //fOpenGroup.fillContextMenu(menu);
I guess that should not be commented out, right?
Comment 5 Michael D. Elder 2006-03-16 21:37:55 EST 
The OpenListener can be modified as you suggest. 

You will also want to ensure that you don't set the open listener unless it is applicable (in the PackageExplorerActionProvider.fillActionBars()). If no Open action is defined against the ICommonMenuActionConstants.OPEN id, then the framework will expand nodes on double click (or whatever the user's open preference is set to).

For IFile, the JDT extension takes over the rendering of the content, and the package explorer action provider takes over the configuration of actions. Therefore you should handle open events on IFile, IStorage, and any other Java cases that you want to support.

You should be able to remove the fOpenGroup.fillContextMenu() line. The open and open with actions are handled by an ActionProvider already defined by the Resource extension. These use the standard Eclipse Open/OpenWith actions.

Comment 7 Markus Keller 2006-03-17 10:40:12 EST 
I've released the second patch to HEAD with these changes:

- added overrides="org.eclipse.ui.navigator.resources.actions.RefactorActions" to the <actionProvider> tag of the JavaNavigatorContentProvider contribution. Without this, I get duplicate reorg actions (CCP, move, rename) when I open the context menu on a non-java file.

- changed OpenAndExpand#expand(..) to:
    if (! fOpenAction.isEnabled())
        fViewer.setExpandedState(target, !fViewer.getExpandedState(target));
This is more in line with what the Package Explorer does, and it does not expand a CU on doubleclick all the time.

Remaining problems:
(1) from comment 4: OpenAction should only be run if it is enabled (maybe even needs a setSelection(..) before)

(2) If I don't add fOpenGroup.fillContextMenu(menu), I don't get the Open and Open With... context menu item. Should the default support for this already be in HEAD, or will that come later?
