Description Litrik De Roy 2006-04-06 04:08:22 EDT 
Both standalone help and infocenter call org.eclipse.help.internal.standalone.Eclipse.ensureEclipseExeExists() during startup.

This method looks for an executable called "eclipse.exe" which makes it impossible to use this in combination with a branded RCP product where the executable can have any name.
Comment 1 Curtis d'Entremont 2006-04-06 10:49:57 EDT 
I think this is ok for standalone and infocenter (but not workbench mode). The reason is that the end user shouldn't be invoking or seeing the executable. End user uses infocenter by pointing their browser to a website, and they see standalone by summoning help from a non-eclipse application. In the case of standalone there is a stripped down eclipse that contains only what you need for serving help, and the application calls standalone help to display, which invokes the eclipse.exe for the user.

So wanting to brand the executable in this case is like branding internal code that the user should never see. I suggest using eclipse.exe. For workbench mode, it should not have this hardcoded (this would be a bug).
Comment 2 Lee Anne Kowalski 2006-04-06 12:19:02 EDT 
Hi Curtis, I don't quite understand your response. Are you saying "it's OK", meaning that the current behavior is OK the way it is?

I may be off-base, but isn't there a -noexec parameter for the command line invocation of standalone help/infocenter, that by-passes the eclipse.exe when starting in standalone/infocenter mode? Would there be a way that for RCP apps to take advantage of that?

--Lee Anne
Comment 3 Curtis d'Entremont 2006-04-06 12:31:59 EDT 
You're correct Lee Anne. RCP apps shouldn't be using standalone help, they should be using integrated help (so called workbench mode). Standalone help is only for non-eclipse apps. As for infocenter, the only reason I can think of for wanting to do this would be if you ship a dual-purpose app that can run as both a GUI app and an infocenter and you don't want to ship an extra copy of eclipse. If you're using the infocenter to just host some help content on a server, I don't see why you would need a custom launcher. Am I missing something?
Comment 4 Litrik De Roy 2006-04-07 02:48:44 EDT 
(In reply to comment #3)
> RCP apps shouldn't be using standalone help, they
> should be using integrated help (so called workbench mode). Standalone help is
> only for non-eclipse apps.

In this usecase there are 2 RCP applications involved. One has its executable branded as "AppWithGUI.exe" and consists of a large number of plugins (including the ones to offer integrated help).

The other application is command line only, has its executable branded as "CommandLineApp.exe" and consists of a subset of plugins of the other (GUI-based) application. The help plugins are part of this subset.

Users have the option to install only the command line application. So now my question is: How can a user who only has the command line app installed read the help that is included? 
IIUC, I'll need to include an eclipse.exe because the standalone help is not able to run without it.

Or is there any other way? If I have to, I'm willing to create another RCP product/application and brand is as "Help.exe". But the Java API to launch standalone help goes back to the command line to start eclipse.exe so I'm not sure this will get my any closer to my goal.

> As for infocenter, the only reason I can think of
> for wanting to do this would be if you ship a dual-purpose app that can run as

Our use case for the infocenter is that we want to embed it in an RCP-based server product. When the server starts (using our branded "Server.exe" I want it to start the infocenter as well so that people can use their browser to read the documentation.

IIUC, I'll need to include an eclipse.exe because the infocenter is not able to run without it.

> If you're using the infocenter to just host some help content on a

See above. I want to be able to start and stop infocenter from within our own RCP with a branded executable.

I hope this clarifies what I would like to achieve.
Comment 5 Lee Anne Kowalski 2006-04-07 10:25:44 EDT 
Hi,

it to start the infocenter as well so that people can use their browser to read
the documentation."

Are you using the infocenter startup commend in Server.exe? 
If so, then why couldn't you add the -noexec parameter to that startup command? I believe then bypasses looking the eclipse.exe for starting infocenter mode (Curtis will have to correct me if I'm wrong. But you could try it and see if you still get the "ensureEclipseExeExists" call when the -noexec param is added).

For your standalone help use case, what command is the command-line program using to invoke the standalone help?

--Lee Anne
Comment 6 Lee Anne Kowalski 2006-04-07 10:31:23 EDT 
OK, I just did a quick test with Eclipse 3.0.1. I cut out eclipse.exe from my eclipse directory and then used the following command line command to start up in infocenter mode:

java -classpath eclipse\plugins\org.eclipse.help.base_3.0.1\helpbase.jar org.eclipse.help.standalone.Infocenter -eclipsehome eclipse -port 8888 -command start -clean -noexec

The infocenter starts up fine, and I can point my browser to it and view the running infocenter. I got no errors in the command prompt window. Log file looks OK.

I will try a sniff test with 3.2, but unless the Eclipse UA team has changed that part of the code between 3.0.1 and 3.2, I predict the same result.

--Lee Anne
Comment 7 Curtis d'Entremont 2006-04-07 10:36:20 EDT 
Thanks Litrik, I understand your use case better now and I can see some value to it.

Lee Anne is correct, you should be able to run without an eclipse.exe by using the -noexec. However I now see this option isn't available for standalone help. Need to investigate why this is the case; in the meantime, does -noexec do the trick for you in infocenter mode?
Comment 8 Lee Anne Kowalski 2006-04-07 11:10:13 EDT 
Hi Curtis,

FWIW:
When I cut out eclipse.exe from eclipse directory in Eclipse 3.1.1 and run command-line for standalone help using the following command (with -noexec)
.\jre\bin\java -classpath eclipse\plugins\org.eclipse.help.base_3.1.0\helpbase.jar org.eclipse.help.standalone.Help -eclipsehome eclipse -command displayHelp -host 127.0.0.1  -noexec

I get my typical standalone help launch: system browser launches and help system is loaded.

--Lee Anne
Comment 9 Litrik De Roy 2006-04-10 03:27:09 EDT 
There's good news:

Using the -noexec parameter I am able to start both the standalone help and the infocenter in my packaged RCP app based on Eclipse 3.2 M5.

So thanks for all your help so far.

But also some bad news:

I've been thinking a little bit more about starting standalone help and the infocenter throught the command line and I don't think it's going to work.

A) I need to construct the command line to make sure org.eclipse.help.base is on the classpath. However, I can never be sure which version of the jar should be used. The user might have used update manager to upgrade the jar and there could be many versions installed in the plugins directory.

B) An additional issue with the infocenter is that it will start a new JVM which not really necesary.

I guess I'll have to figure out a way to start the standalone help components without going through the command line. Even if that means copy pasting some code from internal Eclipse classes.

Or does any of you have a better idea?
Comment 10 Curtis d'Entremont 2006-04-10 12:20:10 EDT 
> A) I need to construct the command line to make sure org.eclipse.help.base is
> on the classpath. However, I can never be sure which version of the jar should
> be used. The user might have used update manager to upgrade the jar and there
> could be many versions installed in the plugins directory.

- Have you looked at the org.eclipse.help.standalone.Help and Infocenter classes? They are useful when called from java code and they'll manage some of the launching for you. From what I see you only need to provide the eclipse home directory and not the plugin path.

> B) An additional issue with the infocenter is that it will start a new JVM
> which not really necesary.

- You're right. Currently it's set up to run in its own JVM with minimal plugins, which is simple but less efficient. We had not encountered a case before where it was required to run an infocenter in the same JVM as a workbench, for example. At the very least, you can try using the org.eclipse.help.standalone.Infocenter API class and make this an implementation detail of the help system. Then if/when in the future we allow running it in the same JVM, you won't have to do anything.
