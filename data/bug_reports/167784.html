Description Willian Mitsuda 2006-12-12 22:53:50 EST 
Hi folks, I think I finally discovered the root cause of the dreadful startup bug ("thread
"Worker-X" timed out waiting (5000ms) for thread "Worker-Y" to finish starting bundle
"org.eclipse.mylar.tasks.ui").

Warning: this is a very long bug... In fact there are
a set of bugs working together to make this happen (or not, according to the situation;
explanation bellow):

1 - Mylar plugins are doing expensive operations on Bundle.start():

I
see that tasks.ui plugin do many heavy-weight operations like reading the tasklist (which
involves XML parsing) on plugin start() method.

This type of operation must be made
on a background job because the platform has a timeout of 5 seconds for each plugin initialization.
The BundleActivator.start() javadoc explicitly says that this method MUST return quickly.

2
- Mylar plugins are doing expensive operations on Display.asyncExec():

Taking a quick
look, I see that at least context.ui, tasks.ui and mylar.team invoke Display.asyncExec()
{ [long-chain-of-method-calls] }.

The problem here is that asyncExec() forces the Runnable
to be executed on UI thread. If the Runnable takes much time to complete, this causes
UI to freeze. Probably this is the cause of UI freeze on Eclipse initialization I see.

The
worst case here is on tasks.ui. Instead of simply wrapping a code block inside asyncExec(),
we must track down the call stack and identify what methods requires to be executed on
UI thread, and wrap just them with asyncExec.

3 - There is a circular dependency between
context.core and tasks.ui:

This is the worst bug. Tasks UI depends on context.core
== OK. However, there is a "involuntary" dependency of context core on tasks UI, because
tasks UI provides a context store extension, so context core activation triggers tasks
UI activation while loading context store extensions, and vice-versa.

I think the easiest
way to fix this is to move the filesystem-based context store implementation to its own
plugin like I suggested on bug#160356 (I'm suposing that the filesystem context store
does not have any dependency on tasks UI classes).

This way, this new plugin (let's
call it "context.filesystem.store") would depend only on context.core. Context core activation
would only trigger context.filesystem.core activation. Tasks UI activation would trigger
context.core and so on.

Conclusion: there is a lot of problems contributing to make
this bug happen sometimes.

Sometimes, depending on VM classloading policy, and because
of busy BundleActivator.start() implementations, bug (1) can be triggered causing the
plugin init timeout.

Most times, bug (2) is causing bug (3) to NOT happen. I suppose
that asyncExec() would be masquerading the circular dependency, otherwise tasks UI would
trigger context core activation, which would trigger tasks UI, and so on. Probably the
"trigger" code would be inside a asyncExec() block, so it is scheduled to run on UI thread,
letting the plugin complete initialization.

However, on some VMs/OSs the thread policy
can be different, making the deadlock/plugin timeout happen.

Taking a quick look at
my bug list, I can remember strong candidates for being dups of this:

- bug#159706
- mylar fails to start on BEA jrockit VM
- bug#167320 - More startup errors after upgrading
to 1.0.0 test

To investigate:

- bug#167639 - 1.0 Errors on Startup.
- all bugs
containing "thread "Worker-X" timed out waiting (5000ms) for thread "Worker-Y" to finish
starting bundle blablabla" on stack trace.

So, fixing these 3 problems would end the
timeout problems.

Comment 10 Mik Kersten 2006-12-19 19:11:53 EST 
I tested various scenarios of (1) and (2) by adding a 30s sleep to both the start() and the asyncExec in the start method of the TasksUiPlugin.  Neither of those caused this failure and resulted in a very slow, but correct startup on both Sun and BEA VMs for both 3.3 and 3.2.  So I think that the underlying cause is (3), and the underlying problem is related to Platform bug 5875.  I believe that our circular dependency is correct, but the circular class loading can time out as described on that bug.

Willian: we can not simply move this extension point to another plug-in because the location of the context store is a preference that comes from the TasksUiPlugin.  And the Task List has to be loaded completely before any mylar.tasks.ui view or editor can show tasks, and note that the workbench restores editors on startup.  These and other issues related to walking this fine line between loading what we need eagerly (e.g. connectors) and deferring everything else to be loaded lazily is what has made this so tricky.  What I have done for 1.0.1 is to break the circularity by removing the extension point based way of specifying the context store, and instead having TasksUiPlugin.start set it explicitly in the code.  Since I can not reproduce this, I tested with a full Callisto install and did not see any problems.

Nathan and others seeing this problem: could you please verify by installing the 1.0.0.v20061219-1530 build from one of the following update sites?  Make sure to use "Search for new features" when updating.

  download.eclipse.org/technology/mylar/update-site/test/e3.2
  download.eclipse.org/technology/mylar/update-site/test/e3.3
