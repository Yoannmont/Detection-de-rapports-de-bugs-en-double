Comment 13 Chris Recoskie 2004-06-30 15:39:23 EDT 
Answer to Leo's question:

If you do not specify an output flag, no output flag is passed, but the name 
of the output file is still passed.

E.g. myCompiler foo.o foo.c

If you specify the output flag (e.g. -o) then you get the usual thing...

mycompiler -o foo.o foo.c

Comment 15 Chris Recoskie 2004-07-14 10:48:03 EDT 
Some additional feedback:

Re: options,

- Currently we have some options which take string arguments, but which have a 
default value that the user will get if they only pass the option without a 
parameter (e.g. "Generate a dependency list for the source file into the file 
filename.pp   Optionally, name the output file.").  Currently it is not 
possible to facilitate this as if the string field is blank, the command does 
not get passed.  I propose that we add boolString option type, which would 
essentially be a checkbox to turn the option off/on, coupled with an edit box 
to edit any optional parameters.  I know that I could do this with a custom 
editor but I think there is enough demand for us to create a built-in editor 
for this.

- Does it make sense to support dependant options? (options which are only 
valid if other options are set).  I do not have a current need for this but I 
am wondering if others might.

Re: Help,

-  Should we have help support for option categories or tools?  What about 
help for the entire toolchain?

Grab bag:

-  I want to have support for switching out toolsets (i.e. keep your 
toolchain, but use new executable tools within it).  Our users often get 
updated release of our code generation tools which they want to switch back 
and forth between.  They might get updated tools, but not a new IDE product, 
and will want to upgrade the CG tools.  Or, they might get a new IDE but they 
are in the middle of their project and cannot switch CG tools (this is very 
common in the embedded world) so they will want to point the IDE at their old 
tools.

The thought I had was to have an extension point mechanism that would allow 
for the registration of sets of tool executables.  Users could then select 
from a GUI which CG tools they wanted.  The CG tools plug-in would specify 
target architecture and a "compiler family" to allow the MBS to determine 
compatibility between tool-chain definitions and installed tool executables.

This was work I was planning to do at some point (I was hoping for our current 
product but probably not until after now...) so I don't mind being prime on 
that feature and comitting it, depending on timing.

-  JUnit testing.  This isn't a product requirement per se but I think we 
should spend some time getting some more comprehensive unit tests put together.

-  All default tool-chain implementations segregated into their own plug-ins 
to facilitate their easy removal for ISV products based on CDT.  Right now I 
have to fork the UI plugin in order to remove all the GNU definitions, as the 
GNU definitions don't make sense for my product.

-  We should specify requirements for the makefile generator(s).  Obviously we 
need the current GNU generator but I think we also need a more generic 
generator that does a pretty good job on arbitrary tool definitions.  The user 
would still be able to resort to creating their own generator but this should 
be looked upon as a last resort to be used by users whose toolchains fall 
under obscure or unusual corner cases.  Right now everything non-GNU requires 
a custom makefile generator.

Comment 17 Leo Treggiari 2004-07-28 10:13:26 EDT 
Sean raised a question at the end of yesterday's meeting asking whether the 
Option, Command attribute needed more sophisticated output formatting (I'm 
paraphrasing, so the question might have been somewhat different...).  I 
thought of a simple extension today that may solve most of the common problems.

The 2 problems that I am aware of are:
 o  A required space between the command and the value
 o  Quotes required around the value - e.g., for when the value may contain em 
bedded spaces

For String and StringList options (and any specializations of these option 
types that we may define...), the Command attribute may contain %s to indicate 
where the option value should be placed in the generated option string.  If no 
%s is present in the Command attribute, the option value is placed after the 
specified Command attribute value (i.e., the current behavior).

Examples of Command attribute values that handle the 2 cases above:
 o Required space:   /foo %s
 o Required quotes:  /bar:"%s"

Below are my responses to Chris' comments:

>Re: options,

>- Currently we have some options which take string arguments, but which have a

We discussed this a little yesterday.  Visual Studio (and other UIs) often use 
2 separate options to handle this.  The first option is a boolean that 
indicates  whether or not a particular "output" is generated, and a separate 
string option is used to override the default name of the "output".  
We could take this approach in the MBS, except that special handling of the
command output is then required, because you don't want to have the "command"
generated twice on the command line when the boolean is set and a string
value is entered by the user.

The problem with defining a new type is that the type would then require
additional type-specific attributes (e.g., there are separate UI "prompt"
strings required for the "check box" and the "edit box" in the UI).  We 
already have at least one option-type-specific attribute in the Option element
 - commandFalse is only used with boolean options (mea culpa - I added it...).
I'm wondering whether there is a better alternative to loading up the Option 
element with option-type specific attributes.  For example, could the Option 
element contain option-type-specific child elements for these option type 
specific attributes?  Then, an optional "booleanOption" element child could 
specify the commandFalse attribute when that needed to be used for a 
particular boolean option.

>- Does it make sense to support dependant options? (options which are only 
>valid if other options are set).  I do not have a current need for this but I 
>am wondering if others might.

Yes, definitely.  Not including it was an oversight on my part.  I suggest
adding an "isVisible" attribute.  It specifies a class name that implements
an interface with a method that returns an enumerated value.  The values
include:
 o  VisibleAndEnabled - the option is visible in the UI and enabled (this is 
    the default when the isVisible attribute is not specified)
 o  NotVisible - the option should not be displayed in the UI
 o  Disabled - the option is displayed in the UI but is disabled (the value
    cannot be changed)
The logic for supporting the dependent options is then up to the tool-chain
integrator.

Re: Help,

>-  Should we have help support for option categories or tools?  What about 
>help for the entire toolchain?

Those make sense to me.  I'll add appropriate attributes to the proposal.

providers.  My current thought is a toolChain attribute that specifies a 
class name that implements an interface to be defined.

>-  JUnit testing.  This isn't a product requirement per se but I think we 
>should spend some time getting some more comprehensive unit tests put 
>together.

Certainly the current tests need to continue to work in order to test backward

Comment 19 Sean Evoy 2004-07-28 16:32:51 EDT 
This is a comment in response to Chris' point on the behaviour of the makefile 
generator in the face of no output flag (#13). As Chris pointed out, if there 
is no output flag defined, the Gnu makefile generator will create a line like
<command> <flags> <outputFlag><output> <inputs> 
if it is a build target and 
<tool_command> <flags> <outputFlag>$@ $< 
if it is a rule.

In the case of a target, this works fine with Gnu tools like the archiver which 
does not want an output flag (but does expect the archive name to be part of 
the command line), and the assembler/linker which require there to be an output 
flag.

Since some tools prefer to decide the name of the output based on the name of 
the input (or some other heuristic out of our control), we need some way to 
format what goes into the make/jam/cmake/whatever file. This goes back to my 
final question of the kick-off meeting, namely do we need additional syntax in 
the description of an option or tool to handle these types of subtle 
differences? Alternatively, should we just offer an extension point that lets 
the makefile generator querry a class supplied by the integrator that answers a 
properly formatted output for the tool?

Leo has raised two points that I completely agree on. We need to know whether 
or not to insert spaces after the ouput flag, and when to quote values with 
embedded spaces. We also need a way to describe what to do if there is no 
output flag. Some toolchains may need the inputs to be specified before the 
output(s). I am not sure if we can come up with a way of encoding all of the 
possible variations in the manifest or not. I am still thinking about this.
Comment 20 Leo Treggiari 2004-07-29 09:31:39 EDT 
What about adding a "commandPattern" attribute to the tool that specifies the 
command line using replaceable arguments.  This is similar to what I proposed 
for generating option commands (%s).  

I don't know what the best syntax for the replaceable arguments would be, but 
let's assume that the replaceable arguments would be delimited by <>.  The 
default for a tool command line would be as it is today:

<command> <flags> <outputFlag><output> <inputs> 

But the tool could change this pattern to suite its needs.  For example, if 
the "inputs" need to come before the "flags" and there needs to be a space 
between the "outputFlag" and "output", then the tool would specify:

<command> <inputs> <flags> <outputFlag> <output> 

All other characters in the pattern would be output as-is.  For example, if 
there needed to be braces around the flags and parenthesis around the output:

<command> {<flags>} <outputFlag>(<output>) <inputs> 

Would this mechanism be able to handle what the "outputPrefix" attribute 
currently does?

On a different issue, does the "outputFlag" really need to be handled using 
unique attribute(s)?  If we use the concept of "special" options being 
recognized using the name-alias mechanism, as in my proposal, could the output 
flag simply be a normal option with a well-known alias name?

Comment 34 Chris Recoskie 2004-08-18 11:28:25 EDT 
Further feedback re: project versioning.

Currently in the MBS you can specify your managed build revision, and if new 
revisions of the MBS come out, your projects will get converted to the new 
revision.

I am currently running into a serious problem in that I need to somehow be 
able to handle versioning of projects with respect to my own toolchain.  As 
time marches on, bugs are fixed, CDT-based products updated, etc., the number 
of options and the representations of existing options changes in my 
toolchain.  As a result, upon going to an updated toolchain, some of the 
options get "lost".

E.g.

1.  Optimization level was a string option
2.  user sets optimization level equal to "3"
3.  Toolchain is updated.  Opt level is now an enumerated option.
4.  User cleans the project.  Since the old option no longer appears, 
optimization gets set to "None".

I would like to see in the toolchain schema a way to specify a toolchain 
converter class that will handle converting the project to a new version of 
the toolchain.

The IToolChainConverter interface would provide the following methods:

1.  GetConvertableToolchains - returns a list of IDs of toolchains and their 
versions which the converter is able to convert.

2.  ConvertProject - take a project and a version number as input, and convert 
the project of the specified version number to the latest version.  

Upon discovery of new toolchains, all applicable projects currently in the 
system are prompted for conversion.  If an existing project is imported, it is 
also prompted for conversion.

Potentially, it may or may not make sense to want to convert between different 
toolchains, not just different versions of the same toolchain.  I am not 
thinking of this so much in terms of converting between disparate toolchains

Comment 36 Chris Recoskie 2004-08-20 11:04:43 EDT 
Interesting issue I'm also running into...

Re: defined macros.  Some of our compiler options, if turned on, will cause 
the compiler to automatically turn on certain macros in the preprocessor.  
E.g. if I turn on the --stric_ansi compiler option, this causes __STDC__ to be 
defined in the code.

Right now I'm going to have to hack around this by making something which 
listens to changes in the build options (specifically to my boolean --
strict_ansi compiler option) and responds to changes to it by adding/removing 
__STDC__ to the list of defined macros as appropriate.

If I don't hack around it, users can create valid code, but the syntax checker 
will highlight errors that don't exist because it doesn't have the true 
version of the macros as the preprocessor sees them.

I suspect that there are other tools (GNU included) which suffer from this.

There are two ways to go about this:

a) options can specify changes to the various special option types that they 
contribute, e.g. an option can specify macros it defines/subtracts, include 
paths it adds/removes, etc.

b) Options can specify changes to arbitrary options that they can make

A related issue is that we actually have an option --undefine which is the 
inverse to --define, that is, it undefines a macro.  As you can see this all 
gets messy very quickly.

I know we talked about turning options on/off based on the values of others 
but it seems that we are going to need to support modifying other options in 
arbitrary ways.

I want to advocate solution b.  Having one big build change listener with a 
huge switch statement (or worse... a gazillion listeners that each get 
notified each time an option is changed) is very ugly.

Ideally, when defining the option in the manifest, I want to specify a change 
listener class for that option.  The listener would have access to the build 
info, and hence could then go in and modify any other options as appropriate.  
Similarly, clients could register programmatically to receive event 
notifications only for changes to the specific options they are interested in.

I think there is also a need as per my recent post on cdt-dev for a separate 
notification mechanism that sends a delta of all changed options for those 
clients that care about the presence of *any* change to the options, or which 
care about large batches of options and thus it would be easier for them to 
handle things in the classic switch statement style rather than individually 
registering for a gazillion different options.  It also makes it handy if they 
care for whatever reason about the options having been set at the same time, 
as opposed to seeing those updates serially as a sequence of separate event 
notifications.

Comment 41 Leo Treggiari 2004-10-25 11:29:02 EDT 
I have submitted our initial patch for the Managed Build System (MBS) work
for CDT 2.1.

New Functionality:

  New model for manifest file and .cdtbuild file:
    o Old model manifest files continue to be supported.
    o The new model should be used in the future since not all new 
      functionality will be available from the old model.  In general, no 
      significant effort will be expended in making new functionality work
      with the old model manifest files.  However, in some cases, new
      functionality will automatically work with old model manifest files.
      The new functionality in this patch appears to work with old model 
      manifest files.  The GNU tool-chain definition has been updated to use
      the new model.
    o Old model project (.cdtbuild) files are converted upon approval
      by end-user.  Converted project files will no longer work with CDT 2.0.
 
  Support for file-specific tool command line options:
    o A specific file in a configuration can use different tool options
      than the rest of the files in the configuration.
    o A specific file can be excluded from the build of a configuration.

  New attributes that add flexibility to tool command line generation:
    o Tool.commandLinePattern:  Specifies a command “pattern” that indicates 
      how the parts of the command line are used to create the entire command 
      line.  The pattern consists of the replaceable variables COMMAND, FLAGS, 
      OUTPUT_FLAG, OUTPUT_PREFIX, OUTPUT and INPUTS.  The default command line 
      pattern is 
      ${COMMAND} ${FLAGS} ${OUTPUT_FLAG}${OUTPUT_PREFIX}${OUTPUT} ${INPUTS}
      White space and other characters are significant and are copied to the 
      generated command.
    o Tool.commandLineGenerator:  The name of a class that implements 
      IManagedCommandLineGenerator (to be defined in the ISV document).
    o Option.command:  The command provides a “pattern” for specifying where 
      the value should be placed for options of type string and stringlist.
      The pattern can contain the replaceable variable VALUE.  If no ${VALUE}
      is specified in the command, the option value is appended to the end of
      the specified command (as happens in CDT 2.0).

  Modified UI:
    o Error parsers are no longer selectable in the project creation wizard.
      This is because error parsers are set per-configuration, and 
      per-configuation changes are made via the property page.
    o C/C++ Build page of MBS project properties now contains all of the
      per-configuration settings for the project.  There are 4 tabs for 
      Tool Settings, Build Settings, Error Parsers, and Binary Parsers.
      Note that binary parsers are not yet applied per configuration because
      the rest of CDT is not currently set up to handle it.
    o File properties now contains a C/C++ Build page where the tool options 
      for a particular file can be set.


To Do:
    o There remain a few functionality problems that need to be worked on, but
      I am not aware of any regressions from the CDT 2.0 MBS functionality.
    o End-user documentation needs to be contributed.
    o ISV documentation needs to be contributed.
    o Additional JUnit tests for the new model.


How you can help:

After the patch is applied, please test your existing tool-chain definitions 
with the new code.  All existing integrations should be able to use the file 
specific tool option functionality.  If you can use the new command line 
generation functionality, try that out.  If you're ambitious, try converting 
your tool definitions to use the new model.
