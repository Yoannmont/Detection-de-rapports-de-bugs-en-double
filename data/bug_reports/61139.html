Comment 7 Jeff McAffer 2004-05-08 00:30:53 EDT 
Doug and I chatted about this.  The summary is that under his work model (which 
is quite reasonable) people are effectively "tweaking" the install all the time 
without telling the configurations.  We certainly could implement more rigorous 
checking and timestamping in the runtime but this is potentially very expensive.

Stepping back however, the problem is more fundamental.  Eclipse does not 
increment the plugin version numbers to indicate changes.  If we did then 
installing a new eclipse would effectvely be "adding a bunch of *new* 
plugins".  Rather than changing in place something we have already processed, 
there would be a mess of new things that would be picked up by Update and 
installed into the framework.

In product scenarios this is pretty much covered since product teams likely inc 
their version numbers when they ship something new.  Doug sees this because he 
is closer to the lifecycle and gets all the intermediate builds.

The eclipse plugin id and versioning design point is that the tuple uniquely 
identifies a set of bytes.  We have been violating that model and should stop. 
PDE build contains support to automatically tagging plugins and features with 
the appropriate "build number" depending on the build context.  Turning this on 
is a trivial (though widespread) change.  The net effect is that all plugin 
versions and their dirs names would have appended a stamp. For example the 
runtime dir
    org.eclipse.core.runtime_3.0.0
would become
    org.eclipse.core.runtime_3.0.0.v20040505  <--- that is the CVS tag

The big issue with this (I forgot when talking to Doug) is that when we ship 
3.0, there is a strong desire for the plugins to be actually called 3.0.0. 
Unfortunately, that sorts before the above mentioned version and so all the 
intermediate builds would appear to be newer than the actual release.  you also 
then have to consider how the versions work going forward (post 3.0).

The net is that we can and should do something about it but likely will not be 
able to for 3.0.  It is too disruptive and given the timeline, there is not 
enough gain.  

I'm going to close this as wontfix.  The workaround is to use -clean when 
running the first (or every ifyou like) time after you have installed a new 
Eclipse.
Comment 8 Douglas Pollock 2004-05-08 08:34:57 EDT 
Toward the end of our conversation, it became clear that this bug has little 
or nothing to do with stale configuration directories.  (I'm still undecided 
as to whether the plug-in versions should be extended in such a way, though I 
think Linux should be distributed with "check" on by default.)  It pointed to 
a timing issue in starting up the OSGI runtime.

Examining the code where the NPE occurs, it points to a situation in which 
memory reads and writes are possibly occurring out of order.  My machine is a 
Pentium-4 configured as an SMP machine in Linux.  Therefore, multiple threads 
can run concurrently.

An interesting article to read about this is "Synchronization and the Java 
Memory Model" ("http://gee.cs.oswego.edu/dl/cpj/jmm.html").  Bill Pugh is 
apparently the main guy to talk to about this kind of stuff, but I've always 
found his website ("http://www.cs.umd.edu/~pugh/java/memoryModel/") kind of 
hard to follow.  To quote from the first article:

"Among other consequences, it is bad practice to make available the reference 
to an incompletely constructed object (see §2.1.2)....  Similarly, if you 
create and start a new thread T and then create an object X used by thread T, 
you cannot be sure that the fields of X will be visible to T unless you employ 
synchronization surrounding all references to object X. Or, when applicable, 
you can create X before starting T."

Reading this article, I can envision a possible explanation for the behaviour 
I saw.  A thread, T1, accesses the debug classes -- forcing plug-in loading.  
It gets far enough along in the plug-in loading to mark the plug-in as loaded 
(so that other threads can access the classes).  T1 is then interrupted.  T2 
comes along and access the debug classes, but, as it has not passed a 
synchronization barrier, it sees only some of T1's memory writes.  T2 tries to 
access the debug plug-in instance, but this happens to be one of the missing 
writes; a NPE is thrown.

Now, I have heard that the Java Memory Model has been revised.  But I'm not 
sure what revisions were made, or which version of the JVM the new 
implementations appear in.  So, I could be pointing you guys at stale 
information.

Also, I've made an assumption that there a synchronization barrier is either 
missing, or that it isn't doing quite what one would intuitively expect.
Comment 9 Jeff McAffer 2004-05-08 22:21:32 EDT 
Just to be clear about this, the intent of the runtime and the autoactivation 
code is that no one sees classes from a plugin before that plugin is fully 
activated.  However...

- the activation code itself sees the plugin in an uninitialized state
- there is nothing we can do if the plugin makes itself available to other 
objects (or threads) before it is ready to do work.
- if activation takes too long we have opt'd for reporting a warning and 
returning the requested class. This avoids deadlock cases that were hounding us 
previously.

There are really three stages of initialization. 
1) Plugin class loading and running of clinit()  
2) Plugin class instantiation and running of init()
3) plugin activation and running of start()

On autoactivation we attempt to run all three of these before giving any 
classes out to others.  We detect the first load of a class for the plugin 
(before we actually try to load it). Then lock the start mechanism and do all 3 
steps.  It is not until step 3 is completed that we release the lock.

If another thread comes along to during this process and tries to load a class, 
it too thinks this is the first load of a class for the plugin and tries to 
activate.  It then sees that another thread is activating and it waits.  It 
will wait for upto 5 sec.  After that, we allow it to just go ahead and load 
the class.  

This last characteristic CAN allow people to see classes that violate the 
invariant defined above.  The flip side of this is deadlock.  I don't recall 
the scenarios but we were seeing deadlock if we blocked indefinitely waiting 
for the activating thread to finish.  Note however that this is not the issue 
here as you would have seen messages related to this in your log.

Pascal, can you look through this again to ensure that we are thread safe?  I 
am a little concerned about safety in true SMP scenarios (think atomic 
writes/reads of ints and pointers...).  If Doug can reproduce
- try to tweak the timing such that it does NOT happen (if you cannot stop it 
happening then it does not seem like a timing issue)
- try to figure out what exactly is going on.

In all of this it is still possible however for the plugin being activated to 
fork threads with objects which are not yet initialized (e.g., the Plugin 
object). This is, as you note, bad practice but not an Eclipse thing.
