Description Martin Oberhuber 2006-10-31 07:15:18 EST 
Open RSE perspective on Solaris-motif.
Select Local system, Shells > New Shell.
The shell output is printed correctly, but no prompt is shown.
I'm using a tcsh shell locally.

When I type "ls" several times, it works OK; after command completion, a prompt is shown about 70% of the tries, the other 30% I see no prompt.
When I export the shell contents to a file, I do see the prompt.

This looks related to bug 142953 and might be a problem with shell event handling. I consider this a major issue since it is very disturbing for the user not to see a shell prompt.

Connecting to a remote Linux host with a tcsh through ssh shows the following:
* Initially, not all output is printed and no prompt is shown
* After pressing ENTER once, the rest of the output is printed, prompt is shown
* When typing "ls" a prompt is always shown
It looks like shell update event handling is timing dependent, and fails on the (very fast) local shell but works on the (slower) remote shell.

Is this something that can be reproduced on non-Solaris clients?
Comment 2 Martin Oberhuber 2006-11-02 04:19:42 EST 
It looks like this is timing dependent.

Comment 3 Martin Oberhuber 2006-11-02 11:50:09 EST 
Reproduced several times now on windows and linux clients, with dstore and other hosts that output from a remote shell would not be printed. This is critical, thus increasing to P1.

Tried content assist on a dstore-windows shell, like
dir te<ctrl+space><enter>
Later, on a dstore-linux shell, typed
ls te<ctrl+space><enter>
ls

--> didnt see any output at all, though lots of files were in the directory.
When pressing enter several times to send empty commands, output was eventually printed.
Please review the shell output event handling!
Comment 4 David McKnight 2006-11-06 10:05:49 EST

Comment 5 Martin Oberhuber 2006-11-06 12:46:16 EST 
I don't think there is a relation to content assist.
The point is, I typed "ls" but did not see any output.

Code should be reviewed with respect to timing sensitivity / race conditions in the Refresh Events.
Comment 6 Martin Oberhuber 2006-11-08 09:44:15 EST 
Fixed the bug in SshServiceCommandShell and LocalServiceCommandShell.

The problem was, that the fix for bug 142953 had been applied for dstore only, but should have been applied for local and ssh as well.

Committers and contributors - whenever you fix a bug somewhere, please think about places where the same problem could probably also occur and check the code.
