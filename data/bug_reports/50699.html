Description Michael Valenta 2004-01-27 13:37:52 EST 
If I have a pessimistic provider and I open an editor on file, modify it and 
commit it, the file will be reset to be read-only. When this occurs, the open 
editor will no longer allow the user to edit the file. To do so, the editor 
must be closed and reopened and then a another validateEdit will be performed. 
The problem is that the editor (the file buffer actually) keeps track of 
whether it has already been validated and does not revalidate. We either need 
to provide notification to the editor that it needs to revalidate or the 
editor (file buffer) needs to keep track of the result of the validate (OK or 
no-edit-allowed) and revalidate if the file becomes read-only again.
Comment 1 Michael Valenta 2004-03-08 23:10:17 EST 
Moving to Text. I think the editor should keep track of the fact that it was 
previously validated and the file should not be read-only. If the file becomes 
read-only at some later point, the editor should revalidate.
Comment 2 David Williams 2004-03-09 00:40:46 EST 
I think its well understood (and is mentioned in original paragraph), but I 
just wanted to emphasize importance of "handling" validateEdit at the model 
(filebuffer) level ... we (downstream editors) have really had to jump through 
hoops in the past when its (closely) tied to the editor. I actually don't 
recall the details, but I think it was related to every action having to do its 
own validateEdit call. Just thought I'd mention this vague "requirement" when I 
saw this bug opened.

Comment 11 Kai-Uwe Maetzel 2004-06-07 11:20:53 EDT 
This is a regression from 2.1.x that happened when introducing file buffers 
and TextFileDocumentProvider. Editors using FileDocumentProvider still worked 
OK. There is one case outlined in bug 14469, resulting in a non-modifyable 
editor. This has been changed as well.

Completely handling state validation at the level file buffers would be nice, 
but is impossible while sustaining the current contracts. Any action working 
on an editor still would have to express its intent to modify the content. We 
could implicitly do that by having a special document that is connected to its 
file buffer. Unfortunately, making assumptions about documents beyond the 
given interfaces is not realistic as it would prevent clients from having 
there own implementations. The same scheme of argumentations can be applied to 
document providers. 

Available in I200506081200.

Assigning to Team as virual owners of validateEdit for testing during the next 
test path.

Comment 13 Jean-Michel Lemieux 2004-06-09 15:54:19 EDT 
The workflows seem better now, but I'm sending back to you to comment on the
following behavior:
1. a read-only editor is opened
2. the user puts the cursor at line 4 and types one character.
3. you are prompted to checkout and say yes
4. the character typed in step 2 is ignored, the file is marked as not-dirty,
and the cursor is placed on the next line.

I would expect the last character to be visible and the editor to me dirty. 

FYI, in the o.e.team.example.filesystem plugin there is a repository provider
that simulates a pessimistic workflow. You can use it to test your validateEdit
changes. To use it, share a project with the repository providers called "File
System Pessimistic Example" and then you can add to control the files and
checkin/checkout will toggle the read-only bit and touch the file. You can
change the behavior of the validateEdit call via the pessimistic preference
page. For example, you can force the operation to fail.
Comment 14 Dani Megert 2004-06-10 12:37:44 EDT 
This is the expected behavior for this provider since
PessimisticFilesystemProvider.hasContentChanged() calls resource.touch(null)
which sends a content changed resource delta. Since the command is called
"Checkout" the seen behavior is actually correct i.e. the file from the server
replaces whatever you have on the client side.

If you use CVS and watch/edit it works as you expected since there's no need to
checkout the file: only an edit command is being sent to the CVS server.

Moving back to Platform/Team to either close this bug or adapt the pessimistic
provider.
