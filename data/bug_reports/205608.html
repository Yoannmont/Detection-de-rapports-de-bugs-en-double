Description Andrew Niefer 2007-10-05 16:02:43 EDT 
Bug 201414 removed the load time dependency on the WS libraries from the launcher.

However, motif.aix.ppc still has these dependencies.

On AIX we load the shared module libXt.a(shr4.o), however the symbol "XtInitialize" does not exist in that module, but instead exists in libXt.a(sharedlib.o).

Attempts to load libXt.a(sharedlib.o) were unsuccessful.  _XtInitialize does exist in shr4.o, but calling this skips vendor initialization which causes problems.

Comment 5 Andrew Niefer 2009-11-18 16:20:38 EST 
Attached is a code change which will attempt to load the motif libraries dynamically if the launcher was compiled with AIX_DYNAMIC_MOTIF.

This attempt is still expected to fail because of the issues involving XtInitialize in comment #0.  The result will be no splash screen, but eclipse will run.

The makefile patch sets AIX_DYNAMIC_MOTIF and removes the linking with the motif libraries.

We can release the code change but not the makefile changes, since the makefile changes will generally result in no splash screen on AIX.
Anyone wishing to run without motif will need to recompile with the patched makefile.

These patches are still untested until I can get on an AIX box.

This is not considered to be a real fix, only a workaround with a specially compiled launcher.

Comment 8 Andrew Niefer 2009-11-18 18:05:22 EST 
New patch.
This patch introduces a new library "libeclipse-motif.so" that goes in the root of the install.  This library has dependencies on the motif libraries.  These depedencies are removed from the launcher "eclipse".

On AIX, we attempt to load 'libeclipse-motif.so'.  If motif is not installed on the machine, this will fail and we proceed without a splash screen.

If motif is present we should succeed in loading libeclipse-motif.so, we call 'eclipseXtInitialize' from that library.  This just calls the normal XtInitialize.  This works because the library is linked against motif so we avoid the problem of trying to load XtInitialize dynamically.

Comment 10 Andrew Niefer 2009-11-20 16:27:28 EST 
It seems that the sharedlib.o member of libXt.a is not a shared module, but rather a normal object file:
Could not load module /usr/X11R6/lib/libXt.a(sharedlib.o).
        The .loader section does not exist.

This implies we need a library that was actually linked against sharedlib.o (which is what libeclipse-motif.so here is).

Also, the other motif initialization fuctions (XtAppInitialize, XtOpenApplication) are all the same, so we can't just change our intialization steps.

This leads me to believe that the libeclipse-motif.so is in fact the correct solution to this bug and not just a workaround.  (Unless there existed some other library that ships with motif on AIX that we could use, I could not find one).

The trick remains finding libeclipse-motif.so at runtime.  We want to place this inside the motif.aix.ppc launcher fragment to avoid releng complications with a new rootfile.   I have been unable to find a method to determine this location internally from eclipse_<version>.so.  See http://www-949.ibm.com/software/rational/cafe/thread/2353 as an example.

As a lasty resort, the location will need to be passed in from outside, which will need to be done from both the main launcher, and also from Java which will require new JNI interface methods.
Comment 11 Andrew Niefer 2009-11-20 16:46:46 EST 
Also, we have the case when the main launcher is unable to find the companion fragment with the shared library.  Today we display an error message, however in this case we will also be unable to find the libeclipse-motif.so library and won't be able to display the error message anymore.
Comment 12 Andrew Niefer 2009-12-10 17:53:13 EST 
Created attachment 154278 [details]
good patch

Here is a new patch against 3.5.x.

Comments:
1) This introduces a new library libeclipse-motif.so.  This library is linked against motif and stored in the org.eclipse.equinox.launcher.motif.aix.ppc fragment.

2) Passing in the location of the eclipse shared library.
We need this to derive the location of libeclipse-motif.so located beside the normal eclipse library.  There are two cases:
   1) passed in from main eclipse executable.  It turns out this was already being done, not sure how I missed this before, just needed to make a global variable.
   2) passed in from java.  I was able to do this without introducing new JNI interface methods.  When java calls showSplash, we have access to the JNIBridge object, we are able to use reflection to get the value of the "library" field.

   Because we didn't actually need to introduce new interface methods, a new eclipse shared library will still work with an old executable and/or launcher.jar.  (Though of course a new executable will be required to run without motif).
