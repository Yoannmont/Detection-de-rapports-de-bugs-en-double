Comment 11 Jeff McAffer 2004-01-29 11:01:44 EST 
this is still ambiguous to me.  Dani, comment 5 #2 requires you to reenter the 
password for each session.  This is a dup of bug 50811.  But everything else 
you say implies that you see this problem only when switching builds (ie., 
comment 5 #1).  Can you clarify?

As for putting the keyring in the config area, this addresses a long held 
security problem (see bug 22220 and bug 50006).  Other people have exactly the 
opposite view where they are annoyed that they have to enter their passwords 
for every workspace.  This analogous to many of the user settings questions we 
have today.

By far most Eclipse users change workspaces more often than they change 
builds.  For the runtime workspace problem we can pass -keyring to the target 
and use the keyring from the host environment.
Comment 12 Wassim Melhem 2004-01-29 11:13:18 EST 
>By far most Eclipse users change workspaces more often than they change 
>builds
I'm not sure we can make such a bold statement.  Builds come and go, but I 
have had the same workspace for over a year.

As for the runtime workbench situation, up until now, we were deleting the 
config area because it only contained files such as platform.cfg and 
config.ini (needed by update and core respectively)that become stale very 
quickly and need to be recomputed with every launch.  Plus, if we didn't clear 
the config area at the termination of the launch, the pde metadata where we 
keep them would accumulate stale directories whose associated launches have 
been removed, but we had no way of knowing that.

As more people are apparently putting files in the config area, including the 
Help indexing, we introduced a new option to turn off the clearing of the 
config area.  It is on the bottom of the new configuration tab.  It is turned 
on by default, as this has always been the behavior.

Comment 15 Jeff McAffer 2004-01-29 11:43:00 EST 
comment 12.  There are many more people who *use* eclipse than who *develop* 
eclipse.  The users almost never get new builds/releases and have many 
workspaces (or at least it is a one to one relationship).

As for retaining the config area for runtime workspaces, yes, that should 
address the issue.  

comment 13. the workflow that includes deleting the .config area is a very 
temporary situation.

The keyring is shared and has the same concurrency issues as configuration 
level preferences and all the other configuration level data files defined and 
managed by various plugins.  This is currently handled individually by each 
plugin.  We are investigating if the runtime can provide a facility to others.

comment 14. yes.  Again, the design point is not that everyone should use -
keyring but that the runtime will manage this file in the config area rather 
than the workspace.  The config area should be in the eclipse install or 
somewhere in the user area (that is a separate discussion).
