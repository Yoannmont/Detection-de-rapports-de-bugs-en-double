Comment 13 Jean-Michel Lemieux 2003-02-20 21:44:13 EST 
Ok here is the state on this PR. The fix I released only partly addressed this
problem. Further investigation shows that the main problem lies in the fact that
the DecoratorManager is dispatching label events in the ModelContext thread:

org.eclipse.jdt.internal.ui.javaeditor.JavaOutlinePage$JavaOutlineViewer.handleL
abelProviderChanged(JavaOutlinePage.java:632)
        at org.eclipse.jface.viewers.ContentViewer$1.labelProviderChanged
(ContentViewer.java:68)
        at org.eclipse.ui.internal.decorators.DecoratorManager.fireListeners
(DecoratorManager.java:145)
        at 
org.eclipse.ui.internal.decorators.DecoratorManager.labelProviderChanged
(DecoratorManager.java:434)
        at org.eclipse.jface.viewers.LabelProvider.fireLabelProviderChanged
(LabelProvider.java:54)

All registered views are getting a chance to respond to the label changes, and
in the stack trace above the java views are doing some work. This work,
compounded by the number of views, can take some time :) All of this is
happening in the *same* thread and context as the CVS action. To really fix this
problem I believe that the DecoratorManager should post the label changes to
it's thread and once the action is complete the deco thread could run an
asynexec and dispatch the label changes at that point. Thus not intefering with
the operation that caused the label changes (e.g. providing the user with timely
operation completion) and decorate afterwards. This is essentialy what the
dedicated CVS Deco thread did in 2.0. 

Another question to pose is why should the decorator manager ask the views if
they should update *then* tell them to update? Couldn't the deco manager simply
*tell* the views to update their labels?

Those are my reasons for moving this over to UI.
Comment 14 Tod Creasey 2003-02-21 10:04:45 EST 
Marking for RC2 as this will need the approval of both Kevin McGuire and Nick 
Edgar.

The new strategy that Jean-Michel and I worked out was to begin decoration 
immediately rather than to wait for the views listening to the decorator 
manager to make a request.

Basically there are two possible assumptions
1) It is more effecient to let the listeners decide if they are interested in 
the updates
2) It is more effecient to assume that the decoration is interesting to 
someone and start right away.

1) is the strategy in 2.1 but evidence suggests that 2 is more effecient now
a) because there are more DecoratingLabelProviders than there were in 2.0
2) strategy 2 requires one less update and is more effecient in cases where 
label providers may be slow.

I advocate this change for 2.0. I will attach a patch for Jean-Michel to 
verify with.

Comment 33 Tod Creasey 2003-03-10 11:13:25 EST 
You are right Matt.

Here is the problem.

Open a PackagesView (looks at CompilationUnits) and a SearchView (looks a 
SourceTypes)

PackagesView requests decoration on a CompilationUnit. We decorate in the 
background and send an update.

The SearchView gets the update and finds that it adapts to the SourceType it 
is showing and requests decoration of the SourceType.

The resultCache is cleared after the updated

Now we service the SourceType. Send and update and the PackagesView does the 
same thing this time asking for a CompilationUnit whose result we have cleared.

This can happen anywhere with two views that are adapting a resource to 
different types which is why we see it on export and file creation dialogs.

Comment 51 Jed Anderson 2003-03-11 15:43:23 EST 
Strongly suggest we re-open this bug.

Outstanding issues:

1. resourceChanged(IResourceChangeEvent event) MUST lock on the resultLock to
perform resultCache.remove(Object)
2. UI lockup (not deadlock, just temporary freeze) during the decoration of
elements.  This occurs when decoration takes a long time, while the asynch exec
posted in the decorated() method is waiting for the resultLock.
3. Deadlock may occur when ISV code needs the UI in order to perform
ILightweightLabelDecorator.decorate(...).  This occurs when the decorator Thread
has the resultLock, the ISV code posts an asynch exec in order to decorate the
code and a previous asynch exec was posted in the decorated method.  This should
not occur because ISV's should not be running their decorate(...) methods on the
UI Thread.  However, the new patch addresses this issue by allowing the decorate
calls to occur without locking on the resultLock.
