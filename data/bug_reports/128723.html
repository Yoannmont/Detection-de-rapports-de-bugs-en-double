Description Paul Slauenwhite 2006-02-20 16:43:21 EST 
Make org.eclipse.hyades.ui.internal.wizard.exampleproject package public.

The org.eclipse.hyades.ui.sampleWizards extension point requires an implementation of the org.eclipse.ui.newWizards extension which requires an implementation of the org.eclipse.ui.INewWizard interface.  Typically, TPTP sample plugins use one of the following classes for the implementation of the org.eclipse.ui.INewWizard interface:


org.eclipse.hyades.ui.internal.wizard.exampleproject.ZipAdjusterCreationWizard
org.eclipse.hyades.ui.internal.wizard.exampleproject.CreationWizard

Unfortunately, these classes are in an internal package and should not be used by consuming products.  Consider making these classes public so consuming products may leverage their function when creating samples.

Comment 4 Valentina Popescu 2007-01-15 13:11:56 EST 
Paul, can you explain why do we need to make this function available as API to consuming products ? 

My take on this is that the function was made available internally to TPTP components as a common way of writting samples. We build on top of the Platform project and if somebody should provide an API for building samples, the Platform team should be the one. They did not do that so we went and build our own structure. Why other products consuming us can't do the same ? 
In an ideal world we should offer this and any other common functions as API to whoever wants to use them. In reality, we don't have enough resources to committ to this support; I also don't see a compeling reason on why we should make this available outside TPTP.

I will close this as wontfix so that the intentions are made clear; I am ready to have a discussion on why this is actually a must fix for TPTP
Comment 5 Paul Slauenwhite 2007-01-16 09:20:18 EST 
(In reply to comment #4)
> Paul, can you explain why do we need to make this function available as API to
> consuming products ? 

It has been a while since I have used the example infastruction in TPTP, but after refreshing my memory by looking at the code, it appears that these classes should have been originally public since they were designed to simplify the creation of example projects in Eclipse (often required when creating a new example).  In my opinion, since we provide public API to create samples, we should expose these two classes as well.  However, this is a Platform Project/AG decision.
Comment 6 Valentina Popescu 2007-01-16 09:56:47 EST 
Paul, as far as I know they were never made public, to be used outside TPTP.
The intention was to provide an easy way for the TPTP project to create Eclipse sample project; we did not intend to offer this as API to the entire world. 
If an API is to be made available to support this then it should be in the Platform and not TPTP.
Comment 7 Paul Slauenwhite 2007-01-16 10:01:52 EST 
(In reply to comment #6)

I am in agreement that the example/sample framework should be part of the platform.  Not exposing these classes is a project/AG decision which the arguments against are valid although making these two classes public would make the sample/example framework more complete.  Consuming products have the choice to copy and support these classes.  As such, closing.
