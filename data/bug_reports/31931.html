Comment 21 Steve Northover 2005-01-21 18:13:41 EST 
This is a Windows "feature".  When the control panel changes, possibly 
changing the font, SWT asks widgets to layout.  The "Find/Replace" dialog gets 
the current contents of the combo box and lays out to a larger size.  Since 
the shell is not normally resizable, the user has no way to fix the clipping.  
For its part, the "Find/Replace" dialog doesn't expect to be told to layout, 
however, the font has changed so Eclipse needs to layout.

Not sure how to fix this.  SSQ, any ideas?
Comment 22 Pepijn Schmitz 2005-02-24 03:25:45 EST 
Why is this resolved as wont fix? It seems to me to be a real bug in SWT. The
layout manager should take the size of the container into account when it's
doing the layout. That's the underlying problem; if it did that, the fact that
it's asked to do the layout again would be no problem.

This is not just an isolated problem in the Eclipse GUI, this is biting my
application as well. I have a non-resizable dialog with a list on it. This list
can get quite long, which is normally not a problem since it has a scroll bar.
But at certain moments it will suddenly be resized so that all its items fit in
it, becoming so big in the process that it will push all the other contents of
the dialog away below the bottom edge of the dialog.
Comment 23 Pepijn Schmitz 2005-02-24 04:30:20 EST 
I made a workaround by making a copy of the GridLayout and GridData classes and
using my own copies instead of the SWT version. I modifed the copies so that
they only perform the layout once, instead of every time when asked. This solves
the problem of the layout becoming corrupted later, but obviously it's only
appropriate for containers which are not resizable.
Comment 24 Steve Northover 2005-02-24 14:27:09 EST 
What?  Why copy the code?  If you want layout to happen only once, why not 
just set the layout to null after the shell has opened?

If you want the layout to take into account the size of the shell, then code 
it that way.  For example, use GridData.grabExcessHorizontalSpace etc.

Comment 27 Veronika Irvine 2005-02-25 10:35:12 EST 
From Comment #22:
>Why is this resolved as wont fix? It seems to me to be a real bug in SWT. The
>layout manager should take the size of the container into account when it's
>doing the layout. That's the underlying problem; if it did that, the fact that
>it's asked to do the layout again would be no problem.

Daniel added line 410 to FindReplaceDialog:

panel.setLayoutData(new GridData(GridData.FILL_BOTH));

This makes the panel take the size of the container.  Without this line, the 
panel takes its preferred size which is big enough to show all of its 
content.  Without this line, the panel changes size if for any reason it is 
laid out (even bogus reasons such as the one found in this bug report) and 
sometimes this means the panel will be larger than its parent and will be 
clipped.  

Line 410 is the right way to make the panel take the size of the container.  
With this line, it is safe to make the dialog resizable.  The combo widget 
will grow as the shell gets wider.
