Description Doug Moran 2006-09-11 10:37:17 EDT 
I am having an issue resolving URLs running BIRT Release Build 2.1.0 (uses Platform 3.2) in JBoss (4.0.4.GA date=200605151000).  

I use a JSP (JPivot) that through the course of execution requests resources that have a "jndi" protocol URL.  Before initializing the Platform (org.eclipse.birt.core.framework.Platform.startup())
everything works fine, after initialization, I get invalid protocol errors for jndi.  If Platform.shutdown() is called, they work again.

This issue is stopping us from including BIRT 2.1 into the Pentaho BI Platform please let me know if a workaround exists.


What I found debugging and tracing:

  - JBoss sets its own StreamHandlerFactory in the URL class ( calls URL.setURLStreamHandlerFactory() )
  - After JBoss initializes, org.eclipse.birt.core.framework.Platform.startup() is called
  - It loads OSGi (.../plugins/org.eclipse.osgi_3.2.0.v20060601.jar) and org.eclipse.osgi.framework.internal.core.Framework initializes
  - Framework.installURLStreamHandlerFactory() is called since JBoss has already installed a URLStreamHandler, the URL class throws an exception.
  - Framework resorts to reflection to circumvent URLs restricted field access and forces its own handler org.eclipse.osgi.framework.internal.protocol.StreamHandlerFactory in

  I think the issue has to do with how this handler processes protocols:

  - Line at comment "<-C--" from below fails, the VM gets a class not found error trying to load org.jboss.net.protocol.jndi.Handler
  - If I force builtInHandlers to equal "" then "<-D--" gets executed and everything is ok.

  Should parentFactory be tried first and then default to the VM?  
  Is there a configuration setting somewhere I need to set?

  Code from org.eclipse.osgi.framework.internal.protocol.StreamHandlerFactory:

        public URLStreamHandler createURLStreamHandler(String protocol) {
                //first check for built in handlers
                String builtInHandlers = secureAction.getProperty(PROTOCOL_HANDLER_PKGS);   <-A-- Returns org.jboss.net.protocol
                Class clazz = getBuiltIn(protocol, builtInHandlers); <-B-- Returns class org.jboss.net.protocol.jndi.Handler
                if (clazz != null)
                        return null; // let the VM handle it         <-C-- VM tries to load org.jboss.net.protocol.jndi.Handler but fails
                URLStreamHandler result = null;
                if (isMultiplexing()) {
                        if (findAuthorizedURLStreamHandler(protocol) != null)
                                result = new MultiplexingURLStreamHandler(protocol, this);
                } else {
                        result = createInternalURLStreamHandler(protocol);
                }
                // if parent is present do parent lookup
                if (result == null && parentFactory != null)
                        result = parentFactory.createURLStreamHandler(protocol);  <-D-- If program flow can get to here then everything is ok
                return result; //result may be null; let the VM handle it (consider sun.net.protocol.www.*)
        }


        private Class getBuiltIn(String protocol, String builtInHandlers) {
                if (builtInHandlers == null)
                        return null;
                Class clazz;
                StringTokenizer tok = new StringTokenizer(builtInHandlers, "|"); //$NON-NLS-1$
                while (tok.hasMoreElements()) {
                        StringBuffer name = new StringBuffer();
                        name.append(tok.nextToken());
                        name.append("."); //$NON-NLS-1$
                        name.append(protocol);
                        name.append(".Handler"); //$NON-NLS-1$
                        try {
                                clazz = secureAction.forName(name.toString());
                                if (clazz != null)
                                        return clazz; //this class exists, it is a built in handler
                        } catch (ClassNotFoundException ex) {
                                // keep looking
                        }
                }
                return null;
        }
Comment 1 Thomas Watson 2006-09-11 11:20:46 EDT 
Are you setting 

java.protocol.handler.pkgs=org.jboss.net.protocol

This VM property expects the prefix protocol packages to be available on the application classloader (stuff available from -classpath).  Is the org.jboss.net.protocol package available on the app classloader in your environment?

I assume in your environment jboss is loaded from a classloader that is not one of the VMs built-in classloaders (boot, ext, app etc).  Then it loads the equinox framework using the same classloader as jboss or in a separate classloader that uses the jboss classloader as its parent classloader.  This means the jboss protocol handlers are available to both jboss and equinox but not to the VMs built-in classloaders.

There are a couple of possible solutions:

1) if jboss protocol handlers are not available in the VM's built-in classloaders then do not set the java.protocol.handler.pkgs property.

2) have the StreamHandlerFactory in Equinox ask the parent factory first, before checking for built-in VM handlers

3) Change the StreamHandlerFactory.getBuiltIn method to load built-in handlers from the application classloader instead of using the framework's classloader.  In your env this should result in a CNFE causing us to search the OSGi registry and then the parent factory.

Option 1 is the current workaround, unless you are not currently setting that property.  If that is the case then I do not understand why you are running into this issue.

Option 2 will cause other issues because we need to look in the service registry for a protocol handler first to properly handle mult-plexing more than one framework in a VM.

Option 3 is probably the ideal solution because it more accurrately mimics what the VM does when searching for a protocol handler from the java.protocol.handler.pkgs property.

Comment 3 Doug Moran 2006-09-11 14:54:30 EDT 
I am not setting java.protocol.handler.pkgs=org.jboss.net.protocol.  I did a search through our jBoss deployment and could not find a file that is setting this config parameter.  

A little digging in the JBoss code found this:

   /**
    * Set up our only URLStreamHandlerFactory.
    * This is needed to ensure Sun's version is not used (as it leaves files
    * locked on Win2K/WinXP platforms.
    */
   private void internalInitURLHandlers() {
      try {
         // Install a URLStreamHandlerFactory that uses the TCL
         URL.setURLStreamHandlerFactory(new URLStreamHandlerFactory());
   
         // Preload JBoss URL handlers
         URLStreamHandlerFactory.preload();
      }
      catch (Error error){ //very naughty but we HAVE to do this or
         //we'll fail if we ever try to do this again
         log.warn("Caught Throwable Error, this probably means " +
            "we've already set the URLStreamHAndlerFactory before"); 
         //Sys.out because we don't have logging yet
      }

      // Include the default JBoss protocol handler package
      String handlerPkgs = System.getProperty("java.protocol.handler.pkgs");
      if (handlerPkgs != null) {
         handlerPkgs += "|org.jboss.net.protocol";
      }
      else {
         handlerPkgs = "org.jboss.net.protocol";
      }
      System.setProperty("java.protocol.handler.pkgs", handlerPkgs);
   }

Looks like JBoss forces this param in. If the class is not available via the System (app) classloader - is that a JBoss bug that needs to be reported?

A modified workaround 1 worked for me.  Just before calling Platform.startup() - I do a System.getProperty("java.protocol.handler.pkgs") and delete the entry for "org.jboss.net.protocol" and do a System.setProperty().  

I don't know what happens with other app servers which is a little scary.  This low level stuff is a bit out of my league.  (or at least any league I want to be in ;) )

Comment 6 Thomas Watson 2006-09-12 09:42:33 EDT 
Thanks Simon.  That is similar to what I was thinking.  After a bit more thought ... 

I'm wandering if we should just remove the getBuiltIn check StreamHandlerFactory.  On most VMs the java.protocol.handler.pkgs is not set.  This means by default we always check the registry and parent factory first for a protocol handler and return null to let the VM find one if we cannot.  If we removed the getBuiltIn check then this would always be the behavior no matter what the java.protocol.handler.pkgs property was set to.

Unfortunately we cannot do the same thing in the ContentHandlerFactory because returning null causes us to never be called again (I love the consistency!! :)  Perhaps we should move the logic to check the built in content handlers into ContentHandlerProxy.setNewHandler.  If the service reference is null then try to load and create a built in handler to use instead of the DefaultContentHandler class.
