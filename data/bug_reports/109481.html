Description Daan de Wit 2005-09-14 05:15:31 EDT 
When searching on a regular expression containing a lookahead, it is not 
possible to replace the found occurences. You can click on the 'replace'-button 
but nothing will be replaced.
I used the following regex: (?<!\w)(?<!\u002e)settings\u002e(?=[\w\d])
Comment 1 Peter VanWylen 2006-10-30 13:03:21 EST 
This occurs with several of the "lookaround" options in regex.  I have tested (on v 3.2.1) and:

(?=X) X, via zero-width positive lookahead
- Regex works properly but replace fails
(?!X) X, via zero-width negative lookahead
- Regex doesn't work properly, but replace works
(?<=X) X, via zero-width positive lookbehind
- Regex works properly but replace fails
(?<!X) X, via zero-width negative lookbehind 
- Regex doesn't work properly, but replace works
Comment 2 Markus Keller 2006-11-03 07:12:15 EST 
To reproduce, e.g. search for the regex in comment 0 in a document containing "()settings.abc".

I looked at the problem in the Find/Replace dialog and found at FindReplaceDocumentAdapter.findReplace(..) line: 176:

Pattern pattern= fFindReplaceMatcher.pattern();
Matcher replaceTextMatcher= pattern.matcher(fFindReplaceMatcher.group());
try {
	replaceText= replaceTextMatcher.replaceFirst(replaceText);
} catch (IndexOutOfBoundsException ex) {
	//..
}

The problem is that the lookahead '(?=[\w\d])' matches 'a' outside of the selection returned by fFindReplaceMatcher.group(). Therefore, pattern.matcher(..) does not find the pattern any more and can thus not replace anything in the given string.

A fix would have to do something similar to Matcher#replaceFirst(..) directly on fFindReplaceMatcher, but using find(int) to start at the right position.

Comment 7 Axel Jebens 2012-11-30 08:18:13 EST 
I have analyzed it quite a bit and here are my statements:

- As already mentioned method ReplaceRefactoring fails since the matcher does not find the pattern in the original text. The original text is just the piece of text which matches with the regular expression. But if there has been a lookahead before, it is not part of the original text any more. 

- Why do we need to use pattern matching twice? 
As you can see, the name of the method is computeReplacementString. The method is especially relevant if you use submatches of the regular expression for the replacement string, e.g. "\1something\2". Purpose of the method is to calculate \1 and \2 in this example. (When using constants as a replacement string, computeReplacementString could simply return the replacement text.)

- When doing the search, why doesn't the system store also the values of the submatches in a map so that we can reuse them for the replacement string?
This would use a lot of memory, and I am not sure if there is an API which returns us all locations found with all submatches. 

- Solution proposal:
In the first search run, we get information about all matching offsets and length of the matches. Now, we are sure that any kind of lookahead and lookbehind matches the expression. Because lookahead and lookbehind has zero length, we can (at least from my point of view) strip off all look behind's and look ahead's for the "inner" matcher. I wouldn't do that in the method computeReplacementString because it is called in a loop and we only need to do it once.
Example: Instead of using (?<!\w)(?<!\u002e)settings\u002e(?=[\w\d]) as a matcher we use settings\u002e. 
If you use the reduced regex in method computeReplacementString, it will work.

Therefore, I would introduce a new private method
private Pattern stripOffLookaheadLookbehind(Pattern originalPattern)

Then, I would call the method at the beginning of createFileChange(...)
...
Pattern replacementPattern = stripOffLookaheadLookbehind(pattern);
...
String replacementString= computeReplacementString(replacementPattern, originalText, fReplaceString, lineDelimiter);

This should do the trick. Any comments?

Comment 15 Markus Keller 2012-12-14 13:37:11 EST 
Comment on attachment 224735 [details]
Solution proposal (patch) for class ReplaceRefactoring

Unfortunately, correctly parsing a regex amounts to writing a full regex compiler. E.g. when I search for "Hello\Q(?!1)" in text "Hello(?!1)Hello", then the patch breaks the replacement. The code in getMatchingClosingBracket also doesn't look like it handles constructs like "\)" or "[()]" correctly. Moreover, the boundary matchers \b and \B should also be handled.

And finally, the lookahead/lookbehind can also contain groups, which would disappear if we just strip away parts of the regex. Example:

        String text= "--- Hello7 ===";
        String regex= "Hello(?=(\\d))";
        String replacement= "$1_";
        
        Matcher matcher= Pattern.compile(regex).matcher(text);
        System.out.println(matcher.find());
        System.out.println(matcher.group(1));
        System.out.println(matcher.replaceFirst(replacement));


I don't think we can correctly implement the replace operation without using a Matcher on the complete input sequence.
