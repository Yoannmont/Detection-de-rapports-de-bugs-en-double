Description Stefan Xenos 2003-02-20 20:06:30 EST 
I’ve been thinking about how to make adapters more general. Several of my
coworkers have encouraged me to make an Eclipse bug report about this, so I’m
attaching it here.

THE PROBLEM:

Eclipse cannot currently use the existance of plugin-provided adapters to filter
UI selections, which often requires messy workarounds (see bug #30453). This
enhancement would allow extension points to offer such filtering without eager
loading of plugins.

If Eclipse allowed menu items, property pages, etc. to only appear for objects
that supported a plugin-defined adapter, it would be a useful way to create UI
elements that act on classes that do not share a common ancestor. This is
currently impossible because Eclipse can only detect that an adapter exists once
the plugin containing the adapter is loaded, and it is likely that the UI
contributions of a plugin will be visible on menus before then.

For example, imagine some plugin declares an adapter called IMyInterface and a
popup menu that should only be enabled for objects supporting this adapter.
Let’s assume that the org.eclipse.ui.popupMenus extension point supported this
by checking if the selection was adaptable to the “objectClass” class. In this
case, the popup menu will always be disabled until some user action causes the
plugin to load. The enablement of the popup menu will only function correctly
after this point. This appears to be the only reason why the selections for
eclipse UI extensions cannot be filtered based on adapters.

Note that org.eclipse.ui.popupMenus can filter based on the existance of an
IResource adapter (using the adaptable flag in the extension's XML), and this
filtering seems to suffer from exactly this problem.

Essentially, it is not possible for IAdaptable#getAdapter to be used for
controlling visibility or enablement, since this would require every plugin that
contributes an adapter to be eagerly loaded... and this is bad. 

The order of loading adapters also causes other problems. For example, assume
that plugin A displays a list of objects, and plugin B wants to make these
objects visible in the generic property view by offering IPropertySource
adapters. This is currently not possible since plugin B’s view might be created
before plugin A loads.


THE CURRENT ECLIPSE OPTIONS:

1. org.eclipse.ui.popupMenus, org.eclipse.ui.propertyPages and others offer an
“objectClass” facility which limits the selection to objects deriving from a
particular base class. This is often useful, but it cannot be used to describe
actions that are common to objects that do not inherit from one another (ie: the
Java Build Path properties page which works for JavaProject and IProject). It
also cannot distinguish between individual instances of the same class (ie:
instances of IProject which have the JavaProject nature).

2. org.eclipse.ui.popupMenus offers an “adaptable” tag that limits the selection
to objects adaptable to IResource. This only works for particular classes and
extension points, and sufers from the plugin initialization problems mentioned
above when the adapter is provided by a different plugin than the selection
provider.

3. Many extension points offer the <enablement> tag in their extension markup.
Although this can limit a selection to objects implementing a particular
interface through the use of multiple <objectClass> tags, the entire list of
objects adapting to the interface must be included for every extension, and
these extensions will not work with classes declared in new plugins that support
the desired adapter. The <objectState> tag can be used to distinguish between
individual instances of a class.


A PROPOSED SOLUTION:

The ideal solution would be to allow eclipse to query for the existence of an
adapter without requiring the plugin that defines that adapter to be loaded.
This would require a slightly different interface than offered by IAdaptable. 

interface IExtendable {

// Determine the existence of an adapter, without loading the 
// plugin that contains it. Returns true if the receiver inherits from
// adapterType.
boolean hasAdapter(Class adapterType);

// Returns the specified adapter. May throw an unchecked exception 
// (ClassCastException?) if the adapter does not exist. Returns
// the receiver if the receiver inherits from adapterType.
Object getAdapter(Class adapterType);
}

It should be considered an error to use “getAdapter” to determine the existence
of an adapter, since this may unnecessarily load a plugin. For this reason, it
may be a good idea to throw an unchecked exception rather than return null when
an adapter does not exist. This means that “hasAdapter” and “getAdapter” would
be to adapters like “instanceof” and downcasting are to java interfaces.

In order for “hasAdapter” to work before the plugin is loaded, we need to store
information about adapters in the plugin registry. We can define a new extension
point (say org.eclipse.resources.adapters) that creates an association between
an adapter and an extendable type. 

For example:

<extension point=”org.eclipse.resources.adapters”>
  <adapterFactory class=”com.ibm.myplugin.MyFileAdapterFactory”>
    <adapter objectClass=”org.eclipse.core.resources.IFile” 
             adapterClass=”com.ibm.myplugin.IMyFileAdapter” />
  </adapterFactory>
</extension>

This XML would replace the adapter factory registration that is currently done
in the plugin’s startup() method and the IAdapterFactory’s getAdapterList()
method. The adapterFactory class must derive from IExtenderFactory, which is
basically a lightweight IAdapterFactory without a getAdapterList method.

interface IExtenderFactory {
  /**
    Returns an object that is an instance of the given class associated
    with the given object or throws an exception if no adapter exists.
  */
  public Object getAdapter(Object adaptableObject, Class adapterType);
}

The difference between this and IAdapterFactory is that an IExtenderFactory must
return an adapter for all instances of the class it is registered for. It may
not choose to only adapt particular instances. This convention ensures that the
extension point’s XML is sufficient to determine the existence of an adapter.
Filtering individual instances can be handled by other means.


WHERE WOULD THIS BE USED:

The following extension points can be updated to check for the existence of
adapters, allowing for filtering by adapter type without eager loading of plugins:

org.eclipse.ui.editorActions
org.eclipse.ui.popupMenus
org.eclipse.ui.propertyPages
org.eclipse.ui.viewActions
org.eclipse.ui.actionSets

The generic property view can use the new facilities to query for
IPropertySource adapters.

The “enablement” and “selection” XML elements appear in many extension points.
These can also check for the existence of an adapter when objectClass is used
for filtering. 

There are probably other areas where this would be useful, but these are the
ones that have been giving me immediate headaches.


MIGRATING FROM THE EXISTING SYSTEM:

This enhancement could either be used as an independent alternative to
IAdaptable, or the IAdaptable code could be migrated to this model in a
backwards-compatible manner. The former approach is self-evident and safer, but
may introduce additional special cases in code that needs to support both
IAdaptable and IExtendable objects.

One way to integrate with the existing IAdaptable code would be as follows.

1.	IExtendable will inherit from IAdaptable
2.	PlatformObject will implement IExtendable, offering IExtendable services to
many existing objects.
3.	IAdapterFactory will inherit from IExtenderFactory, allowing existing
factories to be added to extension points without changing any code.

Since IExtendable will inherit from IAdaptable, it may be desirable to change
the semantics of getAdapter in a backwards-compatible manner. I can see three
alternatives for this:

1.	Rather than returning null if an adapter does not exist, the behavior will
depend on the adapter type being queried for. When querying for an existing
adapter type the function will return null. When querying for a new adapter
registered through org.eclipse.resources.adapters, the function will throw an
exception. Wrapping getAdapter in a call to hasAdapter will always work, while
calling getAdapter and checking for a null return value will continue to work
for existing adapters but will not work for new adapters. The downside is that,
this may be confusing to programmers that are used to the existing model.

2.	OR: getAdapter will become deprecated and continue to behave as it does now.
It will never return an adapter registered through
org.eclipse.resources.adapters and will not throw exceptions. We will create a
new function called IExtendable#getExtender which is capable of finding all both
old and new adapters, and will throw an exception if the adapter does not exist.
getAdapter will continue to work, and the existing code can slowly be migrated
to use getExtender.

3.	OR: getAdapter will detect both old and new adapters. It will return null if
they do not exist and will not throw exceptions. In this case, we will rely on
documentation and coding conventions to ensure that people use hasAdapter for
filtering rather than calling getAdapter and checking for a null return value.
