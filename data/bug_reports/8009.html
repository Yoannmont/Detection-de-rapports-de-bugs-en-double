Comment 14 David Anderson 2004-04-07 10:02:17 EDT 
Once you've had the ability to open multiple editable views of the same file at 
once (as in emacs) so you can focus on different contexts while you're editing -
 switching back and forth while you're making changes - it's really hard to 
give it up. You simply want to be able to see as much at once as you can. The 
only questions in my mind for providing this feature in Eclipse regard the best 
way to represent it: using multiple file tabs seems incorrect, and which view 
gets refocussed when you auto-navigate using the outline and hierarchy views is 
a problem. It's important to do this and do it right.

Basically we want to diminish the counter-intuitive, linear-text-file 
orientation of writing using computers. People just don't create things 
linearly; they skip around to different places, making changes and refining. 
(Authoring has always been incremental; computers have just shrunk the 
increments.) The power of Eclipse is a testament to this - the ease with which 
you can move around in code lets you write software faster. The helper views 
(outline, hierarchy, dependency, etc.) promote the non-linear writing process 
by collecting, summarizing and making explicit the meta-information in code. To 
work fast and accurately, it's not enough to be able to look at different 
objects at once - you also need to be able to easily look at different methods 
of an object (or different parts of the same method) at once.

What's really being asked for is to be able get more in-context information 
while you're analysing and modifying. There are two ways to address the 
problem: through hiding irrelevant information, and through changing the notion 
of context.

One mechanism to accomplish the first could be providing "fisheye" views of 
code on top of a collapsing-expanding code-hiding mechanism (Fumas, CHI 86; 
http://www.si.umich.edu/~furnas/Papers/FisheyeCHI86.pdf). It seems to me like 
this could plug into Eclipse quite nicely. These types of views let 
you "disregard" code that is less-affected by your changes. The only gotcha is 
that specifying and respecifying focus has to be able to be done as quickly and 
painlessly as possible. It'd be really cool if it were done well.

The second could be done by the (at least partial) abandonment of the 
explicit "file" structure of code, relying much more heavily on the outline 
view as a table-of-contents used navigate to methods. (I find myself doing this 
anyway now that I have the feature.) You would view and edit individual methods 
(perhaps with all versions of the method in the override-lineage in the class 
hierarchy) - and let Eclipse handle the management of files behind the scenes. 
The concept is certainly not new (and I admit it might leave me groping in the 
dark for the familiar object-in-a-file paradigm that I'm used to) but this 
could be an extremely powerful break to make in the way programming is done. 
When I'm thinking java, I'm thinking objects, not files; but I know that 
conceptually I still have files in the mix. At this point, nothing really 
forces file-based programming except the compiler, and Eclipse could easily 
tear down those boundaries.

However, even with the development of these alternatives, the old editor 
wouldn't have to disappear. These are just different means to the same end. And 
even within these new interfaces, multiple editable views would still be 
desireable.

So, I guess first things first. Eclipse should be able to have multiple 
editable views into the same file. The sooner the better.

Comment 36 Simon Goldsmith 2004-10-01 11:13:41 EDT

Added a "binary" set of files for people who don't want to build it themselves

Open the attached zip to find changed files.

Mainly changed the AbstractTextEditor to call split function on Ctrl-2. Had to
open the openEditor call in IWorkbenchPage and WorkbenchPage to accomodate it.
Other files are for setting up the action and key-mappings. Note that these
files are all based on source files from the 3.0.1 release of eclipse. This is
not a true split editor as it doesn't split in the same pane it merely opens a
duplicate editor on the current input and you can pull the tab to anchor it
next to the original editor giving you 2 views. The nice thing is that this
plays nicely with all derived text editors. Also note that there is another
function added to the AbstractTextEditor for quickCopyPaste. It is static based
to allow any Text editor to be used as the source of the paste and into the
current editor at the cursor. It is used by Ctrl-C, Ctrl-C then highlight
whatever you want to drop into the current position.
