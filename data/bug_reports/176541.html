Description John Thomas 2007-03-06 18:45:52 EST 
One of the features in STEM 1 was that you could click on a county in the internal map and reflect the exception to the GoogleEarth interface which would cause the selected county to be displayed as a polygon on the GoogleEarth map.

If this feature is deemed to be useful,  we could define a MapSelection event that  the internalmap plugin could issue.  If the GE plugin was active and listening for these events, it could generate the desired KML and launch it to GE.
Comment 1 John Thomas 2007-03-17 19:46:16 EDT 
Comments
Comment 2 Daniel Ford 2007-03-18 14:30:23 EDT 
I don't think we need a special event.  We could simply report the "node" or "edge" in the underlying canonical graph as being selected.  It's instance would be passed to all selection listeners which would allow other views to adjust appropriately.  We would need to implement IAdaptable for the "Identifiable" interface in the "core" project for this to work.
Comment 3 John Thomas 2007-08-31 12:55:18 EDT 
*** Bug 201763 has been marked as a duplicate of this bug. ***
Comment 4 John Thomas 2007-08-31 14:13:19 EDT 
Currently the MapView code is sending a  Structured selection with the RegionImpl instance from which I can retrieve the admin area name.  I can look this up and get the lat-long border info and display the selected polygon on GoogleEarth.  
That is all sort of working.  

But I would also like to mark the area with a placemark (GE thumbtack) that gives its name and perhaps the SEIR values when selected.  But for a placemark I need a point to put it at.  But all I have is the name of a polygon and not the mouse click point.  I could pick an arbitrary point on the border but for many areas there are multiple polygons and where I place the thumbtack may be far different from the mouse click on Mapview.  Also, without an indication of which simulation this is,  I cannot reliably get the relative values.

The StructuredSelection object allows a list of elements to be specified so it would be nice if in addition to the RegionImpl instance, it passed the latlong data, the relative values and a reference to the current simulation.

By the way, this bug should probably be assigned to me.
Comment 5 Daniel Ford 2007-09-04 12:09:56 EDT 
Once you have the ReginImpl instance (which is a sub-class of Node) you can adapt it to a RelativeValueProvider and obtain the relative values.  With the polygon it should be possible to compute a "center" for the place to put the thumbtack.  There is also code to get the NLS name of a geographic location in the geography project.  See org.eclipse.ohf.stem.geography.GeographicNames.getName() and how it is used in the Map view (MapCanvas?).

Does this help?

The function that I am proposing to add is the ability of the user to click on the Mapview and have the results of the click appear both on mapview and GoogleEarth.

Comment 9 Daniel Ford 2007-09-11 12:32:10 EDT 
The patch breaks the processing of the selection event by the Reports view.  It is expecting the selection of just the Identifiable, which is the primary thing being selected and would be what we would feed into other kinds of views as well (Outline?).  I think we can provide both the Identifiable instance and the GSE instance in a single selection event however, and in that case the Reports view would simply ignore the extraneous parts.  Just needs a little investigation.  Also, the path was out of sync with the contents of the repository.  I think if we were using subversion then we could simply reference the (global) revision id as a reference point for the contents of a patch.  With CVS we may need to tag the repository with an id that can be reference by the patch.
