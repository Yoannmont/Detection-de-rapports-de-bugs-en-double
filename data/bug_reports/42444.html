Description Jeff Badorek 2003-09-03 09:28:51 EDT 
I have a situation where I am utilizing a third-party plugin (specifically, a 
WSAD plugin) which I'll call plugin "A". I wrote a plugin called "B" which uses 
one of the extensions from "A". Plugin "A" requires ["org.eclipse.ui" 
version="2.1.0" match="equivalent"]. However, I have eclipse 2.1.1 installed in 
which "org.eclipse.ui" is at version "2.1.1" and version "2.1.0" does not 
exist. Plugin "A" works correctly in my eclipse environment and even my 
plugin "B" works using its extension when running the Runtime Environment. The 
problem is when I try to build the feature. It spits up a message 
saying "Cannot find plug-in: org.eclipse.ui_2.1.0". Shouldn't this be able to 
find "org.eclipse.ui_2.1.1" since match="equivalent" was defined?
Comment 1 Jeff Badorek 2003-09-03 16:55:29 EDT 
I've done some more testing and can reproduce the problem with one plugin...
(I'm assuming Eclipse 2.1.1)

1. Create a new plugin with the "Default Plug-In Structure"
2. Modify the "org.eclipse.ui" dependency "Import Details"
3. Change "Version to match" to..."2.1.0"
4. Change "Match rule" to "Compatible"
5. Create a feature project that includes the newly created plugin.
6. Try to export/build the feature.

You will then notice the error. Is this expected behavior?
Comment 2 Pascal Rapicault 2003-10-09 10:23:19 EDT 
This behavior is expected.

The matching criterias are runtime artefacts allowing flexibility and control 
on the requirements. 

If pde buid was applying the same kind of behavior, there would be 
uncertainties against what a plugin is being compiled.

The only work around is to build a target platform (and refers it from the pde 
preferences) containing all the versions that you need that will match the 
requirements you have in your plugins.
Comment 3 Pascal Rapicault 2003-10-09 11:24:21 EDT 
Here is a concrete example where supporting matching rule can be problematic.

I'm writing plugin P which specifies "import org.eclipse.ui 2.1.0 compatible"
In my development environment I have org.eclipse.ui 2.2.0 available.

In the code of P, I make use of a new API available in 2.2. Everything compile 
and run fine.

Now when I ship my plugin P, nothing prevent it to be run on a 2.1.0 version 
(according to the matching rules) and so the customer of P will have errors 
(undefined method or classes) because P makes use of API not available in 2.1.

I sounds like if the only case where the matching could used in a safe way is 
where "equivalent" is specified, since service releases are only bug fixes.

Comment 7 Jeff Turnham 2003-10-16 16:55:23 EDT 
In cases like this where there are good arguments on both sides of a design 
point, it usually makes sense to allow both to occur.   Can an option be added 
to allow plugin dependencies at build time to be treated the same as they are 
at runtime?   The default could remain as it is today, with strict adherence 
to version numbers, but giving the user the choice would solve this problem I 
believe.

Just to add my 2 cents to the mix, I think that as Eclipse evolves as platform 
for integration of components potentially built/tested by completely different 
teams/companies, this situation will come up more and more.   The fact that 
some component of WSAD was built with version 4.0.7 of a dependency, and 
another was built with 4.0.13 of that same dependency should not force us to 
ship both versions just so that extension writers can use the PDE.   

Thanks.

Jeff Turnham (WSAD Release Team)

Comment 23 Pascal Rapicault 2004-01-21 17:24:09 EST 
Two possible fixes:
- first: resolve the registry and use the 
PluginPrerequisiteModel.getResolvedVersion() method which indicates the way 
matching rule has been resolved. Doing that will prevent the building of the 
multiple version of the same plugin (not library plugins). Although in matter 
of number of line this change is small, the semantics impact are wide.

-second: change the behavior of AbstractBuildScriptGenerator.getPlugin(String, 
String) to take into account the matching rule. The matching needs to be done 
according to the state of the art, which means that the correct plugin need to 
be searched for. This changes require more code changes (more new lines are 
being added) but is safer than the first one.

In either case a preference will be introduced to enable or disable the perfect 
matching.
