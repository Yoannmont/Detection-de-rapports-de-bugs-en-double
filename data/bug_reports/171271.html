Description Michael Giroux 2007-01-22 13:06:28 EST 
Build ID: M20060921-0945

Steps To Reproduce:
1. Write a plugin (A) that does not specify an explicit Bundle-ClassPath.  In this case, the default bundle classpath is '.'
2. Build and install the plugin
3. use some functionality of plugin A to verify it is installed and working correctly.
4. write a plugin (B) that depends on plugin A and imports a package from A.
5. compile plugin B -- JDT reports compile errors due to unresolved references to packages exported in plugin A.

More information:
The issue here is that the default classpath (if not specified explicitely by Bundle-ClassPath) for plugin A is '.' and the plugin executes correctly.

However, if you try to build another plugin that references such a plugin, JDT does not appear to search the default classpath.  That is, if Bundle-ClassPath is not specified, then JDT does not appear to search anything in the bundle.

Comment 15 Michael Giroux 2007-01-22 17:32:39 EST 
Well, now we are getting somewhere.

First, I had the plug-in dependencies filtered from the package explorer.  After turning off the filter, I see that plug-in A is not listed.

Looking at the plug-in A I see that it is built as if it should be packaged in a jar, but is in fact packaged as an exploded folder.

There is no jar file, only a directory structure for the classes:
/extension-folder/eclipse/plugins/plugin-id/**/*.class

I know the PDE build has problems if the plugin is packaged as a nested jar (classes are in a jar, and plugin is distributed as a non-expanded jar).  It almost looks like the inverse of the problem.  Here, the plugin classes are in an expanded directory (there is no jar) and the plugin itself is unpacked.
