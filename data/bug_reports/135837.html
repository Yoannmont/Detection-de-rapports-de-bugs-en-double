Description Markus Keller 2006-04-10 05:44:53 EDT 
N20060410-0010

TreeSelection#equals(..) is not symmetric with StructuredSelection#equals(..):
- TreeSelection#equals(..) is always false if the argument is not instanceof TreeSelection
- StructuredSelection.equals(..) can be true if the argument is a TreeSelection

3 Problems:
 (1) equals(..) should be symmetric, as defined in Object#equals(..)
 (2) hashCode() of both classes must return the same if objects are equal (currently, StructuredSelection#hashCode() is not even implemented!)
 (3) equality is currently only defined for the implementations. Should be defined on interfaces, since clients generally only know about the interface, not the implementation.

Comment 3 Markus Keller 2006-04-19 07:03:53 EDT 
Hmm, difficult to tell. What's sure is that two TreeSelections should not be equal if their TreePaths are not the same (otherwise, we may e.g. miss selection updates because someone thinks the old and the new selections are the same).

I guess a TreeSelection and a plain StructuredSelection should be equal if their elements are equal, but I have no crystal clear reasoning.

Two scenarios that could justify either way:
- If someone sets a StructuredSelection to a TreeViewer and then gets the selection (returns a TreeSelection), he would probably expect the two selections to be equal.
- If someone compares the effect of setting a TreeSelection and a plain StructuredSelection to the same tree, he would probably expect the two selections to be *un*equal if they don't select exactly the same elements.

Adding Dirk, just in case he already had a solution for this in the package explorer.

Comment 6 Boris Bokowski 2006-10-04 13:32:21 EDT 
(In reply to comment #3)
> I guess a TreeSelection and a plain StructuredSelection should be equal if
> their elements are equal, but I have no crystal clear reasoning.

Same here - when I wrote TreeSelection, I thought about this problem but
couldn't find a solution that was better than the one I ended up implementing.

Let's try it again.  We have six interesting cases, given two selection objects
A and B containing the same elements, when calling A.equals(B):

 |                                          |(a) A.equals(B)|(b) B.equals(A)|
-|------------------------------------------|---------------|---------------|
1| A.getClass()==StructuredSelection.class  | true          | true          |
 | &&B.getClass()==StructuredSelection.class|               |               |
-|------------------------------------------|---------------|---------------|
2| A.getClass()==TreeSelection.class        | true          | true          |
 | &&B.getClass()==StructuredSelection.class|               |               |
-|------------------------------------------|---------------|---------------|
3| A.getClass()==TreeSelection.class        | true iff paths| true iff paths|
 | &&B.getClass()==TreeSelection.class      | are the same  | are the same  |
-|------------------------------------------|---------------|---------------|

It seems that the current implementation handles 1a,1b,2b,3a, and 3b but not
2a.  Agree?

To get 2a right, we would have to call super.equals() if the other object is
not instanceof TreeSelection.

BTW, the Javadoc for SWT's getSelection() on Table and Tree says that the order
of the returned items is unspecified.  It seems that both
StructuredSelection.equals() and TreeSelection.equals() should be ignorant of
the order, but the Javadoc on StructuredSelection.equals() clearly states that
it compares the order of the elements too.  Any thoughts?

Comment 9 Alex Bernstein 2006-10-04 14:28:28 EDT 
>> We can only sort if a ViewerSorter is set.

When there is no sorters, the data is displayed to the user in the same order as the elements/children are returned by ContentProvider. I would like to call this natural order. Natural order is used when there is no any ViewerSorters. When this is the case, I expect the selection to return my elements in the same order my content provider gave them to the viewer, which is a natual order. 

Here is a very typical (but not the only) example when this is needed. Select some nodes in the tree (no sorters present) using UI method. Then prompt user if he wants to delete the selected elements. List all the selected elements to be deleted so the user can see them. The prompt should display the candidates for deletion in the same order they are displayed in the tree (without any additional sorting, because natural order is used).
Comment 10 Markus Keller 2006-10-23 13:07:18 EDT 
I basically agree with Alex on the sorting issue. I would expect the elements to be sorted in the way they are presented in the UI (top-down). However, we have to make sure the perfomance hit is acceptable -- especially for TreeViewers, where parent-child relationships have to be considered as well. But I think this can be dealt with in a separate bug.

I also agree with the table in comment 6. It's probably the least surprising solution. Code that does not know about ITreeSelection does not have to care. Code that knows about ITreeSelection can easily do a type check to sort out non-tree selections if it should be necessary in special situations.

Still, I would not use 'X.getClass() == *Selection' to test the argument of equals(), but 'X instanceof I*Selection' (and define the contracts in the interfaces, not the default implementations).
Comment 11 Alex Bernstein 2006-10-23 13:20:25 EDT 
One of the reasons I had asked for this fix was the fact that when I need to programmatically select some elements in a tree, I can only create and pass as an arugment instance of StructuredSelection (no different from pre-3.2) 

Then elsewhere my code checks if the new selecion is the same (contains same elements) as current selection in th tree. If so, some perfromace related optimizatons take place, in order to avoid unnecessary refreshes.

Unfortunately, TreeSelection(X,Y,Z) is not equal to StructuredSelection(X,Y,Z), so my optimizations don't work. I have reasons to believe that there may be more code out there that continue to compare these to classes as before. Since TreeViewer's started to return treeSelections, that code may stop working.
