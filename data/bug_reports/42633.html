Description Dorian Birsan 2003-09-05 14:26:07 EDT 
The browser only knows about a handful of protocols (http, file, etc.) so it 
is difficult to integrate it nicely into a java environment where most of the 
I/O is performed via streams. The browser can neither read from a stream, nor 
from a java URL (plugged in using the standard Java url handlers).

Do you think it is possible for the browser widget to use custom protocol 
handlers or to read from a java InputStream ?

around this by calling Browser.setUrl(myUrl.toString())

Comment 8 Konrad Kolosowski 2005-04-05 02:04:59 EDT 
Hi Christophe,

After the browser being available (and reliable) on all supported platforms, I 
see this as the next most important requirement.

Use cases include all usage of browser where displayed document contains 
(relative) links to resources inside Eclipse.  One can extract data from a 
jar, doc.zip or generate it dynamically and set on the browser, but the linked 
images, CSS, Javascript, and other document do not work (will not be found)
Examples:
About.html - runtime has special rules and workaround to extract content from 
plugin jars.
Help documents in doc.zip or plugin.jar - help uses appserver to be able act 
as a handler for the documents.
Help resources in nl or ws directories - help uses appserver to act as a 
handler and find documents.
Javadoc in zip files - JDT is delegates to help to display Javadoc
Links in Intro - Intro processes all links in XHTML and substitutes with 
custom protocol that it captures when navigation occurs.
Relative resources in Intro - Intro processes all links and resolves them to 
file protocol.  Links from CSS or resources that are not in the same nl 
directory are still a problem.

All of the above could be solved using the http protocol and appserver, but it 
is too heavy.  It is even heavy for help taste.  I do not believe anybody can 
plug-in server-less help and have a good experience because HTML relative 
resources will not work.

It is not necessary for the browser widget to understand URLs defined in 
Eclipse, it is one solution.  Another is if browser instance has a hook for 
plugging in stream handler for all relative links or some other protocol  One 
is enough.  When string is set on the browser, the URL protocol is not 
defined, but it should be possible to supply content of resources relative to 
this undefined URL.  In practice the relative links may resolve to absolute 
URL, and we can share this like a large namespace.

Forced to use external browser and appserver, help implemented its UI in HTML 
using JSPs, tying itself more to the appserver.  In setups where browser 
widget works, help UI could be reimplemented in SWT, but still cannot drop the 
appserver because of supplying content to the browser.

I think what we now have is an HTML browser that can browse resources with 
http or file protocol that have no connection to Eclipse.  We also have HTML 
Displayer that can display single HTML set as a String, but it fails to 
display linked resources or browse to links.  Setting base is not helpful if 
it must be set to http: or file: protocol.  It needs to be allowed to set it 
to some protocol that is handled by Java code inside Eclipse.
