Comment 21 Darin Swanson 2007-05-11 19:56:44 EDT 
I believe I have a workaround for most cases and most importantly for the exporting case.

The incorrect code in Locator is only hit if the URL starts with "file:/"
The default ant home JARs from Eclipse all had a URL of this type so we blew up.

Previous code that resulted in the problem:
URL url = FileLocator.toFileURL(bundle.getEntry(library));
if (new File(url.getPath()).exists()) {
...
}

Changed to:
URL url = FileLocator.toFileURL(bundle.getEntry(library));
File urlFile = new File(url.getPath());
if (urlFile.exists()) {
   url = new URL("file:" +  urlFile.getAbsolutePath()); //$NON-NLS-1$
...
}

I have released the code and will be included in the builds.
Extra verification of the correct behavior is appreciated.

Code change we localized to AntCorePreferences.
Comment 22 John Arthorne 2007-05-14 09:57:18 EDT 
Note the technique recommended in the Java class libraries to convert from java.io.File->java.net.URL is: file.toURI().toURL().  Converting to URI takes care of proper character escaping.  (See javadoc of java.io.File#toURL()). I don't know any cases off-hand where this will obtain a different result from what you have done, but I thought it worth pointing out.

If you search for "URI has an authority component", you can see that NetBeans has been bitten by this, and ant1.7.0 when run off a file share. Don't let eclipse follow.

Ant has special code for the Java1.3 and below JVM; we are thinking of switching to this always, rather than use the file.toURI() logic. It is a shame we are duplicating JVM code, but at least we get consistent behavior that we can test for.

This is the issue that is causing me to consider the current SVN head Locator class to be unstable...I recommend you add some tests for handling these cases too.
