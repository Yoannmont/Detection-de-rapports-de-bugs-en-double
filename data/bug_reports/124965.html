Description John Chan 2006-01-23 21:04:20 EST 
I am not sure whether this is a bug but was told to raise this by our IBM contact.  Anyway, basically we found that the SWT table scrolling performance started to degrade dramatically as the number of columns increases.  In our test scenario, we created a table with over 300 columns and 1000 rows on a Pentium 4 duel 2GHz CPU machine with 1GB of RAM.  The scrolling performance was rather bad (take around a second to scroll one page with a 500 x 500 window).  Further testing showed that the performance problem was caused by the number of columns instead of rows.  The same scrolling action on a table with 10,000 rows but only 60 columns was in fact quite fast.  When we increased the column number to around 80 and above, the performance started to degrade.  By the way, even with the Virtual flag did not help in this case.

Comment 2 Steve Northover 2006-01-25 09:56:06 EST 
Here is the benchmark I am using:

package steve;

import org.eclipse.swt.*;
import org.eclipse.swt.widgets.*;
import org.eclipse.swt.layout.*;

public class PR_124965 {
	static final int ROWS = 128;
	static final int COLUMNS = 300;
	public static void main(String[] args) {
		Display display = new Display();
		Shell shell = new Shell(display);
		shell.setLayout(new FillLayout());
		final Table table = new Table(shell, SWT.NONE);
		for (int i=0; i<COLUMNS; i++) {
			TableColumn col = new TableColumn(table, SWT.NONE);
			col.setWidth(32);
		}
		for (int i=0; i<ROWS; i++) {
			TableItem item = new TableItem(table, SWT.NONE);
			for (int j=0; j<COLUMNS; j++) {
				item.setText(j, i + "-" + j);
			}
		}
		for (int i=0; i<COLUMNS; i++) {
			table.getColumn(i).pack();
		}
		Listener listener = new Listener () {
			long [] time = new long [1];
			public void handleEvent (final Event event) {
				time [0] = System.currentTimeMillis();
				event.display.asyncExec (new Runnable () {
					public void run () {
						System.out.println ("Time=" + (System.currentTimeMillis() - time [0]));
					}
				});
			}
		};
		table.addListener(SWT.KeyDown, listener);
		shell.setSize(500, 500);
		shell.open();
		while (!shell.isDisposed()) {
			if (!display.readAndDispatch()) display.sleep();
		}
		display.dispose();
	}
}

Comment 4 Steve Northover 2006-01-25 10:00:31 EST 
Stupid !#$!#@%$ bugzilla submitted my code twice.  Anyways ...

The problem is that when Windows scrolls a row in the table, it asks for every string in the row, even the ones that are not visible.  The Windows message that does this is LVN_GETDISPINFO.  So, when you hit page down, despite the fact that only 30 columns are showing, Windows ask for the data for 300 x the number of visible rows.  I'm looking into ways to get around this.

Is 300 columns a typical number of columns in your table or are there usually less?
