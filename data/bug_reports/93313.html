Description Ernest Pasour 2005-04-29 15:05:40 EDT 
I am writing a content assistant using the extension point
org.eclipse.jdt.ui.quickAssistProcessors
and my content assistant does not get called.

Here is my problem scenario:

I have some code:
	public static void main(String[] args) {
		Message m=new Message("blah");
		System.out.println(m);
*		JList list=new JList();
	}

I have a lot of warnings turned on, and on the line with the asterisk, I get a
warning for "Local variable list is never read."

Scenario #1
1. I move my cursor just before the second "JList".  
2. press ctrl+1
3. My content assistant does not get called.  The only thing that pops up is the
 item for "Remove declaration of 'list' and assignments without possible side
effects.".  

Scenario #2
1. I move my cursor just before the second "JList" and select the J character
2. press ctrl+1
3. My content assistant gets called.  The proposal I add plus "Rename in file"
pop up.

Scenario #3
1. I move my cursor just before 'l' in 'list'.
2. press ctrl+1
3. My content assistant gets called.  I get 4 proposals (mine, "rename", "remove
declaration", and "split declaration")

The relevant pieces of code are copied below.

In processAnnotation, the content assistant only gets called if the offset has
not been updated (marked with *1*).

The offset gets updated with a call to storePosition (marked with *2*)

storePosition does not get called if there is a selection.  I have no idea why
that logic exists.

storePosition also does not get called if computeOffsetWithCorrection() returns
-1.  computeOffsetWithCorrection(startOffset, endOffset, initalOffset)
apparently returns -1 if there is no correction offset or if the correction
offset is the same as the current cursor position (marked with *3*).  This is
Scenario #3.  

I found defect #42086, in which the idea is discussed of having both
quickassists and quickfixes show up together.  This is certainly related,
although not a dupe IMO.

This is definitely a bug; I can't get to my quickassists because of a java
warning.  If the warning is disabled, I can get to my quickassist.

Comment 2 Daniel Aborg 2007-07-30 11:40:58 EDT 
Why is this marked a dupe of 42086? That has very little to do with this. This bug is about having quick assists and fixes shown together at all times, 42086 is about SEPARATING them to different keystrokes.

I also find the behavior reported in this bug extremely annoying, especially the inconsistency of it based on where the cursor are. As do others in our team.

We just want both quick assists and quick fixes to always show up whether there is a warning or not.

Please reopen this. It is not a dupe of 42086.
