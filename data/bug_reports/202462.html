Description Jeff McAffer 2007-09-06 10:30:45 EDT 
In bug 148782 there is an expressed need to be more flexible in the way we deliver source.  We experienced similar issues in Orbit.  Ultimately it comes down to delivering source in individual bundles where each bundle has a relatively simple layout (e.g., just a JAR of .java files in the normal Java dir structure).

to do this we need to review how PDE is currently taking and handling source contributions.  The proposed approach would significantly increase the number of bundles contributing source and the extension mechanism is a little clumsy for this scenario.  Ideally developers would not have to add a plugin.xml file to the source JAR at all.  Perhaps they can just add particular header to the manifest.mf that identifies the bundle as being a source bundle.

Note that the increase in bundle count will not actually be a runtime perfomrance issue as the souce bundles need never be actually installed since they are never run.  In the current model, source is discovered in the PDE model which in turn is based on the workspace and target platform.  Neither of these is an actual running Eclipse.  Having said that, there may well be some assumptions in the model that make this not as simple as is sounds.

Comment 2 Thomas Watson 2007-09-11 16:00:58 EDT 
I think the new p2 would figure out how not to install these source bundles at runtime.  If we went the header route then the header would have to contain similar information as a current extension to org.eclipse.pde.core.source point.  For example, the org.eclipse.platform.source bundle has the following extension:

<plugin>
 <extension point="org.eclipse.pde.core.source">
  <location path="src" /> 
 </extension>
</plugin>

So something like this could be used in the org.eclipse.platform.source bundle to tell PDE where the source is located.

Eclipse-SourceLocation: src

Any bundle with this header would not get installed by the p2 simple configurator.  That is just the idea as I remember it in discussions with Jeff and others (DJ?) on the Equinox/Orbit teams.

Comment 8 Jeff McAffer 2007-09-11 23:00:22 EDT 
I suspect that we are talking about headers in two different bundles here.  Chris is suggesting that a binary bundle A have a header identifying the bundle that has its source e.g., 
   Eclipse-SourceBundle: A.source;version="3.4.0"

Tom is suggesting that A.source 3.4.0 have a header that simply identifies it as a source bundle and where, in that bundle, the source is located.
   Eclipse-SourceLocation: src
The fact that A.source has the source for A is implicit.  Note that for most source bundles I would actually hope that the header was empty. e.g., 
   Eclipse-SourceLocation: 
indicating that the source was at the root of the bundle.  the src dir is redundant.

Yes there are issues with the implicit nature of the corelation and not to excuse it but this is what we have today.  doing more in the headers is, as tom points out, a build issue because the exact version numbers change alot.

A middle position would be perhaps to say that the source bundle point to the bundle for which it contains source.
   Eclipse-BundleSource:A;version=3.4.0.v122;location=src
Since source bundles are in general generated, the embedded version number burden may not be so great.  In theory it would also allow for one source bundle to contribute source for multiple binary bundles by having a CSV in the header value.  This unfortunately implies that the source lookup mechanism has to manage a mapping table rather than just doing the computational mapping at source lookup time...

Comment 17 Darin Wright 2007-10-29 22:56:52 EDT 
Looking at the spec - it looks like we can specify a root path if wanted, or specify null to have the location automatically detected.

* @param sourceAttachmentRootPath the location of the root of the source files within the source archive or folder or <code>null</code> if this location should be automatically detected.

However, to specify a location means there is only *one* root per bundle. We can have different bundles with different roots in the same archive, but we can't specify multiple roots per bundle.
