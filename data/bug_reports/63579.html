Description Marco Qualizza 2004-05-23 07:55:30 EDT 
M9 (GTK)
KDE 3.2.2
Gentoo
---
If one does a rename (noticed this with a class, but probably affects all
"renamable" artifacts), then the contents of the "New Name" field gets put into
the clipboard, erasing what was there previously.  This, IMHO, is Not Good (ie/
Very Bad (TM)).

To reproduce:
- copy something.
- paste (to prove to yourself that that's what the clipboard contains)
- start a rename (right-click on a class, Refactor->Rename).
- No need to complete the rename, just hit cancel.
- try pasting again -- notice that what you are pasting is the contents of the
"New Name" field.

Comment 5 Grant Gayed 2004-06-03 10:45:02 EDT 
This behaviour is quite pervasive throughout eclipse.  It seems like the base
case can be captured by:

- create the test class that's pasted below
- copy some of its text to the clipboard
- paste it somewhere to show that it is in fact on the clipboard
- run the example
- deselect the selected text (the abc) in the window that's shown
- close the example
- now paste, and note that abc is now on your clipboard

import org.eclipse.swt.widgets.*;

public class Main {

		final Display display = new Display ();

		shell.setBounds(10,10,200,200);
		Text text = new Text(shell, SWT.SINGLE);
		text.setBounds(10,10,180,100);

		shell.open();
		while (!shell.isDisposed()) {
			if (!display.readAndDispatch()) display.sleep();
		}
		display.dispose();
	}
}

Comment 11 Veronika Irvine 2004-10-25 18:02:23 EDT 
Fixed > Oct 25, 2004.

There are a number of things going on here.  

#1 If you are running Klipper, then when you paste something Klipper makes a 
copy of it and therefore it survives after the native widget is disposed.  See 
bug 42632.

#2 StyledText & Clipboard do not support copying the selected text and pasting 
with the middle mouse button but the native Text widget does.  See bug 26878.

#3 Clipboard is bogusly looking in the selection clipboard first and then the 
regular clipboard when the user pasets with Ctrl+V or the menu.

I have fixed case #3.  Clipboard.getContents will only look in GtkClipboard 
for the data to paste.  As a result, it will not pick up the text that is 
highlighted in the Refactor dialog or the Find dialog.

I am looking at case #2 but will persue it in bug 26878.

Case #1 is something GTK should really fix.  I am tracking it in bug 42632.
