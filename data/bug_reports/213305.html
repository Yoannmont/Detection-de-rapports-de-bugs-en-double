> Can you provide any advice on what to search for within SWT's TextLayout to
> address this performance issue?

Start in computeRuns(). First the text is broken into run, this happens in the itemize(). Because your line is very line you will end up with millions of run. The next step to generate the glyphs for each run (see shape()), that is where the time is.

Oh, glad you asked, I assumed that was understood. We don't and users don't want to display this kind of file, normally. But, sometimes users just end up with them. (some servers for example strip out all spaces and line feeds for more compressed transmission over a network, users end up with it in that form, and then want to import into Eclipse). So, user's probably do not even know it's one line when they open it. Once it's open they may try peeking at it, and maybe trying to do a few "manual" line breaks while they figure out what the file is, or what they want to do with it. This sort of "exploration" is near impossible for some, large one-line files because the slightest operation begins to take minutes (well, not sure how long ... but long enough some users assume Eclipse is broken). 

> 
> The right thing to do IMO is to parse this XML adding break lines and tabs for
> formatting, and then display that in the editor.
> 

Yes, I was thinking of this as a work-around. Keep in mind, though, we can't automatically or arbitrarily re-format someone's file ... they may not want it re-formated, for some reason. But, I could envision some "special case" to give them the option .... that we might do extra processing up front (taking time on every file) and decided if some best-guess threshold was exceeded, we could tell the user ahead of time "this is a large, one-line file which will not be usable in the editor. Would you like us to format it first". But, I think that's not completely easy to do ... takes some amount of I/O before allowing the editor to open, hard to set a good threshold, and what if it's two very very long lines (where's the cut-off), etc. 

But, that's why I asked. If this performance problem is considered absolutely unfixable, guess we'll have to start thinking through work-arounds like that. 

Keep in mind, that users do not see this sort of problem with other xml or html editors they use or have used in the past ... which is why they are surprised to see it in Eclipse (or, more precisely, Eclipse based products) and thus why they consider it a bug.

Comment 13 Felipe Heidrich 2008-11-28 15:31:37 EST 
I compared Uniscribe (TextLayout) vs GDI (gc.drawString).

First note that any text longer that 0x7FFF is not drawn by GDI (basically gc.drawString/gc.drawText don't draw anything). The same is true for Uniscribe, it won't draw a run that wider than 0x7FFF pixels.

Uniscribe is as fast as GDI when the text being analised has only one run. When the text has to be broken in runs (potentially thousands/millions) it gets slower.

There is no way TextLayout can pretend the entire text is only one run long without breaking DBCS and other languages.

There is no fix for this problem.
