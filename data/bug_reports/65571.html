Description Panagiotis Korros 2004-06-03 12:36:25 EDT 
The properties file editor corrupts any non ISO-8859-1 characters. It should
implement native2ascii conversion when saving a file and the reverse conversion
and it loads a file.

From 'java.util.Properties' class documentation:

'When saving properties to a stream or loading them from a stream, the ISO
8859-1 character encoding is used. For characters that cannot be directly
represented in this encoding, Unicode escapes  are used; however, only a single
'u' character is allowed in an escape sequence. The native2ascii tool can be
used to convert property files to and from other character encodings.'

http://java.sun.com/j2se/1.4.2/docs/api/java/util/Properties.html#encoding

Comment 9 Dani Megert 2004-07-08 12:45:06 EDT 
In pre-3.0 we did not do anything special in regard to *.properties files
(except providing the Properties Editor which is in fact a normal text editor)
and hence it was up to the user to set the correct encoding directly on the
file. Failing to do so may now cause problems since it got stored in the
workbench's default encoding. Also note, that in respect to Java specification
property files without ISO-8859-1 encoding are invalid.

R3.0 registers two properties file content-types and tells to use the encoding
ISO-8859-1 (declared in core.runtime as org.eclipse.core.runtime.properties
content-type and in jdt.core as org.eclipse.jdt.core.javaProperties). As a
result we now use ISO-8859-1 encoding to open *.properties files unless the
encoding has been explicitly set on the file.

If you have a scrambled properties file then this is the workaround
1. open the file
   ==> it will be scrambled
2. change the encoding via Edit > Encoding to your *workbench* encoding
   ==> it will now correctly be shown
3. Select all, copy to clipboard, delete the text
4. paste into notepad
5. change encoding back to default (ISO-8859-1)
6. select in notepad and copy
7. paste into editor
8. save

see also bug 68270

Changing severity back to major since having Java properties files in
non-ISO-8859-1 encoding is against the spec. However, we could have done better
in 2.x by enforcing this.

We need to add a properties file document provider for the Properties Editor
which will do the native2ascii conversion. The standard text editor will not do
this and allows to look at the plain file.
Comment 10 Panagiotis Korros 2004-07-08 17:53:28 EDT

characters.

If you try to create a .properties file with non ISO-8859-1 characters in
eclipse, so that you can run the native2ascii tool manually, the file is not
scrabled but destroyed. Every non ISO-8859-1 character is replaced by a ?.

I marked the severity as critical, because I think that eclipse should respect

'When saving properties to a stream or loading them from a stream, the ISO
8859-1 character encoding is used.
For characters that cannot be directly represented in this encoding, Unicode
escapes  are used.'

Comment 25 Ivan S. Dubrov 2008-07-15 11:29:06 EDT 
I've implemented a simple workaround to this bug. It is a simple extension to JDK supported charsets.

To use this extension, simply put the binary JAR into the <JDK_HOME>/jre/lib/ext directory,
run the Eclipse, open the Window->Preferences dialog, navigate to General->Content Types,
select "Java Properties File" node under "Text" node on the right, and change the default
encoding to "ISO-8859-1-ESCAPED". After that, the properties files with unicode escapes
will be displayed in "native" form in the properties file editor and converted to the
escaped form automatically upon save.

The binary: http://wfrag.org/files/iso8859-1-escaped-1.0.jar

The source code (for those who afraid running untrusted binary stuff): http://wfrag.org/files/iso8859-1-escaped-sources.zip
