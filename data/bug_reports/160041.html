Comment 6 Anton Leherbauer 2008-06-18 10:31:57 EDT 
Created attachment 105291 [details]
First attempt

Here is an initial draft. 

Interface IStack2 on top of IStack adds support for retrieval of an arbitrary range of stack frames.

There are preferences for the stack frame limit and whether the limit is enabled.

In the launch VM provider, the preference values are copied into the properties of the presentation context so that the VM nodes need not access the preferences directly (I am not sure if this is the right approach here).

The stack VM node limits the number of stack frames retrieved using the new IStack2 interface, but the limit is also applied if the interface is not implemented by the stack service (e.g. PDA).

To support the double click update, a dummy stack frame is added in case the actual number of frames is higher than the limit. A double click listener registered in the view model proxy gives the double click event as-is to the VM provider. The update is done by setting a temporary flag in the stack VM node to ignore the stack frame limit for the subsequent updates. This flag is reset again on the next suspended event.

Unfortunately, the MI stack service still retrieves all frames for the label updates. The reason is that the command cache cannot handle the range arguments of stack-list-frames in a smart way. It is possible to change the update to retrieve only one stack frame, but then the effectiveness of the cache is degraded.
Maybe this requires a specialized cache for the frame data.
Comment 7 Pawel Piech 2008-06-19 00:48:36 EDT 
(In reply to comment #6)
> First attempt
> ...
It's a very good one :-)  I love the double click workflow and I'm really glad you got it to work.  I have a few comments though:

1) I ran into an issue when testing with the StressTests test file.  When I descended into the stack 50 deep, I got a whole bunch of blank lines following the valid stack frames (I'll attach a screen shot).

2) I think it's safe to turn on the limit preference by default, to something like 100 frames.  I think CDI has such a limit on by default as well.

3) The double click is great, but users may not discover it right away.  Perhaps we could add a context menu item to the dummy frame to retrieve more frames.  

4) Maybe it would be better if the IncompleteStackVMContext was based on AbstractVMContext instead of DMVMContext, it seems a little strange to me to see the run control actions enabled when this frame is selected.

5) It's nice to see the number of frames left to retrieve, but it's actually a scalability problem as well.  On some targets, it's almost as expensive to calculate the stack depth as it is to actually retrieve the full stack.  I don't know the details as to why this is, but it's what the DFW guys said.  So when the stack limit is set, the getStackDepth() method should also be called with that limit as the maxDepth argument.

6) It would be nice (but not necessary for DFE parity) if double-clicking would retrieve the next batch of stack frames in the same number as the original limit.  So that if user has a stack trace of 1000 frames, they won't automatically all get retrieved after the double-click.

7) DFE has an additional preference: "retrieve top stack frame only", which is sort of equivalent to just setting the limit to 1, and may actually have been unnecessarily complicated.  But if point 6 is considered it would make a little more sense.  

8) There is definitely a deficiency in the MIStack implementation as far as caching of ranges of data.  In the DSF-TCF implementation, Eugene created some generic caches that deal with arrays of data much better.  I would like to merge them into base DSF framework, and hopefully this will help with this issue as well.

9) Predictably, I would like to extend PDA to support this functionality as well.  Hopefully it won't be as big a feature as multi-threading :-)

10) I'm not crazy about having "DSF" in the labels in the preferences settings.  It's not a term that's going to be meaningful to end users.  But I also understand that not qualifying it will confuse users of other debuggers.  I'm not really sure how to deal with this, maybe we can leave it as is and see if we can resolve it later in the 1.1 cycle.

Overall, this is an excellent patch!

Comment 15 Anton Leherbauer 2008-06-20 06:54:53 EDT 
Created attachment 105484 [details]
Update

The changes:
- The default limit is 100 and enabled.
- getStackDepth() is called with the depth limit +1 (to get an indication whether there are more frames or not). The number of remaining frames is no longer displayed.
- IncompleteStackVMContext derives from AbstractVMContext
- Added an action "Expand Stack" to the context menu to increment the limit
- The limit is doubled at each "increment" via double click or action. This avoids the extra preference value.
- Moved constants to IDsfDebugUIConstants (I had neglected this interface)
- Fixed AbstractDMVMNode.fillUpdateWithVMCs() and PDAStack.getFrames() for startIndex > 0

Cool!
Spend a half a day playing with the patch and debugging and found some interesting stuff, I also have an updated patch that I'll attach:

- The temporary stack frame limit needs to be maintained per-context.  Otherwise we get some weird behavior, like when the stack of one thread shows extra frames, and another thread exits, all threads get reset to the original limit.

- Previously I added an extra frame to be retrieved when suspending to paint it with "..."  This is broken because IStack2 returns an error when requesting too many frames.

- Handling the PropertyChangedEvent in SF VM Node avoids the refresh when limit preferences are changed.

- It probably makes more sense to replace handling of the double-click event in vm nodes with a new event.  The new event could implement IDMEvent and thus would avoid the need to override getContextsForEvent() in thread and container vm node.

I introduced ExpandStackEvent.


Aside from that, I noticed that the model proxies of the Launch view don't get disposed when a launch is removed, so that the double click listeners never get removed. The effect was that the limit was doubled several times on each double click (and possibly some other weird effects).
Therefore I added a check for isDisposed() before handling the double click event in the vm provider. 
It seems that the launch removed event never gets handled, because the vm provider is already marked as disposed at that time in handleLaunchesEvent().

Comment 21 Anton Leherbauer 2008-06-24 07:10:24 EDT 
Created attachment 105689 [details]
Update 4 (+committed)

I found a problem with relying on getStackDepth() to respect the maxDepth argument. PDAStack did not consider this argument, so I think we should keep the check.

Another issue caused my fix to AbstractDMVMNode.fillUpdateWithVMCs(). Previously the method assumed that the dmcs array always starts at index 0 wrt the update request. The fix changed that assumption and fixed it for the stack frame requests but broke others. I added another argument firstIndex which indicates the starting index of the dmcs array. The original method delegates to it with firstIndex=0.
