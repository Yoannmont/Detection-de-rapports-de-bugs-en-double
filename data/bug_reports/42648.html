Description Brent Nicolle 2003-09-05 17:23:55 EDT 
Particularly aggravating on Linux or Solaris:
1. Start up the new Managed Make Project wizard.
2. Provide a name for it. (Accept the default location.)
3. The  "Finish" button is now available.
Impatient users will create projects for Cygwin Executables
without any way of changing that once created.
(Impatient testers did this a lot.)

4. Instead of "Finish", hit "Next"
The default platform is "Cygwin Executable".
The dropdown list is alphabetical on all platforms.

On Linux, the default platform should be Linux Executable.
The other choices should be Linux static or shared library.

Similarly on Solaris, just Solaris choices should show.

Similarly on Windows, just Cygwin choices should show.

Comment 6 Henning Riedel 2003-09-24 23:20:22 EDT 
Well, we could just use MinGW without MSYS, but the MSYS package provides you
with an environment similar to cygwin. The main difference is the dropped
dependency on  the cygwin.dll, as the tools are native ports. What you get with
MSYS is an RXVT using the bash, though you can run the autoconf tools too; and
even ssh is provided. 
The real problem with the MinGW/MSYS stuff is, that if you use MinGW without
MSYS the make tool is called mingw32-make.exe and it sits in the MinGW binaries
path. When using MSYS you get another make tool, actually its the same GNU-Tool.
MinGW and MSYS don't live in the same directory either and as the Docs tell,
they should not. This issue has been asked in the CDT-newsgroup frequently
(Launch error etc.). That's why I added both to the list. 
Another option i could think of would be an approach of selecting an target
output instead of a toolchain in the first place. I think you usually go that way:
* Select a target output you want to build 
  executable, shared-lib, static-lib, embedded-target(.hex/.coff/..)
* Architecture
  win32, linux, qnx, solaris, embedded
* Select Toolchain
  msvc, bcc, gnu (cygwin,mingw,linux)

Actually, a project could be targeted for several architectures and toolchains,
which as I see right now, cannot be accomplished within the same project.
Maybe we could select the target output first (exec/lib..), create Debug and
Release directories for them for each architecure and toolchain pair
(cygwin-win32, gnu-linux) selected, and then map each toolchain from an default
properties file where the default settings are stored and copy this poperties
file to the corresponding directory. The settings dialog can read in then this
properties file, and the changes are saved in there.
default-settings:
org.eclipse.cdt.managedbuilder.ui
+- settings
   +- gnu.linux.properties
   +- gnu.solaris.properties
   +- gnu.win32.cygwin.properties
   +- gnu.win32.mingw.properties
   +- borland.win32.properties

Each file having props like:
---
toolchain.path=D:\MinGW
make.command=make
cc.command=gcc
cxx.command=g++
cxx.flags=-g -Wall
cxx.includes.sys=%toolchain.path/include,%toolchain.path/include/c++
---
This format seems to me even easier to read than going through an xml-file.
The selected toolchain/target settings could be copied over to the project dir,
though you are able to have project specific settings. 

Select 'Executable', then create an Debug and Release dir, and in there create
toolchain specific dirs or otherwise, create toolchain specific dirs and in
there the Debug/Release dir.
<project>
+- .cdtbuild <-- project specific settings (includes,libs,references,targets)
+- cygwin-win32
|  +- Debug
|  +- Release
|  +- gnu.win32.cygwin.properties <-- target specific settings (tools,paths,
|                                     sys-libs,defines,..)
+- mingw-win32
|  +- Debug
|  +- Release
|  +- gnu.win32.mingw.properties
+- gnu-linux-x86
|  +- Debug
|  +- Release
|  +- gnu.linux.x86.properties
+- gnu-linux-ppc
|  +- Debug
|  +- Release
|  +- gnu.linux.ppc.properties
+- gnu-linux-arm
   +- Debug
   +- Release
   +- gnu.linux.arm.properties

The plugin.xml could be kept small, just listing the supported architectures,
toolchains. Maybe I'm wrong, but thats just my thought and I wouldn't mind if
you drop the ideas for now.
