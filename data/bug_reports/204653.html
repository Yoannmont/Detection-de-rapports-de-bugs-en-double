Description Marijn 2007-09-26 04:11:00 EDT 
Hello,

A while ago I filed the bug #200019 at the PDT project, however I was told this was a WTP issue. According to them it should have been resolved in 2.0.1, however I am running 2.0.1RC3 and the bug is still present. My setup:

Mylyn 2.1M1
DTP SDK N091207
PDT SDK 1.0
RSE SDK 2.0.1RC2
Eclipse SDK and test framework M20070921-1145
WTP SDK 2.0.1RC3
GEF SDK 3.3.1
EMF SDO XSD SDK 2.3.1

When I focus my cursor to a function call and type a comma the argument code
assist pops up (or when typing an opening parentheses and it pops up the
arguments required for the function) however it always "starts" at the first
argument.
Example:

some_function($arg1, $arg2, $arg3);

When I click after $arg3 and start typing for example $arg4, $arg5 the code
assist starts at the beginning and highlight "<$arg1>, $arg2, ...." instead of
"$arg1, $arg2, $arg3, <$arg4>, ...". The same thing happens when the code
assist disappears due to another code assist popping up ($this-> or another
function call). When typing a new comma after that it has reset again.

Regards,

Marijn.

Comment 5 Guy Gurfinkel 2007-10-07 07:25:01 EDT 
The problem is that WrappedContextInformation (org.eclipse.wst.sse.ui.internal.contentassist.CompoundContentAssistProcessor$WrappedContextInformation) doesn't implement IContextInformationExtension.


and thus when it gets to org.eclipse.jface.text.contentassist.ContextInformationPopup at line 297 it fails to update the beginOffset.
Comment 6 Amy Wu 2007-10-10 19:00:03 EDT 
Created attachment 80092 [details]
org.eclipse.wst.sse.ui.xml.ui.patch

Thanks for the clarification, Guy.

This patch will make CompoundContentAssistProcessor$WrappedContextInformation implement IContextInformationExtension.  In the case that the original Context Information does not implement IContextInformationExtension, the wrapped context information returns -1 instead.  According to the IContextInformationExtension, that's what you're supposed to return when the begin offset is not known.
