Comment 11 Terry Corbet 2005-02-04 18:26:08 EST 
I don't know if this is the correct way to influence the development of a more 
robust Browser implementation in SWT or not.  I suspect that I ought to be 
providing some formal enhancement request, but having reviewed the various 
postings concerning *some* of the issues, it is clear that you and the other 
people monitoring this forum are aware of and thinking about the issues. So, I 
would like to add a perspective.

Simply having the ability to embed a Mozilla- or Foxfire-based browser instance 
is not enough.  Most of us are on the trail of 'the rich client' and that means 
integrating a lot of nice capability beyond the rendering of <html> pages.  The 
very first level of integration requirement is that all the javascript that we 
have used to enhance those <html> pages has to communicate -- forwards and 
backwards -- with the SWT application.  Granted the present <form> has 
limitations, but the drop-down boxes are there and the calls to our functions 
to modify a selection prior to a submit() are there.  Sometimes we need to grab 
that selection in the SWT application and interceping the StatusText string and 
parsing name-value pairs is just plain wrong.  We need a proper protocol for 
binding data from the embedded scripting language into our java code.

The second level of integration is that we really want to embedd <svg><script> 
in that <xhtml> page and now I need to be sure that I can communicate -- in 
both directions -- with those controls, those events and those functions.  
Forgetting SWT, today there is no cross-platform way to get the parent script 
engine to properly communicate with the embedded script engine.  We want to 
activate a SWT button and have the action percolate all the way down to 
altering the animation behavior of the <circle> inside the <svg>.  As nice as 
SWT's widgets look, I'm sorry, they are not what you can do with <svg> in terms 
of really energizing your current web applications.

I have been working on trying to solve the <svg> embedding so the absurdities 
of not having standards and protocols is fresh in my mind, but I really should 
be doing the same thing with <xforms> -- they are here now, you know.  We 
really should be doing the same separation of model, view and control of our 
forms with this better solution, but it introduces the same problem.  How do I 
populate <xform> content from my SWT application code; how do I retrieve 
<xform> content in my SWT application?

So, don't get me wrong; I wouldn't want to do anything to impede any progress 
towards improving on what you have given us with the current Browser. But, once 
or twice, your reply to a posting has been "... can you tell us what you want 
to do that for ..."  So, what 'I want to do this stuff for' is that the ASP.NET 
Longhorn army is marching right down on our J2EE Servlets with answers to the 
question of how best to partition between client-side and server-side, how best 
to capitalize on the User Interface by enabling images, videos and audio to 
engage the end-user.  The Mozilla army has infrastructure for doing most of 
the 'good things' that we need to do, but it is a lot more than simply tickling 
the rendering engine to get better looking stuff on the screen.  I would like 
to see the SWT.Browser project address full-function, two-way data and control 
flow between the application and the 'contained browser'.

Comment 26 Javier Pedemonte (inactive) 2006-01-10 12:52:01 EST 
Created attachment 32773 [details]
Mozilla SWT fragment

This should be a good starting point for you guys.  This is an SWT fragment project that embeds Mozilla in an SWT widget.

* Code is based on existing SWT/Linux Mozilla code, and the old SWT/Win32 Mozilla code, before it was removed.  It's based on an older version, so missing many changes that went into CVS towards the end of 2005.
* Makes use of JavaXPCOM component (http://developer.mozilla.org/en/docs/JavaXPCOM) and XULRunner (http://developer.mozilla.org/en/docs/XULRunner).  XULRunner is the new "GRE" for Mozilla, and registers itself with the platform.  JavaXPCOM is a Java API for Mozilla, which makes the embedding code much cleaner, and gets rid of JNI code in the embedding code.
* Right now, supports Linux and Win32; haven't had time to work on Mac OS X.
* Platform specific code is split into separate files, so you need to copy the appropriate ".classpath_XXX" file to ".classpath".  If anyone can think of a better way to do this, please let me know.

I wrote the JavaXPCOM code, so if you have suggestions, please let me know.

I don't work with SWT or Eclipse code often, so it would be best if someone here who is more knowledgable could take this code and start working with it.  I'd be happy to help out with the JavaXPCOM and Mozilla interactions.

Comment 36 Sebastien JUST 2006-05-13 07:59:09 EDT 
(In reply to comment #14) and hoping I'm on the right Bug discussion

Our compagny would like to have a Browser plugin based on Firefox or Gecko. And we'll work on it. Help is welcome ! The reasons follows :
- We're looking for a rich client platform to integrate our existing web based solutions and provide more user experience in our future products, so, 
- We want a unique browser implementation through all platforms
- We want Java/Eclipse to interact with DOM to simulate user interaction on the web pages
- We also want Java/Eclipse to listen to DOM events to react to user input and monitor what's happening
- Have a full support for browser preferences
- We do not want the user to be forced to have a particular software on their computer other than Java (no Firefox/Mozilla installation is needed)

Thus we should provide a way to do the following :
- Implement automated test case tools to test our web based solutions by simulating web user interaction with Java
- Cut down development costs with graphical design ( CSS, Javascript and co. does not have to care about IE but just follow standards )
- Focus on a particular version of the browser so we do not need to handle all differences between browsers (version of CSS or Javascript supported for example) and so reducing compatibility tests and development costs
- Extend the time between new browser versions so we can rely on a stable/choosen releases (so we are not dependent on a new IE release, a Windows update, custom installed browser...)
- use most of the technology provided by mozilla/firefox that IE can not handle (SVG, best XML/XSLT support...)
- force client side graphical issues by locking browser preferences towards our web based products so that browser extreme user configurations have no consequences on rendering and usage (so reducing maintenance and hotline)
- tightly integrate our new Rich Client products with existing web based applications by providing hot interaction between the browser and the rest of the application
- ship a package that we are known to work on every platform (Windows, Mac, Linux) without thinking about OS differences and OS installation related stuff
- have multiple Views in Eclipse based on a same opened session on a web application (nowadays, each view is another browser instance, so we can't share sessions for example, in a clean way)

Hope that'll help to understand our motivations and give you an idea of what an embbeded browser component can provide (more that displaying web pages).

Help is needed for this. Hope some of you will understand. Please contribute to this discussion, and forgive me for my so bad english !

Comment 46 Grant Gayed 2007-02-06 12:39:43 EST 
The first part of this work is released, the Mozilla GRE can now be used to render Browser contents on win32 and OS X if a XULRunner is installed and registered on the machine.  To specify that this should be used just create your Browser with style SWT.MOZILLA.

Note that there are some specific version requirements to run this on OS X that do not apply to windows or linux:

- on OS X your jre must be "Java for Mac OS X 10.4, Release 5 Developer Preview 2", which is available on Apple Developer Connection ( http://connect.apple.com/ ) (registration is free)

- on OS X your XULRunner must currently be a 1.9 stream build, such as http://ftp.mozilla.org/pub/mozilla.org/xulrunner/nightly/latest-trunk/xulrunner-1.9a2pre.en-US.mac-pkg.dmg
  -> note that a future 1.8.x stream build of XULRunner may have support for this on OS X; if this build becomes available then a comment will be added here with a link to it
