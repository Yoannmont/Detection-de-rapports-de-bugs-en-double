Comment 3 John Arthorne 2005-10-05 16:50:57 EDT 
I found a possible source of the problem.  The code in StringPoolJob does this:

try {
	Platform.getJobManager().beginRule(rule, monitor);
	start = System.currentTimeMillis();
	savings = shareStrings(toRun, monitor);
} finally {
	Platform.getJobManager().endRule(rule);
}

If shutdown starts before the finally block is run, then the JobManager instance
is nulled out.  The finally block would then cause a new JobManager to be
created. This would cause the mismatched end rule.  The fix is:

final IJobManager jobManager = Platform.getJobManager();
try {
	jobManager.beginRule(rule, monitor);
	start = System.currentTimeMillis();
	savings = shareStrings(toRun, monitor);
} finally {
	jobManager.endRule(rule);
}

I'm not certain if this will fix your case, but it is certainly cleaner.  I will
also investigate if the runtime shutdown sequence can be improved to ensure a
JobManager instance isn't recreated (which would be very bad in any case).
