Description Sean Montgomery 2005-11-07 14:22:40 EST 
Running Eclipse 3.2M3 on Mac OS X 10.4.3, with two windows, one on an external monitor and one on 
the laptop screen.  Both windows are in the Java perspective and each have multiple editors in tabs, 
each containing .java files, all of which are from the same project, with only that one project loaded.

Several times a day while editing (i.e. not debugging or explicitly building, though Build Automatically is 
enabled for the project) one of the windows becomes unresponsive to the mouse and keyboard, i.e. 
text cannot be entered, clicking does not move the caret/insertion point, clicking on tabs does not 
change the editor selected, etc.  I've waited for several minutes but the "stuck" window never becomes 
usable. The other window (and its editors) remains usable, and the CPU usage is normal.

If the Windows menu is chosen and the "stuck" window selected then the "beachball" cursor appears, 
the CPU usage goes to 100% and the entire Eclipse app becomes unresponsive/locked up.  The only 
option at this point is to force quit Eclipse.

A sample of Eclipse while the entire app was locked up and using 100% of the CPU will be attached.
