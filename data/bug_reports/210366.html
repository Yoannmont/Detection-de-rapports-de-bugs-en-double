Comment 41 Marc Khouzam 2012-02-03 14:30:56 EST 
I found a problem with the DSF changes, and since I'm new to using a process factory, maybe someone can suggest a better fix than what I'm thinking of.

Anton's very elegant current proposed solution:

The Test plugins need to intercept the output of the MIInferiorProcess.  To do that, Anton wraps the MIInferiorProcess so as to grab its stream (very nice! I hadn't thought of doing something like that).  The wrapping needs to be done before the inferior is added to the launch, so that the streams can be replaced first, before they are passed to the console.  To achieve that, Anton uses IProcessFactory and DebugPlugin.newProcess.

So, when doing a normal debug session we need a process factory to create an InferiorRuntimeProcess directly, but doing Unit test debugging, we use a different factory which first wraps MIInferiorProcess, then creates InferiorRuntimeProcess using the wrapper process.

Problem:

Using a process factory implies that anyone calling DebugPlugin.newProcess in a DSF-GDB launch, will now create an InferiorRuntimeProcess, while before, it used to create a plain RuntimeProcess.

This can cause problems.  For example, in the org.eclipse.cdt.launch.remote plugin, we create another process (RemoteShell) by using DebugPlugin.newProcess.  This process should not be an InferiorRuntimeProcess, but because the factory is set, it automatically is, and things break.

Solutions:

I have a couple of solutions, but they are not very nice:

1- Fix org.eclipse.cdt.launch.remote.launching.RemoteGdbLaunchDelegate to not use DebugPlugin.newProcess, and continue with the proposed factory solution.  I don't like this approach because if anyone else is extending the DSF launch, they will have trouble if they use DebugPlugin.newProcess.

2- keep everything local to the Unit test plugins, and when launching, after the DSF launch is ready, remove the inferior from the launch, and replace it with a wrapped inferior.  The problem here is that there could be some output sent before the inferior is replaced; especially if the user does not select the "stop on startup" option of the Debugger tab.  

3- Use a factory, but make it smarter.  The factory would need to know when to create an InferiorRuntimeProcess and when to create only a RuntimeProcess.  We could base this decision on the Process parameter of IProcessFactory.newProcess but that seems risky to me.  We could also add a special attribute to the attributes parameter of IProcessFactory.newProcess (in fact, we already have one we can use: IGdbDebugConstants.INFERIOR_GROUPID_ATTR) to indicate that the process is the inferior.

At this time, I think only 3 is a viable solution.

Any thoughts or other suggestions?

As far as I know, Junit communicates with the testing process not via the standard output. It loads the testing framework (Junit library) directly and controls the testing process via Java interface. Unfortunately this model cannot be used for C++ testing frameworks. Unlike Java test module which throws exceptions in case of errors the C++ one can also lead to the crash (e.g. on NULL-pointer dereferencing). So its running should be isolated in a separate process. There are a few ways to communicate with the testing process which have their pros and cons:
- via the standard output or error streams. The easiest way supported by all the testing frameworks. It allows to display the testing progress in "real time" and it is easy to write readers cause many frameworks are able to output in XML. However (as you noticed) this way doesn't allow user to output something to the stream which is used for communication.
- via the file. The most of testing frameworks allows to make a report in file, which can be a simple file or a pipe. Simple files do not allow to display the testing progress, pipes work too differently on different platforms (actually I am not sure that output of test module can be redirected to the pipe file on Windows, however it should works fine on Linux).
- via the sockets. Some of testing frameworks (e.g. Google Test) can output the results via the sockets. It is a nice way, however it is supported only by a few testing frameworks and may trigger some unclear requests to user asking to establish the network connection when starting the testing process (e.g. on Windows).
- via the special loader. This way is applicable only if test module is a library but not an executable. In this case it can be loaded with our custom loader (written in C++ or Java) which will use the testing framework API (for suites browsing and tests execution) and will communicate with the Eclipse plug-in providing the necessary information. This way is the closest to how Junit works, however it is quite hard to implement it and it requires some assumptions about test module internals. So it is not as universal as the above ones. Moreover, it doesn't solve the problem of communication between processes, however it makes it easier because the protocol maybe fully customized.

So analyzing the ways described above I think that the most universal way that allows to see the testing progress and is supported by all the testing frameworks is the first one with stdout/stderr. However the other ways may also be implemented for the special cases as the data providers for the Tests Runner Plug-in.
