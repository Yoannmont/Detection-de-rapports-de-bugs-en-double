Description Neil Hauge 2007-06-28 16:07:09 EDT 
Currently Dali only supports one persistence unit per project.  Multiple persistence units could be found in a persistence.xml, and we need to determine how we can support this use case.  

This bug should be used as a place to discuss what multiple persistence unit support means, and how it could be supported.
Comment 1 Neil Hauge 2007-07-27 15:15:13 EDT 
Proposed for 2.0.  Non-overlapping classes support, with a requirement that classes are listed in persistence.xml.

Comment 3 Neil Hauge 2008-01-04 10:11:11 EST 
Yep.  This is already planned for the next maintenance release.  See bug 201949.
Comment 4 Neil Hauge 2008-01-18 11:50:35 EST 
I think this might be noted somewhere else, but a good solution for 2.0 might be to have an "Active Persistence Unit" setting in the project.  All validation and defaults will be based on the "Active" PU, which will define the current context.  

This will simplify the support, and make it possible for a user to easily work with multiple PU's in a single project.  No overlap would occur since there would never be more than one active PU.

Comment 6 Shaun Smith 2008-02-25 11:21:55 EST 
While I understand the spec supports multiple persistence units per persistence.xml, can someone supporting this bug describe the overlapping persistence unit use cases and how single non-overlapping persistence unit support  cannot be used to solve them?  

The most obvious use case I can think of is wanting different persistence unit configurations for production vs. test.  Typically these are identical except for transaction type and datasource.  However this is easily dealt with by configuring a persistence unit for production and using createEntityManagerFactory properties to override settings.  Section 7.1.4 of the spec includes properties for switching transaction type so you can configure your persistence unit for JTA with a datasource, but override that in test and run in RESOURCE_LOCAL and add provider specific JDBC login info.

public enum PersistenceUnitTransactionType {
   JTA,
   RESOURCE_LOCAL
}

Multiple overlapping persistence units introduce a number of headaches in the area of code assist that ends up in a discussion of "active persistence unit".  Having an "active" anything is the same as having a a modal user interface which is an extremely unpleasant option.  My feeling is that the multiple overlapping use cases are infrequent and that developer energy can be better used elsewhere.

   Shaun
Comment 7 Adrian Goerler 2008-02-28 07:33:48 EST 
I agree that you typically have overlapping persistence units for test purposes only. (Actually I would rather think of it as a different view or configuration of the same persistence unit).

However, to my knowledege, you can't dynamically override the transaction of a PU. At least, there is not standardized way to achieve this. The map argument of the Persistence.createEntityManagerFactory is for dynamically configuring additional (vendor-specific) properties of the persistence unit. It corresponds to the <properties> section of the <persistence-unit>'s section in the persistence.xml. § 7.1.4 describes the PersistenceUnitInfo interface used for the communication between the persistence provider and the container in the Java EE scenario.

Therefore, I think it is more promissing to have an "active" or "default" or "master" persistence unit + additional "views" or "configurations".

Adrian
Comment 8 Shaun Smith 2008-02-28 10:55:13 EST 
>However, to my knowledege, you can't dynamically override the transaction of a
>PU. At least, there is not standardized way to achieve this. 

Actually this is documented in the Spec in section 7.1.3.1 (see below) so switching between "deployment" and "test" as I described should be portable.

   Shaun

7.1.3.1

...When any of these properties are specified in the Map
parameter, their values override the values of the corresponding elements in the persistence.xml file for the named persistence unit. They also override any defaults that the provider might have applied.
The properties listed below are defined by this specification.
• javax.persistence.provider – Corresponds to the provider element in the persistence.xml. See section 6.2.1.4.
• javax.persistence.transactionType – Corresponds to the transaction-type attribute in the persistence.xml. See section 6.2.1.2.
• javax.persistence.jtaDataSource – Corresponds to the jta-data-source element in the persistence.xml. See section 6.2.1.5.
• javax.persistence.nonJtaDataSource – Corresponds to the non-jta-data-source element in the persistence.xml. See section 6.2.1.5.
Comment 9 Neil Hauge 2008-03-05 15:00:18 EST 
Wanted to note that what is decided here will also impact our persistence.xml configuration.  If we only support one PU per project, the persistence.xml editor would be limited to configuring one PU as well (the persistence unit that is active in the model).

This could be good for the standard case (it would greatly simplify the configuration experience), but would make the multiple PU cases more cumbersome (you would need to manually comment out the PU's that you didn't want to work on in order to get a model (and UI) for the PU desired).

So, I think what this all comes down to is how important the Multiple Persistence Unit case is.  Shaun has made a case that completely overlapping PU's for testing purposes can be accomplished through other means.  The other case would be non-overlapping PU's, where a single persistence.xml is being used to manage multiple domains.  

I agree with Shaun that in general, we should avoid "modes" when possible.  Without a mode, or active PU, we don't have a way to support overlapping PU's, but we could still support non-overlapping PU's.  The hard part with this is that we have to be able to handle the case when they do end up overlapping.

Is there any other feedback on this?  I think the default position here will be to continue supporting only 1 PU per project.
Comment 10 Neil Hauge 2008-03-25 12:28:41 EDT 
Deferring this bug post 2.0 as critical mass was never achieved for a solution.  The workaround (when working with multiple PU's) is to comment out all of the PU's in the persistence.xml that are not the particular PU that you want to work with.

We will revisit multiple Persistence Unit support for the next release.  Please feel free to continue the discussion on this bug.

Comment 14 Shaun Smith 2013-04-08 09:01:56 EDT 
If you're using EclipseLink have you considered using a Composite Persistence Unit?  Typically Composite PUs are used to allow you to work with data in multiple heterogenous databases.  http://www.eclipse.org/eclipselink/documentation/2.4/solutions/usingmultipledbs.htm

When using a Composite PU, in Dali you create a project for each composite member containing a single PU and a project for the composite PU.  This is how I structured the Polyglot Persistence example in which some data was in relational and some was in MongoDB: http://wiki.eclipse.org/index.php?title=EclipseLink/Examples/PolyglotPersistence
Comment 15 Shaun Smith 2013-04-08 09:08:06 EDT 
And Craig, putting @PersistenceContext on an entity does not associate it with a persistence unit.   @PersistenceContext is used to inject an EntityManager only. Your entities have to be listed in the persistence.xml for inclusion in your multiple PU scenario to avoid having them included in all PUs by default.

Comment 19 Shaun Smith 2013-04-08 11:27:50 EDT 
Yes you would need to reference composite members using <jar-file> and this is an EclipseLink JPA extension and is not portable to other providers.  It is really designed to support scenarios in which you have related data spread across different databases, as in this diagram: http://wiki.eclipse.org/EclipseLink/Examples/JPA/Composite#Relationship_Example  With Composite PUs you can transparently traverse relationships from objects in one database to objects in another.  Updates are sent to the correct datasource.
