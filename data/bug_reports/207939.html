Description Missing name 2007-10-30 07:04:34 EDT 
Build ID: M20071023-1652

Steps To Reproduce:
1. Create a local copy of a java source file and perform some changes in it
2. Use 'Compare with each other' on them
3. Double click a node in the structure tree view
4. Click in one of the editor windows and manually edit something
5. Save
When you open the file that you manually edited, you will see that the changes did not get saved, which may cause a data loss! I reproduced this on two colleague's windows computers with separate eclipse installations and workspaces, too.


More information:
I recognized this bug after creating a structured compare implementation. While debugging I found out that the problem is triggered by the focus handling of the GUI: Starting in line 132 of org.eclipse.swt.widget.Control, the focus gets set and during this, the org.eclipse.compare.contentmergeviewer.ContentMergerViewer's content changes gets flushed and the dirty state gets resetted (ContentMergeViewer, line 1150). When the actual save mechanism gets triggered later (org.eclipse.jface.window.ApplicationWindow, line 755), the editor isn't marked as dirty anymore, so the content doesn't get saved.

Comment 11 Kevin Tapperson 2008-08-26 16:40:46 EDT 
I have been able to work around this issue by overriding the flushContent method in ContentMergeViewer.  In my subclass of ContentMergeViewer, I coded the following flushContent method.  This is a horrible hack, but it makes this problem go away.

	protected void flushContent(Object input, IProgressMonitor monitor) {
		boolean callFlushContent = true;
		StackTraceElement[] stack = new Throwable().getStackTrace();
		for (int i = 0; i < stack.length; i++) {
			if (stack[i].getMethodName().equals("getAdapter")) {
				if (stack[i].getClassName().endsWith("ResourceCompareInput")) {
					callFlushContent = false;
				}
			}
		}
		if (callFlushContent) {
			super.flushContent(input, monitor);
		}
	}
