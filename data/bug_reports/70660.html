Description Dave Smith 2004-07-22 13:22:33 EDT 
I profiled the GLA code parsing an Apache access log and noticed that the call 
counts for some methods are not correct.  After a certain number records 
parsed, the call counts don't increase.  They stay fixed at a certain value.  
For example, after 37 log records parsed the 
org.eclipse.hyades.logging.adapter.parsers.SubstituteStatement.matchAndSubstitut
e() method has 1661 calls. For every record parsed it should call 
matchAndSubstitute approximately 45 times.  But the number of calls stays the 
same.  I profiled the GLA in standalone mode and verified that after a certain 
time, there are no calls to matchAndSubstitute() recorded in the trace file.

I suspect this is a IBM JDK JIT problem.  The problem occurs with the following 
IBM JDK:

java version "1.4.2"
Java(TM) 2 Runtime Environment, Standard Edition (build 1.4.2)
Classic VM (build 1.4.2, J2RE 1.4.2 IBM Windows 32 build cn1420-20040626 (JIT 
enabled: jitc))

The problem does not occur with the JIT disabled.

The problem does not occur with the SUN JDK:

java version "1.4.2_05"
Java(TM) 2 Runtime Environment, Standard Edition (build 1.4.2_05-b04)
Java HotSpot(TM) Client VM (build 1.4.2_05-b04, mixed mode)

I suspect the method's are being JIT'ed by the IBM JIT after a certain number 
of calls and the JIT'ed code is not calling JVMPI.

I'll attach the files necessary to reproduce this problem.

To reproduce this problem do the following:

1) Check out the org.eclipse.hyades.logging.adapter and 
org.eclipse.hyades.logging.adapter.config plugins from CVS into your workspace.
2) Profile the Main class org.eclipse.hyades.logging.adapter.Adapter with the 
program arguments "-cc f:\apacheaccess.adapter -ac f:\apacheaccess.adapter" and 
the Execution History profiling set.  Also add the following filter to the 
default filters:  org.eclipse.hyades.logging.*  *  INCLUDE
3) Notice the call count for the following methods in the SubstituteStatement 
class:
createResultGraph   1040
matchAndSubstitute  1661
getResultGraph        21
4) Profile the Adapter class again, this time with the VM argument -
Djava.compiler=NONE
5) Notice the call counts now are different:
createResultGraph   1092
matchAndSubstitute  1751
getResultGraph        39

Comment 10 Robert Danek 2004-11-17 11:25:03 EST 
I got the following 'working as designed' response from the JDK person looking
into this bug (basically, turning off inlining seems to be the only way to get
around this problem): 

----- 
I analysed the JVMPI agent and the testcase and found that the method that is
being called  is getting inlined (as per design we do not inline the call to the
JVMPI agent) and hence whenever the method(that is been monitored by the
profiler ) is executed as inlined code into some other method, this wouldn't
trigger a  Method Entry event, which is working as per the current design.
The call to JVMPI agent is not inlined becuase of performance reasons (because
doing so will make lot of optimisations ineffective).
.
Based on certain conditions JIT decides to inline the method. by enabling the
option JITC_COMPILEOPT=NINLINING, we can avoid the method being inlined which in
that case we will be able to see as many method entries that is been mentioned.
.
Since this is working as current design, Would it be possible to use the above
workaround JITC_COMPILEOPT=NINLINING as the solution for the problem , to enable
your logging activity ?
-----------------------------

Unless there are any objections, I will open a README defect for this item, and
close off this bug.
