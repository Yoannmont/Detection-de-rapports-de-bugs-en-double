Description Jay Cagle 2006-02-02 14:15:57 EST 
This is a request to improve the usability of the various WTP project wizards.  My focus is on the Dynamic Web Project wizard but this could apply to all project types.

The request is to move the facet preset selection from the second page of the wizard to the first page and present it more as a "project type" configuration.  Example Web project types might be a Basic Web project (with no additional facets defined), a JavaServer Faces project, or a Struts Web project.

This wizard work would be in conjunction with an enhancement of the facet preset extension point so that an icon and description could be provided for each preset.

In order to make this use of facet presets effective, presets should be defined at the product offering level, so that products building on top of WTP determine which facet presets (project types) they want available to their users.

See the attached screen mockups for how the wizard might look.

Comment 6 Jim Zhang 2006-02-03 10:21:26 EST 
I'd like to comment on the interaction mechanism b/w facets and runtimes.  Currently the interactions go both ways, meaning the initial selection of the runtime on the 1st page primes the facets selection on the 2nd page.  But as you start picking your facets, the available runtimes list gets shrinked/expanded according to the facet support by the runtimes.

Now if we make the facet selection available on the 1st page, this kind of mutual interactions will be confusing.  For instance when the wizard first comes up all the runtimes are available, now if I select "Struts web project" then the runtimes not supporting struts will disappear from the list.  Users have to know that they need to select the empty entry to make all runtimes appear again.  This is very confusing.

The more intuitive way to handle this is to not have facet selections trigger runtime list filtering such that all runtimes are always available in the drop down.
Comment 7 Konstantin Komissarchik 2006-02-03 11:39:39 EST 
I tend to agree. The current behavior even without putting preset selection on the first page is less than optimal. The first page should always be in control. By going back the user expects that any changes made on page 2 may be lost.

Here is how I am thinking it should work if we add preset selection on the first page:

Runtime selection on the first page continues to drive the default facet selection. It now also controls the available presets that are visible on the first page. When a new a runtime is selected, the list of available presets is refreshed. Then, if there is a preset that matches the default facets specified for the runtime, that preset is selected. Otherwise, a "<custom>" preset is selected. Both runtime selection and preset selection should be sticky (locally to that wizard, not globally across all wizards).

The first page of the wizard would only show presets with this attribute set, while the second page (the facet page) could show all presets.  I didn't propose this initially simply because it adds a bit more complexity.

Yet another alternative would be to define a separate extension point used specifically for defining project types.  It could either point to a facet preset or to a set of facets.

Comment 25 Konstantin Komissarchik 2006-04-10 13:26:20 EDT 
In response to Comment #16 ...

1. The reason that the runtime selection is available on the first page is to enable basic project creation usecases to be performed completely from the first page. The reason that the runtime selection is still available on the second  page is to support more advanced use cases, such as selecting multiple runtimes or figuring out which facets are supported by which runtime without going back and forth in the wizard.

2. Target runtime is used for controlling which facets are selectable for the project. User can only select facets that are supported by the target runtime. The target runtime is also used while setting up project classpath.

3. Runtime and facet selection can be done in either order. There is one caveat here. When a runtime is selected on the first page, it resets the list of facets to the default list specified for that runtime. So if the user goes to the second page, picks some facets, goes back to the first and picks the runtime, their facet selection will be overidden. To avoid that, the user should use the second page runtime selection, which does not change the list of selected facets.

1. Runtime selection indirectly controls which presets are available since runtime selection filters the list of available facets and any preset that includes an unavailble facet is not shown.

2-3. Presets explicitly list all facets (and their versions) that should be selected when the preset is selected. If the preset does not list all of the fixed facets or lists any facets or versions that are not available, the preset will not be shown.

4. I would definitely not use the term "Project Type" when refering to presets. Most people would not say that two webapps that have slightly different sets of facets have different project type. I am thinking of using the term "Common Configurations" to refer to presets (at least on the first page).

Comment 31 May Zhu 2006-04-17 13:47:11 EDT 
Kirk Grotjohn, Janet and I had a series discussion to understand the scenarios in order to enhance the wizard. Based on your answers and an assumption that user has to select a target runtime on first page, and the selected runtime determines available presets and facets, we provide the following changes, use cases and corresponding designs. 

Changes: 

On page 1: We use “Project Sets” instead of “Presets”. We add “Other” to the drop down list. We move the contents on the original second page to a dialogue invoked from “Customize” button. (see attachment "Page 1")

On page 2: We paraphrase the headline (just a sample) and use “Show/Hide Details” instead of “Show/Hide Runtimes”. Facet runtime and explanation such as constraints are combined in one panel as “Facet Details”. (see attachment "Page 2")

Use case 1: 

As user selects a target runtime, available presets are displayed in the drop down list on the first page. If user doesn’t find his/her desired pre-defined preset from the list, he/’she will choose “Customize”, then a dialogue will be opened where all facets available for selected runtime will be listed. As user goes through the list and selects his/her desired facet, if “Show Details” window is open, relevant information including explanation of the facet and facet runtime will be displayed. The selected target runtime is always checked in the “Available Runtimes” window and not allowed to be unchecked. After user finishes his/her selection, he/she can persist his/her selection by clicking “Finish” and a new composite of preset will be returned to first page. If user wants to give a name to this new composite, he/she can click “Save” button on the second page. 

Use case 2: 

As user selects a target runtime, available presets are displayed in the drop down list on the first page. If user selects a pre-defined preset and knows what that preset means to him/her, he/she would click “Next” to the page “Web Module” which is the last page of the original wizard. If user wants to look into the content of the selected preset and do something there, he/she will click the “Customize” button where a dialogue is brought up showing the view with facets consist of the selected preset checked and other facets available to the selected target runtime unchecked.  If user de-select or select any facet, “Other” will be populated in the combo box on the second page. This means as long as the composite of a preset is changed, a new preset is formed. User can either give it a name by clicking “Save” or leave it as a default “Other” preset. 

As the relationship among runtimes, presets and facets is complex, we tried to hide the complexity from user as much as possible. Hope our thoughts and mockups here can trigger a better solution to improve the usability of current facets wizard. We will also keep working on it based on your feedback and suggestions.

Comment 34 Konstantin Komissarchik 2006-04-17 17:37:45 EDT 
I very much appreciate the effort that you guys are putting into this and I hope you keep working with me to improve the usability of this ui, however I don't think the latest batch of suggestions are going to work out either. :(

Let me explain point-by-point...

1. **Changing the facets selection ui from a wizard page into separate dialog accessible via a customize button.** The biggest problem with this is that it reduces the discoverability of the facets selection ui. Novice users are not going to know to click the customize button and so are less likely to discover the facility. On the other hand, hitting next on the first page and being faced with a list of labeled checkboxes is pretty intuitive (standard installer experience). The other problem is that it makes the wizard less linear. Facet selection determines which wizards pages are shown. Having a separate dialog control pages in the wizard would not be intuitive. Another problem is that it makes the project creation behave differently than add/remove. For add/remove, users start with the facet selection being the first page. Having creation and add/remove behave the same is benificial for the sake of user familiarity.

2. **The facet selection mockup removes the runtime composition ui.** The runtime composition ui is critical in helping users understand the capabilities of their available runtimes. Without having such ui, it would be difficult for the user to figure out why Tomcat Runtime 1 supports java facet 5.0, but Tomcat Runtime 2 does not. (answer: Tomcat Runtime 1 composition includes jre 5.0, while Tomcat Runtime 2 composition includes jre 1.4).

3. **The facet details section.** This idea is intriguing, however there is not enough space in that area to display all of the information that we need to display. I think that our best bet regarding this would be to utilize the dynamic help ui to display some of this information. Also, in the current mockup the details section and the list of runtimes are grouped together on the hide/show details panel. That seems odd to me as they are not related in any way. 

4. **Renaming preset as "Project Set".** The term "Project Set" to me suggests a set of projects, which is not what we are trying to represent. I will send an e-mail to the dev list to solicit ideas. What do you think about the term "Common Configurations" or other varients around the term "Configuration"?

PS: One of your starting assumptions is incorrect. The user does not have to select a runtime on the first page. In fact, the user can create a project without selecting a runtime at all. Some users will also want to take advantage of the advance facilities regarding runtime selection that are offered only on the second page.
