Description Rob Hughes 2005-04-25 17:41:24 EDT 
The Workbench currently has a list of predefined services/registries/managers
available through the IWorkbench interface.  It would be nice to be able to
register application specific services/registries/managers with the workbench
which could then be accessed through IWorkbench.getAdapter().

This could be done by adding a method to the IWorkbenchConfigurer interface:
public void registerService(Class key, Object service);

And the corresponding WorkbenchConfigurer implementation:
public void registerService(Class key, Object service) {
    Map services = ((Workbench) getWorkbench()).services;
    // prevent applications from changing already configured services
    if (services.containsKey(key)) {
        throw new IllegalArgumentException();
    }
    
    services.put(key, service);
}

Also note that IDisposable (from the experimental component framework) has been copied into our main code base.

Also, take a look briefly at org.eclipse.ui.services and org.eclipse.ui.internal.services.  I apologize in advance for the name "Sources" (it went out late in 3.1 and I regret it). Comment 11 Paul Webster 2007-01-18 08:15:25 EST 
We were looking at perhaps backing IServiceLocator with the OSGi services, but we aren't sure they will support how we use services.  Just for information, here's what I know about how we use services:

For most of the services we've provided so far (IContextService, IHandlerService, ICommandService, IBindingService) we provide 3 things: the service behaviour, potentially service scoping data, and lifecycle.

The IServiceLocator (our service registry) is organized hierarchically.  For example, the workbench had the global service locator, each workbench window has its own SL parented off of the workbench SL, each part site has its own SL parented off of its workbench window SL.  A request for a service returns the service registered with that SL, or it passes the request up the parent chain.

A service can take advantage by creating its own hierarchy.  For example, the IHandlerService registers a global handler service against the workbench SL and provides Handler Service behaviour.  It also registers a slave handler service (parented off of the workbench handler service) against each workbench window SL, and a slave handler service (parented off of the workbench window handler service) against each part site SL.

The service hierarchy allows us to provide the service scoping data.  The slave handler service modifies the active when expressions as appropriate, before passing it up the chain to the global handler service.  The slave service also keeps track of any activations that happen through it.

The lifecycle is provided on dispose.  When the PartSite is disposed, it will dispose its SL.  The SL disposal will dispose all services that are registered against it.  This allows the part site services to clean up.  In the handler service case to deactivate any handlers that were contributed through it.

The http://www.eclipsecon.org/2006/Sub.do?id=185 presentation has some good slides that point out the basic difference between using an extension point, OSGi service, or OSGi declarative service.  If you wanted your service to provide behaviour, loose coupling, and allow pluggable implementations, the OSGi services will provide what you need.

Our approach also allows us to massage data passed through the service, and do cleanup with the lifecycle hooks.

PW
