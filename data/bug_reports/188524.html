Steffen: fyi, I'm nearly done with a solution to this.  It goes along the lines of our last discussion.

This will be a pretty huge change which means that the UI Bridge plug-ins that add focused UI facilities will have a separate lazy load lifecycle that does not get started until there is a context activation.  These plug-ins will still be able to do their usual extension point contributions (e.g. Java stack trace URL detector) and start without initializing their own extension points.  Once something triggers the activation of a task context happens then the focused UI facilities load themselves.  The idea up until now was that it was OK for those facilities to load themselves as long as they didn't do anything if no task was active.  But this was still causing too much activation on startup, a significant portion of which was happening synchronously even though it was in asyncExecs (due to the object locks that we found as per Thomas Watson's comments on bug 191193).  Thankfully all of the trickiness is encapsulated by a new AbstractContextUiPlugin (name is tentative), which extends AbstractUIPlugin.
