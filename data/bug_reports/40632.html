Description MG 2003-07-22 18:50:55 EDT 
When using Open Resource window you should be able to rc on any of the items
found and open choose to open in the text editor instead of having to accept the
default open behavior of the resource.
Comment 1 Tod Creasey 2003-07-29 09:35:45 EDT 
What do you mean by rc?
Comment 2 MG 2003-07-29 12:22:41 EDT 
right click.
Comment 3 Tod Creasey 2003-07-29 12:45:12 EDT 
This is not consistent behaviour with our other dialogs - by doing this we 
would create view like behaviour in the dialogs.
Comment 4 MG 2003-07-29 13:44:16 EDT 
Alright then how about an extra piece to the dialog that says open with which
defaults to default but allows you to select the same thing as the views do when
you rc an item.
Comment 5 Tod Creasey 2003-07-29 13:54:21 EDT 
To be considered for future release.
Comment 6 MG 2003-07-29 14:09:51 EDT 
This is fairly important.  Take the following use case:

I have xml set up to go to textpad an external editor without markers.

When I use open resource to find this resource I like it to expand to the place
where this file exists.  I often use the find resource to go to a place in my
source tree.  

If use the open resource to find an xml file and then click the xml file, that
xml file is opened in an external editor without the navigator synching to the
place where this file exists.

Perhaps for this release you could make the navigator synch to where this file
exists in the tree?  That would at least minimize the inconvinence.  At least
this way I could then close the file in the external editor and go back to
eclipse where I could rc the file and select open with text editor.
Comment 7 MG 2003-07-29 14:46:15 EDT 
I'll log the synch thing as a different bug.

Comment 9 Eugene Kuleshov 2004-10-19 21:50:05 EDT 
It could be quite interesting to have functionality of "Open Type" or "Open
Resource" dialogs as a views. This way you can activate them using Alt-Shift-Q
and provide convinient popup menus to relect Navigator/Package Explorer views
(Open with..., Compare to each other, OpenType Hierarchy, Copy, Delete, etc).

filter feature for any tree-based views. So, if implemented it would allow to
filter out required resources for Navigator and Package Explorer. I'm not sure
about complexity of such generic implementation, but it may be easier to
implement that only for those two views.
