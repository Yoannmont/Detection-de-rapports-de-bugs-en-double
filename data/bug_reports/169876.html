Comment 29 Brad Reynolds 2007-03-26 18:46:35 EDT 
In thinking about this more I'm becoming more comfortable with the idea that returning a java.util.Date or any structure that represents an instance in time would be problematic.  What I'd like to propose is that we return a value that represents exactly what the control represents.  So when the DateTime is constructed with either SWT.DATE or SWT.CALENDAR we return a type, org.eclipse.jface.databinding.swt.Date (naming is up for discussion), that provides access to the date values of the widget.  When constructed with SWT.TIME we return a type, org.eclipse.jface.databinding.swt.Time (naming is up for discussion), that provides access to the time fields of the widget.  By doing this we're externalizing the conversion and defaulting of values into a realm that the consumer has control of (i.e. converters).

The way I see it the benefites are:
1. Observable doesn't default values.
2. Observable doesn't make assumptions of the expected time zone.
3. The conversion of the value is left up to the consumer.

The disadvantages are:
1. The conversion of the value is left up to the consumer.  This forces the Date complexities on the consumer and they will need to take extra care to ensure that the value being set in their model is what they expect.  We'll also need to answer the question of if we will be able to provide default converters for the process.  But this disadvantage is a propagation of the poor support for dates in java which will hopefully be corrected in the future[1].

Thoughts?

Comment 43 Matthew Hall 2009-02-27 17:50:31 EST 
I had an idea that a new custom date+time observable that depended on separate observables for the date and time components would be the right approach.

Example

DateTime dateField = new DateTime(composite, SWT.CALENDAR);
DateTime timeField = new DateTime(composite, SWT.TIME);

// WidgetProperties.selection() would automatically observe either the
// date or time attributes as dictated by DateTime.getStyle()
IObservableValue dateFieldSelection =
    WidgetProperties.selection().observe(dateField);
IObservableValue timeFieldSelection =
    WidgetProperties.selection().observe(timeField);

bindingContext.bindValue(
    new DateAndTimeObservableValue(dateFieldSelection, timeFieldSelection),
    BeanProperties.value("approvalTime").observe(transaction) );

The idea with DateAndTimeObservableValue (DATOV) is that it has two dependents, a date observable and a time observable, both of value type Date.class.  DATOV's takes the date portion from the date observable and the time portion from the time observable.  Setting a value date on a DATOV overrides only the date components of the date observable (leaving the time components intact), and overrides only the time components of the time observable (leaving the date components intact).  Thus we do not make assumptions about the dependent observables, we just get/set only the applicable attributes for each one.

WidgetProperties.selection() for DateTime(SWT.CALENDAR or SWT.DATE) would yield a Date with the year, month and day fields set, and the hour, minute and second fields at zero.

Likewise, WidgetProperties.selection() for a DateTime(SWT.TIME) would yield a Date with the hour, minute and second fields set, and the year, month and day fields at zero (i.e. on the epoch).

This gives clients an easy way to bind dates when they are interested only in the date and not the time of day:

bindingContext.bindValue(
   WidgetProperties.selection().observe(dateField),
   BeanProperties.value("orderDate").observe(order) );

Thoughts?

I came across snippet #30 yesterday and I'm already using it :-). DateAndTimeObservableValue works great, so I would consider it VERIFIED (>04062009).

As a side note: one thing that was a bit non-obvious to me, was how to preserve the other half of the date/time component if one only binds to a single DateTime widget (the snippet uses two). Basically because the DateTime widget only supplied time or date, one has to preserve the other information. I used a writeable value for that:

// bind to DateTime control (SWT.TIME) and preserve date
timeObservable = WidgetProperties.selection().observe(control);
timeObservable.setValue(value.getValue());
dateObservable = new WritableValue(timeObservable.getRealm(), value.getValue(), Date.class);
dbc.bindValue(new DateAndTimeObservableValue(dateObservable, timeObservable), value);
