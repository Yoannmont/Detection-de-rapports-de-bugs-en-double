Description Pascal Rapicault 2007-03-26 11:01:59 EDT 
The processing of artifacts that must be unzipped (plug-ins that are normally stored as folder (e.g. org.eclipse.pde.build)) goes as follow:
 - they are first mirrored into the download cache as part of the collect phase
 - then they are exploded as part of the install phase and bashed into the folder that contains the eclipse.cache. 

Because this last operation does not involve the artifact repo, the repo does not know that it contains the artifact in its runnable form and is therefore never able to share it with other local install, causing the download to occur again.

This should be fixed and the exploded artifact should be added to the repo representing the eclipse.cache.
Comment 1 Simon Kaegi 2007-07-23 12:57:29 EDT 
I don't really like the idea of mirroring to the cache and then removing artifacts and unzipping since this means we can't treat the cache like a local artifact repo. There might be value in insisting that the artifacts are always zipped in the Artifact repo and then let the touchpoint do the work to unzip it to some tp specific location. This of course would waste some disk space.

Perhaps we could avoid involving the touchpoint or somehow integrate the information that the artifact has been unzipped back into the repo? Currently, where is the fact that the artifact needs to be unzipped stored? In the IU?

One option would be to store this information in the artifact metadata and then allow the artifact repo the "option" of unzipping the content as part of its mirroring operation. Any transport of the artifact would still use zipped format. 

e.g. ultimately up to the Artifact Repo to decide what to do.
1. mirroring to the local artifact repo / cache would result in unzipping
2. mirroring to a file store artifact repo would not.
3. a hybrid might hold onto both zipped for byteserving and unzipped format for use in the system.
Comment 2 Jeff McAffer 2007-08-11 19:24:22 EDT 
I agree with the sentiment of reducing copyies etc but we should also avoid building in specific structures to general mechanisms.  That is, it is a quirk of the Eclispe touchpoint that it can use artifacts in their "native" form (i.e., as they come from typical repos).  Certainly we want to leverage this if we can but in the end, if the disk burden of having both the artifact and the touchpoint installed form is too much then we should blow away the artifact repo version.

Comment 4 Simon Kaegi 2007-11-22 22:41:41 EST 
Created attachment 83595 [details]
Folder support in EclipseTouchpoint

This was my first try and although it allows an SDK install and passes our automated tests I really don't like it. I've added a BundlePool artifact repo in the eclipse touchpoint that is folder aware but otherwise is SimpleArtifactRepo.

What I found was that we're not currently well setup to expose SimpleArtifactRepo outside the artifact repo bundle for extension. e.g. lots of internals and package privates. We can do this certainly however we might want to think about this first.

If we wanted to support extension of the SimpleArtifactRepo (like I tried) we would also have to take a look at how SimpleArtifactRepoIO is restoring the repositories as it's currently tightly bound to creating a SimpleArtifactRepo. The data format it uses is relatively generic so this may not be a big deal.

In this patch I actually am getting around the serialization problems by wrapping SimpleArtifactRepos and augmenting their API. Unfortunately as far as the repomanager is concerned it's just a SimpleArtifactRepo. This is what I like least about this current patch.

--
I'm going to try doing something inside the artifact repo bundle and see what it looks like.
Comment 5 Simon Kaegi 2007-11-23 01:08:56 EST 
Fixed in HEAD.

We're now using the proper RepositoryFactory extension points to properly allow creation of a Bundlepool repository. Since our serialized form is identical to SimpleArtifactRepository we can extend the SimpleArtifactFactory and let it deal with the details of interacting with SimpleArtifactRepositoryIO. The BundlePoolFactory wraps the SimpleArtifactRepository but again still lets us register the correct repository type with the RepositoryManager.

We still have to make public a number of methods on SimpleArtifactRepository to allow the BundlePool to wrap it. If we moved some of the common artifact repository code (particularly around creating processing steps) up into AbstractRepository this might be a little cleaner.
