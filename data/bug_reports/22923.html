Description Jeff Brown 2002-08-27 17:53:07 EDT 
The lack of support for Unix permissions causes serious usability problems 
when working with C/C++ projects in such an environment.  It is not practical 
to manually chmod scripts and other executables in (very) large projects.
Comment 1 Michael Valenta 2002-08-28 10:36:01 EDT 
To elaborate, a project on a Linux/Unix machine may contain files whose 
execution bit is set. When this project is first uploaded to a CVS reposiory 
using Eclipse, the execution bit is not transfered. To correct this problem a 
user must manually perform a chmod on the server for each desired file. Once 
the executable bits are set, Eclipse use will not unset them.

Unfortunately, Java has no facility that I know of to obtain the executable bit 
so we would require some other mechanism to get this information.
Comment 2 Jeff Brown 2002-09-02 23:25:05 EDT 
I'm not above suggesting adding stat and chmod natives to make this work.  
Ideally this should be done by Core, but the functionality is sufficiently 
useful to me that I might consider hacking a temporary CVS-only solution in 
the form of a patch.

Comment 9 Tom Tromey 2003-02-18 14:03:36 EST 
If I check out gcc with Eclipse, I lose all the executable
bits on the configure scripts.  This means I have to chmod

Comment 10 Kevin McGuire 2003-02-18 14:58:10 EST

Comment 11 Tom Tromey 2003-02-18 16:41:15 EST 
I think the non-portability isn't a major problem.
If I was going to write a real fix for this, rather
than the hacks that I have, I would put new code in
org.eclipse.core.resources to let us manipulate the
executable bit.  There is already native code there
for similar purposes.

Checking in executable scripts is actually not uncommon.
It is a contentious issue, but many projects have chosen
to check in their configure scripts and other auto* support
code.  At least two major free software projects, gcc and gdb,
do this.  I'm certain many more do as well.

Comment 13 Jeff Brown 2003-03-07 21:43:22 EST 
The problem was a particular nuisance at my last workplace where the CVS 
repository contains a large number (hundreds!) of pre-built binaries and 
modules making up a customized Linux distribution.  Most of the build scripts 
assumed that permissions were set appropriately on all files because there was 
no need to go through the nuisance of doing things otherwise at the time.

I am not concerned about the cross-platform portability issue.  CVS 
permissions are a platform-dependent hack anyways.  I understand that it is 
not desirable to place this type of code in the CVS Team branch, and Core 
Resources are not willing to commit to support at the moment either.

Here is a short-term alternative that requires minimum commitment from 
everyone.  CVS Team provides an internal (non-API) extension point.  A 
separately distributed platform-specific plugin binds to this extension point 
a supplies a proxy interface as described below.  The plugin may be 
independently developed by a third party with the understanding that the 
interface is "unstable".

The actual interface only requires two services like those defined in the 
patch from a few months ago: getCVSPermissionsFor(IResource, String), and 
setCVSPermissionsFor(IResource, String).

Pros:
- Better CVS command line compatibility.
- Boon for many C/C++ developers.
- CVS Team code will already contain the necessary logic and code paths to 
implement the "right fix" if/when Core Resources decides to go ahead with it.

Cons:
- Obviously does not resolve other permissions issues elsewhere in Workbench.
- Therefore may confuse developers.

Nifty non-CVS things I just came up with:

As a temporary hack, someone could implement a plugin action contribution for 
permissions wrangling on POSIX compliant platforms.  On Windows, it would be 
neat to hook the Navigator into the COM interfaces that make File->Properties 
possible throughout the OS.  Presto, instant permissions wrangling on Windows 
too.
