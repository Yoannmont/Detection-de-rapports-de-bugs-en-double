Comment 4 Jason Sholl 2006-06-22 14:53:02 EDT 
The fix will have to be a bit more elaborate because of dynamic component
references.  In the example you mention in the initial bug report, the J2EE
Module dependency from the web to EJB is dyanmic (if it isn't then it should
be).  In this case there should be no entry in the .component file, but rather
the reference is computed from the web's META-INF/MANIFEST.MF file.  So, the
proper fix would be to setup the .project dependnecies when the MANIFEST.MF
file changes or the .component file changes.  Fortunately, a listening
mechanism is already in place which picks up all the key modifications and
filters out all the noise.  This listening mechanism is the
J2EEComponentClasspathUpdater and is used to refresh the
J2EEComponentClasspathContainer (aka EAR Libraries).  The refresh method in
J2EEComponentClasspathContainer is an ideal place to fixup the .project
references.  This way the J2EEComponentClasspathContainer could keep track of
the .project references it created (vs the ones a user might create) and remove
them if the MANIFEST.MF changes to no longer include a particular entry.

Is this really necessary for 1.5 or can it wait for 1.5.1 ?
Comment 5 John Lanuti 2006-06-22 15:38:44 EDT

Comment 6 Rob Frost 2006-06-22 17:16:02 EDT 
Yes, in that example there is no component file dependency (just a manifest).

The assumption for the approach I outlined is that the code which executed the manifest change would explicitly create/remove the project ref (and potentially handle the component dep); so, for both standard (i.e. component) and dynamic (i.e. just manifest) dependencies you always execute the Create/RemoteReferenceComponentOp (and just set a boolean property indicating whether or not you want the operation to create/remove component dependencies).

I had proposed that fix because it seemed like a straightforward patch if we wanted to fix this for 1.5 given that it mirrors what had been done prior to the recent changes. 

It would definitely work to let the J2EEComponentClasspathUpdater handle the .project references (it would need to take into account both manifest and component dependencies but I believe it listens for both) and that may be preferrable; the project reference modification logic can then be stripped out of the operations that are executed via the UI and refactoring logic.

Regarding 1.5 vs 1.5.1: 

When the .project reference is missing, an incorrect build order can be computed (the user might get lucky and the default will be OK but the fact that the user created a dependency between the projects implies a build order constraint); if the build order is incorrect and autobuild is enabled, another build cycle is triggered when errors due to an invalid build order occur (i.e. a required class is contained in a project that has not been built; there may be cases where this doesn't happen...), however, if the user has auto-build off, the build will fail (the error message does clearly indicate the nature of the problem in this case). So, a 1.5.1 target is not ideal but should be OK (would have been good to address in 1.5 if a low risk fix were in hand, but looks like time has run out and the right fix may be more complex).

Comment 9 Rob Frost 2006-07-11 11:47:20 EDT 
Here was the build order scenario I hit:

-Turn off auto-build
-Create a web project named "web" (add it to an EAR)
-Create a util project named "zutil" (add it to the same EAR; you name it zutil because default build order is alphabetical when there are no project refs)
-Create a module dependency from the web to zutil
-Verify that the computed build order lists "web" before "zutil" 
-Create a class in web that depends on a class in zutil (you will be able to resolve the zutil class within the Java editor since the zutil project is exposed via the Ear Libraries cp container)
-Execute a build all and you'll get the following error on the web project:

Severity and Description	Path	Resource	Location	Creation Time	Id
The project cannot be built until its prerequisite zutil is built. Cleaning and building all projects is recommended		web	Unknown	1152632451656	425330
