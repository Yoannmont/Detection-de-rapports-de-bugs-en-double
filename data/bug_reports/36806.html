Comment 13 David Klaus 2009-10-02 14:05:38 EDT 
Created attachment 148673 [details]
Zip archive of research results

This is a zipped archive of research that was done for this issue and tracked in the CygwinX group's bug: http://sourceware.org/bugzilla/show_bug.cgi?id=9848 issue.  I have found a number of X-Server terminal programs that did not have the issue, and used WinXServ as the basis of a comparison to try and determine the source of the issue.  This archive contains:
1) Dialog Demo Commands.txt: This is a minimal program and compile/link instructions to reproduce this problem.
2) Three Wireshark captures: Dialog Demo CygwinX.pcap (where the hang occurrs), Dialog Demo WinXServ.pcap (where it does not), and Dialog Demo CygwinX with twm.pcap (where it does not).  What the 1st capture shows is that this is a run-away process at the X11 level.
3) Three xwininfo captures: Dialog Demo CygwinX xwininfo.txt, Dialog Demo WinXServ xwininfo.txt, and Dialog Demo CygwinX Open Only xwininfo.txt.  The latter capture used a Java program that only opened the window and did not draw the button.
4) Two cropped screen captures: Dialog Demo CygwinX.bmp and Dialog Demo WinXServ.bmp.  These show the drawn window where it freezes and when it completes.
5) A startup file CygwinX startwin.bat: This is a modified starwin.bat for use with CygwinX started in rooted (without -multiwindow) and activates twm (Tiny Window Manager).

After determining that the problem was a run-away X11 process, I used jdb and was able to determine that shell.open(); was the call that was causing it.  With that John Turney (whom we owe a big thank you) was able to compile the Linux Motif version of this and determined the following:

------- Additional Comment #6 From John Turney
Thanks.  Due to this I managed to put together what I think is a reproduction of
the problem with the client running under Linux (after building the SWT Motif)

The client seems to get stuck in a loop in Shell.setVisible()

do {
	display.update ();
	if (isDisposed ()) return;
	iconic = minimized || (shell != null && shell.minimized);
} while (!isVisible () && !iconic);

isVisible() calls getVisible() which fetches the window attributes and checks
the visible state like this:

	if (attributes.map_state == OS.IsViewable && reparented) return true;

And as far as I can see 'reparented' is only set by a ReparentNotify event.

It seems to me the logic of this code is going to fail in 2 situations:
a) when we have no window manager running, so no reparent occurs.
b) when the window manager is rootless, and the parent window containing the
window frame is a native window, not an X window, so no reparent occurs.

Testing with Cygwin/X in rooted mode with twm running, and in an XDMCP session
to a KDE desktop, the DialogClass test application seems to work correctly.
------- Additional Comment 

I have confirmed that running Cygwin/X as requested does work around the issue on our AIX system, indicating that he is one the correct track.  I am very much a beginner at Java, and am simply playing the messanger, so would not feel comfortable working on the patch at this level.  If someone could follow-up on this, I would appreciate it.
Comment 14 Jon TURNEY 2009-10-04 14:28:25 EDT 
Perhaps this bug should be retitled something like "SWT/Motif hangs if Window Manager does not reparent"

I'll attempt to workaround this in Cygwin/X in the rootless Window Manager case, but the 'no window manager is running' case probably needs a fix in SWT.

This 'busy wait until reparented' behaviour was introduced in revision 1.27 (Thu Jun 6 18:07:51 2002 :-) ) of Shell.java, which references bug 18838

https://bugs.eclipse.org/bugs/show_bug.cgi?id=18838

Perhaps a way to fix this is for Shell.createHandle() to walk up the parents of the window up to the root window, checking for a window where SubstructureRedirectMask is selected, and if none is found, turn off the wait for reparent behaviour by setting reparented = true?

This is still somewhat inadequate as, while if nothing is listening for SubstructureRedirectMask, the reparent is definitely not going to happen, but it might be that something listens for it, but doesn't reparent...

On second glance, this kind of heuristic is going to fail on most tiling WMs (which seem to generaly select SubstructureRedirect but don't reparent)

The manpage for dwm tells people to use 'export _JAVA_AWT_WM_NONREPARENTING=1' to avoid this problem, so by analogy, here's a patch to Shell.java which adds a check for the env var _JAVA_SWT_WM_NONREPARENTING to turn this behaviour off.
