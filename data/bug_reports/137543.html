Description Mik Kersten 2006-04-19 13:38:25 EDT 
In order to make working with tasks as simple as possible, so far we have decided not to expose subtasks in the Task List.  However, a few cases pop up where this complicates interaction instead of simplifying it.  Some examples we've come across:

1) Dependent Bugzilla reports.  Several projects use the "depends on" attribute to structure related reports.  The Task List could make this association explicit by showing the dependency tree of the reports.

2) Finer grained local tasks related to repository tasks.  For some, Bugzilla reports represent a large amount of work (e.g. visible to a customer).  It can be desirable to break these up into several local tasks which can have separate contexts.  To support this local tasks could be made children of repository tasks, and both the contexts and times "rolled up" to the parent task as desired.

3) Groupings such as XP stories.  Similar to above, but in this case the parent is more similar to a task container that can contain any kinds of tasks, but also be contained within a folder.  

To support this we would need to ensure that this use of task hierarchies is optional and does not clutter the UI for those who don't need to make use of this support.  We also get a major benefit, because it allows us to relate task context to each other (e.g. to activate all the subtask's contexts together for the purpose of review, etc).  There are also some open questions:

* Should task containers be allowed only at the root level?  This seems simplest and aligns well with (1) and (2), but may compromise (3).  If parent tasks aren't containers, do we allow "go into" for them?

* Do we support indefinite nesting, or just one level?  

* What does it mean to activate a parent task?  One possibility is to activate all subtasks, and to have the contexts distributed evenly among each while all are active.  This is already supported in the context model.

Comment 2 Mik Kersten 2006-04-20 11:04:33 EDT 
Right, that wasn't clear.  If you have the following task structure
- 0: improve UI
   - 1: fix wizard naming
   - 2: add new icons
   - 3: make colors prettier

Then the question is what happens when you activate task #0.  I see two options:
1) Load the context for task#0 only
2) Load the context for tasks 1, 2, and 3 concurrently.  When you're working on task#0, the interaction will get added to each context, so that the work you do on #0 will show up in each of the sub tasks should you activate them individually.

Now, I'm pretty sure that activating multiple tasks is useful for stuff like code review, and we'll have that support before too long.  But back when we allowed multiple tasks to be active I never used this feature for working.  So maybe we just do (1) first and keep in mind whether/when something like (2) would be useful.
Comment 3 Peik Aschan 2006-05-10 02:17:17 EDT 
I'm bringing my views over here from bug 138832, about categories: 

First I must say that I don't work with repository tasks (even though I'd like to), since we don't use Bugzilla or JIRA. So all the tasks I work on are local to me. So I won't comment on any issues related to that.

Could you clarify: 
1) Terminology: Is container=category=folder, as opposed to a task, or is a parent task also a "container"? I would understand "container" as any type of object that can hold tasks within it, be it a category or a parent task.
2) Would allowing for parent and sub-tasks make categories obsolete, or would there be two different container mechanisms: categories and parent tasks? My initial feeling is that there should be only one type of container, but I'm not sure about this...

What about if only parent tasks were task containers, and instead of categories we would have labels, so that a task could belong to several lablels (as in for example Picasa or Gmail)? This would allow for grouping tasks for e.g. code reviews, without breaking the hierarchical model of parent and subtasks.

Comments on the questions: 
1 and 2: I think several levels and nesting should be allowed. It's up to the user if he wishes to use them.
3: I would go with the simplest option (1) at first.

A bit about how I work: 
I usually organize tasks into categories: some are related to a large long-lived task, some to a specific release. Since I'm sloppy, when a release ships, I have a lot of tasks that are still open (=incomplete). At that stage I mark all the tasks related to that release complete. This is curently done using multiple select+Mark complete.

The subtask model would work great in my case: If I want to mark the entire release complete, marking the parent release-related task complete would suffice. 

On the other hand, the long-lived tasks may still be relevant, even though they currently have no open subtasks. Instead of a category that would disappear in this situation, the parent task would still remain open.
Comment 4 Mik Kersten 2006-05-18 15:46:43 EDT 
Those are interesting questions Peik, and the various task management tools I have used have different in this respect (e.g. Microsoft Project makes tasks be containers no categories, Microsoft Outlook allows nestable containers, no subtasks).  I think we have to come up with something that's simple and doesn't force use of subtasks, but flexible enough to support the different representations of subtasks that various repositories have.  Regarding your questions, I'm leaning towards having containers (=category=folder, similar to tag) continue to be their own first class thing, and making tasks simply allow nesting.  All that nesting would be is a field on the task that indicates what the parent task.  So this nesting would only be a view on the tasks, and some other kind of repository could specify nesting that had a different concept of subtasks.  It would also allow local tasks to be nested under repository tasks (an additional reason to keep containers and tasks seperate).

We actually have a very similar use case to you, where we use the dependson/blocks of Bugzilla to structure our tasks, and don't consider the parent task completed until all tasks below it are, or end up wanting to mark all child tasks completed when the parent one is done.  What I think we'll do in the 0.5.3 cycle is start experimenting with Bugzilla tasks that have dependson showing up hierarchically, and then discuss further on this report.

Comment 6 Mik Kersten 2006-07-26 18:00:17 EDT 
In response to the ordering suggestion on bug 151655:

We this kind of dependency-based ordering is definitely important, but the idea is to represent it by nesting tasks as described here.  That way you would see the following in a query that matched one or more of those bugs:
- <query>
  - bug 1
    - bug 2
    - bug 3
      - bug 4 
      
Note that in order to show this as a dependency tree we will have to break cycles (e.g. bug 2 should bre a child of bug 3 if I'm reading right).  But in practice I have not noticed such cycles on bugs.eclipse.org and bugzilla.mozilla.org, which is why Bugzilla's dependency tree support can be useful.  Dependency cycles will be navigable via links in the editor.

If there are no objections to this approach I will bump the priority of this because I think that it would make it easier for us to work with the large numbe rof dependent tasks that we have, and force us to make contexts propagate.
