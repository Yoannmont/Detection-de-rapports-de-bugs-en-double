Comment 12 Wassim Melhem 2004-04-19 22:58:56 EDT 
Actually, if it is ok with the pde-build team, I am going to close this one as 
WORKSFORME, since the PDE Build/UI support is more than sufficient for this 
scenario.  The user just needs to understand the use of the build.properties 
file.

The build.properties file is a required file when you want to build a plug-in 
or a feature.  This file alone contains the rules that guide the build 
process, so it is not a file that would be nice to have, it is a file that you 
must have.

In your case, each plug-in you want to package in a feature must contain this 
file.  When a plug-in is shipped, this file does not usually come with the 
plug-in in the Eclipse install because it is used for development purposes 
only.  That is why when you import the doc plug-in, it is not there, so you 
have to create it manually.  Only you know what should go into it, so it is up 
to you to maintain it.

Now, as for what should go into each build.properties file:
Open the file in PDE's beautiful build.properties editor.  Go to the nice tree 
in the binary build section.  Select the files that you want to be packaged in 
the plug-in (doc.zip, *.xml).  Save the file.  Repeat for all plug-ins you 
want to include in the feature.

Create a feature, include all the plug-ins that you want in it and now you can 
create an update site and build the feature and everything will work fine.

Comment 16 Wassim Melhem 2004-04-20 12:28:16 EDT 
>it also means that the way the PDE tools wants people to work is somewhat 
>inflexible to the way they want to work.  Do you know what I mean?
I don't quite know what you mean.  The pde-build mechanism has a protocol that 
one needs to follow if you need to take advantage of it.  This requires the 
use of a build.properties file.  These are the rules of engagement and cannot 
be viewed as an inconvenience.  

This file contains the list of JARs to build, what source folders to compile 
into each JAR, what folders/files to include in the binary build, etc.  As you 
can see, this is all vital, not gratuitous, information that pde-build needs 
and only the user knows what should go in it.

Note that pde-build is based on the execution of Ant scripts.  So if you find 
the creation of a a build.properties file to be an incovenience, then a 
solution might be for your team to write specialized Ant scripts that packages 
the plug-ins in a way that is specific to your needs and that do not require a 
build.properties file.

I think the summary here is that Mandy is looking for something that has not 
been in the set of target scenarios for PDE.  PDE is focused on plugin 
*development* not particularly product or component packaging.  In the current 
view that implies tha the user has everything in their workspace (since they 
are developing it) and wants to deploy.

This is not to say that the requested function is bogus.  It is more like 
observing that JDT does not support JSP development.  By chance it does most of 
what you want but not quite everything because it was not a targetted scenario.

The packaging scenarios are interesting and going forward they should be 
supported more fully.

In the near term I believe that all the mechanisms needed are available either 
by the PDE UI or by the PDE build packager mechanism.
