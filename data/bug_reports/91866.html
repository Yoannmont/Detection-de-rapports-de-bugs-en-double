Description Steen Lehmann 2005-04-19 07:12:08 EDT 
The content assist in the XML editor makes many more suggestions than might be
expected, e.g. for a content model with a sequence (A?, B?, C?), the content
assist will always suggest A, B and C regardless of the existing content (e.g. B
and C may already be present). This means you can create invalid documents in no
time at all.

This turns out to be partly because AbstractContentAssistProcessor:1593 (in
o.e.wst.xml.ui.internal.contentassist) hardcodes the value
ModelQuery.EDIT_MODE_UNCONSTRAINED in all calls to getInsertActions, which
returns actions for the legal completions.

Setting the value to EDIT_MODE_CONSTRAINED_STRICT is better, but runs into a
bug: the text used for starting the completion ('<') is erroneously considered
#PCDATA, and so all suggestions are evaluated as invalid by the schema validator
used to sieve away illegal suggestions (this happens in
DOMValidator.createContentSpecification:144, in
xml.core.internal.contentmodel.internal.util).

Further, it seems that the DOMValidator determines if something is a valid
insert by inserting it and validating the result. This is a problem if the
content model requires a sequence of elements (e.g. (A, B, C)?). You will never
be allowed to insert 'A' by the validator since A by itself is invalid.

So this is really one enhancement request (stricter validation) and, should that
suggestion be accepted, two bug reports.
Comment 1 David Williams 2005-04-27 23:20:25 EDT 
Thanks Steen for this detailed request and analysis of (future) bugs :)

Might I suggest an alternative? What I'd like our direction to be is 
that content assist continue to be pretty loose and literally help complete
what ever one is typing (whether legal or not). 

I think for "contrainted" editing, there's a couple of other mechanisms. 
1) from the outline view, you should find things like "add child" which 
only proposes legal children of what's selected (e.g if already exists, or 
if order is important, then that one is suggested/allowed). 

2) I'd like to "duplicate" this basic constrained editing in the "quick assist" 
function (ctrl-1). Currently there's only "surround tag" in quick assist, but the 
suggestion I'm proposing is for there to be other constrained editing functions, 
such as "add child <abc>", "add required children", etc.

What do you think? Worth pursuing?

Comment 3 Steen Lehmann 2005-04-28 03:55:45 EDT 
Actually I quite like Max's suggestion - what if legal suggestions, those that
keep the document valid (or allows it to become valid as per my original
comments) are kept first in the list of completions, and those that would make
the document invalid could be shown further down the list in grey, or red with a
warning icon, or whatever (regardless of whether the tag is empty or not, IMHO).

I've already implemented an XML editor in Eclipse with a fairly strict content
assist (and yes, there is an amount of graph traversal involved to get the full
monty - 'potential validity', which could be left for later), but it's a great
help to be able to just hack away on the keyboard and end up with a valid
structure. XML editing by right-clicking outside of the text editing pane is a
lot slower, and in my experience you end up using the schema assistance a lot
less, if at all. It breaks your flow. Bad karma! :)

Another possibility is the UI designer's capitulation - creating a preference to
enable or disable strict validation in the source view. I really don't like
preferences where there is an elegant solution giving you both options in one,
which I think Max's proposal does.

Makes sense?

Comment 5 Steen Lehmann 2005-05-09 09:39:24 EDT 
Good point about badly-formed documents, David. 

Is support for DOM Level 3 Validation considered? Quoting from the spec: 

'This module provides Application Programming Interfaces (APIs) to guide
construction and editing of XML documents. Examples of such guided editing are
queries like those that combine questions like "what does the schema allow me to
insert/delete here" and "if I insert/delete here, will the document still be
valid."'

Instead of investigating too much you could just decide to support what DOM L3
Validation gives you - a go/no go on inserting elements at various points, for
example. 

The editor could then efficiently just always suggest everything (as now), but
have some visual indication in the code complete drop-down as to what elements
would make the document invalid, as discussed previously.

Comment 13 Craig Salter 2005-11-11 04:01:06 EST 
I've checked a fix for this into HEAD. You can enable this feature via the XML
Source preference page.  Change the 'Suggestion strategy' from 'Lax' to 'Strict'

suggestions will have an emphasized icon and will appear at the top of the list
of suggestions.  I'd like to be able to 'gray out' the labels of the 'invalid'

list won't let me (I'll open a bug for that and post back).  If anyone has icon
suggestions that may differentiate more clearly I'm all ears.
