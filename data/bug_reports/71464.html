Description Matthew Hatem 2004-08-05 09:53:39 EDT 
We have cases where we would not like the zoomed state of a view stack to zoom
out when a new view is opened or closed in the same stack.

To reproduce:
1. launch the workbench
2. maximize the "Tasks" view
3. select the "Show view" menu and show the "Console" view.

problem: the console view opens in the same stack that is zoomed, however the
layout is still updated and the stack zooms out.

Some users may expect behavior similar to the editor area, where the editor area
does not zoom out when a new editor part is opened in the same editor stack.

To demonstrate another valid use case for this, 
1. install the rcp browser demo example plug-in
2. change the plugin.xml such that the perspective is not fixed
3. show the history view
4. zoom the browser view
5. show another browser view via File | New Tab

problem: the stack zooms out, showing the history view again, this seems jarring
to the end user.

Comment 6 Stefan Xenos 2004-08-27 12:26:28 EDT 
Okay, the intended behavior change is fine, but the patch doesn't completely fix it.

There are currently several methods that try to decide when to zoom out
(WorkbenchPage.needToZoomOut(...), WorkbenchPage.partChangeAffectsZoom(...), and
PerspectivePresentation.partChangeAffectsZoom(...)), and they don't all agree on
the result.

I believe the correct logic is actually be quite simple:

Zoom out iff the zoomed part is not an ancestor of the part of interest but they
share an ancestor. In the case of stacks, we should treat the stack itself as
the zoomed part, not the active part in the stack.

This should fix a number of unzoom bugs.

Comment 10 Stefan Xenos 2004-08-30 16:41:26 EDT 
The infinite recursion is because WorkbenchPage.setState and PartStack.setState
both believe that they are the one to initiate all state changes.

Actually, each one should initiate different kinds of state changes:
- The workbench decides when to maximize/unmaximize a pane, and it notifies the

- PartStack can decide to change its preferred size at any time (and send a
notification to the workbench). The PartStack can choose between
minimized/restored when it is not maximized.

Comment 32 Stefan Xenos 2004-09-10 15:00:53 EDT 
Fixed in HEAD.

This patch makes zoom out behavior as lazy as possible: 
- activating a view or editor that is hidden due to zoom will cause zoom out.
- IMHO, bringing a view or editor to the front should cause zoom out if
necessary to make it visible, but bug 73500 needs to be fixed before this will
work well in all situations.
- opening or closing views and editors will not cause zoom out unless this also
causes an activation change.
- when selecting the activation order, the workbench will give priority to views
or editors that do not affect the current zoom.
- activating a view or editor that is visible in the current zoom will not
affect the zoom state.

Improved robustness:
- PartSashContainer's layout can be manipulated even when zoomed. 
- There is now only one definitive pointer to the zoomed part (in
PartSashContainer).
- Parts now use localized methods on their parent to zoom in/zoom out rather
than reaching to global objects (this pattern may be also useful for bug 46207,
where parts may want to have their own localized notion of maximizing their
children without actually maximizing the entire workbench).

Comment 35 Stefan Xenos 2004-09-23 12:54:08 EDT 
I200409230800 verified on Windows XP, multimonitor

Note to Matt:

Pre-patch, the workbench set all obscured views to size 0. Post-patch, it makes
obscured views invisible. The pre-patch technique was slow, created artifacts on
linux, and was not sending part hidden/visible messages to the obscured views. 

If you want to keep this part of the fix, you'll need the changes to LayoutTree
and LayoutTreeNode -- they allow PartSashContainer to flush all cached size info
in an entire subtree, which is necessary now that the redundant resizes have
been removed (otherwise, you will get some wonky layouts when you switch to a
maximized perspective and then unmaximize).
