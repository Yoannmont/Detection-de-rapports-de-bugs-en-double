Description Jon Shavor 2002-07-10 16:30:03 EDT 
I am currently running the F4 build of Eclipse 2.0.

I have an editor with some push buttons on them.  As I tab through my controls 
in my editor, I get to a push button which is not specified as being the 
default button.  When tabbing into it, it gets the default.  when tabbing out
of it, it loses the default.  This is working fine, but the next time I tab in, 
it gets the default, but when I tab out, the default remains.  If I tab to 
another button and press enter, I get the action for the default button, which 
is wrong because it is not the one with the focus.

Here is a proposed code fix.  This fixes my problem.  I am not sure if it 
breaks other things???:

Comment 1 Steve Northover 2002-07-11 14:37:27 EDT 
The default button follows focus.  You must mean that when you tab out to 
another widget, say an entry field, the default button where the "default 
remains" gets the return.

This behavior is currently by design but I am rethinking it.
Comment 2 Jon Shavor 2002-07-11 14:49:20 EDT 
Right now it is inconsistent.  The first time I tab into the button and out of 
the button, the button does not recieve the default state.  The second time I 
time into the button and out, it does recieve the default state.  I don't think 
tabbing should change the default state of a button.
Comment 3 Steve Northover 2002-07-12 11:09:04 EDT 
It's not exactly inconsistant.  It's just assuming that every shell that has a 
button wants a default button and if none is assigned, it will be the first 
button that gets focus rather than there never being a default button.  I'm not 
sure this is wrong or right.
Comment 4 Steve Northover 2002-12-06 09:37:14 EST 
*** Bug 27753 has been marked as a duplicate of this bug. ***
Comment 5 Steve Northover 2003-02-17 11:51:36 EST 
OK, after seeing application code that uses return to move from text field to 
text field in a data entry type window I can see that this design decision is 
really causing trouble.  The problem is that when the user gives focus to a 
push button, the "return takes you to the next entry field" strategy, 
implemented in application code, stops working.  The application can work 
around it, using the traverse callbacks but the button that got focus still 
draws as a default button and gives the visual clue that return should 
activate it.  This is wrong.  So, let's fix this.
