Comment 9 Silenio Quarti 2006-03-10 13:35:31 EST 
The problem here is that somehow the linked list that holds all windows in carbon (GetWindowList() and GetNextWindow()/GetPreviousWindow()) is corrupted. The eclipse main window is not found in that queue, so FindWindow() fails and all events for that window are dropped. This explains why the menu bar is alive and actually if a second main window might be alive too. If you press command+comma, it brings the Preferences dialog up. So Eclipse most of the time is just sleeping (not hung) (Display.sleep()) when the stack dump is taken.

Eclipse hangs (show the beachball cursor) when SelectWindow() is called because GetPreviousWindow()/GetNextWindow() for the main eclipse window returns itself (circular list).

We still do not know what causes the list to be corrupted and it is hard to track when it becomes corrupted since the problem cannot be reproduced easily.

Scott, do you if there is any problem in carbon were the window list can be corrupted?

The method below can be used to detect that the problem happen. Add the method to Display and make a call in Display.readAndDispatch() just after the SendEventToEventTarget(). The method checks if all SWT shells are in the carbon list.

public void detectProblem() {
	int count = 0;
	int[] windows = new int[100];
	int w = OS.GetWindowList();
	while (w != 0) {
		windows[count++] = w;
		System.out.print(w + " ");
		w = OS.GetNextWindow(w);
	}
	System.out.println();
	Shell[] shells = getShells();
	for (int i = 0; i < shells.length; i++) {
		int shellHandle = shells[i].shellHandle;
		if (shells[i].getText().length() > 0) System.out.println("main=" + shellHandle);
		int j;
		for (j = 0; j < windows.length; j++) {
			if (windows[j] == shellHandle) break;
		}
		if (windows.length == j) {
//			stats.dumpDiff(System.out);
			System.out.println("BAD");

}
Comment 13 Scott Kovatch 2006-03-13 14:26:46 EST 
According to DTS, the problem is that the Carbon Menu Bar is itself a window and part of the window list. If you're not expecting this, you can mess up the whole window list. The fix is to not use GetWindowList/GetNextWindow/GetPreviousWindow but instead use GetNextWindowOfClass (and related APIs) to dissociate the application's windows from the system's windows.
Comment 14 Silenio Quarti 2006-03-13 15:18:13 EST 
The problem seems to happen when there are two SWT.ON_TOP shells showing on the screen and one is a child of the other one. In this case, the yellow information window is a child of the Quick Fix window. Also,  the child window has to be in the window group (SetWindowGroup()) of the parent window. We are not exactly sure what is needed, since we were not able to reproduce the
problem if a simple snippet (only in Eclipse).

We changed the code to put all SWT.ON_TOP shells in the window group of the kFloatingWindowClass windows and this seems to have worked around the problem. Please try tomorrow's integration build and reopen the bug if you still can reproduce it.

Scott, we are not actually calling GetWindowList/GetNextWindow/GetPreviousWindow. We just used them to track the problem down and show that the window list was getting corrupted. We believe this should never happen.
