Comment 10 Andrew Niefer 2007-06-25 16:59:09 EDT 
I have created a org.eclipse.pde.build.tests bundle that currently has 13 junit plugin tests.  It does not use any code from the attached plugin.  There were a few things I didn't like about the initial proposal:
1) Dependencies on external tools that required approval
2) The need for resources for each test, and the "expected" results.
3) The coarseness of the tests.

I don't believe that comparing entire build.xml's for equivalence is a good way to test pde.build.  Any change in the generated scripts would break all the tests.  And then there is the question of what are we comparing against.  Any new bug fix could potentially change what is a "good" build.xml.

More tests will be added as they are written, but we have a good start.
Comment 11 Barrie Treloar 2007-06-25 20:54:59 EDT 
> 1) Dependencies on external tools that required approval
> 2) The need for resources for each test, and the "expected" results.
> 3) The coarseness of the tests.

1) Can be easily resolved.
2) & 3) are needed because these are integration tests not unit tests.

> I don't believe that comparing entire build.xml's for equivalence is a good way
> to test pde.build.  Any change in the generated scripts would break all the
> tests.  And then there is the question of what are we comparing against.  Any
> new bug fix could potentially change what is a "good" build.xml.

This is an "integration" test not a "unit" test.

I, and probably others, who are not familiar with the black magic that is PDE need confidence that any changes we make are not breaking existing functionality.

The only way to ensure that is to compare against the generated ant build files.

e.g. https://bugs.eclipse.org/bugs/show_bug.cgi?id=178928 "Assembly of config includes features that have root files twice" can only be tested via output of the script file because the java classes are not easily unit tested (no dependency injection or easy mocking of expecting calls) and the output is scattered amongst different files, e.g. assemble.org.eclipse.pde.build.container.feature.* and package.org.eclipse.pde.build.container.feature.* and it is the interaction of all these Ant files that you are testing.

The PDE classes are also tightly coupled, e.g. assembly and packaging in ways not easily understood.  By comparing the known working build files against your changes you have a better change of ensuring nothing is broken.

Is there any way to convince you that this level of testing is essential for PDE?
Comment 12 Andrew Niefer 2007-06-26 10:59:44 EDT 
Barrie, I was not making the distinction in my mind between "integration" tests and "unit" tests.  We do need both.  Of the 13 tests I have written so far, 4 of them run the full build from end to end.  Only 1 is what you would call a pure unit test that tests a java class directly.  The remainder are invoking intermediate ant tasks like eclipse.fetch and genericTargets.xml/generateScript.

5 of the 13 do provide resources as input to the tests, I just didn't want to have to provide resources for all tests.  Most of the others automatically generate the required input files.

I do agree that we need tests that run a full build and assert the results.  However, I don't agree that we can compare generated build scripts.  Those files _will_ change between nightly builds as the version numbers in the base location change.  As well, they will have paths specific to the machine they are generated on.  These changes are all before any changes introduced for bug fixes.  
Currently we can parse a build script using ant to assert its validity.  My hope is to use the ant's model to assert things about the content and even to use the ant debug support to assert runtime values.

The integration tests I wrote take care of the dynamic nature of the output.
Paths are definitely handled.  I'm not sure about versions.

Likewise I am unable to see how we can assert the output is correct if we aren't comparing the Ant build files as the end result of running PDE are Ant build files, so if we aren't comparing the Ant build files how are we asserting the output is correct?
