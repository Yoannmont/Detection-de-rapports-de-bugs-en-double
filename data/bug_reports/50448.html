Comment 9 Rafael Chaves 2004-02-04 17:33:45 EST 
Here is what it seems to be happening: plug-ins not having a <?eclipse
version="3.0"?> tag or requiring org.eclipse.core.runtime.compatibility are
considered old-style plugins, that need more complete support for backwards
compatibility. The plugin class for new plugins must have a no-arg constructor
(as BundleActivators do). The plugin class for old plugins must have a
constructor taking a IPluginDescriptor.

It also happens that currently PDE generates plugin manifests that are tagged as
3.0 but have an old style plug-in class (which inherits from AbstractUIPlugin) -
I just opened a PR to handle this issue: bug 51206.
Comment 10 Rafael Chaves 2004-02-04 17:36:35 EST 
For completeness: the instantiation exception happens because (as pointed out in
comnment 3) the plugin class constructor does not match its plugin style.

Comment 12 Rafael Chaves 2004-02-05 09:55:25 EST 
A 3.0 plug-in that is not in backward compatibility mode (from the viewpoint of
runtime/OSGi) has a plugin class whose constructor takes no arguments (yours is
an example of one that requires 2.1 style support from runtime). 

Currently, a plugin is consided "new style" if it has the <?eclipse
version="3.0"?> processing instruction *and* does not require
org.eclipse.core.runtime.compatibility.

Your problem is that your original manifest was for a "new style" plugin,
although your plugin class has a 2.1 style constructor (expecting
IPluginDescriptor). Requiring org.eclipse.core.runtime.compatibility tells
runtime that your plug-in is a 2.1 style plug-in (again, from the runtime point
of view - the "eclipse" processing instruction has a broader meaning).

Comment 19 Pascal Rapicault 2004-02-09 13:32:02 EST 
Just for clarity, let's summarize the different behavior that are / should be 
supported.

1 - Unconverted 2.1 plugin. The bundle activator will be the default activator 
provided by runtime (PluginActivator) and the plugin-class specified in the 
plugin.xml will be used.

2 - Plugin.xml containing eclipse 3.0 tag, and requirements containing 
org.eclipse.core.runtime.compatibility. The bundle activator will be the 
default activator provided by runtime (PluginActivator) and the plugin-class 
specified in the plugin.xml will be used.

2 - Plugin.xml containing eclipse 3.0 tag, the requires DO NOT contain a 
reference to org.eclipse.core.runtime.compatibility. The plugin-class specified 
in the plugin.xml will be used as a BundleActivator which requires the plugin-
class to have a 0-arg constructor, as well as all the classes in its hierarchy.

This last use case depicts a situation where people really want to take 
advantage of the new OSGi runtime.

Comment 26 Jeff McAffer 2004-02-23 15:13:11 EST 
Note, there likely was a problem in M6 but it has long since been fixed.

As for the current behaviour, we don't believe it to be a problem 
(compatibility or otherwise). Comment 19 describes exactly what is going on.  

The first thing to note is that 2.1 plugins continue to work unchanged.

Next, plugins marked as <?eclipse version=3.0?> and requiring 
runtime.compatibility also continue to work unchanged.

It is only if you are wanting to get off the compatibility layer (why?) that 
you actually need to change your plugin class to have the zero arg constructor.

Note that if your plugin class is exposed as API for others to subclass then 
they cannot get off compatibility until you do as there has to be a zero arg 
constructor all the way up the hierarchy.

Closing.
