Description Tobias Widmer 2006-03-28 09:33:10 EST 
I20060328-0010

Steps to reproduce:
- Open New File wizard

There are some problems with the current field assist implementation:
- The (ASCII-style) asterisk looks ugly IMHO and causes the entire dialog area to be shifted a few pixel! The layout of existing dialogs should NOT be altered by just enhancing text fields with field assist.

- If a dialog comes up with a required text field which has the initial focus, no visual hint is given anymore that the user is required to type something in.

A visual hint should be given if and only if the textual content of the text field is not valid.

I suggest the following behavior:
- Remove the asterisk affordance
- Use background color hint for all required fields which do not have valid content (yet).
- The balloon tooltip can be enabled if hovering over the required text field (positioned on the upper side of the field, at the mouse position)

Comment 3 Susan McCourt 2006-03-28 13:03:04 EST 
Thanks for the feedback, Tobias.  We released this support to obtain community feedback and will be trying to determine the strategy for 3.2 during the RC1 timeframe.

cc'ing Kim, the visual designer providing us the guidance on required field emphasis.  The decisions leading to the current emphasis (discussed in bug #120238) were:

- layout of dialogs will necessarily be altered when fields are decorated.  Decorations take space, and layouts and margins will be adjusted to account for them when they are used.  
- the background color hint is only used when focus has left the field.  The assumption is that the decoration provides the cue that it is required and the color only shows when the user leaves the field
- known problem:  the chosen background color is not very visible in many schemes (see bug #133214)
- known issue:  the current decorator may not be suitable for high contrast situations

These decisions will be revisited in light of feedback.

Comment 6 Kim Peter 2006-03-28 21:01:05 EST 
Created attachment 37170 [details]
Asterisk light blue

(In reply to comment #0)

Initially I was not that keen on using a persistent decorator for required fields, but as we went through the various alternatives in bug #120238, there surfaced two compelling reasons to do so:

1.  It notifies the user immediately that the field is required regardless of there being content in the field or not. There is no guess work and no surprise.
2.  Colour should not be the only cue. The asterisk provides quick visual information that information is required and provides an object for the hover, something consistent with content assist.

The asci-style may not be the most elegant solution, but it is an easily recognizable convention. Using blue instead of black has toned down its contrast some. I've attached a couple lighter colour treatments, one blue, one grey-brown as more subtle alternatives.

These attachments also attempt to make the decorator more suitable for high contrast themes as Susan flagged in comment #3. See related item in bug #133214.

Comment 11 Susan McCourt 2006-03-29 13:33:38 EST 
One implementation/layout concern with the "decoration on the right" case is that the grid layout (used by most dialogs) allows specification of margin on the left, but not on the right.  So when attempting to line up fields on the right edge, it is not as straightforward to make text boxes align.  Instead the text boxes with decorators will appear shorter.  Obviously we need to address this, since DecoratedField allows decorations on the right, I'm just not sure of the time frame for getting a reasonable solution in 3.2

Another treatment I noticed today while filling out a web form was that the asterisk was on the right of the label identifying the field, not on the left of the box.  This is something we might consider trying (making the labels decorated, not the text box) and makes alignment much simpler.  Would have to screen reader issues, because we'd be adding an extra label between the true label and the box.  So this might have its own implementation concerns.
Comment 12 Markus Keller 2006-03-29 14:35:35 EST 
> - layout of dialogs will necessarily be altered when fields are decorated. 
> Decorations take space, and layouts and margins will be adjusted to account
> for them when they are used.
Not sure how you meant that, but I think the visual layout of a dialog should in general *not* be altered when decorations are added. Decorations like the content assist lightbulb and error/warning icons should just go into the (usually blank) space to the left of the field. The layout only needs to be changed if there's not enough space there to show decorations.

Re required field decorations:

I also think we don't need the asterisk. In contrast to the yellow background color, the asterisk is an additional UI element I have to parse visually, although in most cases, it is not interesting. IMO, the best solution would be (as requested in comment 0): Give a field a yellow background if and only if the field is required *and* the field is not empty. This would also neatly eliminate the concerns from comment 5 and comment 10.

Whether the field has focus or not is not relevant for the user to determine which fields need input, so focus should not affect the "required" indication.

Re comment 6 1.: If a field is not empty, then I need no indication whether the field could be cleaned or not. And if I wanted to know whether a particular field can be cleaned (I don't know of a common real-life scenario for this), I could always delete the contents of the field and undo if the field became yellow.

In the rare situation where the user still needs a hint on what's incomplete in a big dialog, there's still the global dialog error message.

Re comment 6 2.: The content assist cue is different, since it only shows up when the field has focus.
Comment 13 Susan McCourt 2006-03-29 15:37:01 EST 
>Not sure how you meant that, but I think the visual layout of a dialog should
>in general *not* be altered when decorations are added. Decorations like the
>content assist lightbulb and error/warning icons should just go into the
>(usually blank) space to the left of the field. The layout only needs to be
>changed if there's not enough space there to show decorations.

This is where we disagree.  If decorations on fields are to be used in the UI, I believe space has to be allocated for them.  I agree it could be implemented differently; for example, a standard margin could be allocated in all dialogs and the decorator superimposed on the margin.  My main point is that hoping everything fits, and fixing oddities on a case by case basis is not a scalable approach as more decorations are added, or if sizes change, etc.  If we are adopting this approach for annotating fields then I think now is the time to also prescribe layout techniques and ensure a sufficient amount of space is allocated.

Comment 21 Kim Peter 2006-03-30 18:19:33 EST 
MVM, Susan, Jin, and I met to discuss the best resolution for 3.2. We addressed
the following questions:

What should we do in 3.2 across the board that is consistent?

What can we do that doesn't destabilize Eclipse? 
- anything that affects layout is a problem

Solution:
Add colour to fields, combos, spinners, text (don't deal with right now with
tables and lists because there are default selections)
- colour always present whether it has content or not
- 15% yellow on top of existing background, unless it is a high contrast theme
(Todd has code to detect)
- keep errors at top as is
- need accessibility hook to tell screen readers that it is required (post 3.2)

The complete discussion is in bug #120238.
Comment 22 Ed Burnette 2006-03-30 21:13:01 EST 
Hey, I had an idea on this. How about delaying the visual indication of missing required fields until it fails the first check and generates an error, and then only highlight the one that has the error? For example, you say File > New > Class. Both the folder and class name are required but there is no error at first and nothing is colored or marked. You start typing a class name, and now you get an error that says the folder name is empty. At that point the folder name field should be red or something indicating which field the error is referring to.

I don't really see a need to clutter up the display with lots of colors and markers telling the user all the problems at once, and neither do I see a need to differentiate between something that's missing and something that has the wrong format, for example if you type "Fred(" in the type name field. Both are errors.

Tying the error/warning at the top (now in the sliding tray) with the highlighting of the one field it's talking about would solve a longstanding usability problem in the UI.
