Comment 8 Jason Sholl 2007-08-13 22:43:43 EDT 
Created attachment 76014 [details]
patch for deadlock

This patch fixes the problem.  I have run the entire test suites for both WTP 1.5.5 and 2.0.1 for this fix several times and will run them a few more.  The underlying problem is the lock structure used by WTP in the EditModel framework is not consistent with the ISchedulingRule lock structure used by Jobs.  This fix relieves the dead lock by acquiring the EditModel locks in the correct order.

Here's a summary of the problem:
The main thread runs and performs an EditModel operation outside a scheduling rule.  This first causes the strutureEdit lock to be acquired at:
StructureEdit.save(IProgressMonitor) line: 361
Later the same thread attempts to save using an IResource as the scheduling rule to save and is blocked by the J2EECompoentClasspathUpdater thread.

The J2EEComponentClasspathUpdater job is running on another thread using the Workspace as its scheduling rule.  The call to set the property is blocked on the StructureEdit lock acquired in the first thread.

The fix to this problem may seem rather complex but is actually very simple.  The J2EEComponentClasspathUpdater needs to update the properties in a way that the locks are acquired in the correct order.  So, rather than updating the properties directly in the job, the J2EEComponentClasspathUpdater recognizes that updates are required and schedules those updates in another Job, ComponentPropertyUpdateJob, which has no scheduling rule.  By scheduling the ComponentPropertyUpdateJob Job with no scheduling the main thread is guaranteed not to be blocked.

The ComponentPropertyUpdateJob first acquires the StructureEdit lock by getting the EditModel for write.  Next it performs the update (this will acquire IResource locks and then release them).  Finally, during the save, it will first acquire the IProject lock (via yet another Job, ModuleCoreSaveJob) and then do the save.  This is necessary to ensure the project is not deleted between the time project.exists() is called and the save finishes.  In a real user environment this is unlikely to happen, but it happens frequently in the JUnit environment.

Comment 13 Jason Sholl 2007-08-16 15:57:53 EDT 
Created attachment 76256 [details]
patch

This patch removes all adding/removing of classpath container from the J2EEComponentClasspathUpdater.  The deadlock is resolved because no component flag is required to be set to keep track of adding/removing classpath containers, and thus that code was removed.

The classpath containers are now added during facet install.  The changes to make this happen are in the various facet install delegates.  The containers are only added there if the appropriate product flag is checked (this logic used to be in the J2EEComponentClasspathUpdater).

This fix vastly simplifies the architecture and is the recommended approach.  This type of fix was not original made with the previous bug because I didn't want to modify the general architecture; however, considering this deadlock, the modifications to the .component file, and the complexities of various Jobs, this truly is the best fix and a great improvement over 1.5.4.
