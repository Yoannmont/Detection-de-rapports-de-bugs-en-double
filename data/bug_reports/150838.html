Description John Lanuti 2006-07-17 13:27:47 EDT 
In a project, create an HTML page (or a JSF page or a JSP page).
In the project navigator, right mouse click on the HTML page.
Try to rename it.
There is no obvious menu item for renaming it.  There is a menu item "Refactor" and if you think to open it, there's a rename under it.
HTML pages are not refactored.  Nor are images, Javascript libraries, CSS stylesheets, etc.  Only Java code is refactored.
There should be no "refactor" for something that isn't Java.  The Rename menu item should not be buried under a menu head that doesn't make sense.

Comment 3 Michael D. Elder 2006-07-25 15:22:48 EDT 
I have renamed the title from "Java refactor actions should be used for java actions only" to "Refactor actions should be overrideable". After discussions with the originator of the defect, we have identified the following issue and a comprimise to solve the problem:

With the fixes from bug 150674, nested action providers may be overridden and contributed with priorities taken into account. 

Now, the only thing that is required is for the Java and Resource extensions to break up their Refactor and Edit action contributions:

+ Refactor the Resource Refactor Action Provider to contribute the same function in the form of a EditActionProvider and a RefactorActionProvider. The former will contribute Copy/Paste/Delete and the second Move/Rename.

+ Refactor the Edit and "Refactor" actions from the JDT PackageExplorerActionProvider into an EditActionProvider and RefactorActionProvider that override the respective Resource Action Providers.

Now, clients (including DTP and products) can choose to override just the Refactor submenu. 

This change does not materially change the behavior or function. It is just repackaging it into neater packages for client consumption. 

I feel that with the demand from DTP and other commercial products for this functionality, it is imperative that it be released for 3.2.1. 

I will attach the relevant patch to this defect. This change must be made at the Resource and JDT level for the scheme to work correctly.

Patches o.e.jdt.ui, o.e.ui.navigator.resources, o.e.ui.tests.navigator


You should load the following R3_2_maintenance plugins to apply the patch:

org.eclipse.jdt.ui
org.eclipse.ui.navigator
org.eclipse.ui.navigator.resources
org.eclipse.ui.tests.navigator

To verify the behavior, ensure that both the Java and Test extension is active for the Project Explorer. All files will assume a light blue background when the test extension is turned on. 

Right-click on a Project and/or a regular Folder. The Java Copy/Paste/Delete and Refactor menu actions will be available. 

Right-click on a regular JDT element (Package, Source Folder, Java Type), the Java CPD+R actions are available. 

Right-click on a plain IFile (e.g. .project), the "CN Test Menu" should replace the JDT Refactor submenu. The Java CPD actions should still be available. 

Right-click on a Java Source Folder and an IFile. Only CPD should be available neither the Java Refactor submenu nor the Test "CN Test Menu" is available for this kind of selection. 

Turn off the Java extension.

Now, the CN Test Menu will override the Resource Extension's Move+Rename.

Right-click on a project or folder. The Resource Copy/Paste/Delete + Move/Rename should appear. 

Right-click on a file. The Resource Copy/Paste/Delete and the Test "CN Test Menu" should appear. 

Right-click on a Folder and a File, the Resource CPD should appear along with the Resource Move+Rename. The "CN Test Menu" is not enabled on this type of selection, but the Resource actions are, so they take over the menu. 

This should satisfy the requirements from DTP (Brian F.) and IBM (Dusko M.).

Comment 6 Martin Aeschlimann 2006-07-26 05:35:29 EDT 
Note that the implementation of the 'Rename' action provided in the JDT refactor menu is doing more than the standard 'Rename' from core resources.

Even if invoked on a resource, Refactor -> Rename notifies registered rename participants (ltk.core.refactoring). I'm not sure if there are clients that listen on HTML or GIF or CSS file renames, but if there were, you really can call this a Refactor > Rename.

So when you replace the refactor menu, better use the JDT Rename and Move actions again to not loose functionality. We might look at this again in 3.3 and maybe move the ResourceRenameAction down to ltk.core.refactoring and try to convince platform.ui to also use rename and move actions that notifies participants.

Comment 8 Michael D. Elder 2006-07-26 09:34:50 EDT 
In most cases, the Project Explorer seeks to mimic the usability patterns of the Package Explorer, as many users will expect this. As the Package Explorer and Project Explorer do the same thing in this scenario, I'm not convinced that we will have widespread confusion among consumers looking for a "Rename" action. 

With the changes for this defect, extenders have the flexibility to customize the menu for their specific users. This is as far as I think we should go for 3.2.1. 

In general, having a different menu for _everything_ based on types makes it harder to anticipate where an action is. If users really are confused by the Refactor> submenu approach used by the JDT, there are drop down menus under File to accomplish the same thing.

The intent of these changes is to allow a CNF extension to provide their own actions for cases where integration with JDT Refactor Participants does not make sense (e.g., clients that must exist in JDT and non-JDT configured environments). It is possible to restructure the menu, but my official recommendation on this is for clients taking advantage of this capability to contribute their own Refactor> submenu when overriding the JDT Refactor Action Provider. 

Beyond the restructuring of the JDT Action Provider, no material changes will be made to the JDT extension for 1.5.1; particularly not to change how JDT displays the menu.
Comment 9 Dusko 2006-07-27 12:00:23 EDT 
This approach will do enough to let us fix our problems with refactoring in 3.2.1 timeframe.

However, post 3.2.1 (aka 3.3) I think we should take this a step further. JDT refactoring (especially Move) is too specific. It deals in terms of packages, not folder, and for most non-Java resources that is just wrong. For example, C++ classes. The proposed patch will help because it will be easier to override the behavior but does not help teams that will be OK with default refactoring and do not have their own.

Going long term I think we should keep this patch and also provide better enablement for JDT actions. They really should be enabled only for the resources that they know how to handle. The generally useful things that JDT refactoring does and that are not JDT specific should be moved to the platform. Prime example of that would be support for participants.

Please remember that Project Explorer is not replacement only for Package Explorer but also a lot of other navigational views. Stuff common for everybody should be in platform and components should override it only for their specific needs.
