Description Carolyn MacLeod 2004-06-16 14:39:52 EDT 
This bug addresses the underlying issue that was flagged in bug 67334.
Please read that bug for background details.

I feel that further analysis of the use cases is required. I think that the UI 
should never make the user type more keystrokes than are strictly required.
I think that once a user understands the concept of code folding, they 
understand completely the implications of selecting one folded line and 
pressing "Delete" (or Ctrl+Delete or Backspace or 'cut'). I think that the 
current behavior of having the code unfold first, necessitating hitting Delete 
again (or one of the other keys) in order to "really delete", is surprising 
and unwelcome. When I select something in an editor and type delete, I expect 
the selected stuff to disappear. I do not expect to have a bunch of stuff 
appear in its place.

I was even further surprised to discover that sometimes the user needs to 
press Delete "n" times, where n is the number of levels of folded code. So 
it's not just twice... it's some potentially unbounded number of delete 
keystrokes required to finally delete that code...

Comment 5 Carolyn MacLeod 2004-06-17 18:11:39 EDT 
Ah, I see - I missed the words "more than" in comment #2 - sorry.
Here is a case where you need to type backspace 3 times to remove the code:

- turn on method and inner class folding.

- open a file that has methods with anonymous inner classes with methods.
  Here is an example of such a class (to get this class to compile, add
  org.eclipse.swt to your workspace, and copy the file .classpath_win32
  to .classpath):

import org.eclipse.swt.*;
import org.eclipse.swt.events.*;
import org.eclipse.swt.layout.*;
import org.eclipse.swt.widgets.*;

public class MultiLevelCodeFolding {
	public static void main (String [] args) {
		Display display = new Display ();
		final Shell shell = new Shell (display);
		shell.setLayout (new GridLayout ());
		Button button = new Button (shell, SWT.PUSH);
		button.setText ("Button");
		button.addSelectionListener(new SelectionAdapter() {
			public void widgetSelected(SelectionEvent e) {
				shell.addDisposeListener(new DisposeListener() 
{
					/**
					 * Print to Console when Shell is 
disposed.
					 */
					public void widgetDisposed
(DisposeEvent e) {
						System.out.println("Shell was 
disposed.");
					}
				});
			}
		});
		shell.pack ();
		shell.open ();
		while (!shell.isDisposed ()) {
			if (!display.readAndDispatch ()) display.sleep ();
		}
		display.dispose ();
	}
}


- now unfold main and select and delete (using backspace) the following line:
      button.addSelectionListener(new SelectionAdapter() {[..]

- count 1, 2, 3 backspaces before the code disappears.

Note that I was wrong when I said that n is potentially unbounded. The maximum 
appears to be 3, because after 2 backspaces all remaining code is unfolded. 
(i.e. in the example above, I deleted main when it was folded, and I expected 
to need 4 backspaces, but it only took 3 backspaces).
Comment 6 Carolyn MacLeod 2004-06-18 15:08:59 EDT 
I have discovered another use case where the current "unfold before clearing" 
model feels counter-intuitive. This is the use case where the user selects 
some (folded) text and then types over it. The expectation is that the text is 
cleared and then the typed text replaces it. However, the reality is that the 
first typed character is used for unfolding, and it is then thrown away.

For example, in the little class above, if I were to select the line:
      button.addSelectionListener(new SelectionAdapter() {[..]
and then start typing
      button.setImage(
then I would be surprised to notice that the 'b' was thrown away (or maybe 
the 'bu', depending on whether 1 or 2 levels of code was unfolded).
      utton.setImage(

I guess I am just trying to say that I would expect folded code to act exactly 
like unfolded code in all of these scenarios. I would expect to not have to 
think or care whether or not my selection contained folded code.

Comment 17 Markus Keller 2007-06-13 07:31:15 EDT 
> We won't change this for 3.3.

Please consider for 3.4, since this is still very annoying and confusing. When the selection contains a folded region,

a) Backspace, Shift+Delete, Cut (Ctrl+X or Shift+Delete), Paste (Ctrl+V), and character keys expand all collapsed regions and set the caret to an arbitrary location

b) Delete and Delete Line (Ctrl+D) delete the selected text, including the folded regions

The actions in group (a) should just do what the user requested and not try to be too smart. Its OK if keystrokes expand folded regions as a side effect, but they should never swallow the original action.

Comment 32 Carolyn MacLeod 2008-09-26 12:50:51 EDT 
Hi, Dani. I tested this on Windows, Mac, and GTK, and I found 3 problems.

1) On Mac and GTK this seems quite broken. I have seen behaviors from simply unfolding the code and eating the backspace, to unfolding the code and deleting seemingly random text and inserting the typed character. Note that typing the Delete key over an area that includes folded code seems to work ok, but typing any other character (including backspace) can have very strange results.

2) On Windows (maybe on Mac and GTK too, but it's hard to tell because of (1)) pasting over folded code can sometimes result in an incorrect unfolding. To see this, fold a (small) method and select only the folded code, so that the end of your selection is at the very beginning of the line immediately following the folded code. Copy/Paste (I typed ^C^V). The code is unfolded, and the copied code (all but the first line) is inserted below the unfolded code. It looks like the code was pasted twice (sort of - the first line of the 2nd copy is missing).

3) On all platforms, selecting code and simply unfolding it by clicking on the + results in the selection either disappearing or being truncated, depending on what was selected. I have opened a new bug report for this one. It is a bit different because no keys are being typed. But it is the same idea: I expect folded code to behave exactly like unfolded code would. Please see bug 248744.

I am reopening this bug to address issues (1) and (2).
