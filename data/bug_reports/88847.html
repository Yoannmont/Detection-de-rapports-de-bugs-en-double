Description Gorkem Ercan 2005-03-23 08:56:02 EST 
Create a publisher API and extension point that may fulfill the below requirements.
1. Ability to create a general purpose publishers that may or may not be
specific to a server.
2. Ability to associate a publisher with a module in the workspace and a server
instance 
3. Ability to chain several publishers, change their orders in deployment time.
4. Provide UI to users so that users can enable disable additional publishers.

Comment 5 Thomas 2005-04-06 20:42:37 EDT 
Sorry for delay, I didn't notice your reply earlier.

Yes. I think that the API needs to allow system wide publishing. 

I have mainly two reasons: control of the ordering, and control of error
handling and reporting.

Depends on situation, I need vastly different order of calling the module. Let's
take clean publish as an example, I need a publisher to generate some artifact
file on the EAR depending on sub modules. This publisher order is WEB modules
first, and then the EAR. 

Another publisher will simply call the deployer with a path in the EAR project.
The publish only need to be run on EAR.

For unpublish, I might want the publisher to be called on each sub modules, so
that each module is stopped. Then, I undeploy the EAR.

Also, a connector module that is not under the EAR might need to be publish
before a EAR does. They are not even in child/root relationship.

(Of course, we can make the life easier for who implements the SPI by providing
a simply visitor for typical ordering.)

Also, the failure of publishing a module matter to the deployment of other
dependent modules. I don't have the flexiblty now to decide what module should
still be publish, or not be publish if failure is detected earlier. I don't not
have such control, unless I do server wide publshing.
Comment 6 Thomas 2005-04-06 21:11:15 EDT 
Regarding two types of publishers, I think that we can keep one publisher. 

A Publisher has only one method .publish(IServer server, IModule module, ...)

The delegate method will walk the modules tree and call the abstract

Another apporach would be to publish against a Publish Set, in which each

Comment 8 Thomas 2005-04-11 02:03:04 EDT 
Gorkem, I had been also working on a patch. For reasons, it took more than two
days... :-P

In my patch, I propose another layer of indirection. I am callilng it
PublisherGenre extension point for now. A publish genre has a publisher id and
associates a server and module (component). A wildcard “*” can be used in place
of server and module. It promotes the reuse of a common Publisher, such as
JSR-88, or an EAR packager etc. A priority can also be defined.

A server.publish() method calls PublisherManager.publish(). And,
PublisherManager.publish() obtains all publish genres, and filter out those that
doesn’t match the current server, and those doesn’t match modules reachable from
the root modules of the server. At the end, a list of publishers following the
defined priority is returned. 

Attributes are saved in Server to override the order of the publishers, and
disable unwanted publishers. Before release 1.0, we can leave the attribute
unused, and always return the list publishers intact as the PublisherManager
returns it. And, it gives us the option of implements the behavior and the UI
later without breaking any API.

There is a common delegate for all publishers to implement. The first one is
PublisherDelegate, which has a publish() method. Those publishers intent to take
care the whole server extends this delegate and implements the method, and has
all freedom to call modules in any order its like.

The second delegate is ModulePublisherDelegate extends PublisherDelegate. A
default implementation is provided for the publish() method. It visits all the
modules much like what the Server.java does today, and call each
publish(IModule) in an order. 

I intent to provide additional methods in the delegate for simple ordering. For
example, int getModulePublishOrder(), and getModuleUnpublishOrder… option
includes DeepFirst, BreathFirst, etc.

Arguably, the ModulePublisherDelegate.publish() method automatically skips all
module unless there is a publish genre matches the publisher id of the delegate,
matches the server in action, and matches the current module.

I think my proposal meet all the following requirements:
    1/ can be implemented in short enough time for 1.0 (and, I have been working
on it for a few days already). 
    2/ it makes minimal assumptions about what a publisher does, 
    3/ supports all 3 common use cases*, 
    4/ leaves the option for ordering and disabling of publishers open, 
    5/ enable reuse of common Publishers, 
    6/ Publisher that is not needed for current publish action will not be
loaded (filtering is done according to publish genre and is not obtained if it
doesn’t match the filter),
    7/ Additional filtering can be added by additional configuration element in
publisher genre extension point, to a future version without affecting
publisher/server/module defined in 1.0, and
    8/ makes common publisher as simple as to implement as what we have today
(implements only publish(IModule[] module) and defines a PublisherGenre
configuration in a plugin.xml). 

*The 3 common use cases are: 
    1/ a publisher associates with a server, for example a JSR-88. 
    2/ a publisher associates with only a specific module, like artifacts files
need to be generated for J2EE module; or, an ant script target should be invoked
pre-publishing.
    3/ a publisher associates with a specific server, for example, Pollinate
artifacts that only support a few servers, but not all.

Comment 18 Thomas 2005-04-19 19:29:13 EDT 
I discovered that IServerTask actually covered the original requirements. 

    1/ IServerTask can be general and may or may not be specific to a server.
    2/ Can be associated with a Server and module
    3/ Ability to chain server tasks and have ordering.
 
(But, I propose that we rename it to IPublishTask. It is the only place they're
invoked. The name "server task" and "publish task" implies different lifecycle,
will have impact of how this API evolve in the future.)


The IPublishTask (IServerTask) itself is richer in features than the patch I
proposed on Apr 14. For example, it supports Ordering, and has some UI support
already in place. With IPublishTask, I think we don't need chaining ablity from
the publisher extension point.


The new patch I proposed on Apr 19th focus on enabling the server adapter to
have freedom of the module publishing order, and freedom of error handling,
which is not covered by previous patches, nor by IServerTask.

The patch factored the publish mechanism into a ServerPublishDelegate, which
enable the publish behaviour to be overridden by the implementation.

During the patching, I have a choice to pull the publish mechanism into a
current delegate (ServerBehaviourDelegate), or make a new one. I found that
making a new one make more sense. The original delegate provide to two aspects
of logical functionalities. First is the lifecycle of the server (ie,
starting/stopping/restarting). The other is to manage the modules in the server
(publish, starting/stopping/restarting). As the publish code over-complicate the
delegate quite a bit, it makes sense to divide them into two separate unit.
After the patch, the two delegates of both tomcat and generic (and weblogic)
never need to invoke method on each other. The separate is pretty logical.

There is some slight behaviour change associated with the patch. The
publishStart() event is fired before IPublishTask is executed. I think it is
more appropriate. The server connection might be obtained by IPublishTask. It is
helpful if the code guarantee that publishStart() is called before hand. It is
also helpful to be sure that publishFinish() is called if error occurs even
before the Publisher is invoked.

Effort has been made to make sure the patch will not introduce new API
violation. IServerPublishInfo, and IPublishInfo are introduced, the visiblilty
of some methods are make public. IServerTask (to be renamed to IPublishTask) is
moved.

Comment 20 Thomas 2005-04-20 03:35:54 EDT 
Thank Gorkem. We want the feature of chaining up generic publisher. But, we also
need it to go deeper.

In addition to chain up a few generic publishers, we also want the ability of
choosing a publishers chain instead of another depending on settings. For
example, to support local and remote server, we have a pretty dissimilar
publisher steps. We also want the flexiblity of adding and removing a step. For
example, depend on user's setting, we might want JARing up the modules as an

publisher API. An extension point somehow static. Making it flexible enough to
cover the use case requires still quite a bit of work. It is not easy to get it
into M4, and is hard to get it right in such a timeframe. It would also be
easier for the first run to be done in code. And, we should know exactly how the
publisher chain to be define after we at least have some prototype. 

I think it is better if we can hold publisher chain and come back to visit this
topic after 1.0.

Comment 24 Tim deBoer 2005-04-20 10:13:32 EDT 
Hi Thomas,

I went over the proposed patch with my team today, and we had a few questions
and concerns:

* We do not understand the separation of the ServerPublishDelegate and think
that there would be many places where these two delegates would need to work
together. What are the reasons that you split this out?
* There are several methods that are allowed to be overridden by service
providers (e.g. publishModules(int, IProgressMonitor)), but the existing
implementation requires internal access to Server to be able to do it's job.
Either these methods should be implemented without access to Server, or they
should be marked final to avoid internal API access. The server instance
variable should be private.
* A fair bit of new API has been exposed (e.g. via IServerTask,
IServerPublishInfo, IModulePublishInfo, and on the delegate) and we don't
understand why all of it is necessary. We need to keep the new API minimal for
this release.
* Why does the publish delegate have it's own copy of the server modules, and
the ability to overwrite the server's copy?
* There is lots of missing javadoc and code cleanup still to be done.

We were wondering if it might make more sense to take this one step at a time,
adding support and exposing features one by one so that we stay close to clean
API and stable code instead of dropping a large change in all at once.
Comment 25 Thomas 2005-04-20 15:55:51 EDT 
Separation of Delegates
-----------------------
I found the separation make the code less clustered. I don't find that many
cases where the spitted delegate need to access each other. I think separating
them logically helps code to evolve. However, I am open if you think they should
be merged back.

Server internal access
----------------------
I intent to avoid client code to have no access to Server.java. But, I think I
still missed some cases. I agree that "either these methods should be
implemented without access to Server, or they should be marked final to avoid
internal API access". I will make the change. It was a good point.

On the other hand, exposing IServerPublishInfo, IModulePublishInfo was my
attempt to avoid implementation to have access to Server.java. We can either
expose them as interface, and add ServerBehaviourDelegate.getServerPublishInfo()
or expose them as a utility.

Even publishModules call server.visit(), but it is not the only way that the
list to be computed. A server adapter who implements its own publishModules()
can compute the list directly from IServer. So, the occurrence of call
server.visit() is purely implementation choice that I inherited from original
code, and doesn't imply anything for the API. (I personally do not like
Server.visit() method. I think at least it should be exposed as a utility and
give user choice of the order that it should visit. 
I personally favor a utility that give me a tree. But, it is probably out of the
scope of this patch.)

> Why does the publish delegate have it’s own copy of the server modules

The list is purely for caching purpose. I notice that the previous
implementation attempts to minimize the number of times that the modules list is
visited. I thought it would help to cache it. I agree for minimizing the API
standpoint, I think it is a good idea to remove it.

> and the ability to overwrite the server's copy?

The method is faithfully inherited from ServerBehaviourDelegate. In fact, I was
really puzzled when I saw that as well. I even notice code that does it in
tomcat or generic server adapter (I forgot which):
    ServerBehaviourDelegate.setModules(server.getModules())

> We were wondering if it might make more sense to take this one step at a time

We are certainly on the same pace on the idea to keep the API simple, and not
exposing Server internal. I think my patch was pretty minimal. A few change that
 you questioned was around the way I avoid exposing Server internal. I am
totally open for better solutions.
Comment 26 Thomas 2005-04-21 16:27:05 EDT 
Tim, I notice that you have made some changes to the CVS already. Like renaming
IPublishTask, moved down where PublishTask got executed etc.

I have an idea. Should we move the implementation of publishModules() into a
“provisional.model” package? 

We can pull down publish(int, IProgressMonitor) into ServerBehaviourDelegate.
Add only an “abstract publishModules()” method.

Then, we can have the ServerBehaviourExtra [need a better name, of course] in
the package of “provisional.model”. We can expose the “resource delta”
management as a utility there. 

- Typical server adapter will extend ServerBehaviourExtra. It requires minimal
changes to any existing implementation.

- Complicated server adapter will override publishModules() method in
ServerBehaviourExtra, and can keep the “resource delta” functionality, by
assuming the risk of API change.

- Server adapter that cannot take the risk of API change can implements it own
publishModules() and extends ServerBehaviourDelegate directly.

- It will buy us time till 1.1 and enables enough flexibility.


If you think it is ok, I can take some time, and create a brand new patch along
this line.
