Description Dennis Quan 2002-12-09 16:30:52 EST 
The Windows implementation of SWT does not correctly identify the threshold 
region (i.e., the area surrounding the mouse cursor outside of which movement 
should be regarded as the initiation of a drag and drop operation). This is 
because the threshold detection code expects screen coordinates while the 
mouse event uses client coordinates. It can be fixed by adding the following 
line to org/eclipse/swt/widgets/Control.java:3397. (I've tested this and it 
works for me.)

        OS.ClientToScreen(handle, pt);

Comment 8 Dennis Quan 2002-12-11 08:55:10 EST 
The trouble is threefold:

(1) Most of the drag and drop support in Eclipse is achieved through native 
controls' implementations of the drag source (under Win32), such as in the 
tree and list controls. They of course handle drag and drop correctly, 
regardless of what is in Control.java.

(2) Calling ::DragDetect with the incorrect set of origin coordinates at worse 
will result in Windows telling the program that it is ok to start the drag 
operation. Take for example a window at (400, 300) and someone clicking on 
(404, 304). The rules say that the mouse cursor must leave the box given by 
(404 +- cx, 304 +- cy), where cx and cy are given by ::GetSystemMetrics
(SM_CXDRAG) and ::GetSystemMetrics(SM_CYDRAG), respectively. Of course, 
if ::DragDetect is called with client coordinates instead of screen 
coordinates, then of course (4, 4) is outside of the (404 +- cx, 304 +- cy) 
box... way outside the box in fact, and the drag operation is then allowed to 
proceed.

(3) The problem is not that drag and drop doesn't work; it is that it works 
TOO well. The moment you click down on the mouse button the system begins the 
drag operation. That is not the correct behavior, as it inhibits support for 
detecting mouse clicks. Sometimes users will move the mouse ever so slightly 
during a click. This is not supposed to initiate a drag operation--
what ::DragDetect is supposed to prevent.

We are developing a huge GUI toolkit as part of the Haystack project at MIT 
(http://haystack.lcs.mit.edu/). To give you a minimal piece of our code that 
implements drag and drop (the support is in one of our base classes) would 
require me to give you about 30,000 lines of code. That is probably 
undesirable. :) I'm more than certain that this can be replicated with SWT 
sample code, which I've seen in the source tree, so long as the drag and drop 
code uses an SWT-based implementation of a widget on a Composite and not some 
native control.

Comment 10 Dennis Quan 2002-12-12 16:55:15 EST 
First, technically, the native widget host is responsible for initiating the 
drag operation itself (e.g., in response to LVN_BEGINDRAG, etc.), but the 
threshold detection is performed by the native widget.

Second, I've tested the code you've supplied, and it does demonstrate the bug. 
Like I had mentioned earlier, the problem is not that drag and drop doesn't 
work. To see the difference, try pressing down on the left mouse button and 
moving the mouse ever so slightly, about one or two pixels in any direction. 
With the bug present, the drag operation should proceed immediately. With the 
bug fix applied, no drag operation should commence, until you move perhaps 5 
or more pixels away.
