Description Alasdair McDowell 2004-07-06 08:53:16 EDT 
Create a new Java project
Create a new Swt-based Visual Class
Compile errors occur because the build path doesn't include the SWT library

We should probably do this automatically (or maybe with a check box), or only 
allow Swt options if the library is in the class path. The first option is 
probably best.
Comment 1 Peter Walker 2004-07-09 17:24:51 EDT 
Comments from Richard Bone in dup bug 69676:
Discussed with Peter briefly as to a solution and there were two possible ones:
1) We disable the SWT options if the project's classpath doesn't have SWT stuff
on it.
2) We add the SWT container to the project's classpath and allow the user to
select the SWT items from the tree.

Comment 5 David J. Orme 2004-07-21 10:45:10 EDT 
re: comment 1:

3) We have a checkbox "Add SWT lib to project" that is enabled if the SWT lib
isn't on the project classpath.

Or maybe another approach that takes into account the concerns raised in bug 70022:

Maybe we need a whole new create project wizard: a "create SWT application"
wizard that adds the SWT lib to the project and the SWT options in the create
class wizard are only enabled if the project already has the SWT lib available.
Comment 6 David J. Orme 2004-07-21 10:55:20 EDT 
Or we allow the SWT options in the create class wizard, but display a warning
message if we detect that the SWT lib hasn't been added yet.

I'm suggesting this because the use case is that the developer wants to create a
SWT application with a visual SWT class.  There is no reason why the developer
should automatically be constrained to add the SWT lib *before* creating the
visual class.  The developer just needs to be reminded that this needs to be done.

Of course, it would be nice if we could offer to add the SWT lib for the
developer if we detect that its missing, but I don't feel that's strictly
required of a 1.0 release.

The whole "add it for the developer" thing is a can of worms because you can't
automatically add the current SWT lib that Eclipse is using to the classpath.  

I'm thinking about how SWT 2.1.2 introduced a bug where new Label(parent,
SWT.RIGHT) left-justified the label.  So Essential Data can't use SWT 2.1.2
because it expects justification to work correctly.  The result is that I might
be developing on Eclipse 3, but need to target any of *several* SWT libraries I
have available on my system, and I can't see how VE could know that.

So we could theoretically have a place in VE where you can register all versions
of SWT that you might want to develop against, and then allow the developer to
choose from that list in the "create SWT project" or "create Visual class"
wizard if we detect that SWT isn't present.  But, IMO, that sounds more like a
VE 2.0 feature than like a VE 1.0 feature.
Comment 7 David J. Orme 2004-07-21 10:57:22 EDT 
So the upshot is that I think we should just have a "create visual class"
wizard, and this wizard should display a *warning* (not an error) if the
developer tries to create a SWT-based class inside a project that does not have
the SWT lib on its classpath.

What do people think?  Does this make sense?
Comment 8 David J. Orme 2004-07-21 11:02:14 EDT 
If someone feels like a "Create SWT Project" wizard that adds the current SWT
lib to the project classpath is really needed in VE 1.0, please file a separate
bug report for that use-case and we'll consider it separately.
Comment 9 Richard Bone 2004-07-21 11:03:43 EDT 
Having the warning there inside the wizard would definitely be better.  You make
a good point about why it might not be a good idea to add the library

the idea of having the different versions of SWT that you could register and

Comment 10 Richard Kulp 2004-07-21 16:21:28 EDT 
The created code will have red-X's in it because Shell won't be found.

Also, though we aren't doing Composite for M2, for composite, you won't even be
able to subclass it because the new class wizard won't allow Composite to be a
super type since it can't see Composite in the project.

And it is not just SWT. Anybody can add their own contributions that depend upon
specific classpaths. They won't be valid either.

At this time, if we can't handle it and it will create invalid code I don't
think we should even show the entry. I feel it should be filtered out in this case.
Comment 11 David J. Orme 2004-07-21 16:45:37 EDT 
For the sake of argument (Gili's kind of argument ;-):

Rich, I'm guessing that the initial contents of a class that VE generates is
governed by a code template somewhere.  If that is correct, then VE doesn't need
to know that it's generating code that can't compile *right now*.  All it has to
do is to fill the right strings into the right places in the template and
generate a new file.

So if this guess is correct, I'm not sure I understand what is wrong with
letting the user type an arbitrary string that matches Java's syntax rules for
an identifier into a text box in some wizard, then dropping that string into the
new class template when we generate the class.

So what if the class won't compile, as long as we warn the user of this in advance?

---

What I really think:

Personally, I don't have very strong feelings about Rich's proposal or mine.  I
think they both have merit.

However, if we require people to already have their classpath in order before we
will generate SWT classes for them, then I feel more strongly that we need a
"New SWT Application" wizard that (in 1.0) puts the current SWT lib on the
classpath for people.  2.0 could add the feature of letting the user select
which SWT library (of possibly several) they want to add.
Comment 12 Gili Mendel 2004-07-21 17:35:12 EDT 
I can see the merit for the warning option for a first time user that is not
familiar with class-path/containers.  This user is most likely to try and create
a SWT (or other type) directly from the create wizard, see the warning ...
(potentially open the help file) augment the class path, and come back again. 

Another option, is to show and disable the other options (e.g., SWT) ... which
will give the same effect as the warning, but will not generate the red Xs code
base.  A user would then open a help file or such to figure out how to enable
these options.
Comment 13 David J. Orme 2004-07-22 12:36:09 EDT 
The advantage of the warning is that if written well, it could save some
percentage of users a trip to the help file...
Comment 14 Jeff Myers 2004-08-04 12:59:07 EDT 
This also impacts the creation of the SWT examples.  After discussing this with
Seth and Sri, we propose two options:

1. Display a warning in the wizard page when selecting an SWT class in the new
wizard/example wizard that the default SWT library will be added to your project.

2. Provide a dialog message when pressing finish along the lines of "You are
creating an SWT class in a Java project without the SWT library. Do you want
this library added to your project? Yes / No"

Switching the level of SWT targeted by the project is a more advanced operation
that would be done by someone who's (hopefully) more familiar with the concept
of Eclipse Libraries.  If they want to switch levels after the fact, they could
select the library, bring up the context menu for it and select Configure (if we
add the target level selection to the container options)

We need to do something on this topic, as it's very confusing for first time
users to use the wizards and have non-compilable code output.  Having the users
have to look at the documentation to find out that they need to add the SWT
library first is a very poor work around.
Comment 15 David J. Orme 2004-08-04 17:15:55 EDT 
I'm happy with this latest proposal.
Comment 16 Gili Mendel 2004-09-08 09:47:34 EDT 
We need a low hanging fruit... let the SWT contributor to add the continer if
used.  The problem of not knowing how to do SWT comes up again and again in the
news groups..... so this is a big usability problem.
Comment 17 Peter Walker 2004-09-08 19:15:59 EDT 
For now I couldn't do the whole UI implementation but I added function to check
the plugin extension for "container" and "pluginId" and to see if the project
for this class had this in the classpath. If it doesn't, it gets added under the
covers. 
So this works for not just SWT, but for any project that needs a specific container.

Changes were committed and should be in the next VE Integration build (targeted
for 9/10/04).
