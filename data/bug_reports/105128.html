Description Alexis Biller 2005-07-26 05:09:22 EDT 
WST M7RC3
IBM Java 1.4.2/SR1

TASKS TO SHOW BUG
A. From a validated schema, select to Generate XML File (PRecord.xsd from
attached schemas). 
B. Provide output location. 
C. Select to generate optional content.
D. Validation of generated XML File reports errors.

Resulting XML contains errors, as (1) a namespace has not been defined and (2)
reference to namespace is not provided in related elements. The namespace that
is missing is referenced within another schema (WPStruct.xsd from attached files).

1. Missing namespace: xmlns:wsa="http://www.w3.org/2005/03/addressing"
2. Example incomplete Element (needs wsa: prefix):
2.1 BAD
	<pr:clientId>
		<EndpointReference>
			<Address>http://tempuri.org</Address>
			<ReferenceParameters />
			<Metadata />
		</EndpointReference>
	</pr:clientId>
2.2 GOOD
	<pr:clientId>
		<wsa:EndpointReference>
			<wsa:Address>http://tempuri.org</wsa:Address>
			<wsa:ReferenceParameters />
			<wsa:Metadata />
		</wsa:EndpointReference>
	</pr:clientId>

Comment 6 Gabriel Indik 2007-03-13 18:56:55 EDT 
Created attachment 60747 [details]
Patch

When the XSD to XML Generation Wizard is opened a table with the schema namespace information is built and shown on screen. This table is then used during the generation process to resolve the contents of the XML document. Unfortunately the table only contains the target namespace and imported namespaces of the top level schema. For this reason not all required namespaces appear in the generated XML document and their corresponding element prefixes are missing.

This patch enhances the build process of the namespace information table by recursively resolving all required namespaces from imported and included schemas. All namespace prefixes are guaranteed to be unique by automatically resolving any potential collition that may appear. Schema locations are resolved and formated as encoded URIs relative to the top level schema.
This ensures that the namespace information table is complete, making it possible to correctly generate XML documents with all the required namespace and prefix information.

Comment 8 David Carver 2007-03-13 21:00:39 EDT 
Question I would have on this, is that xsd:include schemas could themselves have xsd:imports.  So unless the patch also looks into xsd:includes for additional imports it could miss some namespaces.   The schemas in bug 176416 have this scenario.   The tricky part is making sure that you don't get caught in a circular include scenario. i.e an Imported schema has already been identified.   Ideally, the RecordSet a it is loaded into the XSDInfoset should be queried, for the imports, as it automatically will resolve all duplicates.  Unfortunately the way the currrent code is set up, the RecordSet is lost once the schema is loaded.
Comment 9 Gabriel Indik 2007-03-14 14:27:17 EDT 
Dave,
Precisely as you mentioned, included schemas can contain one (or more) imported schemas themselves. Thus, they can also include other schemas and each could have one or more imported and/or included schemas. The algorithm correctly addresses this by efficiently exploring all included and imported schemas and it handles any potential circular reference that may appear.
Comment 10 David Carver 2007-03-14 14:35:46 EDT 
Yeah, I realized that after I posted the message.   Again, my only suggestion would be to look at refactoring the XSDImpl classes, so that the ResourceSet is available.  it would eliminate the need to navigate and track the tree of includes and imports yourself, as when the Resource is loaded, a ResourceSet is created that can be retrieved from resource.getResourceSet() (I believe), which will contain a list of all the resources that were also during the creation of the XSD Model.   Long term I think this method is more efficient and easier on maintenance.  However, it will require either getting that list of namespaces when the XSD is first loaded.

Also, the method as it stands now, will get all the namespaces regardless whether they are used in the actual instance or not.   Many cases the imports may only be used by the schemas (i.e. referencing of complexTypes, and code lists for validation).  In the case of the upcoming STAR schemas, and samples I provided, there are 9 - 13 namespaces created, but only two of them are actually ones that need specification in the instance document that is generated.

The fix should work for now, but the above is something to consider for future releases.
Comment 11 Gabriel Indik 2007-03-14 16:59:16 EDT 
Created attachment 60863 [details]
Enhanced patch

This patch contains the following two enhancements:

 - 1: The namespace information table is now popullated with the minimal set of required namespaces. This is, any namespace that is not referenced from the schema instance is not included in the table.

 - 2: The schemaLocation attribute in the generated XML document now contains only the location hint of the top level schema and the location hint of those schemas that cannot be resolved through it. This is, if a schema can be resolved following the imports/includes of the top level schema, then its location hint is not required and is excluded from the schemaLocation attribute.
Comment 12 Gabriel Indik 2007-03-14 19:34:33 EDT 
Created attachment 60880 [details]
Further enhancement

In this new patch the namespace information table is built not by exploring the contents of the schema files, but by using the ResourceSet that is created after the schema is loaded.
In order to keep consistency between the namespace information table shown at the beginning of the Wizard and the generated XML file, the schemaLocation attribute enhancement has been changed so that the location hints that are not required do not appear in the table. The missing location hint warning no longer applicable) has been removed.
