Description Boris Pruessmann 2007-03-26 03:36:42 EDT 
I think the AbstractRepositoryTaskEditor needs to be split up into several classes to improve its reusability. Although it's pretty easy to write a neat task editor with just a few lines of code (-> TRAC), it's pretty hard if you want to customize the editor in a clean way. I think this could be greatly improved if e.g. the sections would be factored out into their own classes which could be sub-classed.

Here are some examples of things that are hard to implement or at least not possible to implement in a clean way and without duplicating code.
Comment 1 Mik Kersten 2007-03-26 18:45:39 EDT 
I agree that it is too long, that we need to make it easier to extend and that the current approach of making it possible to override methods can be tricky.  

Boris: please feel free to post suggestions.
Comment 2 Eugene Kuleshov 2007-03-26 19:10:18 EDT 
There is one more thing to keep in mind. While breaking editor into many classes might make those classes smaller, but not necessary easier to understand or easier to extend. Also, most likely breaking into the pieces will make editor even more unfriendly to the visual form designers.
Comment 3 Steffen Pingel 2007-05-24 13:58:05 EDT 
I'll start here by factoring out handling of custom attributes.

In addition everything in AbstractRepositoryTaskEditor that is not used by (Mylar) sub-classes will be declared private.
Comment 4 Eugene Kuleshov 2007-05-24 15:07:42 EDT 
BTW, I was thinking to put custom attributes (if you are referring to JIRA ones) under regular Attributes section. Though what is missing in AbstractRepositoryTaskEditor is the control on the rendering of those attributes. I.e. I need to show some of them as single line text fields and others - as multiline text fields. Current implementation has some hardcoded heuristic for comboboxes, but it is really unsufficient.
Comment 5 Steffen Pingel 2007-05-29 17:06:20 EDT 
The idea is to let the connector construct the whole attributes section (I actually meant all attributes not custom attributes). I am planning to provide an editor factory for common attribute types, see Edior/UI section at http://wiki.eclipse.org/index.php/Mylar_Offline_Refactoring .
Comment 6 Eugene Kuleshov 2007-05-29 19:21:34 EDT 
I don't think we should differentiate between "attributes" and "custom attributes". For jira I've added "type" metadata to the visible attributes, so I can construct UI properly. But I had to replace whole Attributes section. So, my thinking was if we can generalize that in AbstractTaskEditor, so connector only need to specify those types for common renderers and contribute own renderers for proprietary types (if any).
Comment 7 Steffen Pingel 2007-05-29 22:32:15 EDT 
Another reason for having the editor construct the whole attributes section is that it decouples the order of controls from order of attribute creation. Also when the repository configuration is not stored as part of the attributes anymore the editor needs to link the editor control to the configuration (e.g. the milestone combo needs to know about the list of milestones).
Comment 8 Eugene Kuleshov 2007-05-29 23:12:02 EDT 
I do agree though that editor should declare order of attributes, but it should not be necessary to do all that code for creating all attribute renderers/editors in each individual editor.

I am also not sure what exactly you have in mind about removing repository configuration from attributes and how editor would link to this stuff.
Comment 9 Steffen Pingel 2007-05-29 23:23:51 EDT 
The basic idea is to get rid of the redundant repository configuration in attributes (e.g. storing a list of milestones in each milestone attribute that is already stored in the repository configuration). 

Additionally I want to store the id for JIRA attributes and not the name (value). Currently JIRA does a two way mapping: for example the priority id received from the repository is mapped to a name which is mapped back to the id when the task data is submitted.

I hope I'll have a patch ready by the end of the week so we can iterate.
