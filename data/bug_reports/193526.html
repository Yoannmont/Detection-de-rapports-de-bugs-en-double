Description Christian Damus 2007-06-20 10:37:28 EDT 
Certain Eclipse views, e.g. the ConsoleView, wrap ActionContributionItems in the local toolbar within SubContributionItems.  This confuses the AGR when the wrapped contribution item has no ID, because it doesn't know how to get the actual contribution's class name by unwrapping the wrapper, and so always produces name-contribId-style widget IDs using the contribclass/org.eclipse.jface.SubContributionItem "action ID."  In consequence, virtually all anonymous toolbar items in the Console view match the same widget ID when playing back a recording.

Moreover, selection of items in a drop-down button in the Console view tool bar does not work because the playback never finds the ActionContributionItem from which to get the Action and, hence, the drop-down menu.  This is the case even for contributions that do have IDs, because the BooleanSelectionCommand looks for ActionContributionItems specifically (not SubContributionItems).

Reproduced in AGR 4.4 RC2:

1. Install Eclipse 3.3 RC2+, TPTP 4.4 RC2, AGR 4.4 RC2.
2. Install EMF 2.3 RC2+, MDT OCL 1.1 RC2+, and the MDT OCL Examples 1.1 RC2+.
3. In a new project, create an Ecore model and add an EClass with an
   EAttribute "name" of type EString.
4. Right-click on the EClass and select "Open OCL Console" in the context menu.
5. Use AGR to record the selection of the "M1" modeling level in the
   "Choose modeling level" pull-down button (which initially shows M2).
   Stop recording.  Observe that the script has
   "contribclass/org.eclipse.jface.action.SubContributionItem" as the action ID
   in the widget ID attribute.
6. Type "self." in the bottom panel of the OCL console and observe the
   content-assist that appears.  It contains "name : String" and a bunch of
   other items starting with "ocl".
7. Restore the M2 modeling level.
8. Play back the recording from step 5 in quick mode.  Observe that no error
   is reported.
9. Focus on the bottom panel of the OCL console again, place the cursor after
   the "self." and hit Ctrl+Space.  Observe that the content-assist options
   are very different.  This is because the M1 modeling level was not selected.

The problem is, that the playback matched the widget ID against the first pull-down action in the toolbar (the "Choose metamodel" option showing "Ecore") but could not get a menu to select "M1", so it failed silently.  It matches because the three-part ID has a threshold of 1 and the generic contribclass together with the style bits match both pull-down buttons, so that the mismatch of the toolitem name is irrelevant.
Comment 1 Christian Damus 2007-06-20 10:43:56 EDT 
Created attachment 71899 [details]
Proposed patch

Attached a patch that makes the following changes:

  - updates the MacroUtil class to unwrap SubContributionItems
    when getting the actionId of a contribution item (in case
    the contribution has no ID of its own and we need to get
    a real action ID or definition ID from the Action) or, in the
    worst case, just an action or contribution item class name

  - updates the BooleanSelectionCommand to look for
    ActionContributionItems in the SubContributionItem, in case
    the widget's data is of the latter type
