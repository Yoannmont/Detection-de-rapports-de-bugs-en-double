Comment 7 Mik Kersten 2007-09-27 15:11:12 EDT

Comment 11 Stuart James Penrose 2007-11-12 13:28:01 EST 
I'm probably not the most sophisticated eclipse user out there, and I'm more or less a brand-new mylyn user, but here are some notes on my use case:

1) I operate with the "Java Declaration Filter" on in package explorer.
2) When designing, prototyping, I often have many java files, most of which contain 'emtpy' classes.  

These classes remain empty for a while until I'm ready to flesh out the concepts/pieces they represent, but It's important that I see them all at a glance because they are essentially 'notes' that I need to create funtionality later, and because looking at all of them at a glance gives me a quick overview of what my emerging design looks like.  So, it gets annoying when I create a class/interface and then a few minutes later it "goes away".  So, naturally, I think to add it manually to my context, but this is difficult because there is nothing that I can select in package explorer which is 'landmarkable', due to the 'Java Declaration Filter' being on.

So, why do I operate with that filter on?  For me, seeing a zillion 'X.java' entries in package explorer, each with it's own 'X class' subentry, just ends up looking to cluttered. This gets especially cluttery when you add mylyn's bolding functionality.

I think I understand the thoughts about excessive bolding being something to avoid.  But I wonder if there is room for a separate concept here - not quite a 'landmark', but something more than 'hidden from view'.  Maybe the thing I'm looking for is not so much of a 'add to task focus' functionality for java files, packages, etc. but an 'add to task periphery' sort of concept.  Or, to put it another way, sometimes I don't necessarily want to have a file be a 'landmark', but rather just 'on the map'.  In the latter case, I wouldn't want it bolded anyway.
Comment 12 Mik Kersten 2007-11-15 02:38:10 EST 
Stuart: thanks, that's a very useful summary.  I'm currently of two minds on this:

1) We should discourage people from using the Java Declaration Filter, because with Mylyn the Package Explorer just ends up showing the exact same contents as the Ctrl+E list, which is almost as accessible and has the additional benefit that you can type at it.

2) We should accept that many people don't use the Ctrl+E list, and make a first-class model for the Package Explorer that causes declarations to be filtered and files to be landmarkable and show up in bold.  All modes of this sort are confusing and add UI complexity, because when engaged the elements behave one way, and when not engaged another.

So I wonder if you could say more about why you use the Java Declaration Filter?  I have heard many users give the very valid reason of the fact that their screen is too small and they see a scrollbar.  I'm wondering whether getting rid of the scrollbar entirely (bug 178891) would address some of the need to hide declarations.  I realize that there is some duplication when a lot of files with one class show.  But what we work on when we code is types and members, not files, so if there is enough space I believe that it is desirable to encourage showing members in the Package Explorer, since they are very representive of the task context.

> I realize that there is some duplication when a lot

> members, not files, so if there is enough space I believe that it is desirable
> to encourage showing members in the Package Explorer, since they are very
> representive of the task context.

My thinking behind using the Java Declarations Filter is something like this: When I think of a given project as a whole, I think in terms of packages & types/classes.  Given Java's 'one class per file' bent, this more often than not boils down to a list of .java files.  Of course the only time this breaks down is in the case of inner classes, but those are usually tied closely to the inner workings of their parent type.  So, for the purposes of getting that high-level view, inner classes are not important to me to see.  So, again, it's both the association between one .java file and one class, and the fact that any more info than 'there is a class named XYZ in package ABC' seems too fine grained for my tastes in a view of this sort (for anything that 'low', I use the 'Outline' view).

Maybe the simplest way to say all that is that I use the filter because:

1) I'd rather not see anything lower than '.java' (given that classes==.java files more or less one-to-one) in the package explorer view.
2) I already have that level of info in the 'Outline' view for the file I'm working on.
3) Because seeing that level of info in the package explorer view ruins my ability to use that view to quickly survey the landscape of my code.

As to ctl-E, I'd never used it before :)  So I'm one of those users (at least I have been till now - nice trick! - I told you I wasn't the most sophisticated eclipse user in the world :) ).

Comment 16 Mik Kersten 2007-11-27 18:05:55 EST 
Fyi, I updated the corresponding FAQ entry some: http://wiki.eclipse.org/index.php/Mylyn_FAQ#How_do_I_stop_declarations_from_showing_in_the_Package_Explorer.3F

Stuart: thanks for your detailed answer.  Some of the reasons you state are why we plan on continuing to support this feature, however, I believe that fixing bug 178891 will remove some of the use cases.

In terms of this bug, we have two options now:
1) Make Mark as Landmark be smart and work on Java File nodes by landmarking the first type in the file.
2) Allow Java file nodes to appear as bold, which could be useful when using the Filter Java Declarations mode.

I still hope to do (1) for 2.2.

Note that (2) would be very involved because it would require having custom decorations for Java file nodes that only work when Filter Java Declarations is enabled.  If anyone wants this please file a new bug with the proposed UI, noting the discussion in bug 111165.
