Comment 23 Philipe Mulet 2004-05-03 06:38:11 EDT 
Even if PDE fixes this deep recursion, we know that our code initializing 
containers is inducing recursions as well. We are going to investigate 
flattening it. Typically, when a container is resolved, we look for dependent 
projects which need to be updated in consequence. This code is walking all 
projects and asking to initialize their classpaths. Each classpath is itself 
recursively entering the same code, doing a container init and then walking 
all projects to find dependents.
On first container initialization, we will try to collect all containers and 
mark them all as being initialized before triggering their initializers. As a 
result, there should be no further recursions.
Comment 24 Philipe Mulet 2004-05-03 13:45:33 EDT 
Something must have changed though in timing due to the fact that PDE now 
listens to POST-CHANGE vs. PRE-AUTO-BUILD. When we ask for container entries, 
PDE isn't initialized yet, and triggers action which induces deep recursion.

Moving to PDE, after talking with Wassim who agreed to avoid triggering such 
actions. Containers are meant to perform only trivial actions.
Comment 25 Wassim Melhem 2004-05-03 13:56:16 EDT 
Here is what is happening:

Take the JDT/Core project for example.  It lists ../org.apache.ant/lib/ant.jar 
as an extra classpath entry in its build.properties.

When PDE goes to look for that library in the org.apache.ant project, it calls 
methods like getPackageFragmentRoots and getSourceAttachmentPath on the 
project.  These calls trigger the resolution of the org.apache.ant container 
in the middle of the resolution of the JDT/Core container, and hence causing 
this problem.

What we do now:
To locate the library and its source attachment in the org.apache.ant project, 
we call getRawClasspath() on the project.  This does not trigger the 
resolution of the container.  We then go through the classpath entries, find 
the one corresponding the ant.jar, retrived its attributes and add a 
corresponding library entry to the resolved classpath of jdt.core.
Comment 26 Philipe Mulet 2004-05-05 09:26:11 EDT 
Just to clarify things a bit more. Jerome was able to reproduce the problem in 
debugger, without PDE fix mentionned above.

The recursion is an infinite one, and isn't directly related to reentering 
container initialization. The PDE container is resolved lazily, when JDTCore 
needs to retrieve its classpath entries. Then container#getClasspathEntries() 
is invoked so as to compute the resolved classpath. PDE container's 
implementation for this protocol, did turn back to the JavaModel and ask 
itself for some operation requiring the same project resolved classpath (no 
need to further initialize container, simply extract its classpath entries 
again), and from thereon we are looping.

PDE solution to use the raw classpath instead is addressing this problem, but 
we should still add some protection in JDT Core against evil clients 
(reentering resolved classpath computation should answer back an empty 
classpath). Entered bug 61040 for extra protection in JDTCore.
