Description Mik Kersten 2006-05-05 14:35:10 EDT 
In order to provide interest-ranked content assit Mylar has its own javaCompletionProposalComputer's, which are configured automatically upon Mylar install via a preference setting wizard:

  http://www.eclipse.org/mylar/doc/faq.php#content-assist

The problem is that when Mylar is uninstalled the workspace is left in the confusing state of having no Java proposal computers enabled and to the user it seems as if content assist has stopped working.  Mylar is not the only plug-in that could get JDT UI into this state, so I'm wondering if JDT UI should be defensive against this somehow.  E.g., if no java computers other than the 2 provided by JDT UI are present, and both of those are disabled, on startup pop up a dialog asking whether content assist should be re-enabled, and ensure the user can click "don't warn me again" if they really meant to turn those proposal computers off?

Comment 17 Mik Kersten 2006-10-19 00:34:13 EDT 
I should have realized this, but mistakenly thought that the template proposal computer was not JDT-specific and that this fix would prompt to restore JDT-specific computers if all were disabled.  But 161372 is no resolved, so Mylar now duplicates a Template computer in order to ensure that the prompt to restore is triggered when the Mylar proposal computers all 'disappear'.  

While this should probably good enough to avoid users' triggering this bad state via Mylar, I wonder if/when we should discuss a more generic solution.  On the UI end of things, it's a a somewhat heavyweight approach for Mylar to be declaring it's own computers when all it's adding is ranking and a separator.  It's great that the computer mechanism allows us to do this.  But if JDT gets enough of a driver to make proposal *presentation* extensible I'd love to discuss it.  Also, has there been any discussion of a generic solution to this problem, e.g. where the publisher of an 'overridable' extension point specifies the default extension, in case the overrider disappears from the configuration?
Comment 18 Dani Megert 2006-10-19 03:00:20 EDT 
Mik, a generic solution would be pretty complicated: a plug-in can access and change any preference via API. This is in the user's workspace and once this plug-in is no longer there we have no way to tell whether the user explicitly changed the setting.

Just look at another scenario (though simpler to detect for the user ;-):
- user installs a CoolOutline plug-in
- user closes Outline view and puts CoolOutline view in-place
- user restarts without cool plug-in
==> no more Outline. There's no way we can detect to put back the Outline view. We cannot assume that this change has been done by a plug-in and not by the user.

Also note that a de-installer won't help much because most often this bug happened when users switched to a newer install and forgot to move their other plug-ins.

The only way this could be solved in a generic way would be to add an originator to each preference and setting. We could then check whether the originator is still there and if not reset the setting.

>But if JDT gets enough of a driver to make proposal *presentation* extensible
Sorry, there's currently no plan and time to do this.
