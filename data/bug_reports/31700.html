Comment 11 Douglas Pollock 2003-08-26 08:22:06 EDT 
There are two separate problems for the upcoming 3.0_M3 release:

1.) The new key binding architecture grabs all key events for processing before
releasing them to the widget hierarchy.  The CyclePartAction dialog listens to
key events to allow the nice "rocker" functionality (releasing and pressing the
non-modifier key repeatedly).  It is possible that we are somehow consuming the
event before it reaches the dialog.

2.) Ctrl+Tab will not be able to be set as a key binding for any command/action
using the new incarnation of the KeysPreferencePage.  This is partly by design
of the page, and partly due to the nature of SWT.  This could be fixed either by
SWT, or by the KeysPreferencePage itself.

Comment 13 Tod Creasey 2003-08-26 09:15:39 EDT

Comment 14 Douglas Pollock 2003-08-26 10:08:53 EDT 
Quick fix for M3.  Workbench now exports an internal API to enable and disable
the global key filter (i.e., the filter that processes keyboard shortcuts). 
CyclePartAction disables the global key filter before opening the dialog, and
enables it after it closes.  The ideal fix (not yet implemented) is to have
CyclePartAction handle it all internally -- without requiring the changes to
Workbench.

"tab" (or other traversal characters, depending on the widget set).
