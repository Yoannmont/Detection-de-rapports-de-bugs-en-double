Description Peik Aschan 2006-06-30 03:22:58 EDT 
This is a followup to bug 87351, see <a href="https://bugs.eclipse.org/bugs/show_bug.cgi?id=87351#c12">comment 12</a>.

HTML in partial JSPF fragments (segments) included in a JSP file can not always be
validated on their own. I get validation errors if I have JSP fragment that contains incomplete HTML, for example I have a footer fragments that ends a table  cell and row and begins with  "</td></tr>". The editor complains "No start tag
(<td>)".
Comment 1 james cook 2006-07-27 18:16:48 EDT 
I have the exact same </tr></td> footer problem - header footer includes are  quite common jsp practice.
Most "errors" disappeared thru the use of .jspf but not these html errors.
Comment 2 David Williams 2006-07-30 04:12:39 EDT 
Seems reasonable to me. (_might_ want to control with a user preference?) 

Can you please evaluate? And recommend alternatives.
Comment 3 Amy Wu 2006-08-29 14:02:20 EDT 
*** Bug 129212 has been marked as a duplicate of this bug. ***
Comment 4 Amy Wu 2006-08-29 14:40:55 EDT 
The HTML validator cannot just look up the "validate jsp fragments" preference because it should not have to know anything about jsp.  Perhaps a wrapper of some sort needs to be placed around the HTML validator for validating jsp fragments so that it'll know to check the perference.
Comment 5 Wolfgang Knauf 2006-08-30 06:07:57 EDT 
I think this is related to bug 125877 (exlude files from validation). 

My conclusion would be that some preference is needed to switch off some validators for specific files/directories (the project properties page "Validation" should be visible also in the file properties).
Comment 6 Peik Aschan 2006-08-30 07:03:31 EDT 
I agree with Wolfgang's comment #5. The validation options need to be addressed in a larger scope, not just for JSPF files. This would perhaps justify marking this bug as a duplicate of bug 125877?

In regard to the JSPF fragment validation problem at hand: Instead of tweaking options, a better solution would of course be that the fragments _could_ in fact be validated (both HTML and JSP). This would require Eclipse to find the context in which the fragment is used, since only the whole of this context is possible to validate. I opened a separate bug 155664 for that feature enhancement request.
Comment 7 Amy Wu 2006-08-30 13:24:39 EDT 
Yes, having a validation preference per file/directories would basically fix this bug.  Thank you for pointing that bug out.  

I'm not going to mark this bug as a duplicate just yet because technically we are offering the "validate fragments" preference now per file, so we should technically be honoring it.  If the other bug gets fixed though, this bug will be fixed rather quickly soonafter.
Comment 8 Amy Wu 2006-09-15 16:05:17 EDT

Comment 9 Amy Wu 2007-02-14 02:58:44 EST 
Created attachment 58954 [details]
jsp.core.ui.html.ui.patch

This patch creates a new validator, "JSP Content Validator" (If anyone has a better name for this validator, please let me know) This validator will validate the content of the content type of the JSP.  Right now this validator only validates the html content in jsp, but it could in the future validate other content types in jsps, like xml or xhtml.

We needed to separate html validation in html vs html validation in jsp because there are quite a few cases where something allowed in jsp is not allowed in html and maybe vice versa.

The new "JSP content validator" follows the Validate JSP Fragments preference, and since it is its own validator, users are able to turn this validation on/off in general (for all jsps)
Comment 10 Amy Wu 2007-02-14 03:26:02 EST

Comment 11 Nitin Dahyabhai 2007-02-14 12:24:54 EST

Comment 12 Amy Wu 2007-02-14 14:01:05 EST
