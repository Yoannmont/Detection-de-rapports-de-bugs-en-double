Description Carolyn MacLeod 2004-06-10 12:04:08 EDT 
build 200406091200

This refers to the default list of "Ignore patterns:" in Window->Preferences...
->Team--Ignored Resources.

I believe that the following patterns should be added to the default list:
*.jnilib
*.sl
*.dll
*.bat
*.com
*.log
*.zip
*.gz

Also, I believe that the following patterns should be UNCHECKED by default:
*.jnilib
*.sl
*.dll
*.so
*.a
*.bat
*.exe
*.com

Also, I believe that the following pattern should be CHECKED by default:
core

Here are my arguments:
1) .jnilib, .sl, and .dll are the functional equivalent of .so and .a files.
   They are all native "shared object libraries" for various platforms.
   Since .so and .a are in the pattern list, the missing 3 should be added.
   Now that we have 5 library extensions in the list, I believe that more
   people will want them IN version control than OUT (ignored). This is
   because people go to a great deal of work to create them. They are the
   final output of many a long software process, and they are typically
   "the thing that is shipped". So I think they should be unchecked by
   default, but they can remain in the list for convenience.

2) .bat and .com are the functional equivalent of .exe files. They are (or
   they launch) programs. Since .exe is in the ignore pattern list, the other
   2 should be also. The same argument as above can be made to have them
   unchecked by default - they are typically a part of the end product. In the
   case of .bat, this particularly needs to be kept in version control by 
   default because it is typically written by hand, i.e. it is typically not
   the output of a process.

3) .zip and .gz are the functional equivalent of .Z files. If you have .Z files
   in the ignore list, you need to add the other 2 extensions. I guess it is
   probably more useful to ignore these by default.

4) .log and core are crash output files. I believe these should be ignored by
   default. When I looked at the ignore pattern list for the first time, I
   noticed that core was unchecked. I can't imagine why anyone might want to
   version control a core file. <grin>


The final argument for having a bunch of unchecked items in the ignore list:

It is far better to have a dialog come up once (and yes, you should have a 
checkbox on the dialog for "Ignore these type(s) of resource in future") than 
to have multiple failed builds, and to have people assume that there is a bug 
in version control, giving them the feeling that they should not trust the 
tool.
Comment 1 Konrad Kolosowski 2004-06-10 12:35:30 EDT 
I believe this bug was the cause of all 2.1.x builds shipping 
org.eclipse.update.core.linux fragment without the native .so library - 
esentially a useless fragment.

.cvsignore allows overriding patterns when they are enabled by default in 
preferences (or not defined).
Is there anything that allows overriding things that are disabled by default?  
If you leave .so checked, it would be nice to be able to specify it as an 
exception for a project and check this preference in the repository for a 
specific file or project, so that people working on the project do not have to 
change defaults and remember to do so for every new workspace.

One more candidate to be ignored: "temp.folder"
Comment 2 Carolyn MacLeod 2004-06-10 13:07:30 EDT 
I agree that a ".cvsdontignore" might be nice, but I feel very strongly that 
it is safer to default to "do not ignore" and let people figure out that there 
is a couple of ways to ignore stuff, than to ignore too many things right off 
the bat.

I *always* do a "Compare with->Released" on all of our projects as a sanity 
check right after I make my final tags and update my map file (the SWT build 
process is somewhat more complicated than the typical build experience, so I 
feel better if I think I have released everything <g>). However, since I 
didn't realize that .so files and .a files were being ignored - even for 
compares! - I bogusly thought that "No differences" actually meant "No 
differences". In retrospect I suppose this was somewhat naive, however I would 
venture to guess that others would easily make the same assumption.

So if you're not absolutely certain that a resource name pattern will always 
match files that should never be version controlled (like 'core' or *~ or 
*.tmp or *.old or *.bak or *.obj or *.o) then please don't ignore them by 
default.
Comment 3 Carolyn MacLeod 2004-06-10 13:08:02 EDT 
Hmmm... another one for the list:  *.o  (same as *.obj)
Comment 4 Jean-Michel Lemieux 2004-06-10 14:01:08 EDT 
If we do anything it will be to uncheck all in a new workspace and use the
prompt to guide users to configure. That way in a clean workspace you won't miss
any *real* changes and if you have hand crafted an extensive list you could
simply import your preference file.

We use the default CVS ignore list and that is the historical reason for the
current list. Not for 3.0 though.
