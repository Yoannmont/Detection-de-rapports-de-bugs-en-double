Description Suyog Mody 2003-02-26 14:40:51 EST 
Popup menus do not work on PocketPC (using HP iPaq 5450 for testing).
On other programs in PocketPC, popup menus come up after holding the stylus 
down for a couple of seconds.

Here is a test case in SWT-

import org.eclipse.swt.SWT;
import org.eclipse.swt.widgets.Composite;
import org.eclipse.swt.widgets.Display;
import org.eclipse.swt.widgets.Menu;
import org.eclipse.swt.widgets.MenuItem;
import org.eclipse.swt.widgets.Shell;

public class SWTPopupTest {

public static void main (String [] args) {
	Display display = new Display ();
	Shell shell = new Shell (display);
	Composite c1 = new Composite (shell, SWT.BORDER);
	c1.setSize (100, 100);
	Composite c2 = new Composite (shell, SWT.BORDER);
	c2.setBounds (100, 0, 100, 100);
	Menu menu = new Menu (shell, SWT.POP_UP);
	MenuItem item = new MenuItem (menu, SWT.PUSH);
	item.setText ("Popup");
	c1.setMenu (menu);
	c2.setMenu (menu);
	shell.setMenu (menu);
	shell.setSize (300, 300);
	shell.open ();
	while (!shell.isDisposed ()) {
		if (!display.readAndDispatch ()) display.sleep ();
	}
	display.dispose ();
}
}

This bug is causing us not to pass a test in the Personal Profile 
compliance test suite on PocketPC.

Comment 4 Stef van Dijk 2003-03-04 10:07:15 EST 
I think that unfortunately this isn't going to resolve our problem with the AWT 
testcase (clean version of a TCK test). The fix would certainly seem to fix the 
provided testcase (note that I haven't actually tried it yet myself), but the 
problem is that the SWT testcase is not representative of the way AWT expects 
popup menus to work (at least as illustrated by the AWT test).
In AWT, MouseEvent supports a method isPopupTrigger(), and the AWT testcase 
uses this to determine whether to show the menu. The problem is that we (AWT) 
don't get sufficient information to be able to identify that an event has 
occurred that should cause the isPopupTrigger() method to return true.

I think we have some more work to do on our end to understand what we need SWT 
to help us provide. For example, how are we able to detect the trigger on Win32 
since it looks to me as though SWT signals no events at all for popup menus?
Comment 5 Stef van Dijk 2003-03-04 10:30:03 EST 
A quick check shows that AWT just detects the MouseUp and appropriate button 
modifier to determine that the event is a popup trigger (and that maybe we're 
broke on motif/gtk). It would seem that unless SWT can provide some new event, 
we in AWT are going to have to detect this ourselves, probably by using the 
same logic Chris has encapsulated in the SHRecognizeGesture native.

Chris, I know this is starting to get outside of SWT now, but do you have any 
thoughts or suggestions?
Comment 6 David Whiteman 2003-03-04 10:55:20 EST

Comment 8 Stef van Dijk 2003-03-04 11:22:13 EST 
Yes, the chain of events you laid out is correct.
The reason there isn't a second popup menu is that bbAWT never sets a menu on

Someone else feel free to correct me if I'm wrong about this.
Comment 9 Suyog Mody 2003-03-04 11:56:17 EST 
That is correct. We never set a popup menu on the SWT widget so it works only 
with our workaround on Win32.

The reason that popup menus dont work on PocketPC is that SWT sets mouse up and 
left button click when you release the stylus (even if you hold the stylus down 
for a popup). Right button click is never set on PocketPC which makes our 
isPopupTrigger() method return false each time for this platform. 

A possible fix for this is SWT sets the type of button 
(org.eclipse.swt.widgets.Event.button) to the right-click button when it 
detects a "tap and hold" on PocketPC.
Comment 10 David Whiteman 2003-03-04 12:14:36 EST 
Actually, java.awt.Component does have:

public synchronized void add(PopupMenu popupMenu)

This adds the AWT Popup Menu to a collection of popups managed by the 
Component.  Of course I can't figure out what we do with that collection :-)  
But Stef is correct that we never set the popup in the SWT widget.

I believe the standard way of using AWT popups is to just register a mouse 
listener, and handle it this way:

	if (e.isPopupTrigger()) {
		myPopupMenu.show((Component)e.getSource(), e.getX(), e.getY());
	}	

Under the covers, it appears the X and Y are being ignored, as we just get the 
corresponding SWT Menu widget and call setVisible(true).

So the thing we need to look at is why SWT doesn't show a menu on PocketPC when 
setVisible(true) is called.

Comment 15 Christophe Cornu 2003-04-10 11:39:45 EDT 
SWT.MenuDetect has been added.

One scenario for bbAWT is to do 1) and 2):
1) listen to SWT.MenuDetect events
2) on PocketPC, maybe AWT should treat SWT.MenuDetect as a "bbAWT Mouse Event 
isPopupTrigger true Right Button Up".

What do you think?

So what does the new SWT.MenuDetect buy us regarding the bbAWT issue? It is 
used on SWT PocketPC to activate the 'popup gesture' even when a popup menu is 
not set.

On this platform, the application is expected to trigger the 'popup gesture' 
only if it really cares about popup menus. In SWT, this now translates into:

1) a popup menu is set -> 'popup gesture' is activated -> if successfull, 
SWT.MenuDetect is sent, popup menu is set visible if event.doit is true
2) no popup, SWT.MenuDetect is hooked -> 'popup gesture' is activated -> 
SWT.MenuDetect is sent
3) none of the above -> 'popup gesture' never activated, no SWT.MenuDetect event

2) is the bbAWT case (no popup set).
