Description Stefan Xenos 2003-10-23 16:22:28 EDT 
The problem:

GridLayout is caching the result of calling computeSize(SWT.NONE, SWT.NONE) on
its children, even though the actual height of the control may change if it is
inserted into a column whose width is different than its preferred width. This
causes problems when creating GridLayouts containing wrapping text.


To reproduce:

1. Create a wrapping text control with the following properties:
- If asked for its preferred size (via computeSize(SWT.NONE, SWT.NONE)), it will
return a fixed width (say 1/4 of the screen) and return the height it needs to
be to wrap itself to that width.
- If given a width hint (via computeSize(somewidth, SWT.NONE)), it will return
the height it needs to be to wrap to somewidth.
- Such a control can be created by encapsulating a wrapping Label within a
Composite that uses a custom layout to return the desired result from computeSize.

2. Insert this control (containing enough text to wrap several lines) into a
1-column GridLayout along with another control (say a button with a very long
label) whose preferred width is significantly larger than the preferred width of
the control.

3. Lay out the controls.

A large empty space will be left below the wrapping text widget.


Why this happens:

When laying out its children, GridLayout first calls computeSize(SWT.NONE,
SWT.NONE) on all controls, and caches the heights. It computes the column width
based on the widest control, then arranges the controls based on the cached
control height.

This is incorrect: once gridlayout knows the column widths, it should invoke
computeSize(actualColumnWidth, SWT.NONE) on all controls to determine the actual
control heights, since the height of some controls (in particular, wrapping
text) may have changed.


Notes:

This problem requires a composite containing a non-GridLayout to reproduce.
Since all the existing wrapping SWT widgets return their unwrapped width as
their preferred width, their height never changes if they are inserted into a
larger column. (Perhaps a DFS for non-GridLayouts could be used as a heuristic
optimization to determine if the additional computeSize(...) calls are necessary)


Why this is important:

This may seem like a rather escoteric problem, but the contents of all JFace
windows are placed within a GridLayout by default, and many objects (such as
wizard pages) are forced inside GridLayouts. This prevents anything within a
GridLayout from using a custom layout to provide more desirable sizing behavior
for wrapping text.

Comment 2 Stefan Xenos 2003-10-24 15:17:55 EDT 
Bug 9866 is about the inability to create wrapping text in a GridLayout. This
bug is about incorrect width hints being passed into child controls. Bug 9866
depends on this bug, it is not a duplicate.

My particular problem is that I have created my own layout class (which fixes
many known issues with GridLayout), but whenever my layout is placed within a
GridLayout it does not know its own width, so cannot compute its preferred
height properly. I only mentioned wrapping text because this is an easily
reproducable implication of this bug.

The workaround in bug 9866 does not solve this. True, it would cause a better
width hint to be passed to my layout if the enclosing grid layout has exactly 1
column and its size is known, however this is not true in most situations. When
computing the initial size of a dialog, the size of the enclosing layout is
still unknown (since this is what we're computing) and my layout would still
receive a bogus width hint. By the time the correct width hint is computed (in
the the workaround's resize listener), the enclosing layout has already been set
to an incorrect size.

Comment 6 Steve Northover 2003-10-29 18:09:50 EST 
Yes (by definition) every layout uses the values that are provided in the 
width hint and height hint fields of the appropriate layout data.  The 
question is under what circumstances (if any) it should make multiple calls to 
computeSize(), substituting the current width (or height).  This requires much 
more thought as the process could be endless.  For example, calling computeSize
(value, SWT.DEFAULT) on one control then calling computeSize (SWT.DEFAULT, 
otherValue) on another could possibly cycle with one control demanding more 
width and another demanding more height (not sure about this).

Comment 8 Stefan Xenos 2003-10-29 22:24:03 EST 
Sorry for the numerous comments, but I already worked through some of these
issues in the new layout I'm writing for JFace. Just in case this is helpful,
I'm including this here...

To fix this bug, child controls only need the ability to compute one dimension
given the actual size of the other. GridLayout still chooses which dimension
each child is allowed to compute, and it can use this to prevent cycles.

For example, GridLayout can call computeSize(SWT.DEFAULT, SWT.DEFAULT) on all
children, and use this to compute the column widths. Then it can call
computeSize(actualColumnWidth, SWT.DEFAULT) on each child and use the results to
compute the row heights. In this way, each child has had the opportunity to
compute one of its dimensions, but there is no possibility for cycles. You never
need to call computeSize(SWT.DEFAULT, someActualHeight) unless you compute the
rows first (which also works). 

This is the technique I'm using for my layout class, and it seems to work well.
