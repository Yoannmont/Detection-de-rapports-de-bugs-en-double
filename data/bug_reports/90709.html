Description Nick Edgar 2005-04-07 15:37:05 EDT 
build I20050405

In order to partially address the split editor feature request (bug 8009) in
3.1, we have decided to add an "Open Again" action in the File menu (just below
Close All) and in the editor tab.

This will open a new editor with the same id and input as the active editor.

For the split editor scenario, the user only has to drag the resulting editor's
tab to the edge of the editor area, to move the editor to a new editor workbook.
Comment 1 Nick Edgar 2005-04-07 16:02:49 EDT 
Stefan had an alternative suggestion, which was to rename Open With to Open New,
and have it always open a new editor (of the selected editor type).

In addition to addressing the current bug whereby this is a no-op if there's
already an editor open with the same input (regardless of editor type), this
would allow us to open a second instance on the same input without adding any
new UI.

Michael, Erich and Dani, what do you think of this proposal?

Some pros and cons I see are:
Pros:
- no new UI required to support the scenario here (just a slight change to
existing UI)
- fixes the "can't open the same input in different types of editors" bug

Cons:
- requires changing multiple views (Navigator, Package Explorer, Plug-ins, other
downstream views), and all of them would have to be changed for consistency
- the UI is not as local to the editor as having "Open Again" on the editor tab

We would also have to consider whether the existing OpenWithMenu API should just
be changed (it is loosely spec'ed right now), or whether new API is required.
But I don't think we should let that affect the decision of what the UI should
look like.

I'd rather focus on performance right now and mark this as a re-visit for 3.2.

Comment 10 Nick Edgar 2005-04-12 14:43:23 EDT 
I agree with all 3 of Ed's points in comment 7.  In particular, I agree that the
expected behaviour of Open With is to activate any existing editor with the same
id and input, rather than opening a new one (and that the current behaviour
where it only matches based on input is a bug).

This is why Stefan proposed changing the name to Open New, and changing its
behaviour to always open a new editor each time.  

However, I don't think we should change Open With to Open New at this stage,
since it requires too many changes to different views, and would lead to
inconsistencies if some but not all navigator-like views changed.  We can't
isolate this change in the workbench -- at a minimum, the views have their own
"Open With" string (and mnemonic).  

It's also unclear to me that Open New is preferable to Open With in the case
where users just want a different kind of editor.  I think having multiple
instances of the same editor type on the same input would not be a desired side
effect.

As for the name "Open Again", I'm fine with changing this to "New Editor".  I
also think it's more likely that users wanting a "split editor" function will
look in the Window menu than the File menu.

So I propose:
- leave Open With as-is (should really fix the matching bug, but that's a
separate issue)
- rename "Open Again" to "New Editor"
- move it to the Window menu, just below "New Window"
- change the action on the editor tab menu to "New Editor" as well

Also need to define a corresponding command (with no default key binding).
Comment 11 Stefan Xenos 2005-04-12 16:56:33 EDT

apply to the whole window. 

(IMO, we should also add it to any context menu that currently contains an
"Open" and "Open With" action -- perhaps limiting its visibility to only show up
if there is already an editor open on that input).

Comment 17 Nick Edgar 2005-04-13 13:57:14 EDT 
If we have "New Editor" in the File menu, I'm concerned about it being
interpreted like File > New in other text editors and word processors.  That is,
open a new empty document.  Our action for this is currently called "Open
Untitled Text File", although it's not really opening a file (it is however
opening an editor).

I'd be less concerned about using "New Editor" if the action only appeared on
the editor tab.

Also, I want to encourage the user to think in terms of their model objects
(files, classes, whatever), not workbench abstractions (editors, views,
perspectives), so I'm not really in favour of adding an "Editor" suffix to the
Close actions.  Doing so would also make Save and the other actions here appear
inconsistent (although most of these can target views too, so that could be an
argument against the suffix for those actions).

For New Editor, it's appropriate for the user to think in terms of opening
another editor.

At this point, I'm inclined to remove the action from the main menus, and just
have "New Editor" on the editor tab.

Erich and Dani, if you feel strongly that it should go in the File menu, I
suggest leaving it as "Open Again" (it's like this in the current I-build).

Comment 32 Nick Edgar 2005-05-09 13:49:28 EDT 
After discussions with Erich, Dani and MvM, we've decided to move the action to
Window > New Editor (under the existing New Window item), for the following reasons:
- It differs from the other File menu items in that the user's focus is on
opening another "window" on their content, not on the content per se. 
J.F.Lanting's comment 14 resonates with us.
- It reduces clutter in the File menu, and hopefuly reduces confusion when new
users are trying to find entrypoints into the IDE.
- It's similar to what Window > New Window does.
- Other apps have similar menu items in the Window menu.

This will involve an API addition: adding ActionFactory.NEW_EDITOR.  This action
was not previously API, and required an internal reference from the IDE's
WorkbenchActionBuilder to the workbench internal OpenAgainAction.

Erich, can you please approve this?

Comment 36 Stefan Xenos 2005-05-09 16:26:56 EDT 
Re: comment 32

The fact that so much discussion was necessary leads me to believe that our
guidelines aren't specific enough. If this PR decides that the window menu is
for "any layout operation that doesn't involve opening or closing a model
element", and another PR decices that the window menu is for "actions that
affect the entire window", then we will end up with an inconsistent UI.

The UI guidelines should make it clear which actions belong on which menus.
There are always fringe cases, but for simple, common operations like this there
should be no ambiguity.
 
I've filed bug 94196 requesting more specific guidelines.
