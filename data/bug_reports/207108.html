Description Bjorn Freeman-Benson 2007-10-23 00:00:06 EDT 
As we gather more innovative projects around Eclipse, we gather more committers and project teams who are not as steeped in the processes (such as the IP Policy) as some of the longer term Eclipse residents. This leads to misunderstandings wherein a project releases a major release without getting appropriate reviews and approvals. (Note that even Apache requires reviews and approvals, so this requirement isn't overly constraining, it just needs to be followed: http://www.apache.org/dev/release.html#what).

At the same time, as we have more Eclipse projects, each one defining its own download and release web pages, it becomes harder for adopters and users to find the correct bits from each project.

I suggest that we knock off two issues (misunderstanding the necessary clearances and complex download pages) with one solution by having standard download pages and a portal-based checklist tool for publishing "releases" on those download pages. The standard download page would still allow projects to publish N, I, milestone and release candidate builds without going through the portal checklist (because those builds do not require IP clearance). Of course the portal checklist would support the abbreviated checklist for bugfix x.y.z and x.y.z.w releases.

We will want to support:
* Platform project style download pages: http://download.eclipse.org/eclipse/downloads/
* CDT style download pages (with descriptions of each release): http://www.eclipse.org/cdt/downloads.php
* Mylyn style download pages (with extra descriptions and images to enhance the user experience): http://www.eclipse.org/mylyn/downloads/
* WTP style download pages (with extra stats about each drop): http://download.eclipse.org/webtools/downloads/

The modeling project, with its myriad of individual components, would need to have separate download pages for each, but it effectively does already, so that shouldn't be a problem.

I suggest having a standard directory layout for each drop, with a standard location for the new & noteworthy, release notes, etc. so that each can be exposed through the download page.

And, of course, all the advantages of having a consistent look (same as the rest of the eclipse.org website) and automatically adding the incubation eggs and using a common code base so that every project doesn't have to maintain their own, etc, etc.

I imagine that most project's releng process would not change: N, I, and S builds would continue to be scp'ed to the download.eclipse.org directories. The releng process would no longer update the index.html or index.php file because it would be the standard single file. When a project wanted to do a release, the project lead would go to the portal and start the release review process. Once that process was complete (all the checklist items checked, including IP clearance approval), the final portal dialog box would be something like "promote the most recent S build as the final release - press ok to continue". Or perhaps "enter the name of the S build to promote". That would mark the build as an official release and the download/index.php script would then show it as the released version.

Note that this bug is only proposing to standardize the top-level of the downloads page structure. If projects want to make a more complex set of download pages past the top-level download page for their project, they can continue to do so. I can imagine that there would be some peer pressure not to make a more complex structure because then the project wouldn't "fit in", but that's neither here nor there with regards to this bug.

Also note that if we did this (standard download pages with portal release process), I could stop policing whether people call something a release or not. Project members could blog "release" when they meant "I-build" - none of that would matter because only official approved releases would show up as "releases" on the download sites.

Comment 22 Eike Stepper 2011-11-12 10:57:17 EST 
> Can you describe the workflow a bit 

Pasting in from my email to you (subject: Building, date: 2011-07-01):

The entire system comrises several infrastructure parts:

- The source repository (SVN in our case)
- Bugzilla to provide input for the generated release notes
- Hudson (2 jobs in our case)
- download.eclipse.org
- our project web space for the generated downloads html page
- a separate web space on build.eclipse.org for a limited number of administrative tasks (copy/rename/delete builds, manage release train staging, etc)
- a working data directory (as of now /tmp/promotion.emf.cdo)
- a "promoter.sh" cron job that can do things under my user id
- a configuration area (I maintain the config data in SVN, too)

The core of the system is the "promoter" job. I've designed a shell script such that it can be started very frequently (e.g. every 1 or 5 minutes). It can recognize very quickly if at least one of the following two conditions is true;

- One of the configured Hudson jobs has new builds (i.e. nextBuildNumber has changed)
- An admin task is enqueued from the admin web interface

If both are false the script ends after milliseconds. In addition the script exits immediately if another instance of the script is already/still running.

If builds are available on Hudson, the script starts a Java process with a two handfull of self-written classes (no other dependencies so far). These classes do the following:

1) The configured Hudson jobs are analyzed in more detail. If new builds are to be promoted (e.g. they're successfully finished, they match configurable criteria like build type etc) they are copied to the configured downloads location (should be a folder called "drops" so that the default mirroring filters exclude N and I builds from mirror distribution)

2) The proper SVN branch is tagged with the build qualifier.

3) The build gets post processed:
    a) content.jar and artifacts.jar are unpacked
    b) from content.xml an additional p2 repository is generated (with xslt) that lists the categories of the original repository under a new unique (i.e. build-specific)  category (e.g. "CDO I20110701-1455"). This is important for a step further down where the build/drop p2 repos get composed into higher level p2 repos (e.g. integration builds, releases, etc). I've learned this from Pascal's blog about category nesting ;-)
    c) p2.mirrorsURL is set to correct values for both repositories (build and categories)
    d) a zip of the p2 repo is created
    e) a zip for the dropins folder is created

4) According to the configuration area a number of p2 composite repositories is generated from the drop repos and the categories repos. Starting with this step all actions happen in a temp folder so that the final result of all the generation can be switched live almost atomically.

5) Release notes get generated according to some configurable rules. The idea is that the release notes show the ID and title of all bugzillas that are mentioned in the SVN commit messages between the last (relevant) build in a composite repo and the current build.

6) The entire downloads page is statically generated (no php in it!)

7) Finally the temporary generation folder is renamed to the public location so that the repos become accessible and the downloads page visible.

http://git.eclipse.org/c/cdo/cdo.infrastructure.git/tree
