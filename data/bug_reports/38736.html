Comment 1 Luc Bourlier 2003-07-10 17:21:26 EDT 
From the stack frame, it looks like we ask the VM for a stack frame at an index
bigger than the stack size.
This request is created when the UI try to get the labels of the tree elements
in the DebugView.
A reason can be that the number of stack frame created in the Java model was
bigger than the number of stack frame in the thread (in the VM).

Darin W: Do you know if this could have been a consequence of the problems there
were in JDIThread#computeStackFrames(boolean) ?
Comment 2 Darin Wright 2003-07-10 17:27:06 EDT 
It's possible that the problem was there (need to see if it's still there). It 
could be that by the time the UI updates, the stack size has gotten smaller - 
so it tries to retrieve an "out of bounds frame".

Comment 11 Jared Burns 2003-11-19 12:10:16 EST 
Fixed in JDIThread. We were allowing IndexOutOfBoundsException to fall through 
to our catch (RuntimeException) block. This block calls code that only wraps the 
exception in a debug exception if it's a "JDI Exception". Since JDI says that 
the method we're calling can throw index out of bounds, we should catch it 
explicitly and always wrap it in a debug exception. The code in the label 
computation correctly handles debug exceptions.

Please verify, DarinW.
