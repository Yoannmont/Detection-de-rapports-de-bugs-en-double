Comment 11 John Arthorne 2004-08-10 09:35:19 EDT 
The first build after a clean must be a full build. The intent of an incremental
build is to update the build state (class files, etc), only for the source that
has changed since the last build. Since a clean instructs builders to discard
all built state, the subsequent build has no choice but to start from scratch
and do a full build.

Comment 13 John Arthorne 2004-08-10 10:15:55 EDT 
If you are using make, then there is no interesting distinction between
FULL_BUILD and INCREMENTAL_BUILD. The significant difference is the availability
of a resource delta in the INCREMENTAL_BUILD case. If you are not using the
delta, then I suggest implementing FULL_BUILD and INCREMENTAL_BUILD to just
"make all" (no clean). FULL_BUILD in Eclipse 3.0 really means "the first build
after a "clean". Thus, if you are adding a free "clean" in the implementation of
FULL_BUILD, you will always get the duplication of cleaning.

I understand that there is a deeper problem that the set of available build
commands in Eclipse doesn't properly map to languages and builders that do not
have true incremental compilation (ant-based builders being the prime example).
In this world autobuild often does not make sense, because it becomes too costly
when builders are not smart enough to properly incrementally build very large
workspaces efficiently. (Because of the difficulty of true incremental
compilation with more complex languages such as C++, not because of poorly
written builders).

The added challenge in Eclipse is that the environment is often heterogenous. A
workspace may contain some effecient incremental builders (such as the Java
builder), alongside less efficient Ant or make builders. See bug 60803 for more
comments on this.

So I notice that this is resolved as invalid.  I guess that means you don't consider it a bug?
Should I open a new feature request then, or would you be open to re-opening this bug?  I do consider it a bug and I think it would be useful to either change the behaviour of this checkbox, or perhaps turn it into a radio box.  Allow me to explain.

I think there are use cases for both scenarios (cleaning and building the entire workspace, or just the selected projects), but the fact is that this checkbox is *only* visible when you have autobuild turned OFF.  I think it is safe to assume that if the user has autobuild turned OFF, that they will undoubtedly have projects in their workspace they do not want to be built automatically.
We use Ant as the main driver for our toolchain and it is very similar to how make would work for C projects.  Our users can have many many projects - all in a state of disarray - when they want to clean and rebuild one project.  This triggers a build of the entire workspace that is not only lengthy, but can result in hundreds of errors (of which they are well aware).

So, I would propose the following behaviour change that I think would satisfy everyone:

With autobuild ON:
- There is no issue as there is no checkbox.

With autobuild OFF:
- If the user selects "Clean all projects", then the current behaviour is fine.  A single option should be shown allowing the user to trigger a rebuild of the entire workspace or not.

- If the user selects "Clean projects selected below", then I think one of the following should be done:

  a) Leave a single option and change the behaviour to only rebuild the selected projects.  I would also change the description of the checkbox in this case to be "Start a build of the selected projects immediately"

  b) Change the checkbox to a radio button (or add a combo box in addition to the checkbox) in this case with two options:
       - "Start a build of the entire workspace immediately"
       - "Start a build of the selected projects immediately"

Personally, I would prefer option "b".

Incidentally - while we are talking about this dialog I can say that almost every single one of our users have been confused about the fact that when you select "Clean all projects", and the checkbox list view of projects is disabled, the checkboxes all look like they are still enabled.  When you glance at the dialog quickly (which you often do when cleaning) it is very easy to think you are only cleaning a subset of projects because the checkboxes are all green and checked, when in fact you are about to accidentally clean your *entire workspace* - which could be a HUGE operation.

Comment 20 John Arthorne 2007-06-22 15:27:15 EDT 
It's certainly a topic we've been struggling with for many years. The problem is that there are many varieties of builders out there, and it's difficult to find a UI presentation that doesn't inundate users with bells and whistles that may not be needed for their workflow. The ideal we have striven for in Eclipse is for building to be invisible, and the builder just figures out what needs to be done, and does it in the background without getting in the user's way. Just like you don't need to explicitly ask a WYSIWYG editor to "render" after changing the document, it would be nice if IDE users never had to ask for a build to occur. I realize this ideal is often difficult to achieve in practice.

Comment 28 James Blackburn 2010-03-22 08:55:30 EDT

The cause of this seems to be BuildManager#basicBuild sets the buildType to FULL_BUILD (appropriately) if lastBuildTree == null.
Unfortunately it checks whether the builder is applicable to the buildType having altered the trigger. The result is that all builders which claim to work for FULL_BUILD are called as a result of the auto build job.

I think the fix is trivial. However there are tests that assert this behaviour i.e. all FULL_BUILD project builders to be run on the first build after a clean (CustomBuildTriggerTest.java).

If a builder has: command.setBuilding(IncrementalProjectBuilder.INCREMENTAL_BUILD, false);, isn't it wrong to cause a FULL_BUILD on INCREMENTAL_BUILD?

CustomBuildTrigger#testBuildAfterClean
