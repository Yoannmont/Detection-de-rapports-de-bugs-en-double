Description Allan Chandler 2007-06-13 01:06:19 EDT 
When getting specific data from a database, the charting work takes forever (as best as I can estimate :-).  I have a table mvs_system_h on DB2 structured as date, time, cpu_load_min_pct, cpu_load_max_pct, cpu_busy_sec and cpu_online_sec.  The first two are date and time types, the others are floats.  There are other columns in the table but they're not extracted by the query.  The date (and another column, mvs_system_id (char(4)) are used to extract the data.

When I graph the data for a specific system id, the BIRT RCP designer and my web app do not return the report; they basically take forever (well, at least 10 minutes at which point I got tired of waiting).  It also sucks up 100% of the CPU which caused grief on the test box (at least my dev box was dual core so I could still function a bit).  If I remove the charts from the report design, the report is shown within a couple of seconds (which is why I'm assuming it's a chart problem).

The query is:
===
   select
      char(date,iso) as isodate,
      left(char(time,iso),2) as isotime,
      round(cpu_load_min_pct,0) as min_cpu_pct,
      round(cpu_load_max_pct,0) as max_cpu_pct,
      round(cpu_busy_sec,0) as busy_secs,
      round(cpu_online_sec,0) as online_secs
   from
      drlkd.mvs_system_h
   where
      (char(date,iso) = ? and mvs_system_id = ?)
   order by
      time
===

Using date=2007-03-19 and mvs_system_id=BEAB is when the problem occurs.  Relevant data for that single record is:
   date = 2007-03-19
   time = 23.00.00
   mvs_system_id = BEAB
   cpu_online_sec = +0.6000219999999999E+03 (~600)
   cpu_busy_sec = +0.3322367280000000E+03 (~332)
   cpu_load_max_pct = +0.5539183363276679E+02 (~55)
   cpu_load_min_pct = +0.5534968184499901E+02 (~55)

I thought it might be something to do with a query returning one record but other queries of that form (and queries returning multiple rows) are okay, one example being:
   date = 2006-07-24
   time = 11.00.00
   mvs_system_id = Z3
   cpu_online_sec = +0.1439988800000000E+05 (~14400)
   cpu_busy_sec = +0.1072397043100000E+05 (~10724)
   cpu_load_max_pct = +0.7631366071736113E+02 (~76)
   cpu_load_min_pct = +0.7292249667497413E+02 (~73)

Attached report designs were made with the BIRT 2.1.3 RCP designer stable build S20070524 (won't load into BIRT 2.1.2 for some reason - you think a minor dot realease would still be compatible but, hey, I'm not the developer :-).

I found only one potential related bug bug it was the CPU going to 100% only for PDF reports (HTML worked okay).  Mine is an HTML report so it doesn't seem to apply.

Will attach qsample.rptdesign and qsample_nograph.rptdesign, the only difference in the latter is the removal of both charts.  The report only started to work once BOTH reports were removed (I tested that).

Okay, did a bit more investigation before submitting this bug (it's getting weird).

I changed the query to see if there was a problem with having min and max cpu % the same (since that's another difference between the rows).
===
   select
      char(date,iso) as isodate,
      left(char(time,iso),2) as isotime,
      50 as min_cpu_pct,
      50 as max_cpu_pct,
      round(cpu_busy_sec,0) as busy_secs,
      round(cpu_online_sec,0) as online_secs
   from
      drlkd.mvs_system_h
   where
      (char(date,iso) = ? and mvs_system_id = ?)
   order by
      time
===

It worked fine, so I then changed both to 55 and it's taking forever again.  WTF?  More investigation needed. Here's what I've tried so far.

So it appears to be a problem around the 55 area, although 49/49 and 50/51 also fail (so does 51/51, although 51/52 works - these were tested but not listed above).  It occurs when the numbers are close together.  It's symetrical (doesn't matter which value is in min and which is in max).  It's not a sum problem alone as 56/54 works where 55/55 doesn't (it may be a combo between sum and proximity).

I also tested 55/55 against the BEAB query (returns 24 rows) and it failed so it appears number of rows is not important.

Best of luck with this one, guys.

Comment 7 Yulin Wang 2007-06-14 02:27:22 EDT 
This bug has been fixed as bug#188695. It won't freeze any longer.
However, there's a workaround in your report, not only for avoiding freezing, but also for a better looking chart. Please select Min value of Y axis as the origin of X axis. The UI is in: Format chart->X Axis->Origin, and Min in Combo.

*** This bug has been marked as a duplicate of bug 188695 ***
Comment 8 Allan Chandler 2007-06-14 03:02:36 EDT 
Thanks Yulin, that appears to have fixed the problem with the chart hanging.

Whilst I agree it's a better looking chart, it's certainly not as useful in terms of showing the information.  It's long been a problem with graphs that don't use zero as an origin point - they seem to indicate movements far greater than reality.

For example, a stock price chart with  min/max of 9.00/10.00 could drop from 10.00 to 9.50 and that looks like a massive 50% drop on the chart, not the more realistic 5% (although the people reading the charts should hopefully know better).

However, I'm happy to use this workaround.  Will the 'real' fix be backfitted to the upcoming 2.1.3?  We're eagerly awaiting that release since we can't yet go to 2.2 due to its non release status (I know 2.1.3 is also non-release, but it's likely to be safer as it involves less changes to the 2.1.2 code).

I ask because the target release for bug # 188695 is set to 2.1.3 but the comments indicate it will be in 2.2 only.

Another question I have is: what if the actual origin is zero so that the workaround does not change the origin by using Min instead of Val=0?

In other words, what if the data for the two series are (A,0),(B,55) and (A,55),(B,55)?  Will that still be a problem?  Would the first series having 0 as the first y-value be equivalent to using Val=0?

Comment 10 Yulin Wang 2007-06-14 03:24:42 EDT 
Answer 1: This bug and #188695 are fixed in 2.1.3 branch. The fix just resolves the hang issue, but the chart looks not so good in the chart with 0 origin.
Answer 2: Origin Min means chart engine will detect the minimum (in one Axis, not one series) as Origin. If you set Origin, will use the fixed value. However, that may produce a bad looking chart. So in your attached image, BIRT chart should do the same way, if you choose Min Origin which is default choice in Excel.
