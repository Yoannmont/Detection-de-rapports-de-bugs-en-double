Description Johan Walles 2002-03-20 16:30:27 EST 
Just like all other Unix apps, Eclipse should default to storing its workspace
somewhere in the user's $HOME directory.  People who want their workspace
somewhere else should have to use the -data switch.

Bug 5927 together with the fact that Eclipse wants to save its workspace in the
current working directory makes the install directory the current default
position for the workspace.  As the install directory will in many cases be
write protected, the current default is broken on many systems.

Comment 6 Michael Toomim 2002-09-27 15:04:44 EDT 
How about a compromise that will satisfy new and old users: If there's no
data argumnet, check the current directory for a workspace directory.  If
one exists, use it.  Otherwise, check the user's home directory.  If
a workspace directory exists there, use it. Otherwise, create one in
the user's home directory.

This way, nobody's existing eclipse installation will break, but new users
won't be subject to the really confusing behavior of pwd-dependent workspaces.

I'd *really* like to see this bug reopened.

Comment 8 Jared Burns 2002-11-14 18:01:51 EST 
I also think this should be reopened. My comments from Bug 26226:

By default, Eclipse puts the workspace in the current working directory. In
general, users launch Eclipse from the install directory. This model is
dangerous in single-user installs because it means that deleting the Eclipse
install will also delete the workspace. This model is wrong in shared
installs because the user cannot write to the Eclipse install dir.

This model is confusing to the user in general because most applications
don't care about where you launch them from. Users have no expectation that there will be a difference if they launch a program from /home/$USER/foo one
day and from /home/$USER/bar the next.

We should put the workspace in a platform-standard location. On Linux, the
normal place to put the user's workspace would be in the user's home
directory, probably in a folder named ".eclipse". I'm not sure what the norm
is for Windows, but I'd speculate that a better place to put the user's
workspace would be their home dir.
Comment 9 John Arthorne 2002-11-15 11:56:12 EST 
For the sake of argument, why would people launch from the install directory on
a multi-user install?  I would never cd to /usr/bin to launch any other
application on unix.  Other apps, such as editors and the java command line
tools, work the same as eclipse.  You cd to where your data is, then you invoke
the tool. If I invoke "emacs foo.txt" or "java A.java", it will look in the pwd
for the data.  If I type "make" or "ant", it looks in the pwd for the build file.

I think the confusion is over the fact that eclipse stores metadata and user
data in the same diectory (the "-data" directory).  Typically a unix program
will take user data from the pwd, and put metadata (such as user settings,
preferences) in the user's home directory.  In eclipse they're stored together,
so it's a toss-up as to which policy makes the most sense. Michael's suggestion
about search in the home directory if a workspace isn't found in the pwd is
worth considering.

Comment 11 Jared Burns 2002-11-20 17:49:31 EST 
The point about Eclipse failing if you launch it from the install directory
was just an example of the more general problem which is the fact that
Eclipse has an unexpected tie to the current working dir. I can think of no
other Linux app that writes files to the current working directory when launched.

I don't think the comparison to Emacs is fair because Emacs is a single file
editor that you pass a filename to. The fact that the filename is resolved to
the current directory is just standard shell behavior. I think Mail is much
closer to Eclipse. Like Eclipse, mail programs contain transient user data
and metadata that the user is presented with each time they launch. Like
Eclipse, the location of the users mail in the filesystem is implicit by
default; You don't specify a file to open when you launch Eclipse or a mail
program. Unlike Eclipse, mail programs follow the *nux convention of storing
their data in the user's home directory.

For clarify, there are two distinct issues here:
1. The current behavior causes problems (user confusion, loss of data, 
failure to launch)
2. The current behavior is not standard behavior on Linux. Standard behavior
for a Linux app is to store program data in the users home dir.

Comment 13 Alex Blewitt 2003-02-19 10:18:01 EST 
I concur with the users below. The current model only works on single-user systems (like Windows) where everyone typically has their own install of the tool.

On multi-user systems, the default should be to use the home directory.

Even on Windows systems, it can be really confusing for the users. One user created a shortcut to Eclipse to discover that the workspace wasn't in the ${Eclipse}/workspace, but in the same directory as the shortcut.

Lastly, on other systems (like Mac OS X) you don't run eclipse as a command line; you run it as an application by double clicking on it in the user interface.

Note also that unlike emacs, Eclipse isn't about editing a single file. It's about editing a set of files in a per-project repository, which may not even be the usual location of the workspace anyway. The only difference is where the per-user workspace is stored, which then can point to projects outside of the Eclipse workspace area.

This bug really should be reopened. DJ Horton's comments that too many people depending on the current behaviour could easily be addressed by a migration tool, or by checking to see if the per-install workspace existed first, and if not defaulting to a per-user install instead.
