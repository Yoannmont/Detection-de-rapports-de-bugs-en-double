Description Helen Beeken 2006-06-22 06:55:13 EDT 
Whilst putting the last version of AspectJ in AJDT too many changes were required within AJDT to cope with updates in AspectJ. This is due to problems on both sides. 

On AJ's side the interfaces we expect AJDT to use aren't clear enough. The javadoc for many of the methods either isn't there or gives the wrong information. One has to look at the implementation to work out what should be returned (see IHierachy for example). Also, looking at IHierachy there are a lot of methods "findElementForXXX". Depending on which one you use depends on whether or not a file node is created if the element can't be found. This is generally not documented although it would be better if there was consistent behaviour. IRelationshipMap also needs to be looked at for example the get methods. For debugging the model, AJDT has a method dumpAJDEstructureModel() which uses the get(IProgramElement) method rather than get(String) version. These two methods return different things if the relationship end is in an injar. Consequently AJDT doesn't get hold of the end of the relationship which is contained in the jar file. AspectJ should also be able to change the implementation and not affect AJDT. This is particularly related to IProgramElement handles and filling in the structure model for injar aspects. Idealy AJDT should be able to cope with these programelements as they are filled in within the aspectj structure model. At the moment this is not the case.

On AJDT's side AJDT is too reliant on specific answers from AspectJ. For example AJDT's AJRelationshipManager class which keeps a record of all relationships. Should AJDT need to have its own list of relationships? Wouldn't it be better for AJDT to be able to ask AspectJ for the relationships it's providing and then deal with those? To compound matters at the moment AJDT uses an array of the relationships and in the persisting of the model relies on particular relationships to be in a specific place within the array. As it turns out in the case of removing uses pointcut/pointcut used by relationship this was ok since they were at the end of the array, however, this is not ideal. Models that were created with the uses pointcut/pointcut used by relationship should be able to be read (since all we've done is remove a relationship) but it needs to be able to cope with the (all be it unlikely) event that a new relationship was added. Again, earlier models should be able to be read since they just don't include this new relationship but the hard requirement of the index within the array makes this hard (especially if the uses pointcut/pointcut used by relationship isn't returned - bug 148027).
Comment 1 Matt Chapman 2006-09-25 10:03:01 EDT 
All good stuff. Here's my top 3:

1. We call the compiler, which launches the compilation in a new thread and then returns. This means AJDT then polls the compiler monitor repeatedly until end of compilation is indicated. This is really inefficient - why can't the compilation happen in-process, and return when the compilation is complete?

2. We pass the compiler a list of filenames to compile, and it writes them out to a .lst file. Is this disk I/O really necessary?

3. There are loads of redundant interfaces and methods - such as much of ProjectPropertiesAdapter, and when we call Ajde.init() we have to create a number of adapters many of which never seem to get called!

Comment 3 Mik Kersten 2006-09-26 14:28:41 EDT 
Agreed on all counts with Matt.  Regarding the approach to improving on the current situation, here's what I suggest, mostly in order.  We can chat about design issues on the AspectJ call.
1) Improve the API for running the compiler (Matt's points 1 and 2), probably bypassing the need to call Ajde.init() or making a lighter-weight init method.  I think this should be done in a way that doesn't break the current AJDE since there seems to be the occasional emergence effort from the JDeveloper and NetBeans folks.
2) Make sure that all the crosscutting structure AJDT needs is provided by ASM (which I think we should refer to as AspectJ Structure Model, not Abstract Structure Model).  The thing that's right about the model is that it's a mapping between elements that can represent arbitrary relationships and is compatible with JDT's java element handle identifiers, which can be easily used for everything including source files.  However, it should be updated to not construct these handle identifiers but to have AJDT pass them to it.  The constructing an IJava/AspectJ/Element will be one line of code via IProgramElement.getHandleIdnetifier(), and the ASM can continue using IProgramElement as a lightweight non-Eclipse-specific proxy.
3) Use the above to smooth out the APIs, but focusing on getting the redundancy out and getting the naming right, and not doing any java doc until the end.  This one is a personal preference based on the fact that most Javadoc that we have written gets out of synch with the implementation and naming anyway, so best wait until it has stabilized.

Comment 11 Mik Kersten 2006-10-12 11:59:48 EDT 
Helen: the doc you created looks good.  One thing to note is that the org.aspectj.ajde.resources.. stuff is just like an icons/ directory, not really API (follows an older Java, not an Eclipse convention of where to put icons).

I think that a key thing to think about going forward is the core/UI split.  There's a reasonable chance that AJDT should only be depending on core stuff and not care about any ajde.ui stuff, only about invoking the compiler and getting back the right messages and data structures.  The org.aspectj.asm component is already a core component so that's good.  But it's probably the wrong place to put the compiler API.  A good intermediary place could be an org.aspectj.ajde.core package, which has no dependency on org.aspectj.ajde.ui.  Or perhaps this could be a good time to bypass this, and to move the compiler API to org.aspectj.ajdt.core (some is already there, and that already has a dependency on org.aspectj.asm).
Comment 12 Helen Beeken 2006-10-16 08:35:50 EDT 
Mik - thanks for looking at the doc :-)

I agree that there should be a core/ui split within ajde. It seems that ajde was designed to also provide the views etc. required for looking at the cross cutting nature of your programs. In order to fit this within an IDE users were required to implement a few interfaces. The result being that the experience was similar regardless of the ide. As we know AJDT tries to fit itself to this model without needing all of the functionality ajde provides. It does seem logical therefore to split ajde into the "core" (which is the bare minimum required for an ide which manages it's own views to call the compiler and get information back) and the "ui" (which is the views). 

I'm not sure the ajde.core should be integrated into the org.aspectj.ajdt.core project. It seems a good idea to keep the distinction: ajdt.core project being the compiler, ajde.core being an interface into the compiler for ide's and ajde.ui being the views.

This enhancement can also be split into four distinct areas:

1. How ide's are expected to invoke the compiler (AJDE)
2. How information about the structure model can be obtained (ASM)
3. How the compiler is accessed for eager parsing information
4. How information about the AST can be obtained

Since these are nicely distinct I propose to look at these separately, starting with (1).
Comment 13 Mik Kersten 2006-10-18 17:24:52 EDT 
Agreed.  I wasn't suggesting that any of the IDE-specific stuff move into org.aspectj.ajdt.core, just as you say, only the what's needed to support invoking compilation and getting the right stuff back.  But part of the right stuff is the crosscutting structure, since so many tools that invoke the compiler use that (e.g. IDEs, ajdoc, binary weavers that want to embed crosscutting relations in the JAR, etc).  This is basically the structure that we have already since org.aspectj.ajdt.core already depends on org.aspectj.asm.  So in line with what you're proposing I suggest that as part of (1) you move more of the compiler invocation currently in org.aspectj.ajde into org.aspectj.ajdt.core.
Comment 14 Helen Beeken 2006-10-19 10:27:11 EDT 
Now I see what you mean :-) I hadn't thought of splitting "the right stuff back" even further. As you say, to ajdoc and binary weavers this is just the fact that the structure model has been created (which can then be accessed via the asm interfaces), to AJDT this is the creation of the structure model plus some feedback about the build (was it successful, what type it was etc) and to ajbrowser this is the structure model, feedback about the build plus views showing the structure model. AJDT and Ajbrowser should still only be using ajde (core/ui) but like you say, this should just provide a layer on top of that which ajdoc uses within org,aspectj.ajdt.core.

Comment 38 Helen Beeken 2007-01-16 06:35:50 EST 
A summary of what users need to do to work with the new interface:

For tools who wish to drive the compiler but manage their own ui representation an org.aspectj.ajde.core.AjCompiler instance is needed for each new configuration (project or build configuration). This is created by providing implementations of the other org.aspectj.ajde.core interfaces (IBuildMessageHandler, IBuildProgressMonitor, ICompilerConfiguration and IOutputLocationMaanger) and then either build() or buildFresh() need to be called on the AjCompiler to drive a compile. For an example of an implementation of this see the MultiProjectIncrementalTests class in the aspectj tests project or the AJDT implementation in the org.eclipse.ajdt.core and org.eclispe ajdt.ui plugin projects.

For tools who wish to drive the compiler and do not want to manage the ui representation then org.aspectj.ajde.Ajde needs to be initialised. This is initialised with implementations of the same ui related interfaces (EditorAdapter etc) but also with the new ajde.core interfaces (IBuildMessageHandler, IBuildProgressMonitor, ICompilerConfiguration and IOutputLocationMaanger). There are helper methods within org.aspectj.ajde.Ajde: runBuildInSameThread(..) and runBuildInDifferentThread(..) which should be called when the user has selected to run a build. See the ajbrowser implementation for an example of this usage.
