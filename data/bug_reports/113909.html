Description Pierre Carlson 2005-10-26 18:17:17 EDT 
It would be advantageous to be able to install a JRE (i.e. have it listed in the
Installed JREs list and usable in the IDE) via an extension point.  When
packaging tools and/or runtime environment, we would like to be able to install
the JRE that is associated with our runtime.  Currently, in requires a good
amount of Java code to get the JRE installed programmatically.  Having an
extension point that allows us to give the location of a JRE, the type of JRE
and a name for the JRE would be useful to many tooling and RCP developers.  It
would also be advantageous to add a description for the JRE (to the UI as well).
 Jeff and Wassim suggested that it would also be useful to point to the JRE
profile from the extension point.  The PDE will be using the profile, so being
able to specify this with the JRE would be very useful.

Comment 2 Darin Wright 2005-10-27 12:18:27 EDT 
After speaking with Wassim and Jeff, we are proposing the following:

The Java launching plug-in will allow JRE's to be annotated with 
the "execution environments" they support. Execution environments will be 
extensible (via an extension point). Extensions can define execution 
environments and provide an "analyzer" that is capable of annotating a JRE 
with the execution environments is supports.

For example, PDE might define environments for J2SE1.2 thru 1.5, and provide 
an analyzer classifies VMs based on their Java version.

An exeuction environment will have a unique identifier and a name (the name 
can be externalized, if required).

We intend to add an new optional interface to IVMInstall's supporting the 
introspection of system properties in that VM (will be named IVMInstall3). It 
is our expectation that analyzers will find system properties generally useful 
when annotating VMs with associated execution environments. 

Users specify JREs on build paths and launch configurations. The UI will be 
enhanced to allow users to choose the workspace default JRE, a specific JRE, 
or a JRE that maps to specific execution environment. This information can be 
shared on the buildpath or in a launch configuration.
Comment 3 Darin Wright 2005-12-01 17:54:01 EST 
I've released initial support for execution environments to HEAD.

New interfaces:
* IExecutionEnvironment
* IExecutionEnvironmentAnalyzerDelegate
* IExecutionEnvironmentsManager
* IRuntimeClasspathEntryResolver2 - this interface was added as an extension to IRuntimeClasspathEntryResolver to be able to determine if a classpath entry references a JRE. Otherwise, we have to guess that only JRE_CONTAINER and JRELIB_VARIBALE reference a JRE. For backwards compatability, we just see if a variable or container resovles to a JRE (but this is not precise when a variable or container refers to a JRE that is not in the workspace... thus the additional API) 

New extension point:
* org.eclipse.jdt.launching.executionEnvironments

New helper API methods on JavaRuntime:
* public static IPath newDefaultJREContainerPath()
* public static IPath newJREContainerPath(IVMInstall vm)
* public static IPath newJREContainerPath(String typeId, String name)
* public static IPath newJREContainerPath(IExecutionEnvironment environment)
* public static IVMInstall getVMInstall(IPath jreContainerPath)
* public static String getVMInstallTypeId(IPath jreContainerPath)
* public static String getVMInstallName(IPath jreContainerPath)
* public static String getExecutionEnvironmentId(IPath jreContainerPath)
* public static IRuntimeClasspathEntry computeJREEntry(ILaunchConfiguration configuration)
* public static IRuntimeClasspathEntry computeJREEntry(IJavaProject project)
* public static boolean isVMInstallReference(IRuntimeClasspathEntry entry)
* public static IExecutionEnvironmentsManager getExecutionEnvironmentsManager()

Comment 6 Darin Wright 2005-12-06 17:27:32 EST 
As discussed with Wassim/Jeff, we removed the "name" attribute from the extension. We will only use an 'id' which is the symbolic non-translatable name for an environment.

Changed UI to use "Exeuction Environment" instead of "JRE Profile" terminology.
Comment 7 Darin Wright 2005-12-07 13:16:28 EST

This is a breaking API change that PDE needs to catch up with. See changes to IExecutionEnvironmentAnalyzerDelegate. The method now returns CompatibleEnvironments.
