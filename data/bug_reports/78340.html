Description Jos Dirksen 2004-11-10 16:02:02 EST 
I'm running the latest integration build on eclipse 3.1 M3 (so this might be
caused by this eclipse version). The problem I have is the following:

The XML editor works on files with 'normal' xml extensions, like xml and xslt.
However when I try to open a non-standard xml extension for instance blabla.xmap
or blabla.myextension I get an error:

"Cannot open default editor on blabla.xmap, editor could not be initialized"

That's all of the error I'm shown, I can't find anything in the log files or
anywhere else.

Comment 2 Nitin Dahyabhai 2004-12-06 02:29:18 EST 
This is similar to bug 62429 in that the registration of the XML content type
and the editor association aren't in agreement.  The difference here is that the
XML Editor _must_ have a compatible document to function--that's why we have an
IDocumentFactory registered.  I've opened bug 80237 against Platform Text for this.
Comment 3 David Williams 2004-12-06 03:36:14 EST 
I wonder, just which document we are getting?, since resource should be of XML 
Content Type, and that should take priority in the DocumentFactory and Setup 
look-ups.  
Is it a matter that there are two xml content type registrations? Or that the 
algorightim for deciding document factory is wrong (I know it used to not give 
priority to content type, but thought that was fixed).
Comment 4 David Williams 2004-12-30 21:18:42 EST 
Here's some interesting information from new groups that points out another 
failing test case. 
= = = = = = = = = = = = = = = = = = =
On Thu, 30 Dec 2004 22:47:58 +0000 (UTC), Will Lopez <will70@gmail.com> wrote:

> <?xml version="1.0" encoding="UTF-8"?>
> <project>
> <parent>
> <child>
> <property></property>
> </child>
> </parent>
> </project>

An interesting test case. It is similar to the bug you reference. 
In this case, the "eclipse system" is recognizing this as an "ant build file"
(due to the initial 'project' tag) so does not create the structuredDocument
our editor is expecting. 

We (WTP editors) do have some work to do to allow/enable this kind of use, 
but I'm wondering ... is it really an ant file you are creating? If so, you
could use
the ant editor in the short term. (guess you'd have to "manually" associate your 
file to the ant editor, for now, since another bug 37688 is still not
fixed/enhanced.

Comment 5 Vincent Partington 2005-01-03 05:31:30 EST 
Not all files that have a top-level <project> tag are Ant build files, e.g.
Maven's project.xml and maven.xml files.

I've observed the same problem as Will Lopez: I cannot edit a file with a
top-level <project> tag using the XML editor because of the "assertion failed:
Editing document must be an IStructuredDocument" error. There is no option to
edit it with the Ant build file editor unless I rename it to build.xml.
Strangely enough, at that point the XML editor also works without throwing the
exception.
