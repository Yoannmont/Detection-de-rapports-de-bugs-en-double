Comment 1 James D. Miles 2007-10-17 18:22:59 EDT 
When I tried to test the relative paths in bundles.txt, I could successfully do it when I added the -bp "<destination>" argument. It is placed bundles.txt in the eclipseApp/configuration/org.eclipse.equinox.simpleconfigurator directory as expected and it is relative. However if I add the "-configuration c:/tmp/workspace/.config -data c:/tmp/workspace" arguments bundles.txt is still placed in the eclipseApp/configuration/org.eclipse.equinox.simpleconfigurator directory. config.ini is also placed in eclipseApp/configuration. All other configuration data flows the configuration location.

This is a problem. In the case of a multiuser scenario we need to be able to configure this file for each user. bundles.txt stored in the eclipseApp tree is of limited use even if nested. Loading all of the bundles specified in this file may not be what each user config requires or wants. While it may be useful to share this file it should only be an option. The location for platform.xml in the current eclipse does not behave this way.
Comment 2 Pascal Rapicault 2007-10-18 10:07:47 EDT 
I'm wondering if there is not a confusion around the meaning of -configuration.
-configuration does *not* control the path of the configuration folder of the provisioned application. -configuration is a runtime option read by OSGi (see http://help.eclipse.org/help33/index.jsp?topic=/org.eclipse.platform.doc.isv/reference/misc/runtime-options.html)

The parameter controlling the place where p2 will write the configuration folder of the provisioned application is set by -destination.

Comment 4 Pascal Rapicault 2007-10-18 12:53:38 EDT 
-destination controls the folder where the launch point will be located. As you mentioned this folder is used as the root to store eclipse.exe, eclipse.ini, config.ini and the bundles.txt.

There is currently no argument to control the location of the configuration folder and it may be needed.

-bp controls the location of bundle pool.

There is no document about the parameters supported by the director app. However they are all pretty clearly called out in the Application class of o.e.e.p2.director.app. Would you mind putting the description of the arguments on the wiki. This would likely be useful to more than one person.

Could you detail the setup you are trying to create?
