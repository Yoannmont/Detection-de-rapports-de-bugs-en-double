Description David Mechner 2003-12-17 15:05:11 EST 
When calling setText() frequently, the cells intermittently flicker when they
are redrawn (even if there are no structural changes in the table).

Steve Northover reports: 
> I took a quick look at it and the problem is in Windows.  I tried to work
> around it and failed (for now).  Please enter a problem report. 

Sample program showing the problem below. The blue row background makes the
flicker more noticable, but it is there even without the colored background
(mostly noticable as the table lines flickering). 

==

import java.text.DecimalFormat;
import java.util.Random;

import org.eclipse.swt.*;
import org.eclipse.swt.layout.*;
import org.eclipse.swt.widgets.*;

public class TableFlicker {

	Table _table;

	TableItem[] _rows;

	Control createContents(Composite parent) {
		Composite panel = new Composite(parent, SWT.NONE);
		FillLayout fill = new FillLayout();
		panel.setLayout(fill);

		_table = new Table(panel, SWT.BORDER);
		_table.setLinesVisible(true);
		_table.setHeaderVisible(true);

		TableColumn a = new TableColumn(_table, SWT.NONE);
		a.setText("Col A");
		a.setWidth(80);
		TableColumn b = new TableColumn(_table, SWT.NONE);
		b.setText("Col B");
		b.setWidth(80);

		_rows = new TableItem[4];
		for (int i = 0; i < 4; i++) {
			_rows[i] = new TableItem(_table, SWT.NONE);
			_rows[i].setBackground(
				_table.getDisplay().getSystemColor(SWT.COLOR_DARK_BLUE));
			_rows[i].setForeground(
				_table.getDisplay().getSystemColor(SWT.COLOR_WHITE));
		}

		_table.pack();

		new Setter().start();
		new Setter().start();

		return panel;
	}

	public static void main(String[] args) {
		TableFlicker test = new TableFlicker();
		Display display = new Display();
		Shell shell = new Shell(display);
		shell.setLayout(new FillLayout());
		test.createContents(shell);
		shell.pack();
		shell.open();
		while (!shell.isDisposed()) {
			if (!display.readAndDispatch())
				display.sleep();
		}
		display.dispose();
		System.exit(0);
	}

	public class Setter extends Thread {
		Random rand = new Random();
		DecimalFormat df = new DecimalFormat("#,##0.00");
		public void run() {
			while (true) {
				try {
					Thread.sleep((long) (rand.nextDouble() * 300));
				} catch (InterruptedException e) {}
				final int row = rand.nextInt(4);
				final int col = rand.nextInt(2);
				if (_table != null && !_table.isDisposed()) {
					_table.getDisplay().syncExec(new Runnable() {
						public void run() {
							if (!_table.isDisposed()) {
								_rows[row].setText(
									col,
									df.format(rand.nextDouble() * 50.0));
							}
						}
					});
				}
			}
		}
	}

}

Comment 2 David Mechner 2004-02-18 19:11:26 EST 
I've learned that there is a fix for this problem. A friend of mine ran into the
problem with the native windows widget, and solved it. He writes: 

> The way around it in native Win32 is to subclass the table widget 
> and do a double-buffered rendering (calling the super-class' paint 
> method, but supplying it with alternating graphic contexts).

Eclipse could do this, implement the swt.widgets.Table widget by wrapping the
"fixed" native table widget, and distribute it in the SWT dll, hey presto. 

Is this something you would be willing to do? If it will help, I can provide the
required Win32 source code. 

I don't see a lot of "CC"s in this bug report, but it's a very important issue
for anyone working with a dynamically updating table - it gives the whole
application a feeling of flakiness, and as much as I love SWT, it alone is
pushing me to consider an alternative. It may be that people who see this
flicker just assume Java or SWT is flakey, and don't bother looking into it
deeply enough to realize the real source of the problem.

Please let me know. 
Thanks.
Comment 3 Steve Northover 2004-02-18 20:05:49 EST 
The Table widget on every platform is currently in the process of being 
rewritten to support virtual tables.  Once the code has been released, we can 
look into the flicker again.  You should all yourself to 37998, 15054 and 
36226 watch the progress.  I wish I could offer more at this time.

As for the double buffer solution ... this is really wild and not something we 
would ever do lightly.  Basically, crazy hacks like this generally burn you 
badly later on for some reason that you could never have determined at the 
time of the hack.  You are welcome to prototype a solution.  SWT is written in 
Java.  Go find Table.WM_PAINT() and hack it but we warned that we will be 
pretty suspicious of the code.

BTW: The flicker is pretty minor on my machine.  I am running Windows XP with 
the manifest.  Could be it is way worse on 2000?
