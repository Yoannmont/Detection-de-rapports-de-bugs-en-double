Description Seth Packham 2007-07-31 09:58:47 EDT 
With Help contributions coming from many different groups, and contributors using different style sheets, a large help system can start looking a bit hodge-podge. 

If we could optionally specify a universal style sheet for a product, it could help a product to bring a consistency across all the help topics. This style sheet could be defined in the plugin_customization.ini.
Comment 1 Debbie 2007-08-01 03:07:24 EDT 
Please make sure to emphasize that it is an "optional setting" so that people who want to customize their help will still be able to without too many complications.
Comment 2 Seth Packham 2007-08-01 09:30:19 EDT 
Yes, this would be optional at the product level for an offering that wants to try to control the look of all help contributions that are added to its help system. 

A help plug-in contributor could still customize how their Help topics look, but a product offering that consumes and ships this help plug-in to customers could try to override the plug-in's styles with this universal style sheet applied by the system at runtime.
Comment 3 Curtis d'Entremont 2007-08-01 10:38:12 EDT 
This is currently faked in eclipse by having the docs pointing at the current running plug-in's book.css file. However it requires that all plug-ins conform to this rule.

Interestingly, this global css mechanism does exist for the help view's narrow_book.css. Rather than being specified in the plugin_customizations, however, it is a special filename (narrow_book.css) that it looks for, similarly to plugin.xml. I believe we should have an equivalent for normal topics.

I will let Chris comment when he returns.

I'm assuming in this situation that the overall.css would override the ourstyle.css, right?

Is it difficult for the system to do this? I mean, I thought that the way the cascading aspect works is the CSS that's coded inside the file "trumps" all of the other previous ones. (But I might have that backwards--please correct me if I'm wrong).

Comment 5 Curtis d'Entremont 2007-08-01 16:00:24 EDT 
If two CSS rules contradict, I believe the one that appears below or after the other one wins. So we could control it somewhat by injecting the global css before or after all other css's, although we can't override everything because there may be some inline style attributes.

I agree with having the global one take precedence over the local one.
Comment 6 Chris Goldthorpe 2007-08-06 14:58:55 EDT 
Are you looking to enforce the same style on all books including books contributed from third parties or just from your organization? 

Allowing for a common CSS seems like a reasonable idea but there is the possibility that some pages will not display as well if you impose a different CSS on them. 

If you are mainly interested in achieving consistency across a set of products over which you have influence you may want to encourage them to follow the Eclipse documentation convention. Most of the eclipse documentation already gets its CSS indirectly from the product, for example org.eclipse.jdt.doc.user/book.css contains this line

@import "../PRODUCT_PLUGIN/book.css" 

as well as a number of additional declarations.

I have mixed feelings about having the global css override the local css, this is contrary to the spirit of css which says that the more specific rules take precedence over global rules.

It may be worthwhile prototyping a change along the lines suggested to see how well it works. I don't think that it would be hard to create a patch which could be tested.
Comment 7 Seth Packham 2007-08-06 16:02:05 EDT 
Yes, we want a "product" to be able to point to a CSS that would be last in the list of style sheets, giving it the ability to define more or less specific rules for *all* contributions to the help system. 

This would allow a product to give all headings a certain color and size, for example. Granted, a third party's <h1> could have a class attribute on it, and they could define a more specific rule for their H1s, but this global CSS could still override it by defining a matching rule for that h1.class value.

I didn't know that your Eclipse docs import a book.css from the product plug-in, so we could add a book.css to our product plug-in to control the look of Eclipse docs, but we have dozens of product plug-ins where we would need to maintain a synchronized version of the book.css file. And this would still only affect docs that do this import.

Rather, if each product plug-in could define an "ultimate and final" CSS to consider, a product could potentially create a sense of consistency across the help topics, which is crucial to good branding.

This does not go against the spirit of CSS. It is allowing a help system owner to add another CSS to the end of the list of all style sheets that any topic links to. The behavior would still adhere to the cascading, specificity, and inheritance rules of style sheets.
Comment 8 Lee Anne Kowalski 2007-08-07 12:16:54 EDT 
So, is the concept of "product" actually codified in some way in the startup process?

I'm thinking that if the rules of the scenarios are clear, then "product owners" and "third party plugin owners" will be clear and easily explain what they 'see' when they view their information under the different scenarios.

For example, if the rule of engagement is spelled out something like "When a Product, a product is instantiated under conditions A, B, C; and the Product Owner can then set an over-riding global CSS that is put on all doc plug-ins. Those of you who are Plug-in Contributors to that Product cede your expectation that your CSS will apply to your doc plug-ins. As part of the agreement to be a part of that Product, you give up the expectation for your own styles under those ABC conditions."

In a sense, it's not terribly different than what the dynamic HTML features like insert allow--I might have a topic with an anchor that Seth then injects a new paragraph into. Haven't I given up the expectation that a new paragraph won't be injected into my topic at that point?

Maybe we could borrow that conceptual pattern: if I'm a doc plug-in owner and I am willing to give up my expectation that my topics will always look a certain way, there's some flag that I put at the top of my XHTML that says "yes, I allow a product to apply a global CSS to my topics"--just like I have to include an anchor point in my XHTML if I'm going to allow someone else to insert a new paragraph or something into my topic.

--Lee Anne Kowalski

Comment 10 Seth Packham 2007-08-07 12:50:38 EDT 
Good thoughts. As you mentioned, Eclipse already allows the "product" to have master control over the help (e.g. filters and inserts) and Welcome (themes and welcome layout control). Furthermore, a product can even ignore an entire help TOC or display it in a different place, essentially hiding it deep in the TOC hierarchy. This does not currently require that the contributor and the product have an explicit agreement, or that the contributor cede control. Today, if my product didn't like your 3rd party help contribution, I could anticipate my users installing it, and we could ignore it in the Help. This global CSS is just an extension of that. If your help contribution "looks" bad, or different, I can make it look like the rest of my product's help, if I want.
Comment 11 Chris Goldthorpe 2007-08-07 13:01:21 EDT 
I think that we want to keep the rules as simple as possible so I don't think
we need an additional flag to say which CSS appears first. If a plugin

be used.

I think we are in agreement that this feature would be useful, at this stage probably the best way to find out exactly what form it needs to take would be to prototype a version which allowed for one CSS file from the product to be included at the end of the <HEAD> and make a patch available for testing.
Comment 12 Chris Goldthorpe 2007-09-10 14:41:14 EDT 
Based on the demand for this feature I'm targeting Eclipse 3.4. Here is a brief description of how CSS works now and a discussion of how the new feature would work.

The documentation for the Eclipse platform contains hard coded references to a css file in evey help topic, they have an entry which looks like this:

<LINK REL="STYLESHEET" HREF="../book.css" CHARSET="ISO-8859-1" TYPE="text/css">
If you don't specify style sheet in your help topic your help page will be unstyled when it displays in the Webapp.

In addition if you display a help topic in the help view we will insert "narrow_book.css" and {$os}narrow_books.css, if you open a disabled topic from a search we insert "disabled_book.css". All of these come from the product plugin.

Because the product already controls the css for narrow and disabled books I think the only thing we need to add is to allow the user to optionally include their own equivalent of "book.css" to every document, which would be controlled by a single option in org.eclipse.help.base/preferences.ini. We would still continue to insert narrow_book.css and disabled book.css where appropriate. The user defined css file will be inserted before the narrow or disabled version.

Is this going to satisfy the requirements of the requestors of this feature?
Comment 13 Seth Packham 2007-09-10 15:33:18 EDT 
Sorry, but I'm a bit confused by your description of the new implementation, so here's how I think it should go down:

1. A product owner would add a new entry to their plugin_customization.ini that points to their product's universal CSS file, wherever it is.
2. The Eclipse help system checks plugin_customization.ini for the universal CSS entry.
3. If entry exists, Eclipse help dynamically inserts a link to this CSS (in the last position possible) in every topic in the help system.

Adding an entry to any file other than the plugin_customization.ini would be non-intuitive. This is a product setting and should be controlled in the product plug-in.

This should apply to Web app and Help view.
Comment 14 Chris Goldthorpe 2007-09-10 15:57:31 EDT 
Yes, you would be able to use plugin_customization.ini to change the preference, the default value is stored in org.eclipse.help.base/preferences.ini. The default will be an empty string, causing the default 3.4 behavior to be the same as 3.3.

We still need to decide what to do for narrow books when the user sets the css. I think that at least some users will find that they want to apply different CSS to narrow books, and including narrow_book.css from the product gives the maximum flexibility, you can always leave that book empty.

Comment 18 Seth Packham 2007-10-23 15:21:14 EDT 
Hi Chris,

I know you delivered this feature, but I've been thinking more about this requirement and its best implementation. At its core, it is about being able to tell the system to add content to every rendered page at runtime. This is valuable not just for universal CSS links, but also for injecting javascripts, universal headers and footers, extra <div>s in body, etc. for cool Ajax stuff, like linking to an external tagging or commenting system.

So, if possible, I'd really like to explore a better implementation of this feature:

Allow a product to define three files with XHTML snippets for injection:

1) A file with snippets to be injected just before </head>
2) A file with snippets to be injected just after <body>
3) A file with snippets to be injected just before </body>

So, a plugin_customization.ini, might look like this:

org.eclipse.help.base/snippet_head=/com.ibm.myplugin.doc/snippet_head.html
org.eclipse.help.base/snippet_body_start=/com.ibm.myproduct.doc/snippet_body_start.html
org.eclipse.help.base/snippet_body_end=/com.ibm.myproduct.doc/snippet_body_end.html

I appreciate your comments on this.

Comment 21 Dave Resch 2007-10-23 20:42:30 EDT 
(In reply to comment #18)
Seth,

This is an interesting idea, XHTML snippets, but it is substantially different from the original CSS override proposal, perhaps deserving a separate bug for due consideration (as Chris just suggested).

Throwing in my 0.02USD... compared to CSS overrides, I think this might introduce a higher risk of things going wrong, as well as a risk of things going more wrong.

For example, if my product inserts XHTML snippets into topics contributed by a 3rd-party plug-in, and those topics already contain running headers/footers and <script> elements, the result could be ugly (e.g., two stacked running headers/footers) and/or indeterminate (weird JavaScript/Ajax results or errors). 

BTW, if you build Eclipse content with DITA-OT, the OT provides properties that you can use to pass in such snippets (args.hdf, args.hdr, and args.ftr) to be embedded in each XHTML output file.
Comment 22 Seth Packham 2007-10-24 14:06:30 EDT 
I'll respond here before opening a new bug report.

Yes, this feature would allow the "product" owner to have more control at the help system level over every page that is displayed in the contents pane.

I don't think that the danger of potential misuse or bad results is a good argument for  not doing this. It is no more dangerous than ignoredTocs, which allows me to totally ignore a 3rd party contribution. And, any use of any feature must be tested and designed to work well in a maximum number of use cases. Any plug-in developer can mess lots of things up -- they just try not to.

In a massively complex product, where contributions come from all over, using strictly build-time insertions is not always feasible, even though we are using DITA.

Allowing the system to insert content in the <head> and at the beginning and end of <body> would give a lot of flexibility for infocenters. I can go on and on with possible cool use cases if the system had this kind of oversight. I believe the system already dynamically inserts links to livehelp.js and breadcrumbs.css. I'm just asking to add some more snippets of my own at runtime, system-wide.
