Description Donal Riordan 2005-10-05 11:08:31 EDT 
As part of our code we need to serve files over HTTP to a remote machine. 
Currently we use the org.eclipse.help.appserver plugin to do this but this code 
is eclipse internal. We only use two classes:
org.eclipse.help.internal.appserver.AppserverPlugin
org.eclipse.help.internal.appserver.IWebappServer

Is it possible to have these externalised into official API.

Comment 3 Curtis d'Entremont 2005-12-20 14:09:38 EST 
Please see the OSGi R4 Compendium Spec section on the HTTP service. Does this spec meet your needs? This spec is being implemented in the new org.eclipse.equinox.http bundle at http://eclipse.org/equinox/bundles
Comment 4 Hasan Ceylan 2006-01-01 19:16:54 EST 
I have briefly looked at the referenced bundle, and it does not look like 
a tomcat container but some sort of static page server. Please correct if I am wrong. But if not, that would not suit my need. I think appserver pluging should
be factored out of help set. It is very mature and very well implemented. 

Would be waste if only used for serving the help pages.

Regards,
Hasan Ceylan
Comment 5 Curtis d'Entremont 2006-01-02 10:46:48 EST 
The HTTP service supports both static content and servlets, so you can do dynamic content as well. It is based on the Java Servlet API v2.1.

The service API is not tied to any particular servlet container implementation, however I believe the initial implementation will use Tomcat, although this could potentially change in the future (the API would remain the same though).

The reason I am quite reluctant to open this is that we would be duplicating the effort of the HTTP service. However, if there is something you cannot do with the HTTP service that you would have been able to do had we opened the appserver, then we have to deal with this. This is why I was asking if the HTTP service is suitable for your needs.
Comment 6 Hasan Ceylan 2006-01-02 10:56:49 EST 
OK, if it is some sort of servlet container then that is a difference. To give you a little bit more hint, I am embedding birt viewer into eclipse for an enterprise application that will run based on the RCP plugin set. 

Birt can be deployed any tomcat based server. I think I can wait some time to see if the new osgi bundle will get along well with the birt viewer. 

I view that you do not like to duplicate the code as a fair argument. Then I propose to wait and see what will come up from the new osgi bundle and see if it will support birt viewer, if not we may ask birt guys to make the viewer compatible with the bundle if the main idea is to centralize this piece of code as general purpose web/servlet server/container. 

Till then I will be fine with some warnings in the ide. Since it is not restricted but simply discouraged to be accessed, it is no problem. But I will kindly ask you to keep it that way, until the above case comes to a maturity one way or another...

Thanks,
H.Ceylan

Thanks for the time.
