Description Michael Garvin 2003-10-06 11:39:54 EDT 
Hi we have never been able to get any version of Eclipse to cut/paste the X11 
selection on Linux (Motif) or HP-UX 11 (Motif).  I searched through the news 
group, and it looks like this feature has already been added to Eclipse.  
Unfortunately it is not working for us, and makes using Eclipse very awkward.  
Is there a work around to write some custom Java code to get Eclipse to work 
with the X11 selection when running on Linux or HP-UX?  

I have already tried the ClipboardTest.java code and it is broken in the same 
way as Eclipse.  However, it does actually past out of itself to the X 
selection but it will not copy anything from the X selection into itself.  I'll 
attach that test code as soon as I save this initial description of the bug.  I 
will also try to reproduce the bug with this test code on HP-UX, I'm expecting 
it to easily be reproduced since cutting/pasting has never worked for us at all.

In the test program when I click on "Available Types" I get this list:

text/uri-list
COMPOUND_TEXT
STRING
text/rtf
TEXT/RTF
application/rtf

As I say I can use the copy/past buttons in the test harness to copy material 
out to the X selection and into other X apps (such as an xterm), but I can not 
copy things from the X selection into this test harness.  When I use the CNTRL-
V keys, then the previously copied out text is pasted into the test harness 
despite the fact that there is a new X selection in another X application.  The 
middle mouse button is is broken in the same way, it works going out of the 
test harness to X, but not coming from X to the test harness.

We are using Eclipse (Motif) 2.1.1 on Linux.  Same version of HP-UX.  

Please let me know if there are any known work-arounds for this.  We use 
Eclipse as the basis for an IDE that we ship to our internal designers who have 
all worked in an X11 environment for years, their expectation is that 
cutting/pasting to/from our IDE (Eclipse) will just naturally be supported.  If 
its not supported then thats a big usability issue for us.

Thanks,

(^_^)/
mike.

Comment 3 Michael Garvin 2003-10-06 14:40:39 EDT 
I found some code that allows me to fetch the X11 selection on demand:

  import java.awt.Toolkit;
  import java.awt.datatransfer.Clipboard;
  import java.awt.datatransfer.Transferable;
  import java.awt.datatransfer.DataFlavor;
  ...

  addMouseListener(new MouseListener () {

    public void mouseDoubleClick(MouseEvent e) {
    }

    public void mouseUp(MouseEvent e) {
    } 

    public void mouseDown(MouseEvent e) {

      System.out.println("Mouse Button Press on #" + String.valueOf(e.button));

      if(e.button==2) {

        // check for the X11 selection...

        System.out.println("Checking for X11 selection...");

        java.awt.datatransfer.Clipboard cb = java.awt.Toolkit.getDefaultToolkit
().getSystemSelection();

        if(cb==null) {

          System.out.println("Could not get system clipboard");

        } else {

          java.awt.datatransfer.Transferable tfb = cb.getContents(null);

          // make sure we can show the selection as a string...

          if(!tfb.isDataFlavorSupported(DataFlavor.stringFlavor)) {

            System.out.println("Current X11 selection can not be formatted as 
string...");

          } else {

            // format the X11 selection...

            String data = "";
  
            try {

              data = (String)tfb.getTransferData(DataFlavor.stringFlavor);

            } catch (Throwable badflav) {

              System.out.println("Could not convert X11 seleciton to a 
string.");

            }

            if(data.compareTo("")!=0) {

              System.out.println("X11 selection is: " + data);

            }
          }
        }
      }

      System.out.println("done mouse proc");
    }
  });


This code seems to work as a way to fake out the middle button select, but it 
isn't very stable.  When I run this inside the test harness, the test harness 
GUI freezes up just after the mouse listener finishes running.  If it freezes 
up inthe test harness, I'm worried that it will have negative side effects on 
the Eclipse Workbench.  Is there a work around along these lines that is known 
to be stable, that I could use?  Also I found that occasionally it seems to 
crash the GUI an spits an X11 error message about not having enough resources 
to complete the request?!?

X Error of failed request:  BadAlloc (insufficient resources for operation)
  Major opcode of failed request:  86 (X_AllocColorCells)
  Serial number of failed request:  165
  Current serial number in output stream:  165

I'm running everything through Exceed, so maybe there is an Exceed interaction 
of some kind, or Exceed doesn't support standard selection operations in X11??
