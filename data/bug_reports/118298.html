Comment 1 Ed Merks 2005-11-29 07:45:19 EST 
James,

Given that it's dying here:

  public List computeMissingUsedGenPackages()
  {
    List missingEPackages = getMissingPackages();
    if (!missingEPackages.isEmpty())
    {
      List allGenModels = new UniqueEList.FastCompare();
      allGenModels.add(this);
      for (int i = 0; i < allGenModels.size(); i++)
      {
        GenModel genModel = (GenModel)allGenModels.get(i);
        for (int j = 0; j < /*here*/ genModel.getUsedGenPackages().size(); j++)
        {
          GenPackage genPackage = (GenPackage)genModel.getUsedGenPackages().get(j);
          allGenModels.add(genPackage.getGenModel());
        }
      }

My guess would be that links to other used GenPackages are broken.  I've added a null check guard to avoid the failure, although without a test case to reproduce the problem I can't know that the problem won't just move to somewhere else.
 
If necessary, you can create a new .genmodel to wrap your .ecores by using "File->New->Other...->Eclipse Modeling Framework->EMF Model". If you want to figure out which package is causing the problem, looking at the proxy URI of the genPackage for which getGenModel() returns null will indicate that.

Comment 3 James Moody 2005-11-30 12:22:52 EST 
I've debugged this, and as you suspected it was a missing genmodel file (since the references are all hardcoded file paths and unless you have the files in exactly the same place as the original author, you're kind of screwed).

The *real* problem here is that the EMF codegen knows exactly what the problem is, and doesn't tell me. This is really really frustrating and has cost me a lot of time on more than one occasion.

When encountering this problem, instead of simply hiding the NPE, the codegen must inform the user of the missing file and where it is referenced from. This information is readily accessible to the codegen. Hiding the problem will simply cause things to break later and the user still has no idea why.
