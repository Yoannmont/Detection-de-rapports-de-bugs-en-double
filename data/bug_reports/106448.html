Comment 1 Tim Austin 2005-10-28 02:39:26 EDT 
A workaround to this issue is to select the ‘Use external browser’ checkbox in 
the 'Help' panel of the 'Preferences' window.  By default this property is 
set 'False'.  Setting to 'True' means that the helpset is viewed using external 
browser, rather than viewer.  The browser is not blocked by modal dialogs.  All 
features of viewer, such as search, tabs, navigation options, appear to be 
available in browser, meaning that help features are not impaired by using 
browser instead of viewer.  Please note, this issue is also reported in posting 
107255.

Comment 19 Brian de Alwis 2013-12-19 10:14:40 EST 
I took a quick look at this, and there's nothing much that can be done.  WizardDialog is APPLICATION_MODAL, and since the internal help browser is a normal shell within the app, so it should be blocked.  The patch alluded to in comment #6 is very specific to Windows and won’t work on other systems:  basically the patch creates a second SWT Display instance and uses that for the help browser.  Creating multiple Display instances is just not possible on the other systems.

The only real solution that I can see is to cause the help to run in the external browser.  You should be able to force this by setting

	org.eclipse.help.base/always_external_browser = true

in plugin_customization.ini.
