Description Ed Burnette 2005-02-21 00:03:45 EST 
3.1M5a
I created the RCP Mail application from the template, created a .product file
for it, and exported it into a zip file, and unpacked the zip file. The
generated config.ini file (d:\Deploy\RcpMail\configuration\config.ini) looked
like this:

#Eclipse Runtime Configuration File
#Sun Feb 20 22:36:58 EST 2005
osgi.bundles=RcpMail,org.eclipse.core.commands,org.eclipse.core.expressions,org.eclipse.core.runtime@2\:start,org.eclipse.help,org.eclipse.jface,org.eclipse.swt,org.eclipse.swt.win32,org.eclipse.ui,org.eclipse.ui.workbench
osgi.bundles.defaultStartLevel=4
eclipse.product=RcpMail.product
osgi.splashPath=platform\:/base/plugins/RcpMail

Then, following the instructions in bug 54919 comment #1 and other places, I
added these lines to the end:

# Put the writable areas in the user's home directory
osgi.instance.area.default = @user.home/RcpMail/workspace
osgi.configuration.area = @user.home/RcpMail/configuration

Then I ran the app by double-clicking the launcher executable, closed it, and
examined both the configuration directory in the install directory
(D:\Deploy\RcpMail\configuration), and the one in the user home directory
(C:\Documents and Settings\userid\RcpMail\configuration). The RCP app had
created new subdirectories and had written files in *both* places, namely a
directory called org.eclipse.osgi and one called org.eclipse.core.runtime.

In the install configuration directory, under org.eclipse.osgi it created a
.manager directory with a .fileTable file. Under the runtime directory it
created .manager/.fileTable and several files like .contributions.1 and .mainData.1.

In the user configuration directory, under org.eclipse.osgi it created several
subdirectories and files. Under the runtime directory it created a .manager
directory with a .fileTableLock file.

I tried some variations in the config.ini including specifying: 

osgi.configuration.cascaded = false

(or true) and specifying:

osgi.sharedConfiguration.area=@user.home/RcpMail/configuration

But could not keep the application from writing into the install directory. Each
time I deleted all the configuration files and directories in the install
directory and the user directory except for config.ini to start with a clean slate.

If there's something else I need to specify I don't know what it is, please
enlighten me.

Comment 4 Rafael Chaves 2005-03-02 16:41:11 EST 
In several places where we want to check the parent configuration, we create and
initialize a FileManager on the parent configuration location even if there is
no relevant metadata there. Creating and opening a FileManager causes changes to
the file system. The fix I propose (affects org.eclipse.osgi and
org.eclipse.core.runtime) is to check if the base location exists before
creating a file manager. The ideal fix would be to be able to open a file
manager in read-only mode (see bug 87033).

Tom, please review the patches (will attach shortly).

Comment 7 Thomas Watson 2005-03-02 18:02:32 EST 
The patches have the following issue:

If the parent configuration directory does exist then you will end up writing 
to it to create the lock and possible the .filetable and instance file.  This 
is the case if the parent configuration is configured and the client has 
FilePermissions to write to the parent.

I think we should fix this bug by implementing a read-only type FileManager as 
suggested in bug 87033.

Comment 11 Rafael Chaves 2005-03-16 10:07:46 EST

Comment 12 Thomas Watson 2005-03-16 10:19:07 EST 
Yes if the state file exists but the lazy file does not then we should null 
out the state file.  These files must exist in pairs.  If we find only a state 
file then we should ingore the content and regenerate the state.
Comment 13 Rafael Chaves 2005-03-16 10:51:31 EST

write).
