Steps To Reproduce:
1. create a new WSDL with skeleton, call it "MyWSDL.wsdl"
2. rename your WSDL file by clicking F2 in package explorer and type "MyWSDL_2.wsdl"
3. try to rename the port type from "MyWSDL" to "MyWSDL_2" using the graphical editor

More information:
You have to close the editor and reopen on the "new" file MyWSDL_2.wsdl.

This patch is just a draft, but it should improve how the wsdl editor reacts after a save as or a rename.

When the editor does a save as or when a rename is done, the wsdl editor will try to update its definition model with the new document.

I say this patch is just a draft because there's still something not quite right with how the wsdl model is updating itself with the new dom document.  The associated schema model seems to get lost somewhere and then NPE's are thrown when you try to set binding or port type after you save as/rename.
Comment 8 Amy Wu 2008-05-13 18:29:43 EDT 
Created attachment 100073 [details]
org.eclipse.wst.wsd.ui.txt

Here is an updated patch.
In WSDLModelAdapter, there's a new method, clear() that does cleanup associated with the model adapter (removing listeners, adapters, etc)

In InternalWSDLMultiPageEditor, buildModel() & dispose()
Changed it so that instead of holding onto the modelAdapter & structuredModel the entire time the editor is up, only the Definition object is saved in the editor.

Added doSaveAs, propertyChanged, and setInputToGraphicalViewer

doSaveAs & propertyChanged do pretty much the same thing.  These methods are called when the file in the editor are being saved as another file name or when the file is being renamed.  So in these methods, we check if we have to resync our models, if so, we do and then we dispose of the old ones.

Both methods call setInputToGraphicalViewer (doSaveAs does so indirectly by calling setInput)  setInputToGraphicalViewer creates a new wsdl model for the editor to use and updates the graphical & outline viewer with this new model.

Comment 12 Amy Wu 2008-05-13 20:34:48 EDT 
* Explain why you believe this is a stop-ship defect. Or, if it is a "hotbug" (requested by an adopter) please document it as such. 
This is a usability issue. It looks bad for the WSDL editor to stop working after you rename the file. Especially if half the editor works (source page) but the other half doesnt (design page)

* Is there a work-around? If so, why do you believe the work-around is insufficient?
User can close and reopen the file after they rename the wsdl file to work around the problem.  But they really shouldn't be forced to have to do this.

* How has the fix been tested? Is there a test case attached to the bugzilla record? Has a JUnit Test been added? 
Fix has been manually tested by me, Keith Chong, and Valentin Baciu.

* Give a brief technical overview. Who has reviewed this fix?
Basically, when the wsdl file in editor is renamed or a save as is performed, a new dom model is created. But the wsdl editor is not aware of this and it continues to try to use the old model.  The fix is to make sure the wsdl editor is aware of when the dom model changes underneath it and update itself with the new model.  (Code changes are explained in more detail in comment #8)
Fix has been reviewed by me. And soon to be reviewed by Keith & Valentin.

* What is the risk associated with this fix? 
Fix is low risk. The code reacts to a save as or if the input of the wsdl editor has changed, and even then, it makes sure the model is changing before disposing of the old one and refreshing with the new one.
Comment 13 Amy Wu 2008-05-14 16:10:57 EDT 
Created attachment 100291 [details]
org.eclipse.wst.wsd.ui.txt

This updated patch has a bit of code cleanup, moving the wsdl model creation code into one method, eliminating duplicate code.

Also, in the previous patch, I had it so that
when the editor needs to be refresh, it was only specifically refreshing the
design page by calling setInputToGraphicalViewer().  This new patch changes it
so that rather than directly refresh only the design page, setInput() is
called.  Calling setInput will ensure that the multipage editor's editorInput
is set to the right value as well as calling setInputToGraphicalViewer().
