Description Pascal Rapicault 2007-09-13 13:20:19 EDT 
The current design of the content of the touchpoint data is such that it can only refer to "actions" contributed by the touchpoint whose type is referred to on the IU containing the data.
This is problematic since it forces some actions (e.g. unzip, chmod) to be defined in multiple touchpoints.

Instead we are thinking of removing this limitation and have the following:
- touchpoint data provides actions
- each IU has a primary touchpoint type
- touchpoint data section of an IU can refer to any action provided by any touchpoint.

Comment 12 Simon Kaegi 2008-09-29 11:50:10 EDT 
I've added an initial cut into HEAD along with some simple tests. This is a relatively small changes and extends the action lookup done in the InstructionParser to do an additional lookup iff the touchpoint can't find the action. This new lookup uses an "actions" extension but does not take things as far as Helmut's patch in terms of integration with the existing actions.

Longer term I'd like to remove the touchpoint.getAction call and just use the extensions however until we sort out some of the issues around initalizing a touchpoint partway through an Engine operation we'll continue to use the touchpoint to do an initial lookup to prevent any breakage. With that said, I think looking at how to handle some of the touchpoint interaction is the next step.
Comment 13 Simon Kaegi 2008-10-17 14:05:42 EDT 
I've merged the engine_changes_3_5 into HEAD now and have done some additional coverage testing. More details are in item (1) here -- http://wiki.eclipse.org/Equinox/p2/Engine/Galileo

Pascal did a review with me yesterday and I wanted to capture some of the ideas that came out, that are not in HEAD so that they don't get lost. Eventually bugs should be opened for these as we get a chance to figure out when and if they fit in the plan.

cleanup
------
refactor redundant actions
look at getting rid of the NullTouchpoint and the NullAction
Both touchpointmanager and actionmanager should be injected into the Phase instead of using a singleton lookup


Removing IU touchpoint relationship
-----------------------------------
Review what happends when IUs don't have an associated touchpoint type (This will impact collect/sizing/checktrust phases)


Other
-----
Do we want a java like model for the action qualification (e.g. an import attribute of "org.eclipse.equinox.p2.touchpoints.eclipse.*)
Look at ensuring uniqueness of the touchpoint instance being created in the context of one provisioning operation (e.g. we don't want interference between two different versions of the same touchpoint)
Consider reifying the TouchpointInstruction import attribute into more than just a String
