Description Samantha Chan 2007-12-12 14:22:35 EST 
3.4 20071211_0010

Testcase:  org.eclipse.debug.manaultesting/stress/TestManyTest

This testcase creates 200 threads.  Each of the threads sleeps a random duration of time.  This is to simulate having threads come and go randomly.

1.  Set breakpoint at line 29.  the breakpoint causes the testcase to stop at thread 122.
2.  When stopped at breakpoint, hold down the F6 key to step.

After stepping a while, you will notice that the thread can be collapsed unexpectedly.  The user has to manually expand the thread and select the stackframe in order to continue stepping.  This seems to happen more often if threads above the suspended thread are being removed from the viewer.  

Sometimes the thread remains expanded, but the selection to the stackframe is lost.

Comment 4 Darin Wright 2008-10-31 16:07:48 EDT 
I discovered the reason this happens:

* A thread terminates, and fires a terminate event
* The model proxy (thread event handler) creates a delta for the event to update the veiwer. The delta does not contain the index of the terminated thread (since the thread has terminated and is no longer a child of the debug target, the model proxy cannot infer the index of the thread)
* The viewer attempts to find the element in the viewer if the index is not supplied - but if the thread being removed has not yet been realized in the virtual viewer, an unrealized item is removed in its place (to keep the child count correct), and we effectively have to update children after that index to ensure the viewer remains consistent with the model.

This refresh can cause elements to be re-positioned in the viewer, so the expanded thread ends up moving to a different item in the tree, which is not expanded, causing the collapse.

I hacked at this to validate the theory - by adding a thread index into the terminate event, so correct deltas could be built. The thread no longer collapsed.
Comment 5 Pawel Piech 2008-10-31 16:58:42 EDT 
Note, you don't absolutely need an index for a REMOVE delta.  But if you don't provide it this bug could still resurface in some extraneous cases, because handleRemove() will remove a random unrealized item if the specified element is not found.

Currently, the burden is on the model to provide appropriate ADDED/REMOVED deltas, which is sometimes unreasonable.  We've managed to implement this logic properly for the expression manager, but it's the only use case that works as far as I know.  

Trying to solve this in the viewer could be extremely tricky.  It would require that we somehow preserve the state of expanded items from before TreeViewer.setChildCount() was called, until the time when all those children were revealed and realized.  We do something similar right now with saving/restoring state for different inputs but even that is not working 100% reliably.
Comment 6 Darin Wright 2008-11-03 12:52:48 EST 
I don't think we want to add support for maintaining expansion state of elements in the viewer as they are moved around. That would compilcate the viewer framework even more :-)

In this case, the model has the information, so it should just provide the proper index (and leverage the information in the model proxy). However, it means the solution will only benefit the Java debugger .

Comment 11 Darin Wright 2008-11-05 15:11:07 EST 
This approach is still not fool proof. For example, when the node containing all the threads is collapsed and a breakpoint it hit, the expansion triggers a refresh of many elements. Since the threads are being added/removed while this is happening, the refresh can cause the originally expanded thread to end up in a different location (causing it to collapse). It's actually a different problem than the "remove", but it appears to the user as the same problem.
Comment 12 Pawel Piech 2008-11-06 17:06:58 EST 
In DSF we have another variation of this bug.  DSF supports a manual refresh mode for the list of threads, so that the user is not overwhelmed by the high number of threads being added/removed in a dynamic system.  But when the user manually refreshes the list of threads, the indexes of added/removed threads is long lost.  

As crazy as it sounds, I think it would be worthwhile to at least investigate saving and restoring the expansion state between structural updates.
