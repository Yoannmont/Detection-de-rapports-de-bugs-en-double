Comment 8 Jason Sholl 2006-09-20 17:20:07 EDT 
I suspect this page is failing because of erroneous .classpath entries in your .classpath file which were added by an earlier version of WTP, by going into the JDT build path preference pages.

Your .classpath file should have this entry for library projects or anything that should map to WEB-INF/libs at runtime:
<classpathentry kind="con" path="org.eclipse.jst.j2ee.internal.web.container"/>

If you have any direct classpath entries that should map to WEB-INF/libs, then you should remove them.  For example make sure you don't have anything like this:
<classpathentry combineaccessrules="false" kind="src" path="/J"/>

If your Web project is a memember of an EAR then it should have an entry like this to resolve MANIFEST.MF entries (this entry will only be present if it is using MANIFEST.MF entries):
<classpathentry exported="true" kind="con" path="org.eclipse.jst.j2ee.internal.module.container"/>

Likewise, if you have any projects, jars, libraries, etc added directly to your .classpath that should be picked up by the MANIFEST.MF entries at runtime then you should remove them.

Once your .classpath file is reduced, then all the entries you need should be resolved by their respective classpath containers.  You can verify this by expanding from the J2EE Perspective WebProject->Java Resources->Libraries->Web App Libraries or EAR Libraries and ensuring all of the classpath entries are resolved.

If this is indeed your problem, then the properties page should start working for you afer the .classpath entries are removed.  The properties page needs an overhaul which will be addressed in a future version of WTP.
