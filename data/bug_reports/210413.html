Description David Carver 2007-11-20 11:56:41 EST 
Currently, the SSE editors, allow for validation based on an XMLCatalog, or by including the DTD or Schema reference within the document.  However, for some editors, like XSLT, a way is needed to allow an editor to choose what schema to be used, instead of using what the user specifies.    Xerces has the ability to use a grammar pool, that can be used to validate the given XML content against, without having to rely on schemalocation, or the DTD specified in the file.

This option is going to be of need in the WTP incubator project so that structure validation as well as content validation can occur, without having to add xsi:schemalocation to the XSL file.  Standard XSL editors have built in support for the XSL syntax, and it will possibly cause some tool interoperability issues if this has to be included, and is a non standard way of editing XSLTs.

This will also provide the XSLT editor with structure and content assist proposals, based on the specified grammar.  Currently in X-Assist this is done by loading a DTD in the ContentAssistProposal routines.  While this method gives content assist, it doesn't provide validation for the structure, and making sure that only the items that are allowed appear in the document.

There are already schemas for XSLT 1.0 and XSLT 2.0, the version attribute needs to be checked, which is why this particular feature needs an API interface so that it can programmatically tell the validators what schema to use for validation from the SSE editors.

XSL typically has the extensions of XSL or XSLT.   A user may specifiy any extension they choose, for a file. This also allows for future editors based on XForms, XProc, SchemaTron, or other xml specifications to work based on their content-type.
Comment 3 Jesper Moller 2008-01-18 20:17:12 EST 
Two things:

1. The contentType recognizer called XMLRootElementContentDescriber is being improved for 3.4, to also take namespace URI into account when matching. See bug 135575 for this.

2. The thing with XSLT is that it's a templating language: It contains fragments of other namespaces (say, XHTML) intertwined with itself, sometimes in an incompatible manner. Assume you are generating a schema from some XML doc using XSLT:

<?xml version="1.0" encoding="UTF-8"?>
<xsl:stylesheet version="1.0" xmlns:xsl="http://www.w3.org/1999/XSL/Transform">
	<xsl:template match="/" >
		<xsd:schema xmlns:xsd="http://www.w3.org/2001/XMLSchema">
			<xsl:for-each select="/datatype/class">
				<xsd:complexType name="{@name}">
				</xsd:complexType>
			</xsl:for-each>
		</xsd:schema>
	</xsl:template>
</xsl:stylesheet>

Due to schema constraints, this will give false warnings from the generic in "xsl:for-each" and the 'name' attribute. "Normal" XML validation just doesn't make sense for XSLT.

That's why it'd be nice with an override mechanism. I imagine an extension handled by the validation framework allowing overrides (or rather exclusions in the new validation framework terms) to be contributed piecemeal). 

But wait - isn't this effectively an outcome of bug 181443?
Comment 4 David Carver 2008-01-18 20:53:57 EST 
One more thing to think about that could potentially bite us when using the XSLT editor that have grammars stored in the XML Catalog.

Take this not unusual snippet of XSLT:

<xsl:stylesheet xmlns:xsl="http://www.w3.org/1999/XSL/Transform"
                xmlns="http://www.w3.org/1999/xhtml"
                xmlns:html="http://www.w3.org/1999/xhtml"
                xmlns:xsd="http://www.w3.org/2001/XMLSchema"
                xmlns:ppp="http://titanium.dstc.edu.au/xml/xs3p"
                version="1.0" exclude-result-prefixes="xsd ppp html">
	<xsl:output method="xml" encoding="ISO-8859-1"
                   standalone="yes" version="1.0"
                   doctype-public="-//W3C//DTD XHTML 1.0 Strict//EN"
            doctype-system="http://www.w3.org/TR/xhtml1/DTD/xhtml1-strict.dtd"
                    indent="no"/>
.
.
.
.
</xsl:stylesheet>

The potential problem here is that if somebody has the XHTML namespace defined in the XML Catalog (which currently it's only defined by DTD), you could potentially have the XML Editor trying to validate portions of the document that are using the XHTML namespace.  However, as Jesper described, an XSLT is a template, and will not have valid content for a particular namespace in a particular catalog.   So there needs to be control over what is validated and what isn't validated.

The above scenario would mark every XHTML fragement as invalid unless there was a way to override it.

Now with that said, it would still also be nice to be able to reference the Catalog and get the content model of any known namespaces so that they could be used for Content Assistance when creating the templates.    This is content assistance on any item not directly used or residing in the XSLT namespace.

We'll know better how this is working once it is out in the field so to speak.

Comment 6 David Carver 2008-03-30 22:03:55 EDT 
I may have found a work around for this with the XSLT 1.0 and 2.0 schemas we use for validation.   Currently items we have an xsd:any namespace="##other" and namespace="##local" with processContents set to "lax".  By setting these to "skip" it allows us to still have content assistance for these areas, but not have those elements that aren't in the XSLT namespace marked as being in error because they don't contain the appropriate content according to the model.   This will be going into 0.5M6 of XSL Tooling for community testing.

See bug 224819 for more information on how XSL Tooling will handle this in it's schemas.
