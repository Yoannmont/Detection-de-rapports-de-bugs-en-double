Comment 7 Tim deBoer 2006-07-13 15:20:43 EDT 
Thanks Nitin, that nails down exactly where the problem is!

J2EE team - The ModuleArtifactAdapterDelegate (implementation of org.eclipse.wst.server.core.moduleArtifactAdapters extension point) is creating an instance of org.eclipse.wst.server.core.util.WebResource but passing in null as the module (first argument to the constructor). Since the resource isn't in a module, the wst.server.ui is returning the error message to the user.

I'll use this bug to improve the error message, but the real problem is that it can't find or isn't returning the web module correctly. Please let me know if you need any help debugging this.

Comment 18 Tim deBoer 2006-11-20 16:21:15 EST 
Re comment 16: Yes, that's the new error message to indicate that incorrect output (the selection isn't within a module) was coming from an underlying component. You should never get this error, and I haven't had any reports of it since this bug was fixed by the J2EE team. If you still see it, the bug should be reopened against jst.j2ee (or another team if the editor selection is not basic J2EE related).

I am closing this bug for two reasons:
 * The original bug was fixed a long time ago by a different team. It tracked a specific hot bug and should not have been used for the unrelated "Could not find a client" issue.
 * The second issue would appear if you're selecting a type of resource that some component in WTP understands but there is no available client app (e.g. web browser) to run it. I can't reproduce or find any examples of this.

Cameron - If you can still reproduce, please open a new bug with details on how to reproduce and if you are always selecting the same thing, or a specific resource/filetype causes the failure message.
