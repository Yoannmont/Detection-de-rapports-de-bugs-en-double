Description Jeff Calcaterra 2006-10-19 15:35:04 EDT 
Users should be able to enter a search term, then optionally, rather than jumping to the event, display a list of events that contain that string.

This is a function that is available in the text editor tools that are used by support organizations.

Please contact me for a design.
Comment 1 Marius Slavescu 2006-11-21 09:24:01 EST 
This function can be achieved today by using the current Filtering capability although you get the result in the log view.

I'm not sure how the Eclipse Search view scales today, we need to look into it. 

The Search view is used in Test Execution Result searching case, Bianca could give you some details (she have been using the query engine that we use to filter to build the result).
Comment 2 Jeff Calcaterra 2006-11-21 10:02:08 EST 
Yes, users can filter the results, but they loose the context.  What was being requested here is for the user to be able to search for a string, then get a list of links that they can click to jump to the event.  This way they can see the events that are happening around this event.  It would be part of a Find dialog (BTW, also please see Bugzilla 162109).
Comment 3 Marius Slavescu 2006-11-21 10:15:20 EST 
I agree that integrating with Eclipse Search capability would be good, we just need to investigate if the scalability of that view is appropiate for our usecases and also see how we define the search context (Bianca added a view context for Test search, to search in the active view instead of the workspace/project/resources).

I think this capability should be extended to all the other TPTP models so I see it build as a core function plus extensions for each model/context/view.
Comment 4 Jeff Calcaterra 2006-11-22 09:53:17 EST 
I agree about the scaling problem.  There needs to be a way to handle if the user types Find * (find everything).

A second thing that's important to make this function useful is that the results need to be displayed in a nonmodal dialog (or maybe in another view, while the log is still visible.  The user needs to be able to jump to an event, scroll up and down around that one, click a link in the results to jump to the next one, scroll up and down again, and so on.
Comment 5 Marius Slavescu 2006-11-22 10:56:43 EST 
There are three ways to filter/search the log entries.

1. Using Filter dialog you can specify what should be shown in the Log View.
3. Using Log View - Find dialog you can search in the filtered set of log records
3. Using Eclipse - Search dialog (our tab) you can search in different contexts (this is not implemented yet, it should be the outcome of this feature).

Filtering data would affect finding and searching, when we use the Find dialog we always apply the find query on top of the filter query that way we avoid finding records that are filtered out.

By using the Eclipse - Search dialog (our tab) you should be able to search in the whole workspace/selected project/resource/working sets/selected view and you'll get a result set across all this containers.

When we will try to view any of the records from the search result set we will need to make sure that they are not filtered out by asking the user to disable view filtering, otherwise we would not be able to select those records in the Log View (in the case of log records, we could search for symptoms in a similar way).

We should aim to make the three methods of filtering/searching mentioned above consistent across our perspectives (Trace/Log/Test/Symptom). The filtering is pretty consistent although not available everywhere.

Comment 9 Jeff Calcaterra 2006-11-22 15:44:32 EST 
That's fine.  That was the original idea that I had mocked up.  The important point is that just searching for a text string should involve the absolute minimum number of steps:
1. User clicks Edit > Find (or Ctrl + F)

3. User types string then presses Find.

Comment 10 Marius Slavescu 2006-11-22 16:04:34 EST 
I mentioned Eclipse Search dialog because Find dialog is not supposed to build a list of matches (that behavior is consistent across all eclipse Find dialog implementations).

See bellow my small changes:

(In reply to comment #9)
> 1. User clicks Search > Search (or Ctrl + H)
> 2. System displays search dialog in Z-dimension, with cursor in "Containing text" field
> 3. User types string then presses Search (or Enter).  
> 4. By default, system searches all fields for the string that was input.
> 

I would try to avoid searching in all fields by default, because that will add a big overhead on the search process (especially if you search in multiple resources), the user could change any time the default behavior and check a checkbox like "Search all relevant fields", where relevant fields list can be computed on the fly depending on the entered data. Let's say it enters a string that contains letters, searching that string in numeric fields won't work, then the numeric fields should not be part of the dynamic query that will be built and executed against the model/resources.
Comment 11 Jeff Calcaterra 2006-11-27 09:31:55 EST 
This is one place that the RCP and Eclipse versions of the application really need to diverge.  We are positioning RCP for non-Eclipse users, it does not make sense to drag in Eclipse baggage, like having multiple ways to search.  Find is critically important in viewing logs and it is important to make searching for a string simple, but with some powerful options and to enable searches from a single dialog.  The dialog should also be named Find, as it is in almost every other application.

If we are not searching all fields by default (and I am not convinced that we should not be searching all fields by default), we need to do it in an intelligent way, that takes into account how it will be used in the real world.  When a user searches in a log in a text editor, they do not have the requirement to specify fields to search and we cannot make this much more difficult than what they are currently doing.

In general, users will be searching for text in the message field around 90% of the time-this should include both the truncated message text that appears in the table, and the extended message text that appears in the Properties window.  They will probably be searching extended properties the rest of the time.  This percentage might shift depending on the adapter that is applied, but the search needs to be optinmzed to search the extended properties and message fields together.  If we provide a default option that says "Search messeage and extended properties" and one that says "Search All", I think that would be a reasonable way to speed up the search.

Comment 13 Jeff Calcaterra 2006-11-28 17:49:31 EST 
I think that what you are saying is on the right track.  So the Find and Search differ in a few major ways:

1. "Output". Find jumps to the string.  Search produces a list of items that the user can then click to jump to the event.

2. Scope Find searches current view.   Search searches workspace.

3. Advanced options Search typically has more options like the ability to search for a "method", etc.

Most applications handle this type of thing as a "Basic" and "Advanced" search (not usually using those names though), but Eclipse does this in a way that is not very intuitive to a non-Eclsipse user by seperating them into two different menus and using two words that are used interchangably in other applications to mean two different things.


I think that for Eclipse, not changing the locations is fine, but for RCP, it makes sense to have them both in the same menu, and to have names that are more descriptive, like Find and Find in Files or Find and Advanced Find.  The key mapping of Ctrl + F and Ctrl + H is fine to keep, I think.  With RCP, it is much more important to be consistent with other desktop applications than to be consistent with Eclipse guidelines, so they should both be under the edit menu.  The user base is much broader than for the Eclipse version, and most of the users will not be Eclipse users.

I think it is important to look at Bugzilla 162109, because that one is critical for the June release and overlaps with this one alot.  We might even be able to use the same code for both-it looks like the Search has a lot of options that would benifit the Find too.

The main idea would be that the Search and Find would be merged together in RCP, and three differences I listed above would be options, for example #1 could be an option to either Find Next or Find all instnances.  #2 could be a scope option, and #3 could be implemented as various options.  Bugzilla 162109 is critical for user acceptance of LTA-RCP, but what I am suggesting would be a nice step above that.

I'll post a mockup as a follow-up.
