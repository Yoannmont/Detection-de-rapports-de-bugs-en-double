Description Lawrence Mandel 2006-01-03 11:41:29 EST 
The top 10 downloads link on the Eclipse downloads page currently says it was last updated Dec 8/05. Although I think this list can be useful, the fact that it is almost a month old means that it does not reflect the current state of top 10 downloads. Specifically, WTP had ~30,000 downloads of WTP 1.0 last week and it is not on the list.

This list should be updated dynamically. I think the refresh period should be daily but at least weekly would provide more accurate information.

As a follow-on question, how are the rankings for this list determined? Simply total downloads, total downloads/time available or some other way? In order to represent the current popular downloads I think this should be total downloads/time available or something along these lines.
Comment 1 Eclipse Webmaster 2006-01-03 15:27:10 EST 
Ah the famous top-10.  I voted to remove it when we redesigned the site, but it ended up staying.

I just re-ran the top-10.  Webtools came in at #11 with 111,069 downloads.

> This list should be updated dynamically. I think the refresh period should be
> daily but at least weekly would provide more accurate information.

I agree.  I'll keep this open as an enhancement for when we get around to it.


> As a follow-on question, how are the rankings for this list determined? Simply
> total downloads, total downloads/time available or some other way? 

Using the Live Download Stats interface, but automated to run through the list of projects. The result is Total files downloaded matching each project's top-level directory.  I only keep the current month + last month's downloads so that's what I run it against.  So for Jan 3, we're looking at 2005-12-01 to 2006-01-03 data (26.3 million downloads).

> In order to
> represent the current popular downloads I think this should be total
> downloads/time available or something along these lines.

We have too many distinct files to make this work with the amount of downloads we get.  Regardless, don't confuse top downloads with project popularity. If you're looking for a health/popularity metric, you should look at the Project dashboard instead - http://www.eclipse.org/projects/dashboard/

D.

Comment 3 Eclipse Webmaster 2006-01-03 16:33:57 EST 
(In reply to comment #2)

> shouldn't I? Seems to me that the top 10 should be the top 10 downloads people

You're mixing downloads with people in the same sentence.  If ProjectX requires one person to download 10 files to be considered "one download" and ProjectY lumps the files in one zip, ProjectX may end up infinitely more popular than ProjectY, even if Y has more people.

This is the case with EMF, which, for this month, has 5M downloads.

> According to these results both 3.2M4 and GEF should fall below WTP and the
> others look to be out of order. Did I perform incorrect queries or is something
> not stirring the kool-aid here?

I'll admit that positions 1 and 2 are hardcoded to the latest release and the latest stable for SDK. Shame on me - perhaps I should lump them together as the Eclipse project, but I know many people use them as shortcuts, so changing the rules now seems moot.

Some projects, like EMF, consistently get unusually high download counts (like EMF 5 million vs. Eclipse project 1.2 million) so for those I usually look at their website URL position in the web stats (also in committer tools) and "readjust" its top-10 position accordingly.  Again shame on me for doing this, perhaps I shouldn't, but otherwise EMF will be #1 each and every month, and I'll be at a loss to explain to Randy Husdon why EMF is higher than GEF despite GEF being required for EMF.

My vote is to remove the top-10 altogether because download counts are confusing and not a reliable metric for popularity. Getting anything accurate out of it is more work than I'm willing to invest in.

Comment 8 Matt Chapman 2006-01-05 12:21:33 EST 
I think it's good having a top 10 by project (or even a top 15 now there are more projects) but it does need to be made more accurate. As Denis no doubt recalls, some time ago I pushed for update site stats to be included, as prior to that the results were even less accurate. The way to get to the next level of accuracy is for each project to define one or more queries which count the real number of "installs" from that project. For example, for AJDT two wildcard queries would be needed:
    "ajdt*.zip" for zip file downloads
and "org.eclipse.ajdt.core_*.jar" for update site downloads (so that you count the number of installs not the raw number of plugins)
It could be made an opt-in process - your project is only considered for the top-10 if you have defined an appropriate query.

Also I think updating it weekly would be more useful, so you can see the spikes caused by major project releases.

If it's included at all, Platform should count as a single entry, rather than stable and latest releases, as separate releases are not considered for other projects.
Comment 9 Randy Hudson 2006-01-05 13:18:41 EST 
I think the top 10 list should re-sort itself based on whether people click on the links on the top 10 list. You could track its usage, and resort as needed. Then, you would just need some mechanism for how the bottom positions get maintained. Or, you could just have a qualification metric like 5-10% of whatever the platform's traffic is. The list could be more or less than 10.

As far as tracking, each project could nominate a single "core" plug-in and the zips which contain at least that "core" plug-in. You can't just count every ZIP since that would include things like examples, src, documentation, test results, add-on plug-ins, build scripts, etc. Hopefully, projects are organized so that downloaders aren't forced to download the core components multiple times to get the pieces they want.

It's GMF, WTP, and VE that require GEF, but not EMF.

Comment 17 Eclipse Webmaster 2006-09-05 11:22:54 EDT 
*gasp* Here goes nothing...

After 6 months, I have updated the top-10.

How?

- I changed the title to indicate "Popular projects"

- I pulled the top 10 popular project home pages from our web stats (Committer tools -> Web Server Statistics -> www.eclipse.org -> September 2006, where the project's home page appears as /projectname/ in the web stats.

This seems like a very reasonable list that is fair to all.

During our Phoenix meetings we agreed that compiling (and maintaining) a list of core zip/jar files for each project is tedious for both the Phoenix team and each project, so the idea was abandoned.  We also discussed having an arbitrary list, such as "Ian's picks" ... "Denis' picks" .. "Matt's picks" ... but that was deemed meaningless.

By compiling the list of the 10 most popular home pages, we get a good feel for what our community is "into" right now. For instance, Callisto is currently number 4, but was #2 in August.  PHP is now #2, which illustrates the immense popularity of the project. The Tools top-level also scores a respectable #9, which is odd, but revealing nonetheless.

I will be bold and close this as "fixed".  IMHO, this is the best solution I can implement in keeping the top-10 up-to-date using a metric that is fair and consistent for each project without imposing a burden on project teams.
