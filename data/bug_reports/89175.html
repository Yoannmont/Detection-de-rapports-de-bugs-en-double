Description Bob Balfe 2005-03-26 18:26:34 EST 
When calling FrameworkCommandInterpreter.execute(<string>) with a command with 
spaces in it like "bundle 123" it doesn't work because it attempts to match a

the passed in cmd string should be tokenized for spaces and only the first one

Comment 2 Bob Balfe 2005-04-04 10:36:46 EDT 
The problem is if a command interpreter's <i>execute</i> is called by the 
command handler (ie. calling other console commands from your own command) it 
doesnt work.  The reason is the command token list in the interpreter is only 
set during instantiation.  The tokens should be updated if there are no tokens 
left and you can tokenize the command coming in.  This means the tokens have 
run out and the <i>execute</i> method is being called again.

Example:

	public void _run(CommandInterpreter intp) throws IOException{
            <b>intp.execute</b>("myothercommand param1 param2");
            <b>intp.execute</b>("myothercommand param1 param2");
            ...
            //do some other code here
        }

Solution:  (inside of FrameworkCommandInterpreter.java, insert at line 135)
this solution just re-allocates the token list under two conditions.
<--New code-->
		String firstCmd = cmd;
		
		//If we are being called by a command handler, check to see if 
there are no tokens left
		//and if there is a cmd then tokenize it from scratch again.
		if (tok.hasMoreTokens() == false && cmd != null && cmd.length
() > 1){
			tok = new StringTokenizer(cmd);
			firstCmd = nextArgument();		
		}
</--New code-->
		for (int i = 0; !executed && (i < size); i++) {
			try {
				Object target = commandProviders[i];

Comment 6 Thomas Watson 2005-06-03 16:42:26 EDT

Proposed fix

The fix from comment 2 was not complete.  The tokenizer was not being
reinitialized for all calls to execute from other commands.  For example if you
had a command like this 

public void _run(CommandInterpreter intp) throws IOException{
  intp.execute("myothercommand param1 param2");
  intp.execute("myothercommand param1 param2");
  String data = intp.nextArgument();
  // do something with "data" ...
}

osgi>run somedata

Running this command at the console would not have reinitialized the tokenizer
for each of the inner execute commands.  This patch creates new
CommandInterpreters for each "inner" execute commands which will give them
their own tokenizer to get parameters from instead of reinitializing the
"outer" one.

Bob, can you test this fix out for your scenario?
