Comment 1 Andre Weinand 2002-09-26 13:10:20 EDT 
On Mac OS X it is possible to have filenames containing a forward slash '/'  like "b/w".
To avoid confusion with the Unix pathname separator the Mac Finder replaces '/' into ':'.
However, if a file containing a ':' is used within Eclipse, everything until ':' is considered 
the device and removed from the pathname. As a result the rest of the pathname is 
crippled and cannot be found on the filesystem.

Comment 11 Gunnar Wagenknecht 2004-09-09 02:33:51 EDT 
The patch is incorrect. It breaks nearly all clients that use IPath and Path 
API because it breaks cross-platform interoperability.

Example:

new Path("/my dir/ myfile")
  would be possible on all platforms except Windows.

new Path("internalpref:/sect1/cat2/prf23")
  would be interpreted different on Windows than on Unix.

new Path("/myproject/a dir\file")
  would be interpreted different on Windows than on Unix.

Especially the last example would forces a lot developers relaying on the 
feature that "\" is also accepted as path seperator to review all their code 
and ensure that every "\" is converted into a "/". 

Please remember that Eclipse is a cross-platform tool and that Eclipse is 
expected to behave equal on any supported platform.
Comment 12 David Daoust 2004-09-09 14:07:45 EDT 
I don't understand the logic behind the previous comment.  Are you saying that 
every platform must follow windows naming conventions?  This is a truely insane 
idea when it comes to migrating existing code bases into eclipse (especially in 
the C++/CDT world, where most existing source bases are unix based).  We cannot 
expect that a potential user would go through a 100,000 file C/C++ source code 
base to change all of the non-windows conformant filenames.  There are enough 
barriers to using eclipse without adding one like this.

We want to make it happen as soon as possible, but at this point the only thing that I can definitively

The first step is to decide what the new API would be, what the impact on the existing API would be,

resources to make the changes. "Fixing" Path/iPath is only a small part of the problem.
