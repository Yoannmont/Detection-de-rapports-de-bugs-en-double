Description Sascha Radike 2007-03-17 15:03:51 EDT 
Build ID: 3.3M5eh

When a thread suspends due to a breakpoint or a step the debug view will select the stack frame where the breakpoint occured. The code window will show the position of the current code line. The variables view will show the variables of that stack frame. The thread can be resumed by clicking 'Continue' or hitting 'F8'.

*** Bug: ***
Unfortunately, there seem to be quite some situations when the stack frame _appears_ selected in the debug view after suspending, but in fact it is not. The variables view remains empty and hitting 'F8' or clicking 'Continue'/'Step' does not work. I will have to click on the current stack frame manually to make this stack frame become active. The variables view will then show the variables and resuming will work, too.

This bug does not happen all the time and it is hard to reproduce, but it happens in various situations (JDT and CDT).

I was able to find one situation which _can_ be reproduced. I can't share this test-case because it depends on third-party stuff, but I put a breakpoint on VariablesView.contextActivated(ISelection) and collected this information:

Situation 1: Stack frame selected correctly after suspending:
	- contextActivated() gets called with the selected context being instanceof IDebugTarget
	- contextActivated() gets called with the selected context being instanceof IThread
	- contextActivated() gets called with the selected context being instanceof IStackFrame

Situation 2: Stack frame NOT selected correctly after suspending:
	- contextActivated() gets called with the selected context being instanceof IDebugTarget
	- contextActivated() gets called with the selected context being instanceof IThread
	- contextActivated() gets called with the selected context being instanceof IStackFrame
	- contextActivated() gets called with the selected context being instanceof IDebugTarget

As you can see, the context has been set to IDebugTarget again in the 4th call.

I attached a copy of the stack frames in .contextActivated for situation 2. You can see that the 4th call to contextActivated() looks different. See attachement.

Comment 9 Darin Wright 2007-03-22 09:48:04 EDT 
This test case demonstrates the problem of the collapsing thread. Set a breakpoint on the "// BREAKPOINT" commented line. Turn "Show System Threads" off (i.e. filter them). Debug - when the breakpoint is hit, the thread will collapse. It will collapse each time a transient thread is added/removed (if you manually expand the thread).

Note - this problem is different than suspending with no stack frame selected, but the thread is expanded and the var view is empty (which is the original report - but that is covered by bug 177788).

package junit.samples;

/**
 * 
 */
public class InvisibleThreads {
	
	public class HangsOut implements Runnable {
		public void run() {
			while (true) {
				try {
					Thread.sleep(1000);
				} catch (InterruptedException e) {}
			}
		}		
	}
	
	public class MakesTransientThreads implements Runnable {
		public void run() {
			while (true) {
				new Thread(new Transient()).start();
				try {
					Thread.sleep(2000);
				} catch (InterruptedException e) {}
			}
		}
	}
	
	public class Transient implements Runnable {
		public void run() {
			try {
				Thread.sleep(1000);
			} catch (InterruptedException e) {}
		}				
	}

	public static void main(String[] args) {
		new InvisibleThreads().go();

	}
	
	public void go() {
		// make 20 hang-out threads
		for (int i = 0; i < 20; i++) {
			Thread thread = new Thread(new HangsOut());
			thread.start();
		}
		Thread thead = new Thread(new MakesTransientThreads());
		thead.start();
		while (thead.isAlive()) {
			try {
				Thread.sleep(1000); // BREAKPOINT
			} catch (InterruptedException e) {
				e.printStackTrace();
			}
		}
	}

}

Comment 15 Darin Wright 2007-03-22 12:04:37 EDT 
Howeve, I now understand the problem with the 'selected frame but empty variables' view. The problem can be reproduced fairly easily by resuming a thread to the same breakpoint. The following happens:

* a thread is suspended, a frame is selected
* resume the thread
* the thread gets selected (context change)
* the thread suspends again - fires delta to expand/select the frame
* the delta carries flag indicating the state of the thread has changed
* since the current context (thread) has changed state, we schedule a UI job to fire the context change notification for the thread
* meanwhile the thread gets expanded, and the frame is selected
* incorrectly, the scheduled job runs saying that the thread context has changed (because the job is run asynch, it is now out-of-date with the current state of the debug view).

We need to ensure that the job only fires a context change iff the context in the view still matches the context that changed.

Comment 21 Darin Wright 2007-04-03 17:36:23 EDT 
This bug is caused by the fix to bug 170857. Filters are cleared when an element is removed from the view (since we don't know the index of the element being removed). This causes us to re-compute content and remove items from the tree as filtered items are retrieved. In this case, we remove an item from the tree that was mapped to the expanded thread since it's position corresponds to a filtered element.
Comment 22 Darin Wright 2007-04-05 15:52:03 EDT 
Fixed. When an element was removed from the viewer, we did not know the index of the element removed, which caused us to refresh the parent element instead (and re-filter). This causes items to be potentially removed as they are re-mapped to elements and then filtered, breaking the expanded/selected state of the items.

The fix is to use the index provided in the delta for the removed element. If the delta does not contain an index for the element, we search the mapped items for the element to determine its index. With the index known, we can remove the proper item and update filters without refreshing the parent, maintaining expanded/selected state. When we can't find the removed element (i.e. it might be mapped to a yet not visible item), we can remove an unmapped item from the tree in it's place and update filters as this maintains the child count correctly).
