Description George C. Hetrick 2002-08-19 13:33:21 EDT 
I have several different ClearCase views onto the "same" directory structure, 
and have a project corresponding to each view.

If I set a breakpoint on one view, it is invisible on other views (which is 
correct), however, execution still stops at the breakpoint. In order to turn it 
off, I need to return to the project in which it was set, and turn it off 
there -- at that point, other projects work as expected.
Comment 1 Darin Wright 2002-08-19 13:56:18 EDT 
I believe this is a duplicate of bug# 5188. The problem is that there is no 
strict relationship between a runtime and "projects". In fact, a java 
application being run/debugged may have no correspoding project in the 
workspace.
Comment 2 Darin Wright 2002-08-19 13:57:44 EDT 
As well, there is ambiguity with class libraries. The same class library is 
oftem shared between many projects, so we cannot distinguish the project in 
which a breakpoint is set, when it is contained in the system libraries (for 
example, java.lang.Object).
Comment 3 George C. Hetrick 2002-08-19 14:09:57 EDT 
Actually, I'm surpised Eclipse works as well as it does when I do multiple 
views onto the same class structure.

All I'd want here is that if a project stops at a breakpoint set in a different 
project, that I be able to edit/delete the breakpoint without having to switcg 
projects.
Comment 4 Darin Wright 2002-08-19 14:25:01 EDT 
I think that would be a "ClearCase" feature request - as the debugger does not 
implement those views.
Comment 5 George C. Hetrick 2002-08-19 15:48:09 EDT 
In this case, each view is brought into Eclipse as a separate project, so from 
Eclipse's point of view, I simply have multiple projects that happen to contain 
classes with the same name.

Although these classes have the same name, changes may have been made in one 
view/project that are not visible in another. This would mean that a breakpoint 
on line 100 of file1.java in project1 would not necessarily logically 
correspond to line 100 of file1.java in project2 (in fact, file1.java might not 
even be 100 lines long in project2). [so this differs from Bug 5188 somewhat]

This is definitely an eclipse issue, rather than a ClearCase issue, since I 
could do the same thing with multiple trees of any sort, not just ClearCase. 
Given that there is a fairly simple work-around, I don't know what the severity 
would be, though.

How would the debugger act if the breakpoint record asked to put a breakpoint 
on a non-existent line? If it deals with it robustly, this is fairly low 
priority.
Comment 6 Darin Wright 2002-08-20 11:02:39 EDT 
The debugger will handle this problem gracefully. The attempt to install the 
breakpoint will fail (silently) in this case (i.e. where a type name matches, 
but the line number is invalid).

Marking as later - perhaps we could add some more useful information - i.e. if 
the install failed, etc.

Comment 11 Darin Wright 2005-05-20 09:07:45 EDT 
Ah, a multiple choice test... the answer is mostly (3). In many cases we 
should be able to determine what project a breakponit orginated from, and 
whether or not it should be installed in a class being loaded. However, there 
will be exceptions to this rule when we can not determine which project a 
class originated from. For example, when remote debugging 
applications "external" to Eclipse, or when an application is using a custom 
classloader (such as Eclipse itself).

Comment 16 Radek Wikturna 2006-05-12 02:26:40 EDT 
In our company we are developing a rather big software project consisting of thousands of classes. And of course, we are working simultaneously on the previous versions (2.0, 3.0. etc -mainly fixing bugs) and on the current version (3.1). Bugs reported in 2.0. need to be fixed and integrated into higher versions. So most us us have all the project versions in our Eclipse workspace as a normal Eclipse projects. The disadvantage is 
1) when you call Open Type and type a class name (e.g. ObjectSorter) then you will see mathes in all the projects and you have to be carefull which one you select.
2) when you set a breakpoint in ObjectSorter, in Project_v2.0, line 100, and run  Project_v2.0 everything works fine, the debugger stops on line 100 in the class ObjectSorter. But when you later start working with Project_v3.0 (maybe solving a completely different problem) and you haven't removed the breakpoint the execution stops in ObjectSorter, in Project_v3.0, line 100, unexpectedly although you don't see a breakpoint set on that line ! and what's worse, since the class ObjectSorter may differ in v3.0 (more lines for example), the execution may be stopped at a completely different line of code or even on a line containing no code at all ! And the annoying thing is that you don't know what's going on unless you remember that 2 days ago you set a breakpoint in the same class but in a different version (Eclipse project). Then you have to go the Breakpoints view and remove the breakpoint to be able to continue. Believe me, this rather annoying. So in our case (multiple opened projects representing different program versions), this problems occurs on a daily basis! Please, please, someone fix it.
