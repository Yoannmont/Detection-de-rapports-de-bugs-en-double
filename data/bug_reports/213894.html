Description Dave Carlson 2007-12-26 12:53:38 EST 
It seems that the UML to Ecore transformation needs to be enhanced to consider Associations where both ends are owned by the Association.  In the OMG's SBVR cmof metamodel, there is NO use of composition, and all associations own both property ends.  Neither property is composition, nor navigable.

I would expect that these associations must be transformed into Ecore classes.  I don't know what tools were used by the SBVR spec authors to create their XSD for serialized XMI models (apparently not EMF...), but the cmof associations appear in XSD as shown below, which implies that EMF would need an EClass for each Association that owns both of its ends.

<xs:element name="placeholderUsesDesignation" type="sbvr:placeholderUsesDesignation"/>
<xs:complexType name="placeholderUsesDesignation">
<xs:all>
<xs:element name="placeholder" form="unqualified" minOccurs="0">
<xs:complexType>
<xs:attributeGroup ref="xmi:LinkAttribs"/>
</xs:complexType>
</xs:element>
<xs:element name="designation" form="unqualified" minOccurs="0">
<xs:complexType>
<xs:attributeGroup ref="xmi:LinkAttribs"/>
</xs:complexType>
</xs:element>
<xs:element ref="xmi:Extension" minOccurs="0"/>
</xs:all>
<xs:attribute name="placeholder" type="xs:IDREF" use="optional"/>
<xs:attribute name="designation" type="xs:IDREF" use="optional"/>
</xs:complexType>

I believe that Christian's comment #13 applies only to AssociationClass, and possibly to an Association where at least one end is navigable, but not to an Association that owns both ends and neither is marked as navigable.

If my patch is refined to apply ONLY when both Association ends are owned, and neither is navigable, then I believe it handles multiplicity correctly.  Both properties are owned by the Association (generated to an EClass), and both are navigable only from the Association instance.  I checked my patch as used with Java generated from SBVR and the multiplicity on classes generated from the CMOF associations is correct.

I reviewed the MOF 2.0 spec and find no constraints or sematnics about how to handle associations where both ends are owned and neither is navigable.  Nor does it disallow this.

In UML 2.1.1, this description from Association (page 39):
"An end property of an association that is owned by an end class or that is a navigable owned end of the association
indicates that the association is navigable from the opposite ends; otherwise, the association is not navigable from the
opposite ends."

It does not disallow binary associations with both ends owned and neither naviagable.

In the case of the SBVR metamodel, I think that it would be incorrect (and a quite different metamodel) to require that both owned association ends be marked as navigable.  This description from the UML spec would then require the Ecore mapping to add navigability (get/set methods) to both end classes.  That may be helpful, but should not be required.

It now appears to me that the Ecore generation from Associations with 2 owned ends (only binary) should generate:

(1) if neither is navigable, then generate an EClass that owns both propeties (as my patch does now).

(2) if at least one end is navigable, then do NOT generate an EClass and add the navigable property to the opposite end class.  This is very similar to current behavior, with the difference that associations with owned navigable ends are processed (they are currently ignored).
Comment 25 Christian Damus 2008-01-04 16:12:58 EST 
(In reply to comment #24)
> > Dave, I think the issue of multiplicities also needs to be addressed, even for
> > your patch - owned association ends need to have multiplicities of 1 when
> > represented in Ecore... and only navigable owned ends should be process, IMHO.
> 
> I believe that Christian's comment #13 applies only to AssociationClass, and
> possibly to an Association where at least one end is navigable, but not to an
> Association that owns both ends and neither is marked as navigable.

Any association-end will have this dual nature of multiplicities from the end perspective vs. the association perspective.

Consider this association:

  A -x------x- B
    0..1     *

(both ends owned and not navigable)

The multiplicities mean that, for any instance of A, there can be any number of Bs linked to it.  For any instance of B, there can be at most one A linked to it, or none.  However, any instance of the association links exactly one A to one B.  The multiplicities only describe how many links an object may have.

Because an association doesn't have any other interesting features (operations or properties), it seems redundant to create so many little EObjects to implement these linked pairs.  It's more efficient to have a single link reference multiple objects at each end.  However, if and when a mapping of AssociationClasses is implemented, then this would be inconsistent because AssociationClasses do have intrinsic properties, with different values for each linked pair.

I updated the patch and the test model as per previous comments.  It now works as follows:

(1) AssociationClass is ignored (until a proper fix is worked out)

(2) An Association with 2 ends, both owned, and neither navigable: generate an EClass with reference multiplicity 1..1 on each end

(3) An Association with 2 ends, both owned, and at least one navigable: do NOT generate an EClass and for each navigable end add a reference to the other end type class.

Thus, cases 1 and 3 will yield the same Ecore model if the navigable end is owned by the Association instead of the originating class.  Case 2 is necessary for CMOF models like SBVR and yield the correct Ecore model for these associations.
