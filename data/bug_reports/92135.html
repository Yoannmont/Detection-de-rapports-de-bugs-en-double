Comment 1 Darin Wright 2005-04-21 10:06:54 EDT

Comment 2 Kevin Barnes 2005-04-21 11:37:36 EDT 
Caused by the introduction of LazyInputStream. It keeps reading until the buffer it was passed is full or 
100 ms have passed. Problem is that if you hit a breakpoint the stream can be in a blocked read forever 
without returning the output that it has already read.
Comment 3 Kevin Barnes 2005-04-21 11:56:09 EDT 
fixed in LazyInputStream. If first read doesn't fill the buffer do not read again unless available() > 0. (ie 
poll until there is something to read or timeout is hit)

Comment 14 Kevin Barnes 2005-04-27 14:49:44 EDT 
The console is slower when running within the performance test framework. Timing it outside of that 
framework shows that it is comparable to 3.0 console. I believe the difference has something to do with 
the amount of garbage that we are creating. 

Changes to reduce garbage (so far):
ConsoleLineNotifier - changed ListenerList to ArrayList. Was calling getListeners() a lot and ListenerList 
was returning a copy everytime
ProcessConsole - don't call setName() unless the name really has changed
TextConsoleViewer - revealEndOfDocument() just schedules a job. Reduces the number of times the 
text widget is accessed.
ConsoleDocumentAdapter - ArrayList of Regions changed to 2 int[] (offsets and lengths). ArrayList was 
creating garbage and Region objects were being created and disposed frequently.
ConsoleManager - showConsoleView() no longer calls page.getViewStack() to determine if part is 
visible. Instead it just asks page.isPartVisible(part). getViewStack() returns a copy that was only used 
briefly.
IOConsolePartitioner - PendingParition contains a StringBuffer instead of a String.
Comment 15 Kevin Barnes 2005-04-27 18:00:02 EDT 
IOConsoleOutputStream - added boolean flag to reduce number of String.equals() 
calls
ConsolePatternMatcher - removed dead code.

Comment 19 Kevin Barnes 2005-05-02 13:47:58 EDT 
ConsoleView now registers/unregisters itself with ConsoleManager on create and dispose. Console 
manager was searching for the view using page.getViewReferences() which returns a new copy 
everytime and was creating a considerable amount of garbage.
Comment 20 Kevin Barnes 2005-05-03 15:30:54 EDT 
Difference between running the test suite and running manually was caused by 
different launching methods. Launching manually launches in a background 
thread via DebugUITools.launch(). The test suite called 
ILaunchConfiguration.launch(). This changes the timing of the console 
initialization more in 3.1 than it did in 3.0. Changed test to use 
DebugUITools.launch since it's more representative of the users experience.

Changed 
ProjectCreationDecorator - set preference to run even if project has errors
PerfConsoleTests - launch in background.

Also deleted LazyInputStream from debug.core. It's not used anymore.
Comment 21 Darin Wright 2005-05-13 14:07:06 EDT

that pass on the slow machine are fail on the fast.
Comment 22 Darin Wright 2005-06-01 22:31:44 EDT

Comment 23 Kevin Barnes 2005-06-02 16:36:33 EDT 
Still hoping to get them to pass.

Made some changes:
OutputStreamMonitor - had some sleep()ing built into it. It was originally there to prevent the Thread 
from hogging all the CPU cycles. Removed the sleep()s, set the threads priority to MIN_PRIORITY.

IOConsolePartitioner - added fBuffer to track how many characters are buffered. Changed 
streamAppended() to block if fBuffer gets too high. Set QueueJob's priority to interactive.

These changes actually slow the console tests down a little on my windows box, but they are still the 
same or faster than 3.0 on that machine. I have a feeling faster machines will benefit from these 
changes.
Comment 24 Kevin Barnes 2005-06-07 13:22:49 EDT 
removing sleep() from OutputStreamMonitor cause a loss of UI responsiveness 
because it allows it to hog the CPU. Added a sleep of 1ms not more than once 
per second.
