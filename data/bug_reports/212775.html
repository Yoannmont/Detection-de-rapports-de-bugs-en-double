Description Susan McCourt 2007-12-12 11:33:25 EST 
When you start up a provisioned SDK and open the end user UI, you'll get a long wait. Not a new thing, I'm opening this bug to discuss how to deal with problem.

Ways to deal with this:

- changing the repo elements to only hold URLs and get names from the prefs may "just fix" the problem.
- don't load the "available features" page when the page to be opened will be the "installed features" page.  
- make sure that auto update checking isn't impacting the wait in anyway

Comment 9 John Arthorne 2008-01-08 22:57:30 EST

The only way to do something like this is to make a job family for those jobs (override Job.belongsTo), and then you can use IJobManager#join(family, IProgressMonitor) to get some rudimentary progress. Comment 11 Susan McCourt 2008-01-09 09:56:49 EST 
The UI model objects hang onto the repo instance once it is needed.  More specifically, the first time a query is run against a repo (getting the categories in the end user UI), that repo is loaded and cached.  It will remain cached until the end user UI closes.  When it reopens, it is likely that the repo will be re-loaded.  Since the manager keeps a soft reference, there's a possibility that the reference would live beyond the UI's life cycle.  I don't have a particular example, but say, for example, a background update check is running.

(Side note: The UI model treats profile objects quite differently.  Since a profile instance is now a snapshot in time, the model objects only refer to the id and they get the profile from the registry every time a profile instance is needed. )

Comment 13 Susan McCourt 2008-01-09 15:30:24 EST 
My understanding is that the repo manager is keeping some info about each repo (URL, name, description, system) but not any content caching. The soft reference to a repo instance is its mechanism for caching contents previously retrieved.  So it's possible that the manager will have to return to the server if the reference is gone.
