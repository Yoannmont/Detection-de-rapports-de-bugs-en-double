Comment 4 Jared Burns 2002-06-06 14:13:06 EDT 
The following program demonstrates a problem here:
public class Hello {
  public static void main(String[] args) {
    new Hello().hello();
  }
  public void hello() {
    System.out.println("Hello!"); // Breakpoint here
    while(true) {
      System.out.println("blah");
    }
  }
  public int hashCode() {
    try {
      Thread.sleep(1000);
    } catch (InterruptedException e) {
    }
    return super.hashCode();
  }
}

Debug to the breakpoint, select "this" in the variables view, and start 
hitting F6. I'm not getting an exception like DarinS gets, but the editor 
stops updating its location as I step.
Comment 5 Jared Burns 2002-06-07 12:28:46 EDT 
The problem here is that the VariablesView happens to get the suspend first. 
It then performs an evaluation to update the details pane. Then the LaunchView 
gets the suspend event and tries to update while the evaluation is taking 
place. It sees that the thread is not suspended and doesn't update.
Comment 6 Jared Burns 2002-06-07 12:31:01 EDT 
A simple fix is to add a flag to the launch view event handler, which says 
"The thread was resumed when I tried to update for a suspend event." Then, the 
next time it updates for an implicit evaluation (which it normally wouldn't 
update for), it can update and reset the flag.

This works, but it's not a good solution. I'll look for something better.
Comment 7 Jared Burns 2002-06-07 16:12:39 EDT 
There is also a problem if the expressions view and variables view are both 
open with a selection while stepping. Only one of them can update after the 
step. The other always says it can't perform the evaluation.

Observation for post-2.0 consideration:
We shouldn't be refreshing details (performing evaluations) immediately after 
a step anyway. The variables view should have a timer like ala the launch 
view. If someone is holding down the F6 key, we're just slowing them down.

Comment 15 Darin Wright 2002-08-13 15:39:43 EDT 
I experimented with serial details computations. I queue up detail computations 
in the JDI debug model presentation, executing one at a time (per IJavaThread). 
This way, a variables view and expression view can both properly update after a 
step. However, this makes the view update timing problem more pronounced - i.e. 
the debug view attempts to update, but an evaluation is taking place, and it 
fails.
