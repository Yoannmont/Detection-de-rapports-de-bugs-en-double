Description Alex Freidin 2007-11-14 09:24:29 EST 
Build ID: M20071023-1652

Eclipse hangs at startup (splash screen) due to deadlock between [Daemon Thread:org.eclipse.cdt.internal.ui.text.CReconsiler] and  [Thread main] that initializes the text editor (CEditor). The following conditions must meet for the deadlock to occur:
1. A workspace must have an open CEditor with a file that requires Synchronization, e.g. a file modified in the file system.
2. A workspace contains C project(s) that requires conversion to CDT 4.0

Unfortunately, I can't reproduce such workspace with GNU tools, since some other condition is still missing.

Before dumping the threads stack, here is a description of what happens:
- [Thread main] initializes CEditor during startup and calls CEditor(EditorReference).createPartControl().
- The above installs the CReconsiler and calls initializeSourceViewer()
- The latter calls TextViewer.setDocument()->fireInputDocumentChanged()
- The latter initiates the [Daemon Thread:CReconsiler] that starts running and builds the AST. During this process it finds that the project has to be converted and opens a question window - ProjectConverter.openQuestion().
- The latter doesn't show a window, but starts waiting. I guess UISynchronizer doesn't allow it to use UI until startup is completed.
- [Thread main] continues running in EditorReference.createPartHelper()->part.createPartControl()
- Once the editor part is activated, the AbstractTextEditor ActivationListener is called and it calls partActivated()->AbstractTextEditor.handleActivation(), 
- The latter checks if editor needs to be synchronized (sanityCheckState()->handleEditorInputChanged()->synchronize())
- The editor synchronization window pops up and answering Yes brings us to deadlock. Synchronization operation is waiting for the other job to complete, I guess it waits for the CReconsiler.

Observations: 
1.Answering "No" in synchronization popup window doesn't cause the deadlock. 
2.Replacing a call to ProjectConverter.openQuestion() with a default answer also resolves the problem. Not a solution of cause.
3.Modifying the "synchronization" to be asynchronous process, resolves the deadlock.

Observe that in AbstractTextEditor.java:
public void windowActivated(IWorkbenchWindow window) {
	if (window == getEditorSite().getWorkbenchWindow()) {
		/*
		* Workaround for problem described in
		* http://dev.eclipse.org/bugs/show_bug.cgi?id=11731
		* Will be removed when SWT has solved the problem.
		*/
		window.getShell().getDisplay().asyncExec(new Runnable() {
			public void run() {
				handleActivation();
			}
		});
	}
} The handleActivation() is asynchronous. But a similar(?) method is NOT:

public void partActivated(IWorkbenchPart part) {
	fActivePart= part;
	handleActivation();	
}

A simple patch to AbstractTextEditor.partActivated() resolves the problem, but I'm not sure this is a correct solution.
public void partActivated(IWorkbenchPart part) {
	fActivePart= part;
	fActivePart.getSite().getShell().getDisplay().asyncExec(new Runnable() {
		public void run() {
			handleActivation();
		}
	});			
}
Perhaps it should be resolved in CReconsiler? Is there any other startup UI operation like editor synchronization that may create a deadlock similar to this one? I'm not sure why CReconsiler has to interact with UI during startup and not wait until startup is completed.

I'm not sure whether this is Eclipse platform bug or CDT bug, so I open it in CDT for investigation.
