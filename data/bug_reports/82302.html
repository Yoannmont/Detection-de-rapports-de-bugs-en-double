Comment 22 Ryan Lowe 2007-03-29 11:55:40 EDT 
>> I don't understand why people insist on having...

We insist because a file named build.xml in the root directory of the source it compiles is long-standing Ant convention.  This convention has obviously been around for a longer time than Eclipse has had Ant support.  That location is where the build.xml file belongs in a Java project.

The main problem I see here is that Eclipse is deleting a user-generated file without warning and it shouldn't.  I don't think there's any debate on that point.  Eclipse can do whatever it wants with files it generates and maintains (like .project for example) but a user-generated file named "build.xml" in the project root does not fall under this category.  Using the custom=true option to prevent this automatic deletion/replacement is very poor usability because it can easily (by accidental or ignorant omission) cause the kind of catastrophic data loss people are describing in this bug.

If it's a complicated problem like some people here are saying, then the people having this problem can't just patch it themselves.  We need the Eclipse team's help on this one.  We're trying to help by providing our input and uses cases.

People use Ant for more than just PDE, even in a plugin project -- so the filename "build.xml" in the project root should probably not be de facto for PDE.  Personally, I compile my Eclipse plugin projects with my own custom Ant scripts and that's why I ran into this problem.  Fortunately I use source control and could recover the build scripts that Eclipse replaced with its own.

The Ant convention of using "build.xml" in the root of a Java project is older and more widespread than Eclipse, so as suggested on this bug already Eclipse could use its own naming convention for its PDE-specific Ant build file.  How about build-pde.xml?  If a user makes a file with this name, Eclipse can show a warning that it could be deleted by Eclipse at any time.

Comment 40 Mike Schrag 2007-10-09 12:31:18 EDT 
I finally tracked down what was going on and found this bug after a couple of hours of getting pretty annoyed at having files disappear.  The current behavior is just really wrong.  If you are autogenerating files that are transient, why would you use a very commonly used file name and more importantly overwrite an existing file.

The only case I can think of is that Eclipse crashed previously during a build and didn't clean up properly, and that strikes me as an acceptable failure case that I might need to attend to manually. The other part of this that I don't understand is that if this is a transient file purely for the use of the automated build system, it seems like naming this something more unique would be appropriate build.pde.xml or something that is unlikely to be already taken (or just a random name, for that matter).

i understand that there is a workaround with custom = true, but it's a workaround that you probably only learn about after your files are already deleted, which I would put squarely in the "uncool" category.

Comment 49 Curtis Windatt 2008-06-27 16:17:22 EDT 
Created attachment 106041 [details]
Fix

Adds a check when the finish button is hit on the plugin export wizard and feature export wizard.  Checks the selected plug-in/feature export projects to see if they have a build.xml file.  If there is a build.xml file and no custom=true entry in the properties, the user is prompted asking them if it is okay to overwrite/delete.  The prompt has a toggle to always ignore existing build.xml files that can be reset from the preference page.

There are limitations to this fix.  It only applies to plug-in/feature export.  The check is only done to the projects selected in the wizard.  Therefore, when exporting a feature, only the feature project is checked, any workspace plug-ins that will be built will have their build.xml files deleted.
