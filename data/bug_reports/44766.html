Description Matthew Conway 2003-10-13 13:04:58 EDT 
This is with eclipse 3.0M4

If I use the find dialog to do a regular expression find/replace, I cannot match
and/or replace the end of line character (\n) - matching/replacing seems to only
work at the line level.

Also, in the help/completion in the find dialog for regular expressions when I
type '\', it reports line feed as '\f', when this is form feed in the F1 help
(javadoc for Pattern) - line feed is '\n'

Comment 7 Tom Hofmann 2006-04-05 06:56:25 EDT 
Fixing this could be done along these lines: 

- Let ILineTrackerExtensionX provide the set of characters that may be part of a delimiter.

- In AbstractDocument::replace, examine the at maximum 6 'border characters' of the replace operation. Example:
  - the document contains 'xxABxxCDxx'
  - replace 'BxxC' by 'ExxF'
  - then the capital letters are the characters of interest

- if any of the border characters may be part of a line delimiter, we have to run some special care code.

- one (API compatible) way to ensure well formed line information would be to extend the replacement to one line before and after the real replacement.

Comment 11 Carl Furst 2006-09-25 12:24:09 EDT 
find/replace regular expression beginning/end of line expressions (^,$) also don't work if you have selected lines. Thus, you need to carry over delimiters into where the processing of selected chunks of text is done.

As to encoding, many editors detect this when a file is open and store a dynamically scoped delimiter variable that is used as the end of line character. This is used in editors like UltraEdit32.

Comment 14 Markus Keller 2007-06-14 13:24:48 EDT 
Note that this is only a problem if the regex pattern does not correctly take line delimiters into account. If the pattern used in the example from bug 130843 comment 1 is "\r?\n" or "\r\n", then everything is fine.

This is a complex problem, since we have diverging interests:

a) want to provide direct access to the Java regex library to allow full flexibility for clients who know what they're doing

b) need to protect users from screwing up their files by e.g. inserting something between the two characters of \r\n line breaks
