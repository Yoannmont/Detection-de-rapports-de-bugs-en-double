Description Doug Schaefer 2007-03-28 11:14:53 EDT 
The next step towards a seemless new user experience for our Windows developer base using MinGW, is to prepopulate the debug launch configuration with the MinGW debugger settings.

These settings are different than the Cygwin debugger so I can’t just default to MinGW for the debugger. What I want to do is to somehow specify that a given build toolchain is associated with a given debugger. Given that in the future we’ll have multiple debugger implementations (CDI and DSF and maybe others), I’d rather do this on the debug side, i.e. the debugger says what toolchains it works with.

So the question is, how do I tell what toolchain is being used for the active configuration of a project. I was able to figure out how to get the ICConfigurationDescription for a project but got stuck there.

As well is the more general question on whether this is the right approach. Ken, is there something in the new launch paradigm that I need to consider as well.

Comment 2 Ken Ryall 2007-03-28 13:10:33 EDT 
I think there are a few places where the launch and project systems need to interact:

When creating a new launch configuration for a project we need to be able to create the correct launch type based on the active build configuration. We then need to be able to fill the new configuration with reasonable default values based on info from the project/build configuration.

The new context launch support in 3.3 uses a launch shortcut's enablement expression to determine which shortcuts to use for the project in context. So we need some property the expression can check to see if it is the right shortcut for the active build configuration for the project.

When you launch and the pref setting says to build before launching we need to build the configuration associated with that launch, which may not be the active one. For example for project can build both ARM and x86, if you launch the x86 debugger it needs to build the x86 build config even though the ARM one be active.

We're doing most of this on the debugger side in our product so we need to think about how to do it in a more general way in CDT.
Comment 3 Doug Schaefer 2007-03-28 13:24:34 EDT 
I think the debugger side is the right place to do most of the work. The trick is to get the right information out of the CoreModel so that we can do it.

I'm not sure we want to have debug depend on managed make, so maybe we do need more info in the ICSettingObject API. Right now we can get the configuration id. So we could store that, but would need API somewhere to ask to build that configuration. As for filling in defaults, maybe we need to be add tool chain to the hierarchy, or make the tool chain id available in the ICCBuildSetting object. If the build system doesn't support tool chain ids, then returning null would be fine and we resort to the current behavior.

BTW, lets wait until after M6 before doing any of this.

> object. If the build system doesn't support tool chain ids, then returning null

An alternative to introducing new elements/methods for getting the tool-chain id is to use the per-configruation session property mechanism. (See ICConfigurationDesription.get/setSettionProperty()). We could define a public TOOL_CHAIN property that the Build system could assign if necessary on the configuration load/apply and adjust subsequently in case the tool-chain is modified.
This approach seems more appropriate for me in that sense that it is more general, i.e. in case we need, e.g. option ids, tool ids in the core in the future we would not end up adding tools and options to the core, but would rather present them as preperties as well.
We could make some adjustments/enhancements to the current configuration properties mechanism if necessary, e.g. introduce configuration persistet properties or allow defining some properties as read only.

What do you think?

Mikhail

Comment 8 Mikhail Sennikovsky 2007-04-12 09:55:31 EDT 
(In reply to comment #7)
> Part of the issue is dealing with inheritance in the build definitions since I
> many need to look up the hierarchy to find the right name, or I do some pattern
> matching on the id and make sure I use id's that show the hierarchy.
If I understand this correctly, the problem is that there are multiple tool-chain definitions (with different ids) matching one and the same tool-chain.
The CDT build System now contains the mechanism for calculating the list of so-called "real" tool-chains/tools/builders. Each "real" element may represent a set of different elements representing one and the same tool-chain/builder/tool.
The problem is that the "real" element is now selected rundomly from the set of identical elements so you can not say for sure which one will be real and use its id for matching. I wonder if we could make the "real" element calculation logic more deterministic, e.g. a tool-chain/tool/builder may have an "isReal" attribute forcing the given element to be selected as "real".

Mikhail

Comment 13 Ken Ryall 2007-04-29 11:44:39 EDT 
I committed my changes that make the shortcut enabled if any CElement is selected, not just a binary. I think that leaves the remaining work in CApplicationLaunchShortcut to construct the launch configuration based on info about the project. For an ISV product like Carbide we should be able to use our own shortcuts that create our own specific launch configs and run our wizards to interview the user if we need launch settings (like connection info) that we can't get from the project settings. When I do this for Carbide I'll see if there is any of that stuff that seems like it would be useful to contribute to CApplicationLaunchShortcut.

Note that this entire mechanism won't work the way C/C++ developers expect unless both of the new launch prefs are turn on and these are disabled by default. In Carbide we can just change the defaults in our product's ini file, but I'm starting to think that CDT should have something like a "first run" wizard that says "Welcome to CDT, here are some settings that help with C/C++ development, do you want me to change them for you?"
