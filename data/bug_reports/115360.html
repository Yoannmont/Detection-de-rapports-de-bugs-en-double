Comment 21 Dani Megert 2007-11-21 05:43:49 EST 
The user sees three different issues here:
a) import archive: the lock is released after some time
b) import existing project from archive: the lock is hold until the VM exits
c) selecting an archive (no import): the lock is released after some time


Reasons:
- ArchiveFileManipulations: isTarFile() and isZipFile() don't close the file
- TarFile constructor can fail with exception where it leaves its stream open
- The caching of the provider(s) in ArchiveFileManipulations doesn't work
  properly and is dangerous ==> best is to just remove it.
- Each WizardProjectsImportPage.ProjectRecord takes the same structure provider
  which isn't released when finished (using the same open archive is OK but
  we need to hold on to it in order to close it at the end). Fortunately this
  code isn't yet made to import in the background.
- TarFile.getInputStream creates file input streams without closing them.
- Unlike ZipFile, TarFile does not override finalize() which explains why
  deletion works after a while.


I've fixed the above mentioned problems in HEAD (builds > N20071121-0010). I also added the TarFile.finalize() as panic code but that's currently not needed/triggered as we now correctly close all opened resources.


I'll attach a patch with my changes in case Platform UI decides to backport this to 3.3.2.
