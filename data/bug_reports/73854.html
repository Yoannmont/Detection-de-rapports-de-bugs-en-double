Description Philipe Mulet 2004-09-14 06:12:01 EDT 
Build 3.1M1

When using shared compiler preferences on a per project basis, the 
generated .setting/org.eclipse.jdt.core.prefs file is quite unreadable.
It could be improved in different ways:
- always sort the keys
- shorten the stored keys when finding prefix matching file name:
   e.g. org.eclipse.jdt.core.property --> .property 
- allow to define associate some category to properties: logical grouping
- allow to define associate some description to properties like in .options file

alternatively this could be delegated to plugin owner, so as to let it 
serialize out its properties in a human readable form, if it cares.

Comment 5 DJ Houghton 2004-12-13 09:43:00 EST 
The preferences are stored in memory in a java.util.Properties object and then
persisted to disk via Properties#store.

The spec for Properties#store can be found here: 
http://java.sun.com/j2se/1.4.2/docs/api/java/util/Properties.html#store(java.io.OutputStream,%20java.lang.String)

The code which controls most of this is in the EclipsePreferences
class/hierarchy. See the #save(IPath) method.

Comment 7 DJ Houghton 2004-12-13 10:31:03 EST 
You can access the eclipse.org CVS repository via an anonymous pserver CVS
connection. 

File -> New -> Other -> CVS -> Checkout projects from CVS -> Create new
server: dev.eclipse.org
repository path: /home/eclipse
user: anonymous
password: <blank>
connection type: pserver

This information can also be found in the Eclipse FAQ located here:
http://www.eclipse.org/eclipse/faq/eclipse-faq.html#users_7

The code for preferences is located in the org.eclipse.core.runtime project.

If you have any suggestions to fix the "whole messed up design" then feel free
to elaborate and include them in a bug report.
