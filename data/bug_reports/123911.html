Description Tim deBoer 2006-01-15 17:56:04 EST 
Validation messages on J2EE DTD files tends to be rather cryptic and un-user friendly, e.g.:

cvc-complex-type.2.4.b: The content of element 'application' is not complete. One of '{"http://java.sun.com/xml/ns/j2ee":display-name, "http://java.sun.com/xml/ns/j2ee":icon, "http://java.sun.com/xml/ns/j2ee":module} is expected.

I doubt there are many users who can easily parse that into the real error message "application element must not be empty" or something even more friendly like "Enterprise applications must contain at least one module".

Validation isn't useful if the user can't understand the error messages it provides. We need to make these messages user friendly.
Comment 1 Neeraj Agrawal 2006-01-16 09:50:39 EST 
Assigning to the wst.dtd component
Comment 2 Lawrence Mandel 2006-02-15 15:07:05 EST 
Kihup, I believe you are looking into the same type of issue with XML error messages. Have you looked at DTD validation messages at all?
Comment 3 Lawrence Mandel 2006-02-15 15:13:41 EST 
Tim, the error messages you see are reused from Xerces. I agree that they can seem cryptic (although I think the one you highlighted is pretty clear) but they do contain correct and complete information based on DTD or XSD grammars. Your suggested message change requires that we include specific validation messages for certain types of XML files. This may be possible for often used XML files such as deployment descriptors but we may have trouble using this strategy for all XML files. To implement something like this we'll need to have an extensible way for validation messages to be modified (this may lead to trouble) so the generic XML validator does not contain J2EE specific logic.
Comment 4 Kihup Boo 2006-02-15 15:35:34 EST 
I have looked at only WSDL and XSD validation messages (opened a couple of defects I plan to work on). I did not pay much attention to DTD. As you noted it would be good to have an extensible way for customizing validation messages in general.
Comment 5 Tim deBoer 2006-02-15 16:26:41 EST 
I agree that the current errors are technically correct, and I'm not asking to have user friendly messages for all types of XML files. I'm just pointing out that for the novice or average J2EE user, the current error messages on J2EE DDs are probably more confusing than the actual error. e.g. 'cvc-complex-type.2.4.b' means nothing to me.

This is a usability issue mainly for the J2EE DDs. Error messages need to be readable text (e.g. like JDT errors) for WTP to be competitive in this area.
Comment 6 Lawrence Mandel 2006-02-15 16:33:27 EST 
Tim, I agree. I'll investigate adding an extensible way to customize validation messages for specific XML file types. With a mechanism like this in place, a JST plug-in can extend the generic XML validator and provide custom 'user friendly' messages for deployment descriptors.

Comment 14 Craig Salter 2006-03-15 01:43:43 EST 
I took a look to see how easy this would be to fix hoping that I could add a couple of easy lines of code to fix it.  Sadly the fix for this problem is really quite tricky. Xerces doesn't provide enough information in it's error handler to recognize the 'missing module' error from any other possible content error with the <application> element. 

It is possible to implement this in a cheap and dirty way... but it's definitely messy and I'm worried that this is just one of many similar problems that would require additional attention.

So my questions are...

1) is an application element without a module the only condition that we're worried about rephrasing
2) what's an appropriate alternative error message
3) am I correct in assume this should simply be a rephrased error ... or would it be better to make it a warning or perhaps just remove any problem indication altogether?
Comment 15 Lawrence Mandel 2006-03-15 11:07:42 EST

Comment 19 Craig Salter 2006-03-16 01:57:06 EST 
We need to think up a good long term approach for this sort of problem.

1) A specialized validator is reg'd for application.xml that takes precedence
(we'd need to fix infamous bug 86385 to make this work nicely).  The application.xml validator extends the xml validator and provides some specialized behaviour.

2) We define some extension points on the xml validator to provide hooks for customizations that are registered for contentType or namespace.  There's a few problems with this approach though
  - if a lot of customization is req'd there's not much value to this
    one would be better of writing their own validator
  - it's hard to augment a SAX based validator to do complex checking 
    (DOM would be more suited to this)their own customized model)

I'm not sure which one would be easier for extenders to work with, but I think (1) seems cleaner ... although I'm not sure if contentType is too limited (can contentType be aware of namespaces?)

OK... now having said all that... I've created a patch (against HEAD) to demonstrate an approach to solving this problem (in the spirit of approach 2).  I've made no attempt to implement extensibility support (for now I've hardcoded some J2EE namespaces and element names) but it should be obvious from the code where extensions would kick in. 

Ted, I'm hoping that if your folks feel this is an important enough problem that your guys can thoroughly test (including running our JUnits) and backport this change to 1.0.2 (it's probably a trivial backport).  I've done very little testing (although the code changes are fairly isolated and look safe enough) and don't have enough cycles to contribute more to this.

Lawrence, feel free to weigh in and veto this rather hasty patch if you spot any issues.

Comment 21 Lawrence Mandel 2006-03-29 22:50:18 EST 
Craig, I reviewed your patch. Thanks for looking into a solution. I think you're on the right track. Modifying your patch to allow for extension may be OK. At this point I see a few problems with your patch as a standalone fix:
1. I haven't had a chance to test your patch and I haven't heard any feedback from anyone else.
2. As you stated, your patch includes a hard coded J2EE error message in the XML validator plug-in. More work needs to be done to allow for further extension.
3. We haven't had a chance to really understand the types of errors that adopters may want to fix. Is this isolated to errors reported on J2EE schemas? Will a lot of the errors need to be replaced? (Should a separate validator be created?)

As we do not have a fully thought out and tested solution and as Ted said in comment #13 that he's OK dropping this bug from 1.0.2, at this point I think it's best to defer this to 1.5. I'll wait to hear feedback on tomorrow's status call before changing the target milestone.
Comment 22 Arthur Ryman 2006-03-30 14:45:05 EST 
I vetoed the proposed fix today because adding J2EE specific error messages to WST is the wrong fix and sets a bad precedent.

There are several better ways to fix this.
1) define smarter validators for J2EE that go beyond XML validation - they should add semantics
2) fix the wizard to it creates valid projects. if the spec says an element MUST NOT be empty then you shouldn't create the project until you have content to give it (or the wizard should create the required content too)

The problem should be very well described in the the Help. Maybe add it to the FAQ.

Comment 24 Lawrence Mandel 2006-04-01 03:40:29 EST 
I've created a fix that provides a better structured solution to this problem. The fix builds on Craig's patch and provides a way to specify an error message customizer for a given namespace. The customizer will be called during XML validation when errors are produced for elements in the namespace.

I'm going to attach 3 patches for this fix. 
org.eclipse.wst.xml.core - contains the framework and extension point for the error message customizers.
org.eclipse.jst.j2ee.core - contains the J2EE error message customizer.
org.eclipse.wst.xml.validation.tests - contains unit tests for the framework added to the XML validator

TODO: I still need to create unit tests for the J2EE error message customizer. The J2EE core test plug-in is giving me some trouble so this will take more time.

Arthur - This fix does not include any J2EE specific error messages in WST. The separation between WST and JST scopes is maintained.

The fix is based on 1.5. If this is found to solve the problem it should be relatively easy to backport this fix to 1.0.2.
Ted - I'd appreciate your feedback on this fix to ensure it solves your problem. Specifically, please comment on the error message and identify additional error messages that you think need to be customized.
