OS:	XP
Build no.:	02242005

Step by step instructions to reproduce:
1. Start Eclipse, name a workplace with national characters.
2. create a testsuite whose name contain national characters by using "HTTP
Proxy Recorder".
3. Switch to "Test Perspective", right click on the testsuite just created and
select "Generate...". This will generate Java file and class automatically. The
automatically generated java and class file name will have the same name as the
testsuite ( they also contain national characters)
4. Make a deployment by associating appropriate artifact and location.
5. Run "Hyades URL Test" for this testsuite.



Problem description:
1. If the workspace or testsuit name contains any national character, the
execution of "Hyades URL test" will be failed. 
Form the "Event tab" of the execution, the error message shows
"java.lang.ClassNotFoundException:test.<NLtestsuiteName>"

ps. <NLtestsuiteName>: the testsuite contains national characters.

Expected result:
1. The execution of "Hyades URL Test" should be successful even workspace or
testsuite name contain national characters.

Comment 13 Bianca Jiang 2005-04-25 11:03:51 EDT 
Currently the generated test script class doesn't compile if it contains
non-ASCII characters. The reason for that is with the encoding of the input
stream when the test script file is created. byte[] is used for the input stream
which will break down in the middle of double byte character when no encoding is
specified. Using "UTF-8" encoding ensures the creation of test scripts works
with DBCS.

Depending on the font settings of the workbench
(Window->Preferences->Workbench->Colors and Fonts), native characters may not
display correctly in Java editor. But this doesn't affect the integrity of the file.

Also need to make sure execution uses the right encoding when reading the test
class file for the test to run properly.

Comment 15 Bianca Jiang 2005-04-27 18:39:25 EDT 
Change to org.eclipse.hyades.test.java.internal.codegen.JUnitGenerator is
checked in to use UTF-8 when generating test.

Second part of this problem is with the test Runner JVM launcher. The launcher
takes the name of the test along with other things as command line arguments in
a string (double bytes) and processed it as ASCII characters. non-ASCII
characters will lose its high end byte during this process and test name becomes
garbled which causes test class unable to be loaded.  

The code in question is on native side in plugin
org.eclipse.hyades.datacollection,
collection/collectors/framework/HCLauncher/hclaunch.c:

/* Creates a char* ASCII representation of a Java String */
static char* copyJavaStringToASCII(JNIEnv *env, jstring str) {
	const jchar *oldChars;
	char *newChars = NULL; /* Bug 68899 */
	jint length;
	int i;

	oldChars=ENV(env)->GetStringChars(ENVPARM(env) str, NULL);
	length=ENV(env)->GetStringLength(ENVPARM(env) str);
	if(length > 0) { /* Bug 68899 */
		newChars=(char *)malloc(length+1);
		for(i = 0; i < length; i++) {
			newChars[i] = (unsigned char)oldChars[i];/*<<DB char loses higher end here*/
		}
		newChars[length] = '\0';
		ENV(env)->ReleaseStringChars(ENVPARM(env) str, oldChars);
	}
	return newChars;
}

Raising this defect to major since it prevent test from running.
