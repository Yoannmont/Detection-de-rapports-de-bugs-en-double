Comment 5 Wassim Melhem 2006-01-27 01:56:51 EST 
If I understand this correctly, you want to be able to freely use and reference any classes you want without making a commitment to having hard-coded dependencies in your manifest.mf until after the fact.

Currently, your plugin project's development classpath (and hence what is available to you via content assist, etc.) is restricted to what you declare in the Require-Bundle and Import-Package of your manifest.

The current workflow is the model that the tooling promotes to Eclipse users, because if the content assist was too generous (as you would want it to be), people would start referencing classes left and right, and before they know it they have a dependency on a thousand plug-ins.  Therefore, we force them to think upfront about adding a dependency and whether they need it.

Having said that, our current model may be too restrictive for people who like to do Import-Package.  So here are a few questions:

If you want PDE to augment your bundle's development classpath without requiring you to add anything to your manifest.mf, how do you expect that to be done?  do you want a subset of bundles to go on your classpath?  If so, where and how would you like to specify that subset?

do you want all bundles in your workspace and in your target (as specified on the Plugin Development > Target Platform preference page) to be available to you?  If so, that is not scalable.  Targets can have thousands of plug-ins.

When you're ready to update your manifest.mf with your dependencies, how do you expect it to be done?  explicitly via some button?  automatically?  

do you want some kind of indicator/warning on your java files if you reference a class from a bundle that is not in your manifest.mf yet?

when you want to test your bundle, you will of course get classloading errors if you have forgotten to update the manifest.mf with dependencies.  do you expect a reminder? do you expect the computation of dependencies to be done automatically prior to launching?  do you expect this task to be left to you and you would update the dependencies manually when you're ready?
Comment 6 Peter Kriens 2006-01-27 03:25:42 EST 
Key is to distinguish the development phase where you are struggling with the problem domain from the packaging phase where you struggle with things like dependencies etc. These phases are VERY orthogonal and have completely different requirements. Programmers are quickly put off when they have to do upfront work; see the discussion about PHP/Ruby versus Java. 

I realize that sometimes in the packaging phase, you have to make changes to the code, but these changes are UNRELATED to the problem domain, they just change where you place code and how you create your dependencies. In larger corporations they should be handled by different people.

My current way of working is to just use the Java Build Path to create a classpath. In a properties file I indicate for each bundle I want to make in that project what packages it should contain and export, e.g.:

   expand.mybundle.jar = com.acme.abc, com.acme.def, [org.osgi.service.http]
   [] = export

This is a lousy syntax, but you get the idea. Btool, a tool I made, will then create a JAR file containing those packages copied from the classpath. 

Note that the pattern of including an OSGi service package in your bundle is VERY common. These sources are NOT in your src directory, the classes are directly copied from the classpath, wherever they come from. I often copy libraries from the classpath because it simplifies my target deployment significantly by not creating these dependencies at the cost of some extra memory. I would not even know how to do this with Eclipse without going to ant? 

Btool will then calculate the imports by analyzing ALL the class files it selected to go into the JAR (including Bundle-Classpath), using the OSGi rules. It can then calculate the Import-Package, Export-Package, and Private-Package headers.

Versions are also handled because we always have a packageinfo file in each package directory which contains the version and other properties. This is not perfect, but it at least means that in the OSGi build we have versioned every imported and exported package. I need to think more about ranges, but I think this is a concern in the packaging phase, not the development phase.

Important in this phase is that manageing the classpath is rather lightweight, and more important, it creates minimal dependencies that can be substituted in runtime. When you start to finalize your bundle, you will need tools to look at your bundle and its siblings from 10.000 feet so you see the dependencies you created, and get help to minimize them. At this level, it should be easy to copy packages, reposition packages to other bundles, and manage version ranges.

> If you want PDE to augment your bundle's development classpath without
> requiring you to add anything to your manifest.mf, how do you expect that 
> to be done?  
The manifest is an output format that contains lots of tidbits from different places. What we need is a bundle definition file that groups all that information. The current Eclipse is treating the manifest as a parameter file which creates all kinds of unwanted constraints, which will only get worse over time because it is a key architectural decision. I currently use properties and macros in my manifest file that are replaced when I generate the JAR. This allows me to put all kinds of runtime info in it without burdening the programmer to maintain it. For example, in several bundles I need a list of the content, I use the macros to create this list.

However, this could be replaced with a bundle definition file that handles the semantics of my properties, but key is understand that the manifest is an output and does not contain all relevant information. I.e. if I want the date in a header, I can not only maintain this by hand.

So the information should be kept separate from the manifest and the manifest should be generated, as I now use the build.properties file (but it would be nicer to have another format).

> Do you want a subset of bundles to go on your classpath?  If so, where
> and how would you like to specify that subset?
In the development phase I am kind of liberal, especially early on. The Java buildpath is fine with me and is extremely well supported in Eclipse.

> Do you want all bundles in your workspace and in your target (as specified on
> the Plugin Development > Target Platform preference page) to be available to
> you?
No, because there is usually a lot of duplication. I would like to be able to specify specific bundles (or non bundle JARS!) on the classpath. I often copy packages from other JARs via the classpath.

>  If so, that is not scalable.  Targets can have thousands of plug-ins.
Right, so I do NOT want to confuse my development environment with my target environment. It would be nice to get a hint when you try to find a class what bundle/JAR/project you should add to the classpath (and do it for you). I.e. if I type HttpService, it would be great if it could show me a list of bundles/JARs/projects that I could put on my classpath.

> When you're ready to update your manifest.mf with your dependencies, 
> how do you expect it to be done?  explicitly via some button? automatically?
The manifest (and declarative services XML) should be generated automatically when you run/debug or make a JAR. It would be nice if you could easily run some script or code at that phase as well.

> Do you want some kind of indicator/warning on your java files if 
> you reference a class from a bundle that is not in your manifest.mf yet?
It would be cool to see on a package and import statement if the package will be imported, exported or private (and its version (range)!). I have to think more about this notifiers ...

> When you want to test your bundle, you will of course get 
> classloading errors if you have forgotten to update the 
> manifest.mf with dependencies.
Not when you generate it from a definition file + class files. 

> Do you expect a reminder? 
Yes, of course. Any help to prevent errors is great.

> Do you expect the computation of dependencies to be done
> automatically prior to launching?
I think you can track it in real time when you have a special definition file and generate the manifest when you run/debug or package.

> Do you expect this task to be left to you and you would 
> update the dependencies manually when you're ready?
No, I obviously want Eclipse to do it.

Key is:

 - To see the manifest as one of the many output artifacts 
   and not as the bundle definition ... 
 - Allow many forms of packaging (in multiple bundles) 
   independent of the development of the code.
 - Warn as early as possible about errors, but not too early

Thanks for listening!
