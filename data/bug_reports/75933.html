==== Ed Merks response ===
 Marcel,

When we deal with https://bugs.eclipse.org/bugs/show_bug.cgi?id=75933 we'll have
to consider this issue.  It looks like the validation will try to instantiate
the instance class in a context where that class might not be visible/loadable;
that shouldn't make the model appear invalid or cause other problems.  It would
be great if you add a comment to that bugzilla.  Perhaps we should change this
method in EClassifierImpl to return null rather than throw a wrapped exception.:

      public Class getInstanceClass()
      {
        if (instanceClass == null && (instanceClassName != null ||
generatedInstanceClassName != null))
        {
          try
          {
            setInstanceClassGen(getClassForName(getInstanceClassName()));
          }
          catch (ClassNotFoundException e)
          {
            Class primitiveClass = getPrimitiveOrArrayClass();
            if (primitiveClass != null)
              setInstanceClassGen(primitiveClass);
            else
              throw new WrappedException(e);
          }
        }
        return getInstanceClassGen();
      }

Comment 26 Ed Merks 2006-12-25 15:27:59 EST 
Created attachment 56176 [details]
Many changes to support generic bound checking better

There is an Example.ecore model showing the types of bound checking that is being done.  The idea being that the generated code for this model should have exactly and only the errors reported in the Ecore model itself. This turned up what appear to be a few problems in the Eclipse compiler reporting bogus errors:
  https://bugs.eclipse.org/bugs/show_bug.cgi?id=169049

Part of the changes here include removing the ETypeParameter.eGenericTypes reference (the inverse of EGenericType.eTypeParameter). The inverse is still maintained internally, but not as a notifying feature.  This allows one to copy things containing references to type parameters without losing them and avoids problems in the future where one might like to construct a resolved generic type including making references to ETypeParameters without having to worry about the inverse holding a reference forever.
Comment 27 Ed Merks 2006-12-28 05:44:35 EST 
The code is committed to CVS.  I believe it will diagnose all the problems reported in the duplicates before the generator tries to generate any code.  If there's anything we've missed, please bring them to our attention.  Note that changes in the runtime include making it tolerant of circular inheritance (no stack overflow), make EAttribute.eAttributeType and EReference.eReferenceType avoid class cast exceptions when the type is set incorrectly, and making EClassifier.instanceClass return null instead of throwing an exception when the instance class name can't be resolved to a class.
