Description Marcio 2003-04-16 15:48:10 EDT 
For my language I'd like to have the currently executing line colored 
depending on state from my VM. I'd imagine that passing such text attributes 
when generating these debug events would be one way to go.

Note that is is a spin-off from Bug 33598
Comment 1 Darin Wright 2003-04-21 14:33:52 EDT 
Generllay we do not promote the inclusions of UI-like presentation attributes 
in core debug models (that is, the debug model and the debug UI are in seperate 
plug-ins). Rather, the debug model contains attributes that the UI interprets 
to create a presentation. Adding information to you debug elements or debug 
events that simply describe more details is fine.

Could you add attributes/info to you debug elements to handle this? Perhaps the 
stack frames that are associated with the text?

I could envision adding a user-data (Object) field to the debug event where 
debug models can append any debug model speicfic data.
Comment 2 Marcio 2003-04-21 14:41:40 EDT 
I agree with separating core from gui. My core, which generates debug events, 
knows the VM state. I could tag the Debug event with this state (an Integer 
actually).

Then if you provide some debug model presentation API (maybe 
IDebugModelPresentation) so that I get asked for presentation colors etc for 
the executing line of text about to be highlighted, I could compute the color 
based on the VM state code (Integer above) that you could pass back to me, to 
this API. So, my core passes the Integer along with the debug event ("user 
data") and you pass it back to my IDebugModelPresentation before you do any 
highlighting, and then you ask me for the text attribs to use.
Comment 3 Darin Wright 2003-07-09 16:28:51 EDT 
I propose to solve this by extending the new IDebugEditorPresentation interface 
to allow a model presentation to decorate an editor for a stack frame. When an 
editor is opened by the debugger, we will call #decorateEditor(IStackFrame, 
IEditorPart) on the editor presention. This will give clients a chance to 
perform any model specific rendering they wish.

The debugger will need to know if it should highlight a line of source code or 
not (when dealing with a text editor). In that case we should first consult the 
editor presentation on whether the debugger should "select and reveal" a line 
of source before doing it.

Comment 5 Patrick Baker 2003-07-09 17:19:29 EDT 
I just posted a comment (Comment #26) for bug #33598 that really should have 
been posted here. Oh well:(

Basically, why not solve this by having a call back that asks for a text 
selection. Right now, if I understand things, a plugin developer has an 
oppurtunity to tell the debug core what the file and line is for the 
instruction that is about to be executed. We need to say a bit more. We need to 
be able to tell the debug core what file and what text selection to use to 
indicate the instruction.

The properties of a text selection are:
  start line
  start column
  end line
  end column
  foreground colour
  background colour

This is not as generic as Darin's solution, but maybe it is good enough. It is 
simpler, and the folks writing the language plugin don't have to worry about 
how to paint an editor. They just return values for the 6 properties above, and 
the debug core will handle the painting.
Comment 6 Darin Wright 2003-07-09 17:30:06 EDT 
Fixed - added two methods to IDebugEditorPresentation - selectAndReveal, and 
decorateEditor. This allows debug implementors to perform any custom 
selection/coloring they desire. Since an ITextEditor or IEditorPart has no API 
for setting foreground/background colors, the debugger cannot support it 
generically. If a debug model is coupled with an editor (which is more likely), 
then it may know the internals of the editor, and can do as it pleases.

Comment 11 Darin Wright 2003-07-14 15:41:26 EDT 
Fixed. I have added the following method to the editor presentation. The method 
is called when a source selection/decoration needs to be cleared when a thread 
resumes or terminates (or when the associated debug target terminates). The 
thread is given (instead of the stack frame the editor was opened for) since 
the stack frame may not exist anymore (when a thread resumes/terminates, it 
does not have stack frames). I'm still working on an example for clients to 
see, but the support is now present.

/**
 * Removes any debug related decorations in the given editor. This method is
 * called when the debugger clears the source selection in an editor opened
 * by the debugger when a debug session is resumed or terminated.
 *   
 * @param editorPart an editor that was decorated 
 * @param thread the thread the editor was decorated for
 */
public void removeDecorations(IEditorPart editorPart, IThread thread);

Comment 20 Darin Wright 2003-11-21 14:54:11 EST 
Note: In Eclipse 3.0M5, the debugger uses background highlighting to show the 
current line of execution in text editors (and for other stack frames on the 
call stack). The debugger contributes "marker annotation specifications" for 
the instruction pointer markers. The following attributes of the rendering can 
now be controlled via preferences (see Text Editor and Java Editor annotation 
tab preferences):

* background highlight color
* whether an icon is displayed in the vertical ruler
* when an icon is displayed in the overview ruler
Comment 21 Patrick Baker 2003-12-01 14:32:35 EST 
I believe that there is a serious problem with the current
implementation. It works fine on Windows, but not on Linux and Solaris
(and quite possible other Unix platforms as well).

On Windows, when I step, removeDecorations() is called and the
selectAndReveal() is called. On Linux, when I step, selectAndReveal()
is called and then removeDecorations() is called.

The net result for me is that the current active line flickers on
Linux and Solaris. This happens because I highlight the selection in
selectAndReveal(). Then, presumably after the user has pressed step, I
remove the text selection in RemoveDecorations(). Because the order in
which these two methods are called is backwards on Linux and Solaris,
my current active line is selected and then deselected immediately.

I have include the call stack for when removeDecorations() is called
by the Eclipse platform on Windows and Linux. Notice that the two are
quite different. The call to removeDecorations() on Windows is a
response to a 'resume event', which makes alot of sense. But this
