Comment 2 Florian Priester 2005-11-24 16:02:30 EST 
(In reply to comment #1)
> I had a go at fixing this but I couldn't get Windows to cooperate.

Out of curiosity, what problem did you encounter? Just to see if it's the
same one that I came across. I gave it a try myself and couldn't do it either
(which isn't saying much, though).

What I tried was handling WM_SETCURSOR for hwndHeader and using HDM_HITTEST
to find out what part of the header the pointer was over in order to prevent
Windows from showing a resize cursor if the result indicated HHT_ONDIVIDER
or HHT_ONDIVOPEN for a non-resizable column. The reason it did not work was
that the hit test results did not exactly match the area for which Windows
decided to display a column-resize cursor. I believe it's a bug in Windows.
(The same effect can be reproduced in the Windows Explorer list view.)

Another approach might be to always set your own cursor based on the hit test
flags but that would make it necessary to somehow obtain the right images for
the column-resize and column-reveal cursors. They do not seem to be among the
IDC_* ones, however. The only idea I have left is to disable the resize cursors
for the entire header area for the special case that the user has set /all/
columns to "non-resizable". Not sure how often that happens and whether it's
really worth it.

Comment 4 Florian Priester 2006-04-05 01:19:14 EDT 
Another attempt. If the approach "veto resize cursor over divider of
non-resizable column but allow the rest" doesn't work, then how about
"allow resize cursor over divider of resizable column but veto the rest"?

That's what the code below does. It's not quite a finished patch but shows
how it could be done. I've used a RECT for the hit test since SWT does not
(yet) have a HDHITTESTINFO structure.

---

case OS.WM_SETCURSOR: {
    final int HDM_HITTEST = OS.HDM_FIRST + 6;
    final int HHT_ONDIVIDER = 0x4;
    final int HHT_ONDIVOPEN = 0x8;
    POINT pt = new POINT ();
    OS.GetCursorPos (pt);
    OS.ScreenToClient (hwndHeader, pt);
    RECT rect = new RECT (); // should use HDHITTESTINFO
    rect.left = pt.x;
    rect.top = pt.y;
    int index = OS.SendMessage (hwndHeader, HDM_HITTEST, 0, rect);
    int flags = rect.right;
    if (index != -1
            && index < columnCount
            && columns [index].resizable
            && (flags & (HHT_ONDIVIDER | HHT_ONDIVOPEN)) != 0) {
        break;
    } else {
        OS.SetCursor (OS.LoadCursor (0, OS.IDC_ARROW));
        return 1;
    }
}

---

import org.eclipse.swt.*;
import org.eclipse.swt.layout.*;
import org.eclipse.swt.widgets.*;

public class TableHeaderTest {
  public static void main(String[] args) {
    Display display = new Display();
    
    Shell shell = new Shell(display);
    shell.setLayout(new FillLayout(SWT.VERTICAL));
    
    // create several tables to test the various scenarios
    createTable(shell, new boolean[] {});
    createTable(shell, new boolean[] {false, false, false, false});
    createTable(shell, new boolean[] {true, false, true, false});
    createTable(shell, new boolean[] {true, true, true, true});
    
    shell.setSize(450, 400);
    shell.open();
    
    while (!shell.isDisposed()) {
      if (!display.readAndDispatch()) display.sleep();
    }
    
    display.dispose();
  }
  
  private static Table createTable(Composite parent, boolean[] resizable) {
    Table tbl = new Table(parent, SWT.BORDER);
    tbl.setHeaderVisible(true);
    tbl.setLinesVisible(true);
    
    for (int i = 0; i < resizable.length; i++) {
      TableColumn col = new TableColumn(tbl, SWT.LEFT);
      col.setWidth((i == 0) ? 0 : 100);
      col.setMoveable(true);
      col.setResizable(resizable[i]);
      col.setText("" + i + (resizable[i] ? ": Resizable" : ""));
    }
    
    return tbl;
  }
}

Comment 9 Steve Northover 2006-08-31 18:38:47 EDT 
Fixed > 20060831

Ok, I finally got to look at this and the code was pretty good.  The changes I made were to change WM_SETCURSOR check for the header HWND and for HTCLIENT before hit-testing the header for the divider.  Also, instead of setting the cursor, I call the DefWindowProc() to pass the WM_SETCURSOR message up the parent chain, avoiding the window proc for the header.  Although the old code worked, the new code is safer because this is the normal behavior when a WM_SETCURSOR is not handled.
