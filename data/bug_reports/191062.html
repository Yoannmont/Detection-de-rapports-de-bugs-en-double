Comment 7 Adam Archer 2007-06-06 10:25:44 EDT 
Created attachment 70330 [details]
Patch

Good catch Curtis.

I looked through the code to see how we are doing this elsewhere. For now I just copied what appeared to be the most elegant method. I also noticed that there is a util method (UrlUtil.getLocale) that will give the appropriate NL string based on help system mode, the result from Platform.getNL and all sorts of other things. I switched to use this for consistency. Unfortunately, if it gets the locale from Platform.getNL, it turns it into a String and I had to switch it back. I think it is worth the overhead to ensure that we are using the same locale everywhere.

In 3.4 however, I think we should try to clean up the Locale situation. We have several different versions of the same conversion from String to Locale in help alone. I can only imagine that other plug-ins have to build a Locale from Platform.getNL() also and are likely duplicating similar code even further. We should get org.eclipse.core.runtime.Platform to provide a getNLLocale() method or something to that effect in 3.4. We should then also add a UrlUtil.getLocaleLocale (perhaps with a better name) that returns a Locale type instead of a String. Then we can find all the places in help where we aren't using this util class and fix things up.

I also noticed, when testing, that some characters (with accents, etc.) were not rendering correctly in the dynamic content, but were in the documents and the table of contents. We were using UTF-8 character encoding to generate the content. I checked the translated docs to find they are using ISO-8859-1. After switching the encoding type, the characters now render correctly as well. Unfortunately this has the bizarre side effect of breaking these characters in the breadcrumbs for the dynamic content. So far I've been unable to figure out why. If anyone can, then by all means, fix my patch. If not, then our choices are incorrectly rendered breadcrumbs or incorrectly rendered content and I think this change provides the lesser of two evils.
Comment 8 Adam Archer 2007-06-06 10:45:46 EDT 
Created attachment 70334 [details]

Curtis helped me figure this one out. The problem was not the character encoding, it was that the PrintWriter in NavServlet was not using the specified encoding, it was using the default.

Switched back to UTF-8 and told the PrintWriter to use this encoding as well.

Comment 12 Adam Archer 2007-06-06 19:24:13 EDT 
Created attachment 70434 [details]
Alternative Patch

Actually, the existence of that method was an oversight on my part. I just looked at it, however, and it does the same thing that Curtis expressed a concern for in comment 6.

This alternative patch uses this method but updates it to create the locale using the StringTokenizer mechanism as in the previous patch.

The only possible concern here is that we are changing the functionality of a method that is used elsewhere. The fact that it is used elsewhere does suggest that it works though. It seems to me, however, that the previous functionality was incorrect based on the JavaDoc for Locale. The fact that it was working is probably a fluke (I'm guessing that the Locale constructor is just hacking off anything other than the first two characters, but can't say for sure).

Given that it was working, though, I think we have three options here:

1) Use this new patch as is and correct the functionality of the getLocaleObj() method. Advantage being correct usage of the constructor. Disadvantage being very minor possibility of a regression by changing the method functionality.

2) Use this new patch without the changes to getLocaleObj(). Advantage being less code than either of the other patches. Disadvantage being that we risk poor functionality of the misused constructor affecting the NavServlet fix.

3) Use the previous patch. Advantage being that we know it will work well and has no risk of causing a regression. Disadvantage being code duplication.

Strictly speaking I think (3) is the safest, but personally I would vote for (1) as all my testing has shown that the StringTokenizer method works very well in all cases and the old implementation of getLocaleObj seems to contradict the Locale JavaDoc.
