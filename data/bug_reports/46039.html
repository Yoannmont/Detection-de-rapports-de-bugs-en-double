Description John Arthorne 2003-11-04 12:11:31 EST 
Eclipse 2.1 and 3.0

1) start with a very large workspace, autobuild is on
2) rebuild all (say it takes 8 minutes)
3) Select a project with many dependents, and run "rebuild project"
4) Select any single file and modify a comment

-> Auto-build after step 4 takes several minutes

The problem is that the user has made a conflicting request.  They have asked
for a single project to be rebuilt, thus invalidating the build state of all
downstream projects.  However, autobuild is on, which requires that all projects
remain in a completely built state.  If we obey the user and only build the
single project in step 3, then the next change will trigger a large build.

Possible solutions:

a) Remove the "rebuild project" menu action when autobuild is on
b) Automatically build downstream projects after an explicit project build when
autobuild is on. I.e., the 4 minute build would happen after step 3 instead of
step 4.

What I like about a) is that it prevents the user from doing something that they
probably didn't mean to do.  Rebuilding a single project from scratch when
autobuild is on generally doesn't make sense (unless an incremental builder is
misbehaving or they want to trigger an external tool builder).  Someone in the
discussion suggested pairing this with a more accessible means of toggling
autobuild, so users can more easily do manual single project builds if they need
to by turning off autobuild.

CCing Nick, Mcq and Philippe for opinions.

Comment 5 John Arthorne 2003-11-05 16:51:20 EST 
Actually, I was planning on changing core to "do the right thing" when autobuild
is on.  That is, if you rebuild a single project when autobuild is on, it will
incrementally build any dependents.  Users understand autobuild as "my workspace
is always in the right state", so I believe we should eliminate this one exception.

The remaining UI issue is that based on feedback from a number of channels, the
meaning of "Rebuild" is misunderstood.  Users don't understand that this is a
destructive operation that wipes out all built state (thus invalidating all
downstream projects).  One solution would be an "are you sure" dialog combined
with "don't ask me again", that describes the implications of the action.

Comment 9 John Arthorne 2003-11-06 12:48:31 EST 
This is a response to Philippe's comment #6.

Scoped autobuilds are an unrelated proposal - it can't be achieved with this
current API. The reason is that if autobuild is on, as soon as the user makes
any change after rebuilding a single project, then all downstream projects will
be rebuilt (regardless of the scope of the rebuild).  The user pain of
downstream projects being rebuilt is simply deferred until the next change (see
the original steps at the start of the bug). I think we need to tackle scoped
builds as a separate issue.

Whether autobuild is on or off currently doesn't make a difference, but I
believe it should. User expectation when autobuild is on is that the world is
"always" in a built state.  Single project rebuild is the only method we have
that doesn't obey this rule.

Comment 12 Nick Edgar 2003-11-11 22:47:30 EST 
When a project is rebuilt, and you then force an incremental build (whether due 
to autobuild due to some other change, or by pressing Ctrl+B), do the 
downstream projects get built incrementally?  I thought that downstream 
projects would have to get fully rebuilt as well since there's no delta from 
the prerequisite project that was rebuilt.

Or did you have some ideas on improving on this, John?  E.g. could Core 
communicate to the Java builder to do a full rebuild, but still generate fine-
grained deltas?  For example, most compiler options do not change the principal 
structure of a type (e.g. NON-NLS warnings should have no impact on downstream 
types), so changing these on a project should have no impact on downstream 
projects.  At one point the builder was smart enough to recognize this, but I'm 
not sure the current builder can (it has coarser-grained dependencies).
Comment 13 Philipe Mulet 2003-11-12 05:11:30 EST 
The compiler compliance level may impact dependent projects, since may include 
default abstract methods (1.3) or not (1.4).

However most setting adjustment are usually not changing much. As of today, 
dependent projects are not rebuild, simply incrementally build against the 
rebuild project which looks like structurally changed on each classfiles (since 
when rebuilding a project, we never bother computing the actual diff as this is 
not an obvious scenario).
Comment 14 John Arthorne 2003-11-12 09:57:53 EST 
To confirm Philippe's comment, downstream projects are built incrementally and
they obtain a delta as usual.  If the java builder optimized the full build to
avoid writing class files that were not structurally changed, then I believe
downstream builds would be fast.  Full build only happens on explicit request,
and on a few corner cases such as natures going missing, upstream projects being
closed when a downstream project is already closed, and on the first build of a
project.
Comment 15 Philipe Mulet 2003-11-12 12:30:47 EST 
In order for full builds to notice the lack of structural changes, we would 
have to keep old classfiles around for the duration of the build, which could 
have some disk footprint ramifications... point is well-taken though, but would 
be a big change in our land.

A more reasonable solution is to ensure that if settings are changed, then the 
workspace is left consistent after closing the UI dialog.
Comment 16 John Arthorne 2003-11-12 15:50:34 EST 
I have released the core change (bug 46204) for build 3.0 M5.  We are no longer
supressing autobuild after a single project build or rebuild.
Comment 17 Dirk Baeumer 2003-11-17 11:08:55 EST

For 3.0, JDT/Core should handle compiler setting changes like class path

Comment 24 John Arthorne 2003-11-21 11:36:36 EST 
The discussion got a bit side-tracked by the compiler preference issue, but the
main intent of this bug was that users did not understand that "rebuild" is a
change that discards all built state, forcing a rebuild of all downstream
projects.  Users were calling rebuild "just to ensure everything is consistent"
without realizing how expensive it is.  See also comment #2.  I suggested in
comment #5 that we add a simple warning with "don't ask me again" semantics,
something like:

"This operation will discard all built state and rebuild all affected projects
from scratch.  Proceed? Yes/No"

Comment 26 Dirk Baeumer 2003-11-24 13:30:34 EST 
I am a little bit puzzled about what we do for 2.1.x and what we do for 3.0. 
My understanding is:

3.0: 
  - core no longer supresses autobuild after a single project build or rebuild
    (see #16).
  - ui will add a dialog telling the user that "rebuild" is more expensinve 
    than he thinks.
  - jdt core will "automatically" compile all needed projects on compiler
    setting changes
  - jdt/ui no further action required

2.1.x (so far):
  - jdt/ui provides a patch which triggers an incremental build when the
    user changes the compiler settings and confirms the rebuild of the project.

IMO this will result in an inconsistent behaviour for 2.1.x. When triggering a 
rebuild via Project->Rebuild downstream projects aren't incrementally build. 
If the user changes the compiler settings and confirms the rebuild downstream 
projects are incrementally build.
Comment 27 John Arthorne 2003-11-24 17:43:03 EST 
I agree this is inconsistent.  The platform core change is very small and would
be quite easy to also backport into 2.1.x.  However, since this affects the
behaviour of API that may be used in a broad variety of situations it feels
risky to change in a maintenance release.  For example it could change the
behaviour of headless build scripts.  Since the change is quite subtle,
downstream components might not notice the change, although they would pay a
signficant performance hit. Our general rule in platform core is to only release
critical fixes in maintenance releases. The platform UI warning dialog could be
a candidate for 2.1.x though.
