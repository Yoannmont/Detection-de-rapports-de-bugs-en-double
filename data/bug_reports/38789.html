Description Adalbert Homa 2003-06-11 13:08:45 EDT 
This is for M1.
I assigned a key to "Show in Package Explorer" (Ctrl+Alt+O).
I open a Java file and press the key, the file is show in the Package Explorer.
I open a text file and press the key, nothing happens. However if I select from
the menu Navigate -> Show In -> Package Explorer, the file is show in the
Package Explorer.
Comment 1 Erich Gamma 2003-07-07 05:39:36 EDT 
Show in Package Explorer is a custom action in the JavaEditor and it isn't the 
same as Navigate -> Show In -> Package Explorer. When you bind Show In Package 
Explorer, then the binding only works for the Java editor. I agree that this 
is confusing since there is no indication anywhere.

There should be a way to bind the Show In actions to a particular key.

Moving to Platform UI.
Comment 2 Douglas Pollock 2003-09-16 16:44:11 EDT 
On I20030916, Navigate > Show In > Package Explorer is empty for files in the
text editor.  I assume this means that the action has never been implemented for
the text editor.  If it is implemented in the text editor (and provided as a
RetargetAction), then the key binding will work.

Punting to platform-text.  Changing to enhancement.
Comment 3 Dani Megert 2003-09-17 05:24:50 EDT 
>On I20030916, Navigate > Show In > Package Explorer is empty for files in the
>text editor. 
Douglas, Show In mechanism is configurable per perspective. Try it out using the
Java perspective instead of the Resource perspective. The
ShowInPackageViewAction for which users can define a key binding is a workaround
for the problem that we cannot assign shortcuts to the Platform's Show In
mechanism contributed by:
	<perspectiveExtension 
		targetID="...">
		<showInPart id="..."/>
	</perspectiveExtension>
Comment 4 Douglas Pollock 2003-09-17 17:10:00 EDT 
Sorry.  The behaviour I was seeing occurs in I20030916, but not in I20030917. 
It would not populate the show-in drop-down menu if it was the first text editor
opened.

This can be fixed by platform-text hooking a local handler on to a global
command defined for "show in package explorer".

However, I think you're right that there should be a capacity to bind to these.
 Even though they are dynamically generated by platform-ui, they tend to be
static within the application.  Therefore, it is reasonable for a user to want
to bind keys; these things won't change very often.

To do so, we'd could dynamically generate commands using
IViewRegistry.getViews().  The identifier could come from
IViewDescriptor.getID(), with some kind of prefix to prevent collisions.  The
name could be created using the IViewDescriptor.getLabel().  A special category
could be used to contain these names -- shielding them from conflicts with other
commands in KeysPreferencePage.

The big trick would be getting persistent key binding preferences to play nicely
with dynamically-generated commands.  I think that this should be as simple as
invalidating the key machine after the new commands are dynamically generated,
but I need to double-check.

Also it looks like this functionality might be duplicated already in a few
places.  Look for classes containing "ShowIn"....

Comment 26 Paul Webster 2008-09-12 08:13:47 EDT 
The general case is already available in 3.4.  You can create keybindings in the Keys preference page for Show In (Target: view) commands.

It could use a little refinement, however (it shows all the views even if they are not IShowInTargets).

PW
Comment 27 Markus Keller 2008-09-12 08:46:36 EDT 
> (it shows all the views even if they are not IShowInTargets).

Note that views don't have to implement IShowInTargets. They can also return it on IAdaptable.getAdapter(IShowInTarget.class). I don't think you can filter out views, at least not those which have not been loaded yet (because the adapter does not have to be declared in the plugin.xml).

The actions in the 'Navigate > Show In > ...' and the 'Show In > ...' quick menus should also show the keybindings.
