Description Kenn Hussey 2005-07-26 13:55:04 EDT 
Provide support for constraints specified in OCL. Requires EMF support for OCL 
(e.g. an OCL parser/interpreter). Consider providing a convenience method on 
Constraint for returning the parsed representation of OCL expressions.
Comment 1 Dave Carlson 2005-09-26 09:27:38 EDT 
Would IBM Rational consider contributing the current Default profile used in 
RSA?  The constraint metadata on stereotypes in that profile is related to OCL 
support, specifically to localize error/warning messages, categorize 
constraints, and assign as live/batch.

Also, the use of documentation and URL stereotypes in that profile are 
generally useful.  As it is now, it's impossible to create a model outside of 
RSA and then import with documentation intact in the properties view.
Comment 2 Kenn Hussey 2005-11-28 10:55:15 EST 
This profile is proprietary, and its purpose is to support product-specific conventions. I'm not sure it's such a good idea to adopt such conventions in UML2 unless they are widely accepted and generally useful (i.e. to others not using or importing into RSA).

Comment 13 Kenn Hussey 2007-05-08 15:57:39 EDT 
A convenience method on Constraint for returning the parsed representation of OCL expressions is not needed since clients can use instances of org.eclipse.ocl.uml.ExpressionInOCL instead of org.eclipse.uml2.uml.OpaqueExpression as the specifications of their constraints.

Support for dynamically evaluating stereotype (or other) constraints will not be provided (at least not in UML2) since this would introduce a circular build dependency on MDT OCL. Besides, this a tool-level capability that would be more appropriately provided by the UML2 Tools component of MDT (seeing as it already depends on both UML2 and OCL).
