Description Olivier Thomann 2006-11-03 15:34:43 EST 
Using I20061102-1715, I have the following problem.
I have:

1) one plugin that has a bundle-classpath like this:
lib.jar,.
'.' corresponds to two output folders (bin1 and bin2)

2) a fragment based on that plugin that provides a lib.jar file. Since lib.jar is first on the bundle-classpath, I expect the classes inside the jar to be selected first.
The jar contains types that are also available in the plugin '.' path through bin2.

I debugged this test case and the entries for this plugin are:
bin1
bin2
lib.jar

This means that my "new" version of the classes (in lib.jar) are never picked up because they are always found in bin2.
I would say that the dev classpath must respect the order specified in the bundle-classpath. lib.jar should definitely added before the two output folders from the plugin.

This is a pretty serious problem for me since I cannot self-host my actual setup. I will need to export the plugin and the fragment to a new target and run from that target hoping to be able to debug it.

Comment 7 Brian Bauman 2006-11-15 12:27:14 EST 
I wanted to make sure to understand this completely before starting to try to fix anything.  Olivier's report does not say whether or not the lib.jar is a binary jar or in compiled in the fragment's source folder.  If it is a source folder, I think things are not as complicated.

Just to make sure we know the limitations, I believe we can run into a misordered dev.properties classpath with the fix from 108466.  If the user creates a plugin with a "Bundle-Classpath: lib.jar, ."  where lib.jar is a binary jar in the plug-in porject. Just wanted to make sure I understand what is going on as much as possible :)
Comment 8 Wassim Melhem 2006-11-15 12:56:08 EST 
Brian,

The point here is that there is no output folder corresponding to library.jar.

Here is how I picture it would work:

Given a "Bundle-Classpath: library.jar, .", here is how the dev.properties value for the plug-in should be composed:

1. Look in the build.properties for a source.library.jar key.  If it exists, then you put the output folders corresponding to the source folders on the classpath and move on to the next library (ie. the dot).  If no source.library.jar exists, you look for a physical library by that name in the plug-in.  In Olivier's case, you will find it and it will therefore be put first on the classpath.

2. Move on to the next entry in Bundle-Classpath (the dot) and do the same as in #1.
Comment 9 Brian Bauman 2006-11-15 14:54:32 EST 
After conversing with Tom and Wassim, the algorithm proposed is similar to Wassim's comment above.

To compute a Plug-in Projects dev.properties we read the Bundle-Classpath to find the libraries and do the following algorithm:

1. Search the plug-in project's build.properties to find a source entry.  If found, add to dev.properties

2. If there is no source entry, look for a binary library representing the library.  If found, do NOT include it in the dev.properties.  This will prevent reintroducing bug 108466.

3. If we can't find a source entry or physical file, find all fragments included in the launch configuration.  For each fragment, look in the build.properties for source entry.  If it exists, add it to the plug-in's dev.properties.  If it does not exist, look for the physical file in the fragment's project.  If it exists, add it to the plug-in's dev.properties.  If not, move on to next fragment.

Drawback: Since we don't check if a fragment is resolved, we could include a reference to an unresolved fragment (a linux fragment in a win32 env.).  To check to see if a fragment would resolved when launched could be time consuming (possibly requiring creating a resolver state whenever we launch).

Before I start doing too much work, I wanted to make sure everyone was in agreement with this algorithm.  Please comment away :)
Comment 10 Thomas Watson 2006-11-16 09:23:28 EST 
Should you only do step 3 for fragments that have the "Eclipse-PatchFragment: true" header?  This would reduce the concern over placing jars from random unresolved fragments on the host classpath.

In step 1 you should continue to use relative paths in the dev.properties when an output folder is found in the host (e.g. bin).

In step 3 you will need to use absolute paths to the output folder or binary library of the fragment (e.g. /myworkspace/my.fragment/bin)
