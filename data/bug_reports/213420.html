Comment 5 Eike Stepper 2007-12-20 04:58:37 EST 
I think I found an important point:

In org.eclipse.emf.edit.provider.ItemProviderAdapter there is:

  public boolean hasChildren(Object object)
  {
    return !getChildren(object).isEmpty();
  }

Although this could eventually made a bit smarter the main issue is in org.eclipse.emf.edit.provider.ItemProviderAdapter.getChildren(Object):

        List<?> children = (List<?>)eObject.eGet(feature);
        int index = 0;
        for (Object unwrappedChild : children)		<==== Iterator loads all child objects!!!
        {
          Object child = wrap(eObject, feature, unwrappedChild, index);
          if (store != null)
          {
            store.getList(feature).add(child);
          }
          else
          {
            result.add(child);
          }
          index++;
        }

This is clearly the reason why more objects are loaded than would be necessary for the immediate deeper level that is to be expanded in the editor (view). I fear I have to discuss that on the EMF newsgroup.

Apart from this issue there are still the points I gave in the email:

Further I realized some performance issues with the DBStore/MappingStrategy:

- Particularly the HorizontalMappingStrategy doesn't require (by definition) a central object table which could be used for id -> type lookups. When I implemented that I decided to let the repository framework (TypeManager) help those stores with hasEfficientTypeLookup()==false. It turned out that it was not a good idea. I'll move this burden back to the store. There I'll provide a central lookup table (cache) that is asynchronously fed through a QueueWorker thread. Each MappingStrategy is prepared anyway to handle cache misses by looking up the id withing the mapped object tables. This can be slow for horizontal mappings and it seems to happen regularly in the current scenario.

- I have not measured/compared times but I have the feeling that tracing to the Eclipse console costs a lot of time.
Comment 6 Eike Stepper 2007-12-20 05:48:46 EST 
Good news:

I was able to fix the main issue with ItemProviderAdapter by adding a CDOItemProviderAdapter base class to the CDO framework with a more efficient hasChildren() method:

  @Override
  @SuppressWarnings("deprecation")
  public boolean hasChildren(Object object)
  {
    Collection<? extends EStructuralFeature> anyChildrenFeatures = getChildrenFeatures(object);
    if (anyChildrenFeatures.isEmpty())
    {
      anyChildrenFeatures = getChildrenReferences(object);
    }

    EObject eObject = (EObject)object;
    for (EStructuralFeature feature : anyChildrenFeatures)
    {
      if (feature.isMany())
      {
        List<?> children = (List<?>)eObject.eGet(feature);
        if (!children.isEmpty())
        {
          return true;
        }
      }
      else
      {
        if (eObject.eIsSet(feature))
        {
          return true;
        }
      }
    }

    return false;
  }

It's also possible to inject this new class into the generated itemprovider hierarchies with a genmodel property (could add this to the migrator). Unfortunately I don't currently see a way to make the generated edit plugin depend on org.eclipse.emf.cdo.edit ;-( Asked that on the newsgroup. Even better would it be if Ed reviews my hasChildren method and is willing to patch EMF (for 2.3 and 2.4)!

Now we have the following characteristics when using a tree viewer:
1) Only the objects of the level to be expanded are loaded (duration is better related with the visible result)
2) Currently there is no way to load all the needed child objects with only 1 Net4j request. I'll look into my code if I can find a location where I could optimize this without knowing about the EMF edit framework.
