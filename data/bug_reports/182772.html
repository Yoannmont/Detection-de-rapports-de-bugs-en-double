Description Mik Kersten 2007-04-17 11:26:55 EDT 
The layout of the Task List could be improved in order to make the considerably amount of visual information being presented easier to view and to better support hierarchical structures.  For example, with the current use of columns showing subtasks would require a considerable expansion of the first column, which will waste more horizontal space.
Comment 1 Mik Kersten 2007-04-17 11:30:38 EDT 
One idea we had for this is to make the tree only have two columns.  The first would have everything but the current "incoming" indication column, and we would deal with the current use of columns for images by making a fancier decorator that handled everything.  We would then right-align the "incoming" column so that those images showed to the right of the Summary labels and would be easy to pick out.

Ben: do you know if this kind of right-aligning is possible?  If not I guess we could do it by setting the width of the main column whenever a resize happened?  The idea is that we would never want to get a horizontal scrollbar, since those waste space.

Comment 13 Mik Kersten 2007-04-26 21:40:13 EDT 
 (In reply to comment #12)
> Is it really? I wouldn't say so considering how many old features it breakes.

As I said: "I've committed my current progress".  Just got off the plane and committed the proper handling of images that are wider than 16px, that was tricky to get right but its mostly there.  

> Anyways, do we really have to change how tree component worked and what is the
> driver for this?

Subtasks (bug 137543) and alternate content models (bug 170681) are key drivers, as they require us to support another level of nesting in the Task List, which we can't do with the current leading columns based approach because it would cause a huge amount of whitespace at the start of the Task List.  The other driver is the various bugs that suggest us combining tasks and hits (e.g. bug 161734), because one key thing I want to accomplish here is make incomings prominent enough that we get rid of the distinction between tasks and hits and just have "new tasks" indicated clearly enough, as you have been suggesting for ages.  I'm about to commit a first try at UI for that soon.
