Comment 5 David J. Orme 2002-06-05 17:04:07 EDT 
(not sure which Dave you want to comment (there's 3 of us now)) :-)

Here are the newsgroups I propose:

eclipse.announce
eclipse.installing
eclipse.using
eclipse.plugins.developing
eclipse.standaloneswtapps

Reasons--

Announce:
- I think that having a separate newsgroup for announcements is (1) both very 
conventional and (2) a good idea for all the standard reasons.

Installing:
- Every time a new major build happens, there are a slew of 

/[V|F|M]?[0-9]+/ doesn't work of foosplat platform

threads.  A significant minority of us do not upgrade to the every bleeding-
edge version of Eclipse and would prefer not to wade through these messages.  
Conversely, when those of us who lag the bleeding-edge do get around to 
upgrading, it would be helpful to have everyone else's tribulations 
conveniently indexed in one place, since others would likely have already run 
into anything we would.  This also would cut down on redundant posts of the 
same problems on the same platforms.

Using:
- Eclipse.using needs its own group, since a large portion of these messages 
seem to be related to

Feature xyz /"doesn't work"|"was moved and I don't like it"|"is incredibly 
wonderful"/

messages.  While these are useful, my main purpose (and that of a lot of 
others, I hope) is to build new cool plugins.  Thus I sometimes would like an 
easy way to ignore these messages for a few days when I'm really busy working 
something, then come back to them later when things slow down.

StandaloneSWT:
- About every other week it seems we get someone asking how to use SWT/JFace 
for building standalone apps.  Less frequently (but still redundantly) we get 
messages about integrating Swing with SWT.  People would be more likely to find 
the old threads if they were in a common location.

Plugins.Developing:
- Plugin development tends to be enough different from the above that I think 
it warrants its own area.

Comment 20 James Moody 2002-06-07 13:50:08 EDT 
Making a distinction between mailing lists and newsgroups based on the type of
conversation that occurs there does *not* make any sense. I also dislike the
idea of mirroring between mailing lists and newsgroups; it seems like a
duplication of resources and bandwidth. Why won't one of the two suffice?

Of all the open source projects that I've observed, only Mozilla makes use of
newsgroups, and they apparently do mirror to mailing lists. Nearly every other
project uses mailing lists exclusively, with no apparent problems.

I have a personal preference for mailing lists, but I'm obviously not going to
convince those who do not. Nearly all of the arguments I've heard boil down to
client (mail & new reader) programs... (i.e. I prefer newsgroups because Lotus
Notes doesn't let me X... or I prefer mailing lists because Forte doesn't let me
Y). The REAL difference between mail and news is separate from any client
issues, and is based on delivery mechanism.

Comment 30 Andrew McCullough 2002-06-24 16:40:53 EDT 
Just my $.02:  I don't think we should try to go 0-60 in 1.2 seconds.  Right 
now, we have a problem:  A cluttered newsgroup with too many different kinds of 
posts.  I would propose this be fixed by adding new newsgroups for relatively 
broad categories, then they can always be subdivided later.  Lets do a little 
Extreme NewsGrouping and go about this "refactoring" iteratively.  I would 
propose replacing eclipse.tools with 3 newsgroups:

eclipse.user
    - someday could be user.install, user.whatever)

eclipse.development 
    - I see no need for "plugin" here -- besides, maybe you just want to ask a 
question only an eclipse developer would know the answer to.  Could be 
subdivided if needed..

eclipse.swt or eclipse.gui 
    - the latter might make JFace seem more "at home" - but either one would 
get the job done IMHO.  I prefer .swt, just because like it or not, people 
think of SWT as the "Eclipse GUI" even if it's technically SWT & JFace.

If this doesn't turn out to be enough, we can always reopen this bug! :)  But I 
think just dividing by 3 would make everything much neater, and since there is 
a pretty clear line here we would avoid rampant cross posting.  Also, we need 
to divide in such a way that the traffic for each NG is still pretty high, 
otherwise people will just post the NG that gets the most attention (see:  the 
Websphere Studio slew of newsgroups, which tend to get ignored in favor of 
the "general" group).   For example, I forsee the "install" newsgroup being 
ignored in favor of .user -- people will post where the traffic is (even more 
likely for the demographic (newbie) targeted by a future eclipse.user NG).  

-Andrew

Comment 49 Randy Hudson 2003-04-16 11:02:06 EDT 
I'm throwing out a proposal for newsgroup names.  Following the current 
pattern, most newsgroups are named: eclipse.(%PROJECT%)[.%SUBPROJECT%]

(or, you could say that the "eclipse." is part of the project name)

This means that the eclipse.tools newsgroup is really poorly named, because it 
should be about the tools subprojects (emerging subprojects, for example), 
instead it is about the Eclipse "base" project.  IMO, it should be renamed, but 
apparently the "base" project has no name, which is probably why we have this 
problem. So Problem #1 is give the base project a name, or decide that it gets 
the "eclipse" namespace.

Problem #2 is to expose the subprojects (minimally) of the base project.  So we 
would need:
eclipse.base.swt
eclipse.base.ui
eclipse.base.jdt
eclipse.base.pde

Problem #3 is to distinguish between DEVELOPERS who are compiling against a 
given subproject, and USERS who are using a given subproject. I think the most 
common newsgroup user is the developer, so "dev" is assumed, and we don't need 
a "dev" prefix.  Besides, it would mean that we would have to rename *every* 
existing newsgroup. Also, some subprojects (like GEF) are small enough to 
combine the "dev" and "user" posts. Therefore, I think some kind of suffix 
like ".user" should be used only for subprojects that get enough traffic to 
require the split.

So in the end, we have:
(%PROJECT%)[.%SUBPROJECT%][.user]
where PROJECT is one of:
    "eclipse[.base??]", "eclipse.tools", "eclipse.technology"
SUBPROJECT is one of "swt", "ui", "gef", "emf", etc.
and user is just the word user :-)
