Description Udo Walker 2007-06-25 10:47:19 EDT 
Build ID: M20070212-1330

Steps To Reproduce:
1. Create a project.
2. Set the hidden flag of the .project file under Windows
3. Edit the .project file
4. Save it -> ERROR that you can't write it.


More information:
If you configure the file explorer under Windows XP so that all system files are hidden then all files which start with a dot are set to hidden state (hidden flag is set).

If you did the above then you can not add e.g. a new project nature to your project because the .project file is set hidden, but the readonly flag is unset.

This error condition is different to a readonly file state because here the file is still writeable.

Here is the workflow how I have run into this:

1. Create an Eclipse project on Linux
2. Using my Samba share, import the project on Windows
   (with "copy files" turned on)
3. Try to modify the project --> .project file can't be written because
   it has been created with "hidden" attribute and the JRE throws an exception.

In this case, I've been manipulating my files / projects only from Eclipse, and I wouldn't ever have had the idea that a file became "hidden" along the way. Just by coincidence a coworker told me to look at the file attributes (note: the "hidden" attribute isn't exposed in the Resource Properties in Eclipse!)

I also validated that the same problem still exists when running on Oracle JRE 7u1 -- so if Oracle claims this has been fixed in Dolphin this doesn't seem true.

Given that EFS does support the ATTRIBUTE_HIDDEN for query as well as put, I think that we should do something about the situation in Eclipse. For Eclipse, the ".* resources" should be normal files which may happen to be hidden in the UI, but the "access denied" when trying to write them just seems like a defect:
  - Maybe expose the "hidden" attribute in resource properties
  - Maybe clear the "hidden" attribute in the destination when copying
  - Maybe ensure that at least the ".project" file can always be written

I'm reopening the bug for discussion.

Comment 21 Richard Gobeli 2014-04-16 14:56:38 EDT 
 Below is a method that makes sure any resource is writable.
 When CDT is going to store the data in .cproject this method is called to make sure it can write to .cproject 
 
 void org.eclipse.cdt.internal.core.settings.model.CProjectDescriptionStorageManager.ensureWritable(IResource resource) throws CoreException


Helper method to ensure that a resource is writable. This means: 
- If the resource doesn't exist, it can be created (its parent is made writable) 
- If the resource exists and its a file, it's made writable 
- If the resource exists and its a directory, it and its (direct) children are made writable

			if (attrs != null) {
				boolean hasChange = false;
				// Nothing to update if we never made the box
				if (hideBox != null
						&& hideBox.getSelection() != previousHiddenValue) {
					attrs.setHidden(hideBox.getSelection());                                  	\\ hidden attribute is set by what is in the hidebox.
					hasChange = true;
				}
				if (hasChange) {
					resource.setResourceAttributes(attrs);        					\\ attribute is changed here.
					attrs = resource.getResourceAttributes();
					if (attrs != null) {
						previousHiddenValue = attrs.isHidden();
						if (hideBox != null) {
							hideBox.setSelection(attrs.isHidden());
						}
					}
				}
