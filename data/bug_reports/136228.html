Description Linda Chan 2006-04-11 20:35:20 EDT 
When a user changes selection of a node in the Data Explorer or Outline tree view, it could trigger query execution of *all* data set nodes in the tree.  This appears to occur due to the call to UIUtil.containElement, as shown in the call stack below.
This is very inefficient, because establishing a live connection to a data source and executing a query could be very time-consuming.  Furthermore, this may trigger unnecessary error messages, e.g. a data source connection may be down.

When a data set is not even related to the user selection node, its child elements checking should be skipped, and should not trigger its query execution.

Comment 6 Linda Chan 2006-04-17 15:26:15 EDT 
This problem exists with a newly created report design (2.1 RC0) as well.

When a new (incomplete) data set is created without result set metadata, then working with the Data Explorer tree would keep trying to instantiate all the child nodes of all data sets in the tree.  
A main issue is why does UIUtil.containElement method need to go instantiate all child nodes of all data sets in the tree, including those that the user has never expanded.  
This by itself is quite inefficient, even if cached metadata is available.
Some data sets used in production could contain hundreds of columns.

Comment 13 jluo 2006-04-28 00:07:11 EDT 
The new mechanism is when there is cached metadata in the report design file, the query will not be executed. If the cached metadata does not exist, the query execution will be done. If the metadata of data set can be retrieved, then the metadata will be inserted into report design file as the cached metadata. If not, an empty cached metadata will be generated and inserted, and meantime, an exception will be thrown to notify users that there is error in defined data set or data source.
