Comment 7 Markus Karg 2003-03-20 03:14:54 EST 
Information from Sun:

   There is a significant bug in Eclipse's native libraries
which appeared to have been tolerated by older JVMs.  The newer
(b17+) JVMs test for this specific application bug, and intentionally
cause the application to fail (as opposed to undefined or unpredictable
behavior, or even C-heap corruption).

  More technical evaluation info:
 ------------------------------------ 
  This is a bug in the application which shows up now because the JNI
  code for JNI Get<Type>ArrayElements has been updated to detect this
  problem.

  The native method Java_org_eclipse_swt_internal_motif_OS_XmCreateScrolledList
  is calling the JNI function GetByteArrayElements, passing it an array
  of length zero.  GetByteArrayElements is expected to make a copy of zero
  bytes and return a pointer to that copy.

  Previously, the JVM would allocate zero bytes with malloc(0) and return 
  a pointer to the resulting buffer.  Since there are zero valid data bytes
  in that buffer, any reference to any data in that buffer would be invalid.
  The application passes this buffer to XmCreateScrolledList(), which passes
  it to strlen(), which looks at the first byte of the (empty copy of the
  empty) array.  The byte returned was probably/coincidentally a null 
  character, allowing strlen() to return 0, which seems not to have 
  interfered with the application.

  Now, in this special case (empty arrays), the JVM deliberately returns
  a pointer to an invalid address.  Calling strlen() on this address
  now produces a fault (SEGV) instead of the previous unpredictable 
  and invalid behavior.
 ---------------------------------------
 
Can you pass this information to Eclipse folks and let me know if it
is necessary that we provide some other "fix"to tolerate this serious
bug in Eclipse apps?

Comment 13 Jim des Rivieres 2003-03-20 20:29:22 EST 
Thanks again to Markus, and to Ingrid and the team at Sun for helping us find 
and diagnose these problems. (Markus: I trust you'll pass this info along to 
Sun via your 4834329.)

Sun's intention is improve early detection of programming errors. Having 
Get<Type>ArrayElements return an invalid address for empty arrays does this, 
but may have other, less welcome, consequences. To see why we are concerned 
that this might be the case, let's see how Eclipse got in trouble (in one 
place, of several).

Eclipse's SWT component provides an API method drawText that does pretty much 
what's you'd expect:

/** 
 * Draws the given string, using the receiver's current font and foreground 
color.
* [...]
 *
 * @param string the string to be drawn
 * @param x the x coordinate of the top left corner of the rectangular area 
where the text is to be drawn
 * @param y the y coordinate of the top left corner of the rectangular area 
where the text is to be drawn
 * @param flags the flags specifing how to process the text
 *
 * @exception IllegalArgumentException <ul>
 *    <li>ERROR_NULL_ARGUMENT - if the string is null</li>
 * </ul>	
 * @exception SWTException <ul>
 *    <li>ERROR_GRAPHIC_DISPOSED - if the receiver has been disposed</li>
 * </ul>
 */
public void drawText (String string, int x, int y, int flags);

On Windows, this method has an implementation, the meat of which is a call a 
Windows-specific native method ("OS.DrawText"):

public void drawText (String string, int x, int y, int flags) {
  if (handle == 0) SWT.error(SWT.ERROR_GRAPHIC_DISPOSED);
  if (string == null) SWT.error(SWT.ERROR_NULL_ARGUMENT);
  RECT rect = new RECT();
  OS.SetRect(rect, x, y, 0x7FFF, 0x7FFF);
  TCHAR buffer = new TCHAR(getCodePage(), string, false);
  int uFormat = OS.DT_LEFT;
  if ((flags & SWT.DRAW_DELIMITER) == 0) uFormat |= OS.DT_SINGLELINE;
  if ((flags & SWT.DRAW_TAB) != 0) uFormat |= OS.DT_EXPANDTABS;
  if ((flags & SWT.DRAW_MNEMONIC) == 0) uFormat |= OS.DT_NOPREFIX;	
  int oldBkMode = OS.SetBkMode(handle, (flags & SWT.DRAW_TRANSPARENT) != 0 ? 
OS.TRANSPARENT : OS.OPAQUE);
  OS.DrawText(handle, buffer, buffer.length(), rect, uFormat);
  OS.SetBkMode(handle, oldBkMode);
}

The C code for this (internal) native method does little more than call the 
Windows DrawText API function:

JNIEXPORT jint JNICALL OS_NATIVE(DrawText)
	(JNIEnv *env, jclass that, jint arg0, jcharArray arg1, jint arg2, 
jobject arg3, jint arg4)
{
	jchar *lparg1=NULL;
	RECT _arg3, *lparg3=NULL;
	jint rc;

	DEBUG_CALL("DrawText\n")

	if (arg1) lparg1 = (*env)->GetCharArrayElements(env, arg1, NULL);
	if (arg3) lparg3 = getRECTFields(env, arg3, &_arg3);
	rc = (jint)DrawText((HDC)arg0, (LPWSTR)lparg1, arg2, lparg3, arg4);
	if (arg1) (*env)->ReleaseCharArrayElements(env, arg1, lparg1, 0);
	if (arg3) setRECTFields(env, arg3, lparg3);
	return rc;
}

The Windows API spec (from MSDN) for the DrawText function seems equally 
innocuous:

"The DrawText function draws formatted text in the specified rectangle. It 
formats the text according to the specified method (expanding tabs, justifying 
characters, breaking lines, and so forth). 
To specify additional formatting options, use the DrawTextEx function.
int DrawText(
  HDC hDC,          // handle to DC
  LPCTSTR lpString, // text to draw
  int nCount,       // text length
  LPRECT lpRect,    // formatting dimensions
  UINT uFormat      // text-drawing options
);"


As you trace what's happening in the above code, an empty string passed to the 
drawText method, begets an empty buffer passed to OS.DrawText, begets an 
invalid address (returned by GetCharArrayElements on the checking JVM) passed 
to the DrawText function.

You would think that this would work. Surely the Windows DrawText function 
would not deference the text string passed to it when the text length is 0. 
Surprise! It does deference its lpString parameter even when length is 0. As a 
result the invalid address returned by GetCharArrayElements causes a fault. 
This is a Windows bug (it's pretty hard to justify actually accessing an empty 
string). But there you are --- it does.

Given that the DrawText function cannot be trusted to not deref the text 
string even when the length is 0, and that GetCharArrayElements cannot be 
trusted to not return an invalid address when the length is 0, there is no 
choice but to tread carefully. Here's how the DrawText native could be 
rephrased to avoid the problem:

JNIEXPORT jint JNICALL OS_NATIVE(DrawText)
	(JNIEnv *env, jclass that, jint arg0, jcharArray arg1, jint arg2, 
jobject arg3, jint arg4)
{
	jchar *lparg1=NULL;
	RECT _arg3, *lparg3=NULL;
	jint rc;
                jsize len;
                jchar dummy1;

	DEBUG_CALL("DrawText\n")

	if (arg1)
	{
	   // do not trust Get<Type>ArrayElements to return a valid address 
for empty arrays
                   len = (jsize) ((*env)->GetArrayLength(env, arg1));
	   if (len != 0)
	      lparg1 = (*env)->GetCharArrayElements(env, arg1, NULL);
	   else
	      lparg1 = &dummy1;
	   // lparg1 is always a valid address, even for empty arrays
	}
	if (arg3) lparg3 = getRECTFields(env, arg3, &_arg3);
	rc = (jint)DrawText((HDC)arg0, (LPWSTR)lparg1, arg2, lparg3, arg4);
	if (arg1)
	{
	   if (len != 0) (*env)->ReleaseCharArrayElements(env, arg1, lparg1, 
0);
	}
	if (arg3) setRECTFields(env, arg3, lparg3);
	return rc;
}


From this simple tale we can make two observations:
(1) It's an easy mistake to make. The Eclipse team has made it in a several 
places despite our best efforts to avoid making these kind of assumptions. We 
didn't even see the bug until there was a JVM that returned an invalid address 
from Get<Type>ArrayElements. We wouldn't be surprised if other JNI clients 
have made similar mistakes.

(2) Once discovered, the fix is to special-case empty arrays. Once the code is 
fixed, performance of the non-empty array case is slightly degraded due to the 
compulsory call to GetArrayLength. 

The benefits of increased early detection of empty array access problems must 
therefore we weighed against these downsides. Clearly (1) would have been a 
non-issue had JNI always worked like this. But this is not how most (all?) 
JVMs work. Some clients, like eclipse, will be surprised to find that they 
will not run on 1.4.2 because of this problem, and have to issue fixes to 
their customers.

On the other hand, (2) is not a one-time hit. JNI clients that wouldn't 
otherwise have to special case empty arrays now have to, and will always be a 
bit slower on the non-empty array case.

Here's a different proposal for improve JNI functionality in this area without 
bumping into either (1) or (2).

(a) Change the spec for Get<Type>ArrayElements to say that the address 
returned is always valid address, even for empty arrays. 
(b) Change the implementation (back) to return a valid address.  However, 
where the JVM used to allocate a zero byte buffer with malloc(0) and return a 
pointer to the resulting buffer, the implementation should do something 
cheaper: just return the fixed address of some 8-byte read-only area. Not only 
would this be cheaper, it ensures that a C client that tries to write into the 
buffer will fault.

If this approach were taken instead, existing JNI clients would not break 
(unless they happened to be attempting to write to the empty buffer, in which 
case they will thank you for finding an ugly bug where they're currently 
trashing memory). Moreover, performance is no worse than the 1.4.1 
implementation, and possibly a bit better because there's no malloc() on the 
empty array case. And there's no need for clients to special-case empty 
arrays. These changes would probably be a greater benefit to the users of JNI 
than the change to Get<Type>ArrayElements introduced in 1.4.2 build 17.
