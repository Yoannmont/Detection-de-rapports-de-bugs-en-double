Comment 10 Brian Bauman 2007-05-30 16:37:29 EDT 
Created attachment 69385 [details]
patch

The problem we are able to reproduce consistently (now), is the PDEState is getting corrupted.

The cause it pretty simple.  In the TargetPluginsTab.performFinish(), if the PDEState we reloaded (fReloaded == true), we run a TargetPlatformResetJob().  This goes ahead and resets the state and also adds all the workspace plug-ins.  

Unfortunately, when the user presses apply, TargetPluginsTab.performFinish() is called.  This resets PDE's main state/models (including adding workspace models to the state).  When the user clicks OK, the same method is called and we again reset PDE's main state/models (which shouldn't really happen for efficency) and REadd the workspace bundles to the state.  I believe it was this that caused the corrupted PDEState which lead to problems throughout PDE.

The fix is simple, when TargetPluginsTab.performFinish is called, if the target was reloaded after resetting the state, change the fReloaded flag to false.  The next time this function is called (without the user reloading another state), a delta from the state will be computed instead of replacing everything in PDE.

Modified PluginModelManager.resetState() to do a check to see if we are reseting the State with our current state.  If so, don't do anything.  This is a safe guard to prevent this from happening if users call this function incorrectly.

Modified TargetPlatformPreferencePage.peformFinish() to update the "original" state path when the function is called.  If this is not included, problem will incur when the user loads a new state, hits apply, and then select the previous state in the drop down.  Since the "original" state's path matches the current target platform drop down, the user will not be prompted to reload and will have bad information in their target platform.

Modified TargetPluginsTab.performFinish() to reset the fReloaded flag and fChangedModels collection upon resetting the PDEState.  Not only will this fix the problem but will improve the performance of the preference page if the user selects Apply before selecting Finish.
Comment 11 Brian Bauman 2007-05-30 16:43:07 EDT 
Wassim, Mike and Dejan, you do not need to read all the explaination for the patch.

This main idea of this bug is we run into problems if we reset the PDEState(TargetPluginsTab:652-654) twice, once upon the Apply and once for the Finish.  Therefore, the patch should only reset the PDEState upon the first Apply and then compute a delta when the user clicks Finish.
