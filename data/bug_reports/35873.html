Description Alex Chapiro 2003-03-31 12:48:17 EST 
This is taken from eclipse.tools newsgroup.
We are using user help plugin with user doc.zip file. The size of this
doc.zip is about 20Mb. Every time I do Help search from IDE (not from Help
application itself!) and then double click in selected list any topic that
belongs to this large doc.zip, help application works in a very strange
manner. It displays requested stuff and updates search results list in the
left pannell more or less quickly. But what else it is doing after that for
a minute or even more? During this time it uses 99% of CPU resource!
I made several experiments. If I at the same time kill main Eclipse process,
it doesn't impact Help application at all. It continues to utilize CPU with
the same intensity for about a minute. So it is nothing to do with exchange
with tomcat server.
If I try it under debugger, I see that in most cases application freezes on
this code (this is IEHost class):
private void runUI() {

    IEHost.printMsgEx("Main thread loop start");

    while (!shell.isDisposed()) {

        if (!display.readAndDispatch()) {

            display.sleep();

.....

at display.readAndDispatch(). At this moment readAndDispatch is usually
blocked on DispatchMessage. The message type is always 0x113 (WM_TIME) and
the window timer belongs to one of "Internet Explorer_Hidden" class. So it
seems to be blocked on some internal IE work.

I have no explanation the fact that there certainly is a misterious
dependency between the size of the doc.zip file which includes the topic I'm
going to display and the period of blocking of help application on WM_TIME.
If I cut large doc.zip to four 5Mb files, things are going much better.

Reply from Dorian Birsan
========================
We've noticed this and concluded the CPU cycles are being consumed
mostly in the browser,by some Javascript code that attempts to synchronize
the displayed topic with the navigation. I can't explain why the doc.zip
size has any effect on the cpu time, but the size of the navigation tree
(the toc) is very important here. Each topic href is compared with the
displayed topic and if the tree is very large then this comparison process
can take a very long time and is very CPU intensive. It would be nice if
there were some threads support in javascript.
