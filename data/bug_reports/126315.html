Description Ralf Ebert 2006-02-03 04:58:55 EST 
I hope this is the right place for this report:

a project contains some files which are excluded from build and it works fine on my local computer. After checking out the project from an other computer, the information which files are excluded from build is lost in the project but the informations in the file .cdtbuild about the files, which are excluded from the build, is correct. After exclude a file from build once more the entry in the file .cdtbuild is duplicated. 

I can reproduce this behaviour in the follow way:
- create a managed make c++ project 'ExcludeTest' with project type 'Executable'
- create a Folder 'src'
- create a file 'main.cpp' in 'src' with an 'hello world"-example
- create an empty file 'exclude.cpp' in 'src'
- set 'Exclude from build' for 'exclude.cpp'
- share the project with CVS
- checkout the project with a different name 'ExcludeTest_new'

The checkmark of 'Exclude from build' for 'Exclude.cpp' in 'EcludeTest_new' isn't set now.

Bye
  Ralf Ebert

That is correct. I have investigated this issue briefly, but did not have time yet to think deeply about the correct way of fixing it.

The root cause of a problem is importing the project as a project with a different name. In this case resource configuration paths, containing the old project name, become invalid.
I will need to check if the resource change handling mechanism implemented in MBS can handle this situation, but I知 almost sure that it can not.

On the other hand, I would say that it is not very correct to import a project from CVS as a project with a different name. In this case the .project file does not get updated and hence contains the old project name. As a result, if I remove the project from the workspace (without deleting contents) and then import it again, the project will contain the old name stored in the .project file.

What I think we could do to handle all this is to use project relative paths as resource configuration paths where appropriate instead of resource full path as done now. We will definitely need to support the old style paths (full paths) for backward compatibility.

Comment 6 Leo Treggiari 2006-03-03 15:29:10 EST 
In the "Project description file" topic, the platform says this:

"If a new project is created at a location that contains an existing project description file, the contents of that description file will be honoured as the project description.  One exception is that the project name in the file will be ignored if it does not match the name of the project being created."

I guess they are not kidding when they say the name is "ignored".  I would also think that they should make the names match.

Regarding the resourcePath, it sounds like using a project relative path is the correct thing to do in the future, but we have to decide how to support the old paths.  Here are 2 thoughts:

1.  Remove the first segment in a converter that runs when loading an older project.

2.  Deprecate the resourcePath attribute and add a new attribute - projectRelativePath.  Use projectRelativePath if present, else use resourcePath.

That’s an interesting question. I will need to investigate this further, but at the first glance it looks like we have the following problem:
When switching to another branch or, in general, when obtaining another version of the .cdtbuild file from the CVS or file system, the in-memory representation of the build information does not get updated.

Comment 9 Mikhail Sennikovsky 2006-03-06 06:17:41 EST 
(In reply to comment #6)
> Regarding the resourcePath, it sounds like using a project relative path is the
> correct thing to do in the future, but we have to decide how to support the old
> paths.  

One thing we should clarify here is whether we might need to support the resource configurations for resources from some other projects.
My assumption is that we need not, I just want to double-check. 

> Here are 2 thoughts:
> 
> 1.  Remove the first segment in a converter that runs when loading an older
> project.
> 
> 2.  Deprecate the resourcePath attribute and add a new attribute -
> projectRelativePath.  Use projectRelativePath if present, else use
> resourcePath.
I agree with #2, as for #1 I think we could perform the first segment removal directly in the Configuration while loading the resource configuration info, rather than in the converter, because the Configuration will have to maintain this logic in any way. 
Here are some addition thoughts on the public API changes:

The ResourceConfiguration will maintain only resource project relative path. The full path stored in the old (v < 3.1) manifest and .cdtbuild files will be converted to the project relative path while loading.
IResourceConfiguration.getResourcePath() is deprecated but remains fully functional for backward compatibility and returns the resource full path, that is calculated by appending the resource project relative path to the project full path.
IResourceConfiguration.getProjectRelativePath() – new method to be used instead of getResourcePath

The resource configuration map maintained by the configuration will hold the “project relative path”-to-“resource configuration” pairs.

Configuration.getResourceConfiguration(String path) - should support both project relative and full paths for backward compatibility. It will check if the given path is absolute (path.startsWith(“/”)), and if so, will remove the first segment from the path. And will use the resulting project relative path as a key for obtaining resource configuration from the map.

Mikhail
