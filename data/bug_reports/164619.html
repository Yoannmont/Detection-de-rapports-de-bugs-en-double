Description Markus Schorn 2006-11-15 06:14:53 EST 
I am trying to understand the interaction between the name resolution 'CPPSemantics.resolveBinding(IASTName)' and the index. I see 3 different paths on how the index is used:

(1) During the (scope-local) lookup in a namespace (CPPScope.getBinding(...))
(2) implicitly via IScope.getBinding(...) in case a name resolution introduces a scope stored in the index.
(3) At the end of the lookup (CPPSemantics.postResolution()), if no result has been found by other means.

From that, it is not clear to me, what the index is expected to return on (3). Does it really have to analyze the context in which the name is used in the AST?

I think it would be more natural to use the index during the (scope-local) lookup in a translation unit, similar to (1) in CPPScope.getBinding(..). I even think that it is not correct moving on without consulting the index there. Doing that should make (3) superfluous. Furthermore, the index would return global symbols, only and does not have to inspect the context of the IASTName.
Comment 1 Andrew Niefer 2006-11-15 11:29:51 EST 
I agree with Marcus. I have not looked at how the PDOM was integrated beyond the one postResolution() method (see bug 164481).  The last thing you want is for the PDOM to essentially duplicate the kind of analysis that goes on in CPPSemantics and CPPVisitor in order to return a binding.

If the PDOM is going to be correct in the long run, it should only be consulted in a "scope restricted" manner, that is (1) and (2).  As well, it should realize that finding nothing is often in fact the correct answer.  The bindings it returns should be indistinguishable from regular bindings as far as the semantic resolution is concerned, in particular with regard to IType information (see for example bug 164659).
Comment 2 Doug Schaefer 2006-11-15 11:53:38 EST 
There is no doubting that the PDOM integration with the resolution algorithms is a big hack. Ideally, yes, the PDOM should be integrated much earlier than it is, i.e. at namespace lookup time. However, I find the resolution code mind boggling and full of assumptions that we have an AST around to help with the lookups. I couldn't see an easy way to integrate the PDOM there.
Comment 3 Andrew Niefer 2006-11-15 13:17:17 EST 
The scopes were introduced as a kind of cache for the bindings to improve performance.  The PDOM should really be considered as an extension of this.

You can consider the resolution to be in 2 parts, the first is to collect a set of bindings to consider, the second is to actually choose between them.  Collecting bindings comes from either the physical tree, or from the scopes.

In theory, resolving between the bindings does not need the physical information although in practice there are likely some places that would need to be changed (In particular, CPPSemantics.declaredBefore would need to do the right thing for PDOM bindings).  There do exist bindings that have no physical information at all.

Comment 7 Markus Schorn 2006-12-06 03:57:20 EST 
Thanks for the feedback!
(In reply to comment #6)
> My two questions would be..
>  (1) In CPPSemantics.resolveAmbiguities, is there an assumption that either all
> function candidates come from the AST or all functions come from the index? if
> so, I'd've thought that might be a problem?
No, theoretically that should work with a combination of both. For the case where there are candidates from the AST I wrote the code such that it ignores the candidates from the index. In 99% of the cases that should be the right thing to do.

>  (2) are PDOMCPPLinkage.resolveBinding and PDOMCPPLinkage._resolveBinding
> needed?
At this point they are still being used, but we should get rid of them. I think it is not hard to do so and I will look into this before applying the patch.
Comment 8 Markus Schorn 2006-12-06 05:52:41 EST 
I have commited the patch, here a summary of what it does:
  (1) cpp-name resolution makes scope-local lookups to the index, only.
  (2) IType.isSameType works accross DOM and PDOM classes
  (3) a general mechanism to prevent stack overflows during the name resolution
  (4) removal of access to the IASTInitializer for default parameters  
  (5) extensions to the index: implicit methods, pointer-to-member-types, 
      array-types, reference types, return types, default parameters.

The fix reduces the problem count for indexing firefox 2.0 from 
576105 to 512334 problems.
