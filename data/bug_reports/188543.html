Comment 5 Frank Becker 2007-05-23 15:28:03 EDT 
Hi Mik,

are you sure that the error is related to my patch.

If I see it right you get win32 from Platform.GetOS() on your Vista box. This is line 284 of BugzillaCorePlugin.

 java2buzillaOSMap returns "Windows XP" for win32 so bugzillaOS is set to "Windows XP".
 
 opSysAttribute.getOptionParameter contains the following list 
 {Windows=Windows, Mac OS=Mac OS, All=All, Other=Other, Linux=Linux} 
 but bugzillaOS is  "Windows XP" so it is rest to null.
 
 Please try to change in BugzillaCorePlugin  line 79 from
 		java2buzillaOSMap.put("win32", "Windows XP");
 to
 		java2buzillaOSMap.put("win32", "Windows");
 
This works for me if I change OS to win32 within the debugger on my IntelMac.

I think we should find a better way to set the OS. Maybe the use of System.getProperty with "os.name" and "os.version" would help.

What do you think about this?

Bugzilla set the Platform and OS by looking for specific Information in the HTTP_USER_AGENT String. Until 3.0 there is no test to find Vista. Bugzilla ues by default only 5 entries for op_sys but can detect much more OS in enter_bug.cgi (subroutine pickos). Bugzilla build a list (most spzific at the beginning of the list) of detected systems and the try to find one of them in the database.

By Example on my Intel Mac the list looks like
"Mac OS X 10.4","Mac OS X 10.3","Mac OS X 10.0","Mac OS"
so depending of the content of table op_sys I get different results.
I think we should use a similar logic for the Mylar Bugzilla connector and not end with the most generic OS Entry.

We have with System.getProperty("os.name") and System.getProperty("os.version") the Information to build the correct value. What do you think about this enhancement?
