Comment 10 Christophe Cornu 2005-04-13 11:41:05 EDT 
Fixed. AMD64 Browser will simply throw a NO MORE HANDLES, Browser is not 
supported for 3.1 on that platform. This will let the build work fine without 
the Browser functionality.

The swt-mozilla native library is no longer built for AMD64. The problem is 
that we currently need to build against Mozilla 1.4 to be upward compatible, 
however there are no standard out of the box Mozilla 1.4 64 bit compliant 
sources. At the moment every distribution has their own set of patches to 
workaround 64 bit issues. As an experiment, I tried to build our library 
against a 1.7.5 patched from Gentoo but that has proven to be causing trouble 
and not very useful since it was not working with most other 64 bit Mozilla 
builds.
64 bit Browser support can be revisited if a future version of the Browser is 
implemented on top of GtkMozembed or when major Linux distributions switched 
to a standard 64 bit 1.7 version of Mozilla.

Comment 13 Billy Biggs 2005-06-01 11:43:55 EDT 
This bug report is a bit confusing.  All of the stack traces on this bug are for
NoClassDefFound errors, which indicates a problem in the Java side and not with
the compiled mozilla code or the version of mozilla on the target machine.  This
indicates a bug either with the build, or with the class loader.  I believe that
these problems are resolved by the new build strategy for 3.1 which puts all of
the SWT Java code in a single jar.

Since:

  1. It is a pain to build the mozilla library for amd64 users that want it
  2. There are no reported crashes from incompatible mozilla builds
  3. amd64 is still an "early access" platform for Eclipse
  4. Users are asking for embedded mozilla support
  5. It's easy to disable if it is causing problems (unset MOZILLA_FIVE_HOME)

I think we should just re-enable it in our builds.
