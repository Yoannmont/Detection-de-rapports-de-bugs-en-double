Comment 3 David Michonneau 2006-06-22 03:57:03 EDT 
Here are some previous comments from Gary regarding this (note that row._outer 
doesn't iterate data in the subquery, so is not a viable solution right now):

?It?s important to understand that the data source of a subquery is the whole 
range of data set rows used to calculate a subset (usually a group) of the 
parent table?s rows. The introduction of the data binding concept has not 
changed this definition. It?s the underlying data set data that a subquery has 
direct access too (via the ?dataSetRow? variable), not the parent table?s bound 
result set (via ?row? variable?). Hence the current restriction on the 
subquery?s access to parent?s ?row?. Note that this does not create a 
regression in functionality compared to 2.0 ? what could be done in 2.0 with 
subqueries are still done the same way today, except that old ?row? reference 
should be replaced by ?dataSetRow?.

 One can argue for an enhancement to allow subqueries access to the parent 
query?s bound result set. However I don?t think the benefit would outweigh the 
cost in this case. Keep in mind that the subquery in most cases can simply 
duplicate the parent table?s binding definition if it needs to obtain the same 
set of values. "

--

"We should not allow subquery bindings to directly use 
(row["parent_binding_col"]) to access the parent's binding column. 
(row._outer["parent_binding_col"]) must be used instead. This maintains the 
structural semantics of the "row" variable - it refers to bindings defined at 
the current table. 

Opening up the "row" variable to parent bound column access will create a host 
of other issues that are tough to deal with. For one, Model would then need to 
maintain bound column name uniqueness across multiple levels of nested tables. 
On the Engine/DtE side, a dynamic "row" variable is a lot harder to implement. 
I don't think these costs are justified.?

Comment 7 Gary Xue 2006-08-03 16:30:56 EDT 
Since the Chart builder is now copying the container's data bindings. This feature has less end user impact. We are postponing it to 2.2.

Included below is a draft design being considered for this feature.

==Objective

Allow a subquery table/chart (which uses the container’s data set) to directly use the container’s data bindings as its source of data. This is currently not possible to do.

==Reference Syntax

The subquery’s expressions (this includes its binding expressions, it’s TOC/Bookmark expression etc.) can directly use the row object to refer to the container’s bindings. The subquery can also define its own bindings and refer to them using the same row object. The subquery binding can continue to use the dataSetRow object, which refers to the container’s data set row. 

Where there is a name conflict between a subquery’s bindings and that of its container, the subquery’s binding definition shadows the same-named binding inherited from the container. This is not considered an error in report design. For example, if both a subquery and its container define a binding named “a”, then expression row[“a”] within the subquery refers to the subquery’s binding definition. The container’s binding “a” becomes inaccessible as the source of the subquery’s data in such a case.

==UI Design 

Data Binding Editor

In the Bindings tab of the property editor of a table/list, if the Data Set is set as “None” (i.e., it is a subquery), all data bindings defined in the container (and the container’s container etc. in case of nested subqueries) are listed, after the locally-defined binding entries. The bindings inherited from the containers must be visually distinguishable (I suggest showing them in grey) from the local bindings. The inherited entries are non-editable, and cannot be deleted. Inherited bindings that are shadowed by a local binding are not shown. 

Same changes are made to the Data Binding dialog.

Chart Builder

Same changes as above are made to the Select Data Binding dialog invoked from the “Data Binding…” button in the chart builder. 

Chart builder currently copies the container’s binding definitions when it uses the container’s data set (presumably implemented this way as a workaround for the lack of binding inheritance support). It will stop doing so.

The Data Preview tab will show preview of all local and inherited binding columns.

Expression Builder

When expression builder is invoked to edit an expression for a subquery item, under Available Column Bindings, when the current table/list/chart is selected in the Sub-Category tab, both the container’s bindings and local bindings are shown in the available binding list. Inherited bindings are shown after the locally defined bindings, and the shadowed bindings are not shown. When a binding X is selected from this list, row[“X”] is generated to refer to that binding, regardless of whether the binding is local or inherited.

Note: If the user selects a container table/list from the Sub-Category tab, the available binding tab will continue to show the container’s bindings like it does today. If the user selects a container’s binding X in this manner, the expression row._outer[“X”] is generated (like it does today). It is important to note that row[“X”] and row._outer[“X”] do not mean the same thing.

Backend Changes

·       Data Engine

Data Engine will be able to evaluate the expression row[“x”] within the subquery when x refer to the parent’s query’s bindings.
DtE’s IResultIterator interface will be able to provide value for binding columns inherited from the parent query.
The row[“X”] and IResultIterator evaluation will follow the shadowing rule as defined above.

·       Engine

No engine changes should be necessary, as it relies on DtE to obtain binding column values

·       Model

No Model change is proposed. It will be UI’s responsibility to merge container bindings by walking the container hierarchy
