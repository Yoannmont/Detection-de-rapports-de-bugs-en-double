Product Build#:         I20070306
Platform/Version :      Windows Vista, XP
Language:                  T.Chinese
Releated Test Case #:    ECLIPSE3.3 - M1 EXPORT WORKING SETS AS PART OF TEAM PROJECT SET
JVM:                    IBM JRE 5.0 SR1

===== DEFECT DESCRIPTION =====
In Export wizard for Team Project Set, the wizard won't display any project when "Export working sets" is enabled

===== STEPS TO RECREATE =====
1. File > Export > Team > Team Project Set
2. Check Export working set
3. Click Edit button
4. Define a new Working Set and select it
5. Click OK button, and will return to "Team Project Set" dialog

===== EXPECTED RESULT =====
1. The projects inculded in the new working set should be displayed on the dialog.

Comment 3 Tomasz Zarna 2007-04-04 10:36:36 EDT 
Created attachment 62930 [details]
Patch #1

This is the first part of the patch with a cosmetic change. It gets rid of resizing issue when switching between exporting projects and working sets.

Here are my comments after reviewing the code:
I can easily add Java Working Sets (or others) to the table on "Export working sets" page. There is a problem with handling them. Export Project Set Wizard expects that selected working set contains only objects of type IProject. This way these projects can be displayed in a column on the right. Java Working Sets don't contain IProject objects ( I think it's JavaProject, but I'm not 100% sure at this moment). This requirement causes problems even when we work with Resource Working Sets. If your Working Set is made up of single files not projects there will be an exception thrown. The reason why the exception is thrown is the same for both cases. So, the question is: should I allow to select (display) working sets that don't contain IProject objects (or similar like JavaProject which I can adapt to IProject somehow) or should I hide the table with projects and don't bother myself with displaying which projects will be exported after a working set has been selected? I'm sure this can be solved in different way too, these are just my first thoughts. What's your opinion?
Comment 4 Michael Valenta 2007-04-04 13:41:39 EDT 
With the provided patch the margins look to small. I don;t see the layout issue mentioned in this bug. Is it mentioned in another bug?

As for the Java working sets, the elements in the working set are instances of IAdaptable. What you can do is adapt the elements to IResource using the Utils.getAdapter method. Then accumulate the projects for the resources that the elements adapt to. The reasoning is this. If you are exporting a project set that contains non-project resources, the only way to rebuild the working set would be to include the whole project. Does that sound reasonable to you?

Comment 8 Michael Valenta 2007-04-05 13:41:23 EDT 
The patch looks like it is almost there but I do have a few more comments:

1) We shouldn't restrict the supported working sets to Resources and Java. We should support any working set that contains at least one resource. 

2) In the getElements method, there are several different conditions for extracting projects. Patch v2

Next version of the patch: it has changes requested in comment 7. I've added third private method named isWorkingSetSupported(IWorkingSet) which checks whether given Working Set contains at least one resource, so we can export it. Now, I'm sure you will have comments to this bug, but let me first ask you some questions:
1) I've assumed that we don't support working sets which are aggregates. Is that OK? And what is related to this question: should Window Working Set be available to export?
2) AFAIK we can't import/export local (non-shared) projects. Should they be displayed while selecting working sets to export? Or maybe I should hide working sets which contain only elements from non-shared projects? They won't be exported anyway.
3) What about importing working sets different than Resource working sets? CheckoutProjectOperation#createWorkingSet (line 517) is only prepared for working with this type of working set (of course it was before this patch allowed to export other types of working sets).

Comment 13 Tomasz Zarna 2007-04-16 07:15:13 EDT 
Created attachment 63890 [details]
Patch v3

Two more questions:

1) Should we display working sets which contain only projects that cannot be exported? We can either hide or disable them. First one is easier to do, but IMO the later is more intuitive.

2) Let's say we have successfully exported  a project set. If we had chosen to export working sets there are is a possibility that the project set file refers to non-shared projects. These projects are part of a working set which contains shared and non-shared projects. So my question is: should we limit what projects from a working set are serialized to a project set file? There will be no error when importing this kind of project set file. A non-shared project from a working set will be ignored (I guess).
Comment 14 Michael Valenta 2007-04-16 11:10:04 EDT 
First, I'll answer your questions:

1) I think it is OK to show the working sets that contain non-shared projects. We could even still export them. Perhaps the right answer is to show all the projects like we did before but disable those that are not shared (having a post-fix on the label that indicated that the project is not exportable would help to). What do you think?

2) We have no control over what parts of the working set are serialized. It's all or nothing. As you say, there is no errors when an imported working set that has entries that correspond to resources that do not exist so I think we are OK the way things are.

As for the patch, the one suggestion I have is to use the test found in the ProjectContentProvider#getElements to see if a project should be shown instead of getting the whole list from the tree and then testing to see if the list contains the project.
