Comment 26 Andrew Low 2006-11-27 15:48:46 EST 
I'd be somewhat concerned that the dump information is not quite telling the truth - since if we choose to believe it:

Module=/lib/libc.so.6
Module_base_address=0FDFB000 Symbol=strlen

we are crashing inside of libc performing a strlen().

Two simple things that could be tried are:

1) Disable the JIT using -Xint on the java command line.  If this resolves the problem we know where to look (it may not be a JIT bug, but some interaction with this OS API and the JIT).

2) Use the -Xrs flag to 'reduce signal usage'.  If this helps resolves the problem, then the file dialog must be making an assumption about signal handling that is not true when you run with the JVM.

Comment 28 Andrew Low 2006-11-27 16:31:22 EST 
In comment #27 it is implied that we are actually crashing in strlen() - but if this is true, why do we not crash using the native command test case?

JNI code runs in the java processes address space, so things can be laid out in memory quite differently from a trivial C test program - this is another potential aspect to this problem.  Maybe there is some bogus pointer math that doesn't expect pointers in high memory to be used, and this is what happens when you run in the JVM address space.

There is a README text in the zip which describes how to build the java part. Assuming you are using the 3.2.2 build :

1) In your Eclipse workspace, create a new Java project and import PR143061.java
2) To add SWT to your classpath:
	i) Right click on the project->Properties
	ii)Java Build Path->Libraries tab. Click on Add Variable...
	iii) Select ECLIPSE_HOME and click on the Extend... button
	iv) Expand the plugins tree and select the org.eclipse.swt.gtk.linux.ppc_3.2.2.v3236.jar file -> click OK
	v) Click OK -> good to go!
4) To run, right click on the file Run As>SWT Application  

Let me know if this works for you.
