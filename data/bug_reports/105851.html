Description Simon Kaegi 2005-08-02 15:58:15 EDT 
[java.lang.]System Properties are being used for:
1) configuring the framework (in the adaptors and internal framework)
2) the underlying Properties of the BundleContext.getProperty(...) lookup.

--
This is not really a problem so long as Eclipse assumes control of the VM 
however if Eclipse is embedded in another environment this assumption might not 
hold.

This request is more than anything a placeholder to investigate alternatives to 
System Properties for the items above.

So, assuming there was an alternative Property holder... here are some thoughts.

The 1st item would probably involve quite a few changes as the framework 
involves a fair bit of configuring and all that configuring is using System 
Properties. There are interesting issues here related to where the property 
object should live (e.g. a singleton off the framework ??) and property 
precedence (e.g default->global->instance) that need some discussing.

The 2nd item seems more straight-forward and would be a matter of just backing 
the BundleContext Properties object with System properties instead of using it 
directly. This would still be inline with the OSGi spec (R3).

Part of the work should assess the impact on existing plug-ins and to sort out 
how to minimize it. Certainly a plug-in that was using System properties 
directly instead of BundleContext.getProperty would be impacted. There are 
likely others.

Comment 5 Simon Kaegi 2005-11-20 17:58:50 EST 
This is a first attempt which at least in basic testing seems to work.
e.g. I seem to be able to run the Eclipse IDE and also in my custom environment.

Instead of System properties a new class, FrameworkProperties is being used both for properties used at set-up time and then later as the backing for BundleContext.getProperty(...) 

Framework properties are still accessed using static methods.
So...
System.getProperty("xyz") becomes FrameworkProperties.getProperty("xyz")

This type of direct access is really only suitable in the System bundle.
Other bundles should use BundleContext.getProperty(...).

The BundleContext does not allow setting Properties so in these cases another approach should be used. For many cases I think Bundle specific properties or preferences might be usable instead.

org.eclipse.core.runtime has a few places where (perhaps for legacy reasons?) it requires to set a framework property. In my patch it uses the FrameworkProperties class directly however I hope this can be cleaned up.

This is all optional and controlled by the System property "eclipse.framework.usingSystemProperties".
By default it uses System.properties to back everything and Eclipse should work the same way it has up until this point and should be fully backwards compatable.

If set to "false" it uses a per instance property set that should allow independent installation of eclipse to co-exist in the same VM.

--
Change Summary:
org.eclipse.osgi 
- Added FrameworkProperties
- changed all System.*Propert* calls to use FrameworkProperties unless they were referring to a "real" System Property

org.eclipse.core.runtime
- changed all System.*Propert* calls to use BundleContext.getProperty except where a "set" is needed in which case FrameworkProperties was used directly.

org.eclipse.update.configurator
- changed all System.*Propert* calls to use BundleContext.getProperty unless they were referring to a "real" System Property.

I also looked at org.eclipse.update.core and org.eclipse.core.runtim.compatability however neither really needed changes.

As I mentioned earlier, a bundle that was using System properties 
directly (to access Framework properties) instead of BundleContext.getProperty would be impacted.

I'm still working away on this problem however I'm also hoping that this might allow work on URLStreamHandlerFactory multiplexing to proceed ( see bug 107909 )

Comment 12 Simon Kaegi 2006-01-16 00:55:05 EST 
Created attachment 33047 [details]
Jan15-HEAD - FrameworkProperties

I'm using a multi-project patch; if that's a problem I can re-submit.
This affects the following:
org.eclipse.osgi
org.eclipse.equinox.common
org.eclipse.equinox.registry
org.eclipse.core.runtime
org.eclipse.update.configurator
--

This latest patch is a clean-up (and update) and uses the following rules:
1) In the system bundle all System property access is done through the FrameworkProperties class(with the exception of those listed in System.getProperties javadoc)
2) No other bundle can access FrameworkProperties directly (This implies no other bundle can set framework-wide properties)
3) FrameworkProperties are used to back BundleContext.getProperty(...) and this is how all other bundles access properties. (e.g. well behaved bundles should use bc.getProperty in place of System.getProperty(...))
--

By default FrameworkProperties really uses System Properties. This is done to maintain backwards compatability with any bundles accessing properties through System.getProperty(...) 
By setting the "eclipse.framework.usingSystemProperties" flag to false FrameworkProperties will use be per instance properties for the framework. Even then the initial set of properties is populated with a snapshot of System Properties.
--
Special cases:
org.eclipse.equinox.registry has had its property access modified slightly to allow it to operate without an active BundleContext for those situations where the registry is used without an OSGi framework (see RegistryProperties). In those situations where it previously set System Properties {eclipse.noRegistryCache, eclipse.noLazyRegistryCacheLoading} internal bundle properties are used. The property accessors take the internal and bundlecontext properties into consideration when determining a particular property value.

org.eclipse.core.runtime was also setting System Properties {eclipse.product, eclipse.application} and now uses internal settings. As with the equinox.registry, the property accessors take the internal settings and bundlecontext properties into consideration when determining a particular property value. I left "eclipse.exitcode" still using System.setProperty because it seemed to not be read inside the framework yet might be read by the launcher(??) That being said I think this has a relatively minor impact on multi-framework interference but still worth bringing up.

--
So...
What do you think?

Comment 29 Richard Backhouse 2006-02-22 14:14:28 EST 
I have just tried this out using the Equinox Incubator Servlet Bridge project running within an OSGi based AppServer. The code in FrameworkProperties does not handle the removal of OSGi System properties added in by the top level framework :

All Line 39 seems to achieve is to add all the System properties into a local properties object. This will include any OSGi ones too. If the lower level framework has not overridden them in its call to EclipseStarter.setInitialProperties then they will affect how it runs. For example if the "osgi.bundles" property is set by top level framework then the lower level one will try and use them too. I changed the code in FrameworkProperties to do this instead :

properties = new Properties();
// snapshot of System properties for uses of getProperties who expect to see framework properties set as System properties
// we need to do this for all system properties because the properties object is used to back
// BundleContext#getProperty method which expects all system properties to be available
//properties.putAll(systemProperties);
for (Iterator it = systemProperties.entrySet().iterator(); it.hasNext();) {
    Map.Entry entry = (Map.Entry) it.next();
    if (((String)entry.getKey()).indexOf("osgi") == -1) {
        properties.put((String) entry.getKey(), (String) entry.getValue());
    }
}

This works but I'm sure there are better ways of achieving this.
