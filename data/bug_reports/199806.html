Description Pascal Rapicault 2007-08-13 18:05:55 EDT 
Originally from Susan
It would be good to have property change events for repo property changes.

Dave do you think property can change? Maybe when the repository content would be reloaded?
Do we want event at the property level or just an event saying "repo as changed"?
Comment 1 Susan McCourt 2007-08-13 18:52:03 EDT 
To be more clear:  There are changes to repo properties, but there are other things I can change via API, such as a repo name, and I need notification when those things change.

So I guess my requirement is a property change event for those properties that can change, and notification for changes to the writable values in an IRepositoryInfo.
Comment 2 Jeff McAffer 2007-08-13 21:16:14 EDT 
Hmmm, this is one of those things that comes up when you have a UI...  We should define a set of guidelines for when we have to generate events.  It feels a bit funny (to a core guy) to put event generation code all over the place like this.  I do understand the reasoning and am sympathetic but personally I would never be able to decide when/where to generate what events so some guidance would be useful.  It would (hopefully) lead to some level of consistency in the events.  Last thing we want is to have the events be all different with different naming, granularity, properties, ...

Also, we should carefully consider the eventing mechanism for scalability.  It is one thing to gen some events to keep the user informed as they click around but completely different if you are a provisioning agent managing a 1000 node server farm or 1000000 cell phones.  That level of events is likely to be lost on anyone.
Comment 3 Susan McCourt 2007-08-14 17:33:35 EDT 
I think we should look at IResourceDelta for guidance here, or at least explain where we diverge from that approach.

I certainly wouldn't expect the provisioning core to know, for example, that a producer changed a repo name or repo property.  That's what "refresh" is for.  I see that as similar to a user changing a workspace file outside of eclipse.  

But from my point of view, if the change happens inside the platform, meaning that the repo manager or repo API allows you to change something about a repo, then it should generate an event, even if it is as granular as "added", "removed", "changed"
Comment 4 John Arthorne 2007-08-15 04:08:42 EDT 
> There are changes to repo properties, but there are other
> things I can change via API, such as a repo name, and I need notification when
> those things change.

It seems really odd to me that repositories would be modifiable by the user like this. I thought the only purpose of the writeable repository APIs was for things like the generator that are programmatically constructing repositories at development time. In end-user workflows, do we really see repositories changing their properties often?  If this is a rare event, or something that only happens on the tooling side, it may be sufficient to just have "repository added" and "repository removed" events, and a changed repository could just be notified as a remove followed by an add.
Comment 5 Pascal Rapicault 2007-08-15 11:54:26 EDT 
I completely agree with John. Repositories are immutable objects (except the few ones that are writable, but we should assume that most of them are read-only). 
Could you elaborate under which circumstances the repo properties would change (except tooling / authoring time)?
Comment 6 Susan McCourt 2007-08-15 12:36:36 EDT 
The only end-user scenario I imagine right now is adding/removing repos.  It's not clear a user would ever want to leave a repo view open, but nonetheless, even update manager today has a wizard where you add new repos.  So if the UI lets you add or remove a repo, it needs to reflect this.  Right now I have to fake an event scheme so that the views are propertly updated.

The "change" event is driven by the fact that the admin UI currently shows the name of the repo, and if it's writable, lets you change the name.  This came out of the M1 discussion of adding a repo name.  I agree this is likely something only done by an administrator, but even so, if some tooling UI allows them to do it, I'd rather not fake an event scheme to accomplish updating the UI.  I don't care so much if it's an add/remove or change event, although an add/remove event would cause superfluous updating and flashy/jumpiness. 

(A separate concern is whether there are user names for repos but that is already covered in bug #194224, and obviously the UI would be in charge of managing those names).

I'm okay with just supporting add/remove for now, and as the UI evolves, if we see that changes happen more frequently than imagined, we could add a change event.
