Comment 9 Wassim Melhem 2005-03-07 17:02:34 EST 
Konrad, please use the latest patch as a starting point to provide manifest.mf 
validation in PDE.

Make sure all the issues pointed out above are addressed.

We will add more validation as we go along.

Two things that render the manifest.mf invalid which the parser does not 
inform us about, the parser only rejects it:

1. the length of characters in a single line.  Jeff can tell us the maxiumum 
beyond which we should flag an error.

2. Gratuitous/missing line breaks.

So if the parser rejects the file, we check for these two things manually and 
see which one breakage scenario applies.  These are the two most common ones.

If neither applies, just put a big red X on the file at line 0, and say that 
the file is malformed but the cause cannot be determined.


Jeff, what does the runtime use to parse the manifests?  do you also use the 
JRE manifest parser or something more sophisticated?  With the JRE parser, we 
get minimal information and we have to do a lot of manual guessing in the dark.

Comment 12 Thomas Watson 2005-03-07 17:23:57 EST 
We use our own internal manifest parser because the OSGi minimum execution 
environment does not include the java.util.jar package.

We do not provide much more feedback than the Manifest class from the JRE.  
But we happen to be lenient on the maximum line length.  We do not put a limit 
here.  I'm not sure why the Manifest syntax does ...

I don't really look forward to making our manifest parser more robust for 
tooling to use.  I'm getting beatup about the size of the Framework already.  
Adding these types of features for tools is going to start becoming 
troublesome.

We definately do not indicate anything about extra or missing line breaks.  
Does the JRE Manfest parser reject these?  Seems like it is valid syntax (to 
have extra/missing line breaks) just not really what the user intended.
Comment 13 Wassim Melhem 2005-03-07 17:34:06 EST 
Yes it does complain about new line characters.  For your amusement, take a 
look at bug 64616.  Things get funny from bug 64616 comment 21 and onwards.

In general, if anything goes wrong syntactically, the JRE manifest parser just 
throws an exception and we have no way of knowing what happened and where it 
happened.  

Whenever these syntactic errors happen, we keep hearing that PDE should add 
more validation bla bla bla.  But given the lack of information from the 
parser, finding syntactic error on our part is like a shot in the dark, based 
on people's previous findings.
Comment 14 Jeff McAffer 2005-03-09 20:58:11 EST 
As we discussed, PDE should have its own manifest parser that gives line 
numbers and more information about problems.  The ones listed in comment 9 are 
a good start 

- continuation lines (starting with a space) that have more than one space
- lines which start with a tab (user may have thought that was ok as a 
continuation)
- lines that start with a space but have something that looks a lot like a 
header (e.g., foo: or some such).  User may have accidentally inserted a 
space.  This one is tricky as the continued value may well have be a URL or 
other string containing a :

Sherry, note that in your latest patch, you introduced a circularity in the 
dependency graph.
org.eclipse.pde is a headless plugin.
org.eclipse.pde.ui is a UI plugin that depends on org.eclipse.pde.

could cover all the validation that we are doing.

Similarly, illegal attributes can cover wrong values for known directives.

Comment 28 Konrad Kolosowski 2005-04-26 19:08:29 EDT 
Validation added, (mostly by M6).  Thanks Sherry, I used much of your code.

Headers are extracted and parsed into header name and value using pde's own 
parser base on code in org.eclipse.osgi.framework.util.Headers.
Header value is parsed into elements using  
org.eclipse.osgi.util.ManifestElement

Validated are:
Jar manifest syntax (line length validated at 512 bytes, not 72)
Bundle-SymbolicName
	value
	singleton attribute
	singleton directive
Bundle-Version
	value
Eclipse-ExtensibleAPI
	value
Fragment-Host
	value
	bundle-version attribute
Bundle-ClassPath
	value (only if not empty)
Require-Bundle elements
	value
	bundle-version attribute
	visibility directive
	reprovide attribute
	resolution directive
	optional attribute
Bundle-Activator
	value
Plugin-Class
	value
Export-Package elements
	value
	version attribute
	specification-version attribute
Provide-Package elements
	value
	specification-version attribute
Import-Package elements
	value
	specification-version attribute
	version attribute
	resolution directive

Is OK to close, or is something missing from the list that must be validated 
for 3.1 release?
