Comment 8 Martin Oberhuber 2006-12-15 12:28:45 EST 
We have a similar problem in bug 168231:

RSE-SDK includes a CDT integration ("remotecdt") that is meant to be optional. Consequently, the remotecdt feature is declared "includes:optional" in rse-sdk. Remotecdt itself requires feature cdt.

When rse-sdk is installed but cdt is not, the configuration is considered invalid, and UM fails to install anything. My naive expectation would have been that the "remotecdt" feature, since its prerequisites are not met, disables itself and thus makes the configuration valid - but that's not the case. In fact, manually disabling the remotecdt feature is a workaround for the problem.

The general problem is that if feature X wants to integrate with features A,B but these integrations are optional - how does one declare that?

Comment 11 Dejan Glozic 2007-11-02 10:31:23 EDT 
Note also that Update Manager is working as designed. Its original specification was to prevent people from shooting themselves in the foot. Installing features that are missing prereqs begs for trouble and UM cannot tell if being more forgiving will be OK or result in a catastrophe (say, not being able to start Eclipse).

Note for example comment #6:
> I had the same trouble when I added just the xml editor plugins (and their
> dependencies) from the wtp but not the whole wtp. I had missed a couple of
> plugins that did not effect the functionality of the xml editor, but when I
> tried to update subclipse, the update manager did not allow me to.

This implies advanced logic in UM to figure out when a missing dependency is benign. We don't have such a code. I suggest passing your ideas to the upcoming new and shiny Equinox provisioning that will replace UM in 3.4.

The problem is: if any prerequisite of any already installed plugin is missing. The update manager refuses to work. To follow your metaphor: it you cut your finger, the update manager thinks you shoot yourself in the foot and therefore refuses to work (the attitude is: "I don't help injured people")

Most tools would allow you to install after giving you a warning that this might invalidate your installation. The current anti-foot-shooting update manager simply refuses to work.

If an airplane would  work like this: if something goes wrong, refuse to work. Imagine you at 10,000m height, the airplane self analysis finds out that your reading light is broken. Well something is wrong with the plane. Lets stop the engine and refuse to work to prevent the passengers from "shooting themselves in the foot"..... 

This is exactly the approach that the current update manager takes. It's to stupid to understand what is going wrong and therefore refuses to work assuming the user is totally stupid. Your configuration is broken? The update manager is innocent because it does what it is designed to do....
