Description Amy Wu 2007-07-23 23:19:42 EDT 
+++ This bug was initially created as a clone of Bug #160548 +++

Two problems of the JSP validator:
1) If an attribute is set to required in TLD and is not specified in JSP, the validator should mark it as error or warning.
2) In JSP if I type a wrong attribute name which is not in TLD, the validator should also mark it as error or warning.
Comment 1 Amy Wu 2007-07-24 00:27:31 EDT 
I would like to backport this fix for this bug from WTP 2.0 to WTP 1.5.5.  It is a hotbug for IBM because previous releases of IBM products contained this functionality, but it is missing from the versions built on WTP, so this is viewed as a regression to customers of IBM products.

There is no workaround for this bug because the problem is with the validator *not* reporting an error/warning on non-existing attributes and missing required attributes for jsp action tags.

The fix has been tested by me and will be tested by component lead.  Since this fix is a backport, it has also been well tested in WTP 2.0.  I've also backported the JUnit tests associated with fixing this bug.

The details of the fix were explained in bug 160548 comment 3:
"This patch will add a new validator, JSPActionValidator, to the JSP validator. 
It will check jsp action tags (jsp tags, custom tags) for missing required
attributes & undefined attribute names.

It also adds a source validator so that partial checking can be done on source
validation.

It also adds a unit test to verify the validator is working."

The one minor change for this fix that is different than the one in WTP 2.0 is that due to NL/UI constraints, I was unable to add the new validation errors/warning strings to WTP 1.5.5.  Instead, I reuse an existing string "Undefined attribute name <name>"

I also want to note that I had to add a dependency on wst.validation for jst.jsp.ui (this was done in WTP 2.0 as well).  Since jsp.ui has a hard dependency on sse.ui, and sse.ui has a hard dependency on wst.validation, this should not cause any problems for users that upgrade to 1.5.5.

This bug will be reviewed by component lead before asking for PMC review.

There should be little risk in fixing this bug since the fix has already been in WTP 2.0 and the fix is limited to jsp validation only.

Comment 11 Amy Wu 2007-08-01 13:59:23 EDT 
While it looks like a lot of code changes in the patch, if you don't count the unit tests, it's actually:
- JSPActionValidator - new class that validates the attributes of JSP action tags 

- JSPActionSourceValidator - new class that turns JSPActionValidator into a source validator (mainly calls into JSPActionValidator)

- JSPBatchValidator - a few lines of code added so that JSPActionValidator is included in JSP batch validation

- plugin.xml - a few lines of code added to register the jsp action validator as a source validator

So the real "meat" of the fix is isolated to the new validator class JSPActionValidator.

Since this is a backport of a fix in WTP 2.0, it was well-tested via fvt/smoke test in that release (and the backport did not depend on any other WTP 2.0 code).  Nitin and I also tested the backport of the fix in WTP 1.5.5.  I've tested this fix by editing and validating several JSP files to make sure it works as expected and nothing regressed.
