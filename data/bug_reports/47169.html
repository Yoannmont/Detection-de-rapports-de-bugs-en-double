Comment 12 Jared Burns 2003-11-22 22:59:10 EST 
Chris and I found a possible scenario here. If a thread has already been resumed 
for a step or evaluation (as with a watch item) before the background label 
provider is asked to fetch labels for its stack frames, the labels will come 
back with junk and the label provider won't recompute the labels when the thread 
suspends again.

One way to fix this is to check if a stack frame's thread has been resumed when 
the label is processed. If it is, we then add the thread to the list of "resumed 
threads" maintained by the label decorator. This will assure that the stack 
frame labels will be recomputed when the thread suspends at the end of the step, 
evaluation, etc.
Comment 13 Jared Burns 2003-11-24 19:57:00 EST 
The LaunchViewLabelDecorator used to respond to thread resume events by looking through the 
queue of elements and adding the thread to a collection if we had any stack frames from that 
thread queued up. Then when a thread suspend event was received, we checked to see if the 
thread was in the collection. If it was, we then recomputed its stack frame labels. However, this 
had serious timing vulnerabilities.

I've changed the label decorator to ignore resume events. Instead, it checks stack frames as it 
pulls them off the queue. If their thread is no longer suspended, the thread is added to the collection. 
The behavior on suspend is the same as it was. This fixes the problem by making sure that the 
stack frame labels are recomputed eventually.

While I was at it, I made some changes to reduce the amount of flicker in the LaunchView. I made 
changes to the LaunchViewEventHandler to queue up a background label computation directly in 
"labelChanged()" instead of telling the UI to refresh. This eliminates some extra "<pending...>" 
updates.

I also changed the queueing mechanism slightly to put non-stack frame elements on the beginning 
of the queue. This lets debug targets and threads get updated a little sooner. I just think the 
presentation is better this way.

Please verify changes to LaunchViewLabelDecorator, LaunchViewEventHandler, and 
LaunchView, DarinW.
Comment 14 Jared Burns 2003-11-25 12:17:17 EST 
Added some more cleanup code to LaunchViewLabelDecorator. We now remove "resumed 
threads" from the collection when their debug target terminates. This is an 
added defense against a memory leak in the collection.
