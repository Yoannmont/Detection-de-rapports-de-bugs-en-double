Description Edoardo Comar 2004-09-28 17:50:53 EDT 
Actually it's ProjectionViewer (ancestor of JavaSourceViewer ) where the bug is.

ProjectionViewer overrides doOperation(int) from 
org.eclipse.jface.text.ITextOperationTarget#doOperation(int)

and deals with CUT 

while its superclass correctly fires a verify event when text is CUT,
that can be picked up in the editor by code like :

getSourceViewer().getTextWidget().addVerifyListener(
       new VerifyListener() {
	     public void verifyText(VerifyEvent event) {
             ....
             }});

in ProjectionViewer the handling of CUT is such that no VerifyEvent is fired.
IMO one should fix 
private void ProjectionViewer.deleteTextRange(int offset, int length, StyledText
textWidget) 

by firing the event before invoking 
getDocument().replace

- comment 0 (missing VerifyEvent)
- bug 185777 (advanced delete operations).

The problem in bug 185777 is that e.g. JavaEditor.DeleteNextSubWordAction#setCaretPosition(int) just calls viewer.getDocument().replace(..), and this doesn't expand foldings. Note that exactly the same problem happens when the document is directly modified by other means (e.g. by choosing "Open With > Text Editor" and deleting the text there).
