Comment 15 Stefan Xenos 2005-01-04 21:49:48 EST 
Cause of this bug: 

When the top part is removed from a PartStack, the stack selects a new tab based
on activation order. The presentation gets the new selection and makes the new
part visible. If the part hasn't been lazily loaded yet, this causes it to be
loaded.

The bug here is that there is no way to defer the selection change until the end
of a batch of removals. When closing all editors, this will cause a lot of
editors to be activated. The problem seems intermittent because it depends on
the order in which editors are closed. It will only occur when a lazily loaded
editors gets closed after the active editor.

I suspect that no activation events are being fired because the editors aren't
being activated -- just opened and brought to top.

This was probably working in previous builds because the part creation is being
triggered by EditorPane(PartPane).setVisible(boolean) line: 323, and in older
builds setVisible simply ignored an attempt to make a part visible if it hadn't
been activated yet (which is another bug).

From what I can tell, there are three solutions to this.
1. Roll back the change to PartPane.setVisible (easy, but kinda scary - will
probably cause other bugs).
2. Add a method to PartPane that defers updates to the presentation (much like
SWT's setRedraw flag), and defer presentation updates while closing a batch of
editors.
3. Add a removeMultiple(List) method to PartPane that allows multiple parts to
be removed at the same time, and only performs one selection change at the end.

I'm working on a patch that uses approach 2.
