Description Boris Bokowski 2006-12-15 23:23:29 EST 
Currently, SafeRunnable opens a modal dialog.  This is problematic because of the side-effects of spinning the event loop (see e.g. bug 154571), and because sometimes many dialogs are displayed after another, or closing one dialog leads to another.  SafeRunnable should use the new error handling facility, and the error handling facility should collect similar errors together to present them in one dialog instead of in many.

Comment 23 Francis Upton IV 2008-02-08 16:29:43 EST 
Created attachment 89316 [details]
Yet another proposal on 3.4M5

It seems that since the patch was originally made, a SafeRunnableDialog has since been added as the means of displaying errors that happen in SafeRunnable.  This is trying to accomplish a similar thing as the work for this bug I think.  

My patch does the following:

1) Change IStatusManager (since it's an abstract class) to AbstractStatusManager.
2) Make a new interface IStatusManagerStyles that has only the style constants; nothing would inherit from this, but the constant values would come from this interface.
3) Removes the SafeRunnableDialog entirely; this functionality is already in the Workbench error handling stuff.  I verified that multiple invocations of the handler (caused by multiple exceptions in SafeRunnable) will correctly display in one dialog when the workbench is active.  I would need some guidance on how to make an automated test for this.
4) Added a small Jface test case for this, just to make sure nothing bad happens.

I ran the JFace and UI tests and they all pass.
Comment 24 Boris Bokowski 2008-02-12 13:59:03 EST 
At first glance, the patch looks good, but: "Removes the SafeRunnableDialog entirely" - this is problematic:

Some clients use just SWT and JFace standalone, i.e. without the Workbench. The JFace viewers use SafeRunnable when they call into third-party code like label providers. This can lead to situations where SafeRunnable needs to handle multiple errors that happen in rapid succession. This is why we had SafeRunnableDialog in JFace.

Alternatively, we could detect when we already have a non-modal dialog open and log the additional errors while changing the dialog's message to "Multiple errors occurred. Please check the log." Not sure if this would simplify things enough to warrant a change of the (currently working) code in SafeRunnableDialog.

I actually took this approach first.  The problem I had was no matter which way you did it, you would create compiler warnings.  By having the constants in AbstractStatusManager, it would create compiler warnings for the existing code that referenced them in StatusManager.  Another alternative is the leave the constants in both places StatusManager and AbstractStatusManager, referring to the values in the superclass, but this then would create compiler warnings in StatusManager (because of hiding a field).

I assume that creating new compiler warnings in the API is not acceptable.  If it is acceptable; I would go with your initial suggestion and do away with IStatusManagerStyles.
