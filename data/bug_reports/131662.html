Comment 7 Linda Chan 2006-06-14 00:13:15 EDT 
Delivered the feature in the oda.jdbc driver to use a JNDI Name Service to look up a Data Source resource factory to get a pooled JDBC connection.  The feature is implemented in the existing oda.jdbc plugin, instead of a new ODA driver, in order to facilitate its use at design-time, plus to simplify deployment installation.

A new connection property is added to the ODA JDBC data source definition for the JNDI Name to look up a Data Source name service.  This optional property expects a full path, for use by a JNDI initial context to look up a Data Source resource factory.  A JNDI name path can be specific to individual JNDI service provider and data source configuration.  For example, "java:comp/env/jdbc/<dataSourceName>".

A text field is also added to the oda.jdbc.ui data source designer pages for user input of the JNDI Name URL property value.  
(To limit the scope of UI changes in BIRT 2.1, driver-specific template for the JNDI Name property value is not supported at this time.)
This feature has been tested with Apache Tomcat 5.5.17 and BEA WebLogic Server 8.1 SP5.

-----------------
Below are some description of this feature and its intended behavior:

<i>JNDI Name service vs. JDBC driver URL</i>

Some JNDI service providers do not support client-side access.  
During design time when using the BIRT report designer, a JDBC data set would need to be designed using direct access to a JDBC driver connection.  The ODA JDBC data set query builder continues to use direct JDBC connection to obtain its metadata.  Only those design functions directly related to a data source design, such as Test Connection and Preview Results of a data set, are enhanced to first attempt to use a JNDI name path, if specified.  And if not successful for any reason, it falls back to use the JDBC driver URL.

Similarly at report runtime, such as during Report Preview, when a JNDI name path is specified, the oda.jdbc run-time driver attempts to look up its JNDI data source name service to get a pooled JDBC connection.  If such lookup is not successful for any reason, it falls back to use the JDBC driver URL directly to create a JDBC connection.

<i>Context Environment Properties</i>

To simplify the task of setting up the JNDI initial context environment required by individual JNDI application, the oda.jdbc JNDI feature supports the use of a "jndi.properties" resource file, installed in the drivers sub-folder of the oda.jdbc plugin.  
When deployed within a web application, it looks for the file in the web application's folder tree for the oda.jdbc's drivers sub-folder.
Its use is optional.  When such file is not found or problem reading from it, an initial context adopts the default behavior to locate any JNDI resource files, as defined by javax.naming.Context.
Note that it is the user responsibilty to configure the proper classpath for the classes referenced by the environment properties.

<i>Potential Future Enhancements</i>

I can think of some potential future enhancements in a future release. For example, support the definition of a driver-specific application resource file and/or a template for the JNDI name path.
It may also be useful for the oda.jdbc.ui data source designer to ask user to specify whether to apply the specified JNDI data source name at design-time.  For example, now when the Test Connection action succeeded, it is not apparent whether it has succeeded using the JNDI data source name service, or has fallen back to using the JDBC driver URL.

Any input from the community on useful enhancements is much appreciated.   Please submit a new Bugzilla entry for your enhancement request.

<i>oda.jdbc.driverinfo extensions<i>

When a custom connection factory is associated with a specific JDBC driver class, by implementating the oda.jdbc.driverinfo extension point, it is responsible for the handling of the JNDI look up service.  In other words, a custom connection factory defined in a driverinfo extension overrides the default JNDI data source handling.

---------

A Special Thank You to Victor Pyankov --
Even though the eventual implementation is different from yours (in order to support design-time usage), your contribution has certainly inspired the delivery of this feature.
