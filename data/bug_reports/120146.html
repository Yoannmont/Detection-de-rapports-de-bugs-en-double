------- Comment #1 From Jim DAnjou 2005-12-08 19:32 [reply] -------

I ran WTP in debug mode. The statement throwing the exception is: 
class: PulishUtil.copyFile(InputStream in, IPath to, long ts)
statement: if (!tempFile.renameTo(file))
                throw new Exception(NLS.bind(Messages.errorRename,
tempFile.toString(), file.toString()));

The problem seems to be that the target file path in variable file is 265
characters which is > 256. 

The same project publish to Tomcat OK because the path for the same class file
is 209 characters.

Publishing to Websphere uses a longer path than publish to Tomcat.

Tomcat path uses path that looks something like this:
C:\eclipse\workspaces\workspace-wtp-eval5\.metadata\.plugins\org.eclipse.wst.server.core\tmp4\webapps....

The Websphere path is considerable longer and looks something like this:
C:\eclipse\workspaces\workspace-wtp-eval5\.metadata\.plugins\org.eclipse.core.resources\.projects\com.ibm.xxxx.xxxx.server.wtpWeb\org.eclipse.jst.server.websphere.core\

We can shorten workspace path name and maybe the project name to get under 256
but it would helpful trim path length contibuted by WTP. For example in the
overall path the project name is included twice. 


I trimed my workspace path to get under the 256 character limit issue. The
publishing succeeded. That is an acceptable workaround for 1.0 but needs to be fixed in 1.0.1, please. I am aware that that the path limit is a JRE issue and some 1.5 JRE's have fixed it. We need to use a 1.4 JRE
