Description Antoine Toulmé 2007-12-20 08:16:49 EST 
See news://news.eclipse.org:119/fgmnno$6im$1@build.eclipse.org for context.

we would like to use DTP in our product, and define a datasource using parameters defined elsewhere in the project or workspace properties. This datasource would then appear by default when opening the data explorer view.

The datasource would be defined according to server settings defined in the preference store of the project, each project could potentially target a different server.
Comment 1 Brian Fitzpatrick 2007-12-20 10:07:01 EST 
In additional comment on the newsgroup, Antoine mentioned:

>I think the settings could be saved on a preference store per project.

In thinking about this some more, I think this may be an interesting case for a specific type of project-based repository profile in the DSE. But it wouldn't be very automatic unless we added an action to the project (perhaps the project has a particular nature that would allow us to tell which projects this would apply to) that allowed the user to create one of these project-based repositories in the DSE or if the profile was created during the project creation process or in the preference pages...

So one possible series of events...

1) User creates a new project with these special connection properties. 
1a) Perhaps as part of the project creation process, a new repository profile is created associated with the new project.
2) User right-clicks on project and says "Create DTP Profile" (or something similar)
3) When the new repository profile is connected, it checks the project properties for the appropriate connection details and creates an in-memory profile in the repository based on those details. 
4) The user can then connect to that internal profile and browse normally.

Something along these lines anyway. 

Is that along the lines of what you were thinking?

--Fitz
Comment 2 Antoine Toulmé 2007-12-20 10:16:32 EST 
I think a nature per project is a good idea.

the IProjectNature of the project could return the profile.
You would just need to define an interface extending IProjectNature that would define something like getProjectProfile().

You would need to define one extension point that would say:
this nature is a DTP nature and we can expect it to implement IDTPProjectNature.

Then when loading the view, grab the project, iterate over the DTP natures registered, and get the natures that define profiles. 

You can ask each of them their getProjectProfile(). Is it possible to merge profiles eventually ?

Does this sound possible ?

Comment 6 Hugues Malphettes 2007-12-20 13:47:42 EST 
I had a slightly different suggestion for this functionality.
If you would be so kind to review this idea I'd be happy to try coding it. 

A testcase: let's say we are a WTP-plugin aware of a deployment to a J2EE server that holds a datasource. It would be nice to contribute to the list of DTP connection profiles the connection to that datasource.

To do this we would use a DTP extension point where we an interface is implemented. For example "IProfileConnectionContributor".

IProfileConnectionContributor#getContributedProfiles() returns a list of IProfileConnection.

IProfileConnectionContributor#isContributedProfile(IProfileConnection) returns whether this profile can be edited by the user.

So DTP loads the list of persisted profile connections.
Then it goes through all the declared IProfileConnectionContributors.
Grabs the profiles generated there. If a profile is read-only, it replaces-or-insert the profile in the list. Otherwise it inserts it only if the profile was not there.

In my opinion, this is lighter than using IProjectNature to plug those.

Please let me know if that would make sense.

Comment 8 Brian Fitzpatrick 2008-02-15 12:50:06 EST 
Hugues, I do like your approach better than using IProjectNature, which locks us into putting the profile provider mechanism with projects and that may not always be the case.

That said... I'm concerned about the editable/not editable quality of profiles produced via your IConnectionProfileContributor (I renamed it slightly to match the IConnectionProfile interface nomenclature that already exists).

How would the user edit one of these profiles coming from a non-DTP source? The implementer of the IConnectionProfileContributor would have to provide such UI if it existed, as it may not be a type of profile that DTP knows about.

Antoine's thought was that the implementer contributing the profiles would need to provide methods for editing at whatever level they wanted to do so. 

So... This leads me to some other questions about this approach. 

The implementer would need to contribute a few connection profile-related bits and pieces to the mix. The connection profile extension would be used to identify a connection factory used to generate a connection to said profile. And you'd need to specify a category. But you wouldn't need to contribute a wizard or property page. So implementation on the DTP side would be minimal except for whatever would be contributed to the DSE's tree via navigator extensions. 

I think this is doable. We might have a few minor glitches here and there like disabling/hiding the Properties menu for a connection profile in the DSE that's contributed via one of these IConnectionProfileContributor classes. But I believe the InternalProfileManager could provide these "contributed" profiles to the list and display them appropriately in the DSE. 

So if you feel up to coding something, feel free. Know however that the 1.6 M6 ganymede drop of DTP is coming hard and fast on the heels of EclipseCon in March, so time is going to run out quickly for this type of API change.

Sorry for the delay in getting back to you. I held off focusing on the Usability side of things until this week to make sure we got as many comments from folks as possible and am just now getting back to that effort. 

--Fitz
