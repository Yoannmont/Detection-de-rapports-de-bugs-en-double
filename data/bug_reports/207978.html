Comment 9 John Arthorne 2007-11-28 17:56:30 EST 
Susan, do you think there's significant value in separating the meta-info about the repository from the loading of the IUs in the repo? I.e., is there much you can do with a repository without asking for its units?  Currently the meta-info and the IUs are stored in a single file, so it would be non-trivial to split this out.  

The fundamental question is whether the laziness should be in the repository manager, or within the IMetadataRepository instances. I.e., should it be really fast to instantiate an IMetadataRepository, and then perhaps slow to perform queries on it. Or, is instantiating IMetadataRepository always going to be expensive.  I can think of two fundamental approaches:

1) A simpler IMetadataRepositoryManager API that essentially just tracks a list of URLs, and is able to instantiate repositories for clients when requested. Something like this:

public void addRepository(URL repository);
public void removeRepository(URL toRemove);
public URL[] getKnownRepositories();

public IMetadataRepository createRepository(URL url, String name, String type);
public IMetadataRepository loadRepository(URL url, IProgressMonitor progress);

With this shape, IMetadataRepositoryManager could still cache IMetadataRepository instances if it wanted to (likely with a soft reference), but it wouldn't be guaranteed. Obtaining an IMetadataRepository instance is always potentially expensive for clients.

2) IMetadataRepositoryManager always holds onto IMetadataRepository instances, as it does today, and its API stays roughly the same.  IMetadataRepository implementations are responsible for being space-efficient, flushing their contents when not in use, and lazily repopulating.

Approach 2) is better if you need to be able to quickly access the repository meta-info (name, description, etc) without necessarily performing queries on it.   Approach 1) is more of a pain for clients, but allows the repository manager to be very simple and efficient.
Comment 10 Susan McCourt 2007-11-29 12:15:33 EST 
The main gain to be had with separating the metainfo is in the admin UI, where we show the repo name and URL.  If getting the name loads the whole repo, we get the long startup times we see today.  In the end-user UI, we show all "available features" from all repos, so separating the metainfo doesn't help.  If you want to see what's out there, you have to pay. 

As it is now, touching a repo manager the first time loads everything, which is bad.  I can live with reading all of a single repo's content as long as it only happens when I need it.  (Approach #1).  Then if that is still too slow, we could always consider not showing the name in the main repo views of the admin UI.  The problem I see with approach #2 is that no matter how efficient a repo implementation is, if it's a remote repo and the connection is slow, then things will be slow, and we should only be penalized when we are working with that particular repo.

That said, we should probably focus our efforts on what is slow in the end-user UI.  The main problem is adding a remote repo.  It takes close to 30 seconds on my machine.  What I really want is a quick validate of the URL and add without getting content.  (Approach #1).  Then the time will be spent on the refresh of the "available features" list where we would expect it, rather than having an async dialog/long progress monitor during a simple add.
Comment 11 John Arthorne 2007-11-30 12:06:41 EST

Now that I think of it, if it's just for the admin UI you could reach directly into the preferences to get this info without loading the repository today.

Comment 20 Scott Lewis 2007-12-24 14:13:27 EST 
I'm not familiar enough (yet) with the artifact repository API to know whether this is helpful or not, but ECF just added an asynchronous directory browsing API to the ECF file transfer API:

http://wiki.eclipse.org/ECF_Filetransfer_API_Bundle
As of now we've got providers for 'file:' URLs, as well as 'efs' URLs (e.g. efs:<efsprotocol>://<host>/path).  We'll be working on others (http, ftp, scp) as soon as possible.

The entry point interface is IRemoteFileSystemBrowserContainerAdapter

URLs are used to refer to remote directories (i.e. FileIDFactory.getDefault().createFileID(namespace,url))

http://www.eclipse.org/ecf/org.eclipse.ecf.docs/api/org/eclipse/ecf/filetransfer/IRemoteFileSystemBrowserContainerAdapter.html

Further, another possibility is the use of the ECF remote services API for access to remote OSGi services

http://wiki.eclipse.org/ECF_API_Docs#Remote_Services_API

Rather than blocking remote calls, it might be helpful to use asynchronous invocation via IRemoteService.callAsynch 1/2, or fireAsynch.

http://www.eclipse.org/ecf/org.eclipse.ecf.docs/api/org/eclipse/ecf/remoteservice/IRemoteService.html

Please let me know if I/we can be of further help with this bug.
Comment 21 John Arthorne 2008-01-04 14:48:37 EST 
Artifact repository support has been released. Both repository managers now reference only repository locations, and maintain a SoftReference to concrete repository implementations if clients have loaded them. This way, all references to large data structures is eventually flushed when no provisioning operations are ongoing. There is still some work on the UI side to avoid holding long term references to IArtifactRepository, IMetadataRepository, and Profile, but I believe there are separate bugs for this.
