Comment 10 Tod Creasey 2006-02-10 15:57:18 EST 
I don't think we want to do anything special for this beyond adding a High Contrast theme in our theme selections. 

We would get it wrong when high contrast is changed on the fly (no callback)> Alsotrying to scale existing fonts and colours frequently results in funny sizes as frequently the value we calculate will not map to an available font or colour.

We certainly don't want to change the jfacefonts.properties as this is a lot of work for the nl teams already.

I think the right answer to this one is to create a HighContrast theme (or themes) that the user can select rather trying to use Display#getHighContrast(). Display#getHighContrast() is not supported on a lot of platforms. On top of that there are frequently totally opposite modes (there are at least 3 on Win 2000) that all High Contrast - for instance white on black vs. black on white.

We would need a series of themes most likely - we can do a reasonable default white on black one (this is the XP default) but people may want to add more.

One possible use is to do the getHighContrast call on startup and choose the theme based on that (i.e. default if false, High Contrast if true).
