Description Anthony Hilario 2006-08-07 20:36:12 EDT 
It appears that when the PDE resolves the linked MANIFEST.MF and plugin.xml files in the manifest editor and when resolving dependencies, etc., that it expects plugin.xml to be at the root of the project. This does not work well with systems such as Maven and any project directory structure that does not have files such as those placed at the root of the project. For example, with Maven, META-INF/MANIFEST.MF and plugin.xml would be placed in src/main/resources as opposed to the project root, but the PDE does not exactly support doing this. When updating a MANIFEST.MF file that is located somewhere other than the root of the project, all updates are directed into a plugin.xml file at the root of the project, regardless of where the actual plugin.xml file may be. It is my suggestion that the PDE resolve plugin.xml relative to the parent parent directory of MANIFEST.MF, or that it be optional to specify the exact location of both files.

Comment 7 Alex Blewitt 2006-10-07 17:44:25 EDT 
Let's be clear here: the restriction is on the PDE side, not the Maven one. Maven allows resources to be placed anywhere; PDE requires them to be at the root. Maven *can* use resources from anywhere (and thus is often bodged around for PDE builds; though Maven more usually generates the Manifest.MF file on the fly).

It is PDE that cannot be configured to put the Manifest.MF anywhere else.

Maven developers often have a 'resources' directory, which can be configured as a source folder in Eclipse builds such that anything in the resource directory is copied across to the build directory (and thus available from e.g. Class.getResource() etc.). Thus, developers *want* to put the resources somewhere else, it's not Maven that's forcing them to.

Ergo, it's PDE that needs to be made more generic; no changes to Maven are required. What Maven is doing is both demonstrating and encouraging a better partitioned directory structure that PDE cannot currently support.

You have also not explained the rationale for why the Manifest.MF has to be where it is *in the project structure*. It is clear that when exported, in the Jar, it must be in /META-INF/MANIFEST.MF. However, that's no reason for requiring that it be that in your source folder; as an example, the WTP uses a folder 'WebContent' that includes, amongst others, META-INF and WEB-INF subdirectories. Why is that PDE is incapable of dealing with a restructuring like this?

I propose that the only files that are needed in the project root are the .classpath and .project files; the .classpath can specify the location of any (sub)directories as output folders, and then files such as MANIFEST.MF can follow the standard Class.getResource() that works already to locate the file. The loading of other bundle resources (e.g. Bundle.getEntry()) could also delegate to the Class.getResource() and find the resources from the build folder(s) that exist in the current project. 

In summary; there is no logical restriction why the project structure needs to be driven based on the output structure.

Comment 9 Alex Blewitt 2006-10-07 20:11:17 EDT 
(In reply to bug 154079 comment 11, in which this subject has been touched upon but discussion copied here since this is the right bug for discussing it)
> the edit/save/run behaviour we all know and love about Eclipse is in part
> enabled by having the manifest in the correct location (where correct is from
> the OSGi runtime point of view).

There's a difference between what the OSGi runtime expects, and the PDE mode. After all, the OSGi 'expects' the classes to be in the root of the bundle by default, but PDE massages the Bundle-Classpath: to add the current project's output folder onto the location. Furthermore, there's no reason why the project structure need map to what the OSGi runtime expects; for example, given a project structure:

/src/java/org/example/Demo.java
/src/resources/META-INF/MANIFEST.MF
/bin/org/example/Demo.class
/bin/META-INF/MANIFEST.MF

with the .classpath looking like:

	<classpathentry kind="src" path="src/java"/>
	<classpathentry kind="src" path="src/resources"/>
	<classpathentry kind="output" path="bin"/>

then the OSGi runtime just need be pointed to the 'bin' directory, and not the project root. So in this setup, the OSGi runtime would get exactly what it expects, and the resources would be copied over (just like any other Java project) into the bin directory. In fact, you could do this today with a Java project (and people do).

> It would be theoretically possible to do one of 
> a) have the manifest somewhere else and then tell Equinox where it was when
> launching in dev mode

You don't need to. It's in the output folder, as defined by the .classpath. In fact, whatever mechanism the JDT uses to obtain resources from the current output path could be exactly the same mechanism used to obtain the MANIFEST.MF.

Point taken re: distinction between the getEntry and getResource; I'm not suggesting that PDE actually use getResource() to find the MANIFEST.MF in the first place, but it seems entirely reasonable to query JDT's state of the project and for each 'bin' type in the .classpath, use that as a prefix for searching for the manifest file.

> OR
> b) have a builder that copied the manifest from wherever it was to the root of
> the plugin project.

This doesn't solve the problem; the goal is for the manifest *not* to be at the root of the project.

> If we did something I would be in favour of #1 however, Dejan's point is dead
> on.  We could fix this one case but there are other plugins that put resources
> (stuff) in the root of the project.  

There's a distinction between (a) resources that have to be there for build purposes (which would include .project, .classpath, build.properties, custom.xml etc.) and (b) resources that need to be present in the final output binary and/or on the classpath. The goal here is to move everything of type (b) into different directories, not everything. (Note that (a) and (b) may intersect; however, the entire of (b) should be moved, even if they're in (a) as well.)

It also makes the build much cleaner; you could move e.g. icons, plugin.properties, and all those other resources underneath the 'resources' directory. The Bundle.getEntry() could be updated to follow the same search path in PDE mode, which is just to scan the contents of the output directories (as mentioned in the .classpath). This has the nice effect that the output of a plugin basically becomes Jaring the contents of the bin/ directory, without any further work needed (or even, as you have commented in the past, being able to run from that bin/ directory as an 'exploded' bundle). It also means that pretty much every build.properties would become:

bin.includes = bin/
src.includes = src/

regardless of what other resources you might happen to have. It also allows for resources to be put in a directory structure that *you* want, without having to put it in the project root, so:

src/resources/icons/example.gif
->
bin/icons/example.gif
->
out.jar!icons/example.gif

Currently, there's no way of having resources in a bundle without exporting the whole directory structure under which they sit.

> For the education of the Eclipse community, can someone outline the drawbacks
> of having these files in the root of the projects?  I don't want to argue with
> them, I just want to understand.

It's not just those files; it's every resource. Let's put it another way; when Eclipse first started, the default was to have the source and output directories the same e.g. src='.' and bin='.'. However, people wanted to have a separate directory structure for Java source files and compiled code. Similar arguments ensued 'Why would anyone want this?' -- but (I'm glad to say) Eclipse became more flexible, and now that's the default.

This is the same situation. All resources *must* be in the root of the project. 'magic' files, like MANIFEST.MF and plugin.xml have to be in the root. Not only that, all other files need to be as well; whether it's plugin.properties, or icon/action.gif, or my/config.xml. There is no way of either setting up a PDE build with those resources other than in the project root, without having extra directories on the front (e.g. if I have resources/icon/action.gif, it *will* be out.jar!resources/icon/action.gif). 

It's pretty much become a de-facto standard in the wider Java community (as seen by the open-source projects using Maven, but not limited to those e.g. Ant's project structure includes a 'resources/' subdirectory, and that's not build with Maven ;-) http://svn.apache.org/viewvc/ant/core/trunk/src/resources/ ) for resources to be packaged as one source directory, and the Java files to be packaged in another source directory. Maven has certainly encouraged the adoption of those structures; but it almost certainly didn't start it and it's not the only type of use in existence. 

It should be observed that the JDT has no problems with this structure; the .classpath is simply setup such that the resources folder is just another Java Source Folder that happens to contain no Java Source files. Many projects built with Eclipse JDT are set up in this way.

The big problem is that many people see this as a restriction on PDE, and that PDE isn't flexible enough to accommodate how they want to layout (or even build) a project. Much like the 'create a separate build folder' argument before; it's like saying 'Why do we need separate output and build folders? What we've got works, why can't you adjust your ant build to use the same directory as the source folder and build folder?' It's not that tools or processes can't be adjusted to work around constraints like this, but it does result in workarounds because of PDE's constraints. Some coding standards actually require projects to be structured in a particular way, including putting all resources into a named directory (e.g. resource/). With PDE, you cannot do this.

So, the drawback of forcing them to be in one location is that it explicitly breaks how people want to structure their code. Furthermore, it's clearly not a problem for JDT -- because people can (and do) structure projects like this currently. There's even integration with Maven in the form of a ClassPathEntry for JDT that allows all the dependent Jars to be driven from a Maven pom file (which yes, has to be in the root directory :-) but is a compile-time file and not part of the generated output, ala .classpath and .project). That works great.

PDE just feels like it's stuck in the Eclipse 2.0 days, and is following the mentality of every resource having to be in the project root. Times (and developer opinions) have changed, and they want to be able to structure their projects differently, or even use different tools. There has been a reasonable amount of interest in getting Eclipse apps built with Maven (see last year's EclipseCon presentation) but PDE forces you down a road that feels as unnatural to Maven developers as today's Java developers feel about having a source and build directory be the same.

I'd also posit that people don't mind if there are some tool-specific files in the project root (e.g. .classpath, .project etc.) in order to allow a build to go ahead smoothly. But there should be a distinction between developer-time files (e.g. build.properties) and run-time files (e.g. plugin.xml) -- where 'run-time' includes 'run-time-and-developer-time' (e.g. MANIFEST.MF). 

After all, isn't that exactly how WTP handles the WebContent subdirectory, under which the META-INF and WEB-INF sit? All we're asking here is for PDE to have the same flexibility in being able to move the run-time resources to a location of our choosing (and you can pretty much guess it's going to be either 'resources' or 'main/resources' or some such).

> As for the last part of comment 7, please distinguish between "resources" and
> "files that happen to be in the bundle".  The former are loaded by a
> classloader.  The latter are accessed via Bundle.getEntry() etc.  The manifest
> in particular is not a resource since it defines the nature of the classloader
> and so by definition has to be loaded before there exists a classloader.

I'm not distinguishing it on purpose; I'm using the term to encapsulate both 'files that happen to be in a bundle' and 'files that happen to be loaded by the classloader'. So, for resource, please read 'file in the bundle that may be loaded directly or via the classloader'. You'll note that e.g. the SVN repo for Ant above and the standard Maven layout (http://maven.apache.org/guides/getting-started/index.html#How%20do%20add%20resources%20to%20my%20JAR?) that the term 'resources' is used pretty much for any runtime data that needs to be exported as part of the output, which (in this case) would include MANIFEST.MF.

I'd also like to point out that at present, there's a real struggle (for RCP developers) to understand the difference between Bundle.getEntry and Class.getResource. That's because they have different behaviours depending on whether you're in PDE-Runtime or exported RCP application. Bundle.getEntry("/hello.txt") and Class.getResource("/hello.txt") in a typical Java bundle will result in different files being loaded:

/hello.txt <- loaded by Bundle.getEntry
/bin/hello.txt <- loaded by Class.getResource

However, when the bundle is exported and run as a separate application, they both end up at the root of the exported bundle (and so at runtime, have the same effect). It would be *so* much easier if the proposed solution outlined above (where bin/ becomes the root of the bundle as far as the Equinox runtime is concerned) because then Bundle.getEntry and Class.getResource have the same behaviour both in PDE-Runtime and in the exported application. You have no idea how many questions come up on EclipseZone about this, and the perceived wisdom is to generally copy the files into both the project root and a source path just so that the testing behaviour works the same.

If you were ever going down a road where you could have two plugins hosted in the same project, you could use this structure to your advantage:

/MyProj/org.example.plugin1/bin <- Bundle.getEntry/Class.getResource looks here for plugin1
/MyProj/org.example.plugin1/src/java
/MyProj/org.example.plugin1/src/resources/META-INF/MANIFEST.MF
/MyProj/org.example.plugin2/bin <- Bundle.getEntry/Class.getResource looks here for plugin2
/MyProj/org.example.plugin2/src/java
/MyProj/org.example.plugin2/src/resources/META-INF/MANIFEST.MF

I merely mention this as an observation of something that could follow on from this change, rather than as a co-dependent requirement.

In fact, it's also worth noting in this bug that Peter Kriens of OSGi thinks that the structure of the project is too restrictive, from http://www.osgi.org/blog/2006/07/eclipse-pde-versus-jdeant.html

"The PDE discussion [on the equinox-dev list http://dev.eclipse.org/mhonarc/lists/equinox-dev/msg01272.html] evolved around the flexibility of the environment. It got started by a request from the maven people to make the location of the manifest flexible. This touched a nerve with me because it is right at the heart why I am not using the PDE. I unfortunately still use the normal JDE because the PDE does not cut it for me. ...

"The reason why I am not using the PDE architecture is because it contains a number of harsh constraints. My key problems are:

"A project = a bundle. It is impossible to deliver multiple artifacts from the same code base with the provided tools. For the OSGi build it would mean 300++ projects.

"The project directory layout must look like the root of the bundle, well eh, sort of. The PDE does perform some magic under the covers before it runs your code or packages the bundle, but the idea is that there is a 1:1 relation.

"The manifest must be up to date at all time, forcing the developer to enter the dependency information in literal form. There is no possibility to generate information for the manifest during build time.
...
"Despite my complaints, rants, or other kinds of obnoxious behavior with the Eclipse guys, I did not get one iota further. I have been going through Eclipse 3.0, 3.1, and 3.2 without making even a dent in this model."

The point is that PDE works to satisfy people who want to develop in Eclipse, but it doesn't encourage people to do so. It's something that has to be tamed, and adapted to (rather than the tool adapting to the way you want to work). Furthermore, there doesn't seem to be any reason why it needs to be this way; the project structure I outlined above seems to be equally capable of hosting a PDE runtime and to boot makes the deployment process simpler in Eclipse. (It also opens up the possibility for other build tools like Maven to become more used for this process.)

I don't really know what else I can say, other than if you look at any other open-source project on the web today, you'll find that they have a separate directory (or subdirectory) for storing resources that are merged in to the output Jar, whether they're built by Maven or Ant. I can imagine that if you've only been exposed to the PDE world for the last 5-10 years, and you're used to doing it the PDE way, then it wouldn't occur to want to do it a different way; but the 43 e-mails discussing the subject on the mailing list suggests that it does touch some nerves.

(I'm going to cut'n'paste this to 153023 because it strictly belongs in that bug, but this bug has yet to be updated with a dependency on bug 153023 which actually is to do with moving plugin.xml etc.)

Comment 14 Alex Blewitt 2006-10-08 08:09:34 EDT 
Re comment 13: OK, let's not use the term 'resource'. Let's use the term 'file' instead. That way, we're clear that we're not discussing OSGi resources or OSGi entries. 

Let me also explain that I don't expect that the behaviour of getEntry to become the same as getResource. For one thing, the getResource will load the resource from the same place that the class is loaded from (which may include individual Jars within a bundle e.g. from Bundle-Classpath: .,example.jar; getResource would have different behaviour depending on whether the class was loaded from the '.' or example.jar)

What I was trying to suggest was the getEntry could be modified to search the output folder(s) of the associated JDT project, rather than the project root. Somewhere inside the PDE initialisation, you've got something akin to 'Bundle.install(/MyProject)' to bring the project in-line. I'm suggesting changing that to allow 'Bundle.install(/MyProject/bin)' instead, or whatever other output folder(s) are denoted in the .classpath. That way, the source for the files can be determined by standard JDT build-and-modify processes. So, to restate: I'm not suggesting getEntry *use* getResource to work (which we both know wouldn't be appropriate) -- I'm just suggesting that Bundle.getEntry(/a.file) resolves to /MyProject/bin/a.file at runtime, and that Bundle.getEntry(/) resolves to /MyProject/bin/ (or whatever the output folders are configured for that project).

It might get a bit tricky if there's multiple output folders for the project, but I'm sure that that could be worked around.

As for the community stepping up to help -- I agree, that would be a way forward (isn't helpwanted the right keyword to add?). However, it's going to impact a number of places, not to mention the build process, so it's unlikely to be something that can be solved with a small patch (as well as finding the other locations that META-INF/MANIFEST.MF is hard coded). This is a big change that is going to need to get help from inside, not just the odd contribution AFAICT.

Guys, you're thinking *way* too much just specifically about MANIFEST.MF and plugin.xml.

Each project has a 'default build directory' (specified in the .classpath, which Eclipse knows how to locate) and the simplest solution is to view the bundle's contents (both resources, bundle entries, plugin, manifest etc.) to be viewed as rooted from there. So, you'd have:

bin\META-INF\MANIFEST.MF
bin\plugin.xml
bin\icons\icon.gif
bin\com\foo\Example.class
bin\thing.properties

Now, this 'bin' directory becomes the root of the expanded bundle (if desired) with no Bundle-ClassPath hacks, and one can 'jar xf bin' to create the deployable bundle if requested. Frankly, it doesn't matter how those files end up in the bin directory (e.g. they could be copied via the standard Java builder by putting a separate source folder -- like src\resources -- in the .classpath; or alternatively, the MANIFEST.MF could be programmatically generated from pom.xml or whatever).

There's two separate issues:

1) Given a runtime PDE, make it look for bundle entries and classes under project/defaultbuildoutput instead of project/
2) Having set up the resources such that they get copied into bin\ from a more structured project layout, how the PDE editor knows where to find MANIFEST.MF (and/or the IClasspathEntry) -- though frankly, if the MANIFEST.MF is viewed as some build's output, then actually the IClasspathEntry could be driven from the output, not the input, anyway.

It would seem trivial for 2) for a lookup to search 
if (project/META-INF/MANIFEST.MF exists)
  return that
else
for srcFolder in srcFolders
  if (project/srcFolder/META-INF/MANIFEST.MF exists)
    return that 

Bear in mind that for the case of PDE build (which would also have to do something different in this case) that most people structuring the project in this way are almost certainly not using PDE build but using something else, like Maven, to build instead. So whilst it would be a follow-on from this, it wouldn't be necessary at this stage to be useful to others. The Harmony project, for example, has its own build mechanisms, but would find it extremely useful not to be limited to having META-INF/MANIFEST.MF  (and other bundle resources) rooted at the top level of a project.

Guys, anyone else on this bug who agrees please post a comment on here -- otherwise the PDE team is fixated on just trying to find a new location for the MANIFEST.MF which is unlikely to suit most of the requirements that others have made in related duplicated bugs.

Comment 23 Ben Spiller 2007-09-28 06:54:08 EDT 
I was very surprised when I discovered the way PDE forces the project root location to be used for MANIFEST.MF, plugin.xml and also - not sure if this has been mentioned before - any bundled classpath jars. JDT allows resources and source files to be located anyway at build time (via 'Link Source'), and jar file dependencies can come from anywhere on the file system.

Examples of why this is annoying/limiting in practice:
1) like many people, we have a separate location in our source repository for third party jars we depend on, away from the source files on each development branch. PDE forces us to commit these jars into our repository alongside the sources files, just so they can be under the project root. There's no good reason to do this - you can setup the JDT build path flexibly, why not PDE?

2) we need our build scripts to make changes to MANIFEST.MF at build time (e.g. to set the Bundle-Version for all our plugins to the current product version number, without having to change each plugin separately). Normally, we'd keep source files (checked into subversion) in one directory and the derived files generated by the build would be put into separate per-platform directories... but PDE requires us to put all files under the project root, so this separation is impossible (hence the different per-platform builds may race with each other to put the generated file under the project root).

I really feel PDE should let developers structure their applications in whatever way they need based on internal processes, policies, etc., ... not force everyone into a particular restrictive structure.

Comment 30 Darin Wright 2010-03-22 10:34:22 EDT 
Created attachment 162673 [details]
modified example

Here's a modified version of Alex's bundle showing how PDE allows for a flexible MANIFEST location in the workspace. I basically modified the pde.core preferences in the .settings folder to specify where the bundle root is with this:

BUNDLE_ROOT_PATH=src/main/resources

I also moved build.properties to the bundle root folder, and modified it's entries to be relative to the bundle root.

Programmatically, the bundle root can be set with the IBundleProjectDescription API (new in 3.6). We have not revealed this in the UI yet. With the modified example I can run an Eclipse Application, start/stop the bundle, hit breakpoints and export the bundle from the IDE.
