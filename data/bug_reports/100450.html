Comment 11 Tom Morris 2005-06-28 16:49:41 EDT 
Well, after several hours of debugging, I figured out the trigger.  I'm so
annoyed at IBM's sense of priorities, I'm tempted to let someone there replicate
the work -- but I won't.

The problem was introduced on June 2 in rev 1.339 of Workbench.java by someone
deciding that the startup splash screen needed beautifying.  How the heck does
someone get approval to beautify a splash screen in a *release candidate*? 
Doesn't a project lead or release manager or *someone* have to sign off on
changes this late in the process?  

It's pretty clear from looking at the volume and type of changes introduced
throughout the project between RC1 & RC2 that the project had slipped and,
rather than adjusting the schedule or feature set, the quality metrics were
lowered instead.  Not good.

So what's the fix?  Replace 'shell' with 'null' in the following line from
org.eclipse.ui.internal.Workbench:

	final ProgressMonitorDialog progressMonitorDialog = new
StartupProgressMonitorDialog(null/*shell*/);

It's clear this isn't the root cause of the problem, but rather a change that
exposed a latent race or memory corruption bug elsewhere.  I have some ideas
where the root bug may lie, but need to investigate further.
Comment 12 Billy Biggs 2005-06-29 13:16:56 EDT 
Tom, thanks a ton for all of the hard work you have put into this problem.

I really hoped that narrowing down which change would help to figure out the
problem but unfortunately I think it's just aligning the planets or causing some
memory corruption that burns us later.  To try and get further, I went through
all of the crash logs posted.  Here is what I know so far:

 - The crash is during IUnknown->Release() on an accessibility object on the
   CLabel created by CompareViewerPane.
 - The CLabel in question does not have any text.
 - The disassembly at the crash is:  call [eax*4+ecx], so this leads me to
   believe it is the pointer to the IAccessible object which is bad or already
   released.
 - In every stack trace posted so far (not just by Tom), the address loaded
   is 0x74c83a58.

One theory is that the pointer returned by CreateStdAccessibleObject is being
released too many times.  To check, add two references in
Accessible.Accessible(Control) and print the return value of
iaccessible.Release() in Accessible.internal_dispose_Accessible().  I tried
calling Release() too many times on one of our objects, but the resulting VM
crash does not seem to quite match those posted.

Maybe the problem is due to the lack of text in the label.  To test this theory,
you could try to set the text of the label to be "nocrash" either in CLabel's
constructor or more specifically in CompareViewerPane and see if this stops the
crashes.

Comment 17 Billy Biggs 2005-06-29 16:04:14 EDT 
It's not a debug DLL, it's with a proposed change.  The code in question is here:

  static HMODULE hm = NULL;
  if (!(hm = GetModuleHandle(CreateStdAccessibleObject_LIB)))
    hm = LoadLibrary(CreateStdAccessibleObject_LIB);

GetModuleHandle() doesn't increment the reference count on the library, which
means that it could be unloaded.  The DLL posted is supposed to change this to
always call LoadLibrary() instead.  Unless maybe we posted the wrong DLL - are
they really identical in size?  :)
