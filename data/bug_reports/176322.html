Description Chris Aniszczyk 2007-03-04 17:53:53 EST 
The ECF code base should be examined to move away from various usage of Eclipse runtime classes. They instead should move to Equinox specific things.

Here are some examples:

- use of Platform for the convenience methods (getExtensionRegistry, ...) 
- use of Plugin to get debugging and logging info 
- access to classes that are now in org.eclipse.equinox.common
Comment 1 Jeff McAffer 2007-03-04 21:00:04 EST 
An easy way to get an idea of what is involved is to simply remove org.ecclipse.core.runtime from the manifest and add in org.eclipse.equinox.common/registry.  The resultant compile errors are candidates to be addressed.  I believe that some of the things you find will have documented alternatives (javadoc or in the 3.2 migration guide).  As a whole though we should write a "Getting off the runtime" wiki.
Comment 2 Scott Lewis 2007-03-05 13:19:16 EST 
I'm starting off on this bug by working on the following three plugins

org.eclipse.ecf.identity
org.eclipse.ecf
org.eclipse.ecf.filetransfer

I thought I would start with the following:

> - use of Plugin to get debugging and logging info 

The main (perhaps only) offender in the ECF code is the org.eclipse.ecf.util.Trace class in org.eclipse.ecf.identity plugin.  This is just a helper class
to make debugging/logging info output easier.  In Trace, we call Plugin.isDebugging().   I couldn't find anything in the 3.1->3.2 migration guide (or the 3.2->3.3 migration guide) about replacing Plugin.isDebugging().

I found the following comment in 3.3m5 version of Plugin.isDebugging()

     * @return whether this plug-in is in debug mode
     * XXX deprecate use the service and cache as needed

So I would be happy to remove all references to Plugin, but Plugin does define an 'isDebugging()' method that our Trace class uses to provide tracing support for any/all ECF plugins.  I would be happy to either

a) throw out/remove the ECF Trace class completely (if there is a 'standard' helper for tracing that we don't currently know about)

or

b) Remove references to Plugin in our Trace class by using the service referred to in the above comment.

So any pointers to info for either 'a' or 'b' strategies above would be appreciated.

Comment 7 Jeff McAffer 2007-03-13 17:57:55 EDT 
The easiest way to track down a replacement is to work backwards from the problem class and see where you get to the real information.  In general Platform and Plugin are just front-ends to deeper function.  For example in the case of Plugin.isDebugging() if you drill down you will find
private DebugOptions getDebugOptions() {
	if (debugTracker == null) {
		if (context == null)
			return null;
		debugTracker = new ServiceTracker(context, DebugOptions.class.getName(), null);
		debugTracker.open();
	}
	return (DebugOptions) debugTracker.getService();
}

Once you have a DebugOptions service you can ask it questions like 

String key = symbolicName + "/debug"; //$NON-NLS-1$
String value = InternalPlatform.getDefault().getOption(key);
this.debug = value == null ? false : value.equalsIgnoreCase("true"); 

In the case of AdapterManager there does not appear to be a service available.  See bug 177224.

Comment 12 Jeff McAffer 2007-03-14 14:24:50 EDT 
Yes, you should stop subclassing Plugin.  Most of the facilities it gives you are available elsewhere.

The logging story is not the best but there is a FrameworkLog service that actually is the underlying mechanism used to write the .log files.  Unfortunately the API is somewhat lower level.

public FrameworkLog getFrameworkLog() {
	if (logTracker == null) {
		if (context == null)
			return null;
		logTracker = new ServiceTracker(context, FrameworkLog.class.getName(), null);
		logTracker.open();
	}
	return (FrameworkLog) logTracker.getService();
}

One thing that might be good in practice is to optionally use Platform.getLog(Bundle) if the Platform class is around and otherwise use the framework log.  it would be a bit of complexity but you should be able to isolate it.  We can help with that if needed.

Comment 16 Jeff McAffer 2007-03-16 11:53:47 EDT 
This is great Scott.  Thanks!   The state location issue is one I had not considered...  Here is a workaround for now.  Not API but it will work.  

/* package */static final String F_META_AREA = ".metadata"; //$NON-NLS-1$
/* package */static final String F_PLUGIN_DATA = ".plugins"; //$NON-NLS-1$

try {
	filter = context.createFilter(Location.INSTANCE_FILTER);
} catch (InvalidSyntaxException e) {
	// ignore this.  It should never happen as we have tested the above format.
}
instanceLocationTracker = new ServiceTracker(context, filter, null);
instanceLocationTracker.open();
Location l = (Location)instanceLocationTracker.getService();
if (l == null) 
	return null;

Path path = new Path(l.getURL().getPath());
return path.append(F_META_AREA).append(F_PLUGIN_DATA).append(bundleName);


See bug 177783
Comment 17 Jeff McAffer 2007-03-16 11:59:54 EDT 
in reply to comment 13, I would suggest trying for the Platform log first and then using the framework log and then the log service.  There is no particular way of testing for Platform class exept for loading it
