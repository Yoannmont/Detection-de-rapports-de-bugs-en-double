Comment 5 Thomas Watson 2004-02-24 09:21:12 EST 
Most of the OSGi headers are a ',' separated list of elements that have a 
single value (optionally a list of values) followed by an optional list of 
attributes or modifiers.  Here is the general syntax:

Manifest-Header ::= element ( ',' element )*
element ::= element-value (';' element-value)* (';' attr-key '=' attr-value)* 

Maybe you could use a general eclipse header like Eclipse-Parameters (bad 
name).  It could look like this:

Eclipse-Paramenters: automate; start=true; stop=false

This way you have the flexibility to extend this header to allow for other 
parameters to the EclipseAdaptor without introducing a new header.  For 
example, we could have an element that indicates that when installing a bundle 
from a jar the bundle should be extracted onto the file system.  When running 
out of jars certain bundles my want to force the framework to extract them to 
the file system (swt probably would so that the libraries do not need to be 
extracted from the jar).  You could then add an element value like this:

Eclipse-Parameters: automate; start=true; stop=true, install; extract=true

The install element could tell the adaptor what to do during install of the 
bundle.

Comment 10 Jeff McAffer 2004-03-16 00:08:02 EST 
moving a relevant comment from a duplicate...

Talked with Jeff about this, and it seems to be interesting to have support for
the contrary situation as well: a mainly non-autoactivatable bundle that may
have some code that would cause auto-activation.

The syntax for the Eclipse-Parameters header as proposed in bug 52393 would be
extended to support an additional attribute: start-exceptions, which would
enumerate package prefixes that would be treated differently than the rest of
the code. It could be specified whether auto-activation is enabled or not. 

Example:

Eclipse-Parameters: 
  automate; start=true;
start-exceptions="org.eclipse.foo.util,org.eclipse.foo.internal.util"; stop=true

Comment 16 Thomas Watson 2004-03-18 22:31:16 EST 
I'm not against additional headers for eclipse.  I only recommend using the 
standard OSGi syntax because that is what the ManifestElement parser 
understands.  This is the syntax from comment 5.

A header in the form
Eclipse-Legacy: legacy=true, plugin-class=foo.MyClass

will not be understood by the ManifestElement parser methods.
Comment 17 Jeff McAffer 2004-03-18 22:37:45 EST 
Yes, I was playing fast and loose with the syntax but is there a reasonable,

Comment 18 Thomas Watson 2004-03-18 22:52:01 EST 
If you want to use the ManifestElement then your syntax needs to have at least 
one value followed by an optional list of attributes

Eclipse-Parameters: 
 automate; start=true;
 start-exceptions="org.eclipse.foo.util,org.eclipse.foo.internal.util";
 stop=true,
 legacy; plugin-class="foo.MyPlugin"

This example has two values (automate and legacy) and each value has its own 
set of attributes.  Its not pretty but if you want something else I would 
recommend the opposite which is just creating a new header for each eclipse 
option like:

Eclipse-AutoStart: (true|false)
Eclipse-StartExceptions: <list of packages>
Eclipse-AutoStop: (true|false)
Eclipse-Legacy: (true|false)
Eclipse-PluginClass: <class name>

Comment 21 Rafael Chaves 2004-03-21 13:45:22 EST 
Just released some changes regarding this to HEAD. 

The current state is:
- Legacy and Plugin-Class headers were preserved (keeping bug open to handle that).
- two new headers are being supported:
Eclipse-AutoStart: (true|false) [;exceptions="<package1>[,...<packageN>]"]
Eclipse-AutoStop: true|false

The default for Eclipse-AutoStart and Eclipse-AutoStop is "false", unless
Legacy: true is specified (this is for temporary backward compatibility, and
Legacy will soon not be taken into account for this regard).

PluginConverterImpl was changed to generate the headers above for converted plugins.

The EclipseClassLoader was changed to take Eclipse-AutoStart into account.

I also moved the plugin stopper to the adaptor (BundleStopper), making use of
the new FrameworkAdaptor#frameworkStopping callback provided in bug 55315. I
removed the shutdownHook extension point, and the corresponding extension
provided by runtime.compatibility.

Comment 25 Rafael Chaves 2004-04-12 18:01:32 EDT 
From a talk with Jeff: Eclipse-AutoStop seems to be actually the wrong solution.
Its only value was to avoid an auto-activated bundle to have its persistent
state forgotten. A more reasonable way to support that is to have a flag in
BundleData that remembers whether the bundle was auto-activated during
classloading or not. If it was, we should ensure its active state is not
persisted (BundleStopper would take that flag into account instead of looking
for the Eclipse-AutoStop header when doing its work).
