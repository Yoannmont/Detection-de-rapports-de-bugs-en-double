Comment 2 Tim deBoer 2006-09-11 18:24:21 EDT 
The reason this is failing during publish appears to be because the server adapter delegate is returning a null entry in the array returned from ServerDelegate.getChildModules(). This trickles through all the way to ServerPublishInfo.getKey() where it causes the NPE. I can add an extra check here to ensure that the null isn't passed on, but that's not really where the problem is coming from, and this could still fail in other cases

---

Received note from Elson Yuen. The IBM server adapter is pretty simple and appears to be returning a null entry in this case directly from the return value of org.eclipse.jst.server.core.IEnterpriseApplication.getModules(). IEnterpriseApplication is implemented in WTP by
org.eclipse.jst.j2ee.internal.deployables.J2EEFlexProjDeployable. I've taken a quick look at getModules() (implemented in org.eclipse.wst.web.internal.deployables.ComponentDeployable) and I don't see any way that this could return a null entry in the array.

At this point this bug is pointing to an error in ComponentDeployable that doesn't exist, and it has already involved three separate teams. We'll need to take another look at this tomorrow and may need some help from the teams involved to track this down.

Comment 5 Tim deBoer 2006-09-12 13:55:16 EDT 
Current status:
Web Services is creating a new project and then calling server tools API (ServerUtil.getModule(iproject)) to get a reference to the module and add it to the server. See AddModuleToServerCommand.java above.

This method sometimes returns null, causing an array with a null entry to be created and passed to ServerUtil.modifyModules(). This fails correctly and the user receives an error, but the null entry still gets added and causes the server to fail during publishing later on.

My guess right now is that either the facet cache or the module cache is failing, out of order, or the resource change events aren't getting fired fast enough and causing neither to be updated and see the new module. See bug 124023 for another example of this.

One possible temporary fix is probably to put a delay or build in to ensure that the resource change events have finished, but it's not clear to me that this will work in all situations given the issues in bug 124023. We'll need to continue to investigate.

As above, these exceptions are all downstream problems based on "IModule imodule = ServerUtils.getModule(iproject);". After it returns null, things go downhill b/c there is a problem adding "null" as a module to the server.

Comment 10 Angel Vera 2006-09-22 11:36:07 EDT 
Since the workaround is acceptable for 1.5.1 release, I am doing a brain dump of the investigation for this bug, so that we can come back later and fix the problem.

The problem is related to the ModuleFactory being out of date because the ResourceChanged events have not reached the Server.

CallSequence:
              (jst.ws) 
AddModuleToServerCommand.execute->
  ServerUtils.getModules---------->
              (wst.server) 
   ServerUtil.getModule(IProject)-->
     ServerUtil.getModule()--------->
      ServerUtil.getModule()--------->
     ModuleFactory.getModules()-------> (here is the problem)
            
Because the ResourceChange events (happening in a different thread) have not reached the server, the modules that the ModuleFactory contains are invalid. 

By using the suggested workaround, the AddModuleToServerCommand forces a clear of the cache in the ModuleFactory, allowing it to pickup the new modules. Thus, the causing the desired behaviour.

For 2.0 we need to re-think the strategy for clearing the cache. 

I will attach 2 traces showing the differences one with the workaround suggested, and one when not using the workaround. I will also add the code that I used to trace this problem.
