Description Dave Orme 2005-06-21 09:01:46 EDT 
With RC3, the Run as... -> SWT Application choice is now gone--and with it the
easiest way to develop and debug SWT applications when parts of a development
team run different operating systems.

The proposed replacement of using <classpathentry ...> nodes in the .classpath
file is broken because it puts platform-specific information into the .classpath
file.  The result is that Windows developers will require a different .classpath
file from Linux developers from MacOS developers.  Whenever anyone syncs to CVS,
they will wind up stomping on the .classpath file and breaking the build for
everyone not on the same platform.

While I can appreciate the need for a method of running SWT JUnit tests as well
as SWT applications, this is *NOT* it!


For reference, I am quoting Veronika's message to the platform-dev newsgroup
that announced this change.


-----

    * From: Veronika Irvine <Veronika_Irvine@xxxxxxxxxx>
    * Date: Wed, 15 Jun 2005 15:07:32 -0400
    * Delivered-to: platform-dev@eclipse.org

For 3.1 RC3, the action Run As ... -> SWT Application will be removed from 
Eclipse.

In 3.1 M7, a feature was added to the .classpath support making it 
possible to specify the location of the SWT JNI libraries in the 
.classpath file:

        <classpathentry exported="true" sourcepath="src.zip" kind="lib" 
path="swt.jar">
                <attributes>
                        <attribute value="org.eclipse.swt" 
name="org.eclipse.jdt.launching.CLASSPATH_ATTR_LIBRARY_PATH_ENTRY"/>
                </attributes>
        </classpathentry>

The advantage this approach has over the  "Run As SWT Application" action 
is that it works for Standalone SWT Java Projects and also for Standalone 
JUnit tests that use SWT.

The SWT team would like to thank the PDE UI team for the work put into 
providing and supporting "Run As SWT Application".   We appreciate all the 
effort this required.  However, since there is a more general mechanism 
now available, we think SWT should take advantage of it.  You know how we 
like to be like everyone else <g>.

If you are importing SWT into your workspace from the SWT plugin shipped 
with Eclipse, here are the steps to make it work:

1) Select "Import..." from the File menu.
2) Select "External Plug-ins and Fragments" in the Import wizard and click 
on the "Next" button.
3) Select "Import As Binary projects" and make sure the target platform is 
correct.  Click on the "Next" button.
4) Select org.eclipse.swt and org.eclipse.swt.<ws>.<os>.<arch> and click 
on the "Add" button. 
(<ws>, <os>, and <arch> should be replaced with the values for the 
platform you wish to run against.  e.g. org.eclipse.swt.gtk.linux.x86)
5) Click on the "Finish" button.
6) Select org.eclipse.swt in the Package Explorer.  From the right mouse 
menu, select "Properties".
7) Click on "Java Build Path" and then click on the "Libraries" tab.
8) Click on the "Add Jars..." button.
9) Select 
org.eclipse.swt.<ws>.<os>.<arch>/org.eclipse.swt.<ws>.<os>.<arch>_3.1.0.jar 
and click on the "OK" button.
10) Expand org.eclipse.swt.<ws>.<os>.<arch>_3.1.0.jar in the Libraries tab 
and select "Native library location".
11) Click on the "Edit..." button.
12) Click on the "Workspace..." button  in the "Native Library Folder 
Configuration" dialog.
13) Select org.eclipse.swt.<ws>.<os>.<arch> and click on the "OK" button.
14) Click on the "OK" button in the "Native Library Folder Configuration" 
dialog.
15) Click on the "Order and Export" tab in the "properties for 
org.eclipse.swt" dialog.
16) Place a check against org.eclipse.swt.<ws>.<os>.<arch>_3.1.0.jar.
17) Click on the "OK" button.

Now your Standalone SWT Java Project can depend on org.eclipse.swt and you 
can launch your application with "Run As -> Java Application".

In the near future, we intend to make the steps above available as an 
Import wizard so it will be simple to use.

Comment 3 Dave Orme 2005-06-21 09:35:45 EDT 
This is not simply a dupe of 100792.

The steps in your message do not involve adding a new <classpathentry to
.classpath, but rather involve importing SWT into the workspace, and then
depending on it.

But your message indicates that the 17-step program I now have to enroll in will
be deprecated in favor of the simpler .classpath file solution at some point in
the future.

This bug is primarily pointing out that the <classpathentry solution described
in your mailing list message (that doesn't appear to be implemented yet) seems
to me to be broken becuase it involves introducing platform-specific information
into the .classpath file.

I could be mistaken here--if CLASSPATH_ATTR_LIBRARY_PATH_ENTRY in the
<classpathentry is a variable that will be substituted at launch time with the
real path, then I don't have a problem with this solution.  But since this is a
fairly large change that is happening very late in the release cycle, I wanted
to make sure that I voiced my concern earlier rather than later.

But I will go on record saying that having to import SWT, wizard or none, is not
something a novice programmer will think of.  It's usability is *significantly*
less than the usability of the Run as... -> SWT Application.  I do not support
that as the final solution for 3.1.

But I didn't that that is what you were saying in your mailing list posting, so
I hope that's not an issue.
Comment 4 Dave Orme 2005-06-21 09:43:14 EDT 
Okay, I just re-read your message and I now understand that the new
<classpathentry goes into the imported SWT plugin.

In that case, I have re-titled this bug report to reflect my position that this
loweres usability compared with the old solution.

I suggest a solution where the <classpathentry is added to the project that
*references* SWT and uses an internal variable to point to the SWT JNI
libraries.  Or something like that.
