Description Panagiotis Korros 2006-08-12 03:45:53 EDT 
By using wtp 1.5 (and the latest 1.5.1 build) I found out that sometimes validation errors in jsp pages are not detected even after a clean. This is happening when the jsp editor is open on the file that has the error.

It seems that the validator somehow relies on the state of the jsp editor and this should not happen. The validator should be completely intependent from the editors.

This is a very important defect cause the user expects to see the correct number of errors reported in the problems view after a clean or an explicit run of the validator.

Steps to reproduce:

1. create a jsp file like the one attached and leave the jsp editor open
2. create a jspf file like the one attached and leave the jsp editor open
3. edit the jspf file, create a compilation error in it and save
4. explicitly run the validator from the menu on the jsp file, or cause a full build by cleaning the project

   the error is not detected!!!
5. close the jsp editor and repeat step 4

   the error is detected

Comment 3 Amy Wu 2006-08-14 10:45:10 EDT 
This appears to be the same problem mentioned here:
https://bugs.eclipse.org/bugs/show_bug.cgi?id=121937#c10

Which is supposed to be covered by bug 147098.  The exact problem is that errors in the included jspf are not correctly displayed in jsp file including them when the jsp file is open in the jsp editor and validate/clean is performed.

The problem has to do with our current limitation of not being able to immediately recognize that when the included jspf is modified, jsps including the jspf file should update their jsp/java translation model.

Users will only run into this problem when they have their jsp file open while they edit any included jspf files.  And this will only stay a problem until the user goes back and edits their jsp file because as soon as a change is made to the java scripting area inside the jsp, the jsp/java translation model will be updated again.

Also, while the jsp file including the jspf file is not immediately updated with the validation error, the jspf file now is, so the user is able to see that the fragment has an error.

Comment 5 Amy Wu 2006-08-21 12:45:21 EDT 
Okay, I see what you are saying now.  There are 2 big issues:
1) getting the right model to validate based on what type of validation we want to do (source-latest/dirty model vs batch-last saved model)

2) reporting the problems in the open editor (there will be confusion over what is reported from batch validation in the problems view vs source validation in the editor when editor is dirty)

For problem #1, currently, I believe we only generate/allow 1 model per file, so when the file is open in the editor, the model we use is always going to be the latest/possibly dirty model.  To get this working, we may need a change to the model manager, and it will probably be a pretty big change.

For problem #2, errors marked in the editor are only updated when something is changed/typed in the editor.  So if an error is fixed/created via batch validation, it will not be reflected in the editor until something causes the source validation to run.

Comment 8 Amy Wu 2008-05-06 19:09:06 EDT 
(In reply to comment #7)
> Amy, wasn't the bug with reporting validation messages fixed during 2.0?
> 

Yes, I believe so, but this bug is reporting a different issue.  I verified this problem still exists in WTP 3.0 M7.

As shown in the example in the bug description, the user has a jsp file open that includes a jspf file. The user then edits the jspf file to cause an error and then saves the jspf file.  However, when the user then tries to validate the jsp file, the user should get an error saying there's something wrong with the jspf file included in the jsp page.

From what I remember about this bug, the problem was that we always use the latest jspf translation ie the current model.  What we should do is when we're doing batch validation, we should always grab last saved model/build jsp translation from last saved file.
