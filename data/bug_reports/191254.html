Comment 27 Andrew Overholt 2007-10-29 17:35:32 EDT 
(In reply to comment #25)
> The major problem here is Linux distros insist to use GCJ for everything so I
> think most people that use Eclipse on Linux are getting it from their distro.

I know you know the reason for this, Chris (and some of the rest of you), but in case others don't, I'll explain:

- many distros (Fedora, Debian, etc.) won't ship non-free (free as in speech, there's some variety in terms of what's allowed) software
- the popular JVMs (Sun, IBM, BEA) fall into the category of software that isn't shipped in these distributions
- the distros *do* want to ship Eclipse
- the distros often (Fedora does, and I'm pretty sure Debian does as well) have a policy of not shipping code that they haven't built themselves or can't rebuild entirely with the distro itself
- gcj provides a free java-like environment
- hence Eclipse is built with (well, and with ecj ;) and by default run with gcj

The problem(s) described in this bug arise when people download from eclipse.org and attempt to run on a system that has gcj.  gcj does not include a JIT compiler and although it *does* have a bytecode interpreter, bytecode should be compiled to native code using gcj's ahead-of-time compilation.  Things will usually run, but they're being interpreted 100%; performance will be poor.

Distributions like Fedora ship the gcj bits (the shared libraries corresponding to the Eclipse jars) as well as the bytecode jars, enabling users to choose between the default of gcj or another JVM if they get it themselves.  With OpenJDK and IcedTea (IcedTea is basically OpenJDK built with and including entirely free components), distributions are now shipping a JVM-like environment with a JIT.  Fedora 8 (due out on 8 November) includes IcedTea and on x86 and x86_64 it is the default /usr/bin/java, etc.

If fixes are required in gcj, they can of course be made, but that's not going to say that these users' distributions -- or more importantly:  their machines -- will be updated to include these fixes.

> I'm pretty confident that most distros don't enforce that Eclipse can only be
> used with the supported vms.

We (Fedora) don't.

Comment 60 Remy Suen 2008-04-08 12:08:34 EDT 
Kim asked me to try to write something on GCJ and Eclipse as "documentation" about this problem so here it is. I'm attaching it here because everybody is here and nobody (except yours truly) is CC'd on bug 225736.

------------

GCJ is an effort by the GCC team to provide an open source Java compiler and runtime environment to interpret Java bytecode. Unfortunately, the GCJ runtime is not a runtime environment that is often tested on by Eclipse developers.

The most common problems surrounding GCJ are:
-Eclipse does not start at all
-Eclipse throws a java.lang.ClassNotFoundException: org.eclipse.core.runtime.Plugin in the logs (located in workspace/.metadata/.log)

The workspace's log file is a good place to identify whether you are using GCJ or not. Every Eclipse log session should be prepended with information about the runtime environment. The log may include something like the following:

java.fullversion=GNU libgcj 4.2.1 (Debian 4.2.1-5)

If Eclipse does start, one can check which runtime environment is being used to run Eclipse by going to 'Help > About Eclipse SDK > Configuration Details'. The About Dialog itself can also provide information, the build identifier can be of particular interest as it is tagged by some distros.

Build id: M20070212-1330 (Ubuntu version: 3.2.2-0ubuntu3)

The two most common workarounds would be to:
-download the Eclipse binary from eclipse.org directly
-run Eclipse using an alternate Java runtime environment

To download Eclipse, try one of the links below:
http://www.eclipse.org/downloads/
http://download.eclipse.org/eclipse/downloads/

It is imperative that 64-bit builds are downloaded and used if a 64-bit Java runtime environment has been installed.

To run Eclipse with an alternate Java runtime environment, the path to the Java virtual machine's binary must be identified. With an Eclipse installation from the distribution, altering the $PATH variable to include the path to the alternate Java runtime environment is often not enough as the Eclipse that Linux distributions package often performs a scan internally to pick up GCJ by itself whilst ignoring the other options.

Once the path to the virtual machine's binary has been identified, try running Eclipse with the following command:

./eclipse -vm /path/to/jre/bin/java

For an actual example, it might look something like the following:

./eclipse -vm /usr/lib/jvm/sun-java-6/bin/java
./eclipse -vm /opt/sun-jdk-1.6.0.02/bin/java

If problems persists after downloading an installation of Eclipse from eclipse.org and using a supported Java runtime environment [1], one should seek assistance from the Eclipse team via its plethora of communication channels like the newsgroups [2], its IRC channels [3], and/or bugzilla.

[1] - http://www.eclipse.org/eclipse/development/readme_eclipse_3.3.2.html
[2] - http://www.eclipse.org/newsgroups/
[3] - irc://irc.freenode.net/#eclipse
[4] - https://bugs.eclipse.org/bugs/
