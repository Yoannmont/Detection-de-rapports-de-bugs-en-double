Found the problem!

The culprit is in the method run() of the class org.eclipse.debug.internal.ui.actions.context.AbstractDebugContextAction.java.  The comment in the run() method says to "disable the action so that it cannot be run again until an event or selection change updates the enablement."  Eclipse Debug framework really should enable the action again, as long as my IThread model is suspended and there is stack frame(s)!  I know that clicking on the stack frame will cause the StepOver action to be enabled again, but the comment also says about event.  What DebugEvent code would I have to fire to get it enabled again?  Remember that I already fire DebugEvent.RESUME code of detail code STEP_OVER every time I do a StepOver.  I think the evidence is very clear here.  Please advise.

public void run() {
        IStructuredSelection selection = getContext();
        if (selection != null && isEnabled()) {
            // disable the action so it cannot be run again until an event or
            // selection change updates the enablement
            setEnabled(false);
            for (Iterator iter = selection.iterator(); iter.hasNext();) {
                Object element = iter.next();
                doAction(element);
            }
        }
    }

Comment 8 Darin Wright 2006-04-10 14:42:29 EDT 
It fires SUSPEND with detail STEP_END when complete. The detail code is the same for all steps (i.e. STEP_END is used for into/over/return):

	/**
	 * Step end detail. Indicates a thread was suspended due
	 * to the completion of a step action.
	 */
	public static final int STEP_END= 0x0008;

javadoc from DebugEvent:

 *	<li><code>SUSPEND</code> - a thread has suspended execution. Event detail provides
 *		the reason for the suspension:<ul>
 *		<li><code>STEP_END</code> - a request to step has completed</li>

public void canStepOver(final Object element, final IBooleanRequestMonitor
requestMonitor) {

Comment 18 Kevin Barnes 2006-04-11 16:14:20 EDT 
The step over action is not enabled by any of the listeners you've found. It's enablement is controlled by the context (selection) in the launch view.

The DebugTargetProxy should fire a delta which the viewer's update manager can use to update the view. This should trigger a selection change in the view (ie.. the top stack frame should be selected).
