Description Rupam Kuehner 2005-07-25 18:03:28 EDT 
When running the Web services bottom-up tutorial,
the browser launches with this URL:

http:/localhost:8080/ConverterProjectClient/sampleConverterProxy/TestClient.jsp?endpoint=http://localhost:9234/ConverterProject/services/Converter

it should be:

http://localhost:8080/ConverterProjectClient/sampleConverterProxy/TestClient.jsp?endpoint=http://localhost:9234/ConverterProject/services/Converter

If if fix the URL, it works.

The default browser on SLES 9 is "Konqueror". If I change the default browser to
Mozilla 1.7.2, the Sample JSP still gets launched with the wrong URL, but
Mozilla corrects it on the fly so the problem is not seen by the user.
Comment 1 Chris Brealey 2005-07-26 08:46:08 EDT 
Kathy, would you please take a peek at GSTCLaunchCommand? From a glance at the 
code, we are getting the leading portion of the TestClient.jsp URL (ie. the 
module context root) from ServerUtils.getWebComponentURL(), so we may be 
getting the hosed up path from Server Tools. If so, please assign to them. 
Thanks - CB.
Comment 2 Rupam Kuehner 2005-07-26 11:05:38 EDT 
I stepped through GSTCLaunchCommand on SLES 9.0 and the problem seems to be with
line 66:

IPath newPath = new
Path(ServerUtils.getWebComponentURL(sampleProject,testInfo.getGenerationModule(),testInfo.getClientServerTypeID(),testInfo.getClientExistingServer()));

ServerUtils.getWebComponentURL(..) is returning the correct URL, but when we
create an org.eclipse.core.runtime.IPath from the URL, it puts only one forward
slash after "http:" instead of two. So, it's not a server tools problem. We
could consider using a StringBuffer instead of an IPath to append things to the
URL we get from server tools.
