Description Sascha Radike 2007-03-17 13:29:00 EDT 
Build ID: 3.3M5eh

There a currently two issues regarding the registers/variables view. I tried to create a simple test case.

- The variables/registers view scrolls up unnecessarily when...
	* [1]: the thread gets suspended (after a few seconds of stepping/running)
	* [2]: the stack frames changes, but the vars/registers are the same - see bug 176627

- [3] The scroll bar of the variables/registers view does not always have the correct size - see bug 176465

I attached a small test java project which reproduces [1] and [3] - tested using eclipse 3.3M5eh / Windows XP.

JDT does not have registers, so the registers view bug [2] can only be tested with CDT or some kind of dummy debug model which does always provide the same registers for all stack frames.

Comment 2 Alan Boxall 2007-03-21 14:25:12 EDT 
Problem [1] in the original bug is also a problem in 3.2

It affects Java debugging as well as debuggers that are built on top of the eclipse debug core.

I have bumped the severity because this is a major usability issue when there are lots of local variables and the stepping response is slow.

A very simple testcase to show the problem:
1) debug any java application where there are 5 or more local variables at the point where the pgm is stopped
2) scroll the variables view such that the top variables aren't visible
3) "deselect" the current stackframe (Ctl + mouse click to deselect the stackframe)
4) the Variables view will go blank
5) select the same stackframe that was deselected in step 3)
6) the variables view will show the same list of variables AND will restore expansion (if there was something expanded) BUT it will be positioned to the top of the variable list.

We have customers with slow connections that can't use the Variables view because it constantly positions to the top.

This must be fixed in 3.3 and I would like a fix in 3.2 if possible.

In the meantime I'd like to pass by you a possible workaround.

Supply a DebugTargetProxy to override the dispatchSuspendTimeout() so that it doesn't collapse the thread.

Marking this as major as this affects our remote debugging scenario where stepping usually takes a long time.  And this makes it very difficult to debug with many variables.
Comment 4 Sascha Radike 2007-03-22 11:24:31 EDT 
I agree on this being major. The registers/variables view is quite unsuable when the view gets scrolled to top too often. I think it will drive customers mad pretty quickly.
Comment 5 Darin Wright 2007-03-22 11:42:36 EDT

Comment 6 Sascha Radike 2007-03-22 12:27:04 EDT 
(In reply to comment #5)
> This is a reasonable request - but I do think this is an enhancement rather
> than a bug, as the register view has always behaved like this.

Just because it has always been like that doesn't make it right ;). I really don't think this is an enhancement request, because the registers view seriously should not behave like that. I would not accept this behavior in a commercial product - and I have never seen this in a debugger. I see the registers view has less priority because it is not required for JDT. Also, it seems there are quite some vendors out there implementing their own registers view - the scroll bug might be just another reason for them doing this. But with new features like the details view pane the original views become more interesting.

Expecting a user to scroll back down to a register everytme he/she steps into a function/resumes the target for more than a few seconds is not acceptable.

Anyway, it's not only about registers. As reported by Alan and Samantha, the scroll bug does affect the variables view, too. See my small test case to reproduce the scrolling.

Inspecting variables and registers is one of the main tasks in debugging. Annoying behavior should be avoided as much as possible. And a scrolling var/reg view is annoying.

Comment 15 Darin Wright 2007-03-30 10:16:20 EDT

A debugger can provide content for any kind of element it wants to provide content for - we intentionally allow for this. It's part of the flexible hierarchy support.
Comment 16 Sascha Radike 2007-03-30 10:56:31 EDT 
(In reply to comment #15)

> A debugger can provide content for any kind of element it wants to provide
> content for - we intentionally allow for this. It's part of the flexible
> hierarchy support.

As far as I know (and I'm no debug expert), the content provider of the registers view does collect the information from IStackFrame.getRegisterGroups().

So when the context (--> the input of the reg/var view) is set to IThread or IDebugTarget the view won't be able to display any data and will forget about the the contents of the last IStackFrame context. That's okay if I manually select a thread or target in the debug view.

But the situation I was talking about was:
- The context is set to stackframe A. Registers are shown in the reg view.
- I resume... and suspend.
- When the target suspends, the context (input) of the reg view is not instantly set back to stackframe A, but it will first be set to a IDebugTarget, then to IThread...and finally to the stackFrame A.

Set a breakpoint on VariablesView.contextActivated(ISelection) and watch the kind of context which does get passed to this method when the target suspends.

How should the tree viewer be able to restore the same state when the input has been set to a IThread or IDebugTarget between those two IStackframes.
Comment 17 Darin Wright 2007-04-04 12:52:38 EDT 
(In reply to comment #16)
> But the situation I was talking about was:
> - The context is set to stackframe A. Registers are shown in the reg view.
> - I resume... and suspend.
> - When the target suspends, the context (input) of the reg view is not
> instantly set back to stackframe A, but it will first be set to a IDebugTarget,
> then to IThread...and finally to the stackFrame A.

If you resume, the debugger may select the thread that was resumed (the Java debugger does, not sure what the C debugger does). When you resume, we do attempt to maintain selection expansion if you suspend at the same location in the future (via an LRU cache of viewer states for stack frames). However, this will not maintain scroll position. 

When stepping, we'd like to maintain more stability. We don't intend to address the scroll/refresh issue when explicity resuming - we want to address the stability issue when stepping for 3.3.
