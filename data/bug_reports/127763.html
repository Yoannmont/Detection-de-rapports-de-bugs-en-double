Description Pierre Tessier 2006-02-14 11:29:21 EST 
For a chart, or an aggregate control, it makes sense to pass the entire dataset for generation and presentation.  However for a simple extended item which processes a single row at a time, this doesn't work.  To make matters worse, there is no way to determine which row is currently being processed by the container.

If my control is in a List's detail band, I would expect to get my onRowSets function called 1 time for each row, and only passed the row in question to be processed.  Same is true if the control is in the header band.  I would expect to receive only 1 row, which is the row for the header.

In the very least, a hook or api call to determine the row number currently being processed, so I could iterate through the rowset to that given number and retreive the data, on the presentation and generation extension points.  

This is particularly important on the generation side for performance reasons.  When my control is in the detail band, onRowSets gets called for every record.  Each time, the entire data set is passed.  Currently I need to step through each row, and each expression to evaluate them, to ensure they are loaded into Context.  This is quite inefficient.  It would make more sense to only evaluate the row pertaining to this instance of the control.  I could easily do this, if I know what row in the set to actualy use.

Comment 4 Juergen Mising name 2006-05-09 09:55:40 EDT 
Actually I was also playing around with the RotatedText code from Pierre and discovered that the mentionend behaviour (complete rows to be passed to onRowsSets()) is only true when binding the element itself to a dataset.
When using the element in the details section of a table, there should be no binding of the element to the dataset, instead the element should use the binding of the table. But this does not work for the RotatedText element where the delivered RowSet is null.

I review the RotatedTextQueryImpl class in the atteached plugin. And I find the reason is that the RotatedText element in the detail may has no data set in itsself, but in RotatedTextQueryImpl class there will return a new IBaseQueryDefinition[] { null }. In engine will regard it has its own query, but the content is null, this will cause the RowSet is null. 

For fix this, RotatedTextQueryImpl should do like this:
   if ( has data set )
   {
      create a query
      return the query
   }
   else
   {
      create a subQuery
      return the subQuery
   }
   return null

Comment 8 Pierre Tessier 2007-03-07 11:14:29 EST 
The issue was not resolved whatsoever.  

The problem is that the PresentationImpl does not have context of the current row number.  It is just sent an array of all rows, however it would only need to process one row at a time.  The current row number is a field to the rowset, however it is always set to -1 while debugging.  Until this issue is resolved, Extended Report items are only useful for items that require the entire data set to do their work, like a Chart.  Simple items like rotated text controls, or perhaps a native SVG control, require the row context to be specified.

Since much of the API to display properties has changed, I have reflected the changes in the plugin and re-uploaded it here.  

Remember the problem is not setting up the queries, the problem is there is no way to determine the current row in context at Presentation time.
Comment 9 Wei Yan 2007-03-30 04:51:43 EDT 
we have defined a new property "getAllRows" in the query extension point to define how many rows needs by the report item, the default value is "false".

In this case, if we left the reportItemQuery extension empty, the extension will receive a row set with only one row.
