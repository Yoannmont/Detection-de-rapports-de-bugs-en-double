Comment 2 Martin Oberhuber 2007-08-06 10:32:19 EDT 
I found what the problem was for my javax.crypto issue:

The project had been imported from an ANT buildfile (File > New > Project > Java >  Java Project from existing ANT build file). By doing so, in its 
Project properties > Java Build Path > Libraries, it had specified "JRE_LIB/rt.jar" as the class path.

I found that JRE_LIB is deprecated, and the "JRE System Library" should be used instead. By doing that, it picked up the jce.jar in addition to the rt.jar, and voila javax.crypto became visible.

So my problem might have been a different one than the originally reported one from the mailing list, which named javax.swing.event (which is defined in rt.jar) -- As Tom said, you should be checking whether you have a Manifest.mf specifying a Bundle-RequiredExecutionEnvironment.

I'm wondering whether I should file a separate bug against JDT, saying that Import from Ant should not use JRE_LIB but rather the JRE System Library.

Comment 19 Thomas Watson 2007-11-30 15:26:27 EST 
(In reply to comment #18)
> Tom, if a user tries to import sun.misc (or sun.reflect) from the BREE for J2SE
> 1.5, should they have access to that package.  

No, these packages are not part of the standard for J2SE 1.5.  They are sun specific packages.

> I went and looked up the
> J2SE-1.5.profile shipped in org.eclipse.osgi and saw that the
> org.osgi.framework.system.packages property did not contain either one of these
> packages.  I figured if the package is not specified in
> org.osgi.framework.system.packages then the resolver does not know the package
> is coming from the VM, right?  

Correct.

> 
> If this is the case, this is the expected behavior.  Then the next question is,
> do we add it to the .profile file?  Should PDE be doing something differently?
> 

We should not add the VM specific packages to the standard profiles in org.eclipse.osgi.  These are meant to represent standard Execution Environment profiles, not vendor specific VMs.  PDE should not promote using packages which are vm specific.

One option could be to allow developers to specify their own profiles for the VMs they have installed and use that for both launching and for populating the platform properties in the PDE State.  I consider that a corner case though and not something that many developers should do.
