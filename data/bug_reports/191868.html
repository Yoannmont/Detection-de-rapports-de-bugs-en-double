Comment 10 Andrew Niefer 2007-06-12 10:20:30 EDT 
David, the existing JarProcessor works for me on a zip file when the pack.excludes property has the correct values.

The problem is the "name" listed in the pack.excludes property is not matched against the the name of the jar, it is matched against ZipEntry.getName().  This name includes the path inside the zip.

This means, in your example, the pack.excludes property you should use is:
pack.excludes=plugins/com.jcraft.jsch_0.1.31.jar,plugins/com.ibm.icu_3.4.5.jar,plugins/org.eclipse.wst.command.env_1.0.203.200706111934.jar,plugins/org.eclipse.wst.common.project.facet.core_1.2.0.200706111934.jar

(path separators must be /).  There is a problem when the input is a directory.

Comment 12 Andrew Niefer 2007-06-12 13:22:58 EDT 
Because the zip file does work with correctly formatted properties, I'm lowering the the severity from blocker. Any changes can go in 3.3.1.

1) We can't change any files (or add or remove files) after the jar is signed.  JarProcessor does not currently know how to detect if a jar is signed.  Though this can likely be done by looking for *.sf and *.rsa files in META-INF.  Running pack on a signed jar that was not conditioned with the JarProcessor will break the signing.

2)  In the current implementation, the PackStep.adjustInf will never even be called on a jar that was excluded in pack.properties.  A jar that is excluded may not even be extracted from the zip at all.

3) I don't believe that in general we want to mark jarprocessor.exclude.pack = true just because the jar was excluded.  There may be many reasons to exclude a jar from a particular run of the jarprocessor, and one exclusion should not preclude the jar from ever being conditioned or packed in the future.  

If you have orbit bundles that you want to always be excluded, it is better to explicitly add a eclipse.inf with the jarprocessor.exclude.pack property to the bundle in CVS.

Marking conditioned=false is entirely equivalent to not being marked at all, the only difference would be to a human looking at the file.  It is likely that marking false would create a file where there was none before.

4) The one case I can see to automatically mark jarprocessor.exclude.pack=true is if the jar is signed without first being conditioned.
