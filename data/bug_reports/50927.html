Description Andre Weinand 2004-01-29 18:20:41 EST 
I have difficulties understanding the policies for hiding files and directories on different platforms.

On Unix and MacOS X the .metadata (and now the .config folders) are hidden.
I have no problem with that.
But I don't understand why most (but not all) files within these hidden directories are itself hidden 
too.
On Unix and Mac OS X it is really a pain for ordinary users to find these hidden files (especially the 
.log file and now the .keyring files) within their hidden folders.
And if a user has decided to look into a hidden folder .metadata, why can't she see the log file 
without further hassles?

The additional odd thing here is that on Windows all this '.' business seems to have no effect at all, 
so ordinary users have access to everything in .metadata and .config without any hurdles.

What am I missing here?

Comment 16 Alex Blewitt 2005-07-22 07:02:16 EDT 
You need to find out what the network drive is doing, and how you're mounting it
or making it available on the mac. I've been able to do this with SMB
(samba/windows file sharing), NFS and also AFP (the Apple Filing Protocol that's
used for sharing Mac-Mac shares) and all copy across the .project file fine.

If your network drive is a standalone product (like a Lacie drive) then a
make/model might be useful to know what server OS it's running to make them
available. Chances are, it could be that the network drive is using an old
version of Linux with an appletalk protocol that is automatically hiding the .
files when the apple server asks for it ...

Can you try the following mounts -- from Finder, Command+K to bring up a
connection window, and then try the following URLs:

afp://mynetworkdrive/
smb://mynetworkdrive/
nfs://mynetworkdrive/

and do the copies from each? You might find that the choice of network protocol
makes a difference; Mac can speak them all, but the network drive might have a
broken AFP driver (the Linux 2.2 kernels were notoriously bad for such things).
Windows should share it over SMB, so I'd expect to see that working exactly the
same way as the Windows box sees it.

Comment 33 John Arthorne 2011-05-13 17:11:31 EDT 
I believe we did actually agree on a resolution to this:

http://wiki.eclipse.org/Architecture_Council/Meetings/August_13_2009#bug_50927_-_JohnA_-_Convention_for_projects_to_use.2Fnot_use_hidden_folders

In short the recommendation is:

- Put metadata under ".settings" when stored under a project in user's workspace
- Put non-shared metadata under workspace ".metadata" folder
- Any metadata files and folders below those two locations should *not* start with a dot.
