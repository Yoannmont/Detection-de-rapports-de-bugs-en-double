Description John Bossom 2006-02-21 12:07:19 EST 
The IDE permits you to add features and plugins from any location on disk.
This allows you to organize your source into logical groups in source
control and on disk.

However, PDE Build forces to you flatten out this directory structure
in order to perform an automated build, either through adopting
a flat directory structure, or through copying all features/plugins 
into a flat directory structure. 

eclipse.buildScript does support an override for this via the pluginPath
parameter, however, it is not available through PDE Build, plus maintaining
such a list is a maintenance headache...

i.e. eclipse.buildScript supports either:
a) features and plugins reside within a flattened directory structure of
   the form:
        <buildDirectory>/
            features/
            plugins/

OR

b) you explicitly specify the full path to each and every plugin directory and
   feature.xml via pluginPath, as in:
        pluginPath="<buildDirectory>/features/f1/feature.xml,\
                    <buildDirectory>/plugins/p1,
                    <buildDirectory>/plugins/p2,
                    <alternateDirectory>/plugins/p3,
                    ..."

--------------
Recommendation:
--------------
    Allow the flexibility provided to the IDE by permitting the specification
    of a list of additional directories to be used to dynamically resolve
    the pluginPath.

    Add a new, optional parameter, searchPath to eclipse.buildScript, which is
    a comma-separated, ordered, list of directories in which to search for
    features and plugins if they are not found under <buildDirectory>.

    Precedence:
       1) if pluginPath specified, use it exclusively as before.

       2) search <buildDirectory>/
                          features/
                          plugins/

       3) foreach dir in searchPath
             search <dir>/
                          /features
                          /plugins


For example:
        <property name="searchPath" 
                  value="c:/src/myOtherComponents,d:/evenMoreComponents" />

		<eclipse.buildScript 
			elements="feature@com.mycompany.myfeature" 
			buildDirectory="c:/src/myComponents"
			configInfo="*,*,*"
			baseLocation="c:/eclipse_3.2M4"
			searchPath="${searchPath}"
			buildingOSGi="true"
			outputUpdateJars="false"
			archivesFormat=""
			forceContextQualifier="false"
			generateJnlp="false"
			signJars="false"
			generateFeatureVersionSuffix="false" />

PDE Build could easily accomodate this by always specifying searchPath as
shown above, and if it is empty, you have the same semantics as you do now.


Comments?
Comment 1 Andrew Niefer 2006-02-21 14:04:20 EST 
Starting in M5, the pluginPaths and workingDirectory are combined.  PDE Build collects plugins from the following locations:
(1) baseLocation
(2) pluginPaths;workingDirectory
pluginPaths is a File.pathSeparator separated list of directories.
From (2) extra Features and plugins are searched for in 
path
path\features\*,
path\plugins\*

The order of the paths does not matter as we collect all plugins/features and resolve the set.

I believe this has the effect you are looking for.
The only thing we should do is add the pluginPaths=${pluginPaths} to the genericTargets.xml script.
Comment 2 John Bossom 2006-02-21 14:22:29 EST 
Thank you for such a speedy response!

Some comments.
1) You state that pluginPaths is a list of directories. in M4 I was
   struggling with not being able to find my feature until I stumbled
   upon a bugzilla report that demonstrated that you had to explicitly
   list the full path down to the feature.xml for features, as in:

       pluginPath="<blah>/features/myFeature.xml,..."

   May I recommend updating the "Generating Ant scripts from the command line"
   document to precisely describe how to use this pluginPath parameter,
   with an example?


2) You indicate that you pull up all features and plugins and then resolve
   the feature you want to build.
   One problem with this is you will not discover feature/plugin dependency
   issues until deployment time if you resolve the list to be everything
   in these directories.
   i.e. If I am missing a plugin dependency on a feature I will still be able
        to build/export the feature. I will only discover I am missing the
        plugin when I am unable to launch the feature.
        However, if you generate the pluginPath based upon the feature/
        plugin dependencies you would catch the missing dependency at
        build time.

3) Yes, please fix PDE Build's genericTargets.xml to allow setting pluginPath!

I'll move up to M5 and try out this new stuff!

Cheers,

John.
Comment 3 Andrew Niefer 2006-02-21 15:04:11 EST 
1) The full path to the feature.xml will work as well.  We will be updating docs and writing an article or two near the end of the release.

2) There is a use case where you are building a feature that is expecting to be deployed into an existing eclipse install somewhere.  In this case, not all your dependencies will be included in the build result.  We can't know if a dependency  not included in your feature is a mistake or on purpose because it is expected to be in the eclipse target you will be deploying to.
