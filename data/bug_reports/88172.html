Comment 5 Susan McCourt 2005-03-17 18:05:32 EST 
Two distinct problems in the OperationApprover inside DefaultUndoManager

1 - if undo is invoked while folding into a compound change, the current 
compound change (fCurrent) will always be invalid.  Once it is determined that 
fCurrent should be committed, the folding parameter must be set to false or it 
won't commit.

2 - if commit fails (for the above reason or any unforeseen reason), then 
invoking another undo will cause operation approval on the same operation, 
causing the infinite loop.  There should be a guard in the method to ensure 
that undo is never invoked a second time if fCurrent didn't actually get 
committed.

Attaching patch.

Comment 9 Dani Megert 2005-03-21 08:39:07 EST 
Applied the patches.
Available in builds > 20050321

Re #1: the open compound change is a bug/open issue on our side: we have
commands (e.g. move/copy lines) which we like to treat as one change and hence
we begin a compound change. Since we do not how many times the user will invoke
the command we have several exit strategies that end the compound change. Until
recently there was no chance to know about other commands being executed and
hence the change is still open when undo is invoked. New experimental API has
been added that allows to listen for command execution. I filed bug 88632 to use
this on our side.

However, the undo manager must gracefully handle such a situation as done with
the attached patch.
