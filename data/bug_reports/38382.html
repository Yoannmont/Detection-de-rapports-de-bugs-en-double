Description Randy Hudson 2003-06-03 15:19:24 EDT 
Using 2.1.1, fresh workspace with 2 java projects: "Proj A" and "Proj B".
Each java project has multiple bogus text files and multiple empty folders.

Using the Packages View, expand all TreeItems. Resize the view such that 
vertical scrolling is necessary.

Drag a text file from one project to the other, such that SWT performs auto-
scroll on the tree.  In other words, try picking the very bottom TXT file, and 
drag it to the very first project, "Proj A".

Now release the mouse.  When you do this, the tree jumps from its current 
vertical location back to some position that allows the original TreeItem to be 
viewed. In doing this, the dragged file gets dropped into whatever folder 
happens to get scrolled into view.

The result is that the user has a moved a file or folder to some unknown 
location.  The user may think that the file was deleted and try restoring from 
local history.

The same is true with classes.  If you have multiple CUs under a TreeItem, and 
you are copying a method from one class to another, and autoscroll occurs, the 
method will end up in some completely unknown target class instead of the 
intended class.  Or, what is more likely, the copy won't happen at all because 
the tree scrolled to some location that is not a valid target of method copy, 
such as the CU or a package.

These problems are severe since there is no undo and no confirmation of what 
happened.

Comment 2 Veronika Irvine 2003-06-03 18:16:00 EDT 
Can show problem with simple example.  Fully expand the tree and select the 
last item.  Drag item up so that tree scrolls upward an release.  Notice the 
item you expect to drop on and compare with the item printed in the cosole.  
Item in console is item under cursor when tree scrolls back to show original 
selected item.

public static void main(String[] args) {
	Display display = new Display();
	Shell shell = new Shell(display);
	shell.setLayout(new FillLayout());
	Tree tree = new Tree(shell, SWT.BORDER);
	for (int i = 0; i < 2; i++) {
		TreeItem item = new TreeItem(tree, SWT.NONE);
		item.setText("item " + i);
		for (int j = 0; j < 20; j++) {
			TreeItem subItem = new TreeItem(item, SWT.NONE);
			subItem.setText("item "+i+" "+j);
		}
	}
	final TextTransfer tt = TextTransfer.getInstance();
	DragSource ds = new DragSource(tree, DND.DROP_COPY | DND.DROP_MOVE);
	ds.setTransfer(new Transfer[] {tt});
	ds.addDragListener(new DragSourceAdapter() {
	});
	DropTarget dt = new DropTarget(tree, DND.DROP_COPY | DND.DROP_MOVE);
	dt.setTransfer(new Transfer[] {tt});
	dt.addDropListener(new DropTargetAdapter() {
		public void dragOver(DropTargetEvent event) {
			event.feedback |= DND.FEEDBACK_SCROLL;
		}
		public void dropAccept(DropTargetEvent event) {
			System.out.println("item "+event.item);
		}
	});
	shell.open();
	while (!shell.isDisposed()) {
		if (!display.readAndDispatch())
			display.sleep();
	}
	display.dispose();
}

Comment 5 Veronika Irvine 2003-06-04 17:20:17 EDT 
This bug has existed since early in 2.0.

Problem:
TreeDragUnderEffect and TableDragUnderEffect are saving the selected item in a 
DragEnter, deselecting all items.  Then in a DragLeave, the original selection 
is being restored.  The act of restoring the selection causes the selected 
item to be scrolled into view.  This changes the item under the cursor and 
consequently the event.item field is incorrect in the DropAccept and Drop 
events.

For 2.1.1, I can do one of the following:
1) Nothing in 2.1.1 since the problem has been there for a while - fix for 3.0.

2) Fix on Windows only. Simple fix.  On Windows, we use the OS defined 
DragSelection.  It is therefore not neccessary to remove and restore the 
selection in the drag under effect.  The only side effect of fixing this on 
Windows is that the table or tree will no longer scroll back to show the 
originally selected item when the drop completes or when you drag outside of 
the drop target.

3) Fix on all platforms.  On non-Windows platforms, we do not use an OS 
defined drag selection behaviour. Also, there is no way to restore the 
selection without scrolling the selection into view.  On these platforms, I 
would propose that we deselect in DragEnter and not restore the selection in 
DragLeave.  The effect will be that if you drag over a table and then leave 
without dropping, the selection will disappear from the table or tree.  In the 
case of a drop, it is acceptable that the selection is removed since the 
application would probably select the new item which resulted from the drop.  
For an unnsuccessful drop, it is not as desirable.  Unfortunately, it is not 
possible in the DragLeave to know whether a drop was successful or not.

Comment 10 Veronika Irvine 2003-06-11 14:29:48 EDT 
2.1.1:

Fixes submitted for Tree and Table as follows:

Windows - on Table use select instead of setSelection (will not scroll).  On 
Tree restore selection using OS calls which will not result in scrolling
GTK - use the native drag selection and remove selection code
Motif - remove selection code and paint a rectangle around the drag selection
Mac OS X - disable all drag under feedback

Leaving bug report open because the 3.0 solution still needs to be defined.
