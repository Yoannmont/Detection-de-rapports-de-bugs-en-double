Comment 8 Vlad Klicnik 2002-05-26 21:10:00 EDT 
There appear to be 2 conditions causing this set of defects.

(1) some of the workspaces were created with a build that had org.eclipse.sdk as 
its primary feature. The configuration code currently does not reset the primary 
feature id in the platform.cfg once it is set. Since then the builds were 
changed and are using org.eclipse.platform as the primary feature. Since 
platform.cfg is not picking up this change, the splash-handling code is looking 
for a splash that does not exist and is defaulting to the one supplied with 
core.boot ("completing install ...")

(2) several of the reported defects are caused by launching eclipse with -dev 
flag. When running in -dev mode, the update startup reconciliation is 
suppressed. The rationale was that when in dev mode you are controlling your 
environment and update should not be getting in the way. So the options here are

(a) start using PDE for self-hosting as you should be by now
(b) startup and shutdown eclipse once before you use it with -dev (this 
will do the 2-splash dance correctly)
(c) run with -dev <bin> -update ... this re-enables startup reconciliation 
processing even in -dev mode
(d) we make a code change not to suppress startup reconciliation in -dev mode


Will fix (1) to always get the default setting from install.ini (and not just on 
first startup).

For (2) will consider (d) but want to discuss this with Jim first. For now use 
any of the other 3 options. I would strongly recommend (2a) for all ... this is 
the only "externally supported" form of self-hosting.

Comment 19 Vlad Klicnik 2002-06-26 09:53:45 EDT 
Gunnar, looking at the configuration history you sent we think the problem was 
caused by the fact the install was updated using the update manager, and then 
freshly re-installed. The packaged builds (in the update format) used 4-part 
version stamps, and the base builds used 3-part (was done that way to enable 
testing of the updates). However, in your case this created a condition where 
the GM1 build looks "older" than the F3 build, so we are not correctly picking 
up the fresh re-install.

This is somewhat of an artificial situation caused by the test update builds. 
The update server on eclipse has been reset, and from now on will only contain 
legitimate updates (with properly sequenced plugin/feature versions). I agree 
that our code should guard itself againts this situation, so am reopenning the 
defect.

As a workaround, delete the .metadata/.config/ directory in your workspace and 
start eclipse again. This should correct your condition and preserve the rest 
of your workspace.
