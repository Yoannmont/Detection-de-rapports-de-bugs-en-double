Description Darin Swanson 2003-04-23 12:26:28 EDT 
For each launch the runtime classpath is resolved twice: once determining the 
classpath and once determining the bootpath.

Comment 5 Darin Swanson 2004-03-31 19:47:26 EST 
This change has completely broken launching Ant in a separate VM.
Comment 6 Darin Swanson 2004-03-31 19:59:54 EST 
Kevin, it appears that your optimization will work for all delegates that are 
org.eclipse.debug.core.model.ILaunchConfigurationDelegate2. Don't we also have 
still provide the old support for delegates that do not implement this 
interface and therefore preLaunchCheck(*) is not called?
Comment 7 Kevin Barnes 2004-04-01 14:46:58 EST 
The changes that I made were in AbstractJavaLaunchConfigurationDelegate which 
IS an ILaunchConfigurationDelegate2. It expects that it's preLaunchCheck(*) 
method will be called.
The problem now is that AntLaunchDelegate create a new 
JavaLocalApplicationLaunchConfigurationDelegate (subclass of 
AbstractJavaLaunchConfigurationDelegate) and then directly calls launch() on 
it w/o calling prelaunchCheck().
Comment 8 Kevin Barnes 2004-04-01 14:56:09 EST 
I'm not sure what the right answer is, switching launch delegates mid-launch 
feels wrong to me. As a quick fix adding the line
delegate.preLaunchCheck(copy, ILaunchManager.RUN_MODE, subMonitor);
before calling launch() will at least keep ant running.
Comment 9 Darin Swanson 2004-04-01 15:00:12 EST 
Thanks Kevin..I am breaking the rules on the calling the lifecycle methods.
Darin and I talked about extending the framework to ask for the correct 
delegate based on a launch configuration. 
Not sure if he was going to file the bug or if I was supposed to...talk amongst 
yourself.
Comment 10 Darin Swanson 2004-04-01 15:01:06 EST 
I will fix up the AntLaunchDelegate to call the prelaunch method.
Then we can see if we will add the new framework support that Darin and I 
talked about.
