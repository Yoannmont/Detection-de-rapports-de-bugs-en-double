Description Wassim Melhem 2006-09-04 19:22:14 EDT 
The org.eclipse.debug.core plug-in exposes API from the core.resources plug-in.

For example,  the return type for ILaunchConfiguration#getMappedResources return an array of IResource

public IResource[] getMappedResources() throws CoreException;

I was just writing a plug-in that implements this interface and got compiler anomalies since core.resources was not in my list of plug-in's direct list of dependencies.

debug.core should re-export core.resources since it exposes API from that plug-in, so that clients are not immediately broken/puzzled when they start using debug API.  

This is one of the rules of thumb of the conditions under which a dependency should be re-exported.

Incidentally, debug.core also exposes IPath from core.runtime:
public IPath getLocation();

Comment 8 Dani Megert 2013-02-04 09:21:41 EST 
The manifest declares the API that a bundle itself provides, not the one it also uses. Debug is not responsible for the Resources API hence it must not re-export it. A client decides what APIs it uses from a bundle. If it needs APIs from Debug that also use Resources API, then it has to add core.resources as required bundle.

Comment 17 Dani Megert 2013-02-07 09:56:34 EST 
The value, is that the plug-in is clean and new clients/users don't copy a wrong pattern. Unfortunately, one cannot add comments into manifest files, otherwise we could just explain the wrong re-export.

Not all clients need to add the bundle, just those who call API on Debug which reference core.resources in their class file and in fact, they should not have to do this, but currently have to due to bug 73957.


But I'm not strongly pushing this. Let's discuss this in our next PMC call and revert the changes again if we decide so.
Comment 18 Martin Oberhuber 2013-02-07 10:33:36 EST 
(In reply to comment #17)
Thanks for referencing bug 73957 .

I understand that re-exporting is more or less a workaround for PDE not being able to auto-compute transitive classpaths.

Let's think about this a bit more. I've been bitten a couple times by the "...indirectly referenced from required .class files" compile error, and scratched my head a couple times how to resolve them. I think nowadays the Plug-in Manifest Editor's button to auto-compute dependencies should work, so that would be a "fix" for the problem, but I'm not sure if it's a good fix since it explicitly adds a run-time dependency for what should only be a build-time dependency when the respective functionality is not needed.

Right now I feel like I'd prefer keeping the reexport in there, since that helps clients avoid accidentally adding an incorrect runtime dependency that they don't really need.

I see why you dislike the reexport, but since it's been in there all the time the damage has happened and I don't quite see the value of removing it now.
