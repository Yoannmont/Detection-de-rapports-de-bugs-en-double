Comment 7 Dirk Baeumer 2005-06-07 13:14:32 EDT 
The problem isn't caused by the Worker-56 thread. The thread is basically
waiting on the same lock as the main thread.

Here is what I think happened:

- refactoring locks the workspace when it gets executed
- the actual workspace modifications are executed in a modal context thread.
  To allow the modal context thread to access the workspace refactoring 
  transfers the workspace rule aquired in the UI thread to the modal context
  thread.
- during the execution of the refactoring a delta occurs which the UI thread
  has to react on. The processing is done in the UI thread using async exec.
  However the refresh in the package explorer triggers some class path
  resolving which sets the class path. This requires a workspace lock which
  the UI thread is trying to get.
- now the modal context threads ends and transfers the rule back to the UI
  thread.

John, what I think is happening is the following: when the modal context thread
transfers the lock back to the UI thread it seems that the job framework doesn't
wake up the UI thread when it is waiting for the lock. Could this be the case. I
assume that this is happening due to the fact that the Modal context thread
doesn't show up in the dump anymore so it must have already finished. I checked
the refactoring correctly transfers the rule between the main and the modal
context thread.

the target project (this is the same location as reported by Bob). When the
break point is hit, the main threads hold the "UI lock" (e.g. no SWT events can
be processed).

The modal context thread is still active and tries to execute the done method of
the AccumulatingProgressMonitor. Since the lock in the modal context thread was
allocated outside of the Workspace.run method the modal context thread still
holds the lock. 

Now assume that the main thread continues and tries to get the workspace lock
(as in the stack trace provided by Bob). This is not possible since the modal
context thread is still hold hit waiting for the lock hold by the main thread. A
classical dead lock.

However, I was not able to reproduce this. All class path containers now get
initialized in the "initialize after load" job run by JDT/Core. Since the job
takes the workspace lock as well, no refactorings can be executed during this time. 

The interesting question is why the class path wasn't initialized in the stack
trace provided by Bob, since in the orignial summary of the PR he put a mark in
saying that it occurred on [RC1]. 

So the questions are: 

- Bob, can you confirm that this happened on RC1

- Wassim, Jerome, can you comment on whether it is still possible that class
  pathes aren't initialized although initialize after load has finished.

- what can we do to avoid such a situation since IMO it can easily occur in
  other scenarios as well. This is not specific to refactoring. I see the 
  following problems:

  o AccumulatingProgressMonitor#done(): a sync exec with an active workspace
    lock

  o Workspace#run finally block: the done method should only be called when
    the thread doesn't hold any other lock

  o the class path initialization shouldn't happen in the UI thread. However
    for the package explorer findPackageFragmentRoots is a simple read method

    UI (but UI could plug-in some synchronizer which core could use and would

This "simple" JUnit test reproduces the problem. The sequence of events is:

1) Lock is acquired in UI thread
2) Modal context thread is started using IThreadListener
3) Lock is transferred to modal context thread
4) Modal context thread performs an asyncExec
5) The asyncExec tries to acquire the same lock held by the modal context
6) The modal context thread exits, transferring the rule back to the UI thread

Now the rule has been transferred back to the UI thread, but the UI thread
is in a wait loop waiting to obtain the rule. The UI thread should realize that

it now owns the lock it is waiting for, and continue with its execution.
