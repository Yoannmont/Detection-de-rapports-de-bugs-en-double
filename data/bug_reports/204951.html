Description Shawn Minto 2007-09-28 16:18:43 EDT 
If you have the "Auto focus navigator views on task activation" preference disabled, there is a way that the action can become disabled with the focus on, resulting in the view having no context.  Whenever the focus action is disabled, the focus filter should be removed from the view as well.

Steps to reproduce:
1. activate a task with or without context and focus the view
2. deactivate the task
3. note that the focus action is now disabled and there is no content in the view

The only workaround is to ensure that the filter is not on when the task is deactivated.

Comment 16 Steffen Pingel 2008-01-26 20:54:20 EST 
It's always reproducible. I have traced the problem down to PluginAction.isOkToCreateDelegate(). The method checks if the bundle that provides the action class has been started and only then creates the object. In my case the o.e.m.ide.ui plug-in is never started since none of it's classes are loaded so the FocusProjectExplorerAction is not created. 

This could be solved by making sure that all plug-ins that extends AbstractContextUiPlugin are activated when a context is activated. That might require an additional extension point that provides a start-up/context activation hook similar to the Eclipse workbench extension. In order to make it work for the project explorer it is also necessary that all context UI plug-ins are activated before ContextUIPlugin.lazyStart() runs which triggers the evaluation of the view action enablements.

Comment 20 Shawn Minto 2008-01-31 16:36:43 EST 
Created attachment 88467 [details]
patch

Some views update the action enablement on selection changed (i.e. Problems view).  This can result in the Focus action being enabled when a task is not active since the "enabler" for the PluginAction says that it should be enabled and the delegate doesn't update the action on selection changed.  The attached patch will update the enablement on selection changed.
