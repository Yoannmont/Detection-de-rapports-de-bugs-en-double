Comment 3 Eugene Kuleshov 2007-10-15 13:18:27 EDT 
Before blaming Mylyn it would be nice if you could open up jconsole and check what is the actual permgen size is used.

In fact, there is a regression bug in Eclipse 3.3.1, so it is not using parameter --launcher.XXMaxPermSize. See bug 203325
In a mean time you could use the following setting instead of that.

-vmargs
-XX:MaxPermSize=120m
Comment 4 Mik Kersten 2007-10-15 23:21:08 EDT 
As far as we know, Mylyn is only related to PermGen errors in that it requires the loading of additional classes.  In this scenario I'm guessing that you're either seeing the problem that Eugene described, or that the VM is not accepting a PermSize of 512M given how much physical memory you have.  I have updated the Mylyn FAQ entry on this with additional steps that you can take to resolve the problem.  If you see Mylyn implicated in any way please feel free to reopen.

http://wiki.eclipse.org/index.php/Mylyn_FAQ#I_am_seeing_java.lang.OutOfMemoryError:_PermGen_space_errors.2C_what.27s_wrong.3F

Eugene: fyi, I recently learned that the setting needs to be a power of two otherwise it will silently fall to the standard size (32M).  Added that note to the FAQ as well.

The current winning version of my eclipse.in is here for what it is worth.

-showsplash
org.eclipse.platform
-vmargs
-Xms40m
-Xmx1024m
-XX:MaxPermSize=128M

Although several people have mentioned that the -XX:MaxPermSize=128M must come AFTER the -vmargs tag (and yes I can confirm it must) this sample makes it easier to see this for those us us blessed with inductive (versus deductive) reasoning skills.

The above ini file is working with Suns JVM:

java version "1.6.0_03"
Java(TM) SE Runtime Environment (build 1.6.0_03-b05)
Java HotSpot(TM) Client VM (build 1.6.0_03-b05, mixed mode, sharing)

I have been crash free since I got the setting *correct*.
