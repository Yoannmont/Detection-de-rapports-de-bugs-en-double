Comment 2 Mikhail Sennikovsky 2005-06-28 05:04:11 EDT 
Doug,

MBS now provides the mechanism that allows tool-integrators to specify 
additional environment variables to be used for the builder process. The gnu 
tool-chain definition uses this mechanism to modify the PATH environment 
variable. The environment variable supplier provided by the gnu tool-chain 
definition obtains the cygwin /bin location from the registry and prepends it 
to the path. 
The initial intension was that if a user wants to override the default 
behavior, he/she has to specify the path of the tools to be used using 
the "User Variables" table of "Environment" tab in the build property page 
(this is what you actually described in the comment #1).
If you think this is not appropriate for the mbs gnu tool-chain, because the 
tool-chain is inteded to be used with mingw also, we might consider making some 
modification for the way PATH is modified for the gnu tool-chain. E.g. we might 
define some shared option (bug# 90481) for the gnu tool-chain which would allow 
user to specify whether cygwin or mingw is to be used. In the latter case the 
path variable will not be modified and also the builder command could be 
changed from "make" to "mingw32-make". We might also consider removing of the 
path modification for the gnu tool-chain, but I suppose that this is quite 
convenient and useful feature, since if cygwin location is automatically 
detected and added to the path, users do not have to add it manually any more 
to make the cygwin projects build, and the issue with mingw appears only if 
there are both cygwin and mingw installed on the system and user wants to use 
mingw.
 
Thank you,
Mikhail

Comment 15 NO ACCOUNT 2006-03-06 10:41:05 EST 
Mikhail,

No offense, but you are making a big assumption here about the demand for CYGWIN and I don't think it is correct. I believe there are market forces greater than CYGWIN driving demand for Eclipse on Windows. For instance, the GPL version of QT 4, which is gaining popularity, does not work with cygwin, but requires the MINGW version of make. QT 4 is the only truly portable way to build native GUI apps on Windows, Linux, and Unix derivatives. Without QT 4, I see no reason to use Eclipse. I would rather just use Visual C++ on Windows. With QT 4 there is a big advantage to using Eclipse because I can use a common IDE and programming environment on all platforms. Cygwin is a nice environment for GNU tools on Windows, but I don't think it is a driving force in the Windows market.

From your perspective, who is driving the usability issues that you mention in your last post? Do you have a requirement from a large number of users who claim that the product would be unusable unless you force the cygwin bin to the front of the PATH? Are you making unfounded assumptions?

The current behavior of Eclipse and the CDT on windows is unacceptable. I will not use Eclipse if it continues to behave as follows:

I install Eclipse and the CDT and then run it. I go to "preferences" under the window menu, and then select C/C+++ and managed build. I click on the Environment tab and check out the PATH variable which has <b>NOT MENTION OF CYGWIN.</b> I then create a new project and <b>SOME IDIOT PREPENDS c:\cygwin\bin TO MY PATH VARIABLE.</b> If I exit Eclipse and append C:\cygwin\bin to the end of my PATH variable and then restart Eclipse and create a new C++ project <b>SOME IDIOT MOVES IT TO THE FRONT OF THE PATH.</b> If you are going to make decisions about configuration by looking for CYGWIN in the registry, <b>YOU SHOULD DO THIS AT INSTALLATION!!!!!</b>

I personally find it very confusing and counter intuitive for any part of the CDT to muck with environmental variables such as PATH. The decision to change environment variables should be in the hands of the user and not the CDT tool. I think you are being way to paternalistic when you claim to know what is best for the user.

In my opinion there are 4 build environments that should be supported by Eclipse and the CDT:

1) plain vanilla windows that does not make use of CYGWIN or MINGW.
2) windows and MINGW and MINGW make (currently required for QT 4).
3) MINGW and MSYS make.
4) CYGWIN

Eclipse is NOT a cygwin tool. It is a WINDOWS tool. It does not require CYGWIN. Therefore I don't believe it should not make any assumptions about CYGWIN. It is nice that you are providing integration with CYGWIN, but it shouldn't interfere with using Eclipse with other build environments. I believe there are ways to do this and have provded some suggestions below:

1) Prompt the user for configuration information during installation or the first time Eclipse is run. For example, Eclipses could search for CYGWIN and ask the user if they want to use CYGWIN, MINGW, or MSYS. In addition, Eclipse could check for changes in the environment during startup and prompt the user if the environment changes.

2) Require the user to set up the PATH variable prior to starting Eclipse. This could be done two ways: 

a) Provide a .bat file and desktop link that starts Eclipse with the CYGWIN variables set or require Eclipse CYGWIN to be started from a CYGWIN shell. This is what CYGWIN itself does. 

b) Require the user to start Eclipse from within a CYGWIN shell.

3) Use an environment variable such as CDT_BUILDPATH to determine the behavior of the CDT on windows. If CDT_BUILDPATH is not set, then the CDT could search for CYGWIN and set the path appropriately during the "create project". If CDT_BUILDPATH is set, then the CDT would prepend CDT_BUILDPATH to PATH.

There are probably more options, but these are the ones that come to mind.

Hope this helps.

Gregg

Comment 26 Mikhail Sennikovsky 2006-06-19 12:29:30 EDT

Hi Tiziano,

> I don't know what you mean with -mno-cygwin ...
> But ... is it really your intention to try to convince people that they have to

No, we do not intend to convince people to use any specific tool-chain.
CDT is shipped with the Gnu tool-chain support as the default one that actually can be used as a reference for integrating any other tools/tool-chains.
It is the Gnu Cygwin tool-chain integration (not the CDT ) who adds the Cygwin bin to the PATH for the projects using this tool-chain.
In your case if you are integrating some custom tool-chain you could either define a new tool-chain/project type or modify the existing Gnu Cygwin tool-chain to not supply the PATH.

There are some others, e.g. Cygwin and MinGW tools treat the environment variables representing includes/lib paths differently.

> Why do you prefer Cygwin over MinGW ?
The Gnu Cygwin tool-chain is the Cygwin integration. Before we implemented the Cygwin PATH discovery there were a lot of complains from users who installed CDT and could not build out of the box because Cygwin bin was not in their PATH.
As for now we are planning to implement a separate MinGW tool-chain in the future.

Mikhail

I am not a Cygwin/MinGW expert, so I can not list for sure all the differences.
The main principle difference I’m aware of is that Cygwin has a Cygwin runtime used to emulate Linux environment and so your application communicates with OS via a Cygwin runtime and AFAIK you can not link with Windows-native libs, while MinGW allows doing this.
The other thing to notice is that MinGW make does not fully support the Gnu make syntax and might fail to build the makefiles generated with the default GnuMakefileGenerator supplied with MBS.
