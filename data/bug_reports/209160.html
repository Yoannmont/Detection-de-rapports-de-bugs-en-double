Description Mariusz Wojtysiak 2007-11-08 06:03:21 EST 
In method GliteSandboxFilesCreator.internalCanCreate(Object)
I have to check, if parent of passed folder is GridJob with GLiteJobId. To do it, I have to find in model grid element based on IResource, using:
IGridElement parentElement = GridModel.getRoot().findElement( folder.getParent().getLocation() );

Unfortunately GridModel.getRoot() cause that GliteSandboxFilesCreator.internalCanCreate() is called again for the same folder :-( and this happens untill stack is overflow :-(

Just uncomment eu.geclipse.glite.internal.GliteSandboxFilesCreator.internalCanCreate(Object) line 50 to see it.

Reason:
During creation of GridRoot, in constructor ResourceGridContainer() there is called fetchChild().
fetchChild() creates all model objects for GridRoot, but in this moment GridRoot.singletone is still null, so next getRoot() is trying to create GridRoot again.

More generally: calling not-static methods like fetchChild() within constructor is very dangerous. During working fetchChild() object is still not initialized (costructor of inherited classes is still not finished).

Problems with calling fetchChild in constuctor of ResourceGridContainer() appeared many times in Poznan.  We always spend a lot of time to investigate what is wrong and try to workaround it somehow. But I think we need clear solutions without workarounds.

For my current problem I just only check parent folder name, but it's not enough.

Please consider if we can remove calling fetchChild() from ResourceGridContainer()

Comment 3 Mathias Stümpert 2007-11-26 12:11:05 EST 
This is a well known problem but there is no way to come around this. The model has to be fully created before one can perform operations on it. This definitely contains to create all elements that depend on local resources and therefore the fetchChildren() has to be called within the ResourceGridContainer. Furthermore it is bad style to require an element creator to depend on other resources (e.g. on parent resources) than on the resource it would like to create an element from.

What exactly does your GliteSandboxFilesContainer contain? Does it really contain physical files or may it also be a virtual container?

A hint that may help you to come around your problem may also be to make use of the IFolder's setPersistentProperty-method in order to mark the folder as a sandbox files container when it is created for the first time.

Just some quick shots in order to give you some input. Please comment ...
Comment 4 Mariusz Wojtysiak 2007-11-27 03:32:34 EST 
GliteSandboxFilesContainer is an IFolder, which contains links to files. These links are created using IFile.createLink(URI, int, IProgressMonitor) and in our model appears as ConnectionRoot objects.
In current implementation links are created during job-submission and when job status is changed to DONE, so now it will get me a lot of time to make GliteSandboxFilesContainer virtual.

I have to check if given IFolder contains staged files for g-Lite, and your hint: setPersistentProperty() will solve my problem. I will use it. Thanks!

Btw. We still have dangerous calling of fetchChild() within ResourceGridContainer(). It's potentially source of errors, because any developer may call within fetchChild() (even not directly) method, which manipulating on object not initialized completely yet.
What I can suggest is to remove calling of fetchChild() in ResourceGridContainer(). Let's creator call fetchChild() for non-lazy objects just after creation.
Comment 5 Mathias Stümpert 2007-11-27 04:19:51 EST 
There may be a problem with setPersistentProperty() that you should be aware of. When sharing a project (i.e. CVS or SVN) and importing it back I do not know if these persistent properties are recovered. So please check if this may be a problem for you.

I know that calling fetchChildren() from within the ResourceGridContainer()-constructor may be dangerous and it proved to be in the past. But for the above mentioned reasons there is no way around this. Furthermore moving this functionality to the creator may be even more dangerous since other creators than the model's own creators may forget about calling this method and may therefore mess up the model completely.

Nevertheless I am aware of this problem and I am thinking about other solutions to come around it. Let's see what will be the result on this...
Comment 6 Mariusz Wojtysiak 2008-04-16 07:34:40 EDT 
Here is simple workaround, not the best but it should work: we can delay a bit fetching children of GridRoot.

Let's override GridRoot.fetchChildren() like this:

  protected IStatus fetchChildren( final IProgressMonitor monitor ) {
    // ignore fetchChildren() during GridRoot creation
    if( singleton != null ) {
      return super.fetchChildren( monitor );
    } else return Status.OK_STATUS;
  }
  
  
  And then let's call fetchChildren, when singleton is set:
  
    public final static GridRoot getInstance() {
    if ( singleton == null ) {
      singleton = new GridRoot();
      singleton.fetchChildren( null );
    }
    return singleton;
  }
Comment 7 Mathias Stümpert 2008-04-16 07:42:25 EDT 
Thanks Mariusz for the comment. But this "hack" (and I think it is somehow a hack, sorry ;-) will only work for the GridRoot but will not solve the problems with the other ResourceGridContainers.

So in fact it would solve the problem with the endless loop but not the one with the empty workspace since the model tree is still loaded completely rather than in "portions". That means if anything happens (e.g. NPE) while the tree is loaded the workspace will still be empty.

I would prefer a solution that is much more general (like the two solutions above). Therefore I would rather change either the ResourceGridContainer's or the AbstractGridContainer's constructor/fetchChildren().
Comment 8 Mathias Stümpert 2008-11-19 04:29:13 EST 
Coming back to this bug the situation here has changed significantly since the grid element creator mechanism has been reworked. Now the element creators do not have the canCreate methods by default any more. The canCreate functionality has moved to the corresponding extension point definition and therefore no endless loops due to canCreate functionality is expected for the default creators.

Nevertheless there is still the possibility to define an ICreatorSourceMatcher for the canCreate pattern and this pattern is still used for some creators. So all in all the situation has improved but is not yet completely solved.

I had a closer look at the problem and I still do not see a solution for omitting the creation of the whole model tree by calling fetchChildren at element creation time. So I would vote for closing this bug as WONTFIX. That still means we have to live with workarounds in some cases when an ICreatorSourceMatcher is needed. But replacing the call of fetcChildren from the ResourceGridContainer's constructor is in my eyes much more dangerous and causes a lot of sideeffects that we may not be have under control.

So Mariusz, please think about it and tell me your thoughts. I would also like to hear Ariel's opinion on that. As I already said I personally would rather leave everything like it is than changing things and risking a highly unstable core model.
Comment 9 Mariusz Wojtysiak 2008-11-20 03:07:37 EST 
We are living with problem since beginning of the project and every time we was able to find workaround.

Important thing is that current solution doesn't affect user's work. It's a problem only for developers, who may trap into endless recursion if she/he use GridModel within constructor/creator. But developer will notice this recursion for sure, so the problem is only to understand how/why it happen...

In this project phase changes in building model is dangerous, so I agree with Mathias we should stay with current solution.
Comment 10 Mathias Stümpert 2008-12-01 04:28:19 EST 
So I am marking this as WONTFIX for now.
