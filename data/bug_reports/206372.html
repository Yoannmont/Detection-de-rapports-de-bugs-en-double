Description Szczepan Holyszewski 2007-10-15 15:44:59 EDT 
Build ID: M20070921-1145

Steps To Reproduce:

1. Create a CDT managed project while GCC 4.2.1 is installed on your system
2. Upgrade GCC to 4.2.2
3. Rebuild the project

When you rebuild, new paths are discovered:

/usr/include/c++/4.2.2
/usr/include/c++/4.2.2/x86_64-unknown-linux-gnu
/usr/include/c++/4.2.2/baskward

However the old paths with 4.2.1 in them are not discarded. They are still shown in the Includes virtual folder for the project, and probably still followed by the indexer because the old paths seem to be higher on the list.


More information:

What do I edit to remove the old paths by hand?

Comment 3 James Blackburn 2008-03-06 12:01:13 EST 
In fact it would appear that the Managed Build PerProjectSICollector collects discovered scanner info (includes and defines) without ever clearing them:

Stack looks like:
PerProjectSICollector.includePathsNeedUpdate(IDiscoveredPathManager$IPerProjectDiscoveredPathInfo) line: 284	
PerProjectSICollector.scannerConfigNeedsUpdate(IDiscoveredPathManager$IPerProjectDiscoveredPathInfo) line: 229	
PerProjectSICollector.updateScannerConfiguration(IProgressMonitor) line: 203	
CfgSCJobsUtil$2.run() line: 148	
SafeRunner.run(ISafeRunnable) line: 37	
Platform.run(ISafeRunnable) line: 857	
CfgSCJobsUtil.updateScannerConfiguration(IProject, CfgInfoContext, SCProfileInstance, IScannerConfigBuilderInfo2, IProgressMonitor) line: 158	

with updateScannerConfiguration(IMonitor) doing:
IDiscoveredPathInfo pathInfo = MakeCorePlugin.getDefault().getDiscoveryManager().getDiscoveredInfo(project, context);

The PerProjectSICollector.scannerConfigNeedsUpdate() method only merges discovered paths/symbols. Moreover  IDiscoveredPathManager.removeDiscoveredInfo(...) appears to have no effect as the removal isn't persisted and PerProjectSICollector.updateScannerConfiguration's getDiscoveredInfo will reload the previous scanner info.

Another problem is that SCProfileInstance.getScannerInfoCollect() only instantiates the scanner once and the PerProjectSICollector's sum* fields are never reset (deleteAll() doesn't ever appear to be called).

Resetting the already PerProjectSICollector instance and ProjectDiscoveredPathInfo on every updateScannerConfiguration works, but I'm not convinced this the right place to do it? 

if (pathInfo instanceof IPerProjectDiscoveredPathInfo) {
    IPerProjectDiscoveredPathInfo projectPathInfo = (IPerProjectDiscoveredPathInfo) pathInfo;
    projectPathInfo.setIncludeMap(new LinkedHashMap());
    projectPathInfo.setSymbolMap(new LinkedHashMap());            
    this.deleteAll(project);

Does anyone know how this is meant ot work?  Surely, in the case of a managed build, we're only fetching built-in compiler symbols/includes, so is it not safe to nuke old discovered values with every build/discovery?

Comment 8 Andrew Gvozdev 2010-06-11 15:06:23 EDT 
Created attachment 171761 [details]
proof of concept patch

This patch provides a button in Paths&Symbols dialog to clean scannerInfoProvider discovered entries (built-in compiler specs). In the patch, the button wipes out all entries for the given language, not only given type. There are 3 kinds of caches to clean, see the meat in BuildEntryStorage. Then Scanner Configuration Builder will re-discover the entries anew during a build.

The patch is intended to be a proof of concept. Ideally, currently I think, the cleaning of the collector needs to be done before running scanner info provider (i.e. GCCSpecsRunSIProvider), perhaps in initialize() method of the superclass, so re-discovery is always done when SI provider runs (and UI like in this patch is unnecessary). But scanner discovery code is like minefield, maybe I'll do a safe patch (I think) based on the attachment in 8.0 and just develop more suitable UI.

Comment 30 Andrew Gvozdev 2013-05-26 08:15:32 EDT 
The current status of this bug is that it was in fact used to give an ability to clean entries with "Clear" button - whether it fits the original description perfectly or not. This was implemented and committed to CDT 7.0.1. This bug is under record for that release.

The problem with *automatic* re-discovery was delegated to be tracked with bug 142619.
Note that this discovery method was being deprecated in CDT 8.1 and made hidden in CDT 8.2 in favor of language settings providers ("Preprocessor Include Paths, Macros, etc." property page), so it is not very likely somebody comes forward to fix bug 142619 although you are welcome to do so.

For language settings providers automatic re-discovery has not been solved yet either. It is tracked with bug 392416.
