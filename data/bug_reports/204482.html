Description James Kaufman 2007-09-24 14:20:57 EDT 
The multicolor display used to show multiple colors for SEIR. There are set in the preferences. 
The preferences page defaults seem to be broken (gain factor field was blank)
and the colors are not being picked up anymore.
Comment 1 Daniel Ford 2007-09-24 17:52:49 EDT 
This resulted from the integration of the NASA World Wind view which required extensive refactoring to create a common geographic view structure in the code base that is extended by both the Map and Globe views.  The refactoring greatly simplified the implementations of both views and allowed for common background processing of polygon rendering. 

The implementation of color processing in the map view wasn't general enough to be included in the factoring so the two views have different color "pallets" and processes for matching variable value to color.  As the two views share high-level preferences it wasn't possible to process preference information consistently.  Also, the inclusion of preferences directly related to disease modeling such as "S", "E", "I" and "R" variable names limits the generality of the interface.  The way both the Map and Globe views are implemented makes them completely independent of the underlying application running in the core STEM components.  In the refactoring these dependencies were commented out to restore the independance  and to allow for future implementation of preferences that matched the new refactored Java package structure.

So, the next step is to implement the preferences so that they match the new organization and to make them independent of the underlying application.  One way to do that is to have a user supply a character string in the preferences that matches the name of the variable they are specifying values for rather than hard coding.

The attached patch restores the ability for the users visualization preferences to assign colors to the simple stem map. This implementation of color processing in the map view may not be general enough for the new GLOBE view but it DOES NOT constitute inclusion of preferences directly related to disease modeling such as "S", "E", "I" and "R". Comment 4 James Kaufman 2007-11-09 17:23:33 EST 
Input re: color 
Investigating the use of color in the GlobeRenderer, the current Globe color is assigned only based on the amplitude of the double value rendered for the polygon. For example,
if (relativeValue > 0.8) {
			...
		} else if (relativeValue > 0.6) {
			 retValue = fillColor06_08;
		} 	else if (relativeValue > 0.4) {
			retValue = fillColor04_06;
		} else if (relativeValue > 0.2) {
			retValue = fillColor02_04;
			
etc.  These are currently hard coded. Using color to distinguish amplitude is a useful mode of display on a map, especially if the map is displaying ONLY one label or variable (eg topographic maps may use white for mountain tops, blue for sea level, etc). However, in STEM we are enabling multiple decorators each with multiple properties to display per decorator. I think it's fine to let all of our views operate in a "topographic" single variable mode but it would be even more general to change the Globe render to use the same color preference selection mechanism as the map viewer. I think this is a better consistent visualization for STEM. It allows multiple variables. Both use awt color so the mechanism should work ok.
Comment 5 Daniel Ford 2007-11-10 15:16:09 EST 
I agree we should make the Map and Globe renders use exactly "the same color preference selection mechanism".  Really, there should be a single color scheme for GeographicViews and taking this further we should consider how that might affect the GoogleEarth view, as it is really a "GeographicViewer" too.  

One general way to approach this is to create a "ColorProvider" interface, instances of which are provided to the GeogrphicRenderer instances (i.e., MapView and GlobeView).  It would take an Identifiable (e.g, a NodeLabel) and produce a color value.  To reproduce our current functionality, we could implement a "RelativeValueColorProvider" that when given a NodeLabel instance tries to adapt it (like the Map and Globe views do today) to a RelativeValue provider and then map the relative values to colors.   How it does this is not the concern of the renderer.  This would make the GeographicViewers much more generic as they wouldn't be limited to processing "relative values", they could be given a different kind of ColorProvider that would take some different aspect of the Identifiable and produce a color value.
Comment 6 Daniel Ford 2007-11-10 16:36:05 EST 
Actually, it just occurred to me that we can make this even more flexible.  We allow for more than one type of ColorProvider in a GeographicControl and add the ability for the user to select which one to use.  For instance, we have one that processes relative values, you might have a different one that processes delta's (first derivative), let the user select which one to display with a combo box.  The set of ColorProviders given to the GeographicControl could be obtained from the Scenario/Canonical graph instance; we adapt the instance to something that can generate a set of ColorProviders for us.  The code in the GeographicView starts to become pretty simple doing it this way.  

The set of ColorProviders is created by looking at the different types of labels in the graph and attempting to adapt them to a known collection of ColorProviders (culled from an extension point).  The ones that can adapt are added to the set, the ones that can't, are excluded.  This would allow new types of ColorProviders to be plugged into STEM, making the displays completely extensible.

The way to handle preferences in this case is to allow the ColorProviders to contribute a preference page at the appropriate place, for example, the RelativeValueColorProvider would allow matches between particular character strings and colors on its page.

public interface ColorProvider {
	Color getColor(Identifiable);
}

We could also think about something like an IconProvider that would work much the same way as the ColorProvider, but would allow Identifiables to be mapped to icons like airports, cities, ships, etc.
