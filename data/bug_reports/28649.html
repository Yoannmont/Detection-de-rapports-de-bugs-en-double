Description Larry Wood 2002-12-18 17:02:50 EST 
Just a suggestion:

It would be nice if the contents tree on the workbench guide would open and 
highlight in conjunction with the current page.   It would make it easier to 
visually see where the page is in the tree.

Larry Wood
Comment 1 Dorian Birsan 2002-12-18 17:39:17 EST 
Larry,

There is a Synchronize button on the main frame toolbar that synchronizes 
content with navigation tree, just like you want (but it is user driver, as 
opposed to automatic).

The operation is relatively expensive if done all the time, but content 
providers can choose to call javascript help API's to support this.

I will mark this as an enhancement.

Comment 5 Curtis d'Entremont 2007-08-24 11:23:00 EDT 
My opinion is that, ideally, this should be done in exactly the same way as the eclipse workbench navigation views; a toggle button that lets you turn synchronization on/off, and when it's on it just happens automagically. No changes required to doc plugins. I see the use of livehelp to do this only as a workaround for the lack of support for it in help.
Comment 6 Chris Goldthorpe 2007-08-24 12:27:18 EDT 
I haven't looked at how livehelp.js does this but I will take a look at that and also Dorian's comments. The approach I'm working on would involve adding javascript to the page which came back from the content servlet to call a function in the TOC view to do a variant of "Show in TOC". In order to get this to work correctly I need to make sure the following occur:

Only add the JavaScript where it makes sense, i.e. only for content pages and not if we were called from the help view.
If the TOC view is hidden, such as when a search result is selected we may not need to synch immediately.
Optimize the performance of Show In TOC. At present this is done using one AJAX call which returns all the ancestors and immediate siblings of the selected node, this is fine for 3.3 functionality but would not be fast enough to handle the automatic synchronization.

Assuming I can get this to work reliably and with good performance then we may not need a toggle button, the button bar is pretty full already. I would just leave the feature turned on by default and see if we get any negative feedback from the milestone builds. There could be a preference setting to satisfy any users who preferred the old non-synching behavior. 

I wasn't even aware of the "Link with Editor" button in the package explorer until Curtis pointed it out in his previous comment.

Comment 9 Chris Goldthorpe 2007-08-29 13:49:40 EDT 
I'm now far enough along in the implementation to see how the automatic synchronization looks, and I'm rethinking my previous comment about the button bar. 

Automatic synchronziation is a nice feature and most people will want to leave it on, but if you don't care about synchronization the changing TOC view could be a visual distraction so I think a button to toggle the autosynch state is appropriate. Another thing which was apparent from testing is that autosynch should not cause the TOC view to be made visible if another view such as the search view had focus.

As of 3.3 each of the views (toc, index, bookmarks, search, links) has a "Refresh/Show current Topic" button. The Content toolbar also has a "Show in Table of Contents" button which is essentially the same thing. I have always felt that having two buttons for the same thing was redundant and if we could get this down to one button it would be an improvement.

Here's what I would like to do:

In the TOC view replace the "Refresh/Show Current Topic" button with a button to toggle the synchronize state.

and either:

Plan A:

Remove the "Refresh/Show Current Topic" button from the other views (search, index etc) and maybe hide or gray out the show in TOC button if the TOC view is visible and autosynch is turned on.

Plan B:

Leave the "Refresh/Show Current Topic" button in the other views but change it's icon to the Show in TOC icon. Remove the "Show in TOC" button.

Plan C:

Leave the other buttons as they are.
Comment 10 Curtis d'Entremont 2007-08-29 14:08:07 EDT 
I like plan A.

Some users, like myself, don't like to see the navigator swishing around as I'm navigating the code via F3 (so I turn auto-sync off), but occasionally do want to see where my file is, e.g. if I know that there's another file next to it I want to now edit. I do this by clicking on the synchronization toggle twice; once to sync, and second to turn it off again. Alternatively, you can right click in the java editor on an element and select "Show in > Package Explorer", which is a longer way of doing the same thing. Since we can't right click in the web world, a toolbar button to manually sync (what we have now) does the trick.
Comment 11 Adam Archer 2007-08-29 14:18:13 EDT

+1 disabling the content toolbar button instead of hiding it.

I experience the same use case Curtis describes and, personally, I would not leave that toggle on in the help system most of the time.
Comment 12 Chris Goldthorpe 2007-08-29 19:38:38 EDT 
I have committed code to HEAD which implements most of Plan A. I still have to get a new icon for the synchronize button, disable the show in TOC button as appropriate and optimize the show in TOC code to make it run faster and only when the TOC view is visible. One nice aspect of the synchronization is that it also works when you use the forward and back buttons. For the most part the performance is good but the synchronization is reading far more data than is necessary if the tree already has the node loaded in memory.

At present the default is for this button to be turned on but I could be persuaded to make turned off the default.
