Description David Biesack 2005-05-12 08:45:49 EDT 
I'm in the midst of submitting a plugin I wrote as a sourceforge.net project and
thought it might make an interesting article -- how to contribute an open source
plugin on sourceforge.net.

Below are some things I had to discover for myself:

   How to structure a sourceforge.net project (i.e. create three Eclipse
projects: one for the plugin, one for the feature, one for the web site)

   Naming the package. Probably should not use org.eclipse just yet :-). I chose
net.sf.eclipseexeditor over org.biesack etc. so that it will feel more "open" as
opposed to an egotistical thing

   Process of registering on sourceforge.net (there's a lot of doc there, almost
too much to wade through...)

   Using sourceforge's cvs server for Eclipse projects

   ssh (OpenSsh or PUTTY), private and public keys

   Uploading files and making a release

   Naming the project

   Trove classification

   Building a web site

   Hosting an update site

   Registering with a plugin promotion site
(http://www.eclipse.org/community/osplugins.html)

Other topics to include:

  Why sourceforge?

  Choosing a license

Comment 9 David Biesack 2005-07-26 09:32:29 EDT 
In response to #8:

I have set aside a section (yet to be written) on hosting an update site, but I
think I will merely mention why not to do it on sf.net. I could not find
anything which actually discourages this (pointers welcome) except that you are
encouraged to have you downloads done through the file release system, and that
is the best way to track the number of downloads. I thought that I would
describe how to create a download site *image* as a zip and release that zip
image as a file release, so that others can make local mirrors of your update
site. However, doing so has two downsides: the mirrors can become out of date
when you release an update, and again you cannot track how downloads of your
plugin performed from such mirrors.

Comment 15 Craig Setera 2005-07-28 08:28:00 EDT 
Geoff and I exchanged notes about how to host an update site a while ago and I
ended up with a hybrid approach for EclipseME.  I have paid hosting through
LunarPages.com and I also use a single file in the FRS so that all downloads get
registered whether they came through the update site or via the archived update
site that I also make available.  I use an .htaccess file in the plugins
directory to have Apache redirect Eclipse to pick up one of the plugins from the
file release system.  The rest are picked up from eclipseme.org web hosting. 
This is relatively easy to set up and gets me relatively accurate statistics.

My .htaccess file which automatically picks up the correct version setting:

RewriteEngine On
RewriteRule ^eclipseme\.core_(.*)\.jar$
http://osdn.dl.sourceforge.net/sourceforge/eclipseme/eclipseme.core_$1.jar

Comment 29 Ed Burnette 2006-07-24 12:25:27 EDT 
I recently had to connect to an existing sf project as a developer in Eclipse. The ssh stuff was easier than indicated in the article and you might want to update it.

You don't need OpenSSH or Puttty - everything can be done in Eclipse. In Window > Preferences > Team > CVS > SSH2 Connection > Key Management, you can just click on Generate DSA Key, save the private key on your local disk, and copy the public key into the clipboard. Then surf to sf.net, log in, click on My Page > Preferences, and then in the Host Access Information settings edit the SSH Keys for Shell/CVS and paste in your public key there, and click Update. You can have more than one public SSH key associated with your account.

Once this is done, wait 10-30 minutes for the keys to sync up, and then you can add the CVS repository location from within Eclipse using the extssh method and start checking files in and out.

If you were previously using the anonymous pserver method (this is common when moving from contributor to committer status), right click on the project, select CVS, and click on the button to Change Sharing, then select the extssh repository location you just created.

sf.net also supports Subversion now but I haven't tried it there. And of course there's no single standard for SVN access in Eclipse.
Comment 30 Lars Ködderitzsch 2006-11-22 02:15:30 EST 
Update on using the SF.net FRS with update sites:

Previously you were not able to use the plain download links of the FRS with the update site, since SF required manual selection of the download mirror to use.
One could work around this by hardcoding the file links on a specific mirror into the site.xml. A downside was that you were losing the download stats for the installs via update site.

Since recently SF provides transparent mirror autoselection, which means that the plain file links from the FRS can be used now in the site.xml. Even the stats are no issue anymore.

Example: http:/eclipse-cs.sourceforge.net/update
