Description Jared Burns 2004-02-13 18:18:52 EST 
Build 20040212 and earlier

I find that the progress monitor dialog very often blocks operations from proceeding when I switch 
Gnome workspaces. I often click a button that starts a long-running operation, then switch to 
another workspace to do other things (email, web browser, etc.) while the operation runs, only to 
find that the operation hasn't proceeded at all.

It's like the dialog is blocking during its creation and doesn't get around to starting the runnable until 
its had a chance to redraw?
Comment 1 Jared Burns 2004-02-19 17:21:15 EST 
The problem also seems to occur at the end of the view's life. For example, I just selected the 
projects in my workspace and chose Team->Update. While that was whirring away, I switched to 
a different Gnome workspace. After a while, I noticed that my CPU was idle so I went back to the 
workspace with Eclipse. At which point the progress dialog closed and a build began which put my 
CPU back to work.

The effect here is that you can't switch Gnome workspaces and expect Eclipse to finish what it's 
doing. Instead, you have to go back to the Eclipse workspace periodically to give the progress 
dialog a nudge (by letting it redraw, presumably).
Comment 2 John Arthorne 2004-02-19 18:27:54 EST 
Interesting... the progress monitor callback spins the event loop for up to 50ms
on each progress report:

long t = System.currentTimeMillis();
for (;;) {
	if (!disp.readAndDispatch()) 
		break;
	if (System.currentTimeMillis() - t > 50) 
		break;
}

It needs to do this to make the cancel button responsive when something runs in
the UI thread. It sounds like Display.readAndDispatch is blocking indefinitely
if the Gnome workspace is switched? I didn't think readAndDispatch was supposted
to be a blocking read...
Comment 3 John Arthorne 2004-02-19 18:35:57 EST 
Sorry, I wasn't clear enough.  Something long running is happening in the
background, and it is reporting progress to the progress view, etc.  Some other
operation tries to run in the UI thread (such as editor quick diff). If that
foreground operation is blocked by the background operation, we open the
"blocked" dialog, and then spin the event loop until the lock is acquired, to
allow the cancel button to work. If readAndDispatch blocks under certain
conditions, this would prevent the UI thread from continuing, even after the
lock it was waiting for is available. In pseudo-code it looks like this:

while (!lock.isAquired()) {
  while (display.readAndDispatch()) {}
  Thread.sleep(50);
}
Comment 4 Jared Burns 2004-02-19 21:12:44 EST

when those workspaces aren't active. So I should be able to tell if the new "blocked" dialog is 
opened and I haven't seen that happen. I just see the normal progress dialog.

Comment 6 Steve Northover 2004-02-26 18:39:14 EST 
It blocks because Shell.setVisible() is defined to return after the Shell has 
been drawn.  This is implemented on GTK by waiting for the shell to be 
mapped.  When you switch Gnome workspaces, it seems the callback doesn't come 
in until you switch back.

Does this happen on Motif as well?
