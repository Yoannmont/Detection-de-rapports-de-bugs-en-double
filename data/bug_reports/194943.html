Comment 15 Thomas Watson 2007-07-06 17:51:45 EDT 
Created attachment 73225 [details]
patch

It may be possible that the bundle file closer (MRUBundleFileList) is closing ZipFiles while an entry inputstream is being read from the ZipFile.  I would expect this to throw an IllegalStateException instead of crashing the VM though.

This patch uses a strategy of counting open inputstream references to a ZipFile and only allowing the background thread "bundle file closer" to close a ZipFile if all open inputstreams have been closed to entries of the ZipFile.  This does mean that if some code leaks InputStreams from bundle entries then we will never close the BundleFile with the leaked inputstream.  Currently the patch just gives up after 1 second.  Another approach could be to queue the bundle file up for closure again later.

I have not been able to reproduce the crash myself with the Sun VM so I cannot confirm that this patch actually fixes the issue.  I have a feeling it will not fix the issue, instead I think there is a VM bug that is preventing the same file path from being opened/closed multiple times with java.util.zip.ZipFile in some situations.

Please give this patch a try in the environments where you can reproduce the crash and report back the results.  Thanks.
