Comment 17 DJ Houghton 2007-10-19 14:19:48 EDT 
Roy, thanks for the information. We are basically looking at the current Eclipse test framework and trying to enable it to run both JUnit3 and JUnit4 test suites without requiring clients to modify their test bundles. I will try and describe the approach that we have taken. (and the problems that we are encountering)

The org.apache.ant bundle contains the ant.jar as well as jars for optional tasks, like JUnit. Unfortunately, the manifest file for this bundle doesn't have JUnit and other in its dependency list. These reasons are outlined in bug 36198 and bug 157622 and we cannot add them to this manifest.

As a work-around, the test framework contains a fragment to the org.apache.ant bundle (called org.eclipse.ant.optional.junit) which is empty except for a requirement on the org.junit bundle. This essentially adds JUnit to the classpath of the Ant bundle. If we were to run without this fragment, we would see NoClassDefFoundErrors when trying to load classes such as XMLJUnitResultFormatter which reference JUnit classes.

This all works great for JUnit3 tests. 

So then JUnit4 came along so we started looking at adding support in the Eclipse test framework for JUnit4 tests. Lucky for us, Junit4 is backwards compatible so the code changes are actually pretty simple. All we need to do is when we load the our class to be tested, we wrap it in a JUnit4TestAdapter and also change the dependency in org.eclipse.test from the org.junit bundle to the org.junit4 bundle.

But with this setup we still have a mix of JUnit3 and JUnit4 dependencies, so we also have to change the dependency statement in the org.eclipse.ant.optional.junit fragment to be on org.junit4.

But of course this doesn't work and actually gives us ClassCastExceptions because we try and cast (for instance) our test class (which is a subclass of TestCase from org.junit - Junit3) to a TestCase (from JUnit4 which is referenced by org.eclipse.test). The problem this time is that my test plug-in is requiring org.junit which is JUnit3. 

If I change the dependency in my test plug-in from org.junit to org.junit4 then it all works ok. BUT since this is a test framework and there are literally hundreds of bundles out there who depend and use us, we cannot just tell everyone that they have to change their dependencies or they will be broken.

So that's basically a summary of where we stand now. We have one more option that we have to investigate. (before we go crazy with all this classloading and classpath stuff)

When we first introduced JUnit4 into Eclipse, we put it in a bundle named org.junit4 because we had problems with multiple versions of the same bundle in our workspace at the same time. A lot of these problems have been fixed so we are looking at having the JUnit4 code in a bundle named org.junit. That way, clients who currently require the org.junit bundle should just get the right one and everyone will be happy.

Off the top of my head, the only thing that might still be a problem is if they put version ranges in their require fields, but I'm sure that no one does that.... ;-)

Any comments or insights are appreciated. We will keep this bug report updated with status.

Comment 20 Allan Godding 2007-10-22 17:04:30 EDT 
Created attachment 80909 [details]
Patch for Eclipse Testing Framework

This is a patch for the org.eclipse.test package.  The approach taken to run JUnit4 style tests is to wrap the test class in a junit.framework.JUnit4TestAdapter.  The org.eclipse.test bundle will still compile against Java 1.4.x, assuming you are not trying to run JUnit4 tests.

To determine what type of test the user is attempting to run, the version of the org.junit bundle is checked, and the tests are run as the highest version of org.junit that could be loaded.  Thus to allow this patch to work, the Bundle ID of the current org.junit4 will need to be changed to "org.junit".

Because the JUnit4TestAdapter is backwards compatible, the user will be able to run JUnit3 tests with a Java 1.4+ VM.  So even if the 4.x version of org.junit is loaded, JUnit3 tests can still be run as a JUnit4TestAdapter.  Essentially, the JUnit4TestAdapter allows us to run either JUnit3 or 4 tests as if they were JUnit3 tests.

*Coles Notes*:
 -JUnit3 testing will require a Java 1.4+ VM
 -JUnit4 testing will require a Java 5+ VM
 -A bundle named org.junit that mirrors the current org.junit4 bundle is required.
Comment 21 Allan Godding 2007-11-02 16:03:54 EDT 
Created attachment 81997 [details]
code for Eclipse Test Framework

This zip file contains new org.junit, org.junit4, and org.eclipse.test.dispatcher bundles, as well as a patch for org.eclipse.test.

The org.junit4 bundle is now empty, and acts as a mirror of org.junit version 4.3.1 by re-exporting its packages.  org.junit version 4.3.1 simply has all the code that org.junit4 previously had, but now has the same bundle ID as org.junit version 3.8.2.

org.eclipse.test.dispatcher is now the home of CoreTestApplication and UITestApplication.  These are basically wrapper applications that prevent the org.eclipse.test bundle from being loaded before we can configure the junit bundles.  These applications are simply responsible for uninstalling org.junit version 4.3.1 if it is not required.  Once this has been done they call either the CoreTestApplication or UITestApplication in org.test.eclipse which have both remained relatively unchanged functionally.

More information will be posted at http://wiki.eclipse.org/ETF_JUnit4_Changes
