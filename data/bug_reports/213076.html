Description Francois Chouinard 2007-12-14 20:25:01 EST 
Typically, we use DsfMIBreakInsert to insert a breakpoint which allows to specify a condition at breakpoint creation time. In DSF, this translates to an MI command that looks like this:

-break-insert -c "condition" [some location]

where the condition, like any String parameter on a DsfMICommand, is quoted.

Unfortunately, this also applies when adding a condition to an existing breakpoint. In that case the MI command is formatted as following:

-break-condition [breakpoint id] "condition"


The problem is that for that command (and a few others), the condition should not be quoted. GDB expects the following form:

-break-condition [breakpoint id] [condition]

If quotes are added, they become part of the condition expression and the condition doesn't work properly.

The basic problem is that GDB doesn't parse the command parameters the same way for MI and CLI commands. More specifically, most commands (like break-insert) are actually implemented in the MI interpreter (actually in libgdb) while some others (like break-condition) are converted and executed in the CLI interpreter. In the second case, the parameters are passed verbatim to the CLI and, in our case, the condition is passed with the enclosing quote marks.

There are at least 5 ways to address this problem:

[1] Provide a GDB fix that standardizes the parameter parsing for MI and CLI. No thanks (way too many impacts and side effects).

[2] Provide a GDB fix that implements break-condition. The best solution although it might break some legacy code (working on it at leasure).

[3] Patch DsfMICommand to format special commands correctly. Obviously not an option either.

[4] Remove the breakpoint and create a new one when a condition has to be added/modified/removed (the CDI way?).

[5] Patch the DsfMIBreakCondition so the command is formatted properly on the constructor:

public DsfMIBreakCondition(IBreakpointsTargetDMContext ctx, int breakpoint, String condition)
{
  super(ctx, "-break-condition " + Integer.toString(breakpoint) + " " + condition); //$NON-NLS-1$ //$NON-NLS-2$
  // setParameters(new String[] { Integer.toString(breakpoint), condition });
}

As a temporary solution, option 5 is retained. Not pretty but it will work until a GDB fix becomes available (or someone comes up with a better DSF alternative).

Best Regards,
/fc

Comment 14 Marc Khouzam 2011-02-04 23:22:10 EST 
Created attachment 188383 [details]
Slightly modified solution

I've tested with double quotes and no double quotes for -break-condition with GDB 6.8, 7.0 and 7.2 and double quotes should never be used in this case.

The command always gets accepted with or without double quotes, but the condition will not actually work if it had double quotes.

So, the current solution would seem to work.

However, a new problem arose:
MI can insert flags such as --thread-group between the command and the parameters.  If we make the entire output be the command, like we do now, then the --thread-group flag will end up at the end, and the syntax will not be valid.

17-break-condition 3 j == 20 --thread-group i1
17^error,msg="A syntax error in expression, near `thread-group i1'."

The syntax should be 
17-break-condition --thread-group i1 3 j == 20

To solve this, there is another trick to avoid the double quotes.  We can split the condition into multiple parameters based on white spaces.

This should work the large majority of the times.  The one case I can think of that will fail is if the condition contains an odd number of spaces in a row.  For example with 3 spaces, one of those spaces will be considered a delimiter and will be removed.  Therefore '   ' will become '  '.
The thing is, I'm not sure what kind of valid condition would have multiple spaces in a row.  I mean, for characters c = ' ', there can only be one space, and for strings, we have to use strcmp and such.

I'll have to give it more thought.
Until then, here is a patch that does this and seems to work well.
