Description Bo Majewski 2005-06-22 16:00:37 EDT 
In some cases one needs to translate plurality dependent strings. A simple
example is:

Found {0} file(s)

For Germanic languages very simple rules exists that allow one to use
java.text.ChoiceFormat. However, the do not apply to other languages. For
example, for Polish language, the following strings would be used
   * for 1 file: Znalaz\lem 1 plik
   * for 2, 3, 4: Znalaz\lem [2|3|4] pliki
   * for 5 .. 21: Znalaz\lem [5 .. 21] plki'ow
   * for 22 .. 24: Znalaz\lem [22 .. 24] pliki
   * for 25 .. 31: Znalaz\lem [25 .. 31] plik'ow

Etc., and so on. A possible solution would be to allow static variables inside
message class to be either of type String or of type String[]. The former is for
translations that do not depend on subject multiplicity. The latter are for
cases like the one described above. Loading of resources would have to be
enhanced so that if a field is an array of strings, all strings with a given key
are fetched and stored. 

Next, the NLS would need to gain a method called, say, choose. The logic of the
method could look like this:

	public static String choose(String[] message, int count) {
		int index = getPluralSelector().index(count);
		
		String text;
		
		if (index < message.length) {
			text = message[index];
		} else {
			if (count == 1) {
				// try the first field ...
				text = message[0];
			} else {
				if (message.length > 1) {
					text = message[1];
				} else {
					text = message[0];
				}
			}
		}
		
		return bind(text, new Integer(count));
	}

The job of the plural selector is to return an appropriate index based on the
supplied count. For English language the selector index method is implemented as

public class EnglishSelector implements PluralSelector {
    public int index(int n) {
        return (n == 0? 0 : n == 1 ? 1 : 2);
    }
}

For Polish language, this can be implemented as

public class PolishSelector implements PluralSelector {
    public int index(int n) {
        return (n == 0? 0 : n == 1 ? 1 : n % 10 >= 2 && n % 10 <= 4
                && (n % 100 < 10 || n % 100 >= 20) ? 2 : 3);
    }
}

In order to print the message about the number of files found one would then use

System.out.println(NLS.choose(Messages.X_FILES_FOUND, n))

And the appropriate .properties files would contain 

# For English language
X_FILES_FOUND=No files found
X_FILES_FOUND=One file found
X_FILES_FOUND={0} files found

# For Polish Language
X_FILES_FOUND=Zero plik'ow znaleziono
X_FILES_FOUND=Jeden plik znaleziono
X_FILES_FOUND={0} pliki znaleziono
X_FILES_FOUND={0} plik'ow znaleziono

If necessary I can supply implementation of selectors that cover the vast
majority of world languages. Unfortunately, MessageResourceBundle uses private
class that prevents one from developing above solution without copying large
portions of the code.

Comment 9 Bo Majewski 2005-06-23 14:13:40 EDT 
As the initial feedback seems to be positive, let me share what I have done
since. First, it is certainly true that certain Asian languages (Japanese,
Korean) have no plural forms. This is easy to implement with PluralSelector. The
second issue is related to this interface. As I am trying to implement
intelligent NLS.choose method, I realized it would be helpful to me if I knew:

a) Whether the selector treats zero case as a special case
b) How many plural forms are there

The reason is that I want to enable not so dilligent translators to specify less
that required number of forms and have the NLS.choose still perform reasonably
(at least for the English language). The possible cases is that 

a) Somebody specifies all necessary forms

X_FILES_FOUND=No files found
X_FILES_FOUND=1 file found
X_FILES_FOUND={0} files found

b) Zero form, being the least important, is dropped (still works, though: 0
files found)

X_FILES_FOUND=1 file found
X_FILES_FOUND={0} files found

c) Only one form is provided (testing?)

X_FILES_FOUND={0} file(s) found

In order to have some shot at guessing which form to take when less than
necessary forms are supplied, I'd use the following format of PluralSelector

public interface PluralSelector {
    /**
     * Returns the choice to be selected for the given multiplicity. 
     * 
     * @param n The value that directs selection of singular or plural form.
     * @return The index of the string to be selected.
     */
    public int index(int n);
    
    /**
     * Indicates whether or not multiplicity of 0 is supported.
     * 
     * @return Whether or not multiplicity of 0 is supported.
     */
    public boolean isZeroSupported();
    
    /**
     * Returns the number of plural forms. For example, for the English 
     * language this method returns 1, while for Japanese it returns 0.
     * 
     * @return The number of plural forms.
     */
    public int getPluralFormsCount();
}

Now, the English (and Danish, Dutch, German, Norwegian, Swedish, Estonian,
Finnish, Greek, Hebrew, Italian, Portuguese, Spanish, and Esperanto) language
PluralSelector takes the form:

public class EnglishSelector implements PluralSelector {
    public int index(int n) {
        return (n == 0? 0 : n == 1 ? 1 : 2);
    }
    
    public boolean isZeroSupported() {
    	return true;
    }
    
    public int getPluralFormsCount() {
    	return 1;
    }
}

While the Hungarian, Japanese, Korean and Turkish may be implemented as:

public class HungarianSelector implements PluralSelector {
    public int index(int n) {
        return (n == 0? 0 : 1);
    }
    
    public boolean isZeroSupported() {
    	return true;
    }
    
    public int getPluralFormsCount() {
    	return 0;
    }
}

Naturally, it would be possible to implement both of the above without
special-casing 0. Then the EnglishSelector would be implemented as

public class EnglishSelector implements PluralSelector {
    public int index(int n) {
        return (n == 1 ? 0 : 1);
    }
    
    public boolean isZeroSupported() {
    	return false
    }
    
    public int getPluralFormsCount() {
    	return 1;
    }
}

Comment 15 Bo Majewski 2005-06-30 13:05:47 EDT 
Notes on the submitted solution:

* I implemented a solution that prevents any plural field to be loaded using
multiple locales. It relies on a secondary map, which is discarded after each
locale specific .properties file is finished loading. This addresses the issue
of having a mix of plural forms coming from multiple files. While not so
confusing for language with a single plural form, it negatively impacts
languages with multiple plural forms

* The guessing that NLS.choose method does allows sloppy resource bundle
preparatios. While very useful in testing, it may be a problem in deployed
applications. I kept it, but I wonder if one should verify that the number of
forms supplied matches that used by the current plural selector and if not,
report a runtime error.
