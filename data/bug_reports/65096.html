Some observations after preliminary investigation:

There seems to be 2 ways to turn on high contrast, and the intro responds
differently to each:
1. using Alt+LeftShift+PrintScreen
2. through Display Properties -> Appearance -> Scheme 

- through method 1 I am able to reproduce the reported results

- through method 2 I am NOT able to reproduce the results, the intro looks normal

- opening the generated HTML in Mozilla browser works fine for both methods

- other CSS-heavy sites seem to suffer from similar problems when method 1 is
used, but are ok with method 2 is used (see various designs on
http://csszengarden.com for examples).  Again, the problems only seem to exist
on IE though, Mozilla looks ok.

Will have to keep investigating this one..
Comment 3 Kim Kelley 2004-06-08 15:06:53 EDT 
The difference between the two methods to turn on high contrast (from comment
#2) is that method 1 switches to High Contrast mode AND sets the following IE
Accessibility options by default:
1 Ignore colors specified on Web pages
2 Ignore font styles specified on Web pages
3 Ignore font sizes specified on Web pages

The first of these options also seems to turn off any background images
specified through CSS.  This becomes a problem for the intro because of the
design of the initial root page: 
- only background images are shown for each link until the user hovers over the
link.
- background images are not content, and they are not displayed when the browser
is told to ignore colors specified on Web pages
- this causes the root page to look empty because no text is displayed until you
happen to hover over the correct location.  

The other content pages are not affected in the same way, because their design
does not force the background image to be used as content (i.e., some text is
always visible).

Note that this is also a problem when in regular (not high contrast) mode, if
the user manually sets the above IE Accessibility options.

possible solutions:
- redesign the root page so at least the link labels are always visible
- allow the links to include first class <img> elements that will not disappear
when the user turns off CSS
- others?

Mazen, Dejan, Lorne, any thoughts on this?

Comment 5 Kim Kelley 2004-06-10 10:49:37 EDT 
Yes, if you use CSS to add any image it is lost when high contrast is turned on
using the method reported by Konrad, or when CSS is turned off manually.  The
reason is that images added via CSS are treated as background only.  Our problem
is not related to having a blank image present for use in the CSS.  The problem
exists because our root page design forces the images to be used as content, by
removing all other content until users hover over the image.

It is very common to use a real image as the link content (not a background
image), with text only on hover - you could even say that the eclipse toolbar
buttons fit this description.  I prefer the approach of allowing links to
contain a first-class <img> element - I think other intro designers will also
want this functionality.  The alt text on that image element would also help
satisfy the accessibility checkpoint.

In our case, we would only need real <img> elements on the root page navigation
links.

The code changes to contain this are actually very small - I've implemented it
locally to just test the theory.
Comment 6 Kim Kelley 2004-06-11 14:30:54 EDT

images: Overview, Tutorials, Samples, What's New.  Hovering over the link images
shows the hover text.
