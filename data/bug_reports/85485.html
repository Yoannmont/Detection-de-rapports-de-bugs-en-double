Description Nick Boldt 2005-02-16 12:53:09 EST 
On this page http://download.eclipse.org/downloads/index.php there are a lot of
N and I builds listed with significant build or test failures, but to verify
that I'm using the latest WORKING 3.1 driver (for EMF builds) I have to click on
each listed one in turn and look at the errors. 

Would it be possible to have some sort of visual indication on the listing page
(index.php) which shows whether or not the build built/passed?

The simplest thing I'd suggest is to parse the individual build pages, like
.../downloads/drops/I20050215-2300/index.php and just scan for any occurrences
of "FAIL.gif". If found, show a similar FAIL icon on the listing.

(Or, even better - if the build fails, don't promote it!) ;-)

If you need sample parser code that's more complex than this pass/fail
condition, like that it reads the following test results page to see which
platform the tests failed on, I've got some code in emf-home that could be used
as a starting point. See emf-home/scripts/downloads-common.php, or ask me for
details.
Comment 1 Nick Boldt 2005-09-23 13:48:23 EDT 
Better idea. Why not publish an RSS file with the status of the latest promoted
driver? One file per branch or perhaps one large file with just the latest from
each contained in it? This way projects that depend on getting the latest,
fully-tested and working I or M drivers can be notified immediately via RSS
reader (there's even an Eclipse plugin that does this:
http://www.eclipse-plugins.info/eclipse/plugin_details.jsp?id=504).

For the curious, here's a couple schema definitions for RSS:

http://www.kbcafe.com/iBLOGthere4iM/rss20.xsd
http://www.thearchitect.co.uk/schemas/rss-2_0.xsd

This file, in RSS2.0 format, would contain notes about the date of the release
(200509220200), the version (3.2.0), the type (I), and the state of the tests
[In Progress | Passed | Failed]. Underneath the test state would have to be a
list of any platforms for which there are failures and a count of how many
failures, with a link to that platform's test result page, eg. 

http://download.eclipse.org/eclipse/downloads/drops/M20050914-1235/testResults.php

Comment 3 Nick Boldt 2005-11-01 15:09:25 EST 
Created attachment 29100 [details]
prototype plugin for viewing Atom RSS 1.0 feeds, and updating feeds via commandline java task

Kim:

The attached plugin contains AtomRSSView, which is a new view for displaying
feeds that conform to the (simplified) Atom RSS 1.0v11 spec. Examples of these
feeds are in the data/ folder in the plugin, and also online here:

http://eclipse.org/emf/news/builds.xml
http://eclipse.org/uml2/news/builds.xml

The script for updating feeds is AtomRSSUpdater, and there's a couple wrappers
for that in scripts/ (.bat and .sh) to give you the commandline syntax, beyond
what's documented in AtomRSSUpdater.java itself. The updater uses a properties
file specific for a given project (in my case, UML2 and EMF). The "xyz" one is
of course a generic one you can use to implement this for the platform, if
you're cool with my way of implementing this.

If you update the plugin.properties file, you can add a platform feed to the
default list of feeds that are preloaded when the view first starts up.

Additionally, if you want to provide different content in the feed (like XHTML
content in the <summary/> field), we'll have to revise the schema to include
support for that, as I've just used a simplified version of the spec to get
this going, so <summary/> only takes unformatted text.

Comment 6 Nick Boldt 2005-11-14 20:11:30 EST 
The intent here is to support different test result statuses, combining the
following platform/result codes:

-results {platform1},{status1},{platform2},{status2},... <recommended, but optional>

where {platform} is one of: ALL, win32, linux-gtk, linux-gtk-x86_64,
linux-gtk-ppc, linux-motif, solaris-gtk, solaris-motif, aix-motif, hpux-motix,
macosx-carbon, etc.
and where {status} is one of: PASS, FAIL, UNKNOWN, PENDING, or some other
descriptive code.

I still need to implement the cron-based feed listener, but this will allow you
to create/update feeds and view them in Eclipse.

To resolve the issue of mirror lag, we've moved the feed file from
www.eclipse.org to download.eclipse.org, so if you're waiting for a file to
appear on a given mirror (such as fullmoon.torolab), you can watch THAT copy of
the feed. As soon as the feed file is updated, the build will be too (since the
promote flow is to upload the build first, then the RSS update).

Comment 17 Kim Moir 2006-09-20 14:28:52 EDT 
The RSS feeds are working now for I-Builds. I disabled them for N-Builds since you said this wasn't useful.

http://download.eclipse.org/eclipse/downloads/builds-eclipse.xml

Issues I have noticed

-An NPE occurs if the RSS file exists, but is blank.  
-The RSS file specifies SIGNED jars when jarSigningStatus=
in the properties file.
-Randomly one of our performance tests show PENDING although I specify UNKNOWN in the properties file

Also, there is this link in the RSS feed for http://www.eclipse.org/2006/BuildFeed, I'm not sure what it's used for and it gives a 404.

Our test results are not reflected in the RSS feeds for our builds. The reasons are 

1.  We need to refactor our builds scripts to display results in a standardized xml format.
2.  We need a way to define what is a "good build". It is very rare for us to have a build where all the JUnit tests pass.  We often have test failures that are 
-not impacting the usability of the build
-issues with the tests themselves
-network or test machine issues

Also, even though we may have a "good build" it may not be the final I-build of the week.  Is it useful for teams to spin their build based on this build.  I don't know.  During milestone week, we run five integration builds a day.  Is it useful for downstream teams to initiate builds for each these milestone builds.  Nope.  It will just initiate builds that nobody will use and consume cpu and bandwidth.  So I think there needs to be some discussion of what should initiate a downstream build so as not to consume unnecessary resources and cause churn among the teams.

You mean the namespace for the schema?

<xs:import namespace="http://www.eclipse.org/2006/BuildFeed" schemaLocation="build.xsd"/>

Ignore that, it's not supposed to resolve to a URL; convention (AFAIK) is to use the domain of the company that created/owns the spec, the year, and a name, similar to:

http://www.w3.org/2001/XMLSchema or http://www.w3.org/2005/Atom ... though admittedly those DO resolve to actual webpages. ;-) 

Denis: if you wanted to clean up those 404s, you could set that URL to redirect to http://wiki.eclipse.org/index.php/Eclipse_Build_Available_RSS_Schema.

I'm thinking of adding these to the <build/> node as simple attributes.
