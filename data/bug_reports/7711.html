Description Duong Nguyen 2002-01-16 10:12:56 EST 
When the pathname of the jar file specified in the plugin.xml file has Japanese 
characters, the Eclipse class loader complains it cannot find the classes in 
the jar file.

This seems to be a problem in the either its XML parser or the path handling of 
its class loader.

To reproduce:

1. Install the workbench to a directory named with double-byte Japanese 
characters with no spaces.
2. Open the workbench.

When opening the workbench a java.lang.NoClassDefFoundError occurs when opening 
the plugin jar file. The workbench displays the message “An error has occurred 
when creating this view”.

Comment 2 John Arthorne 2002-02-14 10:18:25 EST 
This is most likely because we use URLClassLoaders, and non-ASCII characters are 
not legal in URLs.  It's doubtful that we can fix this, since use of URLs is 
pervasive throughout Eclipse.  We should try to give some kind of better error 
reporting, but this might have to be done in eclipse.exe or equivalent.
Comment 3 John Arthorne 2002-03-04 14:41:35 EST 
Won't fix.  Use of URLs for locations is pervasive in Eclipse, and URLs don't 
support non-ASCII characters.
Comment 4 Steve Pequegnat 2002-03-12 10:29:29 EST 
This is a valid requirement for products that intend to offer support for non-
English operating systems. 

Identifying this as a limitation with respect to chosen technologies does not 
remove this as a product requirement.
Comment 5 DJ Houghton 2002-04-10 21:39:59 EDT 
The problem occurs in org.eclipse.core.launcher.Main.getBootPath(). We need to 
be able to set the encoding in the call to URLDecoder.decode(), otherwise it 
is using the Platform encoding and failing. Unfortuanatly this can only be 
done in 1.4.

Some possibilities:

1). Use reflection. This code is in startup so it is not a high traffic area 
and would only be called once.

2). Compile against 1.4. Be tricky and put a try/catch block around the method 
call and catch the NoSuchMethodException in which case the codepath would 
default to the 1.3 behaviour. Would have to think about how this would affect 
the RelEng process. (they compile against 1.3 I think? but we supply them the 
startup.jar manually?)
Comment 6 DJ Houghton 2002-04-11 13:25:11 EDT 
Another alternative might be to look at doing some utf-8 encoding ourselves.

Comment 25 John Arthorne 2002-06-13 15:52:00 EDT 
Attached is a file with a suggested fix to the "decode" method on Main and 
InternalBootLoader for handling decoding of URL strings.  To make a very long 
story short, it turns out that 
class.getProtectionDomain().getCodeSource().getLocation() returns a different 
kind of URL depending on what kind of class loader loaded it.  When loaded with 
a URL classloader, it already correctly handles DBCS characters.  When loaded 
with the system boot loader, it returns a URL with %hex escape sequences, which 
then needs to be decoded.  The attached java class demonstrates this different 
behaviour, when run from a directory containing DBCS characters.  I have tested 
the fix on the following VM/NLs:

Japanese XP: IBM 1.3.1, Sun 1.3.1, Sun 1.4.0, Sun 1.4.1
German XP: IBM 1.3.1, Sun 1.3.1
