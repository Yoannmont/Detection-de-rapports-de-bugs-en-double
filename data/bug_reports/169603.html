Comment 16 Thomas Watson 2007-10-15 14:58:05 EDT 
A few comments on the (original) patch:

1) The BSN for the console bundle should be configurable (perhaps
osgi.console.bundle property)

2) You should move the call to startConsole to after the call to setStartLevel.
 This will allow the console bundle to be installed by update.configurator or
p2 simple.configurator.

3) You should use Bundle.start(Bundle.START_TRANSIENT) to activate the console
bundle so it is not persistently activate.

4) Should the startlevel of the console be set to 1?  I'm not sure this matters
since we want to wait until after setStartLevel is called to actually find and
start the console bundle.

5) You should use PackageAdmin.getBundles(String, String) instead of iterating
over all the installed bundles searching for the correct BSN.  You should use
the first bundle returned in the array.  This would ensure you start the
highest installed version of the console bundle.  What should we do for older
versions installed?  We should probably ensure only one version of the console
bundle is started.  Or maybe we don't care because the console should only be
started transiently.  On restart we would pick the highest version only and the
other bundles should not start.

Comment 39 Lazar Kirchev 2010-06-11 09:31:03 EDT 
Created attachment 171720 [details]
Patch for the console extraction

The console is completely removed from the framework. In the EquinoxLauncher and the EclipseStarter the creation of ConsoleManager is removed.

The symbolic name of the console bundle is given with the property osgi.console.bundle

EclipseStarter goes through the following steps when starting (after starting the initial bundles) if the framework is started with -console:

1. checks if the console bundle is installed
2. if installed but not started - starts it
3. if not installed - prints an error message and exits

I assumed that the console bundle is installed on a previous step - when installing and starting initial bundles. I assumed that, if necessary, it is included in the initial bundles list. 

The other option is to introduce a new property for the console bundle location, and if not installed during install of initial bundles, the EclipseStarter to install it from the given location. But this will be a duplication of the initial bundles functionality. 

I assume that either there will be an initial bundle which will install the console (e.g. through P2) and the console will not be included in the initial bundles list in the configuration, or no other bundle will install it and it will be explicitly included in the initial bundles list. 

Even if two console bundles are installed, the EclipseStarter will try to start only the one with BSN specified with the property osgi.console.bundle

I moved the EclipseCommandProvider and FrameworkCommandProvider in the console bundle.

The changes in the framework are made against the code from the OSGI_4_3_RI branch.
