To reproduce:

Comment 3 Syed Atif 2006-08-30 11:23:07 EDT 
Created attachment 49067 [details]
Patch for GEF

Note: the problem exists with all levels of cyclic connections. That is, application hangs when a connection A tries to connect itself to connection C. Where connection A is connected to connection C through connection B.

-----
This patch fixes the problem by avoiding having connections being attached to other connections they are already connected to (i.e. cyclic connections). It does this by adding all source and target connections to the exclusion list.

However, this patch is for GEF. The problem can be reproduced in all GMF editors.

Class affected:
1. org.eclipse.gef.tools.ConnectionEndpointTracker

Comment 7 Anthony Hunter 2006-09-14 12:58:06 EDT 
Created attachment 50170 [details]
Patch for the defect

I have updated Syed's patch as it adds duplicate connections to the exclusion list.
I also updated the code to make it obvious what we are trying to do.

This is a critical bug for our GMF clients.

We do need to have the recursion as we can have a connection to a connection and then that connection can be connected to the note.

When the user trys to drag his endpoint and drags over a connection that is connected to it, the application hangs in a infinite loop.

Syed's idea looks correct, you do not want to drag the endpoint to a connection that is already connected to you, so we add it to the exclusion list in the ConnectionEndpointTracker.

Comment 10 Randy Hudson 2006-09-14 14:03:21 EDT 
The infinite loop is actually related to laying out the connection, and is not related to whether such a representation makes any sense in the model (it probably doesn't). I could imagine someone creating a different connection router that did not have the infinite layout problem, and eventually stabilized once the interaction among the connections stopped.

Having said that, it seems highly unlikely that anyone would even need to support this type of cyclic dependency. This change sounds like a reasonable behavior for GEF to have at some time.

Having said that, any client can avoid this problem in their own code without any changes made to GEF.

Comment 13 Randy Hudson 2006-09-14 15:38:21 EDT 
One workaround would be to prevent a connection from being targeted if it is "reachable" from the connection being dragged. This search might be faster since you'd be going up the tree from the target, instead of down the tree and finding all the nodes in a branch. So, getTargetEditPart would return null from the connection, and the effect would be the same.

Another workaround would simply be to subclass GEF's tracker, and put the same fix in the subclass.

Comment 19 Randy Hudson 2006-09-14 17:39:17 EDT 
> Perhaps there is another way around this which I am not aware of?

Yes, the other way is to prevent the connection from saying it is a target, instead of having the tracker exclude it. So, the tracker sees the connection, says are you a possible target, and the connection says no because it has identified the cycle condition.

I assume your connections are themselves "nodes", so they probably have some king of graphical node role that allows them to be the target of a "reconnect source/target" request.
