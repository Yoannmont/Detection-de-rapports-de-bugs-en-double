Comment 9 Pascal Rapicault 2006-03-28 22:46:00 EST 
I missed the part where you do the changes while the target is running. Then I know where this is coming from. Here are the details:
When running in -dev mode, on shutdown the registry creates a checksum based on the timestamp of all the plugin.xml and fragment.xml files that where used to build the registry. This checksum is used on startup to decide whether or not the registry should be rebuilt (this allows for some optimization even in dev mode).
Because the changes are done before the target is shutdown, the changes made in the IDE will be accounted in the checksum even though the content of the registry has the old content. Therefore on restart the registry will find that nothing has changed because the timestamps of the files have not changed which will cause the changes to go unnoticed.

Comment 15 Oleg Besedin 2006-06-26 14:30:59 EDT 
Created attachment 45316 [details]
Patch

The one possible solution to the problem is to calculate the checksum on the startup rather than on shutdown. This way if plugin.xml changes after startup, the checksum won't match and registry cache will be tossed out.

Such approach, however, would have some performance implications as calculation of the time stamp involves getting information on all plugin.xml / fragment.xml files.

The proposed solution spawns a separate thread on startup that calculates the checksum. The value calculated is saved with the cache and used for the validation on the subsequent starts.

The proposed patch uses Thread class directly rather than Jobs mechanism as registry is expected to be available as a separate library.

Performance - there was no change:
- No change in the "real" startup time (between enetering startup.jar and passing control to the application);
- No change in the headless startup time as measured by JUnit performance test
- No change in the UI startup time as measured by JUnit performance test

Pascal, could you review the patch?
Comment 16 BJ Hargrave 2006-06-26 14:52:01 EDT 
This sounds like it is only a -dev mode issue (per comment 9), no? If so, then why the need for a thread? Just do it in-line and take the small hit since it is only for -dev mode.

This patch creates a thread at registry start and holds onto the thread object until the registry stops. This is a very expensive use of a thread object (to hold a computed long value). Even if it is decided to use a Thread to compute the value, the thread object should be discarded as quickly as possible to allow the GC to reclaim it. The result can be stuffed back into the registry object (or some other light weight object) as soon as the thread has computed it.

The point of the checksum is just to determine whether the cached information is still valid. If the checksum has changed, then it is best to flush the cache and reprocess the files. If the checksum has not changed, then the files are assumed to not have changed and there is no need to reprocess them.
Comment 19 Thomas Watson 2006-06-26 17:00:00 EDT 
The issue occurs when the framework *is* shutdown correctly and the cache is saved.  At shutdown a checksum is computed according to the timestamp information of each plugin.xml.  If the plugin.xml file was changed while the platform was running and after the plugin.xml was parsed then the saved checksum will use the updated timestamp of the plugin.xml but the old content is saved.

Oleg, could you not compute (add to) the current checksum as you parse new plugin.xml files instead of at the end when the framework is shutdown?

Comment 25 Oleg Besedin 2006-06-30 12:05:51 EDT 
Created attachment 45595 [details]
Expanded patch for the registry

This is the patch that takes approach suggested by Thomas and expands it a bit to make it more generic and to clean the timestamp story of the registry a little bit.

On the SPI side, it modifies the RegistryStartegy.getContributionsTimestamp() usage and adds RegistryStartegy.getContentsTimestamp(). The RegistryStartegy class as marked as "provisional". While I would prefer not to modify it, in this case cleanup of the timestamp story by explicitly separating the value returned into two methods (one for the cache contents, one for the current contributions) makes too much sense to be ignored :-).

The approach used in the RegistryStrategyOSGI will work for the current implementation of registry which does not cache dynamic contributions. Should we decide to change it in future, either cache contents would have to be invalidated on the first contribution removal, or (more likely) we would have to expand the contributions by giving them some unique IDs.

Please review the patch and provide comments. If we go with this approach, it would make sense to back port this into 3.2.1 after some testing on 3.3 stream is done.
Comment 26 Oleg Besedin 2006-07-07 13:47:04 EDT 
Created attachment 45940 [details]
Updated patch

The vastly improved patch :-).

Unlike the previous patch, this one doesn't store individual contributions timestamp, but relies on XOR properties to remove inividual timestamps (A^B^B = A).

Also, it is more backward compatibility friendly as it doesn't require changes to the RegistryStrategy (if registry was modified contents was modified, but not the timestamp, it will revert to the old behaviour). This might matter to people who implementied their own registries. 

The patch does not expose new timestamp functionality in the form of API or SPI as we are not certain as to how those changes are going to fit into a bigger strategy. If we decide to expose it later, it will be most likely in making RegistryTimestamp an SPI and and adding to the IExtensionRegistry an addContribution() method that would be able to take timestamps.
