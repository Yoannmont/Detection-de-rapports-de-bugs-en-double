Description Michael Valenta 2006-04-03 11:19:47 EDT 
See bug 132352 for the motivation for this request.

Basically, bug 132352 requires the ability for the QuickDiff provider to contribute a menu to the annotation bar content menu.
Comment 1 Dani Megert 2007-04-11 06:58:53 EDT 
Allowing quick diff providers to contribute menu items via code is not the right thing to do as it requires the plug-in to get loaded just to contribute the menu which we do not want to happen.

The 'org.eclipse.ui.popupMenus' extension point should be enhanced, so that viewerContributions can use the part input for the visibility testing. Currently this is restricted to the elements in the (structured) selection which doesn't work for parts that don't provide a structured selection (like editors who provide a text selection).

Once this is in place Team providers can contribute their ruler context menu with the same rules as their object contribution but on each concrete editor. Since this isn't that practical I filed bug 181916 to allow ruler and editor context contributes for a whole editor hierarchy.

NOTE: This is a 3.3 polish item. Not sure whether this might be easily doable with the new command and menu contributions story. Adding Paul for comments.
Comment 2 Michael Valenta 2007-04-13 14:55:29 EDT 
Paul and I had a conversation about this and came to the conclusion that we could probably do it with the new menu story but would need to write a custom propertyTester to determine the visibility of the action. The reason we would need a custom property tester is to extract the editor input from the editor part (i.e. the part is provided as an expression variable but the input is not). 

One possible problem I see with this is that, if the propertyTester is provided by CVS, the propertyTester may not be loaded when the menu is first shown (i.e. if the tester is contributed by CVS, it wouldn't be available until the CVS plug-in was loaded). We could force the tester to be loaded but this could lead to eager loading of the CVS plug-in. If we don't force the load, we could end up in a situation where the Show Annotate action doesn't appear on a file mapped to CVS because the menu was opened before the CVS plug-in was loaded.

Of course, if would be possible to provide the tester at a lower level (e.g. workbench) but it is not clear how generally useful such a tester would be. I'm also not sure if such a tester would be the proper solution in the log run. It may be more desirable to provide extensions to the expression language to support property extraction (the editor input is a property of the part).

Comment 6 Paul Webster 2008-10-21 13:48:56 EDT 
I was reading through bug 132352 comment #4 that seemed to imply that you could only target a specific editor and not all text editors ... but that's not the case.

Any one context menu can be registered with multiple IDs, and that will be honoured in both org.eclipse.ui.popupMenus/viewerContributions and org.eclipse.ui.menus/menuContributions

ex: Both the text editor and java editor register #OverviewRulerContext and so it can be used to contribute to both types:

    <menuContribution
          locationURI="popup:#OverviewRulerContext">
       <command
             commandId="org.eclipse.ui.help.aboutAction"
             style="push">
       </command>
    </menuContribution>

Text currently registers  #TextRulerContext and org.eclipse.ui.DefaultTextEditor.RulerContext for the ruler context menu, and the java editor registers  org.eclipse.jdt.ui.CompilationUnitEditor.RulerContext and #CompilationUnitRulerContext.

If AbstractTextEditor registered a common ruler ID, it would be inherited by all text based editors, for example (which might be the intent of Bug 181916 )

A menu contribution of the form popup:#OverviewRulerContext is targeted at a specific context menu, and those contributions could then use the visibleWhen core expression to determine its visibility.

This could either be done by the interested parties providing a property tester against the IEditorPart to extract and test its IEditorInput.  The property tester could then be used with the activeEditor variable.  The downside with this kind of property tester is that it might not be re-evaluated if the editor input changes but not the activeEditor.

Or we can add another variable to the source provider, activeEditorInput, which could then be used as a standard platform variable in our core expressions.  I don't see a problem with the platform providing editor input support.

PW
