Description Eugene Kuleshov 2006-02-20 19:54:04 EST 
Issue repository provider icons (that you can see when choosing repository type on creation) are not shown in repository view.
Comment 1 Mik Kersten 2006-02-21 10:59:41 EST 
We've got two things to consider here:
- a consistent icon that means repository, as CVS and Subclipse do a good job of
- a way of distinguishing the kind of repository

The things we have to work with are icons, overlays, and text.  I'm leaning towards either adding an overlay to the current repository icon, or having a second icon column with the branding icon (and probably getting rid of the "jira:" prefix).  This might take some experimentation to get the visual design right.
Comment 2 Eugene Kuleshov 2006-02-21 11:24:01 EST 
Please don't add second column. It will eat an extra screen estate if repository view will ever became a tree.
Also, the view itself already called "... Repositories", so "Repository" icon is kind of redundant.
I agree about "jira" prefix, though there was another request to allow a custom label for each repository (cvs and svn plugins have that at well).
Comment 3 Mik Kersten 2006-06-28 22:09:28 EDT 
I still prefer the text labels for the kind and all of the repositories having the same icon for consistency.  But we should consider this again in 0.7 when we have more connectors present.
Comment 4 Eugene Kuleshov 2006-07-03 09:51:35 EDT 
(In reply to comment #3)
> I still prefer the text labels for the kind and all of the repositories having
> the same icon for consistency.  But we should consider this again in 0.7 when
> we have more connectors present.

Mik, generally "kind" label is usually irrelevant. Most of the users are working with concrete repositories and they more interested in those repository names (not even urls) in relation to the projects repository used for.
Comment 5 Mik Kersten 2006-07-05 02:06:58 EDT 
I believe that most programmers work with a single repository, which is in part why I have preferred using the blue server icon instead of the branding icon.  That server icon has a meaning and shows up in multiple places (e.g. as a decorator).  I'm not sure if the kind label is necessary, but we need some way of distinguishing repository kinds.  I couldn't make decorators (only 8x7px) out of the connector icons that were meaningful, so the kind labels stuck.  

Are you suggesting that we allow users to name repositories?  That could work well, particularly for those with long URLs or IP addresses.
Comment 6 Eugene Kuleshov 2006-07-05 02:13:17 EDT 
(In reply to comment #5)
> Are you suggesting that we allow users to name repositories?  That could work
> well, particularly for those with long URLs or IP addresses.

Such label makes total sense. Both CVS and Subclipse team providers has it and it is very handy when having few dozens of VCS servers.

Please also change order in representation to: label - url - kind
Comment 7 Mik Kersten 2006-09-08 17:35:11 EDT 
Fixed as part of bug 128737.  Task Repositories now get decorated with overlay icons specified by connectors.

Comment 9 Mik Kersten 2006-09-08 18:26:45 EDT 
They look good to me, could you attach a screenshot?  In general decorators don't cause this kind of problem so it's probably an anti-aliasing/background issue.

Comment 12 Eugene Kuleshov 2006-09-13 23:36:55 EDT

overlays look

This look better, but still could be probably improved. Eg. you can make these overlays at least twice as big, overlap the repository icon and add some space/gap on the right side.

I also think that Task Repositories view should use tree widged and in the future allow repository providers to contribute their own model (e.g. using navigator framework).
Comment 13 Mik Kersten 2006-09-15 11:52:35 EDT 
The overlays are currently the maximum overlay size (7x8px).  It would be nice to give them some space on the right side because it does look a bit weird when selected, but that would be reducing them by 1px, which would cause them to lose shape even further.  While it took some finessing to get them to this point, we should be able to improve them further at some point.

Yes, I agree that the Task Repositories view should use the navigator framework allow providers to show children under the repository node, and have been making changes with that in mind.  While I haven't yet seen the common navigator used for a tree with columns, I think that should be possible.  We'll need a driver for that, not sure yet if we'll have time to do it before 1.0.
