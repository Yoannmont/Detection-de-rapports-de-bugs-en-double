Comment 2 Thomas Watson 2005-04-14 15:54:35 EDT 
I suspect this is because of the fix for bug 50133.  We parse all code jar 
manifests for package information to get the proper information for 
Class#getPackage method.  Now most bundles in Eclipse are being jar'ed 
with '.' on the Bundle-ClassPath.  This means the bundle jar is now a code jar 
with a manifest file that is parsed to get the package information at class 
load time.
Comment 3 Jeff McAffer 2005-04-14 16:40:16 EDT 
when is the manifest parsed?  Seems like there are 50 being parsed.  I would 
not expect 50 classloaders to be created.

Sidenote:  Can we cache a bit that says whether or not the manifest has any 
package info?  Most do not so it seems a waste to parse.  Presumably we know at 
bundle install time if there is any package info in the manifest.  We can mark 
up the BundleData to say.  This way we can avoid reading the manifests.

Comment 11 Thomas Watson 2005-04-27 11:11:10 EDT 
Re comment 10 Question 2:

We can do better for jar'ed bundles.  At install time we can perform an extra 
check of the manifest file to see if the bundle manifest file has extra 
package information in it.  If it does not then we can add some information to 
the bundledata to indicate that the bundle manifest file should not be parsed 
for package information at classload time.

For non-jar'ed bundles we can do some extra work to see if the jars on the 
bundle classpath have manifests with extra package information,  If they do 
not then we can add some information for each of the classpath entries to 
indicate that the manifests should not be parsed.
