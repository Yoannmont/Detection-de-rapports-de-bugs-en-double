Description Douglas Pollock 2006-03-15 11:16:23 EST 
A common pattern in Eclipse is to create extension points which contain a class as one of the attributes (e.g., editors, views, actions, etc.).  These classes are then proxied, and loading is deferred for as long as possible to avoid unnecessarily activating bundles.

However, it is safe to load the class if the bundle is already active.  See org.eclipse.ui.internal.PluginAction#isOkToCreateDelegate (in org.eclipse.ui.workbench) for an example.

Currently, the best block of code I can come up with for this is:

  ServiceReference sr =
      bundlecontext.getServiceReference(PackageAdmin.class.getName())
  PackageAdmin admin = (PackageAdmin) bundlecontext.getService(sr);
  int index = className.lastIndexOf('.');
  String packageName = className.substring(0, index);
  ExportedPackage package = admin.getExportedPackage(packageName);
  if ((package != null) 
      && (package.getExportedBundle().getState() == Bundle.ACTIVE)) {
    return true;
  }

A naive approach is simply to do the following:

  String bundleId = configElement.getNamespace();
  Bundle bundle = Platform.getBundle(bundleId);
  return bundle != null && (bundle.getState() 
    & (Bundle.STARTING | Bundle.ACTIVE | Bundle.STOPPING)) != 0;

However, this leads to Bug 131746.  (Basically, the configElement may not be in the same bundle as the class that is to be loaded.)

We have a utility class in org.eclipse.ui.workbench called BundleUtility.  It would be nice to have something similar in equinox itself.  (Perhaps there is one already?)
Comment 1 Thomas Watson 2006-03-15 11:54:56 EST 
The problem with the PackageAdmin approach is when packages are split across multiple bundles.  In that case you may need to get all the ExportedPackage objects for the package from PackageAdmin#getExportedPackages(String name).  But the problem with that is you do not know which of the bundles from the split package must be ACTIVE for the extension to work.  Also there is no requirement that the contributing bundle export the packages for its extensions.

I think there is some help from the new 3.2 method IConfigurationElement#getContributor().  The IContributor should return you the actual symbolic name of the contributing bundle.  With that you should be able to get the Bundle Object to do your checks.
Comment 2 Douglas Pollock 2006-03-15 13:32:23 EST 
The problem is that the class mentioned in the extension does not need to be a class in the same bundle.  Take the example given in Bug 131884 for example.

  Bundle A is large and has a large dependency chain.
  Bundle B is small and has a small dependency chain.
  Bundle A depends on Bundle B.
  Bundle A has an extension which references a class in Bundle B.

When trying to decide whether its safe to load the executable extension, we need to know whether B is active, but we have a configuration element from A.

Comment 13 Thomas Watson 2007-12-06 15:52:53 EST 
Created attachment 84668 [details]
patch

This patch changes ExecutableExtensionCheck quite a bit from the original patch.  I fixed the following issues.

1) The OSGiUtils class must work without the optional package org.eclipse.osgi.service.resolver.
2) If PlatformAdmin is not available then the operation should *only* use PackageAdmin.
3) If PlatformAdmin is available then the operation should *only* use PlatformAdmin.  PackageAdmin is quite expensive for this type of operation.  Especially with large sets of bundles and exports
4) If a supplier cannot be found for the package then we should assume that the class is coming from the contributing bundle.

Some more tests are needed to test the following:
1) A private class (not exported by anyone)
2) Package imported with Import-Package
3) Package accessed by requiring a bundle which re-exports another bundle who exports the package.

Comment 15 Thomas Watson 2007-12-07 09:13:14 EST 
There is another case which is not covered, but I think it is a corner case which will throw a wrench into the algorithm.  Here is an attempt at a diagram:

        |----------------------|
        |                      |
        | private: foo         |
        |            Bar.class | Bundle A
        |                      |
        |                      |
        |----------------------|
                 |
                 |
                 |Require-Bundle: B
                 |
                 V
        |----------------------|
        |                      |
        | export: foo          |
        |           Baz.class  | Bundle B
        |                      |
        |                      |
        |----------------------|

In this case Bundle A has an extension which uses the foo.Bar class.  We will find that Bundle A is wired to foo from Bundle B *and* Bundle A does not export the package foo.  Since Require-Bundle is used the package becomes split across both Bundle A and B.  But in the current code we will only determine that Bundle B is ACTIVE because we have no way to know about Bundle A's private foo package.  I argue that this is a corner case and not really a best practice anyway.  If we checked Bundle A for ACTIVE in this case it would make the scenario from comment 2 impossible to implement.

Changed names and Javadoc as requested, the method now is RegistryUtilsOSGi#isExecutableExtensionProviderActive(IConfigurationElement configElement, String propertyName).

For the comment #26 : Yes, technical challanges is a good way to describe it :-). Thing is there are several factors that might influence if bundle actually gets activated. Eclipse-LazyStart and it's deprecated version is one factor. Next, package can be excluded from the lazy activtaion policy. You might get a few cases there activation won't happen, but it is really important? The only "drawback" would be that creation of executable extension would happend faster then expected?

Comment 30 Thomas Watson 2007-12-12 09:50:27 EST 
First of all, Rafael, it is very good to hear from you!!!

It has occurred to me that this problem may have no perfect solution.  Both Rafael's approach and the approach in this bug report cannot do this correctly and guarantee that the use of an executable extension will not cause additional bundles to be activated.

Here is an example:

        |----------------------|
        |                      |
        | extension            |
        |   class = foo1.Bar   | Bundle A
        |                      |
        |                      |
        |----------------------|
                 |
                 |
                 |Require-Bundle: B
                 |
                 V
        |----------------------|
        |                      |
        | export: foo1         |
        |           Bar.class  | Bundle B
        | Bar extends foo2.Baz |
        |                      |
        |----------------------|
                 |
                 |
                 |Import-Package: foo2
                 |
                 V
        |----------------------|
        |                      |
        | export: foo2         |
        |           Baz.class  | Bundle C
        |  Baz uses foo3.Zoo   |
        |                      |
        |----------------------|
                 |
                 |
                 |Import-Package: foo3
                 |
                 V
        |----------------------|
        |                      |
        | export: foo3         |
        |           Zoo.class  | Bundle D
        |                      |
        |                      |
        |----------------------|


In this case, in order to use the executable extension from A bundles B, C and D must be activated.  In the solution outlined by this bug we would only check to see that bundle B is ACTIVE.  But loading class foo1.Bar will will cause foo2.Baz to load, triggering bundle C activation.  Then when the executable extension is actually used the foo3.Zoo class will be loaded and bundle D will be activated.

Rafael's approach would solve part of the problem depending on where we entered the "activation causes error" mode.  If we inserted the mode in the process to create the Executable Extension then a check will be done to ensure all bundles required to load class foo1.Bar are ACTIVE; otherwise throw class loading error.  But this may not include the foo3.Zoo class from Bundle D.  Not until foo2.Baz actually uses the foo3.Zoo class will Bundle D be activated.  In order to solve the last issue we would have to proxy the executable extension or expose some API which allows clients to run code which uses the executable extension within the activation error mode.

Is there any indication of what the common case is?  Do we think the solution in this bug will solve a majority of the usecases?

Comment 32 Oleg Besedin 2007-12-19 11:49:49 EST 
From what I see there are three possibilities here:

1. Add extra pieces to answer question "is bundle going to be activated"
   - from what I know there are no APIs on OSGi side, most things could be extracted from manifest files but not readily available
   - it will become very Equinox-specific
   - performance: at some point it will become faster just to activate the bundle when to figure out if it is going to be activated :-)

2. Go ahead with the functionality as is (answers the question "is bundle active")

3. Don't put any code in at all 

As you can see from above, I think that only (2) and (3) are viable options. If the proposed functionality (answering "is bundle active") is interesting, we should put it in. If it is not useful, we probably should just close this. 

So it becomes a question: is the functionality "is the executable extension bundle active" interesting?
