Description michael.t 2007-10-09 06:13:31 EDT 
Consider the following java source : 

public class Test
{
	public static void main(String[] args)
	{
		System.out.println("Hello Eclipse");
	}
}

If you select 'System' then click on menu Navigate / Open External Javadoc it will lunch Safari on the correct page : 
"file:///System/Library/Frameworks/JavaVM.framework/Versions/1.5.0/Resources/Documentation/Reference/doc/api/java/lang/System.html"

If you select 'println' and do the same action, it will try to open : 
"file:///System/Library/Frameworks/JavaVM.framework/Versions/1.5.0/Resources/Documentation/Reference/doc/api/java/io/PrintStream.html%23println(java.lang.String)"

The problem here is the "%23", if you replace it with a '#' it works.

Comment 3 Massimo 2008-02-05 03:13:02 EST 
> This looks like a Safari issue

It is not.

Comment 4 Massimo 2008-02-05 03:20:25 EST 
> The same problem happens with Camino and Firefox (under OS X, of course).

More precisely...

Apparently the fragment identifier (#) in the external javadoc url is escaped even if it should not.

http://en.wikipedia.org/wiki/Percent-encoding

Here is a simple way to reproduce the bug, download and install the last Europa JDT. Create a Java project and a simple "hello wolrd" main, just Shift=F2 on the println part of the output statement. What you get is the (system default) browser opening 

http://java.sun.com/j2se/1.5.0/docs/api/java/io/PrintStream.html%23println(java.lang.String)

and the Sun website return a "Page Not Found" error (exactly as it does if you cut and paste such URL in one of the three OS X browsers cited above). If you manually substitute the %23 in the previous url with # everithing works as it should.

Hope this helps...

Comment 8 Chris Goldthorpe 2008-03-05 01:20:44 EST 
Here's what I've found out so far. The call stack at the point where the URL is opened looks something like this, the Browser class at the top of the call stack will be different depending on the Browser used. The URL in *Browser.openURL() contains a '#' character so the help system is not responsible for encoding the '#'. On the Mac it seems that the # is encoded in org.eclipse.swt.program.execute(). I will follow up with the SWT team.
