Description Mark-Oliver Reiser 2007-06-18 12:00:06 EDT 
The JFace Viewers for Grid should provide support for the height of GridItems.

Things to be considered:
- make row/item heights accessible to client code in JFace style
  (e.g. something like IColorProvider for item heights ??)
- support for resizeable state and min/max height per GridItem ?
- support for automatically updated preferred heights
  (i.e. the Viewer sizes GridItems to their preferred height, e.g.
  as returned by GridCellRenderer#computeSize(), see GridItem#pack() )

Details to be discussed.
Comment 1 Chris Gross 2007-06-19 13:29:46 EDT 
So here's what I'm thinking:

Update GridColumnLabelProvider:
- add getResizeable
- add getMinimumHeight
- add getMaximumHeight
If any of these return modified values then I will automatically add a SWT.Resize listener in update() and do the necessary work.  

Update Grid*Viewer and add get/setAutoPreferredHeight(boolean).  If this is true then Grid*Viewer would automatically call GridItem#pack() just after the call to columnViewer#refresh in doUpdateItem.  Alternatively I could do this in GridColumnLabelProvider#update as well.
Comment 2 Mark-Oliver Reiser 2007-06-20 02:44:07 EDT 
Sounds great.  Just a few more occasions on which a GridItem#pack() could be reasonable:
- when a column is being resized interactively or through GridItem#setWidth()
- when word-wrap is turned on/off on a grid column

In addition, we probably should have a mechanism to detect double-clicks on row resizers to allow client code to implement a more sophisticated custom row height management.  For example, for a behaviour like that of Excel this would be necessary, I think.

Comment 6 Mark-Oliver Reiser 2007-06-22 09:48:33 EDT 
Created attachment 72170 [details]
auto preferred height for GridTreeViewer

Here is a small patch for auto preferred height (for GridTREEViewer only to not unnecessarily blow up the patch).

Two limitations/problems:

(1) While dragging the column resizer, all(!!) GridItems are packed each time the mouse is moved (because there is no way to know when column-resize is finished ; while dragging, it would be enough to only pack the rows currently visible on screen, i.e. from Grid#getTopIndex() to Grid#getBottomIndex()).

(2) Auto-pack() on column resize works only if the column was created with GridViewerColumn (because I don't know how to detect GridColumn createn; see my previous comment).

Comment 12 Chris Gross 2007-06-26 16:06:05 EDT 
Two more questions:  Why the refresh()?  Doesn't seem like we should do that.  Second, do we need the "if (columnResizeListener == null)" in hookColumnResizeListener?
Comment 13 Mark-Oliver Reiser 2007-06-27 03:08:03 EDT 
(In reply to comment #12)
> Why the refresh()?

When setAutoPreferredHeight(true) is called after an input has already been set (as in the snippet), then the existing items need to be pack()ed for the first time in #setAutoPreferredHeight(boolean).  However, it would be enough to do an update() of all existing items instead of the refresh().

Out of curiosity:  What would be the drawback of calling refresh() ?  I thought handling of structural changes does no harm here, even if it may be unnecessary ...


> Second, do we need the "if (columnResizeListener == null)" in
> hookColumnResizeListener?

No, but we need the "if (columnResizeListener !=null)" in unhookColumnResizeListener().  And for consistency between the contracts of hook..() and unhook..() -- i.e. "It is safe to call the method several times in a row" -- I think it would make sense to keep that; it does no harm.  But of course, with the present usage of hook..() you could also remove it.
Comment 14 Chris Gross 2007-06-27 13:18:49 EDT 
> Out of curiosity:  What would be the drawback of calling refresh() ?  I thought
> handling of structural changes does no harm here, even if it may be unnecessary
The refresh will cause the entire viewer to be reloaded.  This is a relatively expensive operation.  The refresh could also cause the user to loose certain state (like what items are expanded and collapsed in a tree, whats selected, etc).  I would avoid calling the refresh for them automatically.  Why don't we just document the method that if they set the auto height on after data has been populated that they will need to call refresh themselves?
Comment 15 Mark-Oliver Reiser 2007-06-28 02:20:03 EDT 
> Why don't we
> just document the method that if they set the auto height on after data has
> been populated that they will need to call refresh themselves?

Well, my feeling was that it would be a bit awkward to provide an auto-update feature and tell the user that - after turining it on - the first thing that needs to be done is to initiate a manual update  :-)
But maybe that is not really a problem ...

But what about calling update() in #setAutoPreferredHeight() for all existing items ?

Anyway, just choose the solution you deem most appropriate ...
