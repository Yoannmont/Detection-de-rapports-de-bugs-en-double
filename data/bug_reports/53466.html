Comment 1 John Arthorne 2004-03-02 11:12:41 EST 
Two threads (ModalContext and Worker-14 ) are waiting for the internal workspace
lock, but nobody appears to own it...
Comment 2 John Arthorne 2004-03-03 17:47:32 EST 
Michael and I think we have found how this can happen:

1) UI thread enters ILock.acquire
2) Lock is owned by other thread, so a Semaphore is added to the queue
3) While waiting, the UI is interrupted and asked to service a syncExec
4) The syncExec attempts to acquire the same lock the UI thread is waiting for
5) The lock is busy, so another semaphore is created
6) The lock enqueue operation does NOT add the semaphore, and instead returns
the semaphore created in step 2) (enqueue avoids enqueuing equal semaphores, and
semaphore equality is based on the thread that created the semaphore).
7) This nested syncExec eventually gets the lock, thus removing the semaphore
from the queue
8) The syncExec completes, and we return to the UI thread that is waiting for
the lock. It tries to acquire the semaphore, but since this semaphore is no
longer in the queue, it will NEVER be notified).
9) The UI thread is now dead, and complete system deadlock quickly follows.
Comment 3 John Arthorne 2004-03-03 17:52:24 EST 
Note that this wouldn't happen in a trivial case, since the UI thread only
services syncExecs from threads that own locks the UI thread is waiting for.
With only one lock involved, the syncExec couldn't be blocked on the same lock
the UI is waiting for (step 4).  In short, I believe this deadlock can only
occur when the UI thread is waiting for two or more locks.  In Michael's test
suite case there were two OrderedLocks, and a Job.join() involved (Job.join()
has locking semantics), as well as two threads trying to syncExec.  Ouch.

Comment 8 John Arthorne 2004-03-04 11:36:01 EST 
I should also describe the fix for future reference.

The fix is in step 8) from my steps in comment #2.  When the UI thread regains
control after servicing a syncExec, the semaphore it created might have been
discarded.  The fix is to make another attempt for the lock by calling
createSemaphore again. There are three possible states at this point:

A) The "normal" use case: the lock is not available, and there have been no
nested acquires. createSemaphore will return the existing semaphore without
adding a new one.  We wait on the existing semaphore as usual.
B) The lock is not available, but the semaphore has been discarded from the
queue due to a nested acquire on the same lock. createSemaphore will add a new
semaphore to the queue and we wait on that one.
C) The lock is available (no owner). createSemaphore will return null and we now
own the lock.  No wait required.

The first patch only addressed A and B (explaining why Michael's test run after
applying it was successful).  The second patch addresses all three cases.
