Comment 4 Wassim Melhem 2004-03-24 05:57:22 EST 
The problem here is that the workspace seems to contain a plug-in with a 
manifest.mf that lacks the Bundle-SymbolicName OSGi header.

In this new piece of code for resolving plug-ins, we don't check for null if 
this header is not present, and hence the NPE.  We do a null check now.

However, as per last week's arch call, with the ongoing changes in the runtime 
and Update, we cannot guarantee that you can launch a new Eclipse on a 
workspace created that contains plug-ins from an older 3.0 Eclipse and expect 
things to work.  Of course, you shouldn't see things blow up like you did and 
we'll make sure this won't happen, but compilation errors and/or other 
symptoms might occur, depending on your workspace setup and what plug-ins you 
have enabled on the Target Platform page.

This really does/should not undermine the robustness of classpath containers, 
as it has nothing do with them.  It has to do with the fact that PDE can 
possibly choke on plug-ins with outdated manifest.mf, and the classpath 
container can fail to initialize as a result.  This is all transient because 
the runtime will stabilize soon enough.  In the meantime, PDE is taking the 
hit for it.
