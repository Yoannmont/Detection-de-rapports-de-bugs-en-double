Description Carolyn MacLeod 2006-12-13 16:55:42 EST 
Build id: I20061212-0010

The fix for bug 36189 now makes bringing up a comparison browser look much slower than before for the typical case.

The user's reaction is "Initializing...? White background? Huh? What is happening in the background? Why is this slower than before?"

Perhaps async initialization was a bit overkill to solve bug 36189. Maybe all that was needed was to show a "progress requester" that would allow cancelling an accidentally-launched large jar compare. This is all that bug 36189 is asking for.
Comment 1 Steve Northover 2006-12-13 17:24:28 EST 
I agree with Carolyn.  It makes Eclipse look slower.
Comment 2 Michael Valenta 2006-12-13 20:05:52 EST 
What you are saying is that it makes compare look slower in the case where the repository is practically next door (or, as in the case with the model-based Sync, the contents are cached as part of the Synchronize operation). Not that that is not a common case but I would like to get an opinion on how it looks for the case where the repository has a significant lag. Dani, can you (or anyone else in Zürich) offer an opinion here?

P.S. We used to show progress with cancellation but in the case where the server round trip had an appreciable cost, the user was blocked for a significant amount of time unnecessarily. That's not to say that the current state is good but it is better for the worst case.

P.P.S. I tried to delay opening the editor until after the contents were fetched which would have been equivalent to what happened before with the option to run in the background. However, this lead to other problems. Opening the editor immediately was the easiest solution (i.e it didn't require additional support from the workbench or additional Compare API).
Comment 3 Michael Valenta 2006-12-13 20:14:52 EST 
As for possible solutions, I tried it and it was fast enough that, if I blinked, I would miss the "Initializing" message. I wonder if one solution would be to delay showing "Initializing" message so it only gets shown if the operation takes longer than a second or two. Another possible solution would be to "optimize" the case that Steve and Carolyn are seeing to not run in the background (i.e. in their case, the contents are cached as part of the Synchronize operation so initializing the editor is not going to take much time. I would only want to do the later solution if the former one wasn't acceptable for some reason. I've copied Kevin in case he has any insight on this (since we had discussed this topic a few weeks ago).
Comment 4 Dani Megert 2006-12-14 03:40:39 EST 
There are two different topics here: load in background vs. locking the UI and secondly, how this is presented in the UI. Currently the load in background is limited and hence I don't profit from it i.e. I still wait until the result is ready and don't switch to another task in the meanwhile (this is tracked by bug 165796).

Assuming loading is done in the background the current feedback was needed for slow connections (see bug 165431) and feels/looks pretty good now in that scenario. For example the "Initialize..." appears for up to 10 seconds. However, when working off a local CVS server it's pretty annoying (as described in comment 0). Couldn't we do something similar to the progress story i.e. start with no special feedback and if the result isn't ready go into the other (current) mode. The ideas from comment 3 go into that direction.
Comment 5 Michael Valenta 2006-12-18 10:45:34 EST 
I've delayed showing the "Initializing" message for 1 second. Hopefully that will improve the situation. Let me know if you feel this is still aproblem.
