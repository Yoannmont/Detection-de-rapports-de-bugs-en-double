Comment 3 Michael Valenta 2007-01-31 13:22:22 EST 
I traced this and determined what the problem is. When a compare editor is opened, it uses a default savable that knows how to save the compare editor the usual way. After a background job is run, the embedded merge viewer for the editor is determined and, in some cases, this causes the saveable to change to one that is file buffer aware. Atthat point, we need to signal that the old saveable should be removed and the new one used. The problem is that, if the background job is fast, the workbench has never asked for a saveable so the remove causes an exception to be logged. 

I think that the solution for this needs to come from the workbench. Possible solutions I can see are to allow an explicit add when the editor is opened or to have a way to query whether the saveable has been added (or we could not log an exception when a remove is performed on a model that is not there).

Comment 5 Michael Valenta 2007-02-01 09:27:50 EST 
After some more investigation, I though I might be able to fix this in Compare. However, I came across the following problem. The CompareEditor wraps a CompareEditorInput and basically passes through any saveables from the input. The problem is that the isDirty of CompareEditor needs to query the saveables of the input. This makes it impossible for the input to know whether it's saveables  have already been obtained by the workbench or not. I've appended the trace to this comment. 

Of course, I could work around this by creating API at the Compare level but I would prefer not to do that. Possible solutions I see are:

1) Workbench never calls isDirty without first obtaining the active saveables. This would help in this situation but could still cause problems if isDirty was called by another client.

2) Allow multiple removes of a saveable. I suspect this has potential problems since the workbench is reference counting.

3) Keep a part to saveable mapping and have an event that clears the saveables for that part regardless of whether it has a saveable or not.

4) Allow a part to query what saveables it has so they can be removed. I think this s similar to 3) and 3 is probably easier on client.
