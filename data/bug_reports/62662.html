Description John Arthorne 2004-05-18 11:21:50 EDT 
I20040517-1219

1) Checkout org.eclipse.swt and whatever native fragment for the given platform
2) Open StyledText.java
3) Ctrl+A (select all)
4) Ctrl+/ (toggle comment)

-> OutOfMemoryError

If you allocate a very big Java heap, it might only occur after step 5)

5) Ctrl+/ (toggle comment again)

According to Windows taskmanager, memory usage before step 4) is 145MB. At the
time of OutOfMemoryError Windows is reporting 570MB used. It seems to require
over 300MB of RAM to comment this file.

Reproducible on Windows with Sidecar and Sun 1.4.1, and on MacOSX with latest
Sun VM.

Comment 23 Mike Wilson 2004-06-16 16:04:02 EDT 
Re: "which results in a real OOME" in comment #21.

OOM is a *fatal* state for Eclipse. Once this has happened, in any thread, Eclipse's state 
may be inconsistent and the only reasonable thing we can do is exit. We should not be 
writing code that unboundedly consumes memory under any circumstances. See bug 
61942 for related info.
Comment 24 Mike Wilson 2004-06-16 16:06:02 EDT 
I did a number of simple tests on SoftReferences and they *appeared* to have 
reasonable behavior. However, when I built a plugin that used them to attempt detect low 
memory conditions, I found that they were often garbage collected too optimistically (at 
least on the Mac VM).

Comment 26 Mike Wilson 2004-06-17 09:23:06 EDT 
You should not be attempting to catch OOM exceptions. These should be perculated to the main 
Eclipse event loop, where they are caught and cause Eclipse to exit. 

In general, when Java VMs run low (or out of) memory, they behave poorly. Separate from that, because 
Eclipse is highly multi-threaded now, it's almost a certainty that if you get an OOM, other threads will 
too.

Your code needs to be written in away that it uses a predictable, bounded amount of memory.

Comment 31 Kai-Uwe Maetzel 2004-06-22 05:02:50 EDT 
Thus the situation is the following:
- there is no way to find out how much memory is available for
  object allocations
- the only way to find out whether an object allocation is possible
  is to allocate the object
- bounding the quick diff algorithm to a fixed number of diff objects
  is not useful
- because of the multi-threaded context there is no complete solution

The best possible approximation to a solution is the following
implementation of LineRangeFactory:
- the factory allocates safety memory
- the factory uses the same up-front check as it already does
- each allocation of a LineRangeDifference object is surrounded by
  a try/catch block catching OOME. In the catch block, the safety
  memory is freed, garbage collection is initiated and a low memory
  handling is activated.

As said, this does not ensure that we are not running out of memory but 
increases the chances to work OK.
