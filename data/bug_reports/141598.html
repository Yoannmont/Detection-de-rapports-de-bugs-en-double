j3daudio.jar, j3dcore.jar, j3dutils.jar, and vecmath.jar.  Once J3D is

C:\j2sdk1.4.2_06\jre\lib\ext\j3dutils.jar;
C:\j2sdk1.4.2_06\jre\lib\ext\j3daudio.jar;
C:\j2sdk1.4.2_06\jre\lib\ext\vecmath.jar

in either the Eclipse launcher as VM arguments or as VM arguments in
the Product Configuration tab.

Doing this, of course, makes the RCP application non-portable, among

something that Eclipse is doing (or not doing) that makes it
necessary.

Comment 6 Thomas Watson 2006-05-13 19:43:20 EDT 
See the option osgi.parentClassloader from Eclipse help.  This option defaults to "boot".  Anything placed in the <j2sdk>/jre/lib/ext directory of the JRE is not accessible from the boot classloader, instead it is available on the extension classloader.  You should set the value osgi.parentClassloader to "ext" if you want stuff from the extension classloader to be available to bundles installed in eclipse.  You can do that in the config.ini file.

osgi.parentClassloader=ext

That is not the ideal solution though because your bundles are making an assuption that J3D is installed on the JRE and they are not declaring their dependancy on J3D.  I would recommend you try placing the J3D jars into your own separate bundle and export the packages included in J3D.  Then your other bundles can use Require-Bundle or Import-Package to access the packages from J3D.

Closing as worksforme, please reopen if either of these solutions do not work for you.  Thanks.

Comment 8 Thomas Watson 2006-05-14 22:40:26 EDT 
I say this worksforme because if you want to expose extensions to bundles then you use the osgi.parentClassloader option.  That works and that is what you stated that you needed.

>> Yes, I do want that. There is still the question of how to do that
>> when this application is used as a plug-in, rather than stand-alone.

You add the option to the config.ini of your eclipse installation.  Does that not work for you?  The article you mention is documentation for a particular product which is built on Eclipse.  This particular product is shipped with this option set osgi.parentClassloader=ext

Eclipse defaults to boot because we want as much control as possible over what bundles have access to.  If we default to ext then there is no way to install a different version of that library in Eclipse.  Eclipse is all about modularity.  The way software is installed in eclipse is by installing new bundles.  That is why the best practice for your situation is to create a library bundle which other bundles can depend on.  This gives you the best modularity because you can accurately state the dependancies on J3D.  Otherwise you will allow users to install your bundle on a system which does not have J3D installed and get classloading errors.  Bundles can include any native libraries you need so that should not be an issue (that is what SWT does for eclipse)

Comment 10 Pascal Rapicault 2006-05-15 18:19:43 EDT 
Kenneth, while selfhosting you need to specify the "osgi.parentClassloader=ext" as a VM arg (in the VM arg of the launch dialog do: -Dosgi.parentClassloader=ext

In order to get the generated config.ini, while the application is running go to the debug view, right click on the second item in the tree representing your proces and do properties, this will open a dialog that will show the value of the -configuration param. Then navigate to the file before the end of the process.

Re: about shipping other people things as jars.
Even though I understand that it is not acceptable for you to ship j3d as a bundle, I still want to reemphasize the fact that shipping a jar as a bundle has nothing weird. As Tom explained previously it is dangereous to rely on jars being present on the system without being able to express any dependency on them since it can cause weird errors if the expected jars are not present. That's why it is actually recommended to "bundle" jars that you did not author. This practice is common in the eclipse eco-system. For example the SDK ships Tomcat, Ant, JUnit, icu4j, ... The best practice to do that indicates that you ship the jar under the namespace of its original provider. Soin your case you would ship visad in the namespace of university of wisconsin (maybe can you get a sanctionned versioned from the author, even though it is not mandatory).

It seems that Tom focused on the global flag (which as you mentionned is not really platform friendly) and forgot to propose the buddy loading solution. In short the buddy loading can be seen as additional steps that are carried to find classes and is specific to a plug-in instaed of being global.
In order to set this up, add the following entry in the manifest.mf of the bundle that needs access to J3d:
   Eclipse-BuddyPolicy: ext

Hope this helps.
