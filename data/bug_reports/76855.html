Description Jeff McAffer 2004-10-22 15:04:50 EDT 
The main job of the Eclipse launcher is to find and exec java.exe.  Currently 
the only way to customize the launching of the VM is through command line 
arguments.  This is ok but forces people who need variations on the norm (e.g., 
need to set a VM arg, set the location of the VM, ...) to either expose the 
command line args to their users or create .bat files that call eclipse.exe or 
write their own wrapper exe for eclipse.exe.

Life would be much simpler if eclipse.exe could read a simple ini file which 
exposes the values currently set via command line args (e.g., -vmargs, -
vm, ...).  This allows products to craft and ship with an ini file configured 
for their requirements.  Their users then can simply run the exe and be assured 
they are getting the right configuration.

The ini file should be named after the exe being run. So eclipse.exe looks for 
eclipse.ini.  If eclipse.exe were renamed to foo.exe it would look for 
foo.ini.  This supports the proposal to allow users to copy/rename/bash 
eclipse.exe to suite the situation and naming requirements of their application 
(see bug 76722).


nominally but not necessarily called eclipse.ini)

Comment 6 Jeff McAffer 2004-12-08 14:51:38 EST 
I think you actually need to know explicitly the difference between 
- launcher directives (i.e., things that tell the launcher how/what to launch)
- VM arguments
- program arguments

This can be done by sections is fine if that is easy.  Alternatively it can be 
something like 
VMARG=-Dsomekey=someValue
ARG=-data c:\foo

Launcher directives are everything else e.g., 
VM=c:\jre\bin\java.exe
foo=bar
etc.

It is important that we are able to distinguish between these cases as they 
need to be processed differently.  Simply ordering them appropriately in the 
ini file does not seem sufficient.  As I say, sections or cool keys or... as a 
mechanism for differentiating are all equal to me.
Comment 7 Ed Burnette 2004-12-08 16:17:15 EST 
Ditto to comment#6. There's no need for the file format to be so simplistic. And
while the [sectional] format is kind of a Windows standard, it would be new to
Eclipse AFAIK. I think you would do better to stick to something we already use.
You can do grouping by name, for example instead of Full=Yes in the [Settings]
section you would just use Settings.Full=Yes.

The person or tool reading and writing this ini file will probably also need to
know how to use the formats in config.ini and manifest.mf so I suggest you just
pick one of those (for example, eclipse.ini in java properties name=value
format, or eclipse.mf in manifest name:value format).
Comment 8 Christophe Cornu 2004-12-08 19:08:48 EST 
v>20041208
released in HEAD

The launcher now looks up for the corresponding .ini file. e.g. eclipse.ini 
for the eclipse launcher, myapp.ini for a launcher called myapp.exe or myapp.

Details
.As indicated in comment 5, the syntax maps to the command line format, one 
argument per line.
.Launcher and VM arguments: command line arguments override the ini arguments.
.Program arguments: program arguments from the ini file are appended to the 
program arguments from the command line arguments.

Sample eclipse.ini
-vm
d:/eclipse/jre/bin/java.exe
-vmargs
-verbose

This ini file causes eclipse to run with the specified VM in verbose mode.

The Eclipse command line arguments as well as the arguments from the ini file 
are used to construct a command line to execute a java process. That command 
line must fit within 4 Ko as such is the limit on certain platforms. Whenever 
possible, program properties should be handled from java and through 
java .properties files where that limit does not apply.
