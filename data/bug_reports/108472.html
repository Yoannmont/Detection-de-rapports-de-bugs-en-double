Comment 8 Matthias Spycher 2005-11-09 15:03:48 EST 
I'm investigating a CDT restart bug on Linux using gdb and here's what I find.
It is probably related to the problem described above.

The inferior's output is piped to a pty which is monitored by an instance of
OutputStreamMonitor. When gdb restarts the program, the monitor may read an
end-of-file which returns -1 (from jni), exits the monitor's thread loop, closes
the pty input stream which removes the pty from the system, and gdb complains
that you can't do that. The mi response reads:

&"/dev/pts/9: No such file or directory.\n"
&"You can't do that without a process to debug.\n"
316^error,msg="You can't do that without a process to debug."

The culprit is (IMHO) in ptyio.c:

JNIEXPORT jint JNICALL
Java_org_eclipse_cdt_utils_pty_PTYInputStream_read0(JNIEnv * env,
                                                          jobject jobj,
                                                          jint jfd,
                                                          jbyteArray buf,
                                                          jint buf_len)
{
    int fd;
    int status;
    jbyte *data;
    int data_len;

    data = (*env)->GetByteArrayElements(env, buf, 0);
    data_len = buf_len;
    fd = jfd;

    status = read( fd, data, data_len );
    (*env)->ReleaseByteArrayElements(env, buf, data, 0);

    if (status == 0) {
        /* EOF. */
        status = -1;
    } else if (status == -1) {
...

Note the conversion of status == 0 -> status = -1.

I'm not sure eliminating this if statement is the correct thing to do under all
circumstances. But it appears to be the right thing for gdb-restart.
Comment 9 Matthias Spycher 2005-11-09 15:24:35 EST 
Regarding my previous message:
I should probably mention that the MIInferior instance takes care of closing the
pty input and output streams upon termination of the program as would be
expected. So there is no danger AFAIK of the pty inadvertently hanging around
after the gdb session has terminated. One just needs to ensure the monitor's
loop is broken (via exception or returning -1 on read()).
