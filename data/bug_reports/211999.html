Comment 1 Hugues Malphettes 2007-12-05 20:16:48 EST 
It looks like the diagram is corrupted.
This happened in older versions of the modeler where calling undo-redo-copy/paste would end-up incorrectly recreating or duplicating GMF-objects.

We have overridden all the operations that were corrupting the diagram and that we were able to reproduce.
Unfortunately there are still situations where the issue occurs.

I am interested by use-cases to reproduce the corruption so we can fix that.

I am marking as "WONT-FIX".
Please re-open this bug with a test case that leads to such corruption of the diagram.

Comment 3 Hugues Malphettes 2007-12-06 13:15:11 EST 
Thanks for following up. We appreciate it.

Your suggestion is a very good idea. It consists of sanitizing the GMF-view model and the BPMN-model after they have been loaded and before they are displayed.
It would try to identify corruptions and try to fix them as much as possible.

For example:
- Check that each BPMN node has its matching view-node. Create such view-node it is misses. The layout will be lost but it is sane.
- Check that each reference to a message actually exist. Remove the reference if the message does not exist. Remove the view if it does not exist.
- Check that each message is referenced only once per activity.

I would like to point to some code samples to give you an idea of what we had to do to avoid corruptions related to undoing the deletion of a message:
MessagingEdgeItemSemanticEditPolicy line 79
ActivityItemSemanticEditPolicy line 85

We had to fix things when the message itself is removed and also when it is removed because it is connected to an activity that is removed.

If this helps you identify actions that corrupt the diagram, please let us know!

Thanks again.
