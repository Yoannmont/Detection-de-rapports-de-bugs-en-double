Description Pascal Rapicault 2007-10-01 13:40:32 EDT 
While artifacts are being downloaded the dialog only shows a "cancel" button. Scott suggested that we may want to have a "pause" button in addition to the "cancel" one. Why should we have one, what is its semantics, what would be the difference with cancel, what happens when the user selects it (does the button turn into a "restart"?), is this "pause" concept needed in other phases.
Comment 1 Scott Lewis 2007-10-01 14:05:18 EDT 
(In reply to comment #0)
> While artifacts are being downloaded the dialog only shows a "cancel" button.
> Scott suggested that we may want to have a "pause" button in addition to the
> "cancel" one. Why should we have one, what is its semantics, what would be the
> difference with cancel, what happens when the user selects it (does the button
> turn into a "restart"?), is this "pause" concept needed in other phases.
> 

My initial suggestion was based upon the following observations:

1) Other popular web-based download/install UI's (e.g. firefox) have a user-level distinction between cancel and pause.  So user expectations might include a real separation between these actions.

2) If 'cancel' is actually 'pause' (where the UI only has cancel, and clicking on cancel results in a 'pause' which can be resumed), there are at least a few obvious questions:  a) how long does the already-downloaded data stay around/cached?  b) When is it 'actually' canceled?  c) What if the data on server changes between the time paused and canceled?  These questions can be answered technically, but there could be some troublesome user-level implications...and we should consider whether implied UI mechanism (e.g. cache clearing) is going to be necessary/expected).

3) Implications WRT multi-threaded downloads.  That is, if a single file (e.g. single plugin) from an installable unit that includes, say, 10 files is paused, what does this do/present in the UI? 

4) Finally, what if users really *do* want to cancel?...e.g. they realize during transfer (which will take the bulk of the time of a provisioning/install/update) that they want to stop the whole thing and do something else...then how will they accomplish this?  Will they have to cancel in several parts?  This could be difficult to make intuitive.

ECF's file transfer API can/will support either approach (pause and cancel, or just pause, since at least for supporting transports (e.g. http 1.1) we already have both pause/resume and a 'true' cancel.

Comment 3 John Arthorne 2007-10-02 11:08:47 EDT 
I agree we will need the ability to pause downloads. The main use case to me is when Eclipse is shutdown while a download is running. The state of that download should be persisted, and resumed again automatically upon startup.

Whether the user should be presented with both "pause" and "cancel" buttons is less clear to me. Presenting both choices implies they will understand/care about the distinction.  I really think simplicity should be the driving consideration in the UI - consider in the RCP space that we aren't necessarily talking about sophisticated or tech savvy users.  The best example of simplicity in this area is the iPod, which replaced the traditional "play", "pause", "stop", and "power off" with a single button that toggles between "make it go", and "make it stop". In that spirit I suggest only pause is needed. Some policy setting would dictate when partial downloads from a pause be discarded. In a simple workflow, the user or administrator would just start the top level install or update again, and when the download manager is asked to mirror artifacts, it may discover some partial downloads that it could resume. Perhaps a more advanced UI would provide a "downloads" view from which downloads could be monitored, paused, resumed, or discarded, but this seems unnecessary - most users likely wouldn't care to micro-manage their updates in this way.

Comment 5 Susan McCourt 2007-10-02 12:00:08 EDT 
I think the only distinction between pause and cancel in a typical end-user's mind is that pause keeps the dialog (and partially filled progress bar) open, and cancel closes the dialog.  It keeps the download on their radar.  Whether the same data is actually kept around on cancel is immaterial....unless you run out of disk space or something, in which case a cache clearing mechanism would presumably get aggressive anyway.

So we could do something in the UI like:
pause button - toggles between "pause" and "resume"
cancel button - closes the dialog

Both buttons tell the operation to pause (for example, we could interpret IProgressMonitor.isCanceled() to mean pause).  The net effect is that the state of the download is preserved as John suggested.  As long as the dialog is open, it will always be preserved.  When the dialog is closed, it falls under whatever the cache clearing policy is.

Comment 11 Jeff McAffer 2007-10-02 16:05:52 EDT 
I suggest that we have the capability to pause/resume and cancel under the covers and choose to expose whatever makes sense in the user's context.  there are several usecases where you would want to pause/resume install activity (e.g. the user is exiting the program).  For the first cut of the simple UI it is likely fine to expose just cancel.

WRT partial files, yes, these should be tracked in a download cache that is at least logically different from the destination repositories.  for example the way downloading works today is you ask the target for repo for an output stream.  you then give that stream to ECF (whoever) to write into.  When the writing is done, the stream is closed and the artifact officially registered as being in the repo.  One approach would have the close() differentiate between close() when the download is done and other cases and take different actions.

To support this case (e.g. needing to take different actions WRT the output stream when paused rather than done), last week I added an event IIncomingFileTransferReceivePaused event (and an associated IIncomingFileTransferReceiveResumed event).  These events are sent to the IFileTranserListener at the appropriate times.  This may be used by the download manager to make decisions about how to handle the received data/what to do with the OutputStream (maybe close then reopen, etc).  The p2 project files use an ECF code snapshot taken a couple of weeks ago, so these event classes are not currently in the p2 workspace, but they will be relatively soon.
