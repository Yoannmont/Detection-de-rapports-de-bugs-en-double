Description Jonathan Amir 2006-12-14 05:30:23 EST 
Build ID: M20060921-0945

Steps To Reproduce:
1. Create a utility jar with /META-INF/tags/mytag.tag
2. Create a Webapp that uses the utility jar.
3. Deploy it on tomcat, and run it.
4. Without stopping tomcat, make changes to mytag.tag. See how tomcat seems to redploy to webapp.
5) Run a jsp that uses the tag, and see that the changes did not take effect


Hello,

I am not sure if this is a tomcat bug or WTP bug, so my apologies if this is the wrong place to report. (I am using tomcat 5.5.20)

When changing a jsp, a servlet, or a class, WTP redeploys the webapp, together with utility jars. the redeployment usually resides in a folder similar to this (this is catalina.base):

/home/yoni/workspace/.metadata/.plugins/org.eclipse.wst.server.core/tmp0

When I access a changed jsp again, tomcat detects that the jsp is newer than the compiled one in ${catalina.base}/work, and it recompiles it (perhaps WTP even cleans-up the compiled files under ${catalina.base}/work, I don't know).

This is also the expected behavior with tag files, but it doesn't occur. I have a utility project, say P1, which has a tag file, say snoop.tag, at P1/src/META-INF/tags/snoop.tag

I have a webapp project, say WEB1. When I deploy WEB1 and access the tag, through a jsp, tomcat creates the following files:

${catalina.base}/work/Catalina/localhost/WEB1/org/apache/jsp/tag/meta/snoop_tag.java
${catalina.base}/work/Catalina/localhost/WEB1/org/apache/jsp/tag/meta/snoop_tag.class

When I make changes to snoop.tag in the workspace, WTP redeploys everything, as expected, but when running the jsp again, my changes are not picked up. I need to browse to the folder mentioned above and manually delete the files. Only then will my changes be picked up.

Is this a WTP bug or a tomcat bug?

Comment 2 Tim deBoer 2007-01-08 13:45:43 EST 
This doesn't sound like a WTP bug to me, but there are several questions/issues here:

 * Does Tomcat support auto-reloading tags?
 * Does WTP have a bug that's blocking Tomcat from reloading?
 * Is there anything we can do in WTP to improve the situation even if we're not the problem, e.g.:
   * Mark the server as requiring a restart, or
   * Providing an action to clean Tomcat's work directory (I think Larry's already working on this)

Larry - I can't remember the spec or what Tomcat supports here :). Do you know what works on Tomcat?
Comment 3 Larry Isaacs 2007-01-10 13:34:52 EST 
Tim is right that this is a bug/quirk in Tomcat.  It has been a while since I have poked around in Jasper code, but it appears that the dependency tracking doesn't extend into jars, at least not for modification dates.  The simple workaround is to touch the JSP file that uses the Tags and Tags that are in jars will be unconditionally recompiled along with the JSP.

Deleting the generated tag files in the work directory didn't work for me, but deleting the generated JSP did.  The disadvantage of this approach, including deletion of the context's or server's work directory, is that Tomcat needs to be stopped and restarted.  As for the spec, I don't recall it saying anything specific about re-compilation of JSPs and Tag files, i.e. live updating of webapps.  I'm not sure if this behavior is an oversight or because it wasn't practical to do better.  I wouldn't be surprised if it was the later.
Comment 4 Jonathan Amir 2007-01-10 16:48:36 EST 
1) You are right that just deleting the tag files doesn't help. I see that too - it only produces a NoClassDefFoundError. I probably deleted the whole work directory earlier on, but wasn't specific enough when I first opened this bug.

2) If I put the tag files directly in the Web Project, under /WEB-INF/tags, then everything works as expects. Therefore, I assume that as far as the spec goes, jsp tags should indeed be auto-reloaded just like jsp pages. The problem seems to be checking the timestamp of a tag file inside a jar file.
Comment 5 Tim deBoer 2007-01-10 17:46:18 EST 
Re #2, I don't think you should assume that because it works in Tomcat in one situation, it is in the spec and should work in all situations. I think Larry is probably right that this is either a Tomcat oversight or a case of what is practical.

In either case, this appears to be confirmed as Tomcat behaviour and not a WTP bug, so I move that we treat this bug as an 'is there anything we can do to improve the usability of Tomcat from within WTP' bug. Some options:

  1. Touch the JSP file for the user when the tag changes - would be a neat feature, but a lot of WTP adopters would frown on modifying user files (especially if under source control). I also don't know if there is any way to reliably map back from the tag to the JSP.

  2. Automatically delete the generated JSP file when the tag changes - Ditto to above on finding the right JSP file(s), and it sounds like the server still needs to be restarted.

  3. Mark the server as requiring a restart when a tag changes, which will cause the user to get prompted next time they run on server. The downside is that the server didn't really need a restart since they could just touch the JSP file.

  4. Leave it up to the user to touch the JSP file or restart Tomcat and close this bug.

Any other options? Options 1-3 don't appear to be great solutions to me.

Comment 10 Larry Isaacs 2008-03-28 11:29:47 EDT 
I've confirmed that "Serve modules without publishing" option is sufficient to workaround this "feature" in Tomcat.  This will require one restart of Tomcat if this option was not already enabled, but another restart while modifying the tag file should not be needed until you choose to disable the "Serve modules without publishing".  Note: If you are using Tomcat 5.5, you will need version 5.5.23 or later for "Serve modules without publishing" to work successfully with tag files in utility projects.

WTP 2.0 also provides a simple way to clean the work directory (i.e. right click on the Tomcat server or module in the Servers View and execute the command to clean the Tomcat or module work directory), though it cleans all of the files and not just those related to the tag file.  This isn't quite as good as the "Serve modules without publishing" workaround as it will require a restart for each clean.

At this point, I'm also doubtful that it would be practical to modify Tomcat to help.  Jars in Tomcat are accessed by URL, which doesn't provide a good way to retrieve time stamps.  The Tomcat modification would be to add a "dev" mode that always compiles tag files in case it had changed.  This would apply to all tag files, and would be much less effective than the "Serve modules without publishing" option.  Thus, I don't think it would be worth the cycles it would require to implement.

As a result, I'm changing this bug to "normal" from "enhancement" and marking as NOT_ECLIPSE.  I don't anticipate trying to improve on the "Serve modules without publishing" solution in WTP 3.0 and beyond.  Cycles would be better spent elsewhere.  For a reminder, I'll refrain from validating or closing this bug until I have updated the Tomcat FAQ to cover this issue.
