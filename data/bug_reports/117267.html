Comment 4 Chris Gross 2006-01-12 10:08:04 EST 
The need is to have a shell which will not steal focus from any other window.  This shell would be used for popups or dropdown shells which do not want to steal focus from the parent shell.  This behavior is evident in CCombo (see bug 19396 or bug 91666).  We need to create similar widgets (emulated combos, menus and more).  

I'm aware that you can show a shell by simply setVisible(true) thus allowing the window to show without initially moving focus, but focus will eventually move when a widget within the shell gets focus (hence Ben's comment).

Comment 5 Steve Northover 2006-01-12 17:24:25 EST 
While there are low level event forwarding hacks possible on Windows, this sort of solution is pretty hard to make portable.

What SWT applications currently do is use an SWT.ON_TOP shell and Shell.setVisible() rather than Shell.open().  The popup shell does not become active and focus stays in the original shell.  Mouse and keyboard events are interpreted by the focus control in the new shell and API is used to manipulate the control in the popup shell.  For example, if the popup shell contained an table and the user pressed SWT.ARROW_DOWN, the control in the original shell might call table.setSelection(table.getSelectionIndex() + 1)).  When the user clicks on a control within the popup shell, focus goes to the popup shell (otherwise, you couldn't type in a text field).

I realize that this isn't exactly what you want, but it is portable.  CCombo doesn't currently work this way, but there is no reason that it couldn't.  As  Benjamin pointed out, the SWT.NO_FOCUS bit can be used with custom draw widget.  However, native widgets attempt to bring the shell forward and make it active.

Your thoughts?

Forgetting about the pain of mapping events through to the popup shell, as you point out the problem still remains:

"When the user
clicks on a control within the popup shell, focus goes to the popup shell"

This steals focus from the main window and in Windows this means your main titlebar redraws as if that application were inactive - which is incorrect. In Eclipse this can be seen when using the scrollbar in the content assist window. Content assist is not meant to be a separate floating window that can be moved/dragged and therefore does not have any affordances with which to indicate focus (other than selected item focus color). It should not steal focus from the main Eclipse window. I don't use a Mac, but I'm guessing this is a much bigger problem there. 

There are certainly os specific style bits than can be given to the window to prevent this. If there are portability issues, can a solution be seen as a hint instead?
