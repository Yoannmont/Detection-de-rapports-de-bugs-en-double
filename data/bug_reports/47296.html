Comment 4 Chris McLaren 2003-11-26 14:42:08 EST 
this is very strange. in the M5 build line 517 of KeyStroke is:
hashCode = hashCode * HASH_FACTOR + modifierKeys.hashCode();
(to see this, import the workbench plugin as source from the M5 drop..)

the NullPointerException can only occur when dereferencing the private field 
modifierKeys, which only is write-accessed once - in the first line of the 
KeyStroke constructor. this first line is guaranteed to return non-null. in 
fact, line 3 of the constructor would throw a NullPointerException if the 
field was null.

if the stack trace was not complete, and the NullPointerException occured 
further down, it would be inside the Set method of hashCode(), which, if you 
follow through TreeSet, HashSet, and Collections.UnmodifableSortedSet, can not 
occur. (they all eventually get to java.util.AbstractSet.hashCode() which is 
safe)

if anyone has a different stack trace, or info on their VM, I would appreciate 
it added to this PR.

i'm going to pass this to doug to double-sanity-check this, otherwise should 
close as 'worksforme'
Comment 5 Douglas Pollock 2003-11-26 15:17:06 EST 
Could one of the three people seeing this bug please specify which VM they are 
using (and version), as well as the specifications of the machine.  Thanks.


Chris: I don't think it is a good idea to close this bug, even if we don't 
know what's wrong.  There are three separate people who have reported this 
problem.


In an extremely unlikely case, I think it is possible for this error to occur 
if two separate threads were trying to access the key stroke at the same 
time.  But, I think it would require a multi-processor machine, which the 
first person specifically says they don't have.  And, I didn't think this 
section of code was multi-threaded.  However, if one thread constructed the 
key stroke and another asked for that stroke's hash code, I believe it might 
be possible that write re-ordering performed by the memory bus could cause a 
NullPointerException.

Seems far-fetched, hunh?
Comment 6 Chris McLaren 2003-11-26 16:47:30 EST 
i don't think that makes sense. KeyStroke has no super. Constructor finishes 
before initial reference is assigned to variable. (the 'x' in this expression:
x = KeyStroke.getInstance(..).

another thread cannot get access to this variable to call hashCode() until the 
initial reference is available, after which point the field modifierKeys is 
not mutable.

i noticed some activity on eclipse-dev about incorrect debug information 
between source and class in M5. i think it possible that we have a case of 
this.
