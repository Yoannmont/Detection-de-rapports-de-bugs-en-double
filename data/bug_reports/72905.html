Comment 4 Paul E. Keyser 2007-04-23 12:49:51 EDT 
(In reply to comment #0)
I see the same bug, as follows: 

0) user clicks on tab => CTabFolder.setSelection(int) is correctly called 
1) that method gets the Control of the TabItem and sets it visible 
2) that setVisible() calls fixFocus() on the given Control 
3) the impl. in Control of fixFocus() climbs the containment hierarchy and calls setFixedFocus() 
4) when that reaches CTabFolder.setFixedFocus() [i.e., when the receiver is CTabFolder], the method will return after the *first* child returns true from setFixedFocus() -- which is incorrect! (E.g., if the code first creates a "TopRight", that will be the first child, and it will return true, and -- voila -- the "TopRight" has snatched focus from the the user-selected TabItem's Control; or, e.g., if there is no "TopRight", or if the "TopRight" was created last, then if the user selects any TabItem but the zeroth, the zeroth TabItem's Control will steal focus.) 

Note -- if there is a "TopRight" that was created first, then immediately after the user selects any tab, you can see that the focus is in the wrong place by noticing that the "TopRight"'s button clearly has focus, and you can use the SHIFT-TAB-key to navigate the focus back to the correct TabItem (you'll see the underline appear). 

THIS BUG EXISTS WHETHER THERE IS A "TOP-RIGHT" OR NOT!

Comment 7 Carolyn MacLeod 2009-03-25 04:39:24 EDT 
(In reply to comment #2 and comment #3)

There is a bug in the test case. It is not necessary to dispose of the tab or its control in the CTabFolderListener itemClosed (or CTabFolder2Listener close) event. So the following line:
  ((CTabItem) event.item).getControl().dispose();
is actually disposing the Text control before it has a chance to get the focusLost event. If you comment out this line, focusLost is correctly called.
Comment 8 Carolyn MacLeod 2009-03-25 04:51:59 EDT 
Created attachment 129813 [details]
CTabFolderFocusBug.java

(In reply to comment #5)

The patch is not quite correct - really close, though. I have attached a simple test snippet that can be used to show the problems described in comment 0, comment 4, and comment 6.

Comment 0 says that if you click on a tab, that tab should get focus. But it is important to note that this *only* applies if the tab was already selected. We do not want the CTabFolder tabs to be grabbing focus every time someone selects a new tab. That could break Eclipse.

To test this, run the snippet and click on Item 0. Item 0 should be underlined, and Text 0 should no longer have focus. Then run the snippet again and click on Item 1 instead. This time, Text 1 should have focus. Item 1 should *not* be underlined.

You need to also run Eclipse from inside Eclipse (if you have never done this before, ask me how in email) to make sure that this does not break the Eclipse views and editors.

Comment 4 says that if a tab's control has multiple children, then focus should be preserved. I tested a regular TabFolder to see what it does, and it doesn't preserve focus... so maybe we shouldn't either. I will discuss this with Steve.
Anyhow, to test this, run the snippet and click on Item 1. Click in Text 2, then click on Item 0, then back to Item 1. If comment 4 is the correct behavior, then Text 2 should still have focus.

Comment 6 (the duplicate) says that using a tab's mnemonic should give focus to the tab. Again, I am not sure if the correct behavior is to focus the tab, or the tab's control. I will attach the regular TabFolder snippet below so that we can compare CTabFolder behavior with TabFolder behavior. Type Alt+0 to activate the mnemonic for tab 0, and Alt+1 to activate the mnemonic for tab 1.

Other small notes for the patch: I would just delete the comment, because it doesn't add any additional information. Also, note that we typically use // comment style for commenting out code, and we use /* Sentence here. */ comment style for actual (non-API) comments.

One final note for the patch - there is another call to setSelection(int, boolean) in CTabFolder.showList that does not have a forceFocus... should it?

Comment 16 Carolyn MacLeod 2009-06-29 16:39:24 EDT 
Re comment 12:

Actually, try running Eclipse inside of Eclipse again, and then create or import a project so that you can open some editors. The Eclipse Editor area is a CTabFolder, and each editor is open in a CTabItem. The CTabItems are very definitely affected by these patches, and so these changes do break Eclipse. Stacked views (i.e. Package Explorer, Hierarchy) are affected, too.

The one change that we can make without breaking Eclipse is to give focus to the tab when the user clicks on an already-selected tab.

Also, Bug 188426 (see comment 6) asks for mnemonics to also give focus to the tabs. Eclipse doesn't use mnemonics for switching between CTabItems (at least, not for Editors or Views). So this might be another area where we could give focus to the tabs. SSQ, what do you think?

But Ctrl+PageUp/Down navigation needs to stay the way it is. So does selecting an unselected tab with the mouse. These are both different from the way TabFolder works, but it is too late to make this consistent because Eclipse is relying on that difference.

We can consider preserving the selected child when selecting an unselected CTabItem, but I'm not sure that that is what a user would typically expect (and it is different from TabFolder). So maybe just setting focus to the first child on selection is the appropriate behavior in that case. SSQ?
