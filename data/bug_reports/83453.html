The timeout exception below was my bad.  Debugging causes this.  What is
interesting is that the response is in the linked list, but since the timeout is
determined first, the reply isn't picked up.

The IllegalArgumentException is a bug in the code since it sends a null message
which is disallowed (not in the javadoc though).

And finally, the loop.  I don't know what causes it but I do know the sequence
now.  It happens on both Java 5 and J9.

-> = invokes    ... = job execution

RemoteTreeContentManager.replaceChildren() -> WorkbenchJob.schedule() ...
VariableViewer.runDeferredUpdates() -> fStateRestorationJob.schedule() ...
StateRestorationJob.runInUIThread() ->
RemoteVariableContentManager.startFetchingDeferedChildren() ... 
DeferredStackFrame.startFetchingDeferedChildren ->
RemoteTreeContentManager.replaceChildren()

Comment 4 Kevin Barnes 2005-01-25 18:44:34 EST 
caused by attempts to restore the variables view to a previous state that was 
invalid. The ViewerState would attempt to decode a path that didn't exist in 
the variables view tree, this would cause the content provider to 
asynchronously getChildren(), the tree would get updated and the we'd attempt 
to restore the state again.
When restoring the state of a remote tree viewer we should only be interested 
in it's current state, the viewer is responsible for maintaining that state. 
Changed AbstractViewerState to getCurrentChildren() instead of getChildren() 
when the view is a remote tree viewer.
