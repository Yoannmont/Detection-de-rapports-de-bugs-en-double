Description John Swanke 2004-10-25 11:14:16 EDT 
Eclipse seamlessly combines multiple applications into one development 
platform. However because Eclipse is so seamless that a simple editing action 
or file open can automatically load up another Eclipse application, using a 
simple wait cursor during plugin loading sends a horrbile message to a user: 
sometimes deleting a "d" in my document takes an instant, sometimes it takes  
minutes. Why? I dunno.

Replace the wait cursor with feedback telling the user what's going on. Put 
back the idea of a welcome screen for each application running in Eclipse if 
only a plain dialog reporting what application is being load and listing the 
plugins as they're loaded. 

Instead of user frustration wondering why Eclipse is running so slow today, you 
have user appreciation that they didn't have to leave Eclipse in order to 
startup this other application. And of course appreciate that because of 
Eclipse's architecture, that application didn't startup until they needed it.

Comment 2 Tod Creasey 2004-10-25 11:45:54 EDT

Comment 3 John Swanke 2004-10-25 11:57:49 EDT 
By application i mean something like Page Designer which is made up up multiple 
plugins. Dan Kehn mentions that "The Platform Core / OSGi framework certainly

Workbench status line, for example." However for the same reason that the build

information should be in a dialog.

Plugin loading typically can't be done in the background--its got functionality 
the user is waiting for now.

Perhaps some extension of busy runnable that uses a dialog instead of a wait 
cursor specfically for plugin loading.
Comment 4 Jeff McAffer 2004-10-25 13:36:40 EDT 
I agree with the overall sentiment.  User does something and it takes a long 
time with no real feedback.  The main issue is that in such a decoupled system, 
no one really has the perspective to know that some lengthy initialization is 
in progress.

This has been talked about in the distant past but we were unable to find a 
good model.  The problem has very little to do with Eclipse.  It may not even 
have to do, technically speaking, with "plugin startup".  When the user hits 
some new function, typically classes are loaded. When classes are loaded there 
is the chance, depending on what else has been done in that session, that some 
plugins will need to be started.  

How many classes need to be loaded and how many plugins activated, as well as 
how long activation takes is completely in the hands of the function developers 
and out of the hands of Eclipse.  If all plugins were good citizens and had 
very fast (or better yet no) statup() methods, Eclipse would not even know that 
something new was happening.   Everything would just appear as classloads.  
Even then, classloading, while expensive, is not the thing that is causing 
multi-minute delays.  That would be due to plugin code execution.

So, despite Dan's assertion, the runtime does not know enough to be able to 
report meaningfully.  To get an idea of how much time the runtime knows about, 
use the Core tools and do/time one of these lengthy operatoins.  Then go into 
Core tools and add up all the time that it reports.  My bet is that you will 
get but a small fraction of the actual time taken.  People are just writing 
big, bloated, slow function that does too much for mainline operations.  

Any ideas on how to manage this would be greatly appreciated.
Comment 5 Dan Kehn 2004-10-25 14:02:38 EDT 
<< So, despite Dan's assertion, the runtime does not know enough to be able to 
report meaningfully. >>

I was alluding to a very primitive feedback mechanism, i.e., "Starting XYZ..." 
in the message status line with the busy spinner going.  That will tell the 
user something is going on, although they won't know their point in the 
progression.  It was my assumption that we're talking about feedback to calm 
the nerves of a user who'll be waiting 10-15 seconds.  If the delay is longer 
than that, I consider it a performance bug.  I certainly don't want to 
implicitly condone delays by providing **too** meaningful progress reports.
Comment 6 Tod Creasey 2004-10-25 14:25:14 EDT

Comment 7 John Arthorne 2004-10-25 14:45:51 EDT 
First step would be to try switching extension activation from a
BusyIndicator.showWhile to an IProgressService.busyCursorWhile. That would at
least switch from a busy cursor to a progress dialog that would allow a more
meaningful user message when activation takes non-trivial time. The message
should probably be something generic like "Loading plug-ins" since it is hard to
determine exactly the set of plugins that will be loaded (due to recursive
activations), and many plugin names are not likely of interest to end users. 
That alone would probably go a long way towards making the end user feel in
control - it allows them to connect a cause to the effect of long loading,
rather than the apparent "random freezing" that many users report.

The major "threading issue" with moving activation out of the UI thread is that
many UI extensions probably assume they are being activated from within the UI
thread. Luckily for most UI extensions the actual widget creation (methods such
as createPartControl) are typically decoupled from their construction, but I
suspect you'll still find some extensions whose constructors try to access or
change widgets, causing invalid thread access exceptions.

Comment 11 Jeff McAffer 2004-10-25 17:12:57 EDT 
re comment 8.  We actually don't know.  The runtime knows that a plugin is 
being activated.  There is no information to indicate that it will take a long 
time.  JohnA did some tests and the longest one was 400ms or so.  Most were 0-
50ms.  The aggregate was a reasonably long time but there were 100+ of these 
little ones.  The little operations, from the Eclipse point of view, are mostly 
unrelated.  You wouldn't want to popup a dialog (or anything for that matter) 
for each of these.  JohnA (comment 7) suggests IProgressService which brings up 
a dialog after some delay.  That sort of idea might work but the delay is too 
long for this scenario and there are the threading issues he mentioned.  
Something could likely be worked out however.

Messages like "Loading additional plugins..." are good but hard to manage.  
When do you take it down?  How do you know when you are done the last one? 

I still maintain that the issue is a lack of coarse grained perspective.  Even 
the summary of this bug implies that there should be something called 
an "application" and that it should have welcome screens.  The vision is great 
(e.g., "now loading J2EE tooling" in response to the user opening a JSP) but 
Eclipse needs help to generate these kinds of high level messages (or even 
grouping ones) since we see only the decoupled micro level events.
Comment 12 Dan Kehn 2004-10-25 17:48:49 EDT 
I agree it is a sticky problem, i.e., how the Workbench recognizes when to 
provide a clue and when to remove it.  My initial thought was a "low key" 
reminder of activity -- certainly no dialogs because that itself could be 
disruptive (e.g., if the triggering event was a pop-up menu request, I don't 
want a dialog flying out).

Limited, automated "I'm busy working for you" reminder is adequate.  The 
triggering events that come to mind are (a) plug-in activation, (b) 
createExecutableExtension invocations, and (c) new classes loaded into 
memory.  A busy reminder that times out after a second of idle time (meaning 
no new classes loaded) may address most cases we're interested in based on our 
performance data, which indicates the majority of the cases fall into first-
time class loading.  

The application's plug-in(s) could provide hints about the beginning / end of 
the delay, but I think that's overkill.  Heck, there's nothing to stop them 
from doing that today, hence our interest in an "automatic" feel-good response.

Comment 17 Jeff McAffer 2004-10-25 21:00:10 EDT 
Some observations:
- The current busy cursor is triggered by the IDE UI in their helper for 
createExecutableExtension if the plugin supplying the extension is not 
currently loaded.  This may capture enough interesting cases to be useful.  Its 
not clear.  

- This approach misses cases where (among others): the pluign is active but 
other are not, the helper API is not used (not sure it is API), the code doing 
the triggering is not UI code, ...  

- Giving progress/activity feedback on some long operations and not others 
*may* be confusing for people.  For example, "Eclipse tells me when it is 
loading plugins.  Its taking a long time and it is not telling me itis 
loading.  It must be hung..."  Not sure how this would play.

- I like the subtlety of using the cursor.

- Is there value in having a different busy cursor?  Does the user particularly 
care?  They have to wait either way.

- Comment 13 and comment 7 net out to about the same thing.  The user gets 
notification if the operation starts to take a long time.  This is nice.

- To get a sense for this approach we need to do some runs where the 
createExecutableExtension is instrumented to say how long the busy cursor is up 
each time as well as how much of the total time this accounts for.
Comment 18 John Swanke 2004-10-26 07:17:23 EDT 
>>Is there value in having a different busy cursor?  Does the user particularly
>>care?  They have to wait either way.

If you look at other applications--even other parts of Eclipse--it's true. A 
busy cursor, no matter what it looks like, is only used for short waits. Longer 
waits are always explained with a message. They have to wait either way, but 
for longer waits most applications tell you why. Otherwise I can see why you'd 
get complaints of "random freezing". Even in response to a popup menu, openning 
a dialog only big enough to hold: "Loading additional plugins..." wouldn't be 
much bigger then 5 adjacent busy cursors.

IOW if some small, time-delayed dialog triggered by createExecutableExtension 
can't provide a visually pleasing message, I don't think there would be any 
point in simply changing the cursor.
Comment 19 Dan Kehn 2004-10-26 10:54:12 EDT 
<< IOW if some small, time-delayed dialog triggered by 
createExecutableExtension can't provide a visually pleasing message, I don't 
think there would be any point in simply changing the cursor. >>

There is some precedence for a cursor augmentation in Windows.  When you 
double-click an executable that's on a CD-ROM, the wait cursor includes a 
little flashing disk in the corner, presumably to alert you to the fact it 
will take longer than usual.  Seeing that means "time to take a break" to 
me.  :-)

I agree that a message / animation / cursor augmentation of some sort is worth 
having.  The busy cursor gives no clue that the delay will be a little 
longer.  Apparently the folks at Microsoft figured out that users complain 
less if you give contextually-sensitive feedback, however limited it may be.
