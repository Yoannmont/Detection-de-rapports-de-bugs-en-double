Description John Lanuti 2006-05-26 10:10:02 EDT 
Opened for IBM (David Slubicki):

Project explorer doesn't properly sort some content.  example:
create a dynamic web project, add a few jsp's to the webcontent
folder (name them a.jsp, b.jsp, z.jsp) then create a new folder (named "myfolder") in that same folder.  Notice that the new folder is sorted alphabetically along with the various files - it should
be amoung the folders.  So the contents of WebContent would look something like this:

META-INF      <---folder
WEB-INF        <---folder
a.jsp               <--- file
b.jsp               <--- file
myfolder         <---folder
z.jsp               <--- file


if you then create a new file, it will get sorted properly.
Comment 1 Michael D. Elder 2006-05-30 10:47:08 EDT 
The problem here is that resources are being added through Viewer.add() so the framework has no knowledge to correlate the item with its source, so it ends up making a best guess on how to sort the items. The solution for this issue will be to enhance the ability of the framework to know which extension added which items. While a little annoying at times, I do not consider this to be a critical issue (so much so that a fix should be made and released to a Platform RC build). Therefore, I am going to target this to 3.2.1.
Comment 2 Michael D. Elder 2006-07-18 11:29:50 EDT

Comment 3 Tod Creasey 2006-08-16 12:53:09 EDT 
If it is not critical we should fix this in 3.3. If you agree please remove the 3.2.1 tag
Comment 4 Michael D. Elder 2006-08-17 09:36:28 EDT 
I would like to release this patch for 3.2.1, but I need an approval because it exposes 1 new API method (2 signatures) on INavigatorContentService. This is important for clients that use the viewer.add() functionality and still want their extension to be the registered source. It is not required for clients to use this API, but some clients will want to take advantage of this feature.
Comment 5 Tod Creasey 2006-08-17 09:57:10 EDT

Comment 6 Mike Wilson 2006-08-30 15:07:38 EDT 
Here are some words that I entered into a bug yesterday, and I'm very disappointed to have to be adding them to another one today:

"
We treat adding new API in maintenance releases as an exremely serious issue. Even though this API seems totally reasonable, the end result is that we would now be in a situation where plug-ins can be written that would run against the maintenance release, but not against the base release. In other
words, anyone who uses this API would need to require 3.2.1 in their plug-in dependencies. Ugh!

In addition, maintenance release cycles are not set up to give the community time to review new API so that we could respond to their concerns. As such, if we have any doubts about the API _at_all_ then it absolutely must be moved to an internal package regardless of who has currently implemented something on some R3.2.1 development stream build. Only if we are 100% confident that this
is *the* answer for R3.3 and beyond can we even consider it.
"

In this case, to me it looks like the problem being solved is visible, but not serious (i.e. no data loss, the environment doesn't crash, etc.) plus the change is in an interface which, of course, breaks binary compatibility. Even if this interface is not intended to be implemented by consumers, I would not like to see this change made in the R3.2.1 stream. 

Adding Boris to the CC for his perspective...
Comment 7 Boris Bokowski 2006-09-05 23:26:50 EDT 
I agree with McQ that we should not introduce new API in maintenance releases unless there is no other way to fix a critical bug.

That said, the interface in question is spec'd as not to be implemented by clients, and even if clients implemented it anyways, I don't see how they could put their implementation to use - the field in CommonViewer is private, and there is no setter for it.  In a nutshell, the patch does not break binary compatibility, but I would rather see us fix this in 3.3.

Setting target milestone to 3.3.
