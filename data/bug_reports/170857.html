Comment 7 Darin Wright 2007-02-02 13:25:20 EST 
The problem happens when threads for a terminated target are being updated after the target has already terminated, when the 'show system threads' filter is filtering system threads.

The viewer's content provider maintains a mapping of which children are filtered. When a node's child count is set to 0 or a node's 'hasChildren' property is false, the filtered chidlren for that node are cleared. However, when an element's child count was set to a non-zero value, we did not clear any filters for items above that count (for example, a node changes from 10 to 5 children and items 7 and 8 were previuosly filtered). Updating the filter indicies fixes this problem, as we no longer attempt the insert.

Changes to:
ChildrenCountUpdate
FilterTransform
ModelContentProvider

Comment 13 Darin Wright 2007-02-07 17:23:27 EST 
Created attachment 58494 [details]
patch

I have a patch for this, but I want to wait for M6 to release it. 

We are not properly updating filtered indexes when an element (thread in this case) is removed from the model. We simply remove the element from the viewer but to not update filter indexes accordinly (remove the filtered element, if it is filtered, or decrement indexes of other element now that an element is removed).

This patch adds support to update filter indexes appropriately. When the index of the removed element is not known, the parent has to be refreshed and filtered elements are recomputed. This adds some flash to viewer as threads off the bottom of the viewer are removed.
