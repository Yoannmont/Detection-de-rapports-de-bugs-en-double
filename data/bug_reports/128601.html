Description Takehiko Ishii 2006-02-20 04:07:25 EST 
Environment:
Windows XP SP2 Japanese
eclipse 3.1.1 and WTP 1.0 ( wtp-all-in-one-sdk )

Description:
eclipse fails to generate bottom up java bean Web Service using Java class whose path contains DBCS.

Reproduce:
1. create new DBCS Dynamic Web Project
2. create new Java class "WebServiceTest" and a new function "echoString" like the following:
	public class WebServiceTest {
		public String echoString(String input){
			return input;
		}
	}
3. right-click on java class just created, then select "Web Services"->"Create Web Service"
4. on the wizard, uncheck "Install Web Service on server". click Next 4 times.

Current Behavior:
MalformedURLException happens. As a result of this error, eclipse fails to generate java bean Web service.

Expected Behavior:
eclipse can generate bottom up Java bean Web Service even if the path of Java class contains DBCS.

Comment 20 andy zhai 2006-05-24 18:38:02 EDT 
Kathy,
I uploaded the new patch. I have tested it. It works when creating Top Down Java web service, Bottom up java web service skeleton and java web service client. 

Notes.
1. Workaround is needed for Tomcat configuration file -- server.xml. 
   Change 
   <Connector acceptCount="100" .....redirectPort="8443"/>
   into
   <Connector acceptCount="100" .....redirectPort="8443" URIEncoding="UTF-8" />

2. The calls of javaWSDLParam_.setInputWsdlLocation(wsdlLocation) are not consistend. 
The string "wsdlLocation" in some calls are file location such as d:\ws\web\aa.wsdl, but in some calls they are URI string (e.g. file:/d:/ws/...). So I changed all of file location into URI formats in patch.

Comment 28 Chris Brealey 2006-06-07 15:57:30 EDT 
Andy, thanks. I approve the first patch (ie. patch 42514) with the one change we've discussed directly, and that you've also tested very well, of calling toURL() instead of toURI() since the former produces file: URLs (from rich Unicode strings) that are more apt to be correctly processed when fed back to java.io.File.

This is a very simple change that insures file: URIs instead of physical filesystem paths get passed to the Axis emitters which are able to handle DBCS and other non-ISO-8859-1 character data via URIs apparently much better than in physical filenames pathnames.

Would the PMC please approve this bug? Without this fix, workspace projects and paths with non-ISO-8859-1 characters (eg. DBCS chars) will tank the Web service wizards and Axis emitters within.

NOTE: The second patch has a few issues, and is in fact for a different, and much less common or critical, bug altogether. As discussed, would you please open a separate bug describing this second problem and add remarks to both bugs simply to cross reference each other? We will NOT commit this patch under this current bug.

Thanks - CB.
