Comment 4 Lin Zhu 2006-12-14 05:53:19 EST 
Hi Valentin,

This is about the strategy of populate parameter value. When a parameter has a position as well as parameter name, both of which are valid, which one should take priority? In our implementation we make position prior to the name.

As the code quoted by you:

if ( isParameterPositionValid(parameterHint.getPosition( )) )
  this.setInputParamValue( parameterHint.getPosition( ),inputValue );
else
  this.setInputParamValue( parameterHint.getName( ),inputValue );

The only chance that above code run to problem is that certain parameter hint is defined as its position not match its name, and you want its name take priority.

However, as parameter hint is freely definable by user (It is usually a mapping populated during design time, which contains values returned IQuery.getParameterMetaData() and custom default values), the problem could possibly be: com.mg.merp.wb.report.birt.data.oda.badi.IQuery.getParameterMetaData() has not returned correct parameter information(its position does not match its name).

The code you quoted is the internal logic of BIRT Data Engine.Its behavior decided by external output, "Parameter Hint" in this case. The parameter hint is setable using BaseDataSetDesign.addInputParameterBindings method. I do not know in your application how you construct BaseDataSetDesign(If you talk to DtE directly), or org.eclipse.birt.report.model.elements.DataSet(If you talk to BIRT model).However we are sure that in both of them you can define you parameter position to "-1".

Thanks.
Lin

In design time, if you are using BIRT Model, you can manipulate data stored in org.eclipse.birt.report.model.elements.DataSet to set a "-1" value manually to param position.

> 2. IMHO when org.eclipse.datatools.connectivity.oda.IDataSetMetaData method 
>   public boolean supportsNamedParameters() throws OdaException; returns "true",
> named parameters must have priority. 
By default, all the driver should support setting parameter using parameter position.

The method supportsNamedParameters() is used to indicate user that the parameter name can also be used to populate the parameter, however, this should not prevent driver from support parameter position. The using of parameter name is a supplement of parameter position rather than a replacement.So maybe we should not let the parameter name take the priority.

>Or, maybe, another way is to set both
> param by name and by index? And developer in his ODA driver implementation
> chooses which of them have higer priority
Actually the ODA driver implementation would not do such work. The conflict exists in BIRT rather than ODA.When position and name have conflict, BIRT needs to decide which method in ODA implementation to call, setXXX(name,value) or setXXX(index, value).ODA driver would not meet name/index conflict at all.

Thank you!

Comment 19 Lin Zhu 2007-04-16 05:36:26 EDT 
Hi,

The valid ParameterHint should qualified for following condition: whether we set parameter value by using its index or its name, we can always get same result after query execution.

If the ParameterHint is invalid, that is, the above process cannot get same results, we will simply take parameter index for priority.

As client for DataEngine, you need to ensure one of following: A)ensure setting of parameter value by index/name can get same execution result B)If you set parameter index to valid value, meanwhile the parameter name are incorrect, then you must make sure that setting parameter value by index can give the result you want C)If you do not want we use parameter index to set parameter, then you should set the index value to -1 so that we will set the parameter value based on parameter name.

Thanks.
Lin
