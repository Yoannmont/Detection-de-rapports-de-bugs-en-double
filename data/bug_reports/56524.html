Description Gunnar Wagenknecht 2004-03-29 01:15:47 EST 
The activities support is confusing and ugly. 

Today I upgraded to 3.0 M8 from I20040316. I had to throw away my workspace 
because of start up problems. I'm using a self hosted workspace with lots of 
plugins imported as binary projects. I'm developing against 2.1 and thus I need 
to change the target platform for importing the plugins.

The first thing that bothered me was when I opened Window-Preferences. The 
preferences for "Plug-in Development" were completly missing. I wondered about 
myself and asked myself if I did something wrong. I checked the available 
perspectives and views. Mhm. No PDE perspective and views. Damn. I checked to 
make sure I downloaded and installed the SDK. I opened "File->Import..." and 
found PDE import wizards. Mhm ... amazing. Then suddendly ... "activities". I 
remembered I read something in the release notes for M8.

I lost 30 minutes to think about why the preferences for PDE missing and what 
is wrong. I am an experienced user and a plugin developer and thus I finally 
figured it out. But what about others?

PDE activity should be enabled by default in the SDK. It is really important 
not to confuse new users. A user first downloading Eclipse SDK to start 
developing plugins doesn't know anything about activities. Nobody will 
understand why he has to enable something that he has already downloaded and 
installed.
Comment 1 Wassim Melhem 2004-03-29 01:58:16 EST 
>Today I upgraded to 3.0 M8 from I20040316. I had to throw away my workspace 
>because of start up problems.

Here is the announcement made to eclipse-dev a couple of weeks ago:

 "The Eclipse team understands the need to target various builds from the most 
recent Eclipse builds.  However, as we move closer to the end of the 3.0 cycle 
it becomes increasingly important that the various teams have the freedom to 
ready their code for shipping.  This generally means deleting function which 
is not going to be part of the final product.   

With that in mind, the Eclipse teams should not bend of over backwards to 
maintain backward compatibility for previous *3.0 milestones*.  For example, 
there is no guarantee that you will be able to use M8 to develop an M7 
target.  People needing to target M7 (or earlier), should use the appropriate 
build as their development environment.   

NOTE: We remain fully committed to backward compatibility for 2.x Eclipse and 
you are free to use any version of 3.0 to develop for 2.x targets or to run 
2.x plugins. "


Moving to Platform/UI to address the dissatisfaction with the activities 
support.

Comment 8 Gunnar Wagenknecht 2004-03-30 09:44:32 EST 
I'm not sure if this is a solution for the problem. Veteran users still will 
not use the next button. They will use the close button of the intro view.

Most applications ask such question during setup ... but Eclipse does not have 
a real setup part. Just unzipping and fine.

As long as the user can close the intro/welcome view without answering these 
questions the problem still exists.

The proposed solution is good if we yarn it to the end: extending the intro 
view to allow some kind of "setup" mode (steps that have to be done before 
Eclipse is usable). This must be extensible to allow clients to hook into (we 
also have a need for configuring some preferences after install/unzip before 
our plugins are really usable).

But I don't think there is enough time to get this into 3.0. Thus, I propose 
the following solution.

For 3.0:
Enable all activities by default in the SDK. Enable activities by default for 
features/plugins downloaded and installed via the Eclipse update system.

Post 3.0:
Consider the extension of the Intro view to provide a setup wizard. Several 
plugins can provide steps/task for this setup wizard that has to be performed 
before Eclipse is usable. Usually this is the first time Eclipse starts up with 
a new workspace. This setup wizard might be reinvoked later at any time my the 
user.
