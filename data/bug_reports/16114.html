Comment 11 Dejan Glozic 2002-06-13 20:16:14 EDT 
I think I know what the problem is, and it is not PDE:

This happens when a new build is used but old workspace is retained. When the 
platform starts, I tried to expand org.eclipse.ui.win32 and delete 
workbenchwin32.jar. It failed with the same error as reported by the Import 
wizard. Obviously, the problem is in some kind of a race condition probably 
caused by JDT whereby workbenchwin32.jar is still open when we try to delete it.

One important prereq for this but to appear is that there has to be cross-
reference between projects. Namely, org.eclipse.ui has workbenchwin32.jar in 
its classpath. This is because PDE must recreate run-time conditions as close 
as possible, and fragment libraries are added to referenced plug-ins by the 
plug-in registry during the registry resolve phase.

I tried to create a small test case with only one fragment and one plug-in and 
couldn't. There is something with the old workspace and new build that sets 
this problem off. It is possible that JDT is persisting some classpath 
information that is not present in the new workspace. Another theory is that
there are errors in metadata that accumulate and cause this that are not 
present in a fresh workspace. BTW, if you try to set up a fresh workspace, 
import binary projects, then try to delete workbenchwin32.jar, it will work.
It has to be a few drivers old workspace.

Moving to JDT for comments. As I said, use F3 build, point to an earlier 
workspace with binary projects and try to delete 'workbenchwin32.jar' in 
org.eclipse.ui.win32. It will fail to delete it. I traced it down to the 
File.delete() call, meaning that it something is keeping the file open.

JDT is never keeping a JAR opened for more than an operation duration. Any 
client might be in the middle of a JAR access when PDE discards it, and if 
clients are not taking workspace locks, then this kind of situation may occur.

The best would be if PDE was a little more resilient and offer to recover from 
these situations.

Comment 23 Dejan Glozic 2002-09-24 10:55:21 EDT 
Translations are delivered using fragments. When resource bundles for a plug-in 
are located in fragments, we are looking in the plug-in first, then in 
fragments to resolve the translated string. We (PDE) are simply recreating the 
run-time behaviour. We must support a scenario where both the fragment with 
translations and the plug-in are in the workspace. By commenting out 
the portion of code as shown above, we would not be able to support this 
scenario.

An alternative solution would be to force the garbage collection as soon
as plug-in model is removed from the list - that should run the 
finalizer and close the zip file.

Comment 25 Philipe Mulet 2002-09-25 04:55:40 EDT 
The Java specs clearly states that no assumption should be made about when 
finalizers run (2.17.7). The only assumption you can make is that it will be 
run when memory for this object is freed. Even a manual GC would not solve this 
issue, because it depends on the GC implementation to free this object or not 
at this very moment (also GC may take quite some time, and cause some IDE 
freeze). 

Again, why are you introspecting the workspace at all for this ? It seems that 
you should only show a picture of the imported plug-ins. The only usage I can 
imagine is a case where the plug-in is external, and the fragment is internal 
and not replaced, but in the case of fragments being available externally, 
internal fragments should be shadowed by external ones (names in internal 
fragment are likely obsolete). 

As a work-around : knowing the library limitation, your code should be more 
resilient and retry deleting fragment until you succeed.

Comment 27 Wassim Melhem 2002-10-02 01:01:33 EDT 
Fixed in the I-20021001 build.
Here is a brief explanation of what was causing the problem:
We were using ResourceBundle.getBundle(String,Locale,ClassLoader) to translate 
strings.  The class loader passed to the resource bundle contains references 
to workspace libraries among other things. When a project is being deleted 
from the workspace, we were making sure to dispose of the resource bundle that 
it was using.  However, the problem was that the ResourceBundle class 
maintains a "static" cache of bundles and the class loaders they reference.
It is the class loaders, maintained in the cache (whose initial size is 25 
according to API doc), that were keeping a reference to jar files we were 
trying to delete, thus preventing their deletion.

What we do now is:
1. Using a URLClassloader, we first get the resource file we're looking for as 
an input stream.
2. We create a PropertyResourceBundle using the PropertyResourceBundle
(InputStream) constructor.
3.  We use bundle from step 2 to translate our strings.

This way, we don't lose any functionality.  No resource loaders are cached, 
and we will never run into this problem ever again.
