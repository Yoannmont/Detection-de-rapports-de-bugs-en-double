Description Gunnar Wagenknecht 2004-08-30 08:30:31 EDT 
CLable defines hIndent and vIndent to a fixed value (3). This prevents the 
widget from fitting into the Forms UI because it is displayed with margin and

CLabel should have setters and getters for both values.

Comment 8 Carolyn MacLeod 2009-03-13 17:40:15 EDT 
Created attachment 128799 [details]
patch for CLabel

In CLabel, we need to allow setting to 0 as a possible value (that is what the bug report was requesting). I also did a check to see if the value is the same, to optimize not calling redraw in that case.

I looked at other "set indent" methods (TextLayout, StyledText, OleClientSite) to see what they wanted the maximum indent to be, and it seems that nobody checks for a maximum. So we don't need to here, either.

Since the call to checkWidget(); may result in an exception, I added the @exception to the javadoc for the set methods. I looked at some typical 'get' methods in the custom controls and it looks like if they just return a field, then they have a commented out line of code for //checkWidget(); so I added that.

Any time we add new API, we always put an @since in the javadoc. Unfortunately, we have just been told that it is now too late to add any new API to 3.5, so I just typed @since 3.6.  We will put the code in right after 3.5 ships.
Comment 9 Carolyn MacLeod 2009-03-13 17:54:39 EDT 
You will note that when you run this code and change the h or v indent using the Set/Get API button, the example CLabels may look clipped (maybe the icon does not draw or maybe the text is even clipped). What is happening here is that the CLabel has become to large to draw at the same size as before, so it needs to recompute its preferred size. To see this, click on 50 X 50 in the "Size" radio group and then click back on "Preferred". The control will resize to its preferred size and it will not be clipped.

There are other examples of controls doing this kind of redraw-but-don't-resize behavior. For example if you set the Text of a Label (in ControlExample), or set long item strings in a List or Table or Tree. The ControlExample and CustomControlExample are intended to show "raw" control behavior.  If you were writing a real application and you changed a control's text, or indent, you would then immediately call layout() on the control's parent (assuming the control's parent was sizing and positioning its control children in a Layout).  Or, if the parent was not using a Layout, the application would be responsible for telling the control to recompute its preferred size by calling widget.computeSize(SWT.DEFAULT, SWT.DEFAULT);
Comment 10 Carolyn MacLeod 2009-06-25 14:47:16 EDT 
I spoke with Silenio about this new API, and we decided that it would be more consistent with other controls to use the name "margin" instead of "indent".
This would also allow us to have right and bottom margins (which both default to 0 pixels for backward compatibility), which would be more flexible when putting CLabels in a form.

So the new API would be:
  public void setLeftMargin (int leftMargin) ... instead of hIndent
  public void setTopMargin (int topMargin) ... instead of vIndent
  public void setRightMargin (int rightMargin)
  public void setBottomMargin (int bottomMargin)

plus the corresponding getters. See StyledText for javadoc.

Can you please rework the patch to use this new API? Of course, you'll have to actually implement the new "right margin" and "bottom margin" feature, too, because they weren't there originally. You will probably want to change the existing field names for hIndent, vIndent, and INDENT while you are there, so that someone reading the code later won't get confused by multiple names for the same thing.

Also, please rework the CustomControlExample patch in comment 7 to use the new names (useful for testing). Thanks!

Comment 15 Lakshmi P Shanmugam(away until August) 2009-08-06 16:21:47 EDT 
Hi Carolyn,

When I tried to set the right margin & top margin to zero, I came across few problems:
1) The CLabel with the SWT.RIGHT alignment is not drawn correctly. This is because in the existing code hIndent is used as the right margin for a CLabel with SWT.RIGHT alignment.

2) Available width is not computed correctly. This is because we compute available width using 
	availableWidth = Math.max(0, rect.width - 2*hIndent)
If we have to use left and right margin then, 
	availableWidth = Math.max(0, rect.width - (leftMargin + rightMargin));
Setting right margin to zero gives a different value for available width.

3) ComputeSize doesn't return the correct size on setting bottom & right margins to zero.
 
Hence, to solve the above problems I have set the right and bottom margins to be same as the left and top margins. 
The patch takes care of backward compatibility. The CLabel with default margins looks the same as the exisitng CLabel.
In the existing code, the image or text in the CLabel is drawn in the center of the control, i.e, when the CLabel is drawn in its preferred size, we get equal margins on all sides. After applying the patch, the CLabel's default look will be the same.

I have tested it with the control example & the attached test-case.
