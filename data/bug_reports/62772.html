Description Brent Nicolle 2004-05-18 15:58:40 EDT 
On CDT 200405180809.

The new Eclipse build interface avoids the word Rebuild,
and instead now has a menu item Project > Clean...
This brings up a "Clean?" dialog, at the bottom of which
is a checkbox "Start a build immediately" (on by default).
So by default, I'd expect Project > Clean... to execute
a "make clean", then a "make all".
Or, simply a "make clean all".

In reality, if you look closely, it's doing a
"make clean", then a "make clean all".
In most cases it has the same effect, but it's less efficient
(particularly if you have a complicated clean) and it looks sloppy.

A minor problem for sure, but if someone starts thinking about
unifying the many different ways to build a Standard project, they
can add this to their pile.
Comment 1 Sean Evoy 2004-07-06 13:18:50 EDT 
I am not sure what else we can do. The semantics (from a builder's 
perspective) of a "Clean" is that the build system will deliver the clean 
notification via a method the builder subclass supplies. It is the builder's 
responsibility to do the right thing in response to this call. That may simply 
be erasing the build artifacts, or forgetting the last build state (thereby 
forcing a rebuild), or calling make with the argument to remove stuff. 

You may be getting a build notification if the auto-build is enabled (in which 
case it will deliver an incremental build notification immediately), or if the 
user has selected the check-box when auto-build is off. At that point, since 
you are essentially rebuilding, the builder is going to call 'make clean all'. 
I doubt that this is going to be a profoundly expensive clean snce the work 
has already been done.

There seems to be 2 scenarios where the "Clean" build is useful. First, with 
auto-build on, this is the only way to do a 'make clean all' and it works 
pretty well. Second, it is a good way to get rid of bad makefiles for the 
managed build system when you go and cancel everything in sight. Otherwise, 
with auto-build off, the "Build" and "Rebuild" items work better for this type 
of thing.
Comment 2 Doug Schaefer 2004-07-06 15:29:29 EDT 
I think we have a misunderstanding of the semantics of the IncrementalBuilder 
interface. FULL_BUILD does not mean "Rebuild". It means do a build and assume 
that the resource delta includes everything. This is in contrast to AUTO_BUILD 
and INCREMENTAL_BUILD which provide resource deltas (AUTO meaning automatic 
build versus INCREMENTAL meaning manual). As a side effect of using Make, we 
ignore the delta so I can see why we have the confusion.

To provide separate clean functionality, the platform team has added the clean 
method to the interface which is called when a CLEAN_BUILD is requested. This 
is then followed with FULL_BUILD call to the build method if a build is 
subsequently requested. The only other time a FULL_BUILD is called is when the 
project is first created.

So, to make a long story short, we should not be doing a 'make clean' on a 
FULL_BUILD. And, in fact, FULL_BUILD, INCREMENTAL_BUILD, and AUTO_BUILD should 
all do the same thing, i.e., run 'make all'.

Comment 17 Mikhail Sennikovsky 2006-02-14 12:31:39 EST 
Hi David,

I agree with your changes for the standard builder and partially for the managed builder as well.

I’ve got a couple of additional concerns:

1. The issue still remains unfixed for the managed projects, because the generated makefile builder currently still implements the programmatic clean for the full build. The programmatic clean is performed to avoid the problem with corrupted .d files that might appear if the previous build was cancelled. We also can not always do a simple “make all” for the managed project because sometimes, the full (or partial) project rebuild is needed because of the managed build settings change, though all resources from the make’s point of view remain up-to-date. So more complicated mechanism is needed to detect what exactly should be done for building based upon the managed build settings changes and resource delta.

2. My other concern is that the “rebuild” and “incremental build” concepts from the user point of view might differ from those from eclipse point of view.
That is why I think the “Rebuild Project” action in the project context menu should exhibit the “make clean all” behavior as was done previously. A user might be confused that when the project rebuild is requested, nothing really happens, or an incremental build happens. We might fix it by first invoking the project clean build and then the project full build for that action.
Note: this issue is currently relevant only to the standard make projects, since the managed make builder still implements the programmatic clean for the full build.
I also think that users might be confused when they request the full build, while the message displayed in the console for the managed project builds always says that the incremental build is running. I think we should remove the word “incremental” from the message.

The Build Description Model I mentioned in my previous comment is going to handle the point 1 mentioned above.
If everyone agrees with the point 2, we could implement and include it along with the changes made by David.

Any thoughts?

Thanks,
Mikhail
