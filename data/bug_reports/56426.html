Description Christophe Cornu 2004-03-26 17:15:47 EST 
I20040324

e.g.
- start eclipse with a workspace containing org.eclipse.swt
- open file Display.java from the java perspective. Do some changes to that 
file.
- Select the node "Display.java" in the Package Explorer. Right click > Team > 
Synchronize with repository

A "Synchronize CVS Workspace" dialog comes up asking me to select the 
ressources to synchronize. But I already choose that set by selecting nodes in 
the Package Explorer tree. Earlier builds did not bring the Synchronize CVS 
Workspace dialog and correctly brought the CVS Synchronize View for the 
selected file Display.java

In short, the following use case was excellent in earlier versions:
- from the java perspective doing real java work, synchronize a single file 
with HEAD and release it:
select a node or a set of nodes in the Package Explorer > team > synchronize. 
Rapidly Compare the contents in head and local, approve and commit.

The new Synchronize CVS Workspace dialog does not allow this use case anymore 
(or it requires extra work to reselect the node(s) already selected on the 
package explorer)). Let me know if I am missing another way to do this.
Comment 1 Jean-Michel Lemieux 2004-03-29 09:10:09 EST 
The nodes should already be selected in the table. The problem is that they
aren't revealed correctly. You can simply press the finish button and you have
pre-M8 behavior.

There are actually usability reasons for the dialog. I consider the 
workflow to be similar to search where the selection provides the 
scope cue but most often you want to synchronize more than simply the
selection. If you think about it, you usually synchronize a working set
instead of a single project/file. I was actually trying to get people away 
from using the context menu to start a synchronize, instead keybind the action 
and select the scope.
We could add the search scope to the menu but this would create a 3-level ctx 
menu, yerk!

It should reveal, that is a bug if it isn't. I'll keep this open to 
investigate the reveal.

Comment 2 Carolyn MacLeod 2004-03-29 11:57:03 EST 
I *often* select a single class and say "synchronize". I can't stand the new 
dialog in my face. I did know that you can just hit 'Finish' - you are correct 
that it doesn't show up properly, but it does synchronize the selected item
(s). Or it seems to, anyhow.

Perhaps the UI could be as follows:
1) if user presses the synchronize button, then bring up the dialog.
    1a) if there are no resources selected, set up the dialog whatever way you 
currently set it up for this case. (I don't know what that is because I never 
do this <g>).
    1b) if there are resources selected, then populate the dialog with the 
selected resources and set the 'Selected Resources' radio.

2) However, if the user chooses 'Synchronize' from the context menu (I assume 
you can tell the difference? If not, then the eclipse framework is bad) then 
you bring up the dialog ONCE. In the bottom left corner, have one of 
those "Don't show me this dialog again" checkboxes. Populate the dialog as in 
1b above. And the next time the user chooses 'Synchronize' from the context 
menu, they have the choice to not have this dialog that simply requires you to 
hit Finish.
Comment 3 Jean-Michel Lemieux 2004-03-29 12:11:33 EST 
I has already thought about your workflow but decided against it because I
wanted all the synchronize workflows to be similar and stay away from the
situation where invoking the same action from different places did different
things. With that said there is a way to see the changes for a single file via
compare with latest. This won't prompt.

I'm curious though why synchronizing a single file is interesting? It's actually
dangerous because you don't know what others have released that would affect
this change, and that is why if you are synchronzing to commit, then you should
minimally sync the project. If you are synchronizing to simply open the compare
editor to see the changes, then use compare with.

I slept on it... and agree with you now. My reason for changing my mind is that
in the Synchronize View there is a ctx menu action called synchronize, this one
doesn't prompt! 

Toolbar buttons prompt - primed with selection, and ctx menus don't.
