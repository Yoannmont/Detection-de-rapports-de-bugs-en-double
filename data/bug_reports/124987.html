Description Josef Haiduk 2006-01-24 06:32:16 EST 
Having a java bean referencing another one causes a web service generation (Button up Java Bean Web Service with Literal/Wrapped style) which does not map the generated wsdl file. This means you get answers not described by the wsdl file and can not validate against it.

bean 1: Deal.java
 private int article;
 private int quantity; 

bean 2; Order.java
 private String id;
 private String personId;
 private Deal[] deals;

Service Class: OrderManager.java
public Order get(String id){..}

After generation the wsdl file OrderManager.wsdl has these complex types (focus on the element named "item"):

<complexType name="ArrayOfDeal">
<sequence>
 <element name="deals" type="impl:Deal" minOccurs="0" maxOccurs="unbounded"/>
</sequence>
</complexType>
<complexType name="Order">
<sequence>
 <element name="item" nillable="true" type="impl:ArrayOfDeal"/>
 <element name="id" nillable="true" type="xsd:string"/>
 <element name="personId" nillable="true" type="xsd:string"/>
</sequence>
</complexType>

But the response looks like this (there is not element named item):
<getResponse xmlns="http://orders">
<getReturn>
<deals>
 <deals>
   <article>2</article>
   <quantity>2</quantity>
 </deals>
</deals>
<id>1</id>
<personId>1</personId>
</getReturn>
</getResponse>

You can have the source code for reproducing the case.

best regards
Josef Haiduk
Comment 1 Jesper Moller 2006-01-24 20:55:43 EST 
The problem lies in Axis (as examined with 1.2.1).
"item" is a default item name used deep inside the Java2WSDL code.

During WSDL generation, the following happens:

In org.apache.axis.wsdl.fromJava.Emitter.java, a call is made to
org.apache.axis.wsdl.fromJava.writeElementDecl with a itemQName == null, while generating schema for elements in WSDL parts. This is the root cause of the problem.

writeElementDecl delegates to createLiteralArrayElement (same class) using the passed itemQName for itemName, i.e. still null. Here, the fallback string "item" is used if the itemName is null.

At the call time, I assume the BeanSerializer does its thing, now knowing of the WSDL.

This being an Axis bug, how is it resolved for WTP, Axis 1.2.1 being phased out?
Comment 2 Chris Brealey 2006-02-10 08:41:54 EST 
Josef,
you will need to open this problem with the Axis runtime in Apache's JIRA bug database [1]. In Eclipse WTP we try to adopt Axis versions as regularly as we can. Next up is RFE bug 116308 in which we will upgrade to Axis 1.3 in WTP 1.5. If Apache reports this bug solved in Axis 1.3, then WTP 1.5 will pick up the fix, and if you need a solution sooner, you can replace the jars in org.apache.axis/lib by their counterparts from the Axis 1.3 distribution [2].

[1] http://issues.apache.org/jira
[2] http://www.apache.org/dyn/closer.cgi/ws/axis/1_3
Comment 3 Chris Brealey 2006-03-22 11:49:31 EST 
Closing. Apache Axis issue.
