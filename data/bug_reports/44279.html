Description Henning Riedel 2003-10-07 04:22:24 EDT 
In JDT when entering paths, usually some predefined values can be entered, 
like ${workspace_root}, ${project_root},... 
I'm not sure, if this works in CDT, but it could help to enter include paths 
alot. Instead of D:\Programme\eclipse\workspace\myproject\include, we could 
use ${workspace_root}/myproject/include and let Makefile generator decide, 
which path to take or, even better, write  
--- 
WS_ROOT   = <${workspace_root}-replacement>  
PRJ_ROOT  = <${project_root}-replacement>  
INCLUDES += $(WS_ROOT)/myproject/include 
--- 
into the makefile. 
 
Same for the compiler stuff. When selecting an compiler, why not write down 
the stuff in the based on the compilers base path. The user could select the 
base, where the compiler is installed, which is pretty much set in Unix/Linux 
environments as either /usr or /usr/local, but on Windows, it could be on any 
drive. The paths underneath are then almost the same. 
--- MinGW --- 
${toolchain_base_path} = C:\MinGW 
${toolchain_tool_prefix} =  
${bin_path} = ${toolchain_base_path}\bin 
${lib_path} = ${toolchain_base_path}\lib 
${c_include_path} = ${toolchain_base_path}\include 
${cxx_include_path} = ${toolchain_base_path}\include\c++ 
 
--- Cygwin --- 
${toolchain_base_path} = C:\cygwin 
${toolchain_tool_prefix} =  
${bin_path} = ${toolchain_base_path}\bin 
${lib_path} = ${toolchain_base_path}\lib 
${c_include_path} = ${toolchain_base_path}\include 
${cxx_include_path} = ${toolchain_base_path}\include\c++ 
 
--- ARM GNU --- 
${toolchain_base_path} = C:\arm-devel 
${toolchain_tool_prefix} = arm-gnu- 
${bin_path} = ${toolchain_base_path}\bin 
${lib_path} = ${toolchain_base_path}\lib 
${c_include_path} = ${toolchain_base_path}\include 
${cxx_include_path} = ${toolchain_base_path}\include\c++ 
 
--- Linux --- 
${toolchain_base_path} = /usr 
${toolchain_tool_prefix} =  
${bin_path} = ${toolchain_base_path}/bin 
${lib_path} = ${toolchain_base_path}/lib 
${sysinclude_path_c} = ${toolchain_base_path}/include 
${sysinclude_path_cxx} = ${toolchain_base_path}/include/c++ 
 
--- MS VC++ --- 
${toolchain_base_path} = "C:\Program Files" 
${toolchain_tool_prefix} =  
${bin_path} = ${toolchain_base_path}\bin 
${lib_path} = ${toolchain_base_path}\lib 
${c_include_path} = ${toolchain_base_path}\include 
${cxx_include_path} = ${toolchain_base_path}\include 
 
You see, you can almost all move around in your system, as long as you have 
one base-path to the toolchain. I _miss_ the FieldEditor to change the 
compilers path in the Preferences Pages for global default compiler and maybe 
the project based Project Properties Page (think about using arm-gnu-gcc, not 
located in /usr but maybe /usr/local/).

Comment 3 Sean Evoy 2004-04-07 16:07:32 EDT 
I have been wrestling with how best to do this going forward. The basic 
problem is that there are two kinds of symbols and includes paths for GNU 
projects (and I am assuming for other toolchains as well but there are likely 
exceptions). The first is the built-in set that the compiler uses regardless 
of what the user enters, and the second is the stuff the user does enter. The 
built-in parser needs the union of both sets to function correctly.

There are two possible solutions, and the suggestion above outlines one 
technique. The other is the way that symbols and paths are discovered 
dynamically for the standard build system (call 'gcc -v <dummy_file>' and 
parse the ouput).

The advantages of coding the paths into the manifest as variables is 
1. I can populate the IStringVariableManager with these settings and use them 
to resolve paths in the managed build UI and in the generated makefile 
2. The user can edit the settings if they are wrong (the manifest would only 
contain defaults) through the UI. 
3. Implementers have to use a consistent set of variables, not whatever they 
want.

The problems are 
1. We cannot reuse the same variable names for every target.
2. GNU tools use the host and version information as part of their path setup, 
that is to say that the include search path on Cygwin 
is /usr/include/c++/<version>/i686-pc-cygwin while on MinGW 
its /usr/include/c++/<version>/i686-pc-mingw and there is no way we can encode 
all the possible combinations.
3. This strategy is really only for paths. Symbols are also built-in and can 
change depending on what flags are used to build (at least for GNU).
4. When should the variable values be put into the IVariableManager? The 
trivial answer is "when the build model reads the manifest" (i.e. during 
project creation and editting the project build properties). But what if the 
user changes the compiler or what if the compiler gets upgraded underneath. 
Will the average user be willing/able to update these settings.

There is an IDynamicVariable type that can resolve the value of the variable 
at run-time and this might solve the problem. Someone supplies the dynamic 
variable resolver as part of a toolchain spec and it figures out things like 
the version and host for GNU tools for a specific project, for example, and 
does whatever magic is requried for toolchain XXX if there is no convenient 
flag to extract this from the compiler.

The second approach involves calling the compiler with the verbose flag, 
extracting the paths and symbols from the output, comparing it with the list 
of built-in symbols, and resolving the differences based on some heuristic. 

The advantages are
1. The toolchain implemeter can get away with no hard-coded built-in symbols 
or paths.
2. Different versions/host types will get captured automatically.
3. Differences caused by build flags will (theoretically) be captured.

The disadvantages are
1. For GNU tools, we know that the only difference in the search paths is 
based on the version and host type, so parsing the output of a build is a lot 
of work.
2. We have no solution for toolchains that do not support this type of output.
3. No build tool in the path, no way these can be resolved. Dumb user is not 
protected from themselves after all.

I am thinking that maybe we need a combination of dynamic variables and an 
automated way to resolve them. The variable names come from the manifest (see 
the suggestion above but add a version and host variable). The toolchain 
provider suplies a variable resolver that gets called to figure them out when 
the string substitution is performed. In the case of Gnu, the resolver 
calls 'gcc -v', which outputs something like:

$ gcc -v
Reading specs from /usr/lib/gcc-lib/i686-pc-cygwin/3.3.1/specs
Configured with: /GCC/gcc-3.3.1-3/configure --with-gcc --with-gnu-ld --with-
gnu-as --prefix=/usr --exec-prefix=/usr --sysconfdir=/etc --libdir=/usr/lib --
libexecdir=/usr/sbin --mandir=/usr/share/man --infodir=/usr/share/info --
enable-languages=c,ada,c++,f77,pascal,java,objc --enable-libgcj --enable-
threads=posix --with-system-zlib --enable-nls --without-included-gettext --
enable-interpreter --enable-sjlj-exceptions --disable-version-specific-runtime-
libs --enable-shared --disable-win32-registry --enable-java-gc=boehm --disable-
hash-synchronization --verbose --target=i686-pc-cygwin --host=i686-pc-cygwin --
build=i686-pc-cygwin
Thread model: posix
gcc version 3.3.1 (cygming special)

We get the host and version from that quite easily so the parsing is 
lightweight. The parser is happy. Any thoughts?

Comment 6 Henning Riedel 2004-07-08 04:38:19 EDT 
Well, as already stated in the newsgroup (and Sean did somehow agree), it would
be good to have at least some settings for toolchain paths. This way, having
multiple installations of different toolchains, they do not have to be set in
the global environment path and you won't need to have multiple Eclipse startup
links.

The paths to the tools can then be added at launch time of the build to the
environment and passed to a newly created shell (like passing a new env in
fork/exec|spawn). This will relief the user from having problems about which
tool has been started (think about several different versions of make).

Also, as I saw in the plugin.xml e.g. the hardcoded cygwin path, once you
provide a way for adding a toolchain and being able to change path settings of
the tools, such settings are easier to handle for the user than grepping through
the plugin.xml looking for paths and one can move its toolchain to its personal
likings (e.g. have cygwin on d:\devel\cygwin).
Comment 7 Sean Evoy 2004-11-01 15:00:38 EST 
I think that we have the outlines of a solution for this problem in the 
current MBS. The solution is to populate the built-in compiler settings 
dynamically (if possible) using the ScannerInfoCollector extensions that Vlad 
created for the SBS. If no dynamic provider is available, then the hard-coded 
values from the manifest are used.

The MBS in 3.0 will go further in supporting the parameterization of paths so 
we can use MACRO-like path definitions for both hard-coded entries in the 
plugin manifest and in the UI.

Just so we don't lose this PR, I am going to mark it as a duplicate of the 3.0 
discussion bug.
