Description David Whiteman 2001-11-09 16:02:58 EST 
If you add a paint callback to a widget and perform a GC.copyArea, then you 
will see a lot of flickering, since the copyArea is forcing another paint 
event.  I.e. this is an endless loop.  Pat Mueller says it's because you used 
the windows scroll method to achieve the copy area, and that scroll method ends 
up causing a repaint.

Here's an example that will illustrate the problem:
Comment 1 David Whiteman 2001-11-09 16:04:38 EST

import org.eclipse.swt.widgets.*;
import org.eclipse.swt.layout.*;
import org.eclipse.swt.events.*;
import org.eclipse.swt.graphics.*;

public class PR_5737 {
	Display display;
	Shell shell;
	
	public static void main (String[] args) {
		new PR_5737().run();
	}
	private void run() {
		display = new Display();
		shell = new Shell(display);
		shell.setText(getClass().getName());
		shell.setLayout(new GridLayout());
		shell.addPaintListener(new PaintListener() {
	      	public void paintControl(PaintEvent event) {
	      		GC g = event.gc;
				g.setForeground(display.getSystemColor
(SWT.COLOR_RED));
				g.drawRectangle(200, 155, 80, 80);
				
				g.setBackground(display.getSystemColor
(SWT.COLOR_BLUE));
				g.fillRectangle(220, 175, 40, 40);
				
				g.setForeground(display.getSystemColor
(SWT.COLOR_BLACK));
				g.drawLine(200, 155, 280, 235);			
						
				g.copyArea(200, 155, 81, 81, 90, 0);
				//g.copyArea(200, 155, 81, 81, -90, -10);	
	
					
				//g.dispose();
	        	}
	    	});
		shell.setSize(400, 400);
	      shell.open();
	
	      while (!shell.isDisposed()) {
	         if (!display.readAndDispatch()) display.sleep();
	      }
	
	}

}

Comment 3 Silenio Quarti 2001-11-15 12:13:16 EST 
This is the correct behavior. If you call
GC.copyArea(int, int, int, int) on a Widget, the
source rectangle will be damaged. You should not
call it in a SWT.Paint event or if for some reason
you need to do that, you should check the event
bounds and avoid copying the same area over and
over.
Comment 4 David Whiteman 2001-11-15 13:11:02 EST 
The problem with this is that AWT lets you call Graphics.copyArea() from a 
paint method.  If you run this in the JDK, you get *no* flickering.  Without 
SWT changing their approach, how do you recommend that we make this work?
Comment 5 Patrick Mueller 2001-11-15 13:38:52 EST 
To clarify, we aren't calling GC.copyArea from WITHIN SWT Paint event, but 
actually, in response to one.  AWT transports most of the SWT events into AWT 
events to be run in the AWT event queue.  Which makes it all the harder to 
determine this case.  

I don't understand why copyArea should force a paint.  It's more like a 
drawImage() than anything else, isn't it?  I mean, if it was called moveArea() 
I could see that, but it's copyArea().  Smells a little fishy to me, like 
someone wanted to make a canvas scrolling algorithm a little simpler, or 
something.

If the SWT copyArea() can't be changed to not force a paint, then I think the 
only answer is that we implement AWT's copyArea to:
   - create offscreen image of desired size
   - copyArea source into offscreen image
   - drawImage the offscreen image into the copyArea target
which will nullify any of the speed you would expect out of a copyArea (ie, 
blt) type of API.

If I run the follwing class under IBM Hursley
JRE/AWT. When I press the button "Copy Area",
the paint method of the Frame is called. The clipping

import java.awt.event.*;
import java.awt.*;

public class FramePaint {

public static void main(String[] args) {
	final Frame f = new Frame() {
		public void paint(Graphics g) {
			System.out.println("PAINT=" + g.getClipBounds());
		}
	};
	f.setLayout(null);
	Button b = new Button("Copy Area");
	b.setBounds(30, 30, 130, 30);
	f.add(b);	
	f.setBounds(0, 0, 600, 300);

	b.addActionListener(new ActionListener() {
		public void actionPerformed(ActionEvent e) {
			Graphics g = f.getGraphics();
			g.copyArea(300, 0, 100, 100, 100, 100);
			g.dispose();
		}
	});

	f.setVisible(true);
	f.addWindowListener(new WindowAdapter() {
		public void windowClosing(WindowEvent e) {
			System.exit(0);
		}
	});
}
}
Comment 7 David Whiteman 2001-11-26 12:55:30 EST 
When I call copyArea() from an overridden Component.paint(Graphics) method in 
JDK AWT, I do not get the paint() method called again.  I'm not sure why it's 
different for doing so from the action listener.  Please run the following and 
see if you see the same results that I did.  I ran this against the IBM JDK 
1.2.2 from Hursley and the 1.1.8-based Microsoft VM (jview.exe).  This is 
basically the equivalent of the SWT-based testcase I originally submitted, so I 
would love to know how to use SWT to achieve the same effect.

import java.awt.*;

public class CopyAreaTest {
	public static void main(String[] args){
		Frame f = new Frame("Test Frame") {
			public void paint(Graphics g) {
				System.out.println("paint called");
				g.setColor(Color.red);
				g.drawRect(200, 155, 80, 80);

				g.setColor(Color.blue);
				g.fillRect(220, 175, 40, 40);

				g.setColor(Color.black);
				g.drawLine(200, 155, 280, 235);			
		
				g.copyArea(200, 155, 81, 81, 90, 0);
	
				g.dispose();
			}
		};
		f.setBounds(0,0,400,400);
		f.setVisible(true);
	}
}
Comment 8 Silenio Quarti 2001-11-26 14:19:28 EST
