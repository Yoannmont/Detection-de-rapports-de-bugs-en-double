Description Michael Valenta 2005-02-14 13:21:00 EST 
Currently, when the RelEng tool plugin changes, I have to send out a message 
to let people know they need to get a new copy. It would be nice if this were 
automated somehow? 

Here's an idea from Dorian:
---------------------------
An alternative is to create a feature for the releng tool, and have eclipse be 
set  to automatically search for updates, on startup, once a day, etc. Or we 
can kick off an update process every once in a while. 
There is still the initial overhead of setting this up, but once in place, 
there shouldn't be too much effort to update. 
-Dorian

Here's my response:
-------------------
This sounds like it may have potential. It sounds like it would still depend 
on the plugin being versioned manually every time it changes but that's 
acceptable. I seem to recall that update sites do not survive a reinstall(i.e. 
swithcing to the next I build). Is this still true? If so, this would prevent 
us from moving forward on this. Also, would it be possible to have the update 
search triggered by an itegration build upgrade of Eclipse? This would be 
ideal.
Comment 1 Michael Valenta 2005-02-14 13:24:20 EST 
From Pascal (for dealing with versioning):
------------------------------------------
Set the version of your plugins to be 1.0.0.qualifier (change the first 3 
digits to reflect the reality), add in the build.properties: 
        qualifier  = context 

Change the feature referring to your plugin to use 0.0.0 instead of the 
precise version number. 

From now on, the version number of your plug-ins will be qualified with the 
CVS tag where the code is coming from.
Comment 2 Dorian Birsan 2005-02-14 19:09:52 EST 
There are a couple of ways (with pros and cons) for dealing with build to 
build updates:

1. install the releng tool, as well as other plugins into an extension 
location that you add to the new build every time you update it

2. install releng tool to another location, but if you keep the configuration 
in a well known place and start eclipse -configuration path_to_config you 
don't have to add it again. This is ok, but there are occasional 
osgi/runtime/configurator changes that may require passing -clean as well. 

3. if releng tool is available from eclipse.org update site (or some discovery 
url contributed by sdk features), you can install it fresh every time you 
install a new build.
Comment 3 Michael Valenta 2005-02-15 09:13:04 EST 
The truth is that I never have a problem because I download the RelEng Tool 
every time I download a new build. The problem is that not everyone is willing 
to do that. Each of the options you have provided requires some sort of manual 
action on the part of the developer. Obviously, the best solution would not 
involve manual developer intervention on each reinstall (as I alluded to in 
the original description above). If this is not possible, then I suspect the 
next best would be a single step. My feeling is that each of the options you 
have provided, on their own, would not be adequate because of the amount of 
manual intervention (i.e. they would be more work than just downloading and 
installing the plugin manually). I think a combination of 1 and 3 might do it. 
That is, have the RelEng tool on an update site and use an extension location 
that you add each time you install a new build. However, I think this would 
only work if the feature performed an automatic check for updates when it was 
installed. Is this possible? Without it, it becomes a multi-step process 
which, again, is more work than just downloading and installing the plugin 
manually.
Comment 4 Dorian Birsan 2005-02-15 09:55:14 EST 
Automatic update searches are disabled by default. However, if the workspace 
is shared across builds, then once you enable it, the preference is saved, and 
the new build will just pick it up.

Another solution is to simply ship the releng tool as part of the sdk during 
the development cycle, and only remove it before eclipse GA.
Comment 5 Dani Megert 2005-02-15 15:21:55 EST 
I'd also prefer to have it within the SDK (or call it SDK-dev) build since we
get the builds pushed to our local server (so that a build is not fetched by n
people but only pushed once).

If this is not possible I'd like to get notified on platform-releng.
