Description Ed Burnette 2003-11-14 18:27:47 EST 
When checking out files from CVS, the client currently always uses the lineend 
characters of the platform: CRLF on Windows and LF on Unix. This needs to be 
under control of an option. For example, the option could have these choices:

   - Default style for the platform (what it currently does), or
   - As stored in the repository.

Some tools I've seen add two additional options:

   - Force CRLF (Windows style), and
   - Force LF (Unix style),

but it may be sufficient just to do the first two.

Use case: We develop C code for both Windows and Unix machines. Our desktop 
environment is Windows so that's where we run Eclipse. However the file system 
used for projects is a networked storage server visible from both Windows and 
Unix. When performing remote compiles on the Unix machines, the compilers 
complain bitterly about the extra CR characters. We could convert the line 
endings one file at a time after checkout but this is too painful. Since 
Eclipse's editors are happy with Unix line endings, and the files in the 
repository are stored with Unix line endings, and the Windows compiler will 
accept that too, we'd like to just avoid the carriage returns altogether. This 
should make checkouts a little faster too since it won't have to do that extra 
filtering step to insert the CR's.

Another bonus would be better interop with those developers using Unix tools. 
Currently it's harder to diff files I checked out on Windows with files they 
checked out on Unix.

I'm willing to contribute code to help implement this if needed.
Comment 1 Michael Valenta 2003-11-21 12:24:28 EST 
So what you are saying is that you would like some way to indicate to Eclipse 
that all files that reside on a particular drive (or live inside a particular 
set of projects) should use a particular line ending. Furthermore, you would 
like all editors and other tools that create new files or modify existing 
files to respect this setting. 

I understand that you get this for free when editing existing files that 
already have the desired line ending since the Eclipse editors respect line 
endings that are already in use in a file. However, I suspect that fixing this 
for CVS would not be enough. Other tools that create new files (e.g. editor, 
language plugins) or modify files (language plugins) would also need to 
respect the setting.

It seems to me that what is required is 

1) a workspace or per-project setting that indicates the line ending to be 
used by that project (along with UI to change the setting). 
2) any tools that create or modify files would need to use this setting. 

Point 1 may be able to leverage what comes out of the 3.0 user settings work 
(bug 36965). However, point 2 is complicated as it effects all plugins that 
create and modify files. Fixing it for CVS and other plugins that live in the 
SDK would be fairly straight forward. However, this new API would need to be 
communicated to all downstream plugins (JDT, CDT, etc.)

There is currently no plan to address this in 3.0 but if you are willing to 
contribute the code, then it would be helpful in making this a reality. I'm 
not saying that you have to implement the full solution mentioned above. If 
you want to start with just the CVS portion that's fine. However, that 
wouldn't solve the entire problem (although it may be enough for you).
Comment 2 Ed Burnette 2003-11-24 11:28:03 EST

Comment 4 Ed Burnette 2003-11-24 15:06:33 EST 
Created attachment 6938 [details]
Patch to the CVS provider to respect the line termination option.

This change will cause the CVS client to only insert carriage returns if the
current platform is a Windows machine AND the line ending option is set to
Default.

One subtle change is that it will ALWAYS remove carriage returns in text files
whether or not we're on a Windows machine, and no matter how the line ending
option is set. The unpatched code was a little inconsistent about this (look at
normal fetches vs. keyword change), and I think it's better to just
consistently remove them to prevent them from ever getting in the CVS
repository. If the user wants CRs in the repository he or she should use the
binary option.
Comment 5 Michael Valenta 2003-11-24 16:55:28 EST 
The stripping of CRs will need to be removed. We used to always strip CRs but 
this caused problems for users who run on Linux clients and Linux servers (and 
never use Windows) so we took it out. We learned that there are quite a few 
Linux users out there who have binary files that are not properly tagged as 
binary. Performing a checkout with the command line client does not corrupt 
their file but if Eclipse removes CRs then performing a checkout using Eclipse 
may corrupt their improperly configured binary files. We prefer not to impact 
clients in this way so we removed the automatic correction of text files. 
Keyword change is different since the user has explicitly requested the 
correction.

Comment 7 Michael Valenta 2003-11-25 10:57:33 EST 
I've looked over the patches in more detail and have the following comments:

1) In CVSTeamProvider, the following constant is defined:

final boolean useCRLF = IS_CRLF_PLATFORM && (CVSProviderPlugin.getPlugin
().getLineendOption() == Command.LINEEND_DEFAULT

This should not be a constant as the expression needs to be rechecked each 
time the keyword change method is invoked (in case the user changed the 
preference).

2) The patch uses a combo box to choose between "default line ending" 
and "repository line ending". I don't know if this is clear enough for most 
users. Why not use options that are more explicit (e.g. "LF (Unix, Mac OS 
X)", "CRLF (Windows)") with the platform default selected by default (I got 
the option text from the Source/Convert Line delimeters menu and I think it 
would be better if we used the same options to be consistent).
Comment 8 Ed Burnette 2003-11-25 16:43:19 EST 
Hmm, I don't think useCR is contstant, because I didn't say "static". It 
should be reevaluated every time the run() method is called. Or am I missing 
something?

I copied the combo box text from another product. Feel free to reword the 
text, however I think you should keep the meanings the same. Why? So that in a 
heterogeneous environment like ours, Unix and PC users can share the same 
option settings (and thus the same documented instructions). In other words, 
the options need to still mean "use the default platform lineend characters" 
and "use what's in the repository without changing it", and the default can be 
the first one regardless of the machine type. The usefulness of this 
outweights consistency with the edit actions.
