Description Vadim Berestetsky 2002-12-09 18:22:54 EST 
I would like to propose the following suggestion for the launch configuration 
API change:

Tab group owners should be notified whenever a 'Debug' or 'Run' button is
pressed. This will give them a chance to perform final checks before continuing
with the launch. Some checks that require long time to run are not practical to
perform when corresponding control or even entire tab is updated. This 
notification should be 'unique' in a sense that they should know that this 
is 'a final' check before.

Usage Scenario:
For example, if one needs to verify a process ID to debug from a remote 
machine. This could be a very expensive operation. It may not be practical to 
perform the validity check every time user changes the value. Farthermore, it 
may not even be practical to make a validity check every time performApply() is 
called. This is a very good candidate for the 'final check' API.
Comment 1 Darin Wright 2002-12-09 19:20:38 EST 
I see a problem with this: the tabs are not always realized when launching. For 
example, when re-launching, or launching a history item, the associated launch 
config is simply launched (no tabs). Then, where does the verification occurr?
Comment 2 Darin Wright 2002-12-10 10:51:56 EST 
PS. The tab group is notified when the config is launched via 
ILaunchTabGroup#launched(ILaunch). However, this is only called when the dialog 
is realized.
Comment 3 Vadim Berestetsky 2002-12-10 11:06:06 EST 
ILaunchTabGroup#launched(ILaunch) does not seem to allow for reporting errors 
back to the Luanch Configurations framework. Namely, if validation performed 
within this method fails, there does not seem to be an obvious way to keep LCD 
up. Is there?

The verification method that I propose is really ment for the verification of 
the UI controls altered by the user. To provide a way to keep the LCD up if 
when 'Debug' or 'Run' buttons are pressed, there are some errors in the UI that 
require an 'expensive' check. In a way, you can re-use performApply() method, 
but indicate via a parameter that this is a 'final' call to performApply() 
prior to calling the delegate.
I don't think re-launching, or launching a history item require such 
verification since they don't put up an LCD.
Currently, an alternative is for the delegate to perform such check and force 
LCD up if check fails. I am cetainly open for other suggestions.
Comment 4 Darin Wright 2002-12-12 15:08:30 EST 
I have the following question:

Does this mean that the delegate will no longer perform "final verification"? 
If this is true, then a new API in the LCD could be appropriate. However, if 
the delegate still needs to perform this "final verification", then it seems 
that the design works. If a client is launching a config programatically, where 
does this verification occurr, if not in the delegate?

NOTE: When a client makes a call to open the LCD, and one is already open, we 
currently re-use the same LCD (i.e. we do not open a new one).
Comment 5 Vadim Berestetsky 2002-12-19 16:41:58 EST 
If the 'final verification' indeed needs to occur in the delegate, what is the

failed?
Comment 6 Darin Wright 2003-01-03 11:28:03 EST

If my delegate fails the 'final' check, it can throw a CoreException, which 
contains an IStatus object (which can represent the actual error message).
As a result, an LCD will come up and display the error message that IStatus 
object contains. Is my understanding correct?

If that is the case, I wonder if this is available in ECLIPSE 2.0.2 as we are 
using 2.0.2 for this release. Also, I wonder if this behavior should be 
documented in your 'Launch Configurations' article - greate palce to start with 
launch configs by the way.

One more question: which tab will display the error in that case? Typically, it 
is up to isValid() API of the individual tab to indicate the error message and 
to indicate which tab is at falt.

Thanks.
Comment 8 Darin Wright 2003-01-03 13:45:09 EST 
The LCD will display the error message in a dialog. The error is not associated 
with a tab. This will also work in 2.0.2.
Comment 9 Vadim Berestetsky 2003-01-06 09:59:02 EST 
If my delegate throws CoreException, I get the 'Launch Configuration Error' 
dialog, which states that 'Exception occured while launching configuraiton. See 
log for more information.' Then it actually shows my own message.

I don't think this is a very user-friendly way of handling errors that are 
detected by the 'final' check in the delegate. The kind of errors that I detect 
this way would include:
- Specified process does not exist on the remote host
- Specified command to start user applicaiton does not exist on remote host
- etc.

In addition, LCD displays the first tab of the launch configuration and there 
does not seem to be a programatic way to switch to the tab that is at fault.

I believe that a proper, user-friendly way to handle these errors would be to 
provide a proper error message (preferably inside the status line in the LCD) 
and clearly indicating which tab is at fault.

Having the 'final check before leaving the LCD' API will eliminate all of the 
above problems.
Comment 10 Darin Wright 2003-01-06 10:52:08 EST 
This brings me back to my original criticism: if the 'final verification' is 
only done by the tabs/tab group, then when happens when a launch config is 
launched programatically? I.e. how does a launch config implementor 
ensure 'final verification' is done when a launch config is launched from the 
history, and how is the error reported in such a case. As well, the error 
handling/reporting should be handled the same in both instances (whether the 
dialog was open or not).

This leads me to another approach. I believe the feature you are requesting is 
the ability to display an error message in the LCD, and display a specific tab. 
So what if we added API to the LCD (which is a non-breaking change, since the 
LCD is read-only - i.e. not to be implemented). I would suggest adding a way to 
set the launch config being displayed (actually, you can already do this by 
opening the LCD on a specific selection), and setting the active tab. (Setting 
the error message is not neccessary, since this is done by the tab).

I would then suggest the following approach:
* the launch delegate performs final verification
* if an error is detected, it opens the LCD on the config (which will re-use 
the current LCD if already open)
* the launch delegate sets the active tab, and then sets the error message 
associated with the tab
* no exception is thrown, since the exception/error has been handled

This has the benefit of working in both launch scenarios - i.e. from the 
dialog, and programatically. No new API needs to be added to the tabs (which 
would be a breaking change - i.e. an addition, unless we created a new 
interface, which was optional to implement). As well, the way in which the LCD 
is opened is up to the implementor. Generally, launch delegates are not 
encouraged to know about (or prereq) the debug UI. Thus, the launch delegate 
could use a status handler extension (see IStatusHandler) to handler the UI 
issues if required, and throw an exception if there is no status handler (thus 
separating the presentation issues from the delegate).
Comment 11 Vadim Berestetsky 2003-01-06 13:31:52 EST 
I think this error handling approach will work well. There are two points that 
I would like to clarify though:
- is there already a way for launch delegate to set the active tab?
- launch delegate setting error message for the tab: Typically, it is up to the 
isValid() method of the tab to detect whether all values in the tab permit 
launching and to set the error message. Because isValid() method can not 
perform the 'final check', it is really the delegate who needs to 
somehow 'communicate' this error message back to the tab. What do you think the 
best way of achieving this would be?
In addition, we will also need a way to re-set the error message back to null 
once the user makes an update to the control in error.
Thanks.
Comment 12 Darin Wright 2003-01-06 14:05:50 EST 
* There is not yet API to set the active tab. This would be a new addition to 
ILaunchConfigurationDialog. I would propose something like #setActiveTab
(ILaunchConfigurationTab tab) or perhaps #setActiveTab(int index). An index may 
be simpler, as otherwise the caller would first have to retrieve the 
appropriate tab from the launch dialog.
* As for setting the error message, my initial thought on this was that it 
would be up to the implementation of the delegate and tabs. For example, the 
delegate (or status handler) and the tabs usually have an "internal" 
relationship. Thus, an implementation could add methods for setting/getting 
a "final verification" error message. The #isValid() method would need to take 
this into account when setting the error message.

For this to work, the delegate (or status handler) would first retrieve the 
appropriate tab (via #getTabs()), set the error message with an internal API 
(i.e. the delegate knows what class the tab actually is, and may cast to the 
class), and then set the active tab (which will have the side effect of 
updating the error message). 

This avoids adding extra API to set the error message. As well, this leaves the 
knowledge if clearing the error message up to the tab.

This leads me to another question. If the tab is displaying an error, how does 
it know when the error is fixed, if the "final verification" is in the delegate?
Comment 13 Vadim Berestetsky 2003-01-06 15:17:36 EST 
That is a VERY good question. Probably, the most difficult one to answer too! 
Obviously, the tab has to defer the 'final re-verification' to the delegate as 
well. The only reasonable approach I could think of is to consider an 
error 'fixed' any time control in error gets updated. This would not guarantee 
that the error is fixed correctly, but at least the user made an effort to 
correct the value that caused the original problem.
Comment 14 Darin Wright 2003-01-07 10:06:52 EST

Comment 17 Darin Wright 2003-01-14 11:38:10 EST 
Even if opening to the proper tab, there is still an issue of communicating 
with the tab to set the error information properly.

I see another "opportunity":

When a launch is performed, and an exception is thrown the dialog currently 
displays the exception & status in an error dialog. We could add a hook to 
check if there are any IStatusHandlers for the associated status. If a status 
handler exists, it could be consulted to handle the error via the #handleStatus
(IStatus status, Object source) method of IStatusHandler. The "source" passed 
into the status handler would be the launch dialog. This allows the active tab 
and error info to be set. If there is no status handler, the default error 
dialog is displayed.
Comment 18 Darin Wright 2003-01-14 11:42:32 EST

Comment 19 Darin Wright 2003-01-14 11:46:56 EST

Comment 20 Darin Wright 2003-01-14 13:22:38 EST 
Released changes to HEAD.

If a launch delegate throws an exception during the launch, the launch dilaog 
looks for a status handler for the exception. If one is found, it is consulted. 
Otherwise, an error dialog is displayed.

If a launch delegate throws an exception when being launched from the history 
(i.e. via the method DebugUITools.launch(config, mode), and a status handler 
exists for the exception, the dialog is opened on the status. Otherwise, an 
error dialog is shown.

I believe this works nicely, as existing implementations do not change. 
Implementors whishing to add function, only need to declare status handlers 
(which also allows launch delegates and UI status handlers to be defined in 
different plug-ins).
