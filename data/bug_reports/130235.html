This could be added as an extension to the new C99 parser. Our main goal with the C99 parser is make it easy to add support for new syntax. You could create a new parser with the additional syntax support by extending the base parser. Take a look at this doc, it describes how support for UPC was added to CDT by extending the C99 parser: http://wiki.eclipse.org/CDT/designs/C99_and_UPC_Parser_Overview.

Keep in mind though that the C99 parser is probably going to change quite a bit for the next release of CDT, and that we don't yet have C++ support (we're working on it).

Comment 13 Richard Gobeli 2008-06-05 17:04:46 EDT 
(In reply to comment #12)

> To make a decision on this I need some kind of specification for that. A link

> * does your compiler accept spaces between '/' and '%'?

I don't have any of the original documenatation of the original Mac compiler.

The /% %/ can be used basically anywhere on a full line to insert a line or more of assembly code or assembly declarations. There should not be any spaces between the '/' and '%'.

This code acts like a comment for the c compiler, but the compiler copies the text between the /% and %/ to the assembly output file.
It also copies the /% and %/ to the assembly file, but makes them a comment on a separate line in the assembly file by adding a ';' before each one.

It does not work in preprocessor c code such as #define as you asked.
It will insert one line of assembly for each line of code between the /% and %/ statements. When used the line of assembly needs to be a valid assemble code line. Therefore c preproccessor code does not work.
It will work anywhere the __asm__( ); line will work.
It will work in file scope and function bodies.
I have replaced most of the single line asssembly code with __asm__ ( );, but our compiler does not support multiline __asm__ statements like below.
__asm__( code    \n
         code    \n
         code    );

Therefore the /%  %/ was used for multiline assembly code like the multiple fuction below in the examples.

Examples below:
Insert module headers:

/%	endsec					%/ /* end a assembly section /*
/%	section	sec_math_0		%/         /* start a new section */

__asm__( "endsec" );                    /* same as above using __asm__() /*
__asm__( "section sec_math_0" );


/%	org		p:$AEB	                   ; assembly comment set origin
	addressing	short                      ; set addressing mode
%/

/%                                                 ; include an assembly file
include 'onchip.i'
%/

/% include 'onchip.i' %/



Whole functions:

#if 0
/*********************************************************************************
**
**  lmul_l_s
**
**  returns
**	The long result of a long by short multiply.
**
**  description
**
*********************************************************************************/

long lmul_l_s( long x, short y )
{
  return ( x * ( long )y );
}
#endif

/%
; note : Must not modify r3, r4, and r5 !

]Flmul_l_s
	global	Flmul_l_s
]Fx_lmul_l_s
	global	Fx_lmul_l_s

	move	Y:-(r6),x1	; lhi
	move	Y:-(r6),x0	; llo
	move	Y:-(r6),y0	; s

	mpysu	y0,x0,b		; s * l lo
	move	b0,a0		; save low product
	dmacss	y0,x1,b		; high product + ( s * l hi )
	move	b0,a1		; save high product
	move	b1,a2		; sign extend
	asr	a

	rts
%/



Anywhere in c code:

    if ( controller_dc_arm_no_field )
    {
      if ( enable )
      {
	/%	bset	#DCR_ARM_ON_BIT,y:DCR_DEVICE_READY_REGISTER	%/
      }
      else
      {
	/%	bclr	#DCR_ARM_ON_BIT,y:DCR_DEVICE_READY_REGISTER	%/
      }
    }

In the function body:

long lmul_l_s( long x, short y )
{
  /%
  assembly code here
  %/
}

Define assembly macro to be used later:

#define DSV_MAX_COUNT_INT		65536

/%

]msb_macro	macro		VALUE,REG

	bclr	#7,REG
	btst	#8,VALUE
	jcc	_msb_macro_0
	bset	#7,REG

]_msb_macro_0

	endm
%/

/*********************************************************************************
**
** global defines
**
*********************************************************************************/


I hope this will be enough information.

Thanks,

Comment 14 Markus Schorn 2008-06-12 09:53:53 EDT 
I think I could add a option to the preprocessor to treat /% %/ as multi-line comment. I can't make this work by default (for most users this is invalid syntax), you would need to define your own language to turn on the extension. 

To define your language you'd need to derive from GCCLanguage or GPPLanguage and provide a different scanner-extension configuration (derive from GPPScannerExtensionConfiguration or GCCScannerExtensionConfiguration). Once the language is provided you can select it in the preferences: C/C++ - Language Mappings.

Would that work for you?
