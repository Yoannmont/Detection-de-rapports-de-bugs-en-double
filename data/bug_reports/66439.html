Comment 3 Nick Edgar 2004-06-10 12:08:27 EDT 
I suspect something like the following, e.g. if B in ABD is pasted to C

- package explorer's paste does a refresh instead of an add
- the tree viewer's updateChildren gets ABCD as the list of children, so the
TreeItem previously used for D gets reused for C, and a new item for D gets
added at the end
- the label update for C talks to the decorator manager, passing the old text "D"
- DecoratingLabelProvider.updateLabel gets decorationReady = false, so does not
update the text in the label
- the update doesn't come in later for some reason
- the label update for D works, since the item had no previous text
- the resulting labels are: ABDD

Comment 12 Nick Edgar 2004-06-11 09:52:45 EDT 
This sounds like a symptom of the same problem.  When the viewer refreshed, it
reused the first N TreeItems for the remaining elements, but their labels were
stale.  The decorator manager was notified, but took a while to respond.

The problem is that the DecoratingLabelProvider is being asked to update the
label, but is being given the wrong original text and image for the element
because the TreeItem has been reused.  This violates the assumptions of
IViewerLabelProvider.update.

Should address this for RC3 by ensuring that we clear the text and image of an
item when it is reused for a different element.
Comment 13 Tod Creasey 2004-06-11 10:12:23 EDT 
Marking for RC3 - I think it is only in one place in the code where this 
happens which is why it is hard to replicate.

This is even more important if the decorators get blocked by some PDE 
operations
Comment 14 Tod Creasey 2004-06-11 11:32:20 EDT 
I think I have found where this can happen.

When internalRefresh is called in the AbstractTreeViewer there is a test to 
see if there is a disconnect between an element and the item it is mapped to - 
if there is there is a refresh.

Previously this was fine as the update was always forced but now we have a 
potential delay by the decorators that can put it out of sync for a while. I 
think this the the CVS decorator blocked issue as once the decorators return 
this will be OK.

The second case happens if updateLabels is not true - we can swap in the 
newElement but we do not neccessary update the labels afterwards. We needed to 
handle the case where the label updates were false but the update still 
occurred.

I will attach a patch to review with Nick during the RC3 cycle.
