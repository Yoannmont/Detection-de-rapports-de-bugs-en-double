Description Erich Gamma 2002-03-14 02:54:14 EST 
The JavaEnvironment tab has a hard dependency on the JavaMainTab to find the 
JavaProject. This means it can be reused for other Java launchers that use a 
different main tab class (e.g. the JUnit plugin). Currently there is no 
default class path when you just contribute the Java Environment tab.

The deeper problem here is that there is no way to set defaults for a launch 
configuration at a more global level. Currently defaults can only be set at 
the tab level. To take the Wizard analogy again, there you can store and 
set "wizard global" state in the Wizard class itself. One solution would be a 
launch Config UI class that can be contributed via the extension point as 
well. It would know which tabs the launch config requires and would add them 
in code directly. So instead of contributing individual tabs you would only 
contribute a single LaunchUI class.
Comment 1 Darin Wright 2002-03-14 09:31:28 EST 
Agreed. This is a problem that we need a solution to.
Comment 2 Darin Wright 2002-03-14 14:44:23 EST 
I propose to add a "tab group" extension point, and a corresponding 
ILaunchConfigurationTabGroup interface. The tab group will be registered for a 
type of launch configuration, and will implement the following methods (debug 
will provide an abstract implementation that does everything except create the 
tabs specific to the group).

#createTabs(ILaunchConfigurationDialog) - instantiates this tab group's tabs, 
and sets their associated launch config dialog (but does not create their 
controls)

#getTabs() - returns this group's tabs

#createTabControls(Composite parent) - calls #createControl(parent) on each tab

#dispose() - disposes this group and tabs

#setDefaults(ILaunchConfigurationWorkingCopy) - initializes the given config 
with default values. The abstract implementataion simply delegates to tabs.

#initializeFrom(ILaunchConfiguration) - initalizes each tab with values from 
the given configuration. The abstract implementation delegates to tabs.

#performApply(ILaunchConfiguration) - ...the abstract impl delegates to tabs

#launched(ILaunch) - ... the abstract impl delegates to tabs


I propose we do this post M4, as the schedule is too tight before M4.

Comment 6 Darin Wright 2002-03-18 15:31:19 EST 
Comments from Matt Lavin

General Problems 
1.  Changes on one page do not get reflected onto another page.  For example, 
if I attach the JavaArgumentsTab and a different tab that analyzes the 
arguements to display some things to the user then if the user makes a change 
in the Arguements tab, my custom tab does not get notifed.  I personally think 
that this is a pretty simple problem to solve since they concept of a 
workingcopy already exists.  Right now the page is intialized from the 
LaunchConfiguration one time and the changes in the page are only applyed at 
the very end.  If the initializeFrom method was changed to use a 
ILaunchConfigurationWorkingCopy instead of  a ILaunchConfiguration, and the 
method was called each time the tab was shown then one page could effect other 
pages.  Another change that would have to be made would be to call performApply 
when each tab is switched away from.  A properly written tab should handle this 
already. &! nbsp;I think that these changes would be very minor since the 
ILaunchConfigurationWorkingCopy is actually a sub interface of 
ILaunchConfiguration.  The methods on ILanchConfiguationTab would be more 
consistant since each method would have the same paramater. 

2.  There is no general way to initialize the values of a page to what you want 
them to be.  For instance, the java pages all populate there settings based on 
the currently selected project, but if I want to override the settings I have 
to subclass the tab and override the setDefaults method.  Some initializer 
class could be made seperate and the setDefaults method could be removed.  That 
would also mean a headless launch configuration could set its defaults without 
loading a UI component. 

Java Problems 
1.  The java launch config delegate is not public.   
2.  The java launch config delegate generally loads all of its paramters from 
the launch configuration passed to it.  The only exception to this is the 
source locator.  If the source locator is not specified in the launch config it 
is created by the project.  This is the only option that this is done for an is 
very inconsistant.  I would suggest that the Project attribute be removed 
compeltly and all needed options should be made part of the launch 
configuration. 
3.  The java tabs are not API and cannot be reused 
4.  If the java tabs where API the widgets are marked as private so the 
subclasses could not programaticly disable certian options.
Comment 7 Darin Wright 2002-03-21 17:30:42 EST 
New interface ILaunchConfigurationTabGroup has been released to HEAD with new 
extension point. To soften the blow, I was able to create a default "group" 
implementation for exsting tabs. The JUnit launch config tabs still work, but 
need to be migrated.

* The PDE launch config tabs are broken since they do not inherit from the 
AbstractLaunchConfigurationTab. The quick fix is to change the superclass of 
your tabs.

* The java tabs have been converted to use the new group mechanism, which you 
can use as an example. There is an AbstractLaunchConfigurationTabGroup, which 
does most of the work for you - see LocalJavaApplicationTabGroup.

* I expect there to be some more breakage for those re-using "java tabs" as we 
make changes to them in the near future (reorgainze the tabs).
