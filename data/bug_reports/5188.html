Comment 6 Jed Anderson 2002-08-12 11:33:08 EDT 
A thought experiement:

What if launch configs stored a set of projects that were associated 
with the launch?  The debugger would know exactly which breakpoints to install and which ones not 
to.

The user would have to be able to set the associated projects using a typical, 
add/remove/default mechanism.  The default would be either the current project, all projects, 
or the current project and every project it depends on (in the java build path sort of 
dependancy).

This would also help source lookup because you could choose between two 
different copies of source for a given class file based off of the project association.  For 
example, given project A & project B both have Foo.java, but only project A is associated with the 
current launch.  When the location of the current stack frame is Foo.class, the Java source lookup 
mechanism would only return the source for A/Foo.java instead of forcing the user to choose.  If we 
made this public others could follow our lead.  This would make it much easier for a user to work on 
separate applications in the same workspace.

Comment 25 Andrey Loskutov 2015-09-10 16:56:10 EDT 
First of all, I was chocked after my colleagues pointed me to this bug. I never ever could imagine that that simple use case is not supported by JDT, so after discovering this issue in our application I was sure it must be some 3rd party plugin problem.

In our specific case we have multiple (not connected) Java projects in same workspace which are often "copies" of each other (with few differences of course), and so containing lot of similar named classes. Debugging such projects is impossible because breakpoint hit always open wrong classes (Murphy's law + Russian roulette = our case), and one has breakpoint hits even if the actual project code does not have any breakpoint installed!

I think the problem can be described (from the user point of view) in this way: 

1) If a JVM is started, only breakpoints for the types visible from the classpath of the project of the main type specified in the currently executed launch configuration should be installed in the JVM. E.g if there are no breakpoints exists in the IDE on "a.b.c.Foo" type accessible from the classpath of the currently launched application, the JVM should not install "a.b.c.Foo" breakpoints, even if some other (unrelated) "a.b.c.Foo" class in the IDE has a breakpoint marker.

2) If there are multiple but unrelated "a.b.c.Foo" classes in the workspace (have same FQN, but different content!), the breakpoint hit on "a.b.c.Foo" type should hit the file/class accessible from the classpath of the project of the main type specified in the currently executed launch configuration, and not some other, completely unrelated "a.b.c.Foo" class in the IDE.

3) If there are multiple "a.b.c.Foo" classes on the classpath of that project, a dialog should be shown to select which one is to open (with "remember the choice" checkbox). This is not our case, but just to complete the story of duplicated classes.


My naive technical point of view on the solution.

We have tree facts (please correct me if I'm wrong): 
1) A breakpoint marker in the IDE, which knows exact location of the file/class/project it was created on
2) A launch config which stores the information about the project and classpath of the executed application.
3) A FQN of a class with breakpoint hit reported by the JVM

Two places to fix (please correct me if I'm wrong): 
1) Before installing breakpoints, they should be filtered *if* the concrete classes they are created on are visible on the classpath of the project of the started application. I think this should be possible, I believe breakpoints carry project information (at least they can be grouped by project), so this should be possible. If not, adding project info to a breakpoint created on a source file should be a no-brainer.
2) On breakpoint hit we can use type search for the FQN of the class *in scope of this project classpath*, and this also should reveal the right source/class file.

With this information, even being non JDT committer (I have very limited knowledge about JDT/debug) I think I can craft a patch.

So it would be nice if someone from JDT/debug could point me to the right place to start.

I think a good first place to start would be to look in JDIDebugTarget as Christian mentioned. Any filtering would have to happen on the client side anyways, since JDI (still) has no knowledge of resources / projects / etc, JDI only knows the name and location (in classfile) during the breakpoint event.

Another thing to consider is that not all breakpoints are associated with a resource, for example exception breakpoints. So a first cut of filtering would likely only work for line, method, watch breakpoints.

As a first cut, during a launch, when we are going to add breakpoints to the VM, we could check the resources from the breakpoints against those from the launch config (reachable from the getMappedResources method on ILaunchConfigurtion) and only send the ones covered by those resources.

We would also need to make sure that no breakpoint requests are being made for the breakpoints during launching / debugging. To do that we would have to add filtering in org.eclipse.jdt.internal.debug.core.breakpoints.JavaBreakpoint.createRequests(JDIDebugTarget) (and its derivatives).

As said in comment 41, it is a "backdoor" for us to have the new behavior switched off by default (paranoia) in the case that there are still "undiscovered" ways to create breakpoints with attached resources that for some reason not discovered by the new code and so rejected to install a breakpoint.

The problem with the proposed filter is: it is strict "yes/no" decision to install the breakpoint. If we don't find the type on the classpath, the breakpoint will be not installed, period.

Because this "no" decision is taken "silently", the user keeps believing that the created breakpoint is also installed at runtime, and in reality we just run over it.

My experience while fixing this issue was that each time I've checked some corner cases I've discovered new creative ways how classpath can be specified and how breakpoints are created. There are for example breakpoints attached to the workspace root, to the external folder, to files linked to the other locations etc. 

I cannot guarantee that I covered all the possible ways to create classpath and to specify a breakpoint, I simply have not enough experience in JDT/debug. I *hope* I covered most cases.

For example, if you check the diff between patch 10 and 9, you will see that I've discovered that linked resources is a big mess and we also have to deal with them, see line 1375 - 1385 here: https://git.eclipse.org/r/#/c/57437/9..10/org.eclipse.jdt.debug/model/org/eclipse/jdt/internal/debug/core/model/JDIDebugTarget.java

The specific problem here was: if someone links a class folder from project B to project A which has not other relationship to B: the breakpoints in this case are created and attached to the class files from B - and so we need to iterate over all projects which might contain the link, otherwise we would reject the breakpoint from B. Funny enough, the java search can't help us here, because it searches for a *type name* - and in case we have duplicated types, we are back to the roots and cannot decide if the breakpoint should be installed or not.

So after this fix I decided that I better not break some not yet discovered debug workflows and create a safety net for this change - the preference.

If you think that this is too much and that we don't need the preference - I'm perfectly fine with it.

> If there's a single match, then use breakpoints for that type and ignore
> breakpoints from all other types with the same name.

Not sure if I can follow. If there is a single match, there cannot be other types with same name.

> If zero or multiple versions of the type are reachable via the resource
> mapping, 

If there are *zero* matches in current resource mappings, the breakpoints will be installed in the new patch (we don't care since there will be no duplicates and no confusion/no stops in the current debug session).

If there are *multiple* matches in current resource mappings, the breakpoints will be installed in the new patch - because the breakpoints are found in *current* resource mappings and we can't do much here. This case is most likely a sign of a broken classpath or broken debug launch or both. Not our job to fix this user error.

Our problem are breakpoints specified for *source* files not reachable via resource mappings we have in the debugger right now, but matching similarly named source types inside the current resource mappings. We don't want the debugger to stop on those breakpoints in this session.

So I've tuned the new patch to solve this only case.

To fix it, we can safely filter out a type existing multiple times in the source tree but *not* related to the current launch - with all the additional constraints below we can rule it out.

> JDT Debug should either use all active breakpoints, or it
> could also ask the user for the correct source type (with the same dialog
> that already asks for the source if it cannot be determined for a selected
> stack frame).

I don't want to introduce a dialog here (it will distract a lot while debugging if you debug "cloned" projects). In a "typical" workspace this filtering should not affect any type anyway. In "our" (Advantest special) case this can be each second type, therefore we can't just pop-up a dialog.

Below is the current logic implemented in the new patch.

TLDR: the new patch filters only breakpoints from special launch configs; only on source files not related to the current launch *and* matching types existing multiple times in the workspace *sources*.

1) Do not filter if org.eclipse.jdt.launching.DEFAULT_CLASSPATH is set to false (this is for any non-standard launch configs like scrapbook thing).
2) Do not filter if resource mappings are not available (this is for PDE/OSGI launches etc).
3) Do not filter a breakpoint without a marker. This allows breakpoints on any type from any library.
4) Do not filter any breakpoint if the marker resource is the workspace root. This allows all exception breakpoints and any other special cases where no real resource exists.
5) Finally, if the marker resource is not related to the launch configuration in any way (not reachable from the resource mapped to the launch config via Java classpath): check if the type specified by the resource exists multiple times in the workspace *sources*. If this type exists more than one time in the sources, filter it out.

So the new patch filters a breakpoint only for "standard" configurations with defined resource mappings, where the breakpoint is set on a resource not related to this mapping and we have the type from this resource defined multiple times in source code. 

Additionally I've updated patch https://git.eclipse.org/r/60265/ which disables the filtering by default, so that we can integrate the main patch without worrying to break something.
