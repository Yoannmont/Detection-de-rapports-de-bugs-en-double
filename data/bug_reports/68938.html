Description Douglas Pollock 2004-06-29 17:51:58 EDT 
3.0, all platforms. 
 
I find it frustrating that it is not possible to do CVS operations by 
right-clicking on the current editor.  This seems like the most intuitive spot 
to do such things.  As a result, I'm forced to keep open an Outline view, 
Package Explorer or Navigator to give me a handle on the resource. 
 
I would like it to be possible to have contributions automatically added for 
the active editor input.
Comment 1 Douglas Pollock 2004-06-29 18:00:11 EDT 
Created attachment 12897 [details]
Patch to "org.eclipse.ui.workbench"

This patch changes the nature of PopupMenuExtender so that -- if the part is an
editor -- they will add contributions based on the editor input for the editor.
 This causes some problems, as there are actually two pop-up menu extenders
(with different ids) for each CompilationUnitEditor currently.	A further
modification was required.

So, PopupMenuExtender now can have multiple menu ids, and PartSite has been
modified appropriately.  Before adding a new extender, it checks to see if
another one exists with the same selection provider, menu manager and part.  If
so, it simply adds another id.

Comment 4 Nick Edgar 2004-07-05 12:33:46 EDT 
While we can try this approach out, my feeling is that editor context menus are
cluttered enough already.  I'd prefer to see object contributions appear in the
editor tab's menu.
Comment 5 Douglas Pollock 2004-07-05 14:02:51 EDT 
It seems to be that most tabbed interfaces provide tab operations on the tab  
menus, and content operations on the content area's context menu.  This change  
is consistent with this.  It seems somehow unintuitive to go looking in the tab  
for some "non-essential" operations.  The question is: how are we making that  
decision as to what is non-essential (from a user perspective, not a technical  
perspective)?  
  
Also, there is prior art (namely, NetBeans) for CVS operations appearing in the  
context menu of editors. 
 
Though, I do agree that the context menu might be too crowded.  It might be 
better to provide these as top-level menu items.
Comment 6 Nick Edgar 2004-07-05 15:04:41 EDT 
I agree with what you're saying about operations on the tabs being distinct from
operations on the content.  In an editor, there's also a distinction between
operations on the content of the file vs. operations on the file itself.
I was actually thinking more along the lines of there being a visible object
representing the file itself, which could provide operations on the file itself
rather than operations on the content.  For example, in IE and other web
browsers, the icon in the address bar represents the URL, and I can drag it into
the links toolbar or elsewhere (although it doesn't have its own context menu).
We could use the icon in the tab to represent the editor input, but then we run
into the problem you describe.

In any case, I think it's worth trying out in the context menu, and that will
give us a better feel for whether having the operations closer at hand improves
usability, and how badly it affects clutter.

Patch to "org.eclipse.ui.workbench" (provides a property on IEditorSite to disable input-based contributions)

This revises the patch to fix Bug 74704 as well as Bug 88552.  It does not deal
with nested editors.  That is, disabling editor input contributions on the
parent multi-page editor will not disable those contributions on the nested
editor parts.
Comment 11 Nick Edgar 2005-04-15 10:05:35 EDT 
Doug proposed adding a providesInputSelection property on the editor site to
allow editors to opt out of the editor input contributions.

I think the flag should be a parameter in a new registerContextMenu method,
rather than a property of the site.  It's possible for parts to register
multiple menus, and they may not all want the same flag.  E.g. in a graphical
editor, the background menu might want the input contributions, but a node's
menu might not.
