Description Doug Schaefer 2004-05-03 15:14:06 EDT 
I have a PDE plugin project with a long running ant builder attached to it. The
ant builder compiles and creates a native library.

I tell the ant builder to run only on manual build but I want the Java and
plugin stuff to build automatically. If I turn build automatically on, I don't
seem to have a way to invoke a manual build. Am I unrealistically mixing both
worlds or just plain missing something.

Comment 2 Doug Schaefer 2004-05-03 16:21:40 EDT 
To be clearer...

With the CDT, build automatically is usually turned off since you generally 
don't want to wait for a compile/link on every save. At the same time, build 
automatically works great for Java projects. When you try to mix the two

Comment 3 John Arthorne 2004-05-03 18:58:24 EDT 
To clarify, this is not a problem that was introduced by the recent renaming of
"Rebuild" to "Clean", although the problem is now more apparent. The problem is
that there is only one temporal granularity of builders (whether autobuild or
not), and this use case suggests that two distinct granularities are
interesting. The "workaround" in the past was to associate Ant builders and
other long running builders with "Rebuild", which would unnecessarily force
other well-behaved incremental builders on the projects to be rebuilt from scratch.

This is something we should discuss, but unfortunately this has been raised too
close to the 3.0 feature freeze for us to do much right now.

Comment 7 John Arthorne 2004-12-13 11:45:08 EST 
I am currently thinking of a simple solution to this bug:

 - Add markup to the builders extension to allow builders to specify if they are
not auto-build friendly (I haven't yet come up with a nice term for it).
 - When a non-auto-building builder is installed on a project in the workspace,
all build actions in the project menu and in the Navigator context menu will
remain enabled when auto-build is turned on.
 - Such builders would ignore the auto-build trigger (the platform could do this
filtering but it feels like that would lock you into not responding when you
might have cases where it makes sense).

The end result is that users can leave autobuild on, and fast incremental
builders can run automatically. The user would have to do a build (Ctrl+B, or
working set build) to get the remaining builders to build. For pure Eclipse Java
IDE users, there would be no change as long as all their builders are fast
auto-builders.

If anyone has more advanced requirements or comments on this proposal, please
add further comments in this report.
Comment 8 John Arthorne 2004-12-13 12:11:51 EST 
In reviewing duplicate bug 74200, the suggestion was made there to allow
auto-building on a per-project basis. The advantage of that approach is that the
user would have more information on what is happening. WIth my suggestion above,
they might be confused about why the build actions are enabled when "Build
Automatically" is turned on. They have no explicit knowledge of exactly what

Since CDT is the main use case for this request, which of those two ideas do you

Comment 13 John Arthorne 2004-12-13 15:29:25 EST 
What configuration options do CDT builders currently have?  I gather that they
already have a per-project setting to determine if they run on autobuild?  If
both CDT and Ant already have a per-project autobuild setting, then how about this:

The platform adds a setting that is per-project AND per-builder to indicate what
"triggers" it responds to (auto-build, manual build, clean, or any combination
thereof). By default, builders inherit the global workspace setting (they
respond to all triggers). The user can change this setting from the "Builders"
property page. This setting replaces the existing Ant and CDT settings.

In the builder extension markup, you can specify if you want to allow the user
to configure your builder (along with a default setting for each build trigger).
 Some builders do not want to provide this level of control to users, and some
do. For example, Ant currently exposes complete flexibility.  The "Builders"
property page currently allows all builders to be disabled, and I know this is a
problem for some builder writers who didn't want to allow their builders to be
turned off. By allowing the builder extension to decide if it is configurable,
and what its defaults are, we get a reasonable out of the box behaviour without
the user needing to do any complicated fiddling.

The build actions (build all, build project, build clean) are only disabled if
the global workspace autobuild is turned on, and ALL builders are configured to
respond to the autobuild trigger.

This way, in the simple Java case, nothing changes.  The JDT team can decide if
they want to allow the user to configure what triggers their builders run on.
CDT builders would specify in their markup that by default they don't respond to
auto-build, but they specify that the user can change that value.  Ant would
probably respond to all triggers by default, but also allow the user to
configure it. The advantage here is that all builder options are localized,
saving the user from hunting through various configuration pages, and since it
lives in the platform the UI can make intelligent decisions about what build
commands should be enabled at what times.

Adding Darin (from Ant) to the discussion.

Comment 20 John Arthorne 2004-12-21 17:43:14 EST 
I have released the following to HEAD:

Builder extensions can now declare an "isConfigurable" attribute on the builder
definition.  A "configurable" builder is one that allows customization of what
build triggers it responds to.

New API on ICommand:
 - isConfigurable() - tells you if a command is configurable
 - isBuilding(int) - tells you whether the command responds to a given trigger
 - setBuilding(int, boolean) - allows turning on or off building of that command
for the given trigger. This method has no effect for builders that are not
configurable.

The idea is that this allows full customization of build triggers for those
builders that want to allow it.  Builders that don't want to allow this level of
user-meddling can do so by not making themselves configurable.

Still to do:
 - Further test suites
 - Use this API to determine enablement of build commands for a given selection
of projects (may need some extra convenience API)
 - Expose this new API in the builder property page (replacing similar
mechanisms already present there and on the ant builder configuration page).
Comment 21 John Arthorne 2005-01-05 10:43:15 EST 
I have released changes to the platform UI build commands to update their
enablement based on the new API.  So, if you select a project that contains a
builder that does not respond to the autobuild trigger, then the manual "Build
Project" command will be enabled.  Similarly, if there exists a builder in the
workspace that does not respond to autobuild, then the manual "Build All"
command will remain enabled.  Once "non-incremental" builders are properly
configured to not respond to autobuild using the new API on ICommand, this bug
will be fixed.

The remaining work is to expose the ICommand API on the Builders property page,
and remove (if deemed appropriate) the external tool builder preferences that
mirror this new functionality.

Comment 27 John Arthorne 2005-03-14 15:04:59 EST 
All changes to commands have to go through the API on IProjectDescription. I.e.,
Find the command in the build spec associated with your builder
(IProject.getDescription().getBuildSpec()), modify it, then call
IProjectDescription.setBuildSpec(), followed by IProject.setDescription.  You
need to go through this central place to ensure that the description change is
propagated and listeners are notified.  Even if you could access the command
object directly referenced by the builder, it's also only a clone of the
"original" stored in the workspace data tree.
