Description Philipp Kursawe 2007-04-24 16:27:39 EDT 
It should be possible to equip the NLS class with the ability to call properties.load not with an InputStream but rather with a BufferedReader that can translate UTF-8 characters. It is way to annoying to convert all UTF-8 files to the escaped versions of properties files Java understands.

I propose a change in the "Externalize Strings" wizard that allows to specify that you wanna use UTF-8 files or even better XML files in whatever encoding you like (its up to properties.load(XML) to decide that then). The generated code could look like this:

NLS.initializeMessages(BUNDLE_NAME, Messages.class, "UTF-8");
NLS.initializeMessages(BUNDLE_NAME, Messages.class, "XML");

It should be only minimal changes and would be very helpful and not break any exciting functionality.
Comment 1 Jeff McAffer 2007-04-24 16:33:47 EDT 
it would be great to have a patch that implemented the desired behaviour.
Comment 2 Philipp Kursawe 2007-04-24 18:35:54 EDT 
I am actually not so firm in the Eclipse source code and would not know where to start to add the functionality to the wizard, let alone creating a source patch. 

However, I have implemented the UTF-8 loading in minutes:

final MessagesProperties properties = new MessagesProperties(fields, bundleName, isAccessible);
                if (format != null) {
                    properties.load(new BufferedReader(new InputStreamReader(input, format)));
                } else {
                    properties.load(input);
                }

Works great. 

Anybody can point me somewhere where I can learn how to create patches for the current source-build?


I was also working today on dynamic locale switching using extension points. When the current locale is switched all registered extensions will be notified about that and can recompose their textual / graphical output after I reloaded all the messages from the new locale properties files.

This already works in the current build of eclipse using a little reflection hackery. I set the NLS.nlSuffixes back to "null" and then reload all Messages classes specified in extension points. This works so far. Would be better of course when NLS would provide a method to set the "nlSuffixes" from outside back to "null" or switch the locale.

I plan on implementing an "in-place" translation feature that will query all untranslated strings in other languages than the default one (English). This view would be an optional plug-in and work with a connection to some kind of datastore to save newly translated messages. In most cases that would be a database. Another extension could provide an update listener on this datastore to download and locally save newly translated messages to all currently running instances of the application. Or upon complete translation of each language some Product Editor is informed and creates a new NL1 fragment for deployment.

Hmm, the wizard is in either JDT UI or PDE UI.  I was thinking more of a patch for the runtime level to read such a file.  I realize it may not be hard but it is useful for people to supply solutions where they are able.

for more info on where stuff is in CVS.

As for the dynamic locale switching you should see bug 75320.  Having a real answer for this would be great.  The big challenge here seems to be introducing a new lifecycle event that tells everyone to refresh.  This is needed for people who use extensions (autotranslated by the registry), users of the NLS class and people using standard ResourceBundles.
