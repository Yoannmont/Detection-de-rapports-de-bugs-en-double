Comment 3 Michael Valenta 2004-05-17 20:46:21 EDT 
The problem is that both files involved have the same modification timestamp 
and renaming a file does not change the timestamp. Since CVS is timestamp 
based, this fools CVS into thinking that the file has not changed (i.e. if the 
timestamp of the local file matches the timestamp on the server then the file 
is deemed in-sync no matter what the contents are). This is a limitation of 
CVS.

Comment 8 Malgorzata Janczarska 2011-01-05 13:25:24 EST 
The source of the problem is situated in method EclipseFile.isDirty(). The file is marked as changed only when our local timestamp is different from the timestamp in SychInfo (the modification timestamp of the file retrieved from CVS). When we delete the file (changing name is also deleting old file) the timestamp is not changed but the SynchInfo is marked. This way this change is interpreted as deletion of the file on our site, not an incoming change (new file in CVS). If we delete the CVS info we will loose this information.
When new file of the same name is created the deleted mark is simply removed from SynchInfo. In majority cases this is enough because the timestamp on the new file on disk is different and we are recoding and update, but in this particular case the timestamp is the same and there is no way to determine that the file is different. If we removed the SynchInfo then on next synchronization we would always get a conflict in this file. After this action I would rather see an update than a conflict, because it's easier to synchronize your changes.
As for updating the SynchInfo I don't have a good idea how could we update it. There are some markers added to SychInfo timestamp when file is merged (see ResourceSyncInfo.getSyncType()). Maybe we could establish something similar for deletion.
BTW: if we update the file somehow without changing its timestamp (for example replace it with a file of different contents but the same name) we will see the same problem, but the original file won't even be deleted in-between. I think this won't be possible to detect at all.

Comment 10 Malgorzata Janczarska 2011-01-14 07:34:08 EST 
Created attachment 186820 [details]
Clear timestamp on restore

This patch clears the timestamp when the file is deleted and then a file with the same name is created. This way the file is always shown as changed. The timestamp is reset by the server while the next synchronization.
It however changes the a little the behavior of the server comparison. When there are no differences in file content (the file has been restored, but not modified) servers resets the date and returns an error "The modification timestamp was changed for '<filename>' but the contents match that of the server. The timestamp has been reset." that we are not prepared for. This causes two test to fail IsModifiedTests#testFileMoveAndCopy and IsModifiedTests#testFolderMoveAndCopy. I don't have a good idea for now how could we handle those errors.
Comment 11 Malgorzata Janczarska 2011-01-14 07:47:18 EST 
Created attachment 186822 [details]
Add mark to the timestamp

This patch solves the problem that I found with attachment 186820 [details]. It does not reset the timestamp but adds a mark to it the same way that merged files are marked. When the server comparison is run the mark is not send to the server. This keeps the old behaviour in server comparisons but still marks the file dirty on the client site.
