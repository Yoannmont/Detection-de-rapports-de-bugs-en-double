a) The jsp editor can't deal correctly with a file which includes another
    file and this included file inclueds another file too!

     <%@page import="java.util.*" %>
<%

%>
<%@include file="include1.jsp"%>

    2. include1.jsp

<%
for(int i=0,n=list.size();i<n;i++){

 <%@include file="include2.jsp"%>
<%}%>

    3. include2.jsp
<%for(int j=0;j<3;j++){%>
 <%=o%>
<%}%>

   now,jsp editor shows an error that "o can not be resolved (in file 
include1.jspinc)"

it is standalone,if not,it is not compiled as a jsp file.

Comment 7 Phillip Avery 2005-09-27 11:29:47 EDT 
Some future JSPTranslation ideas.

If we had an JSP includes dependency tree (wonder where we could get a "table of
links" like that... =) we could just walk up the tree (before translating a
file), and translate the parent most JSP. Then the contents of the translation
would be correct for fragments.

One issue with this is that the offset mapping would have to be more robust than
it is currently, as it doesn't keep track of included file mappings well.
It just maps the code to from the included file to the include="file.jsp"
attribute value.

In the translation we should keep track of which file each chunk of code came
from, and the offset mappings to that file.  Just kind of brain storming here...
There's probably more to be thought of.
Comment 8 Phillip Avery 2005-09-27 13:55:43 EDT 
OK we can probably nix the idea of "dependency tree", must have been too early
in the day.  A fragment or include may (likely) have multiple parents, so it's
impossible which context it's in w/out more info from the user.

I can see that getting complicated.

More importantly I think we can add better support for JSP Fragments, and ignore
them during validation if we don't know the context.

Comment 10 Phillip Avery 2005-09-29 13:47:57 EDT 
I've checked a fix into HEAD (JSPTranslationExtension).

The fix ensures that we don't compile standalone JSP fragments.  As long as you
identify your fragment files as such (name them *.jspf, *.jsf, or associate your
extension w/ JSPFragment), then this should fix your problems.

This should address part b) of the original bug.

As for part a), I think that's a matter of the error message "trickling up" the
include files cauing a misleading error.  

If include2.jsp is a fragment, it won't be compiled and the error won't show up
(with this fix)

If include2.jsp isn't a fragment, it's an invalid standalone JSP anyway (since o
isn't defined).

Marking this bug as fixed.
