Comment 1 Andrew Niefer 2007-07-10 11:23:27 EDT 
Because VMs generally don't start if given an unrecognized vm argument, --launcher.XXMaxPermSize only results in -XX:MaxPermSize if the launcher can positively identify the vm as being from Sun.

The launcher only knows how to identify Sun VMs on Windows, and we currently do not have a good way of identifying Sun VMs on the other platforms.

Comment 8 Andrew Niefer 2007-10-16 12:18:36 EDT 
After looking through the code, if the mac binaries are recompiled for bug 203325, then --launcher.XXMaxPermSize will work if -vm is specified as anything starting with  /System/Library/Frameworks/JavaVM.framework.

3.3.1 as it stands now should work if -vm is specified as
/System/Library/Frameworks/JavaVM.framework
/System/Library/Frameworks/JavaVM.framework/Versions/<version>

but not /System/Library/Frameworks/JavaVM.framework/Versions/1.4.2/Commands/java

If the launcher is left to find java on its own, it will find /usr/bin/java which will not be identified for XXMaxPermSize.

I've posted an experimental program I worked on to test libjava.so for Sun VM compatibility.  It takes an argument which is the path of the JRE libjava.so.  I have tried it on Sun's JDK, Iced Tea, and IBM's JVM on my i386 Fedora 8 box.  It reports success (finding SUNWprivate) for the first two and not for the third.  I offer it in case the logic can be of use in creating a fix for this problem.

To build on my F8 laptop, I had to add -D_LARGEFILE64_SOURCE to the compile arguments.

gcc -D_LARGEFILE64_SOURCE -o telf.out telf.c -lelf
./telf.out {path}/libjava.so

The program is simply looking at the .gnu.version.d section.  For the IBM libjava.so, a readelf -V results in the following snippet:

Version definition section '.gnu.version_d' contains 2 entries:
  Addr: 0x00000000000068fc  Offset: 0x0068fc  Link: 3 (.dynstr)
  000000: Rev: 1  Flags: BASE   Index: 1  Cnt: 1  Name: libjava.so
  0x001c: Rev: 1  Flags: none  Index: 2  Cnt: 1  Name: VERS_1.3

whereas on the aforementioned Sun and Iced Tea JDKs, one gets:

Version definition section '.gnu.version_d' contains 2 entries:
  Addr: 0x0000000000006300  Offset: 0x006300  Link: 4 (.dynstr)
  000000: Rev: 1  Flags: BASE   Index: 1  Cnt: 1  Name: libjava.so
  0x001c: Rev: 1  Flags: none  Index: 2  Cnt: 1  Name: SUNWprivate_1.1

Comment 41 Andrew Niefer 2008-05-07 16:01:29 EDT 
Created attachment 99163 [details]
patch

If the launcher argument --launcher.XXMaxPermSize is specified and we have a java executable (as opposed to only a jni library) then we exec "java -version" with that executable and parse the output.

If "Java HotSpot(TM)" is found, we identify as Sun.

This applies to linux, solaris, hp & aix.  If desired we could turn it off for aix on the assumption that there are no sun vms on aix.
