Description Dejan Glozic 2006-03-06 14:17:53 EST 
Scenario A:

1) SWT presentation implementation is configured to run on a certain os/ws combination.
2) We directly create SWT presentation. Everything is fine.

Scenario B:

1) HTML presentation implementation is configured to run on a certain os/ws combination
2) We try to instantiate SWT Browser widget.
3) It throws a NO_MORE_HANDLES exception.
4) We silently go for the next matching implementation (SWT in our case).

For scenario B, we should warn the user that SWT implementation has been loaded as a fallback. Perhaps a warning dialog with a checkbox to not warn the user the next time would suffice.
Comment 1 Chris Goldthorpe 2007-04-04 13:37:16 EDT 
Hi Dejan

I'd rather leave things the way they are. If a new user loads Eclipse for the first time and for whatever reason the Intro can only show in SWT mode it isn't going to help them to see a dialog with a message that might not make much sense to them. Seeing a warning dialog when you launch the product for the first time doesn't create a good first impression.
Comment 2 Dejan Glozic 2007-04-04 13:46:21 EDT 
Agreed although the warning would be only a first sign of further degradations down the road. There are many uses of the embedded web browser so if we cannot load it, neither can other plug-ins.
Comment 3 Chris Goldthorpe 2007-04-04 14:03:36 EDT 
*** Bug 129808 has been marked as a duplicate of this bug. ***
Comment 4 Jim Perry 2009-02-17 16:30:35 EST 
When Eclipse defaults to the SWT presentation for the Welcome, there is a line of code:

Log.warning("Loaded UI Forms implementation as a default UI implementation.");

The Log.warning() method will only write to the log if the plug-in is in Debug mode, meaning that a generic user running the product will not see any message.  We may want to link about using Log.error() which guarantees output to the Log, so the user has indication that they may have a problem.  If we can provide them a way to recover and get the browser implementation to work, then it would be more useful to the user to see these types of messages.
Comment 5 Chris Goldthorpe 2009-02-17 16:54:43 EST 
There are two reasons why the browser implementation might not load. One is that we are on a system where we would expect it to load, such as most Linux systems or Windows XP. The other is systems such as Solaris or AIX where the Eclipse Browser widget is not supported. In the latter case we would not want to issue a warning because this is the expected behavior. There may be cases on Linux for example where Eclipse is simply not finding the browser but with some preference settings it could and a warning may be appropriate in that case.
