Comment 4 Stefan Xenos 2005-03-28 14:45:43 EST 
Unzooming can trigger part restoration because there may be a view (or editor)
that was obscured by the zoomed part and it may not have been lazily activated
yet. In this situation, unzooming will cause the part to become visible for the
first time and trigger restoration. Making something visible will restore it --
it doesn't need to be the active part.
Comment 5 Nick Edgar 2005-03-28 15:02:40 EST 
OK, makes sense.  There were other stack traces in the console log where the NPE
did not involve an unzoom.  

It looks like the editor mementos are getting blown away too early, e.g. by the
closeAllEditors call.  For close all editors, though, it would be nice if it
could avoid restoring any editors.

Comment 17 Stefan Xenos 2005-04-01 14:01:11 EST 
Spent some time investigating. It appears that this is what is happening: 

When we close the active editor, we first close it and then activate a second
editor. This means that for a short time a disposed editor reference is the
active editor. The solution is to always activate something else (or deactivate
everything) before closing the active editor -- we should never dispose an
editor while it is active.

This regression showed up in the part creation refactoring. Previously, the
pointer to the active editor would have been updated before the editor was fully
initialized. This was changed to update the active editor pointer after the new
editor was fully materialized, but it means when closing editors we end up
pointing to the old disposed editor rather than the new uninitialized editor
(which is just as bad).

By updating activation before closing anything, this problem should happily
disappear. If we also add this to the API contract ("parts are always
deactivated and hidden before closing"), then it will give clients some nice
invariants to code against, and they can remove the common special-case code in
IPartListener/IPartListener2 implementations that needs to check if the part
being closed is the active view/editor.
