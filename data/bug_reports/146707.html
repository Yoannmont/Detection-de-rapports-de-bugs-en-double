Description Mark Hutchinson 2006-06-12 16:47:47 EDT 
I have a WSDL document with a simple type that has a pattern restriction. Pasted below is a fragment of the wsdl, I will attach the full file to this bug.

<wsdl:types>
		<xsd:schema
			targetNamespace="http://www.example.org/webservice/">
			<xsd:element name="NewOperationResponse" type="xsd:string" />
			<xsd:element name="NewOperationRequest" type="tns:initials" />

			<xsd:simpleType name="initials">
				<xsd:restriction base="xsd:string">
					<xsd:pattern value="[A-Z][A-Z][A-Z]" />
				</xsd:restriction>
			</xsd:simpleType>

		</xsd:schema>
	</wsdl:types>


This WSDL document validates fine in the WSDL editor when I right click on it and select validate.  The problem is that when I right click on the wsdl and select "test with webservices explorer" a Validation error appears in the status frame.  

This is what I see in the status frame:

IWAB0380E Errors were encountered while validating XML schemas.
XSD: The pattern facet is not permitted in a type based on 'http://www.w3.org/2001/XMLSchema#string'
IWAB0381I file:/D:/workspaces/WSEBug/PatternTest/WebContent/patternTestWebService.wsdl was successfully opened.

Comment 2 Mark Hutchinson 2006-06-14 16:24:38 EDT 
It looks like this Validation error is found in org.eclipse.wst.ws.internal.explorer.platform.wsdl.datamodel.WSDLElement in the loadWSDL() method.  There is a call to org.eclipse.xsd.XSDSchema.validate() that comes back with this validation error.

This WSDL validates fine in the WSDL editor, and Lawrence Mandel has confirmed that the WSDL looks valid. It looks to me like this could be a bug with org.eclipse.xsd.  Maybe the solution to this would be to use a WTP validator rather than the org.eclipse.xsd one.

Comment 7 Jeffrey Liu 2006-07-11 11:10:39 EDT 
Hi Mark,

First of all, if you believe there's a bug in the o.e.xsd validator, please do open a bug against them.

As for the patch, my first question is, what is the performance impact here? Pre-patch, we are loading schemas using DOM elements (the DOM elements are already loaded at that time when the method is called). Post-patch, we are loading schemas using a WSDL URL. The potential performance problem that I see here is that:

1. Are we making another call to get the WSDL (and everything that it imports)? Network traffic.
2. Since we are loading the schema object from a WSDL URL, I assume that means we will load the WSDL and XSD model again? What is the performance impact here? Is there any caching involved?
3. As I recall, the getSchema(...) method is called quiet often. We are now creating EMF resource set everytime this method is called. Is this going to be expensive?

Looking at the patch, line 381:

EObject eObject = (EObject)resource.getContents().get(0);

Why are we always getting the first object from the list of resource? Say for example I have a WSDL, whose <type> section looks like this:

<type>
  <schema>
  ...
  </schema>
  <schema>
  ...
  </schema>
</type>

Does ".get(0)" only give me the first schema? What if I was trying to load the second schema, what happens here?

Thanks,

Jeff
Comment 8 Mark Hutchinson 2006-07-17 16:22:56 EDT 
Created attachment 46396 [details]
revised patch for org.eclipse.wst.ws.explorer

Thanks for the excellent feedback Jeff.

I don't think there is a problem with org.eclipse.xsd, but I think that we are using it incorrectly.  I asked Ed Merks and he said "In general, unless the XSDSchema instance is in an XSDResourceImpl within an ResourceSet, you won't be able to resolve imports and it won't validate."  The problem is that we do not put our XSDSchema objects in a Resource in a Resource set.

So this new patch adds each XSDSchema to a Resource and adds those to a ResourceSet. We are still creating the schema model from the DOM and now the Resource set is only created when loadWSDL() is called.

Comment 10 Chris Brealey 2006-07-17 23:18:34 EDT 
Mark, Jeff,
thanks. I have reviewed, committed and released the change under tag v200607180315. I made two additional changes after applying the patch:

1. Removed the unnecessary import.

2. Added an "if (xsdSchema != null)" check around the code near the end of gatherSchemas() that adds the Schema resource to a resource set, since it's possible, if very remotely so, for the <wsdl:types> element to contain type system elements other than <xsd:schema>. In such a case, xsdSchema would be null. I'm not sure if resource.getContents().add(null) would explode, but decided to play it safe.

Let me know if you have any concerns.
