Description Kim Moir 2007-02-16 12:22:30 EST 
The Orbit project would like to provide signed bundles.

Downstream components will consume these bundles and include them in their builds.

We don't want downstream components to have to retain a list in their builds of the Orbit bundles they consume and should be excluded from signing.

This enhancement request is to provide an option in the bundle to indicate that that the bundle should not be re-signed by the eclipse.org certificate once it has been signed.

Comment 7 David Williams 2008-02-25 00:05:52 EST 
To answer some of the questions above, we pack.exclude any jar that is prebuilt (as so marked in the map file). This prevents them from being conditioned. 

To date, we do not exclude any from signing. If anyone knows of any that must be or should be excluded, I guess we'll have to maintain a list of them. We could just code them in our pack.properties file with a sign.excludes list.

Comment 8 David Williams 2008-02-25 00:18:15 EST 
There is another efficiency use-case that would justify doing more to automatically not re-process a jar that had already been signed. 

We in WTP, by design, do our builds in a modular way, building smaller things first, producing zips/features for them (and, signing them) and then we build larger things. Sometimes just combining zips or already built features to produce the larger pieces. I haven't studied this in detail, but am pretty sure this means in the latter, large piece builds, we end-up re-processing quite a number of jars that have already been signed. Sure, there are ways we could get around this in our build scripts, but they are all quite a bit of work, and/or make the build less modular. So, seems it'd be nice if a field was added to eclipse.inf that said, basically, eclipse.signed=true, and gave a time stamp, so the jar processor if it found that flag true, and within some reasonable period of time ... such as a week or a month it would not re-sign it (it would seem you'd want some time limit, so, for example, next year if some jar was re-included without being re-built it could get a new signature .... since I assume the official certificate changes from time to time? 

Any volunteers? :)

Comment 14 Martin Oberhuber 2008-09-18 14:10:32 EDT 
Two observations from bug 167864:

(1) Signing a signed Jar again can change the binary bits, because the order
    of entries in the signature file (ECLIPSE.SF) can change. That is, even 
    if the re-signed bundle verifies OK, its binary contents is changed.
    I've seen that when signing the (already signed) commons.net source bundle:
    http://download.eclipse.org/tools/orbit/downloads/drops/R20080807152315/bundles/org.apache.commons.net.source_1.4.1.v200803071935.jar

(2) As a workaround while this is not implemented, clients of the signing
    application can exempt bundles from signing. I'm doing that for my project
    now (remove from master ZIP, sign, re-add to master ZIP). Other projects
    (CDT, I believe) add a package.properties file into the master ZIP before
    sending it to the signer.
    Perhaps such explicit control for marking what's signed and what not is a
    good idea anyways.

It should definitely say something about it, especially because if an already signed jar is submitted to the webservice, it will be resigned (whereas the command line service will *not* resign it). The reasoning is that the webservice never performs the repack, so it leaves the decision about whether the jar need to be re-signed or not to the caller (ie where the repack will be done or not). Note that the maven jarsigner plugin provides options about what to do with already signed plugins. See https://www.eclipse.org/cbi/sitedocs/eclipse-jarsigner-plugin/sign-mojo.html#resigningStrategy. 

I've added a note on the wiki page about that.
