Description Darin Wright 2005-05-18 15:14:41 EDT 
3.1M7

When investigating bug 95679, I found that IWorkspaceRoot.findFilesForLocation
(IPath) fails if the following scenario:

* the workspace has been started with a "-data" argument that specifies 
incorrect case for the workspace (for example "c:\workspace" instead 
of "c:\Workspace")
* the IPath argument specified has the correct OS case (for 
example "c:\Workspace\project\file.txt"

A similar problem exists if a project is a non-default location (not rooted at 
the workspace root), and the case on the OS does not match the case that 
Eclipse thinks the project has.
Comment 1 John Arthorne 2005-05-19 13:58:35 EDT 
Here are results of some investigation:

I believe this problem has existed since 1.0 (I checked in 2.1.3, 3.0.2, and 3.1
M7).  I thought we had actually fixed this once in the past, but what we
actually fixed in the past was locations with a missing device (bug 22751). To
fix that bug we trapped all places where locations were supplied (-data,
Project.create), and converted from IPath -> java.io.File -> IPath.  This caused
the device to be added (java.io.File infers the device from the current working
directory if no device is found on Windows).

It's possible we could fix this problem in a similar way, by doing the following
on every supplied path string: location = new File(location).getCanonicalPath();
 This converts the path case to match the file system path case.  Unfortunately,
this also has the side-effect of resolving symbolic links on Unix platforms,
which may not be desired.

I tried this change (patch to follow), and it seems to solve the two cases
(-data argument doesn't match and project location case doesn't match).  UI
changes would also be required in the case where the data location is user-supplied.

Overall, I don't feel good about considering this for 3.1.  It's behaviour
that's been around forever, so changing in a release candidate without time to
find out any side-effects would be risky.

Comment 8 Logan Colby 2005-11-15 12:57:22 EST 
The challenge of this defect is that third-party users will have no idea why
they see the failures that they do. Say, hypothetically, you have a product that
supports cross-references between files. References that are validated using
workspace-relative paths will succeed; references that are validated using
absolute paths will fail. The end customer will simply see errors reported that
references don't exist, when clearly they do. They will complain bitterly to the
creator of the product that builds on top of eclipse, and problem-determination
can be quite challenging if the person answering the support call doesn't know
about this issue. 

I'll be adding a check in the meantime, in a plugin initializer, comparing the
location returned from Platform and the location returned from the canonical
path, and writing a warning or error to the log to at least assist in problem
determination.

Comment 11 Paulo Merson 2005-11-21 16:39:35 EST 
The workaround I found was to rewrite method org.eclipse.core.runtime.Path.isPrefixOf(IPath)
using equalsIgnoreCase() instead of equals() when comparing the segments. Since Eclipse accepts workspace path and project location typed by users that don't match the file system case, I think the code in Path.isPrefixOf() should use equalsIgnoreCase.
Comment 12 Mark Phippard 2005-12-07 10:21:41 EST

This also causes problems in Subclipse and this one is harder to head off.
Comment 13 Jeff McAffer 2005-12-07 11:07:30 EST 
This is a little more subtle than the patch in comment 10.  The system property may be something like @user.dir.  Doing a getCanonicalPath on that gives something like D:\temp\workspace\test\@user.dir.  Not what we wanted.  Similary, if the instance area prop is "", getCanonicalPath will return the current working dir.  The default instance area property also has to be munged as it may end up being the one that is used to set the actual value.

Comment 11 suggested using equalsIgnoreCase.  The issue of case sensitivity is, well, sensitive.  There are a great many details and variations.  It seems in this situation we are better off with just avoiding the problem in the first place and canonicalizing the path used.  

Perhaps we should just canonicalize it after it has been fully computed and just before setting it into the location.

Comment 16 John Arthorne 2006-01-09 11:15:08 EST 
I have been working on fixing the problems caused by this in the resources plugin (bug 119705).  I am capturing all places where locations are input (project creation, link creation), and replacing the input location with a canonical path.  As part of this I have tried storing a canonical copy of the platform location for use within the resources plugin, but this is causing problems in many tests because the following expression that was always previously true can now be false.

Platform.getLocation().equals(IWorkspaceRoot.getLocation())

I can remove all references to Platform.getLocation() in the resource test suites, but it would be much easier if we could fix this once and for all in the runtime. One line of code inserted in the right place should do the trick...
