Comment 6 Darin Wright 2004-06-16 11:46:34 EDT 
This bug has a similar flavour to bug 55242. The problem is that plugin A is 
activating plugin B after B has been shutdown. A is activating B since B 
provides an extension for an extension point that A defines (and A is 
instantiating/calling the extension implementation). A more encompassing fix 
would be to have plug-in A guard against calling extensions in plug-ins that 
have been shutdown (which we are investigating in bug 55242).

This would mean the fix would be in JCORE - avoid calling initializers in 
upstream, shut-down plugins.

Comment 12 Jeff McAffer 2004-06-16 15:26:36 EDT 
moving to Equinox and assigning to Pascal.  

Tom, please attach a patch for the framework stopping flag.

Pascal, please attach a patch with behaviour something like...

In the eclipse adaptor we need to expose something that we can ask the 
question "has bundle X already been shutdown (as the result of a system 
shutdown)".  If it has, then createExecutableExtension should throw a 
CoreException.  This new mechanism should be controlled by a sytem property (of 
course) called "eclipse.strictShutdown".  By default this value is true (or not 
set).  if it is set to "false" then createExecutableExtension should skip the 
shutdown test and go ahead as normal (i.e., attempt to create the extension)

While this is not an API change in anyway (createEE is already spec'd to throw  
CoreException), it is a change in behaviour.  The new behaviour is much safer 
and more logical.  However, there may be cases where existing plugins have been 
successfully shutting down without receiving any exceptions.

Comment 20 Pascal Rapicault 2004-06-17 12:25:24 EDT 
Although not functionnal problems, the problems that occurend were captured in
the bugs that have been marked as dup.

As a result of that and complementary discussions we slightly changed the story.
A ClassNotFoundException is now being thrown if the class being loaded belongs
to a plugin that has already been shutdown. This exception is then caught by
createExecutableExtension that wraps it into a CoreException.

Note that this new behavior is more permissive and more correct in regard to
activation since plugins can now be activated while shutting down the platform
Of course we don't reactivate plugin that have already been shutdown while
shutting down the platform.
This will solve NPE problems on shutdown.
