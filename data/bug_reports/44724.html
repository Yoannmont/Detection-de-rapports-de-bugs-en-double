Description Jared Burns 2003-10-12 15:09:21 EDT 
We should explore lazy population of the launch view contents. We now compute 
labels in the background but it can also take a significant amount of time to 
compute the structure of a launch with many threads, stack frames, etc.

Investigate using the DeferredTreeContentManager. See Team's 
RemoteContentProvider as an example.
Comment 1 Jared Burns 2004-01-13 19:46:20 EST 
Created attachment 7431 [details]
Lazy content provider patch for the org.eclipse.debug.ui plugin

Implemented lazy content provider for the launch view as specified.

It turns out, however, that most of the time populating the view is spent in
the UI components. Doing the work of getting the content in the background has
no noticable positive effect because the UI still gets locked up with large
content while the items are added to the tree.

Fetching the content lazily looks ugly in the launch view. It causes a
disturbing amount of instability as the "pending..." nodes come and go.

Perhaps most importantly, the asynchronous content fetching introduces
complexity problems related to the expansion and selection of items in the view
which don't seem to be worth the effort to address.

Attached is a patch containing my implementation in case we ever decide to
revisit this issue.

Comment 11 Darin Wright 2004-10-22 11:32:14 EDT 
Implemented background content in the debug view using the framework provided 
by the workbench (DeferredTreeContentManager), similar to the CVS "remote 
view" support.

The debug view's content provider is now an extension of 
the "BaseWorkbenchContentProvider" which uses IWorkbenchAdapter's on the 
elements in the view to retrieve content. Our content provider uses a 
DeferredTreeContentManager, which uses IDeferredWorkbenchAdapter's to retrieve 
deferred content.

The debug plaform provides IWorkbenchAdapters for all debug elements in the 
debug view, and IDeferredWorkbenchAdapters for debug targets and threads. 
Thus, the target and thread content (threads and stack frames) are retrieved 
in the background, while other element content is computed in the maint thread.

Clients that provide their own adapters can provide special content in the 
debug veiw. For example, the java debugger provides its own 
IDeferredWorkbenchAdapter for IJavaThread to provide monitor information.

Comment 20 Darin Wright 2004-11-23 09:16:46 EST 
Just to re-state the goal... Debuggers (generally) communicate with a remote 
process. The communication maybe asynchronous or syncronous, and generally has 
a latency. The communication may also be unreliable and/or timeout (hang in 
worst case). When this does happen, and the communication is in the UI thread, 
the user sees the UI hang/block for periods of time. We want to avoid this, 
and simply update the UI when/if the information becomes available.

We may be able to provide a way for a debugger to opt out of background 
content generation from the debug views, however, there may be other places in 
the workbench where the same problem occurrs (i.e. background access). For 
example, launching occurrs in the background and we process console I/O in the 
background.

If it is a requirement that a debug model can only be called in a single 
(specific) thread, the debug model can provide this function by ensuring 
requests are queued and processed in a single thread (i.e. incoming requests 
in the wrong thread must be dispatched in the appropraite thread, and the 
calling thread then waits for the response). This would be the only "truly" 
safe implementation in a multi-threaded platform. For example, SWT is based on 
such a model, where UI operations must be performed in the UI thread.

If we restrict the debug platform to only call debug models in the UI thread, 
then all debuggers will cause the UI to block when communicating with the back 
end, which is undesirable .
Comment 21 Darin Wright 2004-11-23 09:56:41 EST 
Instant message chat with Alan Boxall reveals the problem is not a single 
threaded issue. Rather the limitation is that the debug model can only handle 
serialized requests. Background jobs accessing the same debug API can result 
in overlapping/parallel requests.
