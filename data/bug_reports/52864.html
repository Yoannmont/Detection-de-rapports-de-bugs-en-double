Description Alain Magloire 2004-02-23 16:06:27 EST 
The idea is to get rid of CoreModel/CModelManager extensions methods

    getHeaderExtensions()
    getSourceExtensions()

etc .. as you know 8-)  in C++ files may not have extensions (stdio)

The idea was to provide via the CoreModel some methods

    isSourceUnit(String name)
    isHeaderUnit(String name)
    isAsmUnit(String name)
    isTranslationUnit(String name) {
       return isSourceUnit(name) || isHeaderUnit(name) || isAsmUnit(name);
    }

And the user could change the list of acceptable extensions/name via
the preference page.

Comment 12 Sam Robb 2004-04-05 12:18:04 EDT 
Created attachment 9215 [details]
Core portion of resolver patch

Attached is a proposed patch for the CDT core model (org.eclipse.cdt.core) for
dealing with detection of translation units.

The patch provides for two new extension points - one to declare a file type,
and another to create an association between a filename pattern and a file
type.  I used two extension points, rather than one, so that plugin authors
could declare additional file type associations for standard file types.  The
two new extension points are used in the core plugin.xml to declare standard
file types and filename associations for C, C++, and asm files.

The current patch is entirely declarative - there is no UI for updating the
file type associations.  I am working on a UI patch to provide a preference
page for managing file types and associations.	The initial iteration for the
UI patch will probably treat file types and associations as a per-workspace
property; Alain has proposed that CDT support per-project file types and
associations.  I can see the reasons for this, but feel that a per-workspace
solution is a good first step towards that final goal.

Comment 31 Sam Robb 2004-05-25 14:30:05 EDT 
Created attachment 11078 [details]
UI patch for file type resolution

Final patch for this particular item, I think.

This patch provides a workspace perference page and a project properties page
that allow a user to specify file types on a workspace or per-project basis.

Some comments:

- I ended up making significant changes to the resolver model.	Alain had the
user of the model fairly well insulated, though, so there were only minor
changes in the code outside of the resolver.

- From what I could see, though, there are places in the core where the
workspace resolver is being accessed (via getResolver() or getResolver(null)),
that will probably need to be switched over to using a project-relative
resolver.

- One assumption in the current model is that language elements and file types
can only be contributed by plugins; they are not user-editable or
user-definable.  I thought about allowing this, but it seemed to me that there
was little utility in allowing users to define new languages or file types
without also providing code (a plugin) that knew what to do with them :-/

- I wasn't able to solve (at least, to my satisfaction) the problem of how to
handle the case where the user installs a new plugin that declares additional
file type associations.  This is really only a problem if the user has altered
the workspace defaults.  In that case, they can acquire the file associations
from the new plugin by choosing to restore the workspace settings to their
default state.

- I've updated the JUnit tests to reflect the capability to set file type
resolvers on a per-project basis.
