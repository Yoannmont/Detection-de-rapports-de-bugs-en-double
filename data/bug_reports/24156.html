Description Simon Archer 2002-09-26 14:15:55 EDT 
I was surprised to see that the CVS data, including the module name, is 
remembered when a Project is copied.  I expected the new Project to have no CVS 
data.

1.  Create a Java Project.
2.  Commit to CVS.
3.  Copy the Java Project.
4.  Paste the Java Project, renaming it appropriately.
5.  Select the Project and choose Properties from its context menu.
6.  Select the CVS page.
7.  Notice that the CVS properties were copied, including the module name.

Perhaps the Copy/Paste dialogs should have an option to retain/forget the CVS 
data and module name.

Comment 2 Christophe Hartwig 2002-10-04 04:23:46 EDT 
I just found out that copying any CVS-controlled directory also copies CVS 
entries (using Eclipse 2.0).

Here is the use case:
- my sources are in src/
- I want to set up translations, and I want the same kind of folders hierarchy 
as in the src/ folder to hold my property files
- I copy the src/ sontents into translations/ to get my com.* hierarchy (let's 
use copy/paste in the navigator)
- I remove all the java files and empty folders...
- Then I commit changes !
- All my source files are removed from the CVS repository ! 

I spent 2 hours this morning restoring them :-)

I really didn't expect Eclipse to copy the CVS directories...
So I really agree with the "retain/forget CVS" data option ! Maybe a popup 
could ask the user whether he wants the CVS data copied or not, if CVS data is 
encountered in the resources to copy...
Comment 3 Michael Valenta 2002-10-04 08:29:29 EDT 
The last stated problem is not the same issue as the one originally mentioned 
in this bug report. CVS folders are only retained when a project is moved or 
copied. If a folder in a project is moved or copied, the CVS folders should be 
purged. If they are not, it is a bug. One possibility I can think of is that if 
you moved the src/ directory instead of copying it, then the contents of the 
src/ directory would be flagged as outgoing deletions. Please open a separate 
bug report with the exact steps you took.

Kevin, I was teaching a WSDD workshop.  I had created an example project and 
committed it to the CVS repository.  Then I wanted to create an independant 
variation of the work in the project, so I simply copied and pasted it, 
renaming it on the way.  When I went to commit it to the CVS repository I was 
amazed to see that it had retained its old module name and CVS meta data.  I 
would have expected this to be forgotton.  While I can imagine a case were 
keeping this information around would be a good thing, this is clearly not 
always the case.  Perhaps a prompt/confirmation should be included since there 
is a risk of bashing the existing CVS module.
Comment 5 Michael Valenta 2002-10-07 15:10:08 EDT 
And now the "interesting" part of the problem. How de we determine when we are 
supposed to prompt the user and/or purge the CVS information? We can't use the 
project creation delta because there are many ways a project gets created 
(checkout, import, copy). Project moves can be handled but I would argue that 
we would only want to "auto-disconnect" project copies and not moves.

Now that I've done the above mental exercise, I think I know what the real 
problem is. When you copy a project in the workspace, the persitant properties 
are copied as well (and a persistant property is used to associate the projetc 
with CVS). Here are the steps of what took place.

1) User copies a project shared with CVS
2) User performs some edits on new project
3) User tries to share the project by opening the Team menu.
  -  the fact that the Team menu was populated should have been a clue that 
something was wrong but it's understandable that the user accepted that the 
project was already shared.
4) User chooses commit
  - this commits over previous resources

I think the best solution would be to have step 3 require a Team >Share 
Project. This would be enough of a clue that the user should figure out that 
they need to purge the CVS info. We could help them out by adding an option to 
the wizard to do this. 

Unfortunately, the above still requires a means of differentiating a project 
copy from the other project creation mechanisms. We will need to discuss 
alternatives with Platform Core.
