Comment 3 Rich Watts 2007-09-05 11:20:40 EDT 
(In reply to comment #1)

The changes (I will create a real diff file) to the eclipse help system allow us to use the eclipse help system + Equinox and package it as a WAR (Web ARchive) to allow it to be easily deployed on an application server, and you can use the app server's admin console to administer the deployed war.

We make 2 changes: 1) we set the help system to infocenter mode. 2) tomcat is more forgiving about url arguments than websphere, so we need to handle url arguments in a more stringent manner so WAS doesn't ignore them.

On #1, we should consider programmatically determining whether we need to be in infocenter mode vs. standalone - we are currently playing with using getbundle(org.eclipse.help.ui) and trapping for the error if it doesn't exist, then we must be in infocenter mode, if we can load it, then we must be standalone mode.

Originally, we created this with Eclipse V3.2 - but the equinox code was in an incubator, so when it was released to the eclipse platform, we made the necessary changes (which after our coordinating with equinox - became very minimal).

As for documenting our efforts. We probably will not document the V3.2 changes, but we can definately document how to build/package a V3.3.1 UA WAR - But that will take me a little bit to put together.

Comment 16 Chris Goldthorpe 2007-11-27 18:52:57 EST 
The OSGI servlet bridge is registered to pick up any servlet requests, it then starts to activate plugins and looks out for plugins which have one of the extension points org.eclipse.equinox.http.* . If it sees an extension like this:

 <extension
         point="org.eclipse.equinox.http.registry.servlets">
      <servlet
            alias="/control"
            class="org.eclipse.help.internal.webapp.servlet.ControlServlet"
            httpcontextId="help">
      </servlet>

It will redirect requests which match the context id to that servlet.

One complication is that the help system uses a service selector which requires initialization parameters to be passed in via web.xml. I'll take a look at the attachment and make sure that it works for me.

- Click Next in the Plug-in Content section
- In the Templates section select "Plug-in with sample help content" and click Next.

Export the created sample doc plug-in to the /plugins directory in your Help WAR and restart the server.

Here's one way to cause customizations such as HELP_DATA to be changed. There may be a cleaner way to do this but this does work. 

In the webapps area locate the plugin which defines the preference you want to change and edit it's preferences.ini file. For example to change the home page to be eclipse.org edit org.eclipse.help.base/preferences.ini and replace 

help_home=/org.eclipse.help.base/doc/help_home.html

with 

help_home=http://www.eclipse.org
