Description coolman5453 2006-10-24 11:57:41 EDT 
I've been trying to get to the update screen for over an hour now, and I had to keep restarting the search for updates, because every mirror I would select would go slower than a snail.

Then, a few minutes ago...I thought everything was going good...because I found a mirror that was relatively quick.  I got halfway through checking for updates, then it pops up with a "there was a network problem" error, and stops scanning for updates!!!

I'm using the initial release of Eclipse 3.2.

You know...other than this issue, Eclipse has been a very stable for me.  If I knew enough about the internal workings of Eclipse, I would lend a hand at improving the performance of the update engine.

For example, the update engine could check all mirrors within your locale, and see which ones respond quicker.  Then, it would use the quickest mirror, and never have to prompt you for what site you want to download from.  Spending a half hour to tell Eclipse every 5 seconds that I want to download from the "OSU Open Source Lab" FTP site is not what I call fun...to say the least...lol.

Also, if there's a network error...it should either try again, or RE-PROMPT you for what mirror you want to download from.  Just stopping the search when a network error occurs wastes my time, and I'm sure many other users feel the same way.
Comment 1 Branko Tripkovic 2007-01-23 16:02:17 EST 
From your bug report it seems obvious that you are experiencing network problems or servers you are trying to use are experiencing problems. I fail to see what client application (update in this case)can do in either case. 
Checking all mirrors or even subset would be a waste of resource both for network and servers and on top of that would not resolve anything, since the response time you would get is meaningless. This response times change all the time depending on current network traffic and server loads. Also response time based on one GET is very unreliable since servers (or group of servers)can be set-up to be very (un)responsive for particular URL or group of urls.
Comment 2 coolman5453 2007-02-26 12:04:17 EST 
I wouldn't think it would waste resources that much, especially since SourceForge uses this method...and they have a heck of a lot more traffic:

"The SourceForge.net download system automatically chooses a download mirror that is near to you. It may choose a different mirror every time in order to balance the downloading over all our available servers."

So, I would think that some sort of load balancing could be put into place that's not resource intensive.  Or, even at the very least, put a counter of how many ftp/http connections are open to a specific update site accessing downloads...which would be updated every 5-10 minutes or so.  That way, the user can choose an update site that has a lower connection count.  Then, you would only be sending 100 bytes for 25 servers (4 bytes per server, using integers.)

And I know at my end there are no networking issues, since we're behind 3-4 T1 connections.

But, even if this couldn't be implemented, then at the very least, get a way to specify a list of default update servers to use for all updates.

I expect that if I go to update an application, that I can tell it what I want to update, and walk away from it to go get a cup of tea, or get lunch.  Having to sit there for 15 minutes to a half hour telling it what server to update from gets annoying.  I can name numerous software applications that have this functionality (yum, apt-get, YaST2 from SuSE Linux, etc.)

Also, have a way to recover from a network error instead of kicking the user out of the entire update process (like, choosing the 2nd, 3rd, 4th, ... server in their list of preferred mirrors.) ;)

If you want, I can download the Eclipse source code and see what I can do with the updater to improve performance...
