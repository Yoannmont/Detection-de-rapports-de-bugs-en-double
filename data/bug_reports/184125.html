Description Rob Frost 2007-04-25 16:54:47 EDT 
The classpath entry publish/export support does not currently allow project cp entries to be marked as dependencies. This has been done during 2.0 because project references are currently handled by WTP via other mechanisms (the Web App Libraries and EAR Libraries cp containers, component references are generatd via other mechanisms). Until these can be reconciled, project references will need to continue to be managed via the existing WTP mechanisms.

Comment 4 Rob Frost 2008-03-10 15:49:09 EDT 
I've have encountered a number of issues during the detailed exploration/prototyping of this enhancement which, due to their associated risk/complexity, are going to necessitate pushing this out of 3.0:

-If an arbitrary project classpath entry can be marked as a dependency, you can add dependencies to non-WTP projects (e.g. a plain Java project which does not have a defined virtual component mapping, facet configuration, target runtime, etc.) Note: if you limited such support to just WTP projects, you don't need a project cp entry and can instead use the current J2EE Module Dependencies UI (the exception being dynamic dependency generation via a cp container).

-This mixing of WTP and non-WTP projects will likely introduce a host of functional and user issues surrounding inconsistency of support for facets, virtual component mappings, target runtime, etc.

-How does one handle the classpath entries in a referenced project which is just a plain Java project? Since this is just a plain Java project, support will not exist for marking classpath entries for publish/export, however, many of these libraries should be included in the published/exported module. Either ignoring or allowing all such secondary dependencies is therefore problematic (and if you allow them, how is the transitive case handled?). The best alternative seems to be not supporting such secondary references.

-Project cp entry-based mixing could result in some odd combinations (e.g. a dynamic web project bundling another dynamic web project inside it's WEB-INF/lib). Using just the current J2EE module dependency tooling constrains the inter-project dependency types.

Some very restricted version of this support may ultimately be the most feasible but will need to weight the implementation cost vs benefit.

So, I think we want to continue to use the current Java EE module dependency tooling to manage the dependencies between module projects for 3.0 and revisit these issues for the next release.

If classpath containers were available to the WST containers then I could make the change in the library and have it reflected in the running container very quickly. 

Is there a better way to do this?

I was all set to try this out but then I realised what the problem would be - the dependencies in the child jar would not be included in the web app.

e.g. Let's say I have a workspace with a WebApp project and a DataAccess project. In the DataAccess project, I specify using ivy or maven that the project depends on the JDBC driver and the Spring JDBC library (which then depends on Spring Core). Ivy knows about these dependencies, downloads them and puts them into a classpath container for me.

In the WebApp project, I now specify that I want the DataAccess project. Ivy does all the work and brings the JDBC driver, Spring JDBC, Spring Core, etc into the classpath container in my WebApp project. Ivy and Maven are also smart enough to link the DataAccess cp entry to the actual project in the workspace.

However, because of this bug, that link doesn't work for WTP - only for executing plain Java.
