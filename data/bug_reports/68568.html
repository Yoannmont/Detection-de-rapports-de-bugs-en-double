Comment 11 Billy Biggs 2004-07-12 15:21:28 EDT 
There seems to be a bug in jface that is also copied in other parts of the UI
code.  Button width and height is derived from the font size plus some internal
constants, but should never allow the button to be smaller than its minimum
size.  This check is being done on the width but not on the height.  The quick
fix is to make sure we check against the height everywhere.

That said, the reason our own size code is too small is simply because these
themes add a lot of padding around their buttons.  Nuvola for example uses a
GTK+ feature which adds extra padding around "default buttons".  The code which
calculates the size in DLUs should probably discount this extra padding, but to
get this size we probably need a stronger API from SWT.

Comment 18 Billy Biggs 2004-08-17 10:53:49 EDT 
A visual style in dialogs is to ensure that all buttons have the same height. 
The Windows User Experience guidelines suggest 14 "dialog units" as the standard
height for buttons, the Mac OS X style guidelines specify 20 pixels for their
height, and the GNOME HIG 2.0 specifies that all buttons in any dialog should
maintain the same height.  JFace follows the Windows recommendation of 14 dialog
units (but unfortunately neglects to check against the minimum height requested
by the button).

If we always went for the exact size returned by ComputeSize, we could end up
with buttons in dialogs not all having the same height, depending on the icons
or text used or misused by different contributing pages in the dialog.  That
said, our current scheme of using dialog units for the height seems to be
breaking down when buttons have trim.

Since this is a pretty major problem under some themes, I did some investigation
into whether we can work around the theme settings for button padding.
Comment 19 Billy Biggs 2004-08-17 10:58:54 EDT 
Created attachment 14007 [details]
Image showing the effects of the GTK+ button border settings

This image shows two of the "difficult" theme engines: "smooth" and
"industrial".  There are two settings commonly used by these themes,
GtkButton::default_border and GtkButton::default_outside_border.  In many
"smooth"-based themes, these are set to give 10 pixels of padding around all
buttons, ballooning their size.  Industrial is similar, its shading for default
buttons seems to require more space.

If we set the padding values all back to 0 like most themes use, our buttons
become legible, but at the cost of the look of the "default button".  In the
smooth-based theme, the default button indicator triangle disappears when I got
rid of the padding.  With the industrial theme, the default button looks
slightly shrunken.

That said, this result seems a lot better than having unreadable buttons.  I
have not yet found a theme that gives a broken result with these hacked
settings, although one may exist.  This workaround may be "good enough" if we
want to solve this problem in 3.0.1.

Comment 30 Billy Biggs 2004-09-11 21:42:24 EDT 
Thanks Steve, looking at the Motif code revealed an inconsistency in how
Button.computeSize works between GTK+ and Motif.  While the JFace code in 3.0
will still break on buttons that add a lot of internal padding, fixing the GTK+
code for computeSize to include the default button padding works around this
problem.  For a discussion and patch, see bug 73717.

Unfortunately, I am unsure if this solution is acceptable for applying in the
3.0 stream, as it changes the behaviour of all buttons.  Still, it solves an
inconsistency between Motif and GTK+, and will not affect users unless they are
using these themes which add extra padding.

For 3.1 I think we will either require some change in SWT or simply deprecate
the button height field in IDialogConstants.  The patches to UI code posted to
this bug break button sizes for OS X users (bug 73027).
