Comment 7 Dejan Glozic 2003-11-26 18:02:28 EST 
I think that is unlikely. We have a table of plug-ins we know about, and table 
keys are plug-in IDs. Each table entry has two references - one for the 
workspace plug-in, another for the external plug-in (one may be null). Most 
operations in PDE are based on this table and only one of the two possible 
models for each entry is used (workspace first, if null, then external). 
Therefore, unless there is a subtle bug somewhere, workspace plug-in 
should 'mask' the external one all the time.

Comment 22 Thomas Watson 2004-03-22 22:35:58 EST 
I have not attempted to reproduce this problem on eclipse but have run into 
similar problems when running applications on OSGi.  There are several 
situations that can cause this problem.  All of them involve a Thread that was 
started by a bundle and that thread continues to run and load classes after 
the bundle has been stopped and the classloader for the bundle has been thrown 
away.

When the framework is shutting down all bundles are stopped and then the 
classloaders are closed.  If a thread is still running after the classloaders 
have been closed and discarded then ClassNotFoundExceptions will start to be 
thrown.  When a bundle is stopped it should ensure that all threads that it 
started are terminated.  In the case of background processes the plugin that 
takes care of the background processes should probably join with the 
background processes when stopping.

Similar problems could happen when updating a bundle.  When a bundle is 
refreshed all dependent bundles classloaders will be closed and discarded.  If 
there are any rogue processes from the refreshed bundles then 
ClassNotFoundExceptions will start to be thrown if these threads try to load a 
class.
Comment 23 Jeff McAffer 2004-03-22 22:51:00 EST 
great summary.  I would add that in the update case, the dependent bundles are 
also stop()'d before they are trashed so well written plugins that clean up

While the current behaviour is likely correct, is there any way to soften the

Comment 24 Thomas Watson 2004-04-19 11:25:29 EDT 
We may be able to add some code to the Framework to prevent closing the 
ClassLoaders during shutdown.  This will involve adding a check to the bundle 
unload code to see if the framework is shutting down.  If it is shutting down 
then don't bother closing the ClassLoader's.  

I do not really like suggesting this because the Framework is designed to be 
able to shutdown and relaunch without restarting the VM.  If we do not ensure 
all files are closed when we are shutdown then we will have problems 
relaunching.  If a fix like this goes in I will likely need to define a java 
property to enable it and keep it disabled by default.

This will not solve the update/uninstall/refresh case where we have to close 
the classloaders inorder to try and GC them.

Comment 27 Thomas Watson 2004-05-06 14:44:19 EDT 
The problem is not when the bundles are stopped.  When a bundle is stopped the 
classloader must be left open and continue to load classes and resources.  
This is specified in the OSGi spec.

Classloaders are only closed by the Framework when we are trying to clean up 
the resources for a bundle.  When a bundle is updated or uninstalled we 
attempt to clean up the resources of the "old" bundle (we only do this if we 
can guarantee that no other bundles depend on our shared resources).  This 
first involves stopping the bundle and then closing all of its resources so 
Garbage Collection can reclaim the objects.

If all bundles stop the threads they may have started when Bundle.stop is 
called then there should be no threads left in the system that are using the 
bundle's classloader when we attempt to clean up the bundle classloaders.

I do not think we should prevent classloaders from being closed on bundle 
updates/uninstalls.  But we should be able to prevent such clean up when the 
Framework is shutting down.

I have checked a fix into HEAD to prevent classloaders from being closed on 
framework shutdown.  Please be aware that if your bundle starts any threads 
but does not ensure the threads are stopped when your bundle is stopped then 
you may run into issues when your bundle is updated or uninstalled since we 
still close classloaders in these cases.
