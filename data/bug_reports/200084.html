Description Nick Boldt 2007-08-15 14:54:54 EDT 
Issues with large install bases and permgen OutOfMemory errors are frequently reported in IRC [1] and are now being blogged, too [2]. 

[1]http://wiki.eclipse.org/IRC_FAQ#I.27m_having_memory.2C_heap.2C_or_permgen_problems.2C_what_can_I_do.3F
[2]http://ketan.padegaonkar.name/2007/08/15/how-much-memory-is-good-enough-memory.html

This suggests two things:

a) it's an annoyance for a lot of people

b) it's becoming a big enough annoyance that people are willing to shame-blog about it, in the hope that it might go away. Since Eclipse installs are likely to INCREASE in size, not decrease, this issue isn't going to go away magically on its own -- in fact, I bet it'll be even more pervasive with Ganymede's release next year. Products can work around this by shipping mod'd eclipse.ini's in their zips, but for people just using Eclipse or Europa with umpteen plugins added in, there ought to be a simple way for them to edit their config without having to resort to hacking text files or using commandline flags.

So, perhaps the default value should be increased from 64M (set by the JVM) to 128M or 256M (set by the out-of-box eclipse.ini) for Eclipse 3.3.1 and 3.4.0?

Or, could the process that starts eclipse after installing plugins have some way to calculate how many plugins it's loading at startup, and if that value is greater than a threshhold value (say, 25?) tell the user something like:

  ---------------------------------------------
  |                                           |
  |  You have 100 plugins that will load when |
  |  Eclipse starts up. If you have memory    |
  |  problems, you should increase the        |
  |  memory allocated for Eclipse to use. For |
  |  details, see >Memory & Performance<.     |
  |                                           |
  |                  [ OK ]                   |
  |                                           |
  ---------------------------------------------

(where >Memory & Performance< is a clickable link that opens the Eclipse Help entry on PermGen / Heap or the FAQ or perhaps even a Preferences page.)

Or, this could be a dialog prompting the user to increase the permgen and write that value directly into their eclipse.ini, then prompt to restart:

  -----------------------------------------------
  |                                             |
  |  You have 100 plugins that will load when   |
  |  Eclipse starts up, but are only allocating |
  |  [64M____] of memory for Eclipse to use.    |
  |  After increasing the above value, you will |
  |  need to restart Eclipse.                   |
  |                                             |
  |     [ Save Changes ] [ Save & Restart ]     |
  |                                             |
  -----------------------------------------------

I'd also suggest adding this option into Window > Preferences... > General, Window > Preferences... > Startup and Shutdown, or a new page for Window > Preferences... > Memory & Performance so that users can find it within Eclipse rather than having to search the web or IRC for a solution.

Comment 3 Ketan Padegaonkar 2007-08-16 01:51:49 EDT 
Nick, I agree with your comment on bug 92250.

I feel that it would be a good idea to have a second look at how jvm options are configured in eclipse.ini. This would help a lot of folks the initial frustration of having to reconfigure eclipse.ini everytime they move to a new eclipse. I for instance use the latest milestone release for development, and almost always forget to change eclipse.ini.

I'd like to propose an eclipse.ini that looks something like this to configure the jvm:

[sun-1.5]
-vmargs
-XX:MaxPermGenSize=512m
-Xms=512m
-Xmx=2g
-server

[jrockit-1.5]
Some jrockit jre specific params

[ibm-1.5]
some IBM jre specific params

Comment 8 John Arthorne 2007-08-16 09:59:53 EDT 
In Eclipse 3.3, the launcher attempts to detect the presence of a Sun VM, and inserts the perm gen argument based on the value of 
--launcher.XXMaxPermSize. However, it's very difficult in the general case to detect the vendor of a particular VM prior to launching it (across the large number of permutations of vendor/version/os). Since there were objections to marking as dup of bug 92250, I'm picking another bug to mark as duplicate of. However, it's all the same issue - trying to work around the bug in the Sun VM that it arbitrarily limits the size of its class space, and provides no standard VM argument for tuning it.

I fail to see the relevance of your comment:

1. You can run Eclipse with a different Java runtime that you use to develop your projects.
2. You can use the standard API Runtime.freeMemory() to monitor the head with any Java VM
3. The problems with CodeCache and PermGen are Sun specific. So a user can either choose another VM or they can use 1.5 to run the VM (so they detect OOM before Eclipse crashes) and 1.4 for their projects.
4. If all else fails, they have to run with 1.4 and will have to live with Eclipse crashing every now and then.

> +1 on this, in addition it would be great if we could add "some" default values
> for these parameters in the eclipse.ini for certain popular and supported JDKs
> to start with.

The problem is to detect which VM it will be without starting it. "C:\prg\jdk-1.5\bin\java.exe" doesn't actually give you any idea.

My idea was to start the VM and then note the information about it in a cached file (like which EXE was used, size and timestamp). After the crash (or when it gets tight and Eclipse can find out), the user gets the dialog so she can modify eclipse.ini for the next try.

Another attempt would be to run the VM twice when you notice a change. Running it with -version isn't that expensive and the resulting String is not too bad to parse.
