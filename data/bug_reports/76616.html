Description Jane Fung 2004-10-19 18:30:17 EDT 
JDK Environment: J9 Refresh 1  

When there is a swtich case, stepping always step to the "break;" statement 
regardless if I have configured the SMAP to step there.  The worse is that not 
only it steps to the break; statement, it would always return a line number 
from the last line from the SMAP file.  Usually, when a java line is not mapped 
to any source line, we would receive a -1 for the line.  But in this case, it 
is very misleading, since it just return whatever the last line in the SMAP 
file.  The last line in the SMAP could have absolutely no relationship to where 
we have stopped.

The correct behaviour is that if I didn't define a line in the SMAP to stop on 
the break; inside a switch case.  It should not have stopped there. And it 
should return -1 for the line that doesn't have a mapping in the SMAP file.

Here is an example:

17		int i=0;
18		switch (i) {
19			case 0:
20			System.out.println ("line1");
21			System.out.println ("line2");
22			System.out.println ("line3");
23			break;
24		case 1:
25			System.out.println ("line4");
26			System.out.println ("line5");
27			break;
28		}

Here is my smap
SMAP
TestClass2.java
JANE
*S JANE
*F
1 TestClass2.jane
*L
333#1,1:20,1
334#1,1:22,1
777#1,1:26,1
*E

What's happening now is...
I installed a breakpoint at source line 333.  It pops.  That's fine.
Then I step over.  It go to source line 334 (= java line 22).  This is fine as 
well.  It correctly skips one line in the stepping.
Then I step over again.  Since case 1 will never get executed, the execution of 
the java class should stop.  However, it would stop on java line 23 (which is 
the break; statement) and then return 777.  This is incorrect.

Comment 9 Jared Burns 2004-12-09 19:31:09 EST 
We were able to reproduce the behavior with the given workspaces. What you're
seeing is the correct behavior based on the generated classfiles. As you can see
below, the compiler inserts an implicit "return" instruction at the end of the
method (line 29).

When we tell the VM to STEP_OVER, it runs until it hits a different line in the
source. The VM is actually stopping on line 29 (the implicit "return"), not line
26. You see "777" in your test case because that's the closest line available in
the SMAP. If you add a mapping from line 999 to line 29, you'll see that the
stepping goes 333, 334, 999.

Comment 14 Jane Fung 2004-12-10 11:58:36 EST 
The purpose of SMAP is to map non-java source file to the translated version of 
the Java file.  During the translation process, there would be lines in the 
java file that the user should not see.  How we identify which lines in the 
real Java files are relavant to customer and which lines are not are thru the 
SMAP file.  We define all the lines that have a valid mappings in the source 
and skip all lines that are not relevant.  That means a user cannot set 
breakpoint on those lines that are not relevant.  And also, during stepping, 
the lines that are not relevant should not be stepped too.

Having said that, I believe you should not just return -1.  You should not even 
stop on the break; line.  The program should keep running until the next valid 
(or relevant) SMAP line mapping.  Thanks.
Comment 15 Luc Bourlier 2004-12-10 13:32:46 EST 
We have no direct influence of where the VM suspend the execution.
On a step over, we are asking the VM to execute the code until it reaches an
instruction mapped, in the current stratum, to a different line.
In your test case, the VM seems to think that line 29 in the java code is mapped
to a different line (compare to the one the step started from), so it suspend
the execution.

If you think the behavoir of the VM is not correct, you should discuss the
problem with the VM developpers.

What we can do (and should) in the Eclipse debugger, is returning -1 (unknow
location) instead of an invalid value, when there is no direct mapping between 
a line in the Java code and the original source.

Btw, the step in and step out actions are likely to create the same type of
problem, as they are always performed in the Java context (the stratum system
has no notion of method).
Comment 16 Jared Burns 2004-12-10 15:18:18 EST 
Fixed (our bug) with change to ReferenceTypeImpl. We'll now correctly return
LINE_NR_NOT_AVAILABLE when the VM suspends on a line that isn't mapped in the SMAP.

To test this, use the workspaces provided by Jane and edit the last parameter of
the line that reads "777(...)" from "1" to "10". With 1 (which says that line 26
maps to 777), you'll get "no line information" when the VM suspends on line 29
(the return statement inserted by the compiler).  With 10 (which says that lines
26 to 36 map to 777), you'll get "777" when the VM suspends on line 29.

Please verify, DW.
