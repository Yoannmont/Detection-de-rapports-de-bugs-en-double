Description Uli Bubenheimer 2003-06-03 11:29:17 EDT 
When I uncheck "use default classpath" in the run/debug configuration for my 
code, I run into problems when using the SunJCE security provider; I tried it 
with JRE 1.4.1_01 through 1.4.1_03, same results in all cases. I reported this 
as a follow-up to bug 25230, but it seems that no one has been monitoring it. 
Here's some sample code for illustration purposes:

import java.security.*;

import javax.crypto.KeyGenerator;

public class Main {

  public static void main(String[] args) {
    try {
      Provider provider = Security.getProvider("SunJCE");
      KeyGenerator keyGenerator = javax.crypto.KeyGenerator.getInstance
("Blowfish", provider);
      System.out.println("success");
    }
    catch (NoSuchAlgorithmException e) {
      e.printStackTrace();
    }
  }
}


With "use default classpath" checked the output is:

success


With "use default classpath" unchecked the output is:

java.lang.SecurityException: The provider SunJCE may not be signed by a 
trusted party
	at javax.crypto.SunJCE_b.a(DashoA6275)
	at javax.crypto.KeyGenerator.getInstance(DashoA6275)
	at Main.main(Main.java:10)
Exception in thread "main"

Comment 4 Uli Bubenheimer 2003-06-03 18:26:55 EDT 
The problem is related to the sunjce_provider.jar in the bootclasspath. If you 
leave it out, things work. The reason may be related to the fact that the 
class javax.crypto.SunJCE_b is defined (differently) in that jar and also in 
jce.jar. Oh, and just switching the order of the two does not help.

I don't know why the class is defined in both places, or why Eclipse regards 
both sunjce_provider.jar and jce.jar as default system libraries when setting 
up the JREs in the Preferences. My guess is that sunjce_provider.jar should 
not be included by default when setting up the JREs.
Comment 5 Uli Bubenheimer 2003-06-05 11:43:03 EDT 
When I debug the application with a default classpath (meaning no 
Xbootclasspath is specified in the command line), the system property 
sun.boot.class.path contains this value:

C:\Program Files\Java\j2re1.4.1_02\lib\rt.jar;C:\Program 
Files\Java\j2re1.4.1_02\lib\i18n.jar;C:\Program Files\Java\j2re1.4.1_02
\lib\sunrsasign.jar;C:\Program Files\Java\j2re1.4.1_02\lib\jsse.jar;C:\Program 
Files\Java\j2re1.4.1_02\lib\jce.jar;C:\Program Files\Java\j2re1.4.1_02
\lib\charsets.jar;C:\Program Files\Java\j2re1.4.1_02\classes

When compared to Eclipse's choice of default system libraries for 1.4.1 JREs
(in the JRE selection under Preferences), Eclipse is missing i18n.jar (but 
that's probably because the jar is not even there on our systems, so Eclipse 
doesn't find it; the fix to bug 25230 seems to claim that it works if it is 
there). The more crucial point, however, is that Eclipse adds four jars under 
the ext directory which are not part of Sun's default bootclasspath. I don't 
know why Eclipse adds those in the first place; it seems that Eclipse should 
leave the ext directory alone.

In addition, Eclipse's default bootclasspath misses the classes directory, 
even if it's there in the file system. Since Eclipse currently only seems able 
to have jars as part of the default system libraries, a mechanism needs to be 
added to also specify class directories; plus the classes directory needs to 
be added to the default system libraries.

Comment 14 Luc Bourlier 2003-11-03 20:28:22 EST 
We now use "-Xbootclasspath/p:" and "-Xbootclasspath/a:" to prepend and prepend
classpath element to the default boot classpath (if the default boot classpath
is in the boot classpath defined by the user).
This works only for 'standart' VMs (sun based). The problem is still there for
J9 and 1.1 VMs (see bug 46001).
I added new methods in VMRunnerConfiguration and
AbstractJavaLaunchConfigurationDelegate to compute and store the new information.
Changes in JavaAppletLaunchConfigurationDelegate,
JavaLocalApplicationLaunchConfigurationDelegate, StandardVMRunner and
StandardVMDebugger.

Comment 16 Darin Wright 2003-11-03 22:39:08 EST 
Luc, I have expanded the javadoc in VMRunnerConfiguration. As well, I have 
specified that if the new attributes (prepend, main, append) are not specified, 
that the old sinlge bootpath attribute should be honored. To support this I 
modified the (single) bootpath computation in 
AbstractJavaLaunchConfigurationDelegate.

Since not all clients using a VMRunnerConfiguration will neccessarily use the 
new attributes (for example, the JUnit launcher does not use the new attributes 
yet), honoring the old attribute (when it's the only one present) prevents 
breakage.

We should file a bug against JDT-UI to let them know about the new API for 
setting the bootpath on the VMRunnerConfiugration.

Please verify, Luc.
