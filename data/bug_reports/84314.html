Description Roland Tepp 2005-02-03 06:05:39 EST 
One of the projects I am working on is using an essentially graph structured
dependancy model and we use TableTreeViewer for browsing (drilling down) through
this graph.

The nature of the model is such that the same model element can occur in
multiple positions (paths) of the three. Current implementation of the
TreeViewer and TableTreeViewer do not support such one-to-many mapping of model
elements to widgets, leading to loads of unnecessary overhead to enshure that
correct items will get updated.

Suggestion: Element -> Item mapper should allow returning of more than one item
per model element that is associated with this element.
Comment 1 Tod Creasey 2005-02-04 08:07:53 EST 
The element mapping is used as a convenient way to look up a mapping from an 
element to an item in O(1) time. It is still possible if you setUseHashlookup 
to false - it is just much less effecient.

I can see the use of this but it would be a significant amount of work on our 
part - you may want to consider refactoring the mapping out and seeing what 
you can do with a pluggable mapping strategy.

Comment 6 Boris Bokowski 2005-11-24 11:29:36 EST 
Would it be valid to make the assumption that whenever there are two equal elements, they will have different parents?

While this restriction makes sense (and simplifies the implementation), the following would not work:

Category 1
  Method 1
    Inner Class 1
  Method 2
Category 2
  Method 1
    Inner Class 1

because "Inner Class 1" occurs two times, but the two occurrences cannot be distinguished from another by just looking at their parent ("Method 1" in both cases).
Comment 7 Dani Megert 2005-11-24 11:52:24 EST 
The scenario of an equal parent in two categories might happen, e.g. if you want to show the class of a method and as main roots have categories and methods are allowed to be in more than one category:

cat1
  C1
   m1()
cat2
  C1
   m2()

I did not yet investigate whether the new @category tag in Java 6 will support multiple categories.
Comment 8 Dirk Baeumer 2005-11-24 12:16:40 EST 
Boris, the assumption will not work for the working copy support of the package explorer. There I only make the assumption that there can't be two equal elements under the same parent.

To solve the problem with multiple elements I introduced the concept of a TreePath. Bug 76482 contains a zip with the code I added to a subclass of the tree viewer to add this support (partially <g>). May be it is of help.
Comment 9 Boris Bokowski 2005-11-24 13:08:19 EST 
Thanks Daniel and Dirk. I agree that the assumption was too restrictive.

When you need a full tree path to identify an element, lots of methods will have to be changed (or rather, variants thereof added) to take a TreePath instead of just the element. So the question is, should I clutter the API of TreeViewer with all these additional methods, or subclass it, or create a different class altogether? Any opinions/comments?

When asking for the selection, do you need to know the tree path of the selected elements? I can imagine cases where this is important.

Comment 12 Boris Bokowski 2005-11-24 13:49:20 EST 
I guess I need to gather more requirements:

1) content provider: may an element have different children depending on its parent path?
2) label/color/font/... provider: do we need new variants of all of these so that an element can look different depending on its tree path?
3) What about filtering and sorting? Will filtering and/or sorting depend on the full path of an element in the tree?
4) addTreeListener(ITreeViewerListener): should the path be available from the TreeExpansionEvent?

For all methods taking an element, I assume they would take a TreePath instead, and methods returning an element would return a TreePath. ICellModifier can probably be used without changes.

For methods taking a TreePath instead of an element (e.g. reveal, setSelection, expand, add, remove, ...), do we need best-effort "old style" methods that just take an element and try to find it?
Comment 13 Michael Van Meekeren 2005-11-24 13:52:14 EST 
In many cases an element may not be duplicated.  It might seem optimal to consider this in the API as well, such that passing null for the path for example means, do the "old way"
Comment 14 Darin Wright 2005-11-24 14:16:32 EST 
Debug has created an 'AsynchronousTreeViewer' that uses the TreePath concept and a TreeSelection (a selection of TreePath's). Our viewer supports duplicate elements. When setting a selection, the selection contains the paths to the specific elements to select.

Our content provider does not require an element to know/provide its parent - as there can be more than one parent for an element, and depending on your model, and element may not know its parent. For example, a variable does not know what objects/variables are referencing it.

In debug views, the label/color/font of our elements are the same - they are not dependant on tree paths (i.e. two different references to the same variable are rendered the same). As well, our filtering & sorting is element based, not path based.
Comment 15 Boris Bokowski 2005-11-24 15:53:44 EST 
Here is my current thinking:

- Say "no" to questions 1-4 from my comment #12, if needed, this could be added later.
- getSelection() returns a subclass of StructuredSelection that allows clients to get the selected TreePaths
- All methods that take elements as parameters continue to work (best effort in the case of multiple elements)
- If you pass in a TreePath instead of an element itself, you can select/reveal/add/remove/editElement... elements with that specific TreePath, i.e. there would be an "instanceof TreePath" in many places.
- For methods like getExpandedElements(), I would add siblings like getExpandedTreePaths().

Does this make sense?
Comment 16 Dirk Baeumer 2005-11-25 04:18:37 EST 
We have the problem that sorting sometimes depends on the parent of the elements to be sorted (this is independent of the multi element story). We added special hacks to support this right now but an official solution for that problem would be appreciated (filtering is already possible based on the parent).

One additional thing that is needed from my experience is to add a special content provider which is able to produce all tree paths for a given element. This is needed when you want to keep selections stable on viewer changes. You can't simply reselect the old element (could randomly reveal another instance of the element) nor can you reselect the tree path (might have changed). So you have to get all possible new tree pathes for an element and try to use those that are "closest" to the remembered tree paths.
