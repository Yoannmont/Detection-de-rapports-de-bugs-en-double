Comment 2 Kim Moir 2006-11-21 12:14:21 EST 
Okay, sorry :-).  I just read it quickly and thought the bug was in the wrong bucket.  Yes, we made significant changes in this script to accommodate our web pages that display drops on a platform basis.  (You can see the changes if you compare the publish.xml with 
the R3_2_1 maintenance branch).  

To the indexResults section we added
platformIdentifierToken="${platformIdentifierToken}"
platformSpecificTemplateList="${platformSpecificTemplateList}"

which is where your build failed with the ArrayIndexOutOfBoundsException.

We didn't update our build pages with this change in the maintenance branch.   I didn't know that other projects use this script specifically, I thought they just used the plugins to build, not publish. I think WTP uses the most recent basebuilder but I don't know if they use the publishing elements.
Comment 3 Nick Boldt 2006-11-22 01:43:50 EST 
So if I understand correctly, the following should work, but I'm not clear what these two new properties do.

<property name="publish.xml" value="../org.eclipse.releng.basebuilder/plugins/org.eclipse.build.tools/scripts/publish.xml"/>

<property name="indexFileName" value="index.html"/>
<property name="result" value="${buildDirectory}/${buildLabel}"/>

<ant antfile="${publish.xml}" dir="${basedir}">
  <property name="platformIdentifierToken" value="%platform%"/> <!-- new, basebuilder M3_33 -->
  <property name="platformSpecificTemplateList" value="Windows,index.html.template,index.html,Linux,index.html.template,index.html"/> <!-- new, basebuilder M3_33 -->
  <property name="dropTokenList" value="%emfruntime%,%emfsdk%,%xsdruntime%,%xsdsdk%,%allsdk%,%alltests%,%allexamples%,%allStandalone%,%allModels%"/>
  <property name="isBuildTested" value="true"/>
</ant>

Note also that I don't see any documentation on these in publish.xml. Perhaps some annotation could be added in the top comment section here:

<!-- Properties that must be passed to this script:
	buildDirectory: Path to perform the build in.  (A working directory)
	buildType:		Type of build (nightly, integration etc.)
	buildId:		Build name
	buildLabel:		<buildType>-<buildName>-<timestamp>
-->

Or perhaps the absence of these parameters could default to an "all platforms" condition so that existing non-platform-specific behaviour is preserved?

If fallback behaviour is not possible and the above code doesn't work, we might just have to simply stop using publish.xml, since we no longer depend on having or creating an index.html in the build's folder for listing zips and test results anyway. All that's now handled by our downloads/index.php pages on www.eclipse instead. 

A dynamic, yet very generic index.php in each downloads folder might be possible too, but we'd want to Phoenixize such a page and running Phoenix on download.eclipse is not the recommended approach (Phoenix should only be needed on www.eclipse).

Thoughts?
Comment 4 Kim Moir 2006-11-23 09:43:06 EST 
We have phoenix web pages on download.eclipse.org because we have internal mirrors for downloads.  We should change this but it hasn't been a priority given our other bugs.

If you can use your existing code to generate your build pages dynamically, please go ahead and do so. I wouldn't recommend using our publishing scripts as they tend to be specific to our needs.  In fact, I would just recommend using basebuilder for the stable plugins used for building, not for publishing.  We used to try to have generic scripts for publishing that other projects could use but we simply don't have time to maintain them.  We have to focus on maintaining our build and don't have time to maintain generic publishing templates that others can teams can just plugin into their build.
