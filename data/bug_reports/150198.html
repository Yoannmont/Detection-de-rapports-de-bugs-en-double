Description Simon Archer 2006-07-10 17:32:44 EDT 
I have been kicking the tires of the Configuration Admin implementation in the incubator and I believe that I've found a case where registering a ManagedService results in a dead-lock, probably caused by Configuration Admin.  

I cannot prove anything yet, since running under the control of the Eclipse debugger results in the problem not happening.  Frankly, while this is frustrating, it does support my believe that there's a dead-lock somewhere.

I'll do some more investigations and then attach my test case.
Comment 1 Simon Archer 2006-07-10 17:49:16 EDT 
Created attachment 46048 [details]
Zip containing two Eclipse plug-in projects.

Extract the zip into an existing workspace and import the projects using File > Import > Existing Projects into Workspace.

Simply launch the test case bundles, plus Configuration Admin and the necessary prerequisites.  It is recommended that you set the Start Level of the Configuration Admin bundle to 1 since my test case bundle that acquires the service does not bother tracking or listening for it.

When the timing is "just right" you should see the following messages coming out of the counter bundle's activator:

  *** About to Register ManagedService ***
  ...
  *** Registered Managed Service ***


When the deadlock happens you don't see the second message.
Comment 2 Simon Archer 2006-07-10 19:27:52 EDT 
As a sanity check, I commented-out all the synchronized blocks in my code, since it occurred to me that maybe the problem was in the test case's handing of the ManagedService updated(Dictionary) call back from Configuration Admin.  Suffice to say, the problem still exists, so I'm more confident that the problem is with the Configuration Admin bundle.  Additionally, I have run an earlier version of the test case on another implementation of Configuration Admin without problem.

The best news is that I managed to get the problem to happen under the control of the Eclipse debugger.  I shall attached a text file containing the stacks.

Comment 13 Simon Kaegi 2006-07-17 20:25:10 EDT 
The first patch prevents the deadlock however after looking at this a bit more thoroughly I think there's a race condition in the two trackers between adding a service and delivering an asynch event. It's possible (although incredibly unlikely) that an update or delete event could be lost or possibly delivered out of order.

I can't really see this coming up in regular use however I'll start working on a patch.
Comment 14 Simon Kaegi 2006-07-18 19:01:32 EDT 
Created attachment 46481 [details]
lock based approach

In this patch the block level synchronization on configuration objects has been replaced with an equivalent mutex mechanism. I found there were a few of the standard list locking situations where block-level synchronization is really akward. Unfortunately this means that a lot of the code has been modified however it's a bit misleading as I really don't believe event ordering or the various callbacks should have been altered in a meaningful way. 

...more testing required, but I've re-run your (SimonA's) testcase succesfully and have started more of my own. I'd appreciate it if you could try this patch out.
Comment 15 Simon Kaegi 2006-07-19 13:25:35 EDT 
Created attachment 46518 [details]
lock based approach (Final)

Same as last patch but cleans up a few things:
1. Adds config.checkLocked() calls when locking is assumed outside of ConfigImpl
2. move the configstore.save() calls inside the config.lock and removed the requirement for it to be synchronized.
3) moved the service return call in ManagedServiceTracker.adding to be inside the synchronized configstore block.

At this point I think the patch is final and I'll move it onto the HEAD in the incubator shortly. There's no real rush but at some point the TCK needs to be run again to make sure I haven't broken anything. (... and apologies as I'd do this myself if I could). After we've confirmed this bug can "finally" be marked FIXED.
