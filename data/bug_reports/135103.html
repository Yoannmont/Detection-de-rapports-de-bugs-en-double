Comment 16 Harm Sluiman 2009-06-09 13:42:55 EDT 
It may be appropriate to close this particular defect, but automated testing of all public api is a base principal at Eclipse, and for some time now has been a point to report on in the release reviews.
The practice is to not declare an API public unless there is a test to validate the api has not been broken from release to release.
This project had api before this practice came to be, but it is an objective the project needs to continue to close the gap on.

If not a functional JUnit, some other form of compliance test is needed.

Agreed on all points Harm.  From a development process point of view, the Test Project creates new unit/functional tests for each symptom resolved in a defect, include new public/provisional API.  Given that the project is in maintenance mode, the amount of new function/API is minimal. Additionally, focus has been given over the last tow major releases on automating our existing manual tests to grow our bucket of automated/repeatable tests, some of which (e.g. JUnits) are now ran with every build.  That said, given that we have 1 virtual person currently working on the Test Project, our current defect backlog, and quantity of public API not covered by unit/functional tests, we cannot complete this defect.  I propose we continue with our current development process to continue to close the gap on untested public API.
