Comment 1 John Arthorne 2005-11-30 11:40:12 EST 
The message is a bit misleading... "crash was detected" means that changes were made to the workspace after it was saved.  This can happen if a plugin modifies the workspace in its plugin's shutdown() or stop() method. So, this could be a problem in the WebTools or mylar plugin.  Can you test the following:

1) Shutdown Eclipse in the Resource perspective, with no Mylar or WebTools views open.
2) Restart Eclipse, wait for initialization to complete, and then shutdown.
3) See if you get the message.

Also note that I believe WebTools M9 is built and tested against Eclipse 3.1.1 release, rather than 3.2 M3. That's not likely the source of this problem, but I just wanted to note you are running a largely untested configuration of WebTools.

Comment 4 Chris Lee 2007-01-25 19:48:42 EST 
I'm seeing a similar error myself; I'm using 3.2 "Callisto" release version.

I've managed to dig into this a bit, and found out it has to do "with /workspace/.metadata/.plugins/org.eclipse.core.resources/.snap", which gets created on some occasions, but never deleted.  Eclipse treats the existence of this file on startup as that the previous run crashed unexpectedly.

The file is read by SaveManager.restoreSnapshots (called during Eclipse startup) - if the file doesn't exist, it calls initSnap (which deletes the file - kinda redundant since it doesn't exist); if it does exist, it logs that a crash occurred.

In debugging into it (starting with a breakpoint in LocalMetaArea.getSnapshotLocationFor) , the .snap file gets created when SaveManager.snapTree is called, which only gets called when SaveManager.save is called with kind=ISaveContext.SNAPSHOT (an example of when this happens is if I create a project).  

Now, it seems that the only way for this file to be deleted when it exists is for SaveManager.save to be called with kind=ISaveContext.FULL_SAVE. - Looking through the call hierarchy, for SaveManager.save, this is never called.  I also have no idea how it would be called; I've tried various ways of shutting down (click the X, File->Exit, end task, etc) and nothing.

From what I can tell, after you create a project, you'll always get the error that your Eclipse workspace "crashed" every subsequent time you start it up.
Comment 5 John Arthorne 2007-01-26 18:18:51 EST 
SaveManager.save is called by Workspace.save, which implements IWorkspace.save.  This method is called by the IDE prior to shutdown (IDEWorkbenchAdvisor.disconnectFromWorkspace).
Comment 6 Chris Lee 2007-01-26 19:40:55 EST 
Thanks for pointing that out - I've figured out why this is happening now: I still think it's a bug in Eclipse, though it's more of a design issue now.

When the ResourcesPlugin starts, it creates a Workspace object and calls Workspace.open, which is what (eventually) checks for the .snap file using the SaveManager.

However, our application doesn't run the Eclipse IDE Application - it is an RCP application.  This means that we can't (and shouldn't) use Eclipse's IDEWorkbenchAdvisor which has the disconnectFromWorkspace, calling ResourcesPlugin.getWorkspace().save

Basically, a solution to stop the message from appearing would be for our RCP application's WorkbenchAdvisor needs to tell the ResourcesPlugin.getWorkspace to save itself... but that doesn't seem like something that a client implementation of WorkbenchAdvisor should be doing / have knowledge about.
Comment 7 John Arthorne 2007-01-29 09:22:51 EST 
Your application is using the workspace, and the workspace requires saving if you want your changes to be persisted. So, your should add a call to IWorkspace#save(true, progress) in your workbench advisor.
Comment 8 Andreas Goetz 2007-03-16 08:55:42 EDT

Comment 9 John Arthorne 2007-03-20 10:08:23 EDT 
Andreas, not sure exactly what you are looking for, but some RCP example applications can be found here:

http://wiki.eclipse.org/index.php/Rich_Client_Platform#Examples

If you're looking for the code snippet to add to your rich client application to save the workspace, it's simply a matter of calling IWorkspace.save in your advisor's postShutdown method. You'll want to use ProgressMonitorDialog to run the save operation so that it gives feedback in the UI during the save.

I'm going to close this bug, because there is no concrete information here to suggest where the problem lies (if there is still in fact a problem).
Comment 10 Chris Lee 2007-03-22 13:57:09 EDT 
I think it's still a bug, but more of a low priority design bug than anything.

Basically, if:
1. the Resource plugin exists in the plugins directory,
2. the RCP app doesn't depend on the resource plugin (and thus can't call IWorkspace.save)

then the problem will present itself.

The design issue is that the *existence* of the resource plugin - even if we don't use it - causes the bug.
