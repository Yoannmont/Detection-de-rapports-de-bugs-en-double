Description Christophe Cornu 2005-03-21 14:41:34 EST 
SWT is packaged with the new plugin format since today's nightly build (bug 
88122). There are no more swt-pi.jar, swt-mozilla.jar swt.jar, just one 
org.eclipse.swt_win32.jar etc. that contains all the classes. 

The native libraries are copied by Core into a location known to only Core 
gurus :-) and the SWT launcher does not find them anymore. We get:

java.lang.UnsatisfiedLinkError: no swt-win32-3126 in java.library.path
	at java.lang.ClassLoader.loadLibrary(ClassLoader.java:1517)
	at java.lang.Runtime.loadLibrary0(Runtime.java:788)
Comment 1 Pascal Rapicault 2005-03-21 14:50:49 EST 
PDE can not rely on having the library exported somewhere, since this has
nothing to do with osgi or runtime.

If PDE wants to provide the same level of comfort than it used to, it will have
to extract the native libs somewhere. However I don't think this is a great story.

Instead I think it would be best if SWT had some "auto extraction" code embedded.
Note to Wassim, SWT has been thinking of having code that automatically extract
the libs at runtime, which would make the standalone case nicer to setup.

Comment 4 Jeff McAffer 2005-03-21 23:28:50 EST 
Yup.  the SWT launcher works if SWT is in the workspace. That shape did not 
change at all.  What we need to do for the external case is extract the 
libraries prior to launching.  The extraction is easy (we have code in OSGi 
that does this so we can "reuse").  Three issues outstanding

1) where do we put the extracted libs (and how do we manage them so they are 
reused)

2) we have to set the permissions on them for *nix systems

3) how do we know what libs there are to extract.

All seem tractible, just details to be set.

1) somewhere in the PDE area and key the location to the launch config
2) DJ wrote this code today for OSGi.  "reuse"
3) presumably we know the OS/WS/Arch we are launching for so reach into the SWT 
jar and extract all files for the current platform.  Could refine this to just 
grab .so and .dll but that might be too refined.
Comment 5 Wassim Melhem 2005-03-21 23:35:52 EST 
>1) somewhere in the PDE area and key the location to the launch config
PDE area is good.  Storing a copy of this set of natives per launch config is 
a bit too much.  Each SWT snippet has its own launch config and they have >150 
snippets.

I will store one copy of the dlls per os-ws-arch combination in the pde 
metadata.  This way all launch configs can reuse them.
When the target platform changes (ie. the timestamp of the swt JARs in the 
target changes), I would dump the cached dlls and extract the new ones.

Comment 8 Christophe Cornu 2005-03-28 11:02:56 EST 
To summarize the current situation: there are 3 scenarios for the SWT launcher.

1. SwtTest class from a java TestProject built against the org.eclipse.swt 
project available in the workspace. org.eclipse.swt.motif or relevant fragment 
is also in the workspace.

'Run as SWT application' works in that case on all platforms except Motif HP. 
HP case should be fixed by SWT in the next build.

Note to SWT: we must use Eclipse 3.1 when checking in our native libraries 
because Eclipse 3.0 CVS does not correctly set the executable flag. That's the 
case with I20050324.

2. SwtTest class from a java TestProject built against the 
org.eclipse.swt_3.1.0.jar and org.eclipse.swt_motif_3.1.0.jar

'Run as SWT application' does not work. PDE launcher needs to be corrected as 
Wassim described in comment 5. At the moment we get:
-
Djava.library.path=/somepath/eclipse/plugins/org.eclipse.swt.motif_3.1.0.jar/os
/hpux/PA_RISC

3. Variant from 2. described in comment 7

Comment 22 Wassim Melhem 2005-04-17 23:09:25 EDT 
Wake the kids and call the neighbors.  The swt launcher is open for business 
again.

Christophe, please have your team test it using this week's I-build the 
following scenarios on all platforms:
1. org.eclipse.swt and its fragment checked out from head into the workspace
2. org.eclipse.swt and its fragment NOT in the workspace
3. org.eclipse.swt and its fragment imported into the workspace via the plugin 
import wizard.

Notes:
1. The launcher and source lookup will work on both the new swt packaging 
format and the old one.
2. When we encounter a JARd fragment, PDE extracts and caches the native libs 
into the pde/ui metadata.
3. The cache is deemed state and the natives are reextracted again if the 
timestamp of the JARd fragment changes or the absolute path to the JARd 
fragment has changed.
4. When extracting resources from the JAR, PDE only extracts resources from 
the root of the JAR (ie. ones that are not in a directory).  

Looking at the different fragments, it seems that the only resources that we 
need to extract are ones whose name contains the substring 'swt'.  Christophe, 
please confirm this.

5. When extracting resources on HP, we chmod the resources with 755

6. I verified that the launcher works great on Windows.
