Description Tetsuro Asahata 2002-08-07 10:09:37 EDT 
On Windows, when I press Alt+TAB, all running programs' icon will be shown,
but Eclipse (and branded product such as WSAD) icon looks ugly.

eclipse.exe (and branded product exe file) should somehow set its large icon 
by calling win32 API like:

HICON hIcon = ::LoadImage(, , IMAGE_ICON, 32, 32, );
::SendMessage(hWnd, WM_SETICON, ICON_BIG, hIcon);

Comment 4 Tetsuro Asahata 2002-08-15 12:50:03 EDT 
I don't know how eclipse implements this function.
But usually, on Windows, icon resource can hold multiple sizes in single 
resource such as 16x16, 32x32, and 48x48. However ::LoadIcon() win32 API
always loads 16x16. Therefore, as I wrote before, program needs to use
::LoadImage(, , IMAGE_ICON, 32, 32, ); to get correct size icon handle
from the resource, and then need to set it as a big program icon.

Note that program still needs to load and set 16x16 icon 
to set a small program icon.

Comment 8 Christophe Cornu 2003-01-15 10:37:52 EST 
From MSDN, it resorts that
- An HICON handle is a handle to a single icon image
- WNDCLASSEX can be used to set the 32*32 'system large' icon (hIcon member) 
and 16*16 icon 'system small' (hIconSm)
- An exe/dll/ico file can define an icon resource, which in turn can define multiple 
icons. (confirm the 'old days souvenirs' from SN)
- The 2 icon sizes we care about here are:
1. System small - displayed in window caption
2. System large - displayed in window int the Alt-Tab dialog

Possibility: new API in Decorations - having an SWT Image internally represent a 
set of hicons is probably crazy and probably requires new api in Image too.

Reference MSDN articles:
'About Icons' and 'Icons in Win32'

Comment 15 Robert Jenks 2003-12-09 11:27:55 EST 
I would humbly ask that something like my proposed solution in bug 48123 be 
considered rather than adding setImage(Image[]).  The problem is that I have 
no idea what image size is needed on different platforms.  In my particular 
case I can generate my application icon in any size I want at runtime. I'm 
using SVG images to do this, but others may simply use a GC and drawing 
commands to do it.  With setImage(Image[]) I would have to generate a superset 
of all of the image resolutions needed for all possible platforms.

The other way to solve this is to allow developers to extend the capability of 
the imageData.scaledTo() method.  In this case the Decorations API would not 
have to change.  decorations.setImage(Image) could simply scale the image 
provided to the platform required sized.  The imageData.scaledTo() method 
could then use either the default bitmap scaling or a custom scaling which 
could simply re-render (via SVG or GC rendering) the image at the new 
resolution.

Comment 17 Robert Jenks 2003-12-10 16:05:28 EST 
With the second option the imageData.scaledTo(...) method doesn't _have_ to 
scale the image.  It could simply return an entirely different imageData.  
i.e. if I had the same image at 4 resolutons I could pass any one of them to 
decorations.setImage(Image).  decorations.setImage(Image) could then call the 
imageData.scaledTo(...) method of that image which could then either return 
one of the other 3 existing sizes or perform bitmap scaling if necessary.

Again, this solution implies that there is a mechanism for the developer to 
override the standard behavior of imageData.scaledTo(...).

If you do go with setImage(Image[]) could you provide a method to return the 
preferred image sizes?  You mentioned that the GTK api has a setImage(Image[]) 
method, is there no way to find out, before calling this, what sizes it 
prefers?  Otherwise, how can a developer determine what size images to pass to 
this method since that information is platform specific?

Adding setImage(Image[]) would be much better than what we have right now and 
does solve a majority of the problem.  I'm just hoping that you can take this 
a step further and give the app developer a way to explicitly set the correct 
size images for each platform.
Comment 18 Christophe Cornu 2003-12-10 18:16:46 EST 
fixed v>20031210

setImages(Image[]) has been added. This API is in line with what Linux window 
managers such as Gnome and KDE support - pass an array of images and they are 
responsible for deciding which image(s) fit best the different areas they 
appear into (window frame, task list etc.).

At the moment, it appears that providing 16x16 and 32x32 icons for a Shell 
provide good results on all supported platforms. The 32x32 icon is commonly 
used on (RH9) Gnome/ (RH9) KDE/Windows XP for the Alt tab menu. The 16x16 icon 
is commonly used in the task list and for the window frame on these platforms.
