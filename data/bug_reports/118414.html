Comment 1 Kim Horne 2005-11-29 09:44:35 EST 
This seems to have been introduced by synchronization changes to EventManager.  For some reason we're doing a syncExec instead of an asyncExec for the event firing in the AbstractWorkingSetManager which is inherently dangerous.  Dirk, Nick tells me that this code was written by you.  Was there a reason you chose a syncExec over an asyncExec?
Comment 2 Dirk Baeumer 2005-11-29 09:58:16 EST 
Kim, when I introduce the class AbstractWorkingSetManager I took the code as it was in WorkingSetManager for the methods I pulled up (as you can see in version 1.10 of WorkingSetManager which shipped with 3.0.).

Turning this into async exec now might cause grief to listeners since they might inspect working sets in a state that doesn't match the event. So if we change anything in firePropertyChange I am more in favour of removing the syncExec all together. However I don't know the consequences of this. May be someone of the UI team remembers why the syncExec was interoduced.
Comment 3 Kim Horne 2005-11-29 10:01:17 EST 
Well, we need either an async, sync, or Job to ensure that the method is fired from the UI thread.  When the bundle events come in we're on the OSGi thread and we can't be firing UI events from there.  

It seems like no one knows why we did sync instead of async unfortunately.
Comment 4 Dirk Baeumer 2005-11-29 10:24:14 EST 
IMO the easiest fix would be as follows:

- AbstractWorkingSetManager doesn't extend EventManager. Instead it uses
  one
- if EventManager is supposed to be subclasses then I think it shouldn't
  synchronize it methods. All it tries to avoid is a current access to the
  list of listeners. So we should sync on the list or some other objects.
  Otherwise we will always have the problem that a "larger" object than the
  EventManager is synchronized.

Another idea how to fix this would be as follows:

When an updater gets installed in getUpdater(WorkingSetDescriptor descriptor) we don't fire the event immediatelly. Instead we remeber the installed updater and fire the installed event when the synchronize block is left.

This is a semantic change as well but the notification of an installation of a updater will not result in inspecting working sets in wrong states (there is only one installation event and no uninstall event).
Comment 5 Douglas Pollock 2005-11-29 10:34:05 EST 
There's an easier fix than that.  Change all the synchronization in AbstractWorkingSetManager to synchronize on the object its modifying (updaters) rather than the whole class.

I've committed that change to CVS.
