Description Martin Oberhuber 2007-02-01 07:28:01 EST 
We should allow users to define their own Filters, where they have complete control over the interpretation of the filter string etc.

They can currently implement API interface ISystemFilter, but the base class for a default filter, SystemFilter, is currently internal. We should make the base class API in order to make it easier for users to implement filters.

Moreover, if we encourage users to derive from the base class rather than implementing the interface directly, it's easier to remain binary compatible in case we add new methods to the interface (and the base class).
Comment 1 Lothar Werzinger 2007-02-01 07:33:31 EST 
I vote for yes
Comment 2 Martin Oberhuber 2007-02-01 07:55:27 EST

Comment 3 David Dykstal 2007-02-01 12:44:39 EST 
SystemFilter is different in than other classes in that it is a persistent model object. Extension by subclassing or implementing ISystemFilter is problematic since any new attributes introduced will have to be captured by the DOM exporter and, more importantly, the exact class captured will have to be restored by the DOM importer. A generalized importer/exporter cannot do this without capturing the class information - making this somewhat fragile. EMF has this "problem".

However, SystemFilters can be typed and should be able to have property sets attached to them. Thus there should be no need to extend SystemFilter directly and reimplementing ISystemFilter should not be necessary. Moreover, instances of SystemFilter are interpreted by their owning subsystem so much of the functionality of what the filter "means" is captured there.

I can't vote "yes" on this until a mechanism is suggested for properly restoring persisted subclasses.
Comment 4 Martin Oberhuber 2007-02-01 12:57:39 EST 
Hm, I thought that a filter was sufficiently described by its filter string and all that needs to be persisted is 
* the subsystem configuration (which references a factory for the filer)
* filter type (so the factory can create the instance)
* and filter string (being read by the filter itself so it can recreate itself).

I was not aware that filters could have other properties which were persisted. API docs could state that nothing is persisted except the filter string and probably property sets.

Tobias has a working extension of RSE where he wrote his own implementation of SystemFilter and it seems to run perfectly fine. Putting him on CC so he can comment on the issue.

Dave - what kind of problems would you expect in Tobias' implementation so we can check?

Comment 6 Tobias Schwarz 2007-02-02 01:38:13 EST 
kushal,

first of all, my extended SystemFilter (lets say MyFilter henceforth) use the standard persistance mechanism from SystemFilter.
I did not overwrite any basic functionallity like get, set or addFilterString.
MyFilter also doesn't store any additional information of holds additional attributes or properties. Thw whole filter information is within the filter string.

there are multiple issues why i chose to extend SystemFilter:

1) SystemViewFilterReferenceAdapter
   to register an adapter for a filter reference is not that easy at it should be. so because it is nearly impossible to use a own filter reference (some methods in the pool managers can't be overwritten because they use a lot of private members without getters and setters), i registered an adapter to ISystemFilterReference to do some work, after e.g. a filter was deleted. to ensure better performance with MyFilter i'm able to just ask "filterRef.getReferencedFilter() instanceof MyFilter" to determine if any work should be done for this filter. when using SystemFilter, i would have to parse the filter string all the time to determine if some work has to be done.

this adapter also make the image decoration.

2) Filter String
   my filter string is much more complicated than e.g. a simple regular expression.
the filter string is xml based and is able to filter over a wide range of properties of the shown items.
for better usability, MyFilter implements methods to create and parse such filter strings and give an easy way to the user to ask for filter properties.

3) renameable, deleteable, changeable
   the decission, if MyFilter is changeable, renameable or deletable is made by the filter itself, because it strongly depends on the xml filer properties and from where the filter has been created.


How did i get use of MyFilter?

To be able to create MyFilter instead of SystemFilter i overwrote ISubSystemConfiguration.createDefaultFilterPool() and in there, i used MyFilterPool to create MyFilter when ISystemFilterContainer.createSystemFilter() is called instead of a SystemFilter.

for editing MyFilter i implemented MyFilterEditPane where the xml filter string can be created using checkboxes, radiobuttons and text fields on multiple tabs.

so i hope it is now a little bit more comprehensible why i wrote MyFilter.

best regards
tobias
Comment 7 Martin Oberhuber 2007-02-02 04:40:57 EST 
(In reply to comment #6)
Ok, let me try to summarize and clarify:

* The use-case is to have customized filter objects which perform a post-action
  when the user deletes the filter in the view (filter references some stuff 
  which can be cleaned up at that time). Also, some utility methods for parsing
  and applying filter strings are desired.
  --> Utility methods for filter parsing can also be done outside the 
      ISystemFilter, e.g. like the current LocalFileNameFilter does it. It
      seems "nicer", though, to do this right in the filter.

* Post-actions can be done by registering a subclass of
  SystemViewFilterReferenceAdapter and overriding doDelete() and its 
  counterparts for move, rename etc. But the problem was that this adapter
  could not easily find out whether it was being called for the "right" kind
  of filter.
  --> Maybe this was already a wrong approach, and 
      SystemViewFilterReferenceAdapter should be internal. It seems that if
      multiple different extenders would use that approach and each register
      their own reference adapter, they would stomp on each other (which 
      adapter would "win" for a given filter reference?)
  --> Finding out whether it is the "right" filter can perhaps be done by
      setting the filter type string.
 
So, the main questions right now are:
* Is it true that registering a custom SystemViewFilterReferenceAdapter is
  problematic? If yes, this must be internal.
* What can we do to support post-actions for filter deletion properly?
  Looks like this could be done by offering doDeleteFilter() in
  ISystemNewFilterActionConfigurator. Or is there an event being sent on
  delete filter?
* Do we want the SystemFilter "public API" for any other reasons?

Generally, I'm inclined to make as much "internal" as possible now.
Once we have 2.0 released, it will be MUCH harder to get rid of any API,
while it will always be possible to make internals stuff API later on.

Comment 10 Martin Oberhuber 2007-02-06 09:04:32 EST 
It turned finally out that Tobias could accomplish all his requirements without implementing ISystemFilter:
- identifying his own filter was possible by looking at the filter type
- post-action was possible by listening to filter remove event
- filtering was possible by separate filter class initialized from filter string

It looks like Tobias went down the wrong path by trying to extend SystemFilter, SystemFilterReferenceAdapter, SystemFilterPoolManager etc. and it would have been better, had all these classes been internal so he'd find the official methods to use easier.

I changed the subject accordingly since we now think that rather than making the SystemFilter API, the adapters etc. should be internal. Especially because if 2 plugins registered different subclasses of SystemViewFilterReferenceAdapter, they would stomp on each other.
Comment 11 David McKnight 2007-02-09 11:51:25 EST

Comment 12 Martin Oberhuber 2007-02-12 14:35:51 EST 
I thought more about internal vs. API, see
http://wiki.eclipse.org/index.php/TM_API_vs_Internal

We should make as much internal as we can, and only keep API for
- what's currently documented in ISV docs (beyond javadoc)
- what's currently used by examples or known products
- what's needed to make the API self-contained (parameters, return values).

Stuff that is needed across plugins (e.g. declared in rse.core and needed in rse.files.ui) needs to be considered specially. We can either keep it internal with ;x-friends="org.eclipse.rse..." or make it API. If in doubt, I'd rather have it internal than API.
